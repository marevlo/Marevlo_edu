MODULE 3.4 · LASSO REGRESSION (L1) Track 3 Module 3.4 5 / 12

Same penalty idea.
Sharper corners.

Lasso replaces Ridge's squared penalty with an absolute one — λ‖w‖₁ instead of λ‖w‖². A single character changes everything. The constraint ball becomes a diamond with corners on the axes, the closed form disappears, and coefficients no longer just shrink — they get pushed exactly to zero, one feature at a time. Lasso is the canonical embedded feature selector, and its algorithm is a single beautifully sharp operator: soft thresholding.

02 / One absolute-value sign breaks the closed form
The loss & why there's no closed form
Lasso looks almost identical to Ridge — only the exponent on the penalty changes from 2 to 1. But that change is fatal to the closed-form solution: the L¹ norm is not differentiable at zero, so "set ∇L = 0" no longer applies along the axes. The price of sparsity is iteration — and the iteration is the soft-thresholding update.
loss : L(w) = ‖y − Xw‖² + λ ‖w‖₁ where ‖w‖₁ = Σⱼ |wⱼ| gradient : ∇‖w‖₁ is undefined when any wⱼ = 0 ← kinks at the axes ∴ no closed form. Use coordinate descent + soft thresholding. sub-gradient condition for optimum (KKT): Xᵀⱼ(y − Xŵ) = λ · sign(ŵⱼ) when ŵⱼ ≠ 0 |Xᵀⱼ(y − Xŵ)| ≤ λ when ŵⱼ = 0 ← coordinate is "off"
The KKT picture: why some coefficients are exactly zero math · intuition

The kinks in ‖w‖₁ live exactly on the coordinate axes. At an interior point (no zero coordinates), the L1 norm is smooth and the usual ∇L = 0 condition applies — but generically those points are not optimal because the constraint ball has corners. The optimum almost always sits on one or more axes, where the L1 gradient becomes a sub-gradient — a set of valid slopes, not a single number.

at the optimum: • for j with ŵⱼ ≠ 0 : ordinary gradient condition gives a specific slope • for j with ŵⱼ = 0 : the loss gradient need only be IN the sub-gradient set [−λ, +λ] → an entire RANGE of data correlations |Xᵀⱼr| can leave ŵⱼ at 0 this "slack zone" of width 2λ is exactly the soft-threshold dead zone below

That slack is the geometric reason Lasso produces sparsity: a coefficient stays at zero whenever its data-residual correlation falls anywhere within [−λ, λ] — a whole region, not just a single point. Raise λ and the region grows, killing more coefficients. This is the same fact as the L¹-ball-with-corners picture below, viewed from the analysis side.

03 / 📐 The diamond, not the ball
The L¹ polytope — corners on the axes mean exact zeros
Same OLS loss ellipses as Module 3.3. Same origin-centered constraint ball. One difference: the ball is now a diamond — |β₁| + |β₂| ≤ t — and its corners sit precisely on the coordinate axes. Slide λ and the diamond shrinks; the loss-ellipse touch-point now slides along an edge of the diamond until it snaps onto a corner, where one coordinate becomes exactly zero. That corner-hitting is sparsity made geometric.

Why corners win — and how Ridge and Lasso compare on the same data geometry · math

The Lagrangian/constrained equivalence is the same as Ridge: for every penalty λ there's a constraint radius t such that the two formulations share an optimum. What's different is the shape of the feasible set.

L² ball ‖β‖₂ ≤ t : smooth boundary tangent to loss ellipse at a generic interior point optimum coordinates all generically nonzero (no sparsity) L¹ ball ‖β‖₁ ≤ t : polytope with 2p corners on the axes optimum almost always at a corner at a corner, (p − 1) coordinates are exactly 0 (sparsity) in higher dimensions: L¹ polytope corners are 1-dimensional EDGES, faces, etc., so the typical number of non-zero coefs is small unless data forces many on

The "almost always" is geometric, not probabilistic: only on a measure-zero set of (X, y) does the touch-point land on an edge rather than a corner. So you get sparsity for free from the algorithm — no thresholding, no feature selector wrapper. The flip side: Lasso is arbitrary when corners are equally close. If two features are perfectly correlated, the diamond's corner could be reached by zeroing either of them, and Lasso will pick one essentially at random. That's the famous Lasso instability on correlated features — and the reason Elastic Net exists (Module 3.5).

04 / The algorithm in one operator
Coordinate descent & the soft-threshold S(z, γ)
Optimize one coordinate at a time, with all others fixed. The L1 penalty makes each one-dimensional sub-problem a tiny piecewise-linear puzzle — and its solution is the soft-thresholding operator. The operator is so simple it fits on one line; the consequence is that small coordinates are crushed to exactly zero, while large coordinates are merely shifted toward it. Slide γ below and see the dead zone widen.

Deriving S(z, γ) from the 1D L1 sub-problem, plus the from-scratch loop math · code · numerics

Fix every coefficient except wⱼ and write the loss as a function of wⱼ alone. After standardizing the column (zⱼ = n), the 1D problem is a quadratic plus an absolute value — convex, piecewise quadratic with a kink at zero. Setting the sub-gradient to zero gives three cases that collapse into one formula:

1D sub-problem : min_w ½ (w − u)² + γ |w| where u = ρⱼ / zⱼ if u > γ : optimum at w = u − γ if u < −γ : optimum at w = u + γ if |u| ≤ γ: optimum at w = 0 ← the kink wins collapsed : w = sign(u) · max(|u| − γ, 0) = S(u, γ) ∴ each Lasso coordinate update is exactly one soft-threshold call.

Ridge contrast. Repeat the same exercise with γ w² instead of γ |w| and you get a smooth quadratic — its minimum is simply w = u / (1 + γ), the purple curve in the demo. Ridge multiplies; Lasso subtracts. Multiplication never produces zero from a nonzero input; subtraction-then-clamp does. That single arithmetic difference is the entire sparsity story.

# from-scratch coordinate-descent Lasso — about 12 lines import numpy as np def soft_threshold(z, gamma): return np.sign(z) * np.maximum(np.abs(z) - gamma, 0.0) def lasso_cd(X, y, lam, n_iter=200, tol=1e-6): n, p = X.shape Xs = (X - X.mean(0)) / X.std(0) # standardize yc = y - y.mean() w = np.zeros(p) z = (Xs**2).sum(axis=0) # column norms squared (= n if standardized) r = yc - Xs @ w # residual for _ in range(n_iter): w_old = w.copy() for j in range(p): rho_j = Xs[:, j] @ (r + Xs[:, j] * w[j]) # add back j's contribution w_new = soft_threshold(rho_j / z[j], lam / z[j]) r += Xs[:, j] * (w[j] - w_new) # incremental residual update w[j] = w_new if np.abs(w - w_old).max() < tol: break return w # sklearn — same algorithm, fast C implementation + LassoCV for lambda selection from sklearn.linear_model import Lasso, LassoCV, LassoLars from sklearn.preprocessing import StandardScaler from sklearn.pipeline import make_pipeline cv = make_pipeline(StandardScaler(), LassoCV(alphas=None, cv=5, max_iter=10_000)).fit(X, y) print("chosen λ:", cv[-1].alpha_, " nonzero coefs:", (cv[-1].coef_ != 0).sum())

For very wide problems (p ≫ n), LassoLars uses the LARS algorithm to compute the entire regularization path in roughly the cost of one OLS fit — exploiting the piecewise-linear structure of ŵⱼ(λ). LassoCV on top of Lasso with the default coordinate-descent solver is the right default for most use cases.

05 / The Bayesian view — Lasso as a posterior mode
Lasso = MAP under a Laplace prior
Ridge corresponded to a Gaussian prior on w. Lasso corresponds to a Laplace prior — a distribution shaped like a tent, with a sharp peak at zero and exponentially decaying tails. That sharp peak is the probabilistic statement of "I believe many coefficients should be exactly zero." The MAP of that posterior is what coordinate descent computes.
prior : wⱼ ~ Laplace(0, 1/λ) p(w) ∝ exp(−λ Σⱼ |wⱼ|) likelihood : y | X, w ~ 𝒩(Xw, σ² I) log-posterior: log p(w | X, y) ∝ −‖y − Xw‖² / (2σ²) − λ Σⱼ |wⱼ| MAP : argmax log p = argmin ‖y − Xw‖² + 2σ² λ ‖w‖₁ ← = Lasso contrast : Gaussian prior is SMOOTH at 0 → Ridge shrinks, never zero Laplace prior is SPIKED at 0 → Lasso pushes to exactly zero
GAUSSIAN PRIOR

Smooth peak at zero

"Coefficients are around zero, with quadratic decay." MAP = Ridge. Soft shrinkage everywhere; nothing ever forced exactly to zero.

LAPLACE PRIOR

Sharp peak at zero

"Most coefficients should be exactly zero, with exponential decay for the rest." Spike at zero ⇒ MAP literally lives on the axes ⇒ sparsity.

HORSESHOE / SPIKE-AND-SLAB

Even spikier priors

Modern Bayesian sparsity priors (horseshoe, spike-and-slab) generalize the Laplace tent into shapes with infinite density at zero — even more aggressive sparsity, accessible via PyMC / numpyro.

06 / The path where features turn off, one by one
The Lasso regularization path
Plot each coefficient as a function of log(λ) and you see the unique Lasso behavior: every line eventually reaches zero — and stays there as λ grows. The order in which features drop is a built-in importance ranking; the count of nonzero coefficients gives the effective model size. LARS & why the Lasso path is piecewise linear algorithm · math

The Lasso path has a remarkable structure: as λ decreases, the active set (features with nonzero coefficients) changes only at finitely many critical values of λ, called breakpoints. Between two consecutive breakpoints the active set is fixed and the coefficients evolve linearly in λ — so the entire path is piecewise linear. LARS (Least Angle Regression) exploits this directly: instead of solving a separate Lasso problem at each grid λ, it walks along the path one breakpoint at a time.

at a breakpoint: a feature either ENTERS the active set (its |Xⱼᵀr| reaches λ) or LEAVES the active set (its coefficient hits exactly 0) between breakpoints: ŵ_active(λ) evolves linearly in λ ∴ entire path computed in O(p) or fewer LARS steps — same cost as one OLS fit

Coordinate descent is the more general workhorse — it parallelizes, handles Elastic Net, and is robust to numerical glitches — but LARS is unbeatable when you need the entire path on a tall, well-conditioned problem.

07 / What Lasso isn't good at — and how to compensate
Pitfalls & production discipline
Lasso's sparsity is its strength and its weakness. Three failure modes you should recognize on sight.
PITFALL 1

Correlated features → arbitrary picks

With two strongly correlated columns, the diamond's corner could equivalently zero either one — Lasso picks one essentially at random, and small data perturbations can flip which. Fix: Elastic Net (Module 3.5) groups correlated features together.

PITFALL 2

Don't trust p-values on coefs

Classical OLS p-values assume the active set was fixed in advance — Lasso selected the active set from the data, breaking that assumption. Naïve p-values are wildly anti-conservative. Fix: post-selection inference (selective inference), stability selection, or refit OLS on the selected subset and report those CIs with caveats.

PITFALL 3

Over-pruning at large λ

Too much penalty kills genuinely useful features. The right defense is LassoCV with a wide grid — not eyeballing the path. Pair with stability selection (refit Lasso on bootstrap resamples; keep features that consistently survive).

⚠️

The scaling rule, restated

Lasso, like Ridge, penalizes the raw coefficient magnitudes. A feature measured in millimeters needs a much larger coefficient than the same quantity in meters, so it gets penalized harder. Always StandardScaler before Lasso, fit inside cross-validation (Module 2.3). The single most common Lasso bug in real code is forgetting to scale.

🧰

Reach for Lasso when…

p > n (more features than rows), you suspect many features are irrelevant, you want an interpretable sparse model with built-in selection, or you're using the chosen subset for downstream OLS refit. Skip it when correlated features should be kept as a group (use Elastic Net → 3.5), when you need every feature in the model with shared variance (use Ridge → 3.3), or when sparsity-induced instability matters (small n, important inference). The standard production recipe is StandardScaler → LassoCV with a wide alphas grid — and stability selection if the chosen subset will be acted on.
