Why "linear in parameters" is the entire point — and a kernel preview math · geometry · code

The model ŷ = β₀ + β₁ x + β₂ x² + … + βₐ xᵈ looks nonlinear in x — and it is. But it is perfectly linear in β. That single fact is the reason all of OLS works without modification: the loss ‖y − Xβ‖² is quadratic in β, its gradient is linear in β, and the same normal equation pops out.

define φ(x) = [1, x, x², …, xᵈ] model ŷ(x) = β · φ(x) loss L(β) = ‖y − Φ(X)β‖² where Φ stacks φ(xᵢ) row-wise ∇L = 0 ⇒ Φᵀ Φ β = Φᵀ y ← literally the normal equation, on φ(x)

The geometric punchline. The curve in input space {(x, ŷ(x))} is a hyperplane in feature space {(φ(x), ŷ)} — flat, but living in a higher dimension. OLS is fitting a flat surface; the bend you see in input space is only the shadow of that flatness when projected back down. This is the trick kernels generalize (Module 3.3's Kernel Ridge, and SVMs later): a kernel K(x, x′) computes the dot product φ(x) · φ(x′) in a possibly infinite-dimensional feature space — without ever materializing φ.

import numpy as np from sklearn.preprocessing import PolynomialFeatures from sklearn.linear_model import LinearRegression from sklearn.pipeline import make_pipeline # the entire algorithm — feature lift, then OLS poly_ols = make_pipeline( PolynomialFeatures(degree=3, include_bias=False), LinearRegression(), ) poly_ols.fit(X_train, y_train) # X_train is (n, p_inputs) # from scratch: same machinery def vandermonde(x, d): """Φ for univariate input — columns x^0 … x^d.""" return np.vander(x, d + 1, increasing=True) Phi = vandermonde(x_train, d=3) w = np.linalg.solve(Phi.T @ Phi, Phi.T @ y_train) # reuse 3.1's solver

One numerical note: a raw Vandermonde of x on [0, 10] at d = 10 has a condition number near 10¹⁰ — disastrous for the normal equation. Standardize x to [−1, 1] before building Φ, or use orthogonal polynomial bases (Legendre, Chebyshev) for production work. PolynomialFeatures standardizes for you when wrapped in a scaling pipeline.

03 / 📐 Why a curve in x-space is a flat thing in φ-space
The lift — and the kernel that generalizes it
There's a single geometric idea behind every polynomial regression, every kernel method, and every basis-function model: lift the input through some φ into a higher-dimensional feature space where the relationship becomes linear, fit OLS there, then project the result back. Polynomial regression is the most concrete instance of that idea — and the easiest one to picture.
INPUT SPACE

The curve you see

In (x, y), the fit looks like a parabola, a cubic, a wave. Genuinely nonlinear in x — that's the geometry you encounter when you plot data.

ŷ(x) = β₀ + β₁x + β₂x²
FEATURE SPACE

The flat thing it really is

Treat x² as its own axis. In (x, x², y), the same fit is a plane — linear in every coordinate. The bend was an artifact of viewing the plane edge-on through x.

ŷ = β · φ(x),  φ(x) = [1, x, x²]
KERNEL TRICK (PREVIEW)

Skip φ, keep its dot products

Many algorithms only need φ(x) · φ(x′). A kernel K(x, x′) computes that directly, even when φ is infinite-dimensional — the same lift idea, taken to its limit. The polynomial kernel is literally this section's φ.

K(x, x′) = (1 + x·x′)ᵈ
🔗

The same equation, two reads

When you write ŷ(x) = β₀ + β₁ x + β₂ x², you can think of it as a quadratic curve in x, or a linear function of the vector φ(x) = [1, x, x²]. Both readings are mathematically identical and computationally equivalent — but the second is the one that scales: it tells you that every linear-model technique applies unchanged. Regularization (Ridge in 3.3, Lasso in 3.4), Bayesian posteriors (3.9), GLM links (3.10) — they all transfer directly because OLS only ever cared about linearity in the parameters.

04 / Model selection as a U-shape
The validation curve picks d
Train R² always goes up with degree — that's the temptation. The honest signal is the held-out R²: it rises, peaks, then collapses as the polynomial starts memorizing noise. Plot both together against degree and you get the most important picture in classical model selection.

Validation curve in code — and the difference from a learning curve code · protocol

Two diagnostics often confused. A validation curve fixes n and varies a hyperparameter — like degree above. A learning curve fixes the hyperparameter and varies n — it tells you whether more data would still help. Both should be CV-averaged, not single-split, to get honest error bars.

validation curve : y = score(model(d), data) d varies, n fixed learning curve : y = score(model_fixed, data[:k]) k varies, d fixed peak validation = bias≈variance crossover for that d
from sklearn.model_selection import validation_curve, learning_curve, GridSearchCV from sklearn.preprocessing import PolynomialFeatures from sklearn.linear_model import LinearRegression from sklearn.pipeline import Pipeline pipe = Pipeline([("poly", PolynomialFeatures()), ("ols", LinearRegression())]) # validation curve: train R² and CV R² across degrees degrees = np.arange(1, 15) tr, va = validation_curve(pipe, X, y, param_name="poly__degree", param_range=degrees, cv=5, scoring="r2") best_d = degrees[va.mean(axis=1).argmax()] # peak held-out R² # or pick directly via grid search inside cross-validation grid = GridSearchCV(pipe, {"poly__degree": degrees}, cv=5, scoring="r2").fit(X, y) print(grid.best_params_, grid.best_score_)

One rule that bears repeating: never peek at the test set to pick d. The validation set is for tuning; a separate, untouched test set is for reporting. Mixing the two reports an inflated number — the same selection-leakage failure mode as in Module 2.6.

05 / The cost of the lift
Feature-count explosion & interactions
Univariate polynomial regression of degree d uses d + 1 features — manageable. With p input variables, every cross-term and power gets its own column, and the count grows combinatorially. The same lift that was a clean idea in 1D becomes a feature-engineering decision in higher dimensions.
UNIVARIATE

d + 1 columns

Polynomial of degree d in one input: [1, x, x², …, xᵈ] — linear in d.

p = 1, d = 10 → 11 features
MULTIVARIATE — FULL

(p + d choose d) columns

All powers and all cross-terms up to degree d. Blows up fast.

p = 10, d = 3 → 286 features
INTERACTIONS ONLY

interaction_only=True

Skip pure powers (xⱼ², xⱼ³, …), keep only the cross-terms xⱼ · xₖ. Far fewer columns, often more interpretable.

p = 10, d = 2 → 55 features
degree-d polynomial features over p inputs: C(p + d, d) p = 5, d = 4 → 126 features p = 20, d = 3 → 1771 features two responses to the blow-up: (1) interaction_only drop pure powers — focus on crossings (2) regularize let L1/L2 zero or shrink useless columns (Modules 3.3-3.4)
06 / When the degree is high, tame it
Ridge as the cure for high-degree oscillation
Fix a high degree like d = 10 and look what happens at λ = 0: the fit threads every training point and oscillates wildly between them, with held-out R² in the gutter. Slide the L2 penalty up and the wild fit smooths out — without changing the model class. λ is a continuous bias-variance dial that complements d — the full story is Module 3.3.

Why high-degree OLS oscillates — and how the penalty fixes it phenomenon · code

The pathology has a name: Runge's phenomenon. Equally-spaced polynomial interpolation through many points produces violent oscillations near the boundary — the polynomial "snakes" up and down to satisfy each interior constraint, with the freedom paid for at the edges. OLS over noisy data does the same, with the additional twist that it isn't even forced through every point — it just gets close, and the slack is spent on wiggle.

at OLS: Var(ŵ) = σ²(ΦᵀΦ)⁻¹ for high d, ΦᵀΦ has tiny eigenvalues → coefficients become huge and noisy → polynomial wanders between data with Ridge: Var(ŵ) = σ²(ΦᵀΦ + λI)⁻¹(ΦᵀΦ)(ΦᵀΦ + λI)⁻¹ λI shifts the spectrum away from zero → bounded coefficients, smooth fit, slight bias

Practically: as soon as you start raising d past CV's chosen optimum, you're trading bias for variance. Once that variance comes from collinearity in the polynomial basis (which it does — high powers of x are nearly linearly dependent), the natural response is regularization, not lower degree. Polynomial + Ridge is so common that it has its own name in some literatures: regularized polynomial regression.

from sklearn.preprocessing import PolynomialFeatures, StandardScaler from sklearn.linear_model import RidgeCV from sklearn.pipeline import make_pipeline import numpy as np # the standard recipe: high degree + scaling + ridge with CV-chosen lambda poly_ridge = make_pipeline( PolynomialFeatures(degree=10, include_bias=False), StandardScaler(), # scaling MATTERS for L2 (Module 2.3) RidgeCV(alphas=np.logspace(-4, 4, 50), cv=5), ) poly_ridge.fit(X_train, y_train) print("chosen λ:", poly_ridge[-1].alpha_) print("held-out R²:", poly_ridge.score(X_test, y_test))

Three rules: (1) standardize first — L2 penalizes raw coefficients, so unscaled features get penalized unevenly. (2) use RidgeCV to pick λ by cross-validation, not by eye. (3) think in (d, λ) jointly — they're two dials on the same bias-variance surface, and the right combination is rarely the corner.

🧰

Reach for polynomial regression when…

You suspect smooth nonlinearity (a curve, not a step) and want to stay inside the linear-model toolkit so you keep its closed form, interpretability, and inference machinery. It's the right tool for low-dimensional regression with mild curvature, for feature engineering before you commit to a heavier nonlinear model, and as a sanity baseline for KNN / trees / GPs (Modules 3.7–3.11). Skip it when: there are sharp non-smoothness or thresholds (use trees), when p is large (the expansion blows up — try kernel ridge or a non-parametric model), or when you must extrapolate (polynomials behave wildly outside the training range — bound your inputs first or switch to splines).
