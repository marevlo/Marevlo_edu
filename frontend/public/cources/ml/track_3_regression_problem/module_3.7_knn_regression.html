Don't fit a function.
Just look up neighbors.

Every regression model so far has been parametric — fix a functional form, fit its coefficients, throw the data away. K-nearest-neighbors regression takes the opposite stance: keep the data, fit nothing. To predict at a new point, look up its k nearest training neighbors and return their average y. No assumptions about the function's shape, no closed form, no gradient — just distances and a sort. The trade-offs are equally stark: KNN follows arbitrary nonlinearity for free, but it pays at every prediction (lazy evaluation), suffers in high dimensions (curse of dim), and is fragile to feature scale.

02 / The rule, in one line
The prediction rule & the 2D Voronoi-like partition
For a query x, find the k training points closest in feature-space distance, and return their average target. That's the entire model. The map that this rule traces out over a 2D input space is a partition into cells — at k=1 these are the classic Voronoi cells (one per training point), and at larger k they're smoother "k-neighbor regions" where the predicted value changes only when the set of k nearest neighbors changes.
prediction rule : N_k(x) = the k training points with smallest ‖xᵢ − x‖ ŷ(x) = (1/k) · Σ_{i ∈ N_k(x)} yᵢ ← uniform-weighted KNN distance-weighted: ŷ(x) = ( Σ wᵢ · yᵢ ) / ( Σ wᵢ ) where wᵢ = 1 / ‖xᵢ − x‖ metric : Euclidean, Manhattan, Mahalanobis, cosine — choose to match your features "training" : store (X, y). no parameters to fit. "predicting" : for each query, sort distances and average — O(n) per query, brute force

The Voronoi partition, k=1 vs k>1, and why scaling matters more than for anything else geometry · code · pitfalls

At k=1, the regression surface is a step function — each training point owns the polygonal region of input space that's strictly closer to it than to any other training point. That's the Voronoi diagram. The predictor jumps discontinuously at the cell boundaries.

At k > 1, the partition is no longer Voronoi; it's the k-th order Voronoi diagram, where each cell is the set of points whose k nearest neighbors are the same set of training points. As you slide k up, the number of distinct cells decreases (more points share the same set of k nearest neighbors), and the surface gets coarser/smoother. Distance-weighting modifies this further by giving close neighbors more influence within a cell.

k = 1 : partition is Voronoi, |cells| = n, predictor piecewise-constant k = 2 : partition is 2nd-order Voronoi, |cells| ≤ n(n−1)/2 (typically much less) k = n : single cell, predictor = global mean ȳ distance-weighted: predictor smooths WITHIN cells, no longer piecewise constant weights wᵢ = 1/dᵢ blow up if a query lands ON a training point — handle the divide-by-zero

Scaling is non-negotiable. Distance-based methods are wholly dictated by which features dominate the distance metric. If x₁ ranges over 0–1 and x₂ ranges over 0–1000, the distance metric will ignore x₁ entirely. Forgetting to scale is the single most common KNN bug — and worse, it's silent: the model will look like it's working, just badly.

from sklearn.neighbors import KNeighborsRegressor from sklearn.preprocessing import StandardScaler from sklearn.pipeline import make_pipeline # The only correct way to use KNN — scale first, ALWAYS knn = make_pipeline( StandardScaler(), # or RobustScaler if outliers in features KNeighborsRegressor( n_neighbors=5, weights="distance", # or "uniform" — try both algorithm="auto", # picks ball_tree/kd_tree/brute by data shape p=2, # Minkowski p — 1=Manhattan, 2=Euclidean, ∞=Chebyshev ), ).fit(X_train, y_train) # Brute-force KNN from scratch — 6 lines, the canonical reference import numpy as np def knn_predict(X_train, y_train, X_test, k=5): # pairwise squared distances via broadcasting d2 = ((X_test[:, None, :] - X_train[None, :, :])**2).sum(axis=-1) nn = np.argsort(d2, axis=1)[:, :k] # indices of k nearest return y_train[nn].mean(axis=1)

The brute-force code above is O(n) per query, fine up to n ≈ 10⁴. For larger n, switch to KD-tree (fast for d < 20) or Ball-tree (better for higher d), both available transparently via the algorithm parameter. Beyond n ≈ 10⁶, exact NN becomes infeasible and you reach for approximate-NN libraries like FAISS or Annoy — covered in the resources.

03 / The one hyperparameter
k is the bias-variance dial
KNN's only meaningful hyperparameter is bias-variance, made tangible. At k=1, predictions are determined by a single training point each — every wiggle of training noise gets reproduced (high variance, zero bias on training). At large k, predictions average over many points, smoothing out everything including real structure (high bias, low variance). 04 / Why KNN fails in high dimensions
The curse of dimensionality, made literal
In low dimensions, "nearest neighbor" carves out a tiny region of the feature space — your closest neighbor is genuinely close. As dimensionality grows, the volume of the unit ball relative to the unit cube shrinks to zero, and all pairwise distances start to look the same: every other point is roughly equidistant from every query. "Nearest" stops meaning anything. The bars below show distances from a fixed query point to 100 random training points, sorted; as d grows, the bars equalize.

Why distance concentration kills KNN — & what to do about it theory · code

The concentration-of-measure result (Beyer et al. 1999, "When is nearest neighbor meaningful?") shows formally that under mild conditions, the ratio of the maximum distance to the minimum distance over n training points approaches 1 as d → ∞. The nearest-neighbor predictor degenerates: at high d, the "closest" training point isn't meaningfully closer than the average training point, so KNN's prediction barely depends on the query — it just averages a near-random subset.

for X ∈ [0,1]^d uniform, query at origin: E[dᵢ²] = d/3 (mean grows linearly with d) Var[dᵢ²] = 4d/45 (variance grows linearly too) CV(dᵢ²) = SD/mean ≈ √(4/45) / √(d/9) · √d → vanishes as d→∞ practical thresholds: d ≤ 10 : KNN works well, distance is informative d ≤ 30 : KNN works with care (feature selection, scaled features) d > 50 : KNN essentially fails for raw features ALWAYS : reduce dimension first (PCA, autoencoder, learned embedding) then run KNN in low-dim embedding space

Counter-measures. If the intrinsic dimension of your data is low (it lies on a low-d manifold inside a high-d ambient space — common for images, audio, text embeddings), then KNN still works in the manifold. The fix is to project onto that manifold first: PCA for linear manifolds, autoencoders / UMAP / t-SNE for nonlinear ones. After projection, KNN in the embedding is usually excellent — that's the recipe behind everything from face recognition (FaceNet + KNN) to semantic search (sentence embeddings + KNN). The lesson isn't "don't use KNN in high dimensions" — it's "use KNN after a learned dimension reduction."

# The standard high-d recipe: dimension reduction + KNN from sklearn.decomposition import PCA from sklearn.neighbors import KNeighborsRegressor from sklearn.pipeline import make_pipeline from sklearn.preprocessing import StandardScaler # Always: standardize, then reduce dim, then KNN pipe = make_pipeline( StandardScaler(), PCA(n_components=10, whiten=True), # or use a learned embedding KNeighborsRegressor(n_neighbors=5, weights="distance"), ).fit(X_train, y_train) # Diagnostic: check if distance is still informative in your embedding import numpy as np def distance_contrast(X, n_queries=50): qs = X[np.random.choice(len(X), n_queries, replace=False)] ratios = [] for q in qs: d = np.linalg.norm(X - q, axis=1) ratios.append(d[d > 0].min() / d.max()) return np.mean(ratios) # > 0.3: distance still informative # < 0.1: distances flat, KNN won't work

One more workaround: use a learned metric. Metric learning (e.g., LMNN, contrastive learning) trains a transformation of the feature space so that semantically similar points are close in the new space. Then KNN in the learned metric works even when the original Euclidean distance was uninformative — the basis for modern face recognition, image retrieval, and recommendation systems.

05 / Making prediction fast — at scale
KD-tree, Ball-tree & approximate NN
Brute force is O(n) per query: tolerable for n < 10⁴, painful by n = 10⁵, impossible by n = 10⁷. The standard speedup is a spatial index built once on the training data, then reused for every query — three flavors, each with its own sweet spot.
KD-TREE

O(log n) for d < 20

Recursively split space by axis-aligned hyperplanes alternating between dimensions. Beautiful when d is small; degrades to brute force as d grows because the bounding boxes become useless.

depth ≈ log₂ n, cost per query ≈ O(log n) for low d
BALL-TREE

Better for d ∈ [20, 100]

Recursively split data into nested hyper-spheres ("balls") instead of axis-aligned boxes. Holds up further into moderate dimensions and works with arbitrary distance metrics (not just Euclidean).

cost per query ≈ O(log n) for moderate d
APPROXIMATE NN

O(log n) at any d — exactness traded for speed

FAISS / Annoy / HNSW return "probably the nearest" neighbors orders of magnitude faster. The workhorses behind semantic search, image retrieval, and large-scale recommenders.

recall@k typically 0.95+, query in microseconds at n = 10⁹
sklearn : KNeighborsRegressor(algorithm="auto") ↳ chooses brute / kd_tree / ball_tree based on n, d, metric ↳ override with algorithm="ball_tree" for d ≥ 20 ↳ leaf_size hyperparameter controls the tree depth / brute-force cutoff at leaves production at scale : swap in FAISS or Annoy — see resources below
⚠️

Three KNN gotchas to internalize

(1) Scale features. Distance is dimensionless only if all features share a scale; otherwise the largest-scale feature dominates the metric. StandardScaler first, every time. (2) Beware categorical features. One-hot encoded categoricals make every distinct category equidistant from every other — KNN treats them as a single binary dimension instead of a meaningful grouping. Consider learned embeddings or target encoding (Module 2.4) for high-cardinality categoricals. (3) KNN doesn't extrapolate. Predictions are bounded by the training-set y range — there's no way to predict outside it. If you need extrapolation, switch to a parametric model.

🧰

Reach for KNN when…

You have low-to-moderate dimensionality (say d < 30 for raw features, any d for learned embeddings), modest data size (or you can afford fast-NN infrastructure), the function shape is unknown (you don't want to commit to linearity, polynomials, or specific nonlinearities), and you can scale features sensibly. Skip when d is large with no good embedding, when you need extrapolation outside the training range, when prediction-time latency budgets are tight without an ANN index, or when you have categorical features dominating the feature set. The natural follow-up question — "what if the regression function isn't smooth and has discontinuous splits?" — leads directly into Module 3.8's decision trees.
