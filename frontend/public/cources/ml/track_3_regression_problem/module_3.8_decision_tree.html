MODULE 3.8 · DECISION TREES · CART Track 3 Module 3.8 9 / 12

Don't average neighbors.
Carve up the input space.

KNN's predictor was a soft average over the closest training points. Decision trees take a completely different stance: recursively split feature space into axis-aligned rectangles, predict a constant on each rectangle. The split decisions are greedy — at every node, search all (feature, threshold) pairs and pick the one that maximally reduces residual variance — and the algorithm runs until a stopping rule fires. The resulting partition is interpretable as a flowchart, handles mixed feature types without preprocessing, and forms the building block for every ensemble in Track 4 (Random Forest, Gradient Boosting, XGBoost, LightGBM, CatBoost). Learn one tree well; the ensembles are then just clever combinations.

02 / The algorithm in one greedy rule
The variance-reduction split & the axis-aligned partition
At every node, look at every feature and every possible threshold; split the data into the two halves that result; pick the split that minimizes the post-split weighted variance. Recurse into each half. Stop when the node is "pure enough," too small to split, or you've hit max depth. That's it — every decision tree algorithm in mainstream use is some refinement of this loop.
at each node with samples (X, y): parent variance : Var(y) · n for every feature j, every candidate threshold t: split into L = {i : Xᵢⱼ ≤ t} and R = {i : Xᵢⱼ > t} weighted child variance : Var(y_L) · n_L + Var(y_R) · n_R gain(j, t) = parent variance − weighted child variance pick (j*, t*) = argmax gain → split, recurse prediction at leaf : ŷ(x) = ȳ over the training points in the leaf geometry : feature space partitioned into axis-aligned rectangles (boxes in d-D) each cell predicts the SAME constant across its entire extent

Below, training points are colored by y on the same dark-violet → purple → cyan ramp as Module 3.7. Slide max_depth and watch the partition grow — at depth 1 a single horizontal or vertical cut splits the plane in two, at depth 2 each half gets cut again, and so on. The cuts are always axis-aligned (greedy CART can't rotate them) and always parallel to the coordinate axes (in contrast to oblique-tree variants).

The greedy split: why it works most of the time and where it fails algorithm · code

CART's split selection is greedy: it picks the locally-best split at each node, without lookahead. This is fast — total fitting cost is roughly O(n · d · log n) for sorted features — and surprisingly competitive in practice. But it's not optimal: there exist datasets where the globally-best tree of depth k can only be reached by accepting a locally-suboptimal split early on. Finding the globally-optimal tree is NP-hard, so we live with greedy.

variance reduction at a single node: parent var · n = parent SSE = Σᵢ (yᵢ − ȳ)² = Σ_L (yᵢ − ȳ_L)² + n_L (ȳ_L − ȳ)² + Σ_R (...) + n_R (ȳ_R − ȳ)² = (child SSE) + (between-children SSE) gain = parent SSE − child SSE = between-children SSE = n_L (ȳ_L − ȳ)² + n_R (ȳ_R − ȳ)² ← maximize how different the two means are ∴ greedy CART = "find the split that pulls the two child means furthest apart"

The geometry of axis-aligned splits has a famous failure mode: data with diagonal structure. A target like y = x₁ + x₂ on [0,1]² has perfectly linear structure but CART needs an exponentially deep tree to approximate it well, because every "diagonal" must be approximated by a staircase of axis-aligned cuts. Linear regression handles this trivially; trees struggle. The fix isn't to abandon trees — it's to ensemble them (Random Forest, Gradient Boosting in Track 4) or to enable feature interactions explicitly. Single trees are best on data with sharp axis-aligned thresholds (income brackets, age cutoffs, categorical splits), worst on smoothly-varying linear structure.

# CART from scratch — the entire algorithm in 25 lines import numpy as np def best_split(X, y): n, p = X.shape parent_sse = ((y - y.mean()) ** 2).sum() best = (0, None, None) # gain, feature, threshold for j in range(p): order = np.argsort(X[:, j]) Xs, ys = X[order, j], y[order] # vectorized: cumulative sums let us compute child SSEs in O(n) sumL = np.cumsum(ys)[:-1]; nL = np.arange(1, n) sumR = ys.sum() - sumL; nR = n - nL sseL = np.cumsum(ys ** 2)[:-1] - sumL ** 2 / nL sseR = ((ys ** 2).sum() - np.cumsum(ys ** 2)[:-1]) - sumR ** 2 / nR gain = parent_sse - (sseL + sseR) gain[Xs[:-1] == Xs[1:]] = -np.inf # can't split between equal values i = gain.argmax() if gain[i] > best[0]: best = (gain[i], j, (Xs[i] + Xs[i+1]) / 2) return best # gain, feat, thresh def build_tree(X, y, depth=0, max_depth=10, min_leaf=1): if depth >= max_depth or len(y) <= 2 * min_leaf: return {"leaf": True, "value": y.mean()} gain, j, t = best_split(X, y) if gain <= 0: return {"leaf": True, "value": y.mean()} mask = X[:, j] <= t return {"leaf": False, "feature": j, "threshold": t, "left": build_tree(X[mask], y[mask], depth + 1, max_depth, min_leaf), "right": build_tree(X[~mask], y[~mask], depth + 1, max_depth, min_leaf)}

That vectorized split routine is exactly what sklearn's compiled Cython does — sort once per feature, then cumulative sums turn the inner loop into O(n) — so the entire fitting cost is dominated by the sort, hence O(n · d · log n) overall. Production trees add categorical-split logic, missing-value handling, and Min-Impurity-Decrease early stopping, but the algorithmic core is what you see above.

03 / Inside a single node
The best-split search
Pick any internal node in the tree above and you'll find this little algorithm running inside it: for the data that landed at this node, scan every feature, scan every candidate threshold along that feature, compute the variance reduction the split would produce, and keep the maximum. 04 / Grow it deep, then prune it back
Cost-complexity pruning & ccp_alpha
The cleanest way to control tree complexity isn't to stop growing early — that's pre-pruning, which can miss good splits that only pay off after a bad intermediate one. The textbook recipe is to grow the tree fully, then walk backwards collapsing subtrees whose contribution to the loss is less than a complexity penalty α per leaf. cost-complexity criterion: R_α(T) = R(T) + α · |T| R(T) = total impurity (Σ leaf SSE) |T| = number of leaves Breiman's weakest-link pruning: 1. grow tree T_max fully 2. for every internal node t, compute g(t) = (R(node-as-leaf) − R(subtree)) / (|leaves(T_t)| − 1) 3. prune the t with smallest g(t) ← weakest contribution per leaf 4. record T at this α-step, increment α threshold, repeat 5. cross-validate over the saved sequence → best α
Why this works better than pre-pruning & the cost_complexity_pruning_path API theory · code

Pre-pruning via max_depth or min_samples_leaf is greedy in the same way the split search is: it terminates branches without checking whether their descendants would have been useful. This is a real problem on data with compositional structure — e.g., a split that's nearly useless on its own but enables an excellent split one level deeper. Post-pruning grows the full tree first, lets the algorithm see all of those interactions, then collapses subtrees whose total benefit doesn't justify their complexity.

cost-complexity at one subtree: R(node as leaf) − R(subtree T_t) ← improvement from keeping the subtree ÷ (|leaves of T_t| − 1) ← per-leaf cost = g(t) ← "rate of return" per leaf prune subtrees with low g(t) first; that's the weakest-link rule. α grows in discrete jumps at each prune step ⇒ sklearn returns the full (α, tree) sequence via cost_complexity_pruning_path() ⇒ CV the α sequence to pick the best generalization complexity
from sklearn.tree import DecisionTreeRegressor import numpy as np # step 1: grow a deep tree and get the full α sequence clf = DecisionTreeRegressor(random_state=0).fit(X_train, y_train) path = clf.cost_complexity_pruning_path(X_train, y_train) ccp_alphas, impurities = path.ccp_alphas, path.impurities # step 2: refit one tree per α and cross-validate from sklearn.model_selection import cross_val_score scores = [] for a in ccp_alphas: t = DecisionTreeRegressor(random_state=0, ccp_alpha=a) scores.append(cross_val_score(t, X_train, y_train, cv=5, scoring="r2").mean()) best_alpha = ccp_alphas[np.argmax(scores)] final = DecisionTreeRegressor(random_state=0, ccp_alpha=best_alpha).fit(X_train, y_train) print(f"best α = {best_alpha:.4f}, leaves = {final.get_n_leaves()}, depth = {final.get_depth()}")

In production, the standard recipe is exactly this two-step CV: get the ccp_alphas sequence, refit one tree per α, CV-score, pick the maximum. It's slow for a single tree but cheap relative to the model-tuning budget — and it's strictly better generalization than picking max_depth by hand. The same path-based pruning is what makes single trees usable as members of an ensemble: prune individual trees lightly, then average across many of them in a Random Forest.

05 / What single trees do well, what they don't
Single-tree strengths & limits
Decision trees have a unique profile in the regression toolkit. They handle things that linear methods can't (sharp non-linearities, automatic feature interactions, mixed feature types) but lose to those same methods on the things linear methods do well (smooth structure, calibrated uncertainty, low variance). The decision panel below distills it.
STRENGTHS

What trees uniquely give you

  • Interpretable as a flowchart of ifs
  • No scaling, no encoding for categoricals
  • Handles non-linearities + interactions automatically
  • Built-in feature importance from total variance reduction
  • Fast prediction (O(log n) via the tree path)
WEAKNESSES

What you give up

  • High variance — different bootstrap → very different tree
  • Step-function predictor — no smooth extrapolation
  • Axis-aligned splits — diagonal structure needs a deep staircase
  • Bias toward features with more unique values
  • Can't extrapolate beyond training-y range
FEATURE IMPORTANCE

Total variance reduction per feature

  • For each feature j, sum the gain across all nodes that split on j
  • Normalize so importances sum to 1
  • Available as .feature_importances_
  • Caveat: biased toward high-cardinality features — use permutation importance for honesty
BRIDGE TO ENSEMBLES

Why one tree is rarely enough

  • High variance is the single weakness that averaging fixes
  • Average many independently-grown trees → Random Forest
  • Sequentially fit trees to residuals → Gradient Boosting
  • Both covered in Track 4 — and both rest entirely on this module
🧰

Reach for a single tree when…

You want maximum interpretability (a tree you can print and explain), quick exploratory understanding of a dataset, a baseline non-linear model with zero preprocessing, or a feature-importance ranking for downstream feature selection. Skip a single tree in favor of a Random Forest or Gradient Boosting (Track 4) whenever predictive performance matters more than interpretability — averaging tames the variance without breaking the rectangular-partition geometry. Skip trees entirely when your target has smooth global structure (linear, polynomial) — single trees can approximate smoothness but inefficiently, with staircase-shaped predictions.
