How you split decides
whether your score is real.

Every technique in this track — imputation, scaling, encoding, selection — is defended by the same wall: the train/test split. But a careless split tears that wall down. Shuffle a time series and you train on the future. Split grouped data row-wise and the model memorizes the group. This module is where leakage is finally made structural — and where shuffle=True has caused real production disasters.

02 / Random — and when it lies
Random vs stratified split
A random split is fine when classes are balanced and n is large. But with imbalance, chance alone can hand a fold almost no minority examples — or none at all — making that fold's metric meaningless. Stratified splitting forces every fold to mirror the global class ratio. Crank the imbalance and watch the random folds scatter while stratified holds steady.

When a random split lies — and how stratification fixes it intuition · code

With a 2% minority class and a 5-fold split, the expected minority count per fold can be a handful of examples — so ordinary sampling variance regularly produces a fold with one or zero. A metric computed on zero positives is undefined or wildly noisy, and your CV mean inherits that noise. Stratified sampling removes the variance at its source by allocating each class across folds deliberately.

E[minority per fold] = n · p_minority / K n=1000, p=0.02, K=5 → ~4 per fold → high relative variance stratified: |fold ratio − global ratio| ≈ 0 by construction
from sklearn.model_selection import train_test_split, StratifiedKFold, cross_val_score # single split: stratify keeps the class ratio in train AND test X_tr, X_te, y_tr, y_te = train_test_split(X, y, test_size=0.2, stratify=y, random_state=0) # cross-validation: StratifiedKFold is the default for classifiers in sklearn skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=0) scores = cross_val_score(pipe, X, y, cv=skf, scoring="average_precision")

Note the scoring: on imbalanced data, accuracy is the wrong metric (Module 2.0) — pair stratification with PR-AUC / average precision so both the split and the metric respect the minority class.

03 / Temporal data · never shuffle
Time-based split & the future leak
If your rows have an order in time, a random split scatters future rows into the training set — so the model "predicts" the past having already seen the future. Offline scores look brilliant; production, which only ever has the past, collapses. A time split trains on earlier data and tests on later, exactly as deployment works. Toggle the split and watch the future leak appear.

TimeSeriesSplit, the expanding window, and why the gap matters code · subtlety

The honest temporal protocol is an expanding (or rolling) window: fold 1 trains on months 1–2 and tests on month 3; fold 2 trains on months 1–3 and tests on month 4; and so on. Training data only ever lies in the past of its test data. TimeSeriesSplit implements exactly this — and crucially it does not shuffle, because shuffling would destroy the ordering the whole scheme depends on.

fold 1: train [■■........] test [..■.......] fold 2: train [■■■.......] test [...■......] fold 3: train [■■■■......] test [....■.....] train window grows, test always in the FUTURE
from sklearn.model_selection import TimeSeriesSplit, cross_val_score df = df.sort_values("timestamp") # order first — non-negotiable tscv = TimeSeriesSplit(n_splits=5, gap=24) # gap = embargo between train & test scores = cross_val_score(pipe, X, y, cv=tscv) # NEVER do this on time series: # train_test_split(X, y, shuffle=True) → leaks the future into the past

The gap (embargo) handles a subtle leak: if a feature is a 7-day rolling mean, the rows just before the test window share information with it. Leaving a gap between train and test prevents that spillover — the temporal cousin of the fit-on-train-only rule.

04 / Clustered observations · split by entity
Group split & the identity leak
When the same entity appears in many rows — a patient with repeated visits, a user with many sessions, a store with daily records — a row-wise random split puts some of that entity's rows in train and others in test. The model then "predicts" a test row by recalling its twin from training. You measure memorization, not generalization. Group splitting keeps every entity entirely on one side.

GroupKFold & why "new patient" is the only honest test code · framing

The question your model will actually face in production decides the split. If deployment means scoring new patients it has never seen, then validation must also test on entirely new patients — otherwise the offline metric answers an easier question ("can you recognize a patient you've already seen?") than the real one. Group splitting aligns the test with the deployment question.

from sklearn.model_selection import GroupKFold, GroupShuffleSplit, cross_val_score groups = df["patient_id"] # the entity that must not split gkf = GroupKFold(n_splits=5) scores = cross_val_score(pipe, X, y, groups=groups, cv=gkf) # pass groups! # single grouped split gss = GroupShuffleSplit(test_size=0.2, random_state=0) train_idx, test_idx = next(gss.split(X, y, groups)) # StratifiedGroupKFold: keep groups whole AND balance classes

The easy mistake is forgetting to pass groups= — the cross-validator silently falls back to row-wise splitting and the leak returns. Whenever an ID column repeats across rows, ask: "would this same entity appear in production's training data?" If not, it must not straddle your split.

06 / The decision
Which split, when
Read the structure of the data, in this order — the first one that applies wins.
1 · temporal?

Time-based split

  • Any time ordering → split by time, never shuffle
  • TimeSeriesSplit, add a gap/embargo
  • This rule overrides the others
2 · grouped?

Group split

  • Repeated entity (user/patient/store) → keep groups whole
  • GroupKFold / GroupShuffleSplit
  • Combine with stratification if also imbalanced
3 · imbalanced?

Stratified split

  • Skewed classes → preserve the ratio in every fold
  • StratifiedKFold, stratify=y
  • Pair with a minority-aware metric
4 · otherwise

Random split

  • Balanced, i.i.d., non-temporal, ungrouped
  • Plain KFold / train_test_split
  • Safe — and only here
🚨

Why shuffle=True has caused real production disasters

The default shuffle is harmless on i.i.d. data and catastrophic on temporal or grouped data — and nothing warns you. Models ship with stellar offline metrics, then degrade the moment real, future, unseen-entity data arrives, because the offline test was answering an easier question. The split is the last line of defense for every preprocessing step in this track: imputation (2.1), scaling (2.3), encoding (2.4), and selection (2.6) are only leak-free if the split underneath them is honest. Choose the split first, then build the pipeline inside it.
