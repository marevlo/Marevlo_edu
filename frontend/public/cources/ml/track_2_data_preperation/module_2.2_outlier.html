Far from the crowd —
but in which metric?

An outlier is a point that doesn't belong. The hard part is defining "belong." A value can look perfectly ordinary on every single axis yet sit absurdly far from the data's shape — and the only way to catch it is to measure distance the way the data is actually spread. This module is about seeing distance correctly, then deciding what to do about the points that are too far.

01 / Learning outcomes
What you should be able to do after this lesson
Treat outliers as a measurement problem before a removal problem. By the end, you should be able to choose a robust detector, measure distance in the right geometry, and pick a treatment that doesn't quietly corrupt the model or leak the split.

Pick a detector that survives contamination

02 / Univariate detection · the implementation view
Z-score · IQR · MAD
The formulas live in your stats course (🌉 Stats 1.4). What matters here is the breakdown behavior: the mean and standard deviation are themselves distorted by the outliers you're trying to find. Add contamination below and watch the Z-score's fence blow outward — masking the very points it should catch — while the median-based MAD barely flinches.

Z-score: flag |x − mean| / std > 3 ← mean & std are NOT robust IQR : flag x ∉ [Q1 − 1.5·IQR , Q3 + 1.5·IQR] ← quartile-based, robust MAD : m = median(x), MAD = median(|x − m|) flag |0.6745·(x − m) / MAD| > 3.5 ← 50% breakdown point
Breakdown point — why the median wins under contamination intuition · math · code

The breakdown point of an estimator is the fraction of the data you can replace with arbitrary garbage before the estimate can be dragged to infinity. The mean has a breakdown point of 0 — a single huge value drags it anywhere. The median has a breakdown point of 0.5 — you must corrupt half the data before it moves freely. That single fact is the whole reason MAD is robust and the Z-score is not.

mean breakdown = 0 (one bad point ruins it) median breakdown = 0.5 (needs 50% corruption) 0.6745 = Φ⁻¹(0.75) rescales MAD so that, for Gaussian data, 1 robust unit ≈ 1 standard deviation

The 0.6745 constant is the trick that makes MAD comparable to a standard deviation: for clean Gaussian data, MAD ≈ 0.6745·σ, so dividing by it converts the robust spread back into familiar "σ units." That's why the threshold 3.5 on the modified Z-score is roughly the analogue of 3σ.

# the three detectors, side by side import numpy as np def flags(x): x = np.asarray(x, dtype=float) # Z-score (non-robust): mean & std are themselves moved by outliers z = np.abs(x - x.mean()) / x.std() z_out = z > 3 # IQR (robust): based on quartiles, ignores the extreme tails q1, q3 = np.percentile(x, [25, 75]) iqr = q3 - q1 iqr_out = (x < q1 - 1.5*iqr) | (x > q3 + 1.5*iqr) # MAD (most robust): modified z-score around the median med = np.median(x) mad = np.median(np.abs(x - med)) mod_z = 0.6745 * (x - med) / mad mad_out = np.abs(mod_z) > 3.5 return z_out, iqr_out, mad_out # inject 6 extreme values into a clean N(0,1) sample and compare counts x = np.r_[np.random.randn(200), np.full(6, 12.0)] z_out, iqr_out, mad_out = flags(x) print(z_out.sum(), iqr_out.sum(), mad_out.sum()) # Z under-flags: masking

Run it and the Z-score row typically flags fewer than 6: the six injected values inflate std so much that they fall back inside the 3σ fence. The MAD row catches all six because the median and MAD never noticed them. This is masking, and it is exactly what the slider above demonstrates.

03 / 🧮 + 📐 · the multivariate centerpiece
Mahalanobis distance & the whitened space
In two or more dimensions, "far from the mean" depends on the correlation of the data. Mahalanobis distance is a Z-score measured in a space where the data has been whitened — stretched and rotated until the cloud is a unit sphere. Equidistance contours in raw space are therefore ellipsoids aligned with the covariance. Click anywhere on the plot to drop a probe and compare its Mahalanobis distance to plain Euclidean.

D_M(x) = √( (x − μ)ᵀ Σ⁻¹ (x − μ) ) Σ = LLᵀ (Cholesky) → z = L⁻¹(x − μ) is the whitened coordinate D_M(x) = ‖z‖ ⟹ Mahalanobis is just a plain Z-score after whitening
Why Σ⁻¹ turns a Z-score into an ellipse — the whitening derivation proof · geometry · code

Start from the 1-D Z-score z = (x − μ)/σ, distance measured in standard-deviation units. Square it: z² = (x−μ)·(σ²)⁻¹·(x−μ). The multivariate version simply replaces the scalar variance σ² with the covariance matrix Σ, and the reciprocal with the matrix inverse:

D_M(x)² = (x − μ)ᵀ Σ⁻¹ (x − μ)

Because Σ is symmetric positive-definite (Track 1.0), it factors as Σ = LLᵀ. Define the whitening map z = L⁻¹(x − μ). Substitute:

D_M² = (x−μ)ᵀ(LLᵀ)⁻¹(x−μ) = (L⁻¹(x−μ))ᵀ (L⁻¹(x−μ)) = zᵀz = ‖z‖²

In the whitened coordinates the distance is the ordinary Euclidean norm — the tilted ellipse in x becomes a unit sphere in z. Push the sphere ‖z‖ = c back through x = μ + Lz and you recover the ellipsoid D_M = c, its axes aligned with the eigenvectors of Σ and scaled by √(eigenvalue) — precisely the contours drawn above.

A principled threshold for free

If the cloud is roughly Gaussian, D_M² follows a chi-squared distribution with p degrees of freedom (here p = 2). That converts "how far is too far?" from a guess into a quantile.

import numpy as np from scipy.stats import chi2 def mahalanobis_sq(X, mu, Sigma): d = X - mu Si = np.linalg.inv(Sigma) # einsum does the per-row quadratic form dᵀ Σ⁻¹ d without a Python loop return np.einsum("ij,jk,ik->i", d, Si, d) mu = X.mean(axis=0) Sigma = np.cov(X, rowvar=False) dm2 = mahalanobis_sq(X, mu, Sigma) cut = chi2.ppf(0.975, df=X.shape[1]) # 97.5% cutoff for p features outliers = dm2 > cut # robust version: estimate mu, Sigma with Minimum Covariance Determinant from sklearn.covariance import EllipticEnvelope ee = EllipticEnvelope(contamination=0.05).fit(X) # MCD-based, resists masking robust_outliers = ee.predict(X) == -1

Note the last block: estimating μ and Σ from all the data lets outliers inflate the covariance and hide inside their own enlarged ellipse — the multivariate version of masking. EllipticEnvelope uses the Minimum Covariance Determinant to estimate the shape from the tightest, cleanest subset first, then measures everyone against that.

04 / Once you've found them · what next
Five ways to treat an outlier
Removal is the lazy default and often the wrong one — a true extreme value can be the most informative row in the dataset. Match the treatment to the cause: a data-entry error gets removed, a heavy tail gets transformed, a rare-but-real event gets kept and flagged.
01 · REMOVE

Drop the row

Only when you're confident it's an error (impossible value, broken sensor). Removing real extremes biases the model and shrinks the variance you're trying to learn.

02 · CAP

Winsorize

Clip values beyond a percentile to that percentile — keep the row, tame its leverage. 1st/99th is common.

x ← clip(x, q₀.₀₁, q₀.₉₉)
03 · TRANSFORM

Log / Box-Cox

A heavy right tail isn't an outlier problem, it's a scale problem. A log or Box-Cox transform pulls the tail in and often makes the extreme disappear honestly. (Full treatment → 2.3.)

y = log(x + c) | Box-Cox
04 · ROBUST MODEL

Let the model absorb it

Don't touch the data — pick an estimator that down-weights extremes: Huber loss, quantile regression, or tree models that are split-based and barely care.

05 · FLAG

Outlier indicator

Add an is_outlier column and keep the value. If "being extreme" is itself predictive (fraud, failure), you've turned a nuisance into a feature.

A right-skewed feature (income, claim size, session length) looks full of "outliers" only because of its scale. Toggle a transform and watch the long tail fold back into a near-symmetric shape — the skew, not the points, was the problem.

Winsorizing & transforms without leaking the split code · pitfall

Both capping and power transforms learn statistics from the data — a percentile, a Box-Cox λ. Learn them on the training fold only, then apply the same learned values to validation and test. Computing a 99th-percentile cap on the full dataset is the same leakage you met in 2.1, just wearing a different hat.

Winsorize, fit on train only
import numpy as np # learn caps on TRAIN lo, hi = np.percentile(X_train["income"], [1, 99]) # apply the SAME caps to every split X_train["income"] = X_train["income"].clip(lo, hi) X_test["income"] = X_test["income"].clip(lo, hi) # reuse train caps, do NOT recompute
Box-Cox / Yeo-Johnson inside a Pipeline
from sklearn.preprocessing import PowerTransformer from sklearn.pipeline import make_pipeline # Yeo-Johnson handles zeros/negatives; Box-Cox needs strictly positive x pt = PowerTransformer(method="yeo-johnson") pipe = make_pipeline(pt, model) pipe.fit(X_train, y_train) # λ is estimated on train only preds = pipe.predict(X_test) # test transformed with the train λ

The transform demo above is exactly this idea visualized: once log(1+x) reshapes the feature, the count of "3σ outliers" usually collapses — because they were never outliers in a multiplicative process, only in the wrong (linear) scale.

05 / Terminology that changes your pipeline
Outlier · anomaly · novelty
These words get used interchangeably and shouldn't be — each implies a different training setup and a different scikit-learn API. Getting the word right is half the design decision.
OUTLIER DETECTION

Bad points inside training data

Your training set is already contaminated and you want to find/clean the rogue rows within it. Unsupervised, fit on the dirty data itself. APIs: EllipticEnvelope, IsolationForest.fit_predict.

ANOMALY DETECTION

The rare event is the target

Fraud, machine failure, intrusion. The "outlier" is what you're trying to predict — usually a supervised, heavily class-imbalanced problem (→ metrics from 2.0). Don't delete it; model it.

NOVELTY DETECTION

Clean training, new point at test

Train on data known to be clean, then ask whether a future observation belongs to that distribution. Semi-supervised. APIs: OneClassSVM, LocalOutlierFactor(novelty=True).predict.

Same data, three different fits — and why the API differs distinction · code

The split between these isn't the algorithm, it's what the training set is assumed to contain. Outlier detection assumes the training data is dirty and learns the bulk anyway. Novelty detection assumes the training data is clean and judges only unseen points against it. That single assumption flips the API.

outlier : fit(dirty X) → label rows of X (unsupervised) novelty : fit(clean X) → predict(new x) (semi-supervised) anomaly : fit(X, y) with y rare (supervised, imbalanced)
from sklearn.ensemble import IsolationForest from sklearn.neighbors import LocalOutlierFactor # OUTLIER: fit & label the same (contaminated) training set iso = IsolationForest(contamination=0.05, random_state=0) labels = iso.fit_predict(X_train) # -1 = outlier, +1 = inlier # NOVELTY: fit on CLEAN data, then judge a brand-new point lof = LocalOutlierFactor(n_neighbors=20, novelty=True) lof.fit(X_clean) # must be novelty=True to call .predict later is_new_normal = lof.predict(x_new) == 1

Call LocalOutlierFactor.predict without novelty=True and scikit-learn raises an error — the library encodes this very distinction so you can't accidentally judge unseen points with a model that only learned to label its own training set.

06 / Two framings · when each applies
Distance vs density
Mahalanobis is a global distance method: one center, one covariance, one ellipse for everyone. It's perfect for a single elliptical blob — and blind to a point that's only an outlier relative to its local neighborhood. Toggle the view: the global method waves the local outlier through; a density method (LOF) catches it.

How LOF measures "locally lonely" intuition · math · code

The Local Outlier Factor compares the density around a point to the density around its neighbors. A point in a region that's much sparser than its neighbors' regions gets an LOF well above 1 — even if, in absolute terms, it sits close to a dense cluster. This is the case a single global ellipse cannot express.

lrd(p) = 1 / avg reachability-distance to p's k neighbors LOF(p) = avg( lrd(neighbor) ) / lrd(p) LOF ≈ 1 : same density as neighbors (inlier) LOF ≫ 1 : much sparser than neighbors (local outlier)

So LOF is a ratio, which is what makes it local: it never compares you to the whole dataset, only to your own neighborhood. That's why it flags the cyan point above while the global-distance view, judging everyone against one center, leaves it alone.

from sklearn.neighbors import LocalOutlierFactor # density-based: outlier-ness is relative to local neighborhood lof = LocalOutlierFactor(n_neighbors=20) labels = lof.fit_predict(X) # -1 = local outlier scores = -lof.negative_outlier_factor_ # larger = more anomalous # rule of thumb for choosing the framing # single elliptical blob, ~Gaussian -> Mahalanobis / EllipticEnvelope # many clusters or varying density -> LOF / DBSCAN # high dimensions, mixed types -> IsolationForest

Rule of thumb baked into the comment block: reach for global distance when the data is one Gaussian-ish cloud, density methods when it has structure (multiple clusters, varying spread), and Isolation Forest when dimensionality is high and you want something cheap that doesn't assume a shape at all.
