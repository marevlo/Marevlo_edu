more signal.

Every irrelevant column you keep adds variance, invites overfitting, slows training, and dilutes interpretability. Feature selection is the disciplined removal of features that don't earn their place — but the methods disagree on what "earn" means, and selecting on the target before splitting is a leakage trap in disguise.

02 / Fast, model-free, univariate
Filter methods & mutual information
Filters score each feature against the target one at a time, before any model. They're fast and great for a first cut — but blind to two things: redundancy (two copies of a good feature both score high) and interactions (two features useless alone, powerful together). Pick a metric and a cutoff and watch the redundant copies rank just as high as their originals.

Why mutual information beats correlation — and what filters still miss intuition · math · code

Correlation only sees linear association: a perfect U-shape y = x² has correlation ≈ 0, so a correlation filter would discard a perfectly predictive feature. Mutual information measures how much knowing X reduces uncertainty about Y in any form — it's 0 only when they're statistically independent.

I(X;Y) = Σ p(x,y) log( p(x,y) / (p(x)p(y)) ) = H(Y) − H(Y | X) ← "uncertainty about Y removed by knowing X" I = 0 ⇔ X ⟂ Y (independent); captures nonlinear dependence corr misses

But every filter shares two blind spots, both visible above: it scores the redundant copy as highly as the original (no notion that they overlap), and it can't reward an interaction — two features that are noise alone but predictive together — because it never looks at pairs.

from sklearn.feature_selection import (VarianceThreshold, SelectKBest, f_classif, chi2, mutual_info_classif) # drop near-constant features (target-free) X = VarianceThreshold(threshold=0.01).fit_transform(X) # keep the k best by ANOVA-F (linear) ... X_f = SelectKBest(f_classif, k=10).fit_transform(X, y) # ... or by mutual information (captures nonlinear dependence) X_mi = SelectKBest(mutual_info_classif, k=10).fit_transform(X, y) # chi-square needs NON-negative (counts / one-hot) categorical features X_chi = SelectKBest(chi2, k=10).fit_transform(X_counts, y)

Use a filter as a cheap first pass to drop obvious junk, then hand the survivors to a wrapper or embedded method that can reason about combinations. And fit the filter on training folds only — F-test and MI both touch the target.

03 / Let the model judge the subset
Wrapper methods — forward, backward, RFE
Wrappers wrap an actual model: they search over feature subsets, scoring each on held-out data. This sees redundancy and interactions a filter can't — at real compute cost. Pick a search and watch the held-out score climb as useful features are added, then plateau (the redundant copy adds nothing) and eventually dip (overfitting).

Forward selection, RFE, and letting CV choose the size code · cost

The key wrapper insight: a feature's value is conditional on what's already chosen. Forward selection adds the strongest first; once income is in, its redundant copy income_v2 gives almost no extra lift, so it never gets added — the plateau you see above. That's minimal-optimal selection: the smallest subset that does the job.

from sklearn.feature_selection import SequentialFeatureSelector, RFE, RFECV from sklearn.linear_model import LinearRegression est = LinearRegression() # forward greedy search, scored by cross-validation sfs = SequentialFeatureSelector(est, n_features_to_select="auto", direction="forward", cv=5) sfs.fit(X, y) # RFE: recursively drop the smallest-coefficient feature rfe = RFE(est, n_features_to_select=5).fit(X, y) # RFECV: let cross-validation pick HOW MANY to keep rfecv = RFECV(est, cv=5, scoring="r2").fit(X, y) print(rfecv.n_features_, rfecv.support_)

The trade-off is cost: wrappers fit the model O(p²) times. For hundreds of features, filter first to thin the candidates, then wrap the survivors. And the whole search must live inside cross-validation — selecting on the full dataset leaks the held-out rows into the chosen subset.

04 / Selection baked into training
Embedded methods — the Lasso path
Embedded methods select while they fit. The Lasso (L1 penalty, Track 3.4) is the cleanest example: as the penalty grows, coefficients are driven exactly to zero, and a zero coefficient is a dropped feature. Slide the penalty and watch features switch off in order — noise first, then a redundant copy, with the strongest features surviving longest.

Why L1 zeros out coefficients but L2 doesn't geometry · code

Picture the constraint region. L2 (Ridge) is a smooth circle; the loss contours touch it at a generic point where no coordinate is exactly zero. L1 (Lasso) is a diamond with sharp corners on the axes — and contours are most likely to first touch a corner, where one or more coefficients are precisely zero. That corner geometry is the entire reason Lasso selects and Ridge only shrinks.

Ridge ball ‖β‖₂ ≤ t : round → optimum rarely on an axis → no exact zeros Lasso ball ‖β‖₁ ≤ t : diamond → corners ON axes → exact zeros (selection)
from sklearn.linear_model import LassoCV, ElasticNetCV from sklearn.feature_selection import SelectFromModel from sklearn.ensemble import RandomForestRegressor # Lasso with CV-chosen lambda; nonzero coefs = selected features lasso = LassoCV(cv=5).fit(X_scaled, y) # SCALE first — L1 is scale-sensitive selected = X.columns[lasso.coef_ != 0] # Elastic Net: keeps correlated groups instead of picking one arbitrarily enet = ElasticNetCV(l1_ratio=0.5, cv=5).fit(X_scaled, y) # tree importance as an embedded selector (no scaling needed) sel = SelectFromModel(RandomForestRegressor(), threshold="median").fit(X, y)

Two cautions: scale before Lasso (the penalty hits raw coefficients, so unscaled features are penalized unevenly — Module 2.3), and with correlated features Lasso picks one arbitrarily and zeros the rest, which is why Elastic Net exists when you'd rather keep the whole correlated group.

05 / All-relevant selection & honest importance
Boruta, permutation importance & SHAP
Lasso and wrappers find a minimal useful set. Sometimes you want every feature with any real signal — that's all-relevant selection, and Boruta nails it with a clever trick: shuffle a copy of each feature to build a "noise ceiling," then keep only features that beat it. Note it confirms both redundant copies — that's the point.

BORUTA

All-relevant, RF-based

Compares each feature to shuffled shadows across many RF runs. Keeps everything genuinely useful — including redundant features — instead of a minimal set.

PERMUTATION

The most honest filter

Shuffle one feature on held-out data and measure how much the model's score drops. Model-agnostic, post-hoc, and it can't be fooled by inflated split importances. (Deep dive in Track 8.)

SHAP

Game-theoretic attribution

Rank and prune by mean |SHAP value| — a fair allocation of each prediction across features. Powerful and consistent; the full treatment is Track 8.

Boruta's shadow trick, permutation importance & SHAP in code code · honesty

Built-in tree importances (Gini/split-count) are biased toward high-cardinality and correlated features and can look large for noise. Permutation importance sidesteps that: break a feature by shuffling it on held-out data and see how much performance falls — if nothing falls, the feature wasn't doing real work, whatever the split-importance claimed.

shadow feature = permutation of a real column (signal destroyed) confirm feature ⇔ importance(feature) > max importance(all shadows) permutation imp = score(X) − score(X with column j shuffled) on HELD-OUT data
from boruta import BorutaPy from sklearn.ensemble import RandomForestRegressor from sklearn.inspection import permutation_importance # Boruta: all-relevant selection against shadow features rf = RandomForestRegressor(n_jobs=-1, max_depth=5) boruta = BorutaPy(rf, n_estimators="auto", random_state=0).fit(X.values, y.values) keep = X.columns[boruta.support_] # permutation importance — honest, model-agnostic, on a HOLD-OUT set rf.fit(X_train, y_train) imp = permutation_importance(rf, X_val, y_val, n_repeats=10) ranked = imp.importances_mean.argsort()[::-1] # SHAP-based pruning (full treatment in Track 8) import shap sv = shap.TreeExplainer(rf).shap_values(X_val) shap_rank = np.abs(sv).mean(axis=0).argsort()[::-1]

Choose by intent: Boruta when you must not miss any relevant feature (scientific discovery), Lasso/wrappers for the leanest predictive set, and permutation/SHAP to rank and explain what a trained model actually relies on.

06 / The discipline of doing nothing
When not to select
Feature selection is a tool, not a ritual. In several common situations it adds risk without reward — and selecting on the target outside cross-validation actively harms you.
skip it

Small p, large n

  • Few features, many rows → overfitting risk is already low
  • The cost of a bad drop outweighs the marginal gain
  • Just train on all of them
let the model do it

Regularized / tree models

  • L1/L2 and trees down-weight junk internally
  • Explicit selection often duplicates their work
  • Embedded selection is usually enough
never like this

Selection outside CV

  • Choosing features on the full dataset = leakage
  • The "best" features were picked using test rows
  • Put selection inside the cross-validation loop
🚨

The selection-leakage trap

Ranking features by their correlation with y on the whole dataset, then cross-validating the survivors, gives a wildly optimistic score — every fold's "held-out" rows already influenced which features survived. Selection that touches the target must be refit inside each fold, exactly like imputation (2.1), scaling (2.3), and target encoding (2.4).
