02 / 📐 Linear in parameters, nonlinear in inputs
Polynomial & interaction features
A quadratic relationship y ≈ βx² is hopeless for a model that only knows x — until you hand it x² as a new column. Then the curve in input space becomes a straight line in the lifted feature space. Switch the view below: the same parabola, plotted against x², is just a line. That single idea is what kernels generalize.

Interactions, the combinatorial blow-up, and the kernel connection intuition · math · code

An interaction term x₁·x₂ lets the effect of one feature depend on another — "price matters more on weekends." Polynomial expansion includes these automatically: degree-2 over (x₁, x₂) gives x₁², x₂², x₁x₂. The catch is combinatorial: degree d over p features produces on the order of C(p+d, d) terms, which explodes fast.

features after degree-d expansion of p inputs ≈ C(p+d, d) p=10, d=2 -> 66 terms p=10, d=3 -> 286 terms (and overfitting risk climbs with each)

The kernel trick is the escape hatch: a kernel like K(a,b)=(aᵀb+1)ᵈ computes the dot product in that huge polynomial space without ever materializing the columns — you get the expressive power of the lift at the cost of the original dimensionality. The polynomial features you build by hand here are the explicit, interpretable version of what an SVM does implicitly.

from sklearn.preprocessing import PolynomialFeatures from sklearn.pipeline import make_pipeline from sklearn.linear_model import Ridge # degree-2 with interactions; ridge tames the variance from extra columns model = make_pipeline( PolynomialFeatures(degree=2, include_bias=False), Ridge(alpha=1.0), # regularize — expansion adds variance ) model.fit(X_train, y_train) # interactions ONLY (no x^2 powers), useful when you suspect crossing effects PolynomialFeatures(degree=2, interaction_only=True, include_bias=False)

Notice the Ridge step: every column you add increases variance, so polynomial expansion and regularization (Track 3.4) almost always travel together. The val-R² readout above is the live version of that lesson — push the degree up and watch train R² climb while val R² collapses.

03 / Discretizing a continuous feature
Binning — equal-width, equal-frequency, supervised
Binning chops a continuous feature into buckets, turning a messy nonlinear relationship into a piecewise-constant one a linear model can grab. Where you place the edges is the whole game. Toggle the strategy and watch the bin means (the step function) track the data — supervised binning puts edges exactly where the target changes.

The three strategies in code — and why supervised binning can leak code · pitfall

Equal-width splits the range evenly (sensitive to outliers — most points may land in one bin). Equal-frequency splits by quantiles so every bin holds the same count (robust to skew). Supervised binning places edges to minimize within-bin target variance — it's literally a one-feature decision tree — which makes it powerful and a leakage risk, because the edges are learned from y.

equal-width : edges at min + k·(max−min)/B equal-freq : edges at the B-quantiles of x supervised : edges chosen to minimize Σ within-bin Var(y) (uses y ⇒ fit on TRAIN only)
from sklearn.preprocessing import KBinsDiscretizer from sklearn.tree import DecisionTreeRegressor # unsupervised: uniform or quantile edges kbd = KBinsDiscretizer(n_bins=5, encode="onehot", strategy="quantile") X_binned = kbd.fit_transform(X_train[["age"]]) # fit on train only # supervised: a shallow tree gives target-aware split points stump = DecisionTreeRegressor(max_leaf_nodes=5).fit(X_train[["age"]], y_train) edges = sorted(stump.tree_.threshold[stump.tree_.threshold != -2]) # the learned cuts

Because supervised edges are derived from the target, computing them on the full dataset leaks test information into the bins — the same cardinal rule as imputation (2.1) and scaling (2.3). Fit the discretizer inside the pipeline so the edges are learned per fold.

04 / Teaching the model that midnight wraps
Datetime engineering & cyclical encoding
A raw timestamp is useless; the parts are gold — hour, day-of-week, month, is_weekend, days-since-event. But a naive hour column lies: it tells the model 23:00 and 00:00 are 23 apart when they're actually adjacent. Encoding the hour on a circle with sine and cosine restores the truth. Drag the hour and watch the two distances diverge.

EXTRACT

Calendar parts

year, month, day, day-of-week, hour, quarter, week-of-year — each often more predictive than the raw timestamp.

FLAGS

Boolean events

is_weekend, is_holiday, is_month_end, is_business_hours — cheap binary features that capture sharp behavioral shifts.

DELTAS

Time-since / time-until

days_since_signup, hours_until_deadline — relative time is usually what drives behavior, not absolute dates.

CYCLICAL

sin / cos pairs

For any periodic part, the sin/cos pair so the model sees the wrap-around as proximity, not a cliff.

Extracting datetime parts & the sin/cos pair, in code code · why pairs

One sine column isn't enough — sin alone maps two different hours to the same value (it's not injective over a full period). You need the cosine partner so the (sin, cos) pair pins down a unique point on the circle. Together they make "distance" on the clock behave correctly.

sin(2π·5/24) == sin(2π·7/24) ? → ambiguous with sin alone (sin, cos) pair → unique (x,y) on unit circle → unambiguous
import numpy as np, pandas as pd ts = pd.to_datetime(df["event_time"]) df["hour"] = ts.dt.hour df["dow"] = ts.dt.dayofweek df["is_weekend"] = (ts.dt.dayofweek >= 5).astype(int) df["days_since"] = (ts - df["signup_time"]).dt.days # cyclical encoding: BOTH sin and cos, period = 24 for hour df["hour_sin"] = np.sin(2*np.pi*df["hour"]/24) df["hour_cos"] = np.cos(2*np.pi*df["hour"]/24) # drop the raw "hour" — keep only the pair

Tree models are an exception: they can split hour at multiple thresholds and don't strictly need the cyclical pair, but linear models, KNN, and neural nets benefit enormously from it.

05 / When one cycle isn't enough
Fourier features for seasonality
Cyclical encoding gives one sine and one cosine — enough for a smooth single cycle. Real seasonality has shape: sharp peaks, plateaus, double humps. Fourier features add higher-frequency sin/cos pairs (harmonics), letting a linear model reconstruct an arbitrary periodic curve. Add harmonics and watch the fit snap onto the signal.

Building Fourier features for a time index code · tip

Fourier features turn periodicity into ordinary regression columns: for a known period P (24 for hourly, 7 for weekly, 365.25 for yearly), generate K sine/cosine pairs at multiples of the base frequency. A linear model then learns the amplitude of each harmonic — reconstructing the curve as a truncated Fourier series.

import numpy as np def fourier_features(t, period, K): cols = {} for k in range(1, K+1): cols[f"sin_{k}"] = np.sin(2*np.pi*k*t/period) cols[f"cos_{k}"] = np.cos(2*np.pi*k*t/period) return pd.DataFrame(cols) X_f = fourier_features(df["day_of_year"], period=365.25, K=4) # concat with other features, fit any linear model # libraries: statsmodels / sktime / prophet do this for you with multiple seasonalities

Choosing K is a bias-variance dial: too few harmonics under-fit the shape, too many start tracing the noise — exactly the Val-R² turnover you can trigger in the demo. Cross-validate it like any other hyperparameter.

06 / Features across rows, not within them
Aggregations & group statistics
Some of the strongest features describe a row by its group: a transaction's amount relative to the customer's average, a product's rank within its category. These groupby aggregations capture context a single row can't see.
GROUP STATS

mean / std / count per group

customer_avg_spend, store_txn_count. The classic: replace or augment a value with statistics of its group.

RELATIVE

value ÷ group mean

amount / customer_avg — "is this unusually large for this customer?" Often far stronger than the raw amount.

WINDOWED

rolling / lag features

7-day rolling mean, value at t−1. Essential for time series — and a leakage minefield if windows peek forward.

groupby transforms & the look-ahead trap code · pitfall

groupby(...).transform(...) is the workhorse: it computes a per-group statistic and broadcasts it back to every row, so you can build relative features in one line. For time series, the danger is look-ahead: a rolling or expanding window must only use the past, or you've leaked the future into the feature.

# group statistic broadcast back to each row df["cust_avg"] = df.groupby("customer")["amount"].transform("mean") df["amt_ratio"] = df["amount"] / df["cust_avg"] # relative-to-customer # time series: shift BEFORE rolling so the window excludes the current row df = df.sort_values("date") df["roll7"] = (df.groupby("customer")["amount"] .transform(lambda s: s.shift(1).rolling(7).mean())) # shift(1) = no peek

The shift(1) is the whole defense: without it, row t's rolling mean includes row t itself (and a centered window would include the future), inflating offline scores that vanish in production.

07 / Scale it up — but lead with judgment
Automation, text & the domain edge
Three more sources, in ascending order of value-per-effort. The last one is the highest-ROI bullet in this entire module.
AUTOMATED FE

Featuretools / DFS

Deep Feature Synthesis stacks primitives (mean, count, time-since) across related tables to generate hundreds of candidate features automatically. Great for breadth — then prune hard with selection (→ 2.6).

TEXT BASICS

Bag-of-Words & TF-IDF

Turn text into counts (BoW) or frequency-weighted counts (TF-IDF) — each term becomes a (sparse) feature. Enough to get a baseline; full NLP and embeddings come in a later course.

tfidf = tf(t,d) · log(N / df(t))
DOMAIN FEATURES

The highest ROI of all

One feature an expert suggests — debt-to-income, BMI, time-on-glass — routinely beats a thousand auto-generated ones. No algorithm can invent knowledge it was never given. Talk to the domain expert first.

DFS & TF-IDF in a few lines code · scope

Featuretools applies primitives to an entity set and stacks them to a chosen depth, producing a wide candidate matrix. TF-IDF down-weights words that appear in many documents, so common-but-uninformative terms ("the", "and") don't dominate.

# Automated FE: Deep Feature Synthesis over related tables import featuretools as ft fm, defs = ft.dfs(entityset=es, target_dataframe_name="customers", agg_primitives=["mean", "count", "std"], trans_primitives=["month", "weekday"], max_depth=2) # stacks primitives 2 deep # Text: TF-IDF features (sparse), then feed any linear model from sklearn.feature_extraction.text import TfidfVectorizer tfidf = TfidfVectorizer(max_features=5000, ngram_range=(1, 2)) X_text = tfidf.fit_transform(df["review"]) # fit on train only

Both broaden your feature set fast — but breadth without selection overfits. Generate widely, then let Module 2.6 cut the candidates down to the ones that actually carry signal, and always anchor the set with hand-built domain features.
