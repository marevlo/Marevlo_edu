Turning words into
numbers — without lying.

A model can only do arithmetic, so every category — "Chrome", "zip 94107", "Business class" — must become a number. The danger is encoding that smuggles in a falsehood: a fake ordering, a column explosion, or — most insidious — the target itself leaking through a mean-encoded feature. This module is about translating categories honestly.

01 / Learning outcomes
What you should be able to do after this lesson
Choose an encoder from cardinality and target type, not habit. By the end you should know when one-hot is fine and when it explodes, how smoothing rescues target encoding from rare categories, and why mean encoding is a leakage trap unless it's computed out-of-fold.
01 · ENCODE HONESTLY

Avoid fake order & explosion

Pick one-hot, label, or ordinal so you never impose an ordering that isn't real — and recognize when one-hot's column blow-up becomes the problem.

02 · SMOOTH THE TARGET

Tame rare categories

Apply the smoothing prior m so a category seen 3 times is pulled toward the global rate instead of memorized — and read the formula that does it.

03 · BEAT CARDINALITY

Handle thousands of levels

Reach for frequency, binary, hashing, or embeddings when a column has thousands of categories, and compute target encoding out-of-fold to stay leak-free.

02 / The four you reach for first
One-hot · dummy · label · ordinal
The choice hinges on one question: does the category have a real order? If not, any single-integer code (label/ordinal) invents one the model will believe. Toggle a strategy below and read the encoded matrix — watch label encoding quietly impose Chrome < Edge < Firefox < Safari.

ONE-HOT

One binary column per level

The safe default for nominal data — no false order. Cost: k new columns. Perfect for linear/distance models at low cardinality.

DUMMY

One-hot minus one column

Drops a reference level (k−1 columns) to avoid the dummy-variable trap — perfect collinearity that breaks a linear model's intercept.

LABEL

Arbitrary integer per level

Compact but imposes a fake order on nominal data. Fine for tree features and for encoding the target itself — not for nominal inputs to linear models.

ORDINAL

Order-respecting integers

Correct only when the category truly orders: S<M<L<XL, Econ<Business<First. Then the integer order is meaningful and one column suffices.

Why label encoding poisons linear models but not trees intuition · math · code

Label-encode browser as Chrome=0, Edge=1, Firefox=2, Safari=3 and feed it to linear regression. The model fits one coefficient β, so it assumes the effect of Safari is exactly three times the effect of Edge and that Firefox sits halfway between Edge and Safari. None of that is true — the integers were alphabetical. A tree, by contrast, can split {0,2} vs {1,3} across several cuts, so it can recover arbitrary groupings regardless of the codes' order.

linear model on label codes: ŷ = β·code(browser) ⇒ forces effect(Safari) = 3·effect(Edge) ← fabricated tree on label codes: a sequence of "code > t?" splits ⇒ can isolate any subset ⇒ order is irrelevant
from sklearn.preprocessing import OneHotEncoder, OrdinalEncoder import pandas as pd # nominal -> one-hot (no fake order, handles unseen levels safely) ohe = OneHotEncoder(handle_unknown="ignore", sparse_output=True) X_browser = ohe.fit_transform(df[["browser"]]) # truly ordered -> ordinal WITH an explicit order (never trust the default) order = [["Economy", "Business", "First"]] oe = OrdinalEncoder(categories=order) df["class_code"] = oe.fit_transform(df[["ticket_class"]]) # pandas one-liner for dummy (k-1) columns: dummies = pd.get_dummies(df["browser"], drop_first=True)

The handle_unknown="ignore" flag matters: at prediction time a browser you never trained on becomes an all-zeros row instead of a crash. And always pass categories= to OrdinalEncoder — its default order is data-appearance order, which is just label encoding wearing a costume.

03 / 🧮 The powerful, dangerous one
Target / mean encoding & smoothing
Replace each category with the mean target value for that category — one column, captures the category's relationship to y directly, scales to any cardinality. The catch: a category seen 3 times with a mean of 1.0 isn't really a perfect predictor, it's noise. Smoothing shrinks small categories toward the global mean. Drag m and watch the rare categories get reined in.

🚨

Smoothing is not enough — you also need out-of-fold

Even smoothed, if you compute ȳ_c using a row's own target, that row's label has leaked into its feature. For a singleton category the encoding equals its target — a perfect, useless predictor that evaporates in production. The fix is to encode each row using only the other folds. See the code below.

Leakage-safe target encoding — the out-of-fold construction code · proof

Naive target encoding fit on the full training set lets every row see its own label through the category mean. The standard cure is a K-fold scheme: to encode the rows in fold i, compute the category means from folds other than i. No row ever contributes to its own encoding.

❌ Leaky (each row sees its own target)
# the rare/singleton categories now equal their own label means = df.groupby("city")["y"].mean() df["city_te"] = df["city"].map(means) # leak: includes this row's y
✅ Smoothed + out-of-fold
import numpy as np from sklearn.model_selection import KFold def smoothed_mean(y, m, global_mean): n = len(y) return (n * y.mean() + m * global_mean) / (n + m) def kfold_target_encode(df, col, target, m=20, n_splits=5): global_mean = df[target].mean() enc = np.full(len(df), global_mean, dtype=float) kf = KFold(n_splits=n_splits, shuffle=True, random_state=0) for trn, val in kf.split(df): stats = df.iloc[trn].groupby(col)[target].agg(["mean", "count"]) smooth = (stats["count"]*stats["mean"] + m*global_mean) / (stats["count"] + m) enc[val] = df.iloc[val][col].map(smooth).fillna(global_mean).values # OOF return enc df["city_te"] = kfold_target_encode(df, "city", "y") # at TEST time: encode with means computed on the FULL training set

scikit-learn ships this as TargetEncoder (with internal cross-fitting), and the category_encoders library offers TargetEncoder, MEstimateEncoder, and CatBoostEncoder. The math in the formula above is exactly what they implement — the only hard part is the discipline of never letting a row encode itself.

from sklearn.preprocessing import TargetEncoder te = TargetEncoder(smooth="auto", cv=5) # cross-fitted, leak-safe by default X_enc = te.fit_transform(df[["city"]], y)
04 / When one-hot explodes
Frequency · binary · hashing

FREQUENCY / COUNT

Replace by how common it is

One column: each level → its count or frequency. Cheap and surprisingly strong when rarity itself is predictive (rare = suspicious). Ties between equally-frequent levels collide.

BINARY

Integer code → its bits

Label-encode, then spread the integer across ⌈log₂ k⌉ binary columns. Far fewer columns than one-hot with no fake ordinal axis. A solid middle ground for trees.

HASHING

Hash into d fixed buckets

The "hashing trick": map any level into one of d columns via a hash. Fixed output size, handles unseen levels for free, streaming-friendly — at the cost of collisions (two levels share a column).

The hashing trick & binary encoding, in code code · trade-off

Hashing buys a fixed dimensionality that never grows with cardinality and needs no vocabulary stored — it computes a hash on the fly, so a brand-new category at test time just lands in some bucket instead of breaking. The price is collisions: with d buckets and k levels, the expected number of pairs sharing a bucket grows like k²/(2d), which the model partly tolerates because informative collisions are rare.

columns: one-hot = k binary = ⌈log₂ k⌉ hashing = d (fixed) collision rate ↑ as k/d ↑ ⇒ pick d to bound expected collisions
from sklearn.feature_extraction import FeatureHasher import category_encoders as ce # HASHING: fixed d columns, no fitted vocabulary, handles unseen levels hasher = FeatureHasher(n_features=32, input_type="string") X_hash = hasher.transform(df["zip"].astype(str).map(lambda v: [v])) # BINARY: ceil(log2 k) columns instead of k be = ce.BinaryEncoder(cols=["zip"]) X_bin = be.fit_transform(df[["zip"]]) # FREQUENCY: one column, the relative frequency of each level freq = df["zip"].value_counts(normalize=True) df["zip_freq"] = df["zip"].map(freq)

Rule of thumb baked in: if you have a target, target encoding (1 column, out-of-fold) usually beats all of these on high-cardinality features; reach for hashing when the vocabulary is unbounded or streaming, and binary when you want one-hot's spirit without its width.

05 / Two more, for completeness
Leave-one-out & embeddings
A sharper take on target encoding, and a preview of how deep learning sidesteps the whole question.
LEAVE-ONE-OUT

Target encoding, minus this row

The cleanest anti-leak idea: encode each row with the category mean computed from every other row. It's K-fold target encoding taken to the limit of one held-out row at a time — strong, but add noise during training so the model can't reverse-engineer the trick.

enc(rowᵢ) = (Σ_{j≠i, same cat} yⱼ) / (n_c − 1)
EMBEDDINGS · PREVIEW

Learn the encoding

For very high cardinality, map each level to a small learned dense vector (a few dimensions), trained jointly with the model. Similar categories end up nearby in the embedding space — the network discovers the geometry instead of you hand-designing it. Full treatment in the deep-learning track.

level → Embedding(k, d) → ℝᵈ (d ≪ k, learned)
Why leave-one-out still needs noise intuition · code

Leave-one-out removes the row's own label, but for a category with two rows it can still be inverted: if you know the LOO encoding and the category size, you can sometimes back out the other row's label. Adding a touch of Gaussian noise during training breaks that invertibility while leaving the signal intact — the same trick CatBoost formalizes with its ordered boosting.

import category_encoders as ce # LOO encoder; sigma injects training-time noise to prevent overfitting loo = ce.LeaveOneOutEncoder(cols=["city"], sigma=0.05) X_train_enc = loo.fit_transform(X_train[["city"]], y_train) X_test_enc = loo.transform(X_test[["city"]]) # test: full-train means, no noise

Note the asymmetry: noise is added at fit_transform (training) but not at transform (test), because test rows never contributed a label to leak in the first place.
