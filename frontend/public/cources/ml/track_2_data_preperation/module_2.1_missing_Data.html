The holes in your data
tell you a story.

A NaN is never just a gap to fill. Why a value is missing decides whether you can safely impute it, whether the missingness itself is a feature, and whether your clever imputer is quietly leaking the test set into training. This module turns missingness from a nuisance into a modeling decision.

01 / Learning outcomes
What you should be able to do after this lesson
Treat missing data as a decision problem. By the end, you should be able to tell which mechanism you are facing, whether missingness is signal, and how to impute without leaking the split.
01 · DIAGNOSE

Identify the mechanism

Use the observed pattern to decide whether the missingness looks MCAR, MAR, or MNAR. The mechanism determines whether any imputation can be unbiased.

02 · PRESERVE SIGNAL

Keep missingness indicators

If missingness correlates with the target, create an indicator column so the model can learn from the pattern instead of losing it during imputation.

03 · AVOID LEAKAGE

Fit transforms on train only

Imputers, scalers, and encoders must learn statistics from the training fold only. Use a pipeline so the split stays honest.

02 / The implementation view of three famous acronyms
MCAR · MAR · MNAR
The mechanism is the process that decides which values go missing. It's the single most important thing to figure out, because it determines whether any imputation can be unbiased. Pick a mechanism and watch which points vanish — and what it does to the observed mean.

MCAR

Missing Completely At Random

P(missing) is the same for every row — a sensor randomly drops readings. Deletion is unbiased (just wasteful); any imputation is safe. The rarest case in practice.

P(M=1 | X_obs, X_mis) = const
MAR

Missing At Random

P(missing) depends only on observed columns — older users skip the income field. Because the driver is in the data, conditional imputation (MICE, KNN) can recover it.

P(M=1 | X_obs, X_mis) = P(M=1 | X_obs)
MNAR

Missing Not At Random

P(missing) depends on the unobserved value itself — high earners hide income. No purely data-driven fix exists; you need a model of the missingness or external data.

P(M=1 | X_obs, X_mis) ≠ P(M=1 | X_obs)
Why the mechanism matters intuition · math · code

The mechanism is the hidden rule that creates the holes. If the rule depends only on what you already observe, you can often model it. If it depends on the value that disappeared, the hole itself carries information that the dataset no longer contains.

MCAR: P(M=1 | X_obs, X_mis) = const MAR : P(M=1 | X_obs, X_mis) = P(M=1 | X_obs) MNAR: P(M=1 | X_obs, X_mis) ≠ P(M=1 | X_obs)

Geometrically, imagine a scatter plot where some points are painted transparent. If the transparent points are sprinkled everywhere, the cloud is still representative. If the transparent points gather in one region, the visible cloud is biased and the observed mean shifts away from the true center.

# quick mechanism probe missing_rate_by_age = df.assign(is_missing=df["income"].isna().astype(int)) \ .groupby(pd.cut(df["age"], bins=[18, 30, 45, 60, 80]))["is_missing"].mean() print(missing_rate_by_age) # if missingness changes with an observed feature, MAR is plausible

That is the practical test: if missingness moves with an observed column, conditional methods like KNN or MICE can use that column to recover structure.

03 / Detection · the part general EDA skips
When missingness is the signal
The modeling-driven question isn't just "how much is missing?" — it's "is the missingness correlated with the target?" If it is, the NaN pattern carries predictive information, and blindly imputing it erases a real feature. The fix is a missingness indicator: a binary column flagging "was this value present?"

A missing last_login almost always means the account was never activated — and those churn at ~7× the base rate. The missingness is one of the strongest predictors in the dataset.

df["last_login_missing"] = df["last_login"].isna().astype(int) # keep BOTH: the indicator (the signal) # and the imputed column (the magnitude)
Why the indicator is often better than the fill value intuition · example

Think of missingness as a binary feature layered on top of the original value. The indicator tells the model whether the entry existed at all; the imputed value gives it a placeholder magnitude. The two together let the model separate absence from size.

x_original → [ x_imputed , is_missing ] is_missing = 1 can carry more signal than any single fill value especially when missingness is correlated with y

For example, if inactive customers tend to have no last_login, then a missing value is not “unknown” — it is close to a label proxy for inactivity. Filling the blank with a mean would erase that clue.

# keep the pattern and the placeholder df["last_login_missing"] = df["last_login"].isna().astype(int) df["last_login"] = df["last_login"].fillna(df["last_login"].median()) # model can now use both columns separately
04 / Filling the holes · with their math
The imputation toolbox
From a constant to a full per-column model. The right choice depends on the mechanism, the feature's relationships, and how much you can afford to distort the distribution.
UNIVARIATE

Mean / Median / Mode

Replace with a single column statistic. Fast, but ignores every relationship — it shrinks variance and biases correlations toward zero. Median resists skew; mode is for categoricals.

x̂ⱼ = mean(xⱼ) | median(xⱼ) | mode(xⱼ)
DISTANCE-BASED

KNN imputation

Find the k nearest complete rows (using the non-missing features) and average their value for the missing column. Preserves local structure — but needs scaled features (→ 2.3) to define "nearest" fairly.

x̂ⱼ = (1/k) Σ x⁽ᵏ⁾ⱼ over k nearest neighbors
ITERATIVE

MICE

Treat each column with missing values as a regression target predicted from all the others; fit f̂ⱼ(x₋ⱼ), fill in, and iterate until values stabilize. The gold standard for MAR data.

repeat: xⱼ ← f̂ⱼ(x₋ⱼ) for each column j
MODEL-BASED

Random Forest imputer

The MICE idea with a Random Forest as f̂ⱼ (e.g. missForest). Captures nonlinearities and interactions automatically, handles mixed types, no scaling needed — at a real compute cost.

f̂ⱼ = RandomForest( x₋ⱼ → xⱼ )

Why each imputer changes geometry differently intuition · math

Each imputation method answers the same question in a different geometric way. Mean fill pulls every missing point toward one horizontal center line. KNN fills from nearby points in feature space. MICE fits a local predictive surface and places the missing value on that surface.

mean fill: x̂ⱼ = μⱼ KNN fill: x̂ⱼ = (1/k) Σ xⱼ⁽ᵏ⁾ MICE fill: x̂ⱼ = f̂ⱼ(X₋ⱼ)

So the decision is not “which method is cleverest,” but “which geometry best matches how my variables relate to one another.” If one feature moves with another, a local or predictive imputer usually preserves that relationship better than a single global statistic.

# compare variance loss after imputation before = df["x2"].var() after = df["x2_imputed"].var() print(f"variance kept: {after / before:.2f}") # the more variance you keep, the less structure you erase
05 / The decision
Drop rows · drop columns · impute
Not every hole is worth filling. The choice is a quick triage on how much is missing, where, and why.
drop rows

Few rows, MCAR

  • Missingness is small (≲5%) and plausibly MCAR
  • You have plenty of rows to spare
  • Never drop rows based on the target or test set
  • Risk: silently biases data if it's really MAR/MNAR
drop column

Mostly empty feature

  • Column is >50–70% missing with little signal
  • Cost of imputation outweighs its value
  • But first: add a missingness indicator — the pattern may still be useful
impute

The default for MAR

  • Moderate missingness with related features present
  • Use KNN / MICE / RF conditioned on those features
  • Pair with an indicator when missingness is informative
  • Always fit the imputer on train only ↓
06 / 🚨 The cardinal trap
Imputation as leakage
This is the mistake that quietly inflates almost every beginner's cross-validation score. If you compute the imputation statistic on the whole dataset before splitting, the mean you fill train rows with already contains information from the validation and test rows. The model has peeked.
🚨

The rule, stated once

Fit the imputer on the training fold only; apply its learned statistics to validation and test. The same rule governs scalers (→ 2.3) and encoders (→ 2.4). A scikit-learn Pipeline inside cross-validation enforces it for free — which is exactly why pipelines exist.

Wrong vs right — and why CV can't catch it for you code · proof
❌ The leak
# fits the mean on ALL rows — train, val, AND test df["income"] = df["income"].fillna(df["income"].mean()) X_train, X_test = train_test_split(df, ...) # too late — the mean already saw X_test

The filled value in every training row is a function of the test rows' incomes. Cross-validation reshuffles rows, but the imputation statistic was frozen before the split, so every fold is contaminated identically. CV looks clean and reports an optimistic score.

✅ The fix — a Pipeline, fit per fold
from sklearn.pipeline import Pipeline from sklearn.impute import SimpleImputer from sklearn.model_selection import cross_val_score pipe = Pipeline([ ("impute", SimpleImputer(strategy="median")), # .fit() runs INSIDE each fold ("model", model), ]) scores = cross_val_score(pipe, X, y, cv=5) # imputer sees only the fold's train data

Now SimpleImputer.fit() is called separately on each fold's training portion, learning that fold's median, and .transform() applies it to the held-out portion. The estimate of generalization is honest.

leaky CV score ≈ 0.91 (optimistic — stats saw the test rows) pipeline CV score ≈ 0.86 (honest — matches production) the 5-point gap is pure leakage, invisible until deployment
Evaluating an imputation

Two checks: (1) distribution shift — compare the imputed column's distribution to the observed one (a mean-imputed column grows a spike at the mean and loses variance). (2) downstream impact — does the imputation actually improve held-out model performance, or just fill cells? Only the second one ultimately matters.

  • You impute a feature with its global mean and your test accuracy jumps from 0.82 to 0.95. What's the most likely explanation?

    Leakage. A 13-point jump from a univariate mean-fill is implausible; almost certainly the mean was computed on the full dataset (or the imputed feature is itself a target proxy). Re-run with the imputer inside a per-fold pipeline.

  • Missingness in income is driven entirely by age (older users skip it). Which mechanism, and can imputation be unbiased?

    MAR — the driver (age) is observed. Conditional imputation (MICE/KNN using age) can be approximately unbiased. A plain mean-fill, which ignores age, would not.
