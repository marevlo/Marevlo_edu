the way the model will.

General EDA asks "what is this data?" Modeling-driven EDA asks a sharper question: "which features carry predictive signal, which ones leak the answer, and what will trip the model up before I've fit a single line?" This lesson turns EDA into a decision-making workflow: you will inspect the target first, compare features by target, and leave with a checklist you can repeat on every new dataset.

The mindset shift

Every plot in this module is colored, split, or sorted by the target. A histogram of age is general EDA. The same histogram split into age | y=churn vs age | y=stay is modeling-driven EDA — it tells you whether age will help a classifier. The lesson is simple: start with the target, then ask whether a feature gives you signal, leakage, or neither.

04 / The most expensive bug in ML · catch it in EDA
The target-leakage scan
Target leakage is when a feature contains information that would not be available at prediction time — it secretly encodes the answer. The model looks brilliant in validation and collapses in production. The cheapest place to catch it is right here, in EDA: a feature with a suspiciously strong association to the target is guilty until proven innocent. In this lesson, use the three scenarios below to practice asking a simple question: would this value exist when I actually make the prediction?
PREDICTING: loan_default ∈ {0,1}  ·  metric: |corr(feature, target)|
plausible signal flagged: likely leak threshold
Why leakage inflates your CV score — and how to interrogate a suspect feature intuition · checklist

Cross-validation estimates how well a model generalizes by hiding part of the data. But it can only hide rows — it cannot hide information that shouldn't be in a column at all. If days_since_last_payment is only populated after a default is recorded, then in every fold the feature carries the label. The model learns the shortcut, CV reports 0.99 AUC, and the moment you score a live applicant — who has no "days since default" — the feature is empty and the model is blind.

The three questions that unmask a leak

1. Timing: Would this value exist, fully populated, at the moment of prediction? If it's recorded at or after the event, it leaks.

2. Provenance: Is the feature derived from, or a proxy for, the label? (e.g., risk_tier assigned by the same process that produces default.)

3. Magnitude: Is the single-feature association implausibly strong for the domain? A lone feature explaining ~all of a hard target is a red flag, not a trophy.

Symptom pattern of a leak: train AUC ≈ 1.00 CV AUC ≈ 0.99 live AUC ≈ 0.62 → the gap appears only in production, never in CV → a single feature dominates importance by a wide margin

Leakage is not always this loud. Subtle leakage (a feature with a small but real future-peek) inflates scores by a couple of points and is far harder to spot — which is exactly why you build the timing/provenance habit now, before Module 2.7 where the train/test split makes leakage structural.

# A first-pass leakage triage in pandas import pandas as np_pd corr = df.corr(numeric_only=True)["target"].abs().sort_values(ascending=False) suspects = corr[corr > 0.85].drop("target") # interrogate each by hand for feat in suspects.index: print(feat, "→ ask: does this exist at prediction time?")
05 / Predictive structure, one feature at a time
Univariate & bivariate by target
For a classification target, the single most useful EDA move is to split each feature's distribution by class. If the two class-conditional distributions sit on top of each other, the feature is useless on its own. If they pull apart, the feature carries signal — and a linear boundary might even suffice. Switch features below and watch the class separation change.

Why target-splitting works — geometry, math, and a quick check intuition · code

Think of a single feature as a one-dimensional axis. Each class forms a cloud on that axis. If the clouds overlap heavily, a threshold cannot separate them well. If their centers are far apart relative to spread, even a simple line on that axis gives useful predictive power.

Cohen's d = (μ₁ − μ₀) / σ_pooled σ_pooled = √((σ₀² + σ₁²) / 2) large |d| ⇒ larger separation in feature space ⇒ easier thresholding
Geometric intuition

Separate the classes by asking how far apart their centers are on the same line. A big gap means a threshold can cut through the empty space between clouds. A tiny gap means the classes are stacked on top of each other, so even a flexible model has little to work with.

Tiny pandas check

Use a groupby to compare class-wise means and spread. If the means are far apart and the standard deviations are small, the feature is likely useful.

# class-wise view of a candidate feature summary = ( df.groupby("y")["monthly_balance"] .agg(["mean", "std", "count"]) ) print(summary) # quick normalized separation score mu0, mu1 = summary.loc[0, "mean"], summary.loc[1, "mean"] sp = ((summary.loc[0, "std"]**2 + summary.loc[1, "std"]**2) / 2)**0.5 d = (mu1 - mu0) / sp

That normalized distance is the reason this module focuses on by-target plots. A good feature does not merely vary; it varies in the right direction relative to the target.

06 / When the target is a class
Categorical-target EDA
Three checks define how you'll model, evaluate, and split. They take five minutes and they change everything downstream.
A — CLASS BALANCE

How skewed is y?

Severe imbalance dictates your metric (PR-AUC, not accuracy), your split (stratify, → 2.7), and whether you resample. Always measure it first.

B — MEAN-ENCODING PEEK

Does category move y?

Group-by category, take the mean target rate per level. Big spread across levels ⇒ the categorical is predictive and target-encoding (→ 2.4) will pay off.

C — MOSAIC PLOT

Category × target, visually

Tile width = how common the level is; the colored split = its target rate. One glance reveals which levels are both frequent and risky.

A model that predicts "stays" for everyone scores 84% accuracy while catching zero churners. This is why accuracy lies on imbalanced targets — and why you decide the metric now.

Reading the mosaic: month-to-month contracts are both the widest tile (most customers) and the reddest (highest churn) — the single most actionable signal in the dataset.

The per-level churn rates are exactly the mean-encoding values: {m2m:0.43, one-yr:0.11, two-yr:0.03}. The wide spread says: encode this categorical by its target rate (carefully, on training folds only → 2.4).

07 / When the target is a number
Continuous-target EDA
For regression, the baseline you must beat is trivially simple: always predict the mean of y. Its error is the standard deviation of the target. So the right EDA question for each feature is: "does knowing this feature shrink the residuals around that flat mean-line?" Scatter the feature against the target, draw the baseline, and look at the residual structure.

Why the mean baseline is the right null model intuition · math

The constant predictor ŷ = ȳ is the best possible model if you are not allowed to use any features. Geometrically, it is the projection of all outcomes onto the space of constant functions. That is why the baseline error is the spread of y itself.

baseline: ŷ = ȳ SSE_base = Σ(yᵢ − ȳ)² RMSE_base = √(SSE_base / n) = std(y) R² = 1 − SSE_fit / SSE_base
Geometric intuition

Imagine the target values as points above a flat horizontal line. The best line without features is the one that sits at the center of mass of those points. Adding a feature tilts the line toward the cloud only if the cloud has structure to tilt toward.

Tiny sklearn check

Fit a one-feature linear model and compare it against the constant baseline. If R² barely moves, the feature adds little linear signal. If the scatter is curved, the relationship may still be real even when R² looks small.

# compare baseline vs one-feature fit from sklearn.linear_model import LinearRegression x = df[["feature_x"]] y = df["target_y"] base_pred = np.repeat(y.mean(), len(y)) model = LinearRegression().fit(x, y) fit_pred = model.predict(x) base_rmse = mean_squared_error(y, base_pred, squared=False) fit_rmse = mean_squared_error(y, fit_pred, squared=False)

For regression EDA, the question is not only “is there a line?” but “does this feature shrink error relative to the no-feature baseline?” That is the right geometric test for predictive value.

09 / Before you touch a single transformer
The pre-modeling checklist
Run this on every new dataset. Each item maps to a decision that, if skipped, silently corrupts everything downstream. Tap each card as you clear it.
0 / 8 cleared
✓
Split decided first

Choose random / stratified / time / group split (→ 2.7) before looking deeply — so EDA never peeks at test.

✓
Leakage scan run

Every feature passes the timing & provenance test. No column knows the future.

✓
Target profiled

Class balance (clf) or distribution & skew (reg) measured — metric and resampling chosen.

✓
By-target structure scanned

Univariate-by-target / scatter-vs-target reviewed — you know which features carry signal.

✓
Missingness mapped

Counts, patterns, and whether missingness correlates with the target (→ 2.1) — it may be signal.

✓
Cardinality & dtypes audited

High-cardinality categoricals flagged for special encoding (→ 2.4); IDs marked drop-or-encode.

✓
Outlier exposure noted

Extreme values located and tagged for treatment (→ 2.2) — but not removed during EDA.

✓
Duplicate / group leakage checked

Same entity (patient/user/store) appearing many times → needs a group-aware split (→ 2.7).
