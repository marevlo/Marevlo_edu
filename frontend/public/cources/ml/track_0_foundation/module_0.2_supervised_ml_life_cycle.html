The supervised
ML lifecycle

Training a model is one step in a much longer loop. The teams that win don't have fancier algorithms — they have a tight loop: frame the problem well, obsess over data, evaluate honestly, ship carefully, and watch for decay. This module is the map of that loop, and the places it tends to break.

01 / Step zero · the most under-taught step
Problem framing
Before a single line of modeling code, three questions decide whether the project can even succeed. Get these wrong and no algorithm can save you; get them right and a simple model often wins.
Q1 — WHAT KIND OF PREDICTION?

Regression or classification?

Is the target a number on the real line, or a label from a finite set? This single choice cascades into your loss, your metrics, and half your tooling.

Q2 — WHAT'S THE UNIT?

What is one prediction about?

One customer? One transaction? One image-per-day? The "unit of prediction" defines what a row is, how you split, and what leakage even means.

Q3 — WHAT DOES ERROR COST?

Which mistake hurts more?

A missed fraud case ≠ a false alarm. Asymmetric costs decide your metric and threshold long before you fit anything.

06 / Derivation
The math behind "wrong metric" — why 96% accuracy can be a total failure definition · numeric

Every classification metric is built from the four cells of the confusion matrix: true/false positives and negatives (TP, FP, TN, FN).

accuracy = (TP + TN) / (TP + TN + FP + FN) precision = TP / (TP + FP) "of those I flagged, how many were right?" recall = TP / (TP + FN) "of the real positives, how many did I catch?" F1 = 2·precision·recall / (precision + recall) (harmonic mean)
Run the demo's numbers

4 positives in 100, model lazily predicts "negative" for everyone: TP=0, FP=0, FN=4, TN=96.

accuracy = 96/100 = 0.96 ← looks great recall = 0/(0+4) = 0.00 ← caught zero of the thing that matters precision = 0/0 = undefined F1 = 0

Accuracy rewarded the model for the easy majority and hid total failure on the minority. Under imbalance, prefer recall / precision / F1, the PR-AUC, or MCC (which uses all four cells and stays honest even when classes are lopsided). This is exactly why "frame the problem → pick the metric" is step zero of the lifecycle.

Practice — frame it, spot the leak exercises + solutions
  1. A fraud model is 99.8% accurate. Fraud is 0.2% of transactions. Is the model useful? What would you measure instead?

    Not necessarily — predicting "never fraud" already scores 99.8%. Accuracy is meaningless here. Measure recall (fraction of fraud caught), precision (cost of false alarms), and the precision–recall curve; pick the operating threshold from the relative cost of a miss vs a false alarm.

  2. To predict 30-day hospital readmission, an engineer includes the feature "discharge destination = hospice." Why is this leakage?

    Discharge destination is decided at or after the outcome window and correlates almost perfectly with the label (hospice patients rarely "readmit"). It won't be available — or won't mean the same thing — at prediction time. Validation looks superb; production collapses. The fix: only use features knowable strictly before the prediction moment.

  3. "Data is 80% of performance." Past a baseline, why does an hour on data usually beat an hour on the model?

    Model tuning moves you along a flattening curve toward the Bayes/irreducible floor set by your current data; you can't cross it by tuning. Better data (more coverage, cleaner labels, fixing leakage) lowers that floor, so it has the steeper payoff. The model can only extract the signal that's present; data work adds signal.
