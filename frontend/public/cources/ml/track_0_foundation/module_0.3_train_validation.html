Train, validate,
test — honestly

A model's score on the data it learned from is a fantasy. The real question is: how will it do on data it has never seen? Answering that honestly is the entire job of splitting and cross-validation — and getting it wrong is how good-looking models quietly fail in production.

02 / Derivation
Derivation — the noise in a single split, and why k-fold beats it proof
1 · A hold-out accuracy is a coin-flip average

Suppose the model's true accuracy is a. Each of the m validation examples is right with probability a, so the measured accuracy â is a scaled Binomial. Its variance:

â = (1/m) Σ 1[correctᵢ] E[â] = a Var(â) = a(1−a) / m std(â) = √(a(1−a)/m)

Plug in the demo's numbers — a = 0.84, m = 32 (20% of 160):

std(â) = √(0.84·0.16 / 32) ≈ 0.065 → ±6.5 percentage points!

That is why a single split is a lottery: the model never changed, but the estimate can swing ±6 points purely on which rows landed in validation. Smaller validation sets are noisier (the 1/m).

2 · Why k-fold shrinks the noise

K-fold validates every example exactly once, so the average uses all n points, not just m. Crudely, the variance of the mean drops toward the 1/n regime instead of 1/m. More precisely, for the CV estimate averaged over k folds with pairwise correlation ρ between fold errors:

Var(CV) ≈ ρ·s² + (1−ρ)·s²/k

The (1−ρ)/k term is the win — averaging decorrelated folds cuts variance. The leftover ρ·s² (folds share training data, so they're correlated) is why CV still has some spread, and why that fold-to-fold std is itself a useful confidence signal a single split can never give you.

The tradeoff in k: large k (e.g. leave-one-out) ⟹ low bias but high variance and k refits; small k (e.g. 5) ⟹ a bit more bias, far cheaper, lower variance. 5–10 is the usual sweet spot.

03 / Interactive · the three roles
The three-way split
Three sets, three jobs, and one strict rule. The train set fits parameters. The validation set picks between models and hyperparameters. The test set is opened exactly once, at the very end, to estimate real-world performance. Drag the proportions.

① Train

The model fits its parameters here. Bigger train set → lower variance, but less data to validate/test on.

② Validation

You tune here — model choice, hyperparameters, thresholds. Touched many times, so it slowly "leaks" optimism.

③ Test

Locked away. Opened once, never tuned on. The moment you optimize against it, it stops being an honest estimate.

04 / Interactive · keep the class ratios
Stratification — when & why
When the target is a class — especially an imbalanced one — a plain random split can hand you folds with wildly different class ratios, or even folds containing none of the rare class. Stratified splitting forces every fold to mirror the overall balance. Crank up the imbalance and flip between the two.

05 / Derivation
From scratch — KFold and StratifiedKFold in NumPy code

The splitters you just visualized are a few lines. Plain k-fold shuffles indices and slices them; stratified k-fold deals out each class separately so every fold mirrors the global ratio.

import numpy as np def kfold(n, k=5, seed=0): idx = np.random.default_rng(seed).permutation(n) for f in np.array_split(idx, k): # f = validation indices val = f train = np.setdiff1d(idx, f) yield train, val def stratified_kfold(y, k=5, seed=0): rng = np.random.default_rng(seed) folds = [[] for _ in range(k)] for cls in np.unique(y): # deal each class round-robin members = rng.permutation(np.where(y == cls)[0]) for j, sample in enumerate(members): folds[j % k].append(sample) for f in folds: val = np.array(f) train = np.setdiff1d(np.arange(len(y)), val) yield train, val

The golden rule, in code: fit every preprocessor (scaler, imputer, encoder) inside the loop on X[train] only, then apply to X[val]. Fitting before the split leaks the validation distribution into training — the next section shows why that quietly inflates your score.

Practice — splitting without fooling yourself exercises + solutions
  1. A validation set has 32 examples; your model scores 84%. Give a rough ± on that number.

    std ≈ √(0.84·0.16/32) ≈ 0.065, so about ±6.5 points (one standard deviation). Reporting "84%" alone hides that the true accuracy could plausibly be anywhere from ~78% to ~90%. Use more validation data or k-fold to tighten it.

  2. You have repeated measurements from the same 50 patients. Why is a plain random split dangerous, and what splitter fixes it?

    Rows from one patient are correlated; a random split puts some of a patient's rows in train and some in test, so the model "recognizes the patient" rather than generalizing — optimistic leakage. Use GroupKFold (group = patient id) so all of a patient's rows stay on one side.

  3. For a time series, why must the test set come after the train set in time?

    Shuffling lets the model train on the future to predict the past — impossible at deployment. A forward-chaining / TimeSeriesSplit (train on [0,t], validate on [t, t+h]) honestly simulates predicting forward. Random k-fold here leaks future information and massively overstates performance.

06 / Preview · the cardinal sin
Leakage — the cardinal sin of splitting
Leakage is when information from outside the training fold sneaks into the model. The most common form hides in preprocessing: fit a scaler or an imputer on the whole dataset, and your training set has secretly "seen" the test set's statistics. The score looks great — and lies.

✗ Fit on everything (leaky)

scaler.fit( ALL data ) X = scaler.transform(X) split → train / test model.fit(train)

The scaler's mean & std were computed using test rows. The model is quietly tuned to the test distribution.

→ optimistic score · fails in production

✓ Fit inside the fold (clean)

split → train / test scaler.fit( train only ) train = scaler.transform(train) test = scaler.transform(test)

Every transform learns its parameters from training data alone, then is applied to held-out data — exactly as it will be in production.

→ honest estimate · generalizes
