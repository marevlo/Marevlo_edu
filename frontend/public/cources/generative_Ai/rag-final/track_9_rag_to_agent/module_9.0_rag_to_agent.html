Every query walks the same path. Input goes in one end, answer comes out the other. The LLM gets one shot, with one retrieval, against one index. Predictable. Cacheable. Cheap. And — for a huge class of problems — completely sufficient.

What RAG Nails

  • ▸ Q&A over docs
  • ▸ Support deflection bots
  • ▸ Enterprise search
  • ▸ Citable summarization
  • ▸ "What does our policy say about X?"

RAG's Superpowers

  • ▸ Grounded, citable answers
  • ▸ Predictable latency (~1s)
  • ▸ Low cost per query
  • ▸ Easy to evaluate
  • ▸ Easy to debug

Where RAG Hits a Wall

  • ▸ Multi-step planning
  • ▸ Performing actions (write, send, run)
  • ▸ Cross-system reasoning
  • ▸ Long, evolving conversations
  • ▸ "Compare A and B, then book the cheaper one"

The New Primitive

The Agent Loop

Where RAG has a pipeline, an agent has a loop. Three steps, repeated until the goal is met.

THINK

The LLM reads the goal and history, then decides: "Given what I know, what's the next step?" Outputs a structured action — a tool call, a query, or "done."

ACT

The runtime executes the chosen tool: hit a RAG retriever, query an SQL DB, call an API, run code, send an email. RAG becomes one tool among many.

OBSERVE

The result of the tool call is fed back into the LLM's context. Now it can THINK again with new information — and the loop continues.

💡 The mental upgrade: RAG is a function — input goes in, answer comes out, deterministic shape. An agent is a program — it executes instructions, branches, calls subroutines (including your RAG system as one of them). When you outgrow functions, you reach for programs. The same is true in AI.
Diagnosis

Have You Outgrown RAG?

Tick the boxes that match your real product needs. The more you check, the more you need an agent.

RAG OUTGROWN CHECKLIST
My users want the system to perform actions, not just answer questions (send email, create ticket, book meeting)
A single retrieval isn't enough — the answer needs cross-referencing across multiple sources (e.g. SQL + docs + email)
The query requires planning over multiple steps ("compare A and B, then pick the cheaper one with delivery this week")
Conversations are long and evolving, with memory of past interactions affecting current behavior
The retrieval target is dynamic and computed, not pre-indexed (live APIs, calculated SQL, runtime data)
You need self-correction — the system should notice a bad result and try again with a different approach
Different specialists should handle different parts of a request (multi-agent orchestration)
