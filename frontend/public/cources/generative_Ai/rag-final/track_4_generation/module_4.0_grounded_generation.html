The Model's Choice

Four LLM Behaviors with Retrieved Context

Given a query plus context, every LLM picks one of four paths. Your prompt decides which one — whether you realize it or not.

GROUNDED

Grounded · use the context

Every claim traces to a retrieved chunk; refuse if not found. This is the goal.

"Refunds are 30 days [doc_213]."

PARAMETRIC BLEED

Parametric bleed · mix context + priors

Cite some claims, silently fill gaps from training-data priors. Plausible-looking, partially unsupported.

"Refunds are 30 days [doc_213]. They take 7-10 days."

CONTEXT IGNORANCE

Context ignorance · answer from priors

Answer purely from training data, no citations. The retrieved chunks may as well not exist.

"Most companies offer 14-day refunds."

FABRICATION

Fabrication · invent unsupported facts

Confidently states details that exist in neither context nor training data — invented citations, made-up policies. The most dangerous failure.

"Refunds are 45 days per [doc_904]."  (doc_904 doesn't exist)

Two independent axes — faithfulness and relevance

Axis 1 · Faithfulness

"Is every claim supported by the retrieved context?"

Measures the relationship between the answer and the context. An answer can be fluent, on-topic, and helpful — and still score 0 on faithfulness if a single claim wasn't in the docs.

High faithfulness · low relevance: model copies the doc verbatim but ignores what the user asked.

Axis 2 · Answer Relevance

"Does the answer address what the user actually asked?"

Measures the relationship between the answer and the question. An answer can be perfectly faithful (citation-clean) and still useless if it answers a question the user didn't ask.

Low faithfulness · high relevance: model answers exactly the question but invents the supporting facts.

⚡ Why this matters for eval (Module 6.1): these axes are independent — improving one doesn't improve the other. RAGAS measures them separately for exactly this reason. The four behaviors above plot onto the (faithfulness, relevance) plane: Grounded = high/high, Bleed = mid/high, Context ignorance = low/varies, Fabrication = low/possibly high.
⚡ Prompts decide which behavior the model takes. A weak prompt invites Parametric Bleed and Context Ignorance. A well-built one (everything in this module) makes Grounded the default and provides a clean refusal path that prevents Fabrication on unanswerable queries.
The Five Parts

Anatomy of a RAG Prompt

Every production RAG prompt has the same five sections. Skip one and you'll feel it in the answers.

PART 01 · SYSTEM

Role & persona

Who is the model? What's its scope? Set the voice, the domain, and the non-negotiables once. "You are a customer support assistant. You answer only from the provided documents."

PART 02 · TASK

Rules & constraints

The behavioral rules. Cite every claim. Refuse if not found. No filler. Keep it tight — 5-10 short rules outperform a wall of text.

PART 03 · CONTEXT

Retrieved chunks

The reranked top-K chunks, each clearly labeled with an ID for citation. Wrap in XML tags or use markdown headers so the LLM can tell where docs start and end.

PART 04 · QUERY

User question

The user's actual question. Place after context, not before — the LLM should "have read the docs" by the time it sees the question.

PART 05 · FORMAT

Output spec

How the answer should look. Markdown bullets, JSON schema, inline brackets, a structured payload — match downstream needs. The last instruction the model reads.

Full prompt assembled 

# PART 1 — SYSTEM
<system>
You are a customer-support assistant for ACME Corp.
You answer only from the documents in <context>.
You never use outside knowledge.
</system>

# PART 2 — TASK / RULES
<rules>
1. Every factual claim must be cited as [doc_id].
2. If the answer isn't in <context>, say
   "I couldn't find this in the documents."
3. Never speculate. Never fill gaps from memory.
4. Quote exact numbers from documents verbatim.
</rules>

# PART 3 — CONTEXT (reranked chunks)
<context>
  <doc id="doc_213" source="refund-terms.md">
    Refunds are issued to the original payment method
    within 5 business days after approval.
  </doc>
  <doc id="doc_847" source="travel-faq.md">
    Travel purchases are refundable within 30 days
    if the booking has not been used.
  </doc>
</context>

# PART 4 — QUERY
<query>
What's the refund window for travel?
</query>

# PART 5 — FORMAT
<format>
- Markdown bullets
- Each bullet ends with [doc_id]
- 1-3 bullets max
</format>
⚡ Order matters: system → rules → context → query → format. The model reads top-down. Putting format before context risks the model forgetting it by the time it answers; format last keeps it primed.
The Liu et al. 2023 Phenomenon

The Lost-in-the-Middle Phenomenon

Formally documented in "Lost in the Middle: How Language Models Use Long Contexts" (Liu, Lin, Hewitt, Paranjape et al., 2023). The result is precise: across GPT-3.5, Claude, MPT-30B, and Longchat, retrieval accuracy from a 10-30 document context follows a U-shape — strong at position 1, strong at the last position, with a measurable trough in the middle that can drop accuracy by 20+ percentage points. This is not a bug to fix; it's a property to design around.

Recall accuracy vs answer position in context

U-SHAPED CURVE

The "needle in a haystack" benchmark across major LLMs. Front and back of the context window dominate; the middle is a recall dead zone.

STRATEGY 01

Put critical chunks first AND last

If you have 5 reranked chunks, place rank 1 first and rank 2 last. The model attends most to both ends.

STRATEGY 02

Keep context short

5 reranked chunks beats 20 raw retrieval hits. Quality > quantity. Lost-in-middle gets worse linearly with context length.

STRATEGY 03

Repeat the query after context

Place the question both before and after the context block. The "end of prompt" attention boost catches what the model would otherwise miss.

Make Every Claim Citable

Three Citation Patterns

Citations are the audit trail. Without them, you can't tell hallucination from truth. Pick a pattern, enforce it in the prompt, validate the output.

# Prompt instruction
<rules>
End every factual claim with the cited doc id in
square brackets, e.g. [doc_213]. Multiple sources
in one claim use [doc_213, doc_847].
</rules>

# Expected model output
Travel purchases are refundable within 30 days
of booking [doc_847]. Refunds are issued back
to the original payment method within 5 business
days of approval [doc_213]. To start the process,
contact support with your order number [doc_159].

# Post-processing — validate
import re
cited = set(re.findall(r"\[(doc_\d+)\]", answer))
retrieved = {c.id for c in context}
assert cited.issubset(retrieved), "hallucinated citation"

When to use

  • Chat UIs, support assistants
  • Human readers, scrollable answers
  • Easy to render as clickable links
⚡ Validate: after generation, regex out every [doc_*] and verify it's a real id from your context. Catches hallucinated citations. 
# Prompt instruction
<format>
Write the answer using [1], [2], [3] markers.
Then list the sources under a "Sources:" section
showing marker → doc id.
</format>

# Expected output
Travel purchases are refundable within 30 days
of booking [1]. Refunds are processed within
5 business days [2]. Contact support with your
order number to start [3].

### Sources
[1] doc_847 · travel-faq.md
[2] doc_213 · refund-terms.md
[3] doc_159 · support-kb.md

When to use

  • Long-form answers
  • Reports, research summaries
  • Reads cleaner than inline ids
⚠ Gotcha: the model has to maintain a mapping from [1] → doc_id. Smaller models drift after 3-4 footnotes. Validate before shipping. 
# Pydantic schema · forced structured output
class Citation(BaseModel):
    claim: str
    doc_ids: list[str]
    verbatim_quote: str | None = None

class RAGAnswer(BaseModel):
    answer: str
    citations: list[Citation] = Field(min_length=1)
    unanswered: bool = False

# Model output (constrained)
{
  "answer": "Refunds: 30 days for travel; 5 biz days to process.",
  "citations": [
    {
      "claim": "30 day window for travel",
      "doc_ids": ["doc_847"],
      "verbatim_quote": "refundable within 30 days"
    },
    {
      "claim": "5 business days to process refund",
      "doc_ids": ["doc_213"],
      "verbatim_quote": "within 5 business days after approval"
    }
  ],
  "unanswered": false
}

When to use

  • API endpoints / downstream tooling
  • Need verbatim quote validation
  • Want structured "unanswered" flag
⚡ Highest discipline: verbatim quotes force the model to literally pull text out of context. Almost impossible to hallucinate around. Pair with strict JSON schema from Track 2.
Cross-Doc Reasoning

Multi-Doc Synthesis

When the answer requires combining facts from 2+ chunks, the failure modes change. Three patterns handle the heavy lifting.

PATTERN 01

Merge facts, cite each

"Travel refunds are 30 days [doc_847]; processed in 5 business days [doc_213]."

Two facts, two sources, one sentence. Cite where each fact came from.

PATTERN 02

Compare & contrast

"Travel: 30 days [doc_847]. Standard products: 14 days [doc_213]. Software: non-refundable [doc_502]."

Same dimension across docs. Use a table or short list. Never mix doc claims without saying which.

PATTERN 03

Surface conflicts

"Two sources disagree: doc_213 says 5 days; doc_088 says 7 days. The most recent (doc_213) supersedes."

Don't paper over conflicts — flag them. Use timestamps from metadata to pick the canonical source.

⚡ Synthesis prompt rule: add "If two documents conflict, surface the conflict explicitly. Never invent a reconciliation." — one of the most underused lines in production RAG prompts.
When to Say "I Don't Know"

Refusal Done Right

A confident wrong answer is the worst outcome. A clean "I couldn't find this" is the second-best outcome — sometimes the best. Engineer it deliberately.

Bad refusal

Vague & unhelpful

"I'm sorry, I can't help with that question."

"I don't have access to that information."

"That's outside my training data."

No signal about what was missing. User can't refine. Looks like a broken system.

Good refusal

Specific & actionable

"I couldn't find a specific refund window for
international orders in the provided documents.
The general refund terms in doc_213 might apply,
but they don't mention international shipping
explicitly. Try contacting support directly."

Names what was missing. Points to the closest available info. Suggests a next step.

Drop-in refusal instruction 

<refusal_policy>
If the answer is not fully contained in <context>:

1. Say "I couldn't find this in the documents."
2. State what specifically is missing.
3. Cite the closest related doc, if any.
4. Suggest a next step (contact support,
   try a more specific query, etc.).

Do not guess. Do not use general knowledge.
Do not apologize beyond one short sentence.
</refusal_policy>
Defense Manual

Six Hallucination Types in RAG

"The model hallucinated" is a diagnosis, not a fix. There are six distinct failure modes, each with a known patch. Recognize, patch, ship.

1 · Prior bleed

Model uses training-data knowledge for a claim that's not in context.

Example: Context says nothing about refund speed. Answer says "5-10 business days" anyway.
Fix: Add "answer only from context" + per-claim citation requirement.

2 · Citation fabrication

Model cites [doc_999] — an id that doesn't exist in your context.

Example: "Refunds take 5 days [doc_500]" — doc_500 was never retrieved.
Fix: Post-validate every citation against retrieved ids; retry on mismatch.

3 · Misattribution

Citation exists, but the cited doc doesn't say what's claimed.

Example: Cites doc_213 for "international shipping" — but doc_213 is about refunds.
Fix: Force verbatim_quote in structured output; verify quote appears in chunk text.

4 · Fact extrapolation

Context says A; model concludes A → B without warrant.

Example: Doc says "refunds processed in 5 days"; answer claims "so you'll have your money by Friday".
Fix: "Don't infer beyond what the documents state. Quote exact phrasing for numbers."

5 · Conflict papering

Two retrieved chunks contradict; model picks one silently or averages them.

Example: doc_213 says 5 days, doc_088 says 7 days. Answer says "5-7 days".
Fix: "If sources disagree, surface the conflict and cite each."

6 · Over-confident refusal

Answer is in context, but model says it isn't.

Example: Context has "refunds take 5 days"; answer says "no info on refund timing".
Fix: Put strong chunks at the start; repeat query after context; raise reranker top-K confidence.
⚡ Diagnose before you fix: when a user reports a "hallucination", inspect the answer against the retrieved chunks. Which of the 6 types is it? Each one has a different patch — guessing wastes prompt engineering effort.

Self-Check

Three Quick Questions

Cement the generation instincts before moving on to prompt structuring.

QUESTION 01

Your reranker put the answer at top-K = 1, but the LLM keeps missing it. The context has 18 other chunks below it. Most likely cause?

Correct. Even with the right chunk at the top, 18 distractor chunks below it pull attention away from the answer. Rerank to 5, place rank-1 first and rank-2 last, drop the rest.
QUESTION 02

The LLM cites [doc_999] in its answer — but you only retrieved doc_213, doc_847, and doc_159. What's the right pipeline-level fix?

Correct. Citation fabrication is one of the six hallucination types. The right fix is mechanical: regex out every [doc_*] after generation, verify it's a retrieved id, retry with a stricter prompt on mismatch.
QUESTION 03

Doc_213 says refunds take 5 days. Doc_088 says 7 days. The user asks "how long do refunds take?" — what's the right model behavior?

Correct. Conflict papering is hallucination type #5. Always surface conflicts and let the user (or downstream logic) decide. "Two sources disagree: doc_213 says 5 days [most recent], doc_088 says 7 days."
