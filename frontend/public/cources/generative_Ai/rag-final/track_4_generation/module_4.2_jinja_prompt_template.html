The Upgrade

Why Jinja2 Beats f-strings

F-strings are fine for one-shot scripts. The moment your RAG prompt has 3 conditionals, 2 loops, and per-doc metadata, f-strings become an unreadable mess.

Feature f-strings Jinja2 Why it matters for RAG
Variable interpolation ✓ f"{x}" ✓ {{ x }} Both work for simple substitution
Loops ~ via "\n".join(...) ✓ {% for %} Iterate over retrieved chunks cleanly
Conditionals ~ ternary or concat ✓ {% if %} Toggle refusal_policy, examples, etc.
Whitespace control ✗ \n hell ✓ {%- -%} Clean output without extra blank lines
Reuse across files ✗ copy-paste ✓ {% include %} / inheritance One prompt library, many endpoints
Filters / transforms ✗ inline Python ✓ value | filter Truncate chunks, escape, format JSON
Auto-escaping ✗ manual ✓ optional, configurable Prevents prompt-injection bugs
Testability ~ function-level ✓ template-level Snapshot-test the rendered prompt
REASON 01

Separation of concerns

Prompt content lives in .jinja files. Python only passes data. Prompts can be code-reviewed independently.

REASON 02

Version control + diffs

PR a prompt change. Reviewers see exactly what the LLM will see. F-strings hide changes behind code formatting.

REASON 03

Non-engineers can edit

PMs and domain experts can edit refund_policy.jinja. They can't safely edit Python f-strings.

⚡ Industry pattern: Most production RAG stacks (LangChain, LlamaIndex, Cohere RAG, Anthropic's quickstart cookbooks) use Jinja2 (or its Python sibling string.Template) under the hood. The pattern is so common it's practically the default.

Whitespace control — the dash matters

WITHOUT DASHES · blank-line garbage
{% for r in rules %}
{{ r }}
{% endfor %}

# renders to:

rule one

rule two

rule three
WITH DASHES · clean output
{%- for r in rules %}
{{ r }}
{%- endfor %}

# renders to:

rule one
rule two
rule three
⚡ Rule of thumb: use {%- ... -%} on for/if blocks unless you specifically want blank lines. Saves tokens, looks clean.
The Logic Layer

Conditionals & Loops

Two patterns handle 90% of dynamic prompt building. Master them once.

Conditionals · branching prompts 

{# Show different rules per user tier #}
<rules>
{%- if user.tier == "enterprise" %}
  - Use formal tone.
  - Include compliance citations.
{%- elif user.tier == "pro" %}
  - Use professional tone.
{%- else %}
  - Use friendly tone.
{%- endif %}

{%- if include_refusal %}
  - If not in context, refuse politely.
{%- endif %}
</rules>

Use cases: tier-specific behavior, A/B prompt variants, feature toggles, multilingual outputs.

Loops · iterate over retrieved chunks 

<context>
{%- for doc in retrieved_docs %}
  <doc id="{{ doc.id }}"
       source="{{ doc.source }}"
       rank="{{ loop.index }}">
    {{ doc.text }}
  </doc>
{%- endfor %}
</context>

{# loop variables you get for free: #}
{# loop.index, loop.index0, loop.first, #}
{# loop.last, loop.length                #}

loop.index = 1-based rank, loop.first/.last = boundary helpers, loop.length = total count.

Loop tricks · empty fallback & loop variables 

{%- for doc in retrieved_docs %}
  {# Mark the top chunk specially (lost-in-the-middle fix) #}
  <doc id="{{ doc.id }}"
       priority="{% if loop.first %}highest{% elif loop.last %}secondary{% else %}normal{% endif %}">
    {{ doc.text }}
  </doc>
{%- else %}
  {# This branch runs if retrieved_docs is empty #}
  <doc id="none">No documents retrieved.</doc>
{%- endfor %}
⚡ Jinja-specific: {% for %}...{% else %}...{% endfor %} — the else branch runs only when the loop is empty. Perfect for empty-retrieval handling.
The Transform Layer

Filters — Inline Transformations

A filter is a function that takes a value and returns a new one. Apply with |. Chain freely.

The 12 filters you'll actually use

| upper | lower | title | trim | truncate(N) | length | default("…") | join(", ") | tojson | replace(a, b) | safe | indent(N)

Truncate long chunks

{{ doc.text | truncate(800, True, "…") }}

# cuts to 800 chars, ends on word, adds "…"

Provide safe defaults

{{ user.lang | default("English") }}
{{ user.name | default("there", true) }}

# avoids "Hello None" when user.name is empty

Join lists into prose

Sources: {{ doc_ids | join(", ") }}.

# → "Sources: doc_213, doc_847, doc_159."

Render JSON safely

<schema>
{{ output_schema | tojson(indent=2) }}
</schema>

# dumps a dict as pretty JSON, properly escaped

Chain filters · transformations in sequence 

{# Read order: left → right, top to bottom #}
{{ doc.title
   | trim
   | truncate(100)
   | title
   | default("Untitled", true) }}

Custom filters · register Python functions 

from jinja2 import Environment, FileSystemLoader

env = Environment(loader=FileSystemLoader("prompts/"))

# register custom filters once at startup
def redact_pii(s: str) -> str:
    # swap emails / phone numbers with [REDACTED]
    return re.sub(r"\S+@\S+|\d{3}-\d{4}", "[REDACTED]", s)

def token_count(s: str) -> int:
    return len(tiktoken.encode(s))

env.filters["redact_pii"] = redact_pii
env.filters["token_count"] = token_count

# now use them like any built-in:
# {{ chunk.text | redact_pii | truncate(800) }}
# Total prompt: {{ rendered | token_count }} tokens
⚡ The big unlock: any Python function becomes a Jinja filter. Token counting, PII redaction, language detection, doc-source formatting — keep the logic in Python, the structure in templates.
Reusable Components

Macros — Functions for Prompts

A macro is a reusable prompt fragment with parameters. Define once, call everywhere. The right abstraction when you have repeated structures.

Define a macro 

{# macros.jinja #}

{% macro render_doc(doc, rank) %}
<doc id="{{ doc.id }}"
     source="{{ doc.source }}"
     rank="{{ rank }}"
     updated="{{ doc.updated | default("unknown") }}">
  {{ doc.text | trim | truncate(800) }}
</doc>
{% endmacro %}

{% macro refusal_policy(style="helpful") %}
<refusal_policy>
{%- if style == "strict" %}
Refuse curtly: "Not in documents."
{%- else %}
Say "I couldn't find this", state what's missing,
suggest contacting support.
{%- endif %}
</refusal_policy>
{% endmacro %}

Import & call from another template 

{# rag_prompt.jinja #}
{% from "macros.jinja" import render_doc, refusal_policy %}

<role>You are a support assistant.</role>

{{ refusal_policy(style=user.tier == "enterprise"
                            and "strict"
                            or "helpful") }}

<context>
{%- for doc in retrieved_docs %}
  {{ render_doc(doc, loop.index) }}
{%- endfor %}
</context>

<query>{{ query }}</query>
⚡ When to make a macro: any time you see the same prompt pattern in 3+ places. Doc rendering, refusal blocks, example blocks, output schema specs — all classic macro candidates. Treat them like Python functions: small, single-purpose, well-named.
Composition at Scale

Template Inheritance

One base.jinja. Many specialized child prompts. Each overrides only what changes. Like class inheritance — but for prompts.

base_rag.jinja · the parent 

<role>
{% block role %}
You are a helpful assistant.
{% endblock %}
</role>

<rules>
{% block rules %}
1. Answer only from <context>.
2. Cite every claim as [doc_id].
3. If absent, follow the refusal_policy.
{% endblock %}
</rules>

{% block extras %}{% endblock %}

<context>
{%- for doc in retrieved_docs %}
  {{ render_doc(doc, loop.index) }}
{%- endfor %}
</context>

<query>{{ query }}</query>

<format>
{% block format %}
Markdown bullets, each ending [doc_id].
{% endblock %}
</format>

support_rag.jinja · the child 

{% extends "base_rag.jinja" %}

{# Only override what's specific to support #}
{% block role %}
You are ACME Corp's customer support assistant.
Be empathetic, concise, and never make promises
beyond what's in <context>.
{% endblock %}

{% block rules %}
{{ super() }}  {# inherit base rules #}
4. Apologize for any inconvenience.
5. Always offer an escalation path.
{% endblock %}

{% block extras %}
<tone>
Warm. Professional. Solution-oriented.
</tone>
{% endblock %}
PATTERN 01

{% block %} defines slots

Parent declares named regions children can override. Each block has a default that wins if no child redefines it.

PATTERN 02

{{ super() }} extends

Inside an overridden block, {{ super() }} pastes the parent's content. Add to it instead of replacing.

PATTERN 03

Multi-level chains

base → industry → tenant → use-case. Three levels of override is typical for multi-tenant production systems.

⚡ The big payoff: a typo in your refusal policy used to mean editing 30 prompts. With inheritance, you edit base.jinja and every child inherits the fix. The DRY principle, applied to prompts.
The Full Stack

A Production RAG Template

Everything you've learned, assembled into one battle-tested template that uses macros, conditionals, loops, filters, and inheritance.

rag_v2.jinja · the full thing 

{% from "macros.jinja" import render_doc, refusal_policy %}

<role>
You are {{ tenant.name }}'s {{ role_name | default("support assistant") }}.
</role>

<rules>
{%- for r in rules %}
{{ loop.index }}. {{ r }}
{%- endfor %}
</rules>

{{ refusal_policy(style=tenant.refusal_style) }}

{%- if examples %}
<examples>
{%- for ex in examples %}
  <example>
    <q>{{ ex.q }}</q>
    <a>{{ ex.a }}</a>
  </example>
{%- endfor %}
</examples>
{%- endif %}

<context>
{%- for doc in retrieved_docs %}
  {{ render_doc(doc, loop.index) }}
{%- else %}
  <doc id="none">No documents retrieved.</doc>
{%- endfor %}
</context>

<query>{{ query | trim }}</query>

<format>
{{ output_format | default("Markdown bullets, each ending [doc_id].") }}
</format>

Why this scales

  • Multi-tenant via tenant.*
  • Optional sections (examples)
  • Graceful empty retrieval
  • Per-call output format
  • Filtered + macro-rendered docs
⚡ Test it: snapshot-test the rendered output with pytest. Any prompt change shows up as a diff in CI.

Calling the template from Python 

from jinja2 import Environment, FileSystemLoader, select_autoescape

env = Environment(
    loader=FileSystemLoader("prompts/"),
    autoescape=select_autoescape(),  # prevent injection via user input
    trim_blocks=True,
    lstrip_blocks=True
)
env.filters["redact_pii"] = redact_pii
env.filters["token_count"] = token_count

tmpl = env.get_template("rag_v2.jinja")

prompt = tmpl.render(
    tenant={"name": "ACME", "refusal_style": "helpful"},
    role_name="customer support assistant",
    rules=[
        "Answer only from <context>.",
        "Cite every claim as [doc_id].",
        "If absent, follow refusal_policy."
    ],
    examples=None,        # skipped — block isn't rendered
    retrieved_docs=reranked[:5],
    query=user_query,
    output_format=None     # falls back to default
)

# Send to the LLM
resp = claude.messages.create(
    model="claude-opus-4-7",
    messages=[{"role": "user", "content": prompt}]
)

Self-Check

Three Quick Questions

Track 4 closes when you ace this. Then you ship.

QUESTION 01

Your retrieval sometimes returns zero docs. What's the cleanest Jinja2 way to handle that without an extra {% if %}?

Correct. The {% else %} clause inside a {% for %} is a Jinja-specific feature for empty iterables. Cleaner than the alternatives, less indentation, less duplication.
QUESTION 02

You have 30 RAG prompts across the codebase. Each one duplicates the same refusal policy. A new lawyer wants you to update the policy text. What's the right Jinja2 pattern?

A
Find-and-replace across 30 files
Correct. This is exactly what macros & inheritance solve. A single shared refusal_policy() macro (or block in base.jinja) means one change propagates to every prompt. DRY for prompts.
QUESTION 03

Why is {%- ... -%} (with dashes) often better than plain {% ... %} for prompt templates?

Correct. Without the dashes, every {% for %} and {% if %} leaves blank lines around it. That's wasted tokens, ugly output, and often messes with structural cues the model uses. The dash form trims it — clean prompt, fewer tokens.
