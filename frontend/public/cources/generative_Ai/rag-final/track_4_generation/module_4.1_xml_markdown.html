The Mechanic

Why Structured Prompts Beat Plain Text

It's not aesthetic preference. Structure changes what the model attends to — and that changes what it outputs.

REASON 01

Attention boundaries

Tags and headers create hard boundaries between sections. The model knows where context ends and rules begin. No bleed.

REASON 02

Named retrieval

"Look in <context>" is a clean reference. "Look at the second paragraph" is fragile and breaks when the prompt changes.

REASON 03

Training distribution

Modern LLMs (especially Claude) are trained on huge amounts of XML & markdown. They've learned the convention — exploit it.

Instruction-following accuracy vs prompt format

ILLUSTRATIVE

As prompts grow longer and more complex, plain text degrades fast. Structure preserves accuracy at scale.

⚡ Anthropic's claim: XML tags consistently improve Claude's instruction-following on complex prompts. The lift grows with prompt complexity.
⚡ The real benefit: structured prompts are also parseable. Your tests can assert "the <rules> section contains X" — observability you don't get from free-form text.
⚡ It compounds: structure + verbatim quotes (4.0) + Jinja2 templates (4.2) is how production RAG prompts stay reliable as they grow to 5000+ tokens.
The Building Blocks

Canonical XML Tag Vocabulary

Tag names are conventions, not standards — but the industry has converged on a small, consistent vocabulary. Use these names and any teammate (and any LLM) instantly understands your intent.

Tag Purpose Where it lives Example
<role>Identity & personaTop of prompt"You are a support assistant."
<task>What to do (1 sentence)After role"Answer the user's question."
<rules>Behavioral constraintsAfter task"Cite every claim. Refuse if absent."
<context>Wraps retrieved chunksMiddle"<doc id="doc_213">...</doc>"
<doc>Single retrieved chunkInside <context>"<doc id="..." source="...">"
<examples>Few-shot demosBefore query"<example>...</example> × N"
<query>The user's questionAfter context"What's the refund window?"
<format>Output shapeNear end"Markdown bullets, max 3."
<thinking>Scratch reasoning areaOutput"Plan in here before answering."
<answer>Final responseOutputWraps the user-facing reply.
<refusal_policy>When to say "no"Inside rules"If not in context, say so."
<output_schema>Strict structured outputNear endJSON schema or Pydantic class.
RULE 01

Names are arbitrary — pick once, reuse

The model doesn't "know" <rules> means rules. It learns from context that text inside that tag is rule-shaped. Pick consistent names across your codebase.

RULE 02

Nest when meaningful, flat otherwise

<context> → <doc> ✓. <rules> → <rule> → <sub_rule> ✗. Deep nesting confuses model and human alike.

RULE 03

Attributes carry metadata

<doc id="doc_213" source="refund.md" updated="2024-08"> tells the model which doc to cite and lets it reason about recency.

RULE 04

Reference tags in instructions

"Answer from <context>", "Output in <answer>". The model bridges your tagged regions and your imperatives.

The Other Tool

Markdown as Prompt Structure

Markdown is XML's lighter cousin. Headers create regions. Bullets create lists. Code fences create literal blocks. Use it where it shines.

Headers as sections

REGIONS

# Role, ## Rules, ## Context — three header levels are usually enough. The model treats each header as a named region just like XML.

Bullets for rules & lists

CRISP

Numbered lists for ordered steps. Hyphen bullets for unordered constraints. Both crush long prose for instruction-following on small & medium models.

Code fences for literals

VERBATIM

Triple-backtick blocks tell the model "treat this as literal text, not instructions". Essential for examples and user-provided strings to prevent prompt injection.

Tables for structured data

DENSE

Pipe tables work surprisingly well for small lookup data — pricing tiers, status maps, etc. Limit to 8-10 rows; beyond that, prefer JSON.

Same RAG prompt · markdown style 

# Role
You are a customer-support assistant for ACME Corp.

# Task
Answer the user's question using only the documents
in the ## Context section below.

## Rules
1. Every factual claim ends with [doc_id].
2. If the answer is not in context, say so explicitly.
3. Never speculate or use outside knowledge.

## Context

### doc_213 · refund-terms.md
Refunds are issued to the original payment method
within 5 business days after approval.

### doc_847 · travel-faq.md
Travel purchases are refundable within 30 days
if the booking has not been used.

## Query
What's the refund window for travel?

## Output format
- Markdown bullets
- Each bullet ends with [doc_id]
- 1-3 bullets max
⚡ Markdown sweet spot: single-purpose prompts, short lists, when humans will edit it directly, when the LLM's output is also markdown (no format confusion). For mixed-content RAG prompts, XML usually wins.
Choose Wisely

XML vs Markdown · The Trade

Both work. Each shines in different conditions. Here's the decision matrix production teams actually use.

XML

Pick when...

  • Prompts have many distinct sections (5+ regions)
  • You need attributes on items (doc_id, source, timestamp)
  • Output uses tags too (<answer>, <thinking>)
  • Programmatic prompt construction (Jinja2 templates)
  • You want to parse sections from the prompt or response
  • Targeting Claude (explicitly trained for XML conventions)
MARKDOWN

Pick when...

  • Prompts are short & single-purpose
  • Mostly lists, headers, rules — no nested structure
  • Humans will edit prompts directly (PMs, ops)
  • Expected output is also markdown
  • You want prompts that read like docs
  • Targeting GPT family (slight preference for markdown in OpenAI training)
⚡ The pragmatic answer: Use both. Wrap regions in XML (<rules>, <context>), but use markdown inside regions (numbered bullets for rules, code fences for examples). It's the best of both — and what most production prompts actually look like.
Production Patterns

Anthropic's XML Patterns

Five patterns that consistently show up in Claude's strongest production prompts. Apply directly to RAG.

PATTERN 01

<thinking> before <answer>

Give the model a scratch area to plan in. The contents of <thinking> never reach the user — you strip them post-generation. But the model uses them to reason carefully.

<format>
First, plan your answer inside <thinking> tags.
Then write the final answer inside <answer> tags.
</format>
PATTERN 02

<examples> with <example> children

Few-shot demos work best when wrapped consistently. Each example has its own input/output structure that mirrors the real prompt.

<examples>
  <example>
    <query>...</query>
    <answer>...</answer>
  </example>
</examples>
PATTERN 03

<doc> with rich attributes

Pack every useful piece of metadata onto the doc tag. The model can reason about source, recency, type, confidence.

<doc id="doc_213"
     source="refund-terms.md"
     updated="2024-08-15"
     retriever_score="0.92">
  Refunds processed in 5 business days.
</doc>
PATTERN 04

Prime the assistant turn

Send a partial assistant message starting with the opening tag. Claude continues from there — locked into your format from token #1.

# messages array
[{"role": "user", "content": "...full prompt..."},
 {"role": "assistant", "content": "<thinking>"}]
# Claude continues from <thinking>
PATTERN 05

Long docs FIRST, query LAST

For prompts with long retrieved context, place <context> before instructions. It fits Claude's positional attention pattern and improves long-context recall.

# Recommended order for long contexts
<context>...</context>        # first (longest)
<rules>...</rules>            # after
<query>...</query>            # last (most attended)
PATTERN 06

Output schema via tag spec

Define the output shape with the same tag system. Easy to parse with regex post-generation. No JSON ambiguity.

<output_schema>
<answer>...</answer>
<citations>
  <cite doc_id="...">quote</cite>
</citations>
<unanswered>true|false</unanswered>
</output_schema>
⚡ Where to learn more: Anthropic's official prompt engineering guide at docs.anthropic.com documents these patterns in detail. The 4-7 family of Claude models is explicitly trained on them.

Self-Check

Three Quick Questions

Cement the structuring instincts before moving on to Jinja2 templating.

QUESTION 01

Your prompt has 6 sections including a few-shot examples block, retrieved context with per-doc metadata, and you need to extract a structured answer from the response. Best format?

Correct. Six sections + metadata-on-items + parseable output = XML's sweet spot. Markdown can't carry attributes (your doc metadata loses recency/score). Plain text fails on attention boundaries at this complexity.
QUESTION 02

You wrote <rules><rule><sub_rule>...</sub_rule></rule></rules>. The model is now confused. Why?

Correct. XML is for region boundaries, not full XML schemas. Once you nest deeper than one level (<context> → <doc> is fine), models start losing track. Use numbered markdown bullets inside a single <rules> block instead.
QUESTION 03

Why is <thinking> a useful tag, even though you strip it from the final response?

Correct. Models perform better when given explicit space to reason step-by-step before producing the final answer — <thinking> is the structural cue for that. You then strip it post-generation. This is the manual version of what reasoning models like Claude 4-thinking do automatically.
