Why It Still Wins

Sparse Beats Dense More Often Than You'd Think

Every "RAG is just embeddings" tutorial skips this lesson. The strongest production systems blend both — and many lean harder on sparse than you'd expect.

Dense Retrieval Fails

When Embeddings Stumble

  • Rare terms: SKU codes, error messages, drug names — embeddings flatten them into noise.
  • Exact match: Looking for "CVE-2024-31082"? Dense retrieval returns loosely related vulnerabilities.
  • Domain drift: Generic embeddings know "doctor" but not "ICD-10 J45.909".
  • Acronyms & jargon: Internal terms like "PR-7842" carry zero semantic signal.
  • Cost & latency: Embedding 10M docs costs real money. BM25 builds for free.
BM25 Shines

Where Sparse Wins

  • Lexical precision: Exact tokens, exact match, every time.
  • Interpretable: You can read the score breakdown — no black box.
  • CPU-fast: Sub-millisecond retrieval over millions of docs.
  • Zero training: Plug in a corpus, get a ranker. No model, no GPU, no warm-up.
  • Hybrid backbone: Pairs with dense via RRF for state-of-the-art results (Module 3.2).
⚡ Industry secret: Most enterprise RAG systems and code-search engines run hybrid by default — and a surprising number of them treat BM25 as the primary retriever, with dense as the rescue net. The reverse is much rarer.
Step 1 of 4

From Text → Tokens

Before scoring, you have to slice. Four small operations turn a paragraph into the bag of terms BM25 actually works on.

STEP 01 · TOKENIZE

Break text into tokens — usually words, sometimes subwords. The simplest tokenizers split on whitespace and punctuation; production tokenizers (Lucene's StandardAnalyzer, ICU) handle Unicode, CJK, and mixed scripts.

in: "The Refund Policy applies to travel purchases."
out: TheRefundPolicyappliestotravelpurchases
STEP 02 · LOWERCASE

Fold case so "Refund" and "refund" match. Almost universal for English. Skip it only when case carries meaning — code search, gene names, acronyms in specific domains.

out: therefundpolicyappliestotravelpurchases
STEP 03 · STOPWORDS

Drop high-frequency words that carry little signal — "the, of, and, to, is". BM25 already down-weights common terms via IDF, so this step matters less than it used to. Many modern engines skip it.

out: therefundpolicyappliestotravelpurchases
STEP 04 · STEM

Reduce words to their stem so "purchases", "purchased", "purchasing" all collapse to "purchas". Porter and Snowball are the classics. Lemmatization is the smarter cousin — slower, more accurate, requires a dictionary.

out: refundpoliciapplitravelpurchas
⚡ Critical rule: Apply the same tokenization pipeline to indexing and queries. Stemming the docs but not the query is the #1 reason "BM25 isn't matching obvious things". The query and index must speak the same tokens.
Step 2 of 4

The Inverted Index

The data structure that makes "search 1M documents in 2ms" possible. Once you understand it, every search engine on Earth makes sense.

Flip the question around

A forward index answers "what's in doc 847?" — useful for displaying but useless for search. An inverted index answers "which docs contain the token 'refund'?" — and you can answer it in O(1).

One token → posting list

For each unique token in the corpus, store a posting list: the doc IDs containing that token, plus their term frequencies. To run a query, look up each query token's posting list, intersect/union, score.

Why it's blazingly fast

Only documents containing at least one query token get scored — usually a tiny fraction of the corpus. Posting lists are typically compressed (variable-byte / Frame-of-Reference) so they fit in CPU cache and disk reads are minimal.

Deep Dive

Inside a Posting List

The inverted index is the map. The posting list is the territory. Here's exactly what's stored, how it's compressed, and how a query intersects two lists in O(min(|A|,|B|)).

What's actually stored per term

A posting list isn't just doc IDs. Each entry carries everything BM25 (and phrase search, highlighting, faceting) needs to score and reconstruct.

term: "refund"
df: 2  # in N=3 docs
postings: [
{doc:1, tf:1, len:3, positions:[0]},
{doc:3, tf:3, len:4, positions:[0,1,2]}
]

Positions enable phrase queries ("refund policy") and proximity boosts. Lengths feed BM25's normalization.

Posting list intersection — O(min(|A|,|B|))

Both lists are sorted by doc ID. Walk them with two pointers — advance the smaller — match in linear time, no random reads.

A = [1, 3, 7, 11, 14]   # "refund"
B = [3, 4, 7, 9, 14]    # "travel"

i=j=0
A[0]=1 < B[0]=3        → i++
A[1]=3 == B[0]=3       → MATCH, i++ j++
A[2]=7 > B[1]=4        → j++
A[2]=7 == B[2]=7       → MATCH, i++ j++
A[3]=11 > B[3]=9       → j++
A[3]=11 > B[4]=14      → j++ (end)

intersect = [3, 7, 14]
Compression 1

Delta (gap) encoding

Store gaps between sorted doc IDs, not the IDs themselves. Smaller numbers compress better.

[1, 3, 7, 11, 14] → [1, 2, 4, 4, 3]
Compression 2

Variable-byte

7 bits payload + 1 continuation bit per byte. Small numbers fit in 1 byte; large in 2-5.

2 → 0x02   200 → 0xC8 0x01
Compression 3

Skip pointers

Insert sparse "skip ahead" pointers every √n entries so intersection can jump past large gaps.

walk → match → skip → walk
⚡ Why this matters: Lucene's segment files (.doc, .pos, .tim) are just compressed posting lists. When you query Elasticsearch, you're walking these lists with skip pointers. "Sub-millisecond search over a billion docs" is literally just clever bit-packing of integers.
Prequel to BM25

From TF-IDF to BM25

BM25 didn't appear from nowhere. It's a careful patching of two flaws in TF-IDF. Derive it from scratch and BM25 stops being a formula you memorize — it becomes one you'd invent.

STEP 01 · TERM FREQUENCY (TF)

Start naive: count how often a query term appears in a document. More mentions of refund → more about refunds.

tf(t, D) = count of t in D

Problem: linear growth. A doc that says "refund" 100 times scores 100× a doc that says it once. That's wrong — diminishing returns kick in after the first few mentions.

STEP 02 · INVERSE DOCUMENT FREQUENCY (IDF)

"The" appears in every doc — useless signal. "CVE-2024-31082" appears in 3 docs — gold. Weight each term by how rare it is.

idf(t) = log(N / df(t))
N = total docs · df(t) = docs containing t

Log dampens the weight — without it, ultra-rare terms (df=1) would crush everything else.

STEP 03 · COMBINE — TF · IDF

Multiply per-term, sum over the query. This is classical TF-IDF — the formula that powered every search engine for ~30 years.

score(D, Q) = ∑t ∈ Q  tf(t, D) · idf(t)

Two remaining flaws: (1) TF still grows linearly — no saturation. (2) A 10,000-word doc has more chances to mention any term — long docs unfairly win.

STEP 04 · PATCH 1 — SATURATE THE TF

Replace tf with tf · (k₁+1) / (tf + k₁). As tf→∞, this approaches k₁+1 — a ceiling. k₁ controls how fast we hit it.

tf_sat = tf · (k₁+1) tf + k₁
STEP 05 · PATCH 2 — LENGTH NORMALIZE

Penalize long docs. Replace k₁ in the denominator with k₁ · (1 − b + b · |D|/avgdl). When the doc is longer than average, the denominator inflates, dampening the score.

Result: we just derived BM25. The "intimidating" formula is TF-IDF + two well-motivated patches.

⚡ Why bother deriving it? Because when BM25 misbehaves — a long doc dominating, a stopword sneaking through — you'll know which patch to adjust. Tuning k₁ is "loosen the saturation." Tuning b is "loosen the length penalty." Memorized formulas can't tell you that.
Step 3 of 4

The BM25 Formula — Decoded

Looks intimidating. Three intuitions. Two knobs. Below: the formula, then each piece in plain English.

BM25 SCORE OF DOCUMENT D FOR QUERY Q
score(D, Q) = ∑t ∈ Q   IDF(t) · tf(t,D) · (k₁+1) tf(t,D) + k₁ · (1 − b + b · |D|/avgdl)
sum over query terms · IDF weight · saturated TF · length-normalized
Idea 1

TF Saturation

"The 50th mention of refund matters less than the 2nd." BM25 caps how much repetition rewards a doc — controlled by k₁.

Idea 2

IDF Weighting

"The is everywhere; CVE-2024 isn't." Rare terms get a higher weight. IDF = log((N − df + 0.5) / (df + 0.5) + 1).

Idea 3

Length Normalization

"A long doc has more chances to hit a term by accident." Compare each doc's length to the corpus average, controlled by b.

b = 0No normalization (long docs win)
b = 0.75Default (sensible balance)
b = 1Full normalization
⚡ Default knobs: k₁ = 1.2 and b = 0.75. These work astonishingly well across almost every English corpus. Don't change them until you've measured.
⚡ When to tune: Increase k₁ (1.5-2.0) for long docs where repetition is meaningful. Lower b (0.4-0.6) for short, uniform-length docs (titles, FAQs).

Step 4 of 4

BM25 in Python — Three Stacks

Same idea, three weights. Use the lightest one that fits your scale.

from rank_bm25 import BM25Okapi
import re

def tokenize(text):
    return re.findall(r"\w+", text.lower())

# 1. tokenize the corpus
corpus = ["Refund policy for travel purchases.",
          "Shipping is free for orders over $50.",
          "Warranty covers manufacturing defects for one year."]
tokenized = [tokenize(d) for d in corpus]

# 2. build the index (k1 and b are the knobs)
bm25 = BM25Okapi(tokenized, k1=1.2, b=0.75)

# 3. score a query against every doc
scores = bm25.get_scores(tokenize("refund travel"))
# → [2.31, 0.0, 0.0]

# 4. top-K
top = sorted(zip(scores, corpus), reverse=True)[:5]

When to use

  • Notebooks, demos, prototypes
  • < 100k documents
  • Slow on big corpora
  • No persistence
⚡ Quick start: pip install rank-bm25. Easiest possible BM25 in Python — 5 lines and you're scoring. 
import bm25s

# 1. tokenize (bm25s has a built-in tokenizer with stopwords + stemming)
corpus = ["Refund policy for travel purchases.", ...]
tokens = bm25s.tokenize(corpus, stopwords="en", stemmer="english")

# 2. build & index (uses scipy sparse matrices — fast)
retriever = bm25s.BM25(k1=1.2, b=0.75)
retriever.index(tokens)

# 3. retrieve top-K — returns (doc_ids, scores)
query_tokens = bm25s.tokenize("refund travel", stopwords="en", stemmer="english")
ids, scores = retriever.retrieve(query_tokens, k=5)

# 4. persist + reload
retriever.save("./bm25_index")
retriever = bm25s.BM25.load("./bm25_index")

When to use

  • 100k–10M documents
  • Local-first RAG apps
  • ~100x faster than rank_bm25
  • Built-in stemming + stopwords
⚡ Sweet spot: If you're not on Elasticsearch already and your corpus fits in RAM, bm25s is the right default.

res = es.search(index="rag-chunks", query={

  • Multi-node, multi-shard scale

  • Production tooling & ops story

A Named Problem

The Vocabulary Mismatch Problem

Sparse retrieval's Achilles heel — and the formal name for the gap that dense and learned-sparse methods exist to close.

Definition

Same meaning, different words

When a user's query and the relevant document express the same concept using different surface tokens, sparse retrieval scores them at zero — the index sees no shared term.

QUERY: "how do I get my money back"
DOC: "our refund policy permits reimbursement within 30 days"
BM25 score: 0.0  (no shared tokens)
Why It Happens

Three flavors

  • Synonymy: "refund" vs "reimbursement" vs "money back" — distinct tokens, identical meaning.
  • Paraphrase: Whole-phrase rewrites — "how do I cancel" vs "termination procedure".
  • Register shift: Conversational query vs formal documentation. Customers talk; docs declaim.
Fix 1

Query expansion

Use a thesaurus or LLM to add synonyms before BM25. Cheap but brittle — synonyms aren't always exchangeable.

Fix 2

Dense retrieval (Module 3.1)

Embed query and docs into a shared semantic space. "Money back" and "refund" land near each other geometrically.

Fix 3

Learned sparse (next ↓)

SPLADE expands each doc/query with predicted related terms, then runs BM25-style scoring. Best of both worlds.

⚡ The honest takeaway: vocabulary mismatch is why "BM25 only" isn't enough for natural-language Q&A — and why hybrid retrieval (BM25 + dense) became the default. Sparse owns exact-match; dense owns paraphrase. You almost always want both.
Beyond BM25

SPLADE — Learned Sparse Retrieval

What if the inverted index stored not just words, but a model's predictions about what words should be there? That's SPLADE — and it sits in the same engine as BM25, no GPU at query time.

The big idea

SPLADE runs each document (and query) through a masked-language-model head over BERT's vocabulary (~30k tokens). The output is a sparse vector: most entries are zero, a few hundred are non-zero, each representing a vocabulary token plus an importance weight.

Crucially, many non-zero entries are tokens that never appeared in the original text — the model expands the doc with semantically related terms. "refund" pulls in "reimbursement, return, money" with learned weights.

SPLADE WEIGHT FOR TOKEN j IN DOC d
wj = ∑i ∈ d log(1 + ReLU(Sij))
Sij = MLM logit for vocab token j at position i
ReLU keeps only "this token belongs here" signal
log(1+·) saturates large activations (echoes BM25!)
∑ over positions = max-pool of evidence
+ FLOPS regularizer during training → enforces sparsity
STEP 01 · ENCODE

Run doc through SPLADE encoder (a BERT variant). Output: ~150 non-zero weights over the BERT vocabulary.

"refund policy" →
{refund:2.3, policy:1.8,
reimbursement:1.2, return:0.9,
money:0.6, ...}
STEP 02 · INDEX

Store the sparse vector in a standard inverted index. Each non-zero token becomes a posting list entry with its learned weight as TF.

Same data structure as BM25 — Lucene/Elasticsearch can serve SPLADE directly via term-impact scoring.

STEP 03 · QUERY

Encode the query the same way. Score = dot product of query & doc sparse vectors — looks up posting lists, sums weighted contributions.

~10–30ms query latency on millions of docs. No GPU at serve time.

Production Code · pyserini
from pyserini.encode import SpladeQueryEncoder
from pyserini.search.lucene import LuceneImpactSearcher

# 1. load the SPLADE encoder + a pre-built sparse index
encoder = SpladeQueryEncoder("naver/splade-cocondenser-ensembledistil")
searcher = LuceneImpactSearcher.from_prebuilt_index(
    "msmarco-v1-passage-splade-pp-ed", encoder
)

# 2. encode + retrieve — same API shape as BM25
hits = searcher.search("how do I get my money back", k=10)

# 3. inspect — note expansion to "refund", "reimbursement"
for h in hits:
    print(h.docid, h.score, h.raw[:80])

When SPLADE wins

  • You want dense-style recall on existing Lucene infra.
  • No GPU budget at serve time.
  • You want interpretable scores (you can read the expanded terms).

When to skip

  • Indexing time is 5–10× BM25 (GPU pass per doc).
  • Domain shift hurts — SPLADE is pre-trained on MS MARCO.
  • If you already run a dense retriever + reranker, gains are marginal.
⚡ The unifying picture: BM25 = hand-engineered sparse vectors. SPLADE = learned sparse vectors. Dense retrieval = learned dense vectors. ColBERT = learned dense token-level vectors. All four ride the same conceptual rail; they just differ in where the learning happens.
Self-Check

Three Quick Questions

Cement the BM25 instincts before moving on to dense.

QUESTION 01

A user types "CVE-2024-31082" into your search bar. Which retriever is most likely to nail it on the first hit?

Correct. Rare, structured identifiers carry zero semantic signal — embeddings blur them. BM25's IDF gives a huge weight to such terms, and exact-token matching surfaces the right doc instantly.
QUESTION 02

In the BM25 formula, what does the k₁ parameter primarily control?

Correct. k₁ sets how aggressively repeated mentions of a term stop adding value. Length normalization is b's job.
QUESTION 03

You stemmed your corpus at index time but forgot to stem the query. What happens?

Correct. The query "purchases" never matches the indexed token "purchas". Always apply the exact same tokenization pipeline to docs and queries — this is the #1 BM25 debugging tip.
