Foundation

Text Becomes a Point in Space

A sentence enters as words. A neural encoder turns it into a fixed-length vector. Vectors with similar meanings sit close. That's the entire trick.

EMBEDDING SPACE · 2D PROJECTION
refund / cancel
travel / flights
warranty / repairs
shipping / delivery
⚡ What you're seeing: 16 chunks projected from 1536-d down to 2D. They cluster by topic — without anyone telling the model what the topics are. The model learned that during pretraining on billions of word pairs.
IDEA 01

Tokens → contextual vectors

An encoder (BERT-style transformer) reads every token in context — "bank" near "river" ends up nothing like "bank" near "loan".

IDEA 02

Pool to one vector

Average (or use the special [CLS] token's output) to collapse N token vectors into one fixed-size vector per chunk.

IDEA 03

Normalize to unit length

Divide by ‖v‖. After this, every vector sits on the unit sphere — and dot product = cosine similarity, the fastest possible comparison.

IDEA 04

Retrieval = nearest neighbors

Embed the query the same way. Find the K vectors closest to it. Return their original chunks. That's the whole search.

The Math

How "Close" Is Measured

Three metrics. Almost always cosine. Here's why — and what each one is actually computing.

Metric 01 · Default

Cosine Similarity

Measures the angle between two vectors. Ignores their length. Range [-1, 1]; identical direction = 1, perpendicular = 0, opposite = -1.

cos(a, b) = (a · b) / (‖a‖ · ‖b‖)

Use when: almost always. It's the right default for text embeddings.

Metric 02 · Fast Path

Dot Product

Element-wise multiply and sum. Cheaper than cosine — no normalization step. When vectors are already unit-normalized, dot product ≡ cosine.

a · b = ∑ aᵢ · bᵢ

Use when: you've pre-normalized — the default for OpenAI, BGE, E5 etc.

Metric 03 · Distance

Euclidean (L2)

Straight-line distance between two points. Smaller = more similar. For unit-normalized vectors, L2 and cosine give the same ranking.

‖a − b‖₂ = √(∑(aᵢ − bᵢ)²)

Use when: the model is trained with L2 loss (some BERT variants, image embeddings).

Worked Example — Cosine in 3 Steps 

import numpy as np

a = np.array([0.2, 0.7, 0.4, 0.5])     # "refund policy"
b = np.array([0.3, 0.6, 0.5, 0.4])     # "return rules"
c = np.array([0.9, 0.1, 0.2, 0.3])     # "shipping zones"

# 1. raw cosine
def cosine(x, y):
    return x @ y / (np.linalg.norm(x) * np.linalg.norm(y))

cosine(a, b)   # → 0.973  (very similar)
cosine(a, c)   # → 0.680  (less similar)

# 2. faster: pre-normalize, then use dot
a_norm = a / np.linalg.norm(a)
b_norm = b / np.linalg.norm(b)
a_norm @ b_norm   # → 0.973 (identical, but no division at query time)

# 3. batched (the real production version)
corpus = np.vstack([b, c])
corpus = corpus / np.linalg.norm(corpus, axis=1, keepdims=True)
scores = corpus @ a_norm   # → [0.973, 0.680]
⚡ The normalize-once trick: normalize all vectors when you index them. Then every retrieval is a single matrix-vector multiply — no division, no norm() calls in the hot path. This is how production engines do millions of comparisons per second.

By Hand — Every Multiplication, Every Sum

VECTORS (4-DIM TOY EMBEDDINGS)
a = [0.2, 0.7, 0.4, 0.5]  # "refund policy"
b = [0.3, 0.6, 0.5, 0.4]  # "return rules"
c = [0.9, 0.1, 0.2, 0.3]  # "shipping zones"
DOT PRODUCT a · b:
= (0.2)(0.3) + (0.7)(0.6) + (0.4)(0.5) + (0.5)(0.4)
= 0.06 + 0.42 + 0.20 + 0.20
= 0.88
MAGNITUDES & COSINE
‖a‖ = √(0.04 + 0.49 + 0.16 + 0.25) = √0.94 ≈ 0.970
‖b‖ = √(0.09 + 0.36 + 0.25 + 0.16) = √0.86 ≈ 0.927
cosine(a, b) = 0.88 / (0.970 · 0.927)
              = 0.88 / 0.899
              ≈ 0.979 → very similar
cosine(a, c):
= (0.18 + 0.07 + 0.08 + 0.15) / (0.970 · √0.95)
= 0.48 / (0.970 · 0.975)
≈ 0.508 → less similar
⚡ The takeaway: cosine is just (dot product) / (product of magnitudes). With unit-normalized vectors, ‖a‖=‖b‖=1, so cosine = dot. That's why every modern embedding model normalizes before shipping — one less division per query.
Architecture

DPR — How Dense Retrievers Are Trained

Dense Passage Retrieval (Karpukhin et al., 2020) showed that two BERTs and a contrastive loss could beat 25 years of BM25 on open-domain QA. Every modern embedding model — BGE, E5, GTE, OpenAI's text-embedding-3 — descends from this recipe.

The bi-encoder architecture

Two BERT encoders, trained independently but in lockstep. The query encoder EQ embeds questions; the passage encoder EP embeds documents. At query time, only EQ runs — every passage was pre-encoded offline.

q ─► E_Q(q) ─► [768-dim vector] ─┐
p ─► E_P(p) ─► [768-dim vector] ─┤ → sim(q, p) = q · p
independent forward passes ┘

Independent encoding is what makes dense retrieval scalable — you precompute passage vectors once and reuse them for every future query.

The contrastive objective

For each training example, you have one query q, one relevant passage p⁺, and a pile of irrelevant ones p⁻. The loss pulls q toward p⁺ and pushes it away from every p⁻.

DPR LOSS (NEG LOG LIKELIHOOD)
L = −log exp(sim(q, p⁺)) exp(sim(q, p⁺)) + ∑ exp(sim(q, p⁻))
softmax-over-negatives · pull positive up, push negatives down
TRICK 01 · IN-BATCH NEGATIVES

For each batch of B (query, positive) pairs, treat the other B−1 passages in the batch as negatives. One forward pass, B² comparisons — free hard negatives.

TRICK 02 · HARD NEGATIVES

In-batch random negatives are easy — model crushes them by epoch 2. Mine BM25-top-K passages that look relevant but aren't → these force the model to learn real distinctions.

TRICK 03 · LARGE BATCHES

More negatives per batch = stronger gradient signal. DPR used B=128; modern recipes (E5, BGE) use B=2048+ with gradient checkpointing.

Bi-Encoder

Two independent towers

  • ✦ Encodes query and doc separately
  • ✦ Precompute passages once → ANN at query time
  • ✦ ~10ms / query over millions of docs
  • ✦ Trained with contrastive / in-batch negatives loss
  • ✦ Misses fine-grained query-doc interactions
Cross-Encoder

Joint query+doc encoding

  • ✦ Concatenates [CLS] q [SEP] d [SEP] → BERT
  • ✦ Full attention between every q-token and d-token
  • ✦ Trained with pointwise / pairwise (margin) loss
  • ✦ ~50-200ms per (q, d) — N× slower
  • ✦ State-of-the-art accuracy → used as reranker
⚡ The two-stage pattern that runs production: bi-encoder for fast first-stage recall (top-100 from millions), then cross-encoder reranker over those 100. Bi-encoder's job is "don't miss the answer"; cross-encoder's job is "find it in the candidate set." (We'll build this in Module 3.3.)
Beyond Single Vectors

ColBERT & Late Interaction

Single-vector retrieval forces the entire passage into one 768-dim point. ColBERT (Khattab & Zaharia, 2020) keeps one vector per token and defers the interaction to query time — getting cross-encoder quality at bi-encoder speed.

The MaxSim operator

Encode the query into n token vectors and the passage into m token vectors. For each query token, find its most similar passage token (max over m). Sum those maxima over the query.

COLBERT SCORE
score(Q, D) = ∑i ∈ Q  maxj ∈ D  (qi · dj)
"For every query token, what's the best match in the doc?"

Why "late"?

Early interaction (cross-encoder): query and doc are concatenated, attended before encoding. Accurate but uncacheable — every (q, d) is a fresh forward pass.

No interaction (bi-encoder): query and doc are encoded independently, pooled to one vector each. Fast but coarse.

Late interaction (ColBERT): independent encoding (cacheable!), but the comparison happens at token granularity. You get most of cross-encoder's expressiveness with bi-encoder's offline-indexable structure.

Worked Example — MaxSim by Hand

QUERY · 3 TOKENS
q₁ = vec("refund")
q₂ = vec("for")
q₃ = vec("travel")
DOC · 5 TOKENS
d₁ d₂ d₃ d₄ d₅ = "our return policy on flights"
PER-TOKEN MAX
max over d for q₁ "refund" → d₂ "return" = 0.85
max over d for q₂ "for" → d₃ "policy" = 0.30
max over d for q₃ "travel" → d₅ "flights" = 0.78
SUM = 0.85 + 0.30 + 0.78 = 1.93

Notice: "refund" matched "return" and "travel" matched "flights" — bi-encoder pooling would have blurred those token-level alignments away.

Pro

Near-cross-encoder quality

On BEIR, ColBERTv2 closes ~80% of the gap between bi-encoders and cross-encoders, with sub-100ms latency.

Con

Index size

M tokens × 128-dim per doc vs. one 768-dim vector. ColBERTv2 fixes this with residual compression — still 4-5× a bi-encoder index.

Tooling

PLAID engine

ColBERTv2's PLAID retrieval engine clusters token vectors offline, then ranks candidates with MaxSim — fast enough for production.

⚡ The big picture — three flavors of dense retrieval: single-vector (DPR, BGE, OpenAI) → token-level late interaction (ColBERT) → full cross-attention (cross-encoder reranker). Latency and quality trade off in lockstep. Most production systems use single-vector + cross-encoder reranker; ColBERT slots in when you can afford its index but not the reranker's per-pair cost.
Measure Twice

BEIR & MTEB — Picking by Numbers

"Which retriever is best?" depends entirely on your domain. Two benchmarks let you stop guessing.

Retrieval-Only Benchmark

BEIR (Thakur et al. 2021)

18 retrieval datasets across 9 domains — bio-medical (TREC-COVID), legal (Touché), financial (FiQA), Wikipedia (NQ, HotpotQA), Twitter (Signal), code (CodeSearchNet)... Zero-shot: no fine-tuning allowed.

Standard metric: nDCG@10. Higher = better ranking quality in the top 10 results.

Embedding Quality Benchmark

MTEB (Muennighoff et al. 2022)

Massive Text Embedding Benchmark — 56 tasks across 8 task families: retrieval, reranking, classification, clustering, pair-classification, STS, summarization, bitext-mining.

Tells you "is this embedding model good in general?" rather than "is it good at retrieval specifically?" — useful when you'll reuse the model for classification or clustering too.

MTEB · Picking a Model — A Decision Tree

STEP 1 · LANGUAGE

English-only? → BGE-large-en / GTE-large. Multilingual? → BGE-m3 / multilingual-e5. Code? → Voyage-code-2.

STEP 2 · DOMAIN MATCH

On MTEB, filter by your closest task family. If your data is medical, check BEIR's TREC-COVID column instead of the overall average.

STEP 3 · BUDGET

Hosted (OpenAI text-embedding-3-large, Voyage-3) vs self-hosted (BGE, E5). Hosted = better dev velocity; self-hosted = $0.0001/query and zero data-egress.

STEP 4 · DIM SIZE

Larger ≠ always better. 384-dim BGE-small often beats 1536-dim hosted on niche domains. Index size is 4× though — measure recall, then decide.

CRITERION 01

Benchmark on YOUR data

MTEB is a leaderboard; your domain is reality. Build a golden set of 30-50 query/answer pairs and measure Recall@10 across two or three candidates.

CRITERION 02

Cost & latency budget

10M chunks × text-3-large ≈ ~$80 one-shot. Self-hosting BGE on a GPU pays back fast at scale; APIs win at low volume.

CRITERION 03

Lock-in & vendor risk

Once embedded, you can't switch models without a full re-index. Document the model name + version in every chunk's metadata.

⚡ The Matryoshka trick: OpenAI's text-embedding-3 family supports configurable dimensions. Request 1536 dims for storage, 256 for fast pre-filtering. Same model, three index sizes. Don't ignore this — it's a real lever on cost.
Scaling Retrieval

Approximate Nearest Neighbor Search

Comparing one query to a million vectors with brute force = 6 GB of math per query. ANN trades a tiny amount of accuracy for a 100-1000× speedup.

HNSW

DEFAULT

Hierarchical Navigable Small World. Build a multi-layer graph where each layer is a "highway" of long links. Search descends from sparse to dense — log(N) hops to the neighborhood.

Recall@10~99%
Latencysub-ms
RAM costhigh
Build timemoderate

Best for: in-memory indexes up to ~10M vectors. The default of Qdrant, Weaviate, pgvector, Milvus.

IVF + PQ

DISK-FRIENDLY

Inverted File + Product Quantization. K-means clusters vectors into buckets; quantization compresses each vector to 16-64 bytes. Search visits only nearby buckets.

Recall@10~95%
Latencylow
RAM cost~50× lower
Build timeslow (training)

Best for: 100M+ vectors that won't fit in RAM at full precision. Faiss' bread and butter.

ScaNN / DiskANN

SPECIALIZED

Anisotropic quantization (ScaNN) or SSD-optimized graphs (DiskANN). Google & Microsoft's answer to "what if you don't have 100GB of RAM?"

Recall@10~97%
Latencylow (with SSD)
RAM costvery low
Maturitygrowing

Best for: billions of vectors on commodity hardware. Pinecone Serverless & Vespa use DiskANN-style designs.

Recall vs Latency · the ANN trade

ILLUSTRATIVE

Brute-force is the gold standard for recall but scales linearly. ANN gives up <5% recall for 100× lower latency. The curve shape is universal.

⚡ Tune ef_search: HNSW's main runtime knob. Higher = better recall, slower queries. Start at 64 and scale up only if recall is bad on your golden set.
⚡ Measure recall@K on YOUR data: 95% recall on MTEB ≠ 95% recall on your corpus. Always run a brute-force baseline on a sample and compare.
⚡ Quantize for storage: int8 or binary quantization shrinks vectors 4-32× with tiny recall loss. Combine with HNSW for the best of both worlds.

Production Recipes

Three Stacks, Same Pattern

Hosted API for speed-to-ship, in-memory for prototypes, full vector DB for production. Pick the lightest one that fits.

from openai import OpenAI
import numpy as np

client = OpenAI()

def embed(texts: list[str]) -> np.ndarray:
    # batch up to 2048 inputs — much cheaper than one-at-a-time
    resp = client.embeddings.create(
        model="text-embedding-3-small",
        input=texts
    )
    vecs = np.array([d.embedding for d in resp.data])
    # OpenAI returns unit-normalized vectors already
    return vecs

# index corpus once
corpus_vecs = embed(corpus_chunks)            # (N, 1536)

# top-K at query time
q = embed([user_query])[0]               # (1536,)
scores = corpus_vecs @ q                       # dot = cos (pre-normalized)
top = np.argsort(-scores)[:10]               # indices of top 10

When to use

  • Speed-to-ship is paramount
  • < 5M chunks (otherwise $$$)
  • No GPU ops in your team
  • Data residency / on-prem constraints
⚡ Always batch: single-call embedding is 50× more expensive than batched per-token. Pipeline 200-1000 texts per request. 
from sentence_transformers import SentenceTransformer
import numpy as np

# loads from HF Hub the first time; ~1.3GB for BGE-large
model = SentenceTransformer("BAAI/bge-large-en-v1.5", device="cuda")

# BGE asks you to prefix queries (NOT chunks) for best recall
corpus_vecs = model.encode(
    corpus_chunks,
    batch_size=64,
    normalize_embeddings=True,
    show_progress_bar=True
)                                                  # (N, 1024) on unit sphere

def search(query: str, k: int=10):
    q = model.encode(
        "Represent this query for searching: " + query,
        normalize_embeddings=True
    )
    scores = corpus_vecs @ q
    return np.argsort(-scores)[:k], scores

When to use

  • Data must stay on-prem
  • Fine-tuning the encoder is on the table
  • Predictable per-call cost
  • Multilingual via E5 / multilingual-e5
⚡ Mind the prefix: BGE, E5, and some other models want a different prefix for queries vs documents. Get this wrong and recall drops 10-20%.

Self-Check

Three Quick Questions

Cement the dense-retrieval instincts before moving on to hybrid.

QUESTION 01

You normalize all your vectors to unit length at index time. Which similarity becomes the cheapest to compute at query time?

Correct. When ‖a‖ = ‖b‖ = 1, cosine simplifies to a · b. A single matrix-vector multiply scores millions of candidates — no division anywhere in the hot path.
QUESTION 02

You have 50 million chunks and need sub-50ms retrieval on a single commodity server. Best ANN choice?

Correct. Pure HNSW at 1536-d × 50M = ~300GB RAM at fp32 — too much. Quantizing to int8 or PQ cuts that 4-32×, keeping recall above 95%. K-d trees collapse in high dimensions.
QUESTION 03

You swap your embedding model from text-embedding-3-small to BGE-large mid-project. What MUST you do?

Correct. Each model carves up the embedding space differently — there's no shared coordinate system. Mixing vectors from two models gives meaningless similarities. Always tag chunks with the model + version used.
