The Lift

Why Rerank at All?

Your retriever's top-50 almost always contains the right answer — it's just not at rank 1. A reranker fixes that. Here's exactly how, and why it works.

Retrieval alone

Recall is high · precision is shaky

A well-tuned hybrid retriever lands the right document in the top-50 about 95% of the time. But the right doc lands at top-1 only ~50-65% of the time. The model sees the wrong context first.

  • Recall@50: ~95%
  • Precision@1: 50-65%
  • Top-K context window often noisy
With a reranker

Recall preserved · precision soars

Take the same top-50, run them through a cross-encoder, and the right doc moves to top-1 about 75-85% of the time. The LLM sees the right context — answer quality jumps.

  • Recall@50: ~95% (unchanged)
  • Precision@1: 75-85%
  • Cleaner context → less hallucination
⚡ The unspoken truth: Retrievers are graded on recall. The LLM is graded on precision (the top-K of context that lands in its prompt). A reranker is the only stage in your pipeline that directly optimizes for the LLM's view of the world.

Four scenarios · when rerank pays off most

SCENARIO 01

Long, mixed-topic chunks

Embeddings blur multi-topic chunks. A cross-encoder reads query + chunk jointly and sees what actually matches.

SCENARIO 02

Subtle paraphrase

"Get my money back" → embeddings recognize loose topic match. The reranker confirms the exact intent.

SCENARIO 03

Distractor docs in the corpus

"Refund travel" hits many travel docs. The reranker separates "refund travel" from "travel deals".

SCENARIO 04

Domain-specific intent

Generic embeddings miss medical / legal / code nuance. Fine-tuned rerankers (or LLM-judges) fix it.

The Tradeoff

Choosing Top-K · Quality vs Latency

Reranking 10 candidates is cheap. Reranking 200 is expensive. The sweet spot exists — and you find it by measuring on a golden set.

nDCG@10 vs Latency · the rerank tradeoff

ILLUSTRATIVE

Quality climbs sharply with the first 25-50 candidates, then plateaus. Past ~100, you pay latency for diminishing returns.

RULE 01

Retrieval top-K ≥ 4× rerank top-K

If you want top-5 after rerank, pull 50 from retrieval. Pulling only 10 strangles the reranker.

RULE 02

Most teams land on 50

50 candidates is the practical sweet spot: covers ~95% of "the answer is in here" cases without breaking your latency budget.

RULE 03

Batch the API call

Send all 50 (query, doc) pairs in one call. Latency scales sub-linearly thanks to batching on the server side.

Production Recipes

Reranking in Production

Cohere API for fastest ship. Sentence-Transformers for self-hosting. LLM-as-judge for heavyweight domain decisions.

import cohere

co = cohere.ClientV2()

# 1. Stage 1 — pull 50 candidates from your retriever
candidates = hybrid_retriever.search(query, top_k=50)
# candidates: list[{id, text, ...}]

# 2. Stage 2 — Cohere reranks them in a single call
resp = co.rerank(
    model="rerank-v3.5",
    query=query,
    documents=[c["text"] for c in candidates],
    top_n=5,                       # return only the best 5
    return_documents=False
)

# 3. Map reranked indices back to your candidate records
reranked = [
    {**candidates[r.index], "score": r.relevance_score}
    for r in resp.results
]

# 4. Send only the top 5 to the LLM
context = "\n\n".join(d["text"] for d in reranked)
answer = llm.generate(prompt=build_prompt(query, context))

When to use

  • Fastest path to "good quality"
  • Multilingual out of the box
  • Single batched API call
  • Data leaves your perimeter
⚡ Production tip: Cache rerank results keyed by hash(query + sorted(candidate_ids)). Many queries repeat. 
from sentence_transformers import CrossEncoder
import numpy as np

# load once at startup — ~1.5GB on GPU
reranker = CrossEncoder(
    "BAAI/bge-reranker-v2-m3",
    max_length=512,
    device="cuda"
)

def rerank(query: str, candidates: list[dict], top_n: int=5):
    # build (query, doc) pairs — that's the cross-encoder input format
    pairs = [(query, c["text"]) for c in candidates]

    # single batched forward pass — much faster than per-pair calls
    scores = reranker.predict(
        pairs,
        batch_size=32,
        show_progress_bar=False
    )

    # sort descending by reranker score
    order = np.argsort(-scores)
    return [
        {**candidates[i], "rerank_score": float(scores[i])}
        for i in order[:top_n]
    ]

# use it
candidates = hybrid_retriever.search(query, top_k=50)
top_5 = rerank(query, candidates, top_n=5)

When to use

  • Data residency / on-prem
  • Volume too high for API cost
  • Want to fine-tune on domain
  • Predictable per-call latency
⚡ Alternatives: Jina-v2 open weights (similar quality, smaller), MS-MARCO-MiniLM (tiny, CPU-friendly, English-only). 
from openai import OpenAI
from pydantic import BaseModel, Field
import asyncio

client = OpenAI()

class RelevanceScore(BaseModel):
    score: int = Field(ge=0, le=10, description="0=unrelated, 10=perfect")
    reason: str = Field(max_length=120)

async def score_one(query, doc):
    resp = await client.chat.completions.parse(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "Score doc relevance to query. Be strict."},
            {"role": "user", "content": f"Q: {query}\nDOC: {doc['text']}"}
        ],
        response_format=RelevanceScore
    )
    return {**doc, "score": resp.choices[0].message.parsed.score}

async def rerank_with_llm(query, candidates, top_n=5):
    # concurrent scoring — 50 candidates fan out as 50 parallel calls
    scored = await asyncio.gather(*[
        score_one(query, c) for c in candidates
    ])
    return sorted(scored, key=lambda x: -x["score"])[:top_n]

When to use

  • Complex domain rules (legal, medical)
  • Need a rubric you can iterate via prompt
  • 10-50× cost of a dedicated reranker
  • Listwise variants are unstable
⚡ Reserve this: for evaluation pipelines and one-shot quality bumps. Don't make it your hot-path reranker without a serious latency budget.
Common Pitfalls

Where Rerankers Go Wrong

Adding a reranker is one of the highest-ROI moves in RAG — if you avoid these traps.

Candidate pool too small

Reranking only the top-5 from retrieval. If the right doc is at rank 23, the reranker never sees it. Fix: rerank at least 30-50 candidates. Quality gains compound.

No latency budget

Reranking 200 candidates with an LLM-judge: 2-3 seconds, every query. Users won't tolerate it. Fix: measure P95 latency, pick top-K accordingly. 50 is usually the sweet spot.

No fallback when reranker fails

API throttles or errors out — your whole RAG pipeline breaks. Fix: wrap rerank in try/except; on failure, return retrieval order. Degrade gracefully.

No golden set to measure on

"Rerank feels better" doesn't survive a quarterly review. Without 50+ labeled query/answer pairs, you can't justify the latency cost. Fix: ship rerank A/B with metrics from day one.

Permission filters skipped

Retrieval filters by ACL → rerank sees the right candidates. But if you rerank before ACL filtering, scores leak across users. Fix: always filter at retrieval, never at rerank.

No caching

The same query hits your API 1000× per day. Each one costs $0.002. Fix: cache rerank results by hash(query + candidate_ids). Costs drop 90%.

Self-Check

Three Quick Questions

Cement rerank instincts. Track 3 closes when you ace this.

QUESTION 01

Why can't a cross-encoder be used directly as your first-stage retriever?

Correct. Bi-encoders pre-compute doc vectors once. Cross-encoders take both query and doc together every time. That's why we cast a wide net with the bi-encoder (or BM25), then refine the small candidate pool with the cross-encoder.
QUESTION 02

Your retriever pulls top-10 and your reranker re-scores those 10. Final P@1 doesn't improve much. Best fix?

Correct. Rerankers can only re-order what they're shown. If retrieval misses the right doc in top-10, the reranker is helpless. Always pull a generous candidate pool (4-10× your final top-K) before reranking.
QUESTION 03

Best place to apply ACL / permission filters in a retrieval+rerank pipeline?

A
After rerank — on the final top-K
Correct. Filter at retrieval. Filtering after rerank wastes compute and risks leakage (you've already paid to score forbidden docs, and a bug could let them through). Filtering in the prompt is the worst option — LLMs don't reliably obey "don't mention X".
