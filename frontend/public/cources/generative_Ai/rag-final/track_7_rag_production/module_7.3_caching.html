The Pareto Win

Why Caching Matters For RAG

Real RAG traffic isn't uniform. 20% of queries account for 80% of volume — and 60% of users ask near-identical questions. Caching turns repetition into a free win.

WIN · 01

Cost

A cache hit costs ~0.01¢ (Redis lookup). A cache miss costs 2-15¢ (LLM tokens).

50% hit rate ≈ 50% cost cut
WIN · 02

Latency

Cache hit: ~10 ms. LLM call: 1-3 seconds. The cached user feels the difference instantly.

~100-300× faster on hits
WIN · 03

Resilience

OpenAI down? Cached answers still serve. A cache is also a backup brain.

Graceful degradation

Real Traffic Looks Like This

Top 5 unique questions 38% of all traffic
Top 20 unique questions 62% of all traffic
Top 100 unique questions 81% of all traffic
Long tail (everything else) 19%
📍 The implication: You only need to cache 100 question answers to cover 81% of traffic. That's a tiny cache (~1 MB) for an enormous cost cut. Caching is one of the highest-ROI moves in RAG.
Don't Mix These Up

Exact vs Semantic vs Prompt Cache

Three caches sit in a RAG stack — same word, three completely different mechanisms. Conflating them is why teams over-cache, under-cache, or trust a hit they shouldn't have.

Type 1

Exact Cache

Hash the normalized query (lowercased, trimmed, optionally tokenized) → look up exact response.

key: sha256("refund window?")
value: stored answer + sources
store: Redis · <1ms
✓ Hit rate: 10-20% (only literal repeats)
✓ Risk: ~zero — same key = same answer
✗ Brittle: "refund window" vs "Refund window?" → miss
Type 2

Semantic Cache

Embed the query, do ANN search over cached query embeddings. Hit if cosine > threshold.

key: emb("refund window?")
value: ANN-matched answer if sim ≥ 0.92
store: Qdrant/Redis-vector · 10-30ms
✓ Hit rate: 30-50% (paraphrases hit too)
⚠ Risk: false hits if threshold too low
✗ Unsafe for: time-sensitive, personalized, factual queries
Type 3

Prompt Cache

A provider feature (OpenAI, Anthropic, Gemini): the shared prefix of consecutive prompts is cached on the provider's side — first-token latency and token cost drop.

key: hash of prompt prefix (provider-side)
value: KV cache state at prefix boundary
store: provider GPU memory · transparent
✓ Hit rate: high — stable system prompts hit every call
✓ Risk: none — answer is still computed fresh
⚠ Different goal: latency & cost, NOT response reuse

Each layer fires only on misses from the one above. L1 catches literal repeats cheap; L2 catches paraphrases; L3 makes the misses themselves cheaper.

⚡ The most common confusion: "we added semantic caching, our prompt-cache hit rate dropped" — these are different things. The prompt cache fires after retrieval (your prompt prefix is the same), regardless of whether L1/L2 hit. They compound.
⚠ When semantic cache becomes a footgun: "What's my refund status?" — cosine-matching across users will hand user A's status to user B. Either partition the cache by user_id (drops the hit rate) or simply don't semantic-cache personalized queries (route them past L2).
⚡ Rule of thumb for which to add first: turn on prompt cache the day you ship — it's free, provider-side, no code. Add exact cache for your top 100 FAQ-shaped queries. Reach for semantic cache only after you have eval coverage to catch false hits — otherwise it's the cache that quietly hurts users.

A Tiered Cache In ~30 Lines

import redis, hashlib, json

cache = redis.Redis(host="redis")

def _key(prefix, text):
    return f"{prefix}:{hashlib.sha256(text.encode()).hexdigest()[:16]}"

async def answer(question):
    # ---------- LAYER 1: semantic cache ----------
    hit = await semantic_lookup(question, threshold=0.95)
    if hit:
        log_cache("semantic_hit")
        return hit["answer"]

    # ---------- LAYER 2: embedding cache ----------
    k_emb = _key("emb", question)
    vec = cache.get(k_emb)
    if vec:
        vec = json.loads(vec)
    else:
        vec = await embed(question)
        cache.setex(k_emb, 86400, json.dumps(vec))   # 24h TTL

    chunks = await qdrant.search(vec, k=5)

    # ---------- LAYER 3: prompt cache (provider-native) ----------
    resp = await llm.messages.create(
        model="claude-sonnet-4-5",
        system=[{
            "type": "text",
            "text": SYSTEM_PROMPT,
            "cache_control": {"type": "ephemeral"}   # ← cached input!
        }],
        messages=[{"role":"user", "content": build_msg(question, chunks)}]
    )

    answer = resp.content[0].text
    await semantic_store(question, vec, answer, ttl=3600)   # 1h TTL
    return answer
The Big Win

Semantic Caching with Redis

Exact-match caching catches "What's our refund policy?" twice. Semantic caching also catches "How do refunds work?" — same answer, paraphrased question. This is where the real volume hides.

Exact-Match Cache

Hash → Answer

Hash the question, store the answer. Lightning fast, but misses any paraphrase.

✓ HIT  "What's our refund policy?"
✗ MISS "What is our refund policy?"
✗ MISS "How do refunds work?"
✗ MISS "refund policy please"
Typical hit rate: 5-15%
Semantic Cache

Embed → ANN Search

Embed the question. If a stored question is similar enough (cosine ≥ 0.95), return its answer.

✓ HIT  "What's our refund policy?"
✓ HIT  "What is our refund policy?"
✓ HIT  "How do refunds work?"
✓ HIT  "refund policy please"
Typical hit rate: 25-50%

Building It — Redis + Embeddings

from redis.commands.search.field import VectorField, TextField
from redis.commands.search.query import Query
import numpy as np

# ---------- ONE-TIME SETUP ----------
r = redis.Redis()
r.ft("semcache").create_index([
    VectorField("vec", "HNSW", {"TYPE":"FLOAT32", "DIM":1536, "DISTANCE_METRIC":"COSINE"}),
    TextField("answer"),
    TextField("question"),
])

# ---------- LOOKUP ----------
async def semantic_lookup(question, threshold=0.95):
    vec = await embed(question)
    q = Query("*=>[KNN 1 @vec $vec AS score]").return_fields("answer","score").dialect(2)
    res = r.ft("semcache").search(q, query_params={"vec": np.array(vec, np.float32).tobytes()})

    if len(res.docs) and float(res.docs[0].score) >= threshold:
        return {"answer": res.docs[0].answer}
    return None

# ---------- STORE ----------
async def semantic_store(question, vec, answer, ttl=3600):
    key = f"semcache:{hashlib.sha256(question.encode()).hexdigest()[:16]}"
    r.hset(key, mapping={
        "vec": np.array(vec, np.float32).tobytes(),
        "question": question,
        "answer": answer
    })
    r.expire(key, ttl)
🎯 The threshold dial: 0.95 = strict, near-paraphrase only — safest default. 0.90 = looser, catches more, occasional wrong-answer risk. 0.98 = nearly exact — barely better than hash. Tune with offline eval.

Where Semantic Caching Goes Wrong

RISK · 01

Same intent, different parameter

"refund for order 4421" vs "refund for order 9930" — embedding may flag them as similar. Wrong answer returned.

✓ Don't cache personalized / parametric queries
RISK · 02

User context bleed

"What's MY balance?" cached from user A served to user B. Privacy incident.

✓ Namespace by user_id or tenant_id
RISK · 03

Stale on doc updates

Cache says "30-day window" but the policy doc changed to 14 days yesterday. Old answer keeps shipping.

✓ Short TTL + invalidation on doc reindex
RISK · 04

Threshold too loose

Hit rate jumps to 80%, but users complain about wrong answers. Semantic ≠ identical.

✓ Eval before tuning down — measure correctness on hits
The Hardest Problem

Cache Invalidation

"There are only two hard things in computer science: cache invalidation and naming things." For RAG, invalidation is doubly hard — your docs change, your prompts change, your model changes.

STRATEGY · 01

TTL Expiry

Every entry expires after N seconds. Dumb, automatic, almost always your default.

1h for hot data
24h for slow-changing
STRATEGY · 02

Tag-Based Eviction

Tag entries with chunk IDs / doc IDs. When a doc updates, evict all entries with that tag.

tags: [doc_42, doc_88]
on update → evict
STRATEGY · 03

Version Namespacing

Cache keys include prompt/model version. New version = whole cache effectively empty.

key: "v3:emb:abc..."
bump v → fresh space
STRATEGY · 04

Manual / Admin Flush

An ops endpoint or feedback button (👎) clears specific entries on demand.

POST /admin/cache/flush
or per-key invalidation

Which Strategy For Which Trigger?

Trigger Strategy Notes
Time-sensitive (news, pricing) TTL (5-60 min) Tradeoff freshness vs hit rate
Doc reindexed in pipeline Tag eviction by doc_id Hook into your ingestion job
New prompt template deployed Version namespace bump Whole cache effectively reset
User clicks 👎 Manual evict that key Don't poison future users
Model switch (gpt-4o → 4o-mini) Version namespace Different model = different answer
🚨 The unforgivable bug: Forgetting to invalidate after a doc update — and serving stale answers for weeks because the cache never expires. Wire your ingestion job to the cache from day one, even if it just calls a no-op endpoint for now. It's far harder to add later.

The Five Cache Metrics That Matter

METRIC · 01

Hit Rate

Percentage of requests served from cache. Track per-layer (semantic / embedding / prompt) separately.

METRIC · 02

Hit-Latency

P95 latency of cache hits. Should be < 50ms. If higher, your cache is the wrong tool or misconfigured.

METRIC · 03

Stale-Serve Rate

% of cache hits where the answer was outdated (caught via re-eval). Should be near 0%.

METRIC · 04

Eviction Rate

If items are evicted before being read, your cache is undersized — or your TTL is too short.

METRIC · 05

$ Saved

hits × cost_per_miss − hits × cost_per_hit. The number that justifies the cache to your CFO.

METRIC · 06

Quality on Hits

Periodically re-eval cached answers. Catches semantic-cache false positives early.

The Exceptions

When NOT to Cache

Caching is a default win — except when it isn't. Six situations where caching does more harm than good.

Personalized Answers

"My balance", "my open tickets", "my account status". The answer depends on who's asking.

✓ Either: bypass cache, or namespace by user_id

Time-Critical Queries

"What's the latest news about X?" "Current stock price?" Even 1-minute stale is wrong.

✓ Bypass cache or TTL < 30 seconds

Creative / Random Outputs

"Write me a joke." "Suggest a name." Users expect variety — same input shouldn't equal same output.

✓ Skip cache for creative endpoints

Tiny / Cheap Calls

If your LLM call costs $0.0001 and takes 200ms, the cache lookup costs $0.00005 and 10ms. Tiny savings + added complexity.

✓ Cache pays off above ~$0.005/call

Low-Volume / Long-Tail

10 requests/day, each a unique question. Cache hit rate ≈ 0%. Pure overhead.

✓ Skip caching until traffic justifies it

High-Stakes Domains

Medical, legal, financial. A stale or near-paraphrase answer can cause real harm. Strict policies, careful audit.

✓ Exact-match only, short TTL, full audit log
🌱 The default decision: Cache by default. Bypass for the 6 cases above. Measure quality on hits to catch silent regressions. A well-instrumented cache is one of the most profitable engineering investments in any RAG system.
