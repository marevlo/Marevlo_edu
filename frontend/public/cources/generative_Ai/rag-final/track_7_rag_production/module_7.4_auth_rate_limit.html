Token Bucket VS Sliding Window

      


      

        

          

            

              
            

            

              
Algorithm A

              
Token Bucket

            

          

          
Bucket of capacity C refills at rate R tokens/sec. Each request takes one token. Excess requests are dropped or queued.


          

            
Allows bursts — full bucket = instant 10 reqs

            
Smooth long-run — caps sustained rate at R

            
O(1) memory & compute per request

            
Hard to reason about exact "N per minute"

          


          
BEST FOR · USER-FACING APIs

        


        

          

            

              
            

            

              
Algorithm B

              
Sliding Window

            

          

          
Count requests in the last N seconds. If count ≥ limit, deny. Implemented with sorted Redis sets or weighted-bucket approximation.


          

            
Precise — "60 reqs / 60 sec" means exactly that

            
No boundary spikes like fixed-window has

            
Headers like X-RateLimit-Remaining are easy

            
More memory per user (timestamp log)

          


          
BEST FOR · BILLING-GRADE METERING

        

      

    


    
    

      

        TOKEN BUCKET · REDIS

# Lua script — atomic check + decrement in Redis
local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refill_rate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

local bucket = redis.call('HMGET', key, 'tokens', 'ts')
local tokens = tonumber(bucket[1]) or capacity
local last_ts = tonumber(bucket[2]) or now

local elapsed = now - last_ts
tokens = math.min(capacity, tokens + elapsed * refill_rate)

if tokens < 1 then
  redis.call('HMSET', key, 'tokens', tokens, 'ts', now)
  return 0  -- denied
end

tokens = tokens - 1
redis.call('HMSET', key, 'tokens', tokens, 'ts', now)
redis.call('EXPIRE', key, 3600)
return 1  -- allowed

      


      

        SLIDING WINDOW · REDIS

async def allow_request(user_id: str, limit: int = 60):
    key = f"rl:{user_id}"
    now = time.time()
    window = 60  # seconds

    # Drop entries older than window
    await redis.zremrangebyscore(key, 0, now - window)

    # Count what remains
    count = await redis.zcard(key)

    if count >= limit:
        return False, 0

    # Record this request
    await redis.zadd(key, {str(now): now})
    await redis.expire(key, window * 2)
    return True, limit - count - 1

# In FastAPI middleware:
@app.middleware("http")
async def rate_limit(request, call_next):
    user = request.state.user_id
    ok, remaining = await allow_request(user)
    if not ok:
        return JSONResponse({"error": "rate_limited"}, 429)
    response = await call_next(request)
    response.headers["X-RateLimit-Remaining"] = str(remaining)
    return response

      

    

  







  

    

       Layer 03
      

        Multi-Tenant Isolation
      

      
When tenant A queries the index, results from tenant B must be cryptographically invisible. Here are the three patterns — and the one Qdrant payload filtering enables.

    


    
    

      

        
01

        
Collection per Tenant

        
Each tenant gets its own Qdrant collection: vectors_acme, vectors_globex.

        

          
✓ Hardest isolation — no shared state

          
✓ Per-tenant index tuning

          
✗ Explodes at 10K+ tenants

          
✗ Cross-tenant analytics painful

        

        
→ USE FOR: Enterprise, <100 tenants

      


      

        

          
02

          RECOMMENDED
        

        
Payload Filtering

        
One shared collection. Every vector carries a tenant_id in its payload. Queries force a filter at the API gateway.

        

          
✓ Scales to millions of tenants

          
✓ Single index, cheaper

          
✓ Gateway enforces filter — apps can't bypass

          
✗ One bug = cross-tenant leak

        

        
→ USE FOR: SaaS, multi-tenant from day 1

      


      

        
03

        
Cluster per Tenant

        
Each tenant gets a dedicated Qdrant cluster — usually their own VPC, region, or hardware.

        

          
✓ Physical isolation

          
✓ Per-tenant data residency (GDPR, HIPAA)

          
✗ 10–100× cost

          
✗ Heavy ops

        

        
→ USE FOR: Regulated, Fortune 500

The #1 Multi-Tenant RAG Bug

          
A dev refactors the search function. They forget to thread the tenant_id through. Now every query is unscoped. Tenant A sees Tenant B's data — silently. Defense: enforce the filter at the gateway middleware, not the application code. Make it impossible to call the vector DB without a tenant scope.

# After each RAG query, report usage to Stripe
async def record_usage(tenant_id: str, tokens_used: int):
    # 1. Update internal counter (Redis HINCRBY for atomic increment)
    await redis.hincrby(f"usage:{tenant_id}:2026-05", "tokens", tokens_used)

    # 2. Check quota
    used = await redis.hget(f"usage:{tenant_id}:2026-05", "tokens")
    quota = await get_tenant_quota(tenant_id)

    if int(used) >= quota * 0.95:
        await send_warning_email(tenant_id, percent=95)

    # 3. Push to Stripe (batched every 5 min to avoid rate limits)
    await usage_queue.push({
        "subscription_item": await get_stripe_item(tenant_id),
        "quantity": tokens_used,
        "timestamp": int(time.time()),
        "action": "increment",
    })

    

  







  

    

       Layer 05
      

        Abuse Detection
      

      
Five attack patterns every public RAG API will see. Detect early, throttle silently, ban hard.

    


    

      

        

          
        

        
Credential Stuffing

        
Botnets try millions of leaked passwords on your login. Solution: per-IP login limits, captcha after 3 failures, account lockout policies.

        
HIGH FREQUENCY

      


      

        

          
        

        
Cost-Bomb Queries

        
Attacker sends 50K-token contexts to drain your OpenAI budget. Solution: cap input tokens per request, alert on cost-per-user spikes.

        
$$$ DRAIN

      


      

        

          
        

        
Scraping

        
Competitor crawls your public RAG endpoint to extract proprietary data. Solution: anomaly detection on query diversity, hidden honeypot docs.

        
SLOW BURN

      


      

        

          
        

        
DDoS / Layer 7

        
Flood your endpoint with auth-less traffic. Solution: Cloudflare / AWS WAF in front, IP reputation lists, JA3 fingerprinting.

        
VOLUMETRIC

      


      

        

          
        

        
Leaked Keys

        
Dev commits a key to a public GitHub repo. Solution: GitHub secret scanning, prefix-based detection, automated revocation pipelines.

        
CATASTROPHIC

      


      

        

          
        

        
Prompt Injection (free tier)

        
Free users probe for jailbreaks at high volume to find your system prompt. Solution: lower free-tier limits, behavioral fingerprinting, shadow-ban suspicious accounts.

        
EXFILTRATION

      

    


    

      🔭 Detection > reaction. Pipe all auth + rate-limit + quota events into your observability stack from Module 7.2. Build dashboards that alert on velocity: spike in 429s from one IP, spike in token-per-request average, spike in unique queries per minute. The fastest tenants to ban are the ones you spotted before they did damage.

QUESTION 01 · AUTH

        
You store API keys in your database. What value should the database row contain?

        

          
          
          
          
        

        

          Correct: B. SHA-256 is fast (millions/sec), one-way, and sufficient for tokens with high entropy. Bcrypt is for passwords — too slow per request. Encrypted keys mean if your DB leaks, so do all keys.
        

      


      
      

        
QUESTION 02 · RATE LIMITING

        
A user is allowed 60 requests per minute, but you want to allow a brief burst of 10 requests at once. Which algorithm fits best?

        

          
          
          
          
        

        

          Correct: B. Token bucket is the canonical answer for "long-run rate + short bursts." Sliding window is precise but spreads requests evenly. Fixed window allows boundary spikes. Leaky bucket forces even drip.
        

      


      
      

        
QUESTION 03 · MULTI-TENANCY

        
A junior dev refactors the search endpoint and forgets to pass tenant_id into the Qdrant query. What's the architectural fix?

        

          
          
          
          
        

        

          Correct: C. Defense in depth — humans forget, code reviews miss things. Make it structurally impossible to query the vector DB without a tenant scope. The gateway is the single chokepoint where the filter is injected. App code never touches Qdrant directly.
