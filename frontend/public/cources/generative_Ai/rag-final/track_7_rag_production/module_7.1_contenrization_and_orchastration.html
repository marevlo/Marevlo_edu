The Problem

Why Containerize a RAG?

Your laptop has Python 3.11. CI has 3.10. Prod has 3.12. Three environments, three sets of bugs. Containers turn three into one.

Before

Bare Python

Install Python. Install pip deps. Hope the system libraries match. Hope the OS matches. Hope CUDA matches.

  • "Works on my laptop, not CI"
  • sklearn / torch / faiss version drift
  • No rollback — you can't ship "the env from last Tuesday"
  • Onboarding new devs takes a day of YAML
After

Container

One immutable image. Same bits on your laptop, in CI, in staging, in prod. Versioned, shippable, rollback-able.

  • Pinned Python + pinned OS + pinned libs
  • docker pull rag:v0.4.2 = exact past behavior
  • One artifact for every target (k8s, Lambda, Cloud Run)
  • New devs onboard with one command

🎯 The takeaway: Containers share the host's kernel — no full OS overhead. That's why a RAG container starts in seconds, and why you can pack 10× more on a single VM. Everything from this module onward — k8s, serverless, compose — sits on this idea.
The Recipe

The Multi-Stage Dockerfile

Bad Dockerfile: 2.4 GB, rebuilds for every code change, runs as root. Good Dockerfile: 280 MB, layer-cached, non-root, secure. Here's the right pattern.

The Naive Dockerfile

FROM python:3.11
# Full Debian + dev tools ≈ 1 GB before you start

COPY . /app
# ALL files invalidate the cache on every change

WORKDIR /app
RUN pip install -r requirements.txt
# Reinstalled every code change — slow CI

CMD python -m uvicorn main:app --host 0.0.0.0
# Runs as root, no health check, no port declared
→ 2.4 GB · 12 min rebuild · root user · no health

Production-Grade Multi-Stage

# ---------- STAGE 1: builder ----------
FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# ---------- STAGE 2: runtime ----------
FROM python:3.11-slim
WORKDIR /app

# Non-root user — security 101
RUN useradd -m -u 1000 app

COPY --from=builder /root/.local /home/app/.local
COPY --chown=app:app . .

USER app
ENV PATH=/home/app/.local/bin:$PATH
EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]
→ 280 MB · 90s rebuild on code-only · non-root · /health

Five Practices That Matter

PRACTICE · 01

Slim Base Images

Use python:3.11-slim (~120 MB) over python:3.11 (~1 GB). For absolute minimum, distroless ships only Python + your code.

PRACTICE · 02

Layer Caching Order

Copy requirements.txt and install before copying source. Code changes invalidate fewer layers — rebuilds drop from 10 min to 30 sec.

PRACTICE · 03

Multi-Stage Builds

Build wheels / compile native deps in builder stage. Copy only the artifacts into the final image. Throws away gcc, headers, build tools.

PRACTICE · 04

Non-Root User

A compromised container as root = host compromise in many environments. USER app with UID > 1000 is non-negotiable for production.

PRACTICE · 05

HEALTHCHECK

Orchestrators (k8s, ECS, Cloud Run) need to know if your app is alive. A /health endpoint + HEALTHCHECK line gives them that signal.

PRACTICE · 06

.dockerignore

Skip .git, __pycache__, .venv, tests/, .env. Smaller context = faster builds + no secret leaks.

⚡ Why size matters: Every push to a registry, every pull on a new node, every cold start downloads the image. A 280 MB image cold-starts ~10× faster than 2.4 GB — and costs ~10× less in egress.
The Dev Stack

docker-compose for Local Stacks

RAG isn't one container. It's an API + a vector DB + maybe Redis + maybe Postgres. Compose stitches them together with one command.

A Full RAG Stack — docker-compose.yml

docker compose up 
services:

  api:
    build: .
    ports: ["8000:8000"]
    environment:
      OPENAI_API_KEY: ${OPENAI_API_KEY}
      QDRANT_URL:    http://qdrant:6333
      REDIS_URL:     redis://redis:6379
    depends_on:
      qdrant: { condition: service_healthy }
      redis:  { condition: service_started }
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s

  qdrant:
    image: qdrant/qdrant:v1.11.0
    ports: ["6333:6333"]
    volumes:
      - qdrant_data:/qdrant/storage
    healthcheck:
      test: ["CMD-SHELL", "timeout 2 bash -c '</dev/tcp/localhost/6333'"]
      interval: 10s

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: dev
    volumes:
      - pg_data:/var/lib/postgresql/data

volumes:
  qdrant_data:
  pg_data:
🎯 One command, full stack: docker compose up brings up API + Qdrant + Redis + Postgres with networking, volumes, and health checks. A new dev clones the repo and is running the full RAG in under 5 minutes — no README treasure hunt.

Service DNS

Services reach each other by name. Your API connects to http://qdrant:6333 — no IPs.

Named Volumes

Vector DB & Postgres data survive restarts. Wipe with docker compose down -v when you want a clean slate.

Dependency Ordering

depends_on + condition: service_healthy = API waits for Qdrant before booting.

Override Files

Keep compose.yml as base, add compose.dev.yml for hot-reload, compose.prod.yml for tight prod settings.

🚨 Compose is NOT for production: It's a dev/CI tool. No autoscaling, no rolling updates, no multi-host networking. Use compose to mirror prod locally, use k8s / Cloud Run / Lambda to actually serve users.
The Heavyweight

Kubernetes — Just Enough To Be Dangerous

You may never write raw k8s YAML. But every engineer needs the mental model — what these objects are, when they fire, and why they exist.

OBJECT · 01

Pod

The smallest unit. One or more containers sharing network & storage. Pods are disposable — k8s kills and restarts them freely.

Think: a single instance
OBJECT · 02

Deployment

Declares "I want N replicas of this pod, always". Handles rolling updates, rollbacks, version pinning. Where your app actually lives.

Think: the desired state
OBJECT · 03

Service

Stable network address + load balancer in front of N pods. Pods come and go; the Service IP stays the same.

Think: the front door
OBJECT · 04

Ingress

External HTTP entry point: hostnames, TLS, path routing. Maps api.example.com → your Service.

Think: the public URL

When To Actually Use Kubernetes

Situation Pick Why
Solo dev, MVP, < 100 users PaaS / Cloud Run k8s ops cost > product value
2-3 services, predictable traffic Managed containers Cloud Run / App Runner does this natively
5+ services, multiple teams Kubernetes Real service mesh / multi-tenant value
Compliance (HIPAA, SOC2 in-VPC) Kubernetes (EKS/GKE) Network policies + audit + IAM granularity
GPU inference at scale Kubernetes + GPU pool Bin-pack GPU nodes, taints/tolerations
⚡ The k8s tax: Kubernetes is powerful and expensive. Plan for 0.5-1 full-time engineer just on cluster ops in any non-trivial deployment. The clusters that look easy (managed control plane) get expensive at the node, networking, and observability layers.
No Servers, Allegedly

Serverless Options for RAG

Four platforms, each with different cold-start, GPU, and runtime trade-offs. Pick the one that fits your traffic shape — not the one with the loudest marketing.

Platform 01

AWS Lambda

The original. Burst to 1000s of concurrent invocations. 15-minute max runtime. Container images up to 10 GB.

Cold start (basic)100-400 ms
Cold start (LLM image)5-15 s
GPU support✗
Best forLight RAG, webhooks
Platform 02

Modal

Python-native serverless built for AI workloads. GPU support first-class. Snapshot-based cold starts. Decorator API.

Cold start (CPU)500 ms - 2 s
Cold start (GPU)3-15 s
GPU supportA10 / A100 / H100
Best forCustom inference, embeddings
Platform 03

Replicate

Model-as-a-service. Push a Cog container, get an HTTPS endpoint with autoscaling GPU. Pay-per-second of GPU time.

Cold start5-30 s (model dependent)
GPU supportT4 → A100 → H100
Pricing modelPer-second GPU
Best forHosted open models
Platform 04

Beam / Banana / RunPod

Newer GPU-serverless platforms targeting AI workloads. Aggressive cold-start optimization. Pricing varies wildly — benchmark before committing.

Cold start1-10 s claimed
GPU supportYes
MaturityLower than Modal
Best forSpecialty / cost optimization
📍 Reality check: Most production RAG calls an external LLM (OpenAI, Anthropic) — meaning your code doesn't actually need a GPU. A CPU container on Cloud Run or Modal is often the right answer. Reach for GPU serverless only when you're self-hosting models for embeddings, reranking, or inference.
The Hidden Tax

Beating Cold Starts

A 12-second cold start is a churned user. Five tactics to crush startup time — pick the ones that match your hosting target.

Where The Seconds Go — A Naive Container

Pull image (2.4 GB) ~6.0 s
Container boot (kernel + filesystem) ~1.0 s
Python interpreter start ~0.4 s
Import torch + transformers + faiss ~3.5 s
Load embedding model into memory ~2.8 s
Connect to vector DB + warm-up call ~1.2 s
Total: ~15 seconds from request → first response. With the tactics below, you can cut this to 2-3 seconds on the same platform.
TACTIC · 01

Shrink The Image

Multi-stage + slim base + .dockerignore. Smaller image = faster pull. Single biggest lever.

2.4 GB → 280 MB = 6s → 0.7s pull
TACTIC · 02

Min-Instances ≥ 1

Cloud Run, Lambda, App Runner all let you keep at least 1 warm. Costs a bit of idle, kills cold starts entirely.

--min-instances 1
TACTIC · 03

Lazy Imports

Don't import torch at the top of main.py if /health doesn't need it. Defer heavy imports into the handlers that use them.

Import inside the function
TACTIC · 04

Model Snapshots

Modal & some platforms checkpoint your loaded process to disk. Next cold start restores memory state instead of reloading.

15s → 1-2s on Modal
TACTIC · 05

External LLM, Tiny Container

If you call OpenAI/Anthropic, your container has no model to load. A CPU-only Python image with httpx is < 200 MB and cold-starts in ~1s.

Match container to workload
TACTIC · 06

Warm Pool / Provisioned

Lambda Provisioned Concurrency, Cloud Run min-instances, k8s HPA min-replicas. Pay for warm capacity that matches your floor traffic.

Floor traffic = always warm
🌱 The pragmatic ladder: Start with tactic 1 (shrink image). Add tactic 5 (external LLM, small container) — these are free wins. If you still see cold starts, add tactic 2 (min-instances ≥ 1) or tactic 6 (provisioned). Reach for snapshots (tactic 4) only when you're stuck on a self-hosted model and need single-digit-second cold starts.
