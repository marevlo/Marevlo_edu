Mission Briefing

Learning Objectives

By the end of this module, you'll have wired up a CI/CD pipeline that versions data, tracks experiments, gates on evals, detects drift, and rolls out new models safely.

Why traditional CI/CD breaks for AI

Your test suite is green. Code coverage is 95%. The PR ships. Three hours later, support tickets explode — the new prompt is hallucinating. Traditional CI/CD checks syntax, not behavior. AI CI/CD bakes in evaluation, drift detection, and gradual rollouts because your "logic" is now probabilistic.

Tool 01

Data Versioning with DVC

Git tracks code. DVC tracks the gigabyte-scale data, embeddings, and vector indexes that code depends on — and ties them together.

What it does

DVC in a sentence

DVC stores large files in object storage (S3, GCS, Azure Blob) and commits tiny .dvc pointer files into Git. Every code commit is paired with the exact dataset that produced it.

Reproducible runs — `dvc repro` re-runs the entire pipeline
Git-friendly — branches and PRs work for datasets
Storage-agnostic — S3, GCS, Azure, SSH, local
Pipeline-aware — DAG of stages with dependencies
dvc.yaml 
stages:
  ingest:
    cmd: python src/ingest.py
    deps:
      - src/ingest.py
      - data/raw/
    outs:
      - data/cleaned/

  chunk_and_embed:
    cmd: python src/embed.py
    deps:
      - src/embed.py
      - data/cleaned/
    params:
      - chunk_size
      - embed_model
    outs:
      - data/embeddings/

  build_index:
    cmd: python src/build_qdrant.py
    deps:
      - data/embeddings/
    outs:
      - artifacts/qdrant_snapshot/

  evaluate:
    cmd: python src/eval.py
    deps:
      - artifacts/qdrant_snapshot/
      - data/eval_set.jsonl
    metrics:
      - reports/ragas_scores.json:
          cache: false
💡 Pro tip: Use dvc exp run for experiments — it creates lightweight branches without polluting Git. When one wins, promote it with dvc exp apply.
Tool 02

Experiment Tracking with MLflow

You'll run hundreds of RAG variants. Without tracking, you lose every insight. MLflow gives you a searchable history of every prompt, model, and metric.

Params

Chunk size, top-K, reranker on/off, model name, temperature, embedding dimensions.

Metrics

RAGAS faithfulness, context precision, answer relevance, latency p50/p95, cost per query.

Artifacts

The prompt template file, eval set snapshot, confusion matrix, per-query traces.

Tags

Git SHA, DVC data hash, author, branch, env (dev/staging/prod), promotion status.

MLFLOW · RUN LOGGING 
import mlflow
from ragas import evaluate

mlflow.set_experiment("rag-prod-v3")

with mlflow.start_run(run_name="chunk-512-rerank-bge"):
    # 1. Log params
    mlflow.log_params({
        "chunk_size": 512,
        "top_k": 5,
        "embed_model": "text-embedding-3-large",
        "reranker": "bge-reranker-v2",
        "llm": "claude-haiku-4-5",
    })

    # 2. Run the pipeline + eval
    results = evaluate(eval_set, rag_chain)

    # 3. Log metrics
    mlflow.log_metrics({
        "faithfulness": results.faithfulness,
        "context_precision": results.context_precision,
        "answer_relevance": results.answer_relevance,
        "latency_p95_ms": results.latency_p95,
        "cost_per_query_usd": results.cost,
    })

    # 4. Log artifacts (prompt, eval set hash)
    mlflow.log_artifact("prompts/rag_v3.jinja")
    mlflow.set_tag("git_sha", os.getenv("GITHUB_SHA"))
    mlflow.set_tag("dvc_data_hash", dvc_hash())

Watchtower

Data Drift Detection

Your model was trained on yesterday's queries. Today's users ask new things. Drift detection catches the silent quality decay before tickets pile up.

Live Drift Monitoring

Query embedding distribution vs training baseline · 30-day window

KS · PSI · COSINE
01

Query Drift

Users start asking about new topics — say a feature launch or a news event. Embeddings cluster differently.

→ DETECT: KL divergence on embedding histograms
02

Corpus Drift

Your knowledge base grows or rebrands — docs go stale, the index loses recall on familiar queries.

→ DETECT: Recall@K on golden eval set over time
03

Quality Drift

RAGAS faithfulness silently drops 4% week-over-week. Nothing changed in code — but model providers updated their endpoints.

→ DETECT: Trend alerts on rolling 7-day metrics
📡 Detection vs reaction. Pipe drift alerts straight into the observability stack from Module 7.2. The fastest teams treat drift like a P2 incident — investigate within 24 hours, retrain or re-embed within a week.
The Gatekeeper

Hard Evaluation Gates

Every PR runs the full RAG eval suite. If any metric crosses its red line, the build fails — period. No manual overrides without a documented exception.

SIMULATION · CI RUN

Will this PR be allowed to merge?

Roll a new evaluation. The 4 gates judge in parallel.

↑ Click "Run Eval" to simulate a CI run

Few, Sharp Gates

3–5 gates max. Each must be objective, threshold-based, and faster than 10 minutes to compute.

No-Regression Floor

Compare against the champion run. New score must be ≥ champion − 1%. No silent regressions.

Golden Eval Set

Curated 200–500 question/answer pairs version-controlled in DVC. The single source of truth for "did we get worse?"

Fast Feedback

Run a 50-question smoke set on every commit. Full eval on PR. Don't make devs wait 30 minutes for a red X.

Implementation

GitHub Actions Workflow

A real CI workflow combining DVC, MLflow, RAGAS evals, and deploy gates. Copy this as the starting point.

.github/workflows/rag_ci.yml 
name: RAG CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # needed for DVC diffs

      - name: Setup Python
        uses: actions/setup-python@v5
        with: { python-version: "3.12" }

      - name: Install deps
        run: pip install -r requirements.txt

      - name: Pull DVC data
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        run: dvc pull

      - name: Run RAGAS eval
        env:
          MLFLOW_TRACKING_URI: ${{ secrets.MLFLOW_URI }}
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: python scripts/eval.py --output reports/scores.json

      - name: Compare to champion
        run: python scripts/gate.py reports/scores.json
        # exits non-zero if any metric drops > threshold

      - name: Comment scores on PR
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const r = require('./reports/scores.json');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner, repo: context.repo.repo,
              body: `## 🛰️ Eval results
              faithfulness:   ${r.faithfulness.toFixed(3)}
              precision:      ${r.context_precision.toFixed(3)}
              relevance:      ${r.answer_relevance.toFixed(3)}
              latency p95:    ${r.latency_p95}ms`
            });

Knowledge Check

Lock It In

Three checkpoint questions. Answer all three to unlock the module checkpoint.

QUESTION 01 · DATA VERSIONING

Your dataset is 12GB. Why is committing it directly to Git a bad idea — and what does DVC fix?

Correct: C. Git was built for source code, not blobs. DVC keeps Git lean and reproducible by externalizing the heavy data while preserving the link between code and data via tiny .dvc files.
QUESTION 02 · EVAL GATES

A PR drops RAGAS faithfulness from 0.94 to 0.91. The author argues "it's still high." What should the eval gate do?

Correct: C. The whole point of the no-regression floor is to be objective and unarguable. 3% drops compound — three of them and you've lost 10%. Make the author justify why the regression is acceptable through a documented exception, not a Slack convince.
QUESTION 03 · CANARY ROLLOUT

Your offline RAGAS evals say the challenger is +2% better. You roll to 5% canary. After 6 hours, p95 latency is 1.6× champion. What now?

Correct: A. The rollback criteria are pre-committed for a reason — to prevent debate during incidents. Offline scores are only one signal. Real users feeling 1.6× slower responses is a worse outcome than +2% on a metric they can't see.
