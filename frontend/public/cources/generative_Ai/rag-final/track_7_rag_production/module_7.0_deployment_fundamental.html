The Default Stack

FastAPI Patterns for RAG

FastAPI is the de-facto standard for Python AI APIs. Async-first, Pydantic-typed, auto-documented. Here's the patterns that matter.

A Minimal RAG Endpoint

~40 lines 
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field
from contextlib import asynccontextmanager

# ---------- LIFESPAN — load heavy stuff ONCE ----------
@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: load embedding model, connect to vector DB
    app.state.retriever = build_retriever()
    app.state.llm        = OpenAI()
    yield
    # Shutdown: close connections (vector DB, HTTP clients)
    await app.state.retriever.close()

app = FastAPI(title="RAG API", lifespan=lifespan)

# ---------- SCHEMAS — Pydantic does input validation ----------
class QueryRequest(BaseModel):
    question: str = Field(..., min_length=3, max_length=500)
    top_k:    int = Field(5, ge=1, le=10)

class QueryResponse(BaseModel):
    answer:   str
    sources:  list[str]
    latency_ms: int

# ---------- ENDPOINTS ----------
@app.post("/v1/query", response_model=QueryResponse)
async def query(req: QueryRequest):
    t0 = time.time()
    try:
        chunks = await app.state.retriever.search(req.question, k=req.top_k)
        answer = await generate(app.state.llm, req.question, chunks)
    except Exception as e:
        raise HTTPException(500, detail=str(e))

    return QueryResponse(
        answer=answer,
        sources=[c.id for c in chunks],
        latency_ms=int((time.time() - t0) * 1000)
    )

@app.get("/health")
async def health():
    return {"status": "ok"}

Lifespan, not Globals

Load embedding models, DB clients, and HTTP pools once at startup via @asynccontextmanager. Never inside request handlers.

✓ Saves 200-2000ms per request — a model load is NOT a request-time op

Pydantic for Everything

Every input gets a BaseModel. Field bounds, enums, regex — caught at the edge, never in business logic.

✓ Free 422 errors for bad inputs + free OpenAPI docs at /docs

async, async, async

LLM calls are I/O-bound. Use async def handlers + async clients (httpx, asyncpg). One worker handles hundreds of in-flight requests.

✓ 10-50× more concurrency than sync Flask

Explicit Errors

Catch upstream failures (rate limits, retrieval down, LLM 5xx). Map to HTTPException with clear codes — never leak stack traces.

✓ Use 429 for rate limit, 503 for upstream down, 422 for bad input
📦 What about LangServe? LangServe wraps LangChain chains as FastAPI routes automatically — useful if you're heavy in the LangChain ecosystem. It gives you /invoke, /stream, /batch for free. Trade-off: you couple your API surface to LangChain's. For most production RAG, plain FastAPI gives more control. Use LangServe for fast internal demos, plain FastAPI when shape matters.
Three Styles

REST · Streaming · WebSocket

Three transport choices. Each fits a different latency, UX, and complexity profile. Pick wrong and you'll regret it for the life of the API.

STYLE 01

REST / JSON

Plain HTTP POST → wait → JSON response. Universal, dead simple, infinite tooling.

✓ Easy to cache, log, test
✓ Works through any proxy/firewall
✗ User waits 3-15s seeing a spinner
Best for
Internal APIs, integrations, short answers, batch jobs
STYLE 02

SSE Streaming

Server-Sent Events — token-by-token push over a single HTTP connection. The ChatGPT experience.

✓ First token in < 1s — feels instant
✓ Still plain HTTP — proxy-friendly
✗ One-way (server → client) only
Best for
Chat UIs, long answers, anywhere users wait
STYLE 03

WebSocket

Full-duplex persistent connection. Client can interrupt, send mid-stream, share state both directions.

✓ Two-way: cancel mid-generation, multi-turn
✓ Lowest latency for highly interactive UX
✗ Hard to load-balance, scale, cache
Best for
Voice agents, collab tools, real-time copilots

Which Style Do You Pick?

Use Case Style Why
Slack bot answering doc questions REST Slack expects one webhook response within 3s
Web chat UI (Perplexity-style) SSE Streaming Users see tokens immediately — feels 10× faster
Backend service calling another backend REST Service-to-service should stay simple & cacheable
Voice assistant with barge-in WebSocket User must interrupt mid-answer; needs two-way
Batch document analysis REST + async job POST → job_id → poll /status — no real-time needed
The UX Multiplier

Streaming with SSE in FastAPI

First token latency is everything. A 12-second wait feels broken. A 12-second stream that starts in 600ms feels instant. Here's how. 

from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import json

@app.post("/v1/query/stream")
async def query_stream(req: QueryRequest):
    chunks = await app.state.retriever.search(req.question, k=req.top_k)

    async def event_stream():
        # 1. Send sources FIRST so UI can render citations early
        yield f"data: {json.dumps({'type':'sources','data':[c.id for c in chunks]})}\n\n"

        # 2. Stream tokens from the LLM
        stream = await llm.chat.completions.create(
            model="gpt-4o-mini",
            messages=build_messages(req.question, chunks),
            stream=True
        )
        async for chunk in stream:
            delta = chunk.choices[0].delta.content or ""
            if delta:
                yield f"data: {json.dumps({'type':'token','data':delta})}\n\n"

        # 3. Done — let the client close cleanly
        yield f"data: {json.dumps({'type':'done'})}\n\n"

    return StreamingResponse(
        event_stream(),
        media_type="text/event-stream",
        headers={"X-Accel-Buffering": "no"}  # disable nginx buffering
    )
GOTCHA · 01

Proxy Buffering

Nginx, CloudFront, ALB buffer responses by default. Your stream becomes one big response.

✓ Header: X-Accel-Buffering: no
✓ Disable in proxy config too
GOTCHA · 02

Mid-Stream Errors

If the LLM fails after 50 tokens, you can't send a 500 — the response started 200.

✓ Send an "error" event:
{"type":"error","data":"..."}
GOTCHA · 03

Connection Drops

User closes tab. Without checks, you keep generating & burning tokens.

✓ Check request.is_disconnected()
✓ Break out of generation loop
Two-way Streaming

Streaming via WebSocket — When SSE Isn't Enough

SSE is one-way. When the client also needs to send mid-stream signals — cancel, follow-up, edit-the-prompt-on-the-fly — WebSocket is the right transport.

SSE vs WebSocket — pick by traffic shape

DimensionSSEWebSocket
Directionserver → client onlybidirectional
TransportHTTP/1.1 long-pollupgraded TCP frame
Proxy supportuniversalneeds explicit pass-through
Reconnectautomatic (browser)manual
Authstandard HTTP headerstoken-via-message
Best forchat completions, logs, pricescollaborative editing, mid-stream cancel, agent tool-confirm
FastAPI · WebSocket endpoint with cancel
from fastapi import WebSocket, WebSocketDisconnect
import asyncio, json

@app.websocket("/v1/query/ws")
async def ws_query(ws: WebSocket):
    await ws.accept()
    cancel = asyncio.Event()

    async def listen_for_cancel():
        try:
            while True:
                msg = await ws.receive_text()
                if json.loads(msg).get("action") == "cancel":
                    cancel.set(); return
        except WebSocketDisconnect:
            cancel.set()

    # first message = the query
    req = json.loads(await ws.receive_text())
    canceller = asyncio.create_task(listen_for_cancel())

    async for token in generate_stream(req["question"]):
        if cancel.is_set(): break
        await ws.send_json({"type":"token", "data":token})

    await ws.send_json({"type":"done"})
    canceller.cancel()
⚡ Pragmatic default: ship SSE first. It works behind every proxy without config, gets free reconnect from the browser, and covers 95% of chat UX. Upgrade to WebSocket only when you need mid-stream client→server messages.
Performance Budgeting

The Latency Budget — Spend Your Milliseconds Deliberately

"Make it faster" is not a plan. Decompose the request into named stages, attach a budget to each, measure against it. Now "the system is slow" becomes "rerank is 350ms over budget."

RULE 01 · MEASURE BEFORE BUDGETING

Instrument first (OpenTelemetry spans per stage). A budget built without baseline data is fiction. Run 1000 production-shape queries, look at P50/P95/P99 per stage, then write the budget.

RULE 02 · BUDGET P95, NOT P50

P50 is what the average user feels. P95 is what an angry user feels. Set the budget to P95 — if you optimize P50 only, the tail eats you.

RULE 03 · STREAMING CHANGES THE GAME

With streaming, only TTFT (first token) matters for "feels instant." If TTFT < 700ms, the rest can take 5+ seconds and feel fine. Budget TTFT separately.

⚡ Where most teams find their first second: reranker over-fetching (K=20 with a slow model), no prompt cache (re-sending the same 3k-token system prompt every turn), Python tokenization in the hot path, and an embedding service one VPC hop away. Each is a single-line fix.
Failing Safely

Circuit Breakers & Graceful Degradation

Every dependency in your pipeline will fail. The reranker, the embedder, the LLM, the vector DB — pick any week of the year. A production RAG endpoint stays responsive when one of them is down, not because it's lucky, but because it was designed to.

The circuit-breaker state machine

CLOSED ─── normal traffic flows
↓  # N consecutive failures or P95 > threshold
OPEN ───── reject calls instantly, return fallback
↓  # after cooldown (e.g. 30s)
HALF-OPEN ─ let one probe through
↓  # success → CLOSED · failure → OPEN

Without a breaker, a slow dependency adds latency to every request until your worker pool is exhausted. With one, the bad path is paid once, then bypassed entirely.

The fallback ladder — what to do when X is down

Component downFallback
Rerankerskip it; use first-stage top-K directly
Cross-encoder modelfall back to dense-only retrieval
EmbedderBM25-only retrieval (sparse path)
Vector DBBM25-only · degrade gracefully · alert
Primary LLMfallback to smaller/cheaper model
Both LLMsreturn cached answer if hit · else "service degraded"

Always communicate the degradation in the response (header or payload field) so the UI can show "running on limited mode" instead of pretending everything is fine.

tenacity + circuitbreaker · Python wrapper
from circuitbreaker import circuit
from tenacity import retry, stop_after_attempt, wait_exponential

@circuit(failure_threshold=5, recovery_timeout=30)
@retry(stop=stop_after_attempt(2),
       wait=wait_exponential(multiplier=0.2, max=1))
async def rerank(query, candidates):
    return await rerank_client.score(query, candidates)

async def retrieve_with_fallback(q):
    cands = await hybrid_search(q, k=20)
    try:
        return await rerank(q, cands), "full"
    except (CircuitBreakerError, Exception):
        # breaker open OR underlying failed
        metrics.inc("rerank.degraded")
        return cands[:5], "degraded"

What good degradation looks like

  • ✦ Latency stays bounded (no unbounded waiting on dead deps)
  • ✦ Behavior changes are visible — metrics + response header
  • ✦ Users get an answer (possibly with a "limited mode" label)
  • ✦ Oncall pages on the breaker state change, not on every retry

Anti-patterns

  • ✦ Infinite retries (cascades into thread-pool death)
  • ✦ Silent fallback (no metric, no log)
  • ✦ Identical fallback path also depends on the dead service
  • ✦ Per-request timeout but no breaker → still wastes 30s/req under outage
⚡ Test the breaker before you need it: chaos-test in staging by forcibly killing the reranker and embedder. If the endpoint still returns answers in under 2s, your degradation is real. If it timeouts, your design is decorative.
Plumbing That Matters

Connection Pooling for Retrievers

Every RAG endpoint talks to 3-5 backends per request: vector DB, LLM, reranker, cache, maybe a SQL warehouse. Open a fresh TCP/TLS handshake to each, on every request, and you've spent half your budget before the first byte of work.

Without pooling

A request's hidden tax

request comes in
→ DNS lookup (~5ms cold)
→ TCP SYN/ACK (~10ms)
→ TLS handshake (~30ms)
→ HTTP/1.1 request
≈ 45ms per dependency · 5 deps = 225ms wasted
With pooling

Warm sockets reuse

request comes in
→ check pool · grab warm socket
→ HTTP/2 multiplex stream
→ first byte
≈ 1-2ms per dependency · same hop, every time
httpx · async pool tuned for RAG
import httpx
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app):
    limits = httpx.Limits(
        max_connections=100,
        max_keepalive_connections=50,
        keepalive_expiry=60,
    )
    timeout = httpx.Timeout(10.0, connect=2.0)

    app.state.llm      = httpx.AsyncClient(base_url="https://api.openai.com", limits=limits, timeout=timeout, http2=True)
    app.state.vector   = httpx.AsyncClient(base_url="http://qdrant:6333", limits=limits, timeout=timeout)
    app.state.rerank   = httpx.AsyncClient(base_url="http://rerank:8000", limits=limits, timeout=timeout)

    yield
    await app.state.llm.aclose()
    await app.state.vector.aclose()
    await app.state.rerank.aclose()

app = FastAPI(lifespan=lifespan)

Pool sizing rules of thumb

  • ✦ max_connections ≥ worker_concurrency × dependencies
  • ✦ max_keepalive ≈ 50% of max_connections
  • ✦ keepalive_expiry < server's idle timeout (usually 60s)
  • ✦ connect timeout 1-2s · read timeout matches downstream P99
  • ✦ Use HTTP/2 when the backend supports it — multiplexing = even fewer sockets
⚡ The mistake that hides: creating a new client inside every request handler. The pool is per-client, so a per-request client is no pool at all. Always instantiate the client once at app startup and reuse it.
⚡ The compounding win: pooling, HTTP/2 multiplexing, async I/O, and a circuit breaker all share the same pool of warm sockets. Each technique is small on its own — together they're the difference between a 1200ms endpoint and a 400ms endpoint, at zero infrastructure cost.
Where It Runs

Managed vs Self-Hosted

Four common hosting targets for a RAG API. The right pick depends on team size, compliance, traffic, and your tolerance for ops work.

OPTION · 01

PaaS

Railway, Render, Fly.io, Heroku. git push deploys. Zero ops, capped flexibility.

✓ < 5 min to live URL
✓ Auto TLS + scaling
✗ Vendor lock + price cliffs
OPTION · 02

Managed Containers

Cloud Run, AWS App Runner, Azure Container Apps. Docker image → URL, autoscale.

✓ Scales to zero (cheap idle)
✓ Standard Docker exit
✗ Cold starts 2-10s
OPTION · 03

Serverless / AI-PaaS

AWS Lambda, Modal, Replicate, Beam. Pay-per-call. Some optimized for GPU inference.

✓ Zero idle cost
✓ Burst to 10,000 reqs
✗ Cold starts brutal for LLM
OPTION · 04

Kubernetes / VMs

EKS, GKE, AKS, or raw EC2. Maximum control. Maximum ops burden.

✓ Full control, any region
✓ Enterprise compliance
✗ Needs an SRE/platform team

A Simple Decision Heuristic

You Are... Pick Why
A solo dev shipping an MVP PaaS (Railway/Fly) Optimize for time-to-live, not infra elegance
A 5-20 person startup with bursty traffic Cloud Run / App Runner Scales to zero, scales up, no cluster to babysit
An ML team needing GPU inference Modal / Replicate GPU autoscale + container exit, no cluster ops
Enterprise with VPC/PII/compliance Kubernetes (EKS/GKE) Auditable, on-network, full IAM control
⚡ The pragmatic ladder: Start on PaaS. When you outgrow it (price cliff or feature gap), move to managed containers. Only consider Kubernetes when you have multiple teams, multiple services, or a compliance reason. Premature k8s is the #1 way RAG startups burn engineering quarters.
The Long-Term Move

Versioning Your API

Day 1 you don't think you need versioning. Day 90 a client breaks because you renamed a field. Version from day 1 — it costs you nothing now and saves you everything later.

STRATEGY · 01

URL Path

Most common. Easy to read in logs, easy to route in proxies.

POST /v1/query
POST /v2/query
✓ Discoverable in OpenAPI · ✓ Cache-friendly
STRATEGY · 02

Header

Cleaner URLs. Used by Stripe, GitHub. Versions by date.

API-Version: 2026-05-01
✓ Date-pinning · ✗ Less obvious in logs
STRATEGY · 03

Field-Level

No URL/header change. Add new fields, deprecate old. Slow rollouts.

sources_v2: [...]
# sources deprecated
✓ Backward-compatible · ✗ Schema bloat

Four Rules That Save You From Pain

RULE 01 · NEVER REMOVE

Never delete a field from /v1. Deprecate, document, but leave it returning something. Old clients break silently otherwise.

RULE 02 · ADDITIVE BY DEFAULT

Need a new field? Add it. Old clients ignore unknown fields — they don't break. Additive changes don't need a new version.

RULE 03 · DEPRECATION WINDOW

Announce v1 deprecation at least 6 months before sunset. Send Deprecation: + Sunset: headers from v1.

RULE 04 · CHANGELOG IS LAW

Every version change goes in a public changelog. "What changed and why" — not a feature you write later. Customers will ask.

🚨 The reality: Your RAG's API surface will change. New retrieval modes, new metadata, new safety rails. Treat your API as a contract from request #1. Even if you're the only consumer today, future-you (or your first paying customer) will thank you.
