The Framework

Observability's Three Pillars

Logs, metrics, traces. Each answers a different question. Skip any one and you'll spend hours debugging what minutes of instrumentation would have solved.

PILLAR 01

Logs

Discrete events with context. "User X asked Y at time Z, retrieval returned these chunks". The narrative record.

Answers: "what happened in this request?"
Granularity: Per event
Cost: Storage-heavy
Best for
Forensic debugging, audit trails, replays
PILLAR 02

Metrics

Aggregated numerical signals over time. P95 latency, tokens/min, error rate. The pulse of the system.

Answers: "is something off RIGHT NOW?"
Granularity: Time-series
Cost: Cheap to store
Best for
Dashboards, alerts, capacity planning
PILLAR 03

Traces

A request's full lifecycle as a tree of timed spans. Embed → search → rerank → LLM → response. The flame graph of a RAG call.

Answers: "WHERE in the call did time go?"
Granularity: Per-span, per-request
Cost: Sampled in prod
Best for
Performance debugging, multi-stage pipelines

The Stack

LLM-Native vs Generic Observability

Datadog tells you a span took 1.2s. LangSmith tells you what tokens were in the prompt. You need both — but you definitely need an LLM-aware tool.

Tool 01

LangSmith

LangChain's first-party platform. Best-in-class for LangChain/LangGraph stacks. Trace UI, eval, prompt management, datasets.

✓ Deepest integration with LangChain
✓ Eval + dataset features built-in
✗ Hosted SaaS · pricier at scale
Best for: LangChain-heavy teams
Tool 02

Langfuse

Open-source LLM observability. Self-host or cloud. Framework-agnostic — works with raw OpenAI, Anthropic, anything.

✓ Open source — self-host for free
✓ Framework-agnostic SDK
✗ Smaller ecosystem than LangSmith
Best for: privacy / on-prem / cost-sensitive
Tool 03

Phoenix / Arize

Arize is the enterprise ML observability platform. Phoenix is its OSS sibling, focused on LLM apps with OpenTelemetry roots.

✓ OpenTelemetry-native
✓ Strong eval & embedding drift tools
✗ Steeper learning curve
Best for: enterprise ML ops teams

Which Tool Fits Your Team?

You Are... Pick Why
LangChain / LangGraph app, want hosted LangSmith Native integration · best-in-class UX for chains
Privacy-sensitive · self-host required Langfuse (self-hosted) Full feature set on your own infra
Bare Python + OpenAI SDK · framework-free Langfuse SDK is framework-agnostic
Enterprise ML team with existing OTel stack Phoenix / Arize OTel-native, slots into existing telemetry
Already deep in Datadog / New Relic Datadog/NR + LLM tool Generic APM + LLM tool · don't dual-pay for traces
⚡ The split-brain trap: Many teams send traces to Datadog (for infra) AND LangSmith (for LLM specifics) — and end up unable to correlate the two. Pick a primary tool and either propagate trace IDs both ways, or ship from OTel SDK to both.
The Vendor-Free Layer

OpenTelemetry for LLM Apps

OTel is a vendor-neutral standard for emitting telemetry. Instrument once with OTel, ship to any backend — no lock-in, no rewrite.

Without OTel

Each tool wants its own SDK. Switching means rewriting all instrumentation.

→ Datadog SDK in code
→ New Relic SDK in code
→ LangSmith SDK in code
→ Migration = rip & replace

With OTel

Instrument once with the OTel SDK. Configure an exporter to any backend.

→ OTel SDK in code (once)
→ Exporter config in env
→ Switch backends = env change
→ Future-proof

Instrumenting a RAG Call with OTel

~30 lines 
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

# ---------- ONE-TIME SETUP ----------
trace.set_tracer_provider(TracerProvider())
trace.get_tracer_provider().add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter())  # reads OTEL_EXPORTER_OTLP_ENDPOINT env
)
tracer = trace.get_tracer("rag-api")

# ---------- INSTRUMENTED RAG ----------
async def answer(question: str):
    with tracer.start_as_current_span("rag.query") as span:
        span.set_attribute("rag.question", question[:120])

        with tracer.start_as_current_span("rag.embed"):
            vec = await embed(question)

        with tracer.start_as_current_span("rag.retrieve") as s:
            chunks = await qdrant.search(vec, k=5)
            s.set_attribute("rag.top_k", 5)
            s.set_attribute("rag.chunks_returned", len(chunks))

        with tracer.start_as_current_span("rag.generate") as s:
            resp = await llm.complete(question, chunks)
            s.set_attribute("llm.model", "gpt-4o-mini")
            s.set_attribute("llm.tokens_in", resp.usage.prompt_tokens)
            s.set_attribute("llm.tokens_out", resp.usage.completion_tokens)

        return resp.text
📍 The 3 power moves:
  • • start_as_current_span creates a parent-child tree — you get the flame graph for free
  • • set_attribute adds searchable metadata (model, top_k, tokens) — filter your traces in any backend
  • • OTEL_EXPORTER_OTLP_ENDPOINT env var switches backends without code changes
SHORTCUT · 01

Auto-Instrumentation

Packages like opentelemetry-instrumentation-fastapi auto-trace every route. Add one line, get HTTP spans free.

SHORTCUT · 02

OpenLLMetry

Open-source semantic conventions for LLM spans. Standardizes llm.tokens_in, llm.model, etc. across vendors.

SHORTCUT · 03

Vendor SDKs Speak OTel

Langfuse, Phoenix, Honeycomb, Datadog all accept OTel-format. Configure the endpoint, you're done.

The Checklist

What To Log, What To Trace

Log everything & you'll never search it. Log nothing & you can't debug. Here's the minimum signal set that makes RAG debugging tractable.

Log it

Discrete Events

Request started · path · user_id
Auth failure / rate limit hit
Retrieval returned 0 results
LLM refused or returned an error
User submitted thumbs-up / -down
Trace it

Pipeline Stages

rag.query (parent span)
rag.embed · rag.retrieve · rag.rerank
rag.generate (with model + tokens)
rag.eval (if scoring inline)
External API calls (OpenAI, Qdrant)

The Anatomy of A Good Log Line

{
  "ts":           "2026-05-19T10:34:12.418Z",
  "level":        "info",
  "service":      "rag-api",
  "version":      "0.4.2",
  "trace_id":     "a4f2c8b9...",    # ← links to a trace
  "request_id":   "req_8c4a",
  "user_id":      "u_4421",
  "event":        "rag.query.completed",
  "duration_ms":  1840,
  "model":        "gpt-4o-mini",
  "tokens_in":    1284,
  "tokens_out":   87,
  "top_k":        5,
  "chunks_used":  ["doc_42#3", "doc_88#1"],
  "refused":      false
}
🎯 The five non-negotiables: trace_id (link to trace), request_id (group a request's logs), user_id (correlate user complaints), version (find regressions to a deploy), structured JSON (parseable, queryable).

What You Should NEVER Log

NEVER · 01

Raw PII

Names, emails, SSNs, addresses in question text. Redact or hash before logging. GDPR/CCPA cares.

NEVER · 02

Full Prompts in Plain

Prompts may contain customer data via retrieved chunks. Log a hash + first 200 chars, NOT the full payload.

NEVER · 03

Secrets / API Keys

It WILL happen by accident. Use scrubbing filters at the logger level — assume devs forget.

What goes on the wall-mounted screen. Six panels you should be able to read at 3 AM half-awake.

Last 24 hours · 1-minute buckets

📍 Why three percentiles? P50 = the median user's experience. P95 = your "slow but acceptable" tail. P99 = the customer who's tweeting about you. An average alone hides all three stories.
PANEL · 01

Latency Percentiles

P50 / P95 / P99 per endpoint. Broken out by stage (embed / retrieve / generate) for instant root-cause.

PANEL · 02

Request Rate (RPM)

Requests per minute. Spikes signal launches or attacks; troughs signal upstream incidents.

PANEL · 03

Error Rate

4xx + 5xx + LLM refusals + tool errors. Bucketed by category — a 429 cluster ≠ a 500 cluster.

PANEL · 04

Cost & Tokens

$/hour, tokens_in/out per request. A 2× spike in tokens_in = retrieved chunks ballooning. Catch it before the bill.

PANEL · 05

Quality (Sampled)

Faithfulness & answer-relevance on 5% of traffic. Rolling 24h. Drops here = retrieval or model issue.

PANEL · 06

User Feedback

👍 / 👎 ratio, comment volume. Ground truth from real users — the most honest signal you have.

⚡ The 3 AM test: Can someone half-asleep look at your dashboard and tell in 10 seconds whether the system is healthy? If you need a doc to interpret it, redesign it. Dashboards should answer one question: "is anything on fire?"

Five Principles For Alerts That Don't Make You Numb

PRINCIPLE · 01

Alert on Symptoms, Not Causes

Alert on "P95 latency > 2s" (user-facing). Don't alert on "CPU > 80%" (a cause that may or may not matter).

PRINCIPLE · 02

Two Tiers: Page vs Warn

PAGE = wake someone up. WARN = open a ticket. Most alerts are WARNs. Pages are rare.

PRINCIPLE · 03

Sustained Duration

"> 2% errors for 5+ minutes", not "any single spike". A 30-second blip of 429s isn't worth a pager.

PRINCIPLE · 04

Quality Regression Alerts

"Faithfulness 24h-avg < 0.85" — quality is a first-class signal. Treat it like latency.

PRINCIPLE · 05

Runbook Per Alert

Every alert links to a runbook: "what does this mean, what to check first, who to ping". An alert without a runbook is a panic attack.

PRINCIPLE · 06

Auto-Mute Known Incidents

When OpenAI has an outage, don't get 47 pages about LLM errors. Detect known root causes, suppress duplicates, page once.

🚨 The alert decay curve: Every false page reduces the team's response speed to real ones. After 5 false alarms in a week, on-call starts hitting snooze. Treat alert hygiene as seriously as code review — a noisy alert IS a production bug.
