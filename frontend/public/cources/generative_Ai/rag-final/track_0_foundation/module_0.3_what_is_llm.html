The Core Idea
02 · Nn
Neural Networks in 5 Minutes

No math, no backprop, no calculus. You don't need to build a neural net — you need to reason about one. Here's the mental model that's enough for everything in this course.

The One-Sentence Definition

An LLM is a function that takes a sequence of tokens and returns a probability distribution over what token should come next. 

llm(["The", " capital", " of", " France", " is"])
→ {" Paris": 0.92, " a": 0.03, " known": 0.01, ...}

That's it. Everything else — chat, reasoning, code generation, function calling — is built on this one primitive. The LLM samples the next token from the distribution, appends it to the input, and runs again. Token by token, the answer emerges.

The Architecture Under the Hood

All modern LLMs are Transformers — a specific neural network design from a 2017 Google paper (Attention Is All You Need).

  • ▸ Input tokens are converted to embedding vectors (think Module 1.0)
  • ▸ Many attention layers (~32 to 128) let each token "look at" every other token in the context
  • ▸ Each layer refines the representation — surface words become syntax becomes meaning becomes reasoning
  • ▸ Final layer outputs probabilities over the entire vocabulary (~100K tokens)
RAG insight: Because every layer looks at the full context, putting your retrieved chunks in the right place — and not too many of them — directly affects answer quality. That's why we obsess over context placement in Track 4.

What LLMs Actually Are

  • ✓ Statistical pattern matchers trained on internet-scale text
  • ✓ Compression algorithms — billions of documents squeezed into ~100GB of weights
  • ✓ Autocomplete on a planetary scale
  • ✓ Surprisingly capable at reasoning when given the right prompt

What LLMs Are NOT

  • ✗ Knowledge bases — they store patterns, not facts
  • ✗ Reasoning engines — they pattern-match what reasoning looks like
  • ✗ Search engines — they have no concept of "looking something up"
  • ✗ Truth oracles — they generate plausible text, not true text
⚡ Engineering takeaway: An LLM's job is to complete patterns plausibly. RAG's job is to give the LLM the right patterns to complete — i.e. your actual data, retrieved at query time. The model never has to "remember" your facts because RAG hands them over fresh, every single query.

How Text Becomes Numbers

LLMs don't see words. They see token IDs — integers from a 100,000-entry vocabulary. Understanding tokenization fixes a dozen production bugs and saves you real money on every API call.

Byte-Pair Encoding (BPE) — the algorithm

BPE is the tokenization scheme used by GPT, Claude, Llama, and almost every modern LLM. The intuition is simple:

  1. Start with every character as its own token (~256 of them)
  2. Scan a giant corpus, find the most-common adjacent pair, e.g. ("t","h")
  3. Merge it into a new token "th"
  4. Repeat ~100,000 times

Result: common words are single tokens; rare words break into pieces; gibberish becomes individual characters.

Rules of Thumb

  • ~1 token ≈ 4 characters of English text. So 1000 tokens ≈ 750 words.
  • ~Code & non-English use more tokens per character — JSON, Chinese, or Python often 2× English ratios.
  • ~Whitespace matters: " Paris" and "Paris" are different tokens. Inconsistency causes subtle bugs.
  • ~Different models = different tokenizers. OpenAI uses cl100k_base, Anthropic has its own, Llama has its own. Token counts differ.

Why Tokenization Matters for RAG

💰 COST

API billing is per-token. A 500-token query at $5/1M tokens costs $0.0025. Times 1M queries = $2,500. Tokenization is your bill.

📏 LIMITS

Context windows are token-counted, not word-counted. "200K tokens" ≈ 150,000 words ≈ a 600-page book. RAG must fit retrieved chunks + question + system prompt inside this limit.

✂️ CHUNKING

In Track 2, we'll chunk documents by token count, not character count. Embedding models have token limits too — exceed them and your chunk gets truncated silently.

Counting Tokens in Python 

# tiktoken — OpenAI's tokenizer library
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o")

text = "RAG is the most important skill in modern AI engineering."
tokens = enc.encode(text)

print(len(tokens))            # → 13
print(tokens)                 # → [49810, 374, 279, 1455, ...]
print([enc.decode([t]) for t in tokens])
# → ['RAG', ' is', ' the', ' most', ' important', ...]
The Working Memory
04 · Context
Context Windows Explained

The context window is the LLM's working memory — the maximum number of tokens it can "see" at once. Everything you give the model (system prompt + history + retrieved chunks + question) plus everything the model writes back must fit inside this single budget.

Inside a 128K Context Window
e.g. GPT-4o standard tier · same shape for all models
128,000 tokens total
SYSTEM PROMPT ~500 tokens
CHAT HISTORY ~3,000 tokens
RETRIEVED RAG CHUNKS (top 5, ~500 tokens each) ~2,500 tokens
USER QUESTION ~50 tokens
↓ MODEL'S RESPONSE (allocated) ~1,000 tokens reserved
UNUSED HEADROOM ~120,950 tokens free
🧠 Realization: Even at "big context" sizes, a well-designed RAG system uses under 5% of the window. Stuffing more chunks isn't always better — it costs money, slows inference, and triggers lost-in-the-middle (covered in Track 4).

RAG-Appropriate Presets

📚 FACTUAL QA
T = 0.0 · top-p = 1.0

Production default for RAG. Maximum determinism. Same query → same answer. Critical for legal, medical, support.

🎯 GROUNDED SUMMARY
T = 0.3 · top-p = 0.9

Slight variation to avoid robotic phrasing while staying anchored to evidence. Good for content rephrasing.

✨ CREATIVE WRITING
T = 0.9 · top-p = 0.95

Marketing copy, brainstorming, ideation. Never use this for factual RAG — hallucinations rise sharply.

⚡ Production rule: For 95% of RAG systems, temperature = 0 is the right answer. You retrieved facts — you don't want the model to "creatively interpret" them.
The Conversation Protocol
06 · Roles
System, User, Assistant Roles

Modern chat APIs aren't free-form text — they're structured message arrays with roles. The role determines how the model treats each message. This is where every RAG prompt lives.

SYSTEM

Sets the model's persona, rules, and behavior for the whole conversation. The user typically never sees this. In RAG, this is where you put your grounding instructions: "Answer only from the provided context. Cite sources with [1][2]. Refuse if the context is insufficient."

→ The model treats system messages with the highest trust.
USER

The human's input. In a RAG system, this is where the question + retrieved context goes. The model treats this as untrusted input — exactly why prompt injection (Track 7) is a real risk when you ingest documents from outside.

→ User messages can be overridden by the system prompt's rules.
ASSISTANT

The model's previous responses, included in multi-turn conversations. This is how chat works — every turn the full message history is replayed so the model can see what it already said. The model treats its own past output as trusted continuation.

→ Long histories burn tokens fast — Track 7 covers history compression.

A RAG Message Array 

from openai import OpenAI
client = OpenAI()

retrieved_chunks = [
    "EU customers may cancel within 30 days...",
    "Annual plans are non-refundable after day 30.",
]

resp = client.chat.completions.create(
    model="gpt-4o-mini",
    temperature=0,
    messages=[
        # ▼ SYSTEM — the rules of engagement
        {
            "role": "system",
            "content": """You are a customer support agent.
Answer ONLY from the Context block below.
If the answer isn't there, say "I don't know."
Cite sources as [1], [2]."""
        },
        # ▼ USER — context + question
        {
            "role": "user",
            "content": f"""Context:
[1] {retrieved_chunks[0]}
[2] {retrieved_chunks[1]}

Question: What is the refund policy?"""
        }
    ]
)
print(resp.choices[0].message.content)
RAG anatomy: System holds the policy; User holds the data + question. This separation is what makes RAG safe — and what makes prompt injection so dangerous when you mess it up.
The Original Sin
07 · Hallucination
Why LLMs Hallucinate

Hallucinations aren't a bug — they're the consequence of the architecture. Once you understand why, you understand exactly why RAG is the right fix.

The Root Cause

The LLM is trained to always produce the most plausible next token — even when it has no idea what comes next. There's no "uncertainty button" in the architecture. When asked about something rare or specific, the model interpolates from similar things in its training distribution — producing fluent, confident, and wrong output.

Mechanically: "I don't know" is just another sequence of tokens. The model only says it when training rewarded that behavior. By default, the path of least resistance is to make something up that sounds right.

Why RAG Fixes This

  1. Provides evidence: The model sees the actual answer in its context window before generating.
  2. Constrains by instruction: "Answer ONLY from the context" turns hallucination into the wrong move.
  3. Enables refusal: If retrieval finds nothing relevant, the system prompt tells the model to say "I don't know."
  4. Makes lies catchable: Every answer cites a chunk. A human auditor can verify the chunk says what the model claims.

The Four Types of Hallucination

01

Factual Fabrication

Model invents specific facts that aren't true.

"The drug was approved by the FDA in March 2019." (actual: never approved)
02

Source Fabrication

Cites studies, papers, URLs, or cases that don't exist.

"As shown in Smith et al. (2022, Nature)..." (Smith never published this)
03

Faithfulness Drift

Subtle distortion of provided context. The fact is real but the model changes a detail.

Context: "Refund within 30 days." Output: "14 days." Track 6 covers measuring this.
04

Reasoning Gaps

Conclusion doesn't follow from the evidence; logical leaps presented as deductions.

"Since prices rose, customers must be unhappy" — projection, not deduction.
🎯 The whole point of this course: Each hallucination type has a specific fix in the RAG pipeline. Factual & source fabrication → retrieval (Track 3) + grounding prompts (Track 4). Faithfulness drift → reranking + faithfulness eval (Track 6). Reasoning gaps → query decomposition (Track 5). You'll learn to diagnose which type is happening and ship the matching fix.

How to Pick a Model for RAG

DEFAULT

GPT-4o-mini

Start every RAG project here. Cheap, fast, more than smart enough for grounded answers. The 80/20 of production RAG in 2026.

HIGH-STAKES

Claude Opus / Sonnet

Legal, medical, complex synthesis. Anthropic models are especially careful about saying "I don't know" — critical for grounded RAG.

HUGE DOCS

Gemini 2.5 Pro

When a single user query needs to reason over a 500-page contract, Gemini's 2M context is unmatched. Hybrid with RAG for cost.

PRIVATE

Llama 4 / Qwen3

Data sovereignty, on-prem, or pre-existing GPU infra. Self-hosted open models for finance, healthcare, defense.
