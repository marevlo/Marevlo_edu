02 · Stream
Why Streaming Changes Everything

A 3-second wait feels like an eternity. A 0.4-second wait feels instant — even though the total response time is identical. Streaming sends tokens as they're generated, so users see Time-to-First-Token (TTFT) instead of total response time. This is the single biggest UX upgrade you can make.

Live Comparison

WITHOUT STREAMING
0.0s
User sees nothing → full response appears at once.
WITH STREAMING
0.0s
User sees the answer forming → feels instant.

The Two Latency Metrics That Matter

TTFT — Time-to-First-Token 300–500ms typical
~400ms
The user's perceived wait. Optimizing this is what makes RAG feel alive.
TPS — Tokens Per Second 50–200 tok/s typical
~120 tok/s
The streaming rate. Determines how fast the answer "types out" — and total response time.
TOTAL RESPONSE TIME TTFT + (tokens / TPS)
~3.4s for 500 tokens
What the user actually experiences without streaming — same wall time, but feels 8× longer.

Streaming Across All Three Providers 

from openai import OpenAI

client = OpenAI()

stream = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Explain HNSW briefly."}],
    stream=True,                                # ← enable streaming
    stream_options={"include_usage": True},   # ← get token counts at end
)

full = ""
for chunk in stream:
    if chunk.choices and chunk.choices[0].delta.content:
        token = chunk.choices[0].delta.content
        print(token, end="", flush=True)   # typewriter UX
        full += token
    elif chunk.usage:                          # final chunk only
        print(f"\nTokens: {chunk.usage.total_tokens}")
from anthropic import Anthropic

client = Anthropic()

with client.messages.stream(                # ← context manager API
    model="claude-sonnet-4-6",
    max_tokens=500,
    messages=[{"role": "user", "content": "Explain HNSW briefly."}],
) as stream:
    for token in stream.text_stream:        # ← clean text iterator
        print(token, end="", flush=True)

    final = stream.get_final_message()
    print(f"\nTokens: {final.usage.input_tokens}+{final.usage.output_tokens}")
from google import genai

client = genai.Client()

# Note: generate_content_stream — different method name
stream = client.models.generate_content_stream(
    model="gemini-2.5-flash",
    contents="Explain HNSW briefly.",
)

for chunk in stream:
    if chunk.text:                            # ← .text shortcut
        print(chunk.text, end="", flush=True)

03 · Async
async/await for LLM Calls

Every API call is 99% waiting — for the network, for the GPU, for the response. Synchronous code wastes that time. Async lets your program fire 100 requests, then do useful work while they all bake in parallel. This isn't an optimization; it's how production AI systems are built.

Synchronous (Sequential)

Each call blocks the program until the response arrives. Ten calls of 2s each = 20s total.

Call 1
Call 2
Call 3
Call 4
Call 5
Total: ~10 seconds

Async (Concurrent)

All calls fire at once. Your code awaits them together. Ten 2s calls = ~2s total.

Call 1
Call 2
Call 3
Call 4
Call 5
Total: ~2 seconds (5× faster)

The Async Pattern

Three changes from sync code: AsyncOpenAI client, await the call, asyncio.gather() to fan out.

import asyncio
from openai import AsyncOpenAI         # ← Async variant

client = AsyncOpenAI()

async def ask(question: str) -> str:
    resp = await client.chat.completions.create(   # ← await
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": question}],
    )
    return resp.choices[0].message.content

async def main():
    questions = [
        "What is HNSW?",
        "What is BM25?",
        "What is reranking?",
        "What is RRF?",
        "What is chunking?",
    ]
    # Fire all 5 in parallel, wait for all to complete
    answers = await asyncio.gather(*[ask(q) for q in questions])

    for q, a in zip(questions, answers):
        print(f"Q: {q}\nA: {a[:80]}...\n")

asyncio.run(main())                          # ← entry point

Three asyncio Patterns You'll Use

FAN-OUT

asyncio.gather()

Fire many, wait for all. Use for parallel RAG queries, fan-out summarization.

await gather(t1, t2, t3)
RACE

asyncio.wait()

First-to-finish wins. Use for fallback models, cascade routing.

FIRST_COMPLETED
CAP

Semaphore(N)

Limit max concurrent calls. Critical for not blowing your rate limit.

sem = Semaphore(10)
⚡ Production rule: Never use gather with hundreds of tasks without a semaphore. You'll hit 429 within seconds. Pattern: Semaphore(20) + gather = safe high-throughput RAG.

04 · Batch
Batching Requests Efficiently

Two ways to batch — one for throughput (concurrent async), one for cost (provider batch APIs at 50% off). Production RAG uses both, for different jobs.

Real-Time Batching

Concurrent async · live results

When you need answers now — user-facing RAG queries, real-time analysis, interactive UIs. Use asyncio.gather with semaphore.

  • ▸ Latency: ~1-3 seconds
  • ▸ Cost: full price per token
  • ▸ Best for: live chat, RAG pipelines, agents
await gather(*[ask(q) for q in queries])

Batch API

Async overnight · 50% discount

Upload a JSONL file of requests; provider runs them within 24 hours; you fetch results. For offline workloads only.

  • ▸ Latency: up to 24 hours
  • ▸ Cost: 50% off standard rates
  • ▸ Best for: bulk evaluation, dataset generation, offline analysis
client.batches.create(input_file_id=...)

Using the Batch API (OpenAI example) 

# Step 1 — prepare a JSONL file with N requests
import json
from openai import OpenAI
client = OpenAI()

questions = ["What is HNSW?", "What is BM25?", "..."] * 1000

with open("batch_input.jsonl", "w") as f:
    for i, q in enumerate(questions):
        f.write(json.dumps({
            "custom_id": f"req-{i}",
            "method": "POST",
            "url": "/v1/chat/completions",
            "body": {
                "model": "gpt-4o-mini",
                "messages": [{"role": "user", "content": q}],
            }
        }) + "\n")

# Step 2 — upload and create the batch job
file = client.files.create(file=open("batch_input.jsonl", "rb"), purpose="batch")
batch = client.batches.create(
    input_file_id=file.id,
    endpoint="/v1/chat/completions",
    completion_window="24h",
)

# Step 3 — poll until done, then download results
# status: validating → in_progress → completed
while client.batches.retrieve(batch.id).status != "completed":
    time.sleep(60)

results = client.files.content(batch.output_file_id).text
# Each line is { "custom_id": "req-N", "response": {...} }
💰 Real numbers: Generating a synthetic RAG eval dataset of 10,000 question-answer pairs at GPT-4o-mini real-time = ~$6. Via batch API = ~$3. For 1M evals at GPT-4o: $25,000 → $12,500. Track 6 (Evaluation) leans on this heavily.
Survive the 429
05 · Ratelimit
Rate Limits & 429 Handling

Providers cap how many calls you can make per minute. Hit the cap → HTTP 429. Naive code retries instantly and stays rate-limited forever. Smart code reads the Retry-After header, backs off, and uses a token bucket to never hit the limit again.

The Three Rate-Limit Dimensions

RPM

Requests per Minute

How many API calls regardless of size.

Tier 1 example: 500 RPM
TPM

Tokens per Minute

Total input + output tokens flowing through.

Tier 1 example: 200K TPM
TPD / RPD

Per-Day Caps

Daily ceiling. Lifts as you spend more.

Tier 1 example: 2M TPD

The Token Bucket Algorithm

Think of your rate limit as a bucket. Tokens drip in at a steady rate (the refill rate). Each API call costs tokens. If the bucket runs dry, you wait. This is exactly how providers enforce limits — and exactly how your client should pace itself.

YOUR RATE-LIMIT BUCKET
Refills at: ~3,333 tok/sec Capacity: 200K TPM
✓ HEALTHY ZONE (> 30%)

Free to fire calls at full speed. Bucket refills faster than you drain.

⚠ THROTTLE ZONE (10-30%)

Pace yourself. Lower concurrency, queue non-urgent calls.

🚨 BACKOFF ZONE (< 10%)

Stop firing. Wait for refill. Otherwise you'll get 429s.

Handling 429 the Right Way

Read the Retry-After header. The provider tells you exactly how long to wait. Honor it.

import asyncio
from openai import AsyncOpenAI, RateLimitError

client = AsyncOpenAI()
# Limit max concurrent in-flight calls — prevents hitting 429 in the first place
semaphore = asyncio.Semaphore(20)

async def ask(question: str, max_retries: int = 5) -> str:
    async with semaphore:                            # cap concurrency
        for attempt in range(max_retries):
            try:
                resp = await client.chat.completions.create(
                    model="gpt-4o-mini",
                    messages=[{"role": "user", "content": question}],
                )
                return resp.choices[0].message.content

            except RateLimitError as e:
                # Provider tells us when to retry — use it.
                retry_after = getattr(e.response.headers, "retry-after", None)
                wait = float(retry_after) if retry_after else (2 ** attempt)
                print(f"[429] Rate limited. Waiting {wait}s...")
                await asyncio.sleep(wait)
        raise Exception(f"Failed after {max_retries} retries")

# Fire 100 requests — semaphore caps to 20 in-flight, 429s gracefully backed off
answers = asyncio.run(asyncio.gather(*[ask(q) for q in questions]))

The Easy Fix Nobody Mentions

💡 Tier upgrades: OpenAI, Anthropic, and Google all upgrade your rate limits as you spend. After ~$5 spent you move to Tier 2 (3-4× higher RPM/TPM). After ~$50, Tier 3 (10×). Most "rate limit problems" in early prototypes vanish once you spend $50 → moving you from 500 RPM to 5,000 RPM. Check your usage dashboard before architecting elaborate throttling.
Mental Model
06 · Concurrency
Concurrent vs Parallel

Most engineers conflate these. They're not the same. Understanding the difference tells you when to use async, when to use threads, and when to spin up another process. For LLM calls — which are I/O-bound — concurrent (async) always wins.

Concurrent

async · single CPU core

Many tasks make progress by taking turns. One worker (the event loop) juggles all of them, switching whenever one starts waiting. Perfect for tasks that wait on something external (network, disk).

  • ▸ Tool: asyncio + await
  • ▸ Best for: I/O-bound (LLM calls, DB queries, file I/O)
  • ▸ 1 process · 1 thread · many tasks
  • ▸ Cheap context switches
→ RAG verdict: Use async for all your LLM and vector DB calls. This is 99% of what you need.

Parallel

multiprocessing · multiple cores

Many tasks run simultaneously on different CPU cores. Real wall-clock simultaneity. Necessary when each task is genuinely doing computation (not waiting).

  • ▸ Tool: multiprocessing, ProcessPoolExecutor
  • ▸ Best for: CPU-bound (heavy embedding math, image processing)
  • ▸ N processes · uses all your cores
  • ▸ Expensive process startup
→ RAG verdict: Useful when running local embedding models or bulk reindexing. Almost never for live queries.

Decision Cheat Sheet

Workload Bottleneck Use Why
API calls to OpenAI/Anthropic Network I/O asyncio Calls spend 99% waiting on response
Vector DB queries (Qdrant/Pinecone) Network I/O asyncio Same as above — remote service
Local sentence-transformers CPU/GPU multiprocessing Actual computation; threads blocked by GIL
PDF parsing (1000s of files) CPU ProcessPoolExecutor Each parse is CPU-heavy
Reading files / DB writes Disk I/O asyncio + aiofiles Disk I/O is a wait, not compute
Hidden Performance Win
07 · Pool
Connection Pooling & HTTP/2

Every fresh HTTP connection costs ~100ms for the TCP + TLS handshake. Do this for every API call and you've added a free second to each query. Connection pooling reuses one warm connection for many requests. HTTP/2 multiplexes many requests over a single connection. Both are free wins most engineers leave on the table.

The Latency Breakdown

FRESH CONNECTION (every request reconnects) ~1300ms total
100ms TCP
150ms TLS
1000ms LLM inference
Wasting 250ms per call on handshakes.
POOLED CONNECTION (reusing warm connection) ~1000ms total
1000ms LLM inference (handshake amortized)
+25% throughput, $0 cost. Just configure your HTTP client.

Configure It Once, Win Forever

The SDKs do most of this for you — but you need to reuse one client across all calls. Creating a new OpenAI() per request kills pooling.

# ❌ DON'T do this — creates a new connection pool every call
def bad_ask(q):
    client = OpenAI()                          # new client → new pool
    return client.chat.completions.create(...)

# ✅ DO this — module-level singleton, reused everywhere
import httpx
from openai import AsyncOpenAI

# Configure the HTTP transport for HTTP/2 + pooling
http_client = httpx.AsyncClient(
    http2=True,                              # multiplex many requests
    limits=httpx.Limits(
        max_connections=100,                  # total pool size
        max_keepalive_connections=20,         # warm connections to keep
        keepalive_expiry=30,                  # seconds before closing idle
    ),
    timeout=httpx.Timeout(30.0, connect=5.0),
)

client = AsyncOpenAI(http_client=http_client)  # one client, app-wide

async def ask(q: str) -> str:
    resp = await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": q}],
    )
    return resp.choices[0].message.content

# Clean shutdown on app exit
async def shutdown():
    await http_client.aclose()
