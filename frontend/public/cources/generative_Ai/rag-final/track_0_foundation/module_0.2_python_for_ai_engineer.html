02 · Type Safety
Type Hints: The Modern Python You Need

Python is dynamically typed — but in 2026, professional Python is statically annotated. Type hints let your IDE catch errors before runtime, let Pydantic validate LLM outputs, and let you reason about complex pipelines without holding 50 things in your head.

Untyped — what you wrote yesterday

def retrieve(query, k=5):
    hits = qdrant.search("docs", query, limit=k)
    return [h.payload for h in hits]

# What's `query`? str? list? embedding vector?
# What does it return? Dict? Object? Sometimes None?
# Your IDE has NO IDEA. Neither do you in 6 months.
  • ✗ No autocomplete
  • ✗ Bugs surface at runtime, in production
  • ✗ Refactoring is terrifying

Typed — what you write today

def retrieve(query: list[float], k: int = 5) -> list[dict]:
    hits = qdrant.search("docs", query, limit=k)
    return [h.payload for h in hits]

# Now: query is a vector, k is an int with default 5,
# returns list of dicts. IDE knows. Linter knows. You know.
  • ✓ Autocomplete works perfectly
  • ✓ mypy catches type bugs pre-commit
  • ✓ Refactoring is mechanical, not scary

The 12 Type Hints That Cover 95% of RAG Code

Python 3.11+ uses the clean list[X] syntax — no more List[X] imports.

PRIMITIVES
x: int
name: str
price: float
ready: bool
COLLECTIONS

meta: dict[str, str]
ids: set[int]
OPTIONAL · UNION
x: int | None
y: str | int
Old: Optional[int]
FUNCTIONS
def f(x: int) -> str:
def g() -> None:
Return type goes after −>
CALLABLES · ITERATORS

SPECIAL CASES

🛠 Tooling stack: Install mypy or pyright for static checking, enable Pylance/Ruff in VS Code, run mypy --strict src/ in CI. Type errors caught here = bugs that never reach production.

03 · Structured Data
Dataclasses vs Pydantic

Both let you define structured records. They look almost identical — but the difference is critical for AI engineering. Pydantic validates data at runtime, dataclasses trust you. With LLMs producing your data, you cannot trust anyone.

Dataclasses

Built-in · simple records
from dataclasses import dataclass

@dataclass
class Document:
    id: str
    text: str
    score: float
    metadata: dict[str, str] = None

# Works fine for trusted internal data
doc = Document(id="d1", text="...", score=0.92)

# Silently accepts wrong types! No validation.
bad = Document(id=42, text=["oops"], score="high")
# ↑ Runs without error. Crashes later in mysterious places.
  • Zero dependencies (built into Python)
  • Light & fast — perfect for internal records
  • No runtime validation
  • No JSON serialization helpers

Pydantic v2

3rd party · validation + serialization
from pydantic import BaseModel, Field

class Document(BaseModel):
    id: str
    text: str
    score: float = Field(ge=0, le=1)
    metadata: dict[str, str] = {}

# Validates on construction!
doc = Document(id="d1", text="...", score=0.92)

# Catches errors immediately:
bad = Document(id=42, score=2.0)
# ↑ ValidationError: id must be string, score must be ≤ 1

# Free JSON + dict conversion:
doc.model_dump_json()         # → JSON string
Document.model_validate_json(s)  # JSON → Document
  • Runtime validation with great errors
  • Built-in JSON serialize / deserialize
  • Field constraints (ge, le, regex, length…)
  • The standard for LLM-facing data

Which to Use When

Scenario Use Why
LLM tool / function-call schemas Pydantic LLM-generated args need validation
API request / response models (FastAPI) Pydantic Validate user input, auto OpenAPI schema
RAG retrieved chunk records Pydantic Validate metadata from external systems
Internal config objects dataclass No external data, no need for validation cost
Throwaway intermediate types dataclass or NamedTuple Lightweight, no runtime overhead

04 · The Throughput Engine
async/await Fundamentals

Every LLM call is 99% waiting — for the network, for the GPU, for the response. Synchronous code wastes that time. Async lets your program fire many calls and do other work while they bake. This isn't an optimization — it's how production AI is built.

Synchronous — wastes time

Each call blocks until done. 5 calls × 2s each = 10s wall-clock.

Call 1
Call 2
Call 3
Call 4
Call 5
Total: ~10 seconds

Async — overlaps the waits

All calls fire at once. The event loop juggles them. 5 calls × 2s = ~2s total.

Call 1
Call 2
Call 3
Call 4
Call 5
Total: ~2 seconds (5x faster)

The Mental Model in 3 Lines

async

Marks a function as awaitable — it can pause and resume. Calling it returns a coroutine, not a value.

await

Says "pause me here until this is done". The event loop steals the moment to run other tasks while you wait.

asyncio.run()

The entry point from sync land. Starts the event loop, runs your top-level async function, shuts it down.

Hello, async

import asyncio

async def fetch_chunk(doc_id: str) -> str:
    print(f"→ start {doc_id}")
    await asyncio.sleep(2)              # simulate I/O wait (e.g., LLM call)
    print(f"← done  {doc_id}")
    return f"chunk for {doc_id}"

async def main() -> None:
    # Sequential (the wrong way) — ~6 seconds total
    a = await fetch_chunk("d1")
    b = await fetch_chunk("d2")
    c = await fetch_chunk("d3")

    # Concurrent (the right way) — ~2 seconds total
    a, b, c = await asyncio.gather(
        fetch_chunk("d1"),
        fetch_chunk("d2"),
        fetch_chunk("d3"),
    )

asyncio.run(main())

Rule 1:

await only works inside an async def function. Top-level scripts use asyncio.run().

Rule 2:

Don't await in a loop when you can gather. Sequential awaits undo the whole point.

Rule 3:

Never call blocking sync code (e.g. requests.get) inside async — it freezes the event loop. Use httpx.AsyncClient.

05 · The Universal Format
Working with JSON in Python

Every API request, every API response, every LLM tool call, every database payload — it's all JSON. Master the four operations and you're never confused about serialization again.

PARSE STRING

json.loads()

JSON string → Python dict/list

data = json.loads(text)
PARSE FILE

json.load()

JSON file → Python object

data = json.load(f)
SERIALIZE STRING

json.dumps()

Python object → JSON string

text = json.dumps(d)
SERIALIZE FILE

json.dump()

Python object → JSON file

json.dump(d, f)

The Patterns You'll Use Daily

import json
from pathlib import Path

# 1. Parse an LLM response that's JSON-formatted
llm_output = '{"answer": "RAG retrieves docs", "sources": [1, 4, 7]}'
parsed = json.loads(llm_output)
print(parsed["answer"])              # → "RAG retrieves docs"

# 2. Pretty-print for logs / debugging
print(json.dumps(parsed, indent=2))

# 3. Save retrieved chunks to disk
chunks = [{"id": i, "text": f"chunk {i}"} for i in range(100)]
Path("chunks.json").write_text(json.dumps(chunks))

# 4. JSONL — one JSON object per line (perfect for streaming + batch)
with open("chunks.jsonl", "w") as f:
    for c in chunks:
        f.write(json.dumps(c) + "\n")

# 5. Read JSONL line by line (memory-friendly for huge files)
with open("chunks.jsonl") as f:
    for line in f:
        record = json.loads(line)
        # process one at a time

# 6. Handle invalid JSON safely
try:
    data = json.loads(maybe_bad)
except json.JSONDecodeError as e:
    print(f"LLM gave invalid JSON: {e}")

💡 JSONL is your friend: Lots of small JSON objects, one per line. Used by OpenAI's batch API, eval datasets, log aggregators, and most ML pipelines. Stream-friendly, append-friendly, grep-friendly.

06 · Talking to the World
HTTP Requests with httpx

requests is the famous sync library. httpx is its modern successor — same API, but also supports async and HTTP/2. In 2026, httpx is the default for production Python.

requests vs httpx

Capability requests httpx
Sync API ✓ ✓
Async API ✗ ✓
HTTP/2 multiplexing ✗ ✓
Connection pooling ✓ ✓
Streaming responses ✓ ✓
Type hints partial ✓

The Patterns You'll Use

import httpx
import asyncio

# ── 1. Simple sync request (debugging / scripts) ──────────────────
r = httpx.get("https://api.example.com/docs")
r.raise_for_status()              # raises on 4xx / 5xx
print(r.json())                     # auto-parse JSON

# ── 2. POST JSON with headers (the API-call pattern) ──────────────
r = httpx.post(
    "https://api.example.com/embed",
    headers={"Authorization": f"Bearer {token}"},
    json={"text": "hello"},
    timeout=30.0,                  # ALWAYS set timeouts
)
data = r.json()

# ── 3. Async + connection pooling (the production pattern) ───────
async def fetch_many(urls: list[str]) -> list[dict]:
    async with httpx.AsyncClient(
        http2=True,                  # multiplex many requests / connection
        timeout=30.0,
        limits=httpx.Limits(max_connections=50),
    ) as client:
        responses = await asyncio.gather(
            *[client.get(u) for u in urls]
        )
        return [r.json() for r in responses]

# ── 4. Streaming a large file ────────────────────────────────────
with httpx.stream("GET", big_file_url) as r:
    with open("dump.bin", "wb") as f:
        for chunk in r.iter_bytes():
            f.write(chunk)

⚡ Production rule: Never make HTTP calls without a timeout. Without one, a hung server keeps your task alive forever — every other call waits — your service slowly chokes. timeout=30 is a sane default.

07 · Disk I/O
File I/O Patterns for AI

You'll constantly read PDFs, write JSONL eval datasets, stream large corpora, walk document trees, and stash intermediate results. pathlib beats string-based paths; context managers (with) prevent file leaks; streaming patterns prevent OOM crashes on big datasets. 

from pathlib import Path
import json

# ── 1. pathlib — the modern path API ──────────────────────────────
data = Path("data")
data.mkdir(exist_ok=True)
docs_dir = data / "docs"             # join paths with /
docs_dir.mkdir(exist_ok=True)

# ── 2. Read / write small files in one line ──────────────────────
(docs_dir / "hello.txt").write_text("RAG rocks", encoding="utf-8")
content = (docs_dir / "hello.txt").read_text(encoding="utf-8")

# ── 3. Always use context managers for streaming ─────────────────
with open("big.txt", "r", encoding="utf-8") as f:
    for line in f:               # memory-friendly
        process(line)
# File auto-closes — no leak even if process() raises.

# ── 4. Walk a directory tree (e.g. ingest all PDFs) ──────────────
for pdf in data.rglob("*.pdf"):    # recursive glob
    print(pdf.name, pdf.stat().st_size)

# ── 5. JSONL — append one record at a time ───────────────────────
def append_jsonl(path: Path, record: dict) -> None:
    with path.open("a", encoding="utf-8") as f:
        f.write(json.dumps(record) + "\n")

# ── 6. Binary files for images / PDFs ────────────────────────────
img_bytes = (data / "logo.png").read_bytes()
(data / "copy.png").write_bytes(img_bytes)

✓ Always use pathlib.Path:

Cross-platform, type-hint-friendly, supports / operator, has .exists(), .stat(), .rglob(). os.path is the old way.

✗ Never f.read() a multi-GB file:

It loads everything into RAM. Iterate line by line for text, chunk by chunk for binary. Streaming is mandatory for production data pipelines.

08 · Survive APIs
Error Handling for API Calls

Networks fail. Providers rate-limit. Models go down. The difference between a hobby script and a production service is how it handles the inevitable. Catch specific exceptions, retry the retryable ones, propagate the rest with context.

🎯

Catch specific

Never except Exception: — it hides bugs. Catch the named exception you actually expect.

🔁

Retry only retryable

429, 500, 503, timeout = retry. 400, 401, 403 = your bug — fix it, don't loop.

📣

Re-raise with context

Use raise X from e to preserve the traceback. Debugging without context is misery.

The Production Pattern

import httpx
from openai import OpenAI, RateLimitError, APIError, APITimeoutError

client = OpenAI(timeout=30)

def embed_safe(text: str) -> list[float] | None:
    try:
        r = client.embeddings.create(
            model="text-embedding-3-small",
            input=text,
        )
        return r.data[0].embedding

    except RateLimitError:                    # 429 — retry-friendly
        logger.warning("rate_limited")
        raise

    except APITimeoutError:                   # timeout — retry-friendly
        logger.warning("timeout", extra={"len": len(text)})
        raise

    except APIError as e:                     # server error
        logger.error("api_error", extra={"status": e.status_code})
        if e.status_code and e.status_code >= 500:
            raise                              # retry-friendly
        return None                            # client error — don't retry

    except Exception as e:                    # UNEXPECTED — log loudly
        logger.exception("unknown_error")
        raise RuntimeError(f"embed failed for text len={len(text)}") from e

Retry or Bail?

✅ ALWAYS retry

429 · 500 · 502 · 503 · 504
ConnectionError · Timeout

Transient — the next attempt will likely succeed.

❌ NEVER retry

400 · 401 · 403 · 404
JSONDecodeError · ValidationError

These are your bug. Retrying just spams the provider.

🔭 Hand-rolling retries is fragile. Module 0.4 introduces tenacity, the production-grade retry decorator. For now, know that specific exceptions get caught and specific recovery happens.

09 · The Invisible Skill
Logging vs print — and Why It Matters

It's 3am. Your RAG service is throwing 500s. You SSH into the box. The only window into what happened is whatever you wrote during normal operation. Print statements got nuked at deploy. Structured logs survive. Master logging now — your future self will thank you.

print() — fine for hobby

  • No severity level (info? error? critical?)
  • No timestamp by default
  • No way to filter or route to a file
  • No structured fields for log aggregators
  • Always on — pollutes output
  • Strips formatting in pipes

logging — production-ready

  • 5 severity levels (DEBUG → CRITICAL)
  • Auto timestamp, logger name, file, line
  • Pluggable handlers: stdout, file, Sentry, Datadog
  • Structured fields via extra=
  • Filter by level in production (only WARNING+)
  • Captures full exception tracebacks

The Five Log Levels

DEBUG Verbose internals — variable values, code path. Off in production.
INFO Normal events — "request received", "embedding cached". The default visibility.
WARNING Something unusual but recoverable — rate limit hit, fallback used, retry needed.
ERROR A request / operation failed but the service is still alive.
CRITICAL Service-level failure — DB down, can't start, must page on-call.

Setting Up Proper Logging

import logging

# One-time setup at the entry point (e.g. main.py)
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)

# Each module gets its own named logger
logger = logging.getLogger(__name__)

def retrieve(query: str, k: int = 5) -> list[dict]:
    logger.info("retrieve start", extra={"k": k, "query_len": len(query)})

    try:
        hits = qdrant.search("docs", embed(query), limit=k)
    except Exception:
        logger.exception("retrieve failed")        # auto-includes traceback
        raise

    if not hits:
        logger.warning("no_hits", extra={"query": query[:80]})
        return []

    logger.info("retrieve ok", extra={"n_hits": len(hits), "top_score": hits[0].score})
    return hits
