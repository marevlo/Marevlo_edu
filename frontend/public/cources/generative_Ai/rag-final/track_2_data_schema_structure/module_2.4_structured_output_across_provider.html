The Strictness Spectrum

Three Tiers of Enforcement

All structured-output mechanisms — across every provider — collapse to one of three tiers. Recognize the tier, you've recognized the technique.

TIER 1 · WEAKEST

Prompt-Only

"Return JSON matching this shape: {…}" — just instructions. Easy. Universal. Unreliable.

  • Works on every model
  • Zero API config
  • 5–20% failure rate
  • May emit prose around the JSON
TIER 2 · STRONGER

JSON Mode

A provider flag that guarantees the response parses as JSON — but not that it matches your shape.

  • Guaranteed valid JSON
  • No prose wrapping
  • Shape still unconstrained
  • Still need post-validation
TIER 3 · STRONGEST

Schema-Constrained

The provider takes your schema and biases the decoder to emit only schema-valid tokens. Near-100% conformance.

  • ≈ 99–100% conformance
  • Types and enums enforced
  • Recoverable schema bugs surface fast
  • Slight schema dialect quirks per vendor
⚡ Pick the strongest tier your provider supports. Then keep the Tier 1 prompt instruction as belt-and-suspenders, and validate with Pydantic anyway. That's the production recipe.
The Big Three

Provider APIs Side by Side

Same Pydantic model. Three different API shapes. Click through to see the exact code.

from openai import OpenAI
from pydantic import BaseModel, Field

class RAGAnswer(BaseModel):
    answer: str
    sources: list[str] = Field(min_length=1)
    confidence: float = Field(ge=0, le=1)

client = OpenAI()

# Modern SDK — auto-parses into the Pydantic model
resp = client.chat.completions.parse(
    model="gpt-4o-2024-08-06",
    messages=[
        {"role": "system", "content": "Answer using ONLY the context."},
        {"role": "user", "content": prompt}
    ],
    response_format=RAGAnswer          # ← pass the model directly
)

answer: RAGAnswer = resp.choices[0].message.parsed

# Lower-level form — pass the raw JSON Schema
resp = client.chat.completions.create(
    model="gpt-4o-2024-08-06",
    messages=[...],
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "rag_answer",
            "schema": RAGAnswer.model_json_schema(),
            "strict": True                # 100% conformance
        }
    }
)
OpenAI

json_schema mode

Tier 3 — true schema constraint. Use .parse() for auto-Pydantic; pass strict: true for hard guarantees.

⚠ Restrictions in strict mode: every property must be in required; additionalProperties must be false; only a subset of keywords is supported.
⚡ Best for: RAG answers, classifications, anything where you control the schema and want zero retry overhead. 
import anthropic
from pydantic import BaseModel, Field

class RAGAnswer(BaseModel):
    answer: str
    sources: list[str] = Field(min_length=1)
    confidence: float = Field(ge=0, le=1)

client = anthropic.Anthropic()

# Anthropic's trick: define a "fake" tool whose input is your schema,
# then force the model to call it. The arguments become your payload.
resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[{
        "name": "emit_rag_answer",
        "description": "Emit the grounded answer payload.",
        "input_schema": RAGAnswer.model_json_schema()
    }],
    tool_choice={"type": "tool", "name": "emit_rag_answer"},
    messages=[{"role": "user", "content": prompt}]
)

# Pull the tool_use block and parse
block = next(b for b in resp.content if b.type == "tool_use")
answer = RAGAnswer.model_validate(block.input)
Anthropic

Tool-schema pattern

No dedicated json_schema mode — instead, use the tool system. Define a tool whose input matches your shape, then force-call it.

⚠ Gotcha: The response is a list of content blocks. Always filter for type == "tool_use" — text blocks can sneak in alongside.
⚡ Strength: Tier 2.5 — very high conformance in practice, but the constraint isn't quite as hard as OpenAI strict mode. Pair with Pydantic validation. 
from google import genai
from pydantic import BaseModel, Field

class RAGAnswer(BaseModel):
    answer: str
    sources: list[str] = Field(min_length=1)
    confidence: float = Field(ge=0, le=1)

client = genai.Client()

# Gemini accepts the Pydantic class directly OR a JSON Schema dict
resp = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=prompt,
    config={
        "response_mime_type": "application/json",
        "response_schema": RAGAnswer           # ← Pydantic class or dict
    }
)

# Parsed Pydantic instance is available directly
answer: RAGAnswer = resp.parsed

# Or raw text if you want it
raw_json = resp.text
Gemini

responseSchema config

Pass your Pydantic class as response_schema; Gemini handles the conversion and returns a parsed object on .parsed.

⚠ Caveats: Gemini's schema dialect is a subset of JSON Schema — no $ref, limited anyOf, some keyword names differ. Test on your real schemas first.
⚡ Best for: Long-context structured extraction, multimodal pipelines, cost-sensitive workloads at Flash-tier pricing.
Under the Hood

How Constrained Generation Actually Works

The mechanic behind every Tier 3 mechanism. Once you see it, every provider's API stops being mysterious.

The Core Idea

An LLM picks the next token from its full vocabulary by sampling from probabilities (logits). Constrained decoding masks out every token that would violate the schema, leaving only valid choices.

Mechanic: FSM / Regex

The schema is compiled into a finite state machine. At each decode step, the FSM lists which next tokens keep the output valid. Invalid tokens get their logits set to -∞.

The Guarantee

The model literally cannot produce invalid JSON — there are no valid tokens that lead there. Conformance becomes a mathematical property of decoding, not a model behavior to coax.

DECODE STEP
"category": "█

Schema requires "category" to be one of ["billing","shipping","technical"]. The decoder masks every token that doesn't lead to one of those.

CANDIDATE TOKENS
EMITTED SO FAR
"category": "█
⚡ Why providers do this differently: OpenAI / Gemini run constrained decoding inside their serving stack. Anthropic enforces shape via the tool system + post-processing. Outlines and Guidance bring constrained decoding to open-source models running locally — same idea, different ownership.
Beyond the Providers

The Three Libraries Worth Knowing

Each fills a gap providers don't cover. You won't use all three — but knowing them is what separates a hobbyist from a practitioner.

Outlines DECODE

For Open-Source Models

Brings constrained decoding to any HuggingFace model — Llama, Mistral, Qwen — without provider lock-in.

import outlines

model = outlines.models.transformers("...")

generator = outlines.generate.json(
    model, RAGAnswer
)
answer = generator(prompt)
  • Hard constraint, like provider Tier 3
  • Regex, JSON Schema, or Pydantic input
  • Requires hosting your own model
Instructor WRAP

For Pydantic-First Workflows

A thin wrapper around provider SDKs that adds Pydantic-driven validation, retries, and async — across all major providers.

import instructor
from openai import OpenAI

client = instructor.from_openai(OpenAI())

answer = client.chat.completions.create(
    model="gpt-4o",
    response_model=RAGAnswer,
    max_retries=3,
    messages=[...]
)
  • Built-in retry-with-feedback loop
  • Unified API across providers
  • Tier 2.5 — relies on provider tools/JSON mode
Guidance DECODE

For Grammar-Driven Control

Microsoft's library for templating prompts as programs — interleaving generation, regex, choices, and code, all with hard constraints.

from guidance import models, gen, select

lm = models.LlamaCpp("model.gguf")

lm += f'category: "'
lm += select(["billing","shipping","technical"])
lm += '", confidence: '
lm += gen(regex=r"0\.\d+")
  • Most expressive — beyond JSON Schema
  • Mix gen with code / control flow
  • Local / self-hosted models only
⚡ When to reach beyond providers: Use Instructor if you want Pydantic-first ergonomics today on closed APIs. Use Outlines if you self-host and want hard constraints. Use Guidance for templated multi-step generation. For 80% of RAG work, the native provider API + Pydantic is enough.

Question 01

Hosted API or self-hosted model?

If you're calling someone else's model over HTTPS, you live in the provider world.

→ Hosted: use the provider's native structured mode
→ Self-hosted: Outlines or Guidance
Question 02

Are you switching providers a lot?

Multi-provider apps want a uniform Python API.

→ Yes: Instructor (wraps all the SDKs)
→ No: use the SDK directly
Question 03

Do you need 99%+ conformance per call?

Some workloads can't tolerate retries; some absolutely can.

→ Yes: Tier 3 only — OpenAI strict, Gemini schema, Outlines
→ Acceptable: Tier 2 + Pydantic + retries
Question 04

Does your "schema" go beyond JSON?

Templated outputs, mixed regex + JSON, multi-step generation.

→ Yes: Guidance — grammars + control flow
→ No: JSON Schema via your provider
⚡ Default recipe for 80% of RAG work: Pydantic model → provider's native Tier 3 mode → Pydantic validation on the response → small retry loop with error feedback. No extra library needed. Reach for Instructor / Outlines / Guidance only when your default hits a real ceiling.
Self-Check

Three Quick Questions

Lock in the matrix before moving on.

QUESTION 01

Which provider doesn't have a dedicated "JSON schema mode" and instead routes structured outputs through the tool system?

Correct. Anthropic's path is to define a "fake" tool whose input_schema matches your desired payload, then force a tool call with tool_choice.
QUESTION 02

What does Tier 2 (JSON mode) guarantee that Tier 1 (prompt-only) doesn't?

Correct. JSON mode kills prose wrapping and guarantees parseability. Schema conformance is Tier 3. Always pair Tier 2 with Pydantic validation.
QUESTION 03

You're self-hosting Llama 3 and need hard schema constraints. Best tool?

Correct. Outlines applies constrained decoding to any HuggingFace-compatible model — same Tier 3 mechanic as OpenAI's strict mode, but on weights you control.
