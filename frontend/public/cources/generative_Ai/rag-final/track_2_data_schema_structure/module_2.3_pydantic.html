The Foundation

BaseModel — Where Everything Begins

Subclass BaseModel, annotate fields with type hints — Pydantic does the rest. Validation, parsing, serialization, JSON Schema generation.

DEFINE THE MODEL
from pydantic import BaseModel
from datetime import datetime

class Chunk(BaseModel):
    id: str
    text: str
    source: str
    page: int | None = None
    created_at: datetime

Five lines define: types, optional fields, defaults, and a JSON-compatible schema — automatically.

USE THE MODEL
# 1. Parse incoming JSON
raw = '{"id":"c1","text":"...","source":"faq.md","created_at":"2026-05-17T10:00:00"}'
chunk = Chunk.model_validate_json(raw)

# 2. Or from a dict
chunk = Chunk.model_validate({"id":"c1", ...})

# 3. Use as a typed object
print(chunk.source)        # "faq.md"
print(chunk.created_at.year) # 2026 (real datetime!)

# 4. Serialize back
chunk.model_dump()        # dict
chunk.model_dump_json()   # str

# 5. Get the JSON Schema, free
Chunk.model_json_schema()

Constraints & Defaults

Field() — Where Constraints Live

Type hints declare the shape. Field() declares the rules: bounds, regex, descriptions, examples, defaults.

from pydantic import BaseModel, Field
from typing import Literal

class RAGAnswer(BaseModel):
    answer: str = Field(
        min_length=1,
        max_length=2000,
        description="The grounded answer to the user's question."
    )

    sources: list[str] = Field(
        min_length=1,                          # at least 1 citation
        description="Chunk IDs supporting the answer.",
        examples=[["chunk_001", "chunk_042"]]
    )

    confidence: float = Field(
        ge=0, le=1,                           # 0 ≤ confidence ≤ 1
        description="Self-reported confidence in [0,1]."
    )

    category: Literal["billing", "shipping", "technical"] = Field(
        description="Closed set of categories. Acts like JSON Schema enum."
    )

    debug_id: str | None = Field(
        default=None,
        pattern=r"^dbg_[a-z0-9]{8}$"            # regex
    )

The Most Useful Field Options

default

Static default. Required if you want any optional field.

default_factory

A callable — use for mutable defaults like list.

ge / le / gt / lt

Inclusive/exclusive numeric bounds.

min_length / max_length

Length bounds for strings, lists, sets, tuples.

pattern

Regex the string must match. Pure Python regex.

description / examples

Docs & examples — passed straight into the generated JSON Schema.

alias / validation_alias

Accept input under a different key (great for messy LLM output).

frozen

Make a field immutable after construction.

discriminator

For tagged unions — pick the right subclass by a field's value.

⚡ Pro tip: Put description on every field. Pydantic injects it into model_json_schema(), which goes straight into your LLM prompt — better descriptions ⇒ better model output.
Custom Logic

Validators — When Constraints Aren't Enough

Most rules fit in Field(). The rest belong in validators — Python functions that run during model construction.

SINGLE-FIELD VALIDATION
from pydantic import BaseModel, field_validator

class Citation(BaseModel):
    chunk_id: str
    quote: str

    @field_validator("chunk_id")
    @classmethod
    def must_be_chunk_id(cls, v: str) -> str:
        if not v.startswith("chunk_"):
            raise ValueError("chunk_id must start with 'chunk_'")
        return v

    @field_validator("quote")
    @classmethod
    def trim(cls, v: str) -> str:
        return v.strip()  # normalize

When to reach for it

  • Format checks beyond what regex can express
  • Normalization (trim, lowercase, parse)
  • Lookup against an enum or database
  • Domain rules ("must be a workday")
⚡ Rule: Return the validated (possibly transformed) value. Raise ValueError to fail. Always decorate with @classmethod.
CROSS-FIELD VALIDATION
from pydantic import BaseModel, model_validator
from typing import Self

class RAGAnswer(BaseModel):
    status: Literal["answered", "refused"]
    answer: str | None = None
    citations: list[str] = []
    refusal_reason: str | None = None

    @model_validator(mode="after")
    def check_consistency(self) -> Self:
        if self.status == "answered":
            if not self.answer:
                raise ValueError("answered → answer required")
            if not self.citations:
                raise ValueError("answered → ≥1 citation required")
        elif self.status == "refused":
            if not self.refusal_reason:
                raise ValueError("refused → reason required")
        return self

When to reach for it

  • Rules involving two or more fields
  • Conditional requirements ("if X, then Y must…")
  • Computed/derived fields
  • Consistency checks across the whole object
⚡ mode="after" runs once all fields are individually validated and assigned — you get a fully-typed self to inspect.

mode="before"

Runs before Pydantic parses the field. Receives raw input. Use to massage data into a parseable shape.

@field_validator("sources", mode="before")
@classmethod
def accept_csv(cls, v):
    if isinstance(v, str):
        return [s.strip() for s in v.split(",")]
    return v

mode="after"

The default. Runs after type parsing. Receives the typed value. Use for business rules and normalization.

@field_validator("confidence")  # default mode
@classmethod
def round_3(cls, v: float) -> float:
    return round(v, 3)

Catching the Error

A failed validator raises ValidationError — a structured object with one error per failing field. This is the gold dust that powers retry loops (next section).

from pydantic import ValidationError

try:
    ans = RAGAnswer.model_validate(llm_output)
except ValidationError as e:
    for err in e.errors():
        print(err["loc"], err["msg"])
    # [("confidence",), "Input should be ≤ 1"]
    # [("citations",), "List should have at least 1 item"]
Self-Healing

The Retry Loop With Error Feedback

The single most-important Pydantic pattern. When the LLM produces something invalid, feed the validation error back in and ask it to fix the output.

1. LLM Call

Prompt + schema description sent to the model.

2. Pydantic Validates

Parses output. Catches type, range, and rule violations.

3. Feedback Retry

Inject error into next prompt. Model auto-corrects.

THE RETRY LOOP
from pydantic import ValidationError
import json

def structured_call(prompt, Model, max_retries=3):
    messages = [
        {"role": "system",
         "content": f"Return JSON matching: {Model.model_json_schema()}"},
        {"role": "user", "content": prompt}
    ]

    for attempt in range(max_retries):
        resp = llm.chat(messages, response_format="json")
        text = resp.choices[0].message.content

        try:
            return Model.model_validate_json(text)
        except ValidationError as e:
            # inject the error so the model self-corrects
            messages.append({"role": "assistant", "content": text})
            messages.append({
                "role": "user",
                "content": f"Your response failed validation:\n{e.errors()}\n"
                           f"Fix it and return ONLY the corrected JSON."
            })

    raise RuntimeError(f"Failed after {max_retries} retries")

Self-Correction Trace

⚡ Why it works: The model sees its own bad output plus the structured error — exactly the kind of signal an LLM is great at acting on. Retry-with-feedback fixes ~85% of validation failures on the first retry.

Cap retries hard

Usually 2–3. Infinite loops waste tokens and lock pipelines.

Bump temperature

Some teams raise temperature on retry to escape sticky failure modes.

Log every retry

First-attempt success rate is a great quality signal — and a cost signal.

When to Use Which

Pydantic ↔ JSON Schema

They're not competitors. They're collaborators. Here's the division of labor.

Dimension Pydantic JSON Schema
Lives where? Inside your Python process Anywhere — files, APIs, prompts
Authored as Python classes with type hints JSON documents
Strength Runtime validation, custom logic, parsing Cross-language, declarative, portable
LLM provider input Via .model_json_schema() Direct — providers already speak it
Custom rules @field_validator, arbitrary Python Limited to spec keywords + pattern
IDE support Full — autocomplete, refactor, types None (it's just JSON)
Best use Inside the Python boundary At the API / LLM contract layer
⚡ The standard recipe: Define one Pydantic model. Call .model_json_schema() to ship the JSON Schema to your LLM provider. Validate every incoming payload with the same model. One source of truth, two surface areas.
Future Outlook

BAML — The Next Wave

Pydantic is the present standard. BAML (Boundary AI Markup Language) is one promising future — a DSL purpose-built for LLM I/O contracts. Worth knowing about; not yet required.

BAML SAMPLE
// rag.baml — declarative LLM contract
class RAGAnswer {
  answer       string
  citations    string[]  @assert(len(this) > 0)
  confidence   float     @assert(this >= 0 and this <= 1)
}

function AnswerWithCitations(question: string, context: string) -> RAGAnswer {
  client GPT4o
  prompt #"
    Answer the question using ONLY this context: {{ context }}
    Question: {{ question }}

    {{ ctx.output_format }}
  "#
}

What BAML adds

  • Schema + prompt in one place — coupled together
  • Generates typed clients for Python, TS, Ruby, Go
  • First-class testing & comparison playground
  • Schema-aligned parsing — recovers from minor LLM mistakes

Trade-offs

  • New language to learn (and review)
  • Build step in your pipeline
  • Smaller community than Pydantic
  • Tool maturity still evolving
⚡ Verdict for this course: Ship with Pydantic. Keep an eye on BAML. The ergonomics it pioneers will likely shape the next generation of structured-output tools — regardless of who wins.
Self-Check

Three Quick Questions

Cement the patterns before moving on.

QUESTION 01

You need to enforce that if status == "answered" then citations must have at least one item. Which tool?

Correct. The rule involves two fields, so it belongs in @model_validator. A single-field validator can't see other fields' values, and Field constraints are unconditional.
QUESTION 02

Your retry loop catches a ValidationError. What's the highest-leverage thing to do with it?

Correct. Feedback-driven retry typically fixes ~85% of failures on the first retry. The structured errors from e.errors() are exactly the kind of signal LLMs respond to well.
QUESTION 03

Where should Pydantic and JSON Schema each live in your RAG pipeline?

Correct. One Pydantic model is the source of truth. It validates every payload inside Python. Export its JSON Schema at the boundary — to LLM providers, API consumers, anyone outside your process.
