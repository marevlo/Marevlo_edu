Fundamentals

What Is JSON Schema?

A vocabulary for describing the shape of JSON data. A schema is itself JSON — so it's machine-readable, language-agnostic, and shippable.

Schemas describe data, not behavior

A JSON Schema answers "what does valid data look like?" It says nothing about how the data is created or consumed — it's a static contract.

Self-describing & composable

A schema is itself JSON. You can validate schemas with schemas, embed them in APIs, version them, and combine them with $ref and allOf.

Universal across providers

OpenAI's structured outputs, Anthropic's tool schemas, Pydantic's .model_json_schema(), OpenAPI specs — all speak JSON Schema. Learn it once, use it everywhere.

ANATOMY OF A SCHEMA
{
  // metadata
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://api.acme.com/schemas/rag_answer.json",
  "title": "RAGAnswer",
  "description": "A grounded answer with citations",

  // type system
  "type": "object",
  "properties": {
    "answer": {
      "type": "string",
      "minLength": 1,
      "maxLength": 2000
    },
    "sources": {
      "type": "array",
      "items": { "type": "string" },
      "minItems": 1,
      "uniqueItems": true
    },
    "confidence": {
      "type": "number",
      "minimum": 0,
      "maximum": 1
    }
  },

  // constraints
  "required": ["answer", "sources"],
  "additionalProperties": false
}
The Vocabulary

Essential Keywords

You don't need every keyword — these are the 12 that cover 95% of real-world schemas.

type CORE

The data type. One of string, number, integer, boolean, array, object, null.

"type": "string"
properties OBJECT

Map of field name → schema. Defines the structure of an object.

"properties": { "name": ... }
required OBJECT

Array of property names that must be present. Missing one ⇒ invalid.

"required": ["id", "text"]
additionalProperties STRICT

Set to false to reject unknown keys. Always set this for LLM outputs.

"additionalProperties": false
enum CONSTRAINT

Restrict a value to a fixed set. Perfect for classifications and labels.

"enum": ["low","med","high"]
const CONSTRAINT

Exactly one allowed value. Useful for discriminators in unions.

"const": "v2"
min/max NUMERIC

minimum, maximum, exclusiveMinimum, exclusiveMaximum.

"minimum": 0, "maximum": 1
minLength / maxLength STRING

Character bounds. Stops LLMs from returning empty answers or 50KB walls of text.

"minLength": 1, "maxLength": 500
pattern STRING

A regex the string must match. Great for IDs, codes, structured strings.

"pattern": "^doc_\\d+$"
format STRING

Named patterns: email, uri, date-time, uuid.

"format": "date-time"
items ARRAY

Schema each array element must satisfy.

"items": { "type": "string" }
minItems / uniqueItems ARRAY

Length bounds + dedup. Force at least one citation; reject duplicates.

"minItems": 1, "uniqueItems": true
⚡ Compound keywords let you express unions and refinements: anyOf (matches at least one), oneOf (matches exactly one), allOf (matches all), not (inverts). You'll meet them in Module 2.2.

Forcing the Model

Make LLMs Obey the Schema

Three levels of strictness — pick the strongest one your provider supports, then add validation as backstop.

1

Prompt-Level Instruction

weakest

Tell the model what JSON shape you want in the prompt. Always do this — even when stronger mechanisms are available. Models obey the prompt 80–95% of the time, but production needs guarantees.

2

JSON Mode

stronger

A provider flag (response_format: { type: "json_object" }) that guarantees some valid JSON comes back. Doesn't guarantee the right shape — just that it parses.

3

Schema-Constrained Generation

strongest

Pass the schema directly to the API (OpenAI json_schema, Anthropic tool schemas, Gemini responseSchema). The model is constrained at decode time to emit only schema-valid tokens. Modules 2.3 covers each provider's exact API.

All Three Together — the Production Recipe 

from openai import OpenAI
from jsonschema import validate, ValidationError

schema = {
    "type": "object",
    "properties": {
        "answer":     {"type": "string", "minLength": 1},
        "sources":    {"type": "array", "items": {"type": "string"}, "minItems": 1},
        "confidence": {"type": "number", "minimum": 0, "maximum": 1}
    },
    "required": ["answer", "sources"],
    "additionalProperties": False
}

resp = OpenAI().chat.completions.create(
    model="gpt-4o-2024-08-06",
    messages=[
        {"role": "system",
         "content": "Answer using ONLY the context. Return JSON matching the schema."},  # LEVEL 1
        {"role": "user", "content": user_question}
    ],
    response_format={                                          # LEVEL 3
        "type": "json_schema",
        "json_schema": {
            "name": "rag_answer",
            "schema": schema,
            "strict": True
        }
    }
)

payload = json.loads(resp.choices[0].message.content)

# LEVEL 0 — ALWAYS validate, even with strict mode. Trust no one.
try:
    validate(instance=payload, schema=schema)
except ValidationError as e:
    # retry with error feedback (see Module 2.2)
    ...
⚡ The mantra: Prompt + Constraint + Validate. Skip any one and you'll eventually ship a bug. The cost of all three is microseconds. The cost of debugging malformed payloads at 3 AM is hours.
Battle-Tested

Schema Patterns for RAG & AI Tasks

Five schemas you'll copy-paste into hundreds of projects. Each one solves a recurring shape problem.

Cited Answer Pattern

Force every answer to ship with traceable sources — the core RAG contract.

RAG · Citations 
{
  "type": "object",
  "properties": {
    "answer":     { "type": "string", "minLength": 1, "maxLength": 2000 },
    "citations": {
      "type": "array",
      "minItems": 1,
      "items": {
        "type": "object",
        "properties": {
          "chunk_id": { "type": "string", "pattern": "^chunk_\\d+$" },
          "quote":    { "type": "string", "minLength": 1, "maxLength": 300 }
        },
        "required": ["chunk_id", "quote"],
        "additionalProperties": false
      }
    }
  },
  "required": ["answer", "citations"],
  "additionalProperties": false
}

Why it works: minItems: 1 forbids zero-citation answers, the regex on chunk_id forces the model to reference real chunks, and maxLength: 300 on quotes prevents lazy paragraph-dumps.

Classification Pattern

Bound the output to a fixed label set — no hallucinated categories.

Classify · Label 
{
  "type": "object",
  "properties": {
    "category": {
      "type": "string",
      "enum": ["billing", "shipping", "technical", "account", "other"]
    },
    "sentiment": {
      "type": "string",
      "enum": ["positive", "neutral", "negative"]
    },
    "urgency": {
      "type": "integer",
      "minimum": 1,
      "maximum": 5
    }
  },
  "required": ["category", "sentiment", "urgency"],
  "additionalProperties": false
}

Why it works: enum closes the output space — the model literally cannot return "complaint" when only the five listed values are valid. Integer with bounds makes urgency tractable for downstream rules.

Entity Extraction Pattern

Pull structured records from unstructured text — invoices, resumes, emails.

Extract · NER 
{
  "type": "object",
  "properties": {
    "invoice_number": { "type": "string", "pattern": "^INV-\\d{6}$" },
    "vendor":         { "type": "string", "minLength": 1 },
    "date":           { "type": "string", "format": "date" },
    "amount":         { "type": "number", "minimum": 0 },
    "currency":       { "type": "string", "enum": ["USD", "EUR", "GBP", "INR"] },
    "line_items": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "description": { "type": "string" },
          "quantity":    { "type": "integer", "minimum": 1 },
          "unit_price":  { "type": "number", "minimum": 0 }
        },
        "required": ["description", "quantity", "unit_price"],
        "additionalProperties": false
      }
    }
  },
  "required": ["invoice_number", "vendor", "date", "amount", "currency"],
  "additionalProperties": false
}

Why it works: Strong pattern on the invoice number, ISO date format, currency enum, and a nested array schema all push the model toward consistent, machine-actionable output.

Router / Tool-Choice Pattern

Make the model decide which downstream branch to run.

Agent · Routing 
{
  "type": "object",
  "properties": {
    "action": {
      "type": "string",
      "enum": ["search_docs", "create_ticket", "answer_directly", "refuse"]
    },
    "reason": { "type": "string", "minLength": 1 },
    "query":  { "type": "string" }
  },
  "required": ["action", "reason"],
  "additionalProperties": false
}

Why it works: Forcing a reason alongside the action dramatically improves routing quality — the model has to commit to a justification, which acts as an inline chain-of-thought.

Refuse-or-Answer Pattern

Let the model gracefully decline when the context doesn't support an answer.

RAG · Grounding 
{
  "type": "object",
  "properties": {
    "status": { "type": "string", "enum": ["answered", "insufficient_context"] },
    "answer":     { "type": ["string", "null"] },
    "citations":  { "type": "array", "items": { "type": "string" } },
    "missing_info": { "type": ["string", "null"] }
  },
  "required": ["status"],
  "additionalProperties": false
}

Why it works: Most hallucinations happen when the model feels pressured to answer. Giving it an explicit refusal slot — "insufficient_context" — gives it permission to say "I don't know."

Long-Term Survival

Schema Versioning

Your first schema is never your last. Plan for evolution from day one — or pay the migration tax with interest.

Backward-Compatible Changes

Old consumers keep working. Ship freely — no version bump needed.

  • Add a new optional property
  • Add new values to an enum (if consumers handle unknowns)
  • Relax a constraint (widen maximum, lower minLength)
  • Add documentation, examples, titles

Breaking Changes

Old consumers will fail. Bump the version. Run both in parallel.

  • Add a new required property
  • Rename or remove a property
  • Change a property's type
  • Tighten a constraint (lower maximum, raise minLength)
  • Remove enum values

Three Versioning Strategies

URL Versioning

Embed the version in the schema's $id.

"$id": ".../v1/answer.json"
"$id": ".../v2/answer.json"

Best for: public APIs where consumers pin a URL.

Discriminator Field

Use a const field inside the schema.

"version": { "const": "v2" }

Best for: storing payloads in a queue or DB where you mix versions.

SemVer in Metadata

Track major.minor in the title or a custom keyword.

"title": "RAGAnswer 2.1.0"

Best for: internal schemas with a clear ownership team.

⚡ The golden rule: "Add — don't change. Deprecate — don't delete." When v2 ships, keep v1 alive for at least two release cycles. Log deprecation warnings. Migrate consumers gradually. Schemas are public contracts.
Self-Check

Three Quick Questions

Lock in the concepts before moving on.

QUESTION 01

You want an LLM to return one of five fixed category labels. Which keyword is the right tool?

Correct. enum closes the output space to exactly your listed values — the model literally cannot return anything else when strict mode is on.
QUESTION 02

Which of these is a breaking schema change?

Correct. Old payloads that don't include the new field will suddenly fail validation. Bump the version and run both in parallel.
QUESTION 03

Why validate with jsonschema even when you've used the provider's strict mode?

Correct. Provider strict modes catch most issues — but they're not perfect, support varies, and your validator is the contract you control. Prompt + Constraint + Validate, always.
