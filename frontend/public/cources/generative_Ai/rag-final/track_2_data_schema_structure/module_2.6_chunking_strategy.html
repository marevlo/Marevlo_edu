Chunking Strategies

Why It Matters

Why Chunking Is Hard

Every chunk competes with every other chunk at retrieval. The wrong boundary makes the right answer disappear from the top-K.

Failure A

Chunks Too Big

A 4,000-token chunk means a 4,000-token bag of mixed topics. Embeddings blur, retrieval pulls in noise, and the LLM wastes context on irrelevant pages.

  • Diluted embeddings → poor recall
  • Wasted context tokens at generation
  • Citations point to giant blobs
Failure B

Chunks Too Small

Sentence-sized chunks miss context. "It costs $99" — but what costs? The answer was two sentences up and got separated by a chunk boundary.

  • Missing referents (it / they / this)
  • Many similar chunks compete for top-K
  • Embedding overhead × N
The Goldilocks Rule A good chunk holds one self-contained idea — enough context to be understood on its own, small enough to embed as a single concept. The trick isn't picking a number; it's respecting the structure of the underlying document.

Split mid-sentence

A fixed-size 256-token splitter chops the sentence "...refunds are accepted within" — making the chunk meaningless and unretrievable.

Lost header context

A chunk says "30 days" but the header "## Refund Policy" sits in the previous chunk. Retrieval finds the number but loses the topic.

Mixed topics

A long-doc chunk holds half the refund policy and the start of shipping policy. Its embedding is the average — neither topic retrieves well.

The Five Strategies

Five Ways to Split a Document

From dead-simple to deeply structured. Listed roughly in order of sophistication — but bigger isn't always better.

1 · Fixed-Size

Slice every N tokens, with overlap

BASELINE

Cut the text every N tokens (or characters). Add a small overlap so concepts that span the boundary survive in at least one chunk. Brutally simple, surprisingly hard to beat as a baseline.

def fixed_chunks(text, size=256, overlap=50):
    tokens = tokenize(text)
    step = size - overlap
    return [tokens[i:i+size]
            for i in range(0, len(tokens), step)]
✓ Pros: Deterministic, fast, easy.
✗ Cons: Cuts mid-sentence, ignores structure.

2 · Recursive Character

Try separators in order of preference

DEFAULT

Try splitting on paragraphs first "\n\n". If a piece is still too big, try sentences ". ". Still too big? Try words. The default choice in LangChain and 90% of tutorials, for good reason.

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=150,
    separators=["\n\n", "\n", ". ", " ", ""]
)
chunks = splitter.split_text(text)
✓ Pros: Respects natural boundaries.
✗ Cons: No semantic awareness.

3 · Semantic

Embed sentences, split on topic shifts

ADAPTIVE

Embed each sentence. Walk through them in order. When two adjacent sentences are semantically far apart (large cosine distance), insert a chunk boundary. Chunks naturally align with topic shifts.

def semantic_chunks(sentences, threshold=0.6):
    embs = embed(sentences)
    chunks, cur = [], [sentences[0]]
    for i in range(1, len(sentences)):
        sim = cosine(embs[i-1], embs[i])
        if sim < threshold:
            chunks.append(" ".join(cur)); cur = []
        cur.append(sentences[i])
    return chunks + [" ".join(cur)]
✓ Pros: Topic-aware boundaries.
✗ Cons: Costs embeddings up-front, variable chunk size.

4 · Structure-Aware

Use the format's own boundaries

FORMAT-AWARE

For Markdown: split on ## headers. For code: by function or class. For HTML: by section. For PDF: by page or heading. Headers go into chunk metadata — the model gets context for free.

from langchain.text_splitter import MarkdownHeaderTextSplitter

splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=[
        ("#",   "h1"),
        ("##",  "h2"),
        ("###", "h3")
    ]
)
chunks = splitter.split_text(markdown)
# each chunk has metadata: { "h1": "...", "h2": "..." }
✓ Pros: Boundaries match meaning perfectly.
✗ Cons: Format-specific; some sections too big.

5 · Sentence-Window

Embed sentences alone, retrieve with surrounding context

ADVANCED

A two-step trick: embed at the sentence level (precise matching), but at retrieval time expand each hit with ±N surrounding sentences (full context). You get pinpoint retrieval with rich context — the best of both ends of the spectrum.

# At ingest time
for i, sent in enumerate(sentences):
    index(
        id=i,
        embedding=embed(sent),       # precise
        payload={
            "sentence": sent,
            "window": " ".join(            # context
                sentences[max(0,i-3):i+4]
            )
        }
    )
# At query time
hits = vector_search(embed(query), k=5)

# Use the window for context, not the sentence
context = "\n\n".join(
    h.payload["window"] for h in hits
)
answer = llm.complete(prompt + context)
✓ Pros: Pinpoint relevance, no lost referents.
✗ Cons: More records to manage; windows can overlap heavily.
Strategy 6 · Decouple Index from Context

Parent–Child Chunking

The size of the chunk you retrieve on and the size of the chunk you send to the LLM don't have to match. Parent–child chunking indexes tiny precise children, but returns their full parent for context. Best of both worlds — and the cleanest answer to the size dilemma.

Two-tier structure

PARENT (~2000 tokens) · stored in a doc store, keyed by id
├─ child_1 (~250 tokens) ─► embedded into vector index
├─ child_2 (~250 tokens) ─► embedded into vector index
├─ child_3 (~250 tokens) ─► embedded into vector index
└─ child_4 (~250 tokens) ─► embedded into vector index
 
each child stores: {parent_id, text, embedding}

Vector search runs against the children (precision). The retriever swaps each hit for its parent (context). The LLM never sees the child directly.

Why this beats picking one chunk size

  • Recall: tiny children embed crisp specific facts — no noise dilution.
  • Context: large parents preserve definitions, antecedents, surrounding rules.
  • Citation: "answer from parent doc X, supporting sentence is child Y" — natural traceability.
  • Deduplication: if two children share a parent, return the parent once — fewer redundant tokens to the LLM.
LangChain · ParentDocumentRetriever
from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore
from langchain_text_splitters import RecursiveCharacterTextSplitter

# parent: 2k tokens · child: 250 tokens
parent_splitter = RecursiveCharacterTextSplitter(chunk_size=2000)
child_splitter  = RecursiveCharacterTextSplitter(chunk_size=250)

vectorstore = Chroma(collection_name="children", embedding_function=emb)
docstore    = InMemoryStore()  # or Redis / Postgres

retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,
    docstore=docstore,
    parent_splitter=parent_splitter,
    child_splitter=child_splitter,
)

retriever.add_documents(raw_docs)

# Retrieval: ANN over children, return parents (deduped)
parents = retriever.invoke("What's the refund window for travel?")
# → list of parent docs · LLM-ready context

Three variants in production

  • Parent = whole doc: when docs are small (FAQs, policies). Indexed children, returned whole doc.
  • Parent = section: when docs are long. Section-level parents, paragraph-level children.
  • Parent = summary: children are sentences from the doc, parent is an LLM-generated TL;DR. Most token-efficient.
⚡ Dedup trick: after ANN returns N children, group by parent_id before lookup. Otherwise three children from the same long parent re-fetch the parent three times — wasted bandwidth and duplicate context.
WHEN IT WINS

Long documents (regulations, manuals, papers) where specific terms occur in a small region but the surrounding rule matters for the answer.

WHEN IT LOSES

Short, atomic content (chat logs, FAQs, tweets). The parent and child are nearly the same size — added complexity, no payoff.

STORAGE COST

~8× more chunks indexed than naive 2k-token chunking, but ANN scales sub-linearly. Doc store sits beside the vector index — Redis or Postgres both fine.

⚡ Mental model: sentence-window (Strategy 5) and parent-child are two flavors of the same idea — retrieve on something small, return something larger. Sentence-window keeps the link positional (±N tokens around the hit); parent-child keeps it structural (the doc/section the child came from). Pick by what your data actually has structure on.

The Two Knobs

Tuning Size & Overlap

There's no universal "best" — but there are sensible starting points, and a curve you can ride to better numbers.

Retrieval Quality vs Chunk Size

ILLUSTRATIVE

Composite of recall, precision, and context-utility on a typical RAG eval. Your curve will differ — but the shape is universal.

Sensible starting points

  • Q&A over policies/FAQs: 200-400 tokens, 10-15% overlap
  • Long-form docs & papers: 500-800 tokens, 15-20% overlap
  • Code: chunk by function; overlap helps less
  • Conversation logs: chunk by turn; preserve speaker boundaries
⚡ The chunk-size paradox: Smaller chunks look like they should win — but split a definition from its term and recall collapses. Bigger chunks look safer — but topic-mixing destroys precision. Tune empirically.
⚡ Token vs character: Use a tokenizer that matches your embedding model. Token-counting prevents surprise truncation; character-counting is fine if you're >25% under the model limit.

Pick Your Strategy

The Decision Tree

Four questions. Answer them in order. Land on a strategy.

Question 01

Does your content have structure (headers, sections, functions, slides)?

Markdown docs, code, slide decks, structured PDFs all expose explicit boundaries.

→ YES: Structure-aware chunking — split on those boundaries first.
→ NO: Continue to Q2.
Question 02

Are your queries precise (entity / value lookups) or broad?

"What's the refund window?" is precise. "Tell me about our return process" is broad.

→ PRECISE: Sentence-window — sentence-level embeddings + windowed context at retrieval.
→ BROAD: Continue to Q3.
Question 03

Do topic shifts inside docs matter for retrieval?

Long-form docs that mix multiple unrelated topics. Reports, books, monolithic wikis.

→ YES: Semantic chunking — let topic shifts define boundaries.
→ NO: Continue to Q4.
Question 04

Is "good enough, fast, and simple" what you actually need?

A baseline that beats most production systems still without you noticing.

→ YES: Recursive character with 500 tokens / 15% overlap.
→ STILL TUNING: Fixed + measure, then upgrade.
⚡ The pragmatic ladder: Most teams ship with recursive character, upgrade structured docs to structure-aware, and reach for sentence-window when precision-critical workloads (legal, medical, code) demand it. Semantic chunking pays off on long-form messy corpora — usually a later optimization.
Self-Check

Three Quick Questions

Cement the chunking instincts before moving on.

QUESTION 01

Your chunks are 1,500 tokens each. Retrieval keeps surfacing chunks that contain the answer mixed with three unrelated paragraphs. What's the most likely fix?

Correct. Mixed-topic chunks produce blurred embeddings. Smaller chunks isolate one idea per chunk, sharpening both the embedding and the retrieval signal.
QUESTION 02

You want pinpoint retrieval for short factual queries but the LLM needs surrounding context to answer well. Best fit?

Correct. Sentence-window separates the embedding granularity (small, precise) from the context delivered to the LLM (large, rich). Best of both worlds.
QUESTION 03

Why is overlap a good default in chunking?

Correct. Overlap is the safety net for sentences, definitions, and references that would otherwise be sliced in half. 10-20% is the standard sweet spot.
