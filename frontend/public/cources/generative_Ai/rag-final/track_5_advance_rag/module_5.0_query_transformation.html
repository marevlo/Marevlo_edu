Mission Objectives

What You'll Master

By the end you'll have four query-transformation techniques in your toolkit — and a clear sense of which to reach for, when, and at what cost.

Rewrite

Generate 3-5 paraphrases of the query, retrieve from each, fuse the results. Highest baseline lift.

Decompose

Multi-hop questions into a list of sub-queries. Retrieve each separately. Synthesize the union.

HyDE

Generate a hypothetical answer with an LLM, then embed that instead of the query. Bridges the query-doc style gap.

Step-Back

Ask the broader principle behind the specific question. Retrieve context for the meta-question first.

The Gap

Why User Queries Aren't Search Queries

Users write the way they talk. Your index expects keyword-dense, statement-shaped chunks. Bridging that gap is what query transformation does.

PROBLEM 01

Question / answer style mismatch

Query: "how do I get my money back?"
Indexed doc: "Refunds are issued within 5 business days."
Zero shared keywords. Embeddings help — but only partially.

PROBLEM 02

Multi-hop questions

"Is my flight refund eligible for international tax recovery?" — needs flight refund policy, international tax rules, AND refund eligibility. Single retrieval misses one or two.

PROBLEM 03

Vague / underspecified queries

"refund??" or "didn't work" — no concept signal, no entity, no intent. Embeddings flatten them into noise.

Technique 1 of 4

Multi-Query Rewrite

Ask the LLM for N different phrasings. Retrieve from each in parallel. Fuse the rankings with RRF. Highest baseline lift; usually your default.

REWRITE FAN-OUT
USER
"how do I get my money back?"
1
refund policy and processing
2
return purchase get refund issued
3
payment reversed to original method
4
cancel order receive money returned
FUSE · RRF over 4 retrieval lists
→ ranked union, deduplicated

Rewrite prompt 

<role>
You generate search-query variants for a
retrieval system. You do NOT answer.
</role>

<rules>
1. Output exactly 4 rewrites, one per line.
2. Each must be a different angle on the
   user's intent — keywords, synonyms,
   formal phrasing, related concepts.
3. Drop greetings, filler, typos.
4. Keep each rewrite under 12 words.
</rules>

<examples>
<example>
<in>didnt get my money back</in>
<out>
refund processing time
how long for refund to arrive
refund status delayed
payment reversal timeline
</out>
</example>
</examples>

<query>{{ user_query }}</query>

Production implementation 

import asyncio
from anthropic import AsyncAnthropic

client = AsyncAnthropic()

async def rewrite_queries(user_query: str) -> list[str]:
    resp = await client.messages.create(
        model="claude-haiku-4-5-20251001",    # cheap + fast
        max_tokens=200,
        messages=[{"role": "user",
                   "content": build_rewrite_prompt(user_query)}]
    )
    lines = resp.content[0].text.strip().split("\n")
    return [l.strip().lstrip("0123456789.- ") for l in lines if l.strip()]

async def multi_query_retrieve(user_query: str, top_k: int=50):
    # 1. fan out: original + N rewrites
    rewrites = await rewrite_queries(user_query)
    all_queries = [user_query] + rewrites

    # 2. retrieve from each in parallel
    rankings = await asyncio.gather(*[
        hybrid_retriever.search(q, top_k=top_k) for q in all_queries
    ])

    # 3. fuse via RRF (Module 3.2)
    doc_id_lists = [[h.id for h in r] for r in rankings]
    return rrf(doc_id_lists, k=60, top_k=50)
⚡ When it wins: vague or short queries, conversational language, vocabulary mismatch with the corpus, multilingual users.
⚠ Caveats: +1 LLM call latency. Limit to N=3-5 rewrites — diminishing returns past that. Use a fast model (Haiku, GPT-4o-mini).
Technique 2 of 4

Query Decomposition

When a question requires combining facts from multiple sources, split it into sub-questions, retrieve each independently, then let the LLM synthesize.

DECOMPOSE FLOW
MULTI-HOP QUERY
"Is my international flight refund eligible for tax recovery in the EU?"
A
What's the international flight refund policy?
→ retrieves refund-policy.md
B
EU travel tax recovery rules
→ retrieves eu-tax-policy.md
C
Refund eligibility criteria for tax recovery
→ retrieves eligibility-rules.md
SYNTHESIZE
LLM combines all 3 contexts → final answer

Decomposition prompt 

<role>
You decompose multi-hop questions into
independently-answerable sub-questions.
</role>

<rules>
1. Output 2-4 sub-questions, one per line.
2. If the question is already simple, output
   it unchanged on a single line.
3. Sub-questions should be self-contained
   (no "this", "it", "they").
4. Output ONLY the sub-questions.
</rules>

<examples>
<example>
<in>What's the warranty for my flight purchase?</in>
<out>
Do flight purchases have a warranty policy?
What's the standard warranty coverage?
</out>
</example>

<example>
<in>How long do refunds take?</in>
<out>
How long do refunds take?
</out>
</example>
</examples>

<query>{{ user_query }}</query>
⚡ When it wins: multi-hop reasoning ("X and Y"), comparison queries ("X vs Y"), conditional queries ("if X then Y"), domain queries spanning multiple doc types.
⚠ Caveats: overdoing it on simple queries adds latency for no gain. Add the "unchanged if simple" rule to keep the LLM honest.
Technique 3 of 4

HyDE — Hypothetical Document Embeddings

A clever trick: don't embed the question. Generate a plausible answer, then embed that. The fake answer sits closer to real documents in vector space than the question ever could.

HYDE FLOW
USER QUERY (question-shaped)
"How long do refunds take to arrive?"
LLM-GENERATED HYPOTHETICAL ANSWER
"Refunds are typically issued back to the original payment method within 5-10 business days after the refund request has been approved. Processing time may vary depending on your bank."
EMBED THE FAKE ANSWER · RETRIEVE
vector closer to refund-terms.md than the question would be
⚡ Why this works: documents are statement-shaped. Questions aren't. A statement-shaped hypothetical answer lives in the same neighborhood of vector space as real documents — even if the hypothetical answer has wrong details.

HyDE implementation 

async def hyde_retrieve(user_query: str, top_k: int=50):
    # 1. Generate a hypothetical answer (~80 tokens)
    prompt = build_hyde_prompt(user_query)
    resp = await claude.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=120,
        messages=[{"role": "user", "content": prompt}]
    )
    hypothetical = resp.content[0].text

    # 2. Embed the hypothetical answer (NOT the question)
    vec = await embed([hypothetical])[0]

    # 3. Vector search with that embedding
    return qdrant.search(
        collection_name="rag-chunks",
        query_vector=vec.tolist(),
        limit=top_k
    )

# The prompt that generates the hypothetical:
HYDE_PROMPT = """
Write a short (2-3 sentence) plausible answer
to this question. It does not have to be true.
Use the same vocabulary and style as a
documentation page would.

Question: {query}

Answer:"""

# Optional: hybrid HyDE — embed both, RRF
async def hyde_hybrid(user_query):
    q_vec, h_vec = await asyncio.gather(
        embed([user_query]),
        embed([await generate_hypothetical(user_query)])
    )
    return rrf([
        qdrant.search(query_vector=q_vec[0]),
        qdrant.search(query_vector=h_vec[0])
    ])
⚡ When it wins: highly conversational queries, queries with little vocabulary overlap with the corpus, low-context queries ("refund??").
⚠ Caveats: if the hypothetical is too generic ("answer the question politely"), HyDE hurts. The prompt design matters. Don't let the model refuse to answer "I don't know" — that defeats the purpose.

HyDE Failure Modes & Advanced Patterns

Failure 1 · Anchored Hallucination

Confidently wrong hypothetical

"Refund window is 14 days" — the LLM invents a specific wrong number, embeds that, retrieves docs near "14 days" — and misses the real 30-day policy doc.

Fix: always RRF-fuse with question-only retrieval (the hyde_hybrid pattern above) so the real query still gets a vote.

Failure 2 · Generic Slop

Hypothetical is too vague

"Refunds depend on the situation and may take some time" — the embedding lands in the middle of every help doc, retrieval becomes random.

Fix: prompt the LLM to write "in the style of a help-center article" with concrete-sounding details, not a hedge.

Failure 3 · Refusal Leakage

"I cannot answer this"

For sensitive queries the model refuses. You embed "I'm sorry, I can't help with that" and retrieve safety boilerplate, not the answer.

Fix: detect refusal-shaped responses and fall back to the question embedding.

Advanced · Multi-HyDE

Generate N hypotheticals, average

Generate N different plausible answers (with temperature>0), embed each, mean-pool the vectors. Reduces variance and counter-balances any single hallucination.

vecs = [embed(h) for h in hypotheticals]
query_vec = np.mean(vecs, axis=0)
Advanced · HyDE for the rerank stage only

Use HyDE downstream, not first

Run normal hybrid retrieval first (cheap). Then use the hypothetical answer to rerank the top-50 candidates — closer to its style wins. Costs one extra LLM call, often beats both alternatives.

⚡ The mental model: HyDE addresses one specific failure mode of dense retrieval — question-shape vs document-shape mismatch. It is not a universal upgrade. On corpora where your queries already look like statements (search bars, code, structured commands), HyDE adds latency for no gain. Measure before you ship it.
Technique 4 of 4

Step-Back Prompting

Sometimes the specific question can't be answered without the underlying principle. Step back, ask the meta-question, retrieve the principle, then answer the original.

STEP-BACK FLOW
SPECIFIC QUERY
"Why was my $239 refund rejected last Tuesday?"
STEP-BACK · BROADER PRINCIPLE
"What are the conditions under which refunds get rejected?"
RETRIEVE BOTH
specific query → user's order history
step-back → refund_rejection_rules.md
SYNTHESIZE
LLM applies the principle to the specific case

Step-back prompt 

<role>
You generate a broader, more abstract version
of a user's question — the underlying
principle that, if known, would help answer it.
</role>

<rules>
1. Output ONE step-back question.
2. Remove specific entities, dates, amounts.
3. Keep the same domain and topic.
4. If the question is already general, output
   "NO_STEPBACK_NEEDED".
</rules>

<examples>
<example>
<in>Why was my $239 refund rejected on Aug 5?</in>
<out>What are the conditions for refund rejection?</out>
</example>

<example>
<in>Can I get a refund on order #4827?</in>
<out>What is the refund eligibility policy?</out>
</example>

<example>
<in>What's your refund policy?</in>
<out>NO_STEPBACK_NEEDED</out>
</example>
</examples>

<query>{{ user_query }}</query>
⚡ When it wins: "why" questions, troubleshooting, edge-case queries with specific dates/amounts, situations where the rule matters more than the instance.
⚠ Caveats: overusing it on simple lookups ("what time is the show?") adds latency for no gain. The "NO_STEPBACK_NEEDED" rule is essential.
Structured + Semantic

Self-Querying Retriever

Most user queries hide a structured filter in plain English. "Show me Q3 invoices over $5k from acme" is half semantic, half WHERE clause. Self-querying lets the LLM extract both — so your vector store does what each was built for.

Without Self-Query

Embed the whole thing

User: "papers from 2023 about transformer architectures"

embed("papers from 2023 about transformer architectures")
→ vector search over ALL years
→ returns 2020, 2021, 2024 papers too
⚠ "from 2023" is a filter, not a topic — wasted in the embedding
With Self-Query

Split into query + filter

LLM extracts:

query: "transformer architectures"
filter: {
year: {$eq: 2023}
}
→ vector search constrained by year=2023
→ precision: every result actually from 2023

How it works — three pieces

PIECE 01 · METADATA SCHEMA

You declare the filterable fields up-front — name, type, description, sample values. This is the menu the LLM picks from.

PIECE 02 · STRUCTURED-OUTPUT LLM CALL

The LLM sees the user query + the schema and emits a (semantic_query, filter_expression) JSON. Use structured-output mode — never free-text.

PIECE 03 · COMPILE TO STORE FILTER

A small compiler turns the filter JSON into your vector store's native syntax (Pinecone $eq, Weaviate where, Qdrant Filter, etc.).

LangChain · SelfQueryRetriever
from langchain.chains.query_constructor.base import AttributeInfo
from langchain.retrievers.self_query.base import SelfQueryRetriever

# 1. declare metadata fields the LLM can filter on
field_info = [
  AttributeInfo(name="year",   type="integer", description="publication year"),
  AttributeInfo(name="author", type="string",  description="first author surname"),
  AttributeInfo(name="venue",  type="string",  description="conference or journal"),
]

# 2. wire LLM + vectorstore + schema together
retriever = SelfQueryRetriever.from_llm(
  llm=llm,
  vectorstore=vstore,
  document_contents="abstract of a research paper",
  metadata_field_info=field_info,
)

# 3. one call — LLM extracts query + filter, runs ANN search
docs = retriever.invoke("papers from 2023 by Karpathy on transformers")
# LLM emits: query="transformers" filter=AND(year=2023, author="Karpathy")

When self-query pays off

  • Documents have rich metadata (dates, authors, tags, prices).
  • Queries routinely mix "what" + "which" (topic + filter).
  • Recall is high but precision drowns the LLM in junk.
⚠ Watch out: the LLM call adds ~300ms latency. Always allow it to emit an empty filter for purely semantic queries — never force a filter that isn't there.
⚡ Mental model: self-querying is the "SQL WHERE clause" half of a hybrid retriever — extracted dynamically by an LLM instead of hand-coded by a developer. Pairs naturally with Module 5.1's NL-to-SQL when your data lives across both stores.
Post-Retrieval

Contextual Compression — Trim Before You Stuff

Retrievers return chunks. Chunks contain paragraphs. Paragraphs contain noise. Compression strips each chunk down to the sentences that actually answer the query — cutting tokens, cost, and lost-in-the-middle dilution before generation.

The pipeline

query
  ↓
base retriever (top-k chunks)
  ↓  # 10 chunks · 8k tokens
compressor (per-chunk filter/extract)
  ↓  # 10 fragments · 1.2k tokens
generation LLM

The compressor sees one chunk + the query at a time, and returns either (a) a relevance score, (b) a yes/keep decision, or (c) a query-conditioned extract.

Strategy 1 · LLM extractor

LLMChainExtractor

Cheap LLM rewrites each chunk into "only the sentences relevant to the query." Highest quality, highest cost — one LLM call per chunk.

Strategy 2 · LLM filter

LLMChainFilter

LLM emits a binary "keep / drop" per chunk. Cheaper than extraction, less granular — keeps whole chunks or nothing.

Strategy 3 · Embeddings filter

EmbeddingsFilter

Re-embed each chunk and drop any below a cosine threshold. No LLM call → ~$0 cost. Acts as a hard recall cutoff.

LangChain · ContextualCompressionRetriever
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import (
  LLMChainExtractor, EmbeddingsFilter, DocumentCompressorPipeline
)

# Stage 1: cheap filter — drop irrelevant chunks fast
emb_filter = EmbeddingsFilter(embeddings=emb, similarity_threshold=0.76)

# Stage 2: LLM extracts only the relevant sentences from survivors
extractor = LLMChainExtractor.from_llm(cheap_llm)

# Compose into a two-stage pipeline
compressor = DocumentCompressorPipeline(transformers=[emb_filter, extractor])

compressed_retriever = ContextualCompressionRetriever(
  base_compressor=compressor,
  base_retriever=base_retriever,
)

docs = compressed_retriever.invoke("What's the refund window for travel?")
# → 10 chunks (8k tokens) become 4 fragments (900 tokens)

Why compress at all

  • ✦ Cuts the generation token bill 50–80%
  • ✦ Reduces lost-in-the-middle dilution (Module 4.0)
  • ✦ Speeds up generation — fewer prompt tokens to attend over
  • ✦ Makes citations sharper — the LLM sees only the source sentence

Costs to weigh

  • ✦ Adds latency — one LLM call per chunk for extractor variants
  • ✦ Wrong threshold drops the right chunk → faithfulness drops
  • ✦ Aggressive extraction can strip context the LLM needed
⚡ Cheap-then-strong pattern: chain EmbeddingsFilter (fast, $0) → LLMChainExtractor (slow, paid). The filter throws out 60% of chunks at almost no cost; the extractor refines the survivors. This is the standard two-stage shape — same as retrieve→rerank, just applied to the chunks instead of the candidates.
Choosing Wisely

Decision Matrix — Pick Your Technique

Not every query needs transformation. Cheap fast-path for simple lookups; smarter techniques only when warranted.

Technique Best for Cost (LLM calls) Latency Recall lift Skip when
Rewrite Vague, conversational, vocabulary mismatch 1 ~250ms +10-15% Very short, keyword-shaped queries
Decompose Multi-hop, compound, comparison 1 (+N retrievals) ~300ms +15-25% on multi-hop Simple single-fact lookups
HyDE Question/doc style mismatch, low-vocab queries 1 ~400ms +5-12% Already keyword-dense queries
Step-Back Specific instance, "why" questions, troubleshooting 1 (+ retrieval) ~350ms +8-15% on specific queries Already general questions

Recall lift across query types

Recommended routing strategy

RULE 01

Cheap classifier first. A small model (or keyword heuristics) decides which technique to apply per query — don't run all four on every query.

RULE 02

Layer cautiously. Rewrite + HyDE is fine. Decompose + Step-Back is overkill. Two LLM calls max in your retrieval path.

RULE 03

Measure on your data. Each technique's gain varies wildly by domain. Build a 50-query golden set first; test before shipping.

⚡ Production default: Most teams start with Rewrite alone — it's the cheapest, simplest, biggest lift across general queries. Layer on Decompose for multi-hop or HyDE for highly conversational domains only after measuring impact.

Self-Check

Three Quick Questions

Cement the transformation instincts before moving on to structured RAG.

QUESTION 01

Your users mostly ask short, conversational questions like "refund??" or "didn't work". Recall is poor. Which technique gives the biggest baseline lift?

Correct. Rewrite is the highest-baseline lift for vague/conversational queries. Adding keyword-richer paraphrases fans out into vocabulary space that matches your indexed docs. Decompose helps multi-hop, not vagueness. Step-back helps specific instances, not lack of detail.
QUESTION 02

What does HyDE embed, exactly?

Correct. HyDE = Hypothetical Document Embeddings. The LLM writes a fake answer; you embed that fake answer instead of the question. Because real docs are statement-shaped, the fake answer sits closer to them in vector space than the question would.
QUESTION 03

You see a query: "Why was my $239 refund rejected last Tuesday?" Which technique is the best fit?

Correct. Specific dates and amounts (Aug 5, $239) live in user history, not policy docs. Step-back generalizes to the policy principle ("what conditions cause refund rejection?"), retrieves the relevant rule, and lets the LLM apply it to this specific case. Classic step-back territory.
