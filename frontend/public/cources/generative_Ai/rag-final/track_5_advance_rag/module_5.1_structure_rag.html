Not every question lives in a paragraph. Some answers are exact numbers, filtered rows, or aggregated metrics. This is how RAG talks to your database — turning natural language into SQL, safely.

Two Worlds

Some Answers Don't Live in Text

Vector search returns paragraphs. But "how many users signed up last week?" needs an exact number — computed, not retrieved.

Unstructured

Vector RAG

Returns passages from documents. Great for "how", "why", "explain".

  • Policy & documentation lookups
  • Conceptual / explanatory answers
  • Cannot do exact aggregates
  • Hallucinates numbers if asked
Structured

SQL RAG

Returns computed values from tables. Great for "how many", "what's the total", "list all".

  • Exact counts, sums, averages
  • Filters by date / user / status
  • Joins across tables
  • Deterministic & verifiable

The Secret Sauce

Your Schema Is The Context

The LLM doesn't know your tables. Schema retrieval IS the RAG step — pull only the relevant tables, columns, and example rows.

Anti-pattern

Raw DDL Dump

Pasting 200 tables of raw CREATE TABLE statements. LLM gets lost. Schema bleeds tokens.

-- 200 of these...
CREATE TABLE tbl_cust_v3 (
  c_id INT,
  c_typ VARCHAR,
  c_dt TIMESTAMP
);
-- no comments, cryptic names, irrelevant tables
Best practice

Curated Schema Card

Description + columns with comments + 2-3 sample rows. The LLM gets exactly what it needs.

-- customers: signup & subscription tier
CREATE TABLE customers (
  id INT PRIMARY KEY,
  plan VARCHAR,    -- 'free'|'pro'|'premium'
  signup_date DATE,
  region VARCHAR     -- ISO country code
);
-- sample: (1, 'premium', '2026-04-12', 'US')

Step-by-Step

NL → SQL — Derived End to End

Watch a single English question become a typed query, executed, then narrated back to the user. Five steps, every intermediate artifact shown.

STEP 01 · USER QUESTION
"Which customers spent more than $5,000 in Q1 2026 across our enterprise plan?"

Free-form, mixes a filter (spend > 5000), a time window (Q1 2026), and a categorical filter (plan=enterprise). A vector store can't answer this.

STEP 02 · RETRIEVE RELEVANT TABLES (RAG OVER SCHEMA)

Embed the question. Retrieve top-K table description cards from a vector index of schema docs.

customers
id (int PK)
name (text)
plan (enum: free, pro, enterprise)
invoices
id (int PK)
customer_id (int FK → customers)
amount_usd (numeric)
issued_at (timestamptz)
STEP 03 · BUILD THE PROMPT

Include only the retrieved tables, current date, dialect hint, and few-shot examples that match this question's shape.

You are a PostgreSQL expert. Today is 2026-05-21.
Use ONLY the tables below. Return JSON: {"sql": "...", "rationale": "..."}.

### Schema
CREATE TABLE customers (id INT PRIMARY KEY, name TEXT, plan TEXT);
CREATE TABLE invoices (id INT PRIMARY KEY, customer_id INT REFERENCES customers(id),
                       amount_usd NUMERIC, issued_at TIMESTAMPTZ);

### Example
Q: "Top 10 customers by revenue last month"
SQL: SELECT c.name, SUM(i.amount_usd) AS rev
     FROM invoices i JOIN customers c ON c.id = i.customer_id
     WHERE i.issued_at >= date_trunc('month', now()) - INTERVAL '1 month'
       AND i.issued_at <  date_trunc('month', now())
     GROUP BY c.name ORDER BY rev DESC LIMIT 10;

### Question
Which customers spent more than $5,000 in Q1 2026 across our enterprise plan?
STEP 04 · LLM EMITS TYPED SQL
{
  "sql": "SELECT c.name, SUM(i.amount_usd) AS spend
          FROM invoices i
          JOIN customers c ON c.id = i.customer_id
          WHERE i.issued_at >= '2026-01-01' AND i.issued_at < '2026-04-01'
            AND c.plan = 'enterprise'
          GROUP BY c.name
          HAVING SUM(i.amount_usd) > 5000
          ORDER BY spend DESC;",
  "rationale": "Joined invoices with customers, scoped by date range and plan,
                summed spend per customer, filtered >5000."
}

Hardening between Step 4 and 5: parse the JSON, validate the SQL with sqlglot, reject DDL/DML, EXPLAIN to estimate cost, attach LIMIT 1000 if missing.

STEP 05 · EXECUTE + NARRATE

name           | spend
---------------|---------
Acme Corp      | 18,420
Globex Ltd     | 11,275
Initech Inc    |  7,840
Soylent Co     |  5,260
LLM-FORMATTED ANSWER
Four enterprise customers spent over $5,000 in Q1 2026: Acme Corp ($18.4k), Globex Ltd ($11.3k), Initech ($7.8k), and Soylent Co ($5.3k). Citations: SQL → invoices, customers (executed 2026-05-21).
⚡ The discipline that separates working NL-to-SQL from a science project: always (1) retrieve schema instead of dumping it, (2) require typed JSON output, (3) validate with a parser before executing, (4) enforce read-only DB roles, (5) cap rows. Skip any one and you have a stochastic exploit.
Tables Without a Database

Table QA — Pandas as the Backend

Not every tabular question deserves Postgres. CSVs, Excel sheets, in-memory DataFrames — pandas is the right answer when the data is fluid, exploratory, or attached to a single user session.

Two prompting patterns

Pattern A — code generation: LLM emits Python that operates on df. You execute in a sandbox. Most flexible, hardest to secure.

Pattern B — typed query DSL: LLM emits a constrained JSON (operation, columns, filter, agg). You compile to pandas calls. Slower to build, much safer.

What the LLM needs to see

  • df.head(3) — sample rows reveal real values
  • df.dtypes — distinguish "2026" the string from 2026 the int
  • Column descriptions — "amount in cents" matters
  • Row count + memory footprint — to size the prompt
Code-Gen Pattern · LangChain pandas agent
import pandas as pd
from langchain_experimental.agents import create_pandas_dataframe_agent

df = pd.read_csv("sales_2026.csv")

# IMPORTANT: allow_dangerous_code=True runs LLM-emitted Python.
# Use ONLY behind a sandbox (firejail, gVisor, restricted-python).
agent = create_pandas_dataframe_agent(
  llm, df,
  allow_dangerous_code=True,
  number_of_head_rows=3,
)

answer = agent.invoke(
  "Which region had the largest YoY revenue drop?"
)
# Behind the scenes: LLM writes pandas, executor runs it,
# LLM reads the result, formats the answer.
Typed-DSL Pattern · safer, slower to build
from pydantic import BaseModel
from typing import Literal

class TableQuery(BaseModel):
    op: Literal["groupby_agg", "filter", "topk"]
    groupby: list[str] = []
    agg: dict[str, str] = {}
    where: str | None = None
    limit: int = 10

q = llm.with_structured_output(TableQuery).invoke(question)

# Compile to pandas — no exec(), no eval()
res = df.query(q.where or "index == index") \
        .groupby(q.groupby) \
        .agg(q.agg) \
        .head(q.limit)

When to use Table QA

  • • User-uploaded CSVs / Excel
  • • Notebook-style ad-hoc analytics
  • • Data that lives in S3, not Postgres
  • • Anything <1M rows that fits in RAM

When SQL wins

  • • Multi-table joins, foreign keys
  • • >10M rows, can't fit in memory
  • • Concurrent users, ACID guarantees
  • • Existing analytics warehouse

Security non-negotiables

  • • Never exec() outside a sandbox
  • • Strip import, open, dunder access
  • • Memory + CPU + wall-clock caps
  • • No filesystem / network from the sandbox
Best of Both

The Hybrid Pattern

Real questions often need both: "What does our refund policy say, and what was our actual refund rate last quarter?"

Router Pattern

A classifier picks vector OR SQL based on question type. Cheap, fast, one query path.

End-to-End

Worked Example — SQL + Vector RAG on One Query

A real product question with one structured half and one unstructured half. Watch both retrieval paths fire, then fuse.

USER QUESTION
"Why is our refund rate up this quarter — what changed in the policy AND what do the numbers say?"

Two halves: "what changed in the policy" is unstructured doc retrieval. "what do the numbers say" is SQL over the warehouse. Neither half alone is a complete answer.

PATH A — VECTOR RAG (POLICY DOCS)
1. Embed "what changed in refund policy"
2. ANN search over policy-docs index, top-5
3. Rerank with cross-encoder, top-3
retrieved chunks:
• policy-v3.md#para8: "extended return window to 30 days · 2026-03-15"
• policy-v3.md#para12: "added partial refund for digital goods"
• changelog.md#L42: "policy v3 launch — broader refund eligibility"
PATH B — SQL RAG (WAREHOUSE)
1. NL→SQL on the numeric half of the question
2. LLM emits typed query against refunds, orders
3. Execute against read-replica
SELECT
  date_trunc('quarter', r.issued_at) AS q,
  COUNT(*) * 100.0 / NULLIF(o.cnt, 0) AS refund_rate_pct
FROM refunds r
JOIN (SELECT date_trunc('quarter',ts) q, COUNT(*) cnt
      FROM orders GROUP BY 1) o
  ON o.q = date_trunc('quarter', r.issued_at)
WHERE r.issued_at >= '2025-01-01'
GROUP BY 1, o.cnt ORDER BY 1;
result:
Q4 2025: 2.1% · Q1 2026: 2.4% · Q2 2026: 3.9%
FUSION — SYNTHESIZER PROMPT
You are a senior analyst. Answer the user's question using BOTH:
- <policy_evidence> (qualitative, what changed)
- <sql_evidence>    (quantitative, the numbers)
Cite policy_evidence with [doc_id]. Cite sql_evidence with [sql].

<policy_evidence>
  <doc id="policy-v3.md#para8">Extended return window to 30 days · 2026-03-15</doc>
  <doc id="changelog.md#L42">Policy v3 launch — broader refund eligibility</doc>
</policy_evidence>

<sql_evidence>
  Refund rate by quarter: Q4-2025 2.1% · Q1-2026 2.4% · Q2-2026 3.9%
</sql_evidence>

<query>Why is our refund rate up this quarter — what changed in the policy AND what do the numbers say?</query>
FINAL ANSWER

The refund rate climbed from 2.4% in Q1 2026 to 3.9% in Q2 2026 [sql] — roughly a 60% relative increase. The driver is the v3 policy launch on 2026-03-15, which extended the return window to 30 days [policy-v3.md#para8] and broadened eligibility to digital goods [changelog.md#L42]. The timing of the rate jump aligns with the policy effective date, so the policy change — not a quality regression — is the likely cause.

⚡ What "hybrid structured + unstructured" actually requires: (1) a decomposer or parallel retrieval that fires both paths, (2) cleanly-labeled evidence blocks the LLM can cite separately, (3) a synthesizer prompt that's explicit about which evidence type covers which kind of claim. Skip any of these and the LLM blends them into a single mush of "the data shows" with no traceable citations.
Structured · Nodes & Edges

Graph RAG — Retrieval by Traversal

When the answer requires following relationships — not just finding similar text — chunks fail. A knowledge graph lets you walk from "Acme" to "its subsidiaries" to "their CFOs" in one hop, exact match every time.

Vector RAG fails on

Multi-hop relationship questions

"Who is the CFO of every subsidiary of Acme that's based in Europe?"

  • ✦ Embedding "subsidiary of Acme in Europe" returns paragraphs mentioning those words, not the actual list.
  • ✦ Three independent facts (parent→child, child→region, child→officer) live in three different chunks. Vector retrieval can't join them.
  • ✦ Recall fails the moment one chunk is missing.
Graph RAG nails it

Walk the edges

(Acme) -[:OWNS]-> (sub)
(sub) -[:BASED_IN]-> (Europe)
(sub) -[:HAS_OFFICER {role:'CFO'}]-> (person)

One Cypher query. Exact, complete, explainable. No similarity threshold to tune.

The three Graph-RAG patterns

PATTERN 01 · NL → CYPHER

Same shape as NL-to-SQL — LLM emits a typed Cypher/SPARQL query from the user's question + graph schema.

Used by Neo4j's GraphCypherQAChain, LlamaIndex's KnowledgeGraphQueryEngine.

PATTERN 02 · ENTITY-LINKED EXPANSION

Extract entities from the query, look them up in the graph, expand N hops, return the subgraph as context.

Used by Microsoft GraphRAG's "local search" — best for entity-centric questions.

PATTERN 03 · COMMUNITY SUMMARIES

Pre-summarize tightly-connected clusters offline (Leiden / Louvain). At query time, retrieve the relevant summaries, not raw triples.

Used by GraphRAG's "global search" — best for "themes across the corpus" questions.

NL → Cypher · Neo4j + LangChain
from langchain_neo4j import Neo4jGraph, GraphCypherQAChain

graph = Neo4jGraph(url="bolt://...", username="neo4j", ...)

# refresh_schema pulls node labels, rel types, property keys
graph.refresh_schema()

chain = GraphCypherQAChain.from_llm(
  llm=llm, graph=graph,
  allow_dangerous_requests=True,   # needs read-only DB role
  verbose=True,
)

chain.invoke({"query":
  "Who are the CFOs of every European subsidiary of Acme?"
})

# Behind the scenes:
# 1. LLM reads the schema, emits Cypher
# 2. Cypher runs against Neo4j
# 3. LLM narrates the rows back as prose

Where Graph RAG earns its complexity

  • ✦ Compliance / org-chart Q&A — entity relationships matter most
  • ✦ Drug interactions, supply chains, citation networks
  • ✦ Fraud / KYC — "show me everyone within 2 hops of this account"
  • ✦ Codebases — call graphs, import trees
⚡ Build cost honesty: graphs don't materialize from PDFs. You either ingest from a structured source (CRM, ERP) or you run an LLM-driven entity-extraction pipeline — which is itself a multi-week project. Don't reach for Graph RAG until vector RAG has visibly failed on relational questions.
⚡ The unified picture of Module 5.1: Vector RAG retrieves on semantic similarity. SQL RAG retrieves on typed filters & aggregations. Graph RAG retrieves on relationships. Real systems route between all three — the question-shape decides the backend.
Real Code

Build It In ~50 Lines

A complete text-to-SQL RAG with schema retrieval, safety guard, and verification. No frameworks. Every line carries weight. 

from openai import OpenAI
from qdrant_client import QdrantClient
import sqlite3, re

llm    = OpenAI()
qdrant = QdrantClient("localhost", port=6333)
db     = sqlite3.connect("app.db")

# ---------- 1. SCHEMA RETRIEVAL ----------
def get_schema_context(question, k=3):
    vec = embed(question)
    hits = qdrant.search("schema_cards", vec, limit=k)
    return "\n\n".join(h.payload["card"] for h in hits)

# ---------- 2. SAFETY GUARD ----------
BLOCKED = re.compile(r"\b(DROP|DELETE|UPDATE|INSERT|ALTER|TRUNCATE|GRANT)\b", re.I)

def is_safe(sql):
    if BLOCKED.search(sql): return False
    if not sql.strip().lower().startswith("select"): return False
    return True

# ---------- 3. SQL GENERATION ----------
def generate_sql(question, schema):
    prompt = f"""You write SQLite SELECT queries.
Schema (only these tables exist):
{schema}

Rules:
- Output ONLY raw SQL, no markdown, no commentary
- SELECT only (no writes)
- Use LIMIT 100 unless asked otherwise

Question: {question}
SQL:"""
    return llm.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role":"user", "content":prompt}],
        temperature=0
    ).choices[0].message.content.strip()

# ---------- 4. EXECUTE + VERIFY ----------
def run_sql(sql):
    if not is_safe(sql):
        raise ValueError("Unsafe SQL refused")
    return db.execute(sql).fetchall()

# ---------- 5. ANSWER SYNTHESIS ----------
def ask_sql(question):
    schema = get_schema_context(question)
    sql    = generate_sql(question, schema)
    rows   = run_sql(sql)
    prompt = f"Q: {question}\nSQL: {sql}\nResult: {rows}\nAnswer plainly with citation."
    return llm.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role":"user", "content":prompt}]
    ).choices[0].message.content

print(ask_sql("How many premium customers signed up last month?"))
📦 What this gives you, line by line:
  • • Schema retrieval as RAG (only relevant tables in prompt)
  • • Regex guard + SELECT-only check (defense in depth)
  • • Temperature 0 for deterministic SQL
  • • Explicit LIMIT instruction prevents runaway queries
  • • Result + question + SQL all fed to synthesis step
  • • Read-only DB user adds another safety layer (not shown — use one!)
Failure Modes

Where SQL RAG Breaks

Six ways production text-to-SQL systems fail — and the fix for each.

SQL Injection / Writes

LLM happily generates DELETE or DROP if asked. Or a retrieved doc says "run this query".

✓ Fix: read-only DB role + regex allow-list + parameterized templates where possible

Runaway Queries

Cross-join returning 10B rows. Full-table scan. Database melts.

✓ Fix: query timeout + mandatory LIMIT + EXPLAIN before execute on large tables

Hallucinated Columns

LLM invents user.full_name when actually it's first_name + last_name.

✓ Fix: retry on SQL error with error message + schema in retry prompt

PII / Row Leakage

User asks for "all customer emails" — and the LLM gives them all customer emails.

✓ Fix: row-level security in DB + tenant_id auto-injection in every query

Date / Timezone Bugs

"Last month" — by user's TZ? UTC? Calendar month? Rolling 30 days? LLM picks. User cries.

✓ Fix: inject current date + TZ in system prompt; define "last month" in schema docs

Silent Wrong Numbers

Query runs fine — but joins wrong table. User trusts the number. Bad decision.

✓ Fix: show the SQL to power users + golden-question eval set + LLM-as-judge for correctness
🚨 The structured RAG tax: A wrong number looks right. There's no "I don't know" surface area like in vector RAG. Always show the generated SQL, always have an eval set, always read-only.
