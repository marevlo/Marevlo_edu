The Diagnostic Lens

Two Halves, Two Eval Stacks

A RAG answer can be wrong in two completely different ways. You need two separate metric stacks to tell them apart.

Half 01

Retrieval Quality

"Did we pull the right chunks?" If the right document never enters the prompt, generation cannot recover. Garbage in, hallucination out.

  • Hit Rate — was the gold chunk in top-k?
  • MRR — how high was it ranked?
  • NDCG — graded relevance ranking
  • Recall@k — how many golds did we hit?
Half 02

Generation Quality

"Did the LLM use the chunks correctly?" Even with perfect retrieval, the LLM can ignore context, fabricate, or answer the wrong question.

  • Faithfulness — every claim grounded?
  • Answer Relevance — addresses the Q?
  • Context Precision — signal in chunks?
  • Context Recall — covered the answer?

The Diagnostic Decision Tree

Retrieval Score Generation Score Diagnosis Where To Look
Low High Retrieval is broken chunking, embeddings, hybrid search
High Low LLM ignores context prompt, model size, context window
Low Low System is broken; fix retrieval first start with chunks & index
High High System works — invest in coverage edge cases, harder questions

Half 02

The Generation Metric Stack

Four reference-free metrics that pin down where generation goes wrong — from hallucinations to half-answers to noisy context.

METRIC · GEN-01

Faithfulness

Every claim in the answer is entailed by the retrieved context. Low score = hallucination.

claims_grounded / total_claims
METRIC · GEN-02

Answer Relevance

Reverse-generate Qs from the answer; embed against original Q. Off-topic answers score low.

mean(cos(Q_orig, Q_gen_i))
METRIC · GEN-03

Context Precision

Of the chunks retrieved, what fraction were actually relevant? Penalizes noisy retrieval.

relevant_chunks / k
METRIC · GEN-04

Context Recall

Of the claims in the ground-truth answer, what fraction can be derived from the retrieved chunks?

covered_claims / gold_claims

The Framework

RAGAS Deep Dive

RAGAS bundles the four core RAG metrics into one library — and uses an LLM as the engine. Here's how it works under the hood.

01
PILLAR

Faithfulness

LLM extracts claims from answer, NLI-checks each against context.

02
PILLAR

Answer Relevance

LLM generates Qs from answer, embedding-similarity vs original Q.

03
PILLAR

Context Precision

LLM marks each chunk relevant/not — weighted by rank position.

04
PILLAR

Context Recall

LLM decomposes gold answer into claims, checks each is in context.

Each Metric — Formula, Procedure, Worked Example

01

Faithfulness

"Does every claim in the answer trace back to the context?"

FORMULA
Faithfulness = |claims supported by context| |total claims in answer|
range [0,1] · 1 = every claim is grounded

Procedure:

  1. 1. LLM extracts atomic claims from the answer ("Refunds are 30 days.", "Process by email.")
  2. 2. For each claim, LLM judges support given the context (yes/no via NLI prompt)
  3. 3. Score = supported / total
WORKED EXAMPLE
Answer: "Refunds are 30 days. Process by email. Available in all 50 states."
Context says: 30-day refunds, email process. (No mention of states.)
claim 1 "30 days" → supported
claim 2 "email" → supported
claim 3 "50 states" → NOT supported
Faithfulness = 2/3 ≈ 0.67
02

Answer Relevance

"Does the answer actually address the question?"

FORMULA
AR = 1 N  ∑i cosine(emb(q), emb(qi′))
qi′ = LLM-generated question from the answer · N typically 3

Procedure (reverse engineer the question):

  1. 1. LLM reads the answer, generates N questions this answer would answer
  2. 2. Embed original q and each generated qi′
  3. 3. Average cosine similarity

No ground truth needed — measures the answer's directional fit to the user's question.

WORKED EXAMPLE
Original Q: "What's our refund window?"
Answer: "Refunds are 30 days from purchase."
LLM reverse-engineers:
q₁' "How long is the refund period?" → cos 0.94
q₂' "When can I get a refund?"      → cos 0.89
q₃' "How many days for returns?"    → cos 0.86
AR = (0.94+0.89+0.86)/3 = 0.897
03

Context Precision

"Of what we retrieved, how much is actually relevant — and is it on top?"

FORMULA (RANK-WEIGHTED · AVERAGE PRECISION-LIKE)
CP@K = ∑k=1..K Precision(k) · vk |relevant items in top-K|
vk = 1 if chunk at rank k is relevant else 0 · LLM-judged

Why rank-weighted: relevant chunk at position 1 is worth more than position 5. Rewards rankers that put the gold on top — punishes ones that bury it.

WORKED EXAMPLE · K=4
rank 1: chunk_42 v=1 · P@1 = 1/1 = 1.00
rank 2: chunk_07 v=0 · skip
rank 3: chunk_88 v=1 · P@3 = 2/3 = 0.67
rank 4: chunk_19 v=0 · skip
|relevant| = 2
CP = (1.00·1 + 0.67·1) / 2 = 0.835
04

Context Recall

"Did we retrieve enough to fully answer? Or did we miss something?"

FORMULA
CR = |gold-answer claims attributable to context| |total claims in gold answer|
range [0,1] · 1 = every fact the gold answer needs is in the retrieved chunks

Procedure:

  1. 1. LLM decomposes the ground-truth answer into atomic claims
  2. 2. For each gold claim, LLM judges whether it's attributable to ≥1 retrieved chunk
  3. 3. Score = attributable / total

This is the only RAGAS metric that requires ground_truth — measures the retriever's ceiling.

WORKED EXAMPLE
Gold answer: "Refunds in 30 days. Process by email. Free over $50."
claim 1 "30 days" → in chunk_42 → attributable
claim 2 "email" → in chunk_88 → attributable
claim 3 "free over $50" → not retrieved → missing
CR = 2/3 ≈ 0.67  # retriever missed the shipping rule
⚡ How the four interact: in the example above, Context Recall drops to 0.67 because the retriever missed the shipping rule. Faithfulness can still hit 0.67 if the LLM hallucinates the missing claim, or 1.00 if it appropriately refuses to mention it. Answer Relevance reports whether the user got something on-topic. Context Precision reports retrieval quality even when the gold-answer ceiling (recall) is low. Each measures one independent dimension — that's why all four together are stronger than any single number.

RAGAS In Practice — ~20 Lines

Production-grade 
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    faithfulness, answer_relevancy,
    context_precision, context_recall
)

# One row per question in your eval set
data = {
    "question":      ["What's our refund policy?", ...],
    "answer":        ["Refunds within 30 days...", ...],
    "contexts":      [["chunk_42 text", "chunk_88 text"], ...],
    "ground_truth":  ["Refunds: 30 days. Email support...", ...]
}
ds = Dataset.from_dict(data)

result = evaluate(ds, metrics=[
    faithfulness,        # generation
    answer_relevancy,    # generation
    context_precision,   # retrieval
    context_recall,      # retrieval (needs ground_truth)
])

print(result)
# → {'faithfulness': 0.87, 'answer_relevancy': 0.91,
#    'context_precision': 0.74, 'context_recall': 0.82}
⚡ The shape of a RAGAS row: Every eval example needs four columns — question, answer, contexts (list of retrieved chunks), and optionally ground_truth (for context_recall). That's the universal RAG eval row. Even outside RAGAS, build your eval set in this shape.
Hidden Failure

The LLM-as-Judge Bias Tax

RAGAS, G-Eval, every modern eval framework leans on an LLM judge. The judge has six well-documented biases. Know them — or your scores lie.

Position Bias

Judges prefer answer A over B simply because A came first. ~60/40 split when both answers are identical.

✓ Fix: run both orders, average. Or use random shuffle per trial.

Length Bias

Longer answers score higher even when terse ones are clearly correct. Verbosity = quality, in the judge's eyes.

✓ Fix: add "concise is good" to the rubric. Normalize for length.

Self-Enhancement

An LLM judging its own outputs (or its family's) scores them higher than equivalent outputs from other models.

✓ Fix: use a different model family as judge. GPT eval ↔ Claude judge.

Verbosity / Tone

Confident, well-formatted, polite answers win — even if they're wrong. Style trumps substance.

✓ Fix: rubric must call out "ignore style, evaluate correctness only."

Compassion Bias

Judges avoid harsh scores. Distributions skew toward 4/5, hiding real differences between systems.

✓ Fix: force binary or pairwise. Provide reference examples per score level.

Format Bias

Markdown lists, code blocks, and structured outputs score higher than plain prose — independent of content.

✓ Fix: normalize formatting before judging, or evaluate semantic content explicitly.

The Mitigation Playbook

TACTIC 01 · CALIBRATE

Score a 50-example human-labeled set with your judge. Measure correlation. If < 0.7, fix the rubric before trusting any production number.

TACTIC 02 · ENSEMBLE

Use 2-3 different judges (different families). Take majority vote or average. Single-judge results are always suspect.

TACTIC 03 · PAIRWISE

Instead of "rate 1-5", ask "is A better than B?" Pairwise is far more reliable than absolute scoring — and dodges compassion bias.

TACTIC 04 · CoT + EVIDENCE

Force the judge to cite spans before scoring. "List which sentences support claim X." This grounds the judgment and exposes weak ones.

🚨 The honest truth: LLM-as-judge is the best automation we have, and it's still systematically biased. A score of 0.87 isn't an absolute number — it's a signal. Use it for relative comparison (A vs B), trend monitoring (last week vs this week), and CI regression — not as a marketing claim.
The Cold Start

Synthetic Test Set Generation

You can't evaluate without questions. Hand-writing 200 takes a week. Here's how to bootstrap a high-quality eval set from your docs in an afternoon.

STEP 01

Sample Chunks

Pick 100-200 chunks stratified by topic, document, length.

STEP 02

Generate Q&A

LLM produces (question, answer) pair from each chunk.

STEP 03

Evolve Difficulty

Rewrite Qs to be harder: multi-hop, conditional, reasoning.

STEP 04

Filter & Audit

Human reviews 10% sample. Drop bad Qs. Keep ~150-200.

Question Evolutions — Get The Mix Right

SIMPLE · ~40%

Direct Lookup

Single-fact, answerable from one chunk.

"What's the refund window?"
REASONING · ~25%

Inference / Compute

Needs combining facts or simple reasoning.

"If I bought on day 25, can I refund?"
MULTI-HOP · ~25%

Multiple Chunks

Answer spans 2+ documents. Real RAG stress test.

"How does refund interact with shipping?"
ADVERSARIAL · ~10%

Out-of-Scope / Unanswerable

Tests if the system correctly refuses.

"What's the CEO's home address?"

Bootstrap An Eval Set — ~30 Lines

from openai import OpenAI
import random, json

llm = OpenAI()

GEN_PROMPT = """Given this passage, produce ONE realistic user question
that can be answered ONLY from this passage. Also produce the ground-truth answer.
Return JSON: {{"question": ..., "answer": ...}}.

Passage:
{chunk}"""

EVOLVE_PROMPT = """Rewrite this question to require multi-hop reasoning
across two facts. Keep the original answer correct.

Original: {q}
Answer it still needs to lead to: {a}"""

def generate_pair(chunk):
    r = llm.chat.completions.create(
        model="gpt-4o-mini", temperature=0.3,
        messages=[{"role":"user", "content":GEN_PROMPT.format(chunk=chunk)}],
        response_format={"type": "json_object"}
    )
    return json.loads(r.choices[0].message.content)

def build_eval_set(chunks, n=200):
    sampled = random.sample(chunks, n)
    eval_set = []
    for c in sampled:
        pair = generate_pair(c)
        pair["source_chunk_id"] = c["id"]
        eval_set.append(pair)
    return eval_set

# Always: human-audit a 10% sample before trusting this set
eval_set = build_eval_set(corpus_chunks)
🎯 Non-negotiable: Human audit ~10% of generated questions. LLM-generated eval sets have systematic blind spots — they tend to be too easy, too lookupy, too aligned with your retrieval model. A quick audit catches the worst patterns and lets you tweak the prompts.
