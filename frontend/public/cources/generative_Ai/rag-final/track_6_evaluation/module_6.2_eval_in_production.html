Two Axes

Online vs Offline Evaluation

Two complementary modes. Offline answers "should we ship this change?" Online answers "is it still working in the wild?" You need both.

Axis 01

Offline Eval

Static eval set. Runs in CI, on commits, in notebooks. Deterministic, repeatable, fast. Answers: "Did this PR break anything?"

  • Runs on a fixed test set (200 questions)
  • Reproducible — same input, same score
  • Gates merges & deployments
  • Catches regressions before users see them
Tools
RAGAS, DeepEval, pytest, GitHub Actions
Axis 02

Online Eval

Live production traffic. Runs continuously on real user queries. Noisy, costly, irreplaceable. Answers: "Is it still good for real users right now?"

  • Scores a sample of real production calls
  • Detects drift & novel failure modes
  • Feeds back into next offline eval set
  • Powers user feedback loops (👍 / 👎)
Tools
LangSmith, Langfuse, Arize Phoenix, OpenTelemetry

When To Reach For Which

Question Mode Why
Did this PR regress quality? Offline (CI) Need deterministic, fast pass/fail
Is faithfulness dropping this week? Online Needs real traffic over time
Which of 3 prompts wins? Offline (A/B) Fixed eval set, head-to-head
Are users hitting new question types? Online Production-only signal
Should this canary go to 100%? Online Real-time comparison vs control
⚡ The mental model: Offline catches the regressions you can predict. Online catches the ones you can't. A team running only offline ships fast but breaks silently. A team running only online flies blind into every deploy. You need both. They feed each other.
Diagnostic Eval

Score Every Stage, Not Just The Final Answer

A single end-to-end score tells you something is broken. Per-stage scores tell you where. Instrument every step.

End-to-End Only

Score: 0.62 — what does that mean?

  • Don't know if retrieval or generation failed
  • Improvements compound or cancel — can't tell
  • Debugging requires re-running everything

Per-Stage Scoring

Retrieval: 0.94 · Rerank: 0.88 · Gen: 0.45

  • Generation is the bottleneck — fix the prompt
  • Retrieval is fine — don't waste cycles on it
  • Score per stage is a debugging shortcut

Every prompt change, every chunking tweak, every model swap goes through the gate. If quality drops below the threshold, the PR cannot merge.

A Working GitHub Actions Workflow

Copy-pasteable 
# .github/workflows/rag-eval.yml
name: RAG Eval Gate

on:
  pull_request:
    paths:
      - "src/rag/**"
      - "prompts/**"
      - "configs/**"

jobs:
  eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install deps
        run: pip install -r requirements.txt ragas pytest

      - name: Run RAG eval
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: pytest tests/eval/ --eval-set=eval/v3.jsonl --json=results.json

      - name: Check thresholds vs baseline
        run: python scripts/check_thresholds.py results.json baseline.json
        # exits non-zero if any metric drops > tolerance

      - name: Comment results on PR
        if: always()
        uses: actions/github-script@v7
        with:
          script: | fs.readFileSync('results.json') → comment table
📦 What this gives you:
  • • Triggered only on changes that could affect quality (path filter)
  • • Eval set version-pinned (eval/v3.jsonl)
  • • Pass/fail driven by check_thresholds.py against baseline
  • • Results posted as a PR comment — reviewers see numbers, not links
  • • Bonus: cache embeddings between runs to keep CI under a few minutes

How To Set Thresholds — Four Patterns

PATTERN 01

Absolute Floor

Faithfulness must be ≥ 0.85. Simple. Brittle.

score >= 0.85
PATTERN 02

Relative Drop

Allow a small regression — typical for noisy metrics.

score >= baseline - 0.02
PATTERN 03

Percent Tolerance

Better for metrics with wide ranges.

score >= baseline * 0.97
PATTERN 04

Stat-Sig Drop

Bootstrap CI — block only if drop is statistically real.

CI_lower >= baseline
Live Signal

Production Monitoring Dashboards

What lives on your wall-mounted RAG dashboard. Six signals you watch every day — and the alerts wired to them.

SIGNAL · 01

Quality (sampled)

Run RAGAS on a 5% sample of production calls. Daily rolling average. Most important number.

Alert: faithfulness < 0.85 for 24h
SIGNAL · 02

Latency (P50/P95/P99)

End-to-end + per-stage. P99 reveals tail problems users actually feel.

Alert: P95 > 2000ms for 5min
SIGNAL · 03

Cost / Query

Tokens-in + tokens-out + embedding calls. Spikes mean retries or runaway agents.

Alert: cost/query > 2× baseline
SIGNAL · 04

Error Rate

Tool errors, timeouts, refusals. Spike = something upstream broke.

Alert: error_rate > 1% for 10min
SIGNAL · 05

User Feedback

👍 / 👎 ratio per day. Single best ground-truth signal.

Alert: 👎 ratio > 15%
SIGNAL · 06

Refusal Rate

% of calls where the LLM refused to answer. Spike = retrieval blew up.

Alert: refusal_rate doubled vs week-1

🎯 The rule: If you can't reconstruct exactly what happened — which chunks, which prompt version, which model — you can't debug. Log the inputs, the intermediates, the outputs, and the timings. Disk is cheap; mystery production bugs are not.
The Silent Killer

Drift Detection

Your RAG didn't change. Users did. Or your docs did. Or the embedding API silently updated. Drift is the most common production failure — and the hardest to spot without instruments.

Faithfulness — Last 90 Days

Rolling 7-day average, sampled production traffic

Drift detected
🚨 What you're seeing: Faithfulness sat at 0.91 for 60 days. Then a quiet decline starting day 65. No deploy, no config change. Cause: new "billing-2026" question cluster hit production after a product launch — the RAG's index never indexed those docs. Drift caught it. Nothing else would have.

Query Drift

User questions shift to a new topic, intent, or phrasing your retrieval wasn't tuned for.

✓ Detect: cluster embeddings of recent Qs vs baseline. New cluster = drift.

Corpus Drift

Docs added/updated/removed. Your answers reference deleted info or miss new info entirely.

✓ Detect: monitor doc count, last-indexed timestamps, missing-citation rate

Model Drift

Provider silently updates the model (or embedding). Behavior shifts under you.

✓ Detect: pin model versions, run weekly canary eval, watch for sudden score deltas

User-Mix Drift

A new customer segment joins (enterprise vs SMB). Their question distribution differs from your eval set.

✓ Detect: track per-segment quality. Periodically refresh eval set from real traffic.

The Living Eval-Set Loop

STEP 01

Sample Live Traffic

Weekly random sample of production queries.

STEP 02

Find Low-Score

Filter to calls with faithfulness < 0.7 or thumbs-down.

STEP 03

Human Label

PM/SME labels gold answers for the worst 50.

STEP 04

Merge → Eval Set

Add to CI eval set. Version-bump. Future PRs catch the regression.

🌱 The flywheel: Online finds new failures → label them → offline catches them next time. After 6 months, your eval set looks like your actual users. That's when production eval stops being a chore and becomes your edge.
