Cross-Case Synthesis

Patterns That Repeat

Across all 10 cases, six patterns kept appearing. Memorize these — they're the predictive signals of trouble brewing.

01

Silent Embedding Swaps

A model provider quietly updates an embedding endpoint. Old vectors and new query vectors no longer share the same space. Recall collapses overnight.

→ DEFENSE: pin model versions, re-embed on upgrade
02

Chunk Boundary Loss

Important context — a definition, a number, a name — gets split across two chunks. Neither chunk alone is retrievable for the right query.

→ DEFENSE: overlap windows, semantic chunking
03

Stale Index Lag

Source-of-truth docs update daily. The vector index updates weekly. Users get last week's policy for this week's question.

→ DEFENSE: incremental re-indexing, doc TTLs
04

Eval / Prod Mismatch

Golden eval set covers 80% of real query distribution. The 20% it misses is exactly where production fails — and you never noticed.

→ DEFENSE: sample real prod queries weekly
05

Prompt Brittleness

A tiny wording change in the system prompt — added "be helpful" — and refusal rate drops from 12% to 0.4%, including legitimate refusals on out-of-scope queries.

→ DEFENSE: prompt regression tests, A/B every change
06

Citation Theater

Model cites a real document but invents a sentence that isn't in it. Users trust the citation; the hallucination ships.

→ DEFENSE: post-gen verifier, span-grounded citations
Run Your Own

The Blameless Postmortem Playbook

When (not if) you have a production RAG incident, follow this template. The goal isn't blame — it's a systemic fix that makes the same failure impossible.

postmortem_INCIDENT-2026-0517.md 
# Incident INC-2026-0517 — Refund queries returning wrong policy

## Summary (1–2 sentences)
Between 14:00–17:30 UTC, ~12% of customer support queries about
refunds returned policy text from a deprecated 2024 KB article.
2,300 users affected, 47 escalated tickets filed.

## Impact
- Users affected:    2,300
- Tickets escalated: 47
- Revenue at risk:   ~$8,400 in mishandled refund disputes
- SLA breach:        yes (Tier-1 latency)

## Timeline (UTC)
- 13:42  Deploy v3.7.2 to prod (chunker upgrade)
- 14:08  First user complaint via support
- 14:55  On-call paged by faithfulness alert (RAGAS < 0.85)
- 15:20  War room opened
- 16:10  Root cause identified: deprecated docs not filtered
- 16:25  Rollback initiated
- 17:30  Faithfulness recovered to 0.94 — incident closed

## Root Cause (use 5 Whys)
1. WHY did users see old policy? Index returned 2024 KB chunks.
2. WHY were 2024 chunks in the index? They were never deleted.
3. WHY weren't they deleted? Ingestion script doesn't tombstone.
4. WHY no tombstoning? Original design assumed "ADD ONLY" sources.
5. WHY assumption wrong? Confluence pages can be deleted in source.

## What went well
- Alerting fired within 47 min of deploy
- Rollback procedure executed cleanly
- Customer support team had pre-drafted apology template

## What went wrong
- No CI check for stale-doc retention
- The chunker upgrade was bundled with an unrelated change
- War room delayed because owner was in 1:1 (no backup paged)

## Action Items (Owner · Due)
- [P0] Add tombstone column to chunks table  · @alex · 2026-05-21
- [P0] CI test: deleted source ⇒ chunk gone   · @priya · 2026-05-22
- [P1] Split deploys: one logical change each · @team  · ongoing
- [P1] Add secondary on-call for every deploy · @leo   · 2026-05-25
- [P2] Run game day simulating stale-doc bug  · @team  · 2026-06-01

## Lessons
Vector indexes need DELETE semantics, not just INSERT. Treat them
like databases, not append-only logs.

Blameless

Talk about systems that allowed the bug, never the person who shipped it. Otherwise people hide incidents.

Five Whys

Keep asking "why" until you hit a process or design root cause — not a human one.

Owned Actions

Every action item has a single owner, a P0/P1/P2 priority, and a real due date — or it won't happen.

Shared Widely

Publish to the whole org. Lessons stay locked away when postmortems are private.

📚 The compounding effect. Teams that publish postmortems consistently develop a shared mental model of failure modes. Within 12 months, your engineers will start preventing the failures other teams just had — because they read the case file last quarter.
Knowledge Check

Lock It In

Three checkpoint questions. Answer all three to unlock the module checkpoint.

QUESTION 01 · DIAGNOSIS

Users report that a familiar query started returning unrelated docs last Tuesday. Nothing was deployed. What's the most likely root cause class?

Correct: B. No deploy + sudden retrieval regression is the classic embedding-drift signature. Old indexed vectors and new query vectors no longer share semantic space. Defense: pin model versions and re-embed on any provider upgrade.
QUESTION 02 · POSTMORTEM CULTURE

A postmortem's "Root Cause" section concludes with: "Engineer Sarah forgot to add the tombstone column." Why is this a poor outcome?

Correct: C. Blameless postmortems ask "why was it possible to forget?" and produce action items like "add CI test that rejects PRs without tombstone column." Blamed engineers hide future incidents — exactly the opposite of what you want.
QUESTION 03 · PATTERNS

Your golden eval set scores 0.93 faithfulness. Production support tickets keep escalating. What pattern from this module explains the gap?

Correct: A. The other patterns are possible, but they'd typically show up in evals too. When evals are green but prod is red, your golden set isn't representative. Defense: sample real production queries weekly and feed them back into the eval pool.
