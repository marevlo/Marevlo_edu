The Wrong Tool

Why Postgres Alone Can't Do This

"Just store the vectors in a column" — every senior engineer's first instinct. It works for 1,000 vectors. At 1 million, your search takes 12 seconds. Here's why classical databases hit a wall.

Brute-Force Scan

naive cosine on a Postgres column

You can compute cosine(query, vec) for every row, then ORDER BY and LIMIT 5. That's a full table scan: O(N) work per query. 

SELECT id, text,
       cosine_sim(embedding, $query_vec) AS score
FROM docs
ORDER BY score DESC
LIMIT 5;
-- 1K rows: 12ms ✓
-- 100K rows: 1.2s ⚠
-- 1M rows: 12s ✗ (user has gone home)
-- 100M rows: 20 minutes ✗ (you're fired)

Why B-Tree Indexes Don't Help

the structural mismatch

Classical SQL indexes (B-tree, hash) accelerate equality and range queries on 1-dimensional values. They sort by ordering.

  • ✗ A 1536-D vector has no natural ordering — you can't put it on a number line.
  • ✗ Similarity is a geometric property, not a comparable scalar.
  • ✗ "Curse of dimensionality": in high-D space, almost everything is roughly equidistant.

The data structure we need is fundamentally different — a graph or tree built around proximity, not value ordering.

The Scaling Cliff

↑ Brute force scales linearly with N. HNSW scales logarithmically — the gap explodes past 100K vectors.

💡 But what about pgvector? The pgvector extension adds proper ANN indexes (HNSW and IVF) to Postgres. So Postgres can do it — just not natively. Whether to use pgvector or a dedicated vector DB is a real decision we'll resolve in the Provider Comparison section below. The point stands: no ANN index = no production RAG at scale.
The Algorithm

HNSW — Hierarchical Navigable Small World

HNSW is the algorithm under the hood of Qdrant, Pinecone, Weaviate, pgvector, and almost every vector DB you'll touch. Get the mental model now — the rest of the course assumes it.

The Highway-Then-Streets Analogy

You're in Berlin and need to find a specific café in Amsterdam. You don't search every street in the world. You:

  1. Top layer (highways): Jump on the autobahn — fast, sparse, gets you near the right city.
  2. Middle layer (regional roads): Exit the highway, take regional roads to the right neighborhood.
  3. Bottom layer (streets): Walk street-by-street to find the exact café.

HNSW does exactly this with vectors. Each vector lives on layer 0; a probabilistic subset also appears on higher layers as "highway nodes." Queries start at the top, greedily descend toward the target. Log-time. ANN-magic.

The Three HNSW Knobs You'll Tune

M

Connections per node

Number of bidirectional edges each node maintains. Higher = better recall, more memory.

Typical: M = 16
ef_construction

Build quality

How thoroughly the graph is built. Higher = slower indexing but higher recall.

Typical: 200
ef_search

Search quality

How many candidates to explore at query time. Higher = better recall, slower queries. Tunable per query.

Typical: 128
⚡ Engineering reality: The defaults baked into Qdrant/Pinecone are excellent for ~95% of cases. You only tune these when you hit recall problems in Track 6 evaluation — and even then, the chunking strategy usually matters more than HNSW params.

The Fundamental Tradeoff

↑ Recall = "what fraction of true nearest neighbors did we find?" Latency is per-query. The curve is the menu.

Compression

Quantization: 32× Less RAM

A 1536-D float32 vector = 6 KB. At 100M vectors, that's 600 GB of RAM — and HNSW wants vectors in memory. Quantization trades a sliver of accuracy for massive compression. In 2026, scalar quantization is essentially a free win.

Scalar Quantization

SAFE DEFAULT · 4× smaller

Each float32 (4 bytes) → int8 (1 byte). Numbers are mapped to 256 buckets. Trivial accuracy loss; massive RAM savings.

float32 · 6 KB
int8 · 1.5 KB
Recall drop: ~0.5%

Product Quantization (PQ)

AGGRESSIVE · 16-32× smaller

Splits the vector into M sub-vectors, encodes each via a small codebook (k-means clusters). Brutal compression — but reconstructions are lossy.

float32 · 6 KB
PQ · 200 B
Recall drop: ~2-5%

Binary Quantization

EXTREME · 32× smaller

Each dimension → 1 bit (positive / negative). Hamming distance replaces cosine. Insanely fast (CPU XOR + popcount). Used as a first-pass filter before exact rescoring.

float32 · 6 KB
bin · 192 B
Recall drop: ~5-10% alone, ~1% with rescoring

Quantization Decision Tree

Your Situation Use Expected Savings
Up to ~1M vectors no quantization Not worth the complexity
1M – 50M vectors Scalar (int8) 4× RAM cut, ~0.5% recall loss
50M – 500M vectors Product Quantization 16-32× RAM cut, 2-5% recall loss
500M+ vectors / cost-extreme Binary + rescoring 32× compression, <1% loss with rerank
🎯 Modern engines (Qdrant especially) make this a one-line config: 
VectorParams(
    size=1536,
    distance=Distance.COSINE,
    quantization_config=ScalarQuantization(  # ← that's it
        scalar=ScalarQuantizationConfig(type="int8")
    ),
)

Hands-On

Quickstart — Up & Running in 5 Minutes

Two paths. Chroma for the easiest possible start (pip install, no Docker, in-process). Qdrant for production-style local dev with Docker. Same conceptual flow: create collection → upsert points → query. 

# pip install chromadb openai
# No Docker. No server. Pure Python.
import chromadb
from openai import OpenAI

openai = OpenAI()
client = chromadb.PersistentClient(path="./chroma_db")

# 1. Create / get a collection
col = client.get_or_create_collection(
    name="docs",
    metadata={"hnsw:space": "cosine"},
)

# 2. Embed + upsert
def embed(text):
    return openai.embeddings.create(
        model="text-embedding-3-small", input=text
    ).data[0].embedding

docs = [
    "EU customers may cancel within 30 days for a full refund.",
    "Annual subscriptions are non-refundable after day 30.",
    "Refunds are issued within 10 business days.",
]
col.upsert(
    ids=["d1", "d2", "d3"],
    embeddings=[embed(d) for d in docs],
    documents=docs,
    metadatas=[{"region": "EU"}, {"region": "GLOBAL"}, {"region": "GLOBAL"}],
)

# 3. Query with optional metadata filter
results = col.query(
    query_embeddings=[embed("What is the EU refund policy?")],
    n_results=2,
    where={"region": "EU"},
)

print(results["documents"])
# → [["EU customers may cancel within 30 days..."]]
WHY CHROMA
  • ▸ Zero infrastructure — pip install and go
  • ▸ Persists to a local folder, restores on restart
  • ▸ Perfect for notebooks, prototypes, demos
  • ▸ Cleanest API of any vector DB
When it breaks: Above a few million vectors, or when you need rich filtering, multi-tenancy, or replication, you graduate to Qdrant. 
# 1. Start Qdrant locally (in another terminal)
# $ docker run -p 6333:6333 qdrant/qdrant

# 2. pip install qdrant-client openai
from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, PointStruct, Filter, FieldCondition, MatchValue,
)
from openai import OpenAI

openai = OpenAI()
qdrant = QdrantClient("localhost", port=6333)

# 1. Create collection (run once)
qdrant.recreate_collection(
    collection_name="docs",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE),
)

# 2. Embed + upsert
def embed(text):
    return openai.embeddings.create(
        model="text-embedding-3-small", input=text
    ).data[0].embedding

docs = [
    (1, "EU customers may cancel within 30 days for a full refund.", "EU"),
    (2, "Annual subscriptions are non-refundable after day 30.", "GLOBAL"),
    (3, "Refunds are issued within 10 business days.", "GLOBAL"),
]
qdrant.upsert(
    collection_name="docs",
    points=[
        PointStruct(id=i, vector=embed(text), payload={"text": text, "region": region})
        for i, text, region in docs
    ],
)

# 3. Query with filter
hits = qdrant.search(
    collection_name="docs",
    query_vector=embed("What is the EU refund policy?"),
    limit=2,
    query_filter=Filter(must=[
        FieldCondition(key="region", match=MatchValue(value="EU"))
    ]),
)
for h in hits:
    print(h.score, h.payload["text"])
WHY QDRANT
  • ▸ Rust = blazing fast, low memory footprint
  • ▸ Powerful payload filters (numeric ranges, geo, full text)
  • ▸ Native quantization & sparse vectors
  • ▸ Same Docker image runs in prod
Production tip: Use create_collection (not recreate_collection) in production — recreate wipes data. The above pattern is for fresh local dev.
🎯 Same conceptual flow, different APIs: Once you've used Chroma and Qdrant, every other vector DB feels familiar — create collection, define dimension + metric, upsert with payload, query with optional filter. The 90/10 rule: 90% of vector DB usage is these four operations.
The Strategic Choice

Self-Hosted or Managed?

The most consequential infra decision you'll make. Self-hosted gives control, freedom from vendor lock, and lower costs at scale — but you own ops. Managed gives instant production-readiness but burns money fast. Here's how to choose, in order.

1
Is your data subject to GDPR / HIPAA / SOC2 with data-residency requirements?
YES → Self-host on infra you control. Or use a managed offering with data-residency guarantees in your jurisdiction.
NO → Continue to Q2.
2
Do you have a dedicated infra/DevOps team?
YES → Self-hosted Qdrant or Milvus pays off above ~10M vectors. Cost crosses over fast.
NO → Continue to Q3.
3
Are you above 50M vectors or 1000+ QPS?
YES → Managed Pinecone / Qdrant Cloud / Zilliz. The ops cost of self-hosting at this scale is real.
NO → Continue to Q4.
4
Do you already run Postgres?
YES → Start with pgvector. No new infra, no new dependency, can host up to 10M vectors comfortably.
NO → Run Qdrant in Docker on a single VM. Best balance of capability and simplicity.
📋 The 2026 default playbook: Prototype with Chroma → move to local Qdrant Docker once you're past 100K vectors or need filters → scale to Qdrant Cloud / managed Pinecone when you cross 10M or need uptime SLAs. You almost never start with managed.
