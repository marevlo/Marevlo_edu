The Core Idea

Why Text Becomes Numbers

Computers don't read. They compute. To search text by meaning — not exact words — we need to turn each piece of text into something machines can compare mathematically. That something is a vector: a list of numbers in high-dimensional space.

Keyword Search

word-level matching (1990s)

Match user words to document words exactly. Different wording → different match.

Query: "how to cancel my subscription"
Doc: "steps to terminate your account"
Result: ✗ no match — zero words overlap

Semantic (Embedding) Search

meaning-level matching (modern)

Convert both to vectors. Compare them in meaning space. Synonyms and paraphrases match.

Query vec: [0.21, -0.48, 0.92, …]
Doc vec: [0.19, -0.45, 0.89, …]
Result: ✓ similarity 0.94 — strong match

What Does a Real Embedding Look Like?

An embedding is just a fixed-length list of decimal numbers — typically 384, 768, 1024, 1536, or 3072 of them depending on the model. Here's the first 12 dimensions of a real one:

# Text → 1536-dimensional vector
"How do I rotate API keys?"

  ↓ embedding model

[0.012 -0.847 0.331 -0.092 0.221 0.654
-0.412 0.078 -0.155 0.483 -0.291 0.062 ... (1524 more)]
🧠 The magic: Each of those 1536 numbers represents a learned dimension of meaning — maybe "is this about animals", "is this formal language", "is this code-related", etc. The numbers aren't human-readable. But two pieces of similar text produce numerically similar vectors. That's all you need to know to start.

Under the Hood

How Embedding Models Actually Work

No deep math. Just enough mechanism to make the right engineering decisions. Modern embedding models are Transformers (just like LLMs), but optimized for a different goal: producing one fixed-size vector for an entire piece of text.

01

Tokenize

Break the text into subword tokens (BPE) — same idea as Module 0.3.

"key rotation"
→ ["key", " rot", "ation"]
02

Transform

Pass tokens through ~12–24 Transformer layers. Each layer adds contextual meaning via attention.

tokens → attention → attention → ...
→ per-token deep representations
03

Pool

Combine all per-token vectors into one. Mean-pooling and [CLS]-pooling are the two common methods.

[t1, t2, t3, ...] → avg() → [vec]
→ one vector for the whole text

How Are They Trained?

Contrastive learning. The model sees pairs of texts known to be similar (or dissimilar) and adjusts its weights so similar pairs produce nearby vectors, and dissimilar pairs produce distant vectors. After billions of pairs, the model knows what "similar" feels like across countless topics.

PULL CLOSE
"How do I cancel my subscription?"
↔ "Steps to terminate your account"
→ teach model: these should be NEAR
PUSH APART
"How do I cancel my subscription?"
↔ "Best Italian pasta recipes"
→ teach model: these should be FAR
Measuring Similarity

Cosine vs Dot Product vs Euclidean

Once everything's a vector, you need a way to score "how similar are A and B?" Three classical options. Picking the wrong one quietly costs you 10–20% retrieval quality. For RAG: cosine wins by default.

Cosine Similarity

RAG DEFAULT

Measures the angle between two vectors. Length doesn't matter — only direction. Range: -1 to +1 (most embeddings stay in 0 to 1). 

cos(A, B) = (A · B) / (|A| × |B|)

# 1.0 = identical direction
# 0.0 = unrelated (90° apart)
# -1.0 = opposite
→ Use when: text length varies, embeddings aren't normalized

Dot Product

FASTEST

Sum of element-wise products. Cares about both direction AND magnitude. No division, so the fastest to compute. 

A · B = a1*b1 + a2*b2 + ...

# Identical to cosine WHEN
# both vectors are length-1
# (normalized).
→ Use when: embeddings are pre-normalized (OpenAI, BGE)

Euclidean (L2)

CLASSICAL

The straight-line distance in N-D space. Cares about absolute position. Larger value = more different (opposite intuition from cosine). 

L2(A, B) = √Σ(ai − bi)²

# 0 = identical
# growing distance = less similar
# unbounded above
→ Use when: image embeddings, position-sensitive tasks

Interactive: Try Cosine Similarity

→ Strong match — likely paraphrases of the same intent.
🎯 The takeaway: In Qdrant, Pinecone, and pgvector you pick the metric at collection creation. Default to cosine for RAG. If your embedding provider explicitly says vectors are normalized (OpenAI does), dot product is mathematically identical and slightly faster. Euclidean is mostly for images and specialty tasks.

Production Optimization

Matryoshka Embeddings & Dimension Reduction

Storing a million 1536-dim vectors takes 6 GB of RAM. At 100M vectors? 600 GB. Matryoshka embeddings — named after Russian nesting dolls — solve this elegantly: the model is trained so that the first N dimensions alone form a usable embedding. Truncate freely. Trade dimension for storage cost with negligible quality loss.

The Nesting Doll Idea

Traditional embedding models concentrate information evenly across all dimensions — chop off half and quality collapses. Matryoshka-trained models pack the most important info into the first dimensions. So vector[:256] is a complete (smaller) embedding by itself.

How to Use It 

from openai import OpenAI

client = OpenAI()

# OpenAI's embedding-3 models are Matryoshka-trained.
# Pass the `dimensions` param to truncate at the source —
# saves bandwidth AND storage.
resp = client.embeddings.create(
    model="text-embedding-3-small",
    input="How do I rotate API keys?",
    dimensions=512,                  # ← Matryoshka magic
)
vec = resp.data[0].embedding
print(len(vec))                       # → 512 (was 1536)

# Quality drops from MTEB ~62.3 → ~61.8 (negligible)
# Storage drops 3× and ANN search is faster.

# Manual alternative: full vector, then slice + normalize
import numpy as np
full = np.array(client.embeddings.create(...).data[0].embedding)
shrunk = full[:512]
shrunk = shrunk / np.linalg.norm(shrunk)  # re-normalize

Quality vs Dimensions (text-embedding-3-large)

↑ Even at 256 dims, you keep ~96% of the quality — and 12× the storage savings.

💡 Production playbook: Start with 1536 dims while prototyping. Once your eval set is stable (Track 6), benchmark at 512 and 768 dims. In 80% of RAG systems, dropping to 512 saves significant storage and HNSW query latency at <2% quality loss. We'll wire this into the production capstone in Track 7.
