Track 02 — Building Servers / Module 2.2

Resources — application-controlled context.

Tools are what the LLM does. Resources are what the host shows it. The distinction sounds small and turns out to be one of the most consequential design choices in MCP — it changes who's in control, what the host can render, and which content is safe to feed back into a model's context window.

01  Tools vs Resources

Same data could go either way. The difference is who decides.

A common confusion in MCP: tools and resources can return the same content. A tool can return the contents of a file, and a resource can be the contents of a file. So why are they distinct primitives? The answer is control. A tool is something the LLM picks; a resource is something the host picks. That single distinction propagates into framing, security, UX, and what the model can do with the result. Compare row by row:

Default rule

If the user is going to browse for it and attach it, it's a resource. If the model is going to think and decide to fetch it, it's either — but a tool is usually the better fit because the model owns timing. When in doubt: action with parameters → tool, addressable content with a URI → resource.

02  Resource anatomy

Five pieces, one resource handler.

FastMCP's resource decorator looks similar to @mcp.tool but does different work. The URI is part of the decorator (tools have a name; resources have an address). The return type is content, not a string. MIME type can be inferred or stated explicitly. Click each highlighted piece to see what role it plays:

Explore · click any element
Anatomy of a Resource Handler Tap a highlighted span → 
from fastmcp import FastMCP, Context

mcp = FastMCP("docs-server")

@mcp.resource(
    "docs://catalog/quickstart",
    name="Quickstart Guide",
    mime_type="text/markdown",
    description="The 5-minute quickstart for new developers.",
)
async def quickstart_doc(
    ctx: Context = None
) -> str:
    """Return the quickstart markdown."""

    # Reach into shared resources via lifespan
    cms = ctx.request_context.lifespan_context["cms"]
    return await cms.fetch("quickstart")
Click any highlighted span on the left to see what role it plays — the metadata the host receives, the field on the wire, and the host-side behaviour it enables.
Resource vs Tool decorator

@mcp.resource(uri) takes the URI as its first positional argument — identification by address. @mcp.tool takes nothing positional — the function name is the identifier. Everything else (name, description, mime_type) is metadata. Watch which decorator you're reaching for; mixing them up generates confusing errors at import time.

03  Static vs templated

One file, or every file?

Static resources have a fixed URI known at server startup. Templated resources have URI patterns — placeholders the host fills in at request time. The difference is between exposing one specific document and exposing every document under a scheme. Toggle below to see both implementations of the same idea:

› When to reach for each


      
    
        
  • Static resources when there's a small, known catalogue: a config file, a few docs, fixed reference data. The host lists them in a sidebar; users pick one.
    • 
        
  • Templated resources when the catalogue is huge or computed: every file in a directory tree, every issue in a repo, every row in a table. Listing all of them in resources/list would be wasteful or impossible.
    • 
        
  • Both at once in the same server is fine — one server can register some static resources and some templated ones. Most production servers do.

04  URI templates

      
How a pattern becomes a match.


      

        URI templates follow RFC 6570 — the same standard used by GitHub's API,
        OpenAPI, and a dozen other places. Placeholders go in curly braces; everything else is a
        literal. At runtime, when a client asks to read a URI, the server tries each registered
        template, picks the first that matches, and extracts the parameters. Try it:

› The matching algorithm in your head


      

        Reading template matches comes up enough that it's worth doing one by hand:
      


      
    
        
  • Template: github://repos/{owner}/{repo}/issues/{id}
    • 
        
  • URI: github://repos/anthropic/courses/issues/42
    • 
        
  • Walk: github:// ✓ · repos/ ✓ · {owner} = anthropic · / ✓ · {repo} = courses · /issues/ ✓ · {id} = 42 · end ✓
    • 
      


      

        Variables consume non-slash segments by default. If the URI ends mid-template — missing
        the id piece — no match. If the literal text in the middle (/issues/)
        doesn't appear, no match. A template with three variables produces a match dict with
        exactly three keys.
      

    


    
    

      
05  Subscriptions

      
The host wants to know when a resource changes.


      

        Resources are typically read once and forgotten. But some resources change — a log file
        being written, a build status updating, a document being collaboratively edited. The host
        wants to keep its view in sync without polling. Subscriptions let the
        client say "tell me when this changes" and the server emits notifications when
        it does. Press play to watch one play out:

› The capability prerequisite


      

        Subscriptions need the server to advertise resources: { subscribe: true } at
        handshake (Module 1.3). Without that sub-flag, the resources/subscribe
        method doesn't exist for this session. The client checks the flag, doesn't even attempt
        the call when missing.
      


      

        
listChanged is not subscribe

        

          Easy to confuse. listChanged means "the catalogue of resources changed" —
          a new file appeared, an old one was deleted. subscribe means
          "the contents of this specific URI changed". The first fires
          notifications/resources/list_changed; the second fires
          notifications/resources/updated. Different capabilities, different
          notifications, different use cases.
        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 2.2

          
Five quick checks before Module 2.3.

        

      


      

        
Q1.Your server can return the contents of any file under a directory. Tool or resource?

        

          
          
          
          
        

        
Right. Files have stable identities (paths), are read-only by nature, and users typically browse for them. A templated resource is the better fit. You can offer a tool alongside if the LLM should be picking files autonomously, but the file content itself fits the resource pattern.

      


      

        
Q2.You have 50,000 documents in a CMS. How do you expose them as resources?

        

          
          
          
          
        

        
Yes. Templates are exactly for this case. The catalogue is infinite from the client's perspective — clients can request any URI matching the template, and the server resolves it on the fly. resources/templates/list announces the template; resources/read resolves a specific URI.

      


      

        
Q3.Given template db://{table}/{id}, which URI matches?

        

          
          
          
          
        

        
Correct. The template requires two non-slash segments after db://. Option A is missing the second segment; C has a slash inside what would have to be the id variable; D has the wrong scheme. Only B walks the template cleanly.

      


      

        
Q4.A log file resource changes hundreds of times per second. Your client wants to display the latest content. Which mechanism?

        

          
          
          
          
        

        
Yes. Subscriptions are exactly for live-changing resources. listChanged is for catalogue changes, not content changes. Polling wastes resources. The server needs to advertise resources: { subscribe: true } at handshake for this to work.

      


      

        
Q5.Your tool returns content that the host should be able to fetch by URI later. Best pattern?

        

          
          
          
          
        

        
Right. The resource_link content type (covered in 2.1) is the bridge between tools and resources. Tool returns a tiny pointer; the host fetches the content via the resource primitive on demand. Keeps tool responses small and lets the LLM decide whether to actually look at the content.
