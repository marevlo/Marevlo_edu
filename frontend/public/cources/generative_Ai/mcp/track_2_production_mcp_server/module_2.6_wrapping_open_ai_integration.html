Track 02 — Building Servers / Module 2.6

Wrapping real systems & OpenAPI.

Most production MCP servers don't implement anything new — they wrap something that already exists. A REST API, a database, an SDK, a CLI. The cheapest path from "we have a backend" to "an agent can use it" is a wrapper, and FastMCP's from_openapi() turns the most common version of that wrap into one line of code. This module is about doing it well: tuning what gets exposed, naming things for LLMs, and learning from servers already in production.

Default to OpenAPI if you already have it

Most modern backends already publish an OpenAPI spec — FastAPI generates one for free, Django REST Framework and Flask have drf-spectacular and flask-smorest, Spring has springdoc. If your team has gone to the trouble of maintaining one, the wrap is almost free. The right work to do is tuning it (Section 3), not rewriting it.

02  OpenAPI → MCP

One spec in, one server out.

FastMCP can read any OpenAPI 3 document and turn it into a working MCP server. Each path operation becomes a tool by default (or a resource for read-only GETs, if you ask). Parameter schemas come from the spec; descriptions come from the spec; auth flows through an httpx client you provide. The mapping is deterministic — see it side by side:

› The minimum-viable wrap

Three lines, no rewrites. The spec is the contract; the httpx client carries auth; FastMCP generates the rest: 

import httpx
from fastmcp import FastMCP

# 1. Load the OpenAPI spec (could be from disk, URL, or a live FastAPI app)
spec = httpx.get("https://api.example.com/openapi.json").json()

# 2. Build an httpx client that handles auth — every generated tool uses this
client = httpx.AsyncClient(
    base_url="https://api.example.com",
    headers={"Authorization": f"Bearer {API_TOKEN}"},
    timeout=30.0,
)

# 3. Generate the MCP server in one call
mcp = FastMCP.from_openapi(openapi_spec=spec, client=client, name="example-api")
mcp.run()

For a FastAPI app, you skip the spec download — pass the app object directly and FastMCP introspects its routes: 

from fastmcp import FastMCP
from myapp import app    # your existing FastAPI app

mcp = FastMCP.from_fastapi(app=app, name="myapp")
mcp.run()
# Tools call the FastAPI routes directly in-process — no HTTP roundtrip
Naming is everything — fix your operationIds

The LLM picks tools by their name. The default name is the OpenAPI operationId; many specs auto-generate ugly ones like get_users_users_get. Fix this in the spec, not by renaming on the MCP side — a clean operationId helps every API consumer, not just the LLM. In FastAPI: @app.get("/users", operation_id="list_users").

03  Tuning the wrap

From "everything" to "the right things".

A raw from_openapi() gives you every endpoint, which is rarely what you want. Internal admin endpoints shouldn't be tools. Read-only GETs often work better as resources. The same wrap can produce different catalogs depending on how you tune it — toggle below and see the catalog change:

› The four knobs that matter

  • Tag filtering. Most OpenAPI specs already tag operations by domain area (users, billing, admin). The single most useful tuning step is excluding the admin tag from MCP exposure — those endpoints exist for human operators, not for agents.
  • Route maps. Tell FastMCP that GET /resources/{id} should become an MCP resource (resource://...), not a tool. Reduces the tool catalogue size; gives the LLM cleaner semantics ("read the thing" vs "call the thing").
  • Method filtering. Read-only deployments can include_methods=["GET"] to exclude every mutating endpoint. Safer for proof-of-concept rollouts and for shared servers where only some users should have write permission.
  • Custom transformations. Override the auto-generated name, description, or argument schema for any operation. Useful when you can't change the upstream API but want a friendlier tool surface.
Authentication flows through the httpx client

FastMCP doesn't reinvent auth — it just calls your httpx.AsyncClient for every generated tool. So whatever auth pattern your API already uses (bearer tokens, mTLS, OAuth, signed query params) works automatically: configure it on the client once, every tool inherits it. For per-user auth flows (the user authenticates, not the server), Track 4's OAuth module covers the patterns.

04  Reference servers

What good wraps look like.

Seven production-grade MCP servers worth studying. None of them use from_openapi() alone — they all blend autogeneration with hand-written tools where it matters. Tab through to see what each chose to expose, and why:

Browse · tap each server
Reference Server Gallery · Production Wraps

› Patterns across all seven

  • Read endpoints become resources when they're "browseable". A list of files, a list of issues, a list of channels — these get a resource URI and a templated form so the host can suggest them in @ mentions. One-off "give me this single record" endpoints stay as tools.
  • Search is always a tool, not a resource. Search takes a query and is fundamentally LLM-driven; you can't make it a resource because the URI would carry user input. All seven servers have a search_* tool.
  • Domain-specific helpers ride alongside the wrap. Postgres wraps SQL but also ships explain and list_tables. Drive wraps the REST API but also ships search and get_file_content. The wrap is the floor; hand-written helpers are the polish.
  • Auth scopes are narrow on purpose. The GitHub server reads PRs/issues but doesn't expose delete_repo. The Drive server can read but, by default, can't share. Scope narrowing is itself a design choice — Module 4.2 builds on this.
05  Anti-patterns

Five ways a generated wrap goes wrong.

The autogeneration is easy. The result is often worse than a small hand-written server, because the LLM is given a maze it can't navigate. Five recurring failure modes; click each:

The 30-tool rule

Empirically — across hosts and models — LLM tool-picking accuracy degrades noticeably above ~30 tools in a single server, and falls off a cliff above ~50. If your wrapped API has 200 endpoints, the right move isn't "ship 200 tools" — it's to pick the 25 the LLM will actually use and either drop the rest or hide them behind a meta-tool (e.g. call_endpoint(path, method, body) for power use). Module 4.1 has more on this from the security angle.

Knowledge Check · End of Module 2.6
Five checks before Module 2.7.
Q1.You have an existing FastAPI app with 35 endpoints, an auto-generated OpenAPI spec, and a deadline. Which approach?
Correct. When you already have a spec and a clean FastAPI app, autogeneration is the cheapest path. The work that earns its keep is tuning: dropping admin endpoints, marking read-only GETs as resources, and fixing names. Don't rewrite what you already have.
Q2.Your generated server exposes 200 tools and the LLM picks the wrong one half the time. What's the right fix?
Right. The 30-tool rule is empirical: above ~30 tools per server, accuracy drops; above ~50 it falls off a cliff. Bigger models help marginally, longer system prompts don't help much, but a thoughtfully-curated catalog helps enormously.
Q3.Your wrapped REST API uses Bearer token auth. Where does the token go in a from_openapi() setup?
Correct. The httpx client is the auth surface. Whatever auth flow your upstream API uses (bearer, mTLS, OAuth, signed params) configures naturally on the client, and FastMCP inherits it automatically for every generated tool. This is exactly why the design is good — no reinvention.
Q4.Your tag-filtered OpenAPI generates a tool called get_users_users_get. What's the right fix?
Yes. The operationId is the API's "name field" — every codegen tool downstream of OpenAPI uses it. Fix it at the source and every consumer benefits. Renaming downstream in MCP works but means the LLM sees a different name than your API docs show, which is a discoverability tax.
Q5.You're wrapping a database. Should SELECT queries be tools or resources?
Correct. The dispatch test: who initiates? For schema (browseable, predictable, addressable), it's a resource. For a SELECT with arbitrary WHERE clauses, it's a tool — the LLM is choosing what to do. The Postgres reference server gets this right: query is a tool, postgres://... URIs are resources.
