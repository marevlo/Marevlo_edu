🌉
Bridge module Track 2 → Track 5/6. This module closes Track 2 by showing how to expose retrieval — the heart of any knowledge-base agent — as an MCP server. It also sets up the production patterns of Track 5 and the capstone in Track 6.

RAG as an MCP server.

Retrieval-Augmented Generation used to be a pipeline you wired up inside the host: embed → retrieve → stuff into prompt → call LLM. With MCP, the retrieval stops being a pipeline and becomes a service: a search tool the LLM chooses to call, with document resources the host can attach. The shift unlocks an agent that decides when to retrieve, can refine queries mid-conversation, and composes retrieval with every other tool you've built in this track.

01  Two paradigms

Always-retrieve vs decide-to-retrieve.

Classical RAG retrieves before the LLM call. The host embeds the user's question, pulls top-k chunks from a vector store, stitches them into the prompt, and only then asks the model. The LLM has no choice; it's served retrieved context whether it needed it or not. Retrieval-as-a-tool flips this: the LLM gets a search tool and decides when to call it. Toggle to compare:

Compare · toggle paradigm
Two Paradigms · Side-by-Side Flow

› When each paradigm wins

  • Classical RAG wins when retrieval is always needed. Customer-support chatbot grounded in a fixed knowledge base, a "your docs as a chat interface" wrap, anything where the question domain is the corpus. Lower latency (one call) and the LLM can't forget to retrieve.
  • Retrieval-as-a-tool wins for general-purpose agents. The same agent that uses GitHub, writes code, and searches Slack should also be able to retrieve from internal docs — but shouldn't retrieve every time you ask it the weather. The LLM's judgement is the gating mechanism.
  • Hybrid is also a thing. Some hosts auto-attach a "primary" knowledge base as classical RAG, while still exposing it as a tool for refinement. Best of both at the cost of some duplicated work.
Why this matters for MCP

The MCP path makes your retrieval system portable. Every host that speaks MCP can attach it; you don't need a Slack-specific RAG plugin and a Cursor-specific RAG plugin and a Claude Desktop-specific RAG plugin. One server, every host. That's the bridge from "RAG pipeline" to "RAG product".

› The minimum-viable RAG server

About 40 lines, including imports. The pattern: hand-written tools that call out to whatever index you happen to use (Postgres+pgvector, Qdrant, Chroma, OpenSearch). The MCP surface stays the same; the backend changes: 

from fastmcp import FastMCP, Context
from mcp.types import TextResourceContents

mcp = FastMCP("company-rag")

@mcp.tool
async def search_docs(
    query:      str,
    collection: str = "all",
    k:          int = 5,
    filters:    dict | None = None,
) -> list[dict]:
    """Search the knowledge base. Returns ranked chunks with source URIs."""
    chunks = await index.hybrid_search(query, collection, k, filters)
    return [
        {
            "text":   c.text,
            "uri":    c.uri,           # for citation
            "score":  c.score,
            "title":  c.metadata.title,
        }
        for c in chunks
    ]

@mcp.resource("docs://collection/{doc_id}")
async def document(doc_id: str) -> TextResourceContents:
    """Whole document by id."""
    doc = await store.get(doc_id)
    return TextResourceContents(uri=f"docs://collection/{doc_id}", text=doc.text)

mcp.run()
Citations come from the URI

Every chunk returned must include a stable URI for the source — that's how the LLM cites. If the URI is a real resource the host can read (the docs:// family), users can click through to the full document. If it's an opaque key, the agent can still drop it into the answer as "[source: docs://kb/12345]". Either way, the LLM citing without a URI in the chunk metadata is doing it from memory — and getting it wrong.

03  Hybrid search

BM25, vectors, and when each is wrong.

Two retrieval mechanisms dominate, and they fail on different queries. BM25 (keyword + TF-IDF) is great when the query contains the exact terminology you indexed — product names, function names, error messages. Vector search is great when the query expresses an intent that doesn't match the docs verbatim — "how do I make the cache reset itself" should find "automatic invalidation strategies". › The "what you actually ship" stack

  • BM25 via Postgres FTS, OpenSearch, or Tantivy. Cheap to run, fast to query, fast to index. Comes for free with most existing databases.
  • Vector search via pgvector, Qdrant, Chroma, or LanceDB. Requires embedding the corpus once (and re-embedding on schema changes). Adds 30–100ms latency per query for the embedding call alone.
  • Hybrid fusion — run both, merge results with Reciprocal Rank Fusion. ~80% of the win of "smart" approaches at ~5% of the engineering effort. Default to this.
  • Cross-encoder reranker (e.g. bge-reranker-base) — re-score top-50 hybrid results, return top-5. Adds 50–200ms but materially improves precision. Add it once the basics are stable; not on day one.
async def hybrid_search(query: str, k: int = 5) -> list[Chunk]:
    # 1. Run both retrievers in parallel
    bm25_hits, vec_hits = await asyncio.gather(
        bm25.search(query, k=50),
        vector.search(await embed(query), k=50),
    )

    # 2. Fuse rankings via Reciprocal Rank Fusion
    fused = rrf_fuse(bm25_hits, vec_hits, k=60)   # classic k=60 constant

    # 3. Optional: cross-encoder reranker for precision
    if reranker:
        fused = await reranker.rerank(query, fused, top_k=25)

    return fused[:k]
Recall vs precision — the agentic shift

Classical RAG tunes for precision in top-5: the LLM gets only those 5 chunks and one bad one is real noise. Retrieval-as-a-tool tunes for recall in top-25: the LLM is happy to ignore the bad ones if the right answer is in there. This means you can run a larger k, rely less on rerankers, and worry less about chunk-level precision — at the cost of some token waste the LLM will tolerate.

One question, one retrieval, one cited answer.

› Where the latency lives

  • Embed query — 30–100 ms if local model, 100–300 ms if cloud API. Vector search needs an embedding before it can search.
  • Hybrid retrieval — 20–80 ms for 100k-chunk corpora with a healthy index. Both BM25 and vector run in parallel; the slower of the two dominates.
  • Reranker (if used) — 50–200 ms for cross-encoders on 25–50 candidates. The biggest single addition in the precision-vs-latency tradeoff.
  • LLM with chunks — 1–4 s depending on chunks-sent and model. Almost always the dominant cost.
Total round trip (typical) ≈ 2.0 – 4.5 s
  ├ embed query     ≈   80 ms
  ├ hybrid search   ≈   60 ms
  ├ rerank top-50   ≈  120 ms
  └ LLM (with k=5) ≈ 2000 ms+

Knowledge Check · End of Module 2.7
Five checks to close Track 2.
Q1.You're building a general-purpose dev assistant that can use GitHub, browse Slack, and answer questions from a 200k-doc internal wiki. Classical RAG or retrieval-as-a-tool for the wiki?
Correct. The agent has multiple knowledge sources and the wiki isn't always relevant. Classical RAG would force wiki retrieval on every question — wasting tokens on a 'fix this Slack bug' question. Letting the LLM choose is exactly what retrieval-as-a-tool delivers.
Q2.Your search tool returns chunks but no source URIs. Symptom?
Right. LLMs cite from what's in their context. No URI in the chunk = no URI to cite = either silent omission or fabrication. The "search-and-cite" pattern is the antidote: every chunk carries a uri field; the system prompt instructs the LLM to cite using those URIs verbatim.
Q3.A user asks "how do I fix EBADF". Best retrieval backend?
Correct. Acronyms, error codes, function names, product SKUs — these are where BM25 shines. Hybrid still works (it includes BM25), but pure vector struggles to match exact tokens reliably. The Section 3 widget shows this exact pattern.
Q4.You're shipping a "reindex" operation that takes 8 minutes. What primitive?
Yes. Reindexing hits all four task-fitness tests from Module 2.5: long, progressive, cancellable, backgroundable. Modelling it as a synchronous tool means the LLM blocks for 8 minutes; modelling it as a task makes it feel like submitting a job.
Q5.Your hybrid search runs BM25 + vector in parallel and merges. Top-5 BM25 has the right doc at position 4; top-5 vector misses entirely. What does Reciprocal Rank Fusion typically do?
Right. RRF is forgiving by design. Each retriever contributes 1 / (k + rank) for every doc it finds; docs found by both win; docs found by one still appear. This is what makes hybrid fusion the "default" choice — neither retriever's blind spot becomes the system's blind spot.

RRF — Reciprocal Rank Fusion outperforms Condorcet…

The 4-page paper that turned RRF into the "default" rank fusion method. Sketches the formula (score(d) = Σ 1 / (k + rank_i(d))) and why the constant k = 60 is the empirical sweet spot. Worth reading end-to-end the first time you implement hybrid search — explains the failure modes of simpler approaches.
