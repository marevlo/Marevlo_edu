Track 02 — Building Servers / Module 2.1

Tools — the workhorse primitive.

Of MCP's three primitives, tools are the one models actually call. Every other module in this track adds polish, but tools are where the work happens. This is the deep dive: the annotation hints hosts use to decide what's safe to auto-run, the rich content types you can return, the two valid error strategies, and the craft of writing descriptions an LLM will pick correctly.

01  Tool anatomy

Eight pieces, one tool definition.

Module 0.2 introduced @mcp.tool as a one-decorator affair. In production you'll layer a lot more on top — explicit names, validation constraints, parameter descriptions, return-type schemas. Click each highlighted piece below to see what it does, what wire change it produces, and where the LLM sees it:

Explore · click any element
Anatomy of a Production Tool Tap a highlighted span → 
from typing import Annotated
from pydantic import Field
from fastmcp import Context

@mcp.tool(
    name="search_issues",
    title="Search GitHub Issues",
    annotations={
        "readOnlyHint": True,
        "openWorldHint": True,
    },
)
async def search_issues(
    query: Annotated[
        str,
        Field(min_length=3, description="GitHub search syntax"),
    ],
    limit: Annotated[int, Field(ge=1, le=100)] = 20,
    ctx: Context = None,
) -> list[str]:
    """Search open GitHub issues matching the query.

    Use for finding existing bug reports or feature requests
    before filing new ones. Returns titles of matching issues.
    """
    await ctx.info(f"searching: {query}")
    rows = await github.search(query, limit=limit)
    return [r.title for r in rows]
Click any highlighted span on the left to see what role that piece plays — what FastMCP does with it, what reaches the LLM, and where it shows up on the wire.
Why Annotated + Field beats raw type hints

query: str tells FastMCP the type but nothing else. Annotated[str, Field(min_length=3, description="...")] tells it the type and emits minLength: 3 into the JSON Schema and attaches a per-parameter description the LLM can read. Three pieces of guidance for the model and runtime, with one annotation. Use it for every non-trivial parameter.

02  Annotations

Four hints. They change whether a host runs your tool without asking.

Annotation hints are how a server tells a host what kind of operation a tool performs. They're advisory — hosts may use them, the spec doesn't compel anyone — but every major host does. Get them wrong and your tool either nags users with confirmation dialogs on every call (false-positive destructive) or fires off database deletes silently with no prompt (false-negative). Toggle the flags below to see how the picture changes:

Configure · toggle the flags
Annotation Inspector

Tool does not modify any state — pure read.

Tool may delete or irreversibly modify data.
idempotentHint
Calling N times has the same effect as calling once.
openWorldHint
Operates on external systems (network, third-party APIs).

Hints are not security

        

          Set the hints correctly because hosts and users use them for safety decisions — not
          because the protocol enforces them. A malicious server can lie. A host that blindly
          trusts a readOnlyHint from an untrusted server is the kind of bug
          covered in Module 4.1. The hints describe intent; security still comes from elsewhere.
        

      

    


    
    

      
03  Rich result content

      
Five content types. Each one looks different on the wire.

Structured output is for chaining

        

          structuredContent (added in 2025-06-18) carries typed data alongside the
          text result. The LLM sees the text for its reasoning; downstream tools — or the host's
          code — can parse the structured JSON without round-tripping through a prose summary.
          Use this whenever the next step in a tool chain needs your result's fields, not
          its narrative.
        

      

    


    
    

      
04  Error handling

      
Two strategies. They look almost identical and behave very differently.


      

        There are two valid ways a tool can fail. Strategy one: let the exception escape — FastMCP
        will translate it into a JSON-RPC -32603 Internal error response. Strategy two:
        return a normal result with isError: true. The wire shapes diverge in one
        crucial way that controls whether the LLM sees the failure and can recover from
        it. Toggle to compare:

The recoverability rule

        

          Use isError: true when you want the LLM to see the failure
          and reason about it. "File not found" — LLM can retry with a different path.
          "Quota exceeded" — LLM can pick a smaller search. Raise an exception only for
          bugs. "Database connection lost" — the LLM has no useful response; the host
          should retry. The protocol layer treats raised exceptions as protocol errors;
          isError results are tool-level errors. The distinction is whether retrying with
          better inputs could possibly help.
        

      

    


    
    

      
05  Writing descriptions

      
The text the LLM reads to decide whether to call your tool.


      

        Tool descriptions are prompt material. Every host running an LLM-driven agent passes the
        description through the model on every turn — that's thousands of model reads per
        production server per day. A vague description means the LLM either ignores your tool
        when it should call it, or calls it when it shouldn't. Compare two descriptions of the
        same tool:

› The four-part description recipe


      
Across thousands of production tools, the descriptions that work consistently share a structure:


      
    
        
  • What it does — one sentence, action-oriented, names the system being touched.
    • 
        
  • When to use it — the trigger conditions. "Use this for…" or "Use when…". This is what the LLM matches against the user's request.
    • 
        
  • Input format guidance — example queries, expected shape, constraints. Eliminates guesswork at call time.
    • 
        
  • Side-effect statement — read-only? destructive? irreversible? Pre-answers the host's confirmation logic and the LLM's risk reasoning.
    • 
      


      

        
Forward reference · Module 4.1

        

          Tool descriptions are also a security surface. Module 4.1 covers tool poisoning —
          attacks where a malicious server writes descriptions that manipulate the LLM into doing
          something against the user's interest. Knowing how to write a good description is also
          knowing what a malicious one looks like; the two skills are the same skill.
        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 2.1

          
Five quick checks before Module 2.2.

        

      


      

        
Q1.You're writing delete_branch(name: str). Which annotations should you set?

        

          
          
          
          
        

        
Correct. Destructive + idempotent + open-world all apply. The host now knows to confirm before running (destructive), to safely retry on transient failures (idempotent), and that the tool touches an external system (open-world).

      


      

        
Q2.Your tool reads a file that doesn't exist. Should you raise an exception or return isError: true?

        

          
          
          
          
        

        
Right. "File not found" is recoverable — the LLM can try a different path. isError result puts the failure in the LLM's context so it can adapt. Raising would trigger a protocol-level error the LLM doesn't reason over.

      


      

        
Q3.What's the difference between name= and title= in @mcp.tool(...)?

        

          
          
          
          
        

        
Exactly. The model sees name and uses it to invoke. The host shows title in confirmation dialogs and tool palettes. Same tool, two audiences, two strings.

      


      

        
Q4.Your tool returns the latest CO₂ measurement: {value: 421.3, units: "ppm", timestamp: "..."}. Best content type?

        

          
          
          
          
        

        
Yes. Structured output is for chaining. Text is for the model's reasoning. Returning both is the best of both worlds — added in spec 2025-06-18 for exactly this case.

      


      

        
Q5.You write a tool description: "Get data." Why does this fail at scale?

        

          
          
          
          
        

        
Correct. Tool descriptions are prompt material the LLM reads every turn. "Get data" has no semantic distinction. The four-part recipe (what · when · input · side-effects) exists exactly to give the LLM the matchable signal it needs.
