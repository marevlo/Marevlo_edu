Track 02 — Building Servers / Module 2.4

Sampling, elicitation & roots.

Three primitives that flip the arrow. Tools, resources, and prompts all run client-to-server. Sampling lets a server ask the host's LLM for a completion. Elicitation lets a server ask the user a question mid-call. Roots lets a server learn which parts of the workspace the user has consented to. All three are capability-gated and all three sit on the trust boundary — which is why this module is the most security-relevant in Track 2.

01  Reversing the flow

Most MCP traffic goes one way. These three go the other.

Up to now, every exchange has been the same shape: the host asks the server something, the server answers. Tools, resources, prompts — all forward. Sampling, elicitation, and roots invert that arrow. The server asks the host something, and the host (often via the user) answers. Toggle:

Compare · toggle forward vs reverse
Forward vs Reverse Requests

› Why reverse calls need extra gates

Forward calls carry implicit consent: the user typed something, the host forwarded the model's tool decision, the server responded. Reverse calls are different — the server is initiating, and servers are often remote software the user doesn't fully trust. So MCP requires:

  • Capability declaration. The client must advertise during initialize that it supports sampling, elicitation, or roots. A server may not even try those methods otherwise — they'll get back -32601.
  • User consent at runtime. Sampling and elicitation must be approvable/deniable by the user. Hosts render approval UIs; servers must handle the deny case gracefully.
  • Scoped trust. Roots aren't an asking primitive — they're a scoping primitive. The user has pre-declared which directories a server may see; the server enforces that boundary itself.
The vendor-lock-in benefit

Reverse primitives are also what make servers portable across LLM providers. A server that needs LLM completion doesn't ship its own OpenAI key or Anthropic key — it asks the host's already-configured LLM via sampling/createMessage. The same server works in Claude Desktop, Cursor, or any other host without modification.

02  Sampling

The server asks the LLM for help.

A server doing summarisation, classification, or any other "soft" task often wants to call an LLM itself — but shouldn't have to ship its own API key. The sampling/createMessage request lets the server send a messages array up to the client, which forwards it to the already-configured LLM, prompts the user for approval, and returns the completion. › Server-side: declaring a sampling-using tool

FastMCP exposes sampling via the request context. Inside any tool you can call ctx.sample(messages=...), and the host handles the round trip, approval, and LLM call. Your tool just gets a string back (or an error if the user denied): 

@mcp.tool
async def summarise_doc(uri: str, ctx: Context) -> str:
    """Read a document and produce a one-paragraph summary."""
    doc = await ctx.read_resource(uri)

    # Ask the host's LLM via sampling — no API key needed server-side
    result = await ctx.sample(
        messages=[
            {"role": "user", "content": f"Summarise:\n\n{doc.text}"}
        ],
        system_prompt="You are a concise technical writer.",
        model_preferences={
            "hints": [{"name": "claude-sonnet"}],
            "intelligencePriority": 0.8,
            "speedPriority":        0.3,
            "costPriority":         0.5,
        },
        max_tokens=500,
    )
    return result.content.text

› The three modelPreferences priorities

Servers don't pick a specific model — that would defeat portability. Instead they declare preferences as three sliders, each 0–1: how much they care about intelligence, speed, and cost. The host picks the closest matching model in its configured stable. The hints array gives a soft preference ("if you have Sonnet, use it; otherwise pick what's closest"). All four fields are optional.

Sampling is dangerous if mis-trusted

A malicious server could ask the host's LLM to do almost anything — leak context, produce phishing copy, generate code with backdoors. This is why sampling must require user approval per call (or per server-session), and why hosts that don't show the messages payload to the user in the approval UI are out of spec. Cover this in detail in Module 4.1.

03  Elicitation

The server asks the user a question.

Sometimes a tool needs information that wasn't in its arguments. The classic case: a "create_pull_request" tool that needs to confirm the target branch before doing something destructive. Pre-2025-06-18 servers either failed (bad) or guessed (worse). The elicitation/create request, formalised in the current spec, lets a server pause mid-tool-call, render a structured form to the user via the host, and resume once the answer comes back. Three input types: string, boolean, enum. Tab through them:

› Server-side: ctx.elicit() with a schema

from pydantic import BaseModel, Field

class PrConfirm(BaseModel):
    target_branch: str  = Field(description="branch to merge into")
    is_draft:      bool = Field(description="open as draft?", default=False)

@mcp.tool
async def create_pr(branch: str, ctx: Context) -> dict:
    """Open a PR from  to a confirmed target."""
    result = await ctx.elicit(
        message=f"Confirm the target branch for PR from {branch}",
        response_type=PrConfirm,
    )

    if result.action == "decline":
        return {"isError": True, "content": [{"type": "text", "text": "User declined."}]}
    if result.action == "cancel":
        return {"isError": True, "content": [{"type": "text", "text": "User cancelled."}]}

    confirmed = result.content   # PrConfirm-shaped
    return await do_pr(branch, confirmed.target_branch, draft=confirmed.is_draft)

› Three response actions

  • accept — user filled the form and submitted. content contains the filled fields.
  • decline — user explicitly said no. Tool should fail cleanly with a clear ToolMessage so the LLM knows.
  • cancel — user dismissed (Esc, closed dialog). Treat like decline for safety.
Elicitation is for missing data, not new permission

The right use is "I need one more piece of information to do my job correctly". The wrong use is "let me ask the user to authorise destructive actions" — that's what the host's tool-approval UI is for, and routing it through elicitation can desensitise users to legitimate confirmation prompts. The 2025-06-18 spec is explicit: elicitation should not be used to bypass existing consent flows.

04  Roots

Filesystem scoping with explicit user consent.

A filesystem MCP server should never have ambient access to the whole machine. Roots are the scoping mechanism: the user tells the host "this server may see these directories", and the server reads the list via roots/list. › Server-side: enforcing roots before every access

      
from pathlib import Path
from urllib.parse import urlparse

async def allowed_roots(ctx) -> list[Path]:
    """Refresh the allowed roots from the client."""
    res = await ctx.list_roots()
    return [Path(urlparse(r.uri).path).resolve() for r in res.roots]

def in_any_root(target: Path, roots: list[Path]) -> bool:
    """True iff target is under at least one allowed root."""
    target = target.resolve()       # resolve symlinks, ..
    return any(target == r or r in target.parents for r in roots)

@mcp.tool
async def read_file(path: str, ctx: Context) -> str:
    roots = await allowed_roots(ctx)
    target = Path(path).resolve()
    if not in_any_root(target, roots):
        raise PermissionError(f"{target} is outside any allowed root")
    return target.read_text()


      

        
Always resolve before comparing

        

          The single most common roots bug is comparing the user-supplied string path
          directly. /etc/../home/alex/dev/app/foo looks suspicious but
          resolves to /home/alex/dev/app/foo. Path traversal attacks (Module 4.3)
          live in this gap. Always call resolve() first, then check
          containment — comparing raw strings is exploitable, comparing canonical paths
          is safe.

› Cheap test: who initiates? Who decides?

      
    
        
  • If the server initiates and the user has to approve in real time → sampling or elicitation. High trust burden. Show your messages and intentions clearly.
    • 
        
  • If the server initiates and consent was given at config time → roots, logging. Lower runtime friction, but the user must have meaningful control during setup.
    • 
        
  • If the server initiates and no consent is needed → progress, ping. Purely operational; can't be used for exfiltration.
    • 
      


      

        
Capability negotiation is your refusal mechanism

        

          A server that wants to require sampling can either probe the
          initialize response and refuse to start if the client doesn't support it, or
          gracefully degrade. Symmetric: a client that doesn't trust this server with
          sampling can simply not declare the capability — the server's
          createMessage calls then fail with -32601, and your
          tool code handles it with a clean error path. Capability declaration is the
          first line of defence; runtime approval is the second.
        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 2.4

          
Five checks before Module 2.5.

        

      


      

        
Q1.Your server needs LLM completion to summarise a doc. The user is on Cursor, which is configured with their own Claude API key. What's the right way to get the summary?

        

          
          
          
          
        

        
Correct. Sampling is exactly this — the server gets LLM access without shipping keys or knowing which provider the user is on. The host handles the model choice and the user approves the request, so it works identically in Claude Desktop, Cursor, or any other compliant host.

      


      

        
Q2.Mid-tool-call, your create_pr tool realises it doesn't know which branch to target. The 2025-06-18 way to ask the user?

        

          
          
          
          
        

        
Right. Elicitation was formalised in 2025-06-18 to handle exactly this — structured mid-call questions to the user. The schema gives the host a typed form to render, and the three-action response model handles "they declined" and "they cancelled" distinctly.

      


      

        
Q3.Your filesystem server's read_file tool is asked to read /home/alex/dev/app/../../etc/passwd. The user's roots are /home/alex/dev/app and /home/alex/notes. What should happen?

        

          
          
          
          
        

        
Exactly. Path traversal works by hiding a different canonical path inside a string that looks like it starts in an allowed root. Always resolve() before doing containment checks — never compare the raw string.

      


      

        
Q4.An elicitation response comes back with action: "cancel". How should your tool handle it?

        

          
          
          
          
        

        
Yes. Cancel is structurally the same as decline for the tool's purposes — no consent was given. Building a tool that proceeds on cancel turns elicitation from a safety net into a footgun. The LLM can read the error and decide whether to retry with explicit args.

      


      

        
Q5.The user just opened a second project folder in their IDE. The roots list changed from one entry to two. How does your server find out?

        

          
          
          
          
        

        
Correct. Same pattern as tools/list_changed and prompts/list_changed — the client emits a list-changed notification, the server re-fetches. No polling, no restart. This is also what keeps long-lived servers honest about which folders they should have access to.
