What "1st party" actually means

First-party SDKs are maintained by Anthropic and the MCP working groups, tracked against the latest spec revision, and ship within days of each revision. Community SDKs are excellent but lag behind on new features; check their changelog before betting a production server on a freshly-spec'd capability.

02  FastMCP vs the low-level SDK

The same server, two different conversations with the protocol.

Python ships two layers in the same SDK family. The low-level SDK (mcp.server) is an imperative protocol library — you register handlers for each method, build Tool objects by hand, manage content serialization yourself. FastMCP sits on top of it and does almost all of that work via introspection of your function signatures and docstrings. Toggle between the two implementations of the same tool to see what FastMCP is buying you:

Compare · toggle the layer
Same Tool · Two Implementations 

        

        

      


      
› When to drop down to the low-level SDK

      

        FastMCP is the right default. There are a few real cases for going lower:
      


      
    
        
  • Dynamic tools. Tool list computed at runtime from a database. FastMCP's decorator model assumes tools are known at import time; the low-level list_tools handler can return anything you want.
    • 
        
  • Non-Python schemas. If your tool's input shape doesn't fit cleanly into Python type hints (deeply nested unions, custom JSON Schema constructs FastMCP doesn't generate), you can ship the schema by hand at the low level.
    • 
        
  • Embedding MCP into a larger app. When you're not running an MCP server as the main process — you're hosting it as one feature of a bigger application — the low-level SDK gives you the integration surface to plug into your existing event loop.
    • 
      


      

        
Mixed-mode is legal

        
You can take a FastMCP app and reach into its underlying low-level server when you need to. mcp._mcp_server exposes the raw Server from mcp.server, and you can register additional low-level handlers next to the decorator-generated ones. Use this sparingly — it's a clear signal in code that something exotic is happening.

04  Lifespan

      
How shared resources survive across requests.


      

        Every tool call doesn't get a fresh DB connection — that would be catastrophic at any
        meaningful traffic. Heavy resources (database pools, HTTP clients, model caches) get
        created once at server startup, kept alive across the server's entire
        lifetime, and torn down cleanly at shutdown. That's what lifespan is. Press play to watch
        a server's lifespan unfold:

› The actual FastMCP shape



from contextlib import asynccontextmanager
from fastmcp import FastMCP
import httpx, asyncpg

@asynccontextmanager
async def app_lifespan(server):
    # Startup: build expensive shared things
    db = await asyncpg.create_pool(dsn=settings.DB_URL)
    http = httpx.AsyncClient(timeout=30)
    try:
        yield {"db": db, "http": http}
    finally:
        # Shutdown: drain and close, in reverse order
        await http.aclose()
        await db.close()

mcp = FastMCP("my-server", lifespan=app_lifespan)


      

        
The reverse-order shutdown rule

        

          Tear down resources in reverse of the order you set them up. If your HTTP client makes
          calls during its aclose() that need the DB pool, closing the DB pool first
          leaves the HTTP client mid-call against a dead connection. try/finally with
          ordered awaits in reverse is the entire pattern. Async context-managers do
          this automatically; manual lifespans need the discipline.
        

      

    


    
    

      
05  The Context object

      
The handle every tool function quietly gets.


      

        FastMCP inspects your tool function's signature. If it sees a parameter typed as
        Context, it skips that parameter in the JSON Schema (the LLM doesn't know
        about it) and injects the live Context at call time. The Context is your interface back
        into the running server — logging, progress reporting, sampling, elicitation, and access
        to lifespan resources all flow through it. Click any highlighted call to see what it does:
      


      Explore · click a ctx call

      

        

          Context Inspector

from fastmcp import FastMCP, Context

mcp = FastMCP("research-server")

@mcp.tool
async def deep_research(
    query: str,
    ctx: Context,
) -> str:
    """Run a multi-source research pipeline."""

    # 1) Log to the host's debug panel
    await ctx.info(f"Starting research: {query}")

    # 2) Reach into the lifespan-shared HTTP client
    http = ctx.request_context.lifespan_context["http"]
    docs = await http.get(...)

    # 3) Stream progress updates as we work
    for i, doc in enumerate(docs):
        await ctx.report_progress(i, total=len(docs))

    # 4) Ask the LLM (via the client) to summarise
    summary = await ctx.sample(messages=[...], max_tokens=500)

    # 5) Ask the user for confirmation before writing
    confirm = await ctx.elicit("Save these findings?", schema=...)

    # 6) The id of this very request — useful for tracing
    log.info(f"Done · req={ctx.request_id}")

    return summary.text

          

            
Each of the highlighted calls is a different facet of the ctx object. Click any one to see exactly what it does on the wire, what host features it depends on, and the capability flag (if any) the client must have advertised.

          

        

      


      

        
Type-hint, not parameter name

        

          FastMCP injects based on type, not parameter name. ctx: Context,
          c: Context, my_context: Context — all work. The convention is
          ctx, but the framework doesn't care. What it does care about: the
          Context parameter does not appear in the tool's JSON Schema, so
          the LLM never knows it exists. Caller LLM passes only the "real" arguments; FastMCP
          fills in the Context automatically.
        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 2.0

          
Five quick checks before Module 2.1.

        

      


      

        
Q1.You need to expose 200 tools whose definitions are computed from a SQL query at server startup. Which approach fits best?

        

          
          
          
          
        

        
Correct. FastMCP's decorator model assumes tools are known at import time. When that doesn't hold, the low-level SDK's list_tools handler can return any computed list at runtime.

      


      

        
Q2.Why is the lifespan context manager strictly preferred over creating a DB pool inside each tool function?

        

          
          
          
          
        

        
Right. Creating a connection pool inside each tool call defeats the purpose of pooling. Lifespan creates expensive things once, shares them across the server's lifetime, and tears them down cleanly.

      


      

        
Q3.A tool function has signature def fetch(url: str, ctx: Context) -> str. What does the LLM see in the input schema?

        

          
          
          
          
        

        
Yes. FastMCP detects the Context type, removes that parameter from the published schema, and injects the live Context at call time. The LLM never knows it exists.

      


      

        
Q4.Your tool calls await ctx.sample(messages=…). The host responds with an error "sampling not supported by client". Most likely cause?

        

          
          
          
          
        

        
Exactly. Sampling is bidirectional with the request going server→client. The client gates it via its own capabilities at handshake. If the host (Claude Desktop, Cursor) hasn't declared sampling, your ctx.sample() will fail no matter what your server says.

      


      

        
Q5.In a lifespan that creates a DB pool then an HTTP client, what's the correct shutdown order?

        

          
          
          
          
        

        
Right. Reverse-order shutdown ensures that anything still using the DB pool (like an HTTP client mid-call) can finish cleanly before its dependencies disappear underneath it.
