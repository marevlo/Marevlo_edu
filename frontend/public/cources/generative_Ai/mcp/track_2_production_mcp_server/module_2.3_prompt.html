Track 02 — Building Servers / Module 2.3

Prompts: the user-invoked primitive.

Tools are invoked by the LLM. Resources are pulled by the host. Prompts are invoked by the user — typically as slash commands. They're reusable, parameterised templates the server publishes so people can trigger consistent, well-engineered prompts without retyping them. Argument completion is what makes them feel native.

01  Three primitives

Tools, Resources, Prompts — and who invokes each.

MCP servers publish three primitives. They look similar on the wire — all three have list and get endpoints — but the actor that invokes each one is different, and that single difference is what determines when to use each. Tools are picked by the model. Resources are pulled by the application (the host, automatically). Prompts are triggered by the user, typically through a slash menu. Click any column:

› Why this matters: prompts encode expertise

A tool is "I want the model to be able to do X". A resource is "I want the model to see X". A prompt is "I want users to be able to invoke X consistently". The classic example: code review. Every team has a carefully-worded "please review this for security issues, style, and edge cases" prompt floating around in a wiki. A review-code MCP prompt bakes that wording into the server, so every user who types /review-code gets the same well-engineered, version-controlled prompt — not whatever they manage to type into the input box that morning.

The dispatch test

When you're trying to decide which primitive to use, ask: who initiates the action? If the model decides, it's a tool. If the host's logic decides (auto-attaching a file, displaying available context), it's a resource. If a human types it on purpose, it's a prompt. Same wire-level shape, completely different UX implications.

› Why the host (not the LLM) drives this

Prompts are not exposed to the LLM. The model never sees a prompts/list — it can't decide to invoke review-code the way it can decide to invoke search_issues. The slash menu lives entirely in the host UI: the host pulls the catalog via prompts/list, renders it, collects arguments from the user, fetches the rendered messages via prompts/get, and only then feeds them to the LLM as if the user had typed them. This separation is intentional: prompts represent user intent, and the user — not the model — picks them.

Prompts vs system messages

A prompt is not a system message. It's a packaged conversation starter that gets injected into the message list as one or more user / assistant turns. Servers can include a system turn in the messages array, but most don't — the host's existing system prompt usually owns that slot, and prompts compose with it. Think of a prompt as "what the user would have typed, only well-engineered".

from fastmcp import FastMCP
from mcp.types import PromptMessage, TextContent

mcp = FastMCP("code-review")

@mcp.prompt(name="review-code",
            description="Carefully review a code change for bugs, security, and style")
def review_code(language: str,
                diff: str,
                strictness: str = "normal") -> list[PromptMessage]:
    """Ask the model to review code with a consistent rubric."""

    return [
        PromptMessage(
            role="user",
            content=TextContent(
                type="text",
                text=f"""You are reviewing a {language} change.
Strictness: {strictness}.

Please flag:
  - logic bugs and off-by-one errors
  - security risks (injection, auth bypass, secret leakage)
  - style inconsistencies with the surrounding code

Be specific. Quote line numbers. Suggest the smallest fix.
"""),
        ),
        PromptMessage(
            role="user",
            content=TextContent(type="text", text=f"```{language}\n{diff}\n```"),
        ),
    ]

Multi-turn prompts unlock few-shot

A prompt's messages array isn't limited to one user turn. The most powerful prompts include a few user / assistant exchanges before the user's actual request — classic few-shot prompting, but version-controlled and shippable. Servers can also include a resource link as a content block, so a prompt can attach a real file from the workspace as context.

› Implementing completion server-side

FastMCP exposes completion via a per-argument hook. The server is told which argument is being completed and the current partial value; you return up to 100 suggestions. The host throttles requests — usually one per 50 ms of idle typing — so you don't need to worry about every keystroke: 

@mcp.prompt(name="summarise-table")
def summarise_table(database: str, table: str, style: str = "concise"):
    return [PromptMessage("user", TextContent("text",
        f"Summarise table {table} from {database} in a {style} style"))]

@mcp.completion()
async def complete(ref, argument, value, context):
    # ref tells you which prompt/resource is being completed
    # argument.name is the field; value is what the user has typed so far
    # context.arguments has already-filled values (e.g. database when completing table)

    if ref.name == "summarise-table" and argument.name == "database":
        return [db.name for db in CONFIGURED_DBS if db.name.startswith(value)]

    if ref.name == "summarise-table" and argument.name == "table":
        db_name = context.arguments.get("database")
        if not db_name: return []
        return await list_tables(db_name, prefix=value)

    return []
Completion is optional but disproportionately impactful

A prompt without completion forces the user to know exactly what valid arguments look like. A prompt with completion turns the field into IDE-grade autocomplete and the same prompt becomes 5× more usable. Almost zero cost server-side; outsized UX win. Implement completion on any argument that accepts a domain-specific identifier (repo, table, channel, file path).

› What separates a kept prompt from a deleted one

  • It encodes wording the team agreed on once. If everyone on the team types something slightly different to ask for the same thing, that's the candidate prompt.
  • It pulls real context, not just a wrapper. A prompt that takes a repo arg and stitches in the repo's README via a resource link is more useful than one that just adds five lines of fluff.
  • It composes few-shot examples. Multi-turn prompts that include 1–2 example exchanges raise the floor of model output more than any single-turn rewording could.
  • It has completion on every argument. If the user has to type a magic string from memory, the prompt loses to "just typing the question directly".
  • It's discoverable by name. The slash menu shows the name first. Boring, predictable names (review-code, summarise-table) beat clever ones (codescope).
Update notifications

When your server's prompt list changes — you added a new prompt, removed one, updated an argument schema — emit notifications/prompts/list_changed. The host re-fetches the catalog, and users see the new prompts in their slash menu without restarting. Same pattern as tools/list_changed from Module 1.2.

Knowledge Check · End of Module 2.3
Five checks before Module 2.4.
Q1.You're adding a new capability to your server: "let users invoke a well-crafted code-review template by name". Which primitive?
Correct. "Invoke by name" + "explicit user intent" + "consistent wording" = prompt. The dispatch test makes this easy — the user picks it, not the model, so it's a prompt.
Q2.Why doesn't the LLM see the list of available prompts?
Right. The actor distinction is intentional. Tools are picked by the model; resources by the application; prompts by the human. If you wanted the LLM to invoke a template, you'd expose it as a tool that returns text — but then it's a tool, not a prompt.
Q3.Your prompt has a table argument and you want autocomplete in the slash menu. What's the minimum implementation?
Correct. Completion is dynamic — the host calls completion/complete with the partial value, and your handler returns live suggestions. This is what makes the field feel like IDE autocomplete instead of a guessing game.
Q4.A prompt returns a messages array with three entries: user, assistant, user. What's the assistant entry for?
Exactly. Prompts aren't limited to one user turn. Including 1–2 user/assistant pairs before the real request is classic few-shot — except now the few-shot is shippable, version-controlled, and the user gets it for free by typing one slash command.
Q5.You add a new prompt to your running server. What does the spec say about telling connected hosts?
Yes. Each primitive has its own list-changed notification. Tools, resources, and prompts each emit theirs independently. Hosts subscribe and re-pull just the affected catalog — no full reconnect, no polling.
