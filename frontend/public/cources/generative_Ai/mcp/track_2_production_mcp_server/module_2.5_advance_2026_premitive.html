Track 02 — Building Servers / Module 2.5 / 2026 primitives

Advanced 2026 primitives.

Two additions to MCP's surface that shipped through 2026 turn it from a protocol for text-and-tools into something closer to an application platform. MCP Apps let servers return rich, interactive UI panes — rendered by the host in a sandboxed iframe. Tasks turn the implicit "tool calls complete in seconds" assumption into an explicit, durable, cancellable lifecycle for work that takes minutes or hours.

01  Why two new primitives

What text tool results couldn't do.

For two years, every MCP tool returned text (or rarely, an image). That works until your tool wants to do either of these:

  • Return something interactive. A query tool that returns 500 rows as a JSON string forces the LLM to summarise. A query tool that returns a sortable table lets the user explore — without burning context tokens.
  • Take a long time. "Generate the quarterly report" or "rebuild the search index" don't finish in a single tool-call. Modelling them as synchronous tools means timeouts, blocked LLMs, and no way to cancel.

MCP Apps and Tasks are the two new primitives that solve these. They're additive — old tools and resources still work — but the servers that adopt them feel qualitatively different, especially for the data-heavy and ops-heavy use cases of Track 5.

› The shape of each, in one paragraph

An MCP App is server-rendered HTML returned as a new content block type, ui_resource, alongside text and image. The host embeds it in a sandboxed iframe and routes a small set of allowlisted messages between the iframe and the server. A Task is what you create instead of returning from a tool: tasks/create returns a handle in milliseconds; the work runs in the background; the host polls via tasks/get or subscribes to notifications/tasks/update; and the user can call tasks/cancel at any time. Both are new capabilities the client must declare.

Both are opt-in for hosts

A host that doesn't support MCP Apps simply won't declare the ui capability, and servers fall back to rendering text. A host that doesn't support Tasks gets synchronous tools or -32601 on tasks/create. Capability negotiation (Module 1.2) is what makes adding these primitives non-breaking — and what makes today's servers backward-compatible with last year's hosts.

02  MCP Apps

Server-rendered HTML, host-sandboxed.

An MCP App is what you get when you let a server return interactive UI instead of text. The wire shape is small: a tool's response contains a content block of type ui_resource whose payload is an HTML string (or a URL the host fetches). The host wraps that payload in a tightly-scoped iframe. Communication runs over the standard window.postMessage channel, but only for a small set of allowlisted message types the spec defines:

The server-rendered HTML runs here with no ambient access to anything. No cookies, no localStorage of the host's origin, no fetch to host APIs. The iframe sees only a window.mcp JS object the host injects — the only legal channel back to the server.

Only allowlisted message types pass: tool, resource, notify, permission. Anything else is dropped silently.

› Returning an MCP App from a tool

The server side is small. Return a content block with type ui_resource; FastMCP gives you a helper: 

@mcp.tool
async def query(sql: str) -> list[ContentBlock]:
    rows = await db.fetch(sql)

    html = render_table(rows)   # your jinja/string template

    return [
        UIResource(
            mime_type="text/html",
            content=html,
            # declare what the iframe can do — host enforces with sandbox attrs
            permissions=UIPermissions(
                tool_calls=["query", "export_csv"],
                storage=False,
                network=[],
                clipboard_write=True,
            ),
            # preferred size; host may ignore
            preferred_size=UISize(width="100%", height=400),
        )
    ]

The host renders the HTML, hooks up the window.mcp object that lets the iframe call back into the server (only the listed tools), and displays the result inline in the chat. Switch tabs to see three real MCP App examples:

The sandbox is not optional — it's the whole point

A server that returns HTML is remote code execution by definition. What makes MCP Apps safe is the iframe sandbox: no same-origin to the host, no ambient credentials, no top-frame access, no DOM access to the chat window. Hosts that strip the sandbox attributes (or don't enforce them server-side via CSP frame-src) are out of spec and would be a serious exploitation surface. Module 4.3 covers this attack class in detail.

03  Tasks · state machine

Long-running work as a first-class object.

A Task is what happens when a tool call would otherwise take too long. Instead of blocking, tasks/create returns a task_id in milliseconds. The server runs the work in the background, emits progress updates, and the client decides whether to poll, subscribe, or wait. The state machine has five terminal-ish states; drive it below:

› The four task methods

  • tasks/create — server immediately returns a { taskId, status: "created" } handle. Almost no latency; the actual work hasn't started yet.
  • tasks/get — polling. Client asks for the current state of a task by id. Returns status, progress, partial results, or final result. Safe to call repeatedly.
  • notifications/tasks/update — push. Server emits these unsolicited whenever the task transitions or makes progress. Replaces polling; the host subscribes once on capability declaration.
  • tasks/cancel — client requests cancellation. Server should stop work promptly and transition to cancelled. Idempotent; calling on an already-terminal task is fine.
@mcp.task
async def generate_report(quarter: str, ctx: TaskContext) -> dict:
    """Long-running report generation. May take 2-10 minutes."""
    sections = ["revenue", "churn", "cohorts", "forecasts"]

    for i, section in enumerate(sections):
        # Cooperative cancellation — check before each chunk
        if ctx.is_cancelled():
            return {"isError": True, "text": "cancelled by user"}

        # Per-section progress update emits notifications/tasks/update
        await ctx.report_progress(
            progress=i + 1,
            total=len(sections),
            message=f"building {section}…",
        )
        data[section] = await compute_section(quarter, section)

    return {"content": [UIResource(...)]}    # Section 5 composes both
Cooperative cancellation

The runtime can't force-stop your task — it can only set a flag your code checks. ctx.is_cancelled() must be polled at every reasonable opportunity (between sections, between expensive operations, in long loops). A task that doesn't check the flag is a task that can't be cancelled, which is a bug. The 2026 spec's task-cancellation contract: respond to tasks/cancel within 30 seconds with a terminal transition.

› The "look-ahead" trick that justifies Tasks

Here's the subtle one: even when your work fits in a tool, switching to a task unlocks a useful LLM pattern. The model gets back the task_id in milliseconds and can immediately make other tool calls in parallel. When it later cares about the report's progress, it polls tasks/get. Translation: tasks let the LLM parallelise instead of serialising a chain of slow tools. The right answer for "generate report" + "summarise recent issues" + "draft email" might be three concurrent tasks all polled at the end, not three sequential tool calls.

Don't replace fast tools with tasks "for symmetry"

The overhead is real: every task adds a handle to track, a polling cycle, an extra round trip. For sub-second work, that overhead is bigger than the work itself. Reserve tasks for the genuinely long, the genuinely cancellable, or the genuinely parallelisable. Everything else stays a tool.

05  Composing both

A task that returns an app.

The pattern, distilled

Submit work as a Task → emit progress as notifications → return an MCP App as the final result → user gets an interactive dashboard with the option to call follow-up tools (drill-down, export) right from the iframe. This is the archetypal "rich agent output" of 2026: not text-and-citations, but submit-and-explore.

› Backwards compatibility · hosts that don't support either

  • Host has neither capability — your tool degrades to synchronous and returns plain text. Check ctx.client_caps in the tool body and branch.
  • Host has Tasks but not MCP Apps — the task completes, but the final result is text/markdown instead of an iframe. The dashboard render becomes a tabular summary in chat.
  • Host has MCP Apps but not Tasks — the tool blocks like a classic tool, then returns the app. Fine for ≤30 s work; ugly for longer.
Knowledge Check · End of Module 2.5
Five checks before Module 2.6.
Q1.An MCP App's HTML wants to read from the host's document.cookie. What happens?
Correct. No allow-same-origin means no access to anything tied to the host's origin — cookies, localStorage, sessionStorage, the parent DOM. The iframe is an island. The only channel back is window.mcp postMessage, which is what the host injects to broker spec-defined message types.
Q2.You're writing a "rebuild search index" operation that takes 8 minutes and emits useful progress. Tool or task?
Right. The four-test rule of thumb: long, cancellable, progressive, backgroundable. Index rebuild hits all four. Modelling this as a tool means the LLM blocks for 8 minutes, the user can't cancel, and a reconnect loses everything.
Q3.The user clicks Cancel on a running task. Your task code is in a tight loop doing matrix math. What's the spec contract?
Exactly. No runtime can safely force-stop arbitrary user code (open files, half-written DB rows, half-released locks). Cancellation must be cooperative. Sprinkle is_cancelled() checks between expensive operations; if your task can't reach a check in 30 s, you've broken the cancellation contract.
Q4.Your tool returns a ui_resource with tool_calls=["export_csv"] in the permissions block. The iframe's JS tries to call delete_table. What happens?
Yes. The permissions block is the iframe's allowlist. Servers must declare upfront which tools the rendered UI can invoke; anything else is rejected at the bridge. This is what prevents a compromised template from escalating privileges via the chat session.
Q5.You want your "generate report" task to be resumable: if the client disconnects and reconnects, it should pick up where it was. How?
Correct. Resumability is the killer feature of Tasks over Tools. Persist the task_id (the host normally does this for you); on reconnect, the same id retrieves the same state. This is what makes a multi-hour task workable in a flaky-network world.
