01  Retries & backoff

The right way to ask again.

The default failure handler in any decent client is "retry, but with thought". Naive constant-delay retries are worse than no retries: when a server recovers, every client that was waiting hammers it simultaneously, knocking it back down. Exponential backoff spreads the load over time; jitter randomises across clients so they don't all hit at the same instant. The formula: 

delay = min(cap, base * 2 ** attempt) * random(0.5, 1.5)
        └─ exponential ─┘   └─ jitter ─┘

Adjust the knobs below to see how attempts spread across the timeline. Outcomes are deterministic for the demo: the call fails for a fixed number of attempts, then succeeds. The relevant story is the spacing:

› In production, use tenacity

Don't hand-roll the loop. The tenacity library has the right primitives — exponential backoff, max-attempts, retry-only-on-specific-errors — and integrates cleanly with async code. The 6-line wrapper that hardens your MCP client's call_tool: 

from tenacity import retry, stop_after_attempt, wait_exponential_jitter,\\
                      retry_if_exception_type

@retry(
    stop=stop_after_attempt(5),
    wait=wait_exponential_jitter(initial=0.2, max=10),
    retry=retry_if_exception_type((TransportError, TimeoutError, RateLimitError)),
    reraise=True,
)
async def call_tool_resilient(client, name, args):
    return await client.call_tool(name, args)
When NOT to retry — and why it matters

Retries are appropriate only for transient failures. Non-idempotent tool calls (creating a payment, sending an email, deleting a file) must not be retried blindly — a duplicate write that you thought failed but actually succeeded is the canonical source of "we paid you twice". Restrict the retry_if_exception_type set to network / timeout / 5xx errors; let 4xx, validation, and auth errors fail immediately. Many MCP tools also return server-side idempotency keys — pass them and you get safety for free.

02  Circuit breakers

Stop hitting a server that's already on fire.

Retries handle one bad request. A circuit breaker handles a bad server. When failures cross a threshold, the breaker trips — subsequent calls fail instantly instead of waiting for a timeout. After a cooldown, it lets one call through to probe whether the server has recovered. Three states, two transitions. Click any state, or hit the failure / success buttons to drive it:

› The three knobs that matter

  • Failure threshold — how many consecutive failures before tripping. Common values: 3–10. Too low and one bad minute trips you; too high and you keep hammering a dead server.
  • Cooldown / reset timeout — how long OPEN stays OPEN before letting a probe through. Common values: 10–60 seconds. Long enough that the server has time to recover; short enough that the user isn't waiting forever.
  • Probe success threshold — how many consecutive successes in HALF_OPEN are needed to declare recovery. Often 1, sometimes 2–3 for noisy upstreams.
from aiobreaker import CircuitBreaker
from datetime import timedelta

breaker = CircuitBreaker(
    fail_max=5,                              # trip after 5 consecutive failures
    timeout_duration=timedelta(seconds=30), # OPEN → HALF_OPEN after 30s
)

@breaker
async def call_mcp_tool(client, name, args):
    return await client.call_tool(name, args)

# breaker handles "this server is dead", retry handles "this call flaked"
One breaker per server, not per tool

The breaker is about upstream health, not call success. All tools against the same MCP server share one breaker because they share one transport and one server process. If github__create_pr keeps timing out, the breaker should also stop sending github__list_issues — both go through the same wire, and both will fail.

03  Checkpointing & memory

An agent that can resume from where it crashed.

LangGraph's checkpointer persists state after every node. Pass a thread_id on ainvoke and the entire message list, position in the graph, and any custom state is snapshotted on every step. Crash mid-run? Restart and invoke with the same thread_id — the agent picks up at the last checkpoint instead of starting over. Pick a backend below to see when each fits:

› The thread_id pattern

A thread_id is just any string you choose — typically a UUID, a conversation id from your app, or the user's session id. All it does is namespace the checkpoints. Same thread_id across calls = same accumulating conversation. Different thread_id = fresh agent run. This is also how multi-turn chat with persistent memory works — the user sends a follow-up question, your code calls ainvoke({"messages": [(user, ...)]}) with the existing thread_id, and the agent sees the full history automatically: 

from langgraph.checkpoint.postgres import AsyncPostgresSaver

checkpointer = await AsyncPostgresSaver.from_conn_string(POSTGRES_URL)
await checkpointer.setup()  # creates tables on first run

agent = create_react_agent(model="...", tools=tools, checkpointer=checkpointer)

# turn 1 — fresh thread
await agent.ainvoke(
    {"messages": [("user", "open issue 1247 in repo app")]},
    config={"configurable": {"thread_id": "convo_8c4f"}},
)

# turn 2 — same thread, agent sees the prior messages from disk
await agent.ainvoke(
    {"messages": [("user", "who is it assigned to?")]},
    config={"configurable": {"thread_id": "convo_8c4f"}},
)
Checkpointing also enables human-in-the-loop

Because state is durably stored after every node, you can pause an agent — say, before a destructive tool call — wait for a user to approve via a separate API call (minutes or hours later), then resume from exactly that checkpoint. This is the foundation Track 4 will lean on for security gates.

04  OTel tracing

One trace, every hop.

When something is slow or broken, "the agent is slow" isn't a diagnosis. You need to see which span was slow: the LLM call? a specific tool? the database the tool hit? OpenTelemetry traces the request across every boundary — host → LLM provider → MCP client → MCP server → backend — and renders it as a nested span tree. The emerging GenAI semantic conventions standardise attributes like gen_ai.system, gen_ai.request.model, and gen_ai.usage.input_tokens so you can query across providers. Click any span:

› Wiring it up — three packages, three lines

from opentelemetry import trace
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor
from openinference.instrumentation.langchain import LangChainInstrumentor

# auto-instrument every httpx call (covers MCP HTTP transports)
HTTPXClientInstrumentor().instrument()

# auto-instrument every LangChain/LangGraph chain (covers agent + LLM)
LangChainInstrumentor().instrument()

# your own MCP client code: wrap the call_tool with a span
tracer = trace.get_tracer("my-mcp-client")

async def call_tool_traced(client, name, args):
    with tracer.start_as_current_span(
        "mcp.tool.call",
        attributes={
            "mcp.server.name":  client.name,
            "mcp.tool.name":    name,
            "mcp.transport":    client.transport,
        },
    ):
        return await client.call_tool(name, args)
Trace context propagation across MCP

For tracing to span the wire, the traceparent header must flow from client to server. For HTTP transports, OTel's HTTPXClientInstrumentor adds it automatically; the MCP server's HTTP layer reads it and links its spans to the same trace. For stdio transports, the spec recommends embedding the trace context as a custom field in the JSON-RPC envelope — your server SDK can extract it and propagate. The result: one trace_id, every hop.

Each pattern composes with the others. In production code, you'll typically see this order, outer to inner: OTel span wraps a circuit breaker wraps a retry policy wraps the raw client.call_tool. Checkpointing operates at a different layer — the LangGraph runtime — and snapshots the whole agent state regardless of what the inner stack is doing.

Knowledge Check · End of Module 3.3
Five quick checks before Module 3.4.
Q1.Why is naive constant-delay retry worse than no retry at all?
Correct. Synchronised retries from many clients can repeatedly take down a recovering service. Exponential backoff spreads attempts in time; jitter spreads them across clients. Both pieces matter.
Q2.A circuit breaker is in OPEN state for the github MCP server. A new tool call arrives. What happens?
Right. OPEN means "we already know this server is broken; don't waste time and resources trying". After the cooldown elapses the breaker moves to HALF_OPEN and allows one probe — only then is real traffic attempted again.
Q3.You want multi-turn chat: a user sends a message, comes back an hour later, sends a follow-up, and the agent has full memory of the earlier exchange. What's the minimum you need?
Exactly. The checkpointer + thread_id pattern is how multi-turn memory works in LangGraph. Same thread_id across invocations gives the agent its prior context; different thread_id = fresh agent. No manual concat needed.
Q4.Your trace shows an MCP tool call span with http.status_code=200 and a nested LLM span with gen_ai.usage.input_tokens=42000 taking 18 seconds. Most likely diagnosis?
Yes. The MCP span returned 200 OK and the slow span is the LLM span, with a giant token count. The fix is upstream of the LLM — fewer messages in context (summarise), smaller tool results (truncate), or a smaller model on shorter prompts.
Q5.An MCP tool call returns a 400 with "validation error: path must be absolute". What's the right move?
Correct. 4xx errors are deterministic — retrying won't fix bad inputs. The right move is to let the error reach the LLM as a ToolMessage with the validation text, and the LLM will adjust its arguments on the next iteration. Retries are only for transient failures.
