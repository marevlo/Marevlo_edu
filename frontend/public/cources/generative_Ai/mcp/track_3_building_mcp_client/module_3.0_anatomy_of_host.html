01  Host vs client vs server

Three roles. One process owns two of them.

The MCP spec uses three words that read like synonyms in casual writing but are precise in the protocol: host, client, and server. Get these wrong and every diagram you draw later will be subtly wrong with them. The trick is that host and client almost always live in the same process — but they're different roles with different jobs. Click each box to inspect:

The naming is unfortunate — and worth pinning down

In a web context, "client" usually means the user-facing app. In MCP, the user-facing app is the host, and "client" means the small protocol object inside the host that talks to one server. When the spec says "the client must advertise sampling capability", it does not mean Claude Desktop — it means the per-server client object Claude Desktop just instantiated.

› The "process owns roles" lens

A common confusion is treating host/client/server as three separate processes. They often aren't. The host process and the client live in the same memory space — the host imports the client SDK and constructs client objects. The server is the only role that's reliably a separate process (or a separate HTTP service). So the process picture for one user with three servers is: 

# one host process (e.g. Claude Desktop)
host_process:
    host  = ClaudeDesktopApp()
    llm   = anthropic.Anthropic(...)        # owned by host
    ui    = MainWindow()                    # owned by host

    clients = [
        MCPClient(transport="stdio", server="filesystem"),
        MCPClient(transport="http",  server="github"),
        MCPClient(transport="stdio", server="postgres"),
    ]

# three separate server processes (or HTTP endpoints)
fs_server_process     # spawned by host via stdio
github_server_process # remote HTTP service
pg_server_process     # spawned by host via stdio
02  The 1:1:N pattern

One host, many clients, each pinned to one server.

The cardinality matters. Every client object in the host talks to exactly one server. Adding a second server means spawning a second client. That's the 1:1 part. The host owns N of these pairs — that's the :N. Press play to watch a host boot three servers, run a tool call against one of them, and shut down:

Why not one client connected to all servers?

Because servers don't share an address space, a transport, or a security domain. One server speaks stdio over a child process you control. Another speaks HTTP to a remote endpoint with OAuth. A third runs in a Docker container with its own filesystem roots. Pinning one client to one server keeps the wire simple (matched JSON-RPC ids, single transport, predictable lifecycle) and the security boundary clean (one set of capabilities and credentials per server).

› Tool name collisions across servers

Two servers can publish a tool called search. The host sees both. To keep them distinguishable when the LLM picks one, every host namespaces the tool name internally — usually {server_name}__{tool_name} or {server_name}.{tool_name}. The exact convention is a host detail; nothing in the protocol forces it. Two important consequences:

  • The LLM sees the namespaced name in the tool schema, not the bare one the server published.
  • When the LLM picks github__search, the host strips the prefix back to search and routes the JSON-RPC tools/call to the github client only.
  • This is why two servers can both publish search and the host doesn't blow up. It's also why two servers configured under the same name in your config does blow up.

Not every host implements all ten

Sampling and elicitation are the two most commonly skipped. A host that doesn't implement sampling simply doesn't advertise the sampling capability at handshake — any server calling ctx.sample() will receive an error. This is allowed by the spec; capabilities are a contract, not a requirement. The Section 4 host grid shows which hosts implement which.

The "configure once, run anywhere" myth

A server you publish doesn't magically run in every host — each host has its own config file shape, its own way of declaring environment variables, and its own stdio vs. http preference. The protocol is identical across them; the configuration is not. Most server READMEs ship a config snippet per host because of this.

The LLM never sees JSON-RPC

Notice that the LLM lane shows tool names and tool arguments, never JSON-RPC envelopes. The host translates between the two — it serialises the LLM's "I want to call read_file with path=foo.txt" into tools/call wire format, sends it via the right client, deserialises the response, and feeds the content back to the LLM as a tool result message. This is the entire job of a host's agent loop.

› The minimum loop, in 30 lines

Strip away the UI, consent, observability, and you're left with the core agent loop. Module 3.2 will plug this into LangGraph; for now, the shape: 

async def agent_turn(user_msg: str, clients: list[MCPClient]):
    # 1. aggregate tools across every connected server
    tools = []
    for c in clients:
        for t in await c.list_tools():
            tools.append({**t, "_client": c, "name": f"{c.name}__{t['name']}"})

    messages = [{"role": "user", "content": user_msg}]

    while True:
        # 2. ask the LLM what to do, given current tools + history
        resp = await llm.complete(messages, tools=tools)

        if not resp.tool_calls:
            return resp.text                # 7. terminal answer

        for call in resp.tool_calls:
            # 3. consent gate (optional but standard)
            if not await ui.confirm(call):
                result = {"error": "denied by user"}
            else:
                # 4. route to the right client by stripped name
                server_name, tool_name = call.name.split("__", 1)
                client = next(c for c in clients if c.name == server_name)
                result = await client.call_tool(tool_name, call.arguments)

            # 5. feed result back into the message log
            messages.append({"role": "tool", "id": call.id, "content": result})

        # 6. loop continues — LLM may issue more tool calls or finalise
Knowledge Check · End of Module 3.0
Five quick checks before Module 3.1.
Q1.A user has Claude Desktop open with 3 MCP servers configured. How many client objects exist in the host process?
Correct. The 1:1:N pattern: one host process, N client objects, each bound to exactly one server. Adding a fourth server means instantiating a fourth client.
Q2.A server tool calls ctx.sample(...) and gets an error "sampling not supported". Where does the responsibility for advertising sampling actually live?
Right. Sampling is a server-to-client request: the server asks the host to run an LLM completion. Only the host knows whether it owns an LLM it's willing to share, so the host declares the capability through its client during initialize.
Q3.Two configured servers both publish a tool called search. The LLM asks to call search. What does a well-behaved host do?
Yes. Tool name collisions across servers are expected. The host namespaces during aggregation, the LLM picks the namespaced one, and the host strips the prefix to route. The protocol itself is silent on the exact separator (__ vs . vs :) — that's a host choice.
Q4.Which of these is not a host responsibility?
Correct. The server publishes the tool's input schema as part of tools/list — that's a server responsibility, not the host's. The host only consumes those schemas and feeds them to the LLM.
Q5.You're switching from Claude Desktop to Cursor. Your existing server keeps working without changes. Why?
Exactly. The whole point of MCP is host-agnosticism. Server code is portable across every host that speaks the protocol. Only the host's config file changes — the JSON-RPC over the wire is identical, and the server is unaware of which host launched it.
