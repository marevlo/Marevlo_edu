01  ReAct in 60 seconds

Three nodes, one loop, that's the whole pattern.

ReAct — short for "Reason + Act" — is the agent pattern that won. Before ReAct, chain-of-thought was monolithic: the model thought, then maybe acted, then was done. ReAct interleaves: think a little, do a little, observe, think again, repeating until the task is solved or the model declares it's done. It's the skeleton inside Claude Code, Cursor's agent mode, Cline, and the LangGraph create_react_agent primitive. Click each phase:

The ReAct Loop · Thought → Action → Observation
The terminating condition

ReAct doesn't loop forever — the LLM decides when to exit. After observing the tool result, it can either propose another action (the thought → action arc fires again) or simply respond with a normal text completion that has no tool_calls. The host treats the absence of tool_calls as the signal to stop and return that text to the user. Module 3.3 will add a hard step limit on top of this as a safety net.

› Mapping ReAct onto OpenAI/Anthropic tool-calling

Modern provider APIs do most of ReAct for you. The model emits a structured tool_use (Anthropic) or tool_calls (OpenAI) block when it wants to act; the host runs the tool; the result goes back as a tool_result / tool message. The provider handles formatting the "Thought" implicitly — you don't see "Thought:" prefixes anymore. The pattern is the same; the surface is cleaner.

02  LangGraph state machine

A graph with two real nodes and one decision.

LangGraph models an agent as a directed graph over a shared state. For a basic ReAct agent there are just two real nodes — agent (calls the LLM) and tools (runs whatever the LLM asked for) — plus the synthetic START and END. The interesting edge is the conditional one out of agent: if the LLM produced tool_calls, route to tools; otherwise, route to END. Click any node or edge:

› The state object that flows through

Every node receives the current state, returns a partial update, and LangGraph merges it. For a chat agent the state is essentially the message list (plus whatever you want to track). The default MessagesState uses an additive reducer — new messages are appended, not replaced — which is what lets the loop accumulate context across iterations: 

from langgraph.graph import MessagesState
from langchain_core.messages import AnyMessage

# MessagesState is just this, pre-defined
class MessagesState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]
    #                                         ↑ the additive reducer

# nodes return partial updates that are MERGED into state
def agent_node(state: MessagesState):
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}   # appended, not replaced

Steps 4–5 collapse into create_react_agent

LangGraph ships a prebuilt create_react_agent(model, tools) that builds the exact graph in Section 2 — agent node, ToolNode, conditional edge — in one call. For 95% of agents that's all you need. Roll your own graph only when you want a non-standard topology (a planner node, parallel branches, guardrails, a critic loop, etc.).

› The whole bridge, twelve lines

from langchain_mcp_adapters.client import MultiServerMCPClient
from langgraph.prebuilt import create_react_agent

client = MultiServerMCPClient({
    "filesystem": {"command": "npx", "args": [...], "transport": "stdio"},
    "github":     {"url": "https://...",      "transport": "streamable_http"},
})

tools = await client.get_tools()                       # aggregated, namespaced

agent = create_react_agent(                                # the whole graph
    model="anthropic:claude-sonnet-4-5",
    tools=tools,
)

result = await agent.ainvoke({"messages": [("user", "summarise /tmp/notes.md")]})
print(result["messages"][-1].content)
04  Message state walk

Watch the list grow, one step at a time.

The clearest way to understand a LangGraph agent is to step through its message state. Each iteration appends one or more messages: a HumanMessage from the user, an AIMessage from the LLM (possibly carrying tool_calls), then one ToolMessage per tool call. The loop continues until an AIMessage arrives with no tool_calls — that's the terminal response. Step forward to watch:

Why messages are appended, not replaced

The LLM has no memory between calls. Each llm.invoke() sees only what's in the messages list, so the list is the agent's working memory. That's why MessagesState uses an additive reducer — replacing messages would amnesia the agent. It's also why long agent runs eventually hit context-window limits and need summarisation or truncation (Module 3.3 covers that).

› The shape of each message type

LangChain unifies provider message shapes into AnyMessage subclasses. The four you'll meet 99% of the time:

  • HumanMessage — input from the user. content: str.
  • AIMessage — output from the LLM. content: str + optional tool_calls: list[ToolCall]. If tool_calls is empty, this is the terminal answer.
  • ToolMessage — result of one tool execution. content: str | list[ContentBlock], tool_call_id: str (matches the AIMessage's tool_call). One per call; if the AI requested 3 tool calls, you get 3 ToolMessages.
  • SystemMessage — agent persona / instructions. Usually first in the list, never appended again.

Streaming, not blocking, in production

Real agents stream rather than block on ainvoke. LangGraph's astream_events() emits an event per node entry/exit and per token inside an LLM call — exactly what you wire into a UI to show the agent "thinking", "calling read_file", "summarising". The graph and message progression above is identical; you're just observing each step as it happens instead of waiting for the final state.

› Streaming, in production

async for event in agent.astream_events(
    {"messages": [("user", query)]},
    version="v2",
):
    kind = event["event"]
    if kind == "on_chat_model_stream":
        # token-by-token streaming from the LLM
        print(event["data"]["chunk"].content, end="")
    elif kind == "on_tool_start":
        render_tool_call(event["name"], event["data"]["input"])
    elif kind == "on_tool_end":
        render_tool_result(event["name"], event["data"]["output"])
Knowledge Check · End of Module 3.2
Five quick checks before Module 3.3.
Q1.A LangGraph ReAct agent loops until what condition?
Correct. The agent's own AIMessage decides the loop's fate. If it has tool_calls, the conditional edge routes to the tools node; if not, to END. Step-limit safeguards (covered in Module 3.3) are layered on top as a safety net, but the natural terminator is the LLM choosing not to call more tools.
Q2.Why does MessagesState use an additive reducer (add_messages) rather than letting each node overwrite the list?
Right. Every iteration of the loop, the agent node calls the LLM with the full messages list. If a node replaced the list instead of appending, the agent would lose all context from prior iterations. Additive reducers are what make multi-step reasoning possible across stateless LLM calls.
Q3.You called client.get_tools() on a MultiServerMCPClient wired to filesystem + github servers. What do you get back?
Exactly. That's the whole point of the adapter: it bridges MCP's tool representation into LangChain's native one. The returned objects subclass BaseTool and integrate transparently with any LangChain/LangGraph primitive that expects tools.
Q4.An AIMessage in the state has tool_calls = [{id: "call_42", name: "read_file", args: {...}}]. After the tools node runs, what gets appended to the messages list?
Yes. The pairing of AIMessage.tool_calls[i] ↔ ToolMessage.tool_call_id is how the LLM correlates which result belongs to which call. When the agent node runs again, it sees this pairing in the messages and incorporates the result into its next decision.
Q5.You want to render the agent's progress in a UI — token-by-token streaming for the LLM, and live updates as each tool fires. Which LangGraph method?
Correct. astream_events is the streaming surface — on_chat_model_stream for tokens, on_tool_start/on_tool_end for tool execution, on_chain_start/on_chain_end for node boundaries. The graph and message progression are identical to ainvoke; you just observe instead of waiting.
