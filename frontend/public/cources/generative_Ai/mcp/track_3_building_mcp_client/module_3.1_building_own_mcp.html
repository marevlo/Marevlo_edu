01  The client stack

Five layers, bottom to top.

A client object isn't one thing — it's a stack. From the byte stream on the wire up to the typed call await client.call_tool("read_file", {...}), five layers are doing work. Knowing where each layer sits is how you find the bug fast when a tool call fails: is it the socket, the JSON-RPC framing, the session state, the capability negotiation, or the host's aggregation? Why this matters for debugging

When a tool call fails, the error you see depends on which layer broke. Connection refused → transport. JSON decode error → framing. "Request id 7 has no matching future" → session. "Server doesn't support sampling" → capability. "Unknown tool github__foo" → aggregation. Diagnosing by layer cuts debug time in half.

› Streamable HTTP vs the older HTTP+SSE

Two HTTP transports are in the wild. The original HTTP+SSE transport (spec revision 2024-11-05) used two endpoints: a POST endpoint for client→server requests and a long-lived GET /sse for server→client streaming. The newer Streamable HTTP transport (2025-03-26 onward) collapses both onto a single endpoint that may optionally upgrade to SSE on a per-request basis. New servers should use Streamable HTTP; many existing servers still expose the legacy SSE endpoint for backward compatibility, so your client will likely meet both in production.

The id-future map is the heart of the client

        

          The thing every from-scratch implementation gets wrong first is correlating
          responses back to callers. JSON-RPC requests fly out with monotonic ids; responses
          come back asynchronously, possibly out of order, interleaved with unsolicited
          notifications. The client must keep a pending: dict[int, asyncio.Future],
          resolve each future when its matching response arrives, and tear it down on
          timeout or transport drop. Get this loop right and the rest is bookkeeping.
        

      


      
› The full 60-line client

      

        Pulled together, the entire bare client fits in one screen. Read it once — Module
        3.3 will harden it with retries and circuit breakers, but the shape stays:
      


      
import asyncio, httpx, json, itertools
from typing import Any

class MCPClient:
    def __init__(self, url: str, headers: dict | None = None):
        self._url = url
        self._http = httpx.AsyncClient(headers=headers or {}, timeout=30)
        self._ids = itertools.count(1)
        self._pending: dict[int, asyncio.Future] = {}
        self._session_id: str | None = None
        self._sse_task: asyncio.Task | None = None

    async def request(self, method: str, params: dict | None = None) -> Any:
        rid = next(self._ids)
        fut: asyncio.Future = asyncio.get_event_loop().create_future()
        self._pending[rid] = fut
        body = {"jsonrpc": "2.0", "id": rid, "method": method, "params": params or {}}
        headers = {"Accept": "application/json, text/event-stream"}
        if self._session_id:
            headers["Mcp-Session-Id"] = self._session_id

        resp = await self._http.post(self._url, json=body, headers=headers)
        # first initialize response carries the session id back
        if sid := resp.headers.get("mcp-session-id"):
            self._session_id = sid

        ct = resp.headers.get("content-type", "")
        if "event-stream" in ct:
            # server upgraded the response to SSE — parse events
            async for line in resp.aiter_lines():
                if line.startswith("data: "):
                    self._dispatch(json.loads(line[6:]))
        else:
            self._dispatch(resp.json())

        return await fut

    def _dispatch(self, msg: dict):
        if "id" in msg and msg["id"] in self._pending:
            fut = self._pending.pop(msg["id"])
            if "error" in msg:
                fut.set_exception(RuntimeError(msg["error"]["message"]))
            else:
                fut.set_result(msg.get("result"))
        else:
            # notification with no id — log to host's debug channel
            self._on_notification(msg)

    async def initialize(self):
        result = await self.request("initialize", {
            "protocolVersion": "2025-06-18",
            "capabilities": {"sampling": {}, "roots": {"listChanged": True}},
            "clientInfo": {"name": "my-client", "version": "0.1"}
        })
        # initialized is a notification (no id, no response)
        await self._http.post(self._url,
            json={"jsonrpc": "2.0", "method": "notifications/initialized"},
            headers={"Mcp-Session-Id": self._session_id})
        return result

    async def list_tools(self):  return (await self.request("tools/list"))["tools"]
    async def call_tool(self, name, args):
        return await self.request("tools/call", {"name": name, "arguments": args})

› The MultiClient shape

      

        The aggregator is roughly 40 lines on top of MCPClient. The two
        interesting bits are parallel fan-out at startup and the reverse routing on a tool
        call:
      


      
class MultiClient:
    def __init__(self, servers: dict[str, MCPClient]):
        self.clients = servers
        self._index: dict[str, tuple[str, str]] = {}  # ns_name → (server, real_name)

    async def connect_all(self):
        # fan-out: every server initializes in parallel
        await asyncio.gather(*(c.initialize() for c in self.clients.values()))

    async def aggregate_tools(self) -> list[dict]:
        # fan-out: list tools across every server in parallel
        results = await asyncio.gather(
            *(c.list_tools() for c in self.clients.values()),
            return_exceptions=True,
        )
        merged = []
        for (srv_name, _), tools in zip(self.clients.items(), results):
            if isinstance(tools, Exception):
                continue  # server flaked, skip — health monitor catches it (Section 4)
            for t in tools:
                ns_name = f"{srv_name}__{t['name']}"
                self._index[ns_name] = (srv_name, t["name"])
                merged.append({**t, "name": ns_name})
        return merged

    async def call_tool(self, ns_name: str, args: dict):
        srv_name, real_name = self._index[ns_name]
        return await self.clients[srv_name].call_tool(real_name, args)


      

        
Why return_exceptions=True matters

        

          With three servers connected, one of them being down should not block the
          aggregate from returning the tools from the other two. asyncio.gather's
          default behaviour — raise on first exception — would make a single flaky server
          poison the whole tool list. return_exceptions=True turns failures
          into values you can filter out. The health monitor (next section) is what then
          flags and surfaces the failure.
        

      

    


    
    

      
04  Health monitoring

      
Three signals, three states, no surprises.


      

        Once a client juggles multiple servers, "is it working?" stops being a yes/no
        question. The answer is per server, over time, and it changes
        independently. A production client tracks three signals per server:
        ping latency (response time to a lightweight method),
        error rate (rolling % of failed calls), and last
        success (when did anything work). Together they classify each server as
        ok, degraded, or down:

› Sliding-window latency & error rate

      

        Per-call timing is too noisy to drive decisions. A rolling window (typically the
        last 20 calls or last 60 seconds, whichever yields more samples) gives you the
        signal a degradation detector can act on. The shape:
      


      
from collections import deque
from time import monotonic

class HealthTracker:
    def __init__(self, window: int = 20):
        self.latencies: deque[float] = deque(maxlen=window)
        self.outcomes: deque[bool] = deque(maxlen=window)
        self.last_success: float | None = None

    def record(self, latency_ms: float, ok: bool):
        self.latencies.append(latency_ms)
        self.outcomes.append(ok)
        if ok: self.last_success = monotonic()

    @property
    def status(self) -> str:
        if not self.outcomes: return "unknown"
        err_rate = 1 - (sum(self.outcomes) / len(self.outcomes))
        p95 = sorted(self.latencies)[int(len(self.latencies) * 0.95)]
        if err_rate > 0.5:               return "down"
        if err_rate > 0.1 or p95 > 2000:  return "degraded"
        return "ok"

    async def ping_loop(self, client: MCPClient, interval: float = 5.0):
        while True:
            t0 = monotonic()
            try:
                await client.request("ping")            # spec-defined no-op
                self.record((monotonic() - t0) * 1000, ok=True)
            except Exception:
                self.record((monotonic() - t0) * 1000, ok=False)
            await asyncio.sleep(interval)


      

        
The spec-defined ping

        

          MCP defines a ping method (no params, empty result) that any
          compliant server must answer. It's free to call, doesn't trigger side effects,
          and roundtrips through the same JSON-RPC plumbing as real calls — which is
          exactly what you want from a health probe. Don't invent your own; use
          ping.
        

      

    


    
    

      
05  LangChain vs raw SDK

      
The same client, two ways. Pick by trade-off.


      

        Almost no one ships a 60-line raw httpx client in production — partly
        because the official mcp Python SDK already wraps everything you saw,
        and partly because most agent stacks prefer to integrate at a higher level. The
        two real options:
      


      
    
        
  • Official mcp SDK — first-party Python SDK with a typed Client class, full Streamable HTTP support, stdio support, and session lifecycle baked in. ~10x less code than from-scratch.
    • 
        
  • langchain-mcp-adapters — third-party adapter on top of the official SDK. Returns LangChain BaseTool objects ready to drop into a LangChain or LangGraph agent. Loses some raw-protocol access but gains a one-line integration with the LangChain ecosystem.

› When to pick which

      
    
        
  • From-scratch (Sections 2–4 of this module): when you're embedding MCP into a larger app that already has its own HTTP layer, or when the official SDK doesn't expose something you need. Rare. Mostly an educational exercise.
    • 
        
  • Official SDK: the default. Use whenever you're building a custom host or agent that isn't already on LangChain. Full protocol access, typed API, maintained by Anthropic and the MCP working groups.
    • 
        
  • LangChain adapter: when your agent loop is already LangGraph or LangChain. The adapter gives you native BaseTool objects and a MultiServerMCPClient with the aggregation you saw in Section 3, but skip-shipped. This is what Module 3.2 will use to bridge MCP into ReAct/LangGraph.
    • 
      


      

        
Don't write the wire by hand in production

        

          The from-scratch client in Section 2 is here to make the protocol concrete, not
          to ship. The official SDK handles edge cases you haven't thought of yet —
          reconnects on stream drop, content negotiation between JSON and SSE responses,
          session id rotation, capability re-negotiation on version change. Use it.
        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 3.1

          
Five quick checks before Module 3.2.

        

      


      

        
Q1.A client sends three requests with ids 1, 2, 3. Responses arrive in order 2, 3, 1, interleaved with a notification. What data structure makes the correlation safe and clean?

        

          
          
          
          
        

        
Correct. JSON-RPC responses are correlated by id, and they can arrive in any order, interleaved with notifications that have no id. The pending: dict[int, Future] pattern is what makes the client safe under concurrent callers without serialising them.

      


      

        
Q2.Your MultiClient.aggregate_tools() calls asyncio.gather(*list_tools_coros). One of the three servers is down. With default gather behaviour, what happens?

        

          
          
          
          
        

        
Right. Default asyncio.gather raises on first exception. To get the partial-success behaviour you want, pass return_exceptions=True and filter the results — the health monitor then flags the failed server separately.

      


      

        
Q3.You hit the POST /mcp endpoint and get back Content-Type: text/event-stream instead of application/json. What does this mean and how do you handle it?

        

          
          
          
          
        

        
Yes. In Streamable HTTP, the server decides per-response whether to reply with a single JSON object or upgrade to an SSE stream — useful when the server wants to send notifications (progress, logs) alongside the actual response. Clients must handle both content types.

      


      

        
Q4.Two servers each publish a tool called search. Your aggregator namespaces them to fs__search and github__search. The LLM calls github__search. What does MultiClient.call_tool have to do?

        

          
          
          
          
        

        
Exactly. The namespace is a client-side construction — the github server published a tool literally called search and that's all it knows. The client adds the prefix when exposing to the LLM and strips it when calling. The server never sees the namespaced form.

      


      

        
Q5.You're integrating MCP into an existing LangGraph agent. Which library gives you the fastest path?

        

          
          
          
          
        

        
Correct. langchain-mcp-adapters is the right abstraction when your agent loop is already LangChain/LangGraph. It returns ready-to-bind tool objects and handles multi-server aggregation for you — which is exactly what Module 3.2 will plug into a ReAct loop.
