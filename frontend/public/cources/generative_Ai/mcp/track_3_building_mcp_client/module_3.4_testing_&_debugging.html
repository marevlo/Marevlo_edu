Track 03 — Clients & Hosts / Module 3.4

Testing & debugging.

The last teaching module of Track 3. We've built clients, wired them into agents, and hardened the runtime. Now we make it debuggable. The official Inspector, raw JSON-RPC replay, a conformance suite wired into CI, and an in-memory test client are the four tools that turn "it works on my machine" into "it's covered". The capstone is next.

01  MCP Inspector

The web UI every MCP developer keeps open.

The MCP Inspector is the official debugging UI for MCP servers — a local web app that connects to any server (stdio or HTTP) and exposes every primitive through a clickable interface. List tools, invoke them with arbitrary arguments, browse resources, see notifications as they arrive, watch the log stream. One command to launch: 

npx @modelcontextprotocol/inspector # launches at http://localhost:6274

# or aimed at a specific server:
npx @modelcontextprotocol/inspector uvx mcp-server-postgres
npx @modelcontextprotocol/inspector --url https://api.example.com/mcp

Which pane for which bug

Tool not in list? Tools pane — checks the catalog itself. Tool throws on invocation? Tools pane → invoke with the exact args your client is sending. Resource URI 404s? Resources pane. Server emits unexpected notifications? Notifications pane. Server-side stack traces? Logging pane (server must call ctx.error() for them to surface). Connection itself flaky? Ping pane.

02  JSON-RPC replay

Read the wire. Then read it again.

The Inspector is great for "does it work". For "why didn't it work", you go one level lower — to the raw JSON-RPC frames between client and server. The Inspector exposes these in a built-in network pane; if you're not running it, every official SDK can dump them via a MCP_DEBUG=1 env flag. Either way, you end up reading something like this. Click any row to see the full frame and what it tells you:

› Replay any frame with curl

For HTTP servers, every frame is reproducible from a terminal. The Mcp-Session-Id header is the only sticky state; once you've grabbed it from the initialize response, every subsequent request can be reissued by hand. This is the lowest-level debugging gesture in MCP: 

curl -X POST https://your.mcp.server/mcp \\
  -H 'Content-Type: application/json' \\
  -H 'Accept: application/json, text/event-stream' \\
  -H 'Mcp-Session-Id: 8c4f-7a21-2046' \\
  -d '{"jsonrpc":"2.0","id":99,"method":"tools/call",
       "params":{"name":"query","arguments":{"sql":"SELECT 1"}}}'
JSON-RPC error codes you'll see

MCP inherits JSON-RPC 2.0's error envelope. Memorise the codes: -32700 parse error · -32600 invalid request · -32601 method not found (your client called an unknown method) · -32602 invalid params (your args don't match inputSchema) · -32603 internal error (uncaught exception in the server). The -32000 to -32099 range is reserved for application-defined errors and is where MCP servers put domain-specific failures.

03  Conformance suite

Does your server pass the spec?

Manual testing finds the bugs you thought to look for. A conformance suite finds the ones you didn't. The suite is a fixed set of assertions about how a compliant MCP server behaves: it must respond to initialize with a protocol version, it must reject unknown methods with -32601, it must validate tools/call arguments against the inputSchema, and so on. You run it as part of CI and it fails the build if your server drifts from the spec. Click any test:

› Conformance as code · a runnable example

The whole point of conformance tests is they're just pytest. Drop the file in your repo, point it at any MCP server, and you get a pass/fail row per spec clause. The Python ecosystem has a few of these — the snippet below shows the pattern for the simplest one (protocol version negotiation): 

import pytest
from mcp import ClientSession
from mcp.client.stdio import stdio_client, StdioServerParameters

@pytest.fixture
async def session():
    params = StdioServerParameters(command="uvx", args=["my-mcp-server"])
    async with stdio_client(params) as (r, w):
        async with ClientSession(r, w) as s:
            yield s

async def test_initialize_returns_protocol_version(session):
    result = await session.initialize()
    assert result.protocolVersion in {"2025-06-18", "2025-03-26", "2024-11-05"}

async def test_unknown_method_returns_minus_32601(session):
    await session.initialize()
    with pytest.raises(Exception) as exc:
        await session.send_request("this/method/does/not/exist", {})
    assert "-32601" in str(exc.value)

async def test_bad_args_returns_minus_32602(session):
    await session.initialize()
    with pytest.raises(Exception) as exc:
        await session.call_tool("query", {"sql": 42})  # wrong type
    assert "-32602" in str(exc.value)
Wire it into CI on day one

The cost of running conformance tests on every PR is seconds; the cost of shipping a spec-violating server to hosts that fall over for inscrutable reasons is hours of bug-hunting. Make this part of the matrix: lint → unit tests → conformance suite → integration tests. The order matters too — a server that violates spec fails fast before downstream tests waste cycles.

04  Unit testing

Two test surfaces, two trade-offs.

For your own server code, there are two ways to write tests. The in-memory approach uses FastMCP's Client hooked directly to the FastMCP instance — no subprocess, no pipes, no transport. Tests run in milliseconds and parallelise freely. The subprocess approach spawns the real server binary over stdio (or hits a real HTTP endpoint) — slower, but exercises the actual deployment path. You want both. Toggle:

› The 80 / 20 split that works

      
    
        
  • 80% in-memory unit tests — one per tool, one per resource, plus tests for any shared lifespan logic and Context-using behaviour. Fast, parallel, no external dependencies.
    • 
        
  • 15% subprocess integration tests — a few end-to-end runs that prove the actual transport works (stdio pipes connected, HTTP routes resolved, session id roundtrips). These are slower and live in a separate pytest mark so you can skip them locally.
    • 
        
  • 5% conformance tests — Section 3's suite, executed as a regular pytest job in CI against your subprocess server. Guards against silent spec drift between SDK upgrades.
    • 
      


      

        
The in-memory pattern is unique to FastMCP

        

          The trick is that FastMCP's Client can take either a transport
          spec or a FastMCP instance directly. When you pass the instance,
          tools dispatch via in-process function calls — same code path as the real
          server uses internally, just skipping the JSON-RPC layer. The lower-level
          mcp SDK doesn't expose this; if you're testing a low-level server,
          you spin up a memory stream pair manually. Either way, no subprocess required.
        

      

    


    
    

      
05  Debug decision tree

      
Symptom in, layer out.


      

        When a tool call fails in production, the symptom is rarely "the LLM didn't get
        the right answer" — it's a stack trace, an error code, a missing response.
        Walking from the symptom down to the broken layer is what experienced MCP
        debuggers do automatically. This widget is that walk, made explicit. Pick a

The five-layer mental model

        

          From Module 3.1: Transport (pipes, sockets), Framing
          (JSON-RPC envelope), Session (id correlation, pending map),
          Capability (handshake, version), Aggregation
          (namespacing, routing). Every bug lives in exactly one of these. Identifying
          which one — usually within a minute, once you have practice — is the entire
          first half of debugging. The second half is just fixing the obvious thing once
          you're in the right layer.
        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 3.4

          
Five quick checks before the Track 3 capstone.

        

      


      

        
Q1.You connect your server with the MCP Inspector. Tools pane shows 0 tools, but logs show the server started cleanly. Most likely cause?

        

          
          
          
          
        

        
Correct. Empty tools pane = server returned an empty tools list. Two likely causes: no tool decorators executed (a missing register(mcp) import in your server.py) or capability not advertised. Open the Notifications pane after restarting to confirm no tools/list_changed ever fires.

      


      

        
Q2.A captured JSON-RPC frame from server to client reads: {"jsonrpc":"2.0","id":7,"error":{"code":-32602,"message":"sql must be a string"}}. What broke?

        

          
          
          
          
        

        
Right. -32602 is JSON-RPC's standard "invalid params" code. The server is rejecting the arguments because they don't match the tool's declared inputSchema. The LLM passed a non-string where a string was required — fix is upstream of the call.

      


      

        
Q3.You're running your test suite. The in-memory tests take 0.4s total; the subprocess integration tests take 18s for the same number of cases. Why is the gap so large?

        

          
          
          
          
        

        
Yes. Process spawn + transport setup + handshake is hundreds of milliseconds; in-memory skips all of that. This is exactly why the 80/20 split exists — most assertions don't need a real transport to verify the logic, so unit tests run in-memory and the integration tier covers the transport itself.

      


      

        
Q4.Your conformance test test_unknown_method_returns_minus_32601 fails because the server returned -32603 instead. What does this tell you?

        

          
          
          
          
        

        
Exactly. Error code precision matters because clients route on the code. A client may consider -32603 transient and retry it, while -32601 is permanent. Returning the wrong code makes the server work for some clients and silently fail for others — the conformance suite's job is to catch this before deployment.

      


      

        
Q5.Your aggregated tool list shows fs__read_file but the LLM calls read_file and gets an error. Which layer of the stack is broken?

        

          
          
          
          
        

        
Correct. The aggregation layer in your client is the only place that knows about the fs__ prefix. If the LLM never sees the prefix, the aggregator never gets a chance to reverse-look-up which server to route to. Check whether your tools are passed to llm.bind_tools() with their namespaced names — that's where the LLM learns what they're called.
