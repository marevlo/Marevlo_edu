Track 01 — The Protocol / Module 1.1

Lifecycle & capability negotiation.

Every MCP session runs the same five-act play: connect, propose, agree, talk, disconnect. The protocol's most-asked questions — "why didn't my server's tools show up?", "why does the host say my version is unsupported?" — all collapse into a misread of one of those acts. This module turns the play into muscle memory.

01  The four phases of a connection

A session is a state machine — and it's small.

Every MCP session moves through four states. Two are transitional (you don't stay in them long); two are stable (most of the connection's life happens there). The rules about what messages are legal change at every transition — and most production MCP bugs come from sending a message in the wrong state. Click any state below to see what's allowed.

The rule that breaks the most servers

Between the server's initialize response and the client's notifications/initialized, the connection is still in the Initializing phase. If your server pushes notifications during that window — say, an eager tools/list_changed — most clients will drop them on the floor. Wait for initialized before volunteering anything.

02  The initialize handshake

Three messages. Order is non-negotiable.

The handshake is three messages with strict ordering: initialize (request), initialize response (success), notifications/initialized (notification). Skipping or reordering any of them puts the session in an undefined state. Watch it unfold:

› Why three messages, not two?

The third message — notifications/initialized — looks redundant. The client already saw the server's response; why send another packet to say "yes, really"? Because until the client acknowledges, the server doesn't know whether the client accepted its proposed protocol version. The server may have downgraded to an older version it supports; the client gets to decide whether to continue or disconnect. The initialized notification is the client's "yes, I'm proceeding under these terms" signal — and only then is the server permitted to push unsolicited traffic.

03  Version negotiation

Date-string versions, and what happens when they don't match.

MCP doesn't use semver. Versions are date strings — 2024-11-05, 2025-03-26, 2025-06-18 — corresponding to spec freeze dates. The client proposes one in initialize; the server either accepts it or counters with a version it actually supports. The client then accepts the counter-proposal or terminates. › Why date strings, not semver?

Semver implies a stable contract with major/minor/patch semantics. MCP's evolution has been choppier than that — features are added, removed, renamed across revisions (batching is gone, OAuth landed, transports were redesigned). A date string is more honest: "this is the spec as it stood on this date." It also avoids the perennial argument about whether a particular change qualifies as breaking.

04  Capability negotiation

An intersection, not a union.

Versions are an agreement about which spec. Capabilities are an agreement about which features within that spec. Both sides declare what they implement. A feature only works if both sides agree — server-side features need the server to advertise them; server-to-client requests (sampling, elicitation, roots) need the client to advertise them. Direction matters

A common mistake: assuming all capabilities are server-advertised. The three client capabilities (roots, sampling, elicitation) unlock messages the server sends to the client. If the client doesn't advertise sampling, the server's sampling/createMessage request will be refused — no matter what the server itself supports.

06  Shutdown & sessions

The protocol doesn't have a "goodbye" message. The transport carries it.

Look hard at the MCP spec and you won't find a shutdown method. There's a philosophical reason — the protocol intentionally defers connection lifetime to the transport, because how you close is fundamentally different between stdio (close a pipe) and HTTP (close a connection). The two procedures look like this:

stdio
Subprocess shutdown
1
Client closes stdin. Sends EOF — the server's read loop unblocks with end-of-stream.
2
Server drains. Finishes any in-flight requests, optionally emits a final notifications/cancelled for the ones it abandons.
3
Server closes stdout and exits with status 0.
4
Client SIGTERMs if step 3 doesn't happen within ~5 seconds. SIGKILL after another 5. Don't be polite forever — hung subprocesses leak.
Streamable HTTP
Network shutdown
1
Client closes the active POST/SSE connections. No message — TCP FIN is the signal.
2
Server detects disconnect via the framework's connection-close hook (FastMCP, asyncio, etc.).
3
Server releases session resources — open files, DB transactions, any state tied to that session id.
4
Server keeps running for the next client. Unlike stdio, the process outlives the connection.
The 2026 stateless turn

The 2026 spec leans into the fact that HTTP servers shouldn't really have sessions at all. Instead of the server tracking "this client id has these pending operations", state moves to explicit handles — opaque tokens the server returns, which the client passes back on every related request. Reconnects become trivial; load balancers stop needing sticky sessions; horizontal scaling stops being a wishful thinking exercise. We'll lean on this all through Track 5.

Knowledge Check · End of Module 1.1
Five quick checks before Module 1.2.
Q1.A server receives tools/call just after it sent its initialize response, but before notifications/initialized arrived. What's the correct behaviour?
Right. Until notifications/initialized arrives, only initialize and ping are legal. Anything else should be refused with an error so the client knows it broke the protocol.
Q2.Client proposes 2025-06-18. Server responds with 2025-03-26. What happens next?
Correct. Versions are a proposal. The server can counter with the highest it supports, and the client gets the final call. If it proceeds, it must stay within the older version's feature set.
Q3.Server wants to call sampling/createMessage (ask the client to invoke the LLM). What's the precondition?
Yes. Sampling is a server-to-client request — and like all server→client capabilities, it requires the client to opt in by declaring the capability during handshake.
Q4.Why does MCP not have an explicit shutdown RPC method?
Exactly. stdio shutdown = close stdin and wait for EOF on stdout. HTTP shutdown = close the connection. The mechanics are unrecognisably different; a single RPC method couldn't cover both.
Q5.In the 2026 stateless core, where does per-session state live?
That's it. Handles are opaque tokens the server hands to the client. The client carries them; the server can be horizontally scaled, restarted, or load-balanced without losing context.
