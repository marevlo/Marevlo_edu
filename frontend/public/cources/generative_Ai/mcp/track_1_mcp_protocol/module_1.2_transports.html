Track 01 — The Protocol / Module 1.2

Transports — stdio & Streamable HTTP.

JSON-RPC is the envelope. The transport is the truck. Two trucks ship MCP messages in 2026: stdio for the local subprocess case, and Streamable HTTP for everything that crosses a network. Knowing which one to reach for — and what each one quietly costs you — is the difference between an MCP server that just works and one that won't scale past your laptop.

01  The transport layer

What the protocol does — and what it leaves to someone else.

Look at the MCP spec and you'll notice a careful split. Some things are defined at length — the wire format, the lifecycle, the capability flags, the error codes. Other things are deliberately abstract: how do bytes actually get from client to server? The spec shrugs at this question on purpose. That's the transport's job, and it differs sharply depending on whether the server lives in your local process tree or behind a load balancer in another data centre.

The protocol layer cares about three things from the transport:

  • Message framing — given a byte stream, where does one JSON-RPC message end and the next begin?
  • Direction — both sides need to be able to send messages without waiting for the other.
  • Reliability — bytes arrive in order, and arrive intact, or the protocol can detect that they didn't.

Everything else — TLS, auth, retries, multiplexing — lives below the protocol's interest. stdio satisfies these three by being a UNIX pipe between parent and child processes. Streamable HTTP satisfies them by being HTTP over TCP with TLS. Both are fine answers to the same question; you pick based on where the server lives.

Two transports, not five

You may see WebSocket, gRPC, named-pipe, and Unix-socket transports mentioned in the wild. None of these are in the current MCP spec. They're either community experiments or things servers use to talk to their own backends — never client↔server in the MCP sense.

02  stdio in depth

Newline-delimited JSON. One channel for traffic. One for logs.

stdio is the simplest transport in MCP and the one you'll meet first. The host spawns your server as a subprocess (command + args + env) and the two processes communicate through three standard streams:

  • stdin  — host writes JSON-RPC messages, one per line, the server reads them.
  • stdout — server writes responses, one per line, the host reads them.
  • stderr — server writes logs. The host ignores this for protocol purposes; it may capture it for debugging.

Framing is dead simple: one JSON message per line, separated by \n (LF, 0x0A). No length prefix, no content-type header. The bytes are UTF-8. A BOM is forbidden. If a message contains a literal newline it must be escaped within the JSON string — never broken across two lines on the wire. The print() trap

In Python, print() writes to stdout by default. Every print statement in your stdio server is a protocol corruption waiting to happen. Use logging.getLogger() configured to write to stderr (FastMCP does this by default), and audit your dependencies — third-party libraries that print on import will brick your server before it speaks its first JSON-RPC byte.

03  Streamable HTTP

One endpoint. Two modes. Streaming when you ask for it.

Streamable HTTP is the network-facing transport, introduced in the 2025-03-26 spec revision. The design is deliberately small: a single endpoint (typically /mcp) that accepts POST requests carrying JSON-RPC messages and a GET request that opens a long-lived stream for server-initiated traffic.

The trick is that the same POST can respond in two different modes, chosen by the server based on whether streaming makes sense for the call. The Content-Type of the response is what tells the client which mode it got:

› The Origin check you must not skip

      

        For any browser-reachable MCP server — and that includes a localhost server, because
        local pages can attack it — the spec mandates validating the Origin header on
        every request. The vulnerability is DNS rebinding: a malicious site
        resolves its domain to 127.0.0.1 after the browser has cached the script
        origin, and now the attacker's JavaScript can call your local MCP server with your user's
        permissions. The fix is one line: allow only origins on a list you control. We'll cover
        the full hardening in Module 4.3.
      

    


    
    

      
04  The legacy migration

      
What HTTP+SSE looked like — and why it's gone.


      

        The original 2024-11-05 spec defined a network transport called HTTP+SSE. It used
        two endpoints: a POST endpoint for client→server and a separate SSE endpoint
        for server→client. The SSE stream was opened during handshake and kept alive for the entire
        session — every message in either direction depended on that long-lived connection being
        healthy.
      


      
It sounds reasonable on paper. In production, it had four sharp edges:


      
    
        
  • Required two endpoints in every reverse-proxy config — easy to misconfigure, common cause of "the handshake works but tool calls hang."
    • 
        
  • SSE was kept open from session start, even for clients that never needed streaming. Wasted connections at scale.
    • 
        
  • The server had to maintain session state mapping client IDs to SSE streams. Load balancers needed sticky sessions; stateless deployment was nearly impossible.
    • 
        
  • Serverless platforms (Cloud Run, Lambda) hate long-lived connections. Cold starts and request timeouts made the design genuinely painful to deploy.
    • 
      


      
The 2025-03-26 spec collapsed both endpoints into one and made streaming opt-in per call. Watch the same task in both designs:

If you're still on legacy

        

          The 2024-11-05 HTTP+SSE transport will keep working with older clients, but no new MCP
          features will be designed for it. Migration is mostly mechanical — collapse the two
          endpoints into one, switch the response handler to dispatch on Accept
          header — and frameworks like FastMCP do this automatically when you set
          transport="http" on a recent version.
        

      

    


    
    

      
05  The stateless turn

      
2026: state moves out of the server, into the handle.


      

        Even with one endpoint instead of two, Streamable HTTP servers traditionally kept session
        state in memory — "this client has these resources subscribed, these tools in flight,
        this auth context". That state had to live somewhere, and it usually lived in
        the process. Which meant: restart the server, lose the state. Scale horizontally, deal
        with sticky sessions. Move the request to a different replica, broken.
      


      

        The 2026 spec turns this inside out. Instead of the server remembering the client, the
        server returns explicit handles — opaque tokens (typically signed or
        encrypted) that encode whatever state was needed. The client carries the handle and
        passes it back on every related request. Any replica can now handle any
        request. Toggle below to see the difference:

What a handle actually contains

        

          Implementations vary, but a handle is typically a base64-encoded JWT or fernet token. Inside:
          a session id, the user/auth principal, any in-progress operation state (cursor positions
          for paginated calls, subscription registrations), and an expiry. The token is signed so the
          client can't forge or modify it. To the client it's opaque; to the server it's
          self-describing — which is the whole point.

Five quick checks before Module 1.3.

        

      


      

        
Q1.How does an MCP stdio server know where one message ends and the next begins on stdin?

        

          
          
          
          
        

        
Correct. MCP stdio uses newline-delimited JSON. No length prefix, no special terminators — just one message per line.

      


      

        
Q2.A Streamable HTTP server POST returns Content-Type: text/event-stream. What does that signal?

        

          
          
          
          
        

        
Right. The same POST endpoint can respond either with application/json (single response, done) or text/event-stream (streaming response). The Content-Type is the dispatch.

      


      

        
Q3.Why did the spec drop the original two-endpoint HTTP+SSE transport?

        

          
          
          
          
        

        
Exactly. The legacy design had real operational problems at production scale. The collapse to one endpoint with content-type dispatch was the structural fix.

      


      

        
Q4.You're building a local-only MCP server for filesystem operations on the user's machine. Which transport?

        

          
          
          
          
        

        
Yes. Anything single-host and process-local should be stdio. No network surface means no auth burden, no firewall rules, and one fewer attack vector.

      


      

        
Q5.In the 2026 stateless model, what does an "explicit handle" actually contain?

        

          
          
          
          
        

        
Correct. The handle is self-describing to the server (it can decrypt/verify it) and opaque to the client. That's what makes any replica able to serve any request.
