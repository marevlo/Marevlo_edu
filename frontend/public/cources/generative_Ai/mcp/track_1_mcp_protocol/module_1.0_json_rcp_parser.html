01  Why JSON-RPC at all

Five constraints. One format that hits them.

Anthropic could have picked anything: gRPC, REST, GraphQL, a bespoke binary protocol. The choice of JSON-RPC 2.0 wasn't aesthetic — it was a response to five hard constraints, and no other mainstream format clears all five.

  • Bidirectional from day one. Servers must be able to initiate calls toward the client (for sampling, elicitation, roots). REST is request/response only. JSON-RPC treats both directions symmetrically.
  • Transport-agnostic. The spec needs to ride on stdio, HTTP, SSE, WebSocket, and whatever comes next. JSON-RPC is just an envelope — the transport is a separate layer.
  • Multiple concurrent in-flight requests. Async tool calls demand that any response can arrive at any time. JSON-RPC's id field exists exactly for this.
  • Battle-tested at scale. The Language Server Protocol — Microsoft's editor↔compiler bridge powering VS Code, Neovim, JetBrains — has used JSON-RPC over stdio since 2016. MCP is downstream of LSP's design.
  • Trivial to debug. Both sides talk plain JSON. You can tail a log and read it. No .proto files, no Wireshark plugins.
The LSP lineage

If a piece of MCP feels familiar, that's because it probably is. Initialize handshake, capability negotiation, notifications as first-class messages, request/cancel pairs — all lifted from LSP. The MCP team has been explicit about the inheritance.

03  Four message types

One discriminator: is there an id?

JSON-RPC defines four message shapes. The single rule that distinguishes them on the wire: presence and position of id. Memorise this before moving on — every debug session you'll ever have on MCP traffic starts here.

  • Request — has id, has method. Expects a response.
  • Success response — has id (echoing the request's), has result.
  • Error response — has id, has error (object, never a string).
  • Notification — has method but no id. No response. Fire and forget.

05  Error objects

      
The codes you'll see — and what they mean.


      

        Errors in JSON-RPC are never bare strings. They are objects with a required code
        (integer) and message (string), plus an optional data for
        context. The codes fall into two ranges: standard JSON-RPC codes (negative four-digit
        numbers prefixed with -326xx) and a server-reserved range
        (-32000 to -32099) where MCP defines its own.
      


      Reference · click any code

      

        

          

            -32700
            Parse errorThe frame wasn't even valid JSON
            +
          

          

            
The server couldn't parse the incoming bytes as JSON at all. Almost always a transport-level bug: a writer not flushing, a stdio framer corrupting newlines, a proxy stripping bytes. If you're emitting this, something downstream is broken — the request never made it intact.


"error": {
  "code": -32700,
  "message": "Parse error",
  "data": "Unexpected token at position 47"
}

            
Most common cause: writing logs to stdout instead of stderr in a stdio server.

          

        


        

          

            -32600
            Invalid RequestParsed fine, but isn't a valid JSON-RPC object
            +
          

          

            
The JSON parsed, but the envelope is malformed: missing jsonrpc: "2.0", wrong field types, both result and error present, neither present. Schema-level violation of JSON-RPC itself.


"error": {
  "code": -32600,
  "message": "Invalid Request",
  "data": { "reason": "missing 'method'" }
}

            
Most common cause: handcrafted client code that forgets "jsonrpc": "2.0".

          

        


        

          

            -32601
            Method not foundNo handler registered for that name
            +
          

          

            
The method name in the request doesn't correspond to anything the server exposes. In MCP, this is what you see when a client tries to call a tool that has since been removed, or when a version mismatch means the client is calling a method the server doesn't implement.


"error": {
  "code": -32601,
  "message": "Method not found",
  "data": { "method": "tools/list_changed" }
}

            
Most common cause: client speaks a newer spec version than the server.

          

        


        

          

            -32602
            Invalid paramsMethod exists, but the arguments are wrong
            +
          

          

            
JSON Schema validation failed on params. The most frequent error you'll see in production MCP because LLMs sometimes hallucinate argument names or types. Good servers put the validation failure detail in data.


"error": {
  "code": -32602,
  "message": "Invalid params",
  "data": {
    "errors": [
      { "path": "/name", "msg": "required" }
    ]
  }
}

            
Most common cause: LLM omitting a required argument or passing a string where a number was expected.

          

        


        

          

            -32603
            Internal errorThe handler itself blew up
            +
          

          

            
An unhandled exception inside your tool implementation. The protocol layer was fine; your business logic threw. Most frameworks (FastMCP included) catch tool exceptions and convert them to this code. Surface the exception class and message in data when safe — but never stack traces for remote servers.


"error": {
  "code": -32603,
  "message": "Internal error",
  "data": { "exception": "ConnectionRefusedError" }
}

            
Most common cause: downstream service the tool calls is down.

          

        


        

          

            -32000 → -32099
            Server-defined rangeMCP-specific codes live here
            +
          

          

            
JSON-RPC reserves -32000 to -32099 for the application to define. MCP uses this range for protocol-specific failures — e.g. resource-not-found, invalid cursor, capability not advertised. Check the spec for current assignments; treat anything in this range as MCP-meaningful, not generic.


// Example MCP-specific error
"error": {
  "code": -32002,
  "message": "Resource not found",
  "data": { "uri": "file:///etc/missing.conf" }
}

            
Rule: never invent codes outside this range for application errors.

          

        

      

    


    
    

      
06  ID correlation

      
How three requests in flight don't collide.


      

        The id field is doing more work than it looks. Both sides of an MCP connection
        can have multiple requests in flight simultaneously — a client might ask the server for
        tool A, tool B, and resource C in parallel, and responses can arrive in any order.
        The id is what lets the client match each response to the right pending
        request. Toggle the switch below and watch what happens when IDs disappear.
      


      Visualise · toggle correlation

      

        

          Correlation Visualizer · 3 concurrent requests
          

            
            
            
          

        

        

          
        

        

          ✓ With IDs: each response carries the id of the request it answers. The client looks up its pending-requests table by id, routes the result to the right callback, and the three calls complete independently — even though they came back as 3 → 1 → 2.
          ⚠ Without IDs: the client gets three responses back but has no way to know which one belongs to which pending request. It either has to (a) serialise everything to one request at a time, killing concurrency, or (b) guess from response shape, which is fragile and breaks the moment two tools return similar JSON.
        

      

    


    
    

      
07  Batching

      
JSON-RPC allows it. MCP killed it.


      

        JSON-RPC 2.0 lets you send an array of messages in a single frame. The server processes
        them and returns an array of responses. For a long time, MCP supported this. Then, in the
        2025-06-18 spec revision, it was explicitly forbidden. Toggle below to see why batching
        looks appealing — and then read why the spec walked away from it.

            MCP 2025-06-18: "JSON-RPC batching MUST NOT be used."
            The spec removed batching because it complicated streaming responses (some tools stream
            progress notifications mid-call), made partial-failure semantics ambiguous, and
            interacted badly with cancellation. The replacement: send N parallel frames and
            correlate responses by id — exactly the mechanism we just visualised.
          

        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 1.0

          
Five quick checks before Module 1.1.

        

      


      

        
Q1.A message has "method": "ping" and "jsonrpc": "2.0", but no "id". What is it?

        

          
          
          
          
        

        
Correct. Absence of id is exactly what makes it a notification. The server won't respond to it, and that's the point.

      


      

        
Q2.You receive an error with code: -32601. What's the most likely cause?

        

          
          
          
          
        

        
Right. -32601 is "Method not found". The handler simply isn't registered, which most often happens when the client is on a newer spec than the server.

      


      

        
Q3.Why does MCP need the id field even though most exchanges look like simple request-then-response?

        

          
          
          
          
        

        
Exactly. Concurrency is the whole point. Without ids, the only safe pattern is one-at-a-time, which kills throughput.

      


      

        
Q4.Did MCP keep JSON-RPC's batching feature in the 2025-06-18 spec?

        

          
          
          
          
        

        
Correct. The spec dropped batching because of conflicts with streaming responses and cancellation. Use parallel frames + id correlation instead.

      


      

        
Q5.An MCP error has code: -32030. Where does that come from?

        

          
          
          
          
        

        
That's it. -32000 → -32099 is JSON-RPC's officially-reserved range for applications. MCP uses it for protocol-specific error semantics.
