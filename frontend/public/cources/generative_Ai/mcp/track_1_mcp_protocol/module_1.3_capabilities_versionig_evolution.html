Track 01 — The Protocol / Module 1.3

Capabilities, versioning & evolution.

Protocols ship and then they keep moving. The MCP you write a server against today is not the MCP you'll be writing against in 18 months. This module is about how MCP changes — capability flags, dated revisions, the deprecation policy that finally arrived in 2026, and the SEP process that decides what makes it into the next version.

01  Capability flags in detail

Not just a boolean — a small structured object per feature.

Module 1.1 introduced capabilities as a yes/no toggle: server advertises tools, client advertises sampling. On the wire, capabilities are richer than that. Each capability is an object that may itself carry sub-flags — small per-feature switches that further refine what's supported. Click a capability below to see its real shape, what it gates, and what its sub-flags actually mean.

Empty object ≠ false

"sampling": {} means "I support sampling — no special sub-flags." Omitting the key entirely means "I do not support sampling at all." These are not the same. A common bug: serialising a Python None as a JSON null ends up as "sampling": null, which most validators reject. Use empty object when the capability has no sub-flags.

02  Spec version history

Three frozen revisions and the road ahead.

MCP versions are date strings, not semver — Module 1.1 established why. What it didn't cover is what actually changed at each revision. Knowing this matters more than it sounds: the difference between a tool call that works and one that returns -32601 Method not found is usually "that method was added in the next revision." 03  Deprecation policy

How a feature dies politely.

Pre-2026, MCP's evolution was choppy. Features arrived, others vanished mid-revision (RIP batching, removed without ceremony in 2025-06-18). The 2026 governance lands a formal deprecation policy: any feature being removed must spend at least one full revision in the deprecated state, with explicit migration guidance and a sunset date published in the changelog. What batching really demonstrated

Batching was the cautionary tale that motivated the formal policy. It was supported through 2025-03-26, then removed in 2025-06-18 with one revision of overlap as deprecation. A lot of servers that hadn't kept up with the changelog broke that quarter. Post-2026, a feature removal of similar magnitude would announce in advance, give at least one revision of dual-support, and ship migration tooling in the SDKs.

05  Extension naming

Reverse-DNS, because the internet ran out of short names.

Not every new capability needs to go through the SEP process. Vendor-specific or organisation-specific extensions can live outside the spec entirely — but they need a naming scheme that won't collide with someone else's extension. MCP borrows reverse-DNS from Java and Apple: your extension ID is your organisation's domain, reversed, plus a name. Try the validator:

When extensions are appropriate

Vendor extensions are for capabilities only your servers and clients understand. If your idea is generally useful, write a SEP instead — extensions outside the spec fragment the ecosystem, and the whole point of MCP is to not fragment. A good rule: build the extension to prove the idea, then upstream it. Anthropic's computer-use extension started exactly that way.

Knowledge Check · End of Module 1.3
Five quick checks to close out Track 1.
Q1.A server advertises "resources": { "subscribe": true } but omits listChanged. What does this mean?
Correct. Sub-flags are independent — each one represents a separate sub-feature. The presence of the parent capability matters; sub-flags are additive.
Q2.Which spec revision finally removed JSON-RPC batching?
Right. 2025-06-18 removed batching outright (MUST NOT be used) and added elicitation, structured output, and resource links in the same revision.
Q3.Under the 2026 deprecation policy, what's the minimum lifespan of a feature in the "deprecated" state before it can be removed?
Yes. The policy guarantees at least one revision of overlap, giving implementers a published sunset date and time to migrate before the feature actually disappears.
Q4.In the SEP lifecycle, at what stage does an SDK typically start shipping the feature?
Correct. SDKs start implementing once the proposal is accepted (stage 4 → 5). Stabilization (stage 6) marks the point where the feature is locked into the spec and treated as mature.
Q5.Your team wants to add a custom capability for an internal RAG service. Which extension ID is valid?
That's it. Reverse-DNS of a domain you control, slash, lowercase feature name. Option B uses a reserved namespace; C uses an unowned naming scheme; D ignores the format entirely.
