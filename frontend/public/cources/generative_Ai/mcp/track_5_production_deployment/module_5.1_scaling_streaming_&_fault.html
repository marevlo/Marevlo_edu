01  Load balancing strategies

Four algorithms, only two of which work for MCP traffic.

A web server with stateless GET requests can use any load-balancing strategy that exists. MCP is not that server. Sessions are sticky, SSE streams are long-lived, and tool calls often run for tens of seconds. Tabbing through the strategies below, watch what each one does to a four-replica cluster handling six concurrent sessions — and which ones survive contact with reality:

Compare · pick an algorithm

› Why round-robin is wrong (specifically)

Round-robin is the default everywhere. It distributes requests evenly across backends, which is what you want for stateless GETs. MCP doesn't have stateless GETs. Every non-initialize call is bound by Mcp-Session-Id to a specific server that already holds the negotiated capabilities, the SSE channel, and any in-progress tool execution. Round-robin scatters those bound requests across all replicas, which produces exactly the "session not found" failure we walked through in 5.0.

› The two algorithms that actually work

  • Consistent hash on Mcp-Session-Id. The dominant choice. Each session lands on the same backend forever (or until that backend dies). Adding/removing a replica reshuffles only ~1/N of existing sessions instead of all of them. This is what Nginx's hash $http_mcp_session_id consistent gives you.
  • Cookie-based affinity. What most cloud LBs default to. The LB sets a cookie on initialize; subsequent requests carry it; the LB reads it and routes accordingly. Same outcome as hashing on the session header — just a different mechanism.
"Least connections" is a trap here

Least-conn is tempting because it sounds intelligent — "send the new request to whichever backend has the fewest active connections." For MCP it's actively harmful: an SSE stream is one long-lived connection that the algorithm sees as load. The replica holding the most active sessions looks the busiest and is sent no new traffic, starving it of new sessions while overloading freshly-restarted replicas with zero long-lived connections. Use consistent hashing.

02  Redis pub/sub fan-out

The simplest backplane that lets any replica notify any client.

Sticky sessions solve one problem: making sure all requests for session S land on the same replica. They don't solve a second: server-side work that needs to notify a session whose SSE channel is on a different replica. A scheduled job, a webhook callback, or a cross-tool dependency may need to push to a client whose POST handler happens to live elsewhere. The standard fix is a pub/sub backplane. A scheduled job runs on Replica 2. It needs to push a notifications/resources/updated to session S1 — whose SSE stream lives on Replica 1. The backplane is what connects them.

› The actual implementation

Every replica subscribes to a channel like mcp:sessions:{session_id} as soon as it accepts an SSE stream. When any replica needs to push to that session — whether the push originated locally or on another replica — it publishes to the same channel. Redis delivers to every subscriber; the replica that owns the stream serializes it as an SSE event and writes it down to the client. 

from contextlib import asynccontextmanager
import redis.asyncio as redis
from fastmcp import FastMCP

@asynccontextmanager
async def lifespan(server):
    r = redis.from_url("redis://redis:6379")
    try:
        yield {"redis": r}
    finally:
        await r.close()

mcp = FastMCP("my-server", lifespan=lifespan)

# On every replica: subscribe to its sessions' channels
async def subscribe_session(ctx, session_id):
    r = ctx.request_context.lifespan_context["redis"]
    pubsub = r.pubsub()
    await pubsub.subscribe(f"mcp:sessions:{session_id}")
    async for msg in pubsub.listen():
        if msg["type"] == "message":
            await ctx.send_notification(json.loads(msg["data"]))

# Any replica: publish to a session's channel
async def notify_session(redis, session_id, payload):
    await redis.publish(f"mcp:sessions:{session_id}", json.dumps(payload))
Pub/sub is fire-and-forget — use Streams for guaranteed delivery

Redis PUBLISH drops messages on the floor if no subscriber is currently connected. For best-effort notifications (progress updates, resource changes) that's fine — they'll be re-derived on reconnect. For deliveries that must survive a brief subscriber disconnect, use Redis Streams with consumer groups, or NATS JetStream. Both give you replay semantics; pub/sub does not.

03  Reconnection & resume

SSE survives a network blip — if you wire it correctly.

SSE is a long-lived HTTP response. On a flaky network it will break. Cell towers swap, NAT tables expire, load balancers time out idle streams. The MCP client is expected to reconnect; the protocol provides an id field on each event and a Last-Event-Id header on resume so the server can replay missed events. › The server's side of the contract

For resume to work the server has to emit IDs on its SSE events and remember (for a bounded window) what it sent. FastMCP handles the framing automatically; what you owe it is a short-lived buffer of "recent events for session S". Redis Streams are perfect for this — bounded length, auto-trim, fast range queries by ID. 

# Stash each emitted event into a per-session Redis stream (max 1k entries)
async def remember_event(redis, session_id, event_payload):
    await redis.xadd(
        f"mcp:events:{session_id}",
        {"data": json.dumps(event_payload)},
        maxlen=1000, approximate=True,
    )

# On reconnect: client sends Last-Event-Id; we replay everything after it
async def replay_since(redis, session_id, last_event_id):
    entries = await redis.xrange(
        f"mcp:events:{session_id}",
        min=f"({last_event_id}",  # exclusive lower bound
        max="+",
    )
    for entry_id, fields in entries:
        yield entry_id, json.loads(fields[b"data"])
Buffer size is a real decision

Too small and a 30-second tunnel outage means losing events the client expected to replay. Too large and a long-running session's stream balloons in Redis. 1k events per session with MAXLEN ~ (approximate trimming) is a good default for typical MCP traffic. If your server emits a high-frequency progress stream, raise it; if you rarely notify, lower it.

04  Health checks & idempotency

Three probes that look identical and do completely different things.

Kubernetes (and most modern orchestrators) ships three different health checks. They are not interchangeable and most production incidents involving "my pod won't stay up" come down to confusing them. Each one answers a different question and triggers a different action:

The Three Probes · What Each One Owns
Startup probe

/startupz

Runs first, with generous timeouts. "Has the app finished initializing?" Once it passes, the orchestrator switches to running the other two. Used to protect slow-starting apps (loading models, building caches) from being killed by liveness before they're ready.

startupProbe: httpGet: {path: /startupz} failureThreshold: 30 periodSeconds: 10 # 5 min total
Action on failurekill + restart
Readiness probe

/readyz

"Should the LB send traffic to me right now?" Runs continuously. Failing it does not restart the pod — it just removes the pod from the Service's endpoint list. Use it to drain traffic during deploys, gate on dependency health, or temporarily refuse load.

readinessProbe: httpGet: {path: /readyz} failureThreshold: 3 periodSeconds: 5
Action on failureremove from LB
Liveness probe

/livez

"Is the process wedged?" Runs continuously. Failing it restarts the pod. Should be the dumbest possible check — does the process respond to HTTP at all. Don't make it depend on Redis, the DB, or anything else; that turns a Redis blip into a cascade of pod restarts.

livenessProbe: httpGet: {path: /livez} failureThreshold: 3 periodSeconds: 10
Action on failurekill + restart

› The FastMCP setup

FastMCP serves Streamable HTTP via Starlette under the hood, which means you can mount plain HTTP routes for health checks alongside the MCP endpoint. Critically — these endpoints should sit outside the MCP session machinery, so probes don't allocate a session every five seconds. 

from fastmcp import FastMCP
from starlette.responses import JSONResponse

mcp = FastMCP("my-server", lifespan=lifespan)

@mcp.custom_route("/livez", methods=["GET"])
async def livez(request):
    # Dumbest possible: the process is up
    return JSONResponse({"status": "ok"})

@mcp.custom_route("/readyz", methods=["GET"])
async def readyz(request):
    redis = request.app.state.redis
    try:
        await redis.ping()
    except Exception:
        return JSONResponse({"status": "degraded"}, status_code=503)
    return JSONResponse({"status": "ok"})

› Idempotency: same tool call, twice, no double effect

A client whose stream dies mid-tool-call has no way to know whether the call ran. When it reconnects it may retry. If your tool's effect is not idempotent — a payment, a file write, a "send email" action — that retry is a bug. The fix is the standard one: accept an Idempotency-Key header (or a tool argument), record it before the side effect, and on a second call with the same key return the original result instead of re-executing. 

@mcp.tool
async def send_email(
    to: str, subject: str, body: str,
    idempotency_key: str,
    ctx: Context,
) -> dict:
    """Send an email exactly once per idempotency_key."""
    redis = ctx.request_context.lifespan_context["redis"]
    cached = await redis.get(f"idemp:{idempotency_key}")
    if cached:
        return json.loads(cached)        # short-circuit: replay result

    result = await mailgun.send(to, subject, body)
    await redis.set(f"idemp:{idempotency_key}",
        json.dumps(result), ex=86400)   # 24h dedup window
    return result
Don't trust the LLM to generate the key

The LLM might generate the same idempotency key for two genuinely different requests, or a different key for what should be the same retry. Generate it inside the tool from the request's stable inputs (a hash of the args) or push the responsibility to the client. Either way: the key is yours to define, not the model's.

05  Leader election

Exactly one of your replicas does the thing.

Replicas are interchangeable for handling user traffic. They are not interchangeable for tasks that should happen exactly once across the cluster: a scheduled "refresh the cache at midnight" job, a periodic cleanup, an outbound webhook poller. If you run such a task in __main__, you get N copies of it, one per replica — every cache refresh happens four times, every email goes out four times. Leader election picks one replica to do these jobs and demotes the others to standby.

› The mechanism, in detail

Every replica, on a loop, attempts SET leader:scheduler <replica-id> NX EX 10 — set the key only if it doesn't exist, with a 10-second TTL. Exactly one replica wins; the others see NX fail and become standbys. The winner re-runs the same SET every 3 seconds (well before TTL) to extend its lease. If the leader crashes, the lease isn't renewed; the key expires; the next replica that tries to set it wins and takes over. Maximum cluster-wide unavailability of the scheduled task: 10 seconds. 

import asyncio, uuid

REPLICA_ID = str(uuid.uuid4())
LOCK_KEY = "leader:scheduler"
LEASE = 10        # seconds — total leader downtime if process dies
RENEW = 3         # renew well inside the lease

async def leader_loop(redis):
    while True:
        # Try to acquire (NX = only if not already set)
        won = await redis.set(LOCK_KEY, REPLICA_ID,
                              nx=True, ex=LEASE)
        if won:
            # I am leader. Run my responsibilities while extending the lease.
            try:
                await run_leader_work_with_renewal(redis)
            finally:
                # Best-effort release on graceful shutdown
                current = await redis.get(LOCK_KEY)
                if current == REPLICA_ID.encode():
                    await redis.delete(LOCK_KEY)
        else:
            # Standby. Try again after lease expires.
            await asyncio.sleep(LEASE)

async def run_leader_work_with_renewal(redis):
    async with asyncio.TaskGroup() as tg:
        tg.create_task(renew_lease(redis))
        tg.create_task(scheduled_jobs())

async def renew_lease(redis):
    while True:
        await asyncio.sleep(RENEW)
        # Only renew if WE still own it (fencing token check)
        if await redis.get(LOCK_KEY) == REPLICA_ID.encode():
            await redis.expire(LOCK_KEY, LEASE)
        else:
            raise RuntimeError("lost leadership")
Use a real coordinator if it actually matters

Redis SET NX EX is good enough for soft cron — refreshing a cache, emitting a daily report — where a brief double-fire or a brief gap is acceptable. For consensus-grade requirements (financial reconciliation, strict-once delivery), use a coordination layer that gives you fencing tokens and proper leadership semantics: etcd's lease API, ZooKeeper, or Consul sessions. The pattern looks the same; the guarantees underneath are much stronger.

Knowledge Check · End of Module 5.1
Five checks before Module 5.2.
Q1.Your cloud load balancer defaults to "least connections" routing. After deploying a four-replica MCP cluster, one replica handles 80% of sessions and the others sit nearly idle. Why?
Right. Least-conn is anti-correlated with what you actually want for sticky workloads. The hot replica is hot because it was warm first; everything that arrives goes there.
Q2.A scheduled job on Replica 3 needs to push notifications/resources/updated to a session whose SSE stream is on Replica 1. What's the smallest correct architecture?
Exactly. The pub/sub backplane decouples "where the work runs" from "where the SSE stream lives". Any replica can publish; only the replica with the live stream actually delivers.
Q3.Your liveness probe does SELECT 1 against Postgres. During a 90-second DB blip, every pod in the cluster gets killed and restarted. The right fix is:
Yes. Liveness restarts. Readiness removes from LB. A DB blip should make replicas unready (so traffic drains), not dead (so they restart). Conflating the two creates restart storms.
Q4.A user's SSE stream drops at event ID 1247. They reconnect. Their server is sitting in Redis Streams; the latest event ID emitted to the session is 1281. What should happen?
Correct. The Last-Event-Id header is the SSE-native resume mechanism. The server's responsibility is to keep a bounded per-session buffer (Redis Streams, in-memory ring) and replay from there.
Q5.You implement leader election with SET leader NX EX 10 and a renewal every 3 seconds. The leader's container is SIGKILL'd at t=0. What's the worst-case time before another replica is doing the work?
Exactly. The lease TTL is what bounds your downtime on hard failure. Shorten EX for faster failover, but not below your renewal interval — that creates a leader-flapping bug.
