Track 05 — Production, Deployment & Ecosystem / Module 5.4

Advanced transports (gRPC & extensions).

Streamable HTTP is what you ship. It works, it's the spec default, and it's behind every production MCP server worth pointing at. So when does anyone reach for gRPC? When the workload demands true bidirectional streaming, sub-millisecond latency, or strongly-typed schemas across language boundaries. This module is the honest case for — and against — adding gRPC to your stack, plus the bridge layer (gRPC-web) that makes either choice reachable from a browser.

01  HTTP/SSE vs gRPC

Two transports, two completely different bets.

Streamable HTTP and gRPC are not interchangeable. They optimize for different things. HTTP is the universal language of the internet — every load balancer, proxy, browser, and ops tool understands it. gRPC is the modern RPC framework — denser on the wire, schema- first, true bidirectional, but harder to deploy. Toggle between them to see what each one actually emits on the wire and what it buys you:

› The honest answer: pick HTTP unless you know why

      

        For 95% of MCP servers, Streamable HTTP is the right answer for the reasons we covered in
        5.0: every reverse proxy, every browser, every cloud LB, every cURL-based smoke test
        understands it. gRPC is a deliberate choice for specific workloads:
      


      
    
        
  • Backend-to-backend MCP traffic at high RPS. Service-mesh sidecar pattern where the "client" is itself a server. Protobuf's compactness and HTTP/2 multiplexing pay off when you're doing thousands of RPCs per second between trusted services.
    • 
        
  • Strictly-typed multi-language teams. One .proto file generates clients in Python, Go, Java, TypeScript, Rust — all with identical types. JSON over HTTP gives you the same wire but every team writes their own (slightly different) types by hand.
    • 
        
  • Genuinely bidirectional, low-latency streaming. SSE is one-way after the initial POST. If you need a continuous stream both directions (think collaborative editing of agent state), gRPC's bidi mode is what was actually designed for that.
    • 
      


      

        
gRPC is not part of the MCP spec (today)

        

          As of the current spec, stdio and streamable-http are the only
          transports MCP officially defines. gRPC servers exist in the wild as extensions
          — typically wrapping the same FastMCP/SDK tool logic behind a hand-written gRPC service
          definition. Clients that consume them are bespoke; Claude Desktop, Cursor, and other
          mainstream hosts speak HTTP. Adding gRPC means accepting that the client side is yours
          to build.
        

      

    


    
    

      
02  The four streaming modes

      
One Protobuf method, four different shapes of conversation.


      

        gRPC's real differentiator over HTTP/SSE is that it natively supports four distinct
        streaming patterns, and you pick per-method by adding (or omitting) the
        stream keyword on either side of the signature. Switch between the modes to
        see what each looks like as a sequence of messages, and what kind of RPC it fits:
      


      Explore · the four modes

› Mapping MCP concepts onto streaming modes

      
    
        
  • Tool call with a single result → unary. The classic case. Send args, get answer. 90% of MCP tool surface fits here.
    • 
        
  • Tool call with progress updates → server-stream. One request, many messages back. This is what SSE gives you over HTTP. gRPC formalizes it with a real stream type.
    • 
        
  • Batch ingest from client → client-stream. The client uploads many small messages, the server collates and replies once. Rare in MCP but useful for "send me this whole document in chunks".
    • 
        
  • Interactive sampling / elicitation conversation → bidirectional. Server asks the host for a model completion; host streams tokens back while server is also streaming more requests. Genuinely two-way, the case where SSE starts feeling inadequate.
    • 
      

    


    
    

      
03  Protobuf, line by line

      
The .proto file is the contract — and the source of truth.


      

        Where REST/JSON-RPC gives you a free-form payload and hopes everyone agrees on the shape,
        gRPC is schema-first. A single .proto file declares your services, methods,
        and message shapes — and then code generators in every language emit identical clients
        and stubs from it. Click any highlighted element to see what it does and what it becomes
        on the wire:

// mcp_extension.proto
syntax = "proto3";

package mcp.extension.v1;

option go_package = "github.com/example/mcp-ext/v1";

// The service is the gRPC counterpart of an MCP server.
service MCPExtension {

  // Unary: one request, one response. Most tool calls.
  rpc CallTool(CallToolRequest) returns (CallToolResponse);

  // Server-stream: progress updates while a long-running tool works.
  rpc CallToolStreaming(CallToolRequest) returns (stream ToolEvent);

  // Bidirectional: full duplex for sampling exchanges.
  rpc Sample(stream SampleRequest) returns (stream SampleResponse);
}

// Request message — keep tag numbers stable forever.
message CallToolRequest {
  string tool_name = 1;
  map<string, google.protobuf.Value> arguments = 2;
  string session_id = 3;
  reserved 4, 5;  // removed fields — never reuse the tag
  reserved "old_user_field";
}

message CallToolResponse {
  oneof outcome {
    ToolResult result = 1;
    ToolError error = 2;
  }
  repeated string warnings = 3;
}

› The schema evolution rules you cannot break

      

        The single biggest reason teams reach for Protobuf — and the single biggest reason teams
        get burned by it — is schema evolution. Three rules:
      


      
    
        
  • Tag numbers are forever. If field user_id was tag 3, it cannot ever become tag 4. If you delete the field, mark its tag reserved so nobody re-uses it for a different field later. Re-using a tag silently corrupts wire-incompatible clients.
    • 
        
  • Adding a field is always safe. Old clients ignore unknown fields. New clients see None/empty when the field isn't present. Pick a fresh tag number and ship.
    • 
        
  • Changing a type is never safe. Changing int32 to int64? Usually fine. Changing string to bytes? Wire-compatible but semantically a different field. Changing repeated-ness? Breaks the world. When in doubt, add a new field with a new tag and deprecate the old one.
    • 
      


      

        
Use buf, not raw protoc

        

          The buf CLI is the modern, sane way to compile and lint .proto
          files. It enforces the evolution rules above (it'll flag tag re-use, type changes, and
          backward-incompatible renames before they ship), generates clients in every language
          from one config, and integrates with CI. Don't reach for raw protoc in
          2026; it's a working tool, but buf is the productivity layer everyone uses.
        

      

    


    
    

      
04  Hybrid architecture

      
The same tools, exposed over both transports, from one process.


      

        The case for gRPC in MCP is rarely "instead of" — it's "in addition to". The same tool
        logic, the same lifespan, the same Context, exposed simultaneously over Streamable HTTP
        (for public/browser/Claude-Desktop traffic) and over gRPC (for internal high-volume
        callers). Pick a client category to see which transport it lands on and why:

› The FastMCP + grpcio shape

      

        The trick to keeping the dual-transport server sane is to share everything above
        the transport layer. One lifespan, one set of tool functions, one set of shared resources.
        The two transport servers (Starlette for HTTP, grpc.aio for gRPC) just call into the
        same handlers.
      



import asyncio
from contextlib import asynccontextmanager
from fastmcp import FastMCP
import grpc.aio
from .generated import mcp_extension_pb2_grpc, mcp_extension_pb2

# --- Shared business logic, transport-agnostic ---
@asynccontextmanager
async def app_lifespan(server):
    db = await create_pool()
    try:
        yield {"db": db}
    finally:
        await db.close()

mcp = FastMCP("hybrid-server", lifespan=app_lifespan)

@mcp.tool
async def search(query: str, ctx: Context) -> list:
    """Full-text search."""
    db = ctx.request_context.lifespan_context["db"]
    return await db.fetch("SELECT ... WHERE body LIKE $1", f"%{query}%")

# --- gRPC servicer reuses the same tool logic ---
class MCPExtensionServicer(mcp_extension_pb2_grpc.MCPExtensionServicer):
    def __init__(self, mcp_app, lifespan_ctx):
        self.mcp_app = mcp_app
        self.lifespan_ctx = lifespan_ctx

    async def CallTool(self, request, context):
        tool = self.mcp_app.get_tool(request.tool_name)
        result = await tool.run(
            arguments=dict(request.arguments),
            lifespan_context=self.lifespan_ctx,
        )
        return mcp_extension_pb2.CallToolResponse(result=result.to_proto())

# --- Run both transports from one process ---
async def main():
    # HTTP transport (Streamable HTTP at :8000/mcp)
    http_task = asyncio.create_task(
        mcp.run_async(transport="streamable-http", port=8000)
    )
    # gRPC transport (at :50051)
    grpc_server = grpc.aio.server()
    mcp_extension_pb2_grpc.add_MCPExtensionServicer_to_server(
        MCPExtensionServicer(mcp, mcp.lifespan_context), grpc_server
    )
    grpc_server.add_insecure_port("[::]:50051")
    await grpc_server.start()
    await asyncio.gather(http_task, grpc_server.wait_for_termination())

asyncio.run(main())


      

        
Keep one transport as the source of truth

        

          Two transports mean two contracts you have to keep in sync. The discipline that works:
          treat your @mcp.tool definitions as the source of truth, and your
          .proto as a thin gRPC-shaped projection of them — auto-generated where
          possible. If the proto starts growing fields the MCP side doesn't have, you've already
          built two servers instead of one.
        

      

    


    
    

      
05  gRPC-web for browsers

      
The bridge that lets a JavaScript client speak gRPC at all.


      

        Native gRPC requires HTTP/2 trailers and binary frames the browser cannot generate
        — the XHR and Fetch APIs don't expose them. Anyone who tells you "just use gRPC in the
        browser" is hand-waving over a real translation layer. That layer is gRPC-web:
        a wire-compatible subset of gRPC the browser can produce, plus an Envoy (or
        equivalent) proxy that translates it into proper gRPC for the backend. Press play to
        follow a request through:

A JavaScript client wants to call CallTool. It can't speak native gRPC because the browser won't emit HTTP/2 trailers. Envoy (or a similar proxy) bridges the gap.

        

      


      
› The choice this forces on you

      

        Adding gRPC means adding a proxy to your stack. Envoy is the canonical choice; alternatives
        are grpcwebproxy (small Go binary) and Caddy with the gRPC-web module. None
        of them are heavy, but all of them are extra moving parts compared to Streamable HTTP,
        which is just HTTP. If your only browser client is the Anthropic-hosted Claude.ai
        connector, you don't need gRPC-web at all — that's what Streamable HTTP is for.
        gRPC-web becomes interesting when you have your own SPA that wants to talk to the same
        gRPC backend your service-mesh sidecars are using.
      



# envoy.yaml — the relevant filter chain
http_filters:
  - name: envoy.filters.http.grpc_web        # the translator
  - name: envoy.filters.http.cors            # browser CORS handling
  - name: envoy.filters.http.router

routes:
  - match: { prefix: "/mcp.extension.v1.MCPExtension/" }
    route:
      cluster: mcp_backend
      timeout: 0s        # long streams
    cors:
      allow_origin_string_match:
        - { exact: "https://app.example.com" }
      allow_methods: "GET, PUT, DELETE, POST, OPTIONS"
      allow_headers: "keep-alive,user-agent,cache-control,content-type,
                     content-transfer-encoding,custom-header-1,x-accept-content-transfer-encoding,
                     x-accept-response-streaming,x-user-agent,x-grpc-web,grpc-timeout"


      

        
gRPC-web doesn't do client-streaming or bidi

        

          gRPC-web supports unary and server-streaming only.
          Client-streaming and bidirectional streaming require capabilities the browser doesn't
          have. If you've designed your gRPC service to use bidirectional streams (the case where
          gRPC's advantage is most visible), browser clients can't consume those methods — you'd
          fall back to Streamable HTTP or WebSockets for those flows. Account for this when you
          choose mode-per-method.
        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 5.4

          
Five checks before Module 5.5.

        

      


      

        
Q1.A teammate proposes "let's switch our MCP server from Streamable HTTP to gRPC for better performance". Your two-user internal tool sees 0.4 RPS at peak. What's the right response?

        

          
          
          
          
        

        
Right. gRPC is the right answer for specific workloads (high-RPS backend-to-backend, multi-language strict typing, true bidi streams). A 0.4 RPS internal tool is none of those. Pick the transport whose constraints fit your traffic, not the one with the better marketing.

      


      

        
Q2.You're defining a tool call with progress updates as the work proceeds. Which gRPC streaming mode fits?

        

          
          
          
          
        

        
Exactly. Progress notifications during a long tool execution are the canonical server-stream case. It's also what SSE gives you over HTTP, so the mental model carries cleanly across transports.

      


      

        
Q3.You remove a field from a CallToolRequest message. To make the schema safely evolvable, you should:

        

          
          
          
          
        

        
Yes. Tag numbers are forever in Protobuf. reserved 4, 5; in the message body tells future maintainers (and the compiler) that those tags must never be reused. This is the rule the buf CLI enforces in CI.

      


      

        
Q4.A teammate ships a browser app that uses gRPC-web to call a service whose schema defines rpc Chat(stream Msg) returns (stream Msg). The app fails at runtime. Why?

        

          
          
          
          
        

        
Correct. This is the hard constraint of gRPC-web that catches teams off-guard. Design your gRPC service knowing browser clients can only consume two of the four streaming modes. Bidi-only methods belong to backend-to-backend traffic.

      


      

        
Q5.You're running a hybrid server with both Streamable HTTP at :8000/mcp and gRPC at :50051. To keep the two transports honest, the right discipline is:

        

          
          
          
          
        

        
Right. The win of a hybrid server is sharing everything above the transport layer. Two contracts to keep in sync is the cost; you minimize the divergence by treating the MCP side as canonical. The gRPC layer is a projection, not a parallel universe.
