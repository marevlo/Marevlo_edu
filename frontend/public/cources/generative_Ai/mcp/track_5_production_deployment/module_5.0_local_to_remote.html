01  Why migrate at all

Three transports, only one of which still belongs in new code.

MCP defines transports as the wire between client and server. The spec has shipped three of them. Knowing which is which — and which exists only for backwards compat — is the gating decision for the rest of this track. Switch between the three to see how their wire shape, topology, and operational properties differ:

Compare · pick a transport
Transport Matrix · stdio vs HTTP+SSE vs Streamable HTTP
Wire & topology
Operational properties

› Why stdio doesn't make it to production

Stdio is a perfect transport for one specific case: the host process launches the server as a child and talks to it over the resulting pipe. One server per user, lifetime tied to the host. It's how Claude Desktop runs your filesystem server today, and the right answer for any tool that is genuinely scoped to one machine and one user. The moment you need any of the following, stdio is wrong: multiple concurrent users, hosting on a machine that isn't the user's, central updates without redistributing the binary, OAuth flows, observability through standard HTTP infra. All of those are why a server moves to a network transport.

› Why HTTP+SSE was deprecated

The original network transport, introduced in 2024-11-05, used two endpoints: client POSTs JSON-RPC to /messages and holds open a separate GET /sse connection to receive responses. This works, but every operational layer between the two of you — load balancers, CDNs, HTTP/2 connection caps — has to understand the two endpoints belong to the same session. Reconnection is awkward, resuming a stream after a network blip is awkward, and the dual-channel pattern fights HTTP/2's single-connection assumptions. 2025-03-26 replaced it with Streamable HTTP.

› Why Streamable HTTP is the answer

One endpoint. The client POSTs a JSON-RPC message; the server chooses per response whether to reply with a single JSON object (Content-Type: application/json) or an SSE stream of events (Content-Type: text/event-stream). The same endpoint also accepts GET to open a long-lived server→client channel for unsolicited notifications. Sessions are identified by an Mcp-Session-Id header. The whole thing fits inside HTTP/1.1 or HTTP/2 cleanly, sits behind any standard reverse proxy, and is what every new server should ship.

The version line in the sand

If the spec date you're targeting is 2025-03-26 or newer, use Streamable HTTP. If you're maintaining a server with hosts pinned to 2024-11-05, you can support both — FastMCP can mount the legacy SSE endpoint next to the modern one — but new code should default to streamable-http only.

› The CLI equivalent

      

        If you'd rather not hard-code the transport in server.py, FastMCP's CLI takes
        it as a flag. Same server module, different command:
      


      
# Local development — what you had in Track 2
fastmcp run my_server.server:mcp

# Remote — what production looks like
fastmcp run my_server.server:mcp \
    --transport streamable-http \
    --host 0.0.0.0 \
    --port 8000 \
    --path /mcp


      

        
The path matters

        

          The default mount path is /mcp. This is conventional, not required, but every
          host that auto-discovers MCP servers (Claude Desktop's remote config, Claude.ai's
          connector form, MCP Inspector) defaults to looking there. Pick a non-default path only
          when you have a real reason — and document it loudly.
        

      


      
› What did not change

      
    
        
  • Tool definitions. Every @mcp.tool function works unmodified. The transport is below them in the stack.
    • 
        
  • Lifespan. Your db_pool and http_client still build at startup, persist across requests, drain at shutdown. Lifespan is bound to the server process, not the transport.
    • 
        
  • The Context object. ctx.info(), ctx.report_progress(), ctx.sample() — identical API. The transport changes how those bytes leave the process; it does not change your code that produces them.
    • 
        
  • JSON-RPC semantics. Same method names, same params, same notifications. A Wireshark capture against either transport shows the same envelope; only the framing differs.
    • 
      

    


    
    

      
03  Sessions & sticky routing

      
What an MCP session is, and why your load balancer needs to know about it.


      

        Going remote is more than swapping bytes for HTTP requests. The protocol assumes a stateful
        conversation between one client and one server: capabilities are negotiated once during
        initialize, ongoing notifications stream from server to client over a held-open
        channel, and the server may keep per-session state in memory. The unit that ties all of
        that together is the session, identified by the
        Mcp-Session-Id response header on the initialize call and echoed
        on every subsequent request from the client.
      


      

        On a single server, none of this matters — every request lands on the only process. The
        moment you put a load balancer in front of two replicas, you're forced to answer a
        question: which replica handles request #2 of session S? Click through the
        three scenarios to see what happens:

› The three failure modes without stickiness

      
    
        
  • Lost session state. The server that handled initialize stored the negotiated capability set, the protocol version, and any per-session data. Replica B never saw it. Replica B returns session not found or, worse, creates a phantom one and the protocol drifts.
    • 
        
  • Broken SSE streams. The held-open GET for server→client notifications is on replica A. When the client POSTs a tool call to replica B and replica B's tool fires a notifications/progress, that notification arrives on... nothing. It's published locally on B; the client is listening to A.
    • 
        
  • Cancellation goes nowhere. Client posts notifications/cancelled for request req_42. The router lands it on B. The actual tool execution is on A. Cancellation is ignored; the tool runs to completion despite the user closing the tab.
    • 
      


      
› The two real options

      

        Sticky sessions are the simple answer: configure your load balancer to
        route all requests carrying a given Mcp-Session-Id to the same backend. Nginx
        does this with hash $http_mcp_session_id consistent. Most cloud LBs do it with
        cookie-based affinity. The downside: that backend becomes a single point of failure for the
        session — if it crashes, the session is gone, the client must initialize
        again.
      

      

        Shared backplane (Redis pub/sub, NATS, etc.) keeps the LB stateless. Any
        replica can handle any request because the session data lives in shared storage and the SSE
        stream is fanned out through the backplane. This is what Module 5.1 will build. More
        moving parts, but you can lose a replica without losing a user.
      


      

        
Default to sticky, graduate to backplane

        

          Sticky sessions cover the vast majority of MCP deployments and they're trivial to set up.
          Move to a Redis-backed backplane when you can quantify the cost of session loss — long
          tool runs, expensive context, paying customers. Don't over-engineer this on day one.

› The decision tree, in three questions

      
    
        
  • Is this for one team behind your VPN? Nginx or Caddy on a single VM. Anything more is over-engineering.
    • 
        
  • Do you need horizontal scale or rolling deploys? Kubernetes, with sticky-session ingress. You'll spend a week on the config and never touch it again.
    • 
        
  • Is traffic bursty (zero to many users in seconds)? Cloud Run / Fargate. You'll pay only for active sessions, and the long-poll SSE problem is what those platforms have all gotten quietly very good at.
    • 
      


      

        
The mistake we keep seeing

        

          Teams reach for Kubernetes for a two-user internal tool and end up debugging ingress
          controllers, cert-manager, and Helm charts for a server that would have run unchanged on a
          $5 VM for three years. Use the smallest thing that fits. Promote up only when the
          workload demands it.

A tools/call from a Claude.ai conversation will cross five layers. Each one terminates something or forwards it onward. Watch how the Mcp-Session-Id header stays attached the whole way.

        

      


      
› The five layers and what each one owns

      
    
        
  • 1 · DNS + TLS. Public DNS resolves your hostname to your edge IP. TLS is terminated here (Cloudflare, your cloud load balancer, or your edge proxy). Below this line, traffic is plain HTTP/1.1 or HTTP/2 inside your network.
    • 
        
  • 2 · Ingress / reverse proxy. Nginx, Caddy, or your cloud ingress. Owns hostname routing, path matching (/mcp → the MCP service), and the long timeout configuration that SSE streams require. Critically: must proxy_buffering off for SSE, or events sit in a buffer until they're "full enough".
    • 
        
  • 3 · Sticky load balancer. Within the MCP service, picks a backend based on Mcp-Session-Id. For Kubernetes, this is the Service with session-affinity annotations; for Nginx, the upstream block with hash $http_mcp_session_id.
    • 
        
  • 4 · Your server process. FastMCP receives the JSON-RPC tools/call, dispatches to your decorated function, builds the result. If the response is streaming, FastMCP writes SSE events back through the chain on the open POST response.
    • 
        
  • 5 · SSE return path. Events flow back through every layer in reverse, each holding the connection open. If any layer buffers, the stream stalls until the buffer flushes. This is the single most common production bug; we'll fix it in Module 5.1.
    • 
      


      
› Reference Nginx config (the parts that actually matter)



upstream mcp_backends {
    # Hash on the session header — sticky sessions, no cookies needed
    hash $http_mcp_session_id consistent;
    server mcp-1.internal:8000;
    server mcp-2.internal:8000;
    server mcp-3.internal:8000;
}

server {
    listen 443 ssl http2;
    server_name mcp.example.com;

    location /mcp {
        proxy_pass http://mcp_backends;
        proxy_http_version 1.1;

        # --- The SSE-critical lines ---
        proxy_buffering off;          # events flush immediately
        proxy_cache off;
        proxy_read_timeout 24h;        # long-lived GET for notifications
        proxy_send_timeout 24h;

        # --- Standard hardening ---
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}


      

        
The four lines that catch everyone

        

          proxy_buffering off, proxy_cache off, and the two long
          *_timeout values are non-negotiable for any reverse proxy in front of an MCP
          server. Without them, SSE streams stutter or die after 60 seconds. This is true for
          Nginx, HAProxy, Caddy (set flush_interval -1), and every cloud ingress —
          read your docs for the equivalent.
        

      

    


    
    

      

        

          
        

        

          
Knowledge Check · End of Module 5.0

          
Five checks before Module 5.1.

        

      


      

        
Q1.A teammate is starting a new MCP server today, targeting Claude.ai's connector form. Which transport should they pick?

        

          
          
          
          
        

        
Correct. Streamable HTTP replaced the dual-endpoint HTTP+SSE pair in 2025-03-26 and is what every new server should ship. WebSockets isn't an MCP transport.

      


      

        
Q2.You have two MCP server replicas behind a round-robin load balancer. A user runs initialize, then calls a tool. The tool call returns session not found. Most likely cause?

        

          
          
          
          
        

        
Right. This is the canonical "no sticky sessions" failure. Either hash on Mcp-Session-Id at the load balancer or move session state to Redis (Module 5.1).

      


      

        
Q3.Your Nginx reverse proxy sits in front of an MCP server. Tool calls work but progress notifications arrive in bursts of 5-10 after long pauses. Which directive fixes this?

        

          
          
          
          
        

        
Yes. The classic SSE-behind-proxy symptom. Nginx buffers responses by default; for SSE you must turn buffering off so each event flushes to the client immediately.

      


      

        
Q4.A small team is deploying an internal MCP server used by ~30 employees behind their VPN. Which pattern fits best?

        

          
          
          
          
        

        
Correct. Use the smallest thing that fits. 30 internal users on a single VM with a reverse proxy is the right answer. Graduate up only when load or reliability demands it.

      


      

        
Q5.You're migrating server.py from stdio to streamable-http. Which line is enough to change the transport?

        

          
          
          
          
        

        
Exactly. The transport switch is a single argument. Every tool, resource, prompt, and lifespan you wrote in Track 2 works unchanged — FastMCP's HTTP server is built in and serves the same protocol.
