Track 05 — Production, Deployment & Ecosystem / Module 5.2

Observability for MCP.

Module 5.1 built a cluster that can take load. The next question is the one every on-call engineer asks at 3am: what is it actually doing right now? This module wires up OpenTelemetry across the MCP lifecycle, instruments per-tool metrics, structures your logs so they're queryable, and turns the numbers into SLOs you can alert on without being woken up by noise.

01  The three pillars

Three signal types, three questions. Don't confuse them.

"Observability" is the union of three signal types: metrics, logs, and traces. Each one answers a different question cheaply, and answers the other two expensively (or not at all). The most common production observability bug is using one to do another's job — counting log lines for rate limits, scanning traces for aggregate latency, building dashboards from individual log events. Switch between the pillars to see what each one is for, what it costs, and what it emits on the wire:

› The mental model: which question, which pillar

  • "Is the system healthy right now?" → metrics. Pre-aggregated time series, designed for fast aggregation and threshold alerts. P95 latency, RPS, error rate. Cheap to store, queryable in milliseconds, lose individual-request detail.
  • "What happened during this specific request?" → traces. A per-request graph of every span (every operation, every downstream call) with parent-child relationships. Expensive but invaluable; you sample (e.g. 1%) in production.
  • "What does the system say happened?" → logs. The narrative. Structured events for grep-after-the-fact diagnosis. Cheap to produce, expensive to query at scale, the only thing that's always there.
Correlate, don't pick

The mature production posture isn't "we use metrics instead of logs" — it's all three, correlated by shared identifiers. A metric alert links to a Grafana panel that links to a Loki log query filtered by trace_id that links to the full Jaeger trace. Tools like Grafana, Datadog, and Honeycomb make that link cheap; designing your instrumentation so the IDs are everywhere is what makes it possible.

02  OTel instrumentation

The trace of a single tool call, span by span.

OpenTelemetry is the vendor-neutral standard for emitting traces, metrics, and logs from your code. You instrument once (in OTel's API), choose an exporter (Jaeger, Tempo, Honeycomb, Datadog), and switch backends without changing code. For an MCP server, the trace shape is very specific: one root span per JSON-RPC method, child spans for downstream HTTP/DB calls, and special spans for sampling/elicitation that go back to the host. › Wiring OTel into FastMCP

The boilerplate is small and stays in lifespan. Set up a tracer provider, register an OTLP exporter pointing at your collector, and decorate the tool functions you care about. The FastMCP middleware can attach a span around every JSON-RPC method automatically — what you add manually are the spans inside your tools. 

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.resources import Resource

# --- Once, in lifespan startup ---
resource = Resource.create({
    "service.name": "my-mcp-server",
    "service.version": "1.4.2",
    "deployment.environment": settings.ENV,
})
provider = TracerProvider(resource=resource)
provider.add_span_processor(
    BatchSpanProcessor(
        OTLPSpanExporter(endpoint="http://otel-collector:4317")
    )
)
trace.set_tracer_provider(provider)

tracer = trace.get_tracer("my-mcp-server")

# --- Inside a tool ---
@mcp.tool
async def deep_research(query: str, ctx: Context) -> str:
    with tracer.start_as_current_span("deep_research") as span:
        span.set_attribute("mcp.tool.name", "deep_research")
        span.set_attribute("mcp.session.id", ctx.session_id)
        span.set_attribute("query.length", len(query))

        docs = await fetch_docs(query)      # auto-instrumented httpx span
        span.set_attribute("docs.count", len(docs))

        return await summarise(docs)
Use the semantic conventions

OTel publishes a stable list of semantic conventions — agreed-upon attribute names like http.request.method, db.system, gen_ai.request.model. Use those names rather than inventing your own; every backend's dashboards and queries are built around them. The gen_ai.* namespace specifically covers LLM/agent observability and is the right place to put model name, token counts, and completion latency.

03  Per-tool metrics

RED, USE, and the dashboard you should be staring at.

The default metric framework for request-driven services is RED: Rate (requests per second), Errors (failures per second or as a ratio), and Duration (latency distribution). For an MCP server, RED is computed per tool — because aggregate "MCP latency" hides the one slow tool dragging down a fast-tool service. › Emitting RED from FastMCP

OTel ships a Metrics API parallel to its Trace API. You get a Meter, create instruments (counters, histograms), and observe per-call. The exporter ships them to your collector which forwards them to Prometheus (or whatever you've configured). Three instruments per tool covers the full RED picture: 

from opentelemetry import metrics

meter = metrics.get_meter("my-mcp-server")

tool_calls = meter.create_counter(
    name="mcp.tool.calls",
    unit="1",
    description="Total tool invocations",
)
tool_errors = meter.create_counter(
    name="mcp.tool.errors",
    unit="1",
    description="Total tool failures",
)
tool_duration = meter.create_histogram(
    name="mcp.tool.duration",
    unit="ms",
    description="Tool execution time",
)

@mcp.tool
async def search(q: str, ctx: Context) -> list:
    attrs = {"tool.name": "search"}
    tool_calls.add(1, attrs)
    t0 = time.monotonic()
    try:
        result = await do_search(q)
        attrs["outcome"] = "ok"
        return result
    except Exception as e:
        attrs["outcome"] = "error"
        attrs["error.type"] = type(e).__name__
        tool_errors.add(1, attrs)
        raise
    finally:
        elapsed = (time.monotonic() - t0) * 1000
        tool_duration.record(elapsed, attrs)
Watch your cardinality

Every distinct combination of attribute values is a new time series. Tagging by tool.name and outcome is fine (~10 tools × 2 outcomes = 20 series). Tagging by session.id or user.id is catastrophic — you'll have millions of series, your Prometheus will hate you, and your bill (if you're on a managed service) will be a surprise. High-cardinality fields belong on traces and logs, not metrics.

› The structlog setup that actually works

      

        Python's structlog wraps the stdlib logging module and lets you
        emit JSON natively without giving up familiar idioms. The OTel logging instrumentation
        injects trace_id and span_id into every log record
        automatically — which is what makes the correlation we keep talking about work in
        practice.
      



import structlog, logging, sys
from opentelemetry.instrumentation.logging import LoggingInstrumentor

# Inject trace_id and span_id into every record
LoggingInstrumentor().instrument(set_logging_format=True)

structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,     # request-scoped vars
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.EventRenamer("message"),
        structlog.processors.JSONRenderer(),         # stdout = JSON lines
    ],
    wrapper_class=structlog.make_filtering_bound_logger(logging.INFO),
)
log = structlog.get_logger()

# In a tool — context vars stick to every log inside this call
@mcp.tool
async def send_email(to: str, body: str, ctx: Context):
    structlog.contextvars.bind_contextvars(
        tool="send_email",
        session_id=ctx.session_id,
        user_id=ctx.user_id,
    )
    log.info("send_email.start", to=to, body_bytes=len(body))
    try:
        result = await mailgun.send(to=to, body=body)
        log.info("send_email.ok", message_id=result.id)
        return result
    except Exception as e:
        log.error("send_email.fail", error=str(e),
                  error_type=type(e).__name__)
        raise


      

        
Sample your logs, not your traces (counter-intuitively)

        

          Production traffic generates way more log volume than trace volume — every tool
          call might emit 5–10 log lines vs one trace. Tail-sample your traces (keep 100% of slow
          or error traces, 1% of the rest) but rate-limit your logs at the source. structlog
          and OTel both ship sampler primitives; use them. A 1% sampling of healthy traffic with
          100% of errors is the standard recipe.
        

      

    


    
    

      
05  SLOs & alerting

      
From metrics to a pager that only rings when it should.

› Multi-window burn-rate alerts (why your pager stops crying wolf)

      

        A naive alert ("error rate > 1%") fires on every blip. SRE practice replaced this with
        burn-rate alerts: alert when error budget is being consumed
        too fast for the chosen window. The trick is using two windows — a short
        one (5 min, fast burn) and a long one (1 hour, sustained burn) — both must trip before
        paging. Quick blips don't cross both; real degradation does.
      



# Prometheus alert: page only if BOTH windows are burning fast
groups:
- name: mcp_slo_burnrate
  rules:
  - alert: MCPToolErrorBudgetBurnFast
    expr: |
      (
        sum(rate(mcp_tool_errors_total{tool="research"}[5m]))
        /
        sum(rate(mcp_tool_calls_total{tool="research"}[5m]))
      ) > (14.4 * 0.01)        # 14.4× budget burn over 5m
      and
      (
        sum(rate(mcp_tool_errors_total{tool="research"}[1h]))
        /
        sum(rate(mcp_tool_calls_total{tool="research"}[1h]))
      ) > (14.4 * 0.01)        # AND also over 1h
    for: 2m
    labels: { severity: page }
    annotations:
      summary: "research tool burning error budget at 14.4× sustained rate"


      

        
Don't alert on everything you measure

        

          The most expensive observability mistake is treating every metric as alert-worthy.
          You should be measuring hundreds of things. You should be paging on maybe five. Alert on
          symptoms (high error rate, latency past SLO, queue depth blowing up) — not causes
          (high CPU, low disk). Causes are what you look at after the symptom alert fires.

Five checks before Module 5.3.

        

      


      

        
Q1.Your dashboard shows aggregate "MCP P95 latency" sitting at 1.4s. The aggregate looks bad. Which signal do you reach for next?

        

          
          
          
          
        

        
Right. Per-tool RED isn't optional — aggregate latency is almost always a Simpson's paradox in disguise. Drill from the aggregate to the per-tool breakdown first, then dive into traces for the offending tool.

      


      

        
Q2.A junior engineer adds session_id as an attribute on the mcp.tool.duration histogram. The Prometheus pod OOMs the next day. Why?

        

          
          
          
          
        

        
Exactly. Cardinality discipline is the single most important rule when designing metrics. Bounded sets (tool name, error type, region) are fine. Unbounded sets (session, user, request id) explode the time series count and break the store.

      


      

        
Q3.A customer reports their tool call hung for 8 seconds before returning. Which signal answers "what exactly was happening in those 8 seconds"?

        

          
          
          
          
        

        
Yes. Metrics answer "is the system healthy" (aggregate). Traces answer "what happened in this one request" (per-request). For a one-off slow call, you need the trace — and you find it by querying for spans with that trace_id (which your customer reported, because you propagated it through your structured logs).

      


      

        
Q4.Your alert "error rate > 1% for 5 minutes" wakes the on-call engineer 12 times a week. Each time it's a 30-second blip that resolved itself. The right fix is:

        

          
          
          
          
        

        
Correct. Multi-window burn-rate alerts are the SRE-standard way to distinguish "real degradation" from "30-second blip". Two windows agreeing eliminates noise while still catching sustained problems quickly. (D) sort of works but delays detection too much for sustained outages.

      


      

        
Q5.You're staring at a JSON log line in Loki. It contains tool=research, session_id=S1, and trace_id=4f17…. What does that trace_id buy you?

        

          
          
          
          
        

        
Exactly. Correlation IDs are what unify the three pillars. A metric alert links to a Grafana panel, which links to a Loki query filtered by trace_id, which links to the full trace. Designing your instrumentation so trace_id is everywhere is what makes that pivot work in production.
