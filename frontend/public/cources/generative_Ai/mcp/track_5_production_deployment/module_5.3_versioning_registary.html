Track 05 — Production, Deployment & Ecosystem / Module 5.3

Versioning, registry & governance.

A server that ships into the world becomes part of an ecosystem. Clients pinned to old spec versions still need to work. Features you shipped last year need to retire without breaking anyone. Someone needs to be able to find your server. This module is about all of that — protocol negotiation, deprecation policy, publishing to the official Registry, and the standards bodies (SEPs, AAIF, NIST) that quietly govern the protocol's evolution.

01  Protocol negotiation

The initialize handshake, byte by byte.

MCP's version contract is negotiated on the very first message of every session. The client opens with the latest spec version it supports; the server replies with the highest version it supports that's ≤ the client's offer, or with the highest it supports if it can't go that high. Both sides also advertise their capabilities — the optional features they implement. Press play to watch three scenarios — clean match, downgrade, and outright incompatibility:

› The version-string rule

Protocol versions in MCP are not SemVer. They're dated revisions — 2024-11-05, 2025-03-26, 2025-06-18, 2026-01-15. The format matters: it makes the rule "newer date wins" trivial and resists the SemVer ambiguity where major-version bumps are politically expensive to release. The latest revision your server supports goes in your serverInfo; the latest your client supports goes in the initialize params. 

# In FastMCP — the negotiation is automatic
from fastmcp import FastMCP

mcp = FastMCP(
    name="my-server",
    version="1.4.2",                          # YOUR server version (SemVer)
)
# FastMCP picks the MCP spec version from the SDK release.
# You don't write it; the SDK declares its highest supported version.

# If you need to inspect what was negotiated for THIS session:
@mcp.tool
async def whoami(ctx: Context) -> dict:
    return {
        "client": ctx.client_info.name,
        "client_version": ctx.client_info.version,
        "protocol": ctx.protocol_version,        # e.g. "2025-06-18"
        "capabilities": ctx.client_capabilities,  # sampling? roots?
    }
Negotiate once, trust forever (per session)

The version chosen on initialize applies for the entire session. Tools you register that depend on a feature added in a newer revision (elicitation, structured tool output, OAuth resource indicators) must check ctx.protocol_version and fall back gracefully if the negotiated version is older. The server doesn't re-negotiate; the client opens a new session with a new initialize if it needs a different version.

› The "two-version window" rule

For most production servers the right policy is to support the current and the immediately previous spec revision. That window is ~6 months wide in practice and covers every host that's been updated within a release cycle. Older than that, you return an explicit "incompatible" error during initialize — the client gets a real signal, the user gets an update prompt, and you don't ship dead code paths for clients that hardly exist anymore.

  • Drop support before features rot. Each retained spec revision is a code path you must test. Two is manageable, three is a chore, four is unmaintainable.
  • Telemetry tells you when it's safe. Module 5.2 tagged metrics by protocol_version. Look at the distribution. When 2024-11-05 traffic drops under 1% over two weeks, you can sunset it.
  • Refuse, don't pretend. When you do drop a revision, fail initialize with a clear error and a link to the upgrade docs. Don't silently downgrade behavior — that creates phantom bugs.
03  Deprecation lifecycle

How a feature retires without breaking the people who depended on it.

Removing something is the hardest thing a protocol ever does. The mistake is to remove first and announce later; the discipline is to announce, mark, sunset, then remove, with months between each step. Click each stage to see what your code does at that stage — and what your communication does:

› The timeline that works in practice

These windows are conventions, not gospel. Adjust them to your traffic. The principle is invariant: each stage must be long enough that any active client has a real chance to see it. For an MCP server with weekly active integrations, the standard cadence is:

  • Active → Deprecated: the day you announce. Documentation updated, replacement linked, no behavioral change.
  • Deprecated → Sunset: 90 days minimum. Server starts emitting a Sunset header and a Warning header on every response touching the deprecated feature. Logs include a per-client deprecation counter.
  • Sunset → Removed: 90 more days. Sunset date is a hard cutoff. After it, the feature still exists but logs every use as a warning. Telemetry tells you who's still calling.
  • Removed: the response is now 410 Gone (HTTP) or a JSON-RPC error (-32601 Method not found). Final.
RFC 8594 already designed this for you

You don't need to invent the deprecation header. RFC 8594 defines the Sunset HTTP response header for exactly this purpose: a UTC timestamp telling clients when a resource will stop responding. Pair it with RFC 7234's Warning header for the human-readable explanation. Standard tooling (Postman, OpenAPI tooling, monitoring agents) already parses these.

04  Publishing to the Registry

› What lives in server.json — and what doesn't

The manifest is small and intentional. It tells the registry who you are (a namespaced ID), what you do (descriptive metadata), where you live (URL or install command), and how to talk to you (transport + spec version). It does not describe your tools — those are discovered dynamically once a client connects. The registry doesn't index your tool list; that would freeze it.

Namespace governance is real

The registry uses reverse-DNS namespaces (com.example.search) and ownership is proven by DNS TXT records or GitHub-org control. This is not a vanity policy — it's how the registry prevents a malicious server from publishing itself as com.anthropic.claude and tricking users. If you can't prove ownership of a namespace, you can't publish under it.

05  The standards landscape

The bodies that govern the protocol, and what each of them owns.

MCP is not Anthropic's private spec. It evolves through a public process with overlapping bodies of authority — protocol changes go through SEPs (spec enhancement proposals), identity flows reference AAIF (Agent Authorization Interoperability Framework), and any enterprise deployment will eventually need to map to NIST AI standards. › The relationship between them

Think of it as four layers stacked on top of each other. Protocol mechanics (transports, capabilities, method shapes) — governed by SEPs and the MCP working groups. Identity and authorization — deferred almost entirely to OAuth 2.1 and the AAIF reference patterns. Discoverability — the Registry, run by Anthropic with community governance. Compliance and risk — NIST AI RMF if you're an enterprise, plus EU AI Act considerations if you're in the EU. None of these are rivals; they cover different concerns and interlock cleanly.

Read the SEP index before you propose anything

New ideas for the protocol go through SEPs — short, structured documents proposing a change, motivating it, sketching the wire impact, and naming an author who will see it through. Read the existing accepted, rejected, and in-flight SEPs first. The vast majority of "I wish MCP had X" thoughts are already covered, either as accepted-but-not- shipped, or as rejected-for-a-good-reason. Save your draft for the actually-novel ones.

Knowledge Check · End of Module 5.3
Five checks before Module 5.4.
Q1.A client opens initialize with protocolVersion: "2025-06-18". Your server's highest supported revision is 2025-03-26. What does the server reply with?
Correct. The server replies with the highest version it can speak, even if the client offered higher. The client gets to decide if the negotiated version is acceptable; downgrade is graceful, not forced to the lowest.
Q2.You're sunsetting a tool you shipped 18 months ago. Your traffic shows 0.4% of calls still using it. What's the minimum-friction retirement sequence?
Right. The four stages — active, deprecated, sunset, removed — give clients the signal trail they need to migrate. RFC 8594's Sunset header carries the date programmatically; logs and telemetry show you when traffic has actually drained.
Q3.You publish a server to the Registry under com.example.search. The Registry rejects the submission. Most likely cause?
Yes. Namespace governance is what stops a malicious actor from publishing as com.anthropic.claude. DNS TXT records or GitHub-org ownership of the namespace's root are the standard proofs.
Q4.You have an idea for a new MCP feature — a standardized "tool cancellation reason" field. What's the right next step?
Exactly. SEPs are the structured front door for protocol changes. Reading existing ones first prevents you from re-proposing rejected ideas. The discipline is identical to Python's PEPs and Rust's RFCs.
Q5.Your client lawyer asks "how does our MCP deployment align with NIST AI RMF?" Where does the AI RMF actually sit in the stack?
Right. The AI RMF is a compliance / governance framework, not a technical spec. MCP's primitives (OAuth, capability negotiation, audit logging) become the implementation surface that your AI RMF program maps to. The two interlock; one doesn't replace the other.
