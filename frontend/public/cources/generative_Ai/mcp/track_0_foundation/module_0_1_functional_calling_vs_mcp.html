Track 00 — Foundations / Module 0.1

Function calling vs MCP.

Function calling is what the model emits. MCP is how the host finds out a function exists in the first place. The two are not competitors — they sit at different layers, and seeing the layering is the conceptual leap that makes the rest of the protocol obvious.

01  Recap

What function calling actually is.

A two-minute reset, because the rest of the module hinges on being precise here. The LLM does not execute anything. When a provider says a model "called a function," what physically happened is this: the model emitted a structured JSON object naming a function and arguments, and your code — not the model — dispatched the call.

Hold onto that distinction, because it is the single most misunderstood thing in the whole space. A language model is a next-token predictorA model trained only to predict the most likely next piece of text. It has no hands, no network access, and no ability to run code — it can only produce text, including text that looks like a function call.. It cannot open a socket, read a file, or hit an API. When it "calls" get_weather, all it did was produce text in a shape your runtime agreed to recognise. Everything downstream — the actual HTTP request, the database read — is ordinary code that you wrote, triggered by you, running on your machine.

The loop looks like this:

Host →
Sends the user's message plus a tools=[…] list (JSON-Schema definitions of available functions) to the model API.
Model →
Responds with either plain text or a tool_use block: { "name": "get_weather", "input": { "city": "Singapore" } }.
Host →
Pattern-matches the name, runs the real function (HTTP call, DB query, whatever), captures the result.
Host →
Sends the result back as a tool_result message and the conversation continues.
Model →
Now sees the result, decides whether to call another tool or write a final answer.

Two things to underline. First, the tool list is static for the duration of the call — whatever was in tools=[…] when you hit the API is what the model can pick from. Second, the tool list lives inside the host's source code. Adding a tool means shipping code. Removing one means shipping code. This is the part MCP changes.

›Deep dive What's actually in the tools=[…] array?

Each entry is a JSON-Schema description of a function — not the function itself. The model never sees your code; it sees a name, a one-line description, and the shape of the arguments. That description is the entire interface the model reasons over, which is why tool descriptions are quietly one of the highest-leverage things you write:

{ "name": "get_weather", "description": "Get current weather for a city. Use when the user asks about temperature, rain, or conditions.", "input_schema": { "type": "object", "properties": { "city": { "type": "string", "description": "City name, e.g. Singapore" } }, "required": ["city"] } }

The model uses description to decide whether to call the tool, and input_schema to decide how. Every tool description is spent from a fixed context budget and is re-sent on every single API request in the conversation — so ten verbose tools can quietly cost you thousands of tokens per turn before the user has typed anything. This token-tax is one of the reasons dynamic, discovered tool lists (the MCP model) matter.

⚡ Check yourself
A model responds with a tool_use block for delete_database(). At that instant, what has happened?
Exactly. A tool_use block is just structured text the model produced. No code runs until your host pattern-matches the name and dispatches it — which is precisely why the host is where permission checks, sandboxing, and human-in-the-loop approval live. The model proposes; the host disposes.

02  What MCP adds

From hard-coded list to discovered list.

Ad-hoc tool calling — what we just walked through — works fine when one team owns both the host and the tools. The moment any of the following becomes true, the cracks show: a tool team that doesn't own the host; a host that wants tools it didn't write; tools that should appear and disappear at runtime; tools that need to outlive a single provider's function-calling format.

MCP adds four properties that ad-hoc tool calling doesn't have. Each one removes a constraint that ad-hoc forces you to live with:

PropertyAd-hoc tool callingMCP
Discoverability Tools listed in code, compiled into the host. tools/list at runtime — the host asks the server what it offers.
Standardisation Each provider's format (OpenAI's tool_call, Anthropic's tool_use, …). JSON-RPC 2.0 over a defined transport, JSON Schema for params, defined error codes.
Dynamic tool lists List frozen per API call. notifications/tools/list_changed — server can add/remove tools mid-session.
Client portability Tools tied to one host's code path. Same server works in Claude Desktop, Cursor, your agent — anything that speaks MCP.

The portability point matters more than it looks. A GitHub MCP server, written once, is reachable from every MCP-capable host on the planet. Compare with the old world: every host that wanted GitHub integration wrote its own GitHub adapter, with its own auth handling, its own rate-limit logic, its own bugs. MCP turns "GitHub integration" into a package you install, not a feature you build.

That's the famous M×N problem, and it's worth feeling rather than just reading. Without a shared protocol, connecting M hosts to N tools means somebody writes a bespoke adapter for every pair: M × N of them. With MCP as a common waist, each host speaks the protocol once and each tool speaks it once: M + N. Drag the sliders below and watch the line count — and the cost of adding one more tool — diverge.

Notice the cruel asymmetry: in the M×N world, adding a single new host means writing N new adapters, and adding a single new tool means writing M new ones. Integration cost grows with the product. Under MCP it grows with the sum — one new conformant connector and you're done. At small scale the protocol looks like overhead; past a handful of each, it's the only thing that keeps the matrix from eating your team.

›Deep dive So what is MCP actually made of, on the wire?

MCP isn't magic — it's a thin, boring, well-specified contract, which is exactly the point. A boring contract is one many parties can implement independently and trust. The pieces:

  • JSON-RPC 2.0 as the message format — request/response with method, params, id, plus one-way notifications. Decades-old, dead simple, language-agnostic.
  • A transport underneath it: stdio for a local subprocess server, or Streamable HTTPAn HTTP-based transport that supports server-to-client streaming (via Server-Sent Events) so the server can push notifications like "my tool list changed" without the client polling. for remote/networked servers.
  • A capability handshake — on connect, client and server exchange an initialize message declaring what each supports (tools, resources, prompts, sampling) before any real traffic. Nobody assumes; everybody negotiates.
  • Three core primitives a server can expose: tools (model-callable actions), resources (readable context/data), and prompts (reusable templated workflows).

The genius is in the restraint: MCP standardises the envelope and the choreography, not your business logic. Your get_weather still does whatever it wants inside — MCP only governs how it's announced, discovered, and invoked across a boundary.

⚡ Check yourself
You run 3 AI hosts and want each to reach 4 different tools. A 5th tool arrives. Without MCP, how many new adapters must you write just for that one tool?
Right — without a shared protocol, every host needs its own integration with the new tool, so a single tool costs M = 3 adapters. Under MCP the tool implements the protocol once and all three hosts reach it for free. That product-vs-sum gap is the entire economic argument for MCP.
03  Where each lives in the stack

MCP, function calling, and A2A are not competitors.

This is the most common confusion online — "Should I use MCP or function calling?" The question is malformed. They occupy different layers. A working agent uses all three at once, doing different jobs. Here's the layering:

A2A — Agent ↔ Agent
top of the stack
How independent agents talk to each other — delegation, task hand-off, shared state across organisational boundaries. Google's Agent2Agent protocol is the prominent example. A2A is about agents-as-peers; the agent you delegate to is itself running its own loop.
Function calling — Agent ↔ LLM
middle of the stack
How the model expresses "I want to invoke X." A capability of the model API (Anthropic's tool_use, OpenAI's tool_calls, Gemini's functionCall). It is the language between an agent and the LLM that powers it. The model is on one side; your code is on the other.
MCP — Agent ↔ Tools / Data
bottom of the stack
How the host discovers and invokes external capabilities — tools, resources, prompts, files, APIs. Below the model entirely. A model never speaks MCP directly; the host translates between the model's function-call format and the MCP wire.
The hand-off

Inside the host, the translation looks like this: at startup, the host calls tools/list on every connected MCP server, collects the results, and reformats each one into the function-calling format its current LLM expects. When the model emits a tool_use, the host routes it back to the right MCP server as a tools/call. Function calling is the model's vocabulary; MCP is the wire; the host bridges them.

›Deep dive Why doesn't the model just speak MCP itself?

A reasonable question — if MCP is how tools are discovered and invoked, why route everything through the host instead of letting the model talk to servers directly? Two reasons, both structural:

  • The model only produces text. It has no execution context, no open connections, no clock. It literally cannot send a JSON-RPC request over a socket — it can only emit a tool_use block and wait. Something else must hold the connections. That something is the host.
  • The host is the trust boundary. Auth tokens, rate limits, user-approval prompts, audit logs, sandboxing, and which servers are even allowed — all of that belongs to the host, not the model. Putting the model directly on the wire would hand an untrusted text-predictor your credentials.

So the division of labour is deliberate: function calling is how the model expresses intent, and MCP is how the host acts on it safely. The host is the adult in the room standing between a creative-but-untrusted model and the real world.

04  Decision framework

When MCP pays for itself — and when it doesn't.

MCP is not free. It adds a JSON-RPC layer, a handshake, a process-or-network boundary, and a schema-validation step that direct in-process calls don't have. For some workloads that cost is irrelevant; for others it's a deal-breaker. Be honest about which side you're on:

Reach for MCP when

  • The same capability must be reachable from more than one host (Claude Desktop, Cursor, your own agent).
  • Tools should be discovered dynamically — you don't want a redeploy every time a tool is added.
  • You're shipping for third parties to plug into, not just your own product.
  • Governance matters: registry publishing, capability versioning, a defined auth story, signed tool descriptions.
  • The tool surface is owned by a different team than the agent surface — MCP is the contract between them.
  • You want the option to run the tool out-of-process for isolation or sandboxing.

Stick with a direct call when

  • Latency-critical inner-loop calls — MCP's framing overhead dominates for sub-millisecond ops.
  • The "tool" is a helper used only by your own agent's code path, never exposed to a model.
  • You control both ends and they will never decouple. A protocol gains nothing here.
  • The capability already has a perfectly good REST API and only one client will ever consume it.
  • You are prototyping — get the first tool calls working in-process, then refactor to MCP once a second consumer appears.

The rule that holds up best in practice: start in-process, move to MCP at the second consumer. The moment a second host (or a second team, or a third-party integration) wants the same capability, the M+N math kicks in and the protocol overhead is repaid many times over.

Lists are easy to nod along to and hard to apply to your situation. So answer five questions about the thing you're actually building, and the wizard will weigh them and make the call — with its reasoning, not just a verdict.

The deeper principle

Strip away the specifics and every question above is really asking one thing: does a boundary already exist, or will one appear? A boundary between teams, between products, between trust domains, between your code and someone else's. MCP is the cost of making a boundary explicit and standard. Where there's no boundary, that cost buys you nothing — so don't pay it. Where there is one, paying it once is far cheaper than the tangle of bespoke adapters you'd otherwise grow.

Recap — the three sentences to remember

If you forget everything else.

1
The model only emits text. "Function calling" is the model naming a tool; your host runs it.
2
MCP is discovery + standardisation. It turns a hard-coded tool list into one fetched at runtime over a shared protocol.
3
They're different layers. Function calling is the model's vocabulary; MCP is the wire; the host bridges them.

The conceptual leap this module was built around: stop asking "MCP or function calling?" and start asking "which layer is my problem on?" Once the stack is in your head, the rest of the protocol — resources, prompts, sampling, transports — slots into place as detail on a frame you already hold.
