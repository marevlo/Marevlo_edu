01  Environment

What you need on the machine.

The MCP Python stack today is built around three pieces: Python 3.10+ (FastMCP uses typing features and asyncio patterns that didn't stabilise until then), uv (Astral's Rust-based package manager — order-of-magnitude faster than pip and the de-facto standard in MCP tutorials), and the fastmcp package itself.

› Why uv and not pip

Two practical reasons. First, uv run server.py creates and caches a per-project virtual environment on the fly — your command in a Claude Desktop config can be a single line, no source venv/bin/activate dance. Second, uvx lets you run any Python tool without installing it globally (we'll use this for the Inspector). Hosts spawn your server fresh on each launch; a tool that handles env setup transparently is what makes this ergonomic.

The one-line uv install differs per OS — pick yours:

Two FastMCPs exist — pick one

fastmcp (the standalone package by Jeremiah Lowin) is the community-maintained 2.x with extras like server composition, OpenAPI generation, and a CLI. It's what most production servers use and what this course uses.

mcp.server.fastmcp ships inside Anthropic's official mcp SDK. It's the same idea but a thinner subset. If a tutorial imports from mcp.server.fastmcp import FastMCP, that's this one. APIs overlap heavily but are not identical — don't mix and match.

02  The hello server

Fifteen lines, dissected.

Create server.py:

SERVER · stdio default server.py 
from fastmcp import FastMCP

mcp = FastMCP("hello-server")


@mcp.tool
def say_hello(name: str, excited: bool = False) -> str:
    """Greet someone by name. Set excited=True for emphasis."""
    suffix = "!" if excited else "."
    return f"Hello, {name}{suffix}"


if __name__ == "__main__":
    mcp.run()  # transport="stdio" by default

Run it:

$ uv run server.py [INFO] starting hello-server on stdio [INFO] protocol version 2025-06-18 [INFO] waiting for client...

That's a real, wire-compliant MCP server. Now the question worth more than the code itself: what did FastMCP actually do?

› What @mcp.tool does under the hood

The decorator is not a marker — it does meaningful work at import time. Press play below to watch a plain Python function become a wire-ready MCP tool, one transformation at a time:

idle

At call time, when a tools/call arrives over the wire, FastMCP looks up the handler by name, validates the incoming arguments against the Pydantic model (rejecting with a JSON-RPC error if validation fails), invokes the function, and serialises the return value into the standard MCP content format ([{ "type": "text", "text": "..." }]). Async functions work identically — FastMCP awaits them.

Design implication

The docstring is not documentation for humans; it is prompt material the model will read every time it considers your tool. Vague docstrings produce vague tool selection. Module 2.1 goes deep on this — for now: write the first line as if it were a sentence the LLM has to act on, not skim.

      ›Deep dive Why is excited optional but name required?
      

        

          It comes straight from the Python signature, and FastMCP mirrors Python's own rules. A parameter
          with no default value (name: str) is mandatory, so it lands in the
          schema's required array. A parameter with a default
          (excited: bool = False) is optional, so it's omitted from required and its
          default is carried into the schema as "default": false.
        

        

          The lesson: your type hintsPython annotations like name: str. FastMCP reads them via typing.get_type_hints to build a Pydantic model, then derives JSON Schema from that. No hints → no usable schema → the model can't reliably call your tool. and defaults are your API contract. Change excited: bool = False
          to excited: bool and you've silently made it a required argument the model must always
          supply. The signature is the single source of truth — there's no separate schema to keep in sync.
        

      

    


    
    

      
⚡ Check yourself

      
You delete the docstring from say_hello but keep everything else. What breaks?

      

        
        
        
      

      

        Right. The server runs and the schema is still derived from the signature, so a client can
        invoke the tool. But description is sourced from the docstring's first line — drop it and
        the model is choosing the tool almost blind. That's why the docstring is prompt material, not a comment.
      

    

  


  
  

    
03  Transports

    
The same server, two wires.


    

      MCP defines the message format (JSON-RPC 2.0) separately from the transport that carries it.
      Two transports matter at this stage:
    


    

      

        
stdio

        
A subprocess the host owns.

        
    
          
  • Host spawns your server with command + args; messages travel as newline-delimited JSON over stdin/stdout.
    • 
          
  • stderr is left free for logging — never write protocol output to stdout.
    • 
          
  • Lifecycle is bound to the host process. Crash = restart on next host launch.
    • 
          
  • No network surface, no auth needed. Use for local-only dev tools, filesystem servers, anything you'd rather not expose.
    • 
          
  • Cannot be shared between users. One client per process.
    • 
        

      

      

        
Streamable HTTP

        
A standalone HTTP service.

        
    
          
  • Single endpoint (typically /mcp). POST for client→server, GET opens an SSE stream for server→client pushes.
    • 
          
  • One server, many clients. Auth required in production (OAuth 2.1 — Module 4.2).
    • 
          
  • Can be deployed behind Nginx, on Cloud Run, on Kubernetes. Sticky sessions matter for the SSE side.
    • 
          
  • Replaces the older two-endpoint HTTP+SSE design from the 2024 spec.
    • 
          
  • The default for any server a team or third party will consume.
    • 
        

      

    


    
Switching transports in FastMCP is a one-line change at the bottom of the file:



# stdio (default)
mcp.run()

# Streamable HTTP on port 8000, endpoint /mcp
mcp.run(transport="http", host="127.0.0.1", port=8000)


    
Hit http://127.0.0.1:8000/mcp with the right headers and you'll get a JSON-RPC response back. The application logic — your say_hello function — never changed.


    

      Choosing between them trips people up because both "work." The real question is about
      ownership and reach. Answer the prompts and the chooser will point you at the right wire —
      and tell you why:

      ›Deep dive Why must logging go to stderr on stdio?
      

        

          On the stdio transportMessages travel as newline-delimited JSON over the subprocess's stdin/stdout. The host writes requests to your stdin and reads responses from your stdout., stdout is the protocol channel — every byte written there is
          parsed by the host as JSON-RPC. A stray print("debug") injects garbage into the message
          stream, the host fails to parse it, and the connection dies with a cryptic error.
        

        

          That's why frameworks route their logs to stderr, which the host leaves alone (it
          typically surfaces in the host's MCP log panel). Rule to internalise now and never forget:
          on stdio, stdout belongs to the protocol; everything human-readable goes to stderr.
          This single mistake accounts for a large share of "my server won't connect" reports.

What actually travels between host and server.


    

      MCP Inspector is the standard tool for poking at a server without bringing a full host into
      the picture. It's a small web UI that connects to your server over stdio or HTTP and lets you
      call every primitive by hand. Critically — it shows the raw JSON-RPC messages.

Five things to notice that will matter for the rest of the course:


    
    
      
  • Every request/response pair shares an id. Notifications (Step 3) have no id and produce no reply — this is how JSON-RPC distinguishes fire-and-forget messages.
    • 
      
  • Protocol version is a date string, not semver. Negotiated at handshake; mismatches fail loudly. See Module 1.3.
    • 
      
  • Capabilities are bidirectional. The client also declares what it supports (here: roots and sampling). The server can refuse to advertise tools that need a capability the client lacks.
    • 
      
  • The inputSchema is plain JSON Schema — nothing MCP-specific. Any JSON-Schema validator works.
    • 
      
  • Tool results are an array of content blocks, not a bare string. Text is one type; image and resource_link are others. We'll use the richer types in Module 2.1.
    • 
    


    
    

      
⚡ Check yourself

      
In the trace, Step 3 (notifications/initialized) has no id field. What does that tell the server?

      

        
        
        
      

      

        Exactly. In JSON-RPC 2.0, the presence of id is the signal: a request with an
        id expects a matching response; a message without one is a notification the
        receiver acts on but never replies to. That's how the protocol cleanly separates "do this and tell me"
        from "just so you know."
      

    

  


  
  

    
05  Wiring into hosts

    
Three configs, one pattern.


    

      Every host that runs local MCP servers does the same thing: spawns your script as a subprocess
      and talks JSON-RPC over its stdio. The only thing that varies is where the host looks
      for its config file. The shape is consistent enough that you can write one mental template
      and translate.
    


    

      Better than copying a template by hand: generate yours. Fill in your real values and
      pick a host — the generator emits the exact JSON with the right top-level key
      (mcpServers vs servers), the correct config path for your OS, and a host-specific
      gotcha note. Copy it straight in.

Note the key is servers (not mcpServers) and the type field is explicit — VS Code's schema diverges slightly. For HTTP servers, use "type": "http" and a "url" instead of command/args.

      

    


    

      
Debug checklist when it won't connect

      
1) Use absolute paths — hosts don't share your shell's $PATH. 2) Check the host's MCP logs (Claude Desktop has View → Open MCP Log Folder). 3) Verify your server runs standalone in a terminal first. 4) Make sure nothing in your code writes to stdout — that channel is reserved for protocol traffic. Logging goes to stderr.
