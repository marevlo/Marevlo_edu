Track 04 — MCP Security / Module 4.3

Sandboxing & defense-in-depth.

Module 4.2 installed auth — who can call, and what they may do. This module assumes auth has failed, or the LLM has been fooled despite valid auth, and asks: how do we contain the blast? Process isolation, filesystem roots, network policy, structural input validation, human gates on action-class tools, an audit trail that survives the incident, and a registry that knows what it's allowed to install. Eight layers stacked deep — each one designed under the assumption that the others failed.

01  The defense stack

Eight layers. Each one assumes the others failed.

Defense in depth is misunderstood as "more is better." It isn't — more layers without coverage logic just adds maintenance. The stack below is complementary: each layer catches a different failure mode of the layer above, and removing any one creates a gap that the others cannot fill. Click a layer to see what it does, what it assumes is broken, and how it's implemented in practice:

Explore · the eight layers
MCP Defense-in-Depth Stack ▸ click any layer
The rule that keeps the stack honest

Every layer's threat model is "the layers above me have all failed." If you can't write that sentence about a defense, it isn't a layer — it's redundancy. Process isolation assumes input validation failed. Network policy assumes process isolation failed. Audit logging assumes everything failed. This is what makes the stack robust to single-point failures, including the most dangerous one — an LLM injection that no filter ever sees.

02  Process & filesystem sandbox

The server runs somewhere. Make it the smallest possible somewhere.

An MCP server is a process. It has a uid, a working directory, environment variables, open file descriptors, and a network namespace. Each of those is an attack surface — and each is something a sandbox can constrain. The two transports (stdio and Streamable HTTP) present different sandboxing problems; the controls below cover both.

› stdio servers — the local-process risk

A stdio server is a subprocess of the host (typically Claude Desktop). It inherits the host's uid, sees the host's filesystem, and can spawn its own subprocesses unless explicitly prevented. The host is its sandbox. A poorly-written stdio server can read your ~/.ssh/, exfil files to anywhere with network access, and persist across reboots via launch agents. Container-based execution is the only safe default for stdio servers from third parties. 

# Run a stdio MCP server in a rootless container. Minimal, but enough.
podman run \
  --rm \
  --read-only \                              # root fs immutable
  --network=none \                            # no network unless declared
  --tmpfs /tmp:rw,size=64m \
  --user 65534:65534 \                        # nobody:nobody
  --cap-drop=ALL \                            # drop all Linux capabilities
  --security-opt=no-new-privileges \
  --pids-limit=32 \                           # can't fork-bomb
  --memory=256m --memory-swap=256m \
  -v $HOME/sandbox/mcp-data:/data:ro \        # mounted read-only
  -i ghcr.io/acme/mcp-docs:v1.4.0@sha256:<digest>

› Filesystem roots — normalize before you trust

For tools that read files (read_doc, search_files), a "filesystem root" is the directory the tool may operate within. The common bug: validating the requested path with startswith(root) before resolving symlinks and Unicode. /data/users/alice/../../../etc/passwd starts with /data/users/alice. /data/users/aliсe (Cyrillic с) is a different path under normalization. Resolve, normalize, then compare: 

# Filesystem root, done correctly.
from pathlib import Path

ROOT = Path("/srv/mcp-data/users").resolve()    # canonical, no symlinks

def safe_path(user_id: str, requested: str) -> Path:
    # 1) Unicode normalize first — defeats homoglyph confusion.
    requested = unicodedata.normalize("NFKC", requested)
    # 2) Build the candidate and resolve symlinks + dotdots.
    candidate = (ROOT / user_id / requested).resolve()
    # 3) Compare against the SCOPED root — not just any prefix.
    scope = (ROOT / user_id).resolve()
    if not candidate.is_relative_to(scope):
        raise PermissionError(f"escape attempt: {requested}")
    # 4) Disallow following symlinks that point outside scope (re-check after open).
    if candidate.is_symlink():
        raise PermissionError("symlinks not permitted")
    return candidate
The four bypasses this function blocks

1. ../../../etc/passwd — resolved before comparison. 2. Symlink to outside the root — checked after resolve. 3. Cyrillic-look-alike usernames — normalized to NFKC first. 4. Cross-user reads (/users/alice/../bob/) — compared against the per-user scope, not the global root. The last one is the most commonly missed; startswith(global_root) would happily allow it.

03  Network policy

The only egress that should work is the egress you wrote down.

The exfiltration step in Module 4.1's attack chain happened because the MCP server's container could talk to arbitrary hosts on the internet. Network policy is the answer: deny by default, allowlist what's needed. For Kubernetes-deployed servers, this is a NetworkPolicy object; for VMs, it's a security group or host firewall. Either way, the rules are the same: 

# NetworkPolicy: MCP server can talk to specific backends, nothing else.
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: mcp-server-egress
spec:
  podSelector:
    matchLabels:
      app: mcp-docs
  policyTypes: [Egress]
  egress:
    # 1) DNS (needed for the allowed destinations below)
    - to: [{ namespaceSelector: { matchLabels: { name: kube-system }}}]
      ports: [{ protocol: UDP, port: 53 }]
    # 2) Postgres backend in the same cluster
    - to: [{ podSelector: { matchLabels: { app: docs-db }}}]
      ports: [{ protocol: TCP, port: 5432 }]
    # 3) Internal storage API — IP-pinned, not just hostname
    - to: [{ ipBlock: { cidr: "10.42.7.0/24" }}]
      ports: [{ protocol: TCP, port: 443 }]
    # NO rule for 0.0.0.0/0 — egress to the public internet is denied by default.

› Three patterns to bake in early

  • Pin destinations by IP/CIDR where possible. Hostname-based rules fail to DNS rebinding; an attacker can flip a record they control to point at an internal address. IP-block allowlists don't.
  • Egress proxy for the open-web case. Tools that legitimately fetch external URLs (fetch_url) should go through a forward proxy that enforces per-call allowlists, logs every connection, and strips response headers that could carry injection signals.
  • No 0.0.0.0/0, ever. Wide-open egress is the difference between a containable injection and a CISO incident. Write the explicit allowlist; it's rarely more than 3-5 destinations for a focused MCP server.
04  Input validation lab

A four-stage pipeline. Each stage fails closed.

Input validation in MCP is a pipeline, not a single check. JSON schema catches shape errors. Allowlists catch unexpected enumerations. Semantic validation catches values that look right but violate domain rules. Policy catches combinations the value-level checks can't see. Pick a payload below and watch every stage decide independently — the request only proceeds if every stage passes:

› The four-stage discipline, named

  • Stage 1 — JSON schema. Types, required fields, formats, length limits. Catches malformed requests. Fast, cheap, runs on every call.
  • Stage 2 — Allowlist. Enumerated values. Tool names, environment names, regions. Anything that has a finite known-good set goes here.
  • Stage 3 — Semantic validation. Domain-specific value checks. Path normalization, URL parsing, regex on free-text. The "would the value cause harm even though it parses?" check.
  • Stage 4 — Policy (OPA). The check from Module 4.2 — role, tenancy, composition. Lives last because it depends on the other stages having normalized inputs.
Fail closed at every stage

A stage's job is to either pass cleanly or stop the request. There is no "warn and continue." A schema mismatch is not a hint — it's a decision. The pipeline runs in a fixed order so each stage can rely on the previous one having normalized its inputs; this is also why your error messages should never echo user input back without structured codes (you saw that footgun in Module 4.1's error-channel injection).

05  Human-in-the-loop gates

Three questions decide whether the model can act alone.

Most tools should not require user confirmation — a search query should run, a calendar read should return. But some tools should never run without a human looking at the arguments. The decision is not subjective; it falls out of three structured questions about the tool's action class. Set the toggles below for any tool and read the verdict — this is the rubric your security review should apply to every new tool ship:

HITL Decision Rubric · Classify a Tool
Q1 · Does this tool change state?
Q2 · What does it touch?
Q3 · Is it reversible?

› The action-class taxonomy

  • Read · internal. Auto-allow. Searching docs, listing files, reading a row. The volume is too high for HITL and the harm is bounded by the read tool's scope.
  • Write · internal · reversible. Auto-allow with audit log. Creating a draft, updating a tag, adding a comment. Reversible means an admin can undo it from the audit trail.
  • Write · external · reversible. Confirm. Sending an email, posting to Slack, creating an issue in a public repo. External means it leaves your control surface.
  • Destructive. Confirm with strong friction (typed confirmation, no keyboard shortcut). Deletes, schema migrations, revocations. Some teams forbid these entirely from the LLM path.
The HITL UI is part of the security boundary

The confirmation prompt is not a UX detail — it's a control. It must show the actual arguments the tool will use (not a paraphrase the LLM wrote), flag external destinations visually, and require a deliberate click. "Don't ask again" should never apply to a tool, only to a specific argument shape. This is where the stop-the-injection-chain layer from Module 4.1's walkthrough actually lives.

06  Audit logging & SIEM

The log is what you'll have left after everything else fails.

An MCP server that does everything else right and logs nothing has no story to tell the incident-response team. The log is the artifact that survives the breach — append-only, structured, complete enough that a SOC analyst can reconstruct the chain in Module 4.1 with nothing but the JSON. › The four fields a SOC actually needs

# Every event. Every time. Nothing more, nothing less in the v1 schema.
{
  "ts":        "2026-05-31T14:23:07.812Z",   # RFC 3339, UTC, ms precision
  "trace_id":  "01HQX...ZQ",                  # ULID, ties together call chain
  "actor": {
    "sub":    "alice@acme.io",                # from JWT
    "role":   "editor",
    "client": "claude-desktop/0.9.4"
  },
  "event": {
    "type":   "tool.call",                   # or auth.deny, sandbox.violation, etc.
    "tool":   "send_email",
    "args_hash": "sha256:7e2b...",           # hash, not args (privacy-safe forensics)
    "verdict": "hitl_required"              # allow | deny | hitl_required
  },
  "context": {
    "recent_tools": ["read_doc", "search_docs"],
    "policy":        "mcp.composition_v3",    # which policy rendered the verdict
    "reason":        "external destination after internal read"
  }
}
Three things this format does that "just log it" doesn't

1. trace_id ties multiple events from one session together — without it, you can see "deny" but not the chain that led there. 2. args_hash instead of args means you can prove an action happened without storing PII; if you need the args later, you cross- reference with an encrypted, shorter-retention store. 3. verdict + reason + policy means every deny is debuggable; an alert on "verdict=deny, policy=composition_v3" tells you both what and why.

Before you install an MCP server, this is the checklist.

The rug-pull and tool-shadowing attacks from Module 4.1 succeed because organizations install MCP servers like they install Chrome extensions — quickly, by individual users, without review. › What "approved" means operationally

A pass on this scorer is necessary but not sufficient. Approved servers still go through the rest of Track 4: they run sandboxed (Section 2), under network policy (Section 3), with input validation (Section 4), with HITL on action-class tools (Section 5), with audit logging (Section 6), and with periodic re-review on description-hash changes (Module 4.1's rug-pull defense). The scorer is the gate that decides which servers even reach the rest of the stack — not a replacement for it.

The org-level version of this rubric

Large organizations should run an internal MCP registry — a vetted catalog of approved servers with pinned versions and reviewed descriptions. Individual users install from the catalog, not from the open internet. This is the same pattern as a private npm registry or a curated container registry: provenance is the foundation everything else builds on. Module 4.4 covers how to red-team servers before they hit the catalog.

Knowledge Check · End of Module 4.3
Six checks before Module 4.4.
Q1.Your filesystem-root check is requested.startswith(root). A user requests /data/users/alice/../bob/notes.md. The literal string starts with /data/users/alice, so the check passes. What's wrong?
Right. This is the most common filesystem-root bug. The fix is to resolve then compare — and to compare against the narrowest scope (per-user), not the broadest root. Pair with Unicode normalization to defeat homoglyph attacks.
Q2.An MCP server's container has --network=host for "easy access to local Postgres." A prompt-injection causes a chained call that fetches an external URL. Why does this configuration make the incident worse?
Yes. Host networking is the #1 "convenient" mistake. It bypasses every network policy you might write. Cloud metadata endpoints (AWS IMDS, GCP metadata) become reachable; lateral movement becomes trivial. Always use a bridged network with an explicit egress allowlist.
Q3.A tool send_internal_notification writes a Slack message to a team channel inside your workspace. Pick the correct HITL category.
Correct. Confirmation-on-every-write would be unusable; users would click through without reading. Reserve HITL for external destinations and irreversible actions. The audit log makes "write, internal, reversible" recoverable without friction in the happy path.
Q4.You log full tool arguments verbatim into a SIEM. A tool argument contains a customer's SSN. Three months later, GDPR / privacy review flags the log retention. Which design choice in Section 6's audit schema avoids this from the start?
Right. The principle is "log what you need to prove, not more." Hashes prove identity-of-action without storing the action's payload. This is the simplest change that makes audit logs both useful for forensics and compliant with retention/privacy laws.
Q5.A third-party MCP server's GitHub repo has 18k stars, weekly commits, and a single maintainer. The README has no security policy, no disclosed past incidents, and no SBOM. The container image isn't signed. Should you install it?
Yes. The choice is rarely "install vs don't" — it's "install with what containment." A popular-but-unverified server can ship in a sandbox, with network policy, with HITL on action-class tools, with hash pinning. The scorer tells you how much containment the server needs, not just yes/no.
Q6.Pick the strongest statement of why this module's stack is "defense in depth" rather than "redundant defenses."
Exactly. If a layer doesn't have a clear failure-mode it catches that nothing else does, it isn't a layer — it's noise. Module 4.4 will test this stack end-to-end with a red-team suite that exercises each layer's failure path independently.
