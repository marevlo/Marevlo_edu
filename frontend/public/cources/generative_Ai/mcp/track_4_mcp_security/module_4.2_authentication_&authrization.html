Track 04 — MCP Security / Module 4.2

Authentication & authorization.

Module 4.1 mapped the attacks. This module installs the first defense layer: knowing who is asking (authentication) and what they may do (authorization). For MCP, that's harder than it sounds — the LLM holds the user's token but acts on instructions from anywhere it has read. OAuth 2.1 with resource indicators, JWTs with audience scoping, RBAC pushed down to per-tool service accounts, and OPA policies over capability combinations. By the end, you'll wire all four together.

01  The auth stack, layer by layer

Authentication, authorization, and the line between them.

Engineers conflate these two constantly. Authentication answers "who is the caller?" and produces an identity. Authorization answers "may this identity perform this action on this resource?" and produces a yes/no decision. MCP needs both, layered, because the identity in the token (Alice) is not the same as the intent in the conversation (whatever the LLM was nudged toward by the last page it read). 

# The four layers, in execution order, on a single tools/call request

1. Transport   TLS 1.3+ (or mTLS) — proves you're talking to the server you think
2. AuthN       OAuth 2.1 + PKCE → bearer token → JWT verification (sig, exp, aud)
3. AuthZ       RBAC role check → OPA policy → tool-specific allowlist
4. Backend     Per-tool service account → least-privilege backend permissions

# If any layer says "deny," the request stops. There's no fall-through.
# Each layer assumes the layers above failed. Defense in depth.

› Why "we have OAuth" is never the whole answer

OAuth gives you a verified caller identity — Alice's token cryptographically proves Alice clicked "authorize" at some point. That's authentication, layer 2. It says nothing about whether the action Alice's token is currently being used for is one Alice intended, or one an injected wiki page told the LLM to attempt on Alice's behalf. The token is Alice's; the intent is the prompt's. That gap is what authorization layers — RBAC, OPA, per-tool service accounts — exist to close.

The mental model to carry through the module

Auth in MCP is not "log in once and trust everything that comes after." It's per-call, per-tool, per-argument. Every layer assumes the model has been fooled. The token proves Alice; OPA proves the action is one Alice's role permits; the per-tool service account proves the backend only sees the data this tool needs. When you wire this up correctly, an attacker who fully controls the LLM's prompt still hits a wall at the policy layer.

02  Method comparison matrix

Four methods. Pick the right one — and the right combination.

Production MCP servers rarely use one auth method in isolation. The common pattern is OAuth 2.1 issuing JWTs over TLS, with mTLS reserved for high-assurance flows (host-to-server in regulated environments). › The four bad patterns this matrix kills

  • "We'll just use API keys for now." Long-lived bearer secrets with no user identity, no scope, no audience. Acceptable for M2M cron jobs; never for user-facing MCP.
  • JWT without an aud claim. A token issued for one server replays against any other. Confused-deputy vulnerability by default.
  • OAuth without resource indicators. Token issued for Server A is silently accepted by Server B that happens to trust the same issuer. RFC 8707 fixes this.
  • mTLS as the only layer. The cert proves which machine is calling, not which user. Always pair with user-identity tokens.
03  OAuth 2.1 flow + PKCE

The wire-level dance, one message at a time.

OAuth 2.1 is OAuth 2.0 with the unsafe parts deleted: implicit flow gone, password grant gone, PKCE now mandatory for all clients, refresh tokens rotated by default. The flow below is the authorization-code + PKCE flow — the one your MCP host runs every time a user adds a server. › The two parameters that exist because of MCP

PKCE (RFC 7636). The host generates a random code_verifier, hashes it to a code_challenge, sends only the challenge with /authorize, and reveals the verifier only when redeeming the code at /token. This binds the authorization code to the specific client that started the flow — an intercepted code is useless without the verifier. Mandatory in OAuth 2.1.

Resource indicators (RFC 8707). The host passes resource=https://server.example.com/mcp on /authorize. The auth server embeds this as the aud claim in the issued JWT. The MCP server later rejects any token whose aud isn't its own URL. Without this, a token issued for Server A is replayable against Server B that happens to trust the same issuer — the confused-deputy attack from Module 4.1.

The MCP spec is explicit about both

The MCP authorization spec mandates OAuth 2.1, PKCE, and resource indicators by name. If your server doesn't enforce all three, you're not spec-compliant — and you have the exact vulnerabilities the spec was designed to prevent. Module 4.4's red-team suite includes a "rogue server replay" test that fails servers missing any of the three.

04  JWT, dissected

Three dot-separated chunks. Each one matters.

A JWT is three Base64URL-encoded JSON objects joined with dots: header.payload.signature. The header says how it was signed; the payload carries the claims; the signature proves the claims weren't modified. For MCP, four specific claims decide whether a token is valid for a given request. › Verification, in code

# Every MCP server needs this. No exceptions.
from jose import jwt
from jose.exceptions import JWTError

EXPECTED_AUD  = "https://mcp.acme.io/v1"            # this server's URL
EXPECTED_ISS  = "https://auth.acme.io"              # the trusted issuer
JWKS_URL      = "https://auth.acme.io/.well-known/jwks.json"

def verify_bearer(token: str) -> dict:
    try:
        claims = jwt.decode(
            token,
            key=fetch_jwks(JWKS_URL),       # pubkey for signature check
            algorithms=["RS256", "ES256"], # NEVER accept "none"
            audience=EXPECTED_AUD,           # aud claim must match (RFC 8707)
            issuer=EXPECTED_ISS,             # iss claim must match
            options={"require": ["exp", "aud", "iss", "sub"]},
        )
    except JWTError as e:
        raise Unauthorized(f"invalid token: {e}")

    # Replay protection: jti not seen recently
    if replay_cache.has(claims["jti"]):
        raise Unauthorized("replay detected")
    replay_cache.add(claims["jti"], ttl=claims["exp"])

    return claims
Three classics that still ship in production

"alg: none" — accepting unsigned tokens because the library defaults allow it. Algorithm confusion — accepting tokens signed with HS256 using the public RSA key as the HMAC secret. No aud check — accepting any token signed by a trusted issuer regardless of who it was issued for. Audit your jwt.decode call. If it's missing algorithms=, audience=, or issuer=, you have at least one of these bugs.

05  RBAC + OPA policies

Authentication ends at the token. Authorization starts here.

Once the JWT is verified, you know who. Authorization decides whether. Simple servers can do this inline: "if role == 'admin' allow else deny." Real servers outgrow that within weeks — tool count grows, argument-shape rules appear, capability combinations matter. Open Policy Agent (OPA) externalizes the decision into a declarative policy language (Rego), so your authz logic isn't scattered across every tool handler. › Where OPA sits in the request flow

OPA runs as a sidecar to your MCP server or as a library call inside the handler. For every tools/call request, the server builds an input document ({user, role, tool, args}) and asks OPA "allow?". OPA evaluates the policy against the input and returns a verdict in milliseconds. The policy lives in version control, gets code-reviewed like any other code, and is hot-reloaded without restarting the server. 

# Wiring OPA into the MCP request path
async def handle_tools_call(request, claims):
    input_doc = {
        "user": claims["sub"],
        "role": claims["role"],
        "tool": request.params["name"],
        "args": request.params["arguments"],
        "recent_tools": session.recent_tools(),  # for composition checks
    }
    decision = await opa.evaluate("data.mcp.allow", input_doc)
    if not decision["allow"]:
        audit_log.deny(input_doc, decision["reason"])
        raise Forbidden(decision["reason"])
    return await dispatch_tool(request, claims)
The argument that wins the OPA budget

OPA is overkill for small servers. For large ones, the alternative is policy code scattered across forty tool handlers, drifting independently, untested as a whole, and impossible to audit. Centralized policy means a security engineer can read one file and know what your server permits. When the compliance team asks "what can the AI do with my data?", you can answer with a file path.

06  Least-privilege scoping

The blast radius is whatever the worst-case tool can do.

Authentication and authorization decide "may this call proceed?" Least privilege decides "if the call proceeds and the model has been fooled, how bad can it get?" The answer is whatever the narrowest set of capabilities you gave the tool can produce. Use the scope builder below — for each tool, tighten the scope until the risk meter drops. The same scope you'd configure in your OAuth issuer and IAM role.

› Three levels of scope, applied together

# Level 1 — OAuth scope (what the user authorized this client to do)
scope = "docs:read"                          # NOT "docs:*"

# Level 2 — Per-tool IAM service account (what the tool can do on the backend)
# IAM policy attached to the lambda/container that backs read_doc:
{
  "Effect": "Allow",
  "Action": ["s3:GetObject"],                 # not s3:* — only Get
  "Resource": ["arn:aws:s3:::acme-docs/users/${user_id}/*"]  # row-level
}

# Level 3 — OPA policy (what combination of tools the role permits per call)
allow {
  input.role == "reader"
  input.tool == "read_doc"
  startswith(input.args.path, "/users/")
  not composition_risk
}
The rule that catches the most production incidents

Tools should never share a single "server" service account. Each tool gets its own IAM role with exactly the backend permissions that tool needs. The capability-composition attacks from Module 4.1 (chain read + external-write to exfil) fail at the backend layer because the read tool has no write permissions and the write tool has no read permissions. The injection still happens; the blast radius is zero.

Knowledge Check · End of Module 4.2
Five checks before Module 4.3.
Q1.Your auth server issues JWTs signed with RS256. Your verifier accepts both RS256 and HS256. What's wrong?
Exactly. The fix is algorithms=["RS256"] only. This is one of the three classic JWT bugs (alg-confusion, alg-none, missing aud) and it's still found in production today.
Q2.A user adds two MCP servers, both authenticated through the same OAuth issuer. Server A is legitimate; Server B is malicious. Server B captures a JWT issued for Server A and replays it. The legitimate Server A's verifier rejects nothing about the replayed token (signature is valid, exp is fine, iss matches). What control was missing?
Right. The MCP authorization spec mandates resource indicators precisely because this attack is trivial without them. Any setup that uses a shared issuer across multiple servers without aud binding is vulnerable.
Q3.A user has the "reader" role and calls read_doc({path: "/users/alice/notes.md"}). Their JWT has sub=bob. Your Rego policy allows reads under /users/. What additional check belongs in the policy?
Correct. Coarse role gates ("readers can read") are usually wrong by themselves. Tenancy lives in the arguments; the policy must inspect them. This is the most common authz bug in MCP servers built quickly.
Q4.You have an MCP server with 12 tools, all running in the same container under a single service account that has s3:* on the data bucket. An injection causes the LLM to chain list_files → read_file → send_to_url. Which layer would have stopped this with the least disruption?
Yes. "Server = service account" is the single most common scoping mistake. Splitting per tool — and treating that split as a deployment requirement, not optional — is the structural fix the rest of the controls compose on top of.
Q5.Your team picks API keys "for simplicity" on a new MCP server that will be added by individual users in a 5,000-employee company. Which of the following is the strongest argument against?
Right. "Simplicity" with API keys is an upfront illusion that becomes operational pain by month three. The MCP spec language ("user-authorized servers MUST use OAuth 2.1") is not arbitrary — it's the answer to all five problems above. API keys remain valid only for true M2M scenarios with no user behind the call.
