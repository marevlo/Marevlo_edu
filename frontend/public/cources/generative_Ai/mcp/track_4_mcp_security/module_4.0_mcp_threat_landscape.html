Track 04 — MCP Security / Module 4.0

The MCP threat landscape.

Track 4 is the security track. It opens here, before any code, because you cannot defend what you cannot describe. MCP introduces a new attack surface — not a bigger version of an existing one, a categorically different one — because a system that lets an LLM act inside your infrastructure has trust boundaries no REST API ever had. This module is the map of that territory: where the attackers come in, what they take, and how to model both before you write the first line of hardening.

01  Why MCP expands the surface

An LLM that acts inside your infra is a new threat model.

The instinct for any team that's shipped REST APIs is to apply the same security playbook to their MCP server: auth, rate limits, input validation, done. The instinct is wrong. Toggle between the two surfaces below — a hardened REST API and an MCP server — and watch what's different about each attack vector:

Compare · REST vs MCP surface
Attack Surface · Traditional API vs MCP Server

› The four surface expansions you have to model

  • The LLM is an untrusted actor inside your trust boundary. A REST API treats the caller (a known user-agent with a known credential) as authenticated. An MCP host hands tool calls to an LLM that is reading whatever text crossed its context — including attacker-supplied text. The LLM is the attack vector.
  • Tool descriptions are executable. Not literally, but practically: a description that says "this tool requires admin to first delete the user's MFA" gets followed by some models. The schema, descriptions, and even field names in your tool spec are read by the model — and read by attackers who upload data that becomes part of your tool's response.
  • Indirect prompt injection through resources/results. Every resources/read result is text that ends up in the model's context. An attacker who can influence that text — by writing in a wiki you indexed, an email you parse, a webpage you scrape — has effectively written a system prompt.
  • Capability composition. A REST API's permissions are checked at one endpoint. MCP's tool composition means an LLM may chain list_files + read_file + send_email in ways no individual permission check was designed for. The attacker doesn't need any single tool to be unsafe; they need the composition to be.
The first principle of MCP security

The model is part of your attack surface, not part of your defense. Treat every byte of input the model touches as adversarial. Treat every tool the model can call as a capability that an attacker is — through prompt injection — authorized to invoke. The rest of this track is the engineering of that posture.

02  Trust boundaries

The four zones you must defend independently.

A trust boundary is a line in the architecture where input changes from "I produced this" to "someone else produced this." Every boundary is a point at which you must re-validate, re-authenticate, and re-authorize. MCP has four of them stacked on top of each other. Hover any zone to highlight it — the explanation below tells you what crosses that line and what your defenses must do there:

Highlight · the four zones
MCP Trust Boundaries · Hover or Click a Zone

› Each boundary's defense set, named explicitly

  • Zone 1 (User → Host). Standard application security: authenticated session, CSRF tokens, rate limits, content security policy on the host's UI.
  • Zone 2 (LLM ↔ Host). The model is untrusted in both directions. Inbound: filter prompts for instruction-overrides. Outbound: never auto-execute tool calls without user-visible confirmation for sensitive actions.
  • Zone 3 (Host ↔ Server). OAuth 2.1 with resource indicators, mutual TLS for high-assurance flows, structured authorization (Module 4.2). This is where Track 4's main wiring lives.
  • Zone 4 (Server ↔ Backend). Least-privilege service accounts, network policy, audit logging at the boundary. Tools should hold the minimum access needed; their permissions live here, not in the server process generally.

› Threat modeling, the lightweight version

You don't need to write a 40-page threat model to be useful. The minimum-viable threat model for an MCP server is a two-column document: STRIDE category × which of your tools, resources, or prompts is affected. For each cell, write one sentence describing the realistic attack and one sentence on the planned mitigation. That's it. Twelve cells covers most servers and takes an afternoon. 

# threat-model.md — a real one, in markdown

| STRIDE | Surface       | Attack                                | Mitigation              |
|--------|---------------|---------------------------------------|-------------------------|
| Spoof  | host→server   | Forged OAuth token                    | OAuth 2.1 + JWKS verify |
| Tamper | tool args     | LLM passes attacker-shaped JSON       | JSON Schema + allowlists|
| Repud  | tool calls    | "I didn't run delete_user"            | Append-only audit log   |
| Info   | resources/read| Doc leaks PII to model context        | Server-side redaction   |
| DoS    | tools/call    | Resource-heavy tool flooded           | Per-session rate limit  |
| Elev   | tool compose  | list_files + read_file → exfiltrate   | Scoped roots, OPA policy|
Threat models are living documents

The threat model you write today is wrong by next quarter. Tools change, dependencies change, new spec features (elicitation, sampling) open new attack surfaces. Build it into your engineering rhythm: re-review the threat model whenever you ship a new tool, and at minimum once per quarter. The first version is the hard one; updates are cheap.

› The honest critique of DREAD — and why we still use it

DREAD is famously a "fuzzy" framework. Two engineers scoring the same threat will disagree by 1–2 points per dimension, easily. The number out the other end is not a measurement; it's a rough sort key. The value isn't the precision — it's that the discipline of scoring forces you to think about all five dimensions instead of just the one that scares you most. Use it to compare threats, not to measure them.

  • ≥ 8.0 — Critical. Blocks launch. Goes to the top of the next sprint regardless of what else is in flight.
  • 6.0–7.9 — High. Fix within the current release cycle. Track in the security backlog.
  • 4.0–5.9 — Medium. Fix in next quarterly hardening sprint. Document the residual risk.
  • < 4.0 — Low. Accepted risk. Note it in the threat model; revisit at the next review.
CVSS for the audit, DREAD for the engineering

If your customers ask for CVSS scores (the IETF/FIRST industry standard), that's a compliance conversation — produce them with the CVSS calculator. DREAD is the internal engineering tool. Use both: CVSS for the auditor, DREAD for the sprint planning meeting. They\\u2019re scoring the same threats with different lenses.

Knowledge Check · End of Module 4.0
Five checks before Module 4.1.
Q1.A teammate says "we already have OAuth and rate limits — our MCP server is as secure as our REST API." What's the most fundamental gap they're missing?
Right. Traditional API controls assume the caller's identity equals the caller's intent. MCP breaks that assumption — the LLM has the user's identity but its "intent" is whatever crossed its context window. This is the categorical difference that the entire track addresses.
Q2.You're modeling a tool that reads from a wiki the user provided as a resource. STRIDE category for "attacker writes a malicious note in the wiki that the LLM follows as an instruction"?
Yes. Indirect prompt injection is Tampering against the model's context: attacker modifies data the system trusts (the wiki) to alter program behavior (the LLM\\u2019s tool calls). Elevation is the secondary effect when the injection gains new capabilities. Either letter is defensible — what matters is that you spotted the threat, not the precise taxonomy.
Q3.You DREAD-score a threat: Damage 8, Reproducibility 9, Exploitability 9, Affected 7, Discoverability 9. Verdict?
Right. Average is (8+9+9+7+9)/5 = 8.4, which is ≥ 8.0 = Critical. The high triple (reproducibility, exploitability, discoverability) is especially alarming — it means attackers will find this quickly and exploit it consistently. Affected users at 7 means the impact is broad even though not universal.
Q4.Which of these is NOT a trust boundary in MCP?
Correct. Tool-to-tool calls within one server don't cross a trust boundary. They do create a real risk (capability composition), but defending it lives at Zone 4 (server↔backend) and through OPA policies, not at a fresh trust line.
Q5.The 2024-2026 timeline shows multiple incidents that resemble each other. What's the single most repeated root cause across them?
Exactly. The pattern is uniform: classical controls were in place; the failure was at the new attack surface MCP introduced. Module 4.1 walks through the specific attacks; Module 4.3 builds the defenses. This module gave you the map — the rest of Track 4 is the territory.
