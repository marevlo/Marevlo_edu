Multi-Agent Systems

Single agents are like freelancers. Multi-agent systems are like companies. Module 3 introduces specialist teams, supervisor coordination, peer-to-peer networks, and hierarchical systems that scale beyond one overloaded prompt.

Focus: Multi-agent orchestration
01 · Framing
This Is Where It Gets Real

Single agents are like solo freelancers. They are good for simple tasks. Multi-agent systems are like companies, where each employee is a specialist and a manager coordinates the team.

The source ties this directly to production: Klarna uses multi-agent systems for customer service at scale, PGA Tour uses specialized agents for content generation, and serious AI systems beyond a simple chatbot depend on multi-agent patterns.

Why one agent is not enough: a single employee trying to answer phones, write code, manage finances, and handle complaints would be overwhelmed. The same problem happens with a single LLM prompt that tries to be everything. Its context fills up with too many tools, too many instructions, and too many competing priorities.

02 · Architectures
The Three Architecture Patterns
Pattern VisualizerSupervisor vs Network vs Hierarchical
Pattern 1

Supervisor

A manager receives every request, decides which specialist should handle it, and gets control back after each specialist finishes.

Pattern 2

Network

No manager. Every agent can talk to every other agent directly, and handoffs happen peer-to-peer.

Pattern 3

Hierarchical

A top supervisor delegates to team leads, and those team leads manage their own subgraphs.

03 · Pattern 1
Supervisor Pattern (Most Common)

The course compares this to a call center. A supervisor takes every incoming request, figures out which specialist should handle it, delegates, then decides whether to route again or finish.

Supervisor PatternSource-preserved
from langgraph.types import Command
from langgraph.graph import StateGraph, MessagesState, START, END

def supervisor(state: MessagesState) -> Command:
    """The manager: decides which specialist handles the task."""
    response = llm.invoke([
        SystemMessage(content="""You manage a team of 3 specialists:
        - 'sales_agent': handles pricing, quotes, and purchases
        - 'support_agent': handles issues, complaints, and technical help
        - 'research_agent': handles product questions and comparisons

        Based on the conversation, decide which agent should respond next.
        If the task is complete, say 'FINISH'.
        Respond with ONLY the agent name or FINISH."""),
        *state["messages"]
    ])
    next_step = response.content.strip().lower()
    if next_step == "finish":
        return Command(goto="__end__", update={"messages": [response]})
    return Command(goto=next_step, update={"messages": [response]})

def sales_agent(state: MessagesState) -> Command:
    response = llm.invoke([
        SystemMessage(content="You are a sales specialist. Help with pricing and purchases."),
        *state["messages"]
    ])
    return Command(goto="supervisor", update={"messages": [response]})

def support_agent(state: MessagesState) -> Command:
    response = llm.invoke([
        SystemMessage(content="You are a support specialist. Resolve issues empathetically."),
        *state["messages"]
    ])
    return Command(goto="supervisor", update={"messages": [response]})

def research_agent(state: MessagesState) -> Command:
    response = llm.invoke([
        SystemMessage(content="You are a research specialist. Provide detailed product info."),
        *state["messages"]
    ])
    return Command(goto="supervisor", update={"messages": [response]})

# Build the graph
graph = StateGraph(MessagesState)
graph.add_node("supervisor", supervisor)
graph.add_node("sales_agent", sales_agent)
graph.add_node("support_agent", support_agent)
graph.add_node("research_agent", research_agent)
graph.add_edge(START, "supervisor")  # Everything starts at the supervisor

aria_v3 = graph.compile()
04 · Pattern 2
Network Pattern (Peer-to-Peer)

No manager. Every agent can talk to every other agent directly. Each agent decides who to hand off to next.

The source compares it to a group of consultants collaborating on a project, where anyone can ask anyone else for help.

Best for: exploratory tasks where the optimal workflow is unknown.

Risky for: anything requiring clear accountability.

05 · Pattern 3
Hierarchical Pattern

The course calls this the enterprise pattern. A CEO delegates to VPs or team leads, and each of those manages its own team. This scales to dozens or hundreds of agents.

Hierarchical PatternSubgraphs as nodes
# Each team is its own compiled graph (subgraph)
research_team = build_research_team().compile()  # 3 agents
writing_team = build_writing_team().compile()    # 2 agents

# Top-level graph treats each team as a single node
main = StateGraph(State)
main.add_node("ceo", ceo_supervisor)
main.add_node("research_team", research_team)  # Entire team = 1 node
main.add_node("writing_team", writing_team)    # Entire team = 1 node
main.add_edge(START, "ceo")
06 · Tradeoffs
Pattern Comparison
Pattern Control Scalability Debugging Best For
Supervisor Centralized Medium Easy (clear flow) Most production systems
Network Distributed Low Hard (unpredictable flow) Creative/exploratory tasks
Hierarchical Layered High Medium (trace through layers) Large-scale enterprise
07 · Master Project
MASTER PROJECT — ARIA v3: Multi-Agent Team

ARIA evolves from a solo agent to a team. The supervisor routes requests to Sales, Support, or Research specialists, and each specialist has its own system prompt and tools.

The source notes that with the supervisor code above, ARIA v3 is already built.

08 · Understanding Check
Mini Quiz

Q1: In the supervisor pattern, who decides when the conversation is done?

A1: The supervisor. When it returns Command(goto='__end__'), the graph terminates.

Q2: Can a specialist agent call tools? How would you add tools to the support agent?

A2: Yes. Each agent can have its own tools via bind_tools(). You would make the support agent into its own tool-calling ReAct agent.

Q3: What happens if the supervisor keeps routing in circles?

A3: Infinite loop. Add a max_rounds counter in state and check it inside the supervisor.
