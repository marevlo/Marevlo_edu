Persistence & Human-in-the-Loop

Module 4 turns an impressive prototype into a real product: persistent conversations, recovery across restarts, human approval before critical actions, and replayable state history for debugging.

Focus: Persistence + HITL
01 · Why This Matters
Where Production Agents Usually Fail

This is where 90% of courses stop, and where 90% of production agents fail. A notebook demo is easy. An agent that remembers conversations across sessions, survives restarts, and pauses for human approval is engineering.

The module calls persistence and HITL production requirements, not optional extras. Klarna’s AI assistant handles 2.3 million conversations per month. Without persistence, those conversations would lose context after a restart. Without HITL, the agent could authorize high-stakes actions like refunds without approval.

The Memory Problem

The source compares an agent without persistence to an employee with amnesia. Every morning they walk in and remember nothing from yesterday. Every phone call starts from zero. That is what a stateless agent does.

Without Persistence

Session 1: User says “My name is Sarah” → agent says “Nice to meet you, Sarah!”

Session 2: User says “What’s my name?” → agent says “I don’t know your name.”

Server restart: all conversations lost. Zero recovery.

With Persistence (Checkpointer)

Session 1: User says “My name is Sarah” → state is saved to PostgreSQL.

Session 2: User says “What’s my name?” → agent answers “Your name is Sarah!”

Server restart: all conversations recovered from the database.

02 · Mechanics
Checkpointers — How It Works

After every node execution, the checkpointer saves the complete state. Each save creates a checkpoint with a unique ID. You can have thousands of concurrent conversations, each identified by thread_id, each with full history of every state change.

Checkpointing TimelineSave after every node
Checkpointer When to Use Persistence Scaling
MemorySaver Development and testing only In-memory, lost on restart Single process
SqliteSaver Single-server production Disk file Single server
PostgresSaver Multi-server production PostgreSQL database Horizontal scaling
03 · Production Memory
Production Setup with PostgreSQL
PostgresSaver SetupSource-preserved
# Production setup with PostgreSQL
from langgraph.checkpoint.postgres import PostgresSaver
from psycopg_pool import ConnectionPool

pool = ConnectionPool(
    "postgresql://user:pass@db-host:5432/langgraph",
    max_size=20  # Handle 20 concurrent conversations
)

with pool.connection() as conn:
    checkpointer = PostgresSaver(conn)
    checkpointer.setup()  # Creates tables on first run

aria = graph.compile(checkpointer=checkpointer)

# Every invoke now persists state automatically
config = {"configurable": {"thread_id": "user-42-session-1"}}
aria.invoke({"messages": [HumanMessage("Hi, I'm Sarah")]}, config)
# ... hours later, same thread_id ...
aria.invoke({"messages": [HumanMessage("What's my name?")]}, config)
# ARIA remembers: "Your name is Sarah!"
04 · Safety Net
Human-in-the-Loop

HITL is how agents stay safe in high-stakes environments. Before executing a critical action like sending an email, creating a ticket, or processing a refund, the agent pauses and waits for human approval.

Pattern 1

Interrupt BEFORE

Approval gate before the critical node runs. This is the most common pattern.

Pattern 2

Interrupt AFTER

Let the node run, then pause to review its output before continuing.

Pattern 3

State Modification

The human can inspect and change the state before the graph resumes.

Interrupt BEFOREApproval gate
aria = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["send_email", "process_refund", "create_ticket"]
)

# Run until pause point
result = aria.invoke(input_data, config)
# → Execution pauses BEFORE send_email
# → Human reviews what the agent wants to send
# → Human approves:
aria.invoke(None, config)  # Resume from pause point!

Pattern 2: Interrupt AFTER means the node runs first, but the system pauses before the next node so a human can review the output.

Pattern 3: State ModificationHuman corrects state
# Human can CHANGE the state before resuming
state = aria.get_state(config)
print(state.values)  # See what the agent planned

# Human modifies the plan
aria.update_state(config, {
    "refund_amount": 50.00,  # Human adjusts from agent's $100
    "approved": True
})
aria.invoke(None, config)  # Resume with human's corrections
05 · Debugging
Time-Travel Debugging

The course makes a strong claim here: this pattern alone can 10x debugging speed. Because every step is checkpointed, you can rewind to any point in a conversation, inspect the exact state, fix something, and replay from a past checkpoint. It is like a debugger with breakpoints, but for AI agents.

Inspect and ReplayCheckpoint history
# See every step the agent took
for checkpoint in checkpointer.list(config):
    step = checkpoint.metadata.get('step', '?')
    node = checkpoint.metadata.get('langgraph_node', 'start')
    print(f" Step {step}: Node '{node}'")

# Replay from any checkpoint
old_config = {"configurable": {
    "thread_id": "user-42",
    "checkpoint_id": "checkpoint_002"  # Go back to step 2
}}
past_state = aria.get_state(old_config)
print(past_state.values)  # Exact state at that moment
06 · Master Project
MASTER PROJECT — ARIA v4: Persistent + HITL

ARIA v4 combines ARIA v3 with PostgreSQL memory and human approval before critical actions like ticket creation and refunds.

ARIA v4Persistent + approval-gated
# ARIA v4 = ARIA v3 + PostgreSQL + HITL
aria_v4 = graph.compile(
    checkpointer=checkpointer,  # Remember everything
    interrupt_before=["create_support_ticket", "process_refund"]  # Human approval
)

# Conversation 1 (Monday)
config = {"configurable": {"thread_id": "customer-sarah-001"}}
aria_v4.invoke({"messages": [HumanMessage("Hi, I'm Sarah. Order #12345 arrived damaged.")]}, config)
# ARIA classifies → routes to support → wants to create ticket → PAUSES
# Human approves → ticket created

# Conversation 2 (Wednesday, same thread)
aria_v4.invoke({"messages": [HumanMessage("Any update on my ticket?")]}, config)
# ARIA remembers Sarah, her order, and the ticket from Monday!
07 · Understanding Check
Mini Quiz

Q1: What is the difference between short-term and long-term memory in LangGraph?

A1: Short-term memory is within a thread, meaning the same conversation. Long-term memory is across threads, using an external store or database to remember a user across sessions.

Q2: If two users send messages at the same time, do they interfere with each other?

A2: No. Each conversation has a unique thread_id, and checkpointers isolate state per thread.

Q3: What happens if the server crashes between checkpoint saves?

A3: The last completed checkpoint is preserved. The in-progress step is lost and can be replayed.

08 · Failure Patterns
Common Mistakes

Mistake #1: Using MemorySaver in Production

Everything is lost on restart. Use PostgresSaver for anything beyond development.

Mistake #2: Inconsistent thread_id Values

If the thread_id changes between messages, the agent treats it as a new conversation. Amnesia returns.

Mistake #3: Interrupting at Every Node

HITL should be surgical, only at critical action nodes. Interrupting everywhere makes the system unusable.
