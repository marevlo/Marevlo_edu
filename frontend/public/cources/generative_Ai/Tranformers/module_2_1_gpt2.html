GPT-2.
Autoregressive
language modeling.

Phase 1 was about understanding — encoding text into representations. Phase 2 is about generation — producing text one token at a time, left to right. GPT-2 (Radford et al., 2019) is the model that proved a single, simple objective — predict the next token — could produce fluent essays, translate languages, summarize articles, and answer questions, all without task-specific training. The architecture is identical to BERT except for one mask. The objective is the simplest possible. The results changed everything.

01 / 08
The autoregressive objective — predict the next token.

BERT masks 15% of tokens and predicts them from context. GPT-2 predicts every token from only the tokens that came before it:

P(x1, x2, ..., xT) = Πt=1T P(xt | x1, ..., xt−1) (1.1)

This is the chain rule of probability applied to sequences. The joint probability of a sequence equals the product of each token's conditional probability given all preceding tokens. Training maximizes this — equivalently, minimizes the negative log-likelihood:

L = −Σt=1T log P(xt | x<t) (1.2)

Unlike MLM (which trains on 15% of tokens), next-token prediction trains on every position. Every token in the sequence contributes a loss term. This is one reason decoder pretraining is more sample-efficient than BERT's MLM — the same insight ELECTRA exploited from the encoder side (Module 1.4).

Next-token prediction — one sequence worked through
Input: "The CineMatch thriller was" → Target: predict each next tokenTokens: [The, Cine, ##match, thriller, was]
Position 0: P("The" | <BOS>) — predict "The" from nothing but the start tokenHard! Many sentences start with "The". Cross-entropy loss here is high (~8.0)
Position 1: P("Cine" | "The") — predict "Cine" given "The"Still hard — "The" could be followed by anything. Loss: ~10.5
Position 3: P("thriller" | "The CineMatch") — easier with context"The CineMatch" narrows possibilities: thriller, movie, app, recommendation... Loss: ~4.2
Position 4: P("was" | "The CineMatch thriller") — very constrained"was", "is", "had" are the main options after "The X thriller". Loss: ~1.8
Total loss = sum of all positions: 8.0 + 10.5 + ... + 1.8 ≈ 26.5. Perplexity = exp(26.5/5) ≈ 200GPT-2 (1.5B) achieves perplexity ~18 on WebText — meaning each position has ~18 "effective choices" on average. Lower = better.
Why next-token prediction works so well

To predict the next word in "The director of the 2010 film Inception is ___", the model must know that Inception is a film (entity type), that it was directed by Christopher Nolan (world knowledge), and that the sentence is asking for the director (syntax). A single objective — predict the next token — implicitly requires learning syntax, semantics, world knowledge, and reasoning. The objective is simple. What it forces the model to learn is not.

ObjectiveQuestion 01 / 07
GPT-2 trains on every token (100% of positions). BERT's MLM trains on 15%. Does this make GPT-2 ~7× more sample-efficient?
AYes — GPT-2 is 7× more efficient per token seen.
BNo — BERT's 15% produces higher quality signal per position.
CNot directly comparable. GPT-2 trains on 100% of positions but each position only sees left context. BERT trains on 15% but each position sees full bidirectional context — a richer signal per position. In practice, GPT-2 needs ~2-3× more data than BERT to reach comparable quality on understanding benchmarks (where BERT has the structural advantage). For generation tasks, GPT-2 is more efficient because the training objective matches the inference task.
DNo — the number of positions doesn't affect efficiency.
02 / 08
The causal mask — the one difference from BERT.

Architecturally, GPT-2 is a transformer with the same building blocks as BERT: multi-head attention, FFN, residual connections, LayerNorm. The only structural difference is the causal attention mask — a lower-triangular matrix that prevents position t from attending to any position > t.

Causal attention mask. Each row shows which positions that token can attend to (blue = visible, red = masked with −∞). Position t sees positions 0..t only. The staggered animation reveals the triangular structure: each new position gets access to one more token. This matches inference, where future tokens don't exist yet.
Causal maskQuestion 02 / 07
During training, GPT-2 processes the entire sequence in parallel (one forward pass computes all positions simultaneously). How does the causal mask enable this?
AIt doesn't — GPT-2 processes tokens sequentially during training.
BThe mask splits the computation across multiple GPUs.
CThe mask is only applied during inference, not training.
DThe causal mask sets future-position attention scores to −∞ before softmax, making their weights exactly 0. This means each position's output depends only on positions ≤ t — identical to what would happen if future tokens didn't exist. Because the mask achieves this within a single parallel matrix operation (QKT + mask), all positions can be computed simultaneously. The loss at each position is valid because each position "saw" only its left context, despite all positions being computed in parallel. This is called "teacher forcing."

Key differences from BERT's architecture: pre-norm LayerNorm (LN before attention and FFN, not after), learned absolute position embeddings (1024 × d, limiting context to 1024 tokens), and no segment embeddings (no sentence-pair input).

import torch, torch.nn as nn, math

class GPT2Block(nn.Module):
    """Single GPT-2 decoder block (pre-norm)."""
    def __init__(self, d=768, h=12, d_ff=3072, drop=0.1):
        super().__init__()
        self.ln1 = nn.LayerNorm(d)
        self.attn = nn.MultiheadAttention(d, h, dropout=drop, batch_first=True)
        self.ln2 = nn.LayerNorm(d)
        self.ffn = nn.Sequential(
            nn.Linear(d, d_ff), nn.GELU(),
            nn.Linear(d_ff, d), nn.Dropout(drop),
        )

    def forward(self, x, causal_mask):
        # Pre-norm: LN before each sub-layer
        h = self.ln1(x)
        x = x + self.attn(h, h, h, attn_mask=causal_mask)[0]
        x = x + self.ffn(self.ln2(x))
        return x

class GPT2(nn.Module):
    """GPT-2 from scratch."""
    def __init__(self, vocab=50257, d=768, L=12, h=12, d_ff=3072, max_len=1024):
        super().__init__()
        self.tok_emb = nn.Embedding(vocab, d)
        self.pos_emb = nn.Embedding(max_len, d)
        self.blocks = nn.ModuleList([GPT2Block(d, h, d_ff) for _ in range(L)])
        self.ln_f = nn.LayerNorm(d)       # final LN (pre-norm needs this)
        self.head = nn.Linear(d, vocab, bias=False)
        self.head.weight = self.tok_emb.weight  # weight tying!

    def forward(self, idx):
        B, T = idx.shape
        pos = torch.arange(T, device=idx.device)
        x = self.tok_emb(idx) + self.pos_emb(pos)
        mask = torch.triu(torch.ones(T, T, device=idx.device), diagonal=1).bool()
        # mask[i,j]=True means position i CANNOT attend to position j
        for block in self.blocks:
            x = block(x, mask)
        logits = self.head(self.ln_f(x))  # (B, T, vocab)
        return logits

Position embedding: 1,024 × 768 = 786,432Learned absolute positions. Max context = 1024 tokens. Cannot extrapolate beyond this.
12 decoder blocks: 12 × 7.1M = 85.1M (same per-block count as BERT-base)Each block: MHA 2.36M + FFN 4.72M + 2 LayerNorms 3K = 7.08M
Final LayerNorm: 768 × 2 = 1,536
Output head: 0 additional params (weight-tied with token embedding)
Total: 38.6M + 0.8M + 85.1M + 0.002M = 124.4MOften reported as 117M or 124M depending on whether biases are counted. The architecture is nearly identical to BERT-base (110M) — the parameter difference is mainly the larger vocabulary (50K vs 30K).
ArchitectureQuestion 03 / 07
GPT-2 ties the output head weights to the token embedding. Why does this work?
AThe embedding maps tokens → vectors. The output head maps vectors → tokens. These are inverse operations — using the same matrix for both means "the vector for token X is also the direction in output space that predicts token X." This is geometrically natural: words with similar embeddings (semantically close words) will also have similar output logits. Weight tying saves 38.6M parameters and acts as implicit regularization — the model can't learn inconsistent input/output representations.
BIt's a memory optimization only — the model would be more accurate with separate weights.
CPyTorch requires weight tying for autoregressive models.
DIt prevents the model from predicting unknown tokens.
04 / 08
Inference — generating text token by token.

Training processes all tokens in parallel (teacher forcing). Inference is inherently sequential: generate token 1, then use it to generate token 2, then use 1+2 to generate token 3, and so on.

The KV cache — avoiding redundant computation.

Without caching, generating token t requires a forward pass through all t tokens — recomputing attention for positions 1..t-1 even though their K and V vectors haven't changed. The KV cache stores the key and value vectors from previous positions, so each new step only computes the new token's Q, K, V and uses the cached K, V from all previous positions.

KV cache — compute savings for generating 100 tokens
Without cache: step t requires attention over t positions. Total: Σ(1..100) = 5,050 position-computationsO(T²) total work — quadratic in the generated sequence length
With KV cache: step t only computes the new token's attention against the cached K,V. Total: 100 single-token stepsO(T) total work — linear. Each step uses cached K,V from previous positions.
Speedup: 5,050 / 100 = 50.5× for 100-token generationFor 1000 tokens: 500,500 / 1000 = 500× speedup. KV caching is not optional — it's essential.
Memory cost: must store K,V for all layers × all past positions. For GPT-2 Small generating 1024 tokens: 12 layers × 2 (K,V) × 1024 × 768 × 2 bytes = 36 MBManageable. For GPT-3 (96 layers × 12288-dim × 2048 tokens): ~55 GB. KV cache memory is the bottleneck for large models.
The inference bottleneck is memory bandwidth, not compute

Autoregressive generation is memory-bound, not compute-bound.

Each generation step produces one token by running the model's full depth (12-48 layers) on a single new token. The matrix multiplications are tiny (1 token × d), but the weights must be loaded from GPU memory for each step. An A100 has 2 TB/s memory bandwidth; loading GPT-2 Small's weights (500 MB) takes 0.25ms, while the actual computation takes ~0.01ms. The model spends 96% of each step waiting for memory. This is why quantization (smaller weights = less to load) and batching (amortize weight loading across multiple sequences) matter so much for serving.

KV cacheQuestion 04 / 07
GPT-2 with KV caching generates 100 tokens. You then want to continue generating 100 more. Can you reuse the existing KV cache?
ANo — the cache must be rebuilt from scratch for each generation call.
BYes — append to the existing cache. The KV cache stores (K, V) for each layer at each position. To generate tokens 101-200, you keep the cache from tokens 1-100 and append each new token's K,V as it's generated. No recomputation of tokens 1-100. This is exactly how chat systems work — the conversation history stays in the KV cache across turns.
COnly if the total length stays under 1024 tokens.
DYes, but the cache quality degrades over time.
05 / 08
Sampling strategies — controlling generation quality.

The model outputs logits over the vocabulary at each step. How you convert those logits into a chosen token determines the text's quality, creativity, and coherence.

StrategyHowTrade-off
GreedyAlways pick the highest-probability tokenDeterministic, repetitive, often boring
TemperatureDivide logits by T before softmax: P = softmax(z/T)T<1: sharper (more greedy). T>1: flatter (more random)
Top-kKeep only top k tokens, zero out the rest, renormalizePrevents extremely rare tokens. k=50 is common
Top-p (nucleus)Keep smallest set of tokens whose cumulative probability ≥ pAdaptive k — narrow when confident, wide when uncertain. p=0.9 is default

In practice, production systems use temperature + top-p together: temperature controls the sharpness of the distribution, top-p truncates the long tail. The combined effect: coherent text (no gibberish from rare tokens) with natural variation (not robotically deterministic).

06 / 08
CineMatch's conversational recommender — GPT-2 in production.
CineMatch Integration

GPT-2 powers CineMatch's conversational recommendation engine.

The product experience: Users type natural language queries — "I loved Inception but hated Interstellar. I want something mind-bending but not too slow." GPT-2 (fine-tuned on CineMatch's review corpus) generates personalized recommendations with explanations.

Architecture: The prompt includes the user's watch history (last 20 movies with ratings), the query, and a system instruction. GPT-2 generates a response token by token, using KV cache for efficient inference. Response length: ~100-200 tokens. Latency: ~500ms on A10G GPU (including all 100+ generation steps).

Why not GPT-4? Cost. CineMatch handles 50K chat queries/day. GPT-4 at $15/M output tokens: 50K × 150 tokens × $15/M = $112.50/day = $3,375/month. A fine-tuned GPT-2 XL on a single A10G: ~$730/month. 4.6× cheaper, with better domain-specific recommendations (fine-tuned on CineMatch data). The trade-off: less general reasoning, shorter context, worse at complex multi-turn conversations.

ProductionQuestion 06 / 07
CineMatch's GPT-2 chatbot occasionally generates "I don't know much about that movie" even for popular films in its training data. Why?
AThe movie wasn't in the training data.
BThe temperature is too low.
CGPT-2 has forgotten the movie due to catastrophic forgetting.
DAutoregressive models don't have reliable retrieval from their parameters. The model learned statistical patterns about movies, but it doesn't have a "database lookup" mechanism. It might generate hedging language if the prompt context doesn't activate the right patterns. Fixes: (1) Retrieval-augmented generation (RAG) — prepend relevant movie data from a database into the prompt, so the model doesn't need to remember facts. (2) Better prompt engineering — include movie metadata in the context. (3) Fine-tune on more movie-specific Q&A pairs.
07 / 08
The GPT-2 legacy — what came next.
Paper breakdown

"Language Models are Unsupervised Multitask Learners" (Radford et al., February 2019)

GPT-2's paper made a bold claim: a language model trained on next-token prediction alone could perform multiple NLP tasks without any task-specific training. By simply prompting with "TL;DR:" after an article, the model produced summaries. By prompting with "Q: ... A:", it answered questions. The paper called this zero-shot transfer — the model had never been explicitly trained on summarization or QA, yet it could do both.

The significance was less about GPT-2's actual quality (it wasn't very good at these tasks) and more about the direction: if next-token prediction on web text produces emergent task-solving at 1.5B parameters, what happens at 175B? That question led directly to GPT-3 (Module 2.2), which answered it: in-context learning — the ability to learn new tasks from examples in the prompt, without any weight updates.

LegacyQuestion 07 / 07
GPT-2 and BERT-base have nearly identical architectures (~110-124M params, 12 layers, d=768). Yet GPT-2 can generate text while BERT cannot. What enables GPT-2's generation ability?
AGPT-2 has a generation-specific layer that BERT lacks.
BTwo things, both stemming from the causal mask. First, the mask ensures each position only sees left context, matching inference (where future tokens don't exist). BERT's bidirectional attention sees the full sequence — at inference, there's no full sequence to see. Second, the training objective: next-token prediction trains the model to output a probability distribution over the next word — exactly what generation needs. BERT's MLM predicts masked tokens, not next tokens. Same architecture, different mask, different objective, fundamentally different capability.
CGPT-2 was trained on more data.
DGPT-2 uses a different tokenizer that supports generation.
