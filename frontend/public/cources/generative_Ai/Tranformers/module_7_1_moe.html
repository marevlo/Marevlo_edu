Switch Transformer — top-1 routing for maximum efficiency.

Mixtral (Module 3.2) routes each token to 2 of 8 experts. Switch Transformer (Fedus et al., 2022) simplifies this to 1 of N experts — each token goes to exactly one expert. This halves the compute per token (one expert instead of two) and simplifies the routing logic.

Switch Transformer vs Mixtral routing comparison
Mixtral (top-2 of 8): Each token activates 2 expert FFNs. Active compute: 2/8 = 25% of FFN params. Output: weighted sum of 2 experts.Redundancy: if one expert is slightly wrong, the second corrects. Higher quality, more compute.
Switch (top-1 of N): Each token activates 1 expert FFN. Active compute: 1/N of FFN params. Output: single expert's output (no blending).No redundancy: the router must choose correctly. Lower compute, slightly lower quality per token.
Switch at extreme scale: Switch-C has 1.6T total params with 2048 experts. Each token uses 1/2048 of FFN params. Active params per token: ~100M (out of 1.6T).Massive capacity (1.6T of knowledge storage) at tiny per-token cost (100M active). Training is 7× more efficient than a dense model of equivalent quality.
The trade-off: top-1 routing is faster but more brittle. If the router sends a token to the wrong expert, there's no second expert to compensate. In practice, top-1 works well for pretraining (large batches, averaged loss), but top-2 often produces better fine-tuning results (robustness to individual routing errors).Mixtral chose top-2 for this reason: open-source users fine-tune extensively, and top-2 is more robust to fine-tuning distribution shift.
SwitchQuestion 01 / 07
Switch-C has 2048 experts. What prevents the router from always picking the same few experts (expert collapse)?
AThe architecture prevents collapse.
B2048 experts is too many to collapse.
CRandom initialization prevents collapse.
DTwo mechanisms. (1) Auxiliary load balancing loss (same as Mixtral, Module 3.2): penalizes uneven expert utilization. L_balance = α · N · Σ f_i · P_i, where f_i is the fraction of tokens routed to expert i. (2) Capacity factor: each expert has a maximum number of tokens it can process per batch (typically 1.25× the even-split amount). Overflow tokens are dropped or sent to a fallback. This hard cap prevents any expert from monopolizing. Together, these ensure tokens are spread across all 2048 experts.
02 / 08
MoE scaling laws — how MoE changes the compute equation.

Dense scaling laws (Module 2.2): loss scales as L ∝ N−α with parameters N. MoE changes this: loss scales with total parameters, but compute scales with active parameters. You get the quality of a large model at the compute cost of a small one.

MoE advantage: quality(Ntotal) at cost(Nactive) where Nactive << Ntotal (2.1)
MoE vs dense — quality at equivalent compute budgets
Dense 7B: 7B params, all active. Quality: MMLU ~60. Compute per token: 7B multiply-adds.This is the baseline: all parameters work on every token.
Dense 70B: 70B params, all active. Quality: MMLU ~80. Compute per token: 70B multiply-adds. 10× more compute than dense 7B.To go from 60 → 80 MMLU with dense models requires 10× more compute per token.
MoE 8×7B (Mixtral): 46.7B total, 12.9B active. Quality: MMLU ~70. Compute per token: 12.9B multiply-adds. ~1.8× dense 7B compute.Gets MMLU 70 (midpoint between 7B and 70B dense) at 1.8× 7B compute, not 5× or 10×.
The MoE rule of thumb: an MoE model with N_total params and top-k/n routing matches a dense model of ~(N_total)^0.6 in quality, at ~N_active compute cost.Mixtral: 46.7B^0.6 ≈ dense ~22B quality. Actual: between 13B and 34B dense — roughly right.
ScalingQuestion 02 / 07
If MoE gets "free" capacity from inactive experts, why not use 1000 experts and make each one tiny?
AMore experts are always better.
BToo many experts cause training instability.
CDiminishing returns + infrastructure constraints. Quality scales as ~N_total^0.6, not linearly — doubling experts (2× memory) gives only ~1.5× quality improvement. And each expert must be large enough to be a useful specialist — a 1M-parameter expert can't store meaningful knowledge. Infrastructure: 1000 experts across GPUs means 1000-way all-to-all communication per layer, which saturates interconnect bandwidth. In practice, 8-64 experts is the sweet spot: enough diversity for quality gains, few enough for manageable communication.
DThe router can't handle more than 64 experts.
03 / 08
Expert specialization at scale — what do experts actually learn?

Research on MoE expert specialization reveals a nuanced picture. Experts don't cleanly divide by topic ("math expert," "code expert"). Instead, specialization is multi-level:

Token-level: Certain token types prefer specific experts — punctuation tokens, function words, content words, and special tokens each have preferred routing patterns. This is the strongest signal.

Domain-level (weak): Some experts handle more code tokens, others more math tokens, but with heavy overlap. No expert is exclusive to any domain.

Layer-level: Early layers show less specialization (more uniform routing). Deep layers show stronger specialization (more peaked routing distributions). This suggests early layers learn shared features while deep layers learn specialized patterns.

SpecializationQuestion 03 / 07
If MoE experts specialize by token type rather than domain, does that mean MoE is just an expensive way to handle different token types?
AYes — MoE is a waste if it only learns token-type specialization.
BNo — token-type routing is more useful than it sounds. Different token types require genuinely different processing: function words ("the," "of") need syntactic processing, content words ("neural," "architecture") need semantic processing, punctuation needs structural processing. Having specialized FFN pathways for each means each pathway can be optimized for its token type — more efficient than a single FFN that must handle all types with the same weights. The quality gain comes from this processing diversity, not from topic-level separation.
CToken-type specialization is a bug, not a feature.
DExperts actually specialize by domain — the research is wrong.
04 / 08
The MoE infrastructure stack.

Serving an MoE model is fundamentally different from serving a dense model because tokens must be routed to the correct GPU during inference:

MoE serving is communication-bound, not compute-bound

In a dense model, each GPU processes its assigned layers — data flows linearly through the pipeline. In MoE, each token must be sent to the GPU hosting its selected expert, processed, and returned — requiring all-to-all communication.

For Mixtral (8 experts, 2 GPUs): each token that routes to an expert on the other GPU requires a round-trip data transfer. With NVLink (600 GB/s), this takes ~1μs per token. With ethernet (25 Gbps), it takes ~50μs — a 50× slowdown. This is why MoE models are fast on single-machine multi-GPU setups (NVLink) but impractical across machines (ethernet). The communication overhead can exceed the compute savings that MoE provides.

InfrastructureQuestion 04 / 07
CineMatch wants to serve Mixtral 8×7B on a cloud instance. The options: (A) 2× A100-80GB on one machine with NVLink, or (B) 2× separate A100-80GB machines with ethernet. Which should they choose?
AOption B — separate machines are more scalable.
BEither works equally well for MoE.
CNeither — Mixtral needs more than 2 GPUs.
DOption A — NVLink is essential for MoE. With NVLink: inter-GPU transfer at 600 GB/s. Expert routing adds ~1-2ms per generation step — negligible. With ethernet (25 Gbps = 3.1 GB/s): routing adds ~50-100ms per step — making generation unusably slow. MoE's expert parallelism requires high-bandwidth interconnect between GPUs. A single machine with 2× NVLink-connected A100s serves Mixtral at near-dense-model latency. Separate machines make it 50× slower.

LandscapeQuestion 05 / 07
DeepSeek-V2 has "shared experts" that process every token (always active) alongside routed experts (conditionally active). What's the advantage?
AShared experts store common knowledge that every token needs (language structure, common patterns), while routed experts store specialized knowledge. This prevents the situation where common knowledge must be redundantly stored in every expert. Without shared experts: each routed expert independently learns basic language patterns — wasting capacity on duplicated knowledge. With shared experts: basic knowledge is centralized, and routed experts focus purely on specialization — more efficient use of total parameters.
BShared experts make routing unnecessary.
CShared experts are just smaller than routed experts.
DShared experts improve training speed only, not quality.
06 / 08
CineMatch's MoE strategy.
CineMatch Integration

When (and when not) to use MoE for CineMatch.

Use MoE (Mixtral 8×7B): Batch processing where quality matters and GPU memory is available. CineMatch's nightly review summarization pipeline processes 50K reviews — Mixtral's MMLU 70.6 produces better summaries than Mistral 7B's 60.1, at only 1.8× the compute cost (12.9B vs 7.3B active params). On a 2× A100 machine with NVLink, the batch job completes in 3 hours instead of 2 — acceptable for overnight processing.

Don't use MoE: Real-time chatbot serving where latency matters and infrastructure is constrained. Mixtral needs 2× A100 (93 GB weights). CineMatch's chatbot runs on 1× A10G (24 GB). A dense Mistral 7B or LLaMA-3 8B serves the chatbot at lower latency on cheaper hardware. MoE's quality advantage doesn't justify 4× the GPU cost for a chatbot where response time matters more than MMLU score.

The pattern: MoE for batch quality. Dense for real-time cost. Same principle as the encoder-only (fast, cheap) vs decoder-only (slower, better) split from earlier phases.

StrategyQuestion 06 / 07
CineMatch's ML team proposes: "Let's switch everything to Mixtral — it's always better quality." What's the counterargument?
AMixtral isn't available commercially.
BMixtral is always worse than dense models.
CInfrastructure economics. Mixtral needs 93 GB GPU memory — requiring 2× A100s (~$8,000/month) vs 1× A10G (~$730/month) for dense 7B. For the chatbot (50K queries/day), Mixtral costs $8,000/month vs $730/month — 11× more expensive. The quality improvement (MMLU 70 vs 60) doesn't translate to 11× better user experience. For batch processing (50K reviews/night), the same 2× A100 setup is already needed for throughput, so Mixtral adds quality at zero additional infrastructure cost. Match the model to the deployment economics, not to the benchmark leaderboard.
DMoE models can't be fine-tuned.

The future of MoE — where it's heading.
FutureQuestion 07 / 07
Will all future LLMs be MoE?
AYes — MoE is strictly superior to dense.
BFrontier models likely yes. Production models: mixed. MoE dominates at the frontier because it's the cheapest path to maximum quality (GPT-4, Gemini). But for production deployment, dense models remain competitive: simpler serving infrastructure, lower GPU memory, easier fine-tuning, and smaller download size. The future is a spectrum: 1-7B dense for edge/mobile, 8-70B dense for single-GPU serving, MoE for batch processing and frontier quality. Dense and MoE will coexist, not converge.
CNo — MoE is a dead end.
DOnly if hardware solves the communication problem.
