Linear attention
& the kernel trick.

Sparse attention skips connections. Linear attention takes a different approach: it replaces the softmax with a kernel function that allows the attention computation to be rearranged from O(T²) to O(T × d). Linformer projects keys and values to lower dimensions. Performer approximates softmax with random features. Both achieve truly linear scaling — but with quality trade-offs that explain why they haven't replaced standard attention. This module completes Phase 5 and maps the full efficiency landscape.

01 / 08
Linformer — low-rank projection of K and V.

The attention matrix A = softmax(QKT/√d) has shape (T, T). Linformer's observation: this matrix is approximately low-rank. If rank k << T captures 95% of the information, we can project K and V from T dimensions to k dimensions before computing attention:

Linformer:   A' = softmax(Q · (E · K)T / √d) · (F · V) (1.1)

where E and F are learned projection matrices of shape (k, T) that compress the sequence dimension. The attention matrix becomes (T, k) instead of (T, T) — and if k=128 and T=4096, that's a 32× reduction.

Linformer cost: projecting T down to k
Standard attention: Q(T×d) × K(T×d)T = A(T×T). Cost: O(T² × d).At T=4096, d=64 (per head): 4096² × 64 = 1.07B multiply-adds.
Linformer: project K(T×d) → K'(k×d) using E(k×T). Project V similarly. Then Q(T×d) × K'(k×d)T = A'(T×k).Projection cost: O(k × T × d). Attention cost: O(T × k × d). Total: O(T × k × d).
With k=128, T=4096: projection = 128 × 4096 × 64 = 33.6M. Attention = 4096 × 128 × 64 = 33.6M. Total: 67.2M.Speedup: 1.07B / 67.2M ≈ 16×. At T=16K: speedup ≈ 64×. At T=128K: speedup ≈ 512×.
The quality cost: projecting K from T to k discards information about position-specific attention patterns. Fine for broad contextual attention, weaker for precise retrieval ("find the exact sentence that mentions X").k is a hyperparameter: larger k = better quality, less speedup. k=128-256 is typical.
LinformerQuestion 01 / 07
Linformer projects K and V from T positions to k positions. What information is lost?
ANo information is lost — projection is lossless.
BAll position information is lost.
CFine-grained position-specific distinctions. The projection E(k×T) compresses T positions into k "summary positions." Positions with similar keys are merged into the same summary. If two positions have very different keys, the projection preserves their distinction. If two positions have similar keys, they're merged — losing the ability to attend to one but not the other. This matters for tasks requiring precise position-level retrieval but not for tasks requiring general contextual understanding.
DThe semantic content of the tokens.
02 / 08
Performer — the kernel trick for attention.

Standard attention: softmax(QKT/√d) × V. The softmax forces materialization of the T×T matrix before multiplying by V. Performer replaces softmax with a kernel approximation that allows the computation order to change:

Standard:   (softmax(QKT)) · V   →   must compute T×T first (2.1)
Performer:   φ(Q) · (φ(K)T · V)   →   compute d×d first! (2.2)

where φ is a random feature map that approximates the softmax kernel. The key insight: by computing φ(K)TV first (a d×d matrix, independent of T), then multiplying by φ(Q), the total cost becomes O(T × d²) — linear in T.

# Performer: FAVOR+ (Fast Attention Via Positive Orthogonal Random features)
import torch, math

def performer_attention(Q, K, V, n_features=256):
    """Linear attention via random feature approximation."""
    d = Q.shape[-1]

    # Random feature map: φ(x) = exp(x·ω - ||x||²/2) / √m
    # ω: (d, n_features) random orthogonal projection
    omega = torch.randn(d, n_features, device=Q.device) / math.sqrt(d)

    # Apply feature map (positive random features for non-negativity)
    def phi(x):
        proj = x @ omega  # (B, T, n_features)
        return torch.exp(proj - proj.max(dim=-1, keepdim=True).values)

    Q_prime = phi(Q)  # (B, T, m)
    K_prime = phi(K)  # (B, T, m)

    # Key trick: compute K'ᵀV first → (B, m, d) — independent of T!
    KV = K_prime.transpose(-1, -2) @ V  # (B, m, d)

    # Then Q' × KV → (B, T, d)
    output = Q_prime @ KV  # (B, T, d) — linear in T!

    # Normalize (approximate softmax normalization)
    normalizer = Q_prime @ K_prime.transpose(-1, -2).sum(dim=-1, keepdim=True)
    return output / (normalizer + 1e-6)
PerformerQuestion 02 / 07
Performer computes φ(K)TV first (d×d matrix), then multiplies by φ(Q). Why can't standard softmax attention do this?
AStandard attention can — it's just not implemented that way.
BThe softmax function prevents reordering. Standard attention computes softmax(QKT) — the softmax is applied row-wise over the T×T matrix, coupling all T key positions for each query. You must compute all T dot products for each query before applying softmax. This coupling is what forces the T×T materialization. Performer's kernel φ replaces softmax with a decomposable function: φ(q)·φ(k) ≈ softmax(q·k). Decomposable functions allow reordering (associativity of matrix multiplication), eliminating the need for the T×T matrix.
CThe V matrix is too large to multiply first.
DStandard attention uses a different V matrix.
03 / 08
The quality gap — why linear attention hasn't won.

If Performer is O(T×d) while standard attention is O(T²), why isn't everyone using Performer? Because the kernel approximation introduces quality loss:

MetricStandard AttentionLinformer (k=256)Performer (m=256)
WikiText-103 perplexity18.319.1 (+0.8)20.5 (+2.2)
GLUE average89.288.4 (−0.8)86.8 (−2.4)
Long-range retrieval95.1%89.3% (−5.8)82.7% (−12.4)
Compute at T=4K1× (baseline)0.06×0.08×
Compute at T=64K256× (quadratic)0.06× (constant)0.08× (constant)

The pattern: linear attention saves massive compute but loses 1-2 points on general benchmarks and 5-12 points on precise long-range retrieval. For CineMatch's movie recommendation chatbot (general language, short context), this quality loss is unacceptable — GPT-quality responses require exact softmax attention. For CineMatch's batch document processing (100K contracts, approximate understanding sufficient), linear attention's 100× speedup at 2% quality loss is a great trade-off.

Linear attention's worst case: precise needle-in-haystack retrieval

Finding a specific fact in a long document drops from 95% to 83% accuracy with Performer.

The kernel approximation smooths out the attention distribution — sharp, peaked attention (one position gets weight 0.95, others get near-zero) becomes diffuse (many positions get small weights). This is fine for tasks requiring general understanding ("what is this document about?") but catastrophic for precise retrieval ("what date is mentioned on page 3?"). The softmax's ability to concentrate weight on exactly one position is what linear attention sacrifices.

QualityQuestion 03 / 07
Performer loses 12.4% accuracy on long-range retrieval but only 2.4% on GLUE. Why is retrieval so much worse?
AGLUE tasks are easier.
BRetrieval needs more compute.
CGLUE uses shorter sequences.
DRetrieval requires sharp attention — concentrating nearly all weight on one specific position (the answer). The kernel approximation diffuses attention, spreading weight across many positions. GLUE tasks require broad attention — integrating information from many positions. Diffuse attention works fine for integration but fails for pinpointing. It's the difference between "understand this paragraph" (broad, Performer works) and "find the one sentence that answers this question" (sharp, Performer fails).
04 / 08
Linformer vs Performer — when to use which.
Practical selection between linear attention methods
Linformer (low-rank K,V projection): better quality (preserves softmax), but k is fixed at model creation — can't handle variable-length sequences without re-projecting.Best for: fixed-length tasks (classification of documents with known max length), batch processing where T is consistent.
Performer (kernel approximation): lower quality, but handles variable-length sequences naturally (φ is applied per-token, no sequence-level projection needed).Best for: streaming/variable-length tasks, when T varies widely across inputs.
Neither (use FlashAttention): when quality matters more than cost, when T ≤ 128K, when the task requires sharp attention (retrieval, QA).This is the default in 2026. Linear attention is the exception for extreme T or extreme volume.
State-space models (Mamba): an entirely different approach that avoids attention altogether. O(T) with different quality trade-offs. Module 7.2.The emerging alternative: not "approximate attention" but "replace attention entirely."
SelectionQuestion 04 / 07
CineMatch needs to classify 50K user reviews per hour (each 200 tokens) as positive/negative. Should they use linear attention?
ANo — at T=200 tokens, standard attention is already fast. O(200²) = 40K operations per head — trivial. Linear attention saves compute at long sequences (T > 2K), but at T=200, the constant factors of linear attention (random projections, feature maps) can make it slower than standard attention. Plus, DeBERTa at T=200 is ~2ms per review — 50K reviews per hour needs only ~28 GPU-seconds. There's no bottleneck to optimize. Use DeBERTa with standard attention.
BYes — always use the most efficient method.
CYes — 50K reviews per hour requires linear scaling.
DUse Performer for the kernel trick advantage.

CineMatch Integration

CineMatch's attention strategy per component.

Chatbot (LLaMA-3 8B): Standard FlashAttention. Context ≤ 8K for conversations. Quality is paramount — users notice when responses are subtly wrong. No benefit from linear attention at this context length.

Review classification (DeBERTa): Standard attention. T=200. Already fast at 2ms/query. Nothing to optimize.

Script analysis (Longformer): Sparse attention (local W=512 + global [CLS]). T=25K-40K. 100× cheaper than standard at this length. Classification task — minor quality loss is acceptable.

Contract processing (batch): Linformer with k=256. T=50K. 10K docs/day. The 16× compute savings at acceptable quality loss saves $4,500/month. Worth the complexity.

StrategyQuestion 05 / 07
A colleague proposes: "Let's replace ALL attention in our stack with Performer for maximum efficiency." What's the problem?
APerformer can't handle short sequences.
BPerformer is not open-source.
CWrong optimization target. Most of CineMatch's attention is already cheap (T ≤ 512). Replacing standard attention with Performer at T=200 adds overhead without benefit (the kernel computation costs more than the 200² attention it replaces). And for the chatbot (quality-critical), Performer's 2.4-point quality drop on language understanding is unacceptable. Efficient attention should be applied surgically — only where T is large enough to create a bottleneck AND quality tolerance exists.
DAll models need to be retrained from scratch.
06 / 08
RAG vs long context — the complementary solution.

Before concluding Phase 5, a critical perspective: the best "efficient attention" method might be avoiding long context entirely. Retrieval-Augmented Generation (RAG) processes only the relevant chunks — typically 5-10 chunks of 200 tokens each = 1K-2K total tokens — regardless of how large the source corpus is.

RAG vs Long ContextQuestion 06 / 07
CineMatch has 10,000 movie descriptions (5M tokens total). To answer "recommend a sci-fi movie similar to Inception": long-context with linear attention, or RAG?
ALong context — the model needs to see all movies.
BLong context — linear attention makes 5M tokens feasible.
CRAG — but only because the model can't process 5M tokens.
DRAG — because it's the right architecture for search. Embed all 10K descriptions with DeBERTa (batch, done once). At query time: embed the query, find top-20 similar movies via FAISS (2ms), feed 20 descriptions (4K tokens) to LLaMA for generation. Total latency: ~350ms. The model doesn't need to "see" all 10K movies — it needs to see the 20 most relevant ones. RAG + short context is faster, cheaper, and more accurate than any long-context attention method on a search task, because the retrieval step pre-filters to relevant content.
