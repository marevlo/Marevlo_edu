LLaMA.
The model that
opened everything.

LLaMA (Touvron et al., 2023) proved that you don't need OpenAI's budget to build a world-class language model. By applying Chinchilla-optimal training to a clean architecture with open weights, Meta created the foundation that spawned an entire ecosystem: Alpaca, Vicuna, CodeLLaMA, and hundreds of fine-tuned variants. This module traces LLaMA's evolution from 1 to 3, dissects the four architectural innovations that define the modern open-source decoder (RoPE, GQA, SwiGLU, RMSNorm), and builds the decoder block that powers CineMatch's self-hosted chatbot.

The progression tells a story: LLaMA-1 validated Chinchilla-optimal scaling with open weights. LLaMA-2 added RLHF alignment (Chat variants) and doubled context. LLaMA-3 went all-in on data quantity — 15T tokens for the 8B model is 37 tokens/param, nearly 2× the Chinchilla ratio — betting that over-training small models is the optimal strategy for inference economics.

Why LLaMA mattered more than GPT-4 for the ecosystem

GPT-4 is the most capable model. LLaMA is the most impactful. Open weights meant anyone could fine-tune, adapt, and deploy without API dependencies. Within months of LLaMA-1's release: Stanford's Alpaca (instruction-tuned for $600), Berkeley's Vicuna (RLHF'd for $300), and hundreds of domain-specific variants for medicine, law, code, and every language. LLaMA didn't just match GPT-3.5 — it democratized it.

EvolutionQuestion 01 / 07
LLaMA-1 65B (1.4T tokens) matches GPT-3 175B (300B tokens). How does a model with 2.5× fewer parameters match a larger model?
ALLaMA has a fundamentally better architecture.
BLLaMA was trained on better-quality data.
CChinchilla-optimal training. GPT-3 was trained on 300B tokens — only 1.7 tokens/param. LLaMA-1 65B was trained on 1.4T tokens — 21.5 tokens/param, right at the Chinchilla optimum. The 4.7× more training tokens compensate for the 2.5× fewer parameters. LLaMA extracted far more learning from each training FLOP by allocating compute optimally between model size and data quantity.
DLLaMA uses a different tokenizer that makes it more efficient.
02 / 08
Innovation 1 — RoPE (Rotary Position Embeddings).

GPT-2 and BERT use learned absolute position embeddings: a lookup table of size (max_len, d). Position 0 gets vector P[0], position 1 gets P[1], etc. This has two problems: (1) no generalization beyond max_len (can't process sequences longer than training), and (2) no explicit encoding of relative distance between positions.

LLaMA uses RoPE (Su et al., 2021), which encodes position by rotating the query and key vectors in 2D subspaces. The angle of rotation is proportional to the position index:

RoPE(x, pos) = x ⊗ [cos(pos·θ₁), sin(pos·θ₁), cos(pos·θ₂), sin(pos·θ₂), ...] (2.1)

where θi = 10000−2i/d — the same base frequencies as sinusoidal position encoding, but applied as rotations to the Q and K vectors rather than added to the input.

Why rotation encodes relative position
If Q at position m is rotated by angle m·θ, and K at position n is rotated by angle n·θ...Q = R(m·θ) · q, K = R(n·θ) · k (where R is a 2D rotation matrix)
Their dot product: Q·K = (R(m·θ)·q) · (R(n·θ)·k) = q · R((m−n)·θ) · kThe rotation matrices combine as R(m)·R(n)^T = R(m−n). The dot product depends only on the DISTANCE m−n, not the absolute positions m and n.
This is the key property: the attention score between positions m and n depends on (m−n), not on m or n individually.Same property as DeBERTa's relative position — but achieved through rotation, not a separate embedding table.
Bonus: RoPE naturally extrapolates to longer sequences. If the model was trained at max_len=4K, it can process 8K by simply applying larger rotation angles. The model may need some fine-tuning at the longer length, but it doesn't fail catastrophically (unlike learned absolute embeddings, which have no vector for position 4001).LLaMA-3 uses RoPE scaling to extend from 8K training to 128K inference context.
import torch

def apply_rope(x, freqs_cos, freqs_sin):
    """Apply Rotary Position Embedding to queries or keys.
    x: (B, T, n_heads, d_head)
    freqs: (T, d_head//2) — precomputed cos/sin frequencies
    """
    # Split into pairs for 2D rotation
    x_r = x.float().reshape(*x.shape[:-1], -1, 2)  # (..., d_head//2, 2)
    x0, x1 = x_r[..., 0], x_r[..., 1]

    # Apply 2D rotation: [cos·x0 - sin·x1, sin·x0 + cos·x1]
    out0 = x0 * freqs_cos - x1 * freqs_sin
    out1 = x0 * freqs_sin + x1 * freqs_cos

    out = torch.stack([out0, out1], dim=-1).flatten(-2)
    return out.type_as(x)

# Precompute frequencies (done once at model init)
def precompute_rope_freqs(d_head, max_len, base=10000):
    freqs = 1.0 / (base ** (torch.arange(0, d_head, 2).float() / d_head))
    t = torch.arange(max_len)
    angles = torch.outer(t, freqs)  # (max_len, d_head//2)
    return angles.cos(), angles.sin()
RoPEQuestion 02 / 07
RoPE is applied to Q and K but NOT to V. Why not?
AV doesn't need position information.
BPosition encoding's role is to make the attention scores position-aware — which tokens should attend to which. Scores are computed from Q·K, so Q and K need position encoding. V carries the information content that gets aggregated by the attention weights. Adding position to V would mix position into the content representation, which is undesirable — the content should be position-invariant, with position affecting only the routing (which content goes where).
CV vectors are too small for rotation.
DApplying RoPE to V would double the compute cost.
03 / 08
Innovation 2 — Grouped-Query Attention (GQA).

Standard multi-head attention (MHA) gives each head its own Q, K, and V projections. At inference, the KV cache stores K and V for every head at every past position — this dominates memory for long sequences.

Multi-Query Attention (MQA) shares a single K and V across all heads (only Q is per-head). This cuts KV cache by n_heads× but can hurt quality. Grouped-Query Attention (GQA) is the middle ground: group heads and share K, V within each group.

KV cache savings: MHA vs GQA vs MQA (LLaMA-3 70B, 128K context)
MHA: 64 heads × 2 (K,V) × 128K positions × 128 (d_head) × 2 bytes = 4.19 GB per layer80 layers × 4.19 GB = 335 GB total KV cache. More than the model weights (140 GB in BF16)!
GQA (8 KV groups, LLaMA-3 70B): 8 KV heads × 2 × 128K × 128 × 2 = 0.52 GB per layer80 layers × 0.52 GB = 41.9 GB. 8× reduction. Fits in a single A100 (80 GB) alongside weights.
MQA (1 KV head): 1 × 2 × 128K × 128 × 2 = 0.066 GB per layer80 layers × 0.066 GB = 5.2 GB. 64× reduction. But quality degrades — one K,V can't serve 64 different query patterns well.
GQA at 8 groups: each KV head serves 64/8 = 8 query heads. Quality loss: ~0.1-0.3% on benchmarks vs MHA. Cache savings: 8×. This is why every modern large model (LLaMA-3, Mistral, Qwen) uses GQA.The 8× cache reduction enables 128K context on hardware that couldn't hold the MHA cache.
GQAQuestion 03 / 07
LLaMA-3 8B uses GQA with 32 query heads and 8 KV heads. Each KV head serves 4 query heads. What does this mean conceptually?
A4 query heads are forced to have identical attention patterns.
BThe model can only attend to 8 positions at once.
CThe model is 4× slower than standard MHA.
D4 query heads share the same "what's available to attend to" (K,V) but each has its own "what am I looking for" (Q). Each query head computes different attention weights over the same key-value pairs — so they attend to the same set of positions but with different weighting patterns. It's like 4 researchers (Q heads) searching the same library (shared K,V) but each finding different relevant books based on their own research question.
04 / 08
Innovations 3 & 4 — SwiGLU and RMSNorm.

SwiGLU — a better FFN activation.

GPT-2 used GELU in the FFN. LLaMA uses SwiGLU (Shazeer, 2020), a gated activation that outperforms GELU and ReLU on benchmarks:

SwiGLU(x) = (x · W₁) ⊗ SiLU(x · Wgate)   then   · W₂ (4.1)

where SiLU(x) = x · σ(x) (Swish activation) and ⊗ is element-wise multiplication. The key difference: SwiGLU has three weight matrices (W₁, Wgate, W₂) instead of two. To keep the total parameter count comparable, d_ff is reduced from 4d to ~2.67d (specifically 8d/3 rounded).

RMSNorm — cheaper normalization.

LLaMA replaces LayerNorm with RMSNorm (Zhang & Sennrich, 2019):

RMSNorm(x) = x / √(mean(x²) + ε) · γ (4.2)

No mean subtraction, no bias term — just divide by the RMS (root mean square) and scale. This removes the mean computation and the learned bias β, saving ~5% of normalization compute. The quality impact: none measurable. The speed impact: ~2-3% faster training and inference.

import torch, torch.nn as nn

class RMSNorm(nn.Module):
    def __init__(self, d, eps=1e-6):
        super().__init__()
        self.weight = nn.Parameter(torch.ones(d))
        self.eps = eps

    def forward(self, x):
        rms = torch.sqrt(x.pow(2).mean(-1, keepdim=True) + self.eps)
        return x / rms * self.weight

class LLaMABlock(nn.Module):
    """Modern decoder block: pre-RMSNorm, GQA, SwiGLU."""
    def __init__(self, d=4096, n_heads=32, n_kv_heads=8, d_ff=11008):
        super().__init__()
        self.norm1 = RMSNorm(d)
        self.norm2 = RMSNorm(d)

        # GQA: n_heads Q projections, n_kv_heads K,V projections
        d_head = d // n_heads
        self.wq = nn.Linear(d, n_heads * d_head, bias=False)
        self.wk = nn.Linear(d, n_kv_heads * d_head, bias=False)
        self.wv = nn.Linear(d, n_kv_heads * d_head, bias=False)
        self.wo = nn.Linear(n_heads * d_head, d, bias=False)

        # SwiGLU FFN: 3 matrices instead of 2
        self.w1 = nn.Linear(d, d_ff, bias=False)     # gate projection
        self.w3 = nn.Linear(d, d_ff, bias=False)     # up projection
        self.w2 = nn.Linear(d_ff, d, bias=False)     # down projection

    def forward(self, x, freqs_cos, freqs_sin, mask):
        # Pre-RMSNorm + GQA attention + RoPE
        h = self.norm1(x)
        q, k, v = self.wq(h), self.wk(h), self.wv(h)
        # Apply RoPE to q, k (not v!)
        q, k = apply_rope(q, freqs_cos, freqs_sin), apply_rope(k, freqs_cos, freqs_sin)
        # ... GQA attention with k,v shared across head groups ...
        x = x + attn_output

        # Pre-RMSNorm + SwiGLU FFN
        h = self.norm2(x)
        x = x + self.w2(nn.functional.silu(self.w1(h)) * self.w3(h))
        return x
ArchitectureQuestion 04 / 07
SwiGLU has 3 weight matrices vs standard FFN's 2, but LLaMA reduces d_ff from 4d to ~2.67d. Is this a net parameter increase or decrease?
AApproximately neutral. Standard FFN: 2 × d × 4d = 8d² params. SwiGLU: 3 × d × (8d/3) = 8d² params. The math works out exactly: 3 matrices at 2/3 the width = same total as 2 matrices at full width. LLaMA gets SwiGLU's quality improvement for free (in parameter count). The only cost: slightly more memory for the third matrix during training (3 smaller matrices vs 2 larger ones).
B50% increase — SwiGLU is much larger.
C33% decrease — the reduced d_ff dominates.
DImpossible to compare without knowing d.

The modern decoder block — all four innovations combined.

AttentionMHA (all heads independent)GQA (8 KV groups)8× KV cache reduction for long context

These four changes — RoPE, GQA, SwiGLU, RMSNorm — define the "modern decoder block" used by LLaMA-3, Mistral, Qwen, and virtually every open-source LLM released in 2024-2026. The architecture is stable; progress now comes from data, scale, and alignment.

Don't confuse architecture innovations with training improvements

LLaMA-3's jump over LLaMA-2 comes mostly from 7.5× more data, not architecture.

LLaMA-2 → LLaMA-3 changed GQA configuration and minor details. But the 8B model went from 2T tokens to 15T tokens — 7.5× more training data. On benchmarks, ~70% of the improvement comes from more/better data, ~20% from longer training, and ~10% from architectural tweaks. This mirrors the RoBERTa lesson from Module 1.2: training recipe > architecture. The modern decoder block has converged; the differentiator is data.

CineMatch on LLaMA — the self-hosted production stack.
CineMatch Integration

LLaMA-3 8B is CineMatch's production chatbot backbone.

Why LLaMA-3 8B: open weights (no API dependency), fine-tunable (DPO on CineMatch preference data), fast inference (single A10G GPU, ~300ms per response), and the 15T-token pretraining gives strong base knowledge about movies, culture, and conversational patterns.

Fine-tuning recipe: SFT on 5K CineMatch (query, ideal_response) pairs → DPO on 10K preference-ranked response pairs. Total training: 4 hours on 4× A100. The fine-tuned model outperforms GPT-3.5 on CineMatch-specific queries (domain fine-tuning > general capability for narrow domains).

Serving: vLLM or TGI for efficient batched inference with continuous batching and PagedAttention. GQA enables 128K context with manageable KV cache. RoPE allows processing long user histories. Cost: ~$150/month on a single A10G for 50K queries/day — 22× cheaper than GPT-4 API.

The ecosystem: If CineMatch needs code generation (movie trivia game), use CodeLLaMA. If it needs multilingual support, fine-tune LLaMA-3 on translated reviews. If it needs a tiny on-device model, use LLaMA-3 8B quantized to 4-bit (~4GB). One model family, many deployment configurations.

ProductionQuestion 06 / 07
CineMatch's LLaMA-3 8B chatbot occasionally generates recommendations for movies that don't exist in CineMatch's catalog. The model "hallucinates" movie titles. What's the best fix?
AFine-tune on more CineMatch data — the model will memorize the catalog.
BRAG (Retrieval-Augmented Generation). Before generating, retrieve the top 20 candidate movies from CineMatch's catalog using the user query + embeddings (from Phase 1's DeBERTa). Prepend these movie titles and metadata into the prompt. The model generates recommendations from the retrieved candidates only, eliminating hallucinated titles. Fine-tuning teaches the model to use the retrieved context; RAG provides the factual grounding.
CIncrease temperature — the model will explore more options.
DSwitch to GPT-4 — it hallucinates less.

LandscapeQuestion 07 / 07
A CineMatch competitor operates in India and needs Hindi + English support. Which open-source model family should they start with?
ALLaMA-3 — it's always the best open model.
BMistral — it's the most efficient.
CGemma — Google has the most multilingual data.
DQwen-2.5. Alibaba specifically optimized Qwen for multilingual performance, including strong Hindi support. LLaMA-3 is English-dominant (most of its 15T tokens are English). Mistral is primarily English/European. Qwen's tokenizer handles Indic scripts efficiently (fewer tokens per Hindi word), and its training data includes substantial non-English content. For a bilingual Hindi+English product, Qwen is the strongest starting point — then fine-tune on domain-specific bilingual data.
