BART.
The denoising
autoencoder.

T5 frames every task as text-to-text. BART (Lewis et al., 2020) frames pretraining as denoising: corrupt the input in various ways (mask tokens, delete sentences, shuffle order), then train the model to reconstruct the original. This flexible corruption scheme produces an encoder-decoder that excels at generation tasks — particularly summarization, where BART was state-of-the-art for years and remains a strong production baseline. For CineMatch, BART powers the review digest feature: condensing long user reviews into 2-sentence summaries for the browse UI.

01 / 08
The denoising autoencoder idea.

BART's pretraining is simple: take clean text, corrupt it, feed the corrupted text to the encoder, and train the decoder to output the original clean text. The model learns to undo corruption — which requires understanding the text deeply enough to reconstruct what was lost.

BART pretraining:   P(x | corrupt(x))   — reconstruct original from corrupted input (1.1)

What makes BART different from BERT's MLM: BERT only masks individual tokens and predicts them in place. BART can apply any corruption — deletion, insertion, shuffling, masking — and the decoder must generate the entire clean sequence, not just fill blanks. This trains a much more capable generator.

Five corruption strategies — and why the combination works.

StrategyWhat it doesWhat the model learns
Token maskingReplace random tokens with [MASK]Fill-in-the-blank (like BERT's MLM)
Token deletionRemove random tokens entirelyDetect what's missing and where to insert it
Text infillingReplace spans with a single [MASK] tokenGenerate multi-token spans (like T5's span corruption)
Sentence permutationShuffle sentence order randomlyReconstruct coherent document structure
Document rotationRotate document to start at a random tokenIdentify the true beginning of a document

BART's best configuration uses text infilling + sentence permutation together. Text infilling (like T5) teaches local language generation. Sentence permutation teaches global document structure. The combination produces a model that understands both local syntax and global coherence — exactly what summarization needs.

DenoisingQuestion 01 / 07
BERT masks tokens. T5 masks spans. BART masks spans AND shuffles sentences. Why is the combination better for summarization?
AMore corruption = harder task = better model.
BSentence shuffling trains the model on longer sequences.
CThe combination is just a regularization trick.
DSummarization requires two capabilities: (1) understanding local content (what each sentence says) and (2) understanding global structure (which sentences are important, how they relate). Text infilling trains capability 1 — reconstruct missing phrases from context. Sentence permutation trains capability 2 — figure out the correct document structure from a scrambled version. Neither alone is sufficient; together they train both capabilities the model needs for summarization.

BART's architecture — BERT encoder + GPT decoder.

BART's architecture is remarkably simple to describe: take BERT's encoder (bidirectional, no causal mask) and GPT's decoder (causal self-attention + autoregressive generation), connect them with cross-attention. That's it.

ConfigEncoderDecoderdHeadsTotal params
BART-base6 layers6 layers76812139M
BART-large12 layers12 layers102416406M
BART-large vs T5-base — architectural differences that matter
BART-large: 12+12 layers at d=1024. Total: 406M. Position: learned absolute (max 1024).Uses BERT-style post-norm LayerNorm and learned absolute positions. GELU activation.
T5-base: 12+12 layers at d=768. Total: 220M. Position: relative bias (32 buckets).Uses pre-norm LayerNorm and relative position bias. ReLU activation (later versions use GELU).
Key difference: BART outputs the full reconstructed sequence (all tokens). T5 outputs only the missing spans.BART's decoder generates T_input tokens. T5's decoder generates ~0.15 × T_input tokens. BART is ~7× more expensive to pretrain per example.
This design choice makes BART a better pretrained generator (it practices generating full sequences) but a less efficient pretraining objective (wasted compute on reconstructing unmasked tokens).For summarization, BART's full-sequence generation pretraining matches the downstream task structure better than T5's span-only output.
ArchitectureQuestion 02 / 07
BART-base has 6 encoder + 6 decoder layers (139M params). BERT-base has 12 encoder layers (110M). Why does BART have more parameters with fewer encoder layers?
ABART uses a larger vocabulary.
BBART has a decoder. Each decoder layer has 3 sub-layers (self-attention + cross-attention + FFN) vs the encoder's 2 (self-attention + FFN). The 6 decoder layers add ~63M params (each with an extra cross-attention module: 4 × d² = 4 × 768² ≈ 2.4M extra per layer × 6 = 14.4M, plus the FFN and self-attention). 6 encoder layers (42M) + 6 decoder layers (56M) + embeddings (24M) + overhead ≈ 139M.
CBART uses wider layers (larger d).
DBART uses more attention heads.
03 / 08
BART for summarization — the production recipe.
# BART summarization — CineMatch review digest
from transformers import BartForConditionalGeneration, BartTokenizer

model = BartForConditionalGeneration.from_pretrained("facebook/bart-large-cnn")
tok = BartTokenizer.from_pretrained("facebook/bart-large-cnn")

review = """The movie started with an incredibly promising premise — a heist
gone wrong in a small town. The first act was gripping, with excellent
performances from the entire cast. However, the second act dragged
significantly, with unnecessary subplots that added nothing to the main
story. The twist at the end partially redeemed it, but by then I had
already lost interest. Visually stunning throughout, with a great
soundtrack that deserved a better film. 6/10."""

inputs = tok(review, return_tensors="pt", max_length=1024, truncation=True)
summary_ids = model.generate(
    inputs.input_ids,
    max_length=60,    # 2-sentence summary
    min_length=20,
    num_beams=4,      # beam search for quality
    length_penalty=2.0,
    no_repeat_ngram_size=3,
)
summary = tok.decode(summary_ids[0], skip_special_tokens=True)
# "A promising heist thriller with strong performances but a dragging 
#  second act. Visually stunning with a great soundtrack. 6/10."
Beam search vs sampling for summarization

Use beam search for summarization, not nucleus sampling.

Summarization requires faithful, factual output — the summary must accurately represent the original text. Beam search (num_beams=4-6) produces the most likely sequence, which tends to be factual. Nucleus sampling (top-p) introduces randomness that can cause hallucinated facts — "the director won 3 Oscars" when the review never mentioned awards. For creative generation (chatbot, story writing), sampling is better. For faithfulness-critical tasks (summarization, translation), beam search is better.

SummarizationQuestion 03 / 07
BART-large-CNN (fine-tuned for summarization) generates: "The movie was praised by critics and won several awards." But the original review says the movie was rated 6/10. What went wrong?
AThe model is too small to understand the review.
BThe tokenizer corrupted the input.
CHallucination. The model generated plausible-sounding but factually wrong content. "Won several awards" doesn't appear anywhere in the input — the model invented it because it's a common pattern in movie summaries it saw during training. This is the same hallucination problem from Module 2.1: the model generates text that looks right (high probability) but isn't right (not supported by the input). Fixes: increase length_penalty (forces shorter, more conservative summaries), use constrained decoding (only allow words from the input), or add a fact-verification post-processing step.
DThe beam search found a wrong path.

The practical answer: use FLAN-T5 for new projects

BART and T5 are architecturally similar (both encoder-decoder). BART's advantage was summarization quality; T5's advantage was multi-task flexibility. In 2026, FLAN-T5 has both: strong summarization AND zero-shot multi-task capability from instruction tuning. BART remains relevant for legacy systems and for specialized summarization pipelines where bart-large-cnn's fine-tuned quality is unmatched. For new projects, start with FLAN-T5.

Fine-tuning BART — the summarization recipe.
# Fine-tune BART for CineMatch review summarization
from transformers import BartForConditionalGeneration, Trainer, TrainingArguments

model = BartForConditionalGeneration.from_pretrained("facebook/bart-large")

training_args = TrainingArguments(
    output_dir="./cinematch-summarizer",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,   # effective batch = 16
    learning_rate=3e-5,              # low LR for fine-tuning
    warmup_ratio=0.1,
    fp16=True,
    eval_strategy="steps",
    eval_steps=500,
    save_strategy="steps",
    save_steps=500,
    load_best_model_at_end=True,
    metric_for_best_model="rouge2",  # ROUGE-2 F1 for summarization
    label_smoothing_factor=0.1,    # prevents overconfident predictions
)

# Dataset: 10K CineMatch (review, human_summary) pairs
# Expected: ROUGE-2 F1 ~22-25 after fine-tuning (competitive with BART-CNN)
Label smoothing is critical for summarization fine-tuning

Without label smoothing, BART over-fits to exact reference summaries and loses generalization.

The reference summaries are just one possible good summary — many valid summaries exist for any text. Without label smoothing (factor=0.1), the model trains to exactly reproduce the reference, penalizing all alternative wordings equally. With smoothing, some probability mass is distributed to non-reference tokens, allowing the model to learn that "great" and "excellent" are both valid descriptions even if the reference uses "great." This produces more flexible, natural summaries at inference time.

Fine-tuningQuestion 05 / 07
ROUGE-2 F1 is the standard summarization metric. It measures bigram overlap between generated and reference summaries. What's its main limitation?
AIt's too slow to compute.
BIt requires the reference summary to be perfect.
CIt can only evaluate English text.
DROUGE measures surface word overlap, not semantic faithfulness. A summary that uses different words to express the same meaning (paraphrasing) gets a low ROUGE score. A summary that copies phrases from the input verbatim gets a high ROUGE score — even if it's incoherent. ROUGE rewards extraction (copying), not abstraction (understanding and rephrasing). Modern evaluation uses model-based metrics (BERTScore, GPT-4-as-judge) that assess meaning rather than surface overlap.

When encoder-decoder beats decoder-only — and when it doesn't.
Architecture selection for seq2seq tasks in 2026
Encoder-decoder wins when: the task is input→output with a clear boundary (summarization, translation, structured extraction), input is fully available before generation starts, quality-per-parameter matters (smaller models, tighter budgets)BART/FLAN-T5: 220M-770M params. Matches decoder-only quality at 10-30× fewer parameters on structured tasks.
Decoder-only wins when: the task is open-ended generation (chat, creative writing, reasoning), you need one model for many tasks, you can afford large models (7B+), or the boundary between "input" and "output" is fuzzy (multi-turn conversation)LLaMA/Mistral: 7B-70B params. Better generality, more ecosystem support, active development.
The trend: decoder-only is winning by ecosystem momentum, not by architectural superiority. FLAN-T5-XXL (11B) matches GPT-3 (175B) on many tasks — but nobody is training new encoder-decoders at 100B+ scale. Investment goes to decoder-only because it's the general-purpose architecture.Encoder-decoder remains the better architecture for structured tasks. But "better architecture at smaller scale" loses to "good enough architecture with massive ecosystem" in practice.
CineMatch Integration

The production encoder-decoder stack.

BART-large-CNN for review digests: 406M params, generates 2-sentence summaries at ~200ms on A10G. Beam search with length penalty produces factual, concise summaries. Batch processing: 50K new reviews/night in ~2 hours.

FLAN-T5-base for structured extraction: 220M params, extracts (actors, genre, sentiment, spoiler) from reviews in a single forward pass. The text-to-text format means one model handles 4 extraction tasks.

Why not decoder-only for these tasks: LLaMA-3 8B can summarize and extract, but at 8B params it costs 10-20× more compute per query than BART/T5 at 220-406M. For batch processing 50K reviews, the encoder-decoder saves $40/night in compute — $1,200/month. Worth it for a clearly-scoped, high-volume task.

Architecture choiceQuestion 06 / 07
CineMatch wants to add a new feature: "Translate this review to Spanish." The review is 200 words. Which model?
ABART — it's the best at structured tasks.
BGPT-4 — it has the best translation quality.
CFLAN-T5 or a dedicated translation model (NLLB). Translation is a classic encoder-decoder task: the full source sentence must be encoded bidirectionally (word order differs between languages), then the target sentence is generated left-to-right. FLAN-T5 handles translation well zero-shot. For production quality, Meta's NLLB-200 (dedicated multilingual translation, encoder-decoder, 600M params) produces better translations than any general-purpose model at 10× lower cost than an API call. Match the model to the task.
DLLaMA-3 8B — it's the default production model.

SynthesisQuestion 07 / 07
Across the entire track so far, you've seen encoder-only (BERT family), decoder-only (GPT/LLaMA family), and encoder-decoder (T5/BART). If you could only keep TWO of the three for CineMatch's entire production stack, which two?
AEncoder-only + Encoder-decoder: covers understanding and structured generation.
BEncoder-only + Decoder-only. Encoder-only (DeBERTa) for classification, NER, embeddings — the high-volume, low-latency tasks that make up 95% of queries. Decoder-only (LLaMA) for generation — chatbot, open-ended recommendations, complex reasoning. This pair covers all of CineMatch's needs. Encoder-decoder (BART/T5) is squeezed from both sides: DeBERTa handles understanding better, and modern decoder-only models handle structured generation well enough. The encoder-decoder niche (summarization, translation) can be filled by decoder-only at slightly higher cost.
CDecoder-only + Encoder-decoder: covers all generation needs.
DAll three are equally essential — can't drop any.
