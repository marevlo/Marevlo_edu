Multimodal
LLMs.
Models that see.

CLIP aligns vision and language in an embedding space. Multimodal LLMs go further — they let language models reason about images. GPT-4V, LLaVA, and Gemini don't just classify images; they answer questions about them, describe scenes, read text in photos, and reason about visual relationships. The architecture is elegantly simple: take a pretrained CLIP image encoder, take a pretrained LLM, and connect them with a projection layer. The vision encoder sees; the language model reasons. This module covers how it works, why it works, and how CineMatch builds a multimodal chatbot that understands movie posters and frames.

01 / 08
The architecture — image encoder + projector + LLM.

Every multimodal LLM follows the same three-component pattern:

ComponentWhat it doesTypical model
Image encoderConverts image to visual tokens (e.g., 576 × d vectors)CLIP ViT-L/14 (frozen or lightly tuned)
Projection layerMaps visual tokens from image encoder's d to LLM's d2-layer MLP (trainable)
Language modelProcesses interleaved visual + text tokens, generates responseLLaMA-3, Mistral, Vicuna (frozen or LoRA-tuned)

The input to the LLM looks like: [image_token_1, image_token_2, ..., image_token_576, "Describe", "this", "movie", "poster"]. The LLM's decoder processes this mixed sequence with standard causal attention — the text tokens attend to both image tokens and preceding text tokens. The image tokens are "understood" through the same attention mechanism that processes text.

LLaVA-1.5 architecture — the open-source multimodal LLM
Image encoder: CLIP ViT-L/14 @ 336px → 576 visual tokens × 1024 dimensions.Frozen during Stage 1 training. The CLIP encoder's visual representations are already language-aligned.
Projection: 2-layer MLP maps 1024 → 4096 (matching LLaMA's d).~8M trainable params. This is the "alignment layer" that adapts CLIP features for the LLM.
LLM: Vicuna-13B (LLaMA-2 chat variant). 13B params, LoRA fine-tuned.The LLM receives 576 image tokens + text tokens. It doesn't know the image tokens came from a ViT — they're just more tokens.
Total sequence: 576 image tokens + ~200 text tokens = ~776 tokens. Standard 2048-token context handles this easily.Multiple images: 2 images = 1152 image tokens + text. Fits in 2048 but gets expensive for many images.
ArchitectureQuestion 01 / 07
LLaVA uses a frozen CLIP image encoder and a fine-tuned LLM. Why freeze the image encoder?
AThe image encoder can't be fine-tuned.
BFine-tuning the image encoder would be too expensive.
CCLIP's visual representations are already excellent — fine-tuning risks degrading them. CLIP was trained on 400M image-text pairs; the visual features are rich, general, and language-aligned. Fine-tuning on a smaller multimodal dataset (600K image-text conversations for LLaVA) could overfit the encoder to the fine-tuning data's visual distribution, losing generality. Freezing preserves CLIP's broad visual knowledge while letting the projection layer and LLM adapt to the multimodal task.
DThe image encoder and LLM use different frameworks.
02 / 08
Training — two stages for multimodal alignment.

LLaVA trains in two stages:

Stage 1 — Alignment pretraining (558K image-caption pairs): Only the projection MLP is trainable. The image encoder and LLM are frozen. The task: given an image, generate its caption. This teaches the projection layer to translate CLIP visual features into tokens the LLM can understand.

Stage 2 — Visual instruction tuning (665K multimodal conversations): The projection MLP and LLM are both trainable (LLM via LoRA). The task: follow visual instructions — "Describe this image," "What's happening in this scene?", "How many people are in this photo?" This teaches the LLM to reason about visual content, not just describe it.

TrainingQuestion 02 / 07
Stage 1 trains only the 8M-parameter projection MLP. Why is this sufficient to connect a 300M image encoder to a 13B LLM?
A8M parameters is actually a lot for a projection layer.
BBecause the hard work is already done. CLIP's image encoder produces features that are already language-aligned (trained with text-image contrastive learning). The LLM already understands language. The projection MLP just needs to map between two representation spaces that already share semantic structure. It's like translating between two languages that share the same grammar — you need a dictionary (projection), not a full language model. 8M parameters is enough for a dimension-matching linear transform with a nonlinearity.
CThe projection doesn't actually work — all the quality comes from Stage 2.
DModern GPUs can train 8M parameters very efficiently.

Gemini is architecturally different: it's trained from scratch as a natively multimodal model (images and text in the same training data from the start), not a bolted-together combination of separate encoders. This may provide deeper visual-linguistic integration but requires enormously more training data and compute. The "CLIP encoder + LLM" approach (LLaVA, GPT-4V) is more practical and produces competitive results.

LandscapeQuestion 03 / 07
Gemini is "natively multimodal" — trained from scratch on mixed image-text data. LLaVA bolts a CLIP encoder onto an existing LLM. Which approach produces better multimodal understanding?
ANative (Gemini) — always better integration.
BBolted (LLaVA) — pretrained components are always better.
CThey're equivalent in all cases.
DIt depends on the scale and task. Native multimodal theoretically enables deeper integration (cross-modal attention from the first layer). Bolted practically is more efficient (reuses existing pretrained components). At Google's scale (trillions of tokens, billions of dollars), native training is feasible and produces slightly better compositional reasoning. For everyone else (including CineMatch), the bolted approach achieves 90%+ of native quality at 1% of the training cost. LLaVA-1.5 achieves GPT-4V-level scores on many visual benchmarks at 1000× lower training cost.

CineMatch's multimodal chatbot.
CineMatch Integration

A chatbot that understands movie posters and frames.

The product experience: Users upload a movie poster or screenshot and ask: "What genre is this?" "Is this appropriate for kids?" "What movies have a similar visual style?" The chatbot sees the image (via CLIP encoder), understands the question (via LLM), and generates a response that references visual details.

Architecture: LLaVA-1.5 with Vicuna-7B (lighter than 13B for serving cost). CLIP ViT-L/14 encodes the poster into 576 tokens. The LLM processes image tokens + question and generates the response. Total inference: ~800ms per query on A10G.

Fine-tuning: Stage 1 (alignment) on 100K CineMatch poster-caption pairs. Stage 2 (instruction) on 20K CineMatch visual Q&A pairs (manually annotated: "What actors are on this poster?" → "The poster features three actors: [name], [name], [name] with [description of visual style]"). Total training: 8 GPU-hours on 4× A100.

Cost: ~$0.01 per multimodal query (image encoding + LLM generation). At 5K visual queries/day: ~$50/day = $1,500/month. Comparable to GPT-4V API cost but with domain-specific quality and no API dependency.

ProductionQuestion 04 / 07
CineMatch's multimodal chatbot is asked: "How many people are on this poster?" for a poster with 5 actors. CLIP-based models notoriously struggle with counting. How can CineMatch improve accuracy?
AUse higher-resolution image encoding. CLIP ViT-L/14 at 224px merges faces into coarse patches. At 336px (LLaVA-1.5's default) or 672px (with patch subdivision), individual faces are better resolved in separate patches, making them countable. Additionally, fine-tune on counting-specific visual Q&A data ("How many X in this image?" with ground truth counts). The combination of higher resolution + counting-specific training improves from ~40% to ~75% counting accuracy.
BUse a separate object detection model (Swin + detector) for counting.
CCounting is impossible for multimodal LLMs.
DAsk the user to count manually.
05 / 08
Video understanding — the next frontier.

Images are single frames. Videos are sequences of frames. Extending multimodal LLMs to video requires processing many frames while managing compute and context costs.

The standard approach: sample K frames uniformly from the video (typically K=8-32), encode each with the image encoder (576 tokens per frame), and concatenate all visual tokens with the text query. For K=16 frames: 16 × 576 = 9,216 visual tokens + text. This fits in modern 128K contexts but is expensive.

VideoQuestion 05 / 07
CineMatch wants to analyze full movie trailers (2 minutes, 30fps = 3600 frames). Encoding all frames with CLIP: 3600 × 576 = 2M tokens. Is this feasible?
AYes — modern models support 2M tokens.
BYes — but only with linear attention.
CNo — subsample. 30fps is redundant; adjacent frames are nearly identical. Sample 1 frame per second (120 frames) or even 1 per 2 seconds (60 frames). 60 × 576 = 34,560 tokens — fits in 128K context. Quality loss from subsampling is minimal for trailer analysis (which needs scene understanding, not frame-perfect motion analysis). For temporal precision (action detection, lip sync), use specialized video models, not frame-sampling multimodal LLMs.
DNo — multimodal LLMs can't process video at all.

SelectionQuestion 06 / 07
A user uploads a movie poster to CineMatch and asks: "What genre is this movie?" Which model should handle this request?
AAlways the multimodal LLM — it's the most capable.
BAlways CLIP — zero-shot is sufficient.
CAlways Swin — best for image understanding.
DCLIP for this specific task. Genre classification is a categorization task — match the image to text labels. CLIP does this zero-shot at $0.001/query in ~50ms. The multimodal LLM would work but costs 10× more ($0.01) and takes 10× longer (~500ms) — overkill for simple classification. Route to the cheapest model that handles the task: CLIP for classification, multimodal LLM for reasoning ("why does this look like a horror movie?"), Swin for detection ("find text in this poster").

Phase 6 — complete. Vision meets language.
SynthesisQuestion 07 / 07
Across Phase 6, what is the single most important architectural insight?
AThe transformer is modality-agnostic. ViT proved images can be processed as patch sequences with the same attention mechanism as text. CLIP proved vision and language can share an embedding space. Multimodal LLMs proved visual tokens can be interleaved with text tokens in a single decoder. The transformer doesn't care what the tokens represent — patches, words, audio frames, video cubes. This universality is what makes multimodal AI possible without modality-specific architectures.
BCNNs should be completely replaced by transformers.
CCLIP is the best image model.
DMultimodal LLMs can replace all vision models.
Phase 6 — Complete

From patches to multimodal reasoning — vision meets language.
