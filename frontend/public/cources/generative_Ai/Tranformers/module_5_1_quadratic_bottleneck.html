The quadratic
bottleneck.

Standard self-attention computes a score between every pair of positions — O(T²) in sequence length. At T=512, that's 262K scores per head. At T=128K, it's 16.4 billion. This quadratic cost is the fundamental bottleneck that limits context length. This module quantifies the problem precisely: where the memory goes, where the compute goes, and why naive scaling breaks. Modules 5.2 and 5.3 present the solutions.

The math of the bottleneck.

Attention computes QKT — a matrix of shape (T, T). Every entry is a dot product between two dk-dimensional vectors. The costs:

Attention cost breakdown at different sequence lengths (d=4096, h=32)
T=512: QKT has 512² = 262K entries. Memory: 262K × 4 bytes = 1 MB per head, 32 MB total.Trivial. This is why BERT (512 tokens) never worried about attention efficiency.
T=4096: 4096² = 16.8M entries. Memory: 16.8M × 4 × 32 heads = 2.1 GB per layer.Significant but manageable on an A100 (80 GB). GPT-3 and LLaMA operate here.
T=32K: 32K² = 1.07B entries. Memory: 1.07B × 4 × 32 = 137 GB per layer.Exceeds a single A100. Must use FlashAttention (memory-efficient) or sparse patterns.
T=128K: 128K² = 16.4B entries. Memory: 16.4B × 4 × 32 = 2.1 TB per layer.Impossible to materialize. Even with FlashAttention, the compute cost is enormous: 16.4B multiply-adds per head per layer.
Scaling: 2× sequence length → 4× attention cost. 10× sequence length → 100× cost. This is why context windows grew slowly (512 → 2K → 4K → 8K over 5 years) while model size grew fast.The bottleneck is quadratic in T, not in d or L. Increasing d or L is linear; increasing T is quadratic.
Quadratic costQuestion 01 / 07
A model processes documents of 8K tokens. A new requirement doubles the context to 16K. How much more attention compute is needed?
A2× — linear scaling with sequence length.
B4×. Attention is O(T²). Doubling T: (2T)² = 4T². The attention layer uses 4× more compute and 4× more memory. Every other component (embedding, FFN, LayerNorm) scales linearly — only attention is quadratic. At 8K→16K, attention goes from the dominant cost to the overwhelming cost.
C8× — cubic scaling.
D16× — exponential scaling.
02 / 06
Where the memory actually goes.

At long context, memory is consumed by two things: the attention score matrix (T × T per head) and the KV cache (T × d per layer for K and V). Both scale with T, but the attention matrix scales with T² while the KV cache scales with T:

ComponentMemory (BF16)ScalingAt T=128K, d=4096, 32 layers
Model weights~15 GBConstant15 GB
KV cache2 × T × d × L × 2BO(T)64 GB
Attention scoresT² × h × 2B per layerO(T²)~2 TB (materialized)

FlashAttention (Dao et al., 2022) solved the attention score memory problem: it computes attention without materializing the full T×T matrix, using a tiling algorithm that keeps only small blocks in fast SRAM. FlashAttention reduces attention memory from O(T²) to O(T) — but the compute is still O(T²). The operations are the same; they're just done in a memory-efficient order.

FlashAttention is not efficient attention — it's memory-efficient standard attention

FlashAttention doesn't change the attention algorithm. It computes the exact same QKT softmax product — just without storing the full T×T matrix. This eliminates the memory bottleneck but not the compute bottleneck. For truly sub-quadratic attention, you need architectural changes: sparse patterns (Module 5.2) or linear approximations (Module 5.3).

MemoryQuestion 02 / 07
With FlashAttention, can you process 128K tokens on a single A100 (80 GB)?
AYes — FlashAttention eliminates all memory concerns.
BNo — 128K tokens are always impossible on one GPU.
CDepends on the model. FlashAttention removes the O(T²) attention score memory. But the KV cache is still O(T): at T=128K, d=4096, 32 layers: 64 GB for KV cache + 15 GB for weights = 79 GB. Barely fits on A100 80GB with GQA (which reduces KV cache 4-8×). Without GQA: doesn't fit. With GQA (8 KV groups): KV cache drops to ~8-16 GB. Then 128K fits. This is why GQA (Module 3.1) and FlashAttention are complementary: FlashAttention handles attention scores, GQA handles KV cache.
DOnly with quantization.
03 / 06
The three approaches to breaking quadratic attention.
ApproachComplexityHowModelsModule
Sparse patternsO(T × w)Each position attends to a fixed subset (local + global)Longformer, BigBird5.2
Low-rank approximationO(T × k)Project K,V to lower dimension before attentionLinformer5.3
Kernel approximationO(T × d)Replace softmax with a kernel, enabling linear decompositionPerformer5.3

All three trade some attention quality for sub-quadratic scaling. The question is: how much quality is lost? The answer depends on the task. For tasks requiring precise long-range retrieval ("find the fact mentioned on page 3 of a 50-page document"), sparse patterns with global tokens (Longformer) work well. For tasks requiring general understanding of long text, any sub-quadratic method works because most attention weights are near-zero anyway.

ApproachesQuestion 03 / 07
In standard attention at T=8K, 64 million attention weights are computed. Empirically, what fraction are near-zero (< 0.001)?
A~10% — most weights are significant.
B~50% — evenly split.
C~80% — most are small but not negligible.
D~95%+. Softmax is inherently peaky: a few positions get most of the weight, and the long tail of positions gets negligible weight. In a typical transformer head at T=8K, each position has ~50-200 "significant" attention targets out of 8K total. The other 7,800+ positions each get < 0.001 weight. This sparsity is what efficient attention methods exploit: if 95% of attention weights are near-zero, you can skip computing them entirely.

CineMatch's long-context problem.
CineMatch Integration

Where CineMatch hits the quadratic wall.

Movie script analysis: Full movie scripts are 20K-40K tokens. CineMatch's script-to-summary pipeline needs to process the entire script to identify key plot points, themes, and character arcs. At 40K tokens with standard attention: 40K² × 32 heads = 51.2B operations per layer. Impractical on a single GPU without efficient attention.

User conversation history: Power users have 500+ conversation turns with CineMatch's chatbot. At ~50 tokens per turn, that's 25K tokens of history. Fitting this into context requires either efficient attention or a retrieval/compression strategy.

Review corpus analysis: Analyzing all reviews for a movie (100+ reviews, ~200 tokens each = 20K tokens) in a single forward pass for aggregate sentiment/theme extraction. Standard attention is the bottleneck; sparse or linear attention makes this feasible.

ApplicationQuestion 04 / 07
CineMatch's standard LLaMA-3 8B (128K context with GQA + FlashAttention) can process a 40K-token script. Why would CineMatch still consider efficient attention methods?
AFlashAttention doesn't work with LLaMA.
BCost. FlashAttention + GQA makes 128K context possible but not cheap. At T=40K, attention compute is still O(40K²) = 1.6B operations per head per layer. Efficient attention (O(T×w) with w=4K) reduces this to O(40K × 4K) = 160M — 10× cheaper. For a batch pipeline processing 1000 scripts/day, this 10× compute saving translates directly to 10× lower GPU cost. "Can we do it?" is a different question from "can we afford to do it at scale?"
CLLaMA-3 can't process scripts — only reviews.
DQuality is always better with efficient attention.
05 / 06
The efficiency landscape — solutions mapped.
How modern systems actually handle long context (2026)
FlashAttention + GQA (standard approach): O(T²) compute, O(T) memory. Used by LLaMA-3, Mistral, GPT-4. Works up to ~128K tokens on modern hardware.This is the default. Good enough for most applications. No quality loss.
Sliding window (Mistral): O(T × W) compute and memory. Each position attends to W=4K nearest positions. Info propagates via stacking.Slight quality loss on precise long-range retrieval. No loss on general understanding.
Sparse patterns (Longformer, BigBird): O(T × (W + G)) with W local + G global tokens. Best for long document understanding.Module 5.2. The document-processing specialist.
Linear attention (Performer, etc.): O(T × d) — truly linear. Replace softmax with a kernel decomposition.Module 5.3. Largest theoretical speedup but quality loss on some tasks.
RAG (retrieval, not attention): O(1) in context length. Don't put everything in context — retrieve the relevant chunks and only process those.Often the best practical solution. Avoids the attention problem entirely by reducing what goes into the context.
LandscapeQuestion 05 / 07
CineMatch needs to search through 100K movie descriptions (total: 5M tokens) to find movies matching a user query. Should they use a 5M-context efficient attention model?
AYes — efficient attention can handle 5M tokens.
BYes, but only with linear attention (Performer).
CNo — use retrieval (RAG). Embedding all 100K descriptions and doing vector search (DeBERTa + FAISS) is O(1) in the number of descriptions at query time (pre-computed embeddings + approximate nearest neighbor). Processing 5M tokens through any attention model — even linear — is vastly more expensive than a vector lookup. Efficient attention solves "long document understanding." Retrieval solves "search through many documents." These are different problems requiring different solutions.
DNo — 100K descriptions is too many for any system.
DesignQuestion 06 / 07
For CineMatch's script analysis (40K tokens, moderate volume), which approach gives the best quality-cost trade-off?
AFlashAttention + GQA (standard, no quality loss). At 40K tokens, standard attention with FlashAttention uses ~1.6B ops per head per layer — manageable on an A100 in ~2 seconds per script. At 100 scripts/day, total GPU cost is ~$0.50/day. The 10× savings from sparse attention ($0.05/day) isn't worth the engineering complexity and potential quality loss. Efficient attention becomes worthwhile at 10K+ scripts/day.
BLinear attention — always cheapest.
CRAG — retrieve relevant script sections only.
DLongformer — designed for documents.
SynthesisQuestion 07 / 07
Will the quadratic attention bottleneck ever be fully "solved"?
AYes — linear attention already solved it.
BYes — hardware will eventually be fast enough.
CNo — it's mathematically impossible.
DIt's being managed, not solved. FlashAttention + GQA handle 128K tokens today. State-space models (Mamba, Module 7.2) achieve O(T) with different trade-offs. Linear attention approximates softmax attention at O(T) but loses some quality. In practice, the "solution" is a portfolio: standard attention for short context, sparse for medium, retrieval for large, and SSMs as a potential replacement. The bottleneck isn't a single problem with a single solution — it's a design space with multiple trade-offs.
