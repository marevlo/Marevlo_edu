The serving stack — vLLM, TGI, and continuous batching.

A trained model is a file of weights. Serving it at scale requires an inference engine that handles: concurrent requests, efficient memory management, batching, and streaming responses. The two dominant open-source engines:

FeaturevLLMTGI (Text Generation Inference)
DeveloperUC BerkeleyHugging Face
Continuous batchingYes (PagedAttention)Yes
KV cache managementPagedAttention (virtual memory for KV cache)Static allocation
Throughput (tokens/sec)~2-4× higher than naive~1.5-2× higher
Model supportLLaMA, Mistral, Mixtral, Qwen, etc.Same broad support
QuantizationAWQ, GPTQ, BitsAndBytesAWQ, GPTQ, BitsAndBytes
PagedAttention — virtual memory for the KV cache

The KV cache grows dynamically as tokens are generated, but GPU memory must be pre-allocated in contiguous blocks. Without PagedAttention: pre-allocate max_length × d per sequence — wasting memory on sequences that end early. With PagedAttention: allocate small KV "pages" on demand (like virtual memory in OS design). A 100-token response only uses memory for 100 tokens, not max_length. This reduces KV cache memory waste by ~60-80%, allowing 2-4× more concurrent sequences on the same GPU.

ServingQuestion 01 / 07
CineMatch's chatbot handles 50 concurrent conversations. Without continuous batching, how are these served?
AAll 50 are processed simultaneously in one batch.
BWithout continuous batching: static batching groups requests that arrive within a time window (e.g., 100ms). If 8 arrive together, they're batched. The remaining 42 wait in queue. Problem: generation lengths vary — a 20-token response finishes in 200ms, a 500-token response takes 5 seconds. With static batching, all 8 sequences must finish before the batch slot is freed. The 20-token response sits idle for 4.8 seconds waiting for the longest response. With continuous batching (vLLM): finished sequences immediately free their slot for new requests. GPU utilization stays near 100% instead of degrading as short sequences finish.
CEach conversation gets its own GPU.
DConversations are processed one at a time.
02 / 08
Quantization — trading precision for speed and memory.

Model weights are trained in BF16 (2 bytes per parameter). Quantization reduces precision to INT8 (1 byte) or INT4 (0.5 bytes), cutting memory and increasing throughput:

Quantization impact on LLaMA-3 8B
BF16 (baseline): 8B × 2 bytes = 16 GB. Fits on A10G (24 GB) with room for KV cache.Full precision. No quality loss. Standard training format.
INT8 (8-bit): 8B × 1 byte = 8 GB. Fits on any 16 GB GPU with ample KV cache room.Quality loss: ~0.1-0.3% on benchmarks. Negligible in practice. 2× memory savings.
INT4 (4-bit, GPTQ/AWQ): 8B × 0.5 bytes = 4 GB. Fits on consumer GPUs (RTX 3090, 4090).Quality loss: ~0.5-1.5% on benchmarks. Noticeable on hard reasoning tasks. Excellent for chatbots.
Speed impact: INT4 is ~1.5-2× faster than BF16 because: (1) smaller weights = less memory bandwidth needed (autoregressive generation is memory-bound), (2) more sequences fit in the same GPU = better batching.CineMatch uses INT4 (AWQ) for the chatbot. Quality is imperceptible in conversation; speed doubles.
QuantizationQuestion 02 / 07
CineMatch quantizes LLaMA-3 8B from BF16 (16 GB) to INT4 (4 GB). Which tasks are most affected by the quality loss?
AAll tasks are equally affected.
BOnly generation tasks — classification is unaffected.
COnly classification — generation is robust to quantization.
DHard reasoning and math are most affected; conversation and simple generation are least affected. Quantization introduces small errors in every weight. These errors accumulate through multi-step reasoning chains — each step amplifies the previous step's error. A 10-step math derivation accumulates 10 layers of quantization error. A casual conversation ("recommend a good thriller") rarely requires multi-step precise reasoning, so quantization errors don't compound. CineMatch's chatbot (casual recommendations) tolerates INT4 well. A math tutoring system would need BF16 or INT8.
03 / 08
Evaluation — benchmarks vs production quality.

Academic benchmarks (MMLU, HumanEval, HellaSwag) measure specific capabilities on standardized tests. Production quality is measured by user satisfaction, task completion rate, and business metrics. These often diverge:

A model that scores 5 MMLU points higher can be worse in production

MMLU measures knowledge recall. Production quality measures helpfulness, tone, safety, latency, and domain accuracy — none of which MMLU captures.

CineMatch's evaluation framework: (1) Automated metrics: latency p50/p95, error rate, cost per query. (2) Model-based evaluation: GPT-4-as-judge scores response quality on a 1-5 scale for helpfulness, accuracy, and safety. (3) Human evaluation: CineMatch's QA team rates 200 random responses/week on a rubric. (4) A/B tests: compare models on real user engagement (click-through, session length, return rate). The benchmark score is the least important signal.

EvaluationQuestion 03 / 07
CineMatch compares two chatbot models: Model A (MMLU 66, great user reviews) and Model B (MMLU 74, mediocre user reviews). Which should they deploy?
AModel B — higher benchmark score always wins.
BNeed more benchmark data to decide.
CModel A — user satisfaction is the ground truth. MMLU measures academic knowledge, not chatbot quality. Model A likely has: better tone/personality, more appropriate response length, better domain-specific knowledge (fine-tuned on CineMatch data), or better safety filtering. Model B might score higher on MMLU because it's larger or trained on more academic data — but that doesn't make it a better chatbot. Deploy Model A, use Model B's architecture as a starting point for fine-tuning, and re-evaluate.
DNeither — switch to GPT-4.
04 / 08
Monitoring — keeping production models healthy.
CineMatch's production monitoring dashboard
Latency: p50 (median response time), p95, p99. Alert if p95 > 2 seconds. Track per-model (DeBERTa ~5ms, LLaMA chatbot ~500ms, BART summarizer ~200ms).Latency spikes usually indicate: GPU memory pressure, batch queue backup, or model serving issues.
Throughput: queries/second per model. Alert if throughput drops below expected capacity. Track GPU utilization (target: 70-90%).Under 70%: over-provisioned (wasting money). Over 90%: at risk of latency spikes under load.
Quality drift: weekly GPT-4-as-judge evaluation on 200 random responses. Track average score over time. Alert if score drops >0.3 points in a week.Quality drift causes: data distribution shift (new movie genres, new slang), model degradation (shouldn't happen with frozen weights), or prompt template issues.
Cost: GPU cost per 1000 queries, broken down by model. Track monthly trends. Alert if cost/query increases >20%.Common cause: context length creep (users sending longer messages) increases KV cache and attention cost.
Safety: % of responses flagged by safety classifier. Track false positive rate (good responses flagged) and false negative rate (bad responses not flagged). Weekly audit of 50 flagged responses.Safety monitoring is continuous, not a one-time check. Distribution shifts can expose new failure modes.
MonitoringQuestion 04 / 07
CineMatch's chatbot quality score drops from 4.2/5 to 3.8/5 over 3 weeks. Model weights haven't changed. What's happening?
AData distribution shift. The model hasn't changed, but user behavior has. Possible causes: (1) New movie releases create queries about films the model wasn't fine-tuned on. (2) Seasonal trends — holiday period brings different query patterns. (3) User base growth — new users have different expectations than early adopters. (4) Prompt template degradation — upstream changes to the RAG pipeline or system prompt. Diagnosis: compare query distributions (topic, length, complexity) between the high-quality and low-quality periods. Fix: update RAG database with new movies, add examples of new query patterns to the fine-tuning set, or adjust system prompts.
BGPU hardware degradation.
CThe model is "forgetting" knowledge over time.
DThe evaluation rubric changed.

The complete CineMatch architecture — all 25 modules in one system.
CineMatch Integration — Final Architecture

The complete ML stack, powered by every architecture in this track.

Understanding layer (encoders, Phase 1): DeBERTa-v3-base for classification (sentiment, topic, helpfulness, spoiler detection). Sentence-BERT for embedding generation (review similarity, user-movie matching). DistilBERT for CPU-based content moderation. Cost: ~$200/month.

Generation layer (decoders, Phases 2-3): LLaMA-3 8B (INT4, vLLM) for the recommendation chatbot. Mistral 7B for real-time autocomplete. Qwen-2.5 7B for Hindi/Japanese market variants. Cost: ~$730/month per model instance.

Structured generation (encoder-decoders, Phase 4): BART-large-CNN for review summarization. FLAN-T5-base for multi-task extraction (entities, genres, structured metadata). Cost: ~$150/month.

Vision (Phase 6): CLIP for zero-shot poster classification and visual search. Swin-Tiny for scene analysis and object detection. LLaVA (ViT + LLaMA) for the multimodal chatbot. Cost: ~$500/month.

Batch processing: Mixtral 8×7B for high-quality overnight review summarization and analysis. Cost: ~$300/month (shared GPU, overnight only).

Total ML infrastructure cost: ~$2,600/month — serving millions of queries across classification, generation, summarization, search, and vision. Using GPT-4 API for everything: ~$40,000+/month.

ArchitectureQuestion 05 / 07
CineMatch uses 8+ different models. A new VP of Engineering says: "This is too complex. Consolidate to one model." How do you respond?
AAgree — one model is always simpler.
BDisagree — more models is always better.
CAgree, but only if the VP funds a GPT-4 API budget.
DPresent the trade-off with data. "One model (e.g., GPT-4 for everything) costs ~$40K/month vs our $2,600/month — a 15× cost increase. It also introduces: single-point-of-failure dependency on one API, higher latency for classification (5ms → 1500ms), and inability to process images without additional API calls. The multi-model stack is complex, but the complexity is managed (each model has clear ownership, monitoring, and deployment). Consolidation makes sense where models overlap — we could merge BART + FLAN-T5 into one FLAN-T5. But collapsing encoders + decoders + vision into one model is an economic and performance regression."

The architecture decision checklist.
How to choose a model for any new ML task
What's the task type? Understanding (classification, NER, similarity) → encoder-only. Generation (chat, creative, reasoning) → decoder-only. Structured transformation (summary, translation, extraction) → encoder-decoder or decoder-only.This determines the architecture family.
What's the volume? < 1K queries/day → API (GPT-4, Claude). 1K-100K → self-hosted on 1-2 GPUs. > 100K → self-hosted with batching and quantization. Cost scales linearly with volume — the crossover from API to self-hosted is typically at ~5K queries/day.This determines the deployment strategy.
What's the latency requirement? Real-time (< 100ms) → encoder-only or small decoder with INT4. Interactive (< 2s) → 7-8B decoder with streaming. Batch (minutes OK) → any model, maximize quality.This determines the model size and quantization.
What's the quality requirement? Must match GPT-4 → use GPT-4 API or fine-tune 70B+. Must beat GPT-3.5 → fine-tune 7-8B on domain data. Must be "good enough" for the task → smallest model that passes evaluation.This determines whether to fine-tune and how much data to collect.
What language(s)? English only → LLaMA-3. Multilingual → Qwen-2.5. Code → Qwen-Coder or CodeLLaMA.This determines the model family.
Does it need vision? Image classification → CLIP or ViT. Object detection → Swin. Image understanding + reasoning → multimodal LLM (LLaVA).This adds the vision component.
ChecklistQuestion 06 / 07
A CineMatch product manager asks: "We want to auto-generate 3-sentence movie descriptions from metadata (title, director, cast, genre, year). 10K movies, run once. Which model?"
ALLaMA-3 70B — highest quality generation.
BGPT-4 API. The task is: one-time batch of 10K structured-to-text generations. Volume: 10K total (not per day). Latency: doesn't matter (batch). Quality: must be publication-ready (visible to all users). Cost: 10K × 150 tokens × $15/M = $22.50 total. At $22.50 for a one-time job, the engineering effort of setting up self-hosted LLaMA is more expensive than the API cost. Use GPT-4 for one-time, low-volume, high-quality batch jobs. Self-host for recurring, high-volume tasks.
CFLAN-T5 — best for structured transformation.
DDeBERTa — it's the most efficient model.

Final synthesisQuestion 07 / 07
Across the entire 25-module track, what is the single most important lesson about transformer architectures?
AThe transformer is a general-purpose computation framework, not a specific model. The same attention mechanism processes text (BERT, GPT), images (ViT), audio, video, and multimodal inputs. The architectural innovations that matter most — RoPE, GQA, SwiGLU, RMSNorm — are improvements to this shared framework. The differences between models are mostly data (what they're trained on), training recipe (how they're trained), and alignment (what behavior they're optimized for). Architecture is the foundation; data, training, and deployment are where the real work happens.
BBigger models are always better.
CThe specific architecture matters more than anything else.
DOpen-source will replace closed-source models.
