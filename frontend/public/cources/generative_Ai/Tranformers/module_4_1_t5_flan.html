T5.
Every task is
text-to-text.

BERT encodes. GPT generates. T5 (Raffel et al., 2020) does both — an encoder-decoder that frames every NLP task as converting input text to output text. Classification? "classify: This movie was great" → "positive". Summarization? "summarize: [long review]" → "[short summary]". Translation? "translate English to Hindi: ..." → "..." The text-to-text framework unifies all tasks into one architecture and one training pipeline. FLAN-T5 (Chung et al., 2022) takes this further: instruction-tune on 1,800+ tasks, creating a model that can follow any text instruction zero-shot.

01 / 08
The text-to-text idea — one framework for everything.

Before T5, each NLP task had its own architecture: classification used [CLS] + linear head, NER used per-token classifiers, translation used encoder-decoder, summarization used copy mechanisms. T5's insight: if the output can be expressed as text, use a text generator.

TaskT5 InputT5 Output
Sentiment"classify: This CineMatch thriller was amazing""positive"
Summarization"summarize: [500-word review]""A gripping thriller with strong performances..."
Translation"translate English to Spanish: Great movie""Gran pelicula"
NER"extract entities: Directed by Christopher Nolan""person: Christopher Nolan"
Similarity"stsb sentence1: ... sentence2: ...""4.2" (similarity score as text)
QA"question: Who directed Inception? context: ...""Christopher Nolan"
Why text-to-text unification matters

One model. One loss function (cross-entropy on output tokens). One training pipeline. One inference API. Every task uses the same architecture, the same optimizer, the same serving infrastructure. This dramatically simplifies production: instead of maintaining separate models for classification, NER, summarization, and translation, you maintain one T5 model that handles all four. The task prefix ("classify:", "summarize:", "translate:") is the only thing that changes.

Text-to-textQuestion 01 / 07
T5 outputs the string "positive" for sentiment classification instead of a class probability. Doesn't this lose information?
AYes — text output is always worse than probability output.
BNo — T5 can output probabilities as text ("0.92").
CThe text output does lose the continuous probability, but the model's internal logits still contain this information. At inference, you can extract P("positive") vs P("negative") from the decoder's first-token logits — they're softmax probabilities over the vocabulary, just like any decoder. The text output format is for framework unification, not because the model lacks probability information. In practice, most applications need the label ("positive"), not the probability — and when you need the probability, it's accessible from the logits.
DT5 can't do classification — it's only for generation.
02 / 08
The encoder-decoder architecture — cross-attention revisited.

T5 uses the full encoder-decoder architecture from Module 0.2: an encoder stack (bidirectional self-attention, no mask) processes the input, then a decoder stack (causal self-attention + cross-attention to encoder outputs) generates the output.

T5 encoder-decoder. The encoder (cyan) processes the full input bidirectionally. The decoder (pink) generates output tokens left-to-right, with cross-attention (gold, animated) reading the encoder's output at each step. The encoder "understands" the input; the decoder "writes" the output.
T5-base vs BERT-base vs GPT-2 — parameter comparison
T5-base: 12 encoder layers + 12 decoder layers = 24 total layers. d=768, h=12220M params total — 2× BERT-base, because it has both an encoder and a decoder
Decoder has 3 sub-layers per block (self-attn + cross-attn + FFN) vs encoder's 2 (self-attn + FFN)Decoder block: ~10.6M params. Encoder block: ~7.1M. Decoder is ~50% larger due to cross-attention.
Encoder params: 12 × 7.1M = 85.1M. Decoder params: 12 × 10.6M = 127.2M. Embeddings: ~24M (shared).Total: 85.1 + 127.2 + 24 ≈ 236M. Often reported as 220M due to weight sharing and rounding.
Inference cost for summarization: encoder runs once on the full input (fast — parallel). Decoder runs T_output times (slow — sequential, one token at a time). For a 500-word input → 50-word summary: encoder = 1 pass, decoder = ~50 autoregressive steps.Encoder cost: O(T_input²). Decoder cost: O(T_output × T_input) for cross-attention + O(T_output²) for self-attention.
ArchitectureQuestion 02 / 07
Why does T5 use an encoder-decoder instead of a decoder-only (like GPT) for summarization?
AEncoder-decoders are always better than decoder-only models.
BThe encoder processes the input bidirectionally — every position sees the full input context. This matters for summarization: understanding "not terrible" requires seeing both words simultaneously (bidirectional), not just "not" when processing left-to-right (causal). The encoder builds a rich representation of the full input, which the decoder's cross-attention reads while generating the summary. A decoder-only model processes the input left-to-right with causal masking, producing less rich input representations.
CDecoder-only models can't generate summaries.
DEncoder-decoders are faster at generation.
03 / 08
Pretraining — span corruption.

T5's pretraining objective is span corruption: randomly mask contiguous spans of tokens (not individual tokens like BERT), then train the model to generate only the masked spans as output.

# Span corruption example
# Original: "The CineMatch thriller was absolutely incredible and worth watching"
# Masked:   "The CineMatch <extra_id_0> incredible and <extra_id_1>"
# Target:   "<extra_id_0> thriller was absolutely <extra_id_1> worth watching"

# Key differences from BERT's MLM:
# 1. Spans, not individual tokens (typically 3-token spans)
# 2. Output is only the masked content (compact target)
# 3. Sentinel tokens (<extra_id_0>) mark span boundaries
# 4. 15% of tokens are masked (same rate as BERT)
Span corruption efficiency vs BERT's MLM
BERT MLM: input = 512 tokens with [MASK]. Target = 512 tokens (predict at masked positions). But only ~77 positions produce loss (15%).Most of the decoder's capacity is wasted on copying unmasked tokens to the output.
T5 span corruption: input = ~435 tokens (unmasked) + sentinels. Target = ~77 tokens (only the masked spans).The decoder generates only the missing content — no wasted capacity on copying unmasked tokens.
Result: T5's decoder processes ~77 output tokens vs BERT's 512. The decoder runs 6.7× fewer steps.More efficient: the expensive autoregressive decoder only generates what's actually missing.
Span corruptionQuestion 03 / 07
T5 masks spans of ~3 tokens instead of individual tokens. Why spans rather than single tokens?
ASpans are easier to implement.
BSingle tokens would be too easy to predict.
CSpans reduce the vocabulary size.
DTwo reasons. First, spans are more efficient: masking 15% of tokens as ~5% spans means fewer sentinel tokens in the input and fewer decoder steps. Second, spans force the model to generate coherent multi-token phrases — not just fill single blanks. Predicting "thriller was absolutely" requires understanding multi-word relationships (adjective-intensifier patterns), which is more useful for downstream generation tasks than predicting individual words.
04 / 08
FLAN-T5 — instruction tuning at scale.

T5 is pretrained on span corruption — it can fill in blanks but doesn't follow instructions well. FLAN-T5 (Chung et al., 2022) fine-tunes T5 on 1,836 tasks formatted as natural language instructions, creating a model that can follow arbitrary text instructions zero-shot.

# FLAN-T5 can handle instructions it was never explicitly trained on
from transformers import T5ForConditionalGeneration, T5Tokenizer

model = T5ForConditionalGeneration.from_pretrained("google/flan-t5-large")
tok = T5Tokenizer.from_pretrained("google/flan-t5-large")

# CineMatch review analysis — zero-shot, no fine-tuning
prompts = [
    "Classify the sentiment of this review as positive or negative:\n"
    "'An absolute masterpiece of storytelling'",

    "Extract the movie genre from this review:\n"
    "'This sci-fi thriller kept me on the edge of my seat'",

    "Summarize this review in one sentence:\n"
    "'The acting was superb, especially the lead. The plot had some "
    "holes but the visual effects more than made up for it. Would watch again.'",
]

for p in prompts:
    ids = tok(p, return_tensors="pt").input_ids
    out = model.generate(ids, max_length=50)
    print(tok.decode(out[0], skip_special_tokens=True))
# "positive"
# "sci-fi thriller"
# "A visually stunning film with great acting despite some plot holes."
FLAN-T5 vs decoder-only instruction models

FLAN-T5 is excellent for structured output tasks, weaker for open-ended generation.

FLAN-T5's encoder-decoder architecture makes it ideal for tasks with clear input-output structure: classification, NER, extraction, translation, and constrained summarization. For open-ended generation (creative writing, long-form chat, brainstorming), decoder-only models (LLaMA, Mistral) are better — they can generate indefinitely without the encoder bottleneck. Use FLAN-T5 when the output is structured and bounded; use decoder-only when the output is free-form and unbounded.

FLANQuestion 04 / 07
FLAN-T5-XL (3B params) often outperforms GPT-3 (175B) on structured NLP tasks despite being 58× smaller. Why?
AThree advantages compound. (1) Bidirectional encoder: FLAN-T5 sees the full input context, while GPT-3 only sees left context. (2) Instruction tuning on 1,836 tasks: FLAN-T5 has seen diverse structured tasks during fine-tuning, while GPT-3 relies on few-shot prompting (in-context learning). (3) Encoder-decoder is structurally suited to input-output tasks — the architecture matches the task. GPT-3's advantage (massive parametric knowledge) is less relevant when the task is "classify this text" rather than "explain quantum physics."
BFLAN-T5 has better training data.
CThe benchmarks are biased toward encoder-decoder models.
DGPT-3 is too old to be competitive.

CineMatch Integration

FLAN-T5 powers CineMatch's structured NLP pipeline.

Review summarization: FLAN-T5-large (770M) generates 2-sentence summaries of CineMatch reviews for the browse UI. Input: full review (up to 512 tokens). Output: 30-50 token summary. Latency: ~200ms on A10G. Quality: comparable to GPT-3.5's summaries at 1/230th the parameter count.

Structured extraction: FLAN-T5-base (220M) extracts structured data from reviews: "Extract: actors mentioned, genres, sentiment, spoiler_present" → JSON-like output. This feeds CineMatch's metadata enrichment pipeline, running batch inference on 50K new reviews nightly.

Multi-task single model: One FLAN-T5-base instance handles classification (sentiment, topic, helpfulness), extraction (entities, genres), and summarization (short summaries). Three tasks, one model, one serving endpoint. The task prefix in the input selects the behavior.

DeploymentQuestion 05 / 07
CineMatch currently uses DeBERTa for classification AND FLAN-T5 for summarization — two separate models. A colleague suggests: "Just use FLAN-T5 for both." Is this a good idea?
AYes — one model is always simpler.
BNo — T5 can't do classification.
CYes — FLAN-T5 is better at classification than DeBERTa.
DTrade-off. FLAN-T5 can do both, but DeBERTa classifies at ~5ms/query (encoder-only, no generation) vs FLAN-T5 at ~50ms/query (must generate the label token by token). For 100K classification queries/day, FLAN-T5 adds $15/day in compute vs DeBERTa's $1.50. If the operational simplicity of one model (one deployment, one monitoring, one update cycle) is worth $13.50/day to CineMatch — do it. If not, keep the specialized stack. There's no universally right answer.
06 / 08
Relative position bias — T5's position encoding.

T5 doesn't use absolute position embeddings (BERT) or RoPE (LLaMA). It uses relative position bias: a learned scalar added to each attention score based on the distance between positions. The bias is bucketed — distances 0-7 get individual biases, then distances 8-128 are grouped logarithmically into fewer buckets.

Aij = (Qi · Kj) / √dk + b(i − j) (6.1)

where b(i − j) is a learned bias that depends only on the relative distance. T5 uses 32 buckets for relative positions, shared across all heads in a layer but different across layers. This is similar to DeBERTa's position-to-content term but simpler — just a scalar bias, not a full vector interaction.

T5's position bias doesn't extrapolate as well as RoPE

T5's bucketed position bias struggles with sequences longer than training length.

The 32 position buckets cover distances up to the training sequence length (512 for T5-base). Beyond this, all distances map to the same bucket — the model can't distinguish position 600 from position 1000. RoPE (LLaMA, Mistral) handles this more gracefully because rotation angles can be extended continuously. For CineMatch's long reviews (sometimes 1000+ tokens), this is a practical limitation of T5 — long inputs must be truncated or chunked.

PositionQuestion 06 / 07
T5 uses 32 position buckets while DeBERTa uses a 512-entry relative position table. Why might fewer buckets be fine?
APosition doesn't matter for T5's tasks.
B32 buckets are faster to compute.
CLogarithmic bucketing captures the key insight: the difference between distance 1 and distance 2 matters much more than the difference between distance 100 and distance 101. By grouping distant positions into coarse buckets, the model uses its 32 parameters on fine-grained local distinctions (distance 1 vs 2 vs 3 — crucial for syntax) and coarse global distinctions (nearby vs far — enough for discourse). This is a form of inductive bias: most useful information is local.
DT5 never processes sequences longer than 32 tokens.
07 / 08
T5's legacy and the encoder-decoder question in 2026.
Paper breakdown

"Exploring the Limits of Transfer Learning with a Unified Text-to-Text Transformer" (Raffel et al., 2020)

The T5 paper is less a model paper and more a systematic study. It tested dozens of design decisions — architecture (encoder-decoder vs decoder-only vs prefix LM), pretraining objective (MLM vs span corruption vs language modeling), dataset (C4 vs Wikipedia), model size (60M to 11B) — in a controlled framework. The paper's lasting contribution is its methodology: by varying one factor at a time on a common evaluation, it established which design choices actually matter. Key finding: encoder-decoder with span corruption on a large, cleaned web corpus (C4) is the strongest combination for structured NLP tasks.

LegacyQuestion 07 / 07
In 2026, most new models are decoder-only (LLaMA, Mistral, Qwen). Is the encoder-decoder architecture dead?
AYes — decoder-only models can do everything encoder-decoders can.
BNo — encoder-decoders still win for specific production tasks. For constrained generation (summarization, translation, structured extraction), encoder-decoder models produce higher quality at smaller parameter counts because the bidirectional encoder captures the full input context. FLAN-T5-large (770M) matches GPT-3.5 on summarization. Decoder-only models need 10-100× more parameters to match encoder-decoder quality on structured tasks. The trend toward decoder-only is about generality and scale, not about encoder-decoder being worse at its niche.
CThe architectures are equivalent — no difference in practice.
DEncoder-decoders are better in every way — decoder-only is a fad.
