Mistral & Mixtral.
The efficiency frontier.

Mistral AI, a Paris-based startup, released Mistral 7B in September 2023 — a 7B model that outperformed LLaMA-2 13B on every benchmark. Three months later, they released Mixtral 8×7B — a Mixture of Experts model with 46.7B total parameters but only 12.9B active per token. The lesson: architecture efficiency matters as much as scale. Sliding window attention for long context, MoE for parameter efficiency, and aggressive optimization made Mistral the quality-per-FLOP champion. For CineMatch, Mistral 7B is the fastest high-quality option for real-time serving.

Mistral 7B outperforms LLaMA-2 13B — a model nearly 2× its size — on every benchmark. How? Better data curation (quality over quantity), sliding window attention (longer effective context at lower cost), GQA (8 KV heads), and RoPE. The architecture is the same modern decoder block from Module 3.1 — the gains come from training quality and one architectural innovation: sliding window attention.

Sliding window attention (SWA).

Standard causal attention at position t attends to all positions 0..t. At t=8000, that's 8000 dot products. Sliding window attention limits each position to the most recent W positions (typically W=4096):

SWA: position t attends to positions max(0, t−W)..t (1.1)

But this doesn't mean the model only "sees" W tokens of context. Through L stacked layers, information propagates across windows: layer 1 position t sees t−W..t, layer 2 sees t−2W..t (via the representations from layer 1), and so on. The effective receptive field is L × W:

Sliding window: local attention, global receptive field
Mistral 7B: W=4096, L=32 layers. Effective receptive field: 32 × 4096 = 131,072 tokensFar exceeding the stated 32K context window. Information from token 0 reaches token 32K through 8 layers of window propagation.
Memory savings: at position t=8000, standard attention stores 8000 KV pairs. SWA stores only 4096.KV cache is capped at W entries per layer, not T. Fixed memory regardless of sequence length.
Compute savings: standard O(T²) per layer. SWA: O(T × W) per layer — linear in T when W is fixed.At T=32K, W=4K: standard = 32K² = 1B ops. SWA = 32K × 4K = 128M ops. 8× cheaper per layer.
SWAQuestion 01 / 07
Mistral 7B uses sliding window attention with W=4096 and 32 layers. A token at position 0 needs to influence the prediction at position 20,000. Can it?
ANo — SWA limits context to 4096 tokens.
BOnly if special long-range attention heads exist.
CYes, through multi-layer propagation. In layer 1, token 0's information reaches token 4096. In layer 2, that information (now at token 4096) reaches token 8192. After 5 layers, the information reaches position 20,000. The effective receptive field is L × W = 131K tokens. The information is attenuated (each hop loses some signal), but it's not blocked. SWA trades some long-range precision for massive compute savings.
DYes, but only through the FFN layers, not attention.

Mixtral — Mixture of Experts at the open-source frontier.

Mixtral 8×7B replaces each dense FFN with 8 parallel expert FFNs and a learned router that selects 2 of 8 for each token:

Mixtral MoE routing. Each token passes through a learned router that selects top-2 experts (green ✓). The other 6 experts are inactive. The selected experts' outputs are weighted by the router scores and summed. Same total knowledge capacity as 8 dense FFNs, but only 2/8 = 25% of FFN compute per token.
Mixtral 8×7B parameter math — total vs active
Shared (MHA + embeddings + norms): ~5.0B params. These are always active.GQA attention with 32 Q heads, 8 KV heads. Same as Mistral 7B.
Per expert FFN: ~5.2B / 8 experts × 8 = 5.2B params. Total across 8 experts per layer: ~41.7BEach expert is a standard SwiGLU FFN: W₁, W_gate, W₂ with d_ff ≈ 14,336
Router: 32 layers × d × 8 = negligible (< 1M params total)The router is tiny — just a linear layer mapping d → 8 scores per token
Total: ~46.7B. Active per token: 5.0B (shared) + 2/8 × 41.7B (2 experts) = 5.0B + 10.4B = ~12.9B activeQuality of a 46.7B model, inference cost of a 12.9B model. Best of both worlds.
MoE mathQuestion 02 / 07
Mixtral has 46.7B total params but only 12.9B active. Does it use more or less GPU memory than a dense 13B model?
ALess — only active params need to be loaded.

Memory = total params × bytes. Mixtral at BF16: 46.7B × 2 = 93.4 GB. A dense 13B model: 13B × 2 = 26 GB. Mixtral needs 3.6× more memory despite similar inference speed. This is MoE's main drawback: capacity is cheap, memory is not.

The router — how tokens choose experts.

The router is a simple linear layer followed by softmax over the 8 expert scores. For each token, it picks the top-2 scores and routes the token to those experts:

g(x) = softmax(x · Wrouter)    select top-k indices    output = Σi∈top-k gi(x) · Ei(x) (3.1) 
class MoEFFN(nn.Module):
    """Mixture of Experts FFN layer (Mixtral-style)."""
    def __init__(self, d=4096, d_ff=14336, n_experts=8, top_k=2):
        super().__init__()
        self.router = nn.Linear(d, n_experts, bias=False)
        self.experts = nn.ModuleList([
            SwiGLUFFN(d, d_ff) for _ in range(n_experts)
        ])
        self.top_k = top_k

    def forward(self, x):
        # x: (B, T, d)
        scores = self.router(x)                    # (B, T, n_experts)
        weights, indices = scores.topk(self.top_k)  # top-k expert indices
        weights = F.softmax(weights, dim=-1)        # normalize top-k weights

        # Compute selected experts only
        out = torch.zeros_like(x)
        for i, expert in enumerate(self.experts):
            mask = (indices == i).any(dim=-1)       # which tokens chose this expert
            if mask.any():
                expert_out = expert(x[mask])
                # Weight by router score and accumulate
                w = weights[indices == i]
                out[mask] += w.unsqueeze(-1) * expert_out
        return out
The load balancing problem

Without a balancing loss, the router collapses — sending all tokens to 1-2 "favorite" experts.

Left to its own devices, the router learns that one expert is slightly better and starts sending more tokens to it. More tokens → more gradient updates → the expert improves further → even more tokens routed to it. This positive feedback loop leads to expert collapse: one expert handles 90% of tokens, the other 7 sit idle. The fix: add an auxiliary load balancing loss that penalizes uneven expert utilization. Mixtral uses Lbalance = α · Σi fi · Pi, where fi is the fraction of tokens routed to expert i and Pi is the average router probability for expert i. α = 0.01 is typical.

RouterQuestion 03 / 07
Do Mixtral's experts specialize in different topics (e.g., expert 1 = math, expert 3 = code)?
AYes — each expert handles a different domain.
BNo — expert assignment is random.
CYes, but only in the first few layers.
DPartially. Analysis shows experts develop soft specializations — not clean topic boundaries. Expert 2 might handle 30% of code tokens but also 15% of math tokens. The specialization is more syntactic than semantic: certain experts prefer punctuation-heavy tokens, others prefer long content words, others prefer function words. This means routing is not "topic classifier → specialist" but rather "adaptive feature selector → diverse processing paths." The diversity helps quality even without clean specialization.

CineMatch Integration

Mistral 7B for CineMatch's real-time features.

Why Mistral 7B over LLaMA-3 8B for real-time: Mistral 7B has 10% fewer active parameters → ~10% lower latency per token. For CineMatch's real-time autocomplete (suggest review completions as the user types), every millisecond matters. Mistral 7B generates the first token in ~30ms vs LLaMA-3 8B's ~33ms. At 1000 keystrokes/second across all users, the 3ms saves 3 seconds of GPU time per second — a measurable cost reduction.

When to use Mixtral instead: CineMatch's batch summarization pipeline (summarize 1000 new reviews overnight). Quality matters more than latency. Mixtral 8×7B produces better summaries (MMLU 70.6 vs 60.1) and runs fine on the 2× A100 already allocated for batch jobs. The 93 GB memory footprint is manageable in batch mode.

05 / 08
MoE deployment challenges.

MoE models have unique serving challenges beyond the memory footprint:

1. Expert parallelism. In a batch, different tokens route to different experts. If 80% of tokens in a batch go to expert 1 and 20% to expert 5, the GPUs hosting experts 2-4 and 6-8 sit idle. Load imbalance wastes hardware. Solutions: expert parallelism (spread experts across GPUs), capacity factors (limit how many tokens each expert processes per batch), and the load balancing loss (train the router to distribute evenly).

2. Communication overhead. When experts are on different GPUs, tokens must be sent to the right GPU (all-to-all communication). This communication can dominate inference time if the interconnect is slow. Mixtral on a single machine with NVLink is fine; Mixtral across machines with ethernet is problematic.

MoE quantization is harder

Quantizing MoE models is less effective than quantizing dense models.

In a dense model, all weights are used every forward pass — quantization errors are consistent and predictable. In MoE, different experts are activated for different tokens. Quantization errors vary per expert, creating inconsistent quality across routing paths. Some experts tolerate INT4 quantization well; others degrade badly. The fix: per-expert calibration (quantize each expert separately on a representative token distribution) or selective quantization (quantize the 6 inactive experts more aggressively than the 2 active ones, since the router already down-weights them).

DeploymentQuestion 05 / 07

With 2 experts per GPU: 46.7B / 4 ≈ 11.7B params per GPU → ~23.4 GB per GPU in BF16. Fits on 4× A10G (24 GB each).

When dense beats MoE — the decision framework.
Dense vs MoE — when to use which
Use dense (Mistral 7B, LLaMA-3 8B) when: single-GPU deployment, latency-critical, limited memory, simple serving infrastructureExample: CineMatch real-time chatbot on 1× A10G. Dense 7B fits, serves fast, no cross-GPU communication.
Use MoE (Mixtral 8×7B) when: multi-GPU available, quality matters more than latency, batch processing is acceptableExample: CineMatch batch summarization on 2× A100. Mixtral's 70.6 MMLU vs Mistral's 60.1 justifies the memory cost.
The crossover point: MoE wins when you need quality > dense-13B but can't afford dense-70B inference cost. Mixtral 8×7B: 70.6 MMLU at 12.9B active compute. LLaMA-3 70B: 79.5 MMLU at 70B compute. If you need 70+ MMLU at <15B compute → MoE is the only option.MoE fills the gap between "7B is too weak" and "70B is too expensive."
