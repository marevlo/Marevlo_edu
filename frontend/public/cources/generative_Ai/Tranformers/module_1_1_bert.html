BERT.
Bidirectional
representations from transformers.

GPT reads left-to-right. BERT reads in both directions simultaneously. By masking random tokens and predicting them from full surrounding context, BERT learns representations where every position encodes the entire sentence — not just the left half. This bidirectional encoding made BERT the dominant model for classification, search, NER, and embeddings from 2018 to 2023, and its architecture remains the backbone of every production encoder system in 2026. This module builds BERT from the paper up: tokenization, architecture, pretraining objectives, fine-tuning, and a complete CineMatch review sentiment classifier.

The input pipeline — how text becomes tensors.

Before any attention can happen, raw text must become a sequence of integers. BERT uses WordPiece tokenization — a subword algorithm that splits rare words into known pieces while keeping common words intact.

# What BERT's tokenizer actually does
from transformers import BertTokenizer

tok = BertTokenizer.from_pretrained('bert-base-uncased')

text = "CineMatch recommends unwatchable movies"
tokens = tok.tokenize(text)
# ['cine', '##match', 'recommend', '##s', 'un', '##watch', '##able', 'movies']

ids = tok.encode(text, add_special_tokens=True)
# [101, 25765, 15318, 16755, 2015, 4895, 6360, 3085, 5765, 102]
#  [CLS]  cine  ##match recommend ##s  un  ##watch ##able movies [SEP]
Tokenization step-by-step — what happens to each word
"CineMatch" → not in BERT's 30,522-word vocabulary → split: "cine" + "##match"The ## prefix means "continuation of the previous word" — a WordPiece convention
"recommends" → split: "recommend" + "##s" (common stem + inflection)WordPiece learns that splitting at morpheme boundaries helps — "recommend" appears in many contexts
"unwatchable" → split: "un" + "##watch" + "##able" (prefix + root + suffix)3 subwords from 1 word. The model can learn that "un##" means negation, "##able" means capability — transferable across all words with these morphemes
"movies" → in vocabulary as-is → kept wholeCommon words stay intact. Rare words decompose. Vocabulary: 30,522 tokens covers any English text.
Special tokens added: [CLS] at position 0, [SEP] at end. Total: 10 tokens from 4 words.[CLS] = "classify here" — its output vector is used for sentence-level tasks. [SEP] = "sentence boundary"

Three embedding layers, summed.

Each input position receives the sum of three embeddings:

Einput(i) = Etoken(idi) + Eposition(i) + Esegment(segi) (1.1)

Token embedding: learned lookup table, 30,522 × 768. Maps each WordPiece ID to a 768-dim vector. Position embedding: learned, 512 × 768. Encodes absolute position (0 to 511). Segment embedding: learned, 2 × 768. Tells the model which sentence a token belongs to (sentence A = 0, sentence B = 1) — needed for the NSP objective.

BERT-base embedding parameter count
Token embeddings: 30,522 × 768 = 23,440,896This is 21% of BERT-base's total 110M parameters — the embedding table is large
Position embeddings: 512 × 768 = 393,216Max sequence length = 512 tokens. Cannot process longer inputs (unlike RoPE in later models)
Segment embeddings: 2 × 768 = 1,536Tiny — just two vectors (sentence A and sentence B)
LayerNorm after sum: 768 × 2 = 1,536 (scale + bias)Total embedding params: 23.84M out of 110M total (21.7%)
TokenizationQuestion 01 / 07
BERT tokenizes "unforgettable" as ["un", "##forget", "##table"]. Why is this better than character-level or word-level tokenization?
AIt produces fewer total parameters.
BSubword tokenization balances vocabulary size and coverage. Character-level would need ~300 tokens for "unforgettable" (one per char) — too many positions, too little semantic density. Word-level would need "unforgettable" in the vocabulary — and every rare word would be [UNK]. Subword splits into meaningful morphemes ("un" = negation, "forget" = root, "##table"/"##able" = capability) that transfer across words, with a fixed 30K vocabulary that covers any English text.
CIt makes the model faster at inference.
DBERT requires exactly 3 subwords per input word.

The BERT architecture — an encoder-only transformer.

BERT is a stack of transformer encoder blocks — exactly the architecture from DL Track Module 4.2. No decoder, no causal mask. Every position attends to every other position bidirectionally.

# Build BERT-base from scratch (no HuggingFace)
import torch, torch.nn as nn, math

class BertEmbedding(nn.Module):
    def __init__(self, vocab=30522, max_len=512, d=768, n_seg=2, drop=0.1):
        super().__init__()
        self.tok = nn.Embedding(vocab, d)
        self.pos = nn.Embedding(max_len, d)
        self.seg = nn.Embedding(n_seg, d)
        self.ln = nn.LayerNorm(d)
        self.drop = nn.Dropout(drop)

    def forward(self, input_ids, segment_ids):
        B, T = input_ids.shape
        positions = torch.arange(T, device=input_ids.device).unsqueeze(0)
        x = self.tok(input_ids) + self.pos(positions) + self.seg(segment_ids)
        return self.drop(self.ln(x))

class BertEncoderBlock(nn.Module):
    def __init__(self, d=768, h=12, d_ff=3072, drop=0.1):
        super().__init__()
        self.attn = nn.MultiheadAttention(d, h, dropout=drop, batch_first=True)
        self.ffn = nn.Sequential(
            nn.Linear(d, d_ff), nn.GELU(), nn.Dropout(drop),
            nn.Linear(d_ff, d), nn.Dropout(drop),
        )
        self.ln1 = nn.LayerNorm(d)
        self.ln2 = nn.LayerNorm(d)

    def forward(self, x, mask=None):
        # Post-norm (original BERT paper style)
        attn_out, _ = self.attn(x, x, x, key_padding_mask=mask)
        x = self.ln1(x + attn_out)
        x = self.ln2(x + self.ffn(x))
        return x

class BertModel(nn.Module):
    def __init__(self, vocab=30522, d=768, L=12, h=12, d_ff=3072):
        super().__init__()
        self.embed = BertEmbedding(vocab, d=d)
        self.blocks = nn.ModuleList([BertEncoderBlock(d, h, d_ff) for _ in range(L)])
        self.pooler = nn.Sequential(nn.Linear(d, d), nn.Tanh())

    def forward(self, input_ids, segment_ids, padding_mask=None):
        x = self.embed(input_ids, segment_ids)
        for block in self.blocks:
            x = block(x, mask=padding_mask)
        cls_vec = self.pooler(x[:, 0])  # [CLS] token output
        return x, cls_vec  # (B,T,d), (B,d)
ArchitectureQuestion 02 / 07
BERT uses 12 heads with dmodel=768. What is dk (the per-head dimension)?
A768 — each head sees the full dimension.
B12 — one dimension per head.
C64. dk = dmodel / nheads = 768 / 12 = 64. Each head operates on a 64-dim subspace. The 12 heads run in parallel on different 64-dim slices, then their outputs are concatenated back to 768-dim. No extra parameters — multi-head is a reshape, not a multiplication.
D9216 — dmodel × nheads.
03 / 09
Pretraining objective 1 — Masked Language Modeling (MLM).

BERT's core pretraining task: randomly mask 15% of input tokens, then predict them from the surrounding bidirectional context. The loss is cross-entropy at the masked positions only.

LMLM = −Σi ∈ M log P(xi | x\M)   where M = set of masked positions (3.1)

The 80/10/10 masking protocol.

Of the 15% selected positions: 80% get replaced with [MASK], 10% with a random token, 10% stay unchanged. This prevents a train/inference mismatch — at inference, [MASK] never appears, so the model must learn good representations even at positions that aren't masked.

Worked example — MLM on a CineMatch review
Input: "[CLS] this movie was absolutely terrible and boring [SEP]"10 tokens. 15% → mask ~1.5 → round to 2 tokens
Selected positions: 4 ("absolutely") and 7 ("boring"). 80/10/10 protocol:
Position 4: 80% chance → [MASK]. Masked input: "[CLS] this movie was [MASK] terrible and boring [SEP]"
Position 7: 10% random chance → replaced with "happy". Input: "[CLS] this movie was [MASK] terrible and happy [SEP]"The model must still predict "boring" at position 7 despite seeing "happy" — forces robust representations
Forward pass through 12 encoder blocks. At position 4: h₄ ∈ ℝ^{768} encodes "what word fits here?"h₄ incorporates bidirectional context: "this movie was ___ terrible" → probably an intensifier
Prediction head: logits = W_head · h₄ + b → ℝ^{30522}. softmax → P("absolutely") = 0.12, P("really") = 0.08, P("very") = 0.15Loss at position 4: −log(0.12) = 2.12. Loss at position 7: −log(P("boring")) = ...
Total MLM loss for this example = average of losses at the 2 masked positions.Only 2 of 10 positions contribute to the loss. The other 8 get no gradient. This is why MLM needs more data than autoregressive pretraining.

The MLM gradient — what the model learns.

∂LMLM/∂hi = WheadT · (p̂i − ex_i) (3.2)

The gradient at masked position i pushes the representation hi to make the correct token more probable. Since hi depends on all other positions via bidirectional attention, this gradient propagates through the entire sequence — every unmasked token's representation is updated to better support prediction of the masked tokens. This is how BERT learns contextual representations: by being forced to reconstruct missing pieces from the whole.

Why 15% and not 50%?

Masking rate is a precision-efficiency trade-off.

At 50% masking, the remaining context is too sparse — predictions become nearly random, gradients are noisy, and learned representations degrade. At 5%, training is too slow — only 5% of positions produce learning signal per sequence. 15% is the empirically validated sweet spot: enough context for accurate predictions, enough masked positions for efficient learning. SpanBERT later showed 15-20% is optimal; T5 explored up to 50% with diminishing returns.

In BERT's MLM, why are 10% of masked positions replaced with random tokens instead of [MASK]?
ATo increase the vocabulary coverage during pretraining.
BTo make training faster by reducing the number of [MASK] tokens.
CTo augment the data with more diverse examples.
DTo prevent the model from learning that [MASK] is a signal to predict. If 100% of masked positions used [MASK], the model could learn a shortcut: "only produce good representations at [MASK] positions." With 10% random and 10% unchanged, the model can't tell which positions are masked — it must produce good representations everywhere, which is what fine-tuning needs (since [MASK] never appears at inference).
04 / 09
Pretraining objective 2 — Next Sentence Prediction (NSP).

BERT's second pretraining task: given two sentences A and B, predict whether B actually follows A in the corpus (positive) or is a random sentence (negative). 50/50 split. The [CLS] token's output feeds a binary classifier.

LNSP = −[y · log p + (1−y) · log(1−p)]   where p = σ(Wnsp · h[CLS]) (4.1)

NSP was designed to help with tasks requiring sentence-pair understanding (natural language inference, question answering). However, later work (RoBERTa, 2019) showed that NSP actually hurts performance — it's too easy a task that doesn't teach useful representations. This is why RoBERTa (Module 1.2) removes it entirely.

Total BERT pretraining loss

Ltotal = LMLM + LNSP. Both losses are averaged and backpropagated through the same encoder. The model learns to simultaneously reconstruct masked tokens and classify sentence pairs. In practice, MLM does most of the work; NSP contributes little.

NSPQuestion 04 / 07
RoBERTa (2019) removed NSP from BERT's pretraining and saw improved performance. Why might NSP hurt?
ANSP is too easy — random negative pairs are trivially distinguishable by topic mismatch alone, so the model learns topic detection rather than inter-sentence reasoning. The signal is misleading: the model spends capacity on a shortcut that doesn't transfer to downstream tasks requiring actual linguistic understanding.
BNSP requires too much compute.
CNSP makes the model too large.
DNSP only works with English text.

Fine-tuning BERT — from representations to task predictions.

After pretraining on unlabeled text, BERT is fine-tuned on labeled data for specific tasks. The pattern: add a task-specific head on top of the pretrained encoder, then train the entire model end-to-end with a small learning rate.

Task typeInputUse which output?HeadCineMatch example
Sentence classificationSingle sentence[CLS] vector (768-d)Linear(768, n_classes)Review sentiment (pos/neg)
Sentence-pairTwo sentences with [SEP][CLS] vectorLinear(768, n_classes)Review similarity
Token classificationSingle sentenceEach token's vectorLinear(768, n_tags) per tokenMovie entity extraction (NER)
Extractive QAQuestion + ContextEach context tokenLinear(768, 2) for start/end"What genre is this movie?"

Build It — CineMatch review sentiment classifier.

from transformers import BertModel
import torch.nn as nn

class CineMatchSentiment(nn.Module):
    """Fine-tune BERT for CineMatch review sentiment (positive / negative)."""
    def __init__(self, n_classes=2, dropout=0.1):
        super().__init__()
        self.bert = BertModel.from_pretrained('bert-base-uncased')
        self.head = nn.Sequential(
            nn.Dropout(dropout),
            nn.Linear(768, n_classes),
        )

    def forward(self, input_ids, attention_mask, token_type_ids):
        outputs = self.bert(input_ids, attention_mask=attention_mask,
                           token_type_ids=token_type_ids)
        cls_vec = outputs.pooler_output      # (B, 768)
        logits = self.head(cls_vec)          # (B, 2)
        return logits

# Fine-tuning recipe
model = CineMatchSentiment()
opt = torch.optim.AdamW([
    {'params': model.bert.parameters(), 'lr': 2e-5},   # pretrained: tiny LR
    {'params': model.head.parameters(), 'lr': 1e-4},   # new head: 5× higher
], weight_decay=0.01)

# Train for 3-5 epochs on labeled CineMatch reviews
# Expected: ~92% accuracy on 10K reviews with just BERT-base + linear head
The #1 BERT fine-tuning mistake

Using the same learning rate for the pretrained backbone and the new head.

BERT's pretrained weights encode millions of examples worth of language knowledge. A learning rate of 1e-3 (appropriate for the randomly-initialized head) will destroy those representations in one epoch. The fix: differential LR — 2e-5 for the backbone, 1e-4 for the head. With warmup: linearly increase from 0 to target LR over the first 10% of steps. If val accuracy drops after epoch 1 (instead of improving), your backbone LR is too high. This is the exact same catastrophic forgetting pattern from DL Track Module 4.4.

You fine-tune BERT on 500 CineMatch reviews. Training accuracy reaches 98% but val accuracy is 71%. What's happening?

Fixes: freeze most of the backbone (only fine-tune the last 2-3 layers + head), increase dropout on the head to 0.3-0.5, use early stopping, or — best — collect more labeled data. With 500 examples, a frozen-backbone + linear head approach often outperforms full fine-tuning.

06 / 09
BERT as an embedding model — sentence and document representations.

Beyond classification, BERT produces dense embeddings that power semantic search, reranking, and similarity matching. The simplest approach: take the [CLS] token's output as the sentence embedding. Better: average all token outputs (mean pooling). Best: train with a contrastive loss (Sentence-BERT, 2019) — but that's Module 1.2's territory.

CineMatch Integration

BERT powers CineMatch's semantic search.

Review embeddings. Each CineMatch review is encoded by BERT into a 768-dim vector (mean-pooled). Stored in a vector database (FAISS or Pinecone). When a user types "movies with surprising endings," the query is encoded by the same BERT → nearest-neighbor search finds reviews mentioning surprises, twists, plot turns — even if the exact word "surprising" doesn't appear.

Movie description embeddings. Each movie's synopsis is BERT-encoded. Similar movies cluster in embedding space: "psychological thriller about a con artist" lands near other con-artist thrillers, not near comedies. This feeds the ranking stage as a 768-dim feature per candidate.

Review reranking. For each candidate movie, CineMatch finds the user's past reviews most similar (by cosine) to the candidate's description embedding. If a user wrote a glowing review of a heist film, and a new heist film's description embedding is close to that review's embedding, the candidate gets a boost.

EmbeddingsQuestion 06 / 07
You use BERT's [CLS] token output as the sentence embedding for CineMatch review similarity. Two reviews about the same movie have cosine similarity 0.45. Is this good or bad?
AGood — 0.45 is a high cosine similarity.
BBad — the reviews should have cosine similarity > 0.95.
CExpected but not ideal. BERT's [CLS] output wasn't trained for sentence similarity — it was trained for NSP classification. Cosine similarity from raw [CLS] vectors is poorly calibrated. The fix: mean-pool all token outputs (typically gives cosine ~0.6-0.8 for similar sentences), or fine-tune with a contrastive loss (Sentence-BERT) for proper similarity structure. Raw [CLS] embeddings are a common mistake — they work okay for classification but poorly for similarity.
DImpossible to assess without knowing the vocabulary size.

SynthesisQuestion 07 / 07
A CineMatch engineer says: "BERT is outdated — we should use GPT-4 for everything instead." How do you respond?
AThey're right — GPT-4 is strictly better than BERT at all tasks.
BThey're wrong — BERT is better than GPT-4 at everything.
CIt depends on the budget — BERT is cheaper but GPT-4 is always more accurate.
DDifferent tools for different jobs. For classification, NER, and embeddings, a fine-tuned BERT (or DeBERTa) is faster, cheaper, and often more accurate than GPT-4 because it's specialized via fine-tuning. For generation, summarization, and few-shot reasoning, GPT-4 wins. In CineMatch's production stack: BERT-family models handle embedding generation (~1ms/query) and review classification (~5ms/query), while a large decoder handles conversational recommendation (~500ms/query). Using GPT-4 for classification would be 100× slower and 50× more expensive for no accuracy gain.

08 / 09
The paper — key decisions and their impact.
Paper breakdown

"BERT: Pre-training of Deep Bidirectional Transformers for Language Understanding" (Devlin et al., October 2018)

BERT's key insight was simple: use the transformer encoder (already well-understood from Vaswani et al., 2017) with a masked-token prediction objective to create bidirectional representations. GPT (June 2018) had shown that pretraining works, but was limited to left-to-right context. BERT showed that bidirectional context is strictly better for understanding tasks, setting new state-of-the-art on 11 NLP benchmarks simultaneously.

What the paper got right: MLM as a pretraining objective (now used in nearly every encoder model). Fine-tuning with a small task head (simple, effective, and universal). The [CLS] pooling mechanism (still used in production).

What later work improved: NSP was dropped (RoBERTa). Static masking was replaced with dynamic masking (RoBERTa). Absolute position embeddings were replaced with relative/disentangled (DeBERTa). The model was made efficient (DistilBERT, ALBERT). And the replaced-token-detection objective of ELECTRA proved more sample-efficient than MLM.
