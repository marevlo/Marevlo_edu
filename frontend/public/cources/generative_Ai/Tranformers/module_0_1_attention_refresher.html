The Attention
mechanism. One operation to rule them all.

Before BERT, GPT, LLaMA, or any model in this track, there is one operation: scaled dot-product attention. It takes a query, matches it against a set of keys, and returns a weighted sum of values. That's the entire mechanism — and every architecture in this track is a variation on how to stack it, mask it, and scale it. This module refreshes the math from the DL track, adds the notation we'll use throughout, and grounds it in CineMatch's semantic search engine.

The single equation you need.

Every transformer — BERT, GPT-4, LLaMA, Mistral, ViT — is built from one operation:

Attention(Q, K, V) = softmax(QKT / √dk) · V (1.1)

Q (Query): "What am I looking for?" — shape (Tq, dk).
K (Key): "What do I contain?" — shape (Tk, dk).
V (Value): "What information do I carry?" — shape (Tk, dv).
Output: weighted mix of values — shape (Tq, dv).

Four steps, always the same: (1) compute similarity scores QKT, (2) scale by √dk, (3) normalize via softmax, (4) weight-sum the values. Every architecture in this track modifies what goes into Q, K, V and what mask is applied — never the core operation itself.

Scaled dot-product attention. Q×KT computes similarity, ÷√dk normalizes variance, softmax turns scores into weights, weights × V produces the output. Animated flow shows the data path.
Core equationQuestion 01 / 07
In scaled dot-product attention, why divide by √dk instead of dk?
AConvention — dividing by dk works equally well.
BTo reduce the computational cost of the division.
CIf Q and K elements have unit variance, the dot product QKT has variance dk. Its standard deviation is √dk. Dividing by √dk normalizes the variance back to ~1, keeping softmax in its smooth, gradient-friendly regime. Dividing by dk would over-normalize, making all scores too close to zero — the softmax output would be nearly uniform, losing discriminative power.
D√dk is always 1 in practice so the division has no effect.
02 / 07
The full computation — worked by hand.
Scaled dot-product attention: Tq=1 (single query), Tk=4 (four movies), dk=3, dv=2
Q = [[1.0, 0.5, −0.3]] — the user's "what I want next" queryShape: (1, 3)
K = [[0.8, 0.3, −0.1], [−0.2, 0.9, 0.4], [1.1, 0.6, −0.5], [0.1, −0.3, 0.8]] — four movie keysShape: (4, 3)
V = [[4.2, 0.8], [1.5, 3.7], [3.9, 1.2], [0.7, 2.1]] — four movie feature valuesShape: (4, 2)
QKT = Q × KT: row 0 = [1.0·0.8+0.5·0.3+(−0.3)·(−0.1), 1.0·(−0.2)+0.5·0.9+(−0.3)·0.4, ...]= [0.98, 0.13, 1.55, −0.39] — movie 3 has highest raw score (1.55)
Scale by √dk = √3 ≈ 1.732: [0.98, 0.13, 1.55, −0.39] / 1.732 = [0.566, 0.075, 0.895, −0.225]Without scaling: softmax([0.98, 0.13, 1.55, −0.39]) = [0.25, 0.11, 0.44, 0.06] — already peaked
With scaling: softmax([0.566, 0.075, 0.895, −0.225]) = [0.26, 0.16, 0.36, 0.12] — smoother, more distributed
Weights α = softmax(scaled) = [0.26, 0.16, 0.36, 0.12]Sum = 1.00. Movie 3 gets 36% weight (highest match), movie 4 gets 12% (lowest)
Output = α · V = 0.26·[4.2, 0.8] + 0.16·[1.5, 3.7] + 0.36·[3.9, 1.2] + 0.12·[0.7, 2.1]= [1.09, 0.21] + [0.24, 0.59] + [1.40, 0.43] + [0.08, 0.25] = [2.82, 1.48]
Interpretation: the output is a blend of all four movies' values, dominated by movie 3 (36%) and movie 1 (26%). These two are most relevant to the query.In CineMatch: this output vector feeds into the next layer or the ranking head

ComputationQuestion 02 / 07
If you double every element of Q while keeping K and V fixed, what happens to the attention output?
AThe output doubles — attention is linear in Q.
BThe scores QKT double, making the softmax distribution sharper (more peaked on the max-score key). The output shifts toward the value vector of the highest-scoring key. The magnitude of the output stays roughly the same (softmax weights still sum to 1), but the content changes — it becomes less of a blend and more of a hard lookup.
CNothing changes — the √dk cancels the doubling.
DThe output becomes zero because softmax saturates.

Multi-head attention — parallel subspace experts.

A single attention head operates on one set of Q, K, V projections. Multi-head attention runs h heads in parallel, each on a dk = dmodel/h dimensional subspace, then concatenates:

MultiHead(X) = Concat(head1, ..., headh) · WO (3.1)
where headi = Attention(X · WQi, X · WKi, X · WVi) (3.2)
Multi-head parameter count — no extra cost vs single-head
Single-head: WQ, WK, WV each ∈ ℝ^{d×d}, WO ∈ ℝ^{d×d}. Total: 4d²At d=768: 4 × 768² = 2,359,296 params
Multi-head (h=12): WQi ∈ ℝ^{d×(d/h)} for each of h heads. Total Q params: h × d × (d/h) = d²Same total! Multi-head is a reshape, not more parameters. 12 × 768 × 64 = 768² = 589,824
All projections: 4d² for multi-head = 4d² for single-head = 2,359,296 at d=768Multi-head gives 12 different attention patterns for the same parameter budget — strictly better
Multi-head attention. Input X is projected into 12 parallel 64-dim subspaces. Each head runs its own attention, specializing in different patterns. Outputs are concatenated back to 768-dim and mixed by WO.
Multi-headQuestion 03 / 07
BERT has 12 heads with dmodel=768. A colleague proposes using 1 head with dk=768. Same parameter count. Why is multi-head better?
AMulti-head is faster on GPU.
BMulti-head uses fewer total parameters.
CMulti-head prevents overfitting.
DA single head computes one attention pattern per position — it must compress syntax, semantics, coreference, and positional structure into one set of weights. 12 heads compute 12 independent patterns, each specializing: head 3 might track subject-verb agreement, head 7 might focus on nearby context, head 11 might learn coreference. Same parameters, 12× more representational diversity.
04 / 07
Self-attention vs cross-attention — the two modes.

Attention has two configurations that produce fundamentally different behaviors:

ModeQ fromK, V fromWhat it doesUsed in
Self-attentionSame sequence XSame sequence XEach position queries all other positions in the same sequenceBERT encoder, GPT decoder
Cross-attentionSequence A (decoder)Sequence B (encoder output)Decoder positions query encoder positions — "reading" the sourceT5 decoder, translation

In self-attention, Q = K = V = X (all projected from the same input). Every position can attend to every other position. This is what BERT uses — and it's why BERT produces contextual representations: each token's output incorporates information from the entire sequence.

In cross-attention, Q comes from one sequence (typically the decoder's current state) while K and V come from another (the encoder's output). This is the mechanism that lets a decoder "look at" the source — essential for translation, summarization, and encoder-decoder models like T5 (Phase 4).

CineMatch Integration

Both attention modes in one system.

Self-attention (BERT-style): CineMatch encodes a user's review with bidirectional self-attention. Each word's representation incorporates the full review context. "The acting was [terrible]" → the representation of "terrible" includes information from "acting" and the entire sentence structure.

Cross-attention (T5-style): CineMatch's summarization model uses cross-attention to generate a 2-sentence summary. The decoder queries the encoder's review representation at each generation step: "what part of the review should I summarize next?"

Self vs crossQuestion 04 / 07
In BERT, all attention is self-attention (Q, K, V all from the same input). What would happen if you replaced BERT's self-attention with cross-attention from a random external sequence?
AThe model would produce representations of the external sequence, not the input. Each position's output would be a weighted sum of the external sequence's values — the input's content is only used as the query, not as the information source. The resulting representations would not encode the input's semantics.
BNothing would change — attention is attention regardless of source.
CThe model would be faster because cross-attention is cheaper.
DThe model would crash because BERT only supports self-attention.
05 / 07
Masking — controlling what attention can see.

The raw attention equation lets every position attend to every other position. In practice, we sometimes need to restrict this. Masking is how:

Mask typeWhat it doesUsed in
No mask (bidirectional)Every position sees every positionBERT, encoder models
Causal mask (triangular)Position i can only see positions ≤ iGPT, decoder models
Padding maskIgnore [PAD] tokens in shorter sequencesAll models with variable-length batches

The causal mask is what makes GPT autoregressive — position t cannot peek at positions t+1, t+2, ... during training, matching the inference scenario where those tokens don't exist yet. BERT has no causal mask — every position sees the full sequence, which is why it produces richer representations for understanding tasks but cannot generate text left-to-right.

The mask that defines the model family

Encoder = no causal mask (bidirectional). Decoder = causal mask (left-to-right).

This single design choice — mask or no mask — is the fundamental architectural difference between BERT-family and GPT-family models. Everything else (layer count, FFN size, training data) is secondary. The mask determines what the model can see, which determines what tasks it can perform: bidirectional → understanding (classification, NER, search). Causal → generation (text, code, recommendations). This distinction is the organizing principle of this entire track.

MaskingQuestion 05 / 07
Could you train BERT with a causal mask (making it left-to-right like GPT)?
ANo — BERT's architecture physically can't support a causal mask.
BYes, and it would perform identically.
CYes — architecturally it's the same transformer encoder with a different mask. But downstream performance on classification/NER would drop significantly because each position only sees left context, not the full sentence. You'd have a model that's worse at understanding (vs BERT) and worse at generation (vs GPT, which is optimized for it). The mask choice should match the intended task family.
DYes, and it would be better than BERT because causal masking prevents data leakage.

The notation for this track.

Every module from here uses these conventions:

SymbolMeaningTypical values
dmodel or dHidden dimension (embedding + all layers)768 (base), 1024 (large), 4096 (LLaMA-70B)
h or nheadsNumber of attention heads12, 16, 32, 64
dk = d/hPer-head dimension64, 128
dffFFN inner dimension (typically 4d)3072, 4096, 11008
LNumber of transformer layers/blocks12, 24, 32, 80
T or seq_lenSequence length512, 2048, 8192, 128K
VVocabulary size30522 (BERT), 32000 (LLaMA), 100K+ (modern)
BBatch size8, 16, 32, 64
One equation, three masks, every model

Attention(Q, K, V) = softmax(QKT/√dk + mask) · V. Change what goes into Q/K/V → different model family. Change the mask → encoder vs decoder. Change d, h, L, dff → different scale. That's the entire design space of transformer architectures, and every module in this track is a point in that space.

Design spaceQuestion 06 / 07
BERT-base has d=768, h=12, L=12, dff=3072. GPT-2 Small has d=768, h=12, L=12, dff=3072. Same hyperparameters. What makes them fundamentally different models?
AThe FFN activation function.
BThe attention mask. BERT uses no causal mask (bidirectional), GPT-2 uses a causal mask (left-to-right). This single difference means BERT sees full context and is trained with MLM, while GPT-2 sees only left context and is trained with next-token prediction. Same architecture, same size, fundamentally different capabilities.
CThe vocabulary — BERT uses WordPiece, GPT-2 uses BPE.
DThe number of parameters.
SynthesisQuestion 07 / 07
For CineMatch's semantic search (encode user queries and movie descriptions into vectors for nearest-neighbor lookup), should the encoder use bidirectional or causal attention?
ACausal — generation is always better.
BNeither — attention isn't needed for search.
CCausal — because we process the query left-to-right.
DBidirectional. Semantic search is an understanding task: encode the full meaning of a query or description into a single vector. Bidirectional attention lets every position see the full input context — "movies with surprising endings" encodes the relationship between "surprising" and "endings" regardless of word order. A causal encoder would only let "endings" see "surprising" but not vice versa, producing a weaker representation.
