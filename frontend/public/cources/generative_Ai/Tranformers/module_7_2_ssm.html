Mamba &
state-space
models.

Every model in this track uses attention — the O(T²) mechanism that computes pairwise token relationships. State-space models (SSMs) ask: what if we replace attention entirely? Mamba (Gu & Dao, 2023) processes sequences in O(T) time and O(1) memory per step by maintaining a fixed-size hidden state — like an RNN, but with the parallelism of a convolution during training. Mamba matches transformer quality on language modeling at 1-3B scale with 5× higher throughput on long sequences. Is this the post-attention future? This module examines the evidence.

01 / 08
The core idea — a fixed-size state replaces the KV cache.

Attention stores the entire past context in the KV cache — which grows linearly with sequence length. SSMs compress the entire past into a fixed-size hidden state (typically d dimensions, same as the model dimension). Each new token updates the state; the state summarizes all previous tokens.

Attention: context = [K₁, V₁, K₂, V₂, ..., Kt, Vt] — grows with t (1.1)
SSM: context = ht ∈ ℝd — fixed size, regardless of t (1.2)
Memory comparison: attention KV cache vs SSM hidden state at inference
Transformer (LLaMA-3 8B, 128K context): KV cache = 32 layers × 2 (K,V) × 8 KV heads × 128K × 128 d_head × 2 bytes = 16.8 GBGrows linearly with context length. At 256K context: 33.6 GB. At 1M: 134 GB. Eventually exceeds model weight size.
Mamba (equivalent 8B): Hidden state = 32 layers × d_state × d_model × 2 bytes. With d_state=16, d_model=4096: 32 × 16 × 4096 × 2 = 4 MBFIXED. Same 4 MB whether context is 1K or 1M or 10M tokens. No growth with sequence length.
Memory ratio at 128K context: 16.8 GB / 4 MB = 4,200× less memory for Mamba's state vs transformer's KV cache.At 1M context: 134 GB / 4 MB = 33,500× less. Mamba's advantage grows with context length.
State vs cacheQuestion 01 / 07
Mamba compresses all past context into a fixed-size state. Attention stores every past token in the KV cache. What does Mamba lose by compressing?
ANothing — compression is lossless.
BAll past information — only the last few tokens are remembered.
CThe ability to do exact retrieval from the past. Attention can look up any specific past token (the KV cache preserves every position's exact representation). Mamba's fixed-size state is a lossy summary — it captures the general trend of past tokens but can't perfectly retrieve a specific fact from 10K tokens ago. This is the fundamental trade-off: O(1) memory at the cost of imperfect memory. For general language modeling (predicting the next token from overall context), SSM states work well. For precise retrieval ("what was the name mentioned in paragraph 3?"), attention's exact KV cache is superior.
DSpeed — SSMs are always slower than attention.
02 / 08
Mamba's selective state mechanism — the key innovation.

Classic SSMs (S4) use fixed state transition dynamics — the same update rule regardless of the input. Mamba's innovation: input-dependent state transitions. The model decides, for each token, how much to write into the state and how much to forget:

ht = At · ht-1 + Bt · xt    where At, Bt depend on input xt (2.1)

At (the "forget gate") controls how much of the previous state to retain. Bt (the "write gate") controls how much of the current input to write into the state. Both are functions of the current input — the model can selectively remember important tokens and forget irrelevant ones.

# Simplified Mamba selective scan (conceptual)
def selective_scan(x, A_params, B_params, C_params, dt):
    """Process sequence with input-dependent state transitions."""
    B, T, d = x.shape
    h = torch.zeros(B, d_state, d, device=x.device)

    outputs = []
    for t in range(T):
        # Input-dependent gating (Mamba's key innovation)
        dt_t = F.softplus(dt(x[:, t]))        # step size — how much to update
        A_t = torch.exp(-dt_t.unsqueeze(-1) * A_params)  # forget factor
        B_t = dt_t.unsqueeze(-1) * B_params(x[:, t])     # write factor

        # State update: forget some, write new
        h = A_t * h + B_t * x[:, t].unsqueeze(1)

        # Output: read from state
        y_t = (C_params(x[:, t]).unsqueeze(1) * h).sum(dim=1)
        outputs.append(y_t)

    return torch.stack(outputs, dim=1)
SelectiveQuestion 02 / 07
Mamba's A_t is input-dependent — the model decides how much to "forget" based on the current token. Why is this important?
AIt allows content-aware memory management. When the model sees an important token (a name, a key fact), it can set A_t low (retain most of the state) and B_t high (write strongly). When it sees a filler token ("um," "the," articles), it can set A_t high (aggressively forget) and B_t low (don't write). This selective attention to input content is what gives Mamba its quality — without it (fixed A), the state treats every token equally, wasting capacity on filler. Input-dependent gating is Mamba's equivalent of attention's ability to selectively focus on important positions.
BIt makes Mamba faster during training.
CIt reduces the number of parameters.
DIt allows Mamba to process images.
03 / 08
Training efficiency — the parallel scan trick.

The sequential loop in the code above is O(T) but not parallelizable — each step depends on the previous state. For training (where the full sequence is available), Mamba uses a parallel scan algorithm that computes all T states simultaneously in O(T log T) work with O(log T) depth — fully parallelizable on GPUs.

At inference, the sequential recurrence is used: each new token updates the state in O(1) time and O(d_state × d) memory. This is the speed advantage — no KV cache growth, constant-time per token regardless of context length.

TrainingQuestion 03 / 07
Mamba is sequential at inference (like an RNN) but parallel during training (like a CNN). How can the same operation be both?
AMamba uses different architectures for training and inference.
BTraining doesn't need to process tokens in order.
CGPUs can parallelize any sequential operation.
DThe parallel scan algorithm. During training, all inputs x_1...x_T are known. The recurrence h_t = A_t·h_{t-1} + B_t·x_t can be rewritten as a prefix sum (cumulative operation), which has a well-known O(T log T) parallel algorithm. It's the same mathematical operation computed two different ways: sequentially for inference (when future tokens don't exist yet) and in parallel for training (when all tokens are available). This dual-mode execution is the key insight — RNN-like semantics with CNN-like training speed.

Mamba's quality gap on recall-intensive tasks

On "needle in a haystack" tests (find a specific fact in a long document), Mamba's accuracy drops from 95% at 4K context to 60% at 32K. Transformers maintain 95%+ at all lengths.

The fixed-size state can't store every fact from a long document — it prioritizes recent and repeated information. A specific fact mentioned once at position 500 may be overwritten by the time the model reaches position 30,000. Transformers don't have this problem because the KV cache preserves every position exactly. This is the fundamental quality trade-off: O(1) memory = lossy compression = degraded recall. For tasks requiring precise long-range retrieval, transformers are still superior.

ComparisonQuestion 04 / 07
Mamba matches transformer perplexity at 1-3B scale. Does this mean Mamba is a transformer replacement?
AYes — matching perplexity means matching quality.
BYes — Mamba is strictly better (same quality, faster).
CNot yet — for three reasons. (1) Perplexity measures average next-token prediction, not downstream task quality. Mamba matches perplexity but underperforms on retrieval-heavy tasks (QA from long documents, in-context learning with many examples). (2) Scale: Mamba is proven at 1-3B, not at 70B+. Transformer scaling laws are well-understood; SSM scaling laws are still being established. (3) Ecosystem: transformers have 7 years of optimization (FlashAttention, vLLM, quantization tooling). Mamba's ecosystem is nascent. Mamba is a compelling alternative for specific use cases, not a general replacement — yet.
DNo — SSMs can never match transformers.
05 / 08
Hybrid architectures — the best of both worlds.

The emerging pattern: combine Mamba layers (fast, memory-efficient) with attention layers (precise retrieval) in the same model. Jamba (AI21, 2024) interleaves Mamba blocks and attention blocks in a 52B MoE model — getting Mamba's throughput advantage for most layers and attention's retrieval precision for a few layers.

Hybrid architecture design patterns (2026)
Pure transformer: All layers use attention. Maximum quality, highest memory. The default.LLaMA-3, GPT-4, Mistral. Proven at all scales.
Pure Mamba: All layers use SSM. Fastest inference, lowest memory. Limited retrieval.Mamba-1, Mamba-2. Best for streaming, on-device, ultra-long context.
Hybrid (Mamba + attention): Most layers are Mamba (fast), every 4th-8th layer is attention (precise). Gets 80% of Mamba's speed advantage and 95% of transformer's quality.Jamba (AI21), Zamba (Zyphra). The emerging sweet spot.
Hybrid + MoE: Mamba layers + attention layers + MoE routing. Maximum flexibility at maximum complexity.Jamba-1.5 uses this pattern. Research frontier — not yet standard.
HybridQuestion 05 / 07
A hybrid model has 28 Mamba layers and 4 attention layers (every 8th layer). Why not use attention for all 32 layers?
ACost and speed. The 28 Mamba layers process tokens at O(1) per step with no KV cache growth. Only the 4 attention layers contribute O(T) KV cache and O(T) per-step cost. Total KV cache: 4/32 = 12.5% of a full transformer. Inference speed: ~3-4× faster than full attention at long context. Quality: the 4 attention layers provide enough "retrieval checkpoints" for the model to look up specific past information every 8 layers — sufficient for most tasks. 28 Mamba + 4 attention = near-transformer quality at near-Mamba speed.
BMamba layers are better for language modeling.
CAttention layers are only needed at the output.
D32 attention layers would make the model too large.
06 / 08
CineMatch's SSM opportunity.
CineMatch Integration

Where Mamba fits in CineMatch's stack.

Streaming recommendation feed: CineMatch's "For You" page generates personalized blurbs as users scroll — a streaming generation task where each blurb depends on the user's session context (what they've scrolled past). Mamba's O(1) per-token inference with constant memory is ideal: the state accumulates user preferences as they browse, generating personalized text without growing a KV cache. A Mamba-based model generates ~3× more blurbs per second than an equivalent transformer.

Not for the chatbot: CineMatch's chatbot needs precise recall ("earlier you mentioned you liked Inception — here's why you'd also like..."). Mamba's compressed state would lose the specific "Inception" mention from 2000 tokens ago. The chatbot stays on LLaMA-3 with attention.

The hybrid future: When hybrid models (Mamba + attention) mature at 7B+ scale, CineMatch could replace both the streaming feed model AND the chatbot with a single hybrid — using Mamba layers for fast generation and attention layers for precise conversation recall.

CineMatchQuestion 06 / 07
CineMatch's recommendation feed generates 100 personalized blurbs per second (across all users). Current transformer-based system costs $2,000/month in GPU. A Mamba-based system would cost $700/month (3× throughput). Should they switch?
AYes — always take the cheaper option.
BNo — Mamba quality is always worse.
CNo — the ecosystem is too immature.
DMaybe — run an A/B test. The $1,300/month savings is significant, but only if quality is maintained. Generate 1000 blurbs with both systems, have CineMatch's QA team rate them blind, and measure user engagement metrics (click-through, time-on-page) in an A/B test. If Mamba blurbs are indistinguishable from transformer blurbs: switch. If users notice a quality drop: keep the transformer. Don't optimize cost at the expense of user experience without measuring the trade-off.

The attention vs SSM question — where we are in 2026.
FutureQuestion 07 / 07
In 5 years (2031), will transformers or SSMs dominate language AI?
ATransformers — they're too established to replace.
BSSMs — they solve the fundamental bottleneck.
CHybrids — and the distinction won't matter. The trend in 2026 already points toward hybrid architectures that combine Mamba's efficient sequence processing with attention's precise retrieval. By 2031, the "transformer vs SSM" framing will likely be as dated as "CNN vs RNN" — both mechanisms will be combined in models where the architecture adapts to the task. The real question isn't which mechanism wins, but how to best combine them. The transformer's legacy isn't attention itself — it's the insight that sequence processing can be done with learnable, composable building blocks.
DSomething entirely new that we haven't invented yet.
