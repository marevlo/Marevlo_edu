Longformer &
BigBird.

If 95% of attention weights are near-zero, why compute them? Sparse attention computes only the connections that matter — local neighbors (for syntax and coherence) and a few global tokens (for document-level reasoning). Longformer (Beltagy et al., 2020) combines sliding-window local attention with task-specific global tokens. BigBird (Zaheer et al., 2020) adds random connections for theoretical completeness. Both achieve O(T) attention cost while preserving most of full attention's quality — the workhorse pattern for long-document understanding.

01 / 08
Longformer — local + global attention.

Longformer's attention pattern has two components:

1. Local sliding window (every token): Each position attends to its W nearest neighbors on each side (typically W=256, so a window of 512 positions). This captures local syntax, coreference, and phrase-level semantics — the patterns that account for most of attention's value.

2. Global tokens (selected positions): A small set of designated positions attend to (and are attended by) every position in the sequence. For classification, [CLS] is global. For QA, the question tokens are global. These global tokens aggregate information from the entire document, enabling long-range reasoning.

Longformer attention cost vs standard attention
Standard attention at T=4096: T² = 16.8M attention scores per head.O(T²) — every position attends to every other position.
Longformer at T=4096, W=512, G=4 global tokens: local = T × W = 4096 × 512 = 2.1M. Global = G × T × 2 = 4 × 4096 × 2 = 32.8K.Total: 2.1M + 33K ≈ 2.13M attention scores. 8× reduction vs standard.
At T=16K: standard = 256M. Longformer = 16K × 512 + trivial = 8.2M. 31× reduction.The savings grow with T: at T=64K, Longformer is 125× cheaper. The gap is O(T²) vs O(T×W).
Memory: standard stores T×T matrix. Longformer stores T×W matrix (banded). At T=16K: standard = 256M × 4B = 1 GB. Longformer = 8.2M × 4B = 33 MB. 30× less memory.Combined with FlashAttention-style tiling, Longformer fits extremely long sequences in limited GPU memory.
LongformerQuestion 01 / 07
Longformer uses W=512 local attention. How can it understand relationships between tokens 2000 positions apart?
AIt can't — W=512 limits understanding to nearby tokens.
BThe FFN layers handle long-range connections.
CTwo mechanisms. (1) Multi-layer propagation: like Mistral's sliding window (Module 3.2), information travels W positions per layer. After 4 layers: 4 × 512 = 2048 positions of effective receptive field. After 12 layers: 6144 positions. (2) Global tokens: [CLS] and other global positions attend to ALL positions, creating direct highways for long-range information. Token at position 0 → global token [CLS] (distance 0) → token at position 2000 (global attends everywhere). Two hops, no limit on distance.
DPosition embeddings carry the long-range information.
02 / 08
BigBird — local + global + random.

BigBird extends Longformer's pattern with a third component: random attention. Each position also attends to R randomly selected positions in the sequence (R ≈ 3-5). The random connections serve a theoretical purpose: they make the attention graph an expander graph, which guarantees that any token can reach any other in O(log T) hops — a stronger theoretical guarantee than local + global alone.

ComponentConnections per tokenPurpose
Local (sliding window)2W (W each side)Syntax, local coherence, adjacent reasoning
Global (designated tokens)T (attend to all)Document-level aggregation, task conditioning
RandomR (randomly chosen)Expander graph property, reduces hop count
BigBirdQuestion 02 / 07
BigBird adds random attention connections. In practice, does this improve quality over Longformer (local + global only)?
AYes — random connections are essential for quality.
BYes — but only for very long sequences (100K+).
CNo — random connections always hurt quality.
DMarginal. Empirically, the random connections add ~0.1-0.3% on most benchmarks — within noise. The global tokens already provide efficient long-range pathways. Random connections matter more in theory (expander graph guarantees) than in practice (global tokens suffice). Most practitioners use Longformer-style (local + global) without random connections. The engineering complexity of random attention (non-deterministic computation, harder to batch, harder to optimize on hardware) outweighs the marginal quality gain.
03 / 08
Choosing global token positions.

Which tokens should be "global" — attending to and attended by every position? The answer depends on the task:

TaskGlobal tokensWhy
Classification[CLS] token only[CLS] must aggregate the entire document for the classification head
QA / ExtractiveQuestion tokensThe question must "see" every passage token to find the answer
SummarizationEvery N-th token (e.g., every 512th)Evenly spaced global tokens sample the document uniformly
NERNone needed (local W=512 suffices)Entity boundaries are local — no long-range reasoning needed
Too many global tokens defeats the purpose

If G global tokens each attend to T positions, the global component costs G × T — which approaches T² when G is large.

Longformer's efficiency comes from having very few global tokens (G = 1-10) among thousands of local tokens. If you make 10% of tokens global (G = 0.1T), the global attention cost is 0.1T × T = 0.1T² — you've recovered 10% of the quadratic cost you were trying to avoid. Keep G ≤ 10 for most tasks. If you need more global information flow, increase the local window W instead — it scales as O(T × W), which is cheaper than O(G × T) when W < G.

Global tokensQuestion 03 / 07
CineMatch uses Longformer for long review analysis (4000 tokens per review). For the task "extract all actor names from this review," how many global tokens should they use?
AZero — NER is a local task. Actor names are recognized from local context: "starring [PERSON NAME]", "directed by [PERSON NAME]". The local window (W=512) provides more than enough context to identify entity boundaries and types. Adding global tokens wouldn't help because NER decisions depend on local patterns (word shape, surrounding words), not document-level structure. Save the global token budget for tasks that need it (classification, summarization).
B1 — always use at least one [CLS].
C10 — one per expected actor name.
DAll tokens — NER needs full context.

Implementation — the Longformer attention kernel.
# Simplified Longformer attention (conceptual)
import torch, torch.nn.functional as F

def longformer_attention(Q, K, V, window_size, global_mask):
    """
    Q, K, V: (B, T, d)
    window_size: int (W — each side)
    global_mask: (B, T) — True for global token positions
    """
    B, T, d = Q.shape

    # --- Local sliding window attention ---
    # Each position attends to [max(0, i-W) : min(T, i+W+1)]
    local_attn = torch.zeros(B, T, 2 * window_size + 1, device=Q.device)
    for i in range(T):
        start = max(0, i - window_size)
        end = min(T, i + window_size + 1)
        # Dot product with local keys only
        local_attn[:, i, :end-start] = (Q[:, i:i+1] @ K[:, start:end].transpose(-1,-2)).squeeze(1)

    # --- Global attention (for global tokens) ---
    # Global tokens attend to ALL positions
    global_indices = global_mask.nonzero(as_tuple=True)[1]
    for g in global_indices:
        # Global token g attends to full sequence
        global_scores = Q[:, g:g+1] @ K.transpose(-1,-2)  # (B, 1, T)
        # All tokens also attend to global token g
        reverse_scores = Q @ K[:, g:g+1].transpose(-1,-2)  # (B, T, 1)

The implementation gap: theory vs CUDA reality

Longformer's O(T × W) complexity is the theoretical cost. In practice, the sliding window pattern is memory-access unfriendly — it requires gathering non-contiguous elements from K and V, which breaks GPU memory coalescing. Custom CUDA kernels (or using FlashAttention's block-sparse variant) are needed to realize the theoretical speedup. A naive Python implementation of Longformer attention is often slower than standard FlashAttention because the irregular memory access patterns undermine GPU parallelism.

ImplementationQuestion 04 / 07
A team implements Longformer with a Python for-loop over the sliding window (like the code above). It's 5× slower than standard FlashAttention. Why?
ALongformer is theoretically slower than standard attention.
BThe Python loop defeats GPU parallelism. Standard FlashAttention processes the full T×T attention in large, contiguous memory blocks — maximizing GPU throughput. The for-loop Longformer processes T individual windows sequentially — underutilizing the GPU's parallel cores. The operations are fewer (O(T×W) vs O(T²)), but the memory access pattern is terrible. Fix: use custom CUDA kernels or FlashAttention-2's block-sparse mode, which tiles the sliding window into GPU-friendly blocks.
CPython is inherently too slow for attention.
DThe sliding window pattern uses more memory than standard attention.
05 / 08
CineMatch's Longformer deployment.
CineMatch Integration

Longformer for CineMatch's document understanding pipeline.

Use case: full script classification. CineMatch receives ~200 movie scripts/month from indie filmmakers submitting for platform inclusion. Each script is 25K-40K tokens. The classification task: "Should CineMatch feature this film?" based on genre fit, quality indicators, and content policy.

Architecture: Longformer-base (149M params) with [CLS] as the global token. Local window W=512. The [CLS] token aggregates the full script through global attention, then a classification head outputs accept/reject/review probabilities.

Why Longformer over LLaMA: This is a classification task, not generation. Longformer-base (149M) classifies at ~$0.001/script. LLaMA-3 8B would cost ~$0.05/script (50× more) for similar accuracy — because the task requires understanding, not generation. The encoder architecture (bidirectional context) is structurally superior for classification.

Why Longformer over DeBERTa: DeBERTa maxes out at 512 tokens. The scripts are 25K-40K tokens. Longformer handles the full script in one pass; DeBERTa would need chunking (process 50-80 chunks, aggregate results), losing cross-chunk context.

DeploymentQuestion 05 / 07
Instead of using Longformer on the full 30K-token script, CineMatch could chunk the script into 512-token pieces, classify each with DeBERTa, and aggregate. When is this chunking approach better than Longformer?
AAlways — DeBERTa is more accurate per chunk.
BNever — Longformer is always better for long documents.
COnly when the chunks are independent (no cross-chunk reasoning needed).
DWhen cross-chunk reasoning is minimal AND DeBERTa's per-chunk quality is much higher. For tasks like "does any scene in this script contain violence?" — each chunk can be independently checked (violence is local), and DeBERTa's superior per-position quality compensates for the lack of cross-chunk context. For tasks like "is this script's overall narrative arc compelling?" — chunking fails because the answer requires understanding the full story structure. The decision depends on whether the task is locally or globally evaluable.

Longformer vs modern approaches — where we are in 2026.
Sparse attention in the context of modern long-context solutions
Longformer/BigBird (2020): O(T × W). Custom sparse attention patterns. Best for encoder tasks (classification, NER on long docs). Limited ecosystem in 2026.Still used for: legal doc classification, medical record analysis, long-form content moderation.
Mistral SWA (2023): O(T × W). Sliding window without global tokens. Simpler than Longformer. Built into decoder-only architecture.The modern equivalent of Longformer's local attention, but for decoders.
FlashAttention-2 (2023): O(T²) compute, O(T) memory. Exact attention with optimized memory access. Makes 128K context practical with standard attention.Reduced the need for sparse attention by making standard attention work at longer contexts.
The 2026 reality: For most applications, FlashAttention-2 + GQA + RoPE scaling handles 128K tokens without sparse patterns. Longformer-style sparse attention remains valuable for: (a) very long documents (>128K), (b) encoder-only models on long docs, (c) efficiency-critical batch pipelines.Sparse attention went from "necessary for long context" to "useful for extreme lengths and encoder models."
Modern contextQuestion 06 / 07
FlashAttention-2 made 128K tokens practical with standard attention. Has this made Longformer obsolete?
AYes — FlashAttention solved the long-context problem.
BNo — Longformer is always faster.
CMostly — but not entirely. FlashAttention makes standard attention memory-efficient, but the compute is still O(T²). For high-volume batch processing of 50K+ token documents, Longformer's O(T × W) compute is still 10-50× cheaper per document. And for encoder-only models on long documents (classification, NER), Longformer remains the best architecture — there's no "FlashAttention for encoders" because the entire encoder model class has less ecosystem investment. Longformer's niche narrowed but didn't disappear.
DThey're equivalent — both achieve O(T) memory.

SynthesisQuestion 07 / 07
CineMatch processes 10,000 legal contracts per day (each 50K tokens) to extract licensing terms. Which attention strategy?
AStandard FlashAttention — simplest to implement.
BLongformer-style sparse attention. At 10K contracts/day × 50K tokens: standard attention costs O(50K²) = 2.5B ops per head per layer per contract. Sparse (W=512) costs O(50K × 512) = 25.6M ops — 100× cheaper. At 10K contracts: standard = $150/day GPU cost. Sparse = $1.50/day. The $148.50/day savings ($4,455/month) easily justifies the engineering investment in sparse attention kernels. Plus, legal NER is a local task (contract terms are identified from nearby clause context), so global tokens are optional.
CRAG — retrieve relevant clauses instead of processing full contracts.
DLLaMA-3 with 128K context — just process each contract as-is.
