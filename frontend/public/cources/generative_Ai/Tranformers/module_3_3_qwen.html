Qwen.
The multilingual
frontier.

LLaMA dominates English. Mistral dominates efficiency. Qwen (Alibaba Cloud, 2023-2025) dominates multilingual — and increasingly, math and code. The Qwen-2.5 family spans 0.5B to 72B parameters with the strongest non-English performance of any open-source model. For CineMatch's expansion into Hindi, Japanese, and Spanish markets, Qwen is the starting point. This module covers Qwen's multilingual tokenizer design, cross-lingual transfer, coding capabilities, and how to choose between LLaMA, Mistral, and Qwen for different deployment scenarios.

02 / 08
The multilingual tokenizer — why it matters more than you think.

LLaMA-3's tokenizer has ~128K vocabulary trained primarily on English text. For Hindi text, it produces 3-5× more tokens per word than for English — because Hindi characters are decomposed into byte-level subwords. Qwen's tokenizer (151,643 vocabulary) is trained on a balanced multilingual corpus, producing near-equal token counts across languages.

Tokenizer efficiency: same sentence in English vs Hindi
English: "This movie was amazing" → LLaMA: 5 tokens. Qwen: 5 tokens.Both tokenizers handle English well — similar vocabulary coverage.
Hindi: "यह फ़िल्म अद्भुत थी" (same meaning) → LLaMA: 18 tokens. Qwen: 6 tokens.LLaMA decomposes each Devanagari character into 2-3 byte-level tokens. Qwen has Hindi subwords in its vocabulary.
Impact on cost: LLaMA processes Hindi at 3× the token cost of English. Same compute budget → 3× fewer Hindi words per context window.For CineMatch Hindi reviews: LLaMA fits ~43K Hindi words in 128K tokens. Qwen fits ~128K Hindi words. 3× more context.
Impact on quality: more tokens per word = more positions for attention = more compute per word. But also: less context per window and more fragmented representations. Hindi text in LLaMA is processing individual bytes, not meaningful subwords — like reading English one character at a time.This is why Qwen's multilingual MMLU leads: the tokenizer makes non-English text first-class.
Tokenizer efficiency IS model quality for non-English

A model with a bad tokenizer for your language will underperform regardless of parameter count.

If LLaMA-3 70B tokenizes Hindi at 3× the cost of Qwen-2.5 7B, the 70B model sees 3× fewer Hindi words per context window. For tasks requiring long-range understanding (document summarization, conversation history), the 7B Qwen model with efficient tokenization can outperform the 70B LLaMA model with inefficient tokenization — even though LLaMA has 10× more parameters. Always check tokenizer efficiency on your target language before choosing a base model.

TokenizerQuestion 02 / 07
CineMatch is expanding to Japan. A Japanese review averages 200 characters. How many tokens will LLaMA-3 vs Qwen-2.5 produce?
ALLaMA-3: ~400-600 tokens (2-3 tokens per CJK character, since most Kanji are split into byte-level subwords). Qwen-2.5: ~150-250 tokens (most common Kanji and Kana are single tokens in its vocabulary). For a 128K context window: LLaMA fits ~50K Japanese characters, Qwen fits ~100K+. This means Qwen can process 2× longer Japanese reviews or 2× more conversation history in the same context window.
BBoth produce ~200 tokens — one per character.
CBoth produce ~100 tokens — Japanese is more compact.
DTokenizer doesn't affect Japanese — Unicode handles it.

Qwen's architecture — the modern standard with multilingual twists.

Qwen-2.5 uses the same modern decoder block as LLaMA-3 — pre-RMSNorm, GQA, SwiGLU, RoPE. The architecture is not the differentiator. What differentiates Qwen:

DifferentiatorLLaMA-3Qwen-2.5
Vocabulary size128,256151,643 (more multilingual tokens)
Training data languages~90% English~60% English, 40% multilingual
Math/code emphasisStandardExtra math + code data (Qwen-2.5-Coder, Qwen-2.5-Math)
Smallest model8B0.5B (edge deployment)
Specialized variantsCodeLLaMAQwen-Coder, Qwen-Math, Qwen-VL (vision-language)
The architecture has converged — differentiation is data

LLaMA-3, Mistral, Qwen-2.5, and Gemma all use essentially the same decoder block (RoPE + GQA + SwiGLU + RMSNorm). The differences are in training data composition, tokenizer design, and model sizes offered. Architecture innovation has plateaued for dense decoders; the competition is now about data quality, multilingual coverage, and specialization (code, math, vision).

ArchitectureQuestion 03 / 07
Qwen-2.5 and LLaMA-3 use nearly identical architectures. What makes Qwen better at multilingual tasks?
AQwen uses a different attention mechanism for multilingual.
BQwen has more parameters dedicated to non-English.
CQwen uses a special multilingual position encoding.
DTwo things, both pre-architectural: (1) Tokenizer — Qwen's 151K vocabulary includes subwords for CJK, Arabic, Devanagari, and other scripts, producing efficient tokenization across languages. LLaMA's vocabulary is English-optimized. (2) Training data — Qwen's training mix includes ~40% non-English text, ensuring the model sees enough of each language to learn its patterns. LLaMA's ~90% English training means non-English languages get ~10% of the learning signal. Same architecture, different data → different multilingual quality.
04 / 08
Qwen-Coder and Qwen-Math — specialized variants.

Alibaba releases specialized Qwen variants trained on domain-specific data. The base architecture is identical — only the training data changes:

Qwen-2.5-Coder (1.5B-32B): trained on additional curated code data. The 7B coder model achieves 83.5 on HumanEval — exceeding GPT-4's 67.0 on the same benchmark. This is the "textbook quality data" effect: a small model trained on exceptionally clean, diverse code outperforms a massive general-purpose model on coding tasks.

Qwen-2.5-Math (1.5B-72B): trained on mathematical reasoning data including step-by-step solutions, proofs, and competition problems. The 7B math model scores 83.6 on MATH (competition math), compared to GPT-4's 52.9. Again: domain-specific data > general-purpose scale for domain-specific tasks.

SpecializationQuestion 04 / 07
Qwen-2.5-Coder 7B beats GPT-4 on HumanEval (83.5 vs 67.0). Does this mean Qwen 7B is "better" than GPT-4?
AYes — benchmark results are definitive.
BNo — it means a specialized 7B model trained on curated code data outperforms a general-purpose model on a specific coding benchmark. GPT-4 is vastly better at reasoning, multi-step problems, natural language generation, and tasks requiring broad world knowledge. Qwen-Coder is better at writing short Python functions (what HumanEval measures). This is specialization vs generalization — the fine-tuned model wins on the narrow task, the general model wins on everything else.
CThe benchmark is wrong — GPT-4 should score higher.
DYes, but only for Python — GPT-4 is better at other languages.

The open-source decoder decision matrix.
Choosing between LLaMA, Mistral, and Qwen — the decision matrix
English-only, general purpose: LLaMA-3 8B or 70B. Largest ecosystem, most fine-tuned variants, best English quality.CineMatch's default for the US/UK market.
Efficiency-critical, single GPU: Mistral 7B. Fewest active parameters, sliding window for long context at low cost.CineMatch real-time autocomplete, on-premises deployment.
Quality-per-FLOP champion: Mixtral 8×7B. Best quality at ~13B active compute. Requires 2+ GPUs for memory.CineMatch batch processing, offline analysis.
Multilingual: Qwen-2.5 7B or 72B. Best non-English tokenizer and training data. Essential for non-Latin scripts.CineMatch Japan, India, Middle East expansion.
Code/math: Qwen-2.5-Coder or Qwen-2.5-Math. Domain-specialized training data beats general-purpose scale.CineMatch's internal tools: automated test generation, analytics dashboards.
On-device / edge: Qwen-2.5-0.5B or Gemma 2B. Sub-1B models for smartphones, IoT, embedded systems.CineMatch mobile app: offline review drafting, local search.
CineMatch Integration

CineMatch's multilingual expansion uses Qwen-2.5.

India launch (Hindi + English): Qwen-2.5-7B fine-tuned on bilingual CineMatch review data. Hindi tokenization at ~1 token per word (vs LLaMA's 3 tokens per word). DPO alignment on Hindi preference pairs collected from Indian movie review sites. Serving: vLLM on 1× A10G, ~200ms per response.

Japan launch (Japanese + English): Qwen-2.5-14B for higher quality on complex Japanese grammar. Fine-tuned on Japanese movie reviews from CineMatch's new Japanese catalog. CJK tokenization handles Kanji, Hiragana, and Katakana efficiently.

The hybrid stack: English market stays on LLaMA-3 8B (largest English fine-tuning ecosystem). Non-English markets use Qwen-2.5 (best multilingual foundation). Both share the same RAG pipeline, the same DeBERTa embedding layer, and the same CineMatch catalog. Only the generation model changes per market.

DecisionQuestion 05 / 07
CineMatch's Spanish market team says: "Let's just use LLaMA-3 8B for Spanish — it's our standard model." Is this reasonable?
AYes — LLaMA works fine for all Latin-script languages.
BNo — LLaMA can't process Spanish at all.
CNo — always use Qwen for non-English.
DIt's reasonable but suboptimal. Spanish uses Latin script, so LLaMA's tokenizer handles it decently (maybe 1.3× more tokens than English, not 3×). LLaMA-3's training data includes meaningful Spanish text (~5-8% of training). For Spanish, LLaMA is workable. For Hindi/Japanese/Arabic, it's not. The recommendation: benchmark LLaMA-3 vs Qwen-2.5 on Spanish CineMatch reviews. If LLaMA is within 2 points, keep it for infrastructure simplicity. If Qwen leads by 5+, switch.
06 / 08
Cross-lingual transfer — why multilingual models work at all.

A striking phenomenon: fine-tune Qwen-2.5 on English movie reviews only, and it also improves at classifying Hindi movie reviews — even though it never saw a Hindi review during fine-tuning. This is cross-lingual transfer.

It works because multilingual pretraining creates shared representations across languages. The concept "this movie was terrible" in English and "यह फ़िल्म भयानक थी" in Hindi map to nearby points in the model's internal representation space. Fine-tuning on English reviews teaches the model "negative sentiment → classify as negative." Because Hindi negative-sentiment representations are nearby, the same classification boundary works for Hindi — zero-shot.

TransferQuestion 06 / 07
You fine-tune Qwen-2.5-7B on 10K English CineMatch reviews for sentiment classification. What accuracy do you expect on Hindi reviews (zero-shot, no Hindi training data)?
A0% — the model only learned English patterns.
B50% — random chance on binary classification.
C~75-85%. Cross-lingual transfer gives substantial zero-shot performance. The model learned "negative sentiment → negative label" in English; because its multilingual representations align English and Hindi sentiment in the same region of embedding space, the classification boundary transfers. Adding even 100 Hindi examples (few-shot) typically pushes this to 88-92%. The gap to fully-supervised (95%+) is closed by collecting 2-5K Hindi-labeled reviews.
D95% — cross-lingual transfer is perfect.

Phase 3 — Complete

The open-source decoder landscape: three families, three strengths.

LLaMA (Module 3.1): RoPE, GQA, SwiGLU, RMSNorm — the modern decoder block. Chinchilla-optimal training. The default open-source foundation. Mistral/Mixtral (Module 3.2): sliding window attention, MoE architecture. Efficiency-first design. Best quality-per-FLOP. Qwen (Module 3.3): multilingual tokenizer, cross-lingual transfer, edge deployment. The non-English frontier.

Phase 4 moves to encoder-decoder models: T5 and BART — architectures that use both encoding (bidirectional) and decoding (autoregressive) for structured generation tasks like translation and summarization.
