SCENARIO 2 · speculative_decoding_calculator.py at Nexus v5 parameters
───────────────────────────────────────────────────────────────────────────────

Command:
  python speculative_decoding_calculator.py --alpha 0.85 --k 5 \
      --draft-ms 1.2 --target-ms 12

Expected key output:

    Configuration:
      α (acceptance rate):  0.85
      K (speculation len):  5
      Draft model step:     1.2 ms
      Target model step:    12.0 ms

    Derived:
      E[tokens per big pass]:    4.15
      Iteration time:            18.00 ms
      Effective TPOT:            4.33 ms
      Baseline (no speculation): 12.00 ms
      Speedup:                   2.77×
      Cost multiplier (vs baseline): 0.36×

   VERDICT
   ✓ Deploy speculation. At α=0.85, K=5, the configuration delivers
     2.77× effective throughput per big-model pass.

INTERPRETATION
  The 4.15 tokens-per-pass is what the M5 course content calls "3.1 tokens
  per big pass" for a slightly different config. The ratio 2.77× is what
  Nexus v5 observed in production. This number is the headline deliverable
  of speculative decoding — all other metrics flow from it.

SCENARIO 3 · speculative_decoding_calculator.py α-sweep showing sensitivity
───────────────────────────────────────────────────────────────────────────────

Command:
  python speculative_decoding_calculator.py --alpha-sweep --k 5 \
      --draft-ms 1.2 --target-ms 12

Expected output:

       α   E[tokens/iter]    iter ms   eff TPOT    speedup
     ------------------------------------------------------
     0.50           1.97      18.00      9.14      1.31×
     0.60           2.38      18.00      7.55      1.59×
     0.65           2.64      18.00      6.81      1.76×
     0.70           2.94      18.00      6.12      1.96×
     0.75           3.29      18.00      5.47      2.19×
     0.80           3.69      18.00      4.88      2.46×
     0.85           4.15      18.00      4.33      2.77×
     0.90           4.69      18.00      3.84      3.12×
     0.95           5.30      18.00      3.40      3.53×

   VERDICT
   Sensitivity: α drop from 0.85 → 0.60 reduces speedup by 43%.
   Monitor α with alarm threshold: alarm if α < 0.70 for 1h sustained.

INTERPRETATION
  This sweep is why the alarm threshold is 0.70, not 0.50. At α=0.70 you've
  already lost 29% of the speedup relative to α=0.85. If you wait until
  α=0.50 to alarm, you've lost 53% and cost-per-token has visibly moved.
  The rule: alarm early (0.70), fix before it gets worse.

SCENARIO 4 · scheduler_simulator.py on mixed workload near capacity
───────────────────────────────────────────────────────────────────────────────

Command:
  python scheduler_simulator.py --arrival-rate 750 --duration-s 60 \
      --policy-compare fifo,sjf,edf,priority

Expected output:

   policy       TTFT p50     TTFT p99   TTFT p99.9    overall   interact   standard      batch
   ---------- ---------- ------------ ------------ ---------- ---------- ---------- ----------
   fifo           8943ms      18432ms      18732ms      3.8%      1.2%      3.9%     21.3%
   sjf              58ms      14235ms      15891ms     93.5%    100.0%     97.1%     45.0%
   edf            2876ms       9132ms       9843ms     52.4%     56.4%     55.1%      9.4%
   priority         57ms      13056ms      13898ms     91.4%    100.0%    100.0%     15.7%

   VERDICT
   Ship priority.
     Interactive SLO compliance: 100.0%
     Overall SLO compliance:     91.4%
     vs FIFO: interactive +98.8 pp, batch -5.6 pp.

INTERPRETATION
  FIFO at 75% load is catastrophic — 3.8% overall SLO compliance means
  the scheduler is the bottleneck, not the hardware. Priority fixes
  interactive immediately. SJF is close but sorts by output length which
  is often unknown a priori. EDF is a balanced compromise.

  The real-world lesson: vLLM's FIFO default is fine for homogeneous
  traffic (all-chat or all-batch), but mixed-SLO workloads NEED priority
  or EDF. Changing this one setting can improve interactive SLO by 95+ pp.

  point-in-time alarms catch it.

───────────────────────────────────────────────────────────────────────────────
WHAT THESE OUTPUTS TELL YOU ABOUT YOUR DEPLOYMENT
───────────────────────────────────────────────────────────────────────────────

1. Before deploying: run batching_throughput_model.py and
   speculative_decoding_calculator.py. The first tells you max_num_seqs
   for your SLO, the second tells you whether speculation is worth it.
   Both answers go in the config and the RFC.

2. When picking a scheduler: run scheduler_simulator.py on a
   representative arrival rate. If your workload is mixed-SLO and your
   arrival rate is >70% of capacity, you need priority or EDF — FIFO
   will fail under load.

3. After deploying speculation: run draft_model_acceptance_tracker.py
   weekly on production decoding logs. Any drift >0.05 over a 4-week
   window means retraining is overdue.

4. When cost-per-token rises unexplainably: first check acceptance rate
   α with the tracker. It's the most common silent cost regression and
   the fix (weekly retraining) is cheap.
