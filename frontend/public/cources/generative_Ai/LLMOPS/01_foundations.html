01 · The Nexus Mandate

A fraud signal, in eighty milliseconds.

Nexus is a real-time fraud detection platform for a large multinational payments processor. It runs in line with the checkout flow: before the transaction is authorized, Nexus receives a structured context window and must return a fraud score between 0 and 1. The requirements are unforgiving.

The Nexus Request · Inputs · Outputs · SLOs

Take a hard look at those numbers. 6,200 input tokens is a real context window for this use case — transaction payloads are dense with semi-structured data. 180 output tokens is a constrained JSON schema. 80 ms p99 is set by the payment network timeout — if checkout sits longer, the customer abandons or the processor falls back to a rules-only decision. And 10,000 requests per second is a quiet Tuesday afternoon for a top-10 global payments processor; Black Friday peaks hit five times that.

The platform team took the most reasonable first approach. They downloaded the fine-tuned 70B model, loaded it with transformers, wrapped it in a FastAPI endpoint, deployed it on a node with 2 × H100 80GB GPUs, and sent 100 warm-up requests. The results killed the project within 48 hours.

⚠

Day-one telemetry, Nexus v1

p99 latency: 4,200 ms (52× the SLO). Throughput: 1,100 tokens/sec aggregate across both GPUs. GPU compute utilization: 34%. Memory utilization: 97% (weights alone), KV cache starved. Projected single-region monthly cost: $2.8M. The failure is not a bug in the code — the code is correct. The failure is a fundamental misunderstanding of what inference actually costs.

To understand why the naive deployment fails — and to understand every optimization technique in the next seven modules — we have to build the problem from first principles. That begins with looking at what the GPU does when it generates a single token.

02 · The Autoregressive Bottleneck

Why one token costs so much.

A transformer decoder, stripped to its essentials, is a stack of layers that each perform two operations — self-attention and a feed-forward network — both of which are dominated by large matrix multiplications. For inference we care about one thing: what does it cost to produce the next token?

The two phases of LLM inference

Every request has two distinct computational regimes. The literature calls them prefill and decode, and they have completely different performance characteristics. This distinction is perhaps the single most important concept in LLM serving — almost every optimization in this course exploits the gap between them.

Prefill vs Decode — The Same Model, Two Different Workloads
🔑

The single most important insight in LLM serving

Prefill is compute-bound. Decode is memory-bandwidth-bound. They are different workloads running on the same hardware. Every serving optimization in existence either (a) makes prefill faster by doing less math or using more efficient kernels, (b) makes decode faster by reading less from memory or reading it in a better shape, or (c) hides one phase behind the other so the GPU is never idle. If you internalize nothing else from this module, internalize this.

Why decode hits the memory wall — a concrete calculation

A 70B-parameter model, in FP16, stores its weights as 70 × 10⁹ × 2 bytes = 140 GB. To produce one output token, the model must — at minimum — read every weight once during the forward pass. That's a hard lower bound set by the architecture: no amount of cleverness can skip weights you genuinely need.

An NVIDIA H100 SXM GPU has HBM3 memory with a peak bandwidth of 3.35 TB/s. The absolute fastest you can possibly stream 140 GB through HBM, assuming 100% efficient utilization (which never actually happens), is:

Decode time lower bound — memory-bound regime t_min = weight_bytes / HBM_bandwidth = 140 GB / 3.35 TB/s ≈ 41.8 ms per token

Forty-two milliseconds per token. For Nexus to emit 180 output tokens, the theoretical minimum is 7.5 seconds on a single H100 — and that's before we account for attention (which gets worse with every new token), Python overhead, kernel launch latency, or the prefill phase that precedes decoding. The naive 4.2-second p99 is not a bug; it is close to the physical floor of what this architecture permits on this hardware, with this batch size.

This is the inescapable truth that motivates every technique in this course. There are exactly three ways out: read fewer weights (quantization, pruning, smaller models, MoE), read the same weights for more work (batching — amortize the read across many requests), or hide the reading behind compute (pipelining, speculative decoding, chunked prefill). Every module from here maps to one of those three escape routes.

Σ

The rule of thumb you should memorize

For decode with batch size 1, the time to generate one token is approximately model_size_bytes / HBM_bandwidth. This formula is accurate to within 20% on modern GPUs for all models from 7B to 405B. If someone tells you they are "making inference faster" without changing model size, data type, batch size, or hardware, they are either mistaken or optimizing a metric that does not matter for single-stream decode latency.

03 · The Roofline Model

One chart that predicts performance.

The roofline model, introduced by Samuel Williams at Berkeley in 2009, is the single most useful tool for reasoning about the performance of any kernel on any accelerator. It plots achievable performance (FLOP/s) against arithmetic intensity (FLOP per byte of memory traffic). For LLM inference, it tells you — at a glance — which regime you are in and which optimization will actually help.

The Roofline — NVIDIA H100 SXM · FP16 Tensor Cores

Read this chart carefully. A 70B decode at batch size 1 sits at arithmetic intensity ≈ 0.5 FLOP/byte. Following the sloped memory-bound roof up to that x-coordinate gives an achievable performance of about 1.7 TFLOP/s. The H100's peak is 990 TFLOP/s. We are extracting 0.2% of the hardware we are paying for.

Now look what happens when we crank batch size up to 128: we move right along the x-axis to AI ≈ 64, and the achievable performance climbs to 214 TFLOP/s — a 125× improvement in compute utilization. We still have not hit the compute roof, but we are in a dramatically more efficient regime. This is exactly why batching is the single most important lever in LLM serving economics. The next modules spend enormous effort on batching strategies not because batching is complicated — it is simple — but because the roofline tells us batching is where all the efficiency lives.

Why the ridge point number is worth memorizing

The ridge point for the H100 — the arithmetic intensity at which memory-bound turns into compute-bound — is peak_FLOPs / HBM_bandwidth ≈ 990e12 / 3.35e12 ≈ 295 FLOP/byte. For H200 with HBM3e at 4.8 TB/s it is about 206. For MI300X it is around 163. These numbers tell you how much arithmetic each byte of memory traffic has to do before the hardware is fully utilized. LLM decoding at batch 1 produces 2/D ≈ 1 FLOP/byte. The gap between 1 and 295 is the entire reason this field exists.

Arithmetic intensity, formally

Arithmetic intensity is defined as the ratio of FLOPs performed to bytes of memory traffic moved between the compute units and main memory. For a matrix multiply C = A @ B where A is M×K, B is K×N, and both are stored in FP16:

Arithmetic intensity of GEMM FLOPs = 2 × M × N × K  ·  Bytes = 2 × (M×K + K×N + M×N)
AI = FLOPs / Bytes ≈ M×N×K / (M×K + K×N + M×N)

For prefill at Nexus's 6,200-token context with hidden dimension 8,192, M=6200 and N=K=8192, so AI ≈ 3,600 — deeply compute-bound. For decode with M=1, AI ≈ 1 — deeply memory-bound. The entire performance difference between the two regimes comes from that single dimension of the matmul.

04 · The Serving Metrics

What to measure, and why average latency lies.

"The average response time is 120 ms" is one of the most dangerous statements in LLM serving. It hides the four metrics that actually determine whether a deployment is healthy, and it averages across two phases (prefill and decode) that have no business being averaged together.

The four numbers that matter

MetricDefinitionWhat bad looks likeWhat fixes it
TTFT — Time to First Token Wall-clock from request arrival to the first output token hitting the wire. > 500 ms for chat; > 150 ms for real-time. Chunked prefill, prefix caching, smaller prompt, faster attention.
TPOT — Time Per Output Token Average ms between successive output tokens during decode. > 50 ms for chat perception; > 20 ms for code gen. Quantization, smaller model, speculative decoding, GQA/MLA.
Throughput Total output tokens per second the cluster delivers, across all requests. Single-digit % of hardware peak. Continuous batching, PagedAttention, higher batch sizes.
Goodput Throughput of requests that met their SLO. Throughput high but goodput falls under load. Priority scheduling, admission control, disaggregated prefill.
⚠

Why you must report percentiles, not means

LLM latency distributions are heavily right-skewed. A system where mean latency is 150 ms but p99 is 3,200 ms is a broken system that happens to look good on average. For a real-time service like Nexus, the p99 is what your users experience during the worst 1% of their interactions — which, at 10k req/s, is 100 requests per second sitting above the SLO. Always report p50, p90, p99, p99.9. If a vendor benchmark reports only a mean, assume the tail is bad.

The anatomy of a single request, visualized

One Nexus Request · The Latency Breakdown

Look where the time actually goes. Queue wait and prefill together — the part that ends at TTFT — are roughly 300 ms. The remaining 3.9 seconds is pure decode: 180 tokens × 22 ms per token. 94% of wall-clock time is spent in the memory-bandwidth-bound decode regime. Any optimization that does not attack decode is optimizing the wrong part of the problem for this workload.

This is not a universal truth — for a chat application with 30-token prompts and 500-token responses, the ratio is different. For a code completion tool with 10-token prompts and 5-token responses, TTFT is everything. Always profile before optimizing. The measurement-first discipline is why this course insists on a benchmark harness from Module 1.

A production benchmark harness

Most LLM benchmark scripts are broken in subtle ways. They forget to warm up the GPU (first iteration includes CUDA context initialization). They use time.time() around GPU operations without synchronizing (CUDA kernels are asynchronous — you are measuring the Python call, not the work). They average across TTFT and TPOT. They report mean instead of percentiles. Here is what correct measurement looks like.

bench_nexus.py · correct measurementPYTHON
# A faithful benchmark harness for Nexus v1. The patterns here — warmup, CUDA
# sync, phase-separated timing, percentile aggregation — show up in every
# later module's measurement code, so we build them right once.

import torch, time, json, numpy as np
from transformers import AutoModelForCausalLM, AutoTokenizer

MODEL_ID   = "nexus/fraud-70b-fp16"
PROMPT_LEN = 6200
OUT_LEN    = 180
WARMUP     = 3
N_TRIALS   = 50

# 1) Load the model. Loading time is not part of the benchmark.
tok   = AutoTokenizer.from_pretrained(MODEL_ID)
model = AutoModelForCausalLM.from_pretrained(
    MODEL_ID, torch_dtype=torch.float16, device_map="auto",
)
model.eval()

# 2) Build a realistic fraud-detection prompt at exactly PROMPT_LEN tokens.
#    Synthetic token ids are fine for throughput/latency measurement — we
#    are not measuring output quality, only wall-clock behavior.
input_ids = torch.randint(100, 30000,
    (1, PROMPT_LEN), device="cuda")

# 3) Warmup. First iterations pay for CUDA context, kernel compilation,
#    KV cache allocation, cuDNN autotuning. Never include them in stats.
with torch.inference_mode():
    for _ in range(WARMUP):
        model.generate(input_ids, max_new_tokens=OUT_LEN,
                       do_sample=False, use_cache=True)
torch.cuda.synchronize()

# 4) Measurement. The key tricks:
#    - torch.cuda.Event gives us GPU-side timing, not Python-side.
#    - We time prefill and decode separately by requesting max_new_tokens=1
#      first (measures prefill+1 token) then OUT_LEN-1 more tokens.
#    - We synchronize before AND after each timed region.
prefill_ms, decode_ms_per_tok, e2e_ms = [], [], []

for _ in range(N_TRIALS):
    e_start  = torch.cuda.Event(enable_timing=True)
    e_ttft   = torch.cuda.Event(enable_timing=True)
    e_end    = torch.cuda.Event(enable_timing=True)

    with torch.inference_mode():
        e_start.record()
        out1 = model.generate(input_ids, max_new_tokens=1,
                              do_sample=False, use_cache=True,
                              return_dict_in_generate=True)
        past = out1.past_key_values
        first_tok = out1.sequences[:, -1:]
        e_ttft.record()

        full = model.generate(first_tok, max_new_tokens=OUT_LEN-1,
                              past_key_values=past, do_sample=False,
                              use_cache=True)
        e_end.record()

    # CUDA operations are async — we MUST wait for completion before reading.
    torch.cuda.synchronize()

    prefill_ms.append(e_start.elapsed_time(e_ttft))
    decode_total = e_ttft.elapsed_time(e_end)
    decode_ms_per_tok.append(decode_total / (OUT_LEN-1))
    e2e_ms.append(e_start.elapsed_time(e_end))

# 5) Report percentiles — means lie about tail behavior.
def stats(xs, name):
    p50, p90, p99 = np.percentile(xs, [50, 90, 99])
    print(f"{name:12s} p50={p50:7.2f}ms  p90={p90:7.2f}ms  p99={p99:7.2f}ms  mean={np.mean(xs):7.2f}ms")

stats(prefill_ms,        "TTFT")
stats(decode_ms_per_tok, "TPOT")
stats(e2e_ms,            "E2E")

# 6) Throughput. Output-token throughput is what vendors quote;
#    for a serving system you also care about request throughput.
total_tokens = OUT_LEN * N_TRIALS
total_sec    = sum(e2e_ms) / 1000.0
print(f"throughput   {total_tokens / total_sec:7.1f} output tok/s (batch=1)")

Running this harness on Nexus v1 on a single H100 produces the numbers from the top of the module. TTFT p99 around 295 ms (that's the prefill roofline being honest with us). TPOT p99 around 22 ms, consistent with HBM bandwidth divided by weight bytes. E2E p99 around 4.2 s, throughput around 50 tokens/second from one request at a time. If you see a vendor claim 4000 tokens/second for a 70B model on a single H100 with batch 1, they are either (a) quoting prefill-only throughput on a long context, (b) counting input tokens, or (c) lying. The physics do not permit it.

05 · The Inference Stack Map

Seven layers, every one optimizable.

Before we dive into individual techniques across the next seven modules, it helps to have a map. Every known LLM optimization operates on exactly one layer of the inference stack. Once you see the layers, every paper you read becomes easier to place — and the techniques combine in predictable ways because they operate on different layers.

The Seven Layers of LLM Inference — and Where Each Module Lives
🔑

How to use this map when you read a new paper

Every inference paper modifies one or two layers. When you see a new technique, ask: which layer does this operate on? FlashAttention is L3. PagedAttention is L4. Speculative decoding is L5. SparseGPT is L2. Once you place the technique on the map, you know three things automatically: (1) which other techniques it composes with (different layers compose cleanly), (2) which techniques it competes with (same layer, same target metric), and (3) how big the win can possibly be (bounded by the gap between current and theoretical performance at that layer).

The counterintuitive H200 story

The H200 has the same peak FP16 compute as the H100 — same tensor cores. What changed is 141 GB of HBM3e at 4.8 TB/s instead of 80 GB of HBM3 at 3.35 TB/s. For a decode-dominated workload like Nexus, this is a 43% throughput improvement from a lower ridge point and 76% more capacity for KV cache — with zero code changes. The H200 is an inference-optimized H100, and that is almost the entire story of the 2024–2026 GPU generation: bandwidth and capacity matter more than peak FLOPs for inference.

The memory hierarchy — why FlashAttention even exists

Inside a single H100 GPU, memory is not homogeneous. There are three tiers, and the bandwidth ratios between them shape every kernel-level optimization in this course.

H100 Memory Hierarchy — Capacity vs Bandwidth Tradeoff

A typical transformer layer, implemented naively, reads the Q, K, V matrices from HBM, computes attention scores as a materialized S = QK^T matrix in HBM (which, at 6,200 × 6,200 × 16 heads × 2 bytes = 1.2 GB per layer, is itself a disaster), applies softmax (another HBM read + write), then multiplies by V. That's six round trips to HBM per layer, 80 layers, every request. FlashAttention rewrites that entire pattern to never materialize the S matrix — it tiles Q/K/V into SRAM-sized blocks and streams the output back to HBM once. That single change is a 3-5× speedup on prefill, and we cover every detail of how it works in Module 3.

For now, the takeaway is structural: memory hierarchy is not just a detail. It is the reason modern attention kernels look nothing like the textbook softmax formula.

07 · Debugging Drills

Four foundational traps you'll walk right into.

The content above gave you the vocabulary. These drills are where vocabulary becomes judgment. Each is a symptom you will see on call — an actual production signal — paired with the wrong hypothesis that wastes hours and the correct one that fixes it. Foundations-level issues are the sneakiest: everything else in the stack depends on these signals being read correctly, so a wrong read here compounds into wasted weeks in later modules.

Drill · 01
"GPU utilization is 95% — we're compute-bound"
SYMPTOM
Team reports nvidia-smi shows GPU util at 92-95%. They conclude the model is compute-bound and plan to upgrade to a faster GPU (H100 → H200). After the upgrade, throughput barely improves.
WRONG HYPOTHESIS
"GPU util = how hard the GPU is working." Not true. nvidia-smi GPU-Util measures whether a kernel is running on ANY SM — not whether the SMs are doing useful work at full throughput.
CORRECT DIAGNOSIS
Check DCGM metric SM_ACTIVE (fraction of SMs doing work) and DRAM_ACTIVE (HBM bandwidth utilization). For a decode-bound LLM, SM_ACTIVE is often 40-60% while DRAM_ACTIVE is 85%+. That's memory-bound, not compute-bound. The roofline model from section 03 tells you which: at arithmetic intensity < op:byte ratio of the hardware, you're memory-bound. Llama-3-70B at batch=1 has arithmetic intensity ~1-2 FLOPs/byte; H100 is 290 FLOPs/byte. Clearly memory-bound.
FIX
For memory-bound workloads, upgrade to faster HBM (H100 → H200 is 1.4× HBM bandwidth, real 30-40% throughput gain) or quantize KV/weights to reduce bytes-per-token. A "faster" GPU with the same HBM bandwidth won't help. Before buying, always classify the workload: compute-bound vs memory-bound. They have different remedies.
Drill · 02
SLO set on p99 latency; p99.9 catastrophic; nobody caught it
SYMPTOM
Team ships with "p99 TTFT < 250ms" SLO. Dashboards green for 3 months. Major tenant complains about occasional 30-second hangs. Investigation finds p99.9 TTFT is 18,000ms.
WRONG HYPOTHESIS
"0.1% is noise — we can't chase every outlier." The 0.1% is where real users suffer. For a service at 10M requests/day, 0.1% is 10,000 users/day experiencing a 30-second hang.
CORRECT DIAGNOSIS
For any SLO involving user-perceived latency, track p99.9 AND a ratio like p99.9 / p99. Healthy = ratio < 2.5. Concerning = 2.5-5. Broken = >5. A p99.9/p99 ratio of 70× (18000ms/250ms) means something catastrophic happens on rare events — almost always preemption, KV pool exhaustion, or a gateway timeout misconfigured to retry indefinitely. None of those are visible on a p99 dashboard.
FIX
Add p99.9 to the SLO dashboard. For a 200ms TTFT service, target p99.9 < 600ms (3× p99). Track p99.9/p99 ratio as an SRE signal — if it's rising, something is breaking, even if p99 still looks fine. For trace investigation, grab the slow requests (p99.9 bucket) and check whether they share a tenant, a region, or a time of day. The pattern tells you the cause.
Drill · 03
"Throughput improved 3× after the new deploy" (but cost went up)
SYMPTOM
Engineering reports 3× throughput gain from the new batch-size config. Latency is within SLO. Finance reports cost-per-million-tokens went UP by 15% over the same window. Engineering is confused.
WRONG HYPOTHESIS
"Finance is measuring wrong — throughput went up, cost must have gone down." Throughput and cost-per-token are different metrics. You can have higher throughput at higher cost-per-token if you're using more expensive hardware less efficiently.
CORRECT DIAGNOSIS
Check what changed alongside the batch config. A common pattern: team raised batch to 128 AND upgraded from 4× H100 to 8× H200. Throughput is 3× (due to hardware + config combined), but cost scaled faster — 2× GPU count × 1.4× cost-per-GPU-hour = 2.8× hourly cost, for 3× throughput = net 7% cheaper per token at best. If the batch change alone was the cause, cost-per-token should drop proportionally. If both changed, decompose the gains. Also check whether GPU utilization on H200s actually rose — if not, the H200 spend isn't pulling its weight.
FIX
Never report throughput improvements without cost-per-million-tokens alongside. The three metrics that tell the truth: p99 TTFT (user-facing), throughput (tok/s) (system), and $/M tokens (economics). Any one alone can mislead. Build a dashboard with all three in a single row. After a deploy, ALL THREE should move in the right direction; if one regresses, understand why before claiming victory.
Drill · 04
Stack trace shows 80% of time in attention; team re-optimizes attention and sees 3% speedup
SYMPTOM
Profiler output (e.g. nsys) shows 82% of decode time in attention kernels. Team spends two weeks swapping to FlashAttention-3 and rewriting scales. Final measurement: 3% TPOT improvement.
WRONG HYPOTHESIS
"Attention dominates wall time; optimize attention for proportional gain." This is Amdahl's law applied backward. If attention is 82% of time and we make it 2× faster, the kernel time drops 50% but total time drops only ~41% — IF attention is actually the bottleneck.
CORRECT DIAGNOSIS
At batch=1 decode, attention dominates wall time in the profiler but the bottleneck is HBM bandwidth, not attention compute. The time in "attention kernels" is largely HBM reads for KV. Upgrading to FlashAttention (a better compute pattern) doesn't help when compute wasn't the bottleneck — it's like optimizing the engine of a car stuck in traffic. Check DCGM: if DRAM_ACTIVE was 90%+ before AND after, the new kernel is reading the same bytes, just reorganized. To actually accelerate, reduce bytes moved (FP8 KV, GQA, MLA) — not the compute pattern.
FIX
Before optimizing any kernel, classify whether it's compute-bound or memory-bound using the roofline. For decode on typical LLMs, it's almost always memory-bound at batch ≤ 32, which means "reduce bytes" is the lever, not "faster compute." FlashAttention helps at batch=256+ where compute starts to matter; at batch=1-8 where most interactive traffic sits, attention-variant swaps give single-digit percentage improvements. The real 10× wins at interactive batch sizes come from quantization (M2) and KV eviction (M4), not from kernel rewrites.
💡

The common thread

All four drills share a fingerprint: a proxy metric is being treated as a ground-truth metric. GPU-Util ≠ SM_ACTIVE. p99 ≠ p99.9. Throughput ≠ cost-per-token. Kernel wall-time ≠ compute boundedness. The foundations-level judgment call is always "does this metric actually measure what I care about?" Ask that question first, pick the right metric, THEN diagnose. Every later module in this course assumes you can do this reliably.

Q1 · Regime identification
A 13B model runs decode on an H100 at batch size 1 in FP16. Is the workload compute-bound or memory-bandwidth-bound, and why?
Compute-bound — the tensor cores dominate the time.
Memory-bandwidth-bound — each token reads all 26 GB of weights once from HBM, but performs only ≈1 FLOP per byte.
Latency-bound — kernel launch overhead dominates.
It depends on sequence length — at 1k context it's compute-bound.
Decode at batch size 1 has arithmetic intensity ≈ 2/D = 1 FLOP/byte for FP16. The H100 ridge point is ≈295. We are deep in memory-bound territory regardless of context length, because the weights — not the activations — dominate bytes moved.

Weight bytes = 405 × 10⁹ × 2 = 810 GB. Time = 810 GB ÷ 8 TB/s ≈ 101 ms. Notice a 405B model will not even fit on a single B200 (192 GB); production serving uses tensor parallelism across multiple GPUs, which splits the bytes across parallel HBM reads.
Q3 · TTFT vs TPOT
A team reports their service has p99 latency of 2.4 s for a chat use case with 100-token prompts and 400-token responses. They want to improve user experience. Which metric should they prioritize reducing, and what should they look at first?
TTFT — users notice the initial delay most.
TPOT — with 400 output tokens at ~6 ms each dominating the 2.4 s, reducing per-token decode cost (quantization, speculative decoding) is where the leverage is.
Queue time — the p99 tail is always queueing.
Both equally — they are always coupled.
Do the arithmetic. 400 output tokens is where the seconds are. A 100-token prefill takes tens of milliseconds even on a 70B. Decode dominates any workload where output length exceeds ~3× input length. The right optimizations (Module 2 quantization, Module 5 speculative decoding) are exactly the ones that move TPOT.
Q4 · Benchmark trap
A colleague's Python benchmark wraps model.generate() in time.time() calls and reports "50 ms average latency" for a 70B model on H100. What is most likely wrong?
Nothing — that's a plausible TPOT.
They are likely not calling torch.cuda.synchronize() — the Python call returns as soon as the kernels are queued, not when they complete. Real wall-clock is much higher.
They forgot to warm up — always a problem but not a 50× error.
time.time() is too imprecise — they should use time.perf_counter().
CUDA kernels are asynchronous. Without a synchronize() call before the second timestamp, you are measuring how fast Python can queue up GPU work, not how long it took to run. This is the single most common benchmarking bug in the field. Use torch.cuda.Event with enable_timing=True for correct GPU-side timing.
Q5 · The batching question
Why does increasing batch size from 1 to 64 dramatically increase throughput of decode, even though the total FLOPs required scales linearly with batch size?
It doesn't — throughput is constant regardless of batch.
The weights are read from HBM once per layer regardless of batch size — so the expensive HBM read is amortized across all 64 requests. Arithmetic intensity rises from 1 to 64 FLOP/byte, moving us rightward along the roofline.
Tensor cores are idle at batch 1 — batching wakes them up.
CUDA kernels have a minimum fixed overhead that gets amortized.
This is the core insight of modern serving. A weight matrix is fetched from HBM once and used to process every request in the batch. At batch 1, each byte of weights does 1 FLOP. At batch 64, each byte does 64 FLOPs. That moves you rightward on the roofline chart until you approach the ridge — which is why continuous batching (Module 5) is the single highest-leverage technique in all of LLM serving.
Q6 · Hardware selection
For a workload of 70B decode at batch size 32, which single-GPU option gives the best tokens/sec per dollar assuming equal hourly rates?
H100 80GB — proven software, best ecosystem.
H200 141GB — same compute as H100 but 43% more bandwidth and capacity for KV cache, so it can run larger batches and has lower TPOT.
A100 80GB — cheaper per hour so better $/token.
B200 192GB — the highest peak FLOP/s wins.
For decode-heavy workloads, the binding constraint is HBM bandwidth and capacity. The H200 is the H100 with more and faster memory — the compute is the same. For 70B decode, that is exactly the right tradeoff. B200 helps too but at a much higher price point; its FLOP advantage is wasted on memory-bound workloads.
Q7 · Prefill vs decode scheduling
A serving system running continuous batching observes that TTFT spikes when many long-prompt requests arrive together, even though decode throughput stays stable. What is happening?
The GPU is out of memory.
Prefill is compute-bound; when many prefills arrive together, they serialize on the tensor cores and later-arriving requests wait. This is why chunked prefill and disaggregated prefill/decode (Module 4) exist.
The scheduler is buggy.
KV cache eviction is slow.
Prefill and decode compete for the same SMs but have different resource profiles. A big prefill can starve several decode steps, spiking TTFT for everyone waiting. Chunked prefill breaks large prefills into smaller chunks that interleave with decode; disaggregated serving puts them on separate GPU pools entirely. Both are covered in depth in Module 4.
Q8 · The ridge point
The ridge point of a GPU (peak FLOP/s ÷ HBM bandwidth) tells you:
The maximum batch size the GPU can handle.
The minimum arithmetic intensity needed to saturate HBM.
The arithmetic intensity above which the workload transitions from memory-bound to compute-bound. At AI below this point, HBM bandwidth is the limit; above it, tensor core FLOP/s is the limit.
The efficiency of the hardware.
The ridge point is the x-coordinate where the two roofs meet on a roofline chart. For an H100 at 990 TFLOP/s and 3.35 TB/s HBM, ridge ≈ 295 FLOP/byte. Any workload below that intensity is HBM-bandwidth-limited; any workload above it is tensor-core-limited. Knowing your workload's intensity and the hardware's ridge tells you instantly which regime you are in — and therefore which category of optimization is worth pursuing.
