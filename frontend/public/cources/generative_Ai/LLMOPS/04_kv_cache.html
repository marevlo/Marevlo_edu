01 · The KV Cache, Precisely

What we actually store per request.

Before we can talk about paging and prefix caching, we need to pin down what the KV cache is — with concrete byte counts for Nexus. A single 6,200-token request to a 70B model with 80 layers and 8 KV heads (GQA-8) allocates enough KV cache memory to shock a first-time engineer. Multiply by concurrent requests and you understand why KV memory is the dominant serving constraint.

The per-request cache, element by element

KV cache memory — one request, one layer kv_bytes = 2 (K and V) × n_kv_heads × head_dim × seq_len × dtype_bytes
total = kv_bytes × n_layers

For Nexus (Llama-3-70B style: 80 layers, 8 KV heads after GQA, head_dim 128, FP16 KV, 6,200-token context):

Nexus — one request, full context kv_per_layer = 2 × 8 × 128 × 6,200 × 2 bytes = 25.4 MB
kv_total = 25.4 MB × 80 layers ≈ 2.03 GB per request

Two gigabytes. For one request. With only GQA-8 and a modest 6k context. A Llama-3-70B without GQA (using 64 KV heads = MHA) would balloon that to 16.3 GB per request, which is why Llama-3 didn't ship without GQA.

Now extend the thought experiment: Nexus's SLO requires 10,000 requests per second across the region, each completing in under 80 ms. That's a steady state of roughly 800 concurrent in-flight requests per node. At 2 GB of KV per request, that's 1.6 TB of KV memory — twenty H200s worth of HBM, just for KV cache. Obviously no one actually pre-allocates that much. The question this module answers is: how do we actually serve 800 concurrent requests on one or two GPUs?

The fragmentation that every naive implementation suffers

Vanilla transformers.generate() allocates a contiguous KV tensor sized for max_seq_length per request — because the attention kernel reads the cache as a tensor of known shape. If max_seq_length is 8k but the request only produces 180 tokens, 97% of that allocation sits empty for the request's entire lifetime. Multiply by hundreds of concurrent requests and the waste eclipses the weights themselves. This is the problem PagedAttention solves — and the solution comes straight from 1960s operating system research.

02 · PagedAttention · Virtual Memory for KV

Break it into blocks, store a table.

PagedAttention (Kwon et al., vLLM, 2023) is the single most important serving-system innovation of the past three years. The idea is disarmingly simple: take an OS virtual-memory textbook, change "page" to "block" and "process" to "request", and implement it in CUDA. The concurrency improvements are 5-20×.

The mental model: KV cache as virtual memory

In an operating system, a process's address space is logically contiguous (0 to 2⁶⁴) but physically fragmented — the OS maps virtual pages to arbitrary physical frames via a page table. New pages are allocated on demand. Freed pages return to the global pool. Two processes can share a physical frame via shared-memory mappings. Each process thinks it has a clean contiguous address space; the hardware MMU translates on every access.

PagedAttention applies this model one level up: each request's KV cache is logically contiguous (token 0 to token N), but physically it is a list of fixed-size blocks scattered across HBM. A per-request block table maps logical positions to physical block addresses. The attention kernel learns to do the translation on the fly — a single extra indirection per block.

PagedAttention · Block Tables and Physical Layout

PagedAttention in (lightly simplified) code

paged_attention_kernel.py · the core ideaPYTHON · CUDA-STYLE PSEUDOCODE
# Simplified PagedAttention attention kernel. The production vLLM kernel
# is CUDA C++ with heavy tiling; this shows the essential indirection logic.

def paged_attention(
    q,                   # query [batch, n_heads, head_dim]              (decode: 1 token at a time)
    k_cache_pool,        # physical KV blocks [n_blocks, n_kv_heads, block_size, head_dim]
    v_cache_pool,        # physical V blocks  [n_blocks, n_kv_heads, block_size, head_dim]
    block_tables,        # per-request block lists [batch, max_blocks_per_req]
    context_lens,        # actual context length per request [batch]
    block_size=16,
):
    output = torch.zeros_like(q)

    for b in range(q.shape[0]):               # per request
        blocks  = block_tables[b]
        ctx_len = context_lens[b]
        n_blocks = (ctx_len + block_size - 1) // block_size

        # Accumulators for numerically stable online softmax.
        m = float('-inf'); l = 0.0
        acc = torch.zeros_like(q[b])

        for blk_idx in range(n_blocks):
            phy_id = blocks[blk_idx]                   # translate logical → physical
            start  = blk_idx * block_size
            end    = min(start + block_size, ctx_len)
            length = end - start

            # Gather K and V for this block from the shared pool.
            K_blk = k_cache_pool[phy_id, :, :length]    # [n_kv_heads, length, head_dim]
            V_blk = v_cache_pool[phy_id, :, :length]

            # Standard attention sub-step for this block (running softmax).
            scores = (q[b].unsqueeze(1) @ K_blk.transpose(-1,-2)) / math.sqrt(head_dim)
            # [n_heads, 1, length] (assuming GQA broadcast handled)

            # Update running max + denominator (online softmax, from Module 3).
            block_max  = scores.amax(dim=-1)
            m_new      = torch.maximum(m, block_max)
            alpha      = torch.exp(m - m_new)
            beta       = torch.exp(block_max - m_new)
            p          = torch.exp(scores - m_new.unsqueeze(-1))
            l          = l * alpha + beta * p.sum(dim=-1)
            acc        = acc * alpha.unsqueeze(-1) + (p @ V_blk)
            m          = m_new

        output[b] = acc / l.unsqueeze(-1)

    return output
💡

The beautiful composition with FlashAttention

Notice the inner loop uses online softmax — exactly the FlashAttention identity we derived in Module 3. PagedAttention and FlashAttention compose cleanly because they operate on different axes: FlashAttention tiles within a single sequence for SRAM efficiency; PagedAttention tiles across sequences for memory efficiency. Modern kernels (vLLM's "PagedAttention v2", SGLang's flash_paged) fuse both patterns into one kernel. The result is what every production system runs today.

Block size — the one tuning knob that matters

PagedAttention has very few knobs. The main one is block_size — how many tokens live in one physical block. The tradeoff is sharp:

Block sizeInternal fragmentationKernel efficiencyMetadata overheadRecommendation
4 tokens< 1% wastePoor — too many blocks per seq, lots of indirectionHigh — 4× more block table entriesAvoid
16 tokens~2% wasteGood balanceLowvLLM default — best general-purpose
32 tokens~3% wasteSlightly better GEMM utilizationVery lowGood for long-context workloads
128 tokens~8-12% waste for short sequencesBest GEMM utilizationMinimalOnly for workloads with uniformly long sequences

The rule: pick block_size ≈ expected request length / 100, rounded to 16 or 32. For Nexus's 6,200-token average, block_size=32 is marginally better than the default 16. For short-chat workloads (average 200 tokens), block_size=16 is clearly correct.

03 · RadixAttention · Share the Prefix

Why every request with the same system prompt should pay once.

In real production workloads, requests are not independent. A chatbot service has the same 2,000-token system prompt on every call. A RAG service has common document chunks. A code assistant has identical instruction templates. For Nexus, the 2,100-token system prompt appears in every single one of the 10,000 requests per second. Recomputing its KV cache for each request is pure waste — identical input, identical output, identical bytes.

The prefix problem

PagedAttention solves the fragmentation problem: within a single request, use memory efficiently. It does not — on its own — solve the sharing problem: across requests with common prefixes, compute once, read many times. That requires a second layer: prefix caching.

The breakthrough here is RadixAttention (Zheng et al., SGLang, 2024). It builds an automatic in-memory trie — a radix tree — keyed on the sequence of tokens produced so far. When a new request arrives, its token sequence is matched against the trie. The longest common prefix is already in HBM from previous requests; we reuse those KV blocks directly and only compute KV for the new suffix.

RadixAttention · Automatic Prefix Sharing via Radix Trie
2,100 tokens × 80 layers × 25.4 MB/layer ÷ savings/req ≈ 34 MB saved per request from prefix hit Prefix hit rate measured on Nexus traffic: 68% average (system + merchant category match). TTFT drops from 275 ms to 72 ms on hits. Net throughput improvement: ~2.3× at peak load, almost entirely from not recomputing prefill KV for shared prompts.

The LRU eviction policy and refcounting

The trie grows as new prefixes arrive. HBM is finite. Two key mechanisms make the system self-regulating:

Reference counting

Each trie node tracks how many in-flight requests currently depend on it. While the refcount > 0, the node's KV blocks cannot be evicted. When a request finishes, it decrements refcounts up the path. Nodes with refcount 0 move to a cold list but are not immediately freed — they are kept as long as HBM allows, to serve future matches.

LRU with prefix awareness

When we need to evict, we pick the least-recently-used leaf node (not an internal node — evicting an internal node would orphan its descendants). In practice this means highly-shared prefixes stay resident forever (the Nexus system prompt is never evicted), while per-transaction tails turn over rapidly. The policy is nearly optimal for most LLM workloads.

The byte-stability rule that catches teams out

For prefix caching to work, the system prompt must be byte-for-byte identical across requests — because the trie is keyed on token IDs. Injecting a timestamp, request ID, or user ID into the system prompt (even for logging) silently destroys the hit rate: every request walks a different path. The fix is to put variable content after the stable prefix, not inside it. "Current date: {date}" as the first line of your system prompt is one of the most common footguns in production LLM serving. We ran into this with Nexus v3 and lost a week to it; budget accordingly.

04 · Disaggregated Prefill / Decode

Two phases, two hardware pools.

Module 1 established that prefill is compute-bound and decode is memory-bandwidth-bound. They are different workloads running on the same hardware — and when they run together, they starve each other. A long prefill blocks every decode step for hundreds of milliseconds; a batch of decode requests leaves tensor cores idle when a prefill would otherwise use them. The final architectural move is to stop running them on the same GPUs at all.

The cohabitation problem, visualized

Cohabitated vs Disaggregated Serving — The TTFT / TPOT Tradeoff

The KV transfer question

Disaggregation sounds simple until you realize: after the prefill pool finishes a request's prefill, the entire KV cache (for Nexus, ~2 GB) has to move from prefill GPUs to decode GPUs before the first decode step can run. That transfer happens on every single request. If it takes 100 ms, you just burned the entire latency budget.

The engineering that makes disaggregation viable is the KV transport layer. Three options, in order of performance:

TransportBandwidthTransfer time (2GB)Use case
NVLink (same node)900 GB/s~2.3 msPrefill and decode pools on same physical host
RDMA InfiniBand / RoCE50 GB/s (400 Gb NIC)~40 msPools on different nodes, tight datacenter
TCP/IP5-12 GB/s160-400 msDon't. Never. Not production.
⚠

When disaggregation is NOT worth it

Disaggregation pays for itself when prefill:decode compute ratio is heavy in one direction and the workload volume justifies separate hardware. For Nexus at 10k req/s with 6,200-token prompts and 180-token responses, disaggregation is obvious. For a chat service at 500 req/s with 50-token prompts, the KV transfer overhead exceeds the scheduling win and a well-tuned cohabitated system with chunked prefill (next section) is strictly better. Rule of thumb: disaggregate when your region has > 4 GPUs serving the same model and prefill:decode compute ratio is > 3:1 or < 1:3.

05 · Chunked Prefill · The Middle Path

Slice the prefill, interleave with decode.

If you can't afford disaggregation but want smoother TTFT/TPOT than cohabitation provides, chunked prefill (introduced in DeepSpeed-FastGen, 2024) is the production-ready middle path. Instead of processing all 6,200 prefill tokens in one forward pass, split them into chunks of 512 or 1024 tokens and interleave chunks with ongoing decode steps.

The chunk size math

Prefill arithmetic intensity scales with chunk size. At chunk size 1, you are in decode regime (memory-bound). At chunk size 8,192, you are deeply compute-bound (near peak FLOP/s). Pick the smallest chunk size that keeps you in compute-bound regime — typically 512-2,048 on H100. A chunk of 1,024 tokens at hidden dim 8,192 has arithmetic intensity ≈ 600 FLOP/byte, comfortably past the H100 ridge point of 295. Anything smaller wastes compute; anything larger adds decode stutter without improving throughput.

06 · SnapKV, H2O, and the Eviction Question

Can you just throw some KV away?

Everything so far has been about storing the KV cache more efficiently. But there's a more radical option: store less of it. Attention is sparse in practice — most positions at decode time attend heavily to a small number of earlier positions (often the first few tokens plus the most recent window). If you can identify which KV entries matter and evict the rest, you free up HBM at a small accuracy cost.

The two families of KV eviction

H2O — Heavy-Hitter Oracle (2023)

During generation, maintain a running accumulation of each token's received attention weight. Tokens with cumulatively high weight are "heavy hitters" — they keep getting attended to — and we keep them. Tokens with low cumulative weight are evicted when KV budget runs low.

Result: often retains accuracy within 1-2 perplexity points while cutting KV by 50-80%. Works well for long-generation workloads where most of the prompt becomes irrelevant after a few tokens.

SnapKV — Observation Window (2024)

Before starting decode, run the prefill forward pass as normal. Look at the attention weights of the last ~64 tokens of the prefill (the "observation window") to each earlier position. Keep only the KV entries that this observation window attends strongly to; discard the rest.

Result: often 90%+ KV reduction on long-prompt RAG workloads with minimal accuracy loss. The observation window turns out to be a remarkably good predictor of what decode will need.

⚠

When NOT to evict KV

Eviction methods are irreversible. If the decoder decides at token 400 that it really wanted to look back at token 20 — and you evicted token 20 — the output silently degrades. For tasks with long reference-and-retrieval behavior (legal analysis, code completion against large files, multi-hop reasoning over documents), KV eviction can cost meaningful accuracy. For Nexus, where output is 180 tokens of structured JSON whose content is fully determined by the input fields, eviction is a good bet and we adopt SnapKV-style compression. For a long-form reasoning assistant, we wouldn't.

The attention sink discovery

StreamingLLM (Xiao et al., 2023) made an important empirical observation: the first few tokens of the KV cache receive disproportionately large attention weights, regardless of content. These are called attention sinks. The softmax has to distribute probability mass somewhere; in well-trained LLMs it learns to dump "residual attention" onto the initial tokens because they are always present.

The practical consequence: if you want to implement a sliding-window KV cache (keep only the last N tokens), you must also always keep the first 4 tokens. Drop them and the model's output degrades catastrophically. Any production long-context system with KV eviction keeps sinks permanently resident. It's a one-line fix that rescued sliding-window decoding as a viable technique.

07 · Offloading · Tier Your Memory

HBM, DRAM, NVMe — the three-tier KV store.

Even with PagedAttention and prefix caching and quantization, some workloads have KV footprints that just don't fit in HBM. Multi-turn chat with 128K context. Agent loops with minute-long histories. Document analysis with long annotations. For these, the final trick is to treat HBM as the cache and DRAM or NVMe as the backing store.

The three-tier memory hierarchy

TierTypical capacityBandwidth to GPUAccess latencyCost vs HBM
HBM (on-GPU)80–192 GB3.35–8 TB/s< 1 μs1× (baseline)
Host DRAM (PCIe / NVLink-C2C)512 GB – 2 TB32–900 GB/s~10 μs0.03×
Local NVMe (PCIe 5.0)4–64 TB14 GB/s~50 μs0.002×
Network storage (RDMA)Effectively unlimited50 GB/s~100 μs0.001×

The offload decision — when it works, when it doesn't

Offloading KV to host DRAM is viable only when the access pattern leaves enough idle GPU time to hide the transfer latency. Two cases where this works well:

✓ Case 1: Multi-turn chat

Between user turns, the conversation sits idle for 1-30 seconds (a human is reading/typing). During that window, we have ~10⁸ μs of time. Moving the KV for one conversation (~500 MB) to DRAM and back costs ~30 ms — invisible. Reload on next turn. This is how ChatGPT-style services serve context windows that no single GPU could hold.

✓ Case 2: Speculative prefix loading

For RAG, you know which document chunks might be needed before the request actually arrives. Pre-populate their KV cache in DRAM, move only the relevant ones to HBM on arrival. Same idea as OS page cache: speculate + prefetch.

⚠

When offloading kills your SLO

For a continuously-active request — decode happening right now, every millisecond — offloading is a disaster. Moving 2 GB over PCIe 5.0 at 64 GB/s takes ~32 ms. Your 5 ms TPOT just became 37 ms. Never offload a request that is currently decoding. Offload only (a) between requests that share tenant context (sessions), or (b) speculatively before requests arrive. For Nexus, which has no cross-request session state, offloading has no role and we skip it in v4. For a chat assistant, it's essential.

08 · Core Operational Math

The formulas for block pools, prefix hits, and chunk sizes.

The conceptual material is in place. This section gives the formulas that turn KV engineering into a sizing exercise you can complete in an RFC rather than a benchmark cycle. Four formulas, four worked examples, four verdicts.

Formula 1 · Block pool sizing

Given a GPU, a model, and a max concurrency target, how many blocks do you allocate in the PagedAttention pool? This is the first number in any KV deployment spec.

Block pool bytes · tokens · count pool_bytes = hbm_bytes − weight_bytes − activation_reserve
  activation_reserve ≈ 3 − 6 GB for safety (varies by TP and model)

tokens_in_pool = pool_bytes / kv_bytes_per_token
num_blocks = tokens_in_pool / block_size

max_concurrency = tokens_in_pool / max_tokens_per_request

Worked example · Llama-3-70B AWQ + FP8 KV on H200 (141 GB HBM)

pool = 141 − 37 (AWQ weights) − 5 (activation reserve) = 99 GB. With FP8 KV (160 KB/token for Llama-3-70B GQA-8), tokens_in_pool = 99 GB / 160 KB ≈ 649k tokens. With block_size=32 that's 20,280 blocks. At max 8K tokens per request, max_concurrency ≈ 81. Matches the Nexus v4 observed ~80 concurrent 6.2K-requests-per-H200. The paged_kv_simulator.py artifact computes this for any model/dtype combination.

Formula 2 · PagedAttention fragmentation

The cost of over-reserving HBM per block. This is the number that decides between block_size 16, 32, or 64.

Expected fragmentation under a request-length distribution For a set of requests with lengths S₁, S₂, ..., Sₙ:
  total_tokens = Σ Sᵢ
  total_blocks = Σ ⌈Sᵢ / block_size⌉
  allocated_slots = total_blocks × block_size
  fragmentation = (allocated_slots − total_tokens) / allocated_slots

Rule of thumb: fragmentation ≈ block_size / (2 × mean_request_length)

Worked example · chat workload with mean 1,200 tokens

fragmentation ≈ 32 / 2400 ≈ 1.3% (block=32). 64 / 2400 ≈ 2.7% (block=64). 128 / 2400 ≈ 5.3% (block=128). On 99 GB of pool, that's 1.3 GB, 2.7 GB, or 5.3 GB of missing KV capacity — the difference between serving 80, 75, or 70 concurrent requests. Block size=32 wins by this arithmetic, and the paged_kv_simulator.py verdict confirms it.

Formula 3 · Prefix cache hit-rate ceiling

The maximum hit rate achievable on your workload, given the prompt template you ship. This is a property of your prompts, not your engine.

Theoretical hit-rate ceiling (chars-based approximation) For a log of N prompts, at prefix length L chars:
  unique_prefixes = |{ p[:L] : p ∈ prompts }|
  max_hit_rate = (N − unique_prefixes) / N

A healthy system has unique_prefixes << N at L ≈ 512 chars
(i.e., most requests share an identical first 512 chars with some earlier request).

Worked example · production prompt log

10,000 sampled prompts · L=512 chars → 142 unique prefixes. max_hit_rate = (10000 − 142) / 10000 = 98.6%. If the engine reports hit rate of 85% but the ceiling is 98.6%, the gap is scheduler eviction under pressure, not a prompt-template problem. If the engine reports 85% and the ceiling is 87%, the gap is prompt-template and the scheduler is fine. radix_prefix_analyzer.py tells you which world you're in.

Formula 4 · Chunked prefill TTFT decomposition

The relationship between chunk size, TTFT, and decode stall. Deriving this is the only way to pick a chunk size without benchmarking.

TTFT and decode-stall under chunked prefill n_chunks = ⌈prefill_length / chunk_size⌉
chunk_time_ms = (chunk_size / prefill_throughput_tok_per_sec) × 1000 + launch_overhead_ms
TTFT_ms ≈ n_chunks × chunk_time_ms + tpot_ms
max_decode_stall_ms = chunk_time_ms    (a concurrent decode gets blocked for one chunk)

Goal: minimize max_decode_stall_ms subject to TTFT_ms ≤ budget.

Worked example · Nexus v4 config

TTFT budget = 400 ms, prefill p99 = 4,096 tokens, TPOT = 12 ms, prefill throughput = 20,000 tok/s, launch overhead = 0.6 ms. At chunk_size=512: 8 chunks × (25.6 + 0.6) ms + 12 ms = 222 ms TTFT. Decode stall per chunk = 26.2 ms. Both inside budget — this is why the Nexus v4 config runs chunk_size=512. At chunk_size=1024, TTFT drops to 215 ms but decode stall jumps to 51 ms, which breaks the TPOT SLO for concurrent requests. At chunk_size=256, decode stall drops to 13 ms but TTFT rises to 231 ms. 512 is the sweet spot. chunked_prefill_calculator.py runs this for any budget.

🔑

The overall framing

KV engineering is fundamentally an allocation problem. Every mechanism in this module (PagedAttention, RadixAttention, disaggregation, chunked prefill, offloading) is a different answer to "how do I allocate scarce KV memory to maximize useful work?" The four formulas above are the numeric backbone of that allocation decision. When deploying, run the artifacts first, read the verdicts, and the config almost writes itself. When debugging, the formulas tell you which number is off and by how much — which is the first step to knowing what to fix.

10 · Debugging Drills

Five symptoms you'll see. Five diagnostic paths.

Five drills covering the most common KV-related on-call symptoms: hit-rate collapse, preemption storms, decode stalls from prefill, fragmentation-driven OOMs, and disaggregation causing latency spikes. Each drill names the symptom, the wrong hypothesis that wastes hours, the exact diagnostic path, and the fix.

Drill · 01
Prefix cache hit rate collapsed overnight
SYMPTOM
Yesterday's hit rate was 78%. Today's is 4%. TTFT p99 tripled. Infra dashboards are all green. No deploy in the last 24h on your team. Alerts for GPU util / memory / error rate are silent.
WRONG HYPOTHESIS
"The engine must be broken — restart the pods." Restart does nothing because the cache is built from incoming prompts, and those are the problem.
CORRECT DIAGNOSIS
Pull 1,000 recent prompts from the request log. Run radix_prefix_analyzer.py on them. If hit rate at L=64 is ~100% but at L=256 it's ~0%, something was added between chars 64 and 256. Diff today's first-100-char prompts against yesterday's — the delta is almost always (a) a timestamp, (b) a request UUID, (c) a session token, or (d) an A/B flag. Check with upstream teams: "did anyone prepend anything to the system prompt recently?"
FIX
Refactor the prompt template to move the variable content AFTER the stable prefix (so it's outside the first 512 chars). If that's not possible immediately, bypass prefix caching for the broken template while upstream fixes it: set enable_prefix_caching: false in the affected tenant's config. Expect hit rate to recover within ~10 minutes of the template fix.
Drill · 02
Preemption rate spiked to 18%; tail latency unstable
SYMPTOM
vllm:num_preemptions_total growing fast. P99 TPOT spiky — sometimes 15 ms (normal), sometimes 90 ms (catastrophic). KV cache usage reading 95-98%. Request queue building.
WRONG HYPOTHESIS
"Throughput is fine, the scheduler is just making some unlucky choices — ride it out." It will not stabilize on its own. Preemption storm is a feedback loop.
CORRECT DIAGNOSIS
Preemption happens when the KV pool can't fit all the in-flight requests. Two possible causes: (a) max_num_seqs is set higher than your actual KV budget supports (over-provisioned scheduler), or (b) request lengths have drifted longer than expected (traffic mix changed). Check vllm:gpu_cache_usage_perc history — if it was 70% last week and is 95% now with same max_num_seqs, traffic drifted. Check a recent prompt-length histogram; if p99 grew, you need to reduce max_num_seqs or reduce max_model_len.
FIX
Immediate: reduce max_num_seqs by 20% to cap concurrency below the preemption threshold — this stops the feedback loop. Then: run paged_kv_simulator.py with the new prompt-length distribution to find the right max_num_seqs for current traffic. Longer-term: if the traffic shift is durable, upgrade to a bigger GPU (H200) or add tensor-parallelism to split weights across more GPUs and free KV budget.
Drill · 03
TPOT p99.9 is 5× p99 — decode is stalling intermittently
SYMPTOM
TPOT p99 = 12 ms (green). TPOT p99.9 = 60 ms (amber). The ratio is 5:1 instead of the healthy ~1.5:1. Happens in bursts, roughly correlating with the arrival of long-prompt requests.
WRONG HYPOTHESIS
"A few slow requests are always expected — tail is noisy." The 5× p99.9/p99 ratio is a specific fingerprint of interleaved-phase stalls, not generic noise.
CORRECT DIAGNOSIS
When enable_chunked_prefill: true and prefill chunks are too large, each chunk blocks decode for its full duration. The stall duration per event = chunk_time. Check your prefill chunk size vs your TPOT budget: if max_num_batched_tokens_prefill_chunk × (1/prefill_throughput) > TPOT target, you'll see this pattern. Confirm via nsys profile: look for gaps in the decode kernel timeline when a new prefill arrives.
FIX
Run chunked_prefill_calculator.py with your current workload and TTFT/TPOT targets. If it recommends a smaller chunk, apply it. If it reports "no chunk size works," you've hit the wall of chunked prefill and need to disaggregate — but first run kv_transport_benchmark.py to verify your network can transport KV in <5ms. If it can't, the only remaining options are (a) reduce max_model_len to cap p99 prefill size, or (b) route long prompts to a dedicated pool.
Drill · 04
Disaggregated P/D deployed; latency got worse
SYMPTOM
Rolled out disaggregated prefill/decode per the design doc. Prefill pool has 98% GPU utilization (great); decode pool has 75% (great). But E2E p99 went up 40%, not down. Nobody can explain why.
WRONG HYPOTHESIS
"Maybe the decode pool is too small — add more nodes." Wastes money and doesn't fix anything because the bottleneck isn't decode capacity.
CORRECT DIAGNOSIS
The KV transport between pools is slower than expected. Measure it directly: run kv_transport_benchmark.py on the actual network between the two pools. If you're getting the "expected" bandwidth of the wire (say 100 Gbps) but actual measurements show 8 Gbps, there's a middlebox, a misconfigured MTU, or TCP overhead dominating. Check: iperf3 across the pool endpoints; verify RDMA is actually being used (look for RDMA counters in DCGM output); check that the KV transport path isn't traversing the public internet (M8 war story #5).
FIX
If the transport is slower than NVLink / RDMA / PCIe 5.0, disaggregation is not viable on this network. Options: (a) physically co-locate the pools in the same rack with RDMA, (b) revert to cohabitated with chunked prefill (often acceptable), or (c) shrink the KV being transferred — lower KV precision to FP8 if not already, or migrate to MLA which sends ~10× less data.
Drill · 05
OOM on a GPU that "should fit" — the sizing math said it would
SYMPTOM
Your back-of-envelope KV math said 80 concurrent requests × 2.5 GB per request = 200 GB, within your 2× H200 = 282 GB budget. Engine starts, serves requests, then crashes with CUDA OOM when batch gets to 60-70. Restart, same pattern.
WRONG HYPOTHESIS
"The engine has a memory leak — open an issue and restart hourly." Almost never a leak; there's usually a reservation you forgot.
CORRECT DIAGNOSIS
You forgot (one or more of): weight memory (30-150 GB depending on quantization), activation reserve during prefill (3-6 GB), CUDA graph capture buffers (can be 4-8 GB for large batches), overhead from the block pool itself (~5% of pool bytes for the page-table and ref counts). Re-run paged_kv_simulator.py or kv_sizing_calculator.py from M3 with the real weight-dtype and correct activation reserve. The likely culprit: you computed KV for max_seq_len requests but weight memory was larger than budgeted.
FIX
Reduce one of: max_num_seqs (lower concurrency ceiling), max_model_len (shorter requests possible), or add tensor parallelism (TP=4 instead of TP=2 splits weights across more GPUs). After change, check engine startup logs — vLLM reports "total KV cache size" and "max concurrency" right at startup. If those numbers match your expectation, you're safe. If they're lower, the engine's internal math disagrees with yours — trust the engine's number.
11 · Failure Reproduction

Watch prefix caching collapse in your terminal.

The artifact pack includes break_prefix_cache.py — a no-GPU script that reproduces the M7 war story #1 failure at the prompt level. Reading about a hit rate drop is one thing; watching 99.9% collapse to 0% because of one embedded timestamp is what makes the lesson stick.

What the script does

Generates 1,000 fraud-detection prompts in two scenarios:

Scenario A · Stable

The system prompt is byte-identical across all 1,000 requests. Only the transaction data (which lives after the prefix window) varies. Simulated radix-trie hit rate: 99.9%.

Scenario B · Broken

Same prompt, but with "The current time is {timestamp}" injected into the system prompt. Every request has a different timestamp, so no two prompts share a prefix beyond the first ~55 characters. Simulated hit rate: 0.0%.

Mapping to real symptoms

In production, prefix cache hit rate doesn't usually drop from 99% to 0% in one step — it's more like 78% to 4% over 20 minutes as a staged rollout propagates. But the mechanism is identical. The three fingerprints that identify this class of bug:

1. Hit rate dropped suddenly

From >70% to <10% in under 30 minutes. Gradual drift is a different problem (scheduler eviction under pressure).

2. Infra is green

GPU, memory, error rate all normal. The only metric moving is hit rate itself and whatever lags it (TTFT).

3. Shallow prefix still hits

radix_prefix_analyzer.py at L=64 shows ~100%, at L=256 shows ~0%. The breaking character is between them.

💡

The design principle

The fix isn't specific to timestamps. It's the general principle: the first N characters of your system prompt must be byte-identical across all production traffic. Every piece of variable content lives AFTER that stable prefix. This constraint is easy to enforce at prompt-template design time and catastrophically hard to retrofit once upstream teams start embedding things. Enforce it in a prompt-template linter; run break_prefix_cache.py semantically against any proposed prompt change before it ships.

Fourteen items before you ship KV changes.

Full checklist lives in artifacts/m04_kv_cache/operator_checklist.md. Inline summary below. Each item corresponds to a failure mode documented in a real postmortem.

Pre-deploy verification · 14 items
  • Block size validated on your workload. paged_kv_simulator.py recommendation matches deployed value.
  • Prefix cache hit rate >80% at L=512 on a 1,000-prompt sample, no timestamp/UUID violations.
  • Chunked prefill chunk_size tuned by chunked_prefill_calculator.py for your TTFT budget and decode-stall target.
  • KV dtype taking effect. Measured KV ≈ expected bytes for your dtype (FP8 = half of FP16).
  • KV calibration file matches your traffic. Not the default Wikipedia calibration.
  • KV transport benchmarked if disaggregating. <5ms per request is safe; >15ms is a trap.
  • Preemption rate below 5% under realistic load. Higher = capacity issue, not scheduler issue.
  • Per-tenant trie keying enabled for regulated tenants (PCI, HIPAA, SOC 2).
  • KV pressure alerts configured. Three alerts: cache usage, hit rate, preemption count.
  • CUDA graph capture sizes cover expected batch distribution including rare sizes.
  • Chunked prefill compatible with speculative decoding, TP, and prefix caching when all are enabled together.
  • Block pool sizing verified against engine startup log.
  • Rollback rehearsed. One-command rollback completes in <90s.
  • Monitor 4 metrics for 24h post-deploy: prefix hit rate, cache usage, TTFT p99, preemption rate.
13 · Nexus v4 Checkpoint

The KV stack, fully deployed.

We've added four mechanisms to the serving engine: PagedAttention for fragmentation-free allocation, RadixAttention for prefix sharing, disaggregated prefill/decode for phase specialization, and chunked prefill as a backup for cohabitated workloads. For Nexus at 10k req/s with heavy system-prompt sharing, the stack is: PagedAttention everywhere, RadixAttention for the Nexus system prompt and merchant profiles, disaggregated pools at 2 prefill nodes + 6 decode nodes per region. No offloading needed (no session state). No aggressive KV eviction (accuracy-critical domain).

▼ Nexus v4 · KV engineering complete

Concurrency jumps 5×. Throughput more than doubles. TTFT drops to SLO-comfortable.

We deployed vLLM with PagedAttention (block_size=32, tuned for Nexus's longer sequences), enabled RadixAttention-equivalent prefix caching (~68% hit rate observed on production traffic — system prompt + merchant profile), and split the cluster into a compute-optimized prefill pool (2× H100, FP16) and a bandwidth-optimized decode pool (6× H200, AWQ W4). The KV transfer between pools runs over RDMA InfiniBand (40 ms for 2 GB). Chunked prefill is enabled as a fallback for off-peak hours when disaggregation is not worth the coordination overhead.

Nine questions to lock it in.

Module 4 · Self-check

Click answers to reveal explanations.

Q1 · KV cache size
For a 70B model with 80 layers, 8 KV heads (GQA), head_dim 128, FP16, at 8,192 tokens, what is the per-request KV cache size?
~200 MB
~2.7 GB
~10 GB
~38 GB
Per layer: 2 × 8 × 128 × 8,192 × 2 bytes ≈ 33.5 MB. Across 80 layers: 2.68 GB. This is why KV memory dominates serving economics at long context — and why GQA (8 KV heads instead of MHA's 64) was critical to make 70B models deployable.
Q2 · PagedAttention premise
What problem does PagedAttention solve that GQA alone does not?
KV compression along the head dimension.
Memory fragmentation — naive allocation reserves max_seq_len per request, wasting HBM. PagedAttention stores KV in fixed-size blocks scattered across HBM, with per-request block tables, eliminating waste on unused sequence positions.
Faster attention kernels.
KV cache quantization.
GQA and PagedAttention operate on different axes. GQA reduces the per-token size of KV. PagedAttention reduces over-allocation waste at the request level. They compose: production engines use both.
Q3 · Block size
A team's vLLM deployment has sequences averaging 400 tokens but max_seq_len of 32k. They set block_size=128 for "better GEMM efficiency." What's the problem?
128 is too small — use 1024.
At 400-token average with 128-token blocks, last-block internal fragmentation averages ~64 tokens per request — 16% waste. block_size=16 or 32 is strictly better for this workload.
Block size doesn't matter for attention accuracy.
vLLM doesn't support block_size=128.
Large blocks pay off only when sequences are uniformly long and the per-block GEMM efficiency gain exceeds the fragmentation cost. For short-sequence workloads, small blocks win — the kernel overhead is already amortized.
Q4 · RadixAttention key
Why does including a timestamp at the top of your system prompt destroy prefix cache hit rate?
The KV values change because the model conditions on it.
The trie is keyed on the exact token ID sequence. Any byte change in the prefix creates a new path, so every request walks a unique prefix and shares nothing. Put variable content AFTER stable content.
Timestamps have too many tokens.
RadixAttention cannot handle numbers.
Byte stability is non-negotiable. The practical fix is to make the first ~1,000 tokens of your system prompt completely static, then inject per-request context afterward. This is one of the easiest production wins — often a 30-50% TTFT improvement from a single prompt-template refactor.
Q5 · Disaggregation tradeoff
When is disaggregated prefill/decode likely NOT worth the complexity?
When the prefill-to-decode compute ratio is 1:1 and your workload has small prompts.
All of the above — disaggregation pays off only when one phase clearly dominates and the KV transfer overhead is a small fraction of total request time. For balanced workloads or small prompts, cohabitated + chunked prefill is strictly simpler.
When KV transfer is over slow network (e.g. TCP).
When you have < 4 GPUs total serving the model.
Disaggregation is a large operational commitment — separate fleets, KV transport layer, coordinated scaling. The breakeven is clear: if prefill:decode is close to 1:1 or the KV transfer cost exceeds 5-10% of request time, don't. Use chunked prefill instead.
Q6 · Chunked prefill
A 2,048-token prefill chunk on H100 at hidden_dim 8,192 has arithmetic intensity of roughly:
~5 FLOP/byte (memory-bound)
~1,200 FLOP/byte (deeply compute-bound — well past H100 ridge of 295)
~50 FLOP/byte (transition zone)
Cannot be computed without knowing output length
Prefill AI ≈ chunk_tokens / 2 for batch 1 at FP16. A 2,048-token chunk gives AI ≈ 1,024 — deep compute-bound territory. This is why chunk sizes of 1,024–2,048 get full GEMM utilization; anything smaller wastes compute.
Q7 · Attention sinks
Why does a pure sliding-window KV eviction policy (keep only last N tokens) catastrophically degrade a well-trained LLM?
Positional embeddings break.
The model learned to use the first few tokens as "attention sinks" — dumping residual softmax probability onto always-present early positions. Evicting them causes softmax to renormalize catastrophically across a distribution it never saw during training.
Causal masking fails.
Layer norms get unstable.
Attention sinks were discovered empirically in StreamingLLM (2023). The fix is a one-liner: always keep the first 4 tokens, then apply sliding window. With sinks, output quality holds; without them, the model collapses within a few tokens.
Q8 · Offloading viability
When is KV offloading to host DRAM a good idea?
Always — DRAM is cheaper than HBM.
When idle time between accesses (e.g. between user turns in a chat) is much larger than the transfer latency. The 30 ms PCIe transfer is invisible if the next access is 10 seconds away; fatal if it's in 5 ms.
Only for models under 7B parameters.
Never — always size HBM for the workload.
Offloading is a temporal-locality bet. For multi-turn chat with seconds of idle time, it's a huge win. For a streaming request that decodes continuously, it kills your SLO. For Nexus, with no cross-request state, offloading has no role.
Q9 · KV eviction risk
A team deploys H2O-style KV eviction on a legal-document-analysis workload and sees a 4-point accuracy regression. What went wrong?
H2O is broken.
Legal analysis requires long-range reference — the model genuinely needs to look back at early-document context 500+ tokens later. Heavy-hitter eviction discards tokens that look "cold" but are actually critical for specific downstream questions. Eviction works only for tasks with local attention patterns.
The eviction budget was set too high.
GQA conflicts with H2O.
KV eviction trades accuracy for memory in an irreversible way. It's excellent for tasks with "forgetful" attention patterns (chat, short-reasoning) and dangerous for tasks that depend on long-range lookup (legal, code, multi-hop QA). Always A/B test eviction on your exact eval set before shipping.
