01 · Why Quantization Works at All

Neural networks are robust to precision loss.

Before we get into methods, pause on the question that trips up every first-time quantizer: why does cutting 12 of the 16 bits of every weight not destroy the model? The answer is a combination of neural network structure, the shape of the weight distribution, and the error-dissipation properties of deep networks — and understanding it tells you exactly when quantization is safe and when it isn't.

Three properties that make quantization possible

1 · Over-parameterization

A 70B model has far more parameters than it strictly needs to represent its learned function. Training finds a solution that uses that capacity inefficiently — most weights end up small, a handful are outliers. Quantization exploits this: we can round 99% of the weights aggressively if we protect the ~1% that carry disproportionate signal.

2 · Gaussian-ish weight distributions

Weights in trained transformers are approximately zero-centered with a heavy-tailed distribution. Most mass lives in a narrow band near zero. A good quantizer places most of its representable values in that band — which is why uniform quantization loses to group-wise quantization, and why asymmetric quantization often wins on ReLU-biased activations.

3 · Error averaging across depth

In a 128-layer transformer, uncorrelated rounding errors in each layer's matmul tend to partially cancel rather than compound geometrically. The catastrophic failure mode — where errors correlate across layers and blow up — happens when quantization systematically biases the output (e.g. clamping all outliers in the same direction). Avoiding bias is 80% of the quantization game.

4 · Activations are harder than weights

Weights are static and observable in advance. Activations are data-dependent and have wild dynamic ranges — a handful of "outlier channels" in LLM activations can be 100× larger than the rest. This is why weight-only INT4 is routine but weight-and-activation INT8 needed SmoothQuant to become viable.

🔑

The memory win you should expect

Cutting weight precision from FP16 to INT4 means 4× less HBM traffic per decode step. In the memory-bound regime (which is every batch-1 decode), that is a 4× speedup — not merely a memory reduction. The memory and the speed are the same axis. This is why INT4 is the default starting point for serving: it is almost free speed, and the accuracy cost, done right, is typically under 1%.

02 · The Quantization Design Space

Six axes, one decision tree.

"Quantize the model to INT4" is not a specification. It's a rough direction. The real decision has six independent axes, each with tradeoffs — and choosing well requires understanding what each one actually controls.

The Six Axes of a Quantization Recipe

The symmetric vs asymmetric quantization math

Every quantization scheme maps floating-point values to a small set of integers. The two canonical schemes are symmetric (no zero-point) and asymmetric (with a zero-point). The choice between them is a tradeoff between runtime cost and representational efficiency.

Symmetric quantization (INT4 example, range [-8, 7]) scale = max(|W|) / 7
W_int = round(W / scale) ← integer in [-8, 7]
W_dequant = W_int × scale ← to recover float
Asymmetric quantization (INT4 example, range [0, 15]) scale = (max(W) - min(W)) / 15
zero_pt = round(-min(W) / scale)
W_int = clip(round(W/scale) + zero_pt, 0, 15)
W_dequant = (W_int - zero_pt) × scale

Symmetric is faster: the matmul kernel doesn't need to subtract a zero-point. Asymmetric wastes less range when the distribution is not centered at zero — which, for weights, is usually unnecessary (they're roughly zero-centered) but for activations after ReLU or SiLU can recover meaningful precision. In practice, production LLM quantization uses symmetric for weights and either symmetric or dynamic per-token asymmetric for activations.

💡

Why smaller group sizes win

At group size 128, every 128 consecutive weights share one scale. At group size 32, that shrinks to 32. Finer groups adapt to local distribution variations, reducing per-weight error at the cost of more scale-storage overhead. The sweet spot for most transformer weights is g=128 — enough adaptivity to handle channel-level variation without blowing up the stored-scale overhead past a few percent of the quantized weight bytes. AWQ papers report < 0.5% accuracy degradation at W4g128 on LLaMA; RTN at the same bit width often loses 2-3%.

04 · GPTQ · The Hessian-Aware Approach

Quantize one column, compensate the rest.

GPTQ (Frantar et al., 2022) is the first quantization method that made INT4 LLMs look genuinely viable. Its key insight: when you round a weight, the resulting error predictably perturbs the layer's output — and you can compensate that perturbation by adjusting the remaining, not-yet-quantized weights. The method comes from Optimal Brain Surgeon theory (1993) applied to transformer weights.

The core optimization problem

Consider a single linear layer with weight matrix W of shape [out_dim, in_dim], processing a batch of calibration activations X of shape [N, in_dim]. The forward pass produces Y = W X^T. We want to find a quantized weight matrix Ŵ such that the reconstructed output Ŷ = Ŵ X^T is as close to Y as possible.

GPTQ objective — minimize reconstruction error Ŵ* = argmin_Ŵ ||W X^T − Ŵ X^T||² s.t. Ŵ is INT4-quantized
     = argmin_Ŵ trace((W−Ŵ) X^T X (W−Ŵ)^T)
     with H ≡ X^T X = the input Hessian (proxy)

H ≡ X^T X is a d × d matrix where d is in_dim (say, 8192 for Llama-3-70B). It captures the statistics of activations — which input dimensions carry more energy, and how they correlate. GPTQ uses this matrix to decide: after I round this column of W, how should I adjust the remaining unquantized columns to minimize the damage?

The algorithm, step by step

GPTQ — Column-by-Column Quantization with Error Compensation

GPTQ in PyTorch — the core of the algorithm

gptq_layer.py · heart of the algorithmPYTHON
# Simplified GPTQ for one linear layer. The real implementation (Frantar et al.)
# adds block-wise processing, activation reordering, and numerical tricks, but
# this captures the core optimization.

import torch

def gptq_quantize_layer(
    W: torch.Tensor,          # shape [out_dim, in_dim], FP32
    X: torch.Tensor,          # calibration activations [N, in_dim]
    bits: int = 4,
    group_size: int = 128,
    damp: float = 0.01,
):
    # 1) Input Hessian proxy: H = 2 * Xᵀ X, damped for numerical stability.
    H = 2.0 * (X.T @ X)
    H += damp * torch.diag(torch.ones(H.shape[0], device=H.device) * H.diag().mean())

    # 2) Handle "dead" input channels (rows of H with zero norm).
    dead = H.diag() == 0
    H[dead, dead] = 1
    W[:, dead] = 0

    # 3) Invert H via Cholesky — faster and more stable than explicit inverse.
    Hinv = torch.linalg.cholesky(H)
    Hinv = torch.cholesky_inverse(Hinv)
    Hinv = torch.linalg.cholesky(Hinv, upper=True)  # upper triangular

    out_dim, in_dim = W.shape
    Q = torch.zeros_like(W)
    Err = torch.zeros_like(W)

    # 4) Process columns in blocks for cache efficiency.
    for g_start in range(0, in_dim, group_size):
        g_end = min(g_start + group_size, in_dim)

        # Compute group scale from weights remaining in this group.
        group = W[:, g_start:g_end]
        absmax = group.abs().max(dim=1, keepdim=True).values
        scale = absmax / (2**(bits-1) - 1)   # symmetric
        qmin, qmax = -2**(bits-1), 2**(bits-1) - 1

        for j in range(g_start, g_end):
            w = W[:, j]
            d = Hinv[j, j]

            # Quantize this column to INT and back to float.
            q_int = torch.clamp(torch.round(w / scale.squeeze()), qmin, qmax)
            q = q_int * scale.squeeze()
            Q[:, j] = q
            err = (w - q) / d

            # KEY STEP: propagate error to columns j+1..g_end-1 within the block.
            # Outer product: each row gets nudged proportionally to its own error.
            W[:, j+1:g_end] -= err.unsqueeze(1) * Hinv[j, j+1:g_end].unsqueeze(0)
            Err[:, j] = err

        # Propagate accumulated block error to columns after this group.
        W[:, g_end:] -= Err[:, g_start:g_end] @ Hinv[g_start:g_end, g_end:]

    return Q, scale
Σ

Why the Cholesky trick matters

Computing H⁻¹ directly for a 8192 × 8192 matrix is numerically unstable — the condition number of activation Hessians in transformers routinely exceeds 10⁸. The Cholesky factorization with small diagonal damping (λI) gives us a numerically stable way to do the same thing. The damping factor is small enough (≈ 1% of mean diagonal) to barely perturb the optimization, but large enough to keep the floating-point math sane. Skipping the damping is one of the most common reasons a homegrown GPTQ implementation produces NaN-filled weights.

When GPTQ wins, and when it doesn't

GPTQ is the gold standard for pure weight-only quantization at INT4 and INT3. It preserves accuracy essentially as well as QAT on dense transformers, requires only 128-1024 calibration samples, and the calibration itself takes 1-3 hours on a single GPU even for 70B models. The catch: GPTQ does not quantize activations — it assumes A16 (FP16 activations). For serving, this means the compute itself is still done in FP16 with dequantized weights. You get the memory win (4×), you get the bandwidth win (4× fewer bytes per weight read), but you don't get the extra math speedup of actually computing in INT8 or FP8.

If you want activation quantization — needed for maximum throughput on TensorRT-LLM or SGLang with INT8/FP8 kernels — you need SmoothQuant or AWQ. That's next.

05 · AWQ · Protect the Salient Weights

Not all weights are equal.

Activation-aware Weight Quantization (AWQ, Lin et al., 2023) takes a fundamentally different approach than GPTQ. Instead of rounding with error compensation, AWQ observes that a small fraction of weight channels are "salient" — they consistently amplify large-magnitude activations — and designs a quantization scheme that protects those channels at the expense of scaling the rest. The result is a simpler, faster method that composes naturally with activation quantization and kernel-level optimizations.

The salient weight observation

Run a calibration forward pass through any trained LLM and look at the activation magnitudes entering each linear layer. You'll find that ~1% of input channels have activation magnitudes 10-100× larger than the rest. The weights connected to these high-magnitude channels are salient: rounding them produces output perturbations that are 10-100× larger than rounding an average weight. This is the core AWQ insight — weights are not uniformly important, and per-channel importance is empirically observable.

AWQ · Salient Channels and the Scaling Trick
🔑

The elegant algebraic identity behind AWQ

The AWQ trick rests on a trivial identity: if we scale a channel of W up by factor s, and scale the corresponding activation channel down by the same s, the layer output is unchanged — (W · s) × (X / s) = W · X. But the quantization of (W · s) is different from quantization of W: the inflated values are farther from zero, which means they land on different quantization levels and suffer less relative rounding error. The activation scaling can be folded into the preceding layer's output scale at no runtime cost. That's the whole method — scale, quantize, bake.

Finding the scale factors — grid search on calibration data

awq_search.py · finding optimal channel-wise scalesPYTHON
# For each linear layer, find per-channel scales s that minimize
# reconstruction error after W4 quantization. This is a cheap grid search.

import torch

def find_awq_scales(
    W: torch.Tensor,        # [out_dim, in_dim], FP16
    X: torch.Tensor,        # calibration activations [N, in_dim]
    bits: int = 4,
    n_grid: int = 20,
):
    # Per-channel activation magnitude — the saliency signal.
    x_mag = X.abs().mean(dim=0)            # [in_dim]
    w_mag = W.abs().mean(dim=0)            # [in_dim]

    # Baseline: no scaling, standard RTN W4.
    def quant_error(W_scaled):
        absmax = W_scaled.abs().max(dim=0, keepdim=True).values
        scale = absmax / (2**(bits-1) - 1)
        Wq = torch.round(W_scaled / scale).clamp(-2**(bits-1), 2**(bits-1)-1) * scale
        # Error on reconstructed output, weighted by activation magnitude.
        return ((W_scaled - Wq).abs() * x_mag).sum()

    # Grid search over alpha in [0, 1]: scale factor = x_mag^alpha / w_mag^(1-alpha).
    # alpha=0 → scale by 1/w_mag (weight-only). alpha=1 → scale by x_mag (pure activation).
    # Optimal usually lives near alpha=0.5 in practice.
    best_err, best_s = float('inf'), None
    for i in range(n_grid):
        alpha = i / (n_grid - 1)
        s = (x_mag.pow(alpha) / w_mag.pow(1 - alpha)).clamp(min=1e-4)
        s = s / s.mean()                           # normalize
        W_scaled = W * s.unsqueeze(0)
        err = quant_error(W_scaled)
        if err < best_err:
            best_err, best_s = err, s

    return best_s

A typical search grid of 20 alpha values takes well under a second per layer on an H100. The full 70B AWQ calibration fits in roughly 30 minutes on one GPU — significantly faster than GPTQ's Hessian inversion. AWQ is what vLLM ships by default for W4A16 quantization because the accuracy is competitive with GPTQ and the kernels are simpler (no per-group zero points, pure symmetric quantization).

06 · SmoothQuant · Making Activations Quantizable

The outlier problem.

GPTQ and AWQ quantize only weights (W4A16). If you want W8A8 — the regime where both weights and activations become 8-bit, enabling TensorRT-LLM's INT8 GEMM kernels — you run into a brutal problem: activations have outliers that make naive quantization destroy accuracy. SmoothQuant (Xiao et al., 2022) introduced the key trick that made W8A8 viable in production.

Why naive activation quantization fails

Look again at that activation histogram from the AWQ section. The salient channels have values ~100× larger than the rest. If you quantize the full activation tensor with one per-tensor scale chosen to cover the max value, the non-outlier channels — which carry the bulk of the signal — get mapped to a tiny fraction of the integer range. With INT8 and one big outlier absorbing half the range, the other 99% of channels are effectively represented at 2-3 bits. Accuracy collapses.

The fix that took the field several years to find: migrate the difficulty from activations (hard to quantize) to weights (easier to quantize) using the same algebraic identity AWQ uses — but applied in the opposite direction.

SmoothQuant · Move the Hard Part from Activations to Weights
Σ

SmoothQuant vs AWQ — same identity, different purpose

SmoothQuant and AWQ both exploit the (W·s) × (X/s) = W·X identity. AWQ scales W up for salient channels so that weight-only quantization preserves them. SmoothQuant scales W slightly up specifically to absorb activation outliers so that activations can be quantized. You can even combine them — recent production recipes do exactly this, using SmoothQuant-style scaling to make activations INT8-friendly and then AWQ-style weight-only INT4 on top. The cleanest production W4A8 pipelines use this composition.

07 · FP8 · Floating-Point at 8 Bits

Hopper's native 8-bit format.

When NVIDIA built the Hopper architecture (H100), they added native tensor-core support for two new 8-bit floating-point formats: E4M3 and E5M2. FP8 is categorically different from INT8 — it's not a post-training trick, it's a first-class numerical format with hardware acceleration. For production serving on H100/H200/B200, FP8 is increasingly the default for both weights and activations because it sidesteps many of the accuracy cliffs of INT8.

Its 4 exponent bits cover the dynamic range of typical activations and weights (roughly 10⁻³ to 10²), while 3 mantissa bits give enough precision for most cases. Because it's floating-point, it handles the outlier problem naturally — a single large value burns one exponent slot, not a huge fraction of the representable range.

💡

Why FP8 sometimes beats INT8 for LLMs

INT8 quantization has a major weakness: it's uniformly spaced. The values -127, -126, ..., 126, 127 cover the range linearly. But LLM activations and weights follow Gaussian-ish distributions where most mass is near zero. FP8 (and all floating-point formats) is logarithmically spaced near zero and coarsely spaced far from zero — which is a much better match for the actual data distribution. The result: on the same model, FP8 PTQ often beats INT8 SmoothQuant by 1-2 perplexity points, and needs zero calibration data because there are no scale factors to fit.

FP8 KV cache — the underappreciated win

Most quantization discussion focuses on weights. But in a production serving system, the KV cache routinely consumes more memory than the weights themselves during long-context workloads. A Nexus request with 6,200 input tokens on Llama-3-70B generates a KV cache of roughly 8,192 × 80 layers × 2 (K and V) × 8 (GQA groups) × 2 bytes × 6,200 tokens = ~14 GB per request in FP16. Quantizing the KV cache to FP8 halves that to 7 GB without measurable accuracy loss, which roughly doubles the concurrent requests you can serve per GPU. Module 4 covers KV cache in depth, but this is the main preview: quantize the KV cache.

08 · The Sub-4-Bit Frontier

QuIP#, BitNet, and the 1.58-bit model.

Below 4 bits, the quantization game changes fundamentally. Per-group uniform scales stop working — the quantization error grows faster than the error-compensation methods can correct. Modern sub-INT4 methods use two very different approaches: (1) non-uniform vector codebooks (QuIP#, AQLM), or (2) training from scratch at low precision (BitNet). Both are research-grade today but increasingly viable for production niches.

QuIP# — lattice codebooks for 2-bit quantization

QuIP# (Tseng et al., 2024) quantizes weight vectors — not individual weights — to entries in a carefully-designed lattice codebook. The key insights: (1) weights in trained LLMs lie on a low-dimensional manifold, so a good vector codebook captures the distribution far better than uniform scalar quantization; (2) pre-processing weights with a random orthogonal matrix (Hadamard transform) makes them more "incoherent" — their distribution becomes more Gaussian — which is exactly the regime where lattice quantization wins.

BitNet — ternary weights trained from scratch

BitNet b1.58 (Ma et al., 2024) takes a fundamentally different path. Instead of post-training quantization, BitNet replaces all linear layer weights with ternary {−1, 0, +1} values during training. The surprising result: at 3B parameters and above, BitNet models match their FP16 counterparts on most benchmarks — and since they have no multiplications (only additions and sign flips), inference is dramatically cheaper on hardware that can exploit it.

⚠

The BitNet caveat you need to know

BitNet's efficiency story depends on specialized kernels that skip multiplications entirely. On current NVIDIA GPUs, the speedup is real but modest (~2× over FP16, because the tensor cores are not designed for ternary arithmetic). The full promise requires custom silicon or repurposed accumulator hardware. For 2026 production, BitNet is realistic only if you can train from scratch — you cannot post-training-quantize a trained FP16 model to 1.58 bits without catastrophic accuracy loss, because the network has to have learned its representations knowing the weights were ternary.

10 · Debugging Drills

Four quantization traps that pass offline eval.

The dangerous failures in quantization aren't the ones that crash the model — those get caught in CI. The dangerous ones pass all unit tests and lab benchmarks, then regress quietly in production where the traffic distribution differs from the calibration set. Each drill below is a real pattern seen in deployed systems.

Drill · 01
AWQ quantized model scores -0.7 F1 on production; WikiText eval looks fine
SYMPTOM
AWQ INT4 model passes all offline evals (WikiText perplexity, MMLU accuracy, lm-evaluation-harness) within 0.3% of the FP16 baseline. Ships to production. Two weeks later: production F1 on fraud detection is 0.87 (vs 0.94 for FP16). Customer complaints follow.
WRONG HYPOTHESIS
"AWQ must be lossy — quantization always degrades quality." AWQ IS lossless enough to match FP16 on most tasks. Blanket quantization isn't the villain; calibration-data mismatch is.
CORRECT DIAGNOSIS
AWQ protects the "salient" 1% of weights based on activation magnitudes measured during calibration. If calibration data doesn't match production traffic, the WRONG weights get protected. Fraud-detection traffic activates different pathways than WikiText — merchant-category tokens, dollar-amount reasoning, geographic mismatch patterns. Calibrating on WikiText means those pathways' weights got quantized aggressively while generic-English weights got over-protected. Check: was the calibration dataset drawn from actual production prompts? If not, that's the cause.
FIX
Re-calibrate on 512-2048 samples drawn from production traffic (stratified across tenants if multi-tenant). Keep a held-out set from the same distribution for offline eval; use F1 on domain-relevant examples, not generic benchmarks. Re-run AWQ. Expect F1 to recover to within 0.1% of FP16. Lesson: quantization eval must use your traffic, not the model-card defaults. Lock this into the release pipeline — no quantized model ships without production-calibrated eval.
Drill · 02
FP8 model TPOT 2× baseline — was supposed to be 0.5× faster
SYMPTOM
Switched weights to FP8 expecting 1.5-2× throughput gain. Measured: TPOT went UP from 12ms to 24ms. GPU utilization rose slightly. Throughput fell.
WRONG HYPOTHESIS
"FP8 must be broken on this GPU" — blame the hardware. FP8 works fine on H100/H200; the issue is almost always the kernel path.
CORRECT DIAGNOSIS
FP8 weights need FP8 matmul kernels to actually see the speedup — cuBLAS FP8, Marlin FP8, or vLLM's FP8 paths. If the kernel path falls back to dequantizing FP8 → FP16 on the fly before matmul, you pay the dequant cost every step AND get FP16 compute speed. Check nsys profile: if you see a "dequant" kernel running before each matmul, you're on the dequant path. Common causes: kernel compiled without FP8 support, wrong scale tensor format, or missing FP8 capability flag in the engine config.
FIX
Verify your engine was built with FP8 matmul support (vLLM: build with FA_FP8=1; TRT-LLM: plugin_config.fp8_context_fmha=True). Confirm nsys shows only FP8-named kernels in the hot path (e.g. gemm_fp8_tn, not dequant_fp8 + gemm_fp16). If the kernel path is correct, expect 1.3-1.8× throughput gain over FP16 on H100, up to 2× on H200. Rule: never deploy FP8 without profiling — "the weights are FP8" is not the same as "the compute is FP8."
Drill · 03
INT4 model scores fine on canary; hallucinations visible in tenant queries
SYMPTOM
INT4-quantized model passes canary eval (10k synthetic prompts, quality delta < 0.5%). Deploys to one tenant. Within hours, that tenant reports outputs including fabricated merchant names, wrong fraud categorizations, occasional token salad in long outputs.
WRONG HYPOTHESIS
"Maybe the tenant has adversarial inputs" — blame the data. Almost never the answer; the canary set was the problem.
CORRECT DIAGNOSIS
Sub-4-bit quantization creates heteroscedastic quality — most prompts are fine, but specific prompt structures have much higher error. Common failure modes: long prompts (>4K tokens) where accumulated quantization noise compounds; prompts heavy with numerical reasoning (INT4 struggles to represent the salient weights for arithmetic); domain-specific terminology (merchant names) where the token embeddings were quantized too aggressively. The canary was 10k short synthetic prompts — none of which hit these failure modes.
FIX
Canary eval must stratify by prompt length bucket (≤512, 512-2K, 2K-8K, >8K), by prompt type (conversational, structured, numerical, code), and by tenant. For INT4 specifically, add a "hallucination audit" — run 1000 production prompts through both FP16 and INT4, flag any token-level divergences by type. If divergence concentrates in one prompt structure, that's your failure mode. Fix options: fall back to INT8 or AWQ4 (more capacity per weight), or shard — use FP16 for long-prompt traffic and INT4 for short. Sub-4-bit is not a drop-in; the savings come with stratification effort.
Drill · 04
Fine-tuned model quantized with old scales; accuracy mysteriously tanks
SYMPTOM
Base FP16 model was quantized to FP8 in January with production-calibrated scales. In March, the fine-tuning team releases a new LoRA adapter merged into the base. Standard pipeline: load base weights with cached FP8 scales, merge LoRA, save. Offline eval drops 3%. No obvious bug.
WRONG HYPOTHESIS
"The LoRA adapter must be faulty — retrain it." Wastes a week retraining a perfectly fine adapter.
CORRECT DIAGNOSIS
FP8 scales are computed from the distribution of each weight tensor. Fine-tuning SHIFTS that distribution — weights move to different ranges. Re-using January's scales to quantize March's weights means the scales don't match the new weight distribution, causing clipping (large weights saturate) and resolution loss (small weights underflow). This is invisible in code because the shapes all match; it's a numerical problem. Check: load both the pre- and post-LoRA weights, compute per-layer min/max/std, compare the distributions. If any layer's range moved >10%, its cached FP8 scale is wrong.
FIX
Re-calibrate FP8 scales whenever the base model changes, even from "just" a LoRA merge. Automate it: the release pipeline should recompute scales on a 512-sample production calibration set and gate on offline eval not regressing. For true drop-in adapter swaps without recalibration, keep the adapter as a separate LoRA head at inference time (don't merge); FP8 only applies to the base, adapter runs in FP16. Costs a small amount of per-token latency (LoRA matmul), but scales stay stable across adapter updates.
💡

The common thread

All four drills share a pattern: quantization is a statistical approximation, and when the statistics of your traffic diverge from the statistics used during calibration, quality degrades. The fixes are all the same recipe in different forms: calibrate on your traffic, eval on your traffic, recalibrate when the traffic changes. Once that discipline is internalized, quantization becomes safe and the 2-4× throughput wins are there for the taking. Without that discipline, quantization is a future outage waiting for its trigger.

Q1 · The core win
Cutting weight precision from FP16 to INT4 reduces TPOT by approximately 4× in decode. What is the primary reason?
INT4 arithmetic is faster than FP16 arithmetic on tensor cores.
Decode is memory-bandwidth-bound — the weights are read from HBM once per token — so reducing bytes-per-weight directly reduces HBM read time, which is almost the entire TPOT.
INT4 kernels use tensor cores; FP16 doesn't.
Quantized models have fewer effective parameters.
In the memory-bound regime (Module 1), time-per-token ≈ weight_bytes / HBM_bandwidth. Cutting bytes by 4× cuts time by 4×. The tensor cores are not the bottleneck at batch 1 — HBM is. This is the entire reason quantization is so powerful for inference.
Q2 · GPTQ mechanism
What does GPTQ do that simple round-to-nearest does not?
Uses more bits for important weights.
After rounding each column, it adjusts the remaining unquantized columns using the input Hessian to compensate for the rounding error.
Uses asymmetric quantization.
Retrains the model.
GPTQ's trick is error compensation: each rounding error is absorbed by adjusting the weights that haven't yet been quantized. The Hessian tells us exactly how much to adjust each one. This drops accuracy loss from ~3 MMLU points (RTN) to ~0.5 points at INT4.
Q3 · AWQ insight
AWQ is built on the observation that:
Weights have wildly different distributions across layers.
~1% of input channels have activation magnitudes 10-100× larger than the rest, and the weights connected to these "salient" channels carry disproportionate output signal.
Transformers store redundant information.
Activations are always larger than weights.
Salient channels are AWQ's core observation. By scaling weights connected to salient channels UP before quantization (and scaling activations DOWN to compensate), the salient weights land in a more quantizable range. This reduces effective rounding error where it matters most.
Q4 · SmoothQuant purpose
Why was SmoothQuant needed to make INT8 activation quantization (W8A8) viable?
INT8 has too few levels.
Activations have outlier channels so large that a single per-tensor scale maps the non-outlier channels into just a few integer levels, destroying precision. SmoothQuant migrates the outliers from activations into weights, where they are quantizable.
INT8 is slower than FP16.
Tensor cores don't support INT8.
The outlier problem is brutal. Without migration, W8A8 LLM perplexity jumps from 4.5 to 38+ — a broken model. SmoothQuant's algebraic identity (W·s × X/s = W·X) moves the outlier burden onto weights (which have a narrower distribution) so that both sides become quantizable.
Q5 · FP8 vs INT8
Why does FP8 E4M3 often outperform INT8 for LLM weights and activations?
FP8 has more bits of information.
FP8 is logarithmically spaced — more precision near zero — which matches the Gaussian-like distribution of LLM weights/activations better than INT8's uniform spacing.
FP8 has a larger dynamic range than INT8.
FP8 doesn't need calibration.
Both statements are partially true, but the key insight is the spacing. LLM weights and activations cluster around zero with heavy tails. FP8's log-spacing gives fine precision where most of the data lives and coarser precision where outliers live — a natural match. INT8's uniform spacing wastes precision in the tails.
Q6 · Group size tradeoff
Changing group size from 128 to 32 in W4 quantization:
Makes accuracy worse because fewer weights share a scale.
Generally improves accuracy because each group adapts to local distribution, but adds ~4× more scale factors to store (increasing effective bits from ~4.1 to ~4.5 bits/weight).
Has no effect — the quantizer adapts automatically.
Makes inference faster.
Smaller groups = better accuracy, more scale overhead, slightly slower kernels. The community has converged on g=128 as a good default because the accuracy delta from g=32 is tiny on well-trained models, while the scale overhead savings are real. For very sensitive models or sub-INT4, g=32 or g=64 is sometimes justified.
Q7 · KV cache quantization
A production LLM system with long context (8k+ input tokens) sees memory-pressure incidents under load. Which quantization target gives the biggest relief?
Weight quantization from FP16 to INT4.
KV cache quantization from FP16 to FP8 — at 8k context per request, the KV cache dominates memory vs. weights.
Activation quantization.
Gradient quantization.
At long context with many concurrent requests, KV cache often exceeds weight memory — each 8k-token request needs ~14 GB of KV in FP16 for a 70B. Halving to FP8 KV doubles concurrent-request capacity, which is usually where the real production pain lives. Module 4 covers KV cache in detail.
Q8 · BitNet caveat
Why can't you post-training-quantize an existing Llama-3-70B to BitNet b1.58 format?
The model is too large.
The model has learned its representations assuming continuous-valued weights. Forcing all weights to {-1, 0, 1} destroys the learned features. BitNet only works when the model is trained from scratch with that constraint.
BitNet kernels don't exist for 70B models.
Llama doesn't support ternary weights.
Training co-adapts the weights: a model trained with ternary weights learns to distribute information differently, using more parameters per concept to compensate for lower per-weight precision. Taking a trained FP16 model and hammering its weights to ternary loses this adaptation — accuracy collapses completely.
Q9 · Hardware selection
You need to deploy a 70B model with < 1% accuracy loss. What's the best quantization recipe on H100 vs A100?
Both should use AWQ W4 for consistency.
On H100, use FP8 (native hardware support, simpler calibration). On A100, use AWQ W4 (no FP8 kernels on Ampere). Both achieve < 1% drop; the H100 FP8 path has slightly better accuracy and much simpler tooling.
On both, use GPTQ W3 for maximum compression.
On H100, use INT8 with SmoothQuant; on A100, use FP16.
Hardware features change the optimal recipe. FP8 on H100 is often underrated because AWQ/GPTQ have better academic exposure, but for production-grade serving with accuracy preservation, FP8 is the simpler and often better choice on Hopper. INT4 paths remain the default on pre-Hopper hardware.
