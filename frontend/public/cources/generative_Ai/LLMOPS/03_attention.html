Attention, properly
engineered.

Attention is where inference spends most of its time and most of its memory. The naive textbook formulation — compute Q@K, softmax, multiply by V — does six round trips to HBM per layer and allocates an N×N intermediate that scales quadratically with context length. Every modern serving stack replaces this with carefully-engineered alternatives: FlashAttention's SRAM tiling, DeepSeek's Multi-head Latent Attention, Grouped-Query Attention's head sharing, Ring Attention's sequence parallelism. This module takes each apart down to the kernel.

01 · The Problem with Vanilla Attention

Six round trips, one quadratic tensor.

Before we talk about any optimization, let's see exactly what the textbook attention formula actually costs when you run it. The math looks clean on paper. The memory-access pattern is a disaster.

The three steps of scaled dot-product attention

Scaled Dot-Product Attention (Vaswani et al., 2017) S = Q K^T / √d_k ← [N × N] scores matrix
P = softmax(S) ← [N × N] probabilities matrix
O = P V ← [N × d_k] output

With N = 6,200 (Nexus prompt length) and 64 attention heads in parallel, the S and P matrices alone consume 6,200 × 6,200 × 64 × 2 bytes = 4.6 GB per layer. Across 80 layers, the intermediate attention tensors — which exist only to be consumed by the next step and then discarded — would demand 370 GB of HBM that we do not have. Implementations work around this by processing one head at a time, but they still do six HBM round trips per layer.

Naive Attention · Six Round Trips to HBM per Layer
⚠

The quadratic-memory blocker

Materializing the S and P matrices means attention memory scales as O(N²). At N=6,200 you pay 4.9 GB per layer. At N=32,768 you pay 137 GB per layer — an entire H100's worth of HBM for one layer's intermediates. This is the primary reason "long context" was impractical before 2022. Every modern attention implementation — FlashAttention, xFormers, PyTorch SDPA, vLLM's kernels — avoids materializing these matrices through some variant of the tile-streaming trick.

The arithmetic intensity of attention

Attention has an interesting cost structure. The QKV projections are regular matmuls — compute-bound during prefill, memory-bound during decode, following the patterns from Module 1. But the attention operation itself (S, softmax, P@V) has different properties: it reads Q, K, V from HBM and produces an output of the same size. In decode, where Q is [1 × d] but K and V are [N × d] (the full past context), the attention operation is overwhelmingly memory-bound — we read N tokens of K and V from HBM to produce 1 token of output. This is why KV cache is the dominant memory bottleneck for long contexts, and it sets up everything we do in Module 4.

02 · FlashAttention · The Tiling Revolution

Never materialize the N × N tensor.

FlashAttention (Dao et al., 2022) changed what's possible in transformer serving. Its single trick: process attention in tiles small enough to live entirely in on-chip SRAM, combining Q@K, softmax, and P@V into one fused kernel that reads Q/K/V from HBM exactly once and writes O exactly once. The scores matrix S is never stored. The result: 3-5× speedup on prefill, O(N) memory instead of O(N²), and the thing that unlocked 128K+ context windows across the industry.

The online softmax — why this even works

The fundamental obstacle to tiling attention is the softmax operation. Standard softmax requires knowing the maximum value across the full row before you can compute any individual probability — because softmax(x_i) = exp(x_i - max(x)) / Σ exp(x_j - max(x)), and we subtract the max for numerical stability. If we process attention in tiles of K, each tile only sees a partial row, so we cannot compute the max or the sum until all tiles are seen.

The online softmax algorithm (Milakov & Gimelshein, 2018) shows we can. Maintain three running statistics as we stream over K tiles: m (running max), ℓ (running sum of exponentials, rescaled to current max), and O (running partial output, also rescaled). When we see a new tile with its own max and sum, we update these three values with a simple rescaling formula that preserves correctness.

Online softmax update — the mathematical core of FlashAttention // when processing tile j with local max m_j and local sum ℓ_j:
m_new = max(m_old, m_j)
ℓ_new = ℓ_old · exp(m_old − m_new) + ℓ_j · exp(m_j − m_new)
O_new = (O_old · ℓ_old · exp(m_old − m_new) + O_j · exp(m_j − m_new)) / ℓ_new

These updates require only O(d) memory per tile — not O(N) — and after processing every tile, O_new is exactly what standard softmax would produce. The math is remarkable: a streaming algorithm that produces bit-identical results to the non-streaming version, enabling the entire N×N attention operation to run in SRAM-sized chunks.

Why FlashAttention-3 is Hopper-only

FA-3 exploits three Hopper-specific features that don't exist on Ampere: (1) WGMMA warpgroup matrix-multiply-accumulate instructions that run async with respect to normal CUDA cores, allowing softmax and matmul to overlap; (2) TMA (Tensor Memory Accelerator) for async DMA-like loads from HBM that don't block the SMs; (3) native FP8 tensor cores. Each of these removes a synchronization point that bottlenecked FA-2. The cumulative effect is 75-85% of the H100's theoretical peak on attention — unprecedented for a non-matmul operation. If you are serving on H100/H200, you should be on FA-3.

03 · FlashAttention in Code

The kernel, minus the CUDA.

Here's a PyTorch implementation of FlashAttention that mirrors the actual CUDA logic. It's slower than the real kernel — we're simulating SRAM with PyTorch tensors — but every data movement and every math operation corresponds line-for-line with what the Triton/CUDA version does. Study this once and the real kernel becomes readable.

flash_attn_simulated.py · the algorithm in clear PyTorchPYTHON
# A direct PyTorch implementation of FlashAttention forward pass.
# This is what the real CUDA kernel does, just slower because we don't
# have access to actual SRAM. The math is identical.

import torch

def flash_attention_forward(
    Q: torch.Tensor,   # [B, H, N, d]
    K: torch.Tensor,   # [B, H, N, d]
    V: torch.Tensor,   # [B, H, N, d]
    Br: int = 64,    # Q-tile row count — fits in SRAM
    Bc: int = 64,    # K,V-tile row count — fits in SRAM
):
    B, H, N, d = Q.shape
    scale = 1.0 / (d ** 0.5)

    # Output tensor lives in HBM (our normal GPU memory).
    O = torch.zeros_like(Q)
    # Logsumexp for the backward pass; not used in forward but recorded.
    L = torch.zeros(B, H, N, device=Q.device)

    # Outer loop: over Q tiles. Each Q tile is loaded into SRAM once
    # and kept there for the whole inner loop.
    for i in range(0, N, Br):
        Q_i = Q[:, :, i:i+Br, :]              # [B, H, Br, d] — in SRAM
        O_i = torch.zeros_like(Q_i)           # running output — in SRAM
        m_i = torch.full((B, H, Br, 1), float('-inf'),
                         device=Q.device)      # running row-max
        l_i = torch.zeros((B, H, Br, 1), device=Q.device)  # running sum

        # Inner loop: stream K, V tiles through the SM.
        for j in range(0, N, Bc):
            K_j = K[:, :, j:j+Bc, :]          # [B, H, Bc, d]
            V_j = V[:, :, j:j+Bc, :]

            # 1) Compute scores for this tile.
            S_ij = (Q_i @ K_j.transpose(-1, -2)) * scale   # [B, H, Br, Bc]

            # 2) Local row-max over this tile.
            m_j = S_ij.max(dim=-1, keepdim=True).values     # [B, H, Br, 1]

            # 3) New running max.
            m_new = torch.maximum(m_i, m_j)

            # 4) Exponentiate with max subtracted — numerically stable.
            P_ij = torch.exp(S_ij - m_new)                   # [B, H, Br, Bc]

            # 5) Rescale accumulated sum to new max reference.
            alpha = torch.exp(m_i - m_new)                   # correction factor
            l_new = alpha * l_i + P_ij.sum(dim=-1, keepdim=True)

            # 6) Online update of O_i — this is the key step.
            #    Old O_i was normalized by old (m_i, l_i); we rescale it
            #    to new max reference, then add this tile's contribution.
            O_i = O_i * alpha + P_ij @ V_j                   # [B, H, Br, d]

            # 7) Move running stats forward.
            m_i, l_i = m_new, l_new

        # After all tiles processed, divide once by final l_i
        # to get the correctly-normalized softmax output.
        O[:, :, i:i+Br, :] = O_i / l_i
        L[:, :, i:i+Br] = (m_i + torch.log(l_i)).squeeze(-1)

    return O, L

# Verification: produces bit-identical results to naive attention.
def naive_attention(Q, K, V):
    d = Q.shape[-1]
    S = Q @ K.transpose(-1, -2) / d ** 0.5
    P = torch.softmax(S, dim=-1)
    return P @ V

B, H, N, d = 2, 8, 512, 64
Q = torch.randn(B, H, N, d, device="cuda")
K = torch.randn(B, H, N, d, device="cuda")
V = torch.randn(B, H, N, d, device="cuda")
O_flash, _ = flash_attention_forward(Q, K, V)
O_naive = naive_attention(Q, K, V)
# Max absolute difference will be < 1e-5 — effectively identical.
print((O_flash - O_naive).abs().max())    # tensor(2.4e-6)
Σ

Why the math is identical, not approximate

FlashAttention produces bit-identical results to standard softmax attention (up to floating-point reordering effects under 10⁻⁶). There's no approximation, no sparsity, no information loss. The online softmax is an algebraic identity: rescaling the partial sum by exp(m_old − m_new) when a new max is seen produces exactly the same final normalized value as if we'd had the full row from the start. This is why FlashAttention was such a profound advance — it turned attention from an O(N²)-memory operation into an O(N)-memory operation with identical output, using pure algorithmic redesign.

04 · MHA, MQA, GQA · Sharing the K/V Heads

Decode's real enemy is the KV cache.

FlashAttention optimizes how attention is computed. The next family of techniques changes what attention is — specifically, how many K and V heads the model has. This matters because during decode, we read the full KV cache from HBM at every step, and with standard Multi-Head Attention the KV cache is enormous. Multi-Query and Grouped-Query Attention cut this by 8-64× with minimal accuracy loss, and they compose perfectly with FlashAttention.

The four head-sharing variants, visualized

MHA · MQA · GQA · MLA — How Many K/V Heads?

The memory math, concretely

Llama-3-70B has 80 layers, 64 attention heads, and head dimension 128. With MHA, the KV cache per token across all layers is 80 × 2 × 64 × 128 × 2 = 2.6 MB. A 6,200-token Nexus request therefore needs 16 GB of KV cache. With Llama-3-70B's actual GQA (8 K/V groups instead of 64 heads), that's already reduced 8× to 2 GB per request. This is why you can even consider running 100+ concurrent Nexus requests on a single H200 — the GQA alone buys you an order of magnitude of request concurrency.

Why Llama-3 uses GQA-8, not MQA

MQA (one K/V head) seems strictly better — 8× less memory than GQA-8. But empirically, MQA loses ~0.5-1 MMLU points, while GQA-8 is essentially free (−0.2). The intuition: different Q heads can attend to genuinely different aspects of the past, and forcing them to share one K/V severely limits expressiveness. Grouping in sets of 8 gives enough diversity for full accuracy while still reducing KV memory 8×. Every major open model family from Llama-3 forward uses GQA as the default.

05 · MLA · DeepSeek's Low-Rank KV Compression

Cache the latent, not the heads.

Multi-head Latent Attention (MLA), introduced with DeepSeek-V2 and refined in V3, takes a fundamentally different approach than GQA. Instead of sharing K/V heads, MLA compresses the full K and V vectors into a low-dimensional latent vector that gets stored in the cache. Per-head K and V are reconstructed on the fly during attention computation via learned up-projection matrices. The result: 93% KV memory reduction with accuracy that matches or slightly exceeds full MHA.

The MLA architecture — compress and reconstruct

MLA forward pass — one new token // Down-projection: compress x into small latent c_KV
c_KV = x @ W_DKV // [1, 512] — this is what we cache

// Later in attention, reconstruct K, V per head as needed:
K = c_KV @ W_UK // [1, 8192] → reshape to [64, 128]
V = c_KV @ W_UV // [1, 8192] → reshape to [64, 128]

// Q uses its own compressed latent with rope-rotation:
c_Q = x @ W_DQ // [1, 1536]
Q_rot = (c_Q @ W_UQ_rot) ⊕ rope // rotary-rotated part
Q_nope = c_Q @ W_UQ_nope // non-rotated part
Q = [Q_rot ; Q_nope] // concatenated

The magic is in the cacheable part: we only store c_KV (512 floats) per token, not the full K and V vectors (8,192 each). For Nexus's 6,200-token context, this is 900 MB instead of 16 GB — an 18× reduction. During decode, the per-head K and V are reconstructed inside the attention kernel via two additional matmuls against the small latent, which is almost free on Hopper tensor cores.

The "absorbed matmul" trick that makes it fast

A naive reading of MLA suggests the extra up-projection matmuls would be expensive. The DeepSeek paper shows they can be absorbed: the attention score Q @ K^T where K = c_KV @ W_UK is equivalent to Q @ (c_KV @ W_UK)^T = Q @ W_UK^T @ c_KV^T. We can precompute Q' = Q @ W_UK^T and then just do Q' @ c_KV^T — one matmul against the tiny latent, which is actually faster than MHA's matmul against the full K.

MLA · Cache the Latent, Absorb the Up-Projection
💡

Why MLA is the design DeepSeek is betting on

For a 236B-parameter MoE model like DeepSeek-V2, KV cache at long context would be catastrophic with MHA — 16k tokens × 236B-scale attention config would consume the entire GPU memory for a single request. MLA makes 128k context practical on a single 8-GPU node. The cost is training complexity: MLA requires a specialized rotary embedding scheme (decoupled rope) and the up-projection matrices must be trained end-to-end with the model. You cannot retrofit MLA onto an existing MHA/GQA model — it must be trained in from the start. This is why we'll see MLA appear in new models going forward, not as a conversion of older ones.

06 · Ring, Sliding Window, and Sparse Attention

When context is very long.

The techniques above help at any context length. But when you push past 128K tokens into genuinely long-context territory — legal document review, codebase understanding, multi-hour transcripts — even MLA's compressed cache becomes expensive and the N² compute of attention itself starts to dominate. Three techniques address this: Ring Attention for sequence-parallel distribution, sliding window for sub-quadratic local attention, and sparse patterns for selective global access.

Ring Attention — sharding the sequence across GPUs

Ring Attention (Liu et al., 2023) solves a specific problem: the full KV cache does not fit on a single GPU. Instead of partitioning the batch or the model, it partitions the sequence across N GPUs, with each GPU holding 1/N of the tokens. During attention, each GPU's Q block has to see every other GPU's K,V block to compute full attention. This is done as a ring: each GPU computes partial attention against the K,V it holds, then passes that K,V to the next GPU in the ring while receiving K,V from the previous one, and combines the partial results using the same online-softmax trick FlashAttention uses. After N steps, each GPU has the full attention for its shard of Q.

Ring Attention · N GPUs, sequence split N ways, KV rotated through the ring

Sliding window attention — give up on distant past

Sliding window attention (used in Mistral-7B, Longformer, and others) limits each token's attention to a window of W most recent tokens. This makes attention O(N·W) instead of O(N²), and importantly, the KV cache only needs to hold the last W tokens per layer. For W=4,096, a 128K-token context uses 32× less KV cache than full attention. The tradeoff is explicit: tokens outside the window are invisible. Models that use sliding window compensate by layering — the deeper layers see farther back through the "funnel" of receptive fields accumulated at lower layers.

Sparse attention and attention sinks

Sparse attention patterns (BigBird, Longformer global tokens, StreamingLLM's attention sinks) allow each token to attend to a structured subset of the past. The StreamingLLM insight is particularly useful: the very first few tokens of the sequence — the "attention sinks" — accumulate disproportionate attention from all later tokens, and removing them (as sliding window does) causes catastrophic quality collapse. The fix: always keep the first 4 tokens plus a sliding window of recent tokens. This gives stable behavior over arbitrarily long contexts and is especially useful for streaming applications like chat where the conversation grows indefinitely.

💡

The "attention sink" phenomenon explained

Sink tokens absorb probability mass that would otherwise have to be distributed across less-relevant past tokens — softmax normalizes to 1, so attention can never "not attend" to anything. The first few tokens are universally visible to every later token, so they become a convenient "dump" for excess attention mass that has no useful target. Removing them forces the attention distribution to spread across the recent window, which distorts the values fed forward and breaks long-context behavior. StreamingLLM keeps them; you should too if you build anything with sliding windows.

08 · Core Operational Math

The formulas you'll actually use in an RFC.

Everything in this module so far explains how attention kernels work. This section gives you the formulas for answering the question every inference engineer eventually faces: will this deployment fit, and if so, how many requests per GPU? Three formulas, three worked examples, and the one-liner verdict each produces.

Formula 1 · Per-request KV size

The single number that determines concurrency on a fixed GPU. Store K and V for every layer, every attention head (or KV-head group under GQA), every token in the sequence.

KV bytes per request KV_bytes = 2 × L × H_kv × d × T × dtype_bytes

  2   = one each for K and V
  L   = number of transformer layers
  H_kv = number of KV heads (= H_q for MHA; H_q / group_size for GQA)
  d   = head dimension (typically 64 or 128)
  T   = context length (tokens)
  dtype_bytes = 2 (FP16/BF16), 1 (FP8 KV), 0.5 (INT4 KV)

Worked example: Llama-3-70B at 8K context, FP16 KV, GQA-8

KV = 2 × 80 × 8 × 128 × 8192 × 2 = 2,684,354,560 bytes ≈ 2.5 GB per request. On an H100 (80 GB HBM) with AWQ W4 weights taking ~37 GB, you have ~43 GB left for KV — serves ~17 concurrent requests at 8K. With FP8 KV (dtype_bytes = 1) that doubles to ~34 concurrent. These are the numbers that decide whether you deploy at TP=1, TP=2, or fail.

Formula 2 · MLA compression ratio

MLA replaces the per-head K and V with a single compressed latent c_kv per layer per token. The compression ratio vs MHA is what produces the oft-cited "93% KV reduction" number.

MLA vs MHA KV ratio ratio = (c_dim + rope_dim) / (2 × H_q × d)

  c_dim = compressed latent dim (DeepSeek-V3 = 512)
  rope_dim = decoupled RoPE dim (typically 64)
  2 × H_q × d = MHA K + V dim per head, summed over all heads

Worked example: DeepSeek-V3 (128 heads, d=128)

ratio = (512 + 64) / (2 × 128 × 128) = 576 / 32,768 ≈ 0.0176. MLA stores 1.76% of what MHA would. The "93% reduction" number in the paper refers to the comparison against GQA-1 (MQA) or similar; vs full MHA the reduction is 98.2%. Either way, MLA is the single biggest architectural lever for KV memory, far larger than any quantization win.

Formula 3 · Arithmetic intensity — where you sit on the roofline

From Module 1: workloads are memory-bandwidth-bound when arithmetic intensity (FLOPs per byte of HBM I/O) is below the hardware's compute/BW ratio. For attention specifically:

Attention arithmetic intensity (prefill) AI_prefill ≈ T_q × T_k × H × d × 4 / (T_q × H × d + T_k × H_kv × d × 2) × byte_scale

  Numerator: FLOPs for (Q·K^T) + softmax + (attn·V), approximate
  Denominator: bytes read for Q, K, V (ignores attention matrix, which stays in SRAM via FA)

For H100 (989 TFLOP/s FP16, 3.35 TB/s HBM): AI_threshold = 295 FLOP/byte
For H200 (989 TFLOP/s FP16, 4.8 TB/s HBM): AI_threshold = 206 FLOP/byte

Worked example: Llama-3-8B prefill at T=4096, H_q=32, H_kv=8

At T=4096 prefill, numerator ≈ 4096² × 32 × 128 × 4 = 2.75 × 10¹¹ FLOPs. Denominator ≈ 4096 × (32 + 16) × 128 × 2 bytes = 50 MB. AI ≈ 5,500 FLOP/byte. Well above H100's 295 threshold → prefill is compute-bound, so FA-3's better compute scheduling (warp specialization, async GEMM) is what wins here. At T=128 prefill, AI ≈ 170 — below threshold → bandwidth-bound → FA-3's compute advantage disappears. This is why FA-3 benchmarks show 2× at long contexts and near-zero gain at short contexts.

🔑

The GQA group-count decision table

When choosing GQA group count for a new model, the tradeoff is KV memory (smaller = better) vs quality (more heads = more expressive). From published ablations: GQA-1 (= MQA) loses ~1.2 points MMLU on a 70B model; GQA-8 loses ~0.15 points; GQA-16 loses essentially nothing. Llama-3 picked GQA-8 as the sweet spot — 8× KV reduction vs MHA, negligible quality cost. For Nexus v3 we inherited this choice; for a de novo model, GQA-8 is the default and only deviates if (a) quality is paramount and you can afford 16-group, or (b) memory is paramount and you've already picked MLA.

10 · Debugging Drills

Four symptoms. Four diagnostic paths.

Four real on-call symptoms for attention-related issues. Each drill names the surface symptom (what shows on the dashboard), the tempting wrong hypothesis (the one that wastes two hours), the correct diagnostic path with exact commands, and the fix. These are the patterns you'll see repeatedly; internalize them and an incident at 3 AM is 15 minutes, not 2 hours.

Drill · 01
FA-3 producing numerically different outputs than FA-2
SYMPTOM
After upgrading from FA-2 to FA-3, a fraction of requests (~2-5%) produce different top-1 tokens than the FA-2 version did on the same input. Quality eval score is within tolerance (0.1 F1 delta), but specific tenants' golden sets regress. Output is not random — the new tokens are reproducible on the same inputs.
WRONG HYPOTHESIS
"The model got worse — roll back to FA-2 and investigate quality drift." Wastes hours chasing model-training explanations when the issue is numerical.
CORRECT DIAGNOSIS
Run numerical parity: take 1,000 inputs, log the top-5 logits from both FA-2 and FA-3 in eager mode. If the logit values differ by >0.01 but ranking differs only on near-ties (top-1 and top-2 within 0.05), the issue is FA-3's softmax numerical precision at long context. Check nsys profile: verify the kernel dispatched is flash_fwd_hopper (FA-3 correct), not flash_fwd_splitkv (FA-2 fallback). Check the causal_mask handling — FA-3's hopper build had a known issue in 2.6.0 where masking at non-multiples-of-8 sequence lengths produced tiny numerical drift.
FIX
Either (a) pin flash-attn==2.6.3 or newer (fixed), (b) pad sequences to multiples of 8 before the attention op, or (c) accept the ~0.003 perplexity drift and update the tenant-specific golden sets to match. Option (c) is actually correct for most deployments — FA-3 is more numerically accurate than FA-2, not less, but "different" trips golden-set comparisons.
Drill · 02
MLA deployed but decode got slower, not faster
SYMPTOM
Switched model to an MLA variant. KV memory dropped as expected (~10× reduction confirmed via nvidia-smi). But TPOT p99 increased 20-30% instead of decreasing. HBM bandwidth utilization (DCGM DRAM_ACTIVE) shows bursty patterns — hitting 95% in spikes then dropping to 40%.
WRONG HYPOTHESIS
"MLA is just slower on our hardware" or "the MLA paper exaggerated the speedup." Leads to abandoning MLA and missing the actual issue.
CORRECT DIAGNOSIS
Run nsys profile during steady-state decode. In the timeline, look for the attention kernel sequence per step. You should see exactly one kernel launch for attention. If instead you see three kernels per step — something like mla_up_project_k, mla_up_project_v, then flash_attn_fwd — the up-projection is happening BEFORE attention on HBM instead of being fused inside the attention kernel. The expanded K and V are being written to HBM and read back, doubling HBM traffic compared to a correct fused integration. This is exactly the failure break_mla_decompression.py reproduces.
FIX
Upgrade to a version of vLLM/SGLang with fused MLA decode (vLLM >= 0.6.3 or SGLang >= 0.3.0). Verify the kernel name after upgrade is mla_decode_fused or equivalent. If running a custom stack, the correct pattern uses the "weight absorption" identity: fold W_up_k into the Q projection and apply W_up_v at the end — see break_mla_decompression.py for the reference implementation.
Drill · 03
Attention wall-time dominated by softmax, not GEMM
SYMPTOM
Profiler output shows softmax_kernel taking 40%+ of attention wall-time, with flash_fwd GEMM only 55%. Expected ratio is roughly inverse — GEMM should dominate on any modern attention kernel. Latency is ~2× what benchmarks predict.
WRONG HYPOTHESIS
"Softmax is the bottleneck — we need a faster softmax kernel." There is no faster softmax; the real issue is elsewhere.
CORRECT DIAGNOSIS
A separate softmax kernel means attention is running in unfused mode — the FA kernel isn't being called at all. Check: (a) is flash-attn actually installed and loaded? Try python -c "from flash_attn import flash_attn_func; print(flash_attn_func)". (b) Is there a custom attention mask (ALiBi, sliding window with certain radius, cross-attention) that FA doesn't support? If yes, the engine falls back to an eager Python attention implementation that runs softmax as a separate kernel. (c) Is the head dim one FA doesn't support? FA-2 requires d ∈ {32, 64, 96, 128, 160, 192, 224, 256}; a nonstandard d forces eager fallback.
FIX
If flash-attn isn't loading, fix the install (pin to exact version in requirements.txt, rebuild the container). If a custom mask is blocking FA, check whether SGLang or Triton-based FlashAttention (e.g., vllm.attention.ops.triton_flash_attention) supports your mask shape — most do. If head dim is nonstandard, unfortunately the only fix is to re-train the model with a supported dim; this is why every new architecture should pick d ∈ {64, 128} unless there's a specific reason not to.
Drill · 04
GQA head-to-group ratio causing quality regression on one task
SYMPTOM
Model quality score on the overall eval set is within noise (±0.1 F1). But on a specific task slice — typically a reasoning-heavy one like math or multi-hop QA — score drops 1.5-3 points. Happens after switching from MHA to GQA-8, or after quantizing KV cache on a GQA model.
WRONG HYPOTHESIS
"The quantization is too aggressive — switch from FP8 to BF16 KV." Sometimes helps, but usually doesn't, because KV quant isn't the root cause.
CORRECT DIAGNOSIS
GQA heads share K and V across groups. For tasks requiring head specialization — specifically tasks where different heads attend to different positional patterns (math reasoning, coreference resolution, multi-hop traces) — the head sharing in GQA degrades the ability of individual Q heads to have "their own" view of the KV. Check: does the regression only appear on tasks where multi-head attention patterns are documented to matter (Olsson et al. 2022, induction heads)? If yes, GQA grouping is the cause. FP8 KV quantization amplifies this because quantization error now affects multiple Q heads at once.
FIX
Three options: (a) accept the regression if the overall eval is good — the "one task" slice may not reflect production traffic weight. (b) Run GQA-16 instead of GQA-8 — doubles KV memory but typically recovers the regression. (c) Keep GQA-8 and switch to MLA: MLA preserves head-level KV expressivity while compressing across the rank dimension, often restoring reasoning performance. Decision depends on production traffic mix.
11 · Failure Reproduction

The MLA integration that kills MLA's benefit.

The artifact pack includes a runnable script that reproduces the #1 MLA integration mistake — up-projection on the critical path of decode. Reading about a failure is one thing; watching it happen in your own terminal, then fixing it, is what makes the lesson durable. Here's what the script demonstrates and how to map it to a real production symptom.

What the script does

Two implementations of MLA decode attention, same mathematical result, very different HBM access patterns:

WRONG · mla_wrong()

Up-projects compressed c_kv to full K and V on HBM before the attention call. Attention then reads K and V back from HBM. Three HBM passes: (1) read c_kv, (2) write K and V, (3) read K and V. MLA's compression is completely wasted.

CORRECT · mla_correct()

Uses the "weight absorption" identity: Q @ (c_kv @ W_up_k)^T = (Q @ W_up_k^T) @ c_kv^T. The up-projection is algebraically folded into the attention computation; c_kv is read once, expanded K/V never materialize on HBM. One HBM pass.

Mapping to real-world symptoms

In production, this bug is almost never obvious. The symptom is "MLA is slower than expected" — not "MLA is broken." The diagnosis path from Drill 02 above is how you catch it without the runnable repro. Having both the drill and the repro means you can (1) recognize the symptom on a dashboard, (2) know what to look for in nsys output, and (3) verify a fix by running the repro after the engine upgrade and confirming the wall-clock ratio is <1.2× instead of >2.5×.

💡

The general pattern

This specific bug is MLA's flavor of a broader class: optimizations that work on paper but lose their benefit to a missing fusion. Quantization has a version of this (dequant on the critical path). Speculative decoding has a version (draft logits not kept on-device). KV compression has a version (decompression as a separate op). When any "should be faster" optimization isn't faster on your hardware, the first thing to check is: how many kernel launches per step, and is there a scratch buffer on HBM that shouldn't be there? Those two questions catch 80% of "inexplicably slow" cases.

12 · Operator Checklist

Twelve items before you ship.

The full operator checklist lives in artifacts/m03_attention/operator_checklist.md — ready to copy into a team runbook. Summarized inline below. Every item has saved a real incident somewhere; treat this as a floor, not a ceiling.

Pre-deploy verification · 12 items
  • Kernel dispatch verification. Run one request under nsys; confirm the attention kernel name matches the expected FA version (e.g. _ZN13flash_hopper for FA-3).
  • Numerical parity vs reference. Diff 100 outputs against a known-good reference; <0.1% tokens flagged is expected. >1% means a numerical bug.
  • KV cache dtype taking effect. Inspect GPU memory at steady-state; confirm KV footprint matches the formula for your dtype (FP8 = half of FP16).
  • GQA group count sanity. Model config, engine startup log, and actual tensor shapes all report the same num_key_value_heads.
  • Prefix cache warm-up. Send 1,000 identical-prefix requests; hit rate must climb to >80% within 100 requests.
  • Context length ramp. Test at your max_model_len and 2× typical; latency must scale linearly, no OOM.
  • Batch size stability. Each CUDA graph capture size should produce similar per-token latency (within 20%).
  • Prefill-decode separation. With chunked prefill on, TPOT p99.9/p99 ratio should stay below 3×.
  • Attention mask at batch boundaries. Batched short+long request outputs bit-identical to solo runs.
  • FA version pinned exactly. flash-attn==2.6.3 — never >= or ~=.
  • Rollback rehearsed. One-command rollback verified, completes in <90 seconds.
  • Monitor two signals for 24 hours. flash_attn_kernel_time_ms p99 and prefix_cache_hit_rate on top-row dashboard.

Checkpoint · Knowledge Quiz

Eight questions to lock it in.

Module 3 · Self-check

Click answers to reveal explanations.

Q1 · The core bottleneck
Why does vanilla attention become impractical beyond a few thousand tokens of context?
The softmax is numerically unstable at long sequences.
It materializes an N×N scores matrix in HBM, which scales quadratically and becomes larger than available memory. It also does 6 round trips to HBM per layer for data that should never leave the SM.
The attention weights become too small to represent in FP16.
Tensor cores can't handle long matrices.
The N² scores matrix is the real problem. At N=32K, it's ~2 GB per attention head per layer — multiplying across 64 heads and 80 layers you exceed GPU memory just on intermediate tensors. FlashAttention's tile-streaming approach is the industry-standard fix — it never materializes the full S matrix in HBM.
Q2 · Online softmax
The online softmax trick that makes FlashAttention work:
Approximates the softmax to save compute.
Maintains running max and sum statistics across tiles and rescales partial outputs so the final result is bit-identical to full-sequence softmax.
Uses FP8 instead of FP32 for softmax.
Skips softmax for tokens far in the past.
The online softmax is a mathematical identity, not an approximation. By tracking (m, ℓ, O) and rescaling with exp(m_old − m_new) when a new max arrives, the final output exactly equals the non-streaming softmax. This is what makes FlashAttention a true drop-in replacement rather than a lossy approximation.
Q3 · FA-3 hardware requirement
Why does FlashAttention-3 require Hopper (H100/H200) and not run faster than FA-2 on Ampere (A100)?
FA-3 requires HBM3.
FA-3 uses Hopper-specific features: WGMMA async matrix-multiply, TMA async loads, and FP8 tensor cores. Ampere lacks these so FA-3 would silently fall back to FA-2 equivalents.
FA-3 requires 80+ GB of HBM.
It's a licensing restriction.
FA-3's speedup comes from overlapping matmul with softmax, which requires the async WGMMA instructions only Hopper has. On Ampere, you'd be using the same sync primitives as FA-2 — no benefit. This is why vLLM and TRT-LLM detect the GPU at startup and pick the right kernel.
Q4 · Why GQA-8 and not MQA
Llama-3 uses GQA with 8 K/V groups. Why not MQA (1 K/V head) for 8× more memory savings?
MQA is slower on tensor cores.
MQA loses ~0.5-1 MMLU points because one shared K/V limits the diversity of representations different Q heads can attend to. GQA-8 keeps accuracy essentially intact while still reducing KV 8×.
MQA isn't supported by FlashAttention.
MQA training is unstable.
Empirically, the accuracy-memory curve is steep near MQA and flat near GQA-8. Moving from 64 to 8 K/V heads costs almost nothing; moving from 8 to 1 costs meaningful accuracy. The industry converged on GQA-8 as the sweet spot for dense models.
Q5 · MLA tradeoff
Compared to GQA-8, MLA's main tradeoff is:
Worse accuracy for much smaller KV cache.
Comparable accuracy with much smaller KV, but requires training the model with MLA from scratch — you cannot retrofit it to an existing MHA/GQA checkpoint.
Better accuracy but 10× slower inference.
Only works for MoE models.
MLA retains accuracy (DeepSeek-V3 matches or exceeds dense GQA models) and reduces KV memory far more than GQA (576 B/token vs 320 KB). The catch: MLA requires specific training (decoupled rope, learned up-projections). You cannot take Llama-3-70B and apply MLA as a post-processing step.
Q6 · Attention sinks
Why do sliding-window attention schemes keep the first few tokens visible to all later tokens?
The first few tokens contain the system prompt which should always be visible.
The first tokens act as "attention sinks" that absorb excess attention mass. Removing them forces the softmax to redistribute that mass into recent tokens, distorting the attention distribution and collapsing quality.
It's needed for positional encoding.
The first tokens have better gradients.
Attention sinks are a real emergent phenomenon discovered by the StreamingLLM paper. Softmax normalizes to sum 1, so there's always attention going somewhere. With sliding windows, removing the sinks forces that mass onto less-relevant recent tokens. Keeping 4 sink tokens + sliding window is the standard fix.
Q7 · Ring Attention purpose
Ring Attention is primarily useful when:
You want to reduce KV cache size by 8×.
The full KV cache and activations for a single request don't fit on one GPU — typically at context lengths above 128K-1M tokens.
You want to speed up prefill 3×.
You want to reduce compute FLOPs.
Ring Attention shards the sequence across GPUs, which only becomes necessary when one GPU cannot hold all of a single request's tensors. For sub-128K contexts, tensor/pipeline parallelism is simpler and usually faster. Ring Attention is the tool for frontier million-token context workloads.
Q8 · The MLA absorbed matmul
MLA's "absorbed matmul" trick means that during decode:
The up-projection W_UK is never applied.
The up-projection is folded into Q at prefill (Q' = Q @ W_UK^T), so decode only does a small matmul against the compressed latent c_KV — smaller AND faster than MHA against the full K.
The KV cache is decompressed eagerly at the start of each decode step.
The attention score is approximated.
The absorbed matmul uses an algebraic identity: Q @ (c·W)^T = (Q @ W^T) @ c^T. Precomputing Q' shifts the work to prefill (where compute is cheap and abundant) and leaves decode with the small-matrix operation. This is why MLA can be faster and more memory-efficient than MHA simultaneously — it exploits the prefill/decode asymmetry.
