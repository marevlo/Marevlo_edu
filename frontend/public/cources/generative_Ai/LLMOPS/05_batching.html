01 · The Roofline, Revisited

Batch size is the highest-leverage knob.

Module 1 established the central fact of LLM serving economics: decode at batch 1 extracts 0.2% of the GPU's peak FLOPs. Every weight is read from HBM once, used for a tiny vector-matrix multiply, and discarded. The roofline says the cure is unambiguous — read each weight once, use it for many requests. That is batching.

What actually happens when batch size grows

For a decode step at batch size B, the arithmetic intensity of the forward pass scales linearly with B. The weights are loaded from HBM once and then multiplied by B different activation vectors sitting in SRAM. The bytes moved from HBM are unchanged; the useful work performed rises B-fold.

Decode arithmetic intensity vs. batch size (FP16) AI(B) ≈ 2B / D_bytes = B FLOP/byte   (for D=2, i.e. FP16)
Achievable = min(AI × HBM_BW, peak_FLOPS)

For Nexus on H100 (ridge at AI ≈ 295): batch 1 achieves 1.7 TFLOP/s (0.2% of peak); batch 32 achieves 54 TFLOP/s (5.4%); batch 128 achieves 214 TFLOP/s (22%); batch 256 sits right at the ridge, fully utilizing memory bandwidth. Throughput per GPU grows roughly linearly with batch size until the ridge, then saturates.

The batching economics, stated plainly

For a 70B FP16 model on H100 at batch 1, you pay for an entire GPU-hour to get ~50 tokens/sec. At batch 64 you pay for the same GPU-hour and get ~2,800 tokens/sec — 56× more output for the same money. Per-request TPOT barely moves (from 20 ms to 23 ms) because the work-per-token grows sub-linearly with batch until the ridge. This is why batching is not a nice-to-have — it is the difference between a viable business model and a loss leader.

02 · Three Generations of Batching

Static → dynamic → continuous.

The field has been through three distinct batching paradigms in five years. Understanding the differences — and why each generation was needed — is the key to choosing serving-engine configurations that actually work. The short version: do not use static or dynamic batching in production in 2026. Continuous batching (Orca, 2022) renders both obsolete for any workload with heterogeneous request lengths. But the path here is instructive.

Three Generations of Batching · The Same Four Requests, Three Scheduling Strategies

The mechanics of iteration-level scheduling

In continuous batching, the serving engine runs a tight loop: (1) collect pending sequences up to the configured max batch size, (2) run one forward pass for all of them simultaneously, (3) for each sequence, check if it emitted an EOS or hit max length, (4) evict finished sequences and admit new ones from the queue, (5) repeat. The forward pass itself is a completely standard decode-step forward pass with the batch dimension set to the current in-flight count.

continuous_scheduler.py · the Orca loop in ~30 linesPYTHON · SIMPLIFIED
# The beating heart of every modern serving engine (vLLM, SGLang, TGI).
# The real implementations add PagedAttention block allocation, prefill mixing,
# priority queues, and preemption — covered later in this module.

class ContinuousScheduler:
    def __init__(self, model, max_batch=256, max_new_tokens=2048):
        self.model = model
        self.max_batch = max_batch
        self.pending = collections.deque()    # newly arrived requests
        self.active  = {}                     # req_id -> SequenceState

    def submit(self, req_id, prompt_tokens):
        self.pending.append((req_id, prompt_tokens))

    def step(self):
        # 1. Admit new requests up to max_batch, run their prefill now or chunked.
        while self.pending and len(self.active) < self.max_batch:
            rid, toks = self.pending.popleft()
            self.active[rid] = SequenceState(toks)    # prefill runs here

        if not self.active:
            return []

        # 2. Build the current batch tensor from all active sequences' last tokens.
        batch_rids  = list(self.active.keys())
        last_tokens = torch.stack([self.active[r].last_token for r in batch_rids])
        kv_blocks   = [self.active[r].kv_block_ids     for r in batch_rids]

        # 3. ONE forward pass for ALL active sequences. This is where the win lives.
        next_tokens, _ = self.model.decode_step(last_tokens, kv_blocks)

        # 4. Update state. Evict finished sequences, yield their outputs.
        finished = []
        for i, rid in enumerate(batch_rids):
            st = self.active[rid]
            st.append(next_tokens[i])
            if next_tokens[i] == EOS or st.length() >= self.max_new_tokens:
                finished.append((rid, st.full_output))
                del self.active[rid]

        return finished

Three things to notice. First, the step function is called many times per second — once per decode step across the whole batch, which is 5-30 ms on a modern GPU. Second, new requests can join the batch as soon as there's room — no head-of-line blocking. Third, the whole scheduler is fewer than 50 lines of Python; the complexity lives in the PagedAttention kernel underneath and in the policy tweaks we discuss in Section 6.

💡

What Orca actually contributed

Before Orca (Yu et al., 2022), serving engines had batches-at-request granularity: you could add a request to a forming batch, but once it started, the batch was fixed. Orca introduced iteration-level scheduling — checking every single forward pass whether to evict/admit. This sounds trivial. It isn't: it required rewriting the attention kernels to handle batches with different sequence lengths (different padded shapes break vanilla attention) and teaching the KV cache to handle sequences coming and going mid-batch. PagedAttention (Module 4) solved the KV side; Orca's contribution was recognizing that iteration granularity was the right abstraction. Every modern engine uses variations of this design.

03 · Speculative Decoding · Emit Multiple Tokens Per Forward Pass

A draft model writes ahead; the big model verifies.

Continuous batching maximized throughput per GPU-second. It did nothing for per-request latency — each request still waits one big-model forward pass per output token. If a Nexus response has 180 tokens and decoding takes 5 ms per step, that is a 900 ms floor that no batching optimization can breach. The technique that breaks this floor is speculative decoding (Leviathan et al. 2023; Chen et al., 2023), and it works via a beautifully simple statistical trick: let a small cheap model draft K tokens ahead, then use the big model to verify them in parallel with a single forward pass. On acceptance, you emit K tokens for the cost of one big-model step.

The core insight — one big-model pass checks many draft tokens

The key observation is that during an autoregressive forward pass on a sequence of length N, the big model already computes the logits at every position simultaneously (attention sees all prior context via the KV cache). Normally we only use the logit at the last position. But if we pre-extend the sequence with K speculated tokens from a draft model, the big model's single forward pass gives us logits at positions N, N+1, N+2, ..., N+K — telling us what the big model would have predicted at each of those positions. We can accept the draft tokens where they agree and reject where they disagree, all in one forward pass.

Speculative Decoding · The Draft-and-Verify Cycle
4 tokens in 1 big-model forward pass. Cost: 1 big forward pass + K=5 cheap draft passes. Draft cost is typically 5-15% of one big-model pass. Speedup: E[tokens per big pass] / big_pass_cost ≈ 2.5–3.5× in practice for well-chosen draft models. THE CRUCIAL GUARANTEE Speculative decoding is mathematically token-for-token identical to running the big model alone. No quality loss. See the proof in Section 4. Acceptance probability per draft token: min(1, p(x)/q(x)) where p is big-model, q is draft. On rejection at position i, we re-sample from the normalized distribution (p - q)₊ — preserves the big model's distribution exactly.

Why the rejection-sampling trick preserves the distribution

This is worth slowing down on because most engineers treat it as magic. It isn't — it's a classical rejection-sampling result. The full statement:

Speculative sampling — the distributional identity Let p = big model probability, q = draft model probability over vocabulary V.

1. Draft samples x ~ q(·).
2. With probability min(1, p(x)/q(x)), accept x.
3. Otherwise reject x and sample from the normalized residual:
    r(y) = max(0, p(y) − q(y)) / Σ max(0, p(z) − q(z))

Then the distribution of the emitted token is exactly p.

The proof is short enough to do here. Let x be the emitted token. We want to show Pr(x) = p(x). Two disjoint paths lead to emitting x:

Path A: draft samples x AND we accept.
  Pr(A, x) = q(x) · min(1, p(x)/q(x)) = min(q(x), p(x))

Path B: draft samples any z ≠ x, we reject, then resample x from r.
  Pr(reject) = Σ_z q(z) · (1 − min(1, p(z)/q(z))) = Σ_z max(0, q(z) − p(z))
  Pr(resample = x | reject) = max(0, p(x) − q(x)) / Pr(reject)
  Pr(B, x) = Pr(reject) · Pr(resample=x | reject) = max(0, p(x) − q(x))

Total: Pr(x) = min(p(x), q(x)) + max(0, p(x) − q(x)) = p(x) ✓
Σ

The two terms always sum to p(x)

For any two non-negative numbers a and b, min(a, b) + max(0, a − b) = a. The proof collapses to that identity — the acceptance probability was engineered precisely to make this work out. This is why speculative decoding is not an approximation: there is no accuracy tradeoff, no quality knob, no "mostly as good as the big model". It is the big model's distribution, sampled through a cheaper procedure. When people claim speculative decoding "degrades quality a little", they are either using an incorrect implementation (common enough — draft-model temperature mismatches are a frequent bug) or conflating speculative with another method.

Greedy decoding — the simpler case

If you are doing greedy decoding (temperature=0), the rejection-sampling step degenerates: accept if the draft's argmax equals the big model's argmax at that position, reject otherwise. No sampling, no residual distribution. This is how speculative decoding is implemented in production for deterministic workloads like Nexus — simpler code path, same guarantee.

04 · Speculative Decoding in Code

Accept, reject, resample.

Here is the actual algorithm, production-faithful. vLLM, TensorRT-LLM, and SGLang all implement variants of this same structure; the kernel-level optimizations differ but the scheduling logic is the same.

speculative_step.py · core decoding loopPYTHON
# Simplified but correct speculative decoding. For each iteration: draft K tokens,
# verify in one big-model forward pass, emit as many as accepted, recurse.

import torch

def speculative_step(
    big_model,          # the 70B target model
    draft_model,        # 1B fast draft
    prefix,             # tokens emitted so far
    K: int = 5,       # speculation length
    temperature: float = 0.0,
):
    # ── STEP 1 · Draft K tokens serially from the cheap model ──
    draft_tokens, draft_probs = [], []
    cur = prefix
    for _ in range(K):
        logits = draft_model(cur)[:, -1, :]
        q = torch.softmax(logits / max(temperature, 1e-6), dim=-1)
        tok = q.argmax(-1) if temperature == 0 else torch.multinomial(q, 1)
        draft_tokens.append(tok)
        draft_probs.append(q)
        cur = torch.cat([cur, tok], dim=-1)

    # ── STEP 2 · ONE big-model forward pass over prefix + all K drafts ──
    # Big model emits logits at EVERY position — we use positions -K-1 to -1.
    big_logits = big_model(cur)[:, -(K+1):-1, :]   # shape [B, K, V]
    big_probs  = torch.softmax(big_logits / max(temperature, 1e-6), dim=-1)

    # ── STEP 3 · Walk left to right, accept / reject each draft token ──
    accepted = []
    for i in range(K):
        p = big_probs[:, i, :]
        q = draft_probs[i]
        d = draft_tokens[i]

        if temperature == 0:
            # greedy: accept iff big's argmax equals the drafted token
            if (p.argmax(-1) == d).all():
                accepted.append(d)
            else:
                # on first disagreement, emit big's choice and stop
                accepted.append(p.argmax(-1, keepdim=True))
                return torch.cat(accepted, dim=-1)
        else:
            # rejection sampling — accept with prob min(1, p/q)
            ratio = p.gather(-1, d) / q.gather(-1, d).clamp(min=1e-9)
            if torch.rand(1) < ratio.clamp(max=1.0):
                accepted.append(d)
            else:
                # rejected — resample from (p - q)+ normalized
                resid = (p - q).clamp(min=0)
                resid = resid / resid.sum(-1, keepdim=True).clamp(min=1e-9)
                new_tok = torch.multinomial(resid, 1)
                accepted.append(new_tok)
                return torch.cat(accepted, dim=-1)

    # All K drafts accepted. As a bonus, we can use big's logits at position K
    # to sample ONE more token "for free" (no additional forward pass needed).
    bonus = big_model(cur)[:, -1:, :].argmax(-1)
    accepted.append(bonus)
    return torch.cat(accepted, dim=-1)

The acceptance rate — how much you actually win

The expected number of tokens emitted per big-model forward pass is:

Expected tokens per big-model pass, speculation length K E[tokens] = (1 − α^(K+1)) / (1 − α)   where α = per-token acceptance rate

speedup ≈ E[tokens] / (1 + c·K)   where c = draft cost / big cost

The critical parameter is α — the probability that a draft token is accepted. Empirically:

When speculative decoding hurts you

If α × draft_cost_ratio falls below a threshold (roughly α < 0.35 for c=0.08), speculative decoding is slower than vanilla decoding — the draft cost exceeds the expected number of accepted tokens. Two situations where this happens in the wild: (1) high-temperature creative sampling where the big model's distribution is genuinely high-entropy and no draft can predict it reliably; (2) domain mismatch where the draft model was trained on different data. The fix in both cases is to either tune the draft for the target distribution (cheap — one fine-tune) or abandon speculation for those workloads. For Nexus's deterministic structured outputs, α=0.88 is routine after a 4-hour draft-model fine-tune on 50k production traces.

Choosing K — the speculation length

K is the only non-trivial knob. Larger K means more drafted tokens per iteration — great when acceptance is high. But draft cost grows linearly with K, and once the chain breaks all subsequent draft tokens are wasted. The optimal K depends entirely on α:

Acceptance rate αOptimal KE[tokens / big pass]Notes
0.957-105.5-7.2Very high-confidence drafts — rare outside specific structured tasks
0.8553.7Nexus-style — this is where most production speculation lives
0.703-42.5-2.9General chat
0.5021.75Borderline — tune draft or disable speculation
0.3511.3Barely worth it

The practical rule: benchmark your actual workload, find α, then pick K such that α^K ≈ 0.5 — that's where the next drafted token has roughly 50% chance of acceptance, balancing the cost of producing it against the expected payoff. For α=0.85 this gives K=5, matching the production default.

05 · Self-Speculation Variants

Skip the draft model entirely.

Classical speculative decoding needs two models — the target and a smaller draft. Running and tuning two models is operationally annoying: you have to version them together, keep them aligned across training cycles, and pay a second set of inference costs. A family of newer methods eliminates the draft model by reusing the big model itself to propose candidate tokens. The three you should know are Medusa, EAGLE-2, and Lookahead decoding. Each trades differently between engineering complexity and achievable speedup.

Medusa — multiple output heads

Medusa (Cai et al., 2024) adds K small "Medusa heads" on top of the target model's last hidden state. Each head is trained to predict the token K positions ahead. During inference, one big-model forward pass emits K+1 candidate tokens simultaneously — the main head's next token plus K speculated futures. These are verified using tree-based speculative decoding: we don't commit to a single draft sequence but to a small tree of likely continuations, any branch of which the big model might accept.

Medusa · Multiple Decoding Heads + Tree Verification

EAGLE-2 — dynamic draft tree

EAGLE (Li et al., 2024) and its successor EAGLE-2 (2024) build a feature-level auto-regressive draft mechanism. A small transformer trained on the target model's hidden states (not output tokens) drafts multiple tokens ahead. EAGLE-2 adds dynamic tree construction: paths in the draft tree are expanded or pruned based on confidence, so compute budget flows to the most promising branches.

Measured: ~3.0-3.5× speedup on standard benchmarks, no separate draft model required. Currently the state-of-the-art self-speculation method for 2026 deployments. Supported in vLLM and SGLang.

Lookahead Decoding — n-gram based

Lookahead (Fu et al., 2024) avoids training anything new. It maintains a pool of n-grams observed in the current generation, uses Jacobi iteration to refine multiple parallel token chains in each forward pass, and verifies n-gram matches against the big model. Works especially well on code generation where repetitive patterns are common.

Measured: 1.5-2.2× speedup, zero additional training cost. Best option when you cannot retrain any auxiliary model. Ships with DeepSeek's serving stack.

💡

How to choose

Decision tree: (a) If you have production traces and compute for fine-tuning a small domain-specific draft, classical speculative decoding with a domain-tuned 1-2B draft wins — α of 0.85+ and clean operational story. This is Nexus's choice. (b) If you want a single-model operational story but can train Medusa/EAGLE heads, EAGLE-2 is the best self-speculation method. (c) If you cannot train anything at all, Lookahead decoding is zero-config and gives 1.5-2× essentially free. All three compose cleanly with continuous batching — you just set the batch's effective decode step to the max accepted prefix length.

06 · Priority, Preemption, and SLO-Aware Scheduling

When multiple tenants, multiple SLOs.

Continuous batching, as described so far, treats every request identically — FIFO within the queue. Real production serves multiple tenants with different contracts: Nexus's checkout path needs 80 ms p99; its offline reconciliation jobs tolerate 30-second latency. Both hit the same cluster. A naive FIFO scheduler serves the reconciliation job when it arrives first, blocking the checkout even though the latter has a much tighter deadline. The fix is SLO-aware priority scheduling.

The scheduling knobs that matter

PolicyWhat it doesWhen to useTradeoff
FIFOStrict order of arrivalSingle-tenant, uniform workloadHead-of-line blocking by big prefill
SJF (Shortest Job First)Prefer short prompts / short expected outputImproves mean latencyStarves long jobs, manipulable

Preemption — and the 95% rule

In an ongoing decode, a new high-priority request arrives. Two choices: (a) wait until an in-flight sequence finishes and release its slot, or (b) preempt — suspend a running request, save its state, admit the new one. Preemption is technically straightforward with PagedAttention (the KV blocks go onto a "swapped out" list; the request re-enters the queue), but it is expensive: on re-admission, you replay from the last committed checkpoint, which is wasted compute.

The practical rule we use for Nexus: never preempt unless the in-flight request is within 5% of its own SLO deadline. If the running request's deadline is not urgent, let it finish to avoid wasted work. Only trigger preemption when a new request's deadline is tight AND a running request's budget is already mostly consumed. This is conservative but empirically gives the best SLO compliance.

nexus_scheduler.py · deadline-aware with conservative preemptionPYTHON
# Simplified version of Nexus's production scheduler. Each request carries
# a deadline (arrival + SLO). The scheduler sorts by earliest deadline first
# (EDF) and preempts only when the cost is justified.

from heapq import heappush, heappop

class EDFScheduler:
    def __init__(self, max_batch=128):
        self.max_batch = max_batch
        self.queue    = []        # heap of (deadline, rid, req)
        self.active   = {}        # rid -> ActiveState(deadline, tokens_emitted, ...)

    def submit(self, rid, req, slo_ms):
        deadline = time.time() + slo_ms / 1000
        heappush(self.queue, (deadline, rid, req))

    def pick_batch(self):
        # Never touch active requests that are in safe margin.
        while self.queue and len(self.active) < self.max_batch:
            deadline, rid, req = heappop(self.queue)
            self.active[rid] = ActiveState(deadline=deadline, req=req)

        # Check for deadline-miss risk — preempt only if needed.
        if self.queue and len(self.active) == self.max_batch:
            new_deadline = self.queue[0][0]

            # Find the active request with the "loosest" deadline.
            victim = max(self.active.values(),
                         key=lambda s: s.deadline - time.time())
            victim_budget_left = victim.deadline - time.time()
            new_budget_left    = new_deadline     - time.time()

            # The 95% rule: preempt only if
            #  (a) new request is already at risk of missing SLO
            #  (b) victim has more slack than the new request
            #  (c) victim has not yet consumed > 95% of its own budget
            if (new_budget_left < 0.08 and
                victim_budget_left > new_budget_left and
                victim.budget_consumed_frac() < 0.95):
                self.preempt(victim)
                _, rid, req = heappop(self.queue)
                self.active[rid] = ActiveState(deadline=new_deadline, req=req)

        return self.active

Admission control — the overlooked last resort

If the system is genuinely overloaded — incoming rate > drain rate — no scheduling policy can save the SLO. Every request will queue longer. At that point you have two choices: (1) degrade everyone, or (2) reject some requests so the remaining ones meet SLO. Production systems do (2) — called admission control. When measured queue depth × expected processing time exceeds some threshold, newly arrived low-priority requests get a 429. This is the mechanism that prevents one bursty tenant from taking down the whole cluster.

⚠

The feedback loop you need observability for

Scheduling decisions are only as good as the signals feeding them. For EDF to work, you need accurate "expected remaining time" for each active request — which requires knowing how many tokens it will still emit. This is unknowable in general. The best proxy: measure the per-request output-length distribution in production, maintain a running histogram, and use the p90 length as the deadline projection. When that prediction is off by > 30% regularly, SLO misses spike. Module 7 covers the observability setup that makes this kind of adaptive scheduling possible.

07 · Core Operational Math

Four formulas for batch sizing, speculation, and scheduling.

Everything in this module so far explains how each technique works. This section gives the formulas that turn batching and speculation into a sizing exercise. Four formulas, four worked examples, four verdicts. The inputs are your model shape, hardware, and SLO; the outputs are the numbers that go directly into your config.

Formula 1 · Decode-step time under continuous batching

Decode is memory-bandwidth-bound. Each step reads all weights (amortized across the batch) plus KV cache for every concurrent request. Per-token throughput rises linearly with batch until HBM bandwidth saturates, then plateaus.

Per-step time for a batch step_ms = (weight_bytes + batch × context_tokens × kv_bytes_per_token) / hbm_bw × 1000
       + launch_overhead_ms

TPOT_per_request_ms = step_ms   (each request gets 1 token per step)
throughput_tok_per_sec = batch / (step_ms / 1000)

Knee of curve: batch at which 90%+ of peak throughput is achieved
SLO-compliant max: largest batch where TPOT < SLO

Worked example · Llama-3-70B AWQ + FP8 KV on H200

weight_bytes = 37 GB, kv_per_tok = 160 KB, hbm_bw = 4400 GB/s, launch = 0.35 ms (CUDA graphs). At batch=64, context=2000: step = (37 + 64×2000×160/1024³) / 4400 × 1000 + 0.35 = 12.85 + 0.35 = 13.2 ms. At batch=128: 17.64 ms. For a 15 ms TPOT SLO, batch=64 is the max. Going to batch=128 doubles concurrency but misses SLO by 17%. This is why Nexus v5 ships max_num_seqs=64. batching_throughput_model.py in the artifact pack computes this for any model/hardware combo.

Formula 2 · Speculative decoding expected tokens per pass

The single number that decides whether speculation is worth deploying. Given acceptance rate α and speculation length K, how many tokens does the big model emit per verification pass on average?

Expected tokens per speculation iteration E[tokens] = (1 − α^(K+1)) / (1 − α)   (when α < 1)
      = 1 + α + α² + ... + α^K

iter_time_ms = K × draft_ms + target_ms
effective_TPOT = iter_time / E[tokens]
speedup = target_ms / effective_TPOT

Worked example · Nexus v5 · α=0.85, K=5, draft=1.2 ms, target=12 ms

E[tokens] = (1 − 0.85⁶) / (1 − 0.85) = (1 − 0.377) / 0.15 = 4.15 tokens/iter. iter_time = 5 × 1.2 + 12 = 18 ms. effective_TPOT = 18 / 4.15 = 4.33 ms. speedup = 12 / 4.33 = 2.77×. Per-token cost drops 64% — this is the headline speculation deliverable. At α=0.60 (drifted draft), speedup drops to 1.59×; at α=0.95 it rises to 3.53×. Every 0.05 of α drift swings speedup by ~10%. speculative_decoding_calculator.py runs this for any α/K/timing combination with a sensitivity sweep.

Formula 3 · Scheduler SLO compliance under load

For a mixed-SLO workload at arrival rate λ and service capacity μ, FIFO breaks when utilization ρ = λ/μ exceeds about 0.7. Above that threshold, interactive-class SLO collapses under FIFO because batch requests hog capacity.

Why FIFO fails on mixed workloads at high utilization Under FIFO at utilization ρ:
  E[wait_time] = service_time × ρ / (1 − ρ)  (M/M/1 approx)

At ρ = 0.75: interactive waits = service_time × 3
At ρ = 0.90: interactive waits = service_time × 9

Priority scheduling gives: interactive waits ≈ service_time (independent of ρ)
because it jumps the queue. Batch requests absorb the full wait.

Worked example · mixed workload at 75% utilization

Capacity 1000 req/s, arrival 750 req/s (ρ=0.75). Service time for interactive is ~60 ms. Under FIFO: expected wait is 60 × 3 = 180 ms, so TTFT = wait + service = 240 ms, which blows the 200 ms interactive SLO. Under priority: interactive wait ≈ 0 (jumps to front of queue), TTFT ≈ 60 ms, well inside SLO — but batch TTFT balloons. The scheduler_simulator.py artifact shows this empirically: at 750 req/s on the Nexus workload, FIFO gives 1.2% interactive SLO compliance; priority gives 100%.

Formula 4 · Preemption threshold

Preemption happens when the KV pool can't fit the max_num_seqs requests in flight. At that threshold, the scheduler evicts requests to make room; each eviction is a latency hit and a potential correctness risk under recompute-mode.

When preemption kicks in expected_kv_demand = max_num_seqs × E[context_tokens] × kv_bytes_per_token

preemption_threshold = expected_kv_demand / kv_pool_bytes

Healthy: preemption_threshold < 0.85 (room for variance)
Warning: 0.85 - 0.92 (preemption rate 1-5%)
Broken: > 0.92 (preemption storm, TPOT instability)

Worked example · Nexus v5

max_num_seqs=64, E[context]=2000 tokens, kv_per_tok=160 KB, kv_pool=99 GB per GPU × 4 GPUs (TP=4) = 396 GB effective. demand = 64 × 2000 × 160 KB = 19.5 GB per GPU → 78 GB pool-wide. threshold = 78 / 396 = 0.20. Very healthy — the system has 5× headroom for context-length variance. If traffic shifted to 16K mean context, demand would be 64 × 16000 × 160 KB ≈ 156 GB, threshold 0.40 — still healthy but with less margin. Preemption is a capacity problem, not a scheduler problem; fix it by reducing max_num_seqs or upgrading hardware.

🔑

The overall framing

Batching and scheduling are fundamentally resource allocation problems. Formula 1 tells you how many requests fit per unit time. Formula 2 tells you whether speculation multiplies that capacity. Formula 3 tells you whether the scheduler can preserve per-class SLOs under load. Formula 4 tells you where the cliff is. When deploying, run the calculators first. When debugging, the formulas tell you which number is off — which is always the first question: "is this a capacity issue or a policy issue?"

09 · Debugging Drills

Five symptoms. Five diagnostic paths.

Five drills covering the most common batching, speculation, and scheduling failures: acceptance-rate drift, preemption storms, FIFO starvation, batch-size misconfiguration, and speculation correctness issues. Each drill names the symptom, the wrong hypothesis that wastes hours, the correct diagnosis, and the fix.

Drill · 01
Speculation acceptance rate dropped 0.85 → 0.62; TTFT still fine; nobody alarmed
SYMPTOM
Cost-per-token has drifted up 35% over the last 8 weeks. Request volume is flat. All infra dashboards green — GPU util, TTFT p99, TPOT, SLO compliance all normal. Finance flags it before engineering does.
WRONG HYPOTHESIS
"Finance must be miscounting — request count probably drifted". Wastes a week doing reconciliation while cost keeps rising.
CORRECT DIAGNOSIS
Run draft_model_acceptance_tracker.py on the last 90 days of decoding logs. The output will show α drifting downward over time. The mechanism: production traffic distribution has shifted away from the draft model's training snapshot. α drops → tokens-per-big-pass drops → cost-per-token rises. Check vllm:spec_decode_accepted_tokens_total / _draft_tokens_total per week; if the ratio has dropped >0.10 vs last retraining, the draft is stale.
FIX
Retrain the draft model on a rolling 14-day window of current traffic. Deploy via canary (10% traffic for 24h, verify α ≥ target on canary, then promote). Set up the retraining pipeline to run weekly automatically; set up a Prometheus alarm at α < 0.70 for 1h. This is exactly the M8 war story #6 scenario — the fix is cheap, missing the alarm is the expensive part.
Drill · 02
Preemption rate spiked to 18%; TPOT p99.9 unstable
SYMPTOM
vllm:num_preemptions_total growing rapidly. TPOT p99 is 15 ms (green) but p99.9 is 110 ms (amber). Request queue depth fluctuating 0 → 40 → 5 → 30. KV cache usage reading 95-98%.
WRONG HYPOTHESIS
"The scheduler is being too aggressive about preempting — tune it." Scheduler isn't the problem. KV budget is.
CORRECT DIAGNOSIS
Preemption happens when the KV pool can't fit max_num_seqs × context_tokens × kv_per_token. Two possible causes: (a) max_num_seqs is set higher than your real KV budget supports, or (b) request lengths drifted longer than when max_num_seqs was tuned. Check the preemption-threshold formula from section 07; if threshold > 0.92, you're in the "broken" zone. Also check vllm:gpu_cache_usage_perc history — if it was 70% last month and 95% now with same config, traffic grew.
FIX
Immediate: reduce max_num_seqs by 25% to pull out of the preemption storm. Then re-run paged_kv_simulator.py (from the M4 artifact pack) with the new request-length distribution to find the right max_num_seqs for current traffic. If traffic growth is durable, add hardware — TP=8 instead of TP=4, or move to H200. Don't try to absorb growth via scheduler tuning; you can't.
Drill · 03
FIFO scheduler at 75% capacity — interactive TTFT p99 at 4×SLO
SYMPTOM
Interactive-class SLO is 200 ms TTFT. Measured: 840 ms p99. System is at 75% of theoretical capacity. FIFO scheduler. GPU util 82% (normal). No preemption. Long-running batch requests dominate the decode pool.
WRONG HYPOTHESIS
"We're close to capacity — add more GPUs." Throws hardware at a policy problem. Capacity IS sufficient; the scheduler is serving the wrong requests first.
CORRECT DIAGNOSIS
FIFO treats all requests identically. At 75% utilization (ρ=0.75), FIFO queue wait approximates service_time × 3 (M/M/1). For a batch request with 800 output tokens, that's ~180 ms of wait — then an interactive request arrives and waits another 180 ms behind it. The scheduler is the bottleneck, not the hardware. Confirm by running scheduler_simulator.py with your current arrival rate and class mix: priority will show 100% interactive SLO where FIFO shows ~1-5%.
FIX
Switch to scheduling_policy: priority with tenant tiers. Tag every request at the gateway with its class. Accept that batch SLO will degrade (priority starves batch by design) — if batch has its own hard SLO, route batch to a dedicated pool instead of competing. This is a 10-line config change; expect interactive SLO compliance to jump from ~3% to ~100% immediately. See vllm_batching_config.yaml for the exact config.
Drill · 04
TPOT doubled after rollout; SLO missing on 40% of requests
SYMPTOM
Deployed new config Friday. Monday morning: TPOT p50 went from 8 ms to 16 ms. p99 from 15 ms to 30 ms. Throughput slightly up (+8%). GPU util slightly up (+5%). 40% of interactive requests now missing TPOT SLO.
WRONG HYPOTHESIS
"Something broke in the kernel — open a vLLM issue." Not broken. A config change increased batch size above SLO threshold.
CORRECT DIAGNOSIS
Friday's deploy raised max_num_seqs from 64 to 128. Batch=128 gives higher throughput (as observed) but TPOT doubles because decode step time is linear in batch (memory-bound regime). Formula 1 shows this directly: at Llama-3-70B-FP8 on H200, batch=64 → step=13 ms, batch=128 → step=17.6 ms. The deploy optimized for the wrong metric (throughput, not SLO compliance). Run batching_throughput_model.py with the current SLO; the SLO-compliant recommendation will be batch=64.
FIX
Revert max_num_seqs to 64. Document that peak-throughput batch ≠ SLO-compliant batch; always pick the SLO-compliant one unless the workload is genuinely batch-only. For teams that want more throughput at the SLO-compliant batch, the lever is hardware (H200 → B200) or quantization (FP16 KV → FP8 KV → INT4 KV), not batch size. This regression is fundamental, not fixable by tuning.
Drill · 05
Speculation enabled but outputs don't match baseline
SYMPTOM
Comparing 1,000 requests with speculation on vs off (same model, same seed, same temp=0): 5-8% of requests produce different top-1 tokens. Quality eval shows a 0.3 F1 regression on speculation-on. Production tenant complains outputs changed.
WRONG HYPOTHESIS
"Speculation is lossy — accept the degradation for the speedup." Wrong on the math. Speculation IS lossless when implemented correctly; any divergence is a bug.
CORRECT DIAGNOSIS
Speculative decoding is mathematically lossless when the rejection sampling is correct: min(p_target, p_draft) accept, resample on reject, with no off-by-one in position indexing. Any token divergence >0.1% means a bug. Check: (a) draft and target use the same tokenizer (version-pin both); (b) sampling temperature applied to target logits only (draft logits are just for proposing); (c) the rejection sampling formula is the actual min(1, p_t/p_d) accept probability, not a shortcut; (d) under temp=0 (greedy), speculation should produce bit-identical output to non-speculation.
FIX
Run a parity test: 1,000 prompts at temp=0, speculation on vs off. Expect bit-identical output. If it's not bit-identical, disable speculation until the bug is found. Common bugs in order of likelihood: tokenizer mismatch between draft and target, off-by-one in accept sampling (target emits at wrong position), or temperature applied at the wrong stage. Do not deploy speculation with known divergence — it's not faster if the outputs are wrong.

Mapping to real symptoms

In production, drift doesn't happen at a defined rate. It happens gradually as tenants onboard, seasonal patterns shift, and new use cases emerge. The three fingerprints:

1. Cost rising faster than volume

Monthly cost-per-request growing >3% while request count is flat. Almost always a speculation (or quantization) regression.

2. All latency metrics still green

TTFT, TPOT, SLO compliance all normal. The regression is in the cost dimension, not the latency dimension.

3. Per-tenant α varies

Old tenants' α still ~0.85; new tenants' α ~0.60. The average drifts downward as new tenants onboard.

💡

The design principle

Any production optimization with a "training-time assumption" eventually drifts. Speculation's assumption is "draft model matches target on real traffic." Quantization's assumption is "calibration set matches production traffic." Prefix caching's assumption is "prompts share a stable prefix." All three need an alarm on the assumption's leading indicator, not on the downstream cost. For speculation, that leading indicator is α. Set the alarm at 0.70. Retrain when it fires. The whole cost regression is avoidable if the alarm is wired.

Q1 · Batching economics
Decoding a 70B FP16 model on H100: batch 1 achieves ~50 tok/s. Approximately what does batch 64 achieve?
~100 tok/s
~500 tok/s
~2,800 tok/s (weights read once, used 64× — still memory-bound but throughput scales linearly with batch up to the ridge)
~50 tok/s (no change — batching doesn't help decode)
At batch 1 the time per decode step is ≈ weight_bytes / HBM_BW (memory-bound). At batch 64 the same HBM read serves 64 requests — so throughput is roughly batch × batch-1 rate, until arithmetic intensity crosses the H100 ridge at ~295 FLOP/byte. For 70B at batch 64, AI ≈ 64 FLOP/byte, still well below ridge, so we stay in memory-bound regime and throughput scales near-linearly.
Q2 · Static vs continuous
What is the core problem with static batching that continuous batching solves?
Static batching doesn't support attention masks.
Static batching completes only when the LONGEST sequence finishes — short requests in the batch wait idly for the longest one to emit EOS. Continuous batching re-admits at every iteration, so short requests return as soon as they're done.
Static batches are limited to batch size 8.
Static batching cannot use FlashAttention.
The head-of-line blocking is the problem. For workloads with varied output lengths (which is essentially all LLM workloads), static batching multiplies the average latency by the ratio of max to mean output length — often 5-20×. Orca's iteration-level scheduling eliminates this by allowing the batch composition to change after every forward pass.
Q3 · Speculative decoding guarantee
True or false: speculative decoding produces a token distribution that is a close but imperfect approximation of the big model's distribution.
True — it's a quality/speed tradeoff.
False. The rejection-sampling construction is designed so the emitted-token distribution is EXACTLY the big model's distribution. It is a pure speed optimization with zero quality loss when implemented correctly.
Only true for greedy decoding.
True only when the draft model is much smaller.
The distributional identity min(p,q) + max(0, p-q) = p is the whole point. Any speculative decoder that is mathematically correct produces bit-identical (in expectation) outputs to vanilla decoding. If you observe quality differences, you have an implementation bug — most commonly a temperature mismatch between draft and target.
Q4 · Acceptance rate
Nexus achieves speculative acceptance rate α = 0.88 with a domain-tuned 1B draft. If each draft-step costs 8% of a big-model step and speculation length K=5, what is the approximate realized speedup?
1.5×
2.0×
≈ 3.1×
5.0×
E[tokens per big pass] = (1 − 0.88⁶) / (1 − 0.88) ≈ 3.97. Cost per big pass with drafting = 1 + 5·0.08 = 1.4 big-pass-equivalents. Speedup ≈ 3.97 / 1.4 ≈ 2.84× — call it 3× after the "bonus token" when all drafts are accepted. The table in Section 4 shows 3.1× measured, matching the formula.
Q5 · Speculation failure mode
A team enables speculative decoding on a creative writing use case and observes it becomes slower. What's the most likely cause?
The draft model is too large.
High-temperature sampling gives the big model a genuinely high-entropy distribution that no draft can predict well. Acceptance α drops below the break-even threshold (roughly 0.35) and the draft cost exceeds expected savings.
Speculative decoding doesn't work with creative writing.
The draft model has fewer parameters than required.
Speculative decoding needs the draft to usefully predict the target. When the target samples at high temperature, even its own next-step distribution is near-uniform — there is nothing a draft can usefully predict. For such workloads either lower temperature, switch to self-speculation (which shares backbone knowledge), or disable speculation.
Q6 · Choosing K
For a workload with acceptance rate α = 0.70, what is the appropriate speculation length K?
K = 1 (minimal speculation)
K ≈ 3-4 (balances draft cost against expected run length ≈ 2.6 tokens)
K = 10 (more speculation = more savings)
K depends only on the draft model size.
Rule of thumb: pick K such that α^K ≈ 0.5, so the "last drafted token" has roughly coin-flip odds of being accepted. For α=0.70, K=2 gives 0.49, K=3 gives 0.34 — so K=2-3 is the sweet spot. Going higher pays for draft tokens that are probably going to be rejected.
Q7 · Self-speculation
What is the main OPERATIONAL advantage of Medusa or EAGLE-2 over classical speculative decoding?
Higher acceptance rates than draft-model speculation.
No separate draft model to version, deploy, tune, or keep aligned with the target across training iterations. One model to ship, one model to monitor.
Lower acceptance rates that are easier to analyze.
They support higher max batch sizes.
The pure speedup is typically comparable or slightly lower than a well-tuned classical draft. The real win is operational: one model, one deployment, no "drift" between target and draft across fine-tuning cycles. For teams with limited ops capacity, self-speculation is often the pragmatic choice.
Q8 · Scheduling policy
A single GPU cluster serves both real-time checkout (80 ms SLO) and offline reconciliation (no SLO). Under FIFO scheduling during a reconciliation burst, checkout latency spikes. Which policy best addresses this WITHOUT starving reconciliation?
Strict priority (checkout always first)
Deadline-aware (EDF) — reconciliation gets distant deadlines but still runs when checkout has slack, so it completes with modestly higher latency but is never starved entirely.
Shortest-job-first
Random sampling
Strict priority starves low-priority work under sustained checkout load. EDF with per-request deadlines naturally time-shares: when checkout queue is light, reconciliation runs; when checkout is heavy, it gets priority because its deadlines are closer. Both classes meet their contracts.
Q9 · Preemption
An in-flight request has emitted 120 of its expected 180 tokens and is within its SLO budget. A new higher-priority request arrives. Should the scheduler preempt?
Yes, always — higher priority wins.
Probably not — the in-flight request is 67% complete and preempting it wastes 120 tokens of compute. Only preempt if the new request is at deadline risk AND the victim is either very early (little work wasted) or very late (near its budget limit).
Yes, if the new request has higher priority by any margin.
Only if the victim has never been preempted before.
Preemption has real cost — the replayed work is pure waste. The right criterion compares the marginal benefit (new request meets SLO) to the marginal cost (victim's replay cost + impact on its SLO). Empirically the "95% rule" — preempt only victims that are > 95% through their budget or new arrivals within 8% of deadline — gives excellent SLO compliance with minimal wasted work.
