The system around
the model.

A serving engine that survives a benchmark is not a production system. Real LLM deployments are traffic-shaped across multiple regions, autoscaled by signals the Kubernetes defaults never heard of, rolled out through shadow mode and canary, monitored for quality drift as well as latency, and observable at the level of tenants, tokens, and tail percentiles. This module covers everything outside the model — routing, scaling, deployment, testing, observability, incident response — the stack that separates "the model is fast" from "the service holds up at 10k req/s on Black Friday." 01 · Multi-Region Topology

One region is not a production system.

A single region of Nexus can serve 10,000 req/s with 48 ms p99. It can also go dark for six hours when AWS us-east-1 has a control-plane incident, when a cloud provider's HBM pool depletes mid-event, or when a BGP misconfiguration routes a continent's traffic into a wormhole. Serious production means two things: multiple geographically independent regions, and a routing layer that fails over between them in seconds without losing in-flight requests. The naive pattern most teams reach for — active/passive with DNS failover — does not work for LLM serving. Here's what does.

The three region topologies and their tradeoffs

TopologyHow it worksFailover timeCost overheadWhen it fits
Active / PassiveOne live region, one warm standby, DNS flip on failure3-15 min (DNS TTL + GPU warmup)~55% (standby is idle)Compliance-heavy use cases that can tolerate minutes of downtime; rarely correct for user-facing LLMs
Active / Active (2 region)Both regions serve live traffic, 50/50 geo-split~5 sec (health-check loss)~40% (each region sized for full load)The sensible minimum for user-facing LLMs
Active / Active (N≥3)3+ regions, any one can absorb failover~5 sec~(N−1)/N headroom (e.g. 33% for N=3)Nexus choice — serves multiple continents, survives regional outage with no capacity loss
Nexus v7 · Three-Region Active/Active Topology
🔑

The non-negotiable: regions cannot depend on each other for inference

The moment you have a "global" database or lookup that every request touches, your global system is only as reliable as that one component. Nexus v7 is designed so any single region can serve its traffic using only its own resources. Shared components — model artifact registry, eval harness, quality monitor — are replicated read-only into each region and refreshed asynchronously. If the global artifact registry goes down, no region's serving path is affected; only the ability to roll out a new model version pauses. Inference must be independent per region; control plane can be global.

Model replication — the pull-not-push pattern

A new Nexus model version is ~85 GB of quantized weights. Getting those bytes into every region without a brittle cross-region dependency is solved by a simple pattern: the model artifact is stored in a globally-replicated object store (S3 with cross-region replication, GCS dual-region, or OCI-compatible registry). Each region's deployment tooling pulls the new version when told to. No region has a push dependency on the control plane. Model rollouts are event-driven ("v6.1 is ready in the registry") rather than command-driven ("push v6.1 to region X right now"), which is more resilient to partial network failures.

02 · Autoscaling · The Right Signals

CPU is not the metric. Queue depth is.

The default Kubernetes Horizontal Pod Autoscaler scales on CPU utilization. For LLM serving this signal is approximately meaningless — a well-configured inference pod pins GPU utilization at 70-90% regardless of load, and CPU is mostly used by the Python scheduler loop. The signals that actually track request health are queue depth, active KV pool utilization, and observed tail latency. Teams that autoscale on CPU consistently over-provision during low load and under-provision during surges. Fixing this is a line-item on every LLMOps roadmap.

A working autoscaling policy

nexus_autoscaler.py · queue-depth + KV-pool driven HPAKEDA SCALEDOBJECT · YAML-LIKE
# KEDA (Kubernetes Event-Driven Autoscaling) allows scaling on any Prometheus
# metric. This is the actual production spec pattern for Nexus, not Kubernetes
# vanilla HPA (which only knows CPU/memory).

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: nexus-v6-inference
spec:
  scaleTargetRef:
    name: nexus-v6-deployment
  minReplicaCount: 2                        # never drop below 2 for HA
  maxReplicaCount: 12                       # absolute cost ceiling
  pollingInterval: 15                       # check every 15s
  cooldownPeriod:  300                      # wait 5min before scaling DOWN (GPU re-init cost)

  triggers:
    # Primary: scale up when pending queue grows
    - type: prometheus
      metadata:
        query: "sum(vllm_pending_requests{deployment='nexus-v6'})"
        threshold: "40"                 # > 40 pending → add a replica

    # Primary: scale up when KV pool is near full
    - type: prometheus
      metadata:
        query: "avg(vllm_kv_pool_utilization{deployment='nexus-v6'})"
        threshold: "0.85"               # > 85% KV used → add a replica

    # Tertiary: predictive scale-up on ingress spikes
    - type: prometheus
      metadata:
        query: "rate(nexus_prompt_tokens_total[1m])"
        threshold: "1800000"            # > 1.8M tok/s input rate → scale

    # Safety: scale DOWN slowly on sustained low queue depth
    - type: prometheus
      metadata:
        query: "sum(vllm_pending_requests{deployment='nexus-v6'})"
        threshold: "5"
        activationThreshold: "5"         # trigger scale-down candidate

# Why the cooldown is 5 minutes on scale-down:
#   • New DGX H200 node provisioning: 90-180 seconds
#   • Model weight load from registry: 40-90 seconds  
#   • Warmup / compile of kernels: 30-60 seconds
#   • TOTAL cold-start: 3-5 minutes
# Scaling down too eagerly causes thrashing when a burst arrives and we pay the
# cold-start latency during a load spike. Better to hold the replica an extra
# few minutes.

The GPU cold-start problem

Scaling a stateless web service is easy — new replicas are ready in seconds. Scaling LLM inference is harder because a new inference replica is not ready for traffic until its model weights are loaded in HBM and its CUDA kernels are compiled. For a 38B-active MoE on H200 that's 3-5 minutes of cold-start. Three strategies to manage this:

Headroom buffer

Always keep 25-40% GPU capacity unused. When load spikes, existing replicas absorb it while new ones warm. Simple, expensive; the default in most production systems. Nexus uses 30%.

Predictive scaling

Forecast 10-minute-ahead load from the last 24h × weekly seasonality. Scale up before predicted surges. Works when your traffic has patterns (daily/weekly); useless for unpredictable bursts.

Warm standby pool

Keep a pool of replicas loaded but not receiving traffic. On scale-up, just flip the routing flag. Zero cold-start. Costs the replica's GPU-hours while idle — worth it for latency-critical tiers.

⚠

Why you cannot "scale to zero" LLM services

"Scale to zero" — drop to 0 replicas when no traffic — is a classic serverless pattern. It does not work for LLM serving because even the first request pays 3-5 minutes of cold-start latency. This is unacceptable for any interactive workload. For Nexus we always maintain minReplicaCount: 2 even in quiet regions. Services that do "scale to zero" either use a very small always-on model and bigger models on-demand (a tiered pattern), or they accept cold-start hits for their most latency-tolerant tier only.

03 · Serving Frameworks · KServe vs Ray Serve

The control plane above vLLM.

vLLM, SGLang, and TensorRT-LLM are inference engines — they handle the model forward pass, the KV cache, the batching. They do not handle deployment, rolling upgrades, autoscaling integration, multi-model routing, or canary splits. That is the job of the serving framework. The two production-viable options in 2026 are KServe (Kubernetes-native, CRD-driven) and Ray Serve (Python-native, dynamic). Each has a natural home.

The choice, in one table

ConcernKServeRay Serve
Integration modelKubernetes CRD · declarative YAMLPython decorators · imperative
Rolling upgradeNative — knative revision + traffic splitManual or via Ray cluster op
AutoscalingKnative + KEDA · great queue-depth integrationBuilt-in, reactive
Multi-model / inference graphsInference pipelines (beta)Trivial — any Python composition
Dynamic batching / routing logicFixed by backend (vLLM)Trivial, full Python
Operational maturityUsed by hyperscalers; rock-solidMature; strong at Anyscale customers
Observability defaultsKnative metrics + Istio distributed tracingRay dashboard; add Prometheus manually
Best forSingle-model, latency-critical, K8s-native orgsMulti-model graphs, RAG, complex routing

Why Nexus uses KServe

Nexus's workload is predictable: one model version live at a time, strict SLO, multi-tenant via priority classes (covered in M5 scheduling). The complexity we need to manage is deployment — rolling new model versions safely, with canary and observability. KServe's CRD model fits this perfectly: a new InferenceService revision is declared, Knative splits traffic gradually, metrics feed back into the rollout decision. Ray Serve would work but the Python-imperative model adds complexity we don't need.

Teams running RAG pipelines — where the LLM is one node in a graph including retrieval, reranking, tool calls — should pick Ray Serve. The ability to compose Python functions into a serving graph with automatic fan-out/fan-in and shared state is where Ray shines, and writing that graph in Kubernetes YAML is painful.

04 · Observability · Three Signals, Not One

Latency, throughput, and quality drift.

A production LLM system has to answer three questions continuously: is it fast enough, is it handling the load, and is it still accurate? Traditional SRE stacks solve the first two; the third is the one that bites teams. Model quality can degrade silently — same latency, same throughput, subtly worse outputs — for weeks before anyone notices. The observability stack for LLMs has to be built around the uncomfortable fact that "correctness" is a continuous distribution, not a binary check.

The three signal layers

Infra signals

GPU utilization (compute and memory separately), HBM pressure, PCIe bandwidth, network RTT between prefill and decode pools, NVLink saturation.

Source: DCGM exporter → Prometheus. Alert on sustained < 60% compute util (over-provisioned) or sustained > 92% (about to tip over).

Serving-engine signals

TTFT p50/p95/p99, TPOT p50/p95/p99, queue depth, admission-rejection rate, KV cache hit rate, speculation acceptance rate α, batch size histogram.

Source: vLLM / SGLang Prometheus endpoint. The serving engine itself knows more than the OS ever can.

Quality signals

Output-distribution drift, refusal rate, hallucination rate against ground-truth samples, downstream task F1 on labeled slices, embedding-distance to previous canonical outputs.

Source: async evaluation pipeline sampling ~0.5% of production traffic through a golden-set evaluator or LLM judge.

The latency percentile trap

The single most common observability mistake in LLM serving is tracking only p99 end-to-end latency. The reason is that E2E latency is a composition — prefill + decode + queue + network — and each component has a different distribution and different sensitivity to load. When p99 spikes, "p99 E2E" alone tells you nothing about which layer caused it.

🔑

Break latency down by phase, always

Track TTFT p99 (prefill + queue), TPOT p99 (steady-state decode quality), TPOT p99.9 (decode jitter — captures cohabitation stalls), queue time p99 (admission health), and the E2E ratio (actual vs. ideal). If E2E p99 regresses, exactly one of those four will tell you which layer to look at. Nexus's dashboard has all four on the top row.

Quality drift — the silent failure

The hardest failure mode to catch is model quality degrading while all infrastructure metrics stay green. It happens in predictable ways: a new upstream service starts sending requests with a different prompt format; a tokenizer update changes byte-to-token boundaries subtly; a KV cache bug produces slightly-wrong logits on a fraction of requests; upstream traffic shifts to a tenant with different characteristics.

The only defense is an always-on evaluation pipeline that asynchronously samples production traffic and scores it. For Nexus this is a dedicated fleet of small H100s running the v5 dense teacher as a reference model, plus a small set of golden-label fraud cases that are injected as synthetic traffic every minute. Divergence between production model and reference model triggers PagerDuty before user-facing quality suffers.

quality_drift_alarm.py · production sampling and scoringPYTHON
# Runs as a sidecar service. Samples 0.5% of production traffic, scores it
# against reference, triggers alert on sustained drift. Critical path for
# catching silent regressions.

import asyncio, prometheus_client as prom
from collections import deque

# Sliding window — 30 minute bucket, 2k recent samples.
score_window = deque(maxlen=2000)

drift_gauge  = prom.Gauge("nexus_quality_drift",  "Mean score drop vs v5 teacher")
refusal_rate = prom.Gauge("nexus_refusal_rate",  "Fraction of requests model refused")

async def score_sample(req, prod_output):
    # Shadow call to the v5 teacher (same prompt, ground-truth output on golden set).
    ref_output = await teacher_client.generate(req.prompt)
    # Task-specific scorer: fraud-label F1 for Nexus, BLEU/ROUGE for chat, etc.
    delta = score(prod_output, ref_output, task="fraud")
    score_window.append(delta)

    # Update alert gauges on a rolling window.
    if len(score_window) >= 200:
        mean_drop = sum(score_window) / len(score_window)
        drift_gauge.set(mean_drop)
        # Flag refusals: empty output or output starting with known refusal patterns.
        refusals = sum(1 for s in score_window if s.is_refusal)
        refusal_rate.set(refusals / len(score_window))

# Prometheus alert rule (alert_rules.yml):
#   alert: NexusQualityDrift
#   expr:  nexus_quality_drift > 0.04
#   for:   15m
#   annotations:
#     summary: "Nexus quality dropped > 4% vs reference for 15 minutes"
# This is the single most important alert we have. Infra alerts fire often and
# recover fast. A quality-drift alert fires rarely and means the product is
# silently getting worse for real users.

Cost attribution per tenant

At Nexus's scale, different tenants have wildly different cost profiles: some send 6,000-token prompts producing 50-token responses (prefill-heavy), others send 100-token prompts producing 2,000-token responses (decode-heavy). Billing, capacity planning, and SLO negotiation all require knowing cost per tenant, not just aggregate cost.

The canonical approach is to emit a per-request cost record as a structured log line — tenant ID, prefill tokens, decode tokens, speculation acceptance rate (determines big-model passes used), model version, region, wall time. These aggregate into per-tenant dashboards. The math is routine; what makes it operationally critical is that it must run for every request, never sampled, because tenants will argue with averages.

05 · Shadow Mode, Canary, and A/B Tests

Three different rollout patterns. Pick deliberately.

Deploying a new model version to a production LLM system is fundamentally different from deploying a new microservice. The new model will behave almost correctly — but subtly differently — on essentially every request. Traditional "roll out to 1% of traffic, check error rate, roll forward" logic breaks down because the error rate is zero in both cases. You need rollout patterns designed for the LLM failure mode.

Three Rollout Patterns · When to Use Each
⚠

The ordering that matters

The rollout sequence for a major version change is always shadow → canary → A/B → full, not any one of them in isolation. Shadow catches "the new model produces different outputs that we didn't anticipate"; canary catches "the new model has an infrastructure interaction bug"; A/B catches "the new model is genuinely worse on the business metric." Skipping shadow and going straight to canary is the most common mistake — it optimizes for throughput of model changes, and pays for it in a rare but inevitable silent regression. Nexus's policy is minimum 7 days in shadow for any version that changes more than the KV cache dtype.

The sticky-routing requirement

A/B tests on LLMs have one non-obvious requirement: the same user / tenant must see the same model arm for the duration of the experiment. If tenant T sees v6 on request 1 and v6.1 on request 2, and v6.1 phrases JSON slightly differently, downstream parsing may fail silently, and the A/B variance will be dominated by within-tenant model switches rather than between-tenant differences. The fix is deterministic hashing: arm = hash(tenant_id, experiment_id) % 2, done once at the gateway, carried as a header. This also makes prefix caching work — RadixAttention's hit rate depends on byte-stable prefixes, which depend on a tenant always hitting the same model replica.

06 · Incident Response · The LLM Runbook

When the dashboards go red, what do you do?

Most LLM incidents fall into a small number of repeating patterns. A mature production team has a runbook keyed on the first-order symptom — TTFT spike, TPOT jitter, quality drop, cost anomaly, 5xx rate — with 3-5 known root causes for each and the diagnostic commands to separate them. The goal is mean time to mitigate < 15 minutes, not "figure out what's broken." Mitigate first, then investigate.

The five incidents that account for 80% of pages

SymptomMost likely causeFast-checkMitigation
TTFT p99 tripled Prefix cache hit rate collapsed (usually prompt template changed) Check nexus_radix_hit_rate — is it < 20%? Roll back prompt template; confirm hit rate recovers
TPOT jitter (p99.9 >> p99) Prefill leaking into decode pool (cohabitation stall) Check chunked-prefill chunk size; check KV transfer latency Reduce chunk size; if cohabitated, enable explicit decode pool
Quality drift alarm Upstream prompt format change OR KV quantization bug Compare recent request samples byte-for-byte to a week ago Roll back model version; contact upstream team
5xx spike on one region HBM OOM from unexpected long context OR pod eviction Check cuda_oom_events; check K8s pod restarts Shift traffic to healthy regions; reduce max_model_len on affected region
Cost-per-token up 40% Speculation acceptance rate dropped (bad draft model match) Check nexus_speculation_accept_rate — below 0.7? Roll back draft model; re-evaluate on recent traffic distribution
💡

The "small but telling" signal

The single most predictive early warning at Nexus is a drop in prefix cache hit rate — it captures prompt-template changes (which cause quality issues downstream), tokenizer drift (which causes billing drift), and routing regressions (which cause latency issues) all at once. Alert when hit rate drops below 80% of its 7-day moving average. This one alert has caught more genuine problems than all infra alerts combined.

The rollback discipline

Model rollback has to be one command. Not three. Not "rebuild the image with the old weights and redeploy." The entire LLM inference stack should treat model versions the way Kubernetes treats image tags — pinned, immutable, one label change to revert. In KServe:

rollback.sh · one-command model revertBASH
# Every model version is a signed, immutable artifact in object storage.
# The InferenceService references a tag; rollback is a tag swap.

kubectl patch inferenceservice nexus-fraud \
  --type=merge --namespace=inference-prod \
  -p '{"spec":{"predictor":{"model":{"modelFormat":{"name":"vllm"},
      "storageUri":"gs://nexus-models/v6.0.3-stable"}}}}'

# Effects propagate in ~90 seconds:
#   1. Knative creates a new revision pointing at v6.0.3-stable
#   2. Traffic shifts 100% to the old stable model (blue/green)
#   3. Pods for the broken version terminate after drain
#
# The stable tag is never moved. The broken v6.1 tag is preserved for
# post-mortem. Rollback is always to a known-good immutable version.
07 · Security and Multi-Tenancy

Isolation when weights are shared.

Nexus serves fraud detection to dozens of merchant tenants. A misbehaving tenant cannot be allowed to starve another tenant's requests; a malicious input cannot be allowed to leak one tenant's data through the KV cache; regulated tenants (payment processors, banks) need evidence that their traffic is isolated. The serving engine's efficiency tricks — prefix caching, continuous batching, shared weights — all work against isolation. The production system has to claw back isolation at layers above the engine.

The four isolation boundaries

Compute isolation — fair share

Per-tenant quotas at the admission-control layer (tokens/sec, concurrent requests). Implemented as a weighted-fair-queuing scheduler on the gateway. Tenant A's burst cannot push tenant B's p99 past SLO. Without this, one noisy tenant silently degrades everyone.

Data isolation — no cross-tenant prefix sharing

RadixAttention will happily share a prefix across tenants if the tokens match — which they sometimes do by coincidence. For regulated workloads, we key the trie on (tenant_id, tokens) rather than tokens alone. Cost: a small hit rate reduction. Benefit: provable no-leakage across tenant boundaries.

Request-content isolation — prompt injection defense

Adversarial inputs attempt to extract system prompts or induce instruction override. Defense: structural tagging (XML/JSON delimiters the model is trained on), input filtering at gateway, output filtering for sensitive pattern matches, dedicated "jailbreak detector" small model on sampled traffic.

Audit isolation — regulated-tenant evidence

Some tenants require "our traffic never touched the same physical GPU as other tenants". For those (typically largest merchants), deploy dedicated GPU pools and route by tenant class at the gateway. More expensive, necessary for SOC 2 / PCI-DSS environments.

The PII handling rule

LLM logs are particularly dangerous. A standard request log includes the full prompt and full completion — which for a fraud-detection model contains credit card numbers, email addresses, transaction descriptions, and merchant IDs. Logging this in the clear is a discovery event waiting to happen.

⚠

The three-tier logging policy

Production-grade LLM logging separates into (1) metadata — latency, token counts, tenant ID, model version; always logged, 90-day retention. (2) hashed content — prompt hash and output hash for debugging correlation; always logged, 30-day retention. (3) full content — only logged when explicitly sampled for quality eval, encrypted at rest with per-tenant keys, 7-day retention maximum, with right-to-deletion honored on request. The instinct to "log everything while you're getting started" is a GDPR violation generator. Build the tiered pipeline on day one, not day 300.

08 · Core Operational Math

Four formulas for load shedding, failover, retries, and timeouts.

Production LLM serving is fundamentally a queueing-theory problem. Every incident described in this module — retry storm, failover lag, load cascade, timeout waterfall — has a closed-form characterization that tells you exactly when it'll happen and where to configure the guardrail. Four formulas, four worked examples, four verdicts.

Formula 1 · Load shedding threshold (Kingman's wait)

For an M/G/1 queue under arrival rate λ and service capacity μ, Kingman's approximation gives expected wait time as a function of utilization ρ = λ/μ. The relationship is nonlinear: doubling ρ from 0.4 to 0.8 quadruples wait time.

Expected wait under M/G/1 (Kingman) E[W] ≈ (ρ / (1 − ρ)) × (CV_a² + CV_s²) / 2 × E[S]

Where:
  ρ    = arrival rate / capacity = utilization
  CV_a = coefficient of variation of arrivals (Poisson: CV_a = 1)
  CV_s = coefficient of variation of service times (LLM: ~0.5)
  E[S] = mean service time

Shedding threshold ρ*: largest ρ where E[W] + E[S] ≤ SLO.
Start shedding above ρ*, before latency goes critical.

Worked example · Nexus v7 at capacity 1100 req/s, 250ms TTFT SLO

Mean service = 60ms, CV_a=1.0, CV_s=0.5. At ρ=0.9: E[W] = (0.9/0.1) × (1.25/2) × 60 = 337 ms. TTFT = 337 + 60 = 397ms. Misses 250ms SLO badly. Binary-searching for ρ where TTFT ≤ 250ms gives ρ* ≈ 0.835, equivalent to 919 req/s arrival. Above this, shedding must kick in. The load_shedding_calculator.py artifact solves this exactly and outputs the gateway rate-limit value to configure.

Formula 2 · Retry amplification

When requests fail, clients retry. Each retry adds load to an already-struggling system. At N retries per failed request and failure rate f, traffic amplification is 1 + N·f. If f is high and N is naive, amplification pushes the system over the capacity cliff.

Traffic amplification during a failure window amplification = (original_requests + retry_requests) / original_requests
        = 1 + (N × f_effective)

Where f_effective = failure_rate(ρ), which itself increases with ρ.
This creates a positive feedback loop: retries → more ρ → more failures → more retries.

Retry budget mitigation: cap total retries to B% of successful traffic.
    amplification_capped ≤ 1 + B
    (independent of failure rate, which is the whole point)

Worked example · 10-second 503 blip, 1000 req/s arrival

Without budget, max_retries=3, failure rate 50%: amplification = 1 + (3 × 0.5) = 2.5× during failure. But amplified load pushes ρ above capacity, causing more failures, more retries — retry_storm_simulator.py shows the system never recovers, 107+ seconds of elevated errors for a 10-second blip. With retry_budget=10%, amplification caps at 1.1× regardless of failure rate; system recovers within seconds of upstream recovery. This is why every production proxy MUST configure retry_budget, not just max_retries.

Formula 3 · Failover budget

During a regional outage, the surviving regions must absorb the failed region's traffic without overflowing themselves. Per-region capacity must be sized for N-1 operation, not N.

Per-region capacity for N-1 survivability With N regions carrying total traffic T at baseline:
  baseline_per_region = T / N
  N-1_per_region = T / (N-1) — the survivor carries more when one fails

Required_capacity_per_region ≥ T / (N-1) × headroom_factor (1.15)

At N=3, T=1000 req/s: each region needs ≥ 575 req/s capacity
At N=2, T=1000 req/s: each region needs ≥ 1150 req/s (2×)
At N=5, T=1000 req/s: each region needs ≥ 287 req/s (much less)

Rule: more regions = cheaper to over-provision.

Worked example · Nexus v7 · 3-region active/active

Total traffic = 10,000 req/s across 3 regions. N-1 math: each region needs 10000/2 × 1.15 = 5750 req/s capacity. That's 57.5% utilization per region at baseline — not 33.3% as naive distribution would suggest. When us-east fails, each survivor absorbs 5000 req/s, hitting 87% utilization. With 30s health-check detection window, regional_failover_simulator.py shows 0.2% traffic drop rate — invisible to users. Shorter windows reduce drop further but risk false-positive region ejection during transient blips.

Formula 4 · Composed timeout budgets

An end-to-end LLM request touches multiple layers: gateway → auth → scheduler → engine → response streaming. Each layer has its own timeout. If the sum of per-layer timeouts exceeds the end-to-end budget, the EE timeout wins but the layers waste work chasing requests that are already dead.

Timeout budget decomposition Total budget B splits across layers:
  B = T_gateway + T_auth + T_scheduler + T_engine + T_streaming_margin

Constraint: Σ per-layer timeouts < B (leave 10% margin)
  For B = 500ms: T_gw = 50, T_auth = 30, T_sched = 20, T_engine = 350, margin = 50

Retry with timeouts composes:
  worst_case = num_retries × per_try_timeout + final_retry_timeout
  For Envoy retry_policy (2 retries, 250ms each): worst_case = 750ms

Worked example · Nexus v7 interactive-class request

End-to-end SLO = 500ms. Envoy timeout: 500ms and retry_policy num_retries: 2, per_try_timeout: 250ms. Math check: 2 × 250 = 500ms, equal to timeout. The third retry can never land within budget even though configured — which is correct, acts as a fail-safe for the edge case where earlier tries complete fast. If you set num_retries=3 with per_try_timeout=250ms, the third retry is pure waste — it'll timeout at the outer 500ms budget before completing its 250ms. This is the composed-timeout trap and it's easy to miss in code review. Always verify: (num_retries + 1) × per_try_timeout ≤ outer_timeout.

🔑

The overall framing

Production resilience is a queueing-theory game. Formula 1 tells you when to shed (before latency breaks). Formula 2 tells you why naive retry amplifies failures. Formula 3 tells you the capacity floor for multi-region. Formula 4 tells you why stacked timeouts compose. Each failure mode in section 11 is one of these formulas going wrong in a specific production context. When deploying, run the calculators to set values. When debugging an incident, the formulas tell you which number is off.

10 · Architecture Failure Modes

Eight patterns every production LLM system must handle.

Every production LLM outage in the last two years maps to one of these eight failure modes. Each one has a precise trigger, a well-understood mitigation, and a specific place in the config to wire it in. Without addressing these, you're not running a service — you're running a time bomb calibrated to your traffic growth curve.

Pattern 01 · Load Shedding Priority: P0 (must-have)
Drop requests at the gateway BEFORE the system is saturated.
TRIGGERS
Arrival rate exceeds capacity for any reason — traffic spike, autoscaler lag, downstream latency regression, bad deploy. Without shedding, every request queues; queues grow; latency goes nonlinear; SLO breaks; clients retry; cascade begins.
MITIGATION
Gateway-level local rate limiter capped at ρ* × capacity (Kingman threshold from section 09). Shed excess traffic with HTTP 503 + clear x-ratelimit-reason header. Shed in priority order: batch class first, interactive last. Never shed; scale up instead when possible.
CONFIG
envoy.filters.http.local_ratelimit with token_bucket.max_tokens = ρ* × capacity. Filter sits BEFORE the routing filter so shedding happens before auth+scheduler cost. See envoy_production_filters.yaml filter 1.
Pattern 02 · Retry Storms (and retry budgets) Priority: P0 (must-have)
Cap retry traffic as a percentage of success, not a count per request.
TRIGGERS
Any upstream blip — a single slow pod, a brief network issue, a deploy-time transient. Clients retry; retries fail under load; more retries; system spirals. A 10-second blip becomes a 2-minute outage.
MITIGATION
Retry budget at the proxy: cap total retries at 10% of successful requests in the recent window. This decouples retry load from failure rate. Add jittered exponential backoff so retries don't synchronize. retry_storm_simulator.py shows naive recovers in 107s, budgeted recovers in <1s.
CONFIG
Envoy retry_policy.retry_budget.budget_percent: 10 + retry_back_off.base_interval: 50ms, max_interval: 500ms. Istio equivalent in istio_traffic_policy.yaml. Free config, massive prevention value.
Pattern 03 · Degraded Mode (graceful fallback) Priority: P1 (nice-to-have for critical services)
When inference fails, return a useful-but-simpler response rather than 5xx.
TRIGGERS
Inference pool is unavailable, overloaded, or too slow. Without degraded mode, every user request fails. With degraded mode, critical responses still arrive — just less personalized or lower quality.
MITIGATION
Fall back to a smaller/cheaper model when primary is unavailable. For Nexus: fall back to a distilled 1B model from a cached response bank. For some use cases: return a templated response. Quality degrades; availability holds. Surface the degradation to the client via x-served-by: fallback header so downstream systems can handle it.
CONFIG
Envoy route.request_mirror_policies to shadow + primary cluster with failover policy. Istio VirtualService with mirror for shadow traffic. Custom logic in the gateway lambda to route on cluster health.
Pattern 04 · Regional Failover Priority: P0 (must-have for multi-region)
One region fails; surviving regions absorb traffic without user-visible impact.
TRIGGERS
Cloud-provider zonal or regional outage; bad deploy that takes down a single region; network partition. Without failover, affected users see 5xx; with failover, they see a few milliseconds of added latency and no errors.
MITIGATION
Health-check-driven routing at the load balancer. 30-second detection window (not 10 — too trigger-happy; not 120 — too slow). N-1 capacity sizing per region (see formula 3 in section 09). Locality-aware routing prefers same-AZ, falls back to sibling AZ, then cross-region.
CONFIG
Istio DestinationRule.trafficPolicy.loadBalancer.localityLbSetting. Global load balancer (Traffic Director, AWS Global Accelerator, Cloudflare Load Balancer) with HTTP health check on /health. regional_failover_simulator.py shows 30s window gives 0.2% drop rate at 3-region topology.
Pattern 05 · Dependency Timeouts (hedged requests) Priority: P1 (meaningful win on tail latency)
Bound the tail by issuing a duplicate request after a time window.
TRIGGERS
One pod in the decode pool is slow — not failing, just slow. Normal request lands on it and blocks. Without hedging, this request contributes to tail latency. With hedging, a sibling request starts after (say) p95 latency; whichever finishes first wins.
MITIGATION
Envoy hedge_on_per_try_timeout: at per_try_timeout, start a second request rather than just failing. For LLM where each request is expensive, tune carefully — aggressive hedging doubles compute cost. Typical setting: hedge at p99 (say 400ms) so only the long tail is duplicated.
CONFIG
Envoy retry_policy.retriable_status_codes + hedge_on_per_try_timeout: true with careful tuning. Not enabled by default in the shipping Envoy config; turn on after A/B testing cost impact.
Pattern 06 · Backpressure (queue depth admission) Priority: P0 (must-have)
Reject new requests when in-flight queue exceeds a bound.
TRIGGERS
Downstream pod slows down for any reason — GPU contention, kernel issue, KV pressure. Without backpressure, the gateway keeps pushing requests onto an overloaded pod, exhausting connections and creating head-of-line blocking. With backpressure, the gateway stops accepting new requests for that pod when queue depth exceeds the bound.
MITIGATION
Envoy circuit breaker max_pending_requests bounds gateway-side queue. Scheduler-level max_num_seqs bounds engine-side concurrency. When both limits are set correctly, a struggling pod gets ejected (outlier detection) rather than dragging the whole cluster down.
CONFIG
circuit_breakers.thresholds.max_pending_requests: 100, max_requests: 500, max_connections: 1000. See envoy_production_filters.yaml circuit_breakers section.
Pattern 07 · Cost Guardrails Priority: P1 (mandatory for any multi-tenant)
Per-tenant spend caps prevent runaway cost from a single tenant or bug.
TRIGGERS
Tenant accidentally sends an infinite loop of requests. Tenant is compromised and attacker tries to rack up cost. Bug in tenant code produces 100× normal volume. Bug on our side (the speculation drift of M8 war story #6) causes cost to drift up globally.
MITIGATION
Per-tenant rate limit (requests/sec) at the gateway. Per-tenant daily token budget, enforced in the billing pipeline. Alert on tenant_cost_per_request > 2× 14-day baseline. Kill switch available to the on-call: ability to downgrade a specific tenant to batch class or throttle to zero.
CONFIG
Envoy rate_limit filter pointing at an external RLS (rate limit service) for per-tenant global limits. prometheus_production_alerts.yaml group 4 (cost) + group 6 (security) has 4+4 = 8 cost-related alerts.
Pattern 08 · Chaos Testing (the discipline) Priority: P1 (quarterly rehearsal)
Inject failures on purpose in staging; fix what breaks; rehearse every 90 days.
TRIGGERS
You don't know your production config is resilient until you try to break it. Without chaos rehearsals, the first real incident is also your first validation of the config — a bad time to find gaps.
MITIGATION
Quarterly rehearsal of 3 scenarios against staging: latency spike (SLO degradation < 5pp), error burst (recovery within 15s), connection drops (no persistent pool damage). Any scenario that breaks SLO for > 60s goes on the fix-before-prod list.
CONFIG
chaos_test_harness.py --compare-scenarios run against the staging endpoint. Istio fault injection enables server-side error bursts (vs client-side). Results logged to a runbook archive for before/after comparison across quarters.
💡

The priority framework

P0 patterns (load shedding, retry storms, regional failover, backpressure) are mandatory before any production deploy. These are free config that prevents common incident classes. P1 patterns (degraded mode, hedged requests, cost guardrails, chaos rehearsals) are strongly recommended but cost real engineering effort — prioritize based on your SLO tier and tenant mix. The operator_checklist.md in the artifact pack has all 16 verification items to check before a production change.

11 · Debugging Drills

Five production incidents. Five diagnostic paths.

Five drills covering the most common production failures: retry storm after upstream blip, slow failover exposing users to errors, autoscaler not scaling, cascading failure from one bad pod, cost drift that alarms miss. Each drill names the symptom, the wrong hypothesis, the correct diagnosis, and the fix.

Drill · 01
10-second upstream 503 blip; error rate stayed elevated for 2 minutes
SYMPTOM
At 14:03:05, upstream dependency returned 503 for ~10 seconds. Our service's error rate jumped to 30% immediately. Upstream recovered at 14:03:15. Our error rate stayed elevated until 14:05:00 — almost 2 minutes after upstream was healthy again.
WRONG HYPOTHESIS
"Something else must have broken — the upstream blip was too brief to cause 2 minutes of elevated errors." Opens the wrong investigation, wastes an hour on pods and networking.
CORRECT DIAGNOSIS
Classic retry storm. During the blip, clients retried failed requests (num_retries=3). Retries piled onto the already-struggling system. Even after upstream recovered, retry load from the blip period was still in flight and exceeded capacity. System oscillated until retry pressure faded. Verify via Envoy metrics: rate(envoy_cluster_upstream_rq_retry[1m]) should show a multi-minute plateau after the blip.
FIX
Configure retry_budget.budget_percent: 10 in Envoy retry_policy. Add retry_back_off.base_interval: 50ms, max_interval: 500ms for jittered exponential backoff. Run retry_storm_simulator.py --compare-mitigations to confirm the fix handles your traffic profile. The fix is free config; the incident cost hours of on-call time + reputational damage.
Drill · 02
Region us-east-1 went down; failover took 2 minutes; users saw errors
SYMPTOM
14:22 · cloud-provider status page reports us-east-1 networking degradation. Our internal monitoring shows us-east-1 pods returning 5xx immediately. Other regions healthy. Failover completed at 14:24 (2 minutes later). In those 2 minutes, ~30% of global traffic (the us-east-1 share) saw errors.
WRONG HYPOTHESIS
"2 minutes is normal for cross-cloud failover — that's the DNS TTL." Not true. DNS TTL is a separate problem; failover can and should be <30 seconds with health-check-driven routing.
CORRECT DIAGNOSIS
Check the global load balancer config: what's the health-check detection window? If it's 120 seconds (a common default), that's your failover floor. The fix is detection window tuning — 30 seconds is the production default. Also check: when us-east-1 was marked unhealthy, did the other regions have capacity to absorb the load? Run regional_failover_simulator.py with your actual traffic and region count.
FIX
Set global load balancer health check to 30s interval, 3 consecutive failures, 10s timeout. Verify per-region capacity is sized for N-1 operation with 15% headroom (formula 3 in section 09). Rehearse quarterly via a controlled region-drain drill. Drop rate during outage should be <0.5% after these changes.
Drill · 03
Queue depth building but autoscaler sitting at minReplicas
SYMPTOM
15:00 · request volume jumped 2× from a marketing event. vllm:num_requests_waiting is at 40 per pod (should be <5). TTFT p99 climbing. Autoscaler log says replica count unchanged at 2 (minReplicas). CPU at 10%, GPU at 95%.
WRONG HYPOTHESIS
"GPU is at 95% — we're already saturated, can't scale more." GPU util is a misleading signal for LLM workloads; it reads high during decode even when the system is waiting.
CORRECT DIAGNOSIS
Check the autoscaler metric: kubectl get inferenceservice -o yaml | grep autoscaling.knative.dev/metric. If it's cpu, that's the bug — CPU is near-idle on GPU workloads, so CPU-based autoscaling never triggers. Must be concurrency or vllm:num_requests_waiting. Also check: is maxReplicas set too low? If maxReplicas=2 and we're already at minReplicas=2, autoscaling can't help.
FIX
Change autoscaler metric to concurrency with target: 48 (aligns with max_num_seqs). Raise maxReplicas to match expected peak traffic. Add the AutoscalerNotScaling alert from prometheus_production_alerts.yaml to catch this pattern in the future.
Drill · 04
One pod slow; all pods appear slow; cluster-wide SLO breach
SYMPTOM
16:45 · one specific pod (let's call it nexus-llm-7k2p) starts taking 4 seconds per request instead of 60ms. Within 5 minutes, ALL pods show elevated latency. TTFT p99 across the cluster spikes. No code change, no deploy, no config push.
WRONG HYPOTHESIS
"Must be a cluster-wide issue — network or Kubernetes problem." Directs investigation to the wrong layer. The actual issue is confined to one pod but visible on all of them.
CORRECT DIAGNOSIS
The single bad pod was holding connections that the gateway kept trying to route to (LEAST_REQUEST load balancer favors pods with fewer active requests — paradoxically sends MORE work to a slow pod because it "has fewer active requests completing"). The bad pod's slow responses created head-of-line blocking in the connection pool, which appeared as cluster-wide slowness. Check envoy_cluster_outlier_detection_ejections_active: if it's 0, outlier detection isn't working. Check individual pod latency percentiles — usually one is dramatically worse than the others.
FIX
Immediate: kubectl delete pod nexus-llm-7k2p; KServe recreates it. Then fix outlier detection: verify consecutive_5xx: 5, interval: 10s, base_ejection_time: 30s are configured. See envoy_production_filters.yaml outlier_detection section. Outlier detection should have auto-ejected the bad pod before it took the cluster down.
Drill · 05
Month-end cost report shows +20% spend; request volume up only 5%
SYMPTOM
Finance team's monthly report shows cost +20% vs last month. Engineering checks: request volume grew 5%, average output length unchanged, no new tenants onboarded. The cost delta doesn't have an obvious source. All latency, error-rate, and SLO metrics are green.
WRONG HYPOTHESIS
"Finance must be miscounting — push back and wait for reconciliation." By the time reconciliation completes, another month of +20% cost has happened.
CORRECT DIAGNOSIS
Cost-per-request has drifted up silently. Common causes: (a) speculation acceptance-rate drift (M5 drill 01) — draft model is stale; (b) prefix-cache hit rate drop (M4 drill 01) — prompt template has a timestamp; (c) traffic shifted to longer-context requests; (d) a deployment changed model quantization without noticing. Run draft_model_acceptance_tracker.py on the last 90 days of decoding logs. Check vllm:prefix_cache_hit_rate trend. Check mean context length vs 30 days ago.
FIX
Wire the cost alerts from prometheus_production_alerts.yaml group 4. The CostPerMillionTokensDrift alert triggers when cost-per-M-tokens is up >15% vs 7 days ago — catches this pattern in days, not a month. Fix the root cause identified by the tracker (retrain draft model, fix prompt template, etc.). This is the M8 war story #6 mechanism — expensive to miss, cheap to monitor.
12 · Failure Reproduction

Watch a 10-second blip turn into a 2-minute outage.

The artifact pack includes break_retry_storm.py — a no-GPU script that reproduces the retry-storm cascade end-to-end. It shows what breaks (naive retry), what fixes it (retry budget + jittered backoff), the exact 3-line config diff to apply in Envoy, and the Prometheus alert rule to catch it BEFORE the next outage.

What the script does

Delegates to retry_storm_simulator.py --compare-mitigations and adds narrative framing. The simulation runs 4 scenarios:

Naive retry (no budget, no jitter)

10-second upstream failure produces 107+ seconds of elevated errors. System never catches up because retry load keeps exceeding capacity. Recovery tied to retry-pressure decay, not upstream health.

Budget + jitter (production baseline)

Same 10-second failure. Retry budget caps load at 1.1× baseline. Jitter prevents synchronized retry waves. System recovers within the failure window itself.

Mapping to real symptoms

In production, retry storms show three specific fingerprints:

1. Error rate outlasts upstream

Error rate stays elevated well past upstream recovery time. Duration from upstream recovery to your recovery = retry-pressure decay time.

2. Retry metric plateau

Envoy upstream_rq_retry rate stays high for minutes after the trigger. The plateau duration = retry-pressure timeline.

3. Capacity oscillation

Load oscillates: serves → overloads → drops retries → recovers → retries flood → overloads. Classic feedback loop fingerprint.

💡

The design principle

Retry storms are caused by naive retry logic, not by infrastructure. The fix is free config that sits in one file (Envoy or Istio). The cost of NOT fixing it is measured in outage postmortems. The action: run break_retry_storm.py once, apply the diff, verify via chaos_test_harness.py --scenario error_burst, move on. This is the single highest-ROI change in production LLM serving.

13 · Operator Checklist

Sixteen items before you ship production changes.

Pre-deploy verification · 16 items
  • Load shedding threshold configured at gateway. Rate limit at ρ* × capacity from load_shedding_calculator.py.
  • Retry budget enabled (not just max_retries). Budget 10% of successful traffic. Verified via chaos_test_harness.py --scenario error_burst.
  • Jittered exponential backoff configured. base_interval 50ms, max_interval 500ms.
  • Circuit breakers set on every upstream. max_pending_requests, max_connections, max_requests all bounded.
  • Outlier detection configured. consecutive_5xx=5, max_ejection_percent=50.
  • Autoscaler uses queue depth, not CPU or GPU utilization. Verify: autoscaling.knative.dev/metric: concurrency.
  • Scale-down delay ≥ 5 minutes. vLLM takes 45-90s to warm up; avoid thrashing.
  • PodDisruptionBudget set with minAvailable ≥ 2. Prevents mass eviction.
  • Multi-region topology with N-1 headroom. Per-region capacity = T/(N-1) × 1.15.
  • Health check detection window = 30s. Fast enough for real outages, slow enough to tolerate transients.
  • SLO alerts configured. Fast-burn + slow-burn + TTFT p99 + error rate alerts all fire-capable.
  • Canary rollout with automated rollback. 10% traffic, rollback on TTFT/error-rate/α regression.
  • Chaos testing rehearsed quarterly. 3 scenarios against staging; any SLO breach >60s is a blocker.
  • Cost alerts wired to on-call. Drift alarm at 15% vs 7-day baseline. Catches silent regressions.
  • Observability 3 signals verified. Metrics + traces + structured logs, all correlated by request ID.
  • Incident response runbook up to date. Each alert has a runbook URL; quarterly review.

Checkpoint · Knowledge Quiz

Nine questions to lock it in.

Module 7 · Self-check

Click answers to reveal explanations.

Q1 · Autoscaling signal
Why is GPU utilization a poor autoscaling signal for LLM serving compared to queue depth?
GPU utilization is always 100% in decode.
GPU util lags the user-perceived problem — by the time it rises, queue has already been growing for tens of seconds. Queue depth is the leading indicator: it reflects arrival rate vs service rate directly, seconds before GPU pressure shows up.
GPU utilization cannot be measured reliably on H100.
Queue depth is easier to expose as a metric.
The rule: autoscale on the signal closest to the user. Queue depth (or queue time) is closest; GPU util is a second-order effect. Scaling on GPU util is reactive; scaling on queue is proactive.
Q2 · Rollout ordering
A team wants to deploy a new model version that swaps the quantization scheme from AWQ to GPTQ. What's the minimum rollout sequence?
Canary only — AWQ and GPTQ produce similar outputs.
A/B test directly — compare F1 scores.
Shadow for 7+ days, then canary staged 1% → 5% → 25% → 100%, ideally with an A/B parallel on the largest tenants. Quantization changes produce subtle distribution shifts that shadow mode surfaces before any user sees them.
Full rollout — it's just a quantization change.
Quantization changes are exactly the case where shadow mode earns its keep: outputs differ in ways that eval benchmarks don't always catch but production traffic reveals. Skipping shadow for "just a quantization change" is how teams ship silent regressions.
Q3 · Quality drift detection
A team monitors E2E latency, throughput, and 5xx rate. Their model started producing subtly worse outputs two weeks ago, no one noticed. What's missing?
Better p99 tracking.
Always-on quality sampling — a pipeline that scores a small fraction of production outputs against a reference model or ground-truth set, with an alert when scores drift. Infra metrics can't catch quality regressions because quality degradation doesn't affect latency or error rate.
More GPUs.
Better rate limiting.
Quality is an explicit third signal alongside latency and throughput. Without a sampling evaluator, quality drift is invisible until users complain — which for B2B LLMs often means a contract review, not a support ticket.
Q4 · Sticky routing
During an A/B test, tenant T's requests sometimes hit arm A and sometimes arm B. Why is this worse than a deterministic 50/50 split?
It makes the test run longer.
Within-tenant variance from model switching dominates between-tenant differences, wrecking statistical power. It also breaks RadixAttention prefix cache hit rate (different arm = different byte prefix). Fix: hash(tenant_id, experiment_id) → deterministic arm.
Costs double.
GPUs can't handle the switching.
Sticky routing is a prerequisite for meaningful A/B statistics on LLMs and for serving-engine efficiency. It's a one-line gateway change that most teams forget on their first experiment.
Q5 · Prefix cache hit rate as an alert
A Prometheus alert fires: "nexus_radix_hit_rate dropped from 78% to 34%". What are the most likely root causes?
GPU failure.
Prompt template change injecting a timestamp into the prefix, tokenizer update shifting byte boundaries, or a new routing rule spraying traffic across replicas that don't share a trie. All three are "things changed upstream of the serving engine."
KV cache corruption.
Network partition.
Hit rate is sensitive to byte-stability of the prefix. A 40-point drop almost always means someone upstream broke the byte-stability assumption without realizing it. First check: diff a recent prompt against one from a week ago.
Q6 · Incident triage
Dashboards show TPOT p99 = 5ms (green), TPOT p99.9 = 48ms (amber). What does the gap between p99 and p99.9 tell you?
The SLO is too tight.
Decode is stalling on rare events — usually prefill intrusions in a cohabitated system, or KV transfer hiccups if disaggregated. The fact that it's only p99.9 means it happens to ~0.1% of requests; the fact that it's 10× the median says it's a genuine stall, not just tail latency.
The network is saturated.
Nothing — 48ms is still fine.
p99 vs p99.9 ratios are how you detect intermittent stalls that average metrics hide. A 5:1 ratio means "mostly fine, occasionally terrible" — which is exactly what interleaved-phase stalls look like. Adding chunked prefill or disaggregating usually fixes it.
Q7 · Multi-tenancy
Why might you disable cross-tenant prefix sharing in RadixAttention for a regulated-workload tenant?
Cross-tenant sharing slows the kernel.
Identical-byte prefixes from different tenants can land on the same trie node — creating a theoretical (and auditable) channel for one tenant's content to influence another's cache state. For SOC 2 / PCI-DSS environments, keying the trie on (tenant_id, tokens) is worth the small hit rate cost.
Prefixes can collide and corrupt outputs.
It consumes too much memory.
The actual information-leakage risk from shared prefix trees is minimal for well-behaved tenants, but "minimal" doesn't satisfy auditors. Per-tenant trie keying is the clean architectural answer.
Q8 · Logging discipline
A team logs every prompt and completion in full to make debugging easy. A year later, a tenant files a GDPR right-to-deletion request. What's the problem?
Deletion is slow.
The "full content everywhere" log store has no structure for per-tenant, per-user deletion — the data is interleaved across backup archives, analytics warehouses, and cold storage. Tiered logging (metadata/hashes/full-with-retention-tiers) is essential from day one, not retrofitted.
Logs are lost.
The model has memorized the prompts.
GDPR/CCPA compliance and LLM "log everything" culture are in direct tension. The resolution is a three-tier policy: metadata forever, hashes for medium retention, full content only when explicitly sampled, encrypted with short retention. Any other approach becomes an expensive rewrite.
Q9 · Rollback discipline
Why should model rollback be one command, not a rebuild-and-redeploy?
Developers don't have time to rebuild.
Rollback speed is the difference between a 10-minute incident and a 2-hour incident. During an incident, every additional step is a chance for a second mistake. Model versions should be immutable artifacts with stable tags; rollback is always a label change on a KServe InferenceService, never a rebuild.
Rebuilds cost money.
Git is too slow.
The incident MTTM target of < 15 min is only achievable with single-command rollback. The architectural cost is discipline around immutable model artifacts and stable tags. It pays off the first time you need it.
