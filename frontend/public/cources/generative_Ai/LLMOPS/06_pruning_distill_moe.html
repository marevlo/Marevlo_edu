Pruning. Distilling.
Sparsifying.

Quantization (Module 2) shrunk every weight to 4 bits. The next frontier is more fundamental: change what weights exist, or change which weights fire per token. Three techniques cover the space — pruning (delete the weights that barely contribute), distillation (train a smaller model to match the bigger one's outputs), and mixture-of-experts (keep all the weights but only activate a fraction per token). Each is viable on its own; the real wins come from combining them. Nexus v6 ends this module as a 38B-active-parameter MoE matching the 70B dense model's fraud-detection F1 score, running on a single H200 node per region.

01 · Pruning · The Design Space

Which weights don't matter?

Pruning is the oldest compression idea in neural networks (LeCun et al., "Optimal Brain Damage", 1989). The premise: trained networks are massively overparameterized, so many weights contribute so little that setting them to zero barely affects the output. The practical question is which weights to zero out, and whether the zeros produce real speedups on modern hardware. For LLMs in 2026, the answers are: use the right method (SparseGPT or Wanda), pick a hardware-friendly pattern (2:4 structured), and target 50% sparsity as the ceiling for dense quality retention.

Three axes that define every pruning method

The Pruning Design Space · Three Orthogonal Axes
Magnitude · |W| — simple, fast, often too naive for LLMs Wanda score · |W| × ||X|| — Wanda method; activation-weighted magnitude OBS / Hessian-based · weight impact on reconstruction loss — used by SparseGPT, most accurate AXIS 3 · SCHEDULE — when in training/inference pipeline we prune One-shot PTQ-style · apply after training · SparseGPT and Wanda both do this · no retraining Iterative · prune, fine-tune to recover, prune again · higher quality but expensive Sparse-from-scratch · train with sparsity constraint from the beginning · research frontier
🔑

The rule that matters most

Unstructured sparsity produces zeros at arbitrary positions — maximally accurate, but modern GPUs do not skip zeros in dense matmul kernels. A 50% unstructured sparse matrix multiplies at the same speed as a 0% sparse matrix on H100. The zeros save memory if you use a sparse-storage format (CSR, etc.), but decode is still memory-bound on the same byte count. For a real inference speedup, you need structured sparsity that hardware can exploit — in 2026 that means NVIDIA 2:4, and nothing else on commodity silicon. Head pruning and MoE routing are also "structured sparsity" in a broad sense, and they speed things up for the same reason: the sparsity pattern aligns with the kernel's natural vectorization.

02 · SparseGPT · Hessian-Aware One-Shot Pruning

Prune and compensate, layer by layer.

SparseGPT (Frantar & Alistarh, 2023) applies the GPTQ machinery (Module 2) to pruning. The same Hessian-aware error-compensation strategy that makes GPTQ a clean 4-bit quantizer makes SparseGPT a clean 50%-sparse pruner. The recipe: process one layer at a time, go column by column, for each column select the K weights with lowest "impact scores" and zero them, then adjust the remaining unquantized columns to compensate. The math is the natural extension of the GPTQ algorithm we already walked through.

The impact score — the key extension from GPTQ

SparseGPT impact score for weight W[i,j] impact(i, j) = W[i, j]² / H⁻¹[j, j]

where H = Xᵀ X is the input Hessian (same as GPTQ)

The intuition: H⁻¹[j, j] is a scalar proportional to "how much adjustment the remaining weights can absorb if we zero out column j". Small H⁻¹[j, j] means the damage from zeroing is easy to compensate; large means hard. Combined with W[i,j]² (straight importance), the score ranks weights by how painful they are to remove given we are allowed to fix up the rest. We pick the lowest-scored weights in each column and zero them.

sparsegpt_layer.py · core pruning loopPYTHON
# SparseGPT for one linear layer, 50% unstructured sparsity.
# Reuses the GPTQ Hessian + Cholesky machinery from Module 2.

import torch

def sparsegpt_prune(
    W: torch.Tensor,          # [out, in]
    X: torch.Tensor,          # calibration activations [N, in]
    sparsity: float = 0.5,
    block_size: int = 128,
):
    out, in_ = W.shape
    # Hessian + damping (same recipe as GPTQ).
    H = 2.0 * X.T @ X
    H += 0.01 * torch.diag(torch.ones(in_, device=H.device) * H.diag().mean())
    Hinv = torch.linalg.cholesky(torch.cholesky_inverse(torch.linalg.cholesky(H)), upper=True)

    mask  = torch.zeros_like(W, dtype=torch.bool)   # True = keep, False = prune
    Err   = torch.zeros_like(W)

    for g_start in range(0, in_, block_size):
        g_end = min(g_start + block_size, in_)

        for j in range(g_start, g_end):
            w = W[:, j]
            d = Hinv[j, j]

            # Impact score within this column across ALL output rows.
            score = w.pow(2) / d.pow(2).clamp(min=1e-9)

            # Keep the top (1 - sparsity) * out by score; zero the rest.
            k_keep = int(out * (1 - sparsity))
            threshold = score.topk(k_keep).values[-1]
            keep = score >= threshold
            mask[:, j] = keep

            # err on pruned entries — propagate compensation to remaining columns in block.
            err = torch.where(keep, torch.zeros_like(w), w) / d
            Err[:, j] = err
            W[:, j+1:g_end] -= err.unsqueeze(1) * Hinv[j, j+1:g_end].unsqueeze(0)

            # Apply mask — zero out pruned entries in the current column.
            W[:, j] = torch.where(keep, w, torch.zeros_like(w))

        # Propagate accumulated block error to columns after this block.
        W[:, g_end:] -= Err[:, g_start:g_end] @ Hinv[g_start:g_end, g_end:]

    return W, mask

SparseGPT + Wanda — two slightly different philosophies

Wanda (Sun et al., 2023) arrived a few months after SparseGPT and produced comparable accuracy with a dramatically simpler algorithm — no Hessian, no compensation, no block-wise error propagation. Wanda's single insight: the importance score should be |W[i, j]| × ||X[:, j]||₂ — magnitude times activation norm. This is AWQ's salient-weight story (Module 2) repurposed for pruning. The method: compute that score, keep the top K per row, zero the rest. That's the whole algorithm.

Why Wanda is almost always the right first call

Wanda gives 80-90% of SparseGPT's accuracy at 1/3 the implementation cost and 1/3 the runtime. For 2:4 structured sparsity — the only pattern with hardware support — the gap shrinks further. Unless you are publishing a paper, the pragmatic pruning workflow in 2026 is: apply Wanda at 2:4, check the accuracy regression on your eval, and if it's above your budget, then escalate to SparseGPT or a recovery fine-tune. For Nexus, Wanda at 2:4 alone gave us 2.1× throughput on H100 with 2.4 F1 points regression — good enough to combine with distillation for full recovery.

03 · 2:4 Structured Sparsity

The sparsity pattern that actually runs fast.

NVIDIA Ampere introduced tensor-core support for a specific sparsity pattern: in every group of 4 consecutive weights along the input dimension, exactly 2 must be zero. The hardware stores the 2 non-zero values plus a 2-bit index telling which positions they occupied, and the tensor core's "sparse GEMM" kernel reads half the bytes and multiplies at twice the rate. 2:4 sparsity is the only sparsity pattern that produces a real speedup on modern GPUs. Unstructured sparsity at any ratio is dead on arrival for inference speed.

How 2:4 Sparsity Actually Works in Silicon
half the memory bandwidth consumed • GEMM reads activations normally, multiplies only at non-zero positions using the index → 2× effective FLOP/s • On H100: measured 1.8-1.95× real throughput improvement on LLM decode workloads For Nexus decode (memory-bound): 1.9× fewer bytes read per token → 1.9× fewer ms per TPOT step. Free speed.

The 2:4 pruning workflow

Applying 2:4 is simpler than it looks. The constraint — every 4 consecutive weights in each row must contain exactly 2 zeros — is a per-row, per-group rule. For each row, for each group of 4 weights, use Wanda or SparseGPT to score all 4 entries, keep the top 2, zero the bottom 2. One pass through the layer, no iteration. Then the weights are stored in the NVIDIA 2:4 compressed format and served through the sparse tensor core path. Every major serving engine (vLLM, TensorRT-LLM, SGLang) supports this today; you flip a flag at model-conversion time.

⚠

The 2:4 ceiling

2:4 is locked at 50% sparsity exactly — you cannot adjust. If 50% sparsity costs more accuracy than you can afford, 2:4 is off the table and you are back to unstructured sparsity with no speedup. For most base Llama / DeepSeek checkpoints at 70B scale, 2:4 via Wanda loses ~2-3 points MMLU — acceptable for many workloads, painful for others. The remedy is a short recovery fine-tune on a few billion tokens, which typically recovers 70-90% of the lost points. This is the path we take for Nexus v6 in combination with distillation.

04 · Knowledge Distillation

A smaller student learns from the full model.

Distillation (Hinton et al., 2015) is the most general compression technique in the toolbox: train a smaller "student" model to mimic a larger "teacher" model's outputs. Unlike quantization or pruning, distillation can change model architecture entirely — the student can be narrower, shallower, use different attention heads, or be an MoE while the teacher was dense. The cost is compute: a full distillation run burns through roughly 10-30% of the pretraining compute. The payoff is a model that, at a fixed parameter budget, outperforms one trained from scratch by several MMLU points.

Three families of distillation signals

Response distillation

Student matches teacher's output logits at each position. Loss = KL(p_student ∥ p_teacher) where both are softmaxed over vocabulary.

Simplest. Works when student capacity is comparable (within ~4× of teacher). Main signal for black-box APIs.

Feature distillation

Student matches teacher's hidden states at intermediate layers. Loss includes MSE or cosine terms between chosen hidden tensors.

Highest-fidelity. Needs white-box teacher access and matching (or learned-projection) layer dimensions. Best when student is much smaller.

Relation distillation

Student matches pairwise relationships between examples — e.g., cosine similarity of hidden states across a batch. Often combined with the above.

Best for retrieval/embedding tasks. Preserves the geometric structure of the teacher's representation space.

The loss function most Nexus-class distillations actually use

Composite distillation loss — production recipe L = α · CE(y_true, p_student) # ground-truth supervision, weight α
  + β · T² · KL(p_teacher/T ∥ p_student/T) # softened logit matching, weight β, temp T
  + γ · Σ_l MSE(h_student^l, W_l · h_teacher^l) # feature match on selected layers l

Typical: α=0.3, β=0.6, γ=0.1, T=2.0

The temperature T > 1 softens both distributions before the KL, which dramatically improves the signal on low-probability tokens. This is where the "dark knowledge" lives — the relative ranking of the top-K tokens the teacher considered but didn't emit. It is often more informative than just the argmax, because it tells the student "you were close to token X, not token Y" at every position.

Data quality is 80% of a good distillation

The single biggest lever in practical distillation is not the loss function — it's the data. You have three choices:

For Nexus, the third and fourth rows combine into the winning recipe: 2 billion tokens of real fraud-adjudication traffic (scrubbed) plus 400 million tokens of synthetic edge cases. The teacher is the 70B dense Nexus v5; the student is the 38B-active MoE we build in the next sections. Total compute: ~45,000 H100-hours (about 4% of pretraining cost). ⚠

The capacity-gap failure mode

Distillation stops working well when the student is too small relative to the teacher. If you try to distill Llama-3-70B into a 1B student, you'll lose double-digit points on most benchmarks no matter how much data or compute you use — there's a minimum parameter count below which the student simply cannot represent what the teacher knows. The rough guideline: student ≥ teacher / 5 for safe distillation without severe quality loss. For Nexus, 38B-active MoE distilling from a 70B dense teacher sits comfortably in the safe zone.

05 · Mixture-of-Experts · Sparse Activation

Keep all the parameters. Use a fraction per token.

MoE is the architectural innovation that dominated 2023-2025: instead of routing every token through a single dense FFN, partition the FFN into N "experts" and route each token to the top-K experts (K ≪ N). Total parameters stay high — preserving the model's knowledge capacity — but active parameters per token drop dramatically. DeepSeek-V3's 671B total / 37B active, Mixtral 8x22B's 141B total / 39B active, and DBRX's 132B total / 36B active all share this shape. For Nexus v6 we take the same playbook.

The top-K routing mechanism

MoE Forward Pass · Top-K Routing Over N Experts
38B active per token — 1.8× throughput on memory-bound decode.

The gating math, precisely

MoE layer forward pass — top-K gating 1. Token hidden state h ∈ ℝᵈ arrives at the MoE layer.
2. Router scores: s = softmax(W_g · h) ∈ ℝᴺ   where W_g ∈ ℝᴺˣᵈ
3. Select top-K indices by score: I = topk(s, K) ⊂ {1, ..., N}
4. Renormalize gates on the selected set: g_i = s_i / Σ_{j∈I} s_j   for i ∈ I, else g_i = 0
5. Output: o = Σ_{i∈I} g_i · Expert_i(h)   — only K forward passes through FFNs

The load-balance problem and auxiliary loss

If you just train MoE with cross-entropy on the output, the router will happily collapse to always using 1 or 2 favorite experts — those get all the gradient, get better, become even more favored, and the rest go unused. This is mode collapse and it kills MoE's capacity advantage. The standard remedy is an auxiliary load-balance loss that penalizes uneven expert assignment:

Auxiliary load-balance loss (Switch Transformer style) For each batch, let f_i = fraction of tokens routed to expert i
and let P_i = mean softmax gate weight on expert i across the batch.

L_balance = N · Σᵢ f_i · P_i

Added to the main loss with coefficient 0.01-0.05. Pushes f_i toward 1/N.

DeepSeek-V3 introduced a subtly better variant — auxiliary-loss-free balancing — where the router's raw scores are shifted per-expert by a bias that is adjusted each step based on observed load. The bias decreases for overloaded experts and increases for underloaded ones, with no gradient flow through the bias itself. This avoids the accuracy regression that pure auxiliary loss sometimes causes and is the current best-practice for large MoE training.

Shared experts — the Nexus choice

DeepSeek-V2 introduced an important architectural refinement: alongside the N routed experts (of which K fire per token), keep M shared experts that always fire on every token. Shared experts capture common knowledge all tokens need; routed experts specialize in domains. The result is better quality at the same active-parameter budget, because the common knowledge doesn't have to be duplicated across specialist experts.

For Nexus v6, we use 2 shared experts + 8 routed experts with top-2 routing. Every token fires the 2 shared experts plus its top 2 routed experts, for 4 total active experts per token. The architecture:

nexus_moe_layer.py · shared + routed experts forward passPYTHON
# Nexus v6 MoE layer. 2 shared experts fire on every token + top-2 of 8 routed.
# This shape is derived from DeepSeek-V2/V3 with hyperparameters tuned for
# the fraud-detection distribution on 2B tokens of production traffic.

import torch
import torch.nn as nn
import torch.nn.functional as F

class NexusMoELayer(nn.Module):
    def __init__(self, d_model=5120, d_ffn=3584,
                 n_routed=8, n_shared=2, top_k=2):
        super().__init__()
        self.top_k    = top_k
        self.router   = nn.Linear(d_model, n_routed, bias=False)
        self.shared   = nn.ModuleList([FFN(d_model, d_ffn) for _ in range(n_shared)])
        self.routed   = nn.ModuleList([FFN(d_model, d_ffn) for _ in range(n_routed)])
        # DeepSeek-style aux-loss-free load-balance biases (updated out-of-band).
        self.register_buffer("route_bias", torch.zeros(n_routed))

    def forward(self, x):    # x: [B, T, D]
        B, T, D = x.shape
        flat = x.view(-1, D)                                   # [BT, D]

        # 1. Shared experts — fire unconditionally on every token.
        out = sum(e(flat) for e in self.shared)              # [BT, D]

        # 2. Routing scores + bias + top-K selection.
        scores  = self.router(flat) + self.route_bias            # [BT, N]
        top_s, top_i = scores.topk(self.top_k, dim=-1)           # [BT, K]
        top_g = F.softmax(top_s, dim=-1)                       # renorm gates on selected

        # 3. Dispatch tokens to experts. In production this uses grouped_gemm;
        # here we show the clear functional version.
        routed_out = torch.zeros_like(flat)
        for e_idx, expert in enumerate(self.routed):
            mask = (top_i == e_idx)                               # [BT, K] bool
            if not mask.any(): continue
            token_idx, slot = mask.nonzero(as_tuple=True)       # which tokens pick this expert
            weights = top_g[token_idx, slot].unsqueeze(-1)        # [n_tokens, 1]
            routed_out.index_add_(0, token_idx,
                                   weights * expert(flat[token_idx]))

        out = out + routed_out
        return out.view(B, T, D)
06 · Expert Parallelism · The All-to-All Bottleneck

Experts live on different GPUs. Tokens travel.

A 70B MoE with 8 routed experts has, per MoE layer, roughly 6× the parameter count of its dense equivalent. At 70B total parameters and 60 MoE layers, you cannot fit all the experts on one GPU. The deployment pattern is expert parallelism: shard the experts across GPUs so each GPU holds only a subset. The price is that every MoE forward pass now requires sending each token's hidden state to whichever GPU owns the selected expert, then sending the result back — a full all-to-all collective operation in every MoE layer.

Expert Parallelism · The All-to-All Communication Pattern
The bottleneck: 2 all-to-alls per MoE layer × 60 MoE layers = 120 all-to-alls per forward pass. Communication budget determines deployment topology.

The deployment rule that matters

⚠

Keep the MoE within one NVLink domain

Each all-to-all has a latency floor set by the slowest link in the communication mesh. On NVLink (intra-node) an all-to-all completes in ~0.5-1.2 ms. On InfiniBand (inter-node) it completes in ~20-60 ms. With 120 all-to-alls per forward pass, inter-node MoE adds ~3 full seconds of pure communication time per token — catastrophic. The rule: the expert-parallel group must fit within one NVLink domain. For H100/H200 nodes with 8 GPUs, that means the MoE is sharded across at most 8 GPUs on one machine. If you need to scale beyond 8, you go tensor-parallel on the dense layers and keep expert-parallel at 8. DeepSeek-V3 runs this exact shape on 8×H800 nodes.

Hot experts — the skew problem

Even with load-balance loss and DeepSeek-style aux-free balancing, real traffic has skew. Some experts end up 2-3× more loaded than others on a given batch, which means their owning GPU becomes the straggler — all other GPUs wait for it. Two techniques mitigate this:

Token drop + capacity factor

Set a per-expert capacity = (tokens_in_batch × K / N) × capacity_factor (typically 1.25). Any tokens routed to an over-capacity expert are dropped (their MoE output becomes zero; residual connection still passes). Trains the router to avoid overload.

Slight accuracy cost (~0.2 MMLU), significant latency stability win.

Replicated hot experts

Identify the top 2-3 "hot" experts from production traces and replicate them across multiple GPUs. The router then round-robins to a replica for tokens hitting those experts, spreading load.

Memory overhead (~10-15% more parameters resident), but eliminates the hot-expert straggler. Nexus v6 uses this for its 2 most-loaded routed experts.

07 · Putting It Together

How Nexus v6 uses all three.

Pruning, distillation, and MoE are often presented as alternatives. In production they compose. For Nexus v6 we use all three: the student is an MoE (not dense), the student is distilled from the dense v5 teacher, and the student's dense layers (attention, shared experts) carry 2:4 structured sparsity. This layered recipe gives compounding benefits that no single technique matches.

The v6 pipeline, end-to-end

Nexus v6 Training + Deployment Pipeline
1 × 8-GPU H200 node per region (DGX-equivalent, NVLink full-mesh) • Expert parallelism 8-way — each H200 hosts 1 routed expert; shared experts replicated everywhere • Tensor parallelism 4-way on attention and shared experts (2:4 sparse GEMMs) • PagedAttention, RadixAttention, EAGLE-2 spec decoding — everything from M3–M5 still in play • All-to-all over NVLink — 0.6-0.9 ms per collective, 60 MoE layers × 2 = 108 ms total per token (absorbed into step time) Down from 3 nodes in v5 → 1 node in v6. Same region throughput, 1/3 the hardware.
08 · Debugging Drills

Four traps across pruning, distillation, and MoE.

Pruning, distillation, and MoE share a pattern: each is a capacity-vs-quality tradeoff that looks clean on paper and becomes murky in production. These four drills catch the common traps — hardware mismatches, teacher drift, expert imbalance, and communication bottlenecks — that turn a "2× faster" deploy into a 2-week investigation.

Drill · 01
2:4 sparsity applied but TPOT unchanged
SYMPTOM
Team prunes a model to 2:4 structured sparsity per NVIDIA's recipe. Verifies sparsity pattern is correct. Deploys. TPOT is within 1% of the dense model. Team is confused — advertised speedup was 1.8×.
WRONG HYPOTHESIS
"2:4 sparsity must not be mature yet." Red herring. 2:4 works well; the issue is almost always a missing prerequisite.
CORRECT DIAGNOSIS
2:4 sparse acceleration requires THREE conditions: (1) Ampere or newer (A100, H100, H200, B200 — NOT V100); (2) the matmul kernel is using the sparse tensor core path (cuBLAS --sparsity flag, cuSPARSELt, or TensorRT-LLM's sparse plugin); (3) the sparsity pattern must be stored in the 2:4 compressed format, not just have 50% zeros in dense layout. Check: if your weights are saved in FP16 with 50% zeros but no compression metadata, kernels see a dense tensor and skip the sparse path. Confirm via nsys: sparse kernels are named differently (e.g. sparse_gemm_sm80_tn). If you see dense kernel names, you're not on the sparse path.
FIX
Save weights in 2:4 compressed format using the vendor tool (NVIDIA's apex.sparsity or cuSPARSELt). Load with a sparse-aware runtime. Confirm kernel names in nsys are sparse variants. Expect 1.5-1.8× speedup on compute-bound layers; smaller gain on memory-bound decode (because the dominant cost is HBM reads, not compute). If most of your wall-time is attention at small batch, 2:4 won't help much — optimize KV first (M4).
Drill · 02
Distilled student matches teacher on eval; drifts in production over weeks
SYMPTOM
Distilled 7B student from 70B teacher. Offline eval: student F1 = 0.92, teacher F1 = 0.94 — acceptable. Ship to production. Over 6 weeks, student F1 slowly drops to 0.85 while teacher (kept as shadow) stays at 0.94. Nothing was redeployed.
WRONG HYPOTHESIS
"Something changed in production" — but nothing deployed. The model weights are identical to week 0.
CORRECT DIAGNOSIS
The traffic changed. Students generalize worse than teachers — they learned the training distribution more narrowly. When production traffic drifts away from the distillation set (new tenants, new merchant categories, new fraud patterns), the student's F1 drops while the larger teacher still generalizes. This is the distillation analog of the speculation-drift problem from M5. Check: sample production prompts from week 0 and week 6, compute embedding distance between the distributions. If the distributions have moved, the student's training set is stale relative to production.
FIX
Re-distill monthly on rolling production traffic. Automate: the distillation pipeline pulls the last 30 days of prompts, generates teacher completions, trains the student, evaluates on the most recent week. Monitor student_F1 / teacher_F1 ratio; if it drops below 0.95, re-distill immediately. Long-term: add a "generalization reserve" during distillation — hold out 10% of the training data at the distribution edges so the student learns to handle distributional variation, not just the median case.
Drill · 03
MoE utilization stuck at 40%; 2 experts handle 60% of traffic
SYMPTOM
Deployed Mixtral 8×7B with expert parallelism (EP=8, one expert per GPU). Per-GPU utilization: 85%, 82%, 40%, 45%, 30%, 35%, 40%, 38%. Two GPUs (experts 0 and 1) are consistently hot; six are nearly idle. Total throughput is 45% of design target.
WRONG HYPOTHESIS
"The GPUs are different" or "something's broken on the idle GPUs." Hardware is fine. This is expert imbalance.
CORRECT DIAGNOSIS
MoE routers are NOT guaranteed to balance load across experts. Left unconstrained, they collapse: two or three experts handle most tokens; the rest starve. Check vLLM/Megatron MoE metrics: moe_expert_load histogram per expert per step. If the top-2 experts hold >40% of load (ideal is 12.5% each for 8 experts), you have routing collapse. Common causes: no load-balancing loss during training; load-balancing loss weight too low; production traffic narrower than training traffic (so the router over-specializes).
FIX
Short-term: enable "capacity factor" in the MoE router (caps how many tokens any single expert can receive per step; overflow tokens are dropped or re-routed). Set capacity_factor=1.25 (25% slack). Medium-term: re-fine-tune the MoE with a stronger load-balancing loss (coefficient 0.01-0.1 depending on model). Long-term: if one expert is permanently over-subscribed in your domain, consider a replication strategy — duplicate the hot expert on two GPUs, route half the tokens to each. Utilization should rise from 40% to 80%+. This is a routing problem, not a hardware problem.
Drill · 04
Scaling EP from 4 to 8 GPUs makes MoE SLOWER
SYMPTOM
Team runs Mixtral with EP=4 at 8k tok/s/GPU. Scales to EP=8 expecting throughput to grow. Measured: EP=8 gives 3.2k tok/s/GPU — per-GPU throughput dropped 60%, total throughput only marginally better than EP=4.
WRONG HYPOTHESIS
"More GPUs means more overhead — expected." Not for any other form of parallelism. EP has a specific failure mode.
CORRECT DIAGNOSIS
Expert parallelism requires an all-to-all communication step per MoE layer: every GPU sends tokens to the expert on another GPU, then receives tokens back. The all-to-all's total bytes scale as O(batch × seq_len × hidden_dim), and the latency scales poorly past 8 nodes — especially on non-NVLink topologies. At EP=4 within a single node (NVLink), all-to-all is ~1-2% overhead. At EP=8 across two nodes (Infiniband), all-to-all becomes 30-50% of step time. The compute savings don't compensate. Check nsys: look for ncclAllToAll kernels and their wall-time percentage. If >15%, you've hit the communication bottleneck.
FIX
Keep EP within NVLink boundaries — single node for EP=8 (H100/H200 DGX has 8 GPUs on one NVLink fabric). If you need more scale, use tensor parallelism (TP) within the node and data parallelism (DP) across nodes instead of stretching EP. Alternative: hierarchical EP where experts are grouped within nodes and tokens are routed within a group when possible (vLLM and Megatron-LM both support this). Last resort: drop expert-parallelism entirely and replicate the whole model, accepting the memory cost. Rule: for MoE, the bottleneck is almost always communication, not compute.
💡

The common thread

Pruning, distillation, and MoE all promise capacity gains via structural techniques. The traps are different in kind but similar in shape: the promised speedup requires a specific compatibility layer (hardware path, training regime, topology) that's easy to miss. 2:4 needs sparse kernels. Distillation needs traffic-matched re-training. MoE needs capacity-balanced routing and low-latency all-to-all. Before adopting any of these techniques, verify the compatibility layer FIRST — the technique is only as fast as its weakest dependency.

Q1 · Hardware & sparsity
A team prunes a 70B model to 60% unstructured sparsity using SparseGPT, packs the weights in CSR format, and expects a 2.5× decode speedup on H100. What actually happens?
They get the expected 2.5× speedup.
Essentially no decode speedup. Dense GEMM kernels don't skip zeros; the tensor cores multiply at the same rate regardless of value. HBM reads drop only if the sparse format is actually read (most kernels re-densify). For real inference speedup, they need structured sparsity (2:4) that hardware recognizes.
They get a 1.5× speedup but a large accuracy regression.
CSR format cannot be used with transformer weights.
The hardware story dominates. Unstructured sparsity is excellent academically but useless for GPU inference speed. The one pattern that translates to real throughput on modern silicon is 2:4 structured sparsity, which NVIDIA tensor cores support natively via a sparse-GEMM path. Anything else saves memory at best, not latency.
Q2 · SparseGPT vs Wanda
What is Wanda's core insight that lets it approach SparseGPT's accuracy with a much simpler algorithm?
Wanda uses a different Hessian formulation.
Wanda scores weights by |W| × ||X|| (weight magnitude times activation norm) rather than needing Hessian inversion and error propagation. The score captures "this weight is large AND sees large inputs" — a surprisingly good proxy for importance without training or Hessian math.
Wanda requires iterative pruning while SparseGPT is one-shot.
Wanda only works for transformer FFN layers.
Wanda's simplicity is the point. No Hessian, no Cholesky, no error compensation, no block structure. Just score = |W| × ||X|| and keep top-K per row. For 2:4 structured sparsity the Wanda-vs-SparseGPT quality gap is often within noise, and the implementation cost is a 10× difference.
Q3 · 2:4 constraint
Can you use 2:4 structured sparsity at 75% sparsity (3 zeros of 4) to get even more compression?
Yes — just change the pattern from 2:4 to 1:4.
No, at least not with current NVIDIA tensor cores. The sparse tensor core only supports exactly the 2:4 pattern (2 of 4 non-zero). Other ratios would need software decompression, which usually costs more than it saves on decode workloads. 2:4 is locked at 50% exactly.
Yes, but only on H200 and newer.
Yes, 2:4 can be chained to produce 75%.
The hardware supports exactly one ratio. Blackwell is rumored to add 4:8 as a variant pattern, but as of 2026 deployments, 2:4 (50%) is the only option that produces real GEMM speedup on mainstream accelerators.
Q4 · Distillation capacity
A team wants to distill Llama-3-70B into a 3B student for a customer-support use case. What's the biggest risk?
Distillation compute will be too expensive.
Capacity gap. At 70B/3B ≈ 23×, the student simply does not have enough parameters to represent what the teacher knows. Even perfect training data will leave the student many points below the teacher on most benchmarks. Rule of thumb: keep student ≥ teacher/5.
3B models cannot be distilled.
The student will overfit.
The capacity gap is the fundamental limit. Under-sized students plateau no matter how much data they see. The practical fix: either accept lower accuracy (often fine for narrow domains with a small eval surface), or pick a larger student (e.g., 14B) that fits the rule. For Nexus, 38B-active from 70B dense is a safe 1.8× ratio — well inside the viable range.
Q5 · Distillation loss
Why does distillation use softmax temperature T > 1 in the KL loss?
To make the student train faster.
Higher T softens both distributions, which amplifies the informative "dark knowledge" in the teacher's low-probability tokens. The relative ranking of non-top tokens carries real signal about how the teacher reasons, and T > 1 makes that ranking visible in the gradient. At T=1, the hard-argmax tokens dominate and most of the teacher's nuance is lost.
To match the teacher's learning rate schedule.
Required by the PyTorch KL-divergence implementation.
"Dark knowledge" is Hinton's original framing. A teacher that says token "A" 80%, "B" 15%, "C" 5% is telling the student something different from one that says "A" 80%, "B" 1%, "C" 19%, even though both argmax to "A". The soft-T loss preserves that distinction and is one of the most important knobs in distillation — typical T is 2-5.
Q6 · MoE economics
A 70B-total MoE with 2 shared + 8 routed experts (top-2 routing) has what active parameter count per token?
70B (all parameters fire)
8.75B (70B / 8 routed experts)
Roughly 38B — active experts per token = 2 shared + 2 of 8 routed = 4 of 10, so about 40% of FFN params fire. Combined with attention (always active), the active fraction is slightly higher — ~55% — giving 38B effective active parameters.
15B
Back-of-envelope: FFN is typically 2/3 of a transformer's parameters, attention is 1/3. Of FFN, only 4 of 10 experts fire → 0.4 × (2/3) ≈ 27% of params from FFN. Attention always fires → 33%. Total active ≈ 60% of parameters. For a 70B total, that's roughly 38-42B active. Confirmed empirically for Nexus v6's final shape.
Q7 · Load balancing
Without any load-balance intervention during MoE training, what happens to the experts?
They stay balanced automatically because softmax is fair.
Mode collapse: a few favored experts get all the gradient, get better, get favored more, and the rest go unused. Without an auxiliary load-balance loss or aux-loss-free bias updates, the router converges to using 1-2 experts and the MoE reduces to a dense model with extra idle parameters.
All experts receive equal traffic because tokens are randomly distributed.
The router learns to match tokens to the expert with lowest initial loss.
The feedback loop is the issue: favored experts → more gradient → better fit → higher router scores → even more favored. The Switch Transformer paper popularized the auxiliary L_balance = N · Σ f_i · P_i loss, and DeepSeek-V3 improved on it with aux-loss-free bias updates that avoid the small accuracy cost of the gradient term.
Q8 · Expert parallelism
You're deploying an 8-routed-expert MoE across a 16-GPU cluster (two 8-GPU NVLink-connected nodes). What expert parallelism degree should you pick?
EP=16 (one expert per 2 GPUs for memory safety)
EP=8, within a single NVLink domain. The second node should run as a separate replica (data parallel). Spanning expert parallelism across InfiniBand inter-node links turns every all-to-all into a multi-millisecond collective and adds seconds per forward pass — unworkable for real-time serving.
EP=2, with tensor parallelism filling the rest.
It doesn't matter — the router handles locality.
The NVLink-vs-InfiniBand gap is the deciding factor. Intra-node all-to-all ≈ 1 ms; inter-node ≈ 30 ms. With 120 all-to-alls per forward pass, the difference is 0.12 s vs 3.6 s per token — not a latency regression, a latency extinction. Nexus v6's deployment pattern (EP=8 within node, data-parallel replicas across nodes) is the only production-viable topology.
Q9 · Composition
For Nexus v6, what was the approximate order of operations for stacking quantization, pruning, distillation, and MoE?
Quantize v5, then prune, then distill, then convert to MoE.
Distill v5 (dense FP16) into a new MoE student first (knowledge transfer needs full-precision teacher). Then prune the student's dense layers (2:4 via Wanda) with short recovery fine-tune. Then quantize everything (AWQ W4 weights + FP8 KV). Ordering matters: quantizing before distilling wastes the teacher's signal; pruning before distilling degrades the teacher.
All four simultaneously during pretraining.
Order doesn't matter — they commute.
The rule is "start with the highest-precision signal and compress progressively". Distillation needs the teacher at full fidelity, so it comes first. Pruning needs a trained student to know which weights are dispensable — second. Quantization is the cheapest and most mechanical step — last. Any other order loses quality with no compensating gain.
