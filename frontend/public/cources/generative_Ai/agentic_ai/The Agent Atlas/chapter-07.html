01 — Why a Vocabulary at All

The model is rarely what breaks

Building GenAI systems can feel like a brand-new discipline demanding brand-new ideas. It isn't. The real engineering challenges — coordination, reliability, scalability, security, observability, integration — are exactly the ones software engineers have wrestled with for decades. The only new wrinkle is that one of our components, the LLM, is an unreliable, slow endpoint we reach over a REST API.

So this chapter introduces pattern-guided coding: the discipline of using named, well-understood software patterns as the primary design vocabulary for building GenAI systems. Crucially, the patterns aren't freshly invented "agentic patterns." They're the established Gang of Four (GoF) design patterns and Enterprise Integration Patterns (EIP) — already documented, widely understood by architects and developers, and (this matters) deeply understood by LLM coding assistants like Claude. The novelty is applying them systematically to agentic architectures, queue topologies, failure handling, and deployment manifests before writing any code.

The code follows from the design; the design follows from the patterns.

A production-ready GenAI system has to clear a demanding bar. Before a single line is written, it's worth stating the ten requirements explicitly — because meeting all ten at once is what forces a specific architectural choice.

1

Documentable — architects, devs, and ops share one unambiguous understanding of what's built.

2

Deployable — no hidden runtime deps; every step scriptable and version-controlled.

3

Monitorable — observable queues, measurable throughput, clear failure signals without reading source.

4

Tunable trade-offs — start simple, scale deliberately, don't over-engineer up front.

5

Adaptable — add consumers, routing rules, failure strategies without rewriting the core.

6

Integrable — connect to back-office systems via known paradigms, not bespoke connectors.

7

Secure — TLS in transit, cert-based auth, vhost isolation, fine-grained permissions.

8

Trusted & proven — built on technology with a long production track record.

9

Easy to evolve — grow capacity without redesigning the messaging layer from scratch.

10

Well-supported — a large, active community with books, docs, and training to onboard people.

The architectural choice these ten force

Use AI to build the application, but deploy it on well-established, battle-tested infrastructure. RabbitMQ satisfies all ten as a messaging layer — observable, scriptable, secure, proven at scale, decades of accumulated knowledge. That leaves one challenge: precision. How do you describe, without ambiguity, exactly what you want the AI to build on top of it? The answer is the shared language of GoF and EIP.

02 — Three Reasons It Works

Why a shared pattern language matters

Pattern-guided coding isn't a radical change — good teams already communicate in patterns. The only difference now is adding an LLM coder to the team. Three concrete reasons explain why making the vocabulary explicit pays off.

1

Communication breaks down without shared words

When one engineer says "I added an agent for routing" and another hears "I added a service that dynamically selects which downstream service to call," they mean the same thing — but without shared pattern names, the conversation is imprecise and error-prone. Code review, architecture review, and onboarding all suffer. Naming the patterns — Strategy, Mediator, Content-Based Router, Scatter-Gather — makes the design vocabulary explicit and shared.

2

Design decisions become invisible

Write code without an explicit pattern vocabulary and the design decisions get baked implicitly into the implementation. A queue acting as a dead-letter channel, a class embodying Strategy, a fanout exchange broadcasting to many agents — all real decisions, all invisible unless you already know what to look for. Pattern-guided coding makes them visible by requiring them to be named and reviewed before implementation. It's really just enforcing best practices when an LLM is doing the coding.

3

GenAI systems fail at the integration layer

Most production failures aren't model failures — they're integration failures: messages that can't be retried safely, agents that crash and lose in-flight work, workflows that stall when a tool times out, no mechanism for malformed LLM output. These are exactly the problems EIP was designed to solve. A system built with explicit dead-letter channels, correlation identifiers, and process managers has the structural machinery to survive. A system built without them is forced to improvise.

Why not "Agentic Patterns"?

A pattern, in the sense Gamma, Helm, Johnson, and Vlissides established, is a named, documented solution to a recurring design problem in a context. Terms like ReAct, Plan-and-Execute, and Reflection describe LLM prompting strategies — they have no canonical implementation, no formal structure, no defined participants and collaborations. Treating them as patterns conflates prompting technique with software architecture.

Pattern-guided coding is precise: GoF and EIP are the patterns. Everything else is the problem to which they're applied. Note too: LLMs are trained on both good and bad code and can't tell them apart — but patterns are different. Patterns are well-defined, thoroughly understood descriptions of structure. That's exactly why steering an LLM with them produces better code.

1Code

"Agent for routing" → Strategy

Naming it Strategy tells the whole team — and the LLM — that routing logic is an injected, swappable object. Suddenly everyone reviews the same thing.

2Business

The 3 a.m. integration failure

It's almost never "the model was wrong." It's a tool timeout with no DLQ, or a crashed consumer that lost work. EIP names the fix before the outage happens.

3Everyday

Recipe vs. "just cook it"

"Make it taste good" is unreviewable. "Sauté, then deglaze, then reduce" names each step so anyone can check the plan before the stove is even on.

03 — How It Works

Wiring agents is just queue topology

Pattern-guided coding maps every significant design decision to one or more named patterns before code is written. The process is iterative: propose a mapping, justify it by name and intent, invite review, proceed. The vocabulary is deliberately constrained to GoF and EIP — and that constraint is productive, forcing precision and killing the vague gesturing that plagues early technical discussions.

In agentic AI, the central question is simply how agents are wired to talk to each other — what the book calls component collaboration. ReAct, Plan-and-Execute, and RAG are just different ways agents can work together. On a messaging system like RabbitMQ, those ways of working together are queue topologies. Queues are superhighways; messages speed between agents as each finishes its task. Change the routing between agents and you change which "agentic pattern" you're implementing — or invent your own.

producer / delivery exchange & queues consumers (LLM / vector DB calls)

You can do pattern-guided coding with nothing but an LLM that knows patterns and RabbitMQ — try it. But to make it easier and safer, the book builds a Claude "skill" called Topologos that provides shortcuts and commands to build, modify, and deploy agentic applications. It enforces security, promotes good design, and makes you explicitly choose the scalability levels you want. (The full Topologos manual lives in the book's Appendix B.)

04 — The Topologos Process

Four steps to a deployable architecture

Topologos is a four-step process for reaching a deployable architecture that meets your exact requirements. Architecture is always juggling compromises — you want enough scalability and reliability without paying for hardware to serve more users than you have, or building in more nines than you need. You can always scale horizontally later, and you can always break out to add custom steps; Topologos doesn't force its way on you. Here's the arc:

1

Clarification before design

Establish five key constraints. The process refuses to proceed until they're answered.

2

Pattern analysis

Analyze the problem through four simultaneous lenses, each mapping to specific patterns.

3

Iterative approval

Present the design one decision at a time; nothing is final until approved.

4

Final topology output

Emit a complete, deployable specification — diagrams, manifests, scripts.

Phase 1 — Clarification before design

Before any design work, the practitioner establishes five constraints: problem scope (which workflows are in scope), scale & reliability requirements, infrastructure constraints, security posture, and the failure tolerance model. This isn't optional — skip it and you get topologies that are technically plausible but contextually wrong (the right pattern for an eventually-consistent internal workflow is the wrong pattern for a regulated, synchronous, multi-tenant one). Topologos refuses to proceed until all five are answered. That refusal is the Template Method pattern applied to the design process itself: a fixed skeleton with mandatory steps that can't advance until each is complete.

Phase 2 — Pattern analysis across four lenses

With context set, the design is analyzed through four lenses at once, each mapping the problem to specific patterns:

Lens A

Agentic decomposition

Break the workflow into message flows. Who produces? Who consumes? Synchronous (Request-Reply) or async fire-and-forget? Loops, branches, human pauses?

→ Request-Reply (EIP)
Lens B

GoF pattern mapping

Map each agent interaction to a GoF pattern. Central orchestrator = Mediator. Behavior by routing key = Strategy. Fixed workflow, swappable steps = Template Method. Replayable action = Command. Naming is mandatory.

→ Mediator · Strategy · Command
Lens C

Security threat model

For each flow: who are producers/consumers, how sensitive is the payload, what's the blast radius if a queue is compromised? Proxy enforces access control; Message Filter is the enforcement point.

→ Proxy (GoF) · Message Filter (EIP)
Lens D

Failure mode analysis

Classify every queue's failures: transient (retry), permanent (quarantine), poison (quarantine+alert), expired, rejected. Each maps to EIP failure patterns.

→ Dead Letter Channel · Process Manager · Competing Consumers · Correlation ID

Phase 3 — Iterative approval, one decision at a time

The design is presented one decision at a time. Each names what's proposed, why (with explicit pattern citations), the trade-offs, and a topology diagram. Nothing is finalized until the team explicitly approves. This embodies the Chain of Responsibility pattern: each decision is a handler in a chain — approved, it passes to the next; rejected, the chain stops there and the decision is revised. The decisions follow a canonical order:

1 Exchange topology 2 Queue naming & durability 3 Routing key schema 4 DLQ topology 5 Security model 6 Correlation strategy 7 Consumer concurrency 8 Persistence 9 Acknowledgment policy
The friction is the point

The discipline in Phase 3 is to resist approving quickly. When Topologos proposes a fanout exchange and you ask "why not a topic exchange with wildcard bindings?", the answer either reveals a real trade-off you'd missed or confirms your instinct — either way you win. You can modify mid-decision ("approved, but the quarantine queue needs 90-day retention for compliance") and the constraint carries forward into the final manifest and operational notes.

Phase 4 — Final topology output

Once all decisions are approved, Topologos emits a complete specification — and notably, the output is not code. It's a design precise enough to drive code generation, infrastructure-as-code tooling, and code review:

◆Output

Topology diagram

Every exchange, queue, binding, routing key, producer, and consumer.

◆Output

Topology manifest

JSON ready for RabbitMQ's management API or a Terraform provider.

◆Output

Pattern reference card

Maps every component back to its GoF and EIP pattern.

◆Output

Security manifest

vhost layout, user permissions, TLS configuration.

◆Output

Standalone DLQ diagram

The dead-letter topology on its own, for ops clarity.

◆Output

Operational notes

Monitoring thresholds and recovery procedures — plus downloadable scripts and config in a zip.

06 — Inside Producers & Consumers

Four GoF patterns do the real work

An agentic microarchitecture is queues plus producers and consumers — and your custom logic (integrations, prompts, vector-DB callouts) lives in those producers and consumers. Topologos uses GoF patterns to generate the boilerplate: security, message creation and sending, correlation, headers, routing-key logic. Four patterns carry most of the load.

Strategy GoF · producer routing

A producer that picks a different routing key, exchange, or serialization format based on runtime conditions implements Strategy. The routing logic is encapsulated in a strategy object and injected at runtime — making the producer testable (mock the strategy), extensible (add strategies without touching the producer), and inspectable (log the active strategy). Different strategies might apply for dev vs. prod, high-priority vs. batch, or internal vs. external consumers.

// Strategy interface interface RoutingStrategy { exchange(): string routingKey(message: AgentMessage): string headers(message: AgentMessage): Record<string, string> } // Producer uses a strategy — it doesn't know which one is active class AgentProducer { constructor(private channel: Channel, private strategy: RoutingStrategy) {} async publish(message: AgentMessage) { const exchange = this.strategy.exchange(); const routingKey = this.strategy.routingKey(message); const headers = this.strategy.headers(message); await this.channel.publish(exchange, routingKey, Buffer.from(JSON.stringify(message)), { headers, persistent: true }); } }

Command GoF · task messages

Every agent task that needs to be replayed, audited, or retried should be a Command: a self-contained, serializable description of the action. It carries everything needed to re-execute — action type, all parameters, agent identity, timestamp, correlation ID — so the consumer needs no external context. This matters enormously for the dead-letter channel: a Command that's NACKed to the DLQ can be inspected, corrected, and replayed without any context lookup. An ad-hoc message depending on external state cannot be replayed safely, because that state may have changed.

// A Command message carries its own context { "command": "GENERATE_EMBEDDING", "payload": { "document_id": "doc_8821", "text": "Pattern Guided Coding is...", "model": "text-embedding-3-small" }, "meta": { "correlation_id": "wf-2025-07-4491", "workflow_id": "rag-ingest-batch-44", "agent_id": "embedding-agent-01", "timestamp": "2025-07-15T10:23:00Z", "retry_count": 0 } }

Template Method GoF · consumer pipelines

A consumer that processes messages through a fixed sequence — validate, deserialize, enrich, process, acknowledge — implements Template Method. The sequence is fixed; individual steps can be overridden or injected. Adding a new step (say, signature validation required by security policy) doesn't change the overall structure. The consumer has a fixed contract with the messaging layer: receive a message, run the pipeline, return an ACK or a NACK with a failure class. That failure class decides routing — transient failures go to the retry exchange with backoff; permanent failures and rejections go straight to quarantine.

Channel Adapter EIP · external integration

An LLM API call, a vector-DB query, or a tool invocation faces a translation problem: external APIs are synchronous and HTTP-based, but the pipeline is asynchronous and message-based. The Channel Adapter translates between them. From the pipeline's view, the LLM becomes just another message consumer producing results on a reply queue. Without it, the orchestrator makes a direct HTTP call inside message-processing code — timeouts block the thread, retries are ad hoc, errors go unclassified, and the LLM becomes a synchronous dependency. With it, the LLM API is wrapped behind a dedicated component that handles the HTTP call, timeouts via its own DLX, and error translation — making the orchestrator fully async and the LLM a replaceable part.

The non-negotiable rule: manual ACK

The Competing Consumers EIP lets many consumer instances bind to one queue and process in parallel. For this to be safe with agent tasks, every consumer must use manual acknowledgment. Auto-ACK removes a message the instant it's delivered — before processing finishes — so a mid-process crash loses it. Manual ACK holds the message in an unacknowledged state until the consumer explicitly ACKs (success) or NACKs (failure); if the consumer crashes, the broker re-queues it. An agent task that triggers a real-world action — writing to a database, calling an API — must be acknowledged only after the action completes and is durably recorded. Premature ACK gives unsafe at-most-once semantics; the correct target is at-least-once, with idempotency enforced at the consumer.

07 — When Things Break

The multi-tier dead-letter strategy

In production agentic systems, failures aren't exceptional — they're structural. Tool timeouts, malformed LLM output, rate-limit errors, schema violations, critic rejections: all expected events the system must handle without losing work. Pattern-guided coding prescribes a three-tier DLQ topology for every agentic task queue, where the tier is chosen by failure class:

The mechanism uses RabbitMQ's native dead-letter exchange (DLX) plus the delayed-message plugin for exponential backoff. Every work queue declares a DLX. When a consumer NACKs with requeue:false, RabbitMQ routes the message to the DLX, which sends it to the right retry tier or quarantine based on the failure class encoded in the routing key. Consumers must check the x-death header on every message: if the retry count exceeds the maximum (typically 3), the consumer publishes directly to quarantine rather than NACKing back to a retry tier. The quarantine queue is durable, has no TTL, and is read-only for the ops/replay service — and a non-empty quarantine queue triggers an immediate alert.

The deeper lesson

This failure-handling strategy shows a broader strength of Topologos: once a GenAI workflow pattern is named, the prompt can infer the required structure and generate a production-ready topology with minimal extra input. You don't hand-design the DLQ tiers — naming the pattern is enough for the machinery to follow.

08 — One Sentence, Whole Architecture

Generate any pattern with a single command

The most powerful property of Topologos: it carries a complete internal reference for every major GenAI workflow pattern. Naming a pattern in the opening message is enough to generate a fully justified, production-ready topology — without specifying individual exchanges or queues. Phase 2 does the decomposition automatically; Phase 3 surfaces every decision for approval. Each of these one-liners opens the full four-phase protocol:

Each invocation opens the full four-phase protocol. The pattern's internal quick reference supplies the starting topology shape — exchange types, queue structure, the EIP patterns the workflow requires — and the approval loop lets the team refine every decision before implementation. The final /manifest produces a JSON topology manifest; /security produces the security manifest. From a single sentence to a production-ready specification, in one session.

From session to deployed project

The Topologos session is only the first phase of a complete project-generation pipeline. Once topology decisions are approved, the outputs drive every downstream artifact — infrastructure code, application code, tests, docs — with no extra manual specification. The full sequence: (1) the session produces /manifest (JSON) + /security (JSON) + /summary (prose); (2) a Terraform RabbitMQ provider reads the manifest and provisions all exchanges, queues, bindings, and policies; (3) Vault PKI reads the security manifest and issues TLS certificates plus minimum-privilege permission triples; (4) code generation reads the manifest + pattern reference card and emits correctly-structured producer/consumer skeletons; (5) an ADR document is generated from manifest + diagrams; (6) an ops runbook is generated from manifest + DLQ topology. At no point does a developer hand-transcribe a queue name, exchange type, routing key, or permission triple.

Two more habits make the loop pay off. You can inject requirements mid-decision — "approved, but the quarantine queue needs 90-day retention for compliance, and add an expiry handler for human-in-the-loop SLA breaches" — and the constraint flows forward into the manifest and ops notes. And the prompt audits existing systems just as well: describe your current topology and ask it to check against its RabbitMQ-specific rules. It flags deviations — queues without a DLX, consumers on auto-ACK, vhosts shared across trust boundaries, payloads carrying credentials that should be claim-checked — and routes each fix through the same pattern-justified approval loop.

09 — Case Study

From one LLM to two, by patterns

Extensibility is a requirement for most real applications — when requirements change, you don't want a rewrite. This is a genuine weakness of the "agentic patterns," which are often wrapped in technologies that assume stable requirements. Pattern-guided coding handles change gracefully. Let's watch it work on a common production ask: a team has a working single-LLM RAG pipeline and wants to run two LLMs at once (GPT-4 and Claude), merging their outputs before returning a result. Ensemble strategies improve reliability and soften single-provider outages.

This change is not trivial. Adding a second LLM to a sequential pipeline isn't duplicating a function call — it requires a new exchange type, two parallel queues, a new aggregator with a correlation window, and a merge strategy as a GoF Strategy. Each addition has a pattern name, and each needs a DLQ decision. The discipline's gift: it makes the full scope of the change visible before a line of code is written.

The "before" topology — a single LLM

The initial pipeline is a linear, sequential flow. A FastAPI producer publishes a query to a topic exchange (rag.query). The retrieval agent consumes from rag.retrieval, calls Pinecone, and publishes an enriched message to a direct exchange (rag.enrich). The generation agent consumes from rag.generate, calls OpenAI GPT-4 through a Channel Adapter, and publishes to a reply queue (rag.response); the FastAPI consumer returns the answer. Every queue has a three-tier DLQ feeding rag.dlx → retry-1 → retry-2 → quarantine.

The single point of failure

Four components sit in the critical path: query exchange, retrieval queue, enrich exchange, generation queue. The generation agent is the single point of failure — if GPT-4 is degraded or rate-limiting, every query queues behind it. The DLQ handles transient API failures with backoff, but there's no fallback path. That's the gap the dual-LLM design closes.

10 — The Session

Phase 2 analysis, before proposing anything

With the existing topology running in production, the team opens a new Topologos session and states the requirement: run two LLMs in parallel (GPT-4 and Claude), merge their outputs before returning, stay within the single-LLM latency budget, and — crucially — stay resilient: if one LLM fails or times out, still return a result from the other. True to its discipline, Topologos responds with pattern analysis first. Three changes surface across the lenses:

Lens A · decomposition

Sequential → fork-and-join

The retrieval agent now produces two messages instead of one — one per LLM queue — and the results must be collected before responding. One input, multiple parallel outputs, one aggregated result.

→ Scatter-Gather shape
Lens B · GoF mapping

Fanout + Aggregator + Strategy

The fork is Scatter-Gather via a fanout exchange (rag.scatter) broadcasting to both generation queues. The join is the Aggregator EIP — a new agent that publishes only when both results for a correlation_id arrive. The merge policy is a GoF Strategy, swappable at runtime.

→ Scatter-Gather · Aggregator · Strategy
Lens D · failure modes

A brand-new failure class

The partial result: one LLM replies, the other times out. Not a transient failure of any queue — a structural failure of the correlation window. It needs its own DLQ (rag.agg.quarantine) and a TTL on the window: if both don't arrive within SLA, publish the one available result with a quality flag rather than nothing.

→ Dead Letter Channel · Correlation ID · TTL

The "after" topology — dual-LLM Scatter-Gather

After Phase 3 approval of four decisions — scatter exchange, dual generation queues, aggregator with correlation window, merge strategy — the topology takes its new shape. The Aggregator's correlation window

The Aggregator is the most complex new component. For each query it must collect exactly two results — one from GPT-4, one from Claude — matched by their shared correlation_id. The merged result can't be published until both arrive, and the agent must handle the case where one never does. The implementation is a short-lived in-memory map keyed by correlation ID: the first result starts a TTL timer; the second cancels it and triggers the merge; if the timer fires first, the fallback branch publishes the single result with a partial quality flag and quarantines the entry.

const window = new Map(); // correlation_id -> { results[], timer } onMessage(msg) { const id = msg.headers['x-correlation-id']; const entry = window.get(id) ?? { results: [], timer: null }; entry.results.push(msg.payload); if (entry.results.length === 1) { // First result — start the TTL timer and wait entry.timer = setTimeout(() => handleTimeout(id), SLA_MS); window.set(id, entry); channel.ack(msg); } else { // Second result — both in: cancel timer and merge clearTimeout(entry.timer); window.delete(id); const merged = mergeStrategy.merge(entry.results); // GoF Strategy channel.publish('rag.merge', merged); channel.ack(msg); } } handleTimeout(id) { const entry = window.get(id); window.delete(id); const fallback = mergeStrategy.fallback(entry.results); // Strategy fallback fallback.headers['x-quality-flag'] = 'partial'; channel.publish('rag.merge', fallback); channel.publish('rag.agg.quarantine', { reason: 'timeout', ...entry }); }

Notice where the work moved. The EIP patterns got us the structure — the fork, the join, the failure path — but the actual merge logic lives in the producers and consumers, so here we lean on a GoF pattern. The Strategy pattern governs how the Aggregator combines the two results: injected at startup, swappable without touching the Aggregator's code. Three strategies cover the common production needs:

Best-of

Score both results with a lightweight heuristic (length, coherence, confidence if available) and return the higher scorer. Fast, deterministic, zero extra LLM calls.

Ensemble

Interleave both into a structured response, each answer labelled with its source model. Useful for high-stakes queries where the caller wants both perspectives.

Fallback

GPT-4 as primary; if it timed out or was quarantined, use Claude. Degrades gracefully to a single-LLM answer instead of erroring; the quality flag tells the consumer which path was taken.

Configuration, not code

Because the merge is a Strategy, switching from best-of to ensemble is a configuration change, not a code change. The Aggregator's structure is stable — receive two results, apply the selected strategy, publish the merged result. This is exactly the extensibility the "agentic patterns" lack: the structure absorbs the variation.

What /manifest produces after the session

Running /manifest after all five decisions are approved emits a JSON block with 3 new exchange declarations (rag.scatter fanout, rag.gather topic, rag.merge direct), 3 new queues (rag.gen.claude.work, rag.aggregate.work, rag.agg.quarantine), 4 new bindings (scatter→gpt4, scatter→claude, gather→aggregate, aggregate→merge), 2 new consumer definitions (claude-gen-agent, aggregator-agent), and updated DLQ policy entries for rag.agg.quarantine. The /security command adds 2 new permission triples for the new agents, scoped to minimum-privilege read/write on their own queues.

The same change, done the naive way

Without pattern-guided coding, adding a second LLM would likely become: duplicate the generation call, add a Promise.all(), merge in a callback. Fine for a prototype — but in production it hides real problems: no failure isolation between the two calls, no independent retry/backoff, no DLQ for partial results, no way to scale the two generators independently, and no way to swap merge logic without editing the generator. The pattern-first approach surfaces all of these before implementation. Scatter-Gather implies the two legs are independent — separate queues, consumers, DLX configs, prefetch. The pattern doesn't just describe the design; it forces the right one.

12 — Check Yourself

Four quick gut-checks

Answer each in your head, then reveal it. If all four land, you can drive a Topologos-style design session and read any agentic topology as patterns.

Why does the book refuse to call ReAct, RAG, and Reflection "software patterns"?
Because a pattern, in the GoF sense, is a named, documented solution to a recurring design problem with canonical participants, collaborations, responsibilities, and failure modes. ReAct and friends are LLM prompting strategies and workflow shapes — they have no canonical implementation or formal structure. Treating them as patterns conflates prompting technique with software architecture. GoF and EIP are the patterns; the workflow shapes are the problems to which they're applied.
An agent task triggers a real-world action like charging a customer. Why must the consumer use manual ACK?
With auto-ACK the broker removes the message the instant it's delivered — before processing finishes — so a mid-process crash loses the task silently. Manual ACK holds the message unacknowledged until the consumer explicitly ACKs (only after the action completes and is durably recorded) or NACKs; a crash re-queues it. This gives at-least-once delivery, which is safe when paired with idempotency at the consumer, instead of the unsafe at-most-once semantics of premature ACK.
In the dual-LLM case study, what genuinely new failure class appears, and how is it handled?
The "partial result" — one LLM replies and the other times out, so the Aggregator gets only one of the two expected results. It's a structural failure of the correlation window, not a transient queue failure, so it needs its own DLQ (rag.agg.quarantine) plus a TTL on the window. If both results don't arrive within SLA, the Aggregator applies the Strategy's fallback branch: publish the single available result with a partial quality flag and quarantine the entry for ops — returning something rather than nothing.
Switching the merge from "best-of" to "ensemble" — code change or config change, and why?
Config change. The merge policy is implemented as a GoF Strategy injected into the Aggregator at startup. The Aggregator's structure is fixed — receive two results, apply the selected strategy, publish — so swapping strategies needs no edit to the Aggregator or the generators. That absorbed variation is precisely the extensibility the wrapped "agentic patterns" tend to lack.
