01 — The Prototype Trap

Why so many GenAI apps die in production

Building an AI app can feel deceptively easy: write a prompt, grab one of the dozens of "agentic frameworks," and you're off. That's genuinely wonderful — more people get to feel the joy of building than ever before. But here's the trap: what you build that way usually cannot survive a real production environment.

Without proper preparation, an application tends to run out of resources, crawl to a halt under peak load, get compromised because nobody designed for security, or end up an isolated island because it's too painful to integrate with anything else. This is not a hypothetical. Reports from MIT indicate that in 2025, fewer than 5% of GenAI applications were judged to have delivered real value to their organizations — and a major reason was teams choosing shiny new tools that simply weren't ready beyond the prototype phase.

A demo proves it can work once. Production proves it works ten thousand times, at 3 a.m., while something is on fire.

So this chapter takes a different path. Instead of reaching for an unproven framework, we'll build a genuinely robust RAG application on RabbitMQ — an open-source message broker hardened over years in mission-critical systems — and we'll design it using pattern-guided coding: describing our architecture to an LLM in the language of well-known software patterns, then letting it generate the implementation. Before we build anything, though, we need a shared definition of the bar we're clearing. What does "production-grade" actually mean?

03 — Choosing Your Tools

Why not just pick a framework?

Research the landscape and you'll find dozens — hundreds — of tools marketing themselves as platforms for agentic apps: LangChain, LlamaIndex, n8n, DSPy, and more. So why not grab one? For enterprise work, the answer is blunt: they simply haven't been around long enough to prove they can handle the rigour of production.

They add a convenience layer to a space where mature, proven products already exist — and with vibe coding rising, the value of low-code convenience is shrinking fast. Factor in the extra learning curve and an uncertain future in a turbulent market, and it's a shaky bet. There's also a deeper truth: GenAI apps don't live in a vacuum. The more value they add, the more urgently they need to integrate with other back-end systems — and IT departments fight constantly to reduce the number of technologies they maintain, because each new tool carries hidden costs in skills, hardware tuning, and obsolescence risk.

1Business

The framework that aged out

A team bet 2024 on a hot agent framework; by 2026 it was abandoned upstream and they owned an unmaintained dependency in a critical path. Maturity is a feature.

2Code

One less language to staff

RabbitMQ has bindings for nearly every language, so the team integrates GenAI using skills they already have — no new runtime to hire for.

3Everyday

Cast iron vs. gadget pan

The novelty non-stick pan is convenient until the coating flakes in a year. The cast-iron skillet your grandmother used still works. Production rewards the skillet.

04 — The Building Blocks

Queues and topics, decoupled

RabbitMQ lets us build with two primitives. A queue is first-in-first-out, and crucially only one consumer can take a given message — once dequeued, it's gone for everyone else. A topic flips that: many consumers can each receive the same message. Two roles move through both: producers send, consumers receive. Compose queues and topics into a topology and you can satisfy almost any requirement.

Producer Consumer / Subscriber Message flow

Why does this matter so much? Because queues and topics decouple producers from consumers — and decoupling buys flexibility, adaptability, and easy scaling. Most bottlenecks dissolve by simply adding more producers, queues, or consumers. Messaging systems also throw in guaranteed delivery (messages can't vanish) and dead-letter queues (if a consumer fails, the message is parked for retry rather than lost). In our app, producers are whatever wants to invoke an LLM, and consumers are the LLM clients — and RabbitMQ guarantees a consumer's reply is routed back to the right producer and correlated with the original request. The best part: we never architect at the raw queue level. We identify the integration patterns we want and let RabbitMQ — or pattern-guided coding — implement them.

06 — The First Build

RAG as a request-reply with a twist

Building our RAG app is two steps: identify the patterns we need, then configure the queue topology. Happily, most LLMs know these patterns cold and can generate reliable implementations — we just verify the output. At its heart, RAG is a plain request-response with one twist: the message gets augmented with retrieved data along the way. The RAG orchestrator asks the vector database client for context, merges it into the message, then forwards the enriched message to the LLM.

request path reply path enrichment fetch

With the modest effort of configuring a few queues, we inherit three battle-tested integration patterns at once — each one a named solution from the Enterprise Integration Patterns canon. (Full descriptions live in the book's Appendix A; here's the intuition.)

07 — Three Patterns, Nearly Free

What the queues give you

Request-Reply EIP

The producer fires off a message and does not block waiting for the answer. It carries on with other work; the reply arrives later through the response queue. In a system serving thousands of concurrent users, not tying up a thread per in-flight request is the difference between scaling and melting.

Content Enricher EIP

Along the message's journey to the consumer, external data is fetched and merged in to give the LLM more relevant context. This is the "augmented" in retrieval-augmented generation — the vector database's chunks are stitched onto the prompt mid-flight.

Correlation Identifier EIP

When many requests and replies are in flight at once, how does a reply find its original request? A correlation ID stamped on each message. The producer doesn't burn threading resources waiting — it simply registers for a notification and gets matched up automatically when the reply lands.

Pause & assess — is this enough?

With just these decisions, our app already has scalability (add queues, producers, consumers), easy integration (any system queues a message and gets notified), and monitoring (the admin UI). For some applications, we'd stop right here. But a high-value RAG app needs to be genuinely robust and flexible — so let's evolve it.

4Business

Request-Reply = the deli ticket

You take a number and sit down instead of standing frozen at the counter. The deli (LLM) calls your number (correlation ID) when your order's ready. Nobody blocks.

5Content Enricher

Middleware that adds headers

Like an HTTP middleware injecting an auth token before the request hits the handler — context gets added in transit, transparently to the sender.

6Correlation ID

The coat-check stub

You hand over your coat, pocket a numbered tag, and wander off. The number is what reunites you with the right coat later — no waiting by the rack.

08 — Evolving for the Real World

One data source? Try many

We quietly assumed a single source of enrichment data — the vector database. But why only one? Companies have content management systems, file stores, relational databases, and often multiple vector databases per department. Enriching an order might need context from a Customer service, a Pricing service, and an Inventory service. Good architecture anticipates this. The move: turn the enrichment queue into a topic, scatter requests to all sources at once, and aggregate the replies — the classic Scatter-Gather pattern layered on top of Content Enricher.

services enricher / aggregator exchanges & queues

The enricher now consumes the base order event, scatters enrichment requests across customer, pricing, and inventory queues, waits for replies (matched by correlation ID), and gathers them — combining everything into one enriched event, or publishing a partial result if some source times out. That graceful-degradation detail is straight out of our production checklist: keep working, even when one dependency is slow.

09 — Making It Truly Robust

Routing, retries, and the safety net

A robust, flexible RAG app needs four more capabilities. Each maps to a real failure you'd hit in production, and each is solved by RabbitMQ machinery rather than custom code.

🔀

LLM routing

Send some requests to LLM A, others to LLM B — for cost, or because one model is better at a task. The producer decides the route.

📥

Dead-letter queueing

If a message can't reach the LLM — network glitch, model down, rejection — don't discard it and don't let it block the queue. Park it in a dead-letter queue for retry or analysis.

🧾

Message history

Track which components a message passed through. Priceless for debugging and for seeing how load spreads across the system.

♻️

LLM reliability

As we learned in Chapter 1, an LLM behaves like an unreliable endpoint with no uptime guarantee. Robust retry and fallback strategies are essential.

To wire all this together we introduce an exchange. The mental model the book gives is perfect: an exchange is a mail sorting centre, and queues are mailboxes. The exchange reads a message's headers and routes it to LLM A or LLM B accordingly. Layer on retry queues with TTLs (wait 30 seconds, then 5 minutes, before retrying), a dead-letter exchange for messages that exhaust their retries, an alternate "fallback" exchange that catches unroutable messages, and an optional trace exchange that copies every message for history — and you have a genuinely production-hardened design.

Figure · LLM routing with robust safe-fallback

7Business

Economy vs. premium routing

Cheap bulk classification goes to LLM A; a high-stakes legal summary goes to LLM B. The router optimizes cost against quality per request.

8Code

TTL-staged retries

First retry after 30s, second after 5m (exponential backoff), then dead-letter. No tight loop hammering a downed model — RabbitMQ TTLs do the waiting.

9Everyday

Exchange = mail sorting centre

Letters (messages) hit the sorting centre (exchange), which reads the address (headers) and drops each into the right mailbox (queue). Undeliverable mail goes to a returns office (dead-letter queue).

10 — The LLM Client

The consumer that actually calls the model

A critical component plays the role of message consumer: the one that invokes the LLM. It must be production-ready enough to handle thousands of transactions and gracefully absorb any trouble with the model. To do that without nasty surprises, it has to account for a daunting nineteen concerns:

01 Authentication 02 Retry policy 03 Rate-limit handling 04 Prompt formatting 05 Response parsing 06 Tool-calling policy 07 Streaming policy 08 Request builder 09 Response parser 10 Tokenizer wrapper 11 Streaming handler 12 Error mapper 13 Logging 14 Metrics 15 Caching 16 Tracing 17 Retries 18 Cost tracking 19 Safety filtering

That's a lot — and the wrong way to handle it is to build a separate client per LLM vendor, duplicating all nineteen each time. The right way is patterns. Three Gang of Four patterns carry almost the entire load.

Template Method GoF

Centralizes the request lifecycle in one place so logic isn't duplicated across providers. An abstract base owns the algorithm — validate input → build provider request → send → parse response → normalize output — and subclasses fill in only the provider-specific steps.

Adapter GoF

Isolates each vendor's API quirks behind one shared internal interface. An OpenAI adapter, an Anthropic adapter, a Gemini adapter, a local vLLM adapter — the client talks to the adapter, never to raw vendor payloads. Switching LLMs becomes, at most, writing a new adapter.

Strategy GoF

Makes variable behaviours pluggable objects instead of hard-coded subclasses — retry policy, authentication, timeout, prompt formatting, streaming, tool-calling. Swap behaviours at runtime and avoid the subclass explosion that would otherwise come from combining every option.

The practical Python rule

You don't need heavy inheritance for every provider. The Pythonic shape is: one reusable orchestrating client, adapter objects per provider, and strategy objects for variable behaviour — letting composition do most of the work. The rule of thumb: Template Method for the stable request lifecycle, Adapter for provider differences, Strategy for policies and behaviour variations.

11 — Generating & Shipping It

Speak in patterns, let the LLM build

Here's the payoff of doing all this design work in the vocabulary of patterns. Pattern-guided coding lets you implement the code and configuration for a pattern-based architecture with very little — sometimes zero — manual coding. Because LLMs know these patterns deeply, you describe intent in the shared language of software engineering, and the model produces the implementation. You stay the architect; the LLM is the very fast junior engineer who has memorized every textbook.

# Describe the architecture in patterns — not in framework-specific code BUILD: a production RAG app on RabbitMQ. PATTERNS: - Request-Reply # producer never blocks - Content Enricher # merge vector-DB context mid-flight - Scatter-Gather # fan out to customer/pricing/inventory, aggregate - Correlation Identifier # match replies to requests LLM CLIENT (message consumer): - Template Method # validate → build → send → parse → normalize - Adapter # OpenAI, Anthropic, Gemini, local vLLM - Strategy # retry, auth, timeout, prompt-format, streaming ROBUSTNESS: - route via exchange on headers (llm.a vs llm.b) - retry queues with TTL backoff (30s, then 5m) - dead-letter exchange for exhausted retries - fanout fallback exchange for unroutable messages - trace exchange copying every message for history DELIVER: a portable RabbitMQ definitions.json plus all deploy files.

Deploying it — infrastructure as code

The entire architecture is captured in a RabbitMQ file called definitions.json, and several LLMs can generate it along with every other file and script needed to install the system — portable across Linux, macOS, and Windows. In the pre-AI era this was onerous hand-work; now there is no need to write any of it manually. The deployable artifacts form one small, reproducible folder, each file with a clear job:

rabbitmq-canonical/ docker-compose.yml # the RabbitMQ service: image (rabbitmq:3-management), # ports 5672 (AMQP) + 15672 (UI), volumes, health checks .env # secrets & config (DEFAULT_USER, DEFAULT_PASS), # kept out of version control, read via VARIABLE_NAME refs rabbitmq/ rabbitmq.conf # server config: listeners, logging, memory, TLS; # enables auto-loading of definitions.json on startup enabled_plugins # activates plugins, e.g. [rabbitmq_management] definitions.json # the whole topology: users, vhosts, permissions, # exchanges, queues, bindings, policies

This single folder bundles declarative, reproducible deployment: Docker Compose orchestration, environment secrets, server configuration, plugin activation, and the entire queue/topic topology. Everything lives in the book's GitHub repo with a README and full install instructions — true infrastructure as code, satisfying the "operable" and "recoverable" rows of our production checklist.

12 — Check Yourself

Four quick gut-checks

Answer each in your head, then reveal it. If all four land, you can architect a RAG app that survives contact with production.

Why turn the single enrichment queue into a topic when adding more data sources?
A queue lets only one consumer take each message; a topic lets many consumers each receive a copy. To scatter one enrichment request to customer, pricing, and inventory services simultaneously, you need every source to get the message — that's a topic, enabling the Scatter-Gather pattern.
An LLM call fails because the model is briefly down. What should happen to the message?
It should be rejected and routed — via TTL retry queues with backoff (30s, then 5m) — and only after exhausting retries land in a dead-letter queue for later inspection. Never discard it (data loss) and never leave it blocking an active queue.
You need to add support for a new LLM vendor. Which pattern minimizes the work, and how much?
Adapter. Because every vendor is hidden behind one shared internal interface, adding a new provider is — at most — writing a single new adapter. The orchestrating client, the Template Method lifecycle, and all Strategy policies stay untouched.
How does building on RabbitMQ help satisfy the production checklist almost for free?
Because the messaging patterns it implements are the accumulated solutions to those exact problems: guaranteed delivery and retries give reliability and resilience; the management UI gives observability; adding queues/consumers gives scalability; dead-letter and fallback exchanges give recoverability. You configure topology instead of reinventing each property.
