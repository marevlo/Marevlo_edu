01 — The Pattern

Thought, Action, Observation

The previous chapter introduced Topologos and walked through designing a RabbitMQ topology. This chapter builds on that foundation by actually implementing an agentic pattern on top of one. The pattern is ReAct — Reasoning and Acting — introduced in a 2022 paper by Yao et al. The name says it all: the agent alternates between two operations in a tight loop.

In a Thought, the agent reasons about the current state of the problem and decides what to do next. In an Action, it calls a tool with a specific input and receives back an Observation. That cycle repeats — think, act, observe, think again — until the agent has gathered enough information to produce a Final Answer. Mechanically, the agent calls an LLM to decide which tool to use, dispatches the tool call as a message, receives the result as an observation, and loops.

Thought (LLM) Action (tool dispatch) Observation (result) Final Answer

By the end of this chapter you'll have a working ReAct agent that reasons about a question, dispatches tool calls over RabbitMQ, receives observations, and produces a final answer. Along the way you'll meet correlation IDs threaded across a reasoning chain, manual acknowledgements, three-tier dead-letter handling, the Competing Consumers pattern, and the classification of failures as poison, transient, or permanent. The system is built from four source files:

react_agent.py

Implements the core Thought/Action/Observation loop and integrates with RabbitMQ.

tool_worker.py

Independently scalable workers for the search, calculator, and weather tools.

publish_command.py

A small CLI for sending questions into the agent.

topology.json

The full RabbitMQ setup — exchanges, queues, a three-tier DLQ, and a quarantine queue.

What you'll need to follow along

RabbitMQ 3.12+, Python 3.11+, and an Anthropic API key. Every example in the chapter is runnable, and the full source is provided. One implementation note: the agent polls react.tool.results.queue using basic_get with a correlation_id filter. That's simple and correct for one agent per question — in a high-concurrency system you'd replace it with a per-correlation reply queue declared by the agent and deleted on completion. We'll return to this when we harden for production.

02 — The Messaging Layer

Setting up the topology

To run the ReAct loop as a distributed system, you need a messaging layer that can route commands, dispatch tool calls, and handle failures. That's exactly what topology.json defines — and you load it straight into RabbitMQ through the management HTTP API, the same API Chapter 7 promised would consume a manifest:

# POST the full definitions file to RabbitMQ's management API curl -u guest:guest -X POST \ http://localhost:15672/api/definitions \ -H 'Content-Type: application/json' \ -d @topology.json

That single POST creates everything. The topology defines a set of exchanges and queues that handle four jobs: receiving commands, dispatching tool calls, collecting results, and routing failures. Here are the core elements:

Reading the whole thing as a picture makes the flow click. A command enters through the commands exchange and lands in the commands queue, where the agent consumes it. The agent publishes tool requests to the dispatch exchange, which routes by tool type to one of three tool queues, each drained by its own worker. Workers publish results to the results exchange, which feeds the results queue the agent polls. And hanging off the left, every queue's failures flow into the same three-tier DLQ chain — retry1, retry2, quarantine.

exchange queue agent / code dead letter

The complete topology.json declares the vhost, two users (react.agent and a management react.operator), their permission triples scoped to react\..*, all six exchanges, every queue, and their bindings. The single most important detail is how the dead-lettering is wired: each work queue sets x-dead-letter-exchange to retry1, the retry1 queues carry a 30-second TTL and dead-letter onward to retry2, and the retry2 queues carry a 5-minute TTL and dead-letter to quarantine. The TTL expiry is the retry timer. Here are the queue arguments that make the chain work:

// Every work queue dead-letters to retry1… { "name": "react.tool.search.queue", "durable": true, "arguments": { "x-dead-letter-exchange": "react.dlq.retry1.exchange", "x-dead-letter-routing-key": "dlq.react.tool.search.queue" } } // …retry1 waits 30s, then dead-letters to retry2… { "name": "react.tool.search.retry1.queue", "arguments": { "x-dead-letter-exchange": "react.dlq.retry2.exchange", "x-dead-letter-routing-key": "dlq.react.tool.search.queue", "x-message-ttl": 30000 } } // …retry2 waits 5 min, then dead-letters to the terminal quarantine. { "name": "react.tool.search.retry2.queue", "arguments": { "x-dead-letter-exchange": "react.dlq.quarantine.exchange", "x-message-ttl": 300000 } }
Why the failures route themselves

This is the elegant part: the application code never decides where a failed message goes. When a consumer NACKs with requeue=false, RabbitMQ reads the queue's x-dead-letter-exchange and moves the message there automatically. The retry "timer" is just a message sitting in a queue with a TTL — when the TTL expires, the message dead-letters again to the next tier. Routing lives in the topology, not scattered through the agent. That's the whole point of doing the design first.

03 — The Agent

Four responsibilities, four functions

With the messaging infrastructure in place, we implement the agent that drives the loop. The agent has exactly four responsibilities, and each maps cleanly to code: receive a command, call the LLM to produce a thought and action, dispatch the tool call, consume the result as an observation — and repeat until a final answer emerges. That's the entire job.

Receive a command

The on_command callback fires for each message from react.commands.queue.

Think

_think calls the LLM with the full history and gets back JSON — an action step or a final answer.

Dispatch a tool

_dispatch_tool publishes a ToolRequest and blocks until the matching result returns.

Loop

run_loop feeds each observation back into context and repeats until done or the step limit.

Message schemas — and the thread that ties them together

Three dataclasses represent the messages moving over the wire. The Command carries the inbound question and the accumulated history; the ToolRequest is published to the dispatch exchange; the ToolResult is consumed from the results queue. The single most important field on all three is correlation_id. It's set once when the command first arrives and threaded unchanged through every request and result in the same reasoning chain. When the agent polls the results queue, it discards any message whose correlation ID doesn't match — those belong to a different in-flight question — and re-queues them for the right agent.

The Correlation Identifier pattern

This is the EIP Correlation Identifier doing exactly what Chapter 7 promised. One UUID, generated at the first command and copied onto every message it spawns, makes the entire causal chain reconstructible from logs alone — across the agent, the search worker, and the calculator worker. Without it, a distributed reasoning loop is an untraceable tangle. With it, you can grep one ID and watch a single question's whole journey.

Here are the three schemas. Notice how each one knows how to serialize itself to and from AMQP properties — the correlation ID and the step counter ride in the message metadata, not buried in the body:

@dataclass class Command: """Inbound message from react.commands.queue.""" correlation_id: str question: str step: int = 0 history: list[dict[str, str]] = field(default_factory=list) @staticmethod def from_delivery(body: bytes, properties) -> "Command": data = json.loads(body) return Command( correlation_id=properties.correlation_id or str(uuid.uuid4()), question=data["question"], step=int((properties.headers or {}).get("x-step", 0)), history=data.get("history", []), ) @dataclass class ToolRequest: """Published to react.tool.dispatch.exchange.""" correlation_id: str tool: str input: str step: int reply_to: str = TOOL_RESULTS_QUEUE def routing_key(self) -> str: return f"react.tool.{self.tool}.dispatch" # routes to the right worker queue @dataclass class ToolResult: """Consumed from react.tool.results.queue.""" correlation_id: str tool: str output: str step: int error: str | None = None

The LLM call — _think

Each call to _think reconstructs the full message history and sends it to the LLM. The system prompt is doing a lot of quiet work: it instructs the model to return valid JSON in exactly one of two shapes — an action step with a tool name and input, or a final answer — and it explicitly forbids the model from guessing tool results. That JSON contract is what makes the loop deterministic to parse. If the model returns non-JSON, _think raises a ValueError, which the loop treats as a poison-message failure: the command is NACKed without requeue and routed straight to quarantine.

def _think(self, question, history) -> dict: """Call the LLM; return a parsed dict — an action step or a final answer. Raises ValueError on unparseable output (the poison-message path).""" messages = [{"role": "user", "content": question}] for entry in history: # replay the whole chain each step messages.append({"role": "assistant", "content": entry["assistant"]}) if "observation" in entry: messages.append({"role": "user", "content": f"Observation: {entry['observation']}"}) response = self.llm.messages.create( model="claude-opus-4-5", # ← see note: parameterize this max_tokens=512, system=SYSTEM_PROMPT, # enforces the strict JSON schema messages=messages, ) raw = response.content[0].text.strip() try: return json.loads(raw) except json.JSONDecodeError as exc: raise ValueError(f"LLM returned non-JSON: {raw!r}") from exc
A note on the model string

The book's code hardcodes model="claude-opus-4-5". The most capable model available today is Claude Opus 4.7, so in your own build you'd point at the current model string — and, as the book itself recommends in its production checklist, you'd lift it into an environment variable so swapping to a faster or cheaper model never requires a code change. The code is shown here exactly as the book presents it; treat the model name as the one knob you'll update first.

Tool dispatch — _dispatch_tool

This method publishes a ToolRequest to react.tool.dispatch.exchange with a routing key of react.tool.<type>.dispatch; each worker queue is bound to its matching pattern, so the message lands on the correct worker type. After publishing, the agent polls the results queue with basic_get in a loop, comparing correlation IDs. Messages that don't match are NACKed with requeue=True so they stay available for other agent instances. If no matching result arrives within 30 seconds, a TimeoutError is raised and the command is NACKed to the Tier 1 retry queue.

def _dispatch_tool(self, request: ToolRequest) -> ToolResult: """Publish a ToolRequest and block until the matching ToolResult arrives. Filters by correlation_id; raises TimeoutError after 30s.""" self.channel.basic_publish( exchange=TOOL_DISPATCH_EX, routing_key=request.routing_key(), body=request.to_body(), properties=request.to_amqp_properties(), ) deadline, elapsed, poll = 30, 0, 0.2 # seconds while elapsed < deadline: method, props, body = self.channel.basic_get( queue=TOOL_RESULTS_QUEUE, auto_ack=False) if method is None: # queue empty — wait and retry time.sleep(poll); elapsed += poll; continue result = ToolResult.from_delivery(body, props) if result.correlation_id != request.correlation_id: self.channel.basic_nack(method.delivery_tag, requeue=True) # not ours — put it back time.sleep(poll); elapsed += poll; continue self.channel.basic_ack(method.delivery_tag) # ours — take it return result raise TimeoutError(f"No result for {request.correlation_id} after {deadline}s")

The loop and the ACK strategy

run_loop ties it together: for each step up to MAX_STEPS, it thinks, checks for a final answer, validates the chosen tool, dispatches it, folds the observation into history, and repeats. If it runs out of steps without an answer, it raises a RuntimeError — a permanent failure, so the question is never silently dropped. The real subtlety is in on_command, which wraps the loop in a try/except that classifies every failure and decides how the message is acknowledged and routed:

Look closely and you'll notice something that seems odd at first: every branch NACKs with requeue=False. So how do different failure types get different treatment? They don't, in the agent code — and that's deliberate. The dead-letter configuration on the queue determines how the message moves through the retry chain, so the agent never needs to encode routing. The application classifies and logs; the topology routes. (In a fuller build you'd also set an x-failure-type header so a queue-level policy can skip retries entirely for permanent failures — more on that in the production section.)

def run_loop(self, cmd: Command) -> str: history, step = list(cmd.history), cmd.step for _ in range(MAX_STEPS - step): step += 1 decision = self._think(cmd.question, history) if "final_answer" in decision: return decision["final_answer"] # done! if "action" not in decision: raise ValueError(f"missing action/final_answer: {decision}") action = decision["action"] tool_name = action.get("tool", "").strip().lower() if tool_name not in {"search", "calculator", "weather"}: raise ValueError(f"Unknown tool: {tool_name!r}") result = self._dispatch_tool(ToolRequest( correlation_id=cmd.correlation_id, tool=tool_name, input=action.get("input", ""), step=step)) observation = f"ERROR: {result.error}" if result.error else result.output history.append({"assistant": json.dumps(decision), "observation": observation}) # feed back into context raise RuntimeError(f"Exceeded MAX_STEPS={MAX_STEPS}") # permanent failure def on_command(self, channel, method, properties, body) -> None: """Per-message callback: run the loop, then ack/nack by failure class.""" cid = properties.correlation_id or "unknown" try: answer = self.run_loop(Command.from_delivery(body, properties)) channel.basic_ack(method.delivery_tag) # success except (json.JSONDecodeError, ValueError, KeyError): # POISON channel.basic_nack(method.delivery_tag, requeue=False) # → straight to quarantine except TimeoutError: # TRANSIENT channel.basic_nack(method.delivery_tag, requeue=False) # → retry1 (30s) via DLX except RuntimeError: # PERMANENT (step limit) channel.basic_nack(method.delivery_tag, requeue=False) except Exception: # UNKNOWN channel.basic_nack(method.delivery_tag, requeue=False)

04 — The Tool Workers

Simple consumers, independently scaled

With the agent in place, we need the workers that actually execute tool calls and return results to the loop. A tool worker is a simple consumer: it reads a ToolRequest, runs a function, and publishes a ToolResult. Three tool types live in the same file and are chosen by a CLI argument — search, calculator, and weather. The implementations here are intentionally simple, using keyword-matched canned responses and a safe arithmetic evaluator, so the example runs with no external API credentials. In production you'd swap each function for a real API call, but the RabbitMQ wiring, the acknowledgment strategy, and the error handling stay identical.

Two kinds of error — and why the distinction matters

The workers draw a careful line between two failure modes, because each deserves a different response. A ValueError means the input was valid but the tool simply can't process it — an unknown city, a malformed expression. Here the worker publishes an error ToolResult with the error field populated and acknowledges the message. The agent receives that error as an observation and can reason about it — perhaps retrying with corrected input. An unexpected exception, by contrast, is a likely-transient system failure; the worker NACKs without requeue, routing it to the Tier 1 retry queue to be retried after the TTL.

The distinction is the whole point. A ValueError is a known condition the agent can handle, so it's surfaced to the LLM as information. An unexpected exception is infrastructure failure, so it's handled by the messaging system and never bothers the LLM. One becomes part of the reasoning; the other becomes a retry.

def on_message(self, channel, method, properties, body) -> None: cid = properties.correlation_id or "unknown" step = int((properties.headers or {}).get("x-step", 0)) try: inp = json.loads(body).get("input", "") output = self.tool_fn(inp) # run the actual tool self._publish_result(cid, step, output=output, error=None) channel.basic_ack(method.delivery_tag) except (json.JSONDecodeError, KeyError): # POISON — malformed request channel.basic_nack(method.delivery_tag, requeue=False) except ValueError as exc: # KNOWN tool error # Publish an error result so the agent can reason about it, # rather than leaving the agent waiting on a timeout. self._publish_result(cid, step, output=None, error=str(exc)) channel.basic_ack(method.delivery_tag) # ← acked! it's an observation except Exception: # TRANSIENT — unknown, allow retry channel.basic_nack(method.delivery_tag, requeue=False)

One worker function deserves a closer look, because it's a small lesson in safety. The calculator must evaluate arbitrary arithmetic from the LLM, and a naive eval() would be a gaping security hole. Instead the worker parses the expression into an abstract syntax tree and walks it, rejecting any node type or name that isn't on a small allow-list — so sqrt(144) * 2 runs but an attempt to call os.system raises a ValueError (which, as we just saw, becomes a clean observation rather than a crash):

def tool_calculator(expression: str) -> str: """Safe evaluator: +, -, *, /, **, sqrt, parentheses. ValueError on anything else.""" allowed_nodes = (ast.Expression, ast.BinOp, ast.UnaryOp, ast.Constant, ast.Add, ast.Sub, ast.Mult, ast.Div, ast.Pow, ast.USub, ast.UAdd, ast.Call, ast.Name, ast.Load) allowed_names = {"sqrt": math.sqrt, "pi": math.pi, "e": math.e} tree = ast.parse(expression.strip(), mode="eval") for node in ast.walk(tree): # reject anything not allow-listed if not isinstance(node, allowed_nodes): raise ValueError(f"Unsafe node: {type(node).__name__}") if isinstance(node, ast.Name) and node.id not in allowed_names: raise ValueError(f"Unknown name: {node.id!r}") result = eval(compile(tree, "<expr>", "eval"), {"__builtins__": {}}, allowed_names) # safe — ast-checked above return str(result)

Scaling — the Competing Consumers pattern

Each tool type runs as its own independent worker process, which means you scale exactly the bottleneck and nothing else. Start one search worker, one calculator (it's CPU-bound and fast — one instance handles real load), and as many weather workers as the slow external API demands. Multiple instances of the same type all consume from the same queue, and RabbitMQ distributes messages across them. This is the EIP Competing Consumers pattern, and the beauty is that no coordination code is required — the broker does the load balancing for free.

# One search worker is often enough for low-throughput use python tool_worker.py search & # Calculator is CPU-bound and fast; one instance handles high load python tool_worker.py calculator & # Scale weather workers if the external API is slow — just start more python tool_worker.py weather & python tool_worker.py weather & python tool_worker.py weather &

Error as observation

Ask for the weather in "Atlantis." The worker returns an error ToolResult, acks it, and the agent reads "no data for Atlantis" — then reasons to ask the user for a real city instead of hanging.

2Business

Scale only the bottleneck

The weather API is slow during a storm. You spin up three weather workers and leave search and calculator at one each — capacity goes exactly where it's needed, no redeploy.

3Everyday

Many cashiers, one line

A supermarket opens more tills when the queue grows; shoppers take the next free one. Competing Consumers is that — the broker is the queue, the workers are the tills.

05 — Sending a Question

The publisher, and the whole thing running

The last source file is the smallest. publish_command.py is a thin CLI wrapper that publishes a command to react.commands.exchange with a freshly generated UUID as the correlation_id — and prints that ID so you can grep the agent logs for one specific reasoning chain. The message is marked persistent and carries an initial x-step of zero in its headers.

def publish(question: str) -> str: correlation_id = str(uuid.uuid4()) # one ID for the whole chain conn = pika.BlockingConnection(pika.URLParameters(RABBITMQ_URL)) channel = conn.channel() channel.basic_publish( exchange="react.commands.exchange", routing_key="react.command.new", body=json.dumps({"question": question}).encode(), properties=pika.BasicProperties( correlation_id=correlation_id, content_type="application/json", delivery_mode=2, # persistent headers={"x-step": 0}, ), ) conn.close() return correlation_id

Running it end to end

With RabbitMQ up, the topology loaded, and an Anthropic key in .env, you start four processes — the agent and one worker per tool — each in its own terminal, then send a question from a fifth:

# Terminal 1 — the ReAct agent python react_agent.py # Terminals 2–4 — one worker per tool type python tool_worker.py search python tool_worker.py calculator python tool_worker.py weather # Terminal 5 — ask a question that needs two tools python publish_command.py "How tall is the Eiffel Tower in feet?"

The publisher prints the correlation ID and tells you to watch the agent's logs. And there, in the agent terminal, the whole reasoning chain unspools — the LLM decides it needs the tower's height, searches for it, gets 330 metres, decides it now needs to convert to feet, calls the calculator, gets the number, and produces a final answer. Two tools, three thoughts, one clean trace:

06 — Watching Failure Work

Triggering the dead-letter chain

The happy path is satisfying, but the real test of an architecture is what happens when something breaks. The best way to understand the three-tier DLQ is to cause a failure on purpose and watch the messages move. The trick is simple: stop the search worker, then ask a question that needs a web search.

# Stop the search worker (Ctrl+C in its terminal), then: python publish_command.py "Who invented the telephone?"

Now two failures unfold in parallel, and watching both is the lesson. On one side, the agent dispatches a search request that no worker consumes. After 30 seconds the request's TTL on react.tool.search.retry1.queue expires and the message dead-letters to Tier 2; after 5 more minutes it reaches react.quarantine.queue. On the other side, the agent's own basic_get poll on the results queue times out after 30 seconds, so the agent NACKs the command — which dead-letters to react.commands.retry1.queue, expires to retry2 after 30 seconds, and lands in quarantine after 5 minutes. Two independent messages, each walking the same staircase down to the same terminal queue.

You can watch this happen live in the RabbitMQ management UI at http://localhost:15672. The message counts on the retry1 queues tick up, drain into the retry2 queues as TTLs expire, then finally drain into the quarantine queue. The staircase becomes visible as numbers moving from one step to the next on a timer.

Quarantine is a holding area, not a dead end

Restart the search worker and you'll notice the quarantined messages do not automatically replay — they require deliberate operator intervention, by design. Every message in quarantine carries its full x-death header chain: the original queue, the reason, the count, and a timestamp for each tier it passed through. That history is enough to diagnose exactly why the message failed and to decide whether to replay it or discard it. To replay, you pull the message from quarantine via the management API, inspect its x-death headers, and republish the body to react.commands.exchange with its original headers.

# Pull one message from quarantine to inspect its failure history curl -u guest:guest -X POST \ 'http://localhost:15672/api/queues/%2Freact/react.quarantine.queue/get' \ -H 'Content-Type: application/json' \ -d '{"count": 1, "ackmode": "ack_requeue_false", "encoding": "auto"}' # Then read the x-death chain, and republish the body to # react.commands.exchange with the original headers to replay it.
4Business

The 3 a.m. page

A worker pod crashes overnight. Messages climb the retry tiers, buying ops time to recover before anything quarantines. Nothing is lost — the system absorbs the outage and waits.

5Code

The TTL is the timer

There's no scheduler, no cron, no sleep loop deciding when to retry. A message sitting in a queue with a 30-second TTL is the retry delay — pure topology, zero application code.

6Everyday

Returned mail

An undeliverable letter doesn't vanish — it's stamped with why and where it's been, then held at the depot until a human decides to re-send or shred. That stamp is the x-death chain.

08 — From Prototype to Production

What to harden before you ship

This implementation is built for clarity and experimentation — it's a teaching system, and a few deliberate simplifications keep it readable. Before running anything like it in production, the book lays out eight concrete improvements. Each one closes a specific gap that the prototype leaves open on purpose:

1

Replace tool stubs with real API clients

Swap the canned responses for genuine calls — and wrap each external call with timeouts and circuit breakers so a slow or down dependency can't stall the loop.

2

Replace the poll with per-correlation reply queues

Have the agent declare an exclusive, auto-delete queue on startup and publish its name in reply_to; the worker then publishes results directly there. This eliminates the polling loop and the correlation-filtering overhead entirely.

3

Persist final answers

The prototype only logs the answer. Add a react.answers.exchange and a downstream consumer that stores answers in a database keyed on correlation_id.

4

Add publisher confirms on every publish

Without confirms, a published message can be silently lost if the broker restarts between the publish and the broker writing to disk. Confirms close that window.

5

Set x-failure-type headers explicitly

The code logs failure types but doesn't set them as AMQP headers. Setting x-failure-type: permanent lets a topology-level policy skip the retry queues and route straight to quarantine — no point retrying a schema violation.

6

Add TLS

Point RABBITMQ_URL at amqps:// and configure certificates. For inter-service connections in production, mutual TLS (mTLS) is strongly recommended.

7

Monitor quarantine queue depth

Any arrival in react.quarantine.queue is an operator alert. Wire the queue to a monitoring exchange and configure an alerting consumer — a non-empty quarantine means something needs a human.

8

Parameterize the model

The model string is hardcoded in react_agent.py. Move it to an environment variable so you can switch to a faster or cheaper model — or to the current Claude Opus 4.7 — without a code change.

The shape of the whole system

Step back and notice what this chapter actually built. The agent, the three tool workers, and the publisher are five independent processes with no direct network dependency on one another. They share nothing but the broker — and that's the design. Every rule the system follows traces back to the previous chapter: durable queues, manual acknowledgements, correlation IDs on every message, topic exchanges with named routing keys, and a three-tier DLQ on every queue. None of it is decoration. Each piece exists because a specific failure mode requires it.

7Code

The poll → reply-queue upgrade

At one agent, polling is fine. At a hundred concurrent questions, every agent scanning the shared results queue is wasteful. A per-correlation reply queue routes each result straight home — the single biggest production change.

8Business

Quarantine as a KPI

Ops dashboards the quarantine depth like a smoke alarm. Zero is healthy; any number is a signal to investigate. The architecture turns silent failures into a visible, alertable metric.

◆ Carry these forward
  1. ReAct is a Thought → Action → Observation loop. The agent calls an LLM to pick a tool, dispatches the call as a message, folds the result back as an observation, and repeats until a final answer.
  2. The broker is the system. Agent, three tool workers, and publisher are five independent processes with no direct network dependency — they share only RabbitMQ.
  3. Load the topology through the management API. One POST to /api/definitions creates the vhost, users, exchanges, queues, and bindings — the manifest-to-broker step Chapter 7 promised.
  4. Correlation IDs make the chain traceable. One UUID, set at the command and threaded through every request and result, lets you reconstruct a whole question's journey from logs across three processes.
  5. The three-tier DLQ routes itself. NACK with requeue=False; the queue's x-dead-letter-exchange and a TTL do the rest. The TTL is the retry timer; routing lives in the topology, not the code.
  6. Classify failures deliberately: poison (bad JSON → quarantine), transient (timeout → retry), permanent (step limit). Tool workers split known ValueErrors (become observations) from unexpected errors (become retries).
  7. Manual ACK and Competing Consumers are non-negotiable. Manual ACK protects in-flight work; multiple workers on one queue scale a bottleneck with zero coordination code.
  8. Quarantine is a holding area with full history. The x-death chain records every tier a message crossed, so operators can diagnose and replay — and a non-empty quarantine is an alert.
  9. Harden before shipping: real API clients with timeouts, per-correlation reply queues, persisted answers, publisher confirms, explicit failure headers, TLS, quarantine monitoring, and a parameterized model.
09 — Check Yourself

Four quick gut-checks

Answer each in your head, then reveal it. If all four land, you can read this ReAct system end to end — and explain why every piece is there.

The agent NACKs every failure with requeue=False. So how do poison and transient failures end up treated differently?
They aren't treated differently in the agent code — and that's deliberate. The NACK just dead-letters the message to the queue's configured x-dead-letter-exchange. From there, the topology decides everything: which retry tier it lands in, how long it waits (the TTL), and where it goes next. Routing logic lives in topology.json, not scattered through the application. The agent classifies and logs; the broker routes. (In a fuller build, an x-failure-type header would let a queue policy skip retries for permanent failures.)
A weather worker is asked for a city it doesn't know. Why does it ack the message instead of NACKing it?
Because an unknown city is a ValueError — a known condition the input was valid but the tool can't fulfill. The worker publishes an error ToolResult and acks, so the agent receives the error as an observation and can reason about it (e.g. ask the user for a real city). NACKing would send it to retry infrastructure, where it would fail identically forever. Only unexpected exceptions — likely transient infrastructure failures — get NACKed for retry. The distinction routes information to the LLM and infrastructure failures to the broker.
You stop the search worker and send a question needing search. Trace both messages that end up failing.
Two independent failures unfold. First, the agent's dispatched search request sits unconsumed in react.tool.search.queue; after its retry1 TTL (30s) it dead-letters to retry2, then after 5 minutes to react.quarantine.queue. Second, the agent's own basic_get poll times out after 30 seconds, so it NACKs the command — which dead-letters to react.commands.retry1.queue, expires to retry2 after 30s, and reaches quarantine after 5 minutes. Same staircase, two messages, one terminal queue. Restarting the worker does not auto-replay them — quarantine requires deliberate operator action.
Why dispatch tool calls over a RabbitMQ topic exchange instead of just making an HTTP call from inside the loop?
Because the broker gives you durability, retry, and dead-lettering for free, and it makes the tool workers independently scalable via Competing Consumers. An in-loop HTTP call makes the tool a synchronous dependency: a timeout blocks the thread, retries are ad hoc, errors go unclassified, and you can't scale one slow tool without touching the agent. Routing the call as a message turns the tool into just another consumer — failures flow into the DLQ chain, and you scale whichever worker type is the bottleneck by starting more processes. The transport choice is what buys all the reliability machinery.
