01 — The Pipeline We're Building

From a bucket of PDFs to searchable meaning

Everything so far has been groundwork. Now we build a genuinely practical pipeline: it reads PDF documents from Amazon S3, runs them through Airbyte to handle the whole extract-transform-load workflow, and lands the results as vector embeddings in the Pinecone vector database. The whole thing is configured through Airbyte's low-code interface — no heavy programming required.

Here's the shape of it. Documents start as raw PDFs in a cloud bucket. Inside Airbyte, three logical stages do the work: Load Data (pull the documents in), Create Embeddings (chunk and vectorize), and Ingest Data (write to the destination). Out the other end, the embeddings sit in Pinecone, ready for semantic queries. Notice how this maps cleanly onto the Extract-Transform-Load shape from the last chapter — the same three movements, now with a concrete tool doing each one.

This chapter has two halves. First we look under the hood at the software patterns Airbyte uses to make this possible — because understanding them is what lets you swap S3 for SharePoint, or Pinecone for Weaviate, without re-architecting anything. Then we roll up our sleeves and build the pipeline step by step.

02 — Under the Hood

The patterns that move the data

Airbyte, like most ETL tools, is built on well-known software design patterns — specifically from the Enterprise Integration Patterns (EIP) catalog by Gregor Hohpe and Bobby Woolf. Two of them do the heavy lifting in our pipeline. Once you recognize them, the tool stops being a black box.

Message Channel EIP

A message channel lets components that produce and consume data talk without being wired directly together. Instead of connecting systems point-to-point, messages travel through a channel. This decouples producer from consumer, so each side can scale independently and neither needs to know the other's internals.

In our pipeline, the producer reads documents from S3 and pushes the extracted data into the channel; downstream components process it and ultimately store it in Pinecone.

Channel Adapter EIP

A channel adapter connects an external communication channel — RabbitMQ, an HTTP API, a file store — to an application by translating between the channel's format and the system's internal representation. It's the universal plug that lets a foreign system speak your pipeline's language.

Airbyte ships two kinds. An inbound adapter brings data in (message-driven, reacting as messages arrive, or polling-based, checking periodically). An outbound adapter sends data out — listening on a channel and pushing to an external file, database, or API. The vector-database connector (Qdrant, Pinecone, Milvus) is an outbound adapter.

These two patterns are the foundation of how ETL tools manage data movement in practice. To build a pipeline, you specify only two things — the source (where information comes from) and the destination (where processed data goes). Airbyte handles the rest with internal strategy and adapter components. And because of that design, a pipeline reading from S3 today can be repointed at SharePoint or a relational database tomorrow without touching the underlying architecture.

04 — The Pattern That Makes It Configurable

One pipeline, many sources and destinations

How does Airbyte let you pick "read from S3, write to Pinecone" without writing custom code for every combination? The Strategy pattern — a behavioral design pattern that lets a program choose an algorithm from a family of algorithms at runtime. It defines interchangeable algorithms, wraps each in its own class, and makes them selectable on the fly, usually by passing a strategy object into a context object. The result is code that's flexible, interchangeable, and extensible without touching its core.

This is exactly why Airbyte minimizes your configuration effort: you simply tell it which adapter to use and hand over the connection details through a no-code interface. The Strategy pattern does the wiring.

The mental model

Think of the context as a power socket and each strategy/adapter as a different plug. You don't rewire your house to charge a different device — you just snap in the right plug. Choosing S3 over SharePoint, or Pinecone over Weaviate, is swapping the plug, not rebuilding the wall.

Strategy on the producer side — picking a source

To make Airbyte read from a source like Amazon S3, you give the context the source type plus the connection and security info the adapter needs to authenticate. The context then routes to the right data-source adapter.

Strategy on the receiver side — picking a destination

Writing to a destination works identically. Give the context the destination type plus its connection and security details, and it routes to the right vector-database adapter — ChromaDB, Pinecone, Weaviate, and so on. Same pattern, mirror image.

1Code

Payment gateways

A checkout that accepts a PaymentStrategy — Stripe, PayPal, or Square — picks the processor at runtime. Adding a new one means writing one class, not rewriting checkout.

2Business

Repointing the source

Legal moves its docs from a file server to SharePoint. The pipeline keeps running — you swap the source strategy and the rest of the architecture never notices.

3Everyday

Travel adapters

One laptop charger, a pocket of plug adapters. You don't buy a new laptop per country — you choose the right adapter. That's Strategy in your carry-on.

05 — Let's Build It

Five steps to a working pipeline

With tool selection done and the patterns understood, the build itself is refreshingly concrete. Five steps take us from nothing to a running pipeline:

Install Pinecone and Airbyte

The open-source Airbyte installs locally with ease, or you can use a free trial (no credit card). Pinecone offers Pinecone Local via Docker for an open-source-like local experience. The book's GitHub repo has setup instructions for Pinecone, Airbyte, and S3.

Analyze the document collections

Survey what you're migrating: categorization, PII, required authentication, file types (PDF/Excel/text), and structural patterns like chapters and subchapters.

Configure Airbyte

Point it at the source (S3) and destination (Pinecone), supplying connection URLs and credentials — via no-code UI, REST API, or Python SDK.

Migrate the documents

Run the pipeline: load from S3, chunk, embed, and write vectors into Pinecone.

Test the results

Run your sample semantic queries and confirm the retrieved chunks are relevant — catching any "drift" before it reaches users.

06 — Know Your Documents First

Survey before you ingest

Before configuring anything, survey and document the collections you intend to migrate. Walk each collection and interrogate it: Does it contain PII? What authentication is needed to access it? What document types are present — PDF, Excel, plain text? What structural patterns appear — do the PDFs have chapters and subchapters? The answers shape every later decision about chunking, security, and connectors.

If you discover several repositories of poorly categorized documents, consider running text-classification or topic-modeling algorithms before the ETL pipeline. These can build a taxonomy that enriches each document's metadata. Options range from a simple taxonomy extractor built with a prompt and a third-party LLM library, all the way to sophisticated cloud tools from Google, Amazon, and Microsoft. Covering every option is beyond scope — but for large, messy collections, a commercial or open-source taxonomy extractor is well worth considering.

4Business

Messy shared drive

Twenty years of files with no folders. A topic-modeling pass tags each doc with a department and category before ingestion, so later queries can filter precisely.

5Code

Prompt-built taxonomy

For a small collection, a single LLM prompt — "classify this doc into a 3-level hierarchy" — produces serviceable metadata without a heavyweight cloud tool.

6Everyday

Sorting the junk drawer

Before storing, you group: batteries here, cables there, manuals in back. Taxonomy is doing that to documents so you can find things fast later.

What each field is really doing

The bucket and file pattern tell Airbyte what to read (** means "every file, any depth"). The access key and secret are the credentials Airbyte attaches to every fetch so AWS lets it in. They're "optional" only because some buckets are public or use other auth — for a real private bucket, they're mandatory. This single screen is the source-side Strategy pattern in action: you picked the S3 adapter and handed the context its connection details.

10 — Step Three: Connect Them

One YAML file to bind it all

With a source and destination defined, the last step is the connection itself. Click the Builder tab and you get three ways to create a connector: import a YAML manifest, fork an existing connector, or start from scratch. We'll take the first route — providing a YAML file you customize and upload. YAML is a human-readable serialization language used for config files; you can open it in any text editor (VS Code, Sublime, Vim, and so on).

# Airbyte connection: S3 (documents) → Pinecone resourceType: connection name: s3-pdfs → pinecone workspaceId: ${AIRBYTE_WORKSPACE_ID} # Reference the source & destination config files source: ../sources/s3_pdfs/configuration.yaml destination: ../destinations/pinecone/configuration.yaml # Minimal catalog — let Airbyte infer the schema syncCatalog: streams: - stream: name: pdf_docs jsonSchema: {} # Airbyte manages schema details supportedSyncModes: ["full_refresh", "incremental"] config: syncMode: incremental # ← only process new/changed files destinationSyncMode: append # ← keep older versions too primaryKey: [["document_key"]] # de-dupe / stable upserts selected: true # Schedule (change as desired) schedule: timeUnit: hours units: 24 status: active

Two lines in that file deserve special attention, because together they decide how your pipeline behaves over time. We set syncMode: incremental and destinationSyncMode: append. In plain terms: when a new document lands in the S3 bucket, or an existing one is updated, only that file gets processed — Airbyte doesn't re-embed the entire collection — and older versions remain available rather than being overwritten. This is precisely the incremental-sync strategy from Chapter 5, now expressed in two lines of config. If you don't need to keep old versions, the Deduped mode saves space by discarding them; the Airbyte docs cover this and many other features in depth.

Why these two settings matter so much

Incremental + append is the difference between a pipeline that costs a few embeddings per night and one that re-embeds everything on every run. Remember the EPS arithmetic from Chapter 5: re-embedding ten thousand documents nightly is hours of compute and a real bill; re-embedding only the 1% that changed is minutes. The mode you pick here directly shapes your operating cost.

11 — Watching for Drift

Did the new data change the answers?

Every run of the pipeline carries a risk the book calls drift — a change in the system's behaviour caused by a new or modified dataset. After ingestion, you want to confirm that your semantic queries still return acceptable results, because fresh data can quietly shift what comes back. This is exactly where the work from Chapter 3 pays off: remember the best practice of writing a few dozen sample queries at the very start of a GenAI project? This is the moment that investment is redeemed. You replay those known queries, inspect the top chunks, and check that quality held steady. If it slipped, you've caught drift before your users did.

10Business

The policy-update surprise

An insurer ingests a revised manual; suddenly a common query returns the new rules — good — but also surfaces a deprecated clause. The sample-query suite flags it before customers see it.

11Code

Drift as a regression test

Wire the sample queries into CI. Every pipeline run re-checks them, so "ingest more data" can never silently degrade retrieval quality without a red build.

12Everyday

Re-tasting the soup

You add an ingredient, then taste again — because one change can alter the whole pot. Sample queries are how you re-taste the knowledge base after every addition.

12 — Check Yourself

Four quick gut-checks

Answer each in your head, then reveal it. If all four land, you can stand up a real ingestion pipeline and reuse it across projects.

Your boss wants to read from SharePoint instead of S3 next quarter. How much of the pipeline must you rebuild?
Almost none. Thanks to the Strategy pattern, the source is just an interchangeable adapter selected at runtime. You configure a new source strategy (SharePoint) and hand the context its connection details — the channel, the destination, and the rest of the architecture stay exactly as they are.
In the YAML, why set syncMode to incremental and destinationSyncMode to append?
Incremental means only new or changed files are processed, not the whole collection — which keeps cost and runtime low (recall the EPS arithmetic). Append means older versions are retained rather than overwritten, preserving traceability. If you don't need old versions, Deduped mode saves space instead.
The Pinecone destination form asks for chunk size and a text splitter. Where did those concepts come from?
Chapter 3's chunking decisions and Chapter 2's chunking discussion. The hard reasoning about chunk size and intelligent splitting didn't vanish — a mature ETL tool simply exposed each as a form field with a sensible default you can override. The expertise is baked in.
You re-ran the pipeline with a fresh batch of documents. What should you do before trusting it, and why?
Replay your sample semantic queries and inspect the top chunks, to check for drift — a change in system behaviour caused by the new dataset. New data can shift what retrieval returns, so the sample queries written at the project's start are how you confirm quality held steady before users are affected.
