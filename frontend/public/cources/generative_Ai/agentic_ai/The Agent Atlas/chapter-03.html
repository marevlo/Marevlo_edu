01 — A Different Kind of Project

Less coding, more knob-turning

If this is your first GenAI project, expect it to behave unlike anything you've built before. The reason is unusual: GenAI projects are dominated by circular dependencies between configuration parameters, and they reward careful tuning far more than they reward writing clever code.

Here's the trap. Success depends on finding good settings for many parameters at once — and those parameters lean on each other. Change a single one and the system can simply stop working, forcing a change to a second parameter, which creates a fresh incompatibility with a third, and around you go. If that pattern sounds familiar, it's because you've met it before: it's the same circular-dependency headache you've fought with software libraries. The comforting news is that, just as package managers tamed library dependencies, a good strategy can tame these.

With 15 interdependent variables, the combinations run into the billions.

That number sounds terrifying, and without a method it genuinely is — you could tune for months. But the goal of this chapter is to hand you that method: a way to group the parameters and order your decisions so the process becomes linear and manageable instead of a billion-armed octopus. Before any of that, four common-sense habits will save you from needless pain.

APractice

Control your environment

When you tune a parameter, make sure nobody else is modifying the system at the same time — otherwise you can't tell what caused what.

BPractice

Log every test (in version control)

Keep a record of what you tried and the result. You'll want to retrace your steps weeks later, and memory won't cut it.

CPractice

Parallelize relentlessly

Nothing forces sequential testing. Spin up several environments and test many parameter combinations at once to compress weeks into days.

DPractice

Document your goals

Write down what the system is for — via customer profiles, focus groups, and so on — so that later, when two settings both "work," you can choose the one that serves the goal.

02 — The Knobs

The fifteen parameters you'll tune

Tunable parameters hide inside three components: the vector database, the document chunker, and the LLM itself. Between them they let you control how documents are grouped and searched, how much text comes back from the store, and how creative the model is allowed to be — all in service of one goal: the best possible response to the prompt. Here is the full set. (This list isn't exhaustive, and the real lesson is learning how to think about tuning, not memorizing names.)

04 — Reading the Arrows

Five chains of cause and effect

The graph above isn't decoration — each arrow encodes a real rule of thumb. Here are the five families of dependency, the ones you'll feel every time you tune. Read them as "if I touch this, watch that."

Chain 1 Document → Chunking

Document types & structures → chunking strategy. Different formats (contracts, invoices, long-form PDFs) and structures (tables, headings, pages, sections) force different chunk boundaries. A spreadsheet doesn't split like a novel.

Chunking strategy → chunk size, overlap & chunk limit. Once you pick the rules — split by headings, by pages, by semantic breaks — you must then tune size and overlap, and set safety limits on how many chunks can exist.

Chain 2 Chunking → Retrieval

Chunk size & overlap → chunk limit. Larger chunks or more overlap means more total chunks for the same document, which pushes against your limits.

Chunk limit → max. chunks returned. If you cap how many chunks exist per document or pipeline stage, you've constrained what retrieval is even able to hand back.

Vector DB structure → similarity function. Repository structure — a single index vs. multiple, per-document-type collections, metadata filters, hybrid indexes — affects which similarity method is appropriate and how it's computed or combined.

Similarity function → max. chunks returned. Switching similarity (cosine vs. dot-product vs. hybrid scoring) changes the score distribution, so the optimal top-k usually shifts with it.

Chain 3 Retrieval ↔ Prompting

Max. chunks returned → prompts. Your prompt has to be built around how many chunks you feed it — their format, citations, compression, and ordering all change with the count.

Prompts → max. chunks returned. And it runs the other way too: if your prompt demands more evidence or citations, you raise top-k; if it demands tighter grounding, you lower top-k and raise the similarity threshold. This is the rare two-way arrow — a true feedback loop.

Chain 4 Model → Context / Tokens → Prompts

LLM → version → context size. Different versions typically come with different context-window constraints.

Context size → max. tokens & prompts. Your prompt template and output limits must fit inside the model's context window — exceed it and the call fails.

Max. tokens → prompts. Lower the maximum output tokens and your prompts often need stronger constraints, more summarization instructions, or a different answer format to stay useful.

Chain 5 Temperature ↔ Prompt behaviour

LLM temperature → prompts. As temperature rises, prompts usually need to get stricter to preserve determinism and faithfulness; when temperature is low, prompts can relax.

LLM version → temperature. Different versions behave differently at the same temperature, so swapping versions almost always means re-tuning temperature.

Why this graph is worth memorizing

A deep knowledge of these arrows can cut your build-and-tune time by orders of magnitude. Just as importantly, it lets you divide work across a team without people fighting each other — because you can see exactly which decisions ripple into which others before anyone starts turning knobs.

06 — A Worked Example

Teaching a machine the world of Harry Potter

Theory only takes you so far. Let's tune a real-ish application: a system that answers any question about the Harry Potter books, starting from all seven novels in PDF. Tuning GenAI is part science, part art — your first guesses will be wrong, and that's expected. The whole craft is an iterative loop of change something, test it, learn, repeat.

(For the uninitiated: the series follows a young wizard at Hogwarts as he battles the dark lord Voldemort across seven books, exploring friendship, bravery, and the fight between good and evil.) To tune anything, we first need a way to measure success — so we write sample questions paired with a "ground truth" answer to compare the LLM against. A real project would use at least 120 questions, including tricky multi-hop ones and broad ones with many valid answers. Four will do for us.

Sample question
Ground-truth answer
What magic is used in Harry Potter?
Witchcraft and wizardry — performed through spells, charms, curses, potions, and magical objects. (A broad question: many valid answers.)
Why does Harry have a lightning-shaped scar?
Voldemort tried to kill him as a baby with the Killing Curse, which rebounded.
What is a wand used for?
It channels a wizard's or witch's magic to cast spells.
What creature is Hagrid's pet, and its name?
A giant three-headed dog named Fluffy. (A specific question: one correct answer.)

We also pin down two goals and constraints: answers should be short and to the point — fewer than five sentences — and a given question should return the same result at least 97% of the time. (That second constraint will matter enormously when we reach temperature.) For the architecture, we use a simple RAG setup: the user's question triggers a semantic search, relevant chunks come back from the vector database, those chunks get added to the prompt, and the LLM generates an answer. We won't actually run an LLM here — the lessons don't depend on any specific model — so we'll focus purely on which chunks come back. With architecture, goals, dataset, and questions in hand, we're ready to start turning knobs. First knob: the chunking strategy.

07 — Tuning the First Knob

Five ways to slice a book

Chunking strategy is the first decision on the right branch of the dependency tree, and it's not fixed before development. We already know the nodes above it — document type is PDF, document structure is chapters. Now we evaluate five candidate strategies. Remember: there's rarely one right choice, but there are plenty of wrong ones. Watch how each option fails or survives contact with our actual questions.

1Treat the entire book as one chunk✗ rejected

Ask "what creature is Hagrid's pet?" and the answer actually lives in two places in Sorcerer's Stone — Fluffy the three-headed dog (Ch. 9) and, confusingly, Norbert the dragon (Ch. 14). But to answer, we'd load the entire 250+ page book. That risks running out of memory or blowing past the token limit of the LLM or embedding model. Even if it squeaks through, latency suffers, and because conversation history is bundled into every new request, each turn balloons in size. And since you pay per token, this gets expensive fast. Fine for tiny documents like a JSON response — hopeless for novels.

2Chunk by chapter✗ rejected

Ask the broad "what magic is used?" and it semantically matches many chapters across all seven books — inheriting every memory and cost problem from Strategy 1. Worse, a chapter sprawls across many topics; maybe only one sentence in it is about magic. Because chapters are big, we must cap how many we return, which means relevant data gets stranded in chunks we had no room to include. Great for a pile of small, unrelated docs like a product inventory — wrong for big, interlinked novels.

3Chunk by sub-chapter— unavailable

Promising in principle — smaller chunks, less dependency between them. But the Harry Potter books simply don't have sub-chapters. A reminder that your strategy is constrained by the document's actual structure, not just what you'd prefer. Off the list.

4Chunks of fewer than 100 tokens✗ rejected

(100 is arbitrary — just a number for the example.) Now we get real precision and small returned chunks, fixing the memory and cost issues of 1 and 2. But a highly specific query like "what creature is Hagrid's pet?" matches just one spot in one book — and when the relevant data sits in a single place, we actually want generous surrounding context, a few sentences before and after. A 100-token cap is so tight it skips the helpful neighbours. It can also slice mid-sentence, leaving a chunk that reads "Hagrid's pet is a kind of…" with the rest orphaned in the next chunk. Better than before, but flawed.

5~600-token chunks with intelligent line breaking✓ chosen

The winner. We take Strategy 4's precision but raise the chunk size to ~600 tokens, so we capture plenty of context before and after a match — and we use intelligent breaking that never cuts a sentence in half. We also cap max chunks returned at 5 (ordered by relevance) so a query matching many spots doesn't drown us; we can revise that number later if we find we're missing data beyond the fifth. Precision, context, and cost control, balanced. This is our pick for the Harry Potter collection.

The fishing metaphor — keep this one

Think of chunk size as choosing your net. A small hand net (small chunks) catches exactly the fish you want with little waste — and you can cast it in many spots. A trawler (huge chunks) scoops up enormous hauls at once, but most of it is bycatch you must laboriously sort, and the trawler can only drag through the one location it chose. Strategy 5 is a well-sized net cast in five good spots.

08 — The Creativity Dial

Temperature: how surprising should it be?

Temperature measures how predictable the LLM's response is. We don't always want sameness — asking for ten opening lines for an important email, we'd want variety. So the temperature dial lets you choose: low temperature → consistent, repeatable answers; high temperature → varied, creative ones.

Take "what is a wand used for?" Suppose the vector data supports two viable answers: "casting spells" and "focusing magical energy." Near zero temperature, call it 100 times and maybe 99 come back "casting spells." Near or above one, the two answers show up with roughly equal frequency. Near zero, one answer dominates — exactly what our "same answer 97% of the time" constraint needs.

Now the design judgment. "What creature is Hagrid's pet?" genuinely demands the same answer every time — Fluffy is Fluffy — so it wants low temperature. But "what magic is used?" has many good answers, and you may neither need nor want identical responses, so a higher temperature is defensible. The choice is subtle; always reason from your use case, and check what other parameters your particular LLM exposes. And recall our constraint: 97% consistency all but forces a low temperature for this application.

09 — Roles & Their Temperatures

Each role wants its own setting

GenAI microarchitectures are built from specialized components, and how much creative freedom you grant the LLM usually depends on the role that component plays. Here are six common roles and the temperature each prefers. Notice the pattern: almost everything that judges, routes, guards, or acts wants determinism — only genuinely creative work wants heat.

Low temp

⚖️ Judge

Evaluates responses from multiple sources and decides which is better or true. Creativity is a liability when judging, so keep it cool and consistent.

Low temp

🚦 Router

Directs a request to the right specialist component using retrieved text. High temperature (or a shifting chunking strategy) could route the same prompt to different places and produce wildly different results — so stay deterministic.

Low temp

🎬 Actor

Performs operations and touches the outside world — calling an API, querying the vector DB, reformatting raw data into something human-readable. Actions should be deterministic.

Low temp

🛡️ Guard

Protects users by detecting and rejecting hallucinations, offensive language, and inappropriate content. A guard is not supposed to be creative — ever.

Low temp

🔍 Critic

Evaluates a response against defined criteria — accuracy, relevance, completeness. Careful, consistent judgment means low temperature.

High temp

✒️ Editor

Shortens, lengthens, or restyles text. Here creativity is the point — a higher temperature yields more diverse, more interesting output. The one role that welcomes the heat.

13Business

A loan-approval pipeline

The component deciding eligibility is a Judge — cold and repeatable, because "same inputs, same verdict" is a legal and ethical requirement, not a preference.

14Code

Intent classifier as Router

It reads a user message and routes to "billing" or "tech-support" agents. Low temperature keeps identical messages from randomly landing in different queues.

15Everyday

Editor as a brainstorm buddy

Asking a friend for "five different ways to phrase this toast" — you want them riffing, not repeating. That's high temperature, doing exactly its job.

10 — Running the Project

Three phases from start to ship

Good process minimizes dependencies between teammates and maximizes their productivity. A GenAI project breaks cleanly into three phases: understand the product and gather test data, tune the parameters, then put on the finishing touches before release.

Phase One

Project initiation — gather your inputs

  • From a deep understanding of your users, write a list of test prompts.
  • Identify the microarchitectures you'll use.
  • Identify the documents for your collection — in big organizations they're scattered across email servers, document management systems, and local drives.
  • Identify the document types (spreadsheets, PDFs, and so on) you'll ingest.
  • Pick a chunking strategy by examining a random sample; thousands of documents may need a more complex strategy that groups them by type, source, size, content, and topic.
  • Identify and document UX factors — acceptable response times, complexity requirements, SLAs.
  • Build a document-classification scheme by type, source, topic, age — using automated AI classification models if needed.
  • Create a repository of representative sample documents, with metadata for source, creation date, and author.
  • Identify the right temperature settings for your sample prompts.
Phase Two

Intermediate goals — build the pipeline

  • Build an ETL pipeline (Extract, Transform, Load) that pulls documents from the source, chunks them, and ingests the chunks into the vector database. This is a crucial step — and the book builds an advanced version of it in later chapters.
  • Use the pipeline to ingest the documents and run your sample prompts.
  • Iterate: ingest, query the vector database, evaluate the results, and re-test the chunking strategy after each pass. This loop is where most of the real tuning happens.
Phase Three

Crossing the finish line — verify before you ship

  • Confirm that, given your microarchitecture, the prompts return the responses you want — use an LLM to generate test inputs if that helps.
  • If you have multiple components, test each one independently (an LLM can help analyze the responses).
  • Ask the hard questions: is document classification working? Are the document groupings independent enough that each group deserves its own collection in the vector database?
16Business

Why Phase One pays off

Spending a week writing 120 good test prompts feels slow — until Phase Two, when every tuning decision is backed by real measurements instead of vibes.

17Code

The ETL pipeline is the backbone

Extract from source → Transform via chunk + embed → Load into the vector DB. The same shape as classic data engineering — your existing skills transfer directly.

◆ Carry these forward
  1. GenAI is a tuning game, not a coding game. Fifteen interdependent parameters, billions of combinations — strategy beats brute force.
  2. The dependency graph is your map. An arrow A → B means changing A forces a change to B. Knowing the arrows cuts tuning time by orders of magnitude.
  3. Group the knobs into three layers — ingestion, retrieval, generation — so a team can divide work without pulling against each other.
  4. Chunking has no single right answer. Whole-book and by-chapter blow up on cost and memory; tiny chunks lose context; ~600-token intelligent chunks with a top-k of 5 win — a well-sized net cast in a few good spots.
  5. Temperature trades consistency for creativity. Low for repeatable facts, high for variety. Match it to the use case — and to constraints like "97% the same."
  6. Roles dictate temperature. Judges, routers, actors, guards, and critics stay cold; only the editor gets to be creative.
  7. Run it in three phases: gather inputs, build-and-tune the ETL pipeline, then verify every component before you ship.
11 — Check Yourself

Four quick gut-checks

Answer each in your head, then reveal it. If all four land, you can walk into a real tuning session with a plan.

You switch your similarity function from cosine to dot-product. What else probably needs attention?
Max chunks returned (top-k). Different similarity methods produce different score distributions, so the cutoff that gave you the best results before will likely need re-tuning. That's Chain 2 of the dependency graph in action.
Why is "treat the whole book as one chunk" such a bad fit for the Harry Potter app?
A 250+ page book can exceed the LLM or embedding model's token limit, cause memory errors, add latency, bloat every request once conversation history is included, and run up a large per-token bill. It only makes sense for tiny documents.
A teammate wants high temperature on the component that routes requests to specialists. Good idea?
No. A router should be deterministic — high temperature (or a shifting chunking strategy) can send identical prompts to different components and produce very different downstream results. Routers, like judges, guards, actors, and critics, want low temperature.
Why bother writing ground-truth answers and test prompts before tuning anything?
Because tuning is iterative change-and-test, and you can't measure "better" without a target to compare against. Ground truth turns subjective impressions into measurable progress — which is exactly what Phase One sets up for Phase Two.
