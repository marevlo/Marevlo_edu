02 / Why Humans At All?
When Code & Judges Aren't Enough
By Modules 3.1–3.4 you have automated graders, judge models, and CI gates. Why still spend money on humans?
REASON 01 · CALIBRATION

LLM judges drift

Judges have biases (length, position, self-preference). Periodic human gold sets anchor them. Without anchoring, your eval drifts and you don't notice.

REASON 02 · UNCAPTURED CRITERIA

Tone, framing, dignity

"Is this answer respectful to a frustrated customer?" — code can't grade that. Often neither can a judge LLM reliably.

REASON 03 · NEW FAILURE MODES

Failures you didn't predict

Automated metrics check what you thought to check. Humans surface the failure modes you hadn't imagined — and those become the next CI tests.

REASON 04 · REGULATION & AUDIT

Compliance demands it

In healthcare, legal, finance, education — auditors want human judgment on file. Automated-only eval is not defensible.

REASON 05 · PREFERENCE DATA

Training fuel

Pairwise human preferences power DPO/RLHF rewards. If you ever want to fine-tune your agent on its own performance, you'll need this data.

REASON 06 · COMMUNICATION

Trust outside engineering

"96% of users rated this 4+/5" carries weight with PMs, execs, customers. "F1 0.86" doesn't.

The Layered Stack — a healthy eval stack has all three layers
Hybrid

Don't ask "code or judge or human?" — ask "what's the right ratio?" The cheap layers cover the volume; the expensive layer calibrates the cheap ones.

LAYER 01 · 90% OF VOLUME

Automated scorers — task completion, tool F1, latency, ASR. Run on every commit.

LAYER 02 · 9% OF VOLUME

LLM-as-judge — trajectory quality, evidence, tone. Run nightly on a 200-sample stratified subset.

LAYER 03 · 1% OF VOLUME

Humans — calibration set + new failure discovery + regulatory audit. Weekly to monthly.

03 / Annotation Task Types
Five Annotation Task Types
Each task has a sweet spot. Pick wrong and you'll burn budget on data that doesn't answer your question.
TASK 01

Direct Rating (Likert)

Show one trajectory; rate 1–5 on a rubric (helpfulness, safety, accuracy).

strong: per-axis insight, easy to aggregate
weak: central-tendency & anchoring bias
cost: low · 30-60s per item
TASK 02

Pairwise A/B

Show two trajectories side-by-side; pick A, B, or tie.

strong: resists rating-scale drift; less biased
weak: position bias; needs counterbalancing
cost: medium · 60-90s per pair
TASK 03

Error Labeling

Annotators select error categories from a fixed taxonomy (hallucination, tool misuse, scope drift…).

strong: diagnostic — tells you what to fix
weak: taxonomy maintenance overhead
cost: medium · 90s-3min per item
TASK 04

Free-Form Critique

"What's wrong with this trajectory? Be specific." Open-text annotation.

strong: surfaces unknown failure modes
weak: hard to aggregate; needs coding pass
cost: high · 3-8min per item
TASK 05

Edit-then-Rate

Annotator edits the agent's answer to be "correct"; rates effort required.

strong: generates training-grade gold data
weak: expensive; expert-level only
cost: very high · 5-15min per item
SELECTION GUIDE

Which one when?

benchmark a release → pairwise
multi-axis quality → likert
root-cause regression → error labeling
discover new failures → free-form
build SFT/DPO data → edit-then-rate

PRINCIPLE 01

Anchor every scale

Number on each endpoint isn't enough. Write the rubric in plain English so 3 ≠ "I'm not sure".

PRINCIPLE 02

One axis per question

"Helpfulness + accuracy" collapses into noise. Force a separate Likert per axis you care about.

PRINCIPLE 03

Counterbalance order

For pairwise, randomize which trajectory is "A". For Likert sets, randomize question order. Defeats position bias.

PRINCIPLE 04

Calibration items

Seed every 10th task with a gold-labeled item. Annotators who fail repeatedly get filtered.

PRINCIPLE 05

Require justification

A short text box per task triples annotation thoughtfulness and gives you debug material later.

PRINCIPLE 06

Cap per-session items

Quality drops sharply after ~40 minutes. Force breaks. Better to use 5 annotators × 30 items than 1 × 150.

METRIC 01

Cohen's κ

Two annotators · categorical labels. The default for binary or small-K classification.

use: 2 raters · 2-5 classes
METRIC 02

Fleiss' κ

3+ annotators · categorical. Use when annotator identity varies per item.

use: many raters per item
METRIC 03

Krippendorff's α

Most general — handles missing data, any scale (nominal/ordinal/interval), any # of raters.

use: real-world messy data
METRIC 04

ICC

Continuous / Likert ratings. Several flavours (ICC(2,k) is the workhorse).

use: Likert 1-5 averaging
Acceptance Gates — when is agreement "good enough"?
Thresholds
REJECT · κ < 0.4
Rubric is broken or annotators are guessing. Stop, re-train, re-write rubric. Do not publish.
CAUTION · 0.4 ≤ κ < 0.6
Useful for trend-spotting; not for headline numbers. Push toward 0.6+ before quoting.
PUBLISH · κ ≥ 0.6
Substantial agreement. Quote the number, with the κ value and CI alongside.

Five Bias Traps

POSITION BIAS
Annotators favor "A". Counterbalance side every pair.
LENGTH BIAS
Longer answer "feels thorough". Track length confound separately.
VERBOSITY BIAS
Confident phrasing wins even when wrong. Calibration items help.
ANCHOR/PRIMING
Reading A first colors B. Allow free re-reading; randomize order.
ANNOTATOR FATIGUE
Last 10% of items show lower IAA. Cap session length.
07 / Who Annotates?
Crowd vs. Domain Experts
Two pools, two cost curves, two quality profiles. Most production teams use both — for different parts of the eval pipeline.

The Hybrid Recipe — the production pattern
Blended

In practice you blend both pools. Experts define the rubric and gold; crowd workers scale annotation; experts re-check disagreement.

STEP 01 · EXPERTS
Write rubric + label 50-item gold set. Establish target κ.
STEP 02 · CROWD
Onboard with gold; filter annotators who fail. Run bulk task.
STEP 03 · CROWD ×3
3+ crowd workers per item. Compute IAA on overlapping items.
STEP 04 · EXPERTS
Adjudicate items where crowd disagreed. Update rubric.

# Detect a difference in two win-rates. Most common pairwise case.

from statsmodels.stats.proportion import samplesize_proportions_2indep_onetail
from statsmodels.stats.power import NormalIndPower

def n_for_winrate_diff(p_main=0.50, p_candidate=0.55,
                        alpha=0.05, power=0.80):
    effect = proportion_effect_size(p_main, p_candidate)
    analysis = NormalIndPower()
    return int(analysis.solve_power(
        effect_size=effect, alpha=alpha, power=power))

n_for_winrate_diff(0.50, 0.55)   # ≈ 770 per arm
n_for_winrate_diff(0.50, 0.60)   # ≈ 195
n_for_winrate_diff(0.50, 0.70)   # ≈ 50
# Estimate a mean rating to a given precision (±half-width).

import math
from scipy import stats

def n_for_mean_ci(sigma, half_width, conf=0.95):
    z = stats.norm.ppf((1 + conf) / 2)
    return math.ceil((z * sigma / half_width) ** 2)

n_for_mean_ci(sigma=1.0, half_width=0.1)    # ≈ 385
n_for_mean_ci(sigma=1.0, half_width=0.2)    # ≈ 97
# Pilot-then-scale: run a small batch, measure variance, compute real N.

def pilot_then_scale(pilot_data, target_half_width=0.05):
    sigma      = np.std(pilot_data, ddof=1)
    n_needed   = n_for_mean_ci(sigma, target_half_width)
    n_pilot    = len(pilot_data)
    remaining  = max(0, n_needed - n_pilot)
    return {"n_total": n_needed,
            "n_pilot": n_pilot,
            "n_remaining": remaining,
            "observed_sigma": round(sigma, 3)}

# Always pilot first — assumed σ is usually wrong.
RULE 01

Pilot first, always

Run 30-50 items, measure σ, then compute the full-scale N. Assumed variance is almost always wrong.

RULE 02

Stratify your sample

If your traffic has segments (intents, customer tiers), stratify. Otherwise the head crowds out the tails.

RULE 03

N per arm, not total

If you A/B two systems, you need N per arm. Double it for "total annotations needed".

RULE 04

Multiple raters per item

3 raters per item is the typical baseline. Pay for redundancy on the items that matter — gold + edge cases.

RULE 05

Pre-register your test

Decide success criteria before seeing data. "We needed p<0.05" after the fact is p-hacking.

RULE 06

Report CIs, not just means

"Win rate 55% [95% CI 50.7–59.3%]" is honest. "Win rate 55%" is a lie of omission.

09 / Recap
Key Takeaways
TAKEAWAY 01

A healthy eval stack runs three layers — automated scorers (90%), LLM judges (9%), humans (1%). The expensive layer calibrates the cheap ones.

TAKEAWAY 02

Pick the task type to match the question: pairwise for benchmarking, Likert for multi-axis, error labeling for diagnostics, free-form for discovery.

TAKEAWAY 03

The UI is the experiment — anchor every scale, one axis per question, counterbalance order, seed calibration items.

TAKEAWAY 04

Report agreement before any number. Use Cohen's κ, Fleiss' κ, Krippendorff's α, or ICC as the data type demands. κ < 0.4 → throw it out.

TAKEAWAY 05

Pairwise studies need counterbalanced order, length controls, and CI reporting — five common biases will eat your signal otherwise.

TAKEAWAY 06

Use crowd for volume, experts for calibration and adjudication. The hybrid recipe is the production pattern.

TAKEAWAY 07

Compute sample size before annotating. N ≈ 385 for ±5% on a binary rate; N ≈ 770 per arm to detect a 5pp pairwise gap.

TAKEAWAY 08

Pre-register your hypotheses, pilot first, report CIs. Skipping any of the three turns human eval back into anecdote.
