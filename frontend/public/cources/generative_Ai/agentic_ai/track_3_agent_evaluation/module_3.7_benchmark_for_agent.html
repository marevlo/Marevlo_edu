02 / Why Public Benchmarks
Why Benchmarks Still Matter
You already have automated metrics, judges, and humans. Why also pay attention to public suites?
REASON 01 · COMPARABILITY

Apples-to-apples

"Our agent hits 71% on SWE-bench Verified" is comparable across teams and time. Custom metrics aren't.

REASON 02 · ANTI-OVERFITTING

Catches narrow optimization

If your internal numbers climb but public numbers don't, you've over-fit your prompts to your own test set.

REASON 03 · COMMUNICATION

Outside-the-team trust

Sales, leadership, customers know SWE-bench. They don't know your internal F1. A public number unlocks doors.

REASON 04 · MODEL SELECTION

Pick your base model

When choosing between models, public benchmarks give you a first-pass ranking before paying for full re-runs.

REASON 05 · REGRESSION DETECTION

A second opinion

If your internal numbers stay flat but public ones drop, something fundamental in your stack regressed.

CAVEAT

Never only benchmarks

Public suites generalize; your business doesn't. Always pair with a stratified domain holdout — covered in §08.

The Layered Eval Pyramid
Public + Domain

Public + Domain · Both Are Required

Public benchmarks tell you where you stand against the field. Domain benchmarks tell you whether you're shippable. Skipping either layer is a failure mode.

LAYER 01 · PUBLIC AGENT SUITES

SWE-bench, AgentBench, WebArena, GAIA, τ-bench, ToolBench. Run quarterly.

LAYER 02 · CAPABILITY MICRO

MT-Bench, HumanEval, MBPP. Cheap, fast, run weekly.

LAYER 03 · DOMAIN HOLDOUT

Your own stratified set, frozen + versioned. Run every commit.

04 / Multi-Turn Benchmarks
MT-Bench & Multi-Turn Suites
If your agent talks for more than one turn, you need a multi-turn benchmark. MT-Bench is the canonical reference — flawed, but everyone reports it.
Canonical · MT-Bench
De-facto standard

80 Curated Two-Turn Dialogues

8 categories (writing, reasoning, math, coding, extraction, STEM, humanities, roleplay). Two turns each. Judged by a strong LLM on a 1-10 scale.

TOTAL
80
prompts × 2 turns
CATEGORIES
8
writing → roleplay
SCORING
1-10

FLAVORS
2
single + pairwise modes
STRENGTHS
Cheap to run (80 items). Industry-comparable. Decent multi-turn coverage. Categorized so you can spot per-skill weaknesses.
WEAKNESSES
Tiny (80). Judge-position bias unless mitigated. Heavily contaminated in modern model training data. Saturated at the top end.
VARIANT 01

MT-Bench-Plus / Hard

Tougher revisions targeting saturated categories. Use when frontier models all score 9+ on vanilla MT-Bench.

VARIANT 02

Chatbot Arena

Crowdsourced pairwise battles + Elo ratings. Slow but cheating-resistant: the data is freshly user-generated.

VARIANT 03

MT-Bench-101

Larger, more-turns extension with per-turn category labels. Useful when turn 1 → turn N drift matters.

CAVEAT

Use it as a smoke test

Don't ship a release on MT-Bench alone. Use it as the cheap, weekly canary; reserve τ-bench / domain set for the real go/no-go.

CAVEAT

Judge != ground truth

MT-Bench scores are LLM-judge scores. Validate against humans (Module 3.5) before quoting an MT-Bench number to leadership.

FOR AGENTS

Pair with τ-bench

MT-Bench tests chat. τ-bench tests chat + tools + policy. Agents need both signals.

05 / Code Benchmarks
HumanEval / MBPP for Code Agents
If you're building a code agent, SWE-bench is the goal — but HumanEval and MBPP are the cheap, fast micro-benchmarks you run before paying for the big one.
MICRO 01

HumanEval

164 hand-written Python problems with hidden tests. Pass@1 is the canonical score. Small, fast, saturated for frontier models.

SIZE164 problems SCOREpass@1, pass@10 RUNTIMEminutes USEfirst sanity check
MICRO 02

MBPP / MBPP+

~1,000 crowd-sourced Python tasks. MBPP+ adds extra test cases that catch shallow solutions. Slightly harder than HumanEval.

SIZE974 (MBPP) / + variants SCOREpass@1 RUNTIME~10 min USEstratified breadth check
MICRO 03

LiveCodeBench / BigCodeBench

Curated to reduce contamination — fresh problems collected over time. Use when contamination concerns are high for HumanEval/MBPP.

SIZE~500 (rolling) SCOREpass@1, contamination-aware RUNTIME~30 min USEwhen SWE-bench too slow
pass@k Explained
Metric choice

Pick Your k With Intent

pass@k is "the model gets k attempts; at least one must pass". A frontier model at pass@10 ≈ 95% may be at pass@1 ≈ 70%. Match the metric to your deployment reality.

pass@1 · PRODUCTION-LIKE
Single shot. The honest number. Always report this — it's what users experience.
pass@10 · RETRY-FRIENDLY
Realistic when your stack has retries or pick-best-of-N. Inflates over pass@1.
pass@100 · MARKETING
Almost meaningless. Use only when you have an oracle to pick the correct attempt.

A Sensible Code-Agent Eval Pipeline

PHASE 01

HumanEval

Every commit. Minutes to run. Catches basic regressions instantly.

PHASE 02

MBPP+ / LiveCodeBench

Nightly. Broader signal, anti-contamination.

PHASE 03

SWE-bench Lite

Pre-release. 300 issues, hours to run, real codebases.

PHASE 04

SWE-bench Verified

Quarterly. The headline number. Compare to industry leaderboards.

07 / Build Your Domain Eval Set
Your Domain Eval Set
Public suites tell you where you stand. Your domain set tells you whether to ship. Seven steps to a holdout that won't get gamed.
STEP 01

Stratify

List the intents, customer tiers, locales, edge cases. Allocate sample count per stratum proportional to business impact, not traffic alone.

STEP 02

Seed

Mine 50-100 real production queries per stratum from logs (with PII redacted). These are your gold seed.

STEP 03

Scale

Use an LLM to generate paraphrases + edge cases. Human-review-and-cull to keep quality high.

STEP 04

Label

Attach expected end-state + gold tool path per case (Module 3.1). Two-annotator agreement; resolve disagreements.

STEP 05

Version

Tag the dataset (v1.0, v1.1, …). Every commit gets a dataset version stamped alongside its eval results.

STEP 06

Freeze + Hide

Split into dev (visible) + test (hidden). Test never enters prompts or training. Period.

STEP 07

Rotate

Every quarter, retire 10-20% of test items and replace with fresh production samples. Defeats long-term overfitting.

★

The 80/20

A 200-item stratified set, well-labeled, used consistently → better signal than 2,000 unstructured ones. Quality > quantity.

# Eval case — the unit of your domain set.

from pydantic import BaseModel
from typing import Any

class EvalCase(BaseModel):
    id:                 str           # stable, e.g. "case_0142"
    stratum:            str           # "refund.duplicate" · "account.login"
    query:              str
    initial_state:      dict[str, Any]
    expected_end_state: dict[str, Any]
    gold_tool_path:     list[str]
    tags:               list[str]    # "vip", "edge", "ood", ...
    created_at:         str
    source:             str           # "prod_log" · "synthetic" · "expert"
# Run domain eval. Stratified rollup. Versioned output.

def run_domain_eval(agent, dataset):
    rows = []
    for case in dataset.cases:
        traj = agent.run(case.query, state=case.initial_state)
        row  = full_agent_eval(traj,
                                case.expected_end_state,
                                case.gold_tool_path)
        row["stratum"]    = case.stratum
        row["dataset_v"] = dataset.version
        rows.append(row)
    return rollup_by(rows, "stratum")   # per-stratum + overall
# Dataset versioning — every commit gets stamped.

class EvalDataset:
    def __init__(self, path):
        self.cases   = load_cases(path)
        self.version = f"{path.stem}@{sha256(self.cases)[:8]}"
        self.test_hidden  = load_cases(path.parent / "test_hidden.json")

    def verify_no_leak(self, prompts):
        # Defensive: ensure test cases never appear in prompts
        for case in self.test_hidden:
            assert case.query not in prompts, f"leak: {case.id}"

# Stamp version on every eval row.  Mismatch in CI = config error.
08 / Benchmark Pitfalls
Six Ways Benchmarks Lie To You
Every published number conceals at least one of these. Train your eye to spot them before quoting a leaderboard.
PITFALL 01

Training-set contamination

Older benchmarks (HumanEval, MBPP, MT-Bench) have leaked into pretraining data. High scores measure memorization, not capability. Cross-check with LiveCodeBench / fresh subsets.
PITFALL 02

Saturation

When frontier models all score 95+, the metric stops discriminating. Move to harder variants (SWE-bench Verified, MT-Bench-Hard) — or your domain set.
PITFALL 03

Subset cherry-pick

"71% on SWE-bench" can mean Lite (300 easy issues) or Verified (~500 cleaned). Always force the subset name + version into the report.
PITFALL 04

pass@k inflation

pass@10 looks better than pass@1. If your prod deployment doesn't do 10 retries with an oracle, pass@10 is marketing — not measurement.
PITFALL 05

Scaffolding-as-score

"Model X hits 65%" — but only with that team's custom agent scaffolding. The model alone may be 20pp lower. Always report scaffold version.
PITFALL 06

Distribution mismatch

Your agent does customer support. WebArena tests shopping. A WebArena score tells you nothing about your shippability — but it photographs well in slides.
Checklist — Before Quoting Any Number
Audit

Six Questions To Ask

① Which exact subset and version of the benchmark?
② pass@1 or pass@k? What k?
③ What scaffolding ran around the model?
④ Is the benchmark plausibly in the training data?
⑤ Does the benchmark distribution resemble production?
⑥ Are we comparing against contemporary frontier baselines?
09 / Download Summary
Key Takeaways
TAKEAWAY 01

Six canonical agent suites: SWE-bench, AgentBench, WebArena, GAIA, τ-bench, ToolBench. Each maps to one agent surface.

TAKEAWAY 02

For multi-turn, MT-Bench is the cheap canary, τ-bench is the production benchmark, and Chatbot Arena is the contamination-resistant alternative.

TAKEAWAY 03

For code agents, run a layered pipeline — HumanEval (every commit) → MBPP+ (nightly) → SWE-bench Lite (pre-release) → SWE-bench Verified (quarterly).

TAKEAWAY 04

Always report pass@1 alongside any pass@k. pass@1 is the production-honest number.

TAKEAWAY 05

Build a stratified domain set in 7 steps: stratify, seed, scale, label, version, freeze, rotate.

TAKEAWAY 06

Allocate test items by business impact, not traffic. A 1% adversarial slice may deserve more eval pressure than the 40% routine slice.

TAKEAWAY 07

Hide and rotate a portion of the test set. Without rotation, even your private set eventually overfits.

TAKEAWAY 08

Ask the six audit questions before quoting any benchmark number — subset, pass@k, scaffolding, contamination, distribution, baselines.
