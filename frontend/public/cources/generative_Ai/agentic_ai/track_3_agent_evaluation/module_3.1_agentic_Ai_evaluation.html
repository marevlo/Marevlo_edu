02 / Metric Architecture
The Seven Pillars of Agent Eval
Every agent eval suite, no matter the framework, decomposes into these seven measurement families. Module 3.1 builds the first five. Modules 3.3 and 3.4 handle the last two.
PILLAR 01

Task Completion

Did the world end in the expected state? Binary outcome over end-state validators.

key: completion_rate, goal_match
PILLAR 02

Tool Accuracy

Did the agent pick the right tools and call them with valid arguments?

key: tool_F1, arg_validity, hallucinated_tool_rate
PILLAR 03

Efficiency

Shortest path vs loops. Steps vs gold. Token waste per trajectory.

key: efficiency, loop_count, step_budget_overrun
PILLAR 04

Trajectory Score

Structural + LLM-judge fusion. One number summarizing whole-path quality.

key: trajectory_quality, edit_distance, judge_score
PILLAR 05

Evidence Binding

Every claim in the final answer must trace back to an observation. No floating assertions.

key: citation_coverage, ungrounded_claim_rate
PILLAR 06

Boundary Enforcement

Did the agent stay inside its allowed tool set, domain, and PII rules?

key: out_of_bounds, scope_drift  → Module 3.3
PILLAR 07

Cost & Latency

Per-trajectory tokens, dollars, wall-clock. p50/p95/p99 distributions.

key: cost_per_traj, p99_latency  → Module 3.4

03 / Pillar 01 Deep Dive
Task Completion
The most important pillar. Get this wrong and every other metric is decorative.
FLAVOR A

Programmatic Validator

Run code against the world state. Fastest, cheapest, deterministic.

def validate(db, case):
    return db.orders[case.id].status == "refunded"
FLAVOR B

Goal-State Subset

Check that all required keys/values are present — extras allowed. Useful for partially-specified goals.

def deep_subset(need, got):
    return all(got.get(k) == v for k,v in need.items())
FLAVOR C

LLM Judge

Open-ended goals (research, summary). Score with rubric. Slow, biased, but flexible. → Module 3.6.

def judge(goal, final):
    return llm.grade(rubric, goal, final)
Partial Credit and the Trap of Pass/Fail
The hard part

Real tasks are compound. "Refund the order, notify the customer, log to audit" is three sub-goals. Binary pass/fail throws away most of your signal.

✗ Naive 
def score(case):
    return 1 if all_subgoals_done(case) else 0
# Loses: did 2/3? did 3/3 but bad email? did nothing?
✓ Weighted subgoals 
weights = {"refund": 0.6, "notify": 0.3, "audit": 0.1}
def score(case):
    return sum(w for g,w in weights.items() if done(case,g))
RULE
Make completion verifiable before you make it fancy.
RULE
Weight sub-goals by business impact, not difficulty.
RULE
Reserve LLM judge for what code truly cannot check.

SUB-METRIC 01

Tool Selection F1

The headline. Set-based F1 over expected vs actual tool names.

SUB-METRIC 02

Argument Validity

% of tool calls whose arguments parse and pass schema validation.

SUB-METRIC 03

Hallucinated Tools

Tool names the agent invented that don't exist in the registry. Should be 0%.

SUB-METRIC 04

Order Sensitivity

For dependent tools, was crm_lookup called before refund? Edit-distance over ordered sequence.

Production-grade tool accuracy scorer
Code
# Production-grade tool accuracy scorer.

def tool_accuracy(trajectory, gold_calls, registry):
    actual   = [s.tool for s in trajectory.steps]
    expected = [c.tool for c in gold_calls]

    tp = len(set(actual) & set(expected))
    precision = tp / max(len(set(actual)),   1)
    recall    = tp / max(len(set(expected)), 1)
    f1 = 2*precision*recall / max(precision+recall, 1e-9)

    arg_valid     = mean([registry[s.tool].validate(s.args) for s in trajectory.steps if s.tool in registry])
    hallucinated  = [s.tool for s in trajectory.steps if s.tool not in registry]
    order_dist   = levenshtein(actual, expected)

    return {
        "tool_f1":           round(f1, 3),
        "arg_validity":      round(arg_valid, 3),
        "hallucinated_tools": hallucinated,
        "order_edit_dist":    order_dist,
    }

METHOD 01

Structural

Pure code. Counts loops, redundant calls, length vs gold. Cheap, deterministic, blind to semantics.

→ length, loops, edit_distance, redundancy_rate
METHOD 02

LLM Judge

Ask another model: "Rate this trajectory on quality / coherence / efficiency 1–5." Catches semantics, but slow + biased.

→ Module 3.6 covers judges for agents
METHOD 03

Fused Score

Weighted blend: 0.4·structural + 0.4·judge + 0.2·cost. The shape every production team converges to.

→ tune weights on a held-out set
06 / Pillar 05 + 06
Evidence Binding & Boundary Enforcement
Every claim grounded in an observation. Every action inside the allowed perimeter. Two checks that catch the most embarrassing failure modes.
Anchor Every Claim
Evidence binding

An agent's final answer should look like a research paper — every factual claim cites the observation that produced it. Untraceable claims are hallucinations waiting to be discovered.

SAMPLE AGENT ANSWER (annotated)

Your order ORD-7711 obs:2 ↗ is currently delayed by 3 days due to a carrier issue obs:4 ↗. As a gold-tier customer obs:2 ↗, you qualify for an automatic $120 refund obs:5 ↗, which I have just issued obs:5 ↗. The refund should appear in your account within 5 business days policy:?.

CITATION COVERAGE
5 / 6 claims
UNGROUNDED CLAIMS
1 (5-day SLA)
EVIDENCE SCORE
0.83
// Detection

Use an LLM judge or NLI classifier to map each claim sentence to candidate observations. Anything that maps to no observation is flagged as ungrounded. Target: ≥ 0.95 coverage on factual claims.

Stay Inside the Perimeter
Boundary enforcement · Module 3.3

The agent has an allowed tool set, an allowed domain, allowed customers, allowed amounts. Every step is a chance to break those rules. Boundary checks turn each one into a measurable event. Deep dive: Module 3.3.

ALLOWED · 04
tools = {crm_lookup, shipping_status, refund, email_user}
ALLOWED · DOMAIN
customer support · orders · refunds
DENIED · ACTIONS
delete_user, change_password, refund > $500
def check_boundaries(trajectory, policy):
    violations = []
    for step in trajectory.steps:
        if step.tool not in policy.allowed_tools:
            violations.append(("unauthorized_tool", step))
        if step.tool == "refund" and step.args["amount"] > policy.max_refund:
            violations.append(("refund_limit", step))
        if domain_classifier(step) not in policy.allowed_domains:
            violations.append(("scope_drift", step))
    return {
        "out_of_bounds_count": len(violations),
        "violations":          violations,
    }
07 / Testing Non-Determinism
pytest for Non-Deterministic Systems
Agents drift. Assertions that demand exact equality break overnight. The fix: run N times, assert distributions.
✗ Breaks in CI by Friday 
def test_refund_flow():
    result = agent.run("refund order 7711")
    assert result.steps == ["crm_lookup", "shipping_status", "refund"]
    assert result.cost == 0.04
# Every prompt edit, model update, or temperature flicker → red CI.
✓ Robust distributional test 
@pytest.mark.parametrize("seed", range(10))
def test_refund_flow(seed):
    r = agent.run("refund order 7711", seed=seed)
    record(r)

def test_distribution():
    assert pass_rate() >= 0.9      # 9-of-10 successes
    assert p95_cost() <= 0.10      # dollars per traj
    assert p95_steps() <= 6         # step budget

PATTERN 01

k-of-n Pass-Rate

Re-run each case N times. Pass iff success on ≥ k. Tunes how flaky you tolerate.

PATTERN 02

Percentile Budgets

Assert p95(cost) ≤ ceiling instead of mean. The long tail is what bills you.

PATTERN 03

Snapshot Diffs

Compare regression run to baseline. Flag if any metric drops > X% with statistical significance.

PATTERN 04

Seeded Replays

For reproducibility in debugging: pin seed + record trajectories. Replay shows exact divergence point.

08 / Public Benchmarks
Agent Benchmark Suites
Public datasets keep your eval honest. Map your agent's domain to the right benchmark — and read the leaderboard's fine print.
SWE-bench CODE

2,294 real GitHub issues from 12 Python repos. Agent must produce a patch that passes the maintainers' tests. Brutal — frontier models score ~50% on Verified.

use when: coding agents, SWE workflows, repo-level tasks
WebArena WEB

Fully reproducible web environments (shopping, GitLab clone, Reddit clone, CMS). 812 tasks. Agent navigates and acts on real-looking UIs.

use when: browser agents, web automation, RPA
AgentBench GENERAL

8 environments: OS, DB, KG, card game, lateral-thinking puzzles, web shopping, web browsing, household. Broad coverage of agent capabilities.

use when: baseline LLM-as-agent comparisons
GAIA REAL-WORLD

466 questions humans answer in < 5 min but agents struggle with. Tests retrieval, multi-modal, reasoning, and tool composition under realistic ambiguity.

use when: generalist assistants, research agents
τ-bench (tau-bench) DIALOGUE

Customer support sims with database-backed users. Tests multi-turn tool use, policy compliance, and reliability across re-runs (pass^k).

use when: support agents, dialogue + tools
ToolBench TOOL USE

16K+ real REST APIs across 49 categories. Tests breadth of tool selection and ability to chain unfamiliar APIs from documentation.

use when: API-heavy agents, tool-selection at scale
Match Benchmark to Use Case

09 / Download Summary
Key Takeaways
TAKEAWAY 01

Agent eval decomposes into seven pillars — five live in this module, two are deferred (3.3 for safety, 3.4 for cost).

TAKEAWAY 02

Task completion is the most important metric. Make it verifiable with code first; reserve LLM judges for what code truly can't check.

TAKEAWAY 03

Tool accuracy = F1 + arg validity + zero hallucinated tools + order sensitivity. One number is not enough.

TAKEAWAY 04

Efficiency is graded against a gold path — count loops, redundancy, and edit distance. Fused trajectory scores blend structural + judge + cost.

TAKEAWAY 05

Every claim in the answer must cite an observation. Ungrounded claims = hallucinations. Target citation coverage ≥ 0.95.

TAKEAWAY 06

pytest for agents asserts k-of-n pass rates and percentile budgets — never single-run equality.

TAKEAWAY 07

Always pair public benchmarks (SWE-bench, WebArena, τ-bench, GAIA) with a domain holdout. Public sets eventually get gamed.

TAKEAWAY 08

A production eval row carries all 7 pillars at once — every module that follows just unpacks one column in more depth.
