03 / Start From Symptoms
Eight Production Symptoms
Eval doesn't start in a notebook — it starts when something looks wrong. These are the eight signals real teams encounter, and where to look first.
SYMPTOM 01

"Wrong answers"

Users / QA report incorrect final outputs.

→ go to: Module 3.1 (task completion, evidence binding)
SYMPTOM 02

"Wrong tool calls"

Agent chooses wrong tool / wrong args / loops.

→ go to: Modules 3.1 (tool F1), 3.6 (tool-call judge)
SYMPTOM 03

"Long, wandering runs"

Step counts exploding; loops; redundant calls.

→ go to: Modules 3.1 (efficiency), 3.4 (budgets)
SYMPTOM 04

"Bill is rising"

Provider invoice grows faster than traffic.

→ go to: Module 3.4 (percentiles, 8 levers, unit economics)
SYMPTOM 05

"Jailbreaks / PII leaks"

Security flags; bug reports; audit demands.

→ go to: Module 3.3 (ASR, TPR/FPR, red-team CI)
SYMPTOM 06

"Sub-agents step on each other"

Wrong routing; specialist drift; bad handoffs.

→ go to: Module 3.2 (router CM, isolation, synthesis)
SYMPTOM 07

"PM says 'feels worse'"

Subjective quality drop; metrics still green.

→ go to: Modules 3.5 (pairwise), 3.6 (trajectory judge)
SYMPTOM 08

"How do we compare to peers?"

Need an external reference number.

→ go to: Module 3.7 (benchmark suites + domain set)

Master Rule

06 / The 4-Layer Eval Stack
Code · Judges · Humans · Benchmarks
Each layer covers what the layer below can't, at a cost the layer above can't sustain. The ratios matter as much as the layers.
LAYER 01 · ~90%

Code Scorers

Deterministic checks: task completion, tool F1, step budgets, cost percentiles, PII regex, safety filters.

cost: ~$0
latency: ms
runs: every commit
LAYER 02 · ~9%

LLM Judges

Trajectory · tool-call · multi-turn · pairwise judges. Score what code can't quantify; calibrated to humans via κ.

cost: ~$0.05/run
latency: seconds
runs: nightly / stratified
LAYER 03 · ~1%

Human Eval

Gold set, judge calibration, regulatory audits. The ground truth that anchors layers 1-2 and catches what neither sees.

cost: ~$1-10/item
latency: days
runs: weekly / monthly
LAYER 04 · QUARTERLY

Benchmarks

Public suites (τ-bench, SWE-bench, etc.) + your domain holdout. External reference and protection against quiet drift.

cost: variable
latency: hours
runs: per release / quarterly
Are Your Ratios Right?
Health Check

Mismatched ratios are the #1 indicator of an unhealthy eval program. Diagnose against these patterns:

100% humans, 0% code
You'll burn through your eval budget in a week. Push toward code-first; reserve humans for calibration.
100% code, 0% humans
Cheap and confident — but you'll never notice when your judges drift or your scorers miss the point. Add a small human gold set.
100% judge, 0% benchmark
Your model and judge are training each other into a corner. Add an external benchmark to anchor reality.
~90% code · ~9% judge · ~1% human · quarterly bench
The healthy default. Adjust per safety/regulatory tier — high-stakes domains push the human share to 3-5%.
07 / The 90-Day Roadmap
Zero to CI-Blocking Eval · 90 Days
Whether you have "we have logs" or "we have a notebook", this is the standard ramp. Six phases. Each phase produces an artifact a stakeholder can see.
WEEK 1-2

Phase 01 · Instrument

Emit structured trajectories from every production run. Fields: goal, steps, tool calls, observations, final answer, model, cost, latency.

→ artifact: trajectory schema + 1k logged runs
WEEK 3-4

Phase 02 · Code Layer

Build deterministic scorers: task completion, tool F1, step budgets, PII regex, basic safety filters. Wire into CI.

→ artifact: CI runs eval suite on PRs
WEEK 5-6

Phase 03 · Cost & Safety Gates

Add cost percentiles (p50/p95/p99), step ceilings, ASR floors, regression-vs-main checks. Hard-block merges on regression.

→ artifact: deploys can be blocked by eval
WEEK 7-8

Phase 04 · Human Gold

Build a 200-500 trajectory gold set with two human annotators per item. Compute IAA. Anchor every judge against it.

→ artifact: versioned gold set + κ ≥ 0.6
WEEK 9-10

Phase 05 · LLM Judges

Build trajectory judge and tool-call judge. Validate against gold. Roll out nightly on stratified samples. Apply bias mitigations.

→ artifact: nightly judge dashboard
WEEK 11-12

Phase 06 · Benchmarks

Run the relevant public suite (τ-bench / AgentBench / SWE-bench / WebArena). Lock in a 200-item versioned domain holdout. Quarterly rotation.

→ artifact: leaderboard you defend in reviews
Where Are You Now?
Maturity Rubric

If your honest answer is L1 or L2, your eval is theatre. Aim for L4 in 90 days; L5 within a year.

L1 · VIBES
"Looks good to me on a few runs." No structured eval.
L2 · NOTEBOOK
Manually-run notebook. No CI. No gold set. Numbers stale within a day.
L3 · CI BASIC
Code scorers run on PRs. No judges. No human anchor.
L4 · 3-LAYER
Code + judges + small human gold. Judges κ-validated. Cost/safety gates block.
L5 · FULL STACK
4 layers + public benchmark + domain holdout. Maturity defended to auditors.
08 / Eight Anti-Patterns
Eight Ways Eval Lies to You
Each of these is a failure mode in your eval program, not in your agent. Spot them in your own team first.
ANTI-PATTERN 01

"50 examples"

Hand-picked dev set. Confidence intervals impossible.

→ fix: use 3.5 sample-size math
ANTI-PATTERN 02

Train = test

Prompted/tuned on the same set you score on.

→ fix: held-out domain set (3.7)
ANTI-PATTERN 03

Mean-only reporting

Means hide tails; tails are where the money and pain live.

→ fix: p50/p95/p99 (3.4)
ANTI-PATTERN 04

Self-judging

Same model family as agent and judge. Self-preference bias guarantees inflation.

→ fix: cross-family judge (3.6)
ANTI-PATTERN 05

Unvalidated judge

"The judge says 4.3" — but never compared to human gold. Number is meaningless.

→ fix: κ ≥ 0.6 before quoting (3.6)
ANTI-PATTERN 06

Single-axis pass/fail

"Pass = correct answer." Ignores tools, evidence, safety, cost.

→ fix: 7 pillars (3.1)
ANTI-PATTERN 07

Benchmark cherry-pick

Quoting only the suite you score well on.

→ fix: select per problem (3.7)
ANTI-PATTERN 08

Eval that doesn't block

Numbers in a dashboard; PRs merge regardless.

→ fix: pytest-mark + CI gate (3.3, 3.4)
09 / Recap
Capstone Takeaways
TAKEAWAY 01

Eval starts from a symptom, not a metric. Route the symptom to the right module — the rest is execution.

TAKEAWAY 02

The master decision tree collapses Track 3 into seven gates: correctness → cost → safety → quality, single vs multi, internal vs external.

TAKEAWAY 03

The lookup matrix gives you metric, tool, and sample size in one row for every common question — bookmark it.

TAKEAWAY 04

Run all four layers — code (~90%), judges (~9%), humans (~1%), benchmarks (quarterly). Mismatched ratios are the #1 program-level red flag.

TAKEAWAY 05

Master rule: code wins ties. Whenever a deterministic check exists, it overrides any judge that disagrees.

TAKEAWAY 06

The 90-day roadmap ramps from zero to CI-blocking eval in six phases: instrument → code → gates → human gold → judges → benchmarks.

TAKEAWAY 07

Use the L1–L5 maturity rubric to grade your own program honestly. L1/L2 = theatre. L4 in 90 days, L5 within a year.

TAKEAWAY 08

Audit yourself against the 8 anti-patterns — every one is a way smart teams convince themselves they have eval when they don't.
