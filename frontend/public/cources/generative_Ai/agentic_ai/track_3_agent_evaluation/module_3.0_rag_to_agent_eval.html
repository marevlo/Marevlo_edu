03 / The Fundamental Shift
Why It Is Fundamentally Different
RAG outputs one string. Agents output a graph. The unit of evaluation changes — and so does almost everything else.

UNIT OF EVAL: the trajectory — a DAG of thoughts, tools, observations.

PATH: different on every run, even for identical inputs.

DECISIONS: every step is a choice the model made.

The Six Fundamental Shifts

SHIFT 01

Output dimensionality

RAG = 1 string. Agent = N steps × M tools × variable depth. Eval surface explodes.

SHIFT 02

Non-determinism is structural

Even at temperature=0, agent paths drift. The same query can take 3, 5, or 11 steps.

SHIFT 03

Cost dispersion

RAG cost ≈ flat. Agent cost = long-tailed. p99 can be 50× the median. Budget per trajectory, not per call.

SHIFT 04

New failure modes

Loops, wrong tool, hallucinated tool, premature termination, off-domain action — none exist in RAG eval.

SHIFT 05

Safety surface widens

Agents take actions. Prompt injection, sandbox escape, PII exfiltration become live concerns — not theoretical ones.

SHIFT 06

Tests must accept variance

"Pass = 100%" breaks. pytest for agents asserts distributions: pass-rate ≥ k of n, p95 cost ≤ X.

04 / The Four New Questions
Four Questions Agent Eval Must Answer
RAG eval answers one: "Was the answer correct and grounded?" Agent eval adds four. Memorize them — every module in this track points back to one of these.
01
SUCCESS

Did the agent complete the task?

Forget the final answer string. Did the agent accomplish the goal — did the refund get issued, did the PR get merged, did the meeting get scheduled?

METRIC FAMILY → task completion rate, end-state validators, goal-state matching
Example: success = db.orders[id].status == "refunded"
02
CHOICES

Did it pick the right tools?

The agent might succeed by accident — or fail by picking delete_user() when it needed deactivate_user(). Tool accuracy measures decision quality, not outcome.

METRIC FAMILY → tool precision/recall, tool-F1, argument validity
Example: expected {crm_lookup, refund}, actual {crm_lookup, email, refund} → recall=1.0, precision=0.67
03
PATH

Was the trajectory efficient?

Two agents finish the same task — one in 3 steps, one in 17 with two loops and a re-try. The second one is a cost bomb in production. Trajectory scoring catches this.

METRIC FAMILY → trajectory length, loop count, edit-distance vs gold path, redundancy
Example: steps=17, gold_steps=3, loops=2 → efficiency = 3/17 ≈ 0.18
04
SAFETY × COST

Did it stay in bounds — and within budget?

Did it leak PII? Try a prompt-injected tool? Burn $4 in tokens? Boundary enforcement and per-trajectory economics are first-class metrics now.

METRIC FAMILY → ASR, TPR/FPR, PII leakage rate, $/trajectory, p95 latency
Example: ASR (Attack Success Rate) over 200 injection prompts → 3/200 = 1.5%
The Memorable Line

RAG eval asks "Is the answer good?" · Agent eval asks "Did it succeed," "did it choose well," "did it take the right path," and "did it stay safe and cheap?"

Choose Your Lenses (You Need Both)

BEHAVIORAL

ASKS
Did the agent reach the right end-state?
SIGNAL
Strong, simple, falsifiable.
BLIND SPOT
Misses cost, loops, bad-but-recoverable choices.
USE WHEN
Closed-domain tasks with verifiable goals (refunds, bookings, code patches).

TRAJECTORY-BASED

ASKS
Was every step justified? Was the path efficient?
SIGNAL
Rich, dense, expensive to label.
BLIND SPOT
"Gold path" bias — penalizes valid alternative routes.
USE WHEN
Open-ended tasks, debugging regressions, cost optimization, safety review.

# RAG-style eval — what you already know (Course 01 / Track 6).
# Input: question + retrieved context + answer.  Output: a number.

from ragas.metrics import faithfulness, answer_relevance, context_recall

def score_rag(sample):
    return {
        "faithfulness":      faithfulness(sample),
        "answer_relevance":  answer_relevance(sample),
        "context_recall":    context_recall(sample),
    }

# The unit of evaluation is a single (question, context, answer) tuple.
# Path is invisible — there is no path.
# Behavioral eval — only the end-state matters.
# Detailed in Module 3.1.

def score_behavior(trajectory, expected_end_state):
    actual = trajectory.final_world_state
    return {
        "task_completed": actual == expected_end_state,
        "goal_match":     deep_subset(expected_end_state, actual),
        "answer_correct": judge_answer(trajectory.final_answer),
    }

# Steps, tools, cost, latency — all invisible.
# Cheap to compute. Strong signal. But path quality is hidden.
# Trajectory eval — judges the path the agent took.
# Detailed in Modules 3.1 (single-agent) and 3.2 (multi-agent).

def score_trajectory(trajectory, gold_path):
    tools_used     = [s.tool for s in trajectory.steps]
    tools_expected = [s.tool for s in gold_path]

    return {
        "tool_f1":        f1(set(tools_used), set(tools_expected)),
        "path_length":    len(trajectory.steps),
        "efficiency":     len(gold_path) / max(len(trajectory.steps), 1),
        "loop_count":     count_repeats(tools_used),
        "edit_distance":  levenshtein(tools_used, tools_expected),
        "arg_validity":   mean([validate_args(s) for s in trajectory.steps]),
    }

# Captures everything behavioral eval misses.  Expensive — needs a gold path.
# Production agents need both, plus cost and safety.
# This is the shape of every eval row Track 3 will build out.

def full_agent_eval(trajectory, expected_end_state, gold_path):
    return {
        # Q1 — Success
        **score_behavior(trajectory, expected_end_state),
        # Q2 + Q3 — Choices and Path
        **score_trajectory(trajectory, gold_path),
        # Q4 — Cost and safety
        "tokens":        trajectory.total_tokens,
        "cost_usd":      trajectory.cost_usd,
        "latency_ms":    trajectory.latency_ms,
        "pii_leaked":    detect_pii(trajectory),
        "injection_hit": detect_injection(trajectory),
        "out_of_bounds": check_boundaries(trajectory),
    }

08 / Download Summary
Key Takeaways
TAKEAWAY 01

The unit of eval changes: from single string to full trajectory DAG. Every other shift cascades from this one.

TAKEAWAY 02

Agent eval answers four new questions: success, tool choice, path efficiency, and safety×cost. RAG eval answers one.

TAKEAWAY 03

Use behavioral eval when you have a verifiable end-state. Use trajectory eval when path quality, cost, or debugging matters. Most production agents need both.

TAKEAWAY 04

From RAG eval, three things carry forward (LLM-judge mechanics, pairwise, IAA), three get re-framed, and six are entirely new.

TAKEAWAY 05

Non-determinism is structural — pytest for agents asserts distributions (k-of-n pass-rate, p95 cost), not exact equality.

TAKEAWAY 06

Annotation cost 5–20× higher than RAG. Plan your sample sizes accordingly — Module 3.5 covers the math.
