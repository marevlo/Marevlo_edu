Multi-Agent Evaluation.

One agent failed? Quantifiable. Five agents collaborating failed? The bug could be in any one of them — or in the router, the handoff, or the synthesis. This module teaches you where to look.

01 / Objective Matrix
Mission Parameters
Six outcomes. By the end you can debug a 5-agent system where the bug is hiding three handoffs deep.
OBJ.01

Isolate Specialists

Evaluate each specialist agent in isolation, with mocked routers and handoffs — so failures attribute cleanly.

OBJ.02

Measure Routing Accuracy

Build a confusion matrix over router decisions. Compute per-specialist precision/recall — and detect class imbalance traps.

OBJ.03

Cross-Domain Tests

Probe handoff boundaries: queries that need two specialists, queries that look like one but require another.

OBJ.04

Score Synthesis Quality

When sub-agents return partials, the synthesizer assembles them. Score for completeness, contradiction handling, and citation preservation.

OBJ.05

Debug Nested Traces

Read parent-child trace trees with confidence. Identify which agent (and which step inside it) caused a regression.

OBJ.06

Enforce Domain Boundaries

Catch scope drift — when a specialist answers outside its allowed domain — before it reaches the user.

02 / Architecture Under Test
Where Eval Has To Happen
A typical multi-agent system has at least six measurable surfaces. Miss one and your bug hides exactly where you didn't look.
Router → Specialists → Synthesizer
Reference Topology

Six numbered eval surfaces. The reference topology routes intent from a router, fans out to specialists, and merges in a synthesizer.

① Router accuracy ② Specialist isolation ③ Handoff fidelity ④ Synthesis quality ⑤ Boundary enforcement ⑥ Whole-system trajectory
①ROUTER ACCURACY
% queries sent to the correct specialist (multi-class classification).
②SPECIALIST ISOLATION
Per-specialist eval (3.1 pillars) with mocked router + handoff inputs.
③HANDOFF FIDELITY
Did the receiving specialist get the context it needed? No info loss across the wire.
④SYNTHESIS QUALITY
Are partials assembled correctly? Contradictions resolved? Citations preserved?
⑤DOMAIN BOUNDARY
Did billing just answer a tech question? Scope drift kills trust.
⑥WHOLE-SYSTEM
End-to-end task completion + cost. The user-facing number.

A Single End-to-End Score Tells You Nothing

If whole-system completion drops from 88% → 71% overnight, was it the router? A specialist? Synthesis? Without per-layer metrics, you're guessing.

WITH LAYERED EVAL

Router accuracy: 94% → 67% · specialists steady. ⇒ Diagnosis in 30 seconds.

WITHOUT IT

All you see: "completion dropped 17%". ⇒ Bisect prompts, models, and graphs for two days.

RULE OF THUMB

Every handoff and every router decision deserves its own metric. Cost: small. Debug savings: enormous.

03 / Isolation Testing
Specialist Isolation
Treat each specialist as a unit under test. Mock the router. Mock the handoff. Apply the 3.1 pillars. Now the failure has a single owner.
REASON 01

Attribution

A bug in the router can corrupt every downstream signal. Isolation gives you ground-truth for "is THIS agent broken?"

REASON 02

Independent shipping

Each specialist can be deployed on its own cadence — only with isolated eval can you certify a change before merge.

REASON 03

Reusable specialists

Same specialist in two different orchestrations? Isolation eval is portable; system eval isn't.

# Isolated specialist eval — feed it the input it WOULD have
# received from the router, with all upstream variance removed.

from dataclasses import dataclass

@dataclass
class IsolatedCase:
    specialist:        str
    routed_query:      str          # exactly what router would send
    handoff_context:   dict         # prior agents' partial state
    expected_end_state: dict
    gold_tool_path:    list

def eval_specialist(specialist, cases):
    rows = []
    for c in cases:
        traj = specialist.run(c.routed_query, context=c.handoff_context)
        rows.append(full_agent_eval(traj, c.expected_end_state, c.gold_tool_path))
    return aggregate(rows)   # reuses Module 3.1 scorers
# Mock the upstream so the specialist sees a *deterministic* input.
# Without this, you can never tell if the router or the specialist drifted.

class MockRouter:
    def __init__(self, target_specialist, payload):
        self.target  = target_specialist
        self.payload = payload

    def route(self, query):
        return self.target, self.payload

class MockHandoff:
    def __init__(self, context):
        self.ctx = context

    def get(self):
        return self.ctx

# In tests: swap real router/handoff for these mocks.
# Specialist sees IDENTICAL inputs across runs.

Six Router Failure Modes

FAILURE 01 · Confident misroute
High-confidence routing to the wrong specialist. The user gets a confident wrong answer. Worst kind.
FAILURE 02 · Boundary ambiguity
"My billing question is failing to load." Tech OR billing? Needs explicit disambiguation pattern.
FAILURE 03 · OOD acceptance
Off-topic query routed to a specialist instead of being rejected. Watch the minority class F1.
FAILURE 04 · Routing oscillation
Specialist hands back; router re-routes elsewhere; loops. Cap with a max-handback budget.
FAILURE 05 · Multi-intent split miss
Query has two intents; router picks one and drops the other. Test cross-domain cases explicitly.
FAILURE 06 · Stale routing rules
New specialist added but router prompt not updated. CI test should fail when a specialist is unreachable.
05 / Cross-Domain Testing
Cross-Domain Probes
A query is rarely pure. "I can't log in to pay my invoice" needs account + billing. Cross-domain test cases stress the handoff machinery the way no in-class eval can.

Pattern A — Compound intent

A + B

"My subscription failed to renew and now I can't log in."

expected billing → account, sequential handoff
common fail router picks one, ignores the other

Pattern B — Disguised intent

B (looks like A)

"My billing page won't load." Sounds like billing — is actually a frontend bug.

expected tech specialist
common fail billing answers and hallucinates an invoice

Pattern C — Mid-task pivot

A → B mid-conversation

User starts asking billing questions, then switches to a tech issue inside the same session.

expected router re-routes with prior context
common fail billing tries to handle the tech question

Pattern D — Out-of-scope probe

OOD

"What's the stock price of NVDA?" — outside the entire system's allowed domain.

expected router rejects gracefully
common fail any specialist tries to answer

Mix the Cases — Don't Just Stack the Easy Ones

A balanced cross-domain eval set is 50% in-class, 30% cross-domain, 15% disguised, 5% OOD. Skewed sets give you false confidence.

Synthesis Quality
Specialists return partials. The synthesizer turns them into one coherent answer. This step has its own failure modes — and they show up nowhere else.

Synthesis-Specific Failure Modes

FAILURE 01 · Partial drop
Specialist B's output got truncated or summarized away. User sees an incomplete answer.
FAILURE 02 · Contradiction blend
A says X, B says ¬X. Synthesizer averages them into incoherence instead of flagging the conflict.
FAILURE 03 · Citation loss
Specialist gave clean obs:N citations; synthesizer dropped them. Evidence-binding score collapses.
FAILURE 04 · Re-hallucination
Synthesizer adds facts that no specialist provided. The merge step becomes a new hallucination surface.
07 / Debug Surface
Nested Trace Debugging
When a 5-agent run fails, you need a parent-child tree, not a flat log. Every observability stack (LangSmith, Langfuse, Arize, OpenTelemetry) produces something like this.
trace_id = t_72ab9f · failed run · Read the Tree from the Top
14 steps · $0.41 · 22.4s · FAIL
▼ router · classify_intent  // 240ms · $0.002
  thought: "Mention of 'invoice' + 'log in' → compound; bias billing"
  ⚠ confidence 0.61 · routed → billing  // account intent dropped

▼ billing · resolve_subscription  // 6 steps · $0.18
  ① crm_lookup(uid="U-9214")  → {tier:"gold", payment_method:"card_xxx"}
  ② invoice_status(uid="U-9214")  → {last:"failed", reason:"card_decline"}
  ③ reset_password(uid="U-9214")  // OUT OF DOMAIN — billing should not call this
  ④ email_user(...)
  ⑤ final_answer(...)  → "We reset your password and updated your invoice"

▼ account · NEVER INVOKED  // router never handed off; should have

▼ synthesizer · assemble  // 3.1s · $0.22
  contradiction unresolved: invoice "failed" + answer claims "updated"
  citation coverage 0.4  // 2 of 5 claims grounded
  final → "Your password was reset and your invoice is updated."

▼ verdict: FAIL · root cause = router (dropped account intent) → cascaded scope drift in billing
// READING THIS TRACE: Start at the leaf failure (synth contradiction). Walk up. The billing specialist made a forbidden call (reset_password) — but only because the router never handed off to account. Root cause = router, even though the visible symptom is in synthesis.

Six Debugging Principles

PRINCIPLE 01

Stable trace_id end-to-end

Propagate one ID across router, specialists, and synthesizer. Without it, joining logs is guesswork.

PRINCIPLE 02

Parent_id on every span

Tools spawned by specialists, specialists spawned by router. The tree only works if every node points up.

PRINCIPLE 03

Inputs + outputs verbatim

Log what the agent received and what it emitted at each boundary. Summaries here cost you the ability to replay.

PRINCIPLE 04

Walk up from the leaf failure

The visible symptom is rarely the root. Read bottom-to-top. Most multi-agent bugs are upstream of where they manifest.

PRINCIPLE 05

Tag span-level metrics

Attach tool_f1, boundary_violation, conf directly on spans. Filtering becomes one query.

PRINCIPLE 06

Replay from any node

Capture seeds + state. Being able to re-run just the failing specialist with its exact input is the fastest debug loop.

08 / Failure Taxonomy
Multi-Agent Failure Modes
Eight failure shapes that only exist when more than one agent is in the system. Memorize these — they're the regression targets for every CI run.
F.01 · Misroute
Router picks the wrong specialist.
F.02 · Scope drift
Specialist answers outside its domain.
F.03 · Handoff loss
Receiver missing context the sender had.
F.04 · Oscillation
Specialists bounce a query between each other.
F.05 · Partial drop
Synthesizer ignores one specialist's output.
F.06 · Contradiction blend
Conflicting partials averaged into nonsense.
F.07 · OOD acceptance
Any specialist answers an out-of-domain query.
F.08 · Cost cascade
Each agent's tokens × N specialists = budget explosion.

Catch Scope Drift Before The User Does

Each specialist gets a declared scope. Before its output reaches the synthesizer, an enforcer validates that the answer stayed inside. This is the multi-agent version of Module 3.1's boundary check.

BILLING · ALLOWED
refunds · invoices · subscriptions · payment methods
TECH · ALLOWED
device · network · app errors · setup
CROSS · DENIED
billing → reset_password, tech → issue_refund, anyone → delete_account
enforce_domain — block scope drift before the synthesizer sees it
Boundary enforcement
def enforce_domain(specialist_id, trajectory, policy):
    scope      = policy.scope_of[specialist_id]
    violations = []
    for step in trajectory.steps:
        if step.tool not in scope.allowed_tools:
            violations.append(("unauthorized_tool", step))
    if domain_classifier(trajectory.final_answer) not in scope.topics:
        violations.append(("scope_drift", trajectory.final_answer))
    return violations

# Block the response or hand back to router if violations > 0.
09 / Recap
Key Takeaways
TAKEAWAY 01

Multi-agent eval has six measurable surfaces: router, isolated specialists, handoff, synthesis, boundaries, and whole-system. Cover all six.

TAKEAWAY 02

Isolate specialists with mocked routers and handoffs. Without isolation, every failure has at least three possible owners.

TAKEAWAY 03

Routing is multi-class classification. Build a confusion matrix and always inspect minority classes — that's where OOD lives.

TAKEAWAY 04

Cross-domain test sets need a balanced 50/30/15/5 mix: in-class, compound, disguised, and out-of-domain. Skewed sets mislead.

TAKEAWAY 05

Synthesis fails in its own ways — partial drop, contradiction blend, citation loss, re-hallucination. Score with a 6-axis rubric.

TAKEAWAY 06

Nested traces with stable trace_id + parent_id + verbatim I/O let you walk a failure from leaf to root in minutes.

TAKEAWAY 07

Eight multi-agent failure modes exist that single-agent eval cannot detect: misroute, scope drift, handoff loss, oscillation, partial drop, contradiction blend, OOD acceptance, cost cascade.

TAKEAWAY 08

Domain boundary enforcement runs per specialist, before the synthesizer sees the output. Catches scope drift while it's still cheap to fix.
