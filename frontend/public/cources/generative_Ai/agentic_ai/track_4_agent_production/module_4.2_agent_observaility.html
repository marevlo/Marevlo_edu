Trace = trajectory, not call

In RAG one trace ≈ one LLM call. In agents one trace = N nodes × M tool calls × retries. Same tool, new mental model.

03 / Trace Anatomy
What an Agent Trace Looks Like
Five span types. Seven standard tags. One tree per trajectory. Get this schema right and 80% of debugging questions answer themselves.
GRAPH

graph_run

Root span. One per trajectory. Carries thread_id and final verdict.

NODE

graph_node

One per node execution. Holds state-in and state-out for replay.

LLM

llm_call

Per model invocation. Prompt, response, tokens, cost, model id.

TOOL

tool_call

Per tool invocation. Args (with PII scrubbed), observation, latency, success.

RETRY

retry / error

Auto-emitted on retry or failure. Attaches the parent that re-ran.

Tag Every Trace With These
Seven Standard Tags

These are the only fields you actually filter by in production. Make them required at instrumentation time — not an afterthought.

tenant_id
Org / workspace owning the run. Multi-tenant slicing.
user_id
End user. Hashed in regulated domains.
thread_id
From the checkpointer (Module 4.1). Joins traces to conversations.
agent_version
Graph + prompt version. Critical for A/B (Module 4.5).
env
dev / staging / prod / shadow.
intent / task
Domain label — refund, onboarding, etc. Powers per-intent dashboards.
release_id
Deploy version. Lets you bisect regressions to a commit.

# Scrub PII before traces leave the boundary. Module 3.3 detector.

from langsmith.run_helpers import trace

PII_FIELDS = {"email", "phone", "ssn", "credit_card"}

def scrub(payload):
    if isinstance(payload, dict):
        return {k: ("[REDACTED]" if k in PII_FIELDS else scrub(v))
                for k, v in payload.items()}
    return payload

# Wrap an arbitrary tool call so its trace doesn't leak PII.
def traced_tool(name, fn):
    def wrapped(**kwargs):
        with trace(name=name, inputs=scrub(kwargs)) as run:
            out = fn(**kwargs)
            run.end(outputs=scrub(out))
            return out
    return wrapped

Diagnostic

Billing and tech overlap from 1.10s → 1.94s — they ran in parallel after the router fanned out. The synth waited on the slowest specialist. Without correlation, you'd see four unrelated traces and miss the parallelism.

# Pass trace_id explicitly when a parent agent invokes a child.

def call_specialist(name, payload, parent_run):
    child_config = {
        "run_id":        uuid4(),
        "parent_run_id": parent_run.id,
        "trace_id":      parent_run.trace_id,   # SAME trace_id
        "tags":          [f"specialist:{name}"],
        "metadata":      {**parent_run.metadata, "specialist": name},
    }
    return specialists[name].invoke(payload, child_config)

# Result: all four agents show up under one trace tree in LangSmith,
# ordered chronologically with proper parent links.
# When sub-agents are different HTTP services, propagate via headers.

from opentelemetry.propagate import inject, extract
import httpx

# Client side — outgoing HTTP carries the W3C traceparent header.
def call_specialist_http(url, payload):
    headers = {}
    inject(headers)                       # adds traceparent + baggage
    return httpx.post(url, json=payload, headers=headers)

# Server side — extract the parent context and continue the trace.
@app.post("/run")
async def run(request):
    ctx = extract(request.headers)
    with tracer.start_as_current_span("specialist_run", context=ctx):
        ...
# Baggage — domain attributes that travel with every span downstream.

from opentelemetry import baggage, context as otel_ctx

def enrich_baggage(tenant_id, user_id, thread_id):
    ctx = otel_ctx.get_current()
    ctx = baggage.set_baggage("tenant_id", tenant_id, ctx)
    ctx = baggage.set_baggage("user_id",   user_id,   ctx)
    ctx = baggage.set_baggage("thread_id", thread_id, ctx)
    otel_ctx.attach(ctx)

# Every downstream span (sub-agent, tool, LLM) inherits these tags.
# Powerful for slicing: "show me all spans for tenant=acme-001 today".
06 / Replay Debugging
Replay Debugging for Agents
A bad production run isn't a mystery — it's a replayable experiment. Re-execute any node from its checkpoint, mock external calls, change the prompt, repeat.
// REPLAY · run=tr_88f0… · node=act · iteration 2
Step through and re-run from any checkpoint
▶ at step 6 · steps 1-5 cached · step 7+ to re-run
01 plan · state_out={plan: ["lookup", "compare", "refund"]} [cached] 0.52s
02 act · crm_lookup → {tier:"gold", charges:[…]} [cached] 0.31s
03 act · compare_charges → identical [cached] 0.18s
04 llm_call · decide_refund → "refund INV-1102" [cached] 0.34s
05 act · refund → 503 upstream timeout [cached] 0.41s
06 ▶ CURSOR · retry decision · re-running with patched prompt —
07 refund (attempt 2) · mock → 200 ok pending
08 llm_call · final_answer · re-run pending
Flow

Replay reuses cached outputs for steps 1-5, jumps the cursor to step 6, and re-runs forward with a patched prompt. External tool calls can be left live or mocked — your choice per debug session.

# Pull a production trace and load its checkpoint history.

from langsmith import Client

ls = Client()
run = ls.read_run("tr_88f0...")

print(run.inputs)
print(run.outputs)
print(run.metadata["thread_id"])

# Use thread_id to pull state history from the checkpointer.
cfg = {"configurable": {"thread_id": run.metadata["thread_id"]}}
history = list(graph.get_state_history(cfg))
# Re-run from any past checkpoint with a patched config.

target = history[5]                       # the state BEFORE the failing step
patched_cfg = {
    **target.config,
    "configurable": {
        **target.config["configurable"],
        "prompt_version": "v2-strict",           # swap the prompt
    },
}
graph.invoke(None, patched_cfg)                  # resume forward

# Diff: compare original outputs vs replayed outputs side-by-side.
# Mock external tools for safe debugging — no real refunds during replay.

from contextlib import contextmanager

@contextmanager
def mocked(overrides):
    orig = {}
    for name, fake in overrides.items():
        orig[name] = tools[name]
        tools[name] = fake
    try:
        yield
    finally:
        tools.update(orig)

with mocked({"refund": lambda **_: {"status": "ok", "refund_id": "mock_1"}}):
    graph.invoke(None, patched_cfg)

Reproduce a bug

Patch & verify

Edit the prompt or graph; replay the same run; diff outputs; ship the patch with confidence.

Build eval datasets

Tag interesting runs and add them to a regression set. Track 3 evals run against real traces, not synthetic ones.

Replay the actual failed runs from the outage window. Show stakeholders what the agent did, not what you guessed.

RULE 01

Tile per question

Every tile must answer a single question a human asks (e.g. "is it down?", "is it slower?"). Decorative tiles get cut.

RULE 02

Slice by intent

Overall numbers hide intent-specific regressions. The dashboard always has an "intent" filter.

RULE 03

Compare vs baseline

Every number shows delta vs 7d / 30d. Static numbers don't trigger action.

RULE 04

Drill-through everywhere

Every chart click should land on a filtered trace list. The dashboard is a lens, not an island.

RULE 05

Cost is a first-class KPI

p50 + p99 cost sit next to latency. Module 3.4 already established why.

RULE 06

Version-aware

Overlay agent_version markers on time series so deploys are visible at a glance.

08 / Alerting
Six Production Alert Rules
Page only on these. Everything else goes to a Slack channel. Alert fatigue is the #1 killer of an observability program.
01
ALERT 01 · PAGE

Success rate < SLO (5m)

Sustained drop below 95% for 5 minutes.

→ On-call · investigate latest deploy
02
ALERT 02 · PAGE

p99 cost ↑ 2× baseline

Sustained 2× spike in p99 cost for 10 minutes.

→ Likely loop or context bloat (Module 3.4)
03
ALERT 03 · PAGE

Tool error rate > 5%

Any single tool exceeding 5% error rate over 5 minutes.

→ Upstream issue or schema drift
04
ALERT 04 · SLACK

Step count p95 > cap

p95 step count breaches its ceiling for 15 minutes.

→ Loop precursor; investigate before paging
05
ALERT 05 · SLACK

Retry storm

Retry rate ↑ 5× baseline. Usually upstream pressure.

→ Coordinate with platform/SRE
06
ALERT 06 · SLACK

Trace volume drop

Ingest down > 30% vs same hour last week.

→ Often a tracer bug, not an agent bug
Four Ways to Make Alerts Useless
Alerting Anti-Patterns
AP 01 · ALERTING ON MEANS
Mean latency lies — p99 tells the truth. Alert on percentiles.
AP 02 · ALERT PER METRIC
15 separate pages for one incident. Group correlated metrics.
AP 03 · NO RUNBOOK LINK
Alert without a runbook = panic at 3am. Embed link in alert payload (Module 4.8).
AP 04 · 100% PAGES
Use the page/slack distinction. Page on customer-visible problems only.
09 / Recap
Key Takeaways
TAKEAWAY 01

LangSmith primitives all carry forward — the SDK doesn't change, but the unit of analysis becomes the trajectory, not the call.

TAKEAWAY 02

Five span types — graph · node · llm · tool · retry — plus seven standard tags are the schema. Get them right and dashboards write themselves.

TAKEAWAY 03

Trace trees show state-in / state-out, retries, and parallel branches — questions RAG-era traces couldn't answer at all.

TAKEAWAY 04

Cross-agent correlation requires propagating trace_id + W3C traceparent + baggage across sub-agent boundaries.

TAKEAWAY 05

Replay debugging — re-run from any checkpoint with patched prompts and mocked tools — is the single biggest MTTR win agents have over RAG.

TAKEAWAY 06

Eight dashboard KPIs cover the on-call view: throughput, success, p50/p95 latency, p50/p99 cost, step count, tool errors.

TAKEAWAY 07

Page on six things: success drop, p99 cost spike, tool error rate, step count breach, retry storm, trace volume drop. Everything else is Slack.

TAKEAWAY 08

Scrub PII at the trace boundary. Once data leaves your runtime it lives in another system's retention rules.
