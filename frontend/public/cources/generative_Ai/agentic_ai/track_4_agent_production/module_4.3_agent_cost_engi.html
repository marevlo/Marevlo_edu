03 / Model Routing
Model Routing
Most queries don't need the premium model. A routing layer picks the cheapest model that's still good enough — and bills the premium one only when you actually need it.
SIGNAL 01 PRIMARY

Intent / task class

A small classifier (or rule set) labels the query: simple lookup vs. multi-step reasoning vs. tool-heavy.

→ Haiku for FAQs, Opus for refunds
SIGNAL 02 SECONDARY

User tier / SLA

Paid tiers get the premium model by default; free tier hits the cheap tier and only escalates on failure.

→ Tier maps to a model budget
SIGNAL 03 SECONDARY

Confidence / retry

Cheap model emits a confidence score (or judge re-checks it). Low confidence → escalate to a stronger model.

→ Bridges into the cascade pattern

# Dispatcher: pick a model per query, fall through to escalation.

from dataclasses import dataclass

@dataclass
class RouteDecision:
    model:      str
    reason:     str
    tier:       int            # 1=cheap, 4=premium

def route(query, user, policy):
    label = classify(query)             # "faq" | "lookup" | "reason" | "tool_heavy"
    tier  = policy[label]["default_tier"]
    if user.sla == "premium":
        tier = max(tier, 3)               # enforce floor for paid users
    if contains_sensitive(query):
        tier = max(tier, 4)               # compliance always uses top tier
    return RouteDecision(
        model=policy["tiers"][tier]["model"],
        reason=f"label={label}, sla={user.sla}",
        tier=tier,
    )
# Tiny classifier — cheap, fast, ~50ms.

from anthropic import Anthropic
client = Anthropic()

CLASSIFY_PROMPT = """Classify the user query into ONE of:
- faq           : a static knowledge question
- lookup        : a single tool/DB lookup
- reason        : multi-step reasoning, no tools
- tool_heavy    : multi-step, needs 2+ tools

Return ONLY the label."""

def classify(query):
    r = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=8, temperature=0,
        system=CLASSIFY_PROMPT,
        messages=[{"role": "user", "content": query}],
    )
    return r.content[0].text.strip()

# A 50ms router that decides where to spend the next few dollars.

The cascade only "wins" if T1/T2 resolve a large share of traffic. Track resolved % per tier in your dashboard (Module 4.2) — if T4 starts handling 30% instead of 4%, your cascade is broken.

cascade.py — run the cascade
Code
# Run the cascade. Each tier returns (answer, confidence).

def run_cascade(query, tiers, min_confidence=0.6):
    for tier in tiers:
        try:
            ans, conf = tier.run(query)
        except ToolError:
            log.warn(f"tier={tier.name} failed; escalating")
            continue
        if conf >= min_confidence:
            emit("cascade.resolved", tier=tier.name, conf=conf)
            return ans
        log.info(f"tier={tier.name} low_conf={conf:.2f}; escalating")
    # Fallback: human handoff if even premium tier was uncertain.
    return escalate_to_human(query)

05 / Batch APIs
Batch APIs · 50% Off
Major LLM providers offer batch endpoints that process requests within hours (not seconds) at roughly half price. Any offline pipeline belongs here.
Sync API vs Batch API
Comparison
Dimension Sync API Batch API
latency seconds ~1-24 hours (typically < 6h)
price list ~50% off list (provider-dependent)
throughput RPS / rate-limited tens of thousands of jobs per submission
best for interactive agents, real-time UX backfills, eval suites, nightly summaries, embeddings
avoid for background pipelines (waste) anything user-facing in < 1 min

Five Batchable Workloads

WORKLOAD 01

Eval runs

Re-running your trajectory eval set against a new model/prompt. Nightly is fine.

WORKLOAD 02

LLM-as-judge sweeps

Scoring last week of production runs (Module 3.6). Result available in dashboard tomorrow.

WORKLOAD 03

Document enrichment

Summarizing or tagging a knowledge base for retrieval. Pure offline pipeline.

WORKLOAD 04

Embedding backfills

Computing embeddings for millions of records on schema changes. Always batch.

WORKLOAD 05

Distillation data

Generating SFT/DPO pairs from a premium teacher model to train a smaller student.

RULE

If hours are OK → batch

Default new offline workloads to the batch endpoint. Promote to sync only if a real-time use-case appears.

06 / Prompt Compression
Prompt Compression
Shave 30-60% off your prompt tokens with structured compression. Smaller prompt = smaller bill and smaller context window pressure.
STRATEGY 01 HIGH

Static prompt caching

System prompt + tool registry are constant. Provider caches them — you pay ~10% of normal rate.

→ already covered in caching recap
STRATEGY 02 HIGH

Schema-driven outputs

Force structured JSON output. Cuts CoT verbosity. Output tokens often cost 3-5× input.

→ Pydantic / response_format
STRATEGY 03 MID

Few-shot deduplication

Replace repeated few-shot examples with rule descriptions. Often equivalent quality at half the tokens.

→ measure on your eval set
STRATEGY 04 MID

LLM-based summarizer

Run a cheap model to summarize long inputs (docs, prior turns) before the expensive call.

→ Haiku as a pre-processor
STRATEGY 05 MID

Token-aware compressors

LLMLingua, Selective-Context — algorithmically drop low-information tokens while preserving semantics.

→ batch eval before adoption
STRATEGY 06 LOW

Trim observations

A web_search returning 5K tokens of HTML never enters the prompt — truncate at retrieval, not at generation.

→ Module 3.4 tool-result budget

07 / Context Pruning
Context Pruning Across Agent Steps
A 10-step trajectory drags every prior observation into step 11 by default. By step 15 you're paying premium model rates to re-read a stale CRM lookup. Prune as you go.
POLICY 01 HIGH

Rolling summary

Past observations older than k steps get summarized into a single line. Recent k stay verbatim.

→ Default for chat agents
POLICY 02 HIGH

Last-k window

Keep last k=4 observations verbatim, drop older entirely. Cheap, predictable, occasionally lossy.

→ When state is fully captured by recent context
POLICY 03 MID

Salient-fact extraction

Extract structured facts (user_id, tier, charges, decisions). Carry only the facts forward.

→ Loads memory store (Module 4.4)
POLICY 04 MID

Token budget enforcement

Hard ceiling on context size. Apply policies 01-03 until under the ceiling, then send.

→ Tightens Module 3.4 budgets in production
POLICY 05 LOW

Drop redundant observations

Same crm_lookup called twice with the same args → keep one observation. Hash-dedup.

→ Cheap to implement, narrow upside
STACK

Use 1 + 2 + 4 together

Rolling summary, last-k verbatim, hard ceiling. Three policies cover ~95% of agent context bloat in practice.

→ Default production config

08 / Circuit Breakers
Circuit Breakers for Runaway Agents
An agent that won't stop on its own will stop your wallet. Per-trajectory and per-fleet circuit breakers are the seatbelt your finance team didn't know to ask for.
LIMIT 01 PER-TRAJ

Max steps

Hard ceiling on graph iterations. 7 for short tasks, 15-20 for long ones. Loops never go past this.

→ exit cleanly to "needs help" state
LIMIT 02 PER-TRAJ

Max cost

Dollar ceiling per trajectory. p99 + 50% headroom. Trip = escalate to human or fail soft.

→ measure with Module 4.2 cost tags
LIMIT 03 PER-TRAJ

Max wall-clock

Cap total time per run. p99 latency + 2×. Long-tail trajectories die at the boundary.

→ Kubernetes / runtime enforced
LIMIT 04 PER-USER

Per-user daily cap

$/day per user-id. Catches abuse and runaway integrations early.

→ enforce in router or gateway
LIMIT 05 PER-TENANT

Tenant burn-rate breaker

Trip the tenant offline if hourly spend exceeds N× their plan. Manual review to re-enable.

→ saved one of our customers $40k once
LIMIT 06 PER-FLEET

Global kill switch

Single toggle to throttle or stop the agent fleet. Runbook entry for the on-call (Module 4.8).

→ feature flag + alert path

circuit_breaker.py

class TrajectoryBreaker:
    def __init__(self, max_steps, max_cost, max_seconds):
        self.max_steps   = max_steps
        self.max_cost    = max_cost
        self.max_seconds = max_seconds
        self.started     = time.time()
        self.cost        = 0.0
        self.steps       = 0

    def check(self, step_cost):
        self.cost  += step_cost
        self.steps += 1
        if self.cost  > self.max_cost:
            raise BreakerTripped("cost")
        if self.steps > self.max_steps:
            raise BreakerTripped("steps")
        if time.time() - self.started > self.max_seconds:
            raise BreakerTripped("timeout")
Fail-Soft, Not Fail-Hard

Tripped breakers should land users in a graceful state — "I need a moment, escalating to a human" — not a 500. Wire the trip events into Module 4.7's HITL workflow so the queue catches them.

10 / Recap
Key Takeaways
TAKEAWAY 01

Caching from RAG Track 7 stays on — but agents multiply LLM calls per task, so caching alone won't keep margins. Stack runtime levers on top.

TAKEAWAY 02

Route by intent class, SLA tier, and confidence. Most queries don't need the premium model — a 50ms classifier picks the cheapest one that's still good enough.

TAKEAWAY 03

The 4-tier cascade delivers 80%+ savings by resolving 60-80% of traffic at the cheapest tier before any escalation.

TAKEAWAY 04

Default offline workloads — evals, judges, embeddings, distillation — to the batch API. 50% off for accepting hours of latency.

TAKEAWAY 05

Prompt compression — caching, structured outputs, summarizers, observation trimming — routinely shaves 30-60% off prompt tokens.

TAKEAWAY 06

Context pruning across steps uses rolling summary + last-k + hard ceiling — three policies that cover 95% of long-trajectory bloat.

TAKEAWAY 07

Circuit breakers fire at six levels: max steps · max cost · max time per trajectory, plus per-user, per-tenant, and a global kill switch. Fail soft, not hard.

TAKEAWAY 08

Track cost per trajectory, per intent, per tenant, per version. Without attribution, anomalies become invoice surprises a month later.
