02 / Why Memory ≠ Checkpoints
Memory ≠ Checkpoints
In Module 4.1 we made trajectories durable. That's not enough for a persistent companion. Memory is the layer above checkpoints — it remembers across threads.
Checkpointer (4.1) vs Long-Term Memory (4.4)
Comparison
Dimension Checkpointer (4.1) Long-Term Memory (4.4) Why It Matters
scope one thread / trajectory across threads, sessions, time "the user told me last Tuesday"
lifetime until thread completes / TTL weeks to years (with retention policy) enables personalization
content graph state, raw messages facts, preferences, summaries, skills distilled, not raw
retrieval by thread_id (point lookup) semantic + filtered + recency-weighted retrieved into prompt at each turn
consistency deterministic snapshot eventually consistent, conflict-resolved two facts can disagree → reconcile
privacy scoped to one run subject to retention + erasure rights GDPR / data protection apply
03 / The Three Types
Episodic · Semantic · Procedural
A working taxonomy borrowed from cognitive science. Production systems use all three — sometimes in the same vendor, sometimes split across stores.
TYPE 01 · EPISODIC

Specific events

"I asked about INV-1102 last week and got a refund." Stores discrete events with timestamps, actors, and outcomes.

  • → Past trajectories, decisions, outcomes
  • → Highly time-indexed
  • → Best for: chat history, audit, "what happened when"
  • → Risk: unbounded growth without eviction
TYPE 02 · SEMANTIC

Stable facts

"User is on the gold tier, lives in Tokyo, prefers email over SMS." Stores derived, deduplicated, longer-lived facts.

  • → Preferences, identity attributes, relationships
  • → Distilled from many episodes
  • → Best for: personalization, routing decisions
  • → Risk: contradiction without conflict-resolution
TYPE 03 · PROCEDURAL

Learned skills

"When refunding gold-tier duplicates, always check INV before issuing." Stores patterns the agent applies as policy.

  • → Heuristics, playbooks, response templates
  • → Often co-edited with humans
  • → Best for: skill acquisition, organizational know-how
  • → Risk: governance — who edits, who approves
Same Conversation · Three Memory Writes
Worked Example

User: "Refund my Oct duplicate again — same as last month. And use my work card next time."

EPISODIC WRITE
event{ ts: now, actor: U-7281, type: "refund_issued", invoice: INV-1102, amount: 24, agent: refund@1.4.2 }
SEMANTIC WRITE
fact{ user: U-7281, key: "preferred_payment_method", value: "work_card", confidence: 0.9, source_event: ev_9381 }
PROCEDURAL WRITE (CANDIDATE)
skill_proposal{ trigger: "duplicate refund + gold tier", action: "verify INV-id before refund", needs_review: true }
The Rule

Episodic writes are automatic. Semantic writes need confidence + conflict-resolution. Procedural writes propose; humans approve. Different policies, different review chains.

04 / Vendors
Mem0 · Zep · LangMem · Roll-Own
Four serious options. None is universally right. The decision driver is usually privacy posture and how much memory governance you want to own.
VENDOR 01

Mem0

SDK-first, opinionated extraction of facts from conversation. Easy quickstart; conflict resolution built in.

✓ low friction
✓ deduplication / merging
✗ less control over schema
VENDOR 02

Zep

Temporal knowledge graph over conversation. Strong at when facts changed; entity-aware.

✓ rich temporal queries
✓ entity extraction
✗ heavier infra footprint
VENDOR 03

LangMem

LangChain-native. Tightest fit with LangGraph Platform store + checkpointer (Module 4.1).

✓ integrates with the platform
✓ same Postgres backend
✗ less mature than Mem0/Zep
VENDOR 04

Roll-Own (Postgres + pgvector)

A few tables, an embedding column, a tiny pipeline. Maximum control, regulator-friendly.

✓ schema is yours
✓ no vendor risk
✗ you maintain everything
When Each Is Right
Decision Frame
PICK MEM0
Fast time-to-value, conversational facts, you're OK with managed SaaS, no air-gap constraints.
PICK ZEP
You need time-aware facts ("when did this preference change?") and entity-graph queries.
PICK LANGMEM
You're already on LangGraph Platform (Module 4.1) and want one Postgres footprint.
ROLL YOUR OWN
Regulated industry, GDPR-strict, residency requirements, or you'd build the abstraction once and amortize it across products.

Reading It

Capture is on the write path (every checkpoint commit). Inject is on the read path (every agent step). Everything between is a background pipeline — usually a batch API job (Module 4.3) running every few minutes per user.

-- Three tables — one per memory type. Same store, separate semantics.

CREATE TABLE episodic_events (
    id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      text NOT NULL,
    ts           timestamptz NOT NULL,
    type         text NOT NULL,             -- "refund_issued", etc.
    payload      jsonb NOT NULL,
    embedding    vector(1536),
    salience     real DEFAULT 0.5,        -- 0..1, used in ranking
    expires_at   timestamptz                -- TTL for retention
);
CREATE INDEX ON episodic_events USING ivfflat (embedding);
CREATE INDEX ON episodic_events (user_id, ts DESC);

CREATE TABLE semantic_facts (
    id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id      text NOT NULL,
    key          text NOT NULL,             -- "preferred_payment_method"
    value        jsonb NOT NULL,
    confidence   real NOT NULL,
    source_event uuid REFERENCES episodic_events(id),
    updated_at   timestamptz DEFAULT now(),
    UNIQUE(user_id, key)
);

CREATE TABLE procedural_skills (
    id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    scope        text,                       -- "global" / "tenant:X" / "user:Y"
    trigger      text NOT NULL,
    action       text NOT NULL,
    approved_by  text,                       -- human reviewer
    approved_at  timestamptz
);
# Stage 02-03: extract candidate facts from a finished trajectory.
# Runs on batch API (cheap, 50% off — Module 4.3).

EXTRACT_SYS = """You extract durable facts from a customer service trajectory.
Return JSON: { "facts": [{ "key": "...", "value": ..., "confidence": 0..1 }] }
Only include facts likely to stay true for > 7 days."""

def extract_facts(trajectory):
    r = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=512, temperature=0,
        system=EXTRACT_SYS,
        messages=[{"role": "user", "content": to_text(trajectory)}],
    )
    return json.loads(r.content[0].text)["facts"]

def resolve_and_store(user_id, facts):
    for f in facts:
        existing = db.fetchone(
            "SELECT value, confidence FROM semantic_facts "
            "WHERE user_id=%s AND key=%s",
            (user_id, f["key"]),
        )
        if existing and existing["confidence"] > f["confidence"]:
            continue                                # keep the older, stronger
        db.upsert("semantic_facts", user_id=user_id, **f)
# Stage 05-06: rank + inject memory into the agent prompt each turn.

def recall(user_id, query, budget_tokens=400):
    q_vec = embed(query)
    rows = db.fetchall("""
        SELECT key, value, ts, salience,
               1 - (embedding <=> %s) AS sim
        FROM episodic_events
        WHERE user_id = %s AND (expires_at IS NULL OR expires_at > now())
        ORDER BY (
            0.6 * (1 - (embedding <=> %s))
          + 0.2 * salience
          + 0.2 * exp(-extract(epoch FROM now() - ts) / 604800)  -- 7-day decay
        ) DESC
        LIMIT 8
    """, (q_vec, user_id, q_vec))

    facts = db.fetchall(
        "SELECT key, value, confidence FROM semantic_facts "
        "WHERE user_id = %s ORDER BY confidence DESC LIMIT 12", (user_id,))

    return render_block(rows, facts, budget=budget_tokens)

06 / Privacy · GDPR · Retention
Privacy for Agent Memory
Memory holds personal data by default. GDPR, CCPA, and sector rules (HIPAA, financial) all reach into your memory store — not just your primary database.
RIGHT 01 HIGH

Right to access

User can ask for everything you remember about them. Need an export endpoint that walks all three memory tables.

RIGHT 02 HIGH

Right to erasure

"Forget me" → cascading delete across episodic, semantic, procedural — including derived facts and training datasets.

RIGHT 03 MID

Right to rectification

"This fact is wrong" → editable semantic facts with audit trail of who changed what when.

RIGHT 04 MID

Data portability

Export memory in a machine-readable format (JSON). Your schema is the gift here — vendors often hide it.

RIGHT 05 HIGH

Purpose limitation

Memory captured for service must not be reused for model training without re-consent. Tag every write with purpose.

RIGHT 06 MID

Data residency

EU users → EU region. JP users → JP region. Force the store + embeddings to obey the user's region.

Defaults That Pass an Audit
Suggested Retention Policy

Not legal advice — but a defensible starting matrix. Tighten for regulated domains (HIPAA, financial, child-facing).

EPISODIC · short events
30-90 days, then summarize into semantic and drop. Audit log of deletes kept 7 years.
SEMANTIC · facts
Lifetime of the account. Deleted on account closure (with grace window).
PROCEDURAL · skills
Organizational, not user-bound. Anonymize before storing. Reviewed quarterly.
SENSITIVE CATEGORIES
Health, financial, biometric → never stored in plain text. Mask at capture or refuse to store.
07 / Eviction Strategies
Six Eviction Strategies
Unbounded memory is a leak. Worse, it's a leak that quietly degrades recall — old, stale, contradictory entries pollute the top-k. These six policies, stacked, keep memory clean.
POLICY 01

TTL by type

Different lifetimes per table. Episodic 30-90d, semantic = account life, procedural = quarterly review. Hard-delete past TTL.

→ trivial, always on
POLICY 02

Salience decay

Each recall bumps salience; non-recalled entries decay. Below a floor → archive or drop. Mimics human forgetting.

→ keeps the useful, drops noise
POLICY 03

Deduplication

"User likes email" written 12 times → keep one with highest confidence. Embedding similarity threshold collapses duplicates.

→ collapses redundancy
POLICY 04

Contradiction resolution

"Lives in Tokyo" vs "Lives in Osaka" → cheap-LLM adjudicator picks the most recent + highest-confidence one, archives the loser.

→ keeps the store coherent
POLICY 05

Manual edits + audit

Users (or support) can edit/delete specific facts. Every change writes an audit row. Mandatory under GDPR rectification.

→ regulator-friendly
POLICY 06

Sensitive redaction

PII detector (Module 3.3) runs on capture and on a sweep. Redact / drop sensitive categories that slipped through.

→ defence in depth
A Sensible Default
Production Stack

Run policies 1 + 3 + 4 + 6 always. Add 2 once you have enough recall traffic to compute salience reliably. Reserve 5 for high-touch products (support agents, healthcare).

RUN NIGHTLY
Dedup sweep · contradiction resolver · sensitive-data sweep · TTL purge.
RUN ON RECALL
Bump salience for entries that helped. Decay the rest gradually.
08 / Recap
Key Takeaways
TAKEAWAY 01

Memory ≠ checkpoints. Checkpointers persist one trajectory; memory persists what the agent learns across trajectories, sessions, and time.

TAKEAWAY 02

Three types — episodic (events), semantic (facts), procedural (skills). Each has its own write rules, retrieval shape, and governance.

TAKEAWAY 03

Pick a vendor by privacy posture: Mem0 for speed, Zep for temporal queries, LangMem for platform fit, roll-own for regulated/sovereign needs.

TAKEAWAY 04

A roll-your-own pipeline has six stages: capture → extract → resolve → store → rank → inject. Knowing them lets you swap providers without re-architecting.

TAKEAWAY 05

Recall blends semantic similarity + salience + recency decay. Always cap by token budget so memory doesn't eat the context window.

TAKEAWAY 06

GDPR rights reach into memory: access, erasure, rectification, portability, purpose limitation, residency. Treat memory as personal data by default.

TAKEAWAY 07

Six eviction strategies — TTL, salience decay, dedup, contradiction resolution, manual edits, sensitive redaction. Stack 1 + 3 + 4 + 6 by default.

TAKEAWAY 08

Procedural memory is organizational, not user-bound. Skill proposals need human approval before they're applied. Governance, not just storage.
