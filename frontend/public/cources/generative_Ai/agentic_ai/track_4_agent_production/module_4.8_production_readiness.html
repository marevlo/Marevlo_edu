SLO program rules
How SLOs bite

How SLOs Bite

  • Each SLO has an explicit error budget over a rolling 30 days.
  • If budget burns > 50% mid-window → freeze risky ramps from Module 4.5.
  • If budget exhausts → all rollouts pause; SREs & product agree the recovery plan.
  • SLOs reviewed quarterly by the Agent Council (Module 4.7) — too tight kills velocity, too loose kills trust.
04 / Runbooks
Six Standard Runbooks
Each runbook fits on one page, has four parts — alert → diagnose → fix → verify — and is linked from the alert payload. Pages at 3am are no time to scroll a wiki.
RB-AGT-01

Success-rate drop

Trajectory success below SLO for ≥ 5m.

RB-AGT-02

p99 cost spike

p99 cost ↑ 2× baseline for ≥ 10m.

RB-AGT-03

Tool error storm

One tool exceeding 5% error rate.

RB-AGT-04

Global kill switch

Emergency throttle/stop of the fleet.

RB-AGT-05

Bad rollout rollback

Sub-60s revert (Module 4.5 drill).

RB-AGT-06

PII / safety leak

Confirmed sensitive-data exposure.

Runbook · RB-AGT-02 — p99 cost spike — one-page playbook
owner: agent-eng-oncall · sla: 15m
01 ALERT. p99 trajectory cost > 2× 7-day baseline for ≥ 10m. PagerDuty fires; payload links here + dashboard.
02 DIAGNOSE. Open the per-intent / per-tenant cost ledger (Module 4.3 table). Sort by p99. The outlier is usually one intent × one tenant × one version.
03 DIAGNOSE. Pull a sample failed trajectory in the trace UI (Module 4.2). Look for: loops, retry storms, huge context, runaway tool calls.
04 FIX. If a tenant outlier: trip per-tenant burn-rate breaker. If a bad rollout: trigger RB-AGT-05. If context bloat: tighten pruning policy and ship a patch.
05 VERIFY. p99 cost returns under threshold for 15m. File the incident report; queue regression test for next CI run.
Rule of Thumb

Every runbook is a 5-line story: alert · two diagnose steps · one fix step · one verify step. If you can't compress it that far, the runbook is fighting two incidents in one.

05 / Incident Response
Incident Response
Same shape as classic SRE incident response — with three agent-specific twists: blast radius is real-world, memory persists past rollback, and regulators may need notification.
Severity matrix
Pick one in < 5 minutes

Four Severities — Pick One in < 5 Minutes

SEV-1 · CRITICAL
Real-world harm, regulatory exposure, or majority of users affected. War room, page exec on-call, customer comms within 30 min.
SEV-2 · HIGH
Significant cost / quality regression, single high-value tenant down, or potential SEV-1 within hours.
SEV-3 · MEDIUM
Degraded performance, isolated feature broken, workaround exists. Business-hours response.
SEV-4 · LOW
Cosmetic, observability gap, internal-only impact. Tracked in normal sprint.
Postmortem template
Eight required sections

Eight required sections

  • 01Summary — one paragraph an exec can read.
  • 02Timeline — UTC, linked to traces (Module 4.2).
  • 03User impact — counts, $, regulatory notes.
  • 04Root cause — agent-side and upstream.
  • 05What went well, what went wrong.
  • 06Action items with owners and dates.
  • 07Memory-poison check — see right.
  • 08Eval-set update — regression test added.
Three agent-specific twists
What classic SRE misses

What classic SRE postmortems miss

  • Memory poison. A bad rollout can write false facts into long-term memory (Module 4.4). Postmortem must inspect & quarantine the affected window.
  • Action irreversibility. Refunds issued, emails sent, records modified. Catalogue every side-effect; some need manual reversal.
  • Regulator notification. For SEV-1 incidents touching personal data, GDPR Art. 33 requires notification within 72h. Legal owns this row.
06 / 30 · 60 · 90 Review
Three Post-Launch Reviews
Same audience never works. Each review has a different question, attendee list, and outcome. Skip any and the launch quietly rots.
DAY 30

Operational Review

"Did it run?" Look at SLO burn, incidents, runbook quality, alert noise.

  • → Owner: eng + SRE
  • → Attendees: agent-eng-oncall, product
  • → Outcome: top 3 ops fixes for next 30d
DAY 60

Quality & Cost Review

"Is it good and affordable?" Trajectory eval vs baseline, cost cascade health, judge-vs-human drift.

  • → Owner: ML lead + product
  • → Attendees: eval lead, eng, ops
  • → Outcome: prompt & cost roadmap
DAY 90

Governance & Strategy Review

"Should it keep running, expand, or wind down?" Compliance, risk, ROI, roadmap.

  • → Owner: agent council (Module 4.7)
  • → Attendees: exec, legal, security, product, eng
  • → Outcome: continue / expand / decommission decision
Why Three

At day 30 you don't have enough signal to judge quality, just operations. At day 60 you can tell whether the cascade economics work. At day 90 you've earned the right (and accumulated the evidence) to put the question to the executive team: is this still the right thing to be doing?

07 / Decommissioning
Decommissioning — Six Stages
Agents get retired — replaced, deprecated, sunsetted. The retirement plan exists from day one; running it cleanly preserves customer trust, memory integrity, and the audit trail.
Six-stage retirement
Decide → Retain
STAGE 01
DECIDE
council decision · day-90 review
STAGE 02
ANNOUNCE
customers · partners · status page
STAGE 03
FREEZE
no new features · only fixes
STAGE 04
MIGRATE
users → successor · with state
STAGE 05
SUNSET
traffic to 0 · in-flight drained
STAGE 06
RETAIN
audit · regulatory minimums only
Critical

Stage 06 isn't "delete everything." Audit trail (Module 4.7) and required regulatory records (Module 4.4) survive on a separate retention schedule — often 7+ years. Code and prompts can go; the evidence stays.

CUSTOMER

Customer-facing checklist

  • → Notice period ≥ contract minimum (often 60-90d).
  • → Migration guide + successor agent or workflow.
  • → Data export available throughout sunset window.
  • → Status page + email cadence (T-90, T-30, T-7, T-1).
DATA

Data & memory checklist

  • → Memory retention reviewed per type (Module 4.4).
  • → Procedural skills migrated or archived.
  • → Audit trail handed to compliance with retention SLA.
  • → DSAR endpoint kept live until retention window ends.
08 / Download Summary
Capstone Takeaways
TAKEAWAY 01

The 50-point audit compresses Track 4 into a single launch gate. Score honestly; ≥ 47 ships, 40-46 ships behind a flag, < 40 doesn't ship.

TAKEAWAY 02

Eight SLOs cover the agent fleet: success, p50/p95 latency, p99 cost, tool error, step-cap breach, approval response, audit completeness. Each carries an error budget.

TAKEAWAY 03

Six runbooks, each five lines and one page: alert → 2× diagnose → fix → verify. Linked from alert payloads.

TAKEAWAY 04

Incident response is classic SRE plus three agent twists: memory poison, action irreversibility, regulator notification. Bake all three into the postmortem template.

TAKEAWAY 05

Run three structured reviews: D-30 operational, D-60 quality & cost, D-90 governance & strategy. Different audiences, different decisions.

TAKEAWAY 06

Decommissioning has six stages: decide → announce → freeze → migrate → sunset → retain. Code goes; audit & regulatory records stay on a separate schedule.

TAKEAWAY 07

The audit is the single document that lets eng, product, security, and legal agree to ship in one meeting. Tag every line to its source module.

TAKEAWAY 08

Track 4 is complete. You can now deploy, observe, optimize, remember, evolve, secure, govern, and retire agent systems with the discipline of a payments platform.
