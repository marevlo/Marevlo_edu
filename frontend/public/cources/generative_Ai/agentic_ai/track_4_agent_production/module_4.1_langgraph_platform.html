Pick · LangGraph Platform

If you want runtime + dashboards out of the box

  • → Small/mid team without dedicated platform engineers
  • → Multi-tenant SaaS where threads/users are first-class
  • → Need cron, webhooks, HITL approvals quickly
  • → Want LangSmith-grade trajectory traces by default
  • → Accept managed-service tradeoffs (latency, vendor)
Pick · Self-Host (FastAPI / Docker)

If you have the team and special needs

  • → Air-gapped / regulated env (no managed plane)
  • → Existing platform engineering with k8s + Temporal/Airflow
  • → Custom checkpointer (e.g., on top of internal state store)
  • → Strict cost / sovereignty / latency SLOs
  • → Patterns covered in Course 01 / Track 7 still apply

# graphs/refund.py — a deployable graph. No FastAPI scaffolding.

from langgraph.graph import StateGraph, START, END
from typing_extensions import TypedDict

class State(TypedDict):
    user_id: str
    goal:    str
    steps:   list[dict]
    verdict: str | None

def plan(state):    ...
def act(state):     ...
def decide(state):  ...

builder = StateGraph(State)
builder.add_node("plan", plan)
builder.add_node("act", act)
builder.add_edge(START, "plan")
builder.add_edge("plan", "act")
builder.add_conditional_edges("act", decide, {"done": END, "continue": "act"})

# The platform wires the checkpointer at deploy time — leave it unset here.
graph = builder.compile()

05 / Checkpointers in Production
Checkpointers — Durability for Agents
A checkpointer commits agent state after each node. Without it, a restart loses everything. With it, you get pause/resume, HITL, rewind, and replay — for free.
Backend Durability Use For Avoid For
MemorySaver lost on restart local dev · unit tests · ephemeral demos anything production
SqliteSaver file-local · single host desktop apps · single-tenant CLI agents multi-replica deployments
PostgresSaver durable · multi-host · ACID default production choice for SaaS agents extreme write QPS without sharding
RedisSaver durable (AOF/RDB) · low-latency very high write QPS · ephemeral sessions strict relational query needs

# Wiring PostgresSaver — the production default.

from langgraph.checkpoint.postgres import PostgresSaver

DB_URI = os.environ["POSTGRES_URL"]

with PostgresSaver.from_conn_string(DB_URI) as checkpointer:
    checkpointer.setup()                       # idempotent; runs migrations
    graph = builder.compile(checkpointer=checkpointer)

    # Every node-boundary now commits to Postgres. 
    # thread_id is the durability key.
    config = {"configurable": {"thread_id": "u-7281-r-001"}}
    result = graph.invoke({"goal": "refund Oct duplicate"}, config)
# Resume a paused thread — e.g. after a human approval comes back.

config = {"configurable": {"thread_id": "u-7281-r-001"}}

# Inspect current checkpoint without running.
snapshot = graph.get_state(config)
print(snapshot.values)        # last committed state
print(snapshot.next)          # nodes pending

# Push an update (e.g. approval), then continue from the pause point.
graph.update_state(config, {"approval": "granted"}, as_node="await_human")
graph.invoke(None, config)            # None input = resume from checkpoint

# Rewind: re-run from any historical checkpoint.
past = list(graph.get_state_history(config))
rewind_cfg = past[3].config          # 3 commits ago
graph.invoke(None, rewind_cfg)
# Migrating in-flight state when the graph schema evolves.
# Run as a one-off job between deploys.

def migrate_v1_to_v2(old):
    return {
        "user_id": old["uid"],          # renamed
        "goal":    old["goal"],
        "steps":   old.get("history", []),  # renamed
        "verdict": old.get("verdict"),
    }

for thread_id in list_threads():
    cfg = {"configurable": {"thread_id": thread_id}}
    snap = graph.get_state(cfg)
    if snap.values.get("_schema") != "v2":
        graph.update_state(cfg, {**migrate_v1_to_v2(snap.values),
                                  "_schema": "v2"})
RULE 01

Thread ID = durability key

Treat thread_id as a first-class business identity (order, conversation, ticket) — not a UUID you toss.

RULE 02

Schema is versioned state

Add a _schema field. Migrate explicitly before deploys that change the graph shape.

RULE 03

Tune commit frequency

Commits happen at node boundaries. Bigger nodes = fewer commits = better throughput, but coarser resume.

RULE 04

TTL old threads

Threads accumulate. Define retention (30d / 90d / per regulation). Without TTL, your Postgres becomes the memory leak.

RULE 05

Index your hot paths

Index by user_id, tenant_id, status. Otherwise dashboards quietly time out.

RULE 06

Test resume in CI

Kill a worker mid-run. Assert another picks it up from checkpoint with identical final state. Make it a release gate.

06 / State Streaming
Streaming Agent State to Clients
A 40-second silent agent looks broken. Streaming partial state is the difference between "thinking" and "abandoned". Pick the right mode per surface.
MODE 01

values

Emits the full state object after each node. Simple, easy to render — but verbose at scale.

→ debugging UIs · admin tools
MODE 02

updates

Emits only the delta per node — the keys that changed. The production default for live UIs.

→ end-user chat · dashboards
MODE 03

messages

Token-by-token streaming of LLM messages. Pair with updates for full chat UX.

→ ChatGPT-style typing UI
MODE 04

events

Every internal event: node enter/exit, tool start/end, retries, errors. High volume; for power UIs.

→ trajectory replay · developer mode
MODE 05

debug

Everything plus checkpoint metadata, task IDs, timing. Internal only — never expose to end users.

→ internal observability only
COMBOS

Multi-mode

Pass a list of modes. The SDK multiplexes — each chunk is tagged so the UI routes it.

→ ["updates", "messages"] = default UX
Live stream · stream_mode=["updates", "messages"]

Design Note

The client renders messages directly into a chat bubble (typing animation), and uses updates to render the "doing X..." status line. Tool calls and observations show up in a collapsible "agent steps" panel.

07 / WebSocket Patterns
WebSocket Patterns for Agents
Long-lived agent connections expose realities you didn't hit with stateless RAG. Keep-alives, reconnects, backpressure, auth handoff — all of it matters.
MODE 01

SSE (Server-Sent Events)

One-way, HTTP-friendly. Browser-native. Easy reconnect. Default for most chat UIs.

✓ trivial reconnect
✓ no proxy issues
✗ no client → server stream
MODE 02

WebSocket (full-duplex)

Two-way. Required when the user interrupts, edits state mid-run, or co-edits a doc. Default for interactive agents.

✓ bi-directional
✓ lower per-message overhead
✗ proxy / CDN quirks
MODE 03

Webhooks (event push)

No persistent connection. Agent calls back when done. Default for cron / batch / B2B integrations.

✓ survives client downtime
✓ trivially horizontal
✗ no in-flight streaming

// Browser client — listen to a thread's run and route chunks by type.

const ws = new WebSocket(`wss://refund-agent-prod.langgraph.app/threads/${threadId}/stream`);

ws.onopen = () => ws.send(JSON.stringify({
  type: "start_run",
  assistant_id: "refund_agent",
  input: { goal: "refund Oct duplicate" },
  stream_mode: ["updates", "messages"],
}));

ws.onmessage = (e) => {
  const chunk = JSON.parse(e.data);
  switch (chunk.mode) {
    case "messages": appendToken(chunk.data); break;
    case "updates":  updateStatusLine(chunk.data); break;
  }
};
// Resilient reconnect — agent runs are durable, the socket is not.

function connect(threadId, lastEventId) {
  const ws = new WebSocket(`${BASE}/threads/${threadId}/stream?since=${lastEventId ?? 0}`);

  ws.onmessage = (e) => {
    const chunk = JSON.parse(e.data);
    lastEventId = chunk.id;          // remember high-water mark
    render(chunk);
  };

  ws.onclose = () => {
    // Exponential backoff; agent run is still alive on the server.
    setTimeout(() => connect(threadId, lastEventId), backoff++);
  };
}
# Server side — auth on socket upgrade, scope check per thread.

from fastapi import WebSocket, WebSocketException, status

@app.websocket("/threads/{thread_id}/stream")
async def stream(ws: WebSocket, thread_id: str):
    token = ws.query_params.get("token")
    user  = verify_jwt(token)
    if not user_owns_thread(user, thread_id):
        raise WebSocketException(code=status.WS_1008_POLICY_VIOLATION)

    await ws.accept()
    async for chunk in subscribe(thread_id):
        await ws.send_json(chunk)
PROD 01

Heartbeat / ping

Send a ping every 15-30s. Many proxies close "idle" connections at 60s.

PROD 02

Event IDs & resume

Every chunk carries an id. On reconnect, pass since=lastId; server replays missed events.

PROD 03

Backpressure

If the client falls behind, buffer up to N events then drop oldest. Don't OOM the server because one browser tab is slow.

PROD 04

Auth at upgrade

Validate JWT on the WS handshake. Verify the user owns the thread. Renew long-lived tokens server-side.

PROD 05

No session affinity

Any worker can serve any thread because state is in the checkpointer. Don't pin sockets to nodes.

PROD 06

Quotas per user

Cap concurrent sockets and concurrent runs per user. Otherwise one buggy client = one full worker pool.

08 / Stateful Deploy Patterns
Six Stateful Agent Deploy Patterns
Same runtime, different shapes. Pick the pattern that matches the product, not the most fashionable one.
PATTERN 01

Interactive Chat

User-driven, WebSocket, token streaming, fast feedback. Threads live for the conversation.

runs: short · state: thread-scoped · UI: chat
PATTERN 02

Long-running Task

Submit, walk away, get notified. Email or webhook on completion. Hours, not seconds.

runs: long · state: durable · UI: queue/status
PATTERN 03

HITL Workflow

Runs auto-pause at approval node, resume on human input. PAUSED is a first-class state.

→ Module 4.7 (governance) builds on this
PATTERN 04

Scheduled / Cron

Agent fires on a schedule. No client. State persists across runs as memory or store.

runs: triggered · state: long-lived · UI: dashboard
PATTERN 05

Event-Driven

Webhook in → run starts → webhook out. Useful for B2B integrations and async pipelines.

runs: per event · state: per source · UI: none
PATTERN 06

Persistent Companion

One agent per user, lives forever. Long-term memory store dominates. Threads come and go; identity persists.

→ Module 4.4 (memory) is the centerpiece

Lifecycle

Every Run Goes Through These States.

running → paused → waiting (human/tool) → resumed → completed → failed (rare)

09 / Recap
Key Takeaways
TAKEAWAY 01

A managed agent runtime exists because eight infra components need to be rebuilt for any stateful agent. Buy unless you have a strong reason to build.

TAKEAWAY 02

The runtime topology is client → gateway → workers + queue + stream hub → checkpointer + memory store + tool sandboxes + LLMs. Internalize the picture.

TAKEAWAY 03

Deploys are versioned, healthchecked, and traffic-shifted. Schema changes pause in-flight threads — migrations are an explicit step, not an afterthought.

TAKEAWAY 04

Production checkpointers default to Postgres; use Redis for hot paths, SQLite only for single-host CLIs, Memory only for tests.

TAKEAWAY 05

thread_id is a business identity — not a UUID. Version the state schema. TTL old threads. Test resume in CI.

TAKEAWAY 06

Pick the right stream mode: updates for UI status, messages for token typing, events for replay tools, debug never for end users.

TAKEAWAY 07

Choose SSE for chat UIs, WebSocket for interactive editing, webhooks for async/B2B. Heartbeat, event-IDs, backpressure, no session affinity.

TAKEAWAY 08

Match the deploy pattern to the product: interactive chat, long task, HITL, cron, event-driven, persistent companion. The runtime supports all six.
