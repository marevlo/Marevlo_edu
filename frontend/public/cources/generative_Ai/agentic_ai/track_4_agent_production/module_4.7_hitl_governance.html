02 / Risk Tiers
Four Action Risk Tiers
Every tool the agent can call belongs in exactly one tier. The tier dictates the policy — and the policy is auditable, testable, and constant across releases.
TIER 01 · AUTO

Auto · runs alone

Read-only, idempotent, low blast radius. No human required.

  • → crm_lookup, search, embed
  • → idempotent reads
  • → cap p99 cost only
TIER 02 · NOTIFY

Notify · proceed + log

Side-effects but reversible / bounded. Human sees a notification, not a gate.

  • → send templated emails
  • → create draft tickets
  • → tag records
TIER 03 · APPROVE

Approve · wait for human

Real money, real writes, real users. Agent pauses; checkpoint commits; human releases.

  • → refund > $X
  • → send to all users
  • → change permissions
TIER 04 · REFUSE

Refuse · agent never does

Above the agent's authority. Tool simply isn't available to it; human-only domain.

  • → close customer accounts
  • → issue regulatory filings
  • → hire / fire
Classifying Actions — Four Criteria, One Tier
Criteria

For every tool, ask these four questions. The combination determines the tier — and the tier is written in code as a policy, not a comment.

CRITERION 01 · REVERSIBILITY
Can the action be undone within minutes / hours / not at all? Irreversible → at least APPROVE.
CRITERION 02 · BLAST RADIUS
How many users, dollars, records, or partners affected on failure? Big → at least APPROVE.
CRITERION 03 · REGULATORY EXPOSURE
Does the action touch PHI / payments / personal data / safety? Often forces REFUSE or APPROVE.
CRITERION 04 · USER TRUST
Would the user expect to see this happen without confirmation? "Surprise factor" alone can require APPROVE.

# An approval is a graph node that returns & checkpoints. 
# LangGraph's interrupt mechanism does the pause cleanly.

from langgraph.types import interrupt

def await_human(state):
    tier = classify_tier(state["proposed_action"])
    if tier != "approve":
        return {}                          # pass-through for auto/notify

    decision = interrupt({
        "action":     state["proposed_action"],
        "reason":     state["reason"],
        "confidence": state["confidence"],
        "sla_due":    now() + timedelta(minutes=15),
    })
    # interrupt() commits checkpoint, returns when a human resumes.
    return {"approval": decision}                 # approve / edit / reject
# Human responds via the approval UI; backend resumes the thread.

@app.post("/approvals/{thread_id}/decide")
async def decide(thread_id, decision: Decision, user=Depends(auth)):
    require_role(user, "approver_L1")        # scope check (Module 4.6)

    audit(
        action="approval_decision",
        thread_id=thread_id,
        approver=user.id,
        decision=decision.verdict,
        note=decision.note,
    )

    config = {"configurable": {"thread_id": thread_id}}
    graph.update_state(config, {"approval": decision.model_dump()})
    await graph.ainvoke(None, config)            # resume from checkpoint

Six Approval UX Rules

RULE 01

Show the trajectory

Approvers see the full agent reasoning + tool calls (Module 4.2 trace), not just the proposed action.

RULE 02

Approve · Edit · Reject

Three first-class actions — never just "OK / Cancel". Edit lets a human change the action without exiting the loop.

RULE 03

Mandatory reason on reject

Reject without text = poor governance. Reason feeds the eval set (Track 3) for future regressions.

RULE 04

SLA per tier

Each tier has a target response time. Breach → escalate (next section), not silently rot.

RULE 05

Separation of duties

For Tier 03+ in regulated domains: the approver can't be the user who triggered the run.

RULE 06

No batch-approve

Click-through fatigue erodes the value of HITL. Approvers see one decision at a time.

04 / Escalation Ladder
Four-Rung Escalation Ladder
When humans don't respond — and they won't, sometimes — the run can't sit forever. Each rung has a clear owner, SLA, and what happens when it lapses.
Ladder · refund-agent — Rung · SLA · Owner
Ladder
L1 First-line agent reviewer — picks up most approvals SLA 15 min support agent
L2 Senior reviewer — > $1k, sensitive intent, edge cases SLA 1 hour support lead
L3 Manager — policy exceptions, multi-tenant, repeated reject SLA 4 hours ops manager
L4 Compliance / on-call — regulatory edges, breach, broken state SLA 30 min (page) compliance + SRE
// THE FAILSAFE: If L1 doesn't respond within SLA → auto-escalate to L2. Same at every rung. If L4 doesn't respond → the run fails soft with a "needs review" status and the customer gets a graceful "we're looking into this" message — not silence, not a wrong action.
SLA POLICY

Auto-escalation on SLA breach

A scheduled job (Module 4.1 cron) wakes every minute, scans paused threads, and promotes to the next rung if SLA expired. The promotion itself is an audit event.

PAGE POLICY

Page only at L4

L1-L3 use email / Slack / queue widgets. Pages reserved for compliance edges and SRE-level incidents. Otherwise on-call fatigue undoes the program (see Module 4.2 anti-patterns).

FAIL-SOFT

Graceful customer-facing fallback

An unresolved approval after L4 SLA → user sees "we're reviewing this and will follow up". Never silent, never the wrong action.

BREAKER LINK

Circuit-breaker integration

Trips from Module 4.3 (cost / step / tenant breakers) drop straight into L2 of this ladder. Same UI, same audit trail, same SLA logic.

05 / Audit Trails
Audit Trails — Nine Fields
An audit row exists for every consequential decision — by agent or human. Append-only, signed, replayable. Auditors don't ask for one missing field; they assume the worst when one is missing.
Audit Row Schema — Nine Standard Fields
Schema
01 · ts
UTC, monotonic. ISO-8601 with millisecond precision.
02 · actor
user_id · agent_version · system. One source per row.
03 · action
Canonical name — refund_issued, approval_decision, etc.
04 · target
What was acted on — invoice id, account id, doc id.
05 · before / after
Hash-diff or full snapshot for mutations. PII scrubbed.
06 · context
thread_id, trace_id, agent_version, release_id (Module 4.2 tags).
07 · justification
Reason text from agent or human. Mandatory for rejects.
08 · approver_chain
Ordered list of [actor → decision → ts] for the approval ladder.
09 · signature
HMAC of row content with a server-held key. Tamper-evident.

Six Audit-Store Rules

STORE 01

Append-only

No UPDATE / DELETE on the audit table. Corrections are new rows that reference the original.

STORE 02

Separate retention

Audit retention typically 7+ years (regulator-driven). Decoupled from trace retention (Module 4.2) or memory retention (Module 4.4).

STORE 03

Cold archive

Older rows roll into object storage with WORM (write-once). Cheap to keep, hard to tamper.

STORE 04

PII scrubbed

Same scrubber as Module 4.2. Audit needs what happened, not raw PII. Reference by hash when needed.

STORE 05

Linked to trace

Every audit row carries the trace_id. Auditor clicks → full trajectory in LangSmith. No mystery.

STORE 06

Export ready

Have an export endpoint that filters by user / tenant / time range. GDPR access requests will hit this.

07 / Governance Committees
Standing Governance
Workflows + audit trails don't govern themselves. A cadence of standing meetings keeps the program from rotting after launch.
Committee Charter Cadence · Members
Agent Council Sets policy, owns the risk tier matrix, approves new high-tier actions, signs off on major releases. Monthly · product · eng lead · legal · ops · security
Model Risk Review Reviews eval regressions, post-incident reports, model-version changes, fairness reports. Bi-weekly · ML lead · eval lead · ops
Compliance Working Group Tracks SOC2 / HIPAA / GDPR / EU AI Act mappings, DSARs, sub-processor BAAs. Monthly · legal · security · DPO · ops
Incident Review Postmortems for outages, breaches, regulatory near-misses; runbook updates. As needed · on-call · SRE · product · legal
Governance Anti-Patterns — Four Ways Governance Goes Theatrical
Anti-patterns
AP 01 · POLICY-WITHOUT-CODE
A doc says "we require approval on refunds > $500" but the code path doesn't check. Policies belong in code, not just markdown.
AP 02 · COMMITTEE-OF-RECORD
Council meets, nods, exits. Without decisions, action items, and follow-up the meeting is a calendar block, not governance.
AP 03 · BATCH-APPROVE FATIGUE
If reviewers approve hundreds per shift, they don't review — they click. Tighten tier criteria so the queue is short and meaningful.
AP 04 · OWNERLESS RISK
No single name on each tier-3+ action. Compliance loves named owners. Engineering loves them too — they prevent debate at incident time.
08 / Recap
Key Takeaways
TAKEAWAY 01

Every action belongs in one of four tiers — auto, notify, approve, refuse. Tier is policy-in-code, not policy-in-prose.

TAKEAWAY 02

Classify by reversibility, blast radius, regulatory exposure, user trust. Anything irreversible or surprising starts at APPROVE.

TAKEAWAY 03

Approval is a graph node that pauses + checkpoints. Resume from any state via update_state. Three actions: approve · edit · reject.

TAKEAWAY 04

Four-rung escalation with SLA per rung. Auto-escalate on lapse; page only at L4; fail soft for the customer.

TAKEAWAY 05

Every consequential decision writes an audit row with nine fields — append-only, signed, linked to its trace_id.

TAKEAWAY 06

SOC2, HIPAA, GDPR, EU AI Act are satisfied by your engineering work when you tag it. Don't build a parallel compliance program.

TAKEAWAY 07

Standing committees — Agent Council, Model Risk Review, Compliance WG, Incident Review — keep governance alive after launch.

TAKEAWAY 08

Avoid the four anti-patterns: policy-without-code, committee-of-record, batch-approve fatigue, ownerless risk. Each one quietly turns governance into theatre.
