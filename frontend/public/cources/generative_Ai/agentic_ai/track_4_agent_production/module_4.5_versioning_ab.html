02 / Why This Is Hard
Six Prompt Versioning Challenges
Code has compilers, type-checkers, and unit tests. Prompts have eval sets, judges, and tail risk. The challenges below are why this module exists.

Code has compilers, type-checkers, and unit tests. Prompts have eval sets, judges, and tail risk. The challenges below are why this module exists.

CHALLENGE 01

Soft diffs

A reworded sentence can flip behavior; a refactor of formatting often doesn't. Line-diff ≠ behavior diff.

→ eval set is the only honest diff
CHALLENGE 02

Non-deterministic regressions

Same prompt, same model, same temp — different runs. Single-trial comparisons lie.

→ multi-seed, statistical tests
CHALLENGE 03

Coupled changes

Prompt + graph + tool schema + memory schema often ship together. "Which one regressed?" needs isolation.

→ change-only-one-thing discipline
CHALLENGE 04

Distribution drift

Eval set was built last quarter. Today's traffic is different. A 5% lift on eval can be a 0% lift in prod.

→ keep prod-like holdouts fresh
CHALLENGE 05

Long-tail blast

A "good" prompt change can break the 0.5% case that costs you a refund. Means look fine; tails bite.

→ p99 metrics matter (3.4 + 4.2)
CHALLENGE 06

Hidden coupling with memory

A prompt change can quietly rewrite how facts get extracted (Module 4.4). Memory poisoned by a bad rollout outlives the rollback.

→ memory writes need their own version
03 / Tools
Promptfoo · LangSmith Hub · BAML · git
Four serious approaches. Most teams use two — one for the registry, one for evaluation. They compose; they don't compete.
Tool Best At Weak At Best When
Promptfoo side-by-side eval across prompts & models · YAML configs long-lived registry · live A/B routing pre-deploy regression gates in CI
LangSmith Hub versioned prompt registry · linked to traces (Module 4.2) heavy statistical analysis teams already on LangSmith / LangGraph
BAML type-safe prompt+function bundles · compile-time schema checks non-Python/TS stacks · pure prompt drafting when prompts are structured-output heavy
git + plain files universal · diff-friendly · auditable no eval · no rollout · no traffic split always — as the source of truth under any of the above
A Default Combination
Sensible Stack

No single tool covers everything. A working production stack usually layers three or four:

LAYER 1 · SOURCE
git — every prompt + graph file under version control. PR reviews mandatory.
LAYER 2 · STRUCTURE
BAML (optional) — typed prompts with schema for structured outputs.
LAYER 3 · CI GATE
Promptfoo — runs the eval set on every PR; blocks merges below threshold.
LAYER 4 · REGISTRY
LangSmith Hub — versioned production prompts, linked to traces & runs.

04 / Prompt Semver
Semantic Versioning for Prompts
Borrow the discipline from software, define what each level means for an agent. Without this, every change feels like a major and nothing ever ships.
MAJOR · 2.0.0

Behavioral break

Output schema changes, removed tools, new mandatory inputs, response style fundamentally different.

  • → requires data migration
  • → requires client coordination
  • → full A/B + eval set refresh
MINOR · 1.5.0

Additive change

New tool available, optional new instruction, new fallback path. Backward-compatible behavior.

  • → shadow → canary → ramp
  • → eval gates must hold
  • → dashboards stay green
PATCH · 1.4.3

Tweak

Wording fix, typo, clearer formatting, refusal tightened on the same intent. No new behavior.

  • → CI eval gate passes → ship
  • → small canary recommended
  • → revert path stays warm
Five Changes · Five Versions
Worked Example · refund-agent
1.4.3 PATCH Fixed a typo in the refusal message
1.5.0 MINOR Added shipping_lookup tool as an optional fallback
1.5.1 PATCH Tightened wording on the gold-tier escalation phrase
1.6.0 MINOR New optional output field recommended_followup
2.0.0 MAJOR Rewrote output as a typed JSON schema — clients must update
The Rule

Anything that changes how callers use the agent is MAJOR — even if the prompt diff looks tiny. Anything new and optional is MINOR. Anything that ships at the same behavior level is PATCH.

05 / A/B Infrastructure
A/B Infrastructure
Most teams already have an A/B framework (LaunchDarkly, Statsig, GrowthBook, in-house). The agent-specific work is in assignment, tagging, and reading the right metric.
LANE 01 · SHADOW

0% user-facing

Run candidate in parallel, output discarded. No side effects. Compare offline.

→ 100% of traffic, 0% impact
LANE 02 · CANARY

1-5% user-facing

Small fraction sees candidate. Watch dashboards for regressions before widening.

→ 5% user-facing
LANE 03 · PARTIAL

25-50% user-facing

Real A/B. Big enough sample for power; small enough to revert without drama.

→ 50% A / B split
LANE 04 · FULL

100% user-facing

Candidate becomes production. Previous version held warm for fast revert.

→ 100% on candidate

# Sticky variant assignment — same user gets same variant across runs.

import hashlib

def variant(user_id, experiment, split):
    # Deterministic bucket in [0, 1)
    h = hashlib.sha256(f"{experiment}:{user_id}".encode()).hexdigest()
    bucket = int(h[:8], 16) / 0xFFFFFFFF
    acc = 0.0
    for name, frac in split.items():
        acc += frac
        if bucket < acc: return name
    return list(split)[-1]

SPLIT = {"v1.4.2": 0.5, "v1.5.0": 0.5}
v = variant("u-7281", "refund_decide_v15", SPLIT)
# → always returns "v1.4.2" or "v1.5.0" deterministically for u-7281
# Tag every trace with the variant — Module 4.2 dashboards split by it.

def run_with_variant(graph, user_id, payload):
    v = variant(user_id, "refund_decide_v15", SPLIT)
    config = {
        "configurable": {
            "thread_id":      make_thread(user_id),
            "prompt_version": v,
        },
        "tags": [f"variant:{v}", f"experiment:refund_decide_v15"],
        "metadata": {
            "agent_version": f"refund@{v}",
            "experiment":    "refund_decide_v15",
            "variant":       v,
        },
    }
    return graph.invoke(payload, config)
# Inside a graph node — load the prompt corresponding to the variant.

def decide(state, config):
    v = config["configurable"]["prompt_version"]
    prompt = load_prompt(f"refund-agent/decide:{v}")
    msg = model.invoke([prompt.format(**state)])
    return {"verdict": msg.content}

CAVEAT 01

Non-IID traffic

Users send many turns; assign by user, not by request, or stratify by intent. Otherwise variance is wrong.

CAVEAT 02

Sequential peeking

Don't keep refreshing the p-value. Use sequential / always-valid tests (e.g. mSPRT) or pre-register the stop time.

CAVEAT 03

Multiple metrics

8 KPIs means 8 chances to find a "false" win. Correct for multiple comparisons (Bonferroni / BH).

CAVEAT 04

CUPED for variance

Use pre-experiment success as a covariate. Cuts required sample sizes 30-60% on stable metrics.

07 / Shadow + Rollback
Shadow Mode + Fast Rollback
Shadow runs the candidate alongside production with zero side effects. Rollback brings the previous version back without losing in-flight state. Both are agent-specific.
SHADOW · RULE 01

No real tools

Shadow runs use read-only tool stubs. Refunds, emails, writes — all mocked. The point is comparison, not execution.

SHADOW · RULE 02

No memory writes

Module 4.4's capture pipeline must skip shadow runs. Otherwise the candidate poisons memory before you decide.

SHADOW · RULE 03

Separate budget

Shadow runs cost real money. Tag them in the cost ledger (Module 4.3) and cap their fraction of fleet spend.

SHADOW · RULE 04

Eval offline, not live

Compare candidate vs baseline as a batch nightly job (cheap, 50% discount). Don't run an LLM judge inline during shadow.

One-Minute Revert · Step Sequence
Rollback · Sub-60s SLO

When dashboards or alerts (Module 4.2) flag a bad rollout, this is the sequence the on-call follows. Practice it in drills.

01 Alert fires — success drop > SLO over 5m (Module 4.2 PAGE rule) +0s
02 On-call confirms variant tag is the cause via split dashboard +15s
03 Flip rollout split: variant v1.5.0 → 0% via feature flag +30s
04 New runs route to v1.4.2; in-flight runs continue on their assigned variant (state-safe) +35s
05 Memory poison check: pause Module 4.4 capture pipeline; flag the window +40s
06 Verify success rate recovers; open incident; file regression test for next CI run +55s
The Principle

Old version is held warm for 24-72 hours after a full ramp. Rollback is a flag flip, not a redeploy. Anything slower than that and your blast radius gets larger every second.

08 / Recap
Key Takeaways
TAKEAWAY 01

Prompt versioning has its own pathologies — soft diffs, non-deterministic regressions, coupled changes, drift, tails, memory coupling. Recognize all six.

TAKEAWAY 02

Use the right tool per job: git for source · BAML for structure · Promptfoo for CI gates · LangSmith Hub for registry. They compose.

TAKEAWAY 03

Apply semver rigorously: MAJOR for behavior breaks, MINOR for additive, PATCH for tweaks. Diff size doesn't matter; caller impact does.

TAKEAWAY 04

Ramp through four lanes — shadow → canary → partial → full. Skip lanes only when the risk demands it.

TAKEAWAY 05

Use deterministic, user-sticky variant assignment and tag variant + experiment into every trace. Module 4.2 dashboards split the rest for you.

TAKEAWAY 06

Significance is hard: non-IID traffic, sequential peeking, multi-metric correction, CUPED. Pick proper methods or wait for a clean stop time.

TAKEAWAY 07

Shadow mode has four rules: no real tools, no memory writes, separate budget, offline eval. Break any one and shadow becomes silent production.

TAKEAWAY 08

Rollback is a flag flip, not a redeploy. Hold old versions warm 24-72h. Practice the six-step sub-60s drill regularly.
