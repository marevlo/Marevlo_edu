QUERY: "My order ORD-7711 didn't arrive. What do I do?"

⟶ "The user's order didn't arrive. I need its shipping status before I can act. I'll call shipping_status."
STEP 02 — ACTION
⟶ shipping_status(order_id="ORD-7711")
STEP 03 — OBSERVATION
⟵ { status: "delayed", eta_days: 3, carrier: "DHL" }
STEP 04 — THOUGHT
⟶ "Order is delayed. I should check refund policy for delays > 2 days."
STEP 05 — ACTION
⟶ policy_lookup(topic="shipping_delay_refund")
STEP 06 — ANSWER
⟶ final_answer(answer="Your order ORD-7711 is delayed 3 days. Per policy, a partial refund of 15% is automatic — applied to your card within 24h.")
Key insight

the LLM never executes anything. It only emits structured intent. A deterministic runtime — your code — actually calls the API. This separation is what makes agents debuggable, sandboxable, and replayable.

Inside the ReAct Loop

What actually happens at runtime. State, messages, context window.

Message History Growth

Step 1:
[system, user_query]
Step 2:
[system, user_query, assistant(thought+action)]
Step 3:
[system, user_query, assistant(...), tool_result]
Step 4:
[system, user_query, assistant(...), tool_result, assistant(thought+action)]

Each loop appends 2 messages. After 10 steps, token count grows exponentially. This is the cost driver.

When Does It Stop?

✓ SUCCESS

LLM emits final_answer() with the result.

⚠ MAX STEPS

Counter reaches limit (usually 10–20). Return partial answer.

⏱ TIMEOUT

Wall-clock time exceeds budget. Abort gracefully.

✗ ERROR

Tool throws unrecoverable error. Escalate or return error message.

Context Window Pressure

The Problem: Each step appends to the message history. Long trajectories (8+ steps) can exhaust context, especially if tool results are verbose. Claude 3.5 has 200K tokens, but agents routinely hit the "we're at 95% capacity" wall by step 8.

STRATEGY 1: Summarize

After 5 steps, replace old steps with a one-line summary: "Earlier, we checked X and found Y. Moving on..."

STRATEGY 2: Filter

Only keep the last 3 steps + system prompt. Trades completeness for token efficiency.

STRATEGY 3: Checkpoint

At step 5, save the trajectory to a database. Start fresh with a new LLM call that references the checkpoint ID.

Common Failure Modes

Infinite Loop

LLM keeps calling same tool. Fix: track tool frequency, error on repetition.

Invalid Tool Calls

LLM invokes tool that doesn't exist. Fix: tight tool schema validation before dispatch.

Silent Failures

Tool returns null. LLM doesn't know how to react. Fix: always return a structured error message.

Hallucinated Results

LLM assumes tool result without waiting. Fix: very clear prompt: "Wait for OBSERVATION before reasoning."

Control Flow (Pseudocode)

while step_count < MAX_STEPS and not timeout:
  1. response = llm.invoke(messages, tools=[...])
  2. if response.type == "final_answer":
       return response.content
  3. if response.type == "tool_call":
       messages.append(response)
       4. tool_result = runtime.execute(response.tool_name, response.args)
       5. messages.append(tool_result)
       6. step_count += 1
       7. continue
  8. else:  // Invalid response
       raise ParseError

if timeout:
  return "Agent timeout. Partial work: " + summarize(messages)
if step_count == MAX_STEPS:
  return "Max steps reached." + messages[-1]

ReAct as a Pattern

Why it works. Why it's expensive. Why it sometimes fails.

What ReAct Guarantees

  • Determinism: Same input + same seed → same trace.
  • Auditability: Full trace preserved. Every step logged.
  • Observability: You see exactly what the LLM saw at each step.
  • Replayability: Feed old trace back to resume at any step.

The Cost Trade-off

  • Token Multiplier: Each step re-sends history. Step N costs ~O(N) tokens.
  • Latency: Sequential. No parallelism. 10 steps = 10 LLM roundtrips.
  • Unpredictability: Number of steps is emergent. Can't know cost upfront.
  • Timeout Risk: Slow tools can cause SLA violations.

Hard Constraints

  • Sequential Only: Tool N+1 must wait for N to complete.
  • Tool Visibility: LLM must "know" all tools upfront.
  • No Early Exit: Until final_answer is called, loop continues.
  • Tool Output Size: Huge results truncate or cause parsing errors.

ReAct vs. Plan-Then-Execute

ReAct (Step-by-Step)

Thought → Action → Observe → Thought → ...

  • ✓ Adaptive to surprises
  • ✓ Good for unknowns
  • ✗ High token cost
  • ✗ Slow (sequential)
Plan-Then-Execute

Thought → [Plan steps] → Execute all → Done

  • ✓ Cheap (1 planning call)
  • ✓ Fast (parallel execution)
  • ✗ Brittle to deviations
  • ✗ Plan can be wrong
04 / Tool Anatomy
What "Tools" Actually Mean to an LLM
A "tool" is not a function. To an LLM, it is a contract — described in natural language and JSON Schema — that promises a runtime will execute the call.
WHAT THE LLM SEES
{
  "name": "shipping_status",
  "description": "Resolve current shipping state of an order. Use when the user asks about delivery, lateness, or location of a package.",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "e.g. 'ORD-7711'"
      }
    },
    "required": ["order_id"]
  }
}
ANATOMY
NAME

A stable identifier the LLM emits when calling. Treat it as an API endpoint.

DESCRIPTION

Natural-language prose. This is the most important field. The LLM uses it to decide when to call.

PARAMETERS

JSON Schema. The LLM must produce arguments matching this shape — runtime validation rejects malformed calls.

RETURN (implicit)

Whatever the runtime gives back is serialized as text and fed into the next prompt. The LLM never executes — it only reads results.

THREE CONTRACTS A TOOL SCHEMA EMBODIES
01
SEMANTIC

The description teaches the LLM when this tool is the right pick.

02
SYNTACTIC

The parameters schema constrains arg shape. Validation runs before execution.

03
OPERATIONAL

The runtime promises to execute. The LLM trusts that promise; bugs there look like agent hallucinations.

07 / Decision Framework
When You Need an Agent — and When You Don't
Four questions. Answer them in order. If any answer is NO, you do not need an agent yet.
Q1 · Is the flow unknown at design time?
Can you draw the flowchart in advance?

If you can draw the path on a whiteboard before run-time, you have a workflow. Agents earn their cost only when the next step legitimately depends on what the model has just observed. Example: "summarize this PDF" is a workflow. "Triage this support ticket" — where the system might need a refund, a tech escalation, or just a knowledge-base lookup — is agent territory.

Q2 · Are there multiple tools to choose between?
Or is there one obvious next move every time?

Tool selection is the place where the LLM's judgment is most useful. A system with one tool is barely an agent — that's a function call. Three or more meaningfully different tools, where the right pick depends on context, is where the agentic abstraction pays for itself.

Q3 · Does the system need to iterate on its own output?
Try → check result → adjust → retry?

Looping on observations is the second core agent capability. Coding agents run tests, read failures, and re-edit. Research agents read a result, decide it's insufficient, and search again. If your workload never needs to react to its own intermediate results, a single LLM call probably suffices.

Q4 · Can you tolerate 10–100× the cost & latency?
Each loop iteration is another LLM call.

Agents accrue tokens at every step. A 6-step ReAct trajectory issues 6 LLM calls, each carrying the growing trajectory in context. If your latency budget is sub-second or your unit economics demand sub-cent queries, you must either downgrade to a workflow, route to a cheaper model, or rethink the product. Module 4.3 (Agent Cost Engineering) teaches the levers.

The pragmatic rule

Build the workflow first. Promote to agent only when the workflow demonstrably fails on real inputs.

Context Growth

Every observation gets appended to the prompt. By step 10, the agent is re-reading thousands of tokens it already processed.

Sequential Latency

ReAct is intrinsically sequential. Step N+1 cannot start until step N's observation arrives. There is no easy parallelism.

Looping Risk

A misbehaving agent can burn through your step budget retrying the same call. Hard step caps + circuit breakers are non-optional.
