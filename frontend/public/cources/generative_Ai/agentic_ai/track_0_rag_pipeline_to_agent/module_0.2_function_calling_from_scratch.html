02 / Tool / Function Schemas
Writing Schemas the LLM Trusts
A schema is a contract. Every field below is load-bearing. Get any of them wrong and the model will hallucinate, drop arguments, or pick the wrong tool.
tools/get_weather.json// LLM-FACING CONTRACT
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "Resolve the current weather for a city. Use whenever the user asks about temperature, conditions, rain, snow, or wind.",
    "parameters": {
      "type": "object",
      "properties": {
        "city": {
          "type": "string",
          "description": "City name. May include country, e.g. 'Bhopal, IN'."
        },
        "units": {
          "type": "string",
          "enum": ["celsius", "fahrenheit"],
          "default": "celsius"
        }
      },
      "required": ["city"]
    }
  }
}

Field Guide

NAME

Stable, snake_case, < 32 chars. The LLM emits this verbatim in tool_calls[i].function.name.

DESCRIPTION

Verbs + use-cases, not just nouns. Bad: "weather tool". Good: "Resolve current weather for a city. Use when the user asks about temperature…"

PARAMETERS

Always JSON Schema, type object. Use enum for closed sets — the model respects them. Use default for optional args.

REQUIRED

List the names that MUST be present. The runtime validates before calling. Anything not required is optional with a default.

DO

  • Describe when to call, not just what it does.
  • Constrain with enum where the answer is closed.
  • Include format hints in arg descriptions ("ISO-8601", "URL", "snake_case").
  • Keep the tool's effect idempotent when possible — the agent may retry.

DON'T

  • Bundle 5 unrelated capabilities into one tool. Split them.
  • Use ambiguous arg names like data or params.
  • Forget required — the model will fabricate args.
  • Expose internal IDs the user has no way to provide.

RULE OF THUMB

If a senior engineer can't pick the right tool from the registry by reading only the descriptions, the LLM can't either. Schemas are docs for a non-human reader who reads them every turn.

agent.py — the loop// PYTHON 3.11+
from openai import OpenAI
import json

client = OpenAI()

def run_agent(user_msg, tools, tool_impls, max_steps=10):
    messages = [
        {"role": "system", "content": "You are a careful assistant. Call tools when needed."},
        {"role": "user",   "content": user_msg},
    ]

    for step in range(max_steps):
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=tools,
        )
        msg = resp.choices[0].message
        messages.append(msg)   # preserve assistant turn

        if not msg.tool_calls:
            return msg.content        # terminal answer

        for call in msg.tool_calls:
            name = call.function.name
            args = json.loads(call.function.arguments)
            result = tool_impls[name](**args)
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": json.dumps(result),
            })

    raise RuntimeError("step budget exceeded")
L02

Single import. No agent framework anywhere.

L06

tool_impls is a dict[str, Callable]. Name → Python function. This is the bridge between LLM intent and real execution.

L13

The while-loop with a step budget. Every framework wraps this same construct.

L22

If the model emits no tool_calls, it's done. Return the natural-language answer.

L25

Iterate over tool_calls — there can be multiple. We'll exploit this in the parallel section.

L29

Append a role="tool" message with the same tool_call_id. This is how the model correlates request and response.

06 / Parallel Tool Calls
Parallel Tool Calls
Modern models emit several tool calls in one turn. Execute them concurrently and return all results before the next LLM call.
parallel.py// CONCURRENT EXECUTION
import asyncio

async def exec_calls(tool_calls, tool_impls):
    async def one(call):
        name = call.function.name
        args = json.loads(call.function.arguments)
        # run sync tool in thread pool
        result = await asyncio.to_thread(
            tool_impls[name], **args
        )
        return {
            "role": "tool",
            "tool_call_id": call.id,
            "content": json.dumps(result),
        }
    # all tool calls run concurrently
    return await asyncio.gather(
        *[one(c) for c in tool_calls]
    )
ONE TURN, MANY CALLS
get_weather(Bhopal)
get_weather(Delhi)
get_weather(Mumbai)
↓ concurrent

All three execute at once via asyncio.gather. Latency = slowest call, not sum.

SPEEDUP

3 sequential calls @ 400ms = 1200ms. Parallel = ~400ms. 3× faster.

07 / Error Handling
When Tools Go Wrong
Production tools fail. The agent must observe the failure and adapt — not crash. Return errors as tool results, not exceptions.
safe_exec.py// ERROR AS DATA
for call in msg.tool_calls:
    name = call.function.name
    try:
        args = json.loads(call.function.arguments)
        result = tool_impls[name](**args)
        status = "ok"
    except KeyError:
        result = {"error": f"unknown tool: {name}"}
        status = "error"
    except json.JSONDecodeError:
        result = {"error": "bad arguments JSON"}
        status = "error"
    except Exception as e:
        result = {"error": str(e)}
        status = "error"

    messages.append({
        "role": "tool",
        "tool_call_id": call.id,
        "content": json.dumps(result),
    })
FAILURE MODEMalformed JSON args

Model occasionally emits invalid JSON. Catch JSONDecodeError, return it as an observation. The model self-corrects next turn.

FAILURE MODETool throws exception

API down, timeout, bad input. Wrap in try/except, serialize the error. Never let it bubble up and kill the loop.

RECOVERYError becomes observation

The model sees {"error": ...} as a normal tool result and decides whether to retry, try a different tool, or report failure to the user.

08 / Why This Matters
Why Build It By Hand?
You will use frameworks after this module. But this is the one time you see the whole machine with the cover off.

Debug Anything

When LangChain throws a cryptic error at 2am, you'll know it's just a malformed tool message in the history.

Optimize Cost

You can see every token in the message history. You know exactly what you're paying for and where to trim.

Customize Freely

Custom retry logic, caching, logging, guardrails — trivial when you own the loop. Painful when fighting a framework.

The payoff

Frameworks are convenience, not magic. Every AgentExecutor, every create_react_agent, every graph node is a wrapper around the exact loop you just built. You now read framework source code as documentation, not mystery.
