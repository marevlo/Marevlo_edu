RAG-Course Recap

Pydantic is a type-driven validation library. You declare a class with typed fields; Pydantic auto-validates inputs and exports a JSON Schema. Already used in the RAG course for structured outputs, retrieval payloads, and config models.

recap.py — the four things you must remember
Pydantic v2
from pydantic import BaseModel, Field
from typing import Literal

# 1. Declare a model with typed fields + descriptions
class SearchQuery(BaseModel):
    query: str = Field(..., description="Search terms")
    top_k: int = Field(5, ge=1, le=50)
    source: Literal["docs", "web"] = "docs"

# 2. Validate dict → instance (raises if invalid)
q = SearchQuery.model_validate({"query": "agents", "top_k": 10})

# 3. Instance → dict
payload = q.model_dump()

# 4. The killer feature for this module: emit JSON Schema
schema = SearchQuery.model_json_schema()
# → { "type": "object", "properties": {...}, "required": [...] }

The Four Load-Bearing Primitives

BaseModel

Inherit from it. Every field declared on the class becomes a validated attribute.

Field()

Attach metadata: description, defaults, ge/le, regex. Descriptions flow into the LLM's tool schema.

model_validate()

Coerce + check a dict. Raises ValidationError on failure. We will catch this in the loop.

model_json_schema()

Auto-generates valid JSON Schema. This is the single feature that makes Pydantic the right tool-schema source.

The Pydantic-for-Agents Thesis

One Pydantic class = LLM schema + runtime validator + type-safe Python object. No drift. No duplication. No hand-written JSON.

tools_pydantic.py — single source of truth2 / 4 — typed, validated, compact
from pydantic import BaseModel, Field
from typing import Literal

class SearchDocs(BaseModel):
    """Search the documentation index.

    Use when the user asks about API behavior, function signatures,
    or how to use a feature.
    """
    query:  str = Field(..., description="Search terms")
    top_k:  int = Field(5, ge=1, le=50)
    source: Literal["docs", "web"] = "docs"

# That's it. The class IS the schema, the validator, AND
# the type-safe object the tool function consumes.
pydantic_tools.py — the conversion helper3 / 4 — 15 lines, reused forever
from pydantic import BaseModel
import inspect

def to_openai_tool(model: type[BaseModel]) -> dict:
    """Convert any Pydantic model into an OpenAI tool spec."""
    schema = model.model_json_schema()
    schema.pop("title", None)
    return {
        "type": "function",
        "function": {
            "name": model.__name__,
            "description": inspect.getdoc(model) or "",
            "parameters": schema,
        },
    }

# Usage
tools = [to_openai_tool(SearchDocs), to_openai_tool(GetWeather)]
resp = client.chat.completions.create(model="gpt-4o-mini", messages=..., tools=tools)
providers.py — same model, multiple envelopes4 / 4 — DRY across providers
def to_anthropic_tool(model: type[BaseModel]) -> dict:
    return {
        "name": model.__name__,
        "description": inspect.getdoc(model) or "",
        "input_schema": model.model_json_schema(),   # note: different key
    }

def to_gemini_tool(model: type[BaseModel]) -> dict:
    return {"function_declarations": [{
        "name": model.__name__,
        "description": inspect.getdoc(model) or "",
        "parameters": model.model_json_schema(),
    }]}

# Your Pydantic class never changes. The envelope is per-provider.
# This is exactly what frameworks like Instructor and LangChain do internally.
VALIDATION AT EXEC

When the LLM emits args, run SearchDocs.model_validate(args). Coerces types, enforces ranges, raises on bad args — before your tool function runs.

ZERO DRIFT

Rename a field, change a default, tighten an enum — schema regenerates automatically. The LLM and the Python function can no longer disagree.

IDE LOVES IT

Inside the tool function, args: SearchDocs gives full autocomplete on args.query, args.top_k. Refactors propagate automatically.

Discriminated Union — Force a Branch Choice

Make the LLM pick one of N variants, each with its own arg shape. When an action has variants — e.g. "issue refund" can be partial or full with different required args — discriminated unions force the LLM to pick a variant and supply only the args for that variant.

from typing import Literal, Annotated, Union
from pydantic import BaseModel, Field

class PartialRefund(BaseModel):
    kind: Literal["partial"]
    order_id: str
    amount_pct: int = Field(..., ge=1, le=99)
    reason: str

class FullRefund(BaseModel):
    kind: Literal["full"]
    order_id: str
    requires_approval: bool = True

class IssueRefund(BaseModel):
    refund: Annotated[Union[PartialRefund, FullRefund], Field(discriminator="kind")]
# LLM now must choose 'partial' or 'full' and supply only the matching fields.

Nested Models — Structured Sub-Objects

Express richer arg shapes than a flat dict — addresses, date ranges, filters. For tools whose arguments contain logical sub-objects, nest them. The LLM produces a nested JSON dict; Pydantic validates the whole tree in one shot.

class DateRange(BaseModel):
    start: str = Field(..., description="ISO-8601 date, e.g. 2026-05-01")
    end:   str = Field(..., description="ISO-8601 date")

class SearchOrders(BaseModel):
    """Find orders matching a customer and date range."""
    customer_id: str
    range: DateRange
    statuses: list[Literal["open", "shipped", "delivered"]] = []

# LLM emits: {customer_id, range: {start, end}, statuses: [...]}
# Pydantic validates the whole tree in one call.

Defaults + Validators — Cheap Guardrails

Move policy into the schema. The agent can't bypass it. Use field_validator to enforce business rules at the boundary. If the LLM violates them, the validator raises — and the error becomes an observation the model can correct on.

from pydantic import BaseModel, Field, field_validator

class SendEmail(BaseModel):
    """Send an outbound email. Internal addresses ONLY."""
    to: str = Field(..., description="Recipient email")
    subject: str = Field(..., max_length=120)
    body: str = Field(..., max_length=5000)
    priority: Literal["low", "normal", "high"] = "normal"

    @field_validator("to")
    def internal_only(cls, v: str) -> str:
        if not v.endswith("@acme.com"):
            raise ValueError("Recipient must be an @acme.com address")
        return v.lower()
# Bypass-proof: the agent simply cannot send external email through this tool.

Result Models — Type the Return Too

The tool's output becomes the LLM's next observation. Shape it explicitly. Tool results feed back into the LLM context. A typed result model makes the observation predictable, parseable, and easy to evaluate later.

class SearchResult(BaseModel):
    doc_id: str
    title: str
    score: float = Field(..., ge=0, le=1)

class SearchResults(BaseModel):
    hits: list[SearchResult]
    total: int

def run_search(args: SearchDocs) -> SearchResults:
    raw = vector_db.query(args.query, k=args.top_k)
    return SearchResults(
        hits=[SearchResult(doc_id=r.id, title=r.title, score=r.score) for r in raw],
        total=len(raw),
    )
# Serialize with .model_dump_json() before appending to messages.

async_batch.py — run N trajectories concurrently
asyncio.gather + Pydantic
import asyncio
from openai import AsyncOpenAI
from pydantic import ValidationError

aclient = AsyncOpenAI()

async def run_one(user_msg, tools_spec, tool_models, tool_impls, max_steps=8):
    messages = [{"role": "user", "content": user_msg}]

    for _ in range(max_steps):
        resp = await aclient.chat.completions.create(
            model="gpt-4o-mini", messages=messages, tools=tools_spec)
        msg = resp.choices[0].message
        messages.append(msg)

        if not msg.tool_calls:
            return msg.content

        async def _exec(call):
            model = tool_models[call.function.name]
            try:
                args = model.model_validate_json(call.function.arguments)
                result = await tool_impls[call.function.name](args)
                payload = result.model_dump()
            except ValidationError as e:
                payload = {"error": "validation", "detail": e.errors()}
            return {"role": "tool", "tool_call_id": call.id, "content": str(payload)}

        results = await asyncio.gather(*[_exec(c) for c in msg.tool_calls])
        messages.extend(results)

# Run 100 trajectories at once
async def batch(queries):
    return await asyncio.gather(*[run_one(q, ...) for q in queries])

answers = asyncio.run(batch(my_100_queries))
PARALLEL INNER LOOP

Inside one trajectory: when the LLM emits N tool calls, they run with asyncio.gather. Already familiar from Module 0.2.

PARALLEL OUTER LOOP

Across many trajectories: batch() dispatches all trajectories concurrently. This is the agent-workflow level.

VALIDATION FIRST

model_validate_json happens before awaiting the tool. Bad args become a typed error observation in microseconds.

RATE LIMITS

Gating with asyncio.Semaphore(20) is non-optional once you cross provider TPM/RPM. Cap concurrency.

06 / BAML for Production
When Pydantic Isn't Enough
For production agent pipelines, BAML (Boundary AI Markup Language) lifts schema + prompt + provider config into a typed DSL. You still write Pydantic-like models — but the compiler owns the LLM contract.
Prototype → Mid-scale

Pydantic — In-Language

  • ✓ Lives in your Python. Zero learning curve for any Python dev.
  • ✓ Schema, validator, type — all one class.
  • ✗ Prompt strings still live in Python; drift between schema + prompt is your problem.
  • ✗ Provider switching = adapter code. Each model's quirks leak through.
  • ✗ No first-class test harness for prompt regressions.
Production Pipelines

BAML — DSL + Codegen

  • ✓ Schema, prompt template, and provider config — versioned together in one .baml file.
  • ✓ Compiler emits typed clients per language (Python, TS, Ruby).
  • ✓ Provider-agnostic — swap OpenAI ↔ Anthropic by changing one line.
  • ✓ Built-in test harness for prompts. Run cases in CI.
  • ✗ Extra build step. Team must learn a small DSL.
agents.baml — schema + prompt + client in one file
Versioned · Typed · Provider-agnostic
// 1. Declare data types (the Pydantic equivalent)
class RefundDecision {
  kind         RefundKind
  order_id     string
  amount_pct   int?  @description("1-99 for partial, null for full")
  reason       string
}
enum RefundKind { Partial Full Reject }

// 2. Declare the agent function — schema + prompt + client all here
function DecideRefund(ticket: string) -> RefundDecision {
  client GPT4oMini
  prompt #"
    You are a support agent. Given the ticket, decide a refund action.
    Return ONLY a RefundDecision object.

    Ticket:
    {{ ticket }}

    {{ ctx.output_format }}
  "#
}

// 3. CI-grade tests, in the same file
test partial_refund_for_late_delivery {
  functions [DecideRefund]
  args { ticket "Order #7711 arrived 5 days late..." }
  @assert({{ this.kind == "Partial" }})
}
TIER 01 — AUTHOR

Write .baml files. Declare types, prompts, providers, tests side-by-side. Single source of truth per agent function.

TIER 02 — COMPILE

Run baml-cli generate. Emits typed client code (Pydantic models + thin client class) per language.

TIER 03 — CONSUME

Import the generated client; call b.DecideRefund(ticket=...) exactly like a regular Python function. Fully typed return.

Migrate to BAML When

  • ✓ You have 10+ agent functions across multiple services.
  • ✓ You need to swap providers or A/B-test models frequently.
  • ✓ Multiple languages consume the same agent contracts.
  • ✓ Prompt regression tests must run in CI (covered in Module 4.5).

Stay on Pydantic When

  • ✗ Single Python service, <5 tools, no provider churn.
  • ✗ Team aversion to extra build steps or learning a DSL.

08 / Recap
Key Takeaways
  1. One Pydantic class = LLM schema + runtime validator + typed Python object. Eliminates drift between the three.
  2. Conversion is a ~15-line helper. Each provider gets its own envelope; the model stays untouched.
  3. Four patterns: discriminated unions, nested models, defaults+validators, typed result models. Each closes a specific failure mode.
  4. Two-level async: gather() inside a trajectory, gather() across trajectories. ~10× throughput uplift.
  5. BAML earns its complexity when you have 10+ agents, multi-language consumers, or in-CI prompt regression tests.
  6. Everything generic about JSON Schema, Pydantic basics, structured outputs, Jinja2, CI/CD stays in the RAG course. This track was deliberately slim.
