02 / 3-Min MCP Recap
What You Already Built
You completed the MCP course. This is a fast callback to lock in the vocabulary — not a re-teach. If anything below feels unfamiliar, swing back before continuing.
3-Min Callback

From the MCP course: MCP is a protocol that lets an LLM client talk to a server exposing tools, resources, and prompts — over JSON-RPC 2.0, with three transports (stdio, streamable HTTP, SSE). You shipped a server with the Python SDK using @mcp.tool() decorators. All of that stays there. This module focuses only on the agent's side of the connection.

MCP SERVER (recap)
  • Exposes Tools (callable), Resources (readable data), Prompts (templates).
  • Speaks JSON-RPC over stdio / HTTP / SSE.
  • Declares its capabilities at handshake.
  • Python SDK: FastMCP + @mcp.tool().
MCP CLIENT (the agent)
  • Discovers a server's tools via tools/list.
  • Calls them via tools/call.
  • Reads resources, fetches prompt templates.
  • This module: client = your LangGraph agent.
The Reframe

MCP course: "how do I build a server?"
This module: "how does my agent use one?"

03 / Why MCP Matters for Agents
M×N → M + N
Without a protocol: every agent framework writes its own adapter for every tool service. That's M×N integrations. MCP collapses it to M servers + N clients — anyone speaks to anyone.
Chaos

Before — M × N

Every framework writes its own adapter for every service. 4 × 4 = 16 integrations. Add a service or framework → quadratic explosion.

Protocol

With MCP — M + N

Each side implements MCP once. 4 + 4 = 8 connections. Add a service → 1 new integration. Add a framework → 1 new client.

ECOSYSTEM LEVERAGE

By 2026 there are hundreds of community MCP servers (GitHub, Postgres, Slack, Filesystem, Notion, Linear, Stripe…). Your agent gets them for free.

FRAMEWORK-NEUTRAL

A tool exposed as an MCP server works in Claude Desktop, Cursor, your LangGraph agent, and competitors — without rewriting.

SECRETS BOUNDARY

The server holds credentials, not the agent. API keys, OAuth tokens, DB strings — all stay in the server process.

04 / The Three MCP Primitives
Tools · Resources · Prompts
MCP servers expose three things. Each maps to a familiar agent concept you've already used.
01

TOOLS

MODEL-INVOKED · SIDE EFFECTS

Callable actions the LLM picks. tools/call via JSON-RPC. Maps 1-to-1 with LangChain's @tool.

// create_issue, run_query, send_message
02

RESOURCES

APP-CONTROLLED · READ-ONLY

Readable data the application (or agent) injects into context. Like a retriever, but address-based — URIs not queries.

// file://repo/README.md · postgres://schema/users
03

PROMPTS

USER-INVOKED · TEMPLATES

Parameterized prompt templates the server provides. Maps to Module 2.1's ChatPromptTemplate — but discoverable across the ecosystem.

// /review_pr · /summarize_ticket
Mapping to your existing toolbox
Reference
MCP Primitive Controlled By Equivalent in Our Stack
Tools Model decides @tool (Module 1.0, 2.5)
Resources Application decides Retrieved docs / fetched files (Module 2.5)
Prompts User triggers ChatPromptTemplate (Module 2.1)
For Track 2 Purposes

You'll spend ~95% of your MCP time on the Tools primitive. Resources and Prompts are useful but optional — Tools is what makes an MCP server actionable from an agent.

connect_filesystem.py — one MCP server, stdio transport1 / 3
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client
from langchain_mcp_adapters.tools import load_mcp_tools

# Server config — points at any existing MCP server binary/package.
server = StdioServerParameters(
    command="npx",
    args=["-y", "@modelcontextprotocol/server-filesystem", "/workspace"],
)

async with stdio_client(server) as (read, write):
    async with ClientSession(read, write) as session:
        # Handshake — server announces its capabilities
        await session.initialize()

        # Discover + convert to LangChain BaseTool instances
        tools = await load_mcp_tools(session)
        print([t.name for t in tools])
        # → ['read_file', 'write_file', 'list_directory', 'search_files', ...]

        # Now `tools` works in ANY pattern from Module 2.4 / 2.5.
connect_multi.py — many servers at once2 / 3 · MultiServerMCPClient
from langchain_mcp_adapters.client import MultiServerMCPClient

client = MultiServerMCPClient({
    "filesystem": {
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"],
        "transport": "stdio",
    },
    "github": {
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-github"],
        "env": {"GITHUB_PERSONAL_ACCESS_TOKEN": os.environ["GH_TOKEN"]},
        "transport": "stdio",
    },
    "postgres": {
        "url": "http://mcp-postgres.internal:9000/mcp",
        "transport": "streamable_http",    # remote MCP server
    },
})

async with client:
    tools = await client.get_tools()
    # tools is a flat list across ALL servers.
    # Names are namespaced: 'filesystem__read_file', 'github__create_issue', etc.
graph.py — MCP tools into LangGraph ReAct3 / 3 · Module 2.4 pattern unchanged
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI

# `tools` came from MultiServerMCPClient above.
model = ChatOpenAI(model="gpt-4o-mini").bind_tools(tools)

def agent_node(state):
    return {"messages": [model.invoke(state["messages"])]}

tool_node = ToolNode(tools)

def route(state):
    return "tools" if state["messages"][-1].tool_calls else "__end__"

builder = StateGraph(AgentState)
builder.add_node("agent", agent_node)
builder.add_node("tools", tool_node)
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route, {"tools": "tools", "__end__": END})
builder.add_edge("tools", "agent")
graph = builder.compile()

# Agent now has filesystem + GitHub + Postgres tools — all via MCP.
# Zero custom integration code. Add a new server → add 5 lines of config.
YOUR AGENT

LangGraph + langchain-mcp-adapters. The pattern above. Most flexible — every MCP server, your control flow.

CLAUDE DESKTOP / CURSOR

Native MCP hosts. Point them at the same server config — they auto-discover tools.

CLAUDE CODE

CLI agent. Same servers, same tools, no rewrite. This is the cross-host portability MCP enables.

06 / Exposing Your Agent's Tools
Wrap Your Tools as an MCP Server
The other half of the bridge. Make your internal tools consumable by Claude Desktop, Cursor, other agents — without writing N adapters.
mcp_server.py — expose your internal tools
FastMCP (Python SDK)
from mcp.server.fastmcp import FastMCP
from pydantic import BaseModel, Field

mcp = FastMCP("acme-internal")

# Anything you'd normally write with LangChain's @tool — same shape.
@mcp.tool()
def search_acme_docs(query: str, top_k: int = 4) -> list[dict]:
    """Search Acme Cloud's internal product documentation."""
    return docs_retriever.invoke(query)[:top_k]

@mcp.tool()
def create_support_ticket(title: str, body: str, priority: str) -> dict:
    """Open a new support ticket. Returns {ticket_id, url}."""
    return support_api.create(title=title, body=body, priority=priority)

# A resource — read-only data, addressable by URI.
@mcp.resource("acme://policies/{policy_id}")
def get_policy(policy_id: str) -> str:
    """Return the full text of a policy by ID."""
    return policies_db.get(policy_id).text

# A prompt — parameterized template the server provides.
@mcp.prompt()
def triage_ticket(ticket_id: str) -> str:
    """Standard triage prompt: severity + owner team + ETA."""
    return f"Triage ticket {ticket_id}: severity (P0–P3), owning team, ETA."

if __name__ == "__main__":
    mcp.run(transport="stdio")   # or "streamable_http" for remote
Pick a transport
Reference
Transport When Pros Cons
stdio Local · single-user · IDE plugins Simple, secure, no network Not remotable
streamable HTTP Remote · multi-user · production Standard infra, auth, scaling Need TLS + auth setup
SSE (legacy) Older deployments Browser-friendly streaming Superseded by HTTP transport
The Two-Way Door

Your agent is a consumer of community MCP servers, and a producer of MCP services for other agents. This bidirectionality is what makes MCP an ecosystem, not just a protocol.

What's Out There Already
Hundreds of community MCP servers. A representative sample of what your agent can plug into today, without writing any integration code.

Issues, PRs, comments, file contents, search.

Postgres

Slack

Send/search messages, channel history, user lookup.

Pages, databases, blocks — full read/write.

Issues, projects, cycles — full workflow control.

Puppeteer / Playwright

Browser automation as a tool — for agents that do web tasks.

AWS / GCP / Azure

Cloud APIs surfaced as tools — provisioning, monitoring.

A Word on Trust

MCP servers run code on your machine (stdio) or on a server you trust (HTTP). Audit before adding. Treat them like browser extensions — review the source, scope their permissions (e.g. read-only filesystem roots), and limit what credentials you hand over. Module 2.11 (Memory · Guardrails · Safety) goes deeper on agent-side defenses.
