03 / As an Agent Pattern
A Control Pattern, Not a Doc Pattern
In MapReduce-the-distributed-system, the input was always a dataset. In MapReduce-the-agent-pattern, the input is any list of independent things the LLM should reason about in parallel.
SUPPORT

Triage 200 open tickets → group by theme, summarize each cluster.

REVIEWS

Analyze 1k product reviews → executive sentiment + top-N issues.

RESEARCH

A research agent fans 1 question into 8 sub-questions → fuses answers.

PLANNING

Generate 5 candidate plans → score each → pick or merge the best.

MAP STEP

One LLM call per input, run in parallel. Each produces a typed partial result (Pydantic, from Module 1.0).

REDUCE STEP

One LLM call that receives the list of partials and produces the final combined output. The hard step — coherence lives here.

04 / Stuff / Map-Reduce / Refine
Three Strategies for Many Inputs
Same problem — process N items, produce 1 output — three control-flow shapes. Each has a sweet spot and a failure mode.
STUFF 1 LLM CALL

Concatenate all N items into one giant prompt. One model call. Fastest, simplest, cheapest when it fits.

CALLS1
LATENCYlow
COST$
COHERENCEbest
FAILS WHENN × size > ctx
USE WHEN

Inputs are small AND total tokens fit comfortably in context with budget for output.

MAP-REDUCE N + 1 CALLS

One call per input (parallel) → one combining call. Scales horizontally. The general-purpose answer.

CALLSN + 1
LATENCY~map + reduce
COST$$
COHERENCEdepends on reduce
FAILS WHENN partials > ctx
USE WHEN

Inputs are independent of each other (no cross-references) and you want speed via parallelism.

REFINE N CALLS · SEQUENTIAL

Process input 1 to produce draft 1. Feed input 2 + draft 1 → draft 2. Repeat. The answer evolves.

CALLSN
LATENCYhigh (serial)
COST$$
COHERENCEexcellent
FAILS WHENdrift on long chains
USE WHEN

Order matters, later inputs build on earlier conclusions, or you need a single evolving narrative.

Latency Profile — 8 Inputs
Comparison
STUFF · ~ 1 × call_time
MAP-REDUCE (parallel map) · ~ 1 map + 1 reduce
REFINE (serial) · ~ 8 × call_time
Decision Flow

Fits in context? → STUFF. Independent items? → MAP-REDUCE. Order or evolution matters? → REFINE.

05 / Two-Level Map-Reduce
When N is Too Big
If 200 partial outputs won't fit into one reduce prompt — group them. Reduce each group. Reduce the groups. Hierarchical map-reduce in two levels.

SIZING THE FAN-OUT

Group size k = how many partials fit comfortably into one reduce prompt. For most production models:

  • Partials ~ 400 tokens each → k ≈ 12–20 per reduce
  • Partials ~ 1k tokens each → k ≈ 6–10 per reduce
  • Leave 30%+ context budget for the reduce prompt + output
WHEN TO GO DEEPER

If even the meta-reduce overflows — add another level. The recursion bottoms out at the root. In practice:

  • 2 levels handles N up to ~ k² (≈ 100–400)
  • 3 levels handles N up to ~ k³ (≈ 1k–10k)
  • Past that, you have a streaming / batch-job problem — different architecture
two_level_mr.py — hierarchical reduce
LCEL · Async · Typed
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
import asyncio, math

model = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# Per-item: map prompt
map_prompt = ChatPromptTemplate.from_messages([
    ("system", "Extract issues + sentiment from one review."),
    ("human",  "{item}"),
])
map_chain = map_prompt | model | map_parser

# Group reduce — combines k partials
group_prompt = ChatPromptTemplate.from_messages([
    ("system", "Synthesize a group summary from these partials."),
    ("human",  "{partials}"),
])
group_chain = group_prompt | model | group_parser

# Meta reduce — final answer from group summaries
meta_chain = meta_prompt | model | meta_parser

async def two_level(items: list[str], k: int = 10):
    # LEVEL 0 → 1: parallel map
    partials = await map_chain.abatch([{"item": x} for x in items],
                                       config={"max_concurrency": 16})

    # LEVEL 1 → 2: group + parallel group-reduce
    groups = [partials[i:i+k] for i in range(0, len(partials), k)]
    summaries = await group_chain.abatch(
        [{"partials": g} for g in groups],
        config={"max_concurrency": 8},
    )

    # LEVEL 2: single meta-reduce
    return await meta_chain.ainvoke({"summaries": summaries})
06 / Narrative Coherence
Map-Reduce's Defining Failure Mode
When you reduce partials produced in isolation, the output reads like a stapled-together pile. The fix is not better prompting — it is structural anchoring.
Uncontrolled MR

❌ Incoherent Splice

"Users praised performance. Latency was a problem in some regions. The pricing tier is competitive. Several customers asked about onboarding. Bug reports mentioned crashes. Pricing came up again..."

FAILURE: repeated topics, no progression, no synthesis. The reduce step concatenated rather than reasoned.

Structured MR

✓ Anchored Output

"Themes (ranked): 1) Performance (positive, 142 mentions); 2) Pricing (mixed, 87); 3) Onboarding gaps (negative, 64). Action items: latency audit in APAC region; refresh onboarding tutorial."

WORKS BECAUSE: shared schema across map outputs + reduce prompt that aggregates fields, not text.

01 SHARED SCHEMA

Every map call returns the same Pydantic model. The reduce step receives a typed list — not free text. Aggregating fields beats concatenating prose.

02 PROPAGATED OUTLINE

First, run a quick pass to extract a global outline (themes, sections). Pass that outline into every map call so partials slot into a shared scaffold.

03 ANCHORING REDUCE

Reduce prompt explicitly demands ranking, dedup, synthesis verbs ("compare", "rank", "merge") — not "summarize". Verbs guide structure.

07 / Async Hierarchical Reduction
Concurrency, Properly Gated
Async makes hierarchical map-reduce fly — but uncapped fan-out will trip rate limits. The right control = parallel within a level, gated by semaphore, serialized across levels.

async_hierarchy.py — gated parallelism
Semaphore per level
import asyncio
from typing import Awaitable

async def gated_gather(coros: list[Awaitable], limit: int):
    """Run coros concurrently but no more than `limit` at once."""
    sem = asyncio.Semaphore(limit)
    async def _run(coro):
        async with sem:
            return await coro
    return await asyncio.gather(*[_run(c) for c in coros])

async def hierarchical_mr(items, k=10, map_limit=16, group_limit=4):
    # Level 0 → 1: gated parallel map
    partials = await gated_gather(
        [map_chain.ainvoke({"item": x}) for x in items],
        limit=map_limit,
    )

    # Level 1 → 2: gated parallel group-reduce
    groups = [partials[i:i+k] for i in range(0, len(partials), k)]
    summaries = await gated_gather(
        [group_chain.ainvoke({"partials": g}) for g in groups],
        limit=group_limit,
    )

    # Level 2: single meta-reduce
    return await meta_chain.ainvoke({"summaries": summaries})
GATE PER LEVEL

Each level gets its own semaphore. Map fan-out is wide (16+); group reduce is narrower (4–8); meta-reduce is 1.

WATCH RATE LIMITS

Provider TPM (tokens/min) is the real ceiling. Compute it: (map_limit × ~tokens_per_call) ≤ TPM / 60.

WALL TIME LAW

Total ≈ map_time + group_time + meta_time. Each term is set by its slowest call, not its count — if gating is correct.
