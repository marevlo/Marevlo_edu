02 / ChatPromptTemplate
The Canonical Template Class
Every prompt in this course flows through ChatPromptTemplate. It accepts a list of role-tagged templates, exposes .partial() for pre-filling, and returns a ChatPromptValue piped to the model.
bad.py — fails at scale

f-string in app code

def build_prompt(question, persona):
    return f"""You are {persona}.
    Answer: {question}
    """

# ❌ No type checking on variables.
# ❌ No partial / reusable subprompts.
# ❌ Cannot be traced or versioned by LangSmith.
# ❌ Role boundaries lost — model just sees one blob.
good.py — LCEL-native

ChatPromptTemplate

from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are {persona}. Be concise."),
    ("human",  "{question}"),
])

# ✓ Roles preserved as a list of BaseMessage
# ✓ Variables are tracked — input_variables = {persona, question}
# ✓ Plays nicely with .partial(), .stream(), LangSmith.

chain = prompt | model | parser
VARIABLE TRACKING

prompt.input_variables tells you exactly which keys the chain expects. Missing one = clear error at chain definition, not runtime.

PARTIAL APPLICATION

prompt.partial(persona="...") pre-fills one variable and returns a new template with one fewer input. Use for format instructions, tenant configs, system prompts.

COMPOSABLE

A template is a Runnable. Pipe it. Batch it. Cache it. Trace it. The same machinery from Module 2.0 applies.

partial.py — pre-fill format instructions once
Reuse across requests
from langchain_core.output_parsers import PydanticOutputParser

parser = PydanticOutputParser(pydantic_object=SentimentReport)

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a sentiment analyst.\n{format_instructions}"),
    ("human", "{review}"),
])

# Pre-fill format_instructions once at module load.
# prompt.input_variables now = {"review"} only.
prompt = prompt.partial(format_instructions=parser.get_format_instructions())

chain = prompt | model | parser
chain.invoke({"review": "Fast and friendly."})    # only one input now
03 / System / Human / Assistant
Role-Tagged Messages
Chat-tuned models attend differently to each role. Mixing them up is one of the most common prompt-engineering mistakes — and one the LCEL templates make impossible.
SYSTEM

Identity & Standing Rules

Persistent context the model treats as constant: persona, capabilities, constraints, format rules, refusal policies.

USE FOR:
• Persona & tone
• Output format contracts
• Hard constraints / refusals
• Tool-use guidance
HUMAN

The Actual Request

The query, task, or input under processing. This is what changes per turn. The variable-heavy role.

USE FOR:
• User question
• Document to summarize
• Data to extract
• Conversation context
ASSISTANT

Prior Model Turns

What the model said before. Drives few-shot exemplars, multi-turn conversations, and tool-call traces.

USE FOR:
• Few-shot AI responses
• Prior assistant turns
• Tool-call traces
• Prefilling output start
roles.py — every role used correctly
Multi-turn agent prompt
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    # SYSTEM — identity, format, and rules (persistent)
    ("system", "You are an order-resolution agent. Always cite a policy ID."),

    # HUMAN — prior turn from the user
    ("human", "My order #7711 hasn't arrived."),

    # ASSISTANT — model's previous turn (kept in context)
    ("ai", "Let me check the shipping status of ORD-7711."),

    # HUMAN — the new request now under processing
    ("human", "{new_request}"),
])

# Note the role keys: "system", "human", "ai".
# Templates accept tuples (role, text) OR explicit message classes.
PITFALL 01
Stuffing the user's query into the system message. Chat models down-weight inputs that look like rules — your task becomes invisible.
PITFALL 02
Skipping the assistant role for few-shot examples. Without it, the model can't tell where its turn begins and parrots back the example text.
PITFALL 03
Multiple consecutive system messages. Most provider APIs collapse them silently — your second system prompt vanishes.
04 / Few-Shot with Templates
FewShotChatMessagePromptTemplate
Hand-concatenating examples into a string is fragile. The framework gives you a typed exemplar list + an example-formatting template = clean injection at runtime.
Few-shot anatomy
Diagram
A. EXAMPLES (LIST OF DICTS)
EX 01
input: "It works!"
output: "positive"
EX 02
input: "Broken."
output: "negative"
EX 03
input: "It's fine."
output: "neutral"
+
EXAMPLE PROMPT
B. PER-EXAMPLE TEMPLATE
ChatPromptTemplate.from_messages([
    ("human", "{input}"),
    ("ai", "{output}"),
])

Each row in A is formatted by B → produces N×2 messages threaded into the final prompt.

few_shot.py — end-to-end
FewShotChatMessagePromptTemplate
from langchain_core.prompts import (
    ChatPromptTemplate, FewShotChatMessagePromptTemplate,
)

# 1) Exemplar bank — pure data
examples = [
    {"input": "It works flawlessly.", "output": "positive"},
    {"input": "Crashed twice today.",  "output": "negative"},
    {"input": "It's fine, nothing special.", "output": "neutral"},
]

# 2) How each example becomes a (human, ai) pair
example_prompt = ChatPromptTemplate.from_messages([
    ("human", "{input}"),
    ("ai",    "{output}"),
])

# 3) The few-shot block
few_shot = FewShotChatMessagePromptTemplate(
    example_prompt=example_prompt,
    examples=examples,
)

# 4) Final composition — system + few-shot block + real input
prompt = ChatPromptTemplate.from_messages([
    ("system", "Classify product reviews: positive / neutral / negative."),
    few_shot,
    ("human", "{review}"),
])

chain = prompt | model | parser    # pipes like anything else
STATIC FEW-SHOT

A fixed exemplar list at module load. Use when your task is narrow and a small handful covers it. Cheap, deterministic, easy to version.

DYNAMIC (SEMANTIC) FEW-SHOT

Use an ExampleSelector to retrieve the most relevant K exemplars per query — embed once, search at runtime. Returns in Module 2.5 (RAG-as-tool).

05 / Chain-of-Thought Patterns
Make Models Think Out Loud
Asking the model to produce reasoning steps before the final answer lifts accuracy on multi-step tasks — at the cost of tokens and latency. Three reliable patterns to know.

ZERO-SHOT COT

+ 1 LINE

Append "Let's think step by step." to the system prompt. Free, often surprisingly effective on math & reasoning.

("system",
 "... Let's think step by step before answering.")

FEW-SHOT COT

EXEMPLARS WITH REASONING

Show worked examples — each AI turn includes the reasoning trace, not just the answer. Combines § 4 + this pattern.

{"input": "23 + 19?",
 "output": "23 + 19 = 23 + 20 - 1 = 42"}

STRUCTURED COT

PYDANTIC FIELDS

Add a reasoning field to your output schema. Forces the model to produce reasoning + answer in one parseable object.

class Answer(BaseModel):
    reasoning: str
    final: str
Structured CoT trace
Trace
[REASONING.1] Customer reports "didn't arrive" → first action: resolve order_id from CRM.
[REASONING.2] Order ORD-7711 exists. Status = delayed by 3 days. Tier = gold.
[REASONING.3] Policy SHIP-014 grants 15% auto-refund on delays ≥ 2 days for gold tier.
[FINAL] Apply 15% refund per SHIP-014. Notify customer.

Why this beats free-form CoT: the reasoning lives in a typed field. You can log it, eval it, redact it from the user-facing answer, and run regression tests against it. The next module reuses this exact pattern.

Accuracy vs Cost per Pattern

CoT pays off for multi-step reasoning. On simple classification or extraction, it is pure overhead. Pick per task.

Pattern Accuracy (% on reasoning tasks) Tokens per response (×)
No CoT621.0×
Zero-Shot CoT781.8×
Few-Shot CoT842.3×
Structured CoT862.6×
06 / 5 Components of a Production Prompt
The Skeleton of Every Real Prompt
Once you ship more than two prompts, write them with this 5-component structure. Each component has a stable role — and a stable home in the template.
01
ROLE

Who the model is. Persona + tone + standing capabilities.

02
TASK

What it must do this turn. One concrete objective.

03
CONSTRAINTS

What it must NOT do. Refusals, length limits, safety.

04
FORMAT

Exact output shape. JSON Schema, fields, ordering.

05
EXAMPLES

1–3 worked input/output pairs. Few-shot from § 4.

production_prompt.py — all 5 in one template
Annotated
prompt = ChatPromptTemplate.from_messages([
    # [01 ROLE] Identity + capabilities (persistent)
    ("system",
     "You are a senior support agent for Acme Cloud. "
     "You have access to the order, billing, and shipping APIs.\n\n"

     # [02 TASK] One sentence — what this prompt must accomplish
     "TASK: Triage the customer ticket below and decide a next action.\n\n"

     # [03 CONSTRAINTS] Hard rules. Be explicit.
     "CONSTRAINTS:\n"
     "- Do not promise refunds > 50% without HITL approval.\n"
     "- Never expose internal IDs or stack traces to the customer.\n"
     "- If unsure, escalate.\n\n"

     # [04 FORMAT] Pydantic output parser instructions
     "FORMAT:\n{format_instructions}"),

    # [05 EXAMPLES] Few-shot block from § 4
    few_shot,

    # Real user input
    ("human", "{ticket}"),
]).partial(format_instructions=parser.get_format_instructions())
VERSIONABLE

Five named blocks = clean diffs. Reviewers can argue about constraints without re-reading the whole prompt.

REUSABLE

Role + Constraints often hold steady across tasks for the same product. Pull them into a shared helper.

TESTABLE

Track 3 (Eval) runs regression tests by holding 4 components fixed and varying 1. Impossible without this structure.

07 / 6 Prompt Types in LCEL
The Full Template Catalog
LCEL ships six prompt-template classes. Each one solves a distinct problem. You'll touch all of them across this course.
01
PromptTemplate
f-string style · returns str

Single-string templates for completion (non-chat) models or as a sub-component in a chain. Predecessor of ChatPromptTemplate. Use only for legacy completion APIs.

02
ChatPromptTemplate
role-tagged · returns messages

The default. Section 2. Use this for every chat-tuned model — which is essentially every model worth shipping in 2026.

03
FewShotChatMessagePromptTemplate
exemplar injector · composes inside template

Section 4. Embeds N input/output exemplars into a parent ChatPromptTemplate. Pair with an ExampleSelector for dynamic retrieval.

04
MessagesPlaceholder
slot for chat history / tool messages

Reserves a slot in the template where a list of messages — chat history, tool-call traces — gets spliced in at runtime. Foundational for memory and ReAct loops. Returns in Module 2.4 (LangGraph).

05
PipelinePromptTemplate
compose templates from sub-prompts

Build a final prompt by piping together named sub-prompts. Useful when the 5 components live in separate files / get owned by different teams. Less common than the others — reach for it only when composition pressure is real.

06
StructuredPrompt
schema-bound · pairs with model.with_structured_output

A template that knows its output schema. Pairs with model.with_structured_output(Schema) so chains return validated Pydantic objects directly — no separate parser. The convergence of Module 1.0 and prompt templating.

Pick Order in Practice

Start with ChatPromptTemplate · add FewShot when accuracy needs it · reach for MessagesPlaceholder for memory · use StructuredPrompt in production.
