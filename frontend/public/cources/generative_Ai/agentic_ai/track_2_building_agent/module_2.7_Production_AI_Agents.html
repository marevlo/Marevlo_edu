04 / Pillar 1: State Management
From MemorySaver to Production
Module 2.4 introduced checkpointers with MemorySaver — for development. Production needs durability, isolation, and lifecycle control.
Checkpointer matrix
Reference
Backend Use For Pros Watch Out
MemorySaver Dev · tests · ephemeral CLI Zero setup Lost on restart
SqliteSaver Single-process apps · desktop Durable, simple Not concurrent-safe
PostgresSaver Production · multi-user Transactional, queryable Connection-pool sizing
RedisSaver High-frequency · short TTLs Sub-ms reads Memory cost · less durable
state_prod.py — durable Postgres checkpointer
Multi-tenant
from langgraph.checkpoint.postgres import PostgresSaver

# One pool, reused across all requests. Sized for your QPS × avg-steps.
checkpointer = PostgresSaver.from_conn_string(
    os.environ["DATABASE_URL"],
)
checkpointer.setup()   # first-run: creates tables / indexes

graph = builder.compile(checkpointer=checkpointer)

# Multi-tenancy: namespace thread_id by (tenant_id, conversation_id).
# NEVER cross-rent — collision = data leak.
def request_config(tenant_id: str, conv_id: str):
    return {
        "configurable": {
            "thread_id": f"{tenant_id}:{conv_id}",
            "checkpoint_ns": tenant_id,    # hard isolation
        },
        "recursion_limit": 25,            # Pillar 5 — max steps
    }

# Lifecycle: periodically GC old checkpoints by TTL.
# A nightly cron deletes threads with last_activity < now - 30 days.
ISOLATE TENANTS

Namespace every thread_id by tenant. Use the checkpoint_ns field for hard separation. Audit cross-tenant queries.

SET TTLS

Checkpoints accumulate. Pick a retention window (7/30/90 days) and GC nightly. Otherwise your DB grows unboundedly.

VERSION YOUR STATE

When you change the AgentState schema, old checkpoints become invalid. Migrate or version-gate. Document the contract.

05 / Pillar 2: Error Boundaries
Four Layers of Defense
Agents fail at four boundaries. Each needs its own catch-and-recover policy. Mix them up and the agent either lies or loops forever.
B1 · TOOL

A tool raises (bad args, downstream 5xx, timeout).

Catch → return error as ToolMessage → let agent retry/repair.

B2 · MODEL

LLM call fails (429 rate limit, 500, parse error).

Exponential backoff retry · fallback to cheaper model.

B3 · TRANSPORT

DB / vector store / network blip.

Retry with circuit breaker · degrade gracefully.

B4 · AGENT

Agent loops, hits step cap, or stalls.

Hard recursion_limit · timeout · safe-default response.

errors.py — production-grade error boundaries
All four layers
from langchain_core.messages import ToolMessage
from langgraph.prebuilt import ToolNode
from tenacity import retry, wait_exponential, stop_after_attempt

# [B1] Tool boundary — ToolNode catches and re-injects as ToolMessage by default.
# Customize the error message the model sees:
def handle_tool_error(state) -> dict:
    err = state.get("error")
    tool_calls = state["messages"][-1].tool_calls
    return {"messages": [
        ToolMessage(
            content=f"Tool failed: {err!s}. Try a different argument or tool.",
            tool_call_id=tc["id"],
        ) for tc in tool_calls
    ]}

tool_node = ToolNode(TOOLS).with_fallbacks(
    [RunnableLambda(handle_tool_error)],
    exception_key="error",
)

# [B2] Model boundary — retries with backoff + a cheaper fallback model.
primary = ChatOpenAI(model="gpt-4o").bind_tools(TOOLS)
fallback = ChatOpenAI(model="gpt-4o-mini").bind_tools(TOOLS)
model = primary.with_retry(
    stop_after_attempt=3,
    wait_exponential_jitter=True,
    retry_if_exception_type=(RateLimitError, APITimeoutError),
).with_fallbacks([fallback])

# [B3] Transport — wrap any external call (DB, vector store) in tenacity:
@retry(wait=wait_exponential(min=0.5, max=8), stop=stop_after_attempt(3))
async def vector_search(query: str): ...

# [B4] Agent boundary — recursion_limit + outer timeout.
config = {"configurable": {"thread_id": tid}, "recursion_limit": 25}
try:
    result = await asyncio.wait_for(graph.ainvoke(input, config), timeout=90)
except (GraphRecursionError, asyncio.TimeoutError):
    result = safe_default_response(input)   # human handoff or apology
Crash-Only Design

Because every step is checkpointed, killing and restarting the worker is a valid recovery strategy. Don't write elaborate in-memory retry logic for catastrophic failures — let the process die, let your orchestrator restart it, let LangGraph resume from the last checkpoint. Simpler is more correct.

06 / Pillar 3: Streaming
Three Stream Modes, Three Audiences
Different layers of your stack want different granularity from the agent. LangGraph emits three stream types — pick the right one per consumer.
Three stream modes
Reference
values FULL STATE PER STEP
heavy · for debugging

Emits the entire AgentState after every node runs. Best for debugging dashboards and replay tooling.

updates DELTAS PER NODE
medium · for progress UI

Emits only the state-update each node returned — "tools just ran", "agent is thinking". Powers progress bars and step indicators.

messages TOKEN-BY-TOKEN
light · for end-user UX

Streams LLM tokens as they arrive. The "ChatGPT typing effect". Lowest latency for user-perceived responsiveness.

stream.py — wire all three to a FastAPI endpoint
SSE
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.post("/chat/stream")
async def chat_stream(req: ChatRequest):

    async def events():
        config = request_config(req.tenant_id, req.conv_id)

        # stream_mode="messages" — token-level for the chat UI
        async for token, meta in graph.astream(
            {"messages": [("user", req.text)]},
            config=config,
            stream_mode="messages",
        ):
            if token.content:
                yield f"data: {{'type':'token','v':{token.content!r}}}\n\n"

    return StreamingResponse(events(), media_type="text/event-stream")

# Multi-mode: pass a LIST and you get a stream of (mode, chunk) tuples.
# Useful for the debug dashboard:
async for mode, chunk in graph.astream(
    input, config, stream_mode=["updates", "messages"]
):
    if mode == "updates": ...   # progress bar
    if mode == "messages": ...  # typing effect
TTFB Is a Product Metric

Time-to-first-byte (or first-token) drives user perception of speed far more than total response time. Stream messages from the first agent step — don't block on tool calls before emitting anything. Show "Thinking…" updates from updates mode while tools run, then resume token stream.

07 / Pillar 4: Observability
Traces · Metrics · Logs
Three tiers, three audiences. Without all three you're flying blind when a customer says "the agent did something weird at 3:47am yesterday".
TRACES

Step-by-step trajectory of one request through the agent. Every node, every tool call, every prompt sent to the LLM.

  • Tool: LangSmith (LangGraph-native)
  • Alt: OpenTelemetry → Jaeger / Honeycomb
  • Audience: engineers debugging trajectories
METRICS

Aggregates over many requests. Cost, latency p50/p95/p99, tool call frequency, error rate, step count distribution.

  • Tool: Prometheus + Grafana
  • Or: Datadog / New Relic
  • Audience: SRE / on-call
LOGS

Structured event records — exceptions, business events, audit trail. Searchable, retainable, evidence for incidents.

  • Tool: structlog + JSON to stdout
  • Sink: Loki / Splunk / Cloud logs
  • Audience: support / forensics
What to log — and what never
Reference
✓ Log ✗ Never
Request ID · tenant ID · model used · tokens in/out · cost · latency Raw user prompts containing PII (mask before log)
Tool name + arg-key + status (not full args) API keys, OAuth tokens, DB strings
Step count, recursion-limit hits, retry count Full conversation history (use trace IDs to look up)
Guardrail trips (input rejected, output blocked) Embedding vectors (re-identifiable + bulky)
obs.py — LangSmith + structlog setup
One-line tracing
import os, structlog

# LangSmith — set 3 env vars, every graph run is traced automatically.
os.environ["LANGSMITH_TRACING"]   = "true"
os.environ["LANGSMITH_PROJECT"]   = "acme-agent-prod"
os.environ["LANGSMITH_API_KEY"]   = os.environ["LS_KEY"]

# Structlog with PII redaction processor.
structlog.configure(processors=[
    structlog.processors.add_log_level,
    structlog.processors.TimeStamper(fmt="iso"),
    redact_pii,                              # your redactor
    structlog.processors.JSONRenderer(),
])
log = structlog.get_logger()

# Per-request: bind context once, every log line inherits it.
log = log.bind(
    request_id=req_id,
    tenant_id=tenant_id,
    conv_id=conv_id,
    model="gpt-4o",
)
log.info("agent.start")
# ... graph runs, LangSmith captures the trace ...
log.info("agent.done", steps=n_steps, cost_usd=round(cost, 4))
08 / Pillar 5: Cost & Latency
Stay Within Budget
Production agents have a per-request budget — in dollars and in seconds. Four levers push you under both, simultaneously.
LEVER 01 · TIERING

Cheap model for routing & tool-args. Strong model only for synthesis steps. 10-100× cost gap, often invisible in quality.

LEVER 02 · PROMPT CACHE

Mark the long system prompt as cacheable. Repeat invocations pay ~10% of base prompt cost. Huge for multi-turn agents.

LEVER 03 · STEP CAPS

recursion_limit hard-stops runaway loops. Bound parallel tool fan-out. Refuse high-depth requests above tier.

LEVER 04 · TTFB FIRST

Stream tokens immediately. Users tolerate longer total times if they see progress in < 500ms.

✓

[STATE] TTL / GC policy documented and scheduled

[ERR] recursion_limit + outer timeout + safe-default response
