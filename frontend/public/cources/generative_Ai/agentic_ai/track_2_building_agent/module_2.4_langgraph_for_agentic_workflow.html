02 / The Pivot
Pipeline → State Machine
LCEL composes straight chains: A → B → C. Agents don't fit that shape. They loop, branch, pause, retry. The graph model is built for exactly this.
DAG

LCEL — Straight Pipeline

  • ✓ Composes with pipe |, easy to read.
  • ✓ Same interface: invoke / batch / stream.
  • ✗ No native cycles → no ReAct loop.
  • ✗ No persistent state across steps.
  • ✗ No pause-and-resume; no HITL gating.
Cyclic Graph

LangGraph — State Machine

  • ✓ Cycles are first-class — ReAct fits naturally.
  • ✓ Conditional edges dispatch on state.
  • ✓ Persistent state with reducers.
  • ✓ Checkpointers — resumable, debuggable runs.
  • ✓ HITL: interrupt, inspect, resume.
The One-Line Reading

LCEL composes data.
LangGraph composes control flow over evolving state.

They're Not Competitors

LangGraph uses LCEL. Each node in your graph is typically a Runnable. The graph orchestrates between those Runnables — branching, looping, resuming. You don't pick one or the other; you pick which one owns the outer flow.

03 / The Five Primitives
Every Graph Uses Only These
Five concepts. Memorize them and you can read any LangGraph codebase — including the production agents you'll build in Track 4.
01
STATE

A typed dict (or Pydantic model) that lives across the whole run. Every node reads it, writes to it, returns updates.

TypedDict | Pydantic BaseModel
02
NODES

A function (or Runnable) that takes the State, does work, returns a partial-state update. Pure computation.

State → dict (state updates)
03
EDGES

Directed "go from node X to node Y" instructions. Wire the nodes into a graph. Always unconditional.

graph.add_edge("A", "B")
04
CONDITIONAL EDGES

A routing function reads the State, returns the next node name. The dispatcher that makes cycles and branches possible.

add_conditional_edges("agent", route_fn)
05
REDUCERS

How partial-state updates merge. add_messages appends. Custom reducers handle union/replace semantics.

Annotated[list, add_messages]
✦
START & END

Two reserved node names. Every graph has them as anchor points. END terminates the run.

from langgraph.graph import START, END
state.py — typed state with a reducer
The Entire Graph's Memory
from typing import Annotated, TypedDict, Literal
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage

class AgentState(TypedDict):
    # `messages` accumulates — reducer = add_messages (appends & de-dupes).
    messages: Annotated[list[BaseMessage], add_messages]

    # Plain fields use the default reducer: REPLACE (last writer wins).
    step_count: int
    status: Literal["running", "awaiting_approval", "done"]

# A node returns a partial dict; LangGraph applies reducers per field:
def my_node(state: AgentState):
    new_msg = AIMessage(content="thinking...")
    return {
        "messages": [new_msg],         # appended (reducer = add_messages)
        "step_count": state["step_count"] + 1,   # replaced
    }

from typing import Annotated, TypedDict
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage

class AgentState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]
nodes.py2 / 3
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import ToolNode
from tools import TOOLS               # [@tool]-decorated Python functions

model = ChatOpenAI(model="gpt-4o-mini").bind_tools(TOOLS)

# NODE 1 — the "agent": one LLM call
def agent_node(state: AgentState):
    response = model.invoke(state["messages"])
    return {"messages": [response]}       # reducer appends

# NODE 2 — "tools": LangGraph's prebuilt ToolNode runs the tool_calls
tool_node = ToolNode(TOOLS)             # auto-handles parallel calls + errors

# ROUTING FUNCTION — read state, decide next node
def route(state: AgentState) -> str:
    last = state["messages"][-1]
    if last.tool_calls:
        return "tools"
    return "__end__"
graph.py — wire it up3 / 3
from langgraph.graph import StateGraph, START, END
from state import AgentState
from nodes import agent_node, tool_node, route

builder = StateGraph(AgentState)

builder.add_node("agent", agent_node)
builder.add_node("tools", tool_node)

builder.add_edge(START, "agent")

# CONDITIONAL EDGE — the heart of the ReAct loop
builder.add_conditional_edges("agent", route, {
    "tools":   "tools",
    "__end__": END,
})

# Unconditional return from tools → agent (creates the CYCLE)
builder.add_edge("tools", "agent")

graph = builder.compile()

# Same interface as any Runnable — Module 2.0's LCEL contract holds.
final = graph.invoke({"messages": [("user", "weather in Bhopal?")]})
Key Insight

The compiled graph is itself a Runnable. Every LCEL pattern from Module 2.0 — invoke, batch, stream, ainvoke — works on it. LangGraph extends LCEL; it doesn't replace it.

05 / Four Canonical Patterns
Off-the-Shelf Graph Shapes
Four shapes cover ~ 90% of real workloads. Recognize them, reach for them, customize the nodes.
01 — LINEAR A → B → C

Just nodes wired in sequence. State threads through. Used when you want LangGraph's persistence / observability but don't need branching yet.

USE: structured extraction pipelines · the research engine from 2.3 (when ported to graphs)
02 — BRANCHING ROUTER

Conditional edges fan out to N destinations based on the state. Use one when each branch handles a distinct task.

USE: triage agents · intent routing · escalation paths
03 — REACT (CYCLE) LOOP

The canonical agent loop from Section 5. Agent decides → optionally calls tools → tools return → agent decides again. Loops until done.

USE: tool-using agents · most single-agent production systems
04 — SUPERVISOR HUB-AND-SPOKE

A supervisor agent routes to N specialist worker agents and aggregates results. The hierarchical multi-agent pattern — full deep dive in Module 2.8.

USE: customer support · ops orchestration · multi-domain assistants
06 / Checkpointing + HITL
What Makes Agents Shippable
Two graph features that have no LCEL analog. They turn a fragile loop into a production-grade system.
CHECKPOINTERS
Persistence

Persist the full state at every step. Each step gets a checkpoint ID. You can resume mid-run, replay for debugging, or fork into a new branch.

from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()           # dev — swap for Postgres/Redis in prod
graph = builder.compile(checkpointer=checkpointer)

config = {"configurable": {"thread_id": "user-7711"}}
graph.invoke({"messages": [...]}, config=config)
# Inspect history: graph.get_state_history(config) → every step ever taken.
  • ✓ thread_id = one conversation / run.
  • ✓ Production backends: PostgresSaver, SqliteSaver, RedisSaver.
HUMAN-IN-THE-LOOP
interrupt_before

Pause the graph at a known node, surface the proposed action to a human, then resume on approval. Powered by checkpoints + interrupt_before.

graph = builder.compile(
    checkpointer=checkpointer,
    interrupt_before=["tools"],         # pause BEFORE any tool call
)

# 1) Run — it stops at the interrupt
state = graph.invoke({"messages": [...]}, config=cfg)

# 2) Human reviews state.values["messages"][-1].tool_calls
#    (e.g. an "issue_refund" call). Approve OR edit args.

# 3) Resume — None input continues from the checkpoint
state = graph.invoke(None, config=cfg)
  • ✓ Approve · edit args · reject (rewind to prior checkpoint).
  • ✓ Foundational for high-stakes agents (Module 4.7 deep dive).

More Graph-Runtime Capabilities

STREAMING

graph.stream(...) yields per-node updates. Plug straight into a WebSocket — the user sees the agent think in real time.

TIME-TRAVEL

graph.get_state_history() + a checkpoint id = rewind and re-run with edited state. Invaluable for eval (Track 3) and incident replay (Module 4.2).

SUBGRAPHS

A compiled graph is a Runnable — it can be a node inside another graph. Composition all the way up. Returns in Module 2.8.

07 / Decision Framework
LCEL vs LangGraph
Four questions. If any answer is yes, you want LangGraph. Otherwise LCEL is fine.
Q1Does the flow need to loop on its own output?

ReAct, retry-with-correction, plan-and-execute…

LCEL has no native cycle. You'd handroll a while-loop around it — defeating the framework. LangGraph's conditional edges make loops first-class.

Q2Does control flow depend on intermediate state?

Branch by tool result, escalation level, confidence…

LCEL can branch with RunnableBranch, but the routing logic lives inline with the data flow — hard to test. LangGraph's routing functions are pure, named, and inspectable.

Q3Do you need to pause and resume?

HITL approval, long-running, user gone offline…

Checkpointers + interrupt_before are unique to the graph runtime. Without them, pause/resume means rebuilding the entire conversation context manually from your own DB.

Q4Multiple agents collaborating?

Supervisor + workers, swarm, hand-offs…

Subgraphs + shared state = clean multi-agent topology. LCEL can compose two chains, but coordination, routing, and handoff become tangled. Module 2.8 builds the full pattern.

The Pragmatic Rule

Pipelines → LCEL.
Agents → LangGraph.

Anything fancier than a single prompt-model-parser is usually a graph.
