02 / 2-Min RAG Recap
What You Already Built
You completed the RAG course. This is a 2-minute callback — not a re-teach. If anything below feels foreign, swing back to the RAG course before continuing.
2-Min Callback

In the RAG course you shipped: a pipeline that takes a query, embeds it, retrieves K relevant chunks from a vector store, stuffs them into a prompt, and lets the model answer. You also covered chunking, hybrid search (BM25 + dense), reranking, and faithfulness eval. All of that stays there. This module is concerned only with how an agent consumes retrieval.

RAG-COURSE TAKEAWAYS
  • Retriever is a Runnable: query in → list[Document] out.
  • Embedding model + vector store back it. Hybrid (BM25 + dense) is the default in 2026.
  • Reranker (cross-encoder) tightens the top-K. Eval = faithfulness + answer-relevance.
  • The classic shape: retriever | stuff | model | parser.
WHY THAT SHAPE BREAKS FOR AGENTS
  • Every turn retrieves — even greetings, small talk, function-calls.
  • Context bloats with documents the LLM didn't need.
  • Multi-step agents end up retrieving the same chunks repeatedly.
  • Only one knowledge source — agents often need many.
The Reframe

For pipelines: retrieval is the architecture. For agents: retrieval is a tool the LLM picks — when, where, how often.

03 / Two Architectures
Retrieval-as-Tool vs Always-Retrieval
Two shapes. One forces retrieval on every turn. The other lets the agent decide. Each has a sweet spot.
Pipeline

Always-Retrieval

QUERY → RETRIEVE
(forced) → STUFF +
LLM → ANS
  • ✓ Predictable, single-shape — easy to evaluate (faithfulness, recall@k).
  • ✓ Right for narrow Q&A workloads with one corpus.
  • ✗ Wastes tokens on turns that don't need retrieval.
  • ✗ Cannot pick between multiple knowledge sources.
Agent

Retrieval-as-Tool

QUERY → AGENT
decides ⇄ retrieve() other tool
  • ✓ Retrieves only when needed — saves tokens, latency, cost.
  • ✓ Composes with other tools (calendar, CRM, web search).
  • ✓ Supports multiple retrievers as distinct tools (§ 5).
  • ✗ Harder to test — model may skip retrieval when it shouldn't.
The One-Line Discriminator

Always-retrieval = you control when. Retrieval-as-tool = the LLM controls when.

04 / When Agents Should Retrieve
Five Signals That Say Retrieve
Five signals. If any are present, give the agent a retrieval tool. If none are — skip RAG entirely, you'll add latency for nothing.
SIGNAL 01

Private / Domain Knowledge

Information the model can't possibly know — internal docs, your codebase, customer history. Retrieval is the only way in.

SIGNAL 02

Fresh / Post-Cutoff Facts

News, prices, current product specs. The model's training data is months old by the time you ship.

SIGNAL 03

Citation Requirement

The product must link claims to sources. RAG produces source IDs; an unaided model can only confabulate URLs.

SIGNAL 04

Corpus > Context Window

Your knowledge base is bigger than even long-context models can swallow. RAG narrows it; long-context can't.

SIGNAL 05

Hallucination Cost Is High

Medical, legal, financial, compliance. Grounding to retrieved evidence + faithfulness eval beats raw inference.

DON'T RETRIEVE FOR

Pure Reasoning · Code · Chitchat

Math, logic, code synthesis, casual back-and-forth. Retrieval adds noise — and the model is already good at these.

Two Failure Modes to Avoid

FAILURE 01 — UNDER-RETRIEVAL
The agent has the tool but skips calling it on questions it should answer with sources. Result: confident hallucinations.
Fix: tighten the tool description with explicit triggers. Or hybrid: detect intent first, force-call the retriever on knowledge-intent turns.
FAILURE 02 — OVER-RETRIEVAL
The agent calls retrieve on every turn — even "hi" and follow-ups. Latency explodes, context fills with junk.
Fix: add a refusal clause to the tool description ("do NOT call for greetings, code, or pure reasoning"). Cap calls per trajectory.
05 / Multi-Retrieval Agents
Multiple Knowledge Sources

The LLM reads tool descriptions — same mechanism as Module 0.2 — and picks the right one (or several in parallel) per turn. Module 1.0 tool patterns + Module 2.4 graph nodes assemble cleanly.

Three Patterns

P1 — ROUTER

LLM picks one retriever per turn. Cheapest. Use when sources are clearly disjoint.

if intent=docs → search_docs
P2 — PARALLEL FAN-OUT

LLM emits N tool calls at once — Module 0.2's parallel pattern. Use when a query genuinely spans sources.

[search_docs, search_crm]
P3 — CASCADE

Cheap retriever first; only escalate to expensive ones on miss. Best cost/latency profile.

docs → if empty → web
06 / Code: Retrieval as a Tool
Wrap a Retriever as a Tool
A 15-line bridge from the RAG course's retriever to an agent tool the LangGraph ReAct loop can call.
From single source to multi-source into the graph
Code
1 · SINGLE SOURCE
2 · MULTI-SOURCE
3 · INTO THE GRAPH
tools/search_docs.py — retriever → @tool1 / 3 · ONE KNOWLEDGE SOURCE
from langchain_core.tools import tool
from pydantic import BaseModel, Field

# The retriever itself comes from the RAG course — black box here.
from rag import docs_retriever     # Runnable: query → list[Document]

class SearchDocsArgs(BaseModel):
    query: str = Field(..., description="Specific question to search the docs for.")
    top_k: int = Field(4, ge=1, le=10)

@tool(args_schema=SearchDocsArgs)
def search_docs(query: str, top_k: int = 4) -> list[dict]:
    """Search Acme Cloud's product documentation.

    Use this when the user asks how to use a feature, what an API does,
    or what the platform supports. Do NOT use for greetings, casual chat,
    code synthesis, or pure reasoning questions."""
    docs = docs_retriever.invoke(query)[:top_k]
    return [
        {"source_id": d.metadata["id"], "title": d.metadata["title"],
         "content": d.page_content[:1200]}
        for d in docs
    ]
# Description is load-bearing: it teaches the LLM WHEN to call this tool.
tools/multi.py — four retrievers, one agent2 / 3 · MULTIPLE SOURCES
from langchain_core.tools import tool

@tool
def search_docs(query: str) -> list[dict]:
    """Product documentation. Use for HOW-TO and FEATURE questions."""
    ...

@tool
def search_crm(user_id: str, query: str) -> list[dict]:
    """Customer history. Use ONLY when a specific user_id is in context."""
    ...

@tool
def search_code(query: str) -> list[dict]:
    """Internal codebase. Use for "where is X implemented?" questions."""
    ...

@tool
def search_web(query: str) -> list[dict]:
    """Open web. Use for FRESH facts (news, prices, new releases).
       AVOID for anything that should live in docs or code."""
    ...

KNOWLEDGE_TOOLS = [search_docs, search_crm, search_code, search_web]
# LLM reads all 4 descriptions and picks. Often emits 2+ in parallel.
graph.py — bind to the LangGraph ReAct agent3 / 3 · MODULE 2.4 PATTERN
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI
from tools.multi import KNOWLEDGE_TOOLS

# Identical to Module 2.4's ReAct graph — just bind the retrieval tools.
model = ChatOpenAI(model="gpt-4o-mini").bind_tools(KNOWLEDGE_TOOLS)

def agent_node(state):
    return {"messages": [model.invoke(state["messages"])]}

tool_node = ToolNode(KNOWLEDGE_TOOLS)

def route(state):
    return "tools" if state["messages"][-1].tool_calls else "__end__"

builder = StateGraph(AgentState)
builder.add_node("agent", agent_node)
builder.add_node("tools", tool_node)
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route, {"tools": "tools", "__end__": END})
builder.add_edge("tools", "agent")

graph = builder.compile()
# The agent now retrieves only when its own reasoning calls for it —
# and picks among 4 knowledge sources by reading the tool descriptions.
Insight

Reuse compounds: the retriever is whatever the RAG course built. The tool wrapper is 15 lines. The agent loop is Module 2.4 unchanged. RAG-as-tool is a composition pattern, not new infrastructure.

07 / Long-Ctx vs RAG vs Fine-Tuning
The Three Levers, Agent Lens
Module 2.0 covered this for chains. Here we look at the agent-specific implications — especially how 1M-token contexts have changed when RAG is worth it.
LONG CONTEXT
MECHANISM
Stuff the whole corpus into the prompt
AGENT IMPACT
Every step pays the full context cost — token bills compound across the loop.
SWEET SPOT
Corpus is small + static, query touches most of it (codebase Q&A on a single repo).
RAG
MECHANISM
Retrieve K relevant chunks per call
AGENT IMPACT
As a tool: pays only when called. Multiple sources sit as separate tools.
SWEET SPOT
Corpus >> context, citations needed, multi-source, frequent updates.
FINE-TUNING
MECHANISM
Bake knowledge / style into weights
AGENT IMPACT
Best for specialist sub-agents — extraction, routing — where calling cheap is critical.
SWEET SPOT
High-volume narrow tasks, format/style enforcement, prompt-compression.

Agent-Specific Decision Table

Situation Long Ctx RAG-as-Tool Fine-Tuning
Corpus 50 MB · static · queries span much of it ★ best ok overkill
Corpus 50 GB · frequently updated · citations required impossible ★ best no
Specialist intent-classifier called 1M times/day too expensive unneeded ★ best
Multi-domain agent (docs + CRM + code + web) no ★ best (multi-tool) router only
Strict house style for every output prompt heavy no ★ best
The 2026 Shift

With 1M-token contexts, long-ctx beats RAG on small/static corpora — but for multi-source agents with citations, RAG-as-tool still wins. All three coexist in production. They are not competitors; they are levers.
