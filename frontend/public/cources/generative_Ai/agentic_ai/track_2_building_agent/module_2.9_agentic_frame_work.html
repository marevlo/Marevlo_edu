02 / Three Discriminating Axes
Don't Compare Frameworks. Compare Axes.
Frameworks come and go. The dimensions they trade off don't. Internalize three axes and any new framework becomes legible at a glance.
AXIS 1 · ABSTRACTION

How much the framework decides for you vs leaves under your control.

Low graph-level (LangGraph, Pydantic AI)
Mid role-level (AutoGen)
High crew-level (CrewAI)

More abstraction = faster start, less control when things go wrong.

AXIS 2 · LOCK-IN

How tied are you to one vendor — for models, hosting, observability?

Open LangGraph · CrewAI · AutoGen · Pydantic AI
Closed-ish LlamaIndex (own infra)
Vendor OpenAI Agents SDK

Vendor SDKs ship faster — but tie you to one model provider.

AXIS 3 · SCOPE

Does the framework handle just the agent loop, or full app concerns?

Narrow Pydantic AI · OpenAI SDK
Broad LangGraph · LlamaIndex
Suite CrewAI · AutoGen

Broad scope = batteries included; narrow = you wire your own stack.

The Frame

Pick by what you control, what you trade, and what you're willing to wire yourself.

03 / Six Frameworks
The Working Landscape
Six frameworks dominate production agents in 2026. Each card: mental model, primitives, sweet spot, watch-outs.
LANGGRAPH YOUR DAILY DRIVER

Mental model: state machines (graphs) over Runnables. The substrate Modules 2.4–2.9 are built on.

PRIMITIVES: State, Nodes, Edges, Conditional Edges, Reducers, Checkpointers
SWEET: Production agents · cycles · HITL · resumable runs
WATCH: Steep learning curve · graph mental model
FROM: LangChain team — vendor-neutral, open
CREWAI ROLE-FIRST

Mental model: a "crew" of agents with named roles + goals collaborating on tasks. Org-chart shape.

PRIMITIVES: Agent, Task, Crew, Process (sequential/hierarchical)
SWEET: Multi-agent demos · business-team metaphor · fast prototyping
WATCH: High abstraction hides control flow · debugging the "crew" is hard at scale
FROM: crewAI Inc — open source, paid platform
AUTOGEN CONVERSATIONAL

Mental model: agents that talk to each other via messages. Group chat is the primitive, not a graph.

PRIMITIVES: ConversableAgent, GroupChat, UserProxyAgent, RoutedAgent
SWEET: Code-generation tasks · research workflows · multi-agent chats
WATCH: v0.4 actor model is a rewrite — many docs still target v0.2
FROM: Microsoft Research — open source, broad community
PYDANTIC AI TYPE-FIRST

Mental model: agents as typed functions. Inputs, outputs, dependencies — all Pydantic. Modern, FastAPI-like.

PRIMITIVES: Agent (generic over deps + result type), Tool, RunContext
SWEET: Backend Python services · strict I/O contracts · single-agent flows
WATCH: Younger ecosystem · multi-agent patterns less mature than LangGraph
FROM: Pydantic team — open source, vendor-neutral
LLAMAINDEX AGENTS DATA-FIRST

Mental model: agents on top of LlamaIndex's RAG primitives. Workflows + Agents API.

PRIMITIVES: Workflow, Event, AgentRunner, QueryEngineTool
SWEET: Heavy RAG + agent · doc-Q&A products · enterprise data pipes
WATCH: Agent layer thinner than LangGraph · more useful with their RAG infra
FROM: LlamaIndex / Run-LLaMA — OSS + cloud
OPENAI AGENTS SDK VENDOR-FIRST

Mental model: minimal SDK — agents, tools, handoffs, guardrails. Optimized for OpenAI's models + Responses API.

PRIMITIVES: Agent, Tool, Handoff, Guardrail, Runner
SWEET: Pure OpenAI shops · fastest TTL · built-in tracing dashboard
WATCH: Best with OpenAI models · model swapping non-trivial · ecosystem effect
FROM: OpenAI — open SDK, vendor-aligned

05 / Same Agent, 4 Ways
One Spec, Four Codebases
A simple support agent with one tool — lookup_order. Each tab shows the same agent implemented in a different framework. The differences are surface; the underlying loop is the same.
Same agent — four frameworks

langgraph_agent.py — Module 2.4 pattern// GRAPH
from langgraph.prebuilt import create_react_agent
from langchain_core.tools import tool
from langchain_openai import ChatOpenAI

@tool
def lookup_order(order_id: str) -> dict:
    """Look up a customer order by ID."""
    return orders_db.get(order_id)

agent = create_react_agent(
    model=ChatOpenAI(model="gpt-4o-mini"),
    tools=[lookup_order],
    prompt="You are a support agent. Use lookup_order when asked about orders.",
)

result = agent.invoke({"messages": [("user", "Where is order #7711?")]})
crewai_agent.py — role-first// CREW
from crewai import Agent, Task, Crew, Process
from crewai.tools import tool

@tool("lookup_order")
def lookup_order(order_id: str) -> dict:
    """Look up a customer order by ID."""
    return orders_db.get(order_id)

support_agent = Agent(
    role="Support Specialist",
    goal="Resolve customer order questions.",
    backstory="You handle order lookups for Acme Cloud.",
    tools=[lookup_order],
    llm="gpt-4o-mini",
)

task = Task(
    description="Find out where order #7711 is.",
    expected_output="Shipping status of the order.",
    agent=support_agent,
)

crew = Crew(agents=[support_agent], tasks=[task], process=Process.sequential)
result = crew.kickoff()
pydantic_ai_agent.py — type-first// AGENT[Deps, Result]
from pydantic_ai import Agent, RunContext
from dataclasses import dataclass

@dataclass
class SupportDeps:
    orders_db: OrdersDB

agent = Agent(
    "openai:gpt-4o-mini",
    deps_type=SupportDeps,
    system_prompt="You are a support agent. Use lookup_order for order queries.",
)

@agent.tool
async def lookup_order(ctx: RunContext[SupportDeps], order_id: str) -> dict:
    """Look up a customer order by ID."""
    return await ctx.deps.orders_db.get(order_id)

deps = SupportDeps(orders_db=orders)
result = await agent.run("Where is order #7711?", deps=deps)
openai_sdk_agent.py — vendor SDK// AGENT / TOOL / RUNNER
from agents import Agent, Runner, function_tool

@function_tool
def lookup_order(order_id: str) -> dict:
    """Look up a customer order by ID."""
    return orders_db.get(order_id)

agent = Agent(
    name="SupportAgent",
    instructions="You are a support agent. Use lookup_order when asked about orders.",
    model="gpt-4o-mini",
    tools=[lookup_order],
)

result = Runner.run_sync(agent, "Where is order #7711?")
print(result.final_output)
WHAT'S SHARED

All four bind a tool, give it a description, and let an LLM call it in a loop. The ReAct pattern is universal — only the wrapping differs.

WHAT DIFFERS

Entry shape (invoke / kickoff / run / run_sync). Persona model (prompt vs role+goal+backstory vs system_prompt). Async vs sync defaults.

SO WHAT

Concepts transfer in hours. Idioms transfer in days. Skill in one framework = working knowledge of all.

06 / Pick by Use Case
A 5-Question Decision Tree
Walk top-down. First "yes" sets your default. Adjust for organizational constraints (existing stack, team skill).
Q1
Building a production agent with HITL, resume, multi-step state?
→ LangGraph

Checkpointers, conditional edges, interrupts, streaming. No one else handles all four as fluently. Track 4 of this course assumes LangGraph as the runtime.

Q2
Demo-quality multi-agent crew with low ceremony?
→ CrewAI

Stakeholders love the role + task language. You'll ship a demo today. Re-evaluate at production scale — the abstraction starts to leak when debugging matters.

Q3
Multi-agent code generation / dev-tool agents?
→ AutoGen

AutoGen's group-chat metaphor maps cleanly to code-gen workflows where agents critique each other. Be sure to use v0.4 (actor model) — v0.2 docs you'll find online are deprecated.

Q4
Backend Python service with strict types + single-agent flow?
→ Pydantic AI

Generic Agent[Deps, Result], async-first, FastAPI-like ergonomics. Especially good if your stack is already Pydantic-heavy. Less mature for multi-agent than LangGraph.

Q5
Heavy RAG product where agent is a thin layer on data?
→ LlamaIndex Agents

Their workflow API + query engines + index abstractions compose smoothly. The agent layer is thinner than LangGraph's but it inherits the entire LlamaIndex data stack.

Q6
Pure OpenAI shop, want fastest TTL + built-in tracing?
→ OpenAI Agents SDK

Cleanest 5-primitive surface. Tied to OpenAI's Responses API; non-OpenAI models work through adapters but lose some features. Use when vendor alignment is a deliberate choice.

The Pragmatic Rule

Pick by what you'll need at month 6, not by what's easiest at week 1. Demo frameworks rarely scale; production frameworks always demo.

07 / Reading Other Codebases
Spot the Pattern, Ignore the Syntax
Any agent codebase you'll read uses the same five primitives — just with different names. Build a translation table once; it works everywhere.
The Universal Translation
Table
CONCEPT LANGGRAPH CREWAI AUTOGEN PYDANTIC AI OPENAI SDK
Tool @tool @tool register_for_* @agent.tool @function_tool
Agent identity prompt role+goal+backstory system_message system_prompt instructions
Run / invoke .invoke / .stream .kickoff .initiate_chat .run / .run_sync Runner.run
State TypedDict + reducers crew context chat history deps + history conversation history
Handoff Command(goto=...) hierarchical process GroupChat speaker sub-agent call Handoff()
Structured output with_structured_output output_pydantic function-based result_type output_type
Tracing LangSmith crew.usage_metrics callbacks Logfire built-in dashboard
STEP 1 · TOOLS

Grep for @*tool decorators or register_*. This shows you the agent's actions — the most concrete surface.

STEP 2 · PROMPTS

Find the system_prompt / role / instructions. That's the agent's identity — what it thinks it is.

STEP 3 · FLOW

Trace the entrypoint (invoke / kickoff / Runner.run). Walk one trip through the loop. You now understand the agent.
