02 / When to Go Multi
Default to One Agent
"Multi-agent" sounds impressive. Often it's the wrong answer. Default to one agent with more tools. Split only when at least one of these five signals is genuinely present.
5 signals · default is "add a tool"
Decision
Signal Split? Cheaper Alternative
Distinct system prompts / personas needed ★ split Try: dynamic system message routing first
10+ tools confusing the agent ★ split Try: tool filtering by intent first
Independent failure domains required ★ split No real alternative — split.
Parallel work over the same input ★ split Module 2.2 map-reduce is often enough
Different model tiers per sub-task ★ split Cost lever from Module 2.7 forces this
"It sounds cool" / org diagram fit no Stay single-agent. Add tools.
What multi-agent gives

The Upside

  • ✓ Smaller, more focused prompt per agent
  • ✓ Specialist models per task tier
  • ✓ Independent debugging & eval
  • ✓ Reusable agents across products
What it costs

The Downside

  • ✗ 2-3× total LLM calls per request
  • ✗ Higher latency (sequential handoffs)
  • ✗ Harder to trace and debug
  • ✗ Coordination becomes its own failure mode
The Pragmatic Rule

Start single-agent. Add tools until 10+ confuse the model. Then — and only then — split.

03 / The Four Topologies
Four Canonical Shapes
Every multi-agent system you'll meet is one of these — or a combination. Pick by question shape, not by aesthetic preference.
01 — SUPERVISOR HUB-AND-SPOKE

A supervisor agent routes each turn to one of N specialist workers, collects their result, decides the next step. The workers don't talk to each other.

USE: Distinct specialties · clear routing rules · centralized control needed
WATCH: Supervisor becomes bottleneck · over-summarization
02 — HAND-OFF PEER-TO-PEER

Each agent owns a domain. When it hits a request outside its expertise, it hands off to another agent via a special tool call. No central router.

USE: Customer support escalation · workflow stages · dialog systems
WATCH: Ping-pong loops · loss of high-level intent
03 — HIERARCHICAL SUPERVISORS OF SUPERVISORS

A team-of-teams pattern. Top supervisor routes to sub-supervisors, each managing their own workers. Each sub-team is its own subgraph.

USE: Org-shaped workflows · complex domains · >6 specialists
WATCH: Cost compounds quickly · 2-hop debugging
04 — SWARM EVERYONE TALKS

Agents can hand off to any peer — no fixed routing. The system self-organizes. Most flexible, hardest to reason about, last resort.

USE: Research / exploration · open-ended creative tasks
WATCH: Termination is hard · costs explode · rarely justified in production
Pick Order in Practice

90% of real systems: SUPERVISOR. Stage-based workflows: HAND-OFF. Many specialists: HIERARCHICAL. · Open-ended research: SWARM.

The 90% Pattern, Built
A supervisor agent decides which worker runs next. Each worker is a compiled LangGraph (Module 2.4) treated as a node in the parent graph.

state.py — schema shared across supervisor + workers1 / 4
from typing import Annotated, TypedDict, Literal
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage

class TeamState(TypedDict):
    # Conversation messages — shared, append-only.
    messages: Annotated[list[BaseMessage], add_messages]

    # Supervisor decision — which worker to run next.
    next: Literal["billing", "refund", "shipping", "techsupport", "FINISH"]

# Discipline: workers READ messages, WRITE only their AIMessage.
# `next` is owned exclusively by the supervisor.
workers.py — each is a Module 2.4 ReAct subgraph2 / 4
from langgraph.prebuilt import create_react_agent

# Each worker = its own ReAct graph. Different prompts, different tools.
billing_agent = create_react_agent(
    model=ChatOpenAI(model="gpt-4o-mini"),    # cheap tier
    tools=[lookup_invoice, list_payments, retry_charge],
    prompt="You are a billing specialist. Only handle billing questions.",
)

refund_agent = create_react_agent(
    model=ChatOpenAI(model="gpt-4o"),         # strong tier — high-stakes
    tools=[lookup_order, issue_refund, policy_check],
    prompt="You are a refund specialist. Cite policies; escalate if > 50%.",
)
shipping_agent     = create_react_agent(...)
techsupport_agent  = create_react_agent(...)

# Wrap each as a node that takes parent state and returns updates.
def billing_node(state: TeamState):
    result = billing_agent.invoke({"messages": state["messages"]})
    return {"messages": [
        HumanMessage(content=result["messages"][-1].content, name="billing")
    ]}

supervisor.py — routes to next worker3 / 4 · ROUTING DECISION
from langchain_core.prompts import ChatPromptTemplate
from pydantic import BaseModel, Field

WORKERS = ["billing", "refund", "shipping", "techsupport"]

class Route(BaseModel):
    """Supervisor output schema — one decision per turn."""
    next: Literal["billing", "refund", "shipping", "techsupport", "FINISH"]
    reason: str = Field(..., max_length=120)

prompt = ChatPromptTemplate.from_messages([
    ("system",
     "You are a supervisor managing: {workers}.\n"
     "Given the conversation, decide which worker should act NEXT.\n"
     "Reply 'FINISH' only if the user's request is fully resolved.\n"
     "Do NOT do the work yourself — only route."),
    ("placeholder", "{messages}"),
]).partial(workers=", ".join(WORKERS))

# Structured output — supervisor returns a Route, not free-form text.
supervisor_model = ChatOpenAI(model="gpt-4o-mini").with_structured_output(Route)
supervisor_chain = prompt | supervisor_model

def supervisor_node(state: TeamState):
    route = supervisor_chain.invoke({"messages": state["messages"]})
    return {"next": route.next}
team.py — assemble the supervisor graph4 / 4 · COMPILED RUNNABLE
from langgraph.graph import StateGraph, START, END

builder = StateGraph(TeamState)

builder.add_node("supervisor",    supervisor_node)
builder.add_node("billing",       billing_node)
builder.add_node("refund",        refund_node)
builder.add_node("shipping",      shipping_node)
builder.add_node("techsupport",   techsupport_node)

builder.add_edge(START, "supervisor")

# Conditional edge — supervisor's `next` field dispatches the worker
builder.add_conditional_edges(
    "supervisor",
    lambda s: s["next"],
    {"billing": "billing", "refund": "refund",
     "shipping": "shipping", "techsupport": "techsupport",
     "FINISH": END},
)

# Every worker reports back to the supervisor for the next decision.
for worker in WORKERS:
    builder.add_edge(worker, "supervisor")

team = builder.compile(checkpointer=checkpointer)
# `team` is itself a Runnable — composable, observable, resumable.
Composition All the Way Up

The team graph is a Runnable, like everything else in the LangGraph world. Drop it into another supervisor and you have a hierarchical pattern — same primitives, more depth.

05 / Hand-Off Pattern
Hand-Off Is a Tool Call
No central supervisor. Each agent has a special transfer_to_X tool. When the LLM decides to call it, control transfers to agent X with shared state. The simplest, most LLM-native multi-agent pattern.
The mechanism
Flow
STEP 1

Triage agent recognizes the request is outside its scope.

STEP 2

It calls transfer_to_billing(reason=...) as a tool.

STEP 3

Graph routes to the billing agent — same state, same messages.

Because hand-off is just a tool call, the LLM picks when and where to transfer using normal tool reasoning. No separate routing model.

handoff.py — hand-off as a Command-returning tool
LangGraph Command pattern
from langgraph.types import Command
from langchain_core.tools import tool
from langchain_core.messages import ToolMessage

def make_handoff(target: str):
    """Factory: produce a transfer_to_{target} tool the LLM can call."""
    @tool(name=f"transfer_to_{target}")
    def _handoff(reason: str, tool_call_id: str) -> Command:
        """Transfer control to the {target} agent. Provide a reason."""
        return Command(
            goto=target,                # <-- LangGraph routes here
            update={"messages": [ToolMessage(
                content=f"Transferred to {target}: {reason}",
                tool_call_id=tool_call_id,
            )]},
            graph=Command.PARENT,       # jump across subgraph boundary
        )
    return _handoff

# Each agent gets hand-off tools for the OTHER agents.
triage_agent = create_react_agent(
    model=model,
    tools=[make_handoff("billing"), make_handoff("tech")],
    prompt="You triage support tickets. If it's billing or tech, hand off.",
)
billing_agent = create_react_agent(
    model=model,
    tools=[lookup_invoice, make_handoff("triage")],
    prompt="You handle billing. If unrelated, transfer back to triage.",
)
tech_agent = create_react_agent(...)
Wins

Why It's Cheap

  • No extra supervisor LLM call per turn — cheaper
  • Hand-off logic is in the agent's prompt + tool description
  • Natural fit for staged workflows (intake → process → close)
Watch out

Failure Modes

  • Ping-pong loops: A → B → A → B. Add a max-handoffs cap.
  • Receiving agent inherits all prior messages → context bloat
  • No single agent owns the user's overall intent
06 / Shared State Discipline
Public Channels, Private Scratchpads
Without discipline, multi-agent state becomes a god-object. Three rules keep coupling under control as the system grows.
R1
PUBLIC vs PRIVATE

The parent state holds public fields all agents see (messages, next, summary). Each subgraph has its own private state for scratchpad work that never leaks up.

R2
OWNERSHIP PER FIELD

Each state key has one writer. next belongs to the supervisor. messages is append-only (any agent). refund_amount belongs to refund_agent only.

R3
REDUCERS FOR FAN-IN

When multiple agents write the same key (parallel work), declare a reducer — add_messages, list-concat, or custom merge. Never last-write-wins for shared fields.

shared_state.py — public + private with subgraphs
Schema discipline
from typing import Annotated, TypedDict
from operator import add
from langgraph.graph.message import add_messages

# PUBLIC — what the parent supervisor and every worker can see.
class TeamState(TypedDict):
    messages: Annotated[list, add_messages]
    next: str
    summary: str                        # supervisor writes
    artifacts: Annotated[list, add]         # list-concat across writers

# PRIVATE — billing agent's internal scratchpad.
# Lives only inside the billing subgraph; never leaks to the parent.
class BillingState(TypedDict):
    messages: Annotated[list, add_messages]
    invoice_cache: dict                  # scratchpad
    retry_count: int

# Compile the billing subgraph with its OWN state schema.
billing_subgraph = StateGraph(BillingState). ... .compile()

# When mounting in the parent: only `messages` crosses the boundary.
def billing_node(state: TeamState) -> dict:
    sub_result = billing_subgraph.invoke({"messages": state["messages"]})
    return {"messages": sub_result["messages"][-1:]}
    # invoice_cache / retry_count NEVER reach the parent → clean boundary.
07 / Five Pitfalls
Where Multi-Agent Goes Wrong
Five failure modes every team hits the first time they ship a multi-agent system. Recognize them early and you'll save weeks.
Five failure modes — click to expand the fix
Pitfalls
P1 · Cost Spiral
P2 · Context Bloat
P3 · Infinite Hand-Offs
P4 · Fragmented Memory
P5 · Debugging Hell

P1 — Cost Spiral

Every turn = supervisor LLM call + worker LLM call (+ tool calls). 3-5× cost vs single agent.

Fix: tier models aggressively — small model for supervisor routing, strong only for synthesis workers. Cache the supervisor prompt. Cap worker steps per turn.

P2 — Context Bloat

Every worker inherits the full message history. By turn 10, prompts are huge.

Fix: pass only the relevant slice to each worker. Maintain a summary field the supervisor refreshes. Workers see summary + last_3_messages, not the full transcript.

P3 — Infinite Hand-Offs

Agent A hands to B, B hands back to A, forever.

Fix: track handoff_count in shared state. Hard-cap at 3. On overflow, route to a human-handoff node or a safe default. Use Module 2.7's recursion_limit as a back-stop.

P4 — Fragmented Memory

Workers store details in private state; later turns can't find it.

Fix: any fact the team will need later must surface to public state. Workers append structured findings to artifacts. Module 2.11 covers long-term memory layers in depth.

P5 — Debugging Hell

"Which agent failed at step 14 of a 23-step trajectory?" Stack traces don't help.

Fix: structured tracing per Module 2.7 — tag every step with its agent_name. LangSmith renders subgraph timelines natively. Add a name= argument when messages cross agent boundaries.

The Pattern Behind the Fixes

Most multi-agent failures are accounting failures — unbounded cost, context, recursion, or memory. Bound everything. Cap everything. Trace everything.
