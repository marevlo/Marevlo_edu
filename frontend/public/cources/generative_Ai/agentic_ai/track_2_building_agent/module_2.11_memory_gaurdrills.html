02 / Threat Model
Why Agent Safety Is Different
A chatbot says things. An agent does things. The asymmetry — bounded input, unbounded action surface — is the whole game.
The Core Asymmetry — untrusted text → trusted action
Diagram

Every byte the agent reads — from users, retrievers, tool outputs, MCP servers, even prior turns — is potential instruction. The model can't reliably distinguish data from commands. If those bytes can reach a tool call, that's an attack vector.

Six Agent-Specific Threats

T1 · CRITICAL

Direct Prompt Injection

User types "ignore previous instructions and …". The LLM may obey.

T2 · CRITICAL

Indirect Prompt Injection

Malicious instructions embedded in a retrieved doc, email, or tool result. The agent reads them and executes.

T3 · HIGH

PII / Secret Leak

Agent reproduces or transmits confidential data — emails, names, keys — to the user or to external tools.

T4 · HIGH

Tool Misuse / Excessive Action

Agent calls a destructive tool (DELETE, refund_all, send_to_X) inappropriately — too often or with wrong args.

T5 · HIGH

Cross-Tenant State Leak

Long-term memory written for tenant A surfaces in a session for tenant B. Module 2.7 thread namespacing was step 1; memory needs the same discipline.

T6 · MEDIUM

Hallucinated Action

Agent invents a tool, an order ID, or a policy that doesn't exist — then claims to have acted on it.

The Frame

Treat every byte the LLM reads as potentially adversarial. Treat every tool call as a privileged action.

03 / Memory Architecture
Three Layers, Three Lifetimes
An agent has more than the messages list. Three memory layers, each with a different scope and lifetime. Place data in the wrong layer and you either forget it or leak it.
SHORT-TERM
SCOPE
One agent turn
LIFETIME
Seconds
CONTENT
Scratchpad, intermediate tool results, reasoning

Lives in subgraph-private state (Module 2.8). Dies when the turn ends. Never leaks across requests.

WORKING
SCOPE
One thread (conversation)
LIFETIME
Hours to days
CONTENT
Messages, decisions, structured context

The checkpointed AgentState from Module 2.7. Persists across the conversation. Tenant-namespaced thread IDs.

LONG-TERM
SCOPE
Across threads, per user
LIFETIME
Indefinite (with TTL)
CONTENT
Profile, preferences, learned facts, summaries

Separate store (vector DB or key-value). Explicit writes only. This is the highest-risk leak surface.

long_term_memory.py — LangGraph Store API
Scoped · Typed · Auditable
from langgraph.store.postgres import PostgresStore
from pydantic import BaseModel

# A separate, namespaced store — NOT the checkpointer.
store = PostgresStore.from_conn_string(os.environ["MEMORY_DB"])

class UserMemory(BaseModel):
    preferences: dict[str, str]
    learned_facts: list[str]
    last_updated: str

# NAMESPACE = (tenant, user) — never share across tenants.
def memory_ns(tenant_id: str, user_id: str):
    return ("memories", tenant_id, user_id)

# WRITE — explicit + audited.
async def remember(state, config):
    ns = memory_ns(config["tenant_id"], config["user_id"])
    fact = extract_durable_fact(state["messages"])    # LLM extractor
    if fact and not contains_pii(fact):                # PII guard
        await store.aput(ns, key=f"fact_{uuid4()}", value=fact.dict())
        log.info("memory.write", ns=ns, fact_type=fact.kind)

# READ — scoped retrieval, capped at top-K.
async def recall(query: str, config) -> list[dict]:
    ns = memory_ns(config["tenant_id"], config["user_id"])
    hits = await store.asearch(ns, query=query, limit=5)
    return [h.value for h in hits]
# Memory injected as a system message; never as a tool result.
Memory Rules
  • ✓ Always namespace by (tenant, user) — never global
  • ✓ Write is explicit (not auto-everything) — extract durable facts only
  • ✓ Read returns top-K via semantic search, not full dump
  • ✓ Inject as system context, not as tool output
Memory Anti-Patterns
  • ✗ "Remember everything" auto-store on every turn
  • ✗ Memory written by LLM with no PII filter
  • ✗ Read scope only by user_id (tenant ignored)
  • ✗ No TTL — memory grows unboundedly forever
04 / Input Guardrails
At The Front Door
The earliest, cheapest layer. Block inputs that have no business reaching the agent — before any tokens are billed.
G1 · LENGTH & RATE

Cap input characters. Cap requests per user per minute. Cheapest defense — most abuse is high-volume.

G2 · AUTH & SCOPE

Reject before the loop if the user lacks scope for the requested action. Cheaper than refusing later.

G3 · INJECTION HEURISTICS

Pattern-match obvious payloads ("ignore previous instructions", role-spoofing, role-tag injection). Catches the low-hanging fruit.

G4 · CLASSIFIER

Cheap-model classifier: is this a harmful or off-topic request? Reject before the main agent burns tokens.

G5 · LOCALE / TOPIC

Restrict to supported languages and topics. Off-topic = clearer refusal, less attack surface.

DO NOT

Rely on a single regex. Injection has thousands of variants. Pattern matching is one signal, not the gate.

input_guards.py — staged input pipeline
Cheap → Expensive
from pydantic import BaseModel
from langchain_core.runnables import RunnableLambda

class GuardResult(BaseModel):
    allowed: bool
    reason: str
    severity: str          # info / warn / block

def length_guard(text: str) -> GuardResult:
    if len(text) > 8000:
        return GuardResult(allowed=False, reason="too_long", severity="block")
    return GuardResult(allowed=True, reason="", severity="info")

def injection_heuristics(text: str) -> GuardResult:
    patterns = [
        r"(?i)ignore (?:all |previous |the )?instructions",
        r"(?i)you are now (?:a |an )?[\w\s]+",
        r"<\s*system\s*>|<\|im_start\|>",
    ]
    if any(re.search(p, text) for p in patterns):
        return GuardResult(allowed=True, reason="injection_signal", severity="warn")
    return GuardResult(allowed=True, reason="", severity="info")

async def classifier_guard(text: str) -> GuardResult:
    # Cheap small-model classifier — runs only after cheaper checks pass.
    label = await classifier.ainvoke(text)
    if label in {"off_topic", "harmful"}:
        return GuardResult(allowed=False, reason=label, severity="block")
    return GuardResult(allowed=True, reason=label, severity="info")

async def input_pipeline(text: str) -> GuardResult:
    for g in [length_guard, injection_heuristics]:
        r = g(text)
        if not r.allowed: return r
    return await classifier_guard(text)
# Wire as the FIRST node of the LangGraph; on block → safe response.
Indirect Injection — The Harder Case

Input guards run on the user's message. The bigger surface is everything the agent reads after: retrieved docs, tool outputs, MCP server responses, prior memories. Each is also an injection vector — and you can't classify all of them.

Mitigations: wrap untrusted content in clear delimiters (e.g. <untrusted> … </untrusted>), instruct the model to treat it as data not instructions, and — most importantly — rely on the tool-call guardrails downstream (§ 6). Defense in depth: assume injection succeeds at the prompt layer; ensure it fails at the action layer.

05 / Output Guardrails
Before It Leaves the System
Even if input passed and the agent reasoned, the response may still be unsafe to ship. Last-mile filters protect users and the business.
G1 · PII REDACTION

Mask emails, phone numbers, SSNs, names from out-of-scope domains. Use a deterministic redactor — not the LLM — for compliance.

G2 · CITATION CHECK

For RAG agents: enforce that every factual claim cites a source_id present in retrieved docs. Otherwise: reject or warn.

G3 · SCHEMA VALIDATION

Pydantic on the boundary (Module 1.0). Wrong shape = block, not retry. Schema is non-negotiable contract.

G4 · TOXICITY GATE

Run a classifier on the final draft. Block obvious slurs, harassment, self-harm content. Cheap small models are good at this.

G5 · REFUSAL ENVELOPE

When blocking, return a structured refusal — never raw error. Includes reason_code, redirect, contact-support fallback.

WATCH

LLM-as-judge for output safety can itself be jailbroken via the candidate output. Combine with deterministic rules — never single layer.

output_guards.py — last-mile filters
Deterministic First
import re
from pydantic import BaseModel, ValidationError

# Deterministic PII patterns — supplement, never replace, a proper PII engine.
PII_RE = {
    "email": re.compile(r"[\w.+-]+@[\w-]+\.[\w.-]+"),
    "phone": re.compile(r"\+?\d[\d\s().-]{8,}"),
    "ssn":   re.compile(r"\b\d{3}-\d{2}-\d{4}\b"),
}

def redact_pii(text: str) -> str:
    for kind, pat in PII_RE.items():
        text = pat.sub(f"[REDACTED_{kind.upper()}]", text)
    return text

def enforce_citations(answer: str, sources: list[str]) -> bool:
    # Every "src:<id>" tag must match a retrieved source_id.
    cited = re.findall(r"src:(\w+)", answer)
    return all(c in sources for c in cited)

class Refusal(BaseModel):
    refused: bool = True
    reason_code: str
    user_message: str
    contact: str = "support@acme.cloud"

async def output_pipeline(draft: str, sources: list[str]) -> dict:
    text = redact_pii(draft)
    if sources and not enforce_citations(text, sources):
        return Refusal(reason_code="missing_citations",
                       user_message="I can't verify that answer; please rephrase.").dict()
    if await tox_classifier.ainvoke(text) == "unsafe":
        return Refusal(reason_code="unsafe_content",
                       user_message="I can't help with that.").dict()
    return {"answer": text}
06 / Tool-Call Guardrails
The Agent-Only Surface
Chatbots don't have these. Every tool call is privileged code execution on behalf of an LLM that just read untrusted text. This is the layer that has to hold even when injection has already succeeded upstream.
TG1 · ALLOWLIST

Per-tenant, per-role tool allowlist. Reject any tool call not in scope — even if the model "decides" to make it.

TG2 · ARG SCHEMA

Pydantic validation on every tool input — bounded ranges, enums, max lengths. Module 1.0 was foundational; this is the security layer.

TG3 · RATE LIMITS

Per-tool, per-user, per-window. send_email capped at 5/min. delete_record at 1/hour. Prevents runaway loops.

TG4 · BLAST RADIUS

Cap impact per call: max records updated, max $ refunded, max recipients emailed. Even a successful injection can't escalate.

TG5 · HITL FOR IRREVERSIBLES

Irreversible or high-stakes tools (payments, deletes, public posts) require human approval. Use LangGraph interrupt_before.

TG6 · AUDIT TRAIL

Every tool call: who, what, when, args (sanitized), outcome. Append-only. Use for forensics, refund disputes, compliance.

tool_guards.py — defense around the tool call
The Load-Bearing Layer
from langgraph.types import Command, interrupt
from langchain_core.tools import tool
from pydantic import BaseModel, Field

# TG1 — Allowlist enforced at the graph edge, not inside each tool.
ALLOWED = {"customer": {"lookup_order", "open_ticket"},
           "agent":    {"lookup_order", "open_ticket", "issue_refund"}}

def tool_gate(state, config):
    role = config["role"]
    for tc in state["messages"][-1].tool_calls:
        if tc["name"] not in ALLOWED[role]:
            raise PermissionError(f"Tool {tc['name']} not allowed for {role}")

# TG2/TG4 — Pydantic schema with hard bounds = blast radius cap.
class RefundArgs(BaseModel):
    order_id: str = Field(..., pattern=r"^ord_\w{8,16}$")
    amount_cents: int = Field(..., ge=1, le=50_000)        # max $500
    reason: str      = Field(..., max_length=500)

# TG3 — Rate limit lives in the tool wrapper, keyed by (tenant, tool, window).
@tool(args_schema=RefundArgs)
async def issue_refund(order_id, amount_cents, reason, *, ctx):
    await rate_limiter.check(ctx.tenant_id, "issue_refund", per_min=2)

    # TG5 — irreversible action: HALT for human approval before executing.
    decision = interrupt({"kind": "approve_refund",
                          "order_id": order_id,
                          "amount_cents": amount_cents,
                          "reason": reason})
    if not decision["approved"]:
        return {"status": "declined_by_human", "by": decision["user"]}

    # TG6 — append to immutable audit log BEFORE the side effect.
    await audit.log(
        actor=f"agent:{ctx.agent_id}", action="issue_refund",
        target=order_id, amount=amount_cents, approver=decision["user"],
    )
    return await payments.refund(order_id, amount_cents, reason)
The Load-Bearing Principle

Assume input guards fail. Assume the model is compromised. Tool guards must still hold.

07 / Defense in Depth
Five Layers on Every Request
No single guard is perfect. Stack five so an attacker has to defeat all of them. Each layer is fast and falsifiable on its own.

Request Lifecycle — Where Each Guard Fires

Layer When Blocks Techniques
L1 · EDGE Before agent invocation spam, auth fails, oversize Rate limit · WAF · auth · length cap
L2 · INPUT After auth, before LLM direct injection · off-topic Heuristics · classifier · scope
L3 · PROMPT Inside the LLM call indirect injection from RAG/tools Delimiters · system reminders · structured I/O
L4 · TOOL Around each tool call malicious or excessive actions Allowlist · schema · rate · HITL · audit
L5 · OUTPUT Before response goes out PII leaks · uncited claims · toxicity Redaction · citation check · classifier

08 / Safety Pitfalls
Where Safety Theatre Begins
Five mistakes teams make in their first safety pass. Each looks correct in code review. Each fails in production.
Five pitfalls — and their structural fixes
Patterns
P1 · System Prompt Only
P2 · Trusting Retrieval
P3 · Memory No TTL/Scope
P4 · LLM-as-Guard
P5 · No Audit Trail

Putting Safety Only in the System Prompt

"You must never reveal X" — easily overridden by injection.

Fix: the system prompt is a hint, not a gate. Critical rules are enforced in code — output redaction, tool allowlists, schema validation. The prompt may collaborate with the model; only deterministic layers can be trusted.

Trusting Retrieved Content

RAG docs are treated as ground truth — but anyone who can write to the corpus can inject instructions.

Fix: wrap retrieved docs in <untrusted> delimiters, instruct the model to treat them as data not commands, and — most importantly — rely on tool-call guards (§ 6) downstream. Module 2.5's signal-5 (high hallucination cost) applies twice for adversarial corpora.

Memory Without TTL or Scope

Long-term memory grows forever, mixes tenants, or persists deletion requests.

Fix: namespace by (tenant, user); set TTLs per memory type (preferences: 1y, derived facts: 90d, support history: 30d); expose a delete endpoint and honor it within audit-logged SLAs.

LLM-as-Guard, No Backstop

A "safety classifier LLM" is the only protection — and can itself be jailbroken by the input.

Fix: LLM judges are a useful signal, not a gate. Pair every LLM-based check with deterministic rules. For tool calls especially: schema, allowlist, and rate limits do not depend on the model's good behavior.

No Audit Trail on Tool Calls

An incident occurs — and you can't answer "what did the agent do?" because tool calls were never logged.

Fix: log every tool call (actor, target, sanitized args, approver, result) to an append-only store BEFORE the side effect. Tie to request_id so traces (Module 2.7) join up. Required for refund disputes, compliance audits, and forensics.

The Working Principle

Safety is not a classifier. Safety is a stack of cheap, falsifiable checks — each protecting a real action boundary.
