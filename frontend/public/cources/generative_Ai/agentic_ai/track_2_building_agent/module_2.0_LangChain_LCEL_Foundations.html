02 / LangChain Architecture
The Five-Tier Stack
LangChain is not one package — it is a layered ecosystem. Understanding the tiers is the difference between productive use and dependency hell.

Stack Tiers, Top to Bottom

TIER 05 — APPLICATIONS LangSmith · LangGraph Platform · LangServe

Tracing, deployment, hosted platform. The production layer. Returns in Track 4.

TIER 04 — langchain pip install langchain

High-level chains, agents, retrieval, and memory abstractions. Where most of your code imports from.

TIER 03 — PARTNERS langchain-openai · langchain-anthropic · ...

Per-provider integrations, version-pinned by official partners. Production-grade.

TIER 02 — COMMUNITY langchain-community

Long-tail integrations maintained by the community. Quality varies — pin versions.

TIER 01 — CORE langchain-core

Foundational abstractions: Runnable, BaseMessage, BasePromptTemplate, the LCEL machinery itself. Zero third-party deps.

install.sh — the correct way
Pin by tier
# Core abstractions — almost always required
pip install langchain-core

# Provider partner package — pick what you actually use
pip install langchain-openai            # for OpenAI / Azure
pip install langchain-anthropic         # for Claude

# High-level chains/agents (orchestration)
pip install langchain

# Only if you need a long-tail integration:
pip install langchain-community

# Observability and tracing (Track 4)
pip install langsmith
Why the Split Matters

The langchain-core package is intentionally tiny. Heavy partner integrations (and their dependency trees) only land in your venv when you explicitly install them. Production Dockerfiles thank you.

The One-Line Reading

langchain-core defines the Runnable protocol. Every other tier exists to implement it for some new component.

03 / Modular Building Blocks
The Six Primitives
Every LangChain chain or agent is built from these six things. Each is just a Runnable that takes one type and returns another.

PROMPTS

in: dict → out: messages

Templates that turn input variables into a list of BaseMessages — system, human, AI. Covered in detail in Module 2.1.

ChatPromptTemplate · PromptTemplate

MODELS

in: messages → out: message

LLM wrappers. Same interface, swappable backends. Bind tools, set temperature, enable streaming.

ChatOpenAI · ChatAnthropic

OUTPUT PARSERS

in: message → out: typed

Convert AIMessage.content into a Pydantic object, dict, or string. Validated using the schema work from Module 1.0.

PydanticOutputParser · StrOutputParser

RETRIEVERS

in: query → out: docs[]

Vector stores, BM25, hybrid. RAG internals → RAG course; agentic usage → Module 2.5.

VectorStoreRetriever · BM25Retriever

TOOLS

in: args → out: result

A Runnable exposed to the LLM. Backed by the Pydantic schemas you built in Module 1.0.

@tool · StructuredTool · BaseTool

RUNNABLES

protocol — composes all

The unifying contract. Every block above implements it. Pipe two together and you get another Runnable. This is LCEL.

Runnable · RunnableLambda · RunnableMap
Architectural Symmetry

Any of the six is interchangeable with any custom Python function wrapped as a RunnableLambda. Once you grasp this, you can pipe arbitrary logic into a chain — formatting, post-processing, validation, retries.

04 / LCEL Basics
LangChain Expression Language
LCEL = the Runnable protocol. Four uniform methods. Every block in the stack honors them. Same code paths whether you call once or batch a thousand.
runnable.py — the four-method contract
The whole protocol
class Runnable:
    """Every LangChain component honors this interface."""

    def invoke(self, input):
        """Run once, sync. The base case."""
        ...

    def batch(self, inputs: list):
        """Run N inputs in parallel."""
        ...

    def stream(self, input):
        """Yield partial results as they arrive."""
        ...

    async def ainvoke(self, input):
        """Async variant. Also: abatch, astream."""
        ...

# Prompts, models, parsers, retrievers, tools, custom fns —
# every one implements these four. That is LCEL.

The Four Runnable Methods

.invoke()

Run once with one input. The simplest case. Returns when complete.

.batch()

Pass a list of inputs. LCEL parallelizes — limited by max_concurrency config.

.stream()

Yields chunks as the model produces them. Critical for low-perceived-latency agent UIs.

.ainvoke() / .abatch() / .astream()

Async variants. Drop straight into the asyncio.gather patterns from Module 1.0.

Why LCEL Matters for Agents

Same code path runs one query, a million queries, or a streaming token feed. No rewrites between dev → eval → production. This is the foundation of Track 3 (eval at scale) and Track 4 (deployment).

05 / The Pipe Operator
One Symbol. Typed Composition.
The pipe | is not syntactic sugar. It is Python operator overloading on Runnable.__or__ that builds a typed graph of stages.

before.py — manual orchestration// MODULE 0.2 STYLE
def run(question: str):
    # 1. Build messages by hand
    msgs = [
        {"role": "system", "content": "..."},
        {"role": "user", "content": question},
    ]
    # 2. Call the SDK
    resp = client.chat.completions.create(
        model="gpt-4o-mini", messages=msgs)
    # 3. Parse manually
    raw = resp.choices[0].message.content
    # 4. Validate manually
    return Answer.model_validate_json(raw)
# ❌ No streaming. No batching. No async. Brittle.
after.py — LCEL chain// SAME LOGIC, ONE PIPELINE
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import PydanticOutputParser

parser = PydanticOutputParser(pydantic_object=Answer)
prompt = ChatPromptTemplate.from_messages([
    ("system", "... {format}"),
    ("human", "{question}"),
]).partial(format=parser.get_format_instructions())

model = ChatOpenAI(model="gpt-4o-mini")

chain = prompt | model | parser
# ✓ Streams, batches, async — all for free.
usage.py — same chain, four execution modes
LCEL uniformity
# Sync, single input
answer = chain.invoke({"question": "What is LCEL?"})

# Sync, batched (parallel under the hood)
answers = chain.batch([
    {"question": q} for q in ["What is LCEL?", "What is a Runnable?"]
])

# Streaming chunks
for chunk in chain.stream({"question": "What is LCEL?"}):
    print(chunk, end="", flush=True)

# Async with concurrency control
answers = await chain.abatch(inputs, config={"max_concurrency": 8})

# Same `chain` object. No rewrites. This is the win.
06 / Prompt Eng. vs RAG vs Fine-Tuning
Three Levers, Briefly Re-Surfaced
Quick recap — the RAG course covered these in depth. Re-anchored here because agents touch all three, often in the same system.
Build-Up Callback

Three ways to make an LLM behave differently. Prompt engineering changes the input. RAG changes the context. Fine-tuning changes the weights. Cost, latency, and persistence go up in that order — and so does effort.

PROMPT ENG.
CHANGES WHAT?
The input string only
COST
$ — token cost only
LATENCY
Same as base call
USE WHEN
Behavior tweaks, format control, style
RAG
CHANGES WHAT?
The context retrieved at runtime
COST
$$ — retrieval + larger prompt
LATENCY
+ retriever round-trip
USE WHEN
Fresh facts, large corpora, citations
FINE-TUNING
CHANGES WHAT?
The model weights themselves
COST
$$$ — train + host
LATENCY
Inference often faster
USE WHEN
Style, narrow domain, prompt compression

At-a-Glance Comparison

Dimension Prompt Eng. RAG Fine-Tuning
Effort to ship Minutes Hours-days Days-weeks
Adds fresh knowledge? No Yes Frozen at train time
Best for agents Tool selection + planning Grounding + memory Specialist sub-agents
Reversible? Trivially Swap index Retrain
Agent-Specific Framing

In production agents you will use all three — prompt eng. for the loop, RAG for memory, fine-tuning for sub-agents. Module 2.5 reframes RAG as a tool. Module 2.11 handles long-term memory architectures. RAG internals stay in the RAG course.

models.py — Pydantic from Module 1.0// 1 / 4 — TYPED CONTRACT
from pydantic import BaseModel, Field
from typing import Literal

class SentimentReport(BaseModel):
    """Structured sentiment analysis result."""
    sentiment: Literal["positive", "neutral", "negative"]
    confidence: float = Field(..., ge=0, le=1)
    keywords: list[str] = Field(..., max_length=5)
    rationale: str = Field(..., max_length=200)
chain.py — the pipe composition// 2 / 4 — 5 LINES OF CHAIN
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import PydanticOutputParser
from models import SentimentReport

parser = PydanticOutputParser(pydantic_object=SentimentReport)

prompt = ChatPromptTemplate.from_messages([
    ("system",
     "You are a sentiment analyst. Be terse and accurate.\n{format}"),
    ("human", "Review:\n{review}"),
]).partial(format=parser.get_format_instructions())

model = ChatOpenAI(model="gpt-4o-mini", temperature=0)

chain = prompt | model | parser    # <-- the whole thing
main.py — execute the chain// 3 / 4 — INVOKE / BATCH / STREAM
from chain import chain

# Single review
report = chain.invoke({"review": "Loved it. Fast support, fair price."})
print(report.sentiment, report.confidence)
# > positive 0.94

# 100 reviews in parallel
reports = chain.batch(
    [{"review": r} for r in reviews_100],
    config={"max_concurrency": 10},
)

# Stream tokens for a live UI
for chunk in chain.stream({"review": "Decent product, sluggish app."}):
    print(chunk, end="", flush=True)
output.log — typed result// 4 / 4 — VALIDATED, USABLE
