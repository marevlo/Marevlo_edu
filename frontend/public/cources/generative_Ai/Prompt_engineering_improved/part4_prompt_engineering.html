06.1 / Systematic Prompt Optimization Frameworks
DSPy: Turn Prompts into Optimizable Code

Manual prompt engineering is like cooking without a recipe — sometimes it works, often it doesn't, and you can never reproduce it. DSPy (Declarative Self-improving Language Programs) changes this by turning prompts into Python programs that a compiler can automatically optimize.

The core insight: Instead of writing prompts by hand, you declare what you want (input → output signatures), provide training examples, and let the compiler figure out the best instructions, few-shot examples, and reasoning chains automatically.

DSPy Core Concepts

1. Signatures — Declare What, Not How

A signature defines the input/output spec without writing the actual prompt text:

Python — DSPy Signatures
import dspy

# Simple signature: question → answer
class ProductQA(dspy.Signature):
    """Answer customer questions about products accurately."""
    question: str = dspy.InputField(desc="customer question")
    context: str = dspy.InputField(desc="product catalog data")
    answer: str = dspy.OutputField(desc="helpful, accurate answer")

# The docstring becomes the task description
# Input/Output fields define the prompt structure
# DSPy compiles this into an optimal prompt automatically

2. Modules — Composable Building Blocks

Python — DSPy Modules
# Predict = simple LLM call
qa = dspy.Predict(ProductQA)

# ChainOfThought = forces step-by-step reasoning
qa_cot = dspy.ChainOfThought(ProductQA)

# ReAct = tool-using agent
qa_react = dspy.ReAct(ProductQA, tools=[search_catalog, check_stock])

# ProgramOfThought = generates + executes code
pricing = dspy.ProgramOfThought("cost, margin, competitor_price -> optimal_price")

3. Compilers (Optimizers) — The Magic

This is where DSPy becomes powerful. Instead of manually tweaking prompts, you provide a metric and training examples, and the compiler finds the best prompt automatically:

Python — DSPy Optimization
def accuracy_metric(example, prediction, trace=None):
    return example.answer.lower() in prediction.answer.lower()

# Load training data
trainset = [
    dspy.Example(
        question="Is this laptop good for gaming?",
        context="RTX 4060, 16GB RAM, 144Hz display",
        answer="Yes, the RTX 4060 GPU and 144Hz display make it excellent for gaming"
    ),
    # ... 20-50 examples
]

# BootstrapFewShot: auto-selects best few-shot examples
optimizer = dspy.BootstrapFewShot(
    metric=accuracy_metric,
    max_bootstrapped_demos=4,
    max_labeled_demos=4
)
optimized_qa = optimizer.compile(qa_cot, trainset=trainset)

# MIPROv2: optimizes BOTH instructions AND examples
optimizer2 = dspy.MIPROv2(
    metric=accuracy_metric,
    num_candidates=10,
    init_temperature=1.0
)
best_qa = optimizer2.compile(qa_cot, trainset=trainset)

06.2 / Evaluation & Testing Frameworks
The Missing Piece: Measuring Prompt Quality

Most teams write prompts and "vibe check" the output. This doesn't scale. You need systematic evaluation: golden datasets, automated metrics, and testing pipelines.

Building Evaluation Datasets

Golden Sets

A golden set is a curated collection of input-output pairs with verified correct answers. It's your ground truth:

Python — Golden Set Structure
golden_set = [
    {
        "input": "Is this laptop good for video editing?",
        "context": "M2 Pro chip, 32GB RAM, 16-inch Liquid Retina XDR",
        "expected_answer": "Yes — the M2 Pro and 32GB RAM handle 4K editing smoothly",
        "expected_citations": ["M2 Pro", "32GB RAM"],
        "tags": ["product_qa", "technical", "positive"]
    },
    # ... 50-200 carefully curated examples
]

LLM-as-Judge (G-Eval)

When human evaluation is expensive, use a stronger LLM to judge the output of a weaker one:

Python — LLM-as-Judge
judge_prompt = """Rate this product description from 1-5 on:
1. Accuracy (facts match specs): __/5
2. Persuasiveness (would you buy?): __/5
3. Tone (matches brand voice): __/5
4. Length (within 100-150 words): __/5

Product specs: {specs}
Generated description: {description}

Return JSON: {"accuracy": N, "persuasion": N, "tone": N, "length": N}"""

# Use GPT-4o as judge, test outputs from GPT-4o-mini
judge_response = client.chat.completions.create(

Automated Testing Pipelines

Python — Prompt Testing Pipeline
import json
from datetime import datetime

class PromptTestRunner:
    def __init__(self, golden_set, prompt_template, model="gpt-4o-mini"):
        self.golden_set = golden_set
        self.prompt = prompt_template
        self.model = model
        self.results = []

    def run_all(self):
        for case in self.golden_set:
            output = self.call_llm(case["input"], case["context"])
            score = self.evaluate(output, case["expected_answer"])
            self.results.append({
                "input": case["input"],
                "output": output,
                "expected": case["expected_answer"],
                "score": score,
                "timestamp": datetime.now().isoformat()
            })
        return self.summary()

    def summary(self):
        scores = [r["score"] for r in self.results]
        return {
            "total": len(scores),
            "accuracy": sum(scores) / len(scores),
            "passed": sum(1 for s in scores if s >= 0.8),
            "failed": sum(1 for s in scores if s < 0.8)
        }

A/B Testing Prompts in Production

Deploy two prompt versions simultaneously, route traffic 50/50, and measure which performs better on real user interactions:

Python — A/B Test Router
import random

def ab_route(user_id, prompt_a, prompt_b, split=0.5):
    # Deterministic assignment based on user_id
    bucket = hash(user_id) % 100
    variant = "A" if bucket < split * 100 else "B"

    prompt = prompt_a if variant == "A" else prompt_b
    response = call_llm(prompt)

    # Log for analysis
    log_experiment({
        "user_id": user_id,
        "variant": variant,
        "response": response,
        "latency_ms": elapsed,

    })
    return response

06.3 / Prompt Management & Versioning
Treat Prompts as Code

In production, prompts are not casual text — they are critical infrastructure. A bad prompt change can break your entire customer experience. You need versioning, rollback, and experimentation tracking.

Prompt Version Timeline — InteractiveANIMATED

Watch prompt versions evolve — each version shows accuracy changes and what was modified.

Prompt Registry Pattern

Python — Prompt Registry
class PromptRegistry:
    def __init__(self, storage="postgres"):
        self.db = connect(storage)

    def register(self, name, template, version, metadata={}):
        """Save a new prompt version with metadata"""
        self.db.insert({
            "name": name,          # e.g., "product_qa_v3"
            "template": template,  # the actual prompt text
            "version": version,    # semantic version "3.1.0"
            "created": now(),
            "author": metadata.get("author"),
            "eval_score": metadata.get("accuracy"),
            "model": metadata.get("model"),
            "is_active": False   # promote explicitly
        })

    def promote(self, name, version):
        """Make this version the active one in production"""
        self.db.update({"name": name}, {"is_active": False})
        self.db.update({"name": name, "version": version},
                       {"is_active": True})

    def rollback(self, name):
        """Revert to previous version"""
        versions = self.db.find({"name": name}).sort("version", -1)
        self.promote(name, versions[1]["version"])

Cost Control & Model Routing

def smart_route(query, complexity_threshold=0.6):
    """Route to mini or full model based on query complexity"""
    complexity = estimate_complexity(query)  # 0-1 score

    if complexity < complexity_threshold:
        model = "gpt-4o-mini"    # 10x cheaper, 3x faster
        cost_per_1k = 0.00015
    else:
        model = "gpt-4o"         # full power for complex Qs
        cost_per_1k = 0.0025

    response = call_llm(query, model=model)
    log_cost(model, response.usage.total_tokens, cost_per_1k)
    return response

06.4 / Structured Output & Reliability
Force the LLM to Follow Your Schema

In production, you can't have the LLM return free-form text when your code expects JSON. Structured output techniques guarantee the model's response matches your exact schema.

JSON Mode + Pydantic Validation

Python — Pydantic + OpenAI Structured Output
from pydantic import BaseModel, Field
from typing import List
from openai import OpenAI

class ProductRecommendation(BaseModel):
    product_name: str = Field(description="Name of the product")
    price: float = Field(ge=0, description="Price in INR")
    match_score: float = Field(ge=0, le=1, description="0-1 relevance")
    reason: str = Field(description="Why this product matches")

class RecommendationResponse(BaseModel):
    recommendations: List[ProductRecommendation]
    total_found: int

# OpenAI will GUARANTEE the output matches this schema
response = client.beta.chat.completions.parse(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Best laptops under 60k for students"}],
    response_format=RecommendationResponse
)
recs = response.choices[0].message.parsed  # typed Pydantic object!

Output Guardrails

Python — Output Validation Pipeline
def validate_output(response, schema, rules):
    """Multi-layer output validation"""
    # Layer 1: Schema validation (Pydantic)
    try:
        parsed = schema.model_validate_json(response)
    except ValidationError as e:
        return {"valid": False, "error": "schema_violation", "details": str(e)}

    # Layer 2: Business rules
    if parsed.price < 0:
        return {"valid": False, "error": "negative_price"}
    if len(parsed.reason) < 10:
        return {"valid": False, "error": "reason_too_short"}

    # Layer 3: Content safety
    if contains_harmful_content(parsed.reason):
        return {"valid": False, "error": "content_safety"}

    return {"valid": True, "data": parsed}

06.5 / Prompt Chaining & Workflow Orchestration
Complex Tasks Need Multiple Steps

Real production tasks rarely fit in one prompt. A customer support system might need to: classify the issue → retrieve relevant policy → generate response → check for compliance → format output. This is prompt chaining.

Three Chain Patterns

1. Sequential Chain

Output of prompt A becomes input of prompt B. Simple and predictable:

3. Conditional Chain

Branch based on the output of a classifier:

Python — Conditional Chain
async def support_chain(query):
    # Step 1: Classify intent
    intent = await classify(query)  # "order_status" | "complaint" | "product_qa"

    # Step 2: Branch based on intent
    if intent == "order_status":
        data = await fetch_order(query)
        response = await generate_status_reply(data)
    elif intent == "complaint":
        sentiment = await analyze_severity(query)
        policy = await retrieve_policy(intent, sentiment)
        response = await generate_complaint_reply(query, policy, sentiment)
    else:
        docs = await search_catalog(query)
        response = await generate_product_reply(query, docs)

    # Step 3: Validate and return
    validated = await validate_output(response)
    return validated
