03.1 / Chain-of-Thought
Think Step by Step Before Answering

This method improves an LLM's ability to solve complex reasoning problems by explicitly instructing it to think step by step before giving the final answer.

How it works: The prompt encourages the model to think step-by-step, either by showing reasoning examples or explicitly asking for reasoning before the final answer.

03.2 / Auto-CoT
Let the Model Create Its Own Examples

This is an automated version of regular Chain-of-Thought where the model itself creates step-by-step examples.

How it works: You select a set of questions from a dataset, prompt the LLM with "Let's think step-by-step" to generate multiple reasoning chains per question, then cluster and pick diverse chains to form high-quality few-shot demonstrations that are used in the final prompt for better execution on new problems.

💰Example 1: Multi-step Pricing & Margin Calculation▶ FULL PROMPT
Auto-CoT — Pricing (Complete JSON)
{
  "instruction": "Automatically generate 3 similar pricing problems and
   solve them step-by-step before solving the main problem.",
  "requirements": [
    "Each example must involve cost price, target margin, and discount impact.",
    "Use different reasoning styles for each example.",
    "Ensure each chain arrives at a logically correct final answer.",
    "Keep examples concise but structured."
  ],
  "reasoning_chains": {
    "chain_a": "Calculate minimum margin price first.",
    "chain_b": "Estimate revenue impact first.",
    "chain_c": "Compare competitor or psychological pricing angle first."
  },
  "output_requirements": [
    "Provide clear numbered steps.",
    "Provide final pricing recommendation."
  ],
  "problem": "<PUT_THE_QUESTION_HERE>"
}
🤔 Question

Cost ₹1500. Normal price ₹2499. Target min 25% margin. If discounted to ₹1999, expect +80% sales volume. What should be the sale price?

CoT vs Auto-CoT Comparison

Chain of Thought (CoT)Automatic Chain of Thought (Auto-CoT)
Manually written by humans (in few-shot CoT) — each example includes question + detailed reasoning steps + answer. Zero-shot CoT skips examples entirely and just adds the phrase.Automated method: 1) Cluster a large set of questions for diversity, 2) Pick one representative question per cluster, 3) Generate its reasoning chain using zero-shot CoT, 4) Apply simple filters to remove bad chains.

03.3 / Consistency
Multiple Paths, Majority Vote

Self-Consistency is a decoding strategy that improves reasoning accuracy by generating multiple diverse step-by-step reasoning paths from a language model and selecting the most consistent final answer across them, rather than relying on a single greedy reasoning path.

How it works: Instead of producing one chain-of-thought, the model samples several different reasoning chains and then chooses the final answer that appears most frequently or consistently across those sampled solutions.

📊Example 1: Estimating Sales Lift from a Discount▶ FULL PROMPT
Self-Consistency — Sales Lift (Complete YAML)
instruction: >
  You are a senior e-commerce pricing analyst specializing in
  demand forecasting and profit optimization. Your task is to
  estimate sales lift and profit impact.

task:
  description: Generate 5 independent estimation paths.

baseline_data:
  - Current volume
  - Selling price
  - Cost
  - Planned discount
  - Typical accessory lift range

estimation_steps:
  - Estimate expected percentage volume increase (60–100%)
  - Calculate new daily sales volume
  - Calculate old daily profit
  - Calculate new daily profit after discount
  - Conclude whether profit increases, decreases, or stays same

path_rules:
  - Each path must use a slightly different % lift assumption
  - Show short structured calculations
  - Produce the final answer in the required format

aggregation_logic:
  after_all_paths:
    - Count how many times each outcome appears
    - Identify the most frequent estimated new volume
    - Identify the most frequent profit direction

output_format:
  paths:
    - "Path 1: New volume = X units/day. Profit: ___"
    - "Path 2: New volume = X units/day. Profit: ___"
    - "Path 3: New volume = X units/day. Profit: ___"
    - "Path 4: New volume = X units/day. Profit: ___"
    - "Path 5: New volume = X units/day. Profit: ___"
  final_decision: >
    Final Majority Decision: "New volume: X units/day.
    Profit: increases / decreases / same."

input: "<PUT_THE_QUESTION_HERE>"

03.4 / Logical CoT
Verify Each Step with Logic

LogiCoT Prompting improves multi-step reasoning by verifying each intermediate step using logical contradiction checks and iteratively correcting errors before producing the final answer.

How it looks: The model generates reasoning steps, then symbolically verifies each step by testing if assuming it false leads to contradiction; incorrect steps are revised and re-generated in an iterative think-verify-revise loop.

💵Example 1: Discount Approval for Bulk Order (If-Then + Margin Logic)▶ FULL PROMPT
LogiCoT — Pricing Discount (Complete Markdown Prompt)
# Logical Pricing Discount Evaluation Prompt (LogiCoT)

## Role
You are a **pricing logic expert** responsible for evaluating
bulk discount requests using **formal logical reasoning**.

Your task is to apply **LogiCoT** reasoning. Follow a structured
deduction process and ensure every conclusion logically follows
from the given premises and rules.

## Step 1: State the Premises
List all given facts clearly:
- Cost per unit
- Minimum required margin
- Normal selling price
- Customer requested price or discount
- Total order value or quantity

## Step 2: Derive Minimum Price for Required Margin
Use the formula:
  Minimum Price = Cost / (1 − Margin)

## Step 3: Apply Logical Rules
**Rule A**: If proposed price ≥ required minimum → Approve Full Discount
**Rule B**: Else if expected volume lift > 200% AND margin ≥ 15%
           → Partially approve discount

## Step 4: Calculate the Actual Margin at Proposed Price
  Margin = (Price − Cost) / Price

## Step 5: Check for Contradictions
Ensure:
- The calculated margin is **positive**
- The final decision **follows the defined rules**
If a contradiction exists, **reject the request**.

## Step 6: Output Format
**Decision:** Approve Full / Partial / Reject
**Logical Justification:** Derived margin + rule evaluation
  + contradiction checks
🤔 Question

Cost ₹1200/unit. Min margin 25%. Normal price ₹1999. Customer wants 50 units at ₹1500 each (total ₹75,000).

📋Example 2: Return Policy Contradiction Check▶ FULL PROMPT
LogiCoT — Return Policy (Complete Markdown Prompt)
# Return Policy Logical Reasoning Prompt (LogiCoT)

## Role
You are a **return policy logician**. Analyze return policy
statements using **LogiCoT** reasoning and determine whether
a logical conclusion follows from the given premises.

Use **formal deductive reasoning**.
Structure: Premises → Derivations → Consistency Check
           → Contradiction Test → Conclusion

## Step 1: State the Premises
- **Premise 1:** All returns within **7 days** are accepted **if unopened**.
- **Premise 2:** **Some unopened returns are denied** because of
  hygiene rules (e.g., underwear).

## Step 2: Convert to Logical Form
- Returns within 7 days ∩ Unopened ⊆ Accepted
- Some Unopened ⊆ Denied

## Step 3: Evaluate Simultaneous Validity
Check whether:
- An unopened return could fall into both Accepted AND Denied
- Policy exceptions may override general rules

## Step 4: Test the Assumption
**Assumption:** All returns within 7 days that are unopened
are accepted.
Compare with Premise 2: Some unopened returns are denied.
→ Does this create a logical contradiction?

## Step 5: Check for Contradictions
If contradiction occurs:
→ The general rule must contain exceptions
→ Determine: can we conclude some returns within 7 days are denied?

## Final Output
**Logical Conclusion:** Yes / No
**Justification:** Reference premises + contradiction test
                   + logical deduction
🤔 Question

Can we conclude that some returns within 7 days are denied?

When to use it: Use LogiCoT for tasks requiring rigorous logical deduction, such as mathematical proofs, complex puzzles, scientific hypothesis testing, or legal reasoning, where standard CoT might introduce unchecked errors.

ProsCons
Highly suitable for reducing logical inconsistencies in intricate, verification-heavy scenarios, leading to measurable performance gains over CoTNot suitable for simple, non-reasoning tasks like factual recall or creative writing, as the iterative loop can increase computation time unnecessarily

Learn More

Concept: Verify reasoning steps and use symbolic or formal logic tools when natural language reasoning is too loose.

03.5 / Chain-of-Symbol
Symbols Instead of Words

This method improves an LLM's ability to reason about spatial, structural, or relational problems by replacing ambiguous natural language with compact symbolic representations that encode objects, positions, and relations explicitly.

How it works: The prompt converts entities and relationships into structured symbols (grids, coordinates, tokens), enabling the model to manipulate symbolic states step-by-step instead of interpreting vague language descriptions.

🖥️Example 1: Product Page Layout Optimization (UI Spatial Reasoning)▶ FULL PROMPT
CoS — UI Layout (Complete YAML)
instruction: >
  You are a UX strategist and interface planner. Solve the given
  UX/UI problem using Chain-of-Symbol (CoS) reasoning, where
  layouts and relationships are represented using symbols instead
  of long textual reasoning.

symbol_system:
  grid_structure:
    vertical: "V1, V2, V3, V4 (top to bottom)"
    horizontal: "L, M, N (left to right)"
  symbols:
    "^": "above"
    "v": "below"
    "R>": "to the right of"
    "<L": "to the left of"
    "P1": "high-priority element"
    "P2": "supporting element"
    "NO": "avoid overlap or clutter"
    "SWP": "swap positions"

optimization_goals:
  - Critical actions must appear early and visible
  - Important information before secondary content
  - Trust signals before decisions
  - Avoid clutter and overlapping elements
  - Maintain logical reading flow

steps:
  step_1: Represent current interface using symbolic grid
  step_2: Mark priority elements with P1
  step_3: Chain-of-Symbol adjustments (step by step)
    examples:
      - "Hero ^ Price"
      - "Price P1 R> CTA P1"
      - "CTA P1 ^ Reviews"
      - "NO overlap between Hero and CTA"
      - "Description v Reviews"
  step_4: Provide optimized layout using symbolic grid
  step_5: Explain improvements

output:
  format: >
    Optimized interface blueprint using symbolic grid
    + short explanation.

input: "<PUT_THE_UX_QUERY_HERE>"

symbol_system:

03.6 / Tree-of-Thoughts
Explore Multiple Branches, Pick the Best

This method improves an LLM's ability to solve complex, multi-path reasoning problems by exploring multiple possible reasoning directions simultaneously in a structured tree and selecting the most promising path.

How it works: The model generates multiple intermediate reasoning "thoughts," evaluates them, and explores them using search strategies like breadth-first or depth-first to reach the best solution.

⚡Example 1: Choosing the Best Flash Sale Strategy▶ FULL PROMPT
ToT — Flash Sale Strategy (Complete YAML)
instruction: >
  You are a marketing strategist and revenue optimization expert.
  Solve the problem using Tree-of-Thought (ToT) reasoning. Instead
  of generating one answer, explore multiple strategy branches,
  evaluate them, and choose the best one.

reasoning_method:
  name: Tree-of-Thought
  process:
    - Decompose the problem
    - Generate multiple strategic branches

03.7 / Graph-of-Thoughts
Ideas That Connect, Merge, and Loop

This method improves an LLM's ability to solve complex, multi-path reasoning problems by exploring multiple possible reasoning directions simultaneously in a structured tree and selecting the most promising path.

How it works: The model generates multiple intermediate reasoning "thoughts," evaluates them, and explores them using search strategies like breadth-first or depth-first to reach the best solution.

  name: Graph-of-Thought
  process:

    - Connect dependencies between nodes

    - Refine nodes using constraints

steps:
  step_1: Generate 4–5 bundle concept nodes (different customer needs)

03.9 / Logic-of-Thought
Embed Formal Logic Into Prompts

Logic-of-Thought (LoT) Prompting improves reasoning consistency by embedding explicit logical structure (propositions and relations) into prompts so model conclusions remain aligned with intermediate reasoning.

How it looks: The method extracts logical statements from text, expands them using formal logic rules, and reinserts them into the prompt so the model reasons over logically complete information.

🎟️Example 1: New-User Coupon Eligibility (Propositional Expansion)▶ FULL PROMPT + I/O
LoT — Coupon Eligibility (Complete YAML)
instruction: >
  You are a policy logic engine. Apply Logic-of-Thought (LoT) reasoning.

steps:
  - Extract atomic propositions from policy and customer query (P1, P2...)
  - Apply logical rules: modus ponens, contraposition, conjunction
  - Convert expanded logic into clear natural language
  - Append logical reasoning to query and decide outcome

policy:
  - "If first-time buyer AND cart ≥ ₹2000 AND no sale items
     → 15% new-user coupon applies."
  - "If any sale item present → coupon blocked."

output_format:
  propositions: "List of P1, P2..."
  logical_expansion: "Applied logical rules"
  explanation: "Natural language reasoning"
  decision: "Coupon applies / Coupon blocked"

input:
  customer_query: "<PUT_QUERY_HERE>"
Input

"First-time buyer here. My cart is ₹2200 including a 20% off T-shirt. Can I apply the new-user 15% coupon?"

Output

Propositions: P1 = First-time buyer (true), P2 = Cart ≥ ₹2000 (true), P3 = Sale item present (true)
Logical Expansion: Policy rule: (P1 ∧ P2 ∧ ¬P3) → Coupon applies. Since P3 is true, ¬P3 is false → condition fails → ¬eligible.
Explanation: Although the customer is a first-time buyer and the cart exceeds ₹2000, the presence of a sale item violates the policy condition.
Decision: Coupon blocked.
