Module 04
Reducing Hallucination in LLMs
Hallucinations happen when the model makes up facts that aren't true. It's common because models predict based on patterns, not perfect knowledge.
4.1 / ReAct (Reason + Act) Prompting
Think → Act → Observe → Repeat

ReAct is a prompting framework where an LLM combines reasoning steps and external actions (like searching or tool use), allowing it to gather information, update plans, etc.

How it works: A simple repeating loop of Thought → Action → Observation → repeat until the answer is ready.

ReAct Circular Loop — Live AnimationANIMATED

Watch the orbiting particle cycle through Thought → Action → Observation. The loop counter increments each full cycle.

🔍Example 1: Finding the Best Matching Product▶ FULL PROMPT + I/O
ReAct — Product Search (Complete YAML)
instruction: >
  You are a smart shopping assistant. Use the ReAct
  reasoning format strictly.

format:
  loop:
    Thought: "Reason step by step about the query"
    Action: "Take a specific action (e.g., search catalog,
             compare items, calculate discount)"
    Observation: "Record the result of the action"

rule: "Repeat Thought → Action → Observation until the
       final answer is ready."

input:
  customer_query: "<PUT_QUERY_HERE>"

output:
  reasoning_trace: "ReAct steps"
  final_answer: "Recommended product or response"
Input

"I need wireless earbuds under ₹2000 with good noise cancellation and at least 20 hours battery."

Output

Thought: Need to find earbuds under ₹2000 with noise cancellation and ≥20h battery.
Action: Search catalog for "wireless earbuds noise cancelling under ₹2000 20h battery".
Observation: Found 3 models — Model X: ₹1799, ANC, 22h; Model Y: ₹1999, ANC, 18h; Model Z: ₹1499, no ANC.
Thought: Model X satisfies price (<₹2000), ANC present, and battery ≥20h.
Final output: Recommend Model X at ₹1799.

🎁Example 2: Personalized Bundle Recommendation with Price Check▶ FULL PROMPT + I/O
ReAct — Bundle Builder (Complete YAML)
instruction: >
  You are a bundling expert. Use a ReAct loop to create
  product bundles.

loop_format:
  - Thought: "Reason about the bundling step"
  - Action: "search_related_products(item) |
             calculate_bundle_price(items) | check_stock"
  - Observation: "Result of the action"

rule: "Repeat Thought → Action → Observation until the
       bundle is finalized."

output:
  reasoning_trace: "ReAct steps"
  final_answer: "Bundle item and price"
Input

Customer just added smartphone (₹24999) to cart. Suggest a value bundle.

Output

Thought: Smartphone buyer often needs protection + accessories.
Action: search_related_products("smartphone accessories")
Observation: Top: case ₹799, screen protector ₹499, charger ₹999.
Thought: Bundle case + protector = good protection, total add ₹1298.
Action: calculate_bundle_price([smartphone, case, protector])
Observation: Bundle price ₹26298 (5% discount).
Final output: Recommend "Protection Bundle": case + screen protector for ₹26298 total.

When to use it: For tasks needing real-world info, tools, or interaction (search, calculators, APIs) — very powerful in agents or tool-using AIs.

ProsCons
Enhances interpretability and accuracy in interactive or knowledge-intensive tasks, handles exceptions wellNeeds tool access (not plain chat); more complex to set up

Learn More

Concept: Interleave reasoning with tool calls: think, act, observe, repeat.

04.2 / Chain-of-Verification (CoVe)
Write, Verify, Fix

CoVe is a structured prompting method where the model first produces an answer, then deliberately verifies its own claims through targeted questions and revises the final response to reduce hallucinations and logical errors.

How it works: The model generates an initial response, creates verification questions about its claims, answers them independently, and then revises the final answer using the verified facts.

Do you know? CoVe is like making the AI write an answer and then become its own strict teacher — it creates a checklist and fact-checks itself before finalizing!

CoVe 4-Step Pipeline — Live AnimationANIMATED

Watch data packets flow through Draft → Verify → Answer → Revise with transformation effects at each stage.

Example 1: Aggregating Customer Review Highlights

Task: Summarize common pros/cons from reviews of smartphone. The CoVe flow generates a draft summary, creates verification questions about each claim, answers each independently from the reviews, then revises to remove unverified claims.

Example 2: Verifying a Product Feature Summary

Task: Summarize key features of wireless earbuds accurately (no exaggeration).

CoVe Flow (multi-prompt):

  1. Draft Prompt: "Summarize the main features of XYZ wireless earbuds from the product specs: noise cancelling, 30-hour battery, IPX5 waterproof, deep bass."
    Draft: "XYZ earbuds offer top noise cancelling, incredible 40-hour battery, fully waterproof, and booming bass — best in class!"
  2. Plan Verification Questions Prompt: "Given the draft summary above and original specs, generate 4 verification questions to check accuracy, completeness, and no exaggeration."
    Questions: 1. What is the actual battery life claimed? 2. Is "top noise cancelling" supported by specs? 3. What is the exact waterproof rating? 4. Does it say "best in class"?
  3. Independent Answer Questions Prompt: "Answer this verification question based only on original product specs: [question]"
    Answers: 1. 30 hours. 2. No — specs say "active noise cancellation" but no "top". 3. IPX5. 4. No such claim.
  4. Revise Prompt: "Original draft: [draft]. Verification Q&A: [paste answers]. Revise the summary to be 100% factual and accurate."
    Final: "XYZ wireless earbuds feature active noise cancellation, 30-hour battery life, IPX5 water resistance, and deep bass sound."

When to use it: Best for hallucination-prone tasks such as multi-hop question answering (e.g., HotpotQA), fact extraction, list-based queries, or long-form content generation where deliberate self-verification identifies inconsistencies better than single-pass methods like CoT.

ProsCons
Effectively reduces hallucinations by promoting self-correction; suitable for QA, verification, and generation needing coherenceRequires multiple steps, increasing latency and compute; not ideal for simple queries or real-time applications

Learn More

Concept: Generate a draft, create verification questions, answer them independently, then revise.

04.3 / Chain-of-Note (CoN)
Read Carefully, Ignore the Noise

Chain-of-Note prompting improves retrieval-augmented language models by forcing them to explicitly evaluate, filter, and summarize retrieved documents into structured "notes," helping the model ignore noisy or irrelevant information and correctly admit when knowledge is insufficient.

How it works: The model first converts retrieved documents into concise relevance-checked notes, then reasons only over those validated notes to produce an answer or decide "unknown."

Do you know? When you give the AI lots of documents or search results, it often gets confused by junk. CoN forces it to read carefully, make short honest "notes", and ignore the noise.

Watch chunks get scanned — relevant ones glow green and pass, noisy ones get rejected with a red flash.

📋Example 1: Answering Policy Question from Mixed Docs▶ FULL PROMPT + I/O
CoN — Policy Q&A (Complete YAML)
instruction: >
  You are a policy Q&A agent. Use Chain-of-Note reasoning.

process:
  - For each retrieved policy/document chunk, write a Note.
  - Note fields: relevance to "return window for custom items",
    key rule, contradiction/noise.
  - Chain reliable notes and ignore noisy or contradictory ones.
  - Answer only using the reliable notes.

output:
  notes: []
  final_answer: ""

input:
  query: "<PUT_QUERY_HERE>"
Input

"Can I return a custom-printed T-shirt?"
Retrieved:
Chunk 1: "Standard returns 15 days..."
Chunk 2: "Custom/personalized items non-returnable except defect."
Chunk 3: "Sale items final sale."
Chunk 4: "Free shipping over ₹999."

Output — Notes

• Chunk 1: Not relevant (general 15-day return window).
• Chunk 2: Relevant. Key rule: Custom/personalized items are non-returnable except for manufacturing defects.
• Chunk 3: Not relevant (sale item rule).
• Chunk 4: Noise (shipping policy).

Note utilized: Chunk 2
Final Answer: No, custom-printed T-shirts are non-returnable except for manufacturing defects.

🎧Example 2: Product Recommendation with Catalog + Reviews▶ FULL PROMPT + I/O
CoN — Product Recommendation (Complete YAML)
instruction: >
  You are a recommender. Use Chain-of-Note reasoning.

process:
  - For each retrieved chunk, write a short Note
    (relevant / irrelevant + key info).
  - Ignore noisy or irrelevant chunks.
  - Recommend only from reliable notes that satisfy the query.

input:
  query: "<PUT_QUERY_HERE>"
  retrieved_chunks: []

output_format:
  notes: []
  recommendation: ""
Input

Recommend budget noise-cancelling earbuds under ₹2500.
Retrieved chunks:
Chunk 1: "Model A: ₹1999, ANC, 20h battery."
Chunk 2: "Model B review: Best ANC under 3k!"
Chunk 3: "Old blog post about wired headphones..."
Chunk 4: "Model C: ₹2999, no ANC."

Output — Notes

Chunk 1: Relevant, key specs.
Chunk 2: Relevant, positive sentiment for Model A/B range.
Chunk 3: Irrelevant (wired, old).
Chunk 4: Relevant but over budget + no ANC.

Output: "Model A at ₹1999 — solid ANC and battery, good reviews in budget range."

When to use it: Ideal for open-domain question-answering tasks where retrieved documents may contain noise, irrelevance, or gaps.

ProsCons
Enhances robustness in noisy environments by improving exact match accuracy and rejection ratesIncreases response latency and token usage; less efficient for clean retrievals

04.4 / Chain-of-Knowledge (CoK)
Build Knowledge Piece by Piece

This prompting method improves factual accuracy and complex reasoning by guiding the model to iteratively gather, verify, and integrate knowledge from multiple sources before producing an answer, reducing hallucinations compared to single-step reasoning.

How it works: The prompt forces the model to first frame the problem, then progressively retrieve and validate relevant knowledge from internal or external sources, and finally synthesize a grounded answer.

Do you know? This one makes the AI act like a detective — it collects facts piece by piece from different areas, connects them, then gives a super grounded answer.

💰Example 1: Deciding Dynamic Pricing for a Flash Sale▶ FULL PROMPT + I/O
CoK — Flash Sale Pricing (Complete YAML)
instruction: >
  You are a pricing strategist. Use Chain-of-Knowledge reasoning.
  Build the knowledge chain step-by-step, then give a recommendation.

chain_steps:
  - Stock level and demand trend
  - Average daily sales (no discount)
  - Expected sales lift from discount
  - Profit margin (full vs discounted)
  - Competitor pricing
  - Target customer segment
  - Brand dilution risk

output_format:
  knowledge_chain: []
  recommendation: "Yes | No"
  suggested_discount: ""
  reasoning: ""

input:
  question: "<PUT_QUESTION_HERE>"
Input

Should we run a 25% flash sale on wireless earbuds (normal ₹2499, cost ₹1499) for the next 24 hours?

Chain steps applied: 1. Current stock level and trend. 2. Average daily sales without discount. 3. Expected sales lift from 25% discount. 4. Profit margin at full price vs discounted. 5. Competitor pricing. 6. Customer segment most likely to buy. 7. Risk of devaluing brand or training customers to wait for sales.

Output

Knowledge: High stock + strong lift expected + competitors already discounting
Output: Yes, but only 20% to protect margin.

🚨Example 2: Approve Instant Refund for "Not Received" Claim (₹12,999 UPI)?▶ FULL PROMPT + I/O
CoK — Refund Risk (Complete YAML)
instruction: >
  You are a risk decision agent. Use Chain-of-Knowledge reasoning.
  Build knowledge step-by-step across domains, refine rationales,
  then give the final decision.

process:
  step_1_initial_rationales:
    - Generate 2–3 preliminary hypotheses about the case.
  step_2_identify_domains:
    - Determine knowledge domains needed (tracking, payment risk,
      fraud signals, customer history, policy).
  step_3_build_knowledge_chain:
    - Insert verified facts from each domain.
  step_4_refine_reasoning:
    - Update rationales using the collected knowledge.
  step_5_decision:
    - Combine the knowledge chain to produce final decision.

output_format:
  rationales: []
  knowledge_chain: []
  conclusion: ""
  action: ""

input:
  case: "<PUT_CASE_DETAILS_HERE>"

Initial rationales:

  • Rationale 1: "If tracking shows delivered, probably not genuine non-delivery."
  • Rationale 2: "But UPI payments sometimes have fraud patterns."

Dynamic Knowledge Building — 6 domains:

  • Domain 1 (Tracking): Order ORD98765 — "Delivered" 2 days ago, proof-of-delivery photo shows package at gate, signed as received.
  • Domain 2 (Payment Risk): Paid via UPI (instant, low chargeback protection), new account created 3 days before order.
  • Domain 3 (Customer History): First order, no previous returns or disputes.
  • Domain 4 (Fraud Patterns): Common scam = claim "not received" after COD/UPI delivery with photo proof (~18% of such claims were fraudulent per internal data).
  • Domain 5 (Policy): Investigate "not received" claims with proof required; refund only if confirmed lost after 72-hour re-verification.
  • Domain 6 (Business Impact): Instant refund = ₹12,999 fraud exposure. Investigation = possible negative review but protects revenue.

Consolidated: "Strong delivery proof + new account + UPI + known fraud pattern → high fraud likelihood → policy requires investigation."

Final Output

Do not approve instant refund. Send polite messages asking customer to check with family/neighbor/guard (reference delivery photo), allow 72 hours for re-verification. If no update, escalate to full investigation/refund only if confirmed missing.

When to use it: Best for knowledge-intensive QA spanning multiple domains, outperforming CoT by dynamically grounding responses in heterogeneous sources.

ProsCons
Significantly improves factual reliability by integrating diverse sourcesAdds computational overhead; unsuitable for simple factual questions

Learn More

Concept: Decompose a question, retrieve and verify evidence for each piece, then synthesize.

Module 05
Context Optimization Techniques
LLMs support 128K to 1M+ tokens, but real apps can exceed limits. Exceeding increases latency, cost, and sometimes reduces performance.

Prompts may contain instructions, conversation history, examples, documents, and user inputs. When all included together, tokens become too large. Context optimization techniques help reduce tokens while keeping the most important information.

05.1 / Naive Truncation
The Simplest Cut

5.1.1 Removing Extra Tokens

Tokens are simply removed when total exceeds the limit. Most APIs use left truncation (remove oldest first). Some use right truncation (remove from end), less common since recent info is usually more important.

ProsCons
Zero compute, trivial to implement, predictable token usageNon-semantic — critical info discarded arbitrarily. Causes hallucinations, errors, loss of intent.

5.1.2 Sliding Window Approach

Keeps most recent context while summarizing key info from earlier parts:

  • Extract relevant sections based on keywords or similarity
  • Maintain fixed-size buffer of recent conversation turns
  • Optionally include short summary of earlier interactions

Learn More

Concept: Drop tokens, keep a sliding window of recent turns, or summarize older context.

05.3 / Natural Language Compression
Shrink Without Losing Meaning

Advanced techniques reduce prompt size by rewriting or summarizing while keeping meaning intact. Can compress text 5–20×.

5.3.1 Prompt Compression

Methods like LLMLingua and LongLLMLingua use smaller LMs to remove low-information tokens: stopwords, redundant phrases, repeated formatting/boilerplate. Soft prompts replace long text with learned token embeddings.

5.3.2 Token Optimization

Removing excessive formatting, merging repeated instructions, shortening descriptions. Some methods estimate token importance using entropy/perplexity scores.

5.3.3 Semantic Summarization (ACON)

ACON (2025) iteratively improves summarization for long-running AI systems:

05.4 / Dynamic Context Prompt Optimization (DCPO)
Adjust as Conversation Progresses

5.4.1 Context Rewriting

Every 5–10 turns or at 80% token limit, the system rewrites: summarize history, extract entities, remove repeated info, rewrite instructions concisely.

5.4.2 Prompt Refinement Loops

After generating output, the model evaluates whether it meets criteria. If not, adjusts prompt before next output. Some frameworks compress reasoning traces over time.

5.4.3 Adaptive Prompts

  • Increase/decrease compression based on remaining token budget
  • Inject stored memory when needed
  • Switch instruction style (detailed → concise) as task becomes clearer

Advanced agent frameworks monitor token usage and auto-switch to summarized context when limits approach.

Learn More

Concept: Rewrite queries, refine prompts, and adapt context as the conversation progresses.
