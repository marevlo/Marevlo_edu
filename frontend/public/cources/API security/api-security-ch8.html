The ACL explosion

Back in chapter 3, Natter controlled access with access control lists — a simple table linking each user to their permissions on each object. ACLs are wonderfully easy to understand. The trouble is arithmetic: they don't scale. As your users and objects multiply, the list of individual permissions multiplies with them, and eventually it collapses under its own weight.

The worst case is genuinely alarming. With a million users and a million objects, you could in principle need a billion ACL entries, one for every possible user-object pairing. Long before you reach that extreme, a subtler problem bites: permissions that are never cleaned up. Users drift between jobs, accumulating access they no longer need, quietly violating the principle of least privilege. This is what this chapter sets out to fix.

Reading the diagram

The point isn't the exact numbers — it's the shape of the growth. Direct user-to-permission mappings grow as users × objects, which is multiplicative.

  1. With a handful of users and objects, an ACL is perfectly manageable — you can read the whole table at a glance.
  2. Double the users and double the objects, and you've roughly quadrupled the entries. The grid fills fast.
  3. The problem compounds when permissions are managed centrally (mandatory access control), because no individual user is responsible for pruning their own stale entries — so privileges accumulate.

The cure, in every model that follows, is the same idea: insert an intermediary layer between users and permissions, so you assign access to a few shared things rather than to every individual pairing.

Definition · Identity-Based Access Control (IBAC)

Identity-based access control determines what you can do based on who you are. The user is first authenticated, then a check decides whether that identity is authorized for the requested action. Every model in this chapter — groups, roles, attributes — is a way of organizing IBAC so it scales. (Chapter 9 covers non-identity-based models.)

2

Users and groups

The first and most familiar intermediary is the group. Instead of always assigning permissions to individual users, you collect related users into a group and assign permissions to the group. Add a new engineer to the "software engineers" group and they instantly inherit the right access — no need to remember every individual permission they need.

The relationship is many-to-many: a group has many members, and a user can belong to many groups. And because a group's members can themselves be groups, you can nest them into hierarchies — "project managers" can be a member of "employees," so every project manager is automatically an employee too.

Reading the diagram

The whole story of this chapter is visible in these three pictures: every model adds a layer that collapses many connections into few.

  1. Direct (ACLs) — every user links straight to every permission they hold. Simple, but the links multiply with users and objects.
  2. Groups — users join groups, and permissions attach to groups. Now adding a user means one link (join the group), not many. But permissions can still be assigned to individuals too, so you must check both.
  3. Roles — the next section's refinement: permissions attach only to roles, never to individuals, giving the cleanest separation of all.

Implementing groups in Natter is straightforward: add a group_members table, look up the user's groups during authentication, and attach them as a request attribute. The permission check then accepts either the username or any of the user's groups.

Principle · keep the layers separated

It's tempting to fold group lookup into a single clever SQL JOIN at permission-check time. Resist it. Collect all the attributes you need for access control — including group membership — during the authentication step, then make the authorization decision purely from those attributes. Why? Because if you later switch to an external user store like LDAP or an OIDC identity provider, group membership arrives as part of authentication (e.g. in the ID token), and code that assumed a local database JOIN would break.

3

LDAP groups

In most large organizations, users aren't stored in your API's own database at all — they live in a central LDAP directory (Lightweight Directory Access Protocol), purpose-built for storing user information and with groups baked right in. If you're integrating with a corporate environment, this is almost certainly where group membership comes from.

The LDAP standard defines two kinds of groups, plus a widely-implemented hybrid, and the distinction is really about how membership is decided:

Static groups

Explicitly list their members (groupOfNames / groupOfUniqueNames). You know exactly who's in by reading the member attribute. Simple and fast, but every member must be listed.

Dynamic groups

Defined by a search query (groupOfURLs): anyone matching the query is a member. Great for huge groups where listing everyone is impractical — but evaluating the query can be expensive.

Virtual static groups

Non-standard but common: look like static groups but resolve membership by querying a dynamic group behind the scenes. The best of both, where supported.

To find a user's static groups, you search the directory for every group whose member attribute contains that user's distinguished name. The critical security detail: always parameterize the search filter rather than concatenating user input into it, exactly as you parameterize SQL to prevent injection. JNDI's numbered placeholders ({0}, {1}) let the library escape input safely, defeating LDAP injection.

Looking up a user's LDAP groups safely
var results = directory.search(
    "ou=groups,dc=example,dc=com",
    "(&(objectClass=groupOfNames)" +              // static groups...
    "(member=uid={0},ou=people,dc=example,dc=com))", // ...containing this user
    new Object[]{ username },                         // {0} is safely escaped — no LDAP injection
    searchControls);

while (results.hasMore()) {
  var result = results.next();
  groups.add((String) result.getAttributes().get("cn").get(0));  // the group's name
}
A faster shortcut · the isMemberOf attribute

Searching for every group a user belongs to can be slow. Many directory servers offer a non-standard virtual attribute on the user entry itself — often called isMemberOf — that lists all their groups (static and dynamic), kept up to date automatically by the server. Reading one attribute off the user is far cheaper than running a directory-wide search. And if you must search regularly, cache the results for a short while.

4

Role-based access control

Groups help, but they don't fully solve the problem. They still usually allow permissions on individual users (so you must check both), they're often defined organization-wide and may not match your API's needs, and a single group can accumulate a sprawling pile of fine-grained permissions that nobody can review. RBAC tightens all of this up by inserting a cleaner intermediary: the role.

The defining move of RBAC is that permissions are assigned only to roles — never directly to users or groups. Users are then assigned to roles. Assigning someone the "moderator" role is far easier than remembering the exact set of permissions a moderator needs, and if those permissions change, you update the role once instead of touching every user.

In principle anything you can do with RBAC you could do with groups — but in practice they're used quite differently, and the differences are what make RBAC powerful:

Groups organize users

They answer "who belongs together?" — usually defined centrally for the whole organization (in LDAP), and they often still permit per-user permission grants.

Roles organize permissions

They answer "what can this kind of actor do?" — usually specific to one API, and they forbid per-user grants entirely, so reviewing access is far simpler.

Separation of duties & sessions

RBAC's split between defining a role's permissions and assigning users to it is a valuable separation of duties — different people do each job, and it's much harder to slip in unauthorized access. Roles can also be dynamic (a "duty officer" role active only during a shift) and the NIST RBAC model adds sessions, where a user activates only a subset of their roles at a time — just like scoped tokens in OAuth, this honors least privilege by limiting the damage if a session is compromised.

5

Mapping roles to permissions

There are two ways to connect roles to what they can actually do. You can annotate each API operation with the roles allowed to call it, or you can keep explicit lower-level permissions and define a mapping from roles to those permissions. Each suits a different situation.

Annotate operations with roles

Java EE / JAX-RS does this with @RolesAllowed("owner") on each method. Clean and declarative, but the role-to-operation mapping lives in source code.

Map roles to permissions in data

Keep a role_permissions table. Lets admins define new roles at runtime and review exactly what each role grants without reading code. This is the path Natter takes.

For Natter, we define four roles by mapping each to the existing rwd (read/write/delete) permissions — a clean, reviewable table:

schema.sql — four Natter roles
INSERT INTO role_permissions(role_id, perms) VALUES
    ('owner',     'rwd'),   -- full control: read, write, delete
    ('moderator', 'rd'),    -- read posts and delete offensive ones
    ('member',    'rw'),    -- read and write, but never delete
    ('observer',  'r');     -- read-only

Users are then mapped to roles in a user_roles table, scoped per social space — so the owner of one space has no power over another. (When users, roles, or groups are confined to one slice of an application like this, that slice is called a security domain or realm.) At request time, a single before filter joins the two tables to resolve the user's permissions once and stashes them in a request attribute, so the existing requirePermission checks keep working unchanged:

Resolving role → permissions in one query
var perms = database.findOptional(String.class,
    "SELECT rp.perms" +
    "  FROM role_permissions rp JOIN user_roles ur" +   // role → permission
    "    ON rp.role_id = ur.role_id" +                    // user → role
    " WHERE ur.space_id = ? AND ur.user_id = ?",
    spaceId, username).orElse("");
request.attribute("perms", perms);   // cache for the rest of the request

Because roles are usually API-specific (unlike groups), it's fine to resolve them in one query here — you wouldn't typically be told a user's roles during authentication.

6

Static and dynamic roles

Most role assignments are static: you decide a user has a role, and they keep it until you take it away. That's what the user_roles table above implements, and it's what most Java EE servers do via configuration files. But some situations call for roles that depend on context.

A dynamic role is granted only when certain conditions hold. The classic example is a call-center worker who gets the "customer support" role — with access to customer records — only during their contracted shift hours. Outside those hours the role simply isn't granted, and access is denied. There's no single standard for dynamic roles; implementations lean on database queries or rule languages like Prolog or OWL. Some RBAC systems also support constraints, such as making two roles mutually exclusive so one person can't hold both at once — a clean way to enforce separation of duties (you can't be both the system administrator and the auditor of that same system).

Where RBAC runs out of road

Notice the strain in that call-center example. Restricting by shift hours already stretches static roles — but what we'd really like is to let the agent see only the record of the customer they're currently on a call with, and only for a few minutes afterward. That depends on live, per-request context that no fixed role assignment can capture. When access rules need to react to the specifics of this request, RBAC gives way to something more flexible: ABAC.

7

Attribute-based access control

When access decisions need to depend on the live details of a request — who, what, how, and under what circumstances — we reach for ABAC. Instead of a fixed role lookup, ABAC makes a fresh decision for every request by feeding a bundle of attributes into a policy engine that returns allow or deny.

Reading the diagram

The power of ABAC is that any of these attributes can drive the decision, alone or in combination — which is exactly what lets it express rules RBAC can't.

  1. Subject — everything about the user: username, groups, how and when they authenticated.
  2. Resource — what's being accessed: the URI, a security label like TOP SECRET, which space it belongs to.
  3. Action — what they're trying to do: the HTTP method, the operation name.
  4. Environment — the context the request arrives in: time of day, the caller's IP or location. Pulling these out explicitly also makes rules easy to test, since you can feed in a fake time.

Now the call-center rule becomes expressible: permit if subject=agent and resource=the-customer-on-this-call and environment.time within shift. That's four categories working together.

8

Combining decisions safely

Real ABAC policies are built from many independent rules, and sooner or later two of them will disagree about the same request. That raises two unavoidable design questions — and the safe answers to both lean the same way: toward caution.

What if no rule matches?

Default to deny. Require at least one rule to actively permit the request; silence means refusal. (The exception, below, is when layering ABAC on top of an existing system.)

What if rules conflict?

Let deny win. A single deny decision overrides any number of permits. It only takes one rule spotting a problem to stop a request.

Natter takes a pragmatic variant. Because we're layering ABAC on top of the RBAC system we already built — to add extra restrictions, not replace existing checks — we use default-permit with deny-overrides: if no ABAC rule objects, the request passes through to the RBAC checks underneath. The Decision class encodes exactly this: it starts permitted, deny() flips it off permanently, and permit() is a deliberate no-op (since permit is already the default and must never override a deny).

Decision — default permit, deny always wins
public static class Decision {
  private boolean permit = true;   // start permitted (we layer over RBAC)
  public void deny()   { permit = false; }   // any deny is final
  public void permit() { }                       // no-op: never override a deny
  boolean isPermitted() { return permit; }
}

For the actual rule evaluation, Natter uses the Drools rules engine rather than hand-coded Java — a domain-specific rule language reads far more clearly for access policy. Here's the call-center-style rule that blocks moderation outside office hours, combining an action attribute with an environment attribute:

accessrules.drl — a Drools ABAC rule
rule "deny moderation outside office hours"
  when
    Action( this["method"] == "DELETE" )            // the action: a delete
    Environment( this["timeOfDay"].hour < 9          // the context: before 9am...
        || this["timeOfDay"].hour > 17 )            // ...or after 5pm
  then
    decision.deny();                              // → forbidden
end

Wired in as a Spark before("/*") filter, this runs ahead of the RBAC checks — defense in depth, with each layer able to deny independently.

9

Policy engines and XACML

As policies grow, you don't want every API hand-rolling its own rule evaluation. Two architectural ideas address this. A policy agent plugs into an existing server or reverse proxy to enforce rules; a standalone API gateway intercepts requests at the HTTP layer. And to keep rules consistent across many APIs, you can centralize the policy definitions in a dedicated server that agents consult — which is exactly what XACML formalized.

Definition · XACML

XACML (eXtensible Access-Control Markup Language) is an OASIS standard defining a rich XML policy language and — more influentially — a reference architecture for distributed ABAC. The XML language has fallen out of fashion, but the architecture lives on, now folded into NIST's ABAC recommendations.

That architecture splits the job across four cooperating components, each with one clear responsibility. Tap each to see how a request flows through them.

Reading the diagram

Trace a single request through the four and the division of labor becomes clear — this is the same separation Natter built by hand, just named and standardized.

  1. PEP (Policy Enforcement Point) — sits in front of the API like a policy agent, intercepting each request and enforcing whatever the decision comes back as. Natter's Spark before filter is a PEP.
  2. PDP (Policy Decision Point) — the brain. It holds the policy engine and returns the allow/deny decision. Natter's Drools evaluation is a PDP.
  3. PIP (Policy Information Point) — retrieves and caches the attribute values a decision needs, from local databases or remote services like an OIDC UserInfo endpoint.
  4. PAP (Policy Administration Point) — the admin interface where policies are written and managed.

The payoff of separating them: the PDP and PAP can be centralized for a whole organization, so many APIs (each with its own PEP) share one consistent, reviewable set of policies — while open-source engines like the Open Policy Agent bring the same model to modern infrastructure.

10

ABAC best practices

ABAC's flexibility is its strength and its trap. The same expressiveness that lets you write precise rules also lets you build a tangle of thousands of them where nobody can say for certain who has access to what. The author of this book recounts a real system where a tiny change to a document's structure silently broke many XPath-based rules — letting invalid requests through for weeks before anyone noticed. So treat ABAC's power with respect.

The flexibility trap

Small changes to flexible rules can have outsized, hard-to-predict effects, and rich rule languages can make it impossible to automatically tell when a rule has silently stopped matching anything. A rule that no longer fires fails open — quietly allowing what it was meant to block. That's exactly why the practices below center on layering, testing, and review.

A handful of disciplines keep ABAC's power from becoming a liability:

Layer over something simpler

Run ABAC on top of RBAC (as Natter does), so a mistake in an ABAC rule doesn't mean total loss of security — the RBAC layer underneath still holds. Defense in depth.

Test endpoints automatically

Have automated tests that alert you the moment a policy change grants access to the wrong party — the only reliable defense against silently-broken rules.

Version-control policies

Keep policies in source control so changes are reviewed and can be rolled back instantly. Treat a policy edit with the same care as a code change.

Centralize thoughtfully

Centralizing everything adds bureaucracy that can itself violate least privilege — broad rules linger because changing them is painful. Centralize what benefits from consistency; leave the rest local. And measure performance early.

Key takeaways

  • Direct user-to-permission ACLs don't scale — entries grow as users × objects, and stale permissions accumulate. Every model in this chapter inserts an intermediary layer to fix this.
  • Groups collect related users so permissions attach to the group, not each user; LDAP has built-in static, dynamic, and virtual-static groups. Collect group membership during authentication, not at decision time.
  • RBAC goes further: permissions attach only to roles, never to individuals. Roles organize permissions and are API-specific; groups organize users and are usually org-wide.
  • RBAC separates defining role permissions from assigning users to roles — a useful separation of duties — and NIST sessions let users activate a subset of roles for least privilege.
  • Roles can be static or dynamic (granted only in context, like during a shift), but when decisions must react to live per-request detail, RBAC gives way to ABAC.
  • ABAC decides every request dynamically from four attribute categories — subject, resource, action, environment — fed into a policy engine.
  • Combine rule decisions safely: default deny and let deny override (or default-permit-with-deny-overrides when layering ABAC over an existing system).
  • XACML's reference architecture splits enforcement (PEP), decision (PDP), information (PIP), and administration (PAP), enabling centralized, consistent policy across many APIs.
  • Tame ABAC's flexibility by layering it over RBAC, testing endpoints automatically, version-controlling policies, and centralizing only what truly benefits.

Test yourself

Pop quiz · tap to reveal
True or false: in general, can groups contain other groups as members?
True. Many group models allow groups to be members of other groups, creating hierarchies — e.g. "project managers" can be a member of "employees."
Which three are common types of LDAP groups? (static · abelian · dynamic · virtual static · dynamic static · virtual dynamic)
Static, dynamic, and virtual static groups. Static and dynamic are standard; virtual static is non-standard but widely implemented. (Abelian groups belong to mathematics, not LDAP!)
In the filter (&(objectClass=#A)(member=uid=alice,...)), which object class searches for static groups? (group · groupOfURLs · groupOfNames · groupOfSubjects)
groupOfNames (or groupOfUniqueNames). groupOfURLs defines dynamic groups.
Which apply more to roles than groups? (all permissions assigned via roles · better separation of duties · application-specific · permissions assignable to individuals)
All permissions assigned via roles, better separation of duties, and application-specific. RBAC never assigns permissions directly to individuals; defining role permissions and assigning users to roles are typically done by different people; and roles are usually per-API while groups are org-wide.
What is a session used for in the NIST RBAC model?
To let a user activate only a subset of their roles at a given time — supporting least privilege, much like scoped tokens in OAuth.
What are the four attribute categories in ABAC, and which XACML component defines and manages policies?
The four categories are subject, resource, action, and environment. Policies are defined and managed at the Policy Administration Point (PAP).
