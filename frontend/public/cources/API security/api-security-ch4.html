1

Authentication in a real browser

So far, Natter has used HTTP Basic authentication — the client sends the username and password on every single request. That's fine for a command-line script, but the moment you put a real web UI in front of it, the cracks show. This chapter is the story of replacing it with something browsers actually enjoy using.

The trouble starts as soon as you build a UI. Browsers can't POST JSON from a plain HTML form, so Natter's UI uses JavaScript's Fetch API to call the API. And because Basic auth credentials aren't sent automatically by Fetch, you have to opt in with credentials: 'include'. The first time the API replies with 401 Unauthorized and a WWW-Authenticate header, the browser does something jarring: it pops up its own ugly, uncustomisable login box.

natter.js — calling the API from the browser
function createSpace(name, owner) {
  let data = {name: name, owner: owner};
  fetch(apiUrl + '/spaces', {
    method: 'POST',
    credentials: 'include',            // send cookies / Basic creds with the request
    body: JSON.stringify(data),
    headers: { 'Content-Type': 'application/json' }  // required, or the API rejects it
  })
  .then(response => response.ok ? response.json()
                                : { throw Error(response.statusText); })
  .then(json => console.log('Created space:', json.name, json.uri))
  .catch(error => console.error('Error:', error));
}

Fetch is asynchronous — it returns a Promise. The .then() block runs on success, .catch() on a network error. response.ok is true for any 2xx status.

But before any of this works, you hit a more fundamental wall: open that HTML file straight from disk and clicking "Create" does nothing. The browser silently blocks the request. The reason is the single most important rule in browser security — the same-origin policy — and you can't understand cookies, CSRF, or any of this chapter without it.

2

The same-origin policy

The same-origin policy (SOP) is the browser's core defence: by default, JavaScript loaded from one origin can freely talk to that same origin, but is heavily restricted when talking to a different one. Get comfortable with what counts as "the same origin" — almost everything later depends on it.

Definition · Origin

An origin is the combination of three things: the protocol (http vs https), the host (e.g. api.example.com), and the port (80 for HTTP, 443 for HTTPS by default). Two URLs share an origin only if all three match exactly. The path (/test/1) is not part of the origin.

That "all three must match" rule trips everyone up, so let's make it concrete. Below, the reference origin is https://api.example.com:8443. Tap each candidate URL to see whether the browser treats it as the same origin — and why.

Reading the checker

Walking through each one cements the rule — notice how a single mismatched component is enough to make it a different origin:

  1. http://…/test/1 — different: the protocol is http, not https (and the port implicitly changes to 80).
  2. https://…/test/2 with no port — different: no port means the default 443, but our reference uses 8443.
  3. http://…:8443/test/2 — different: same port number, but the protocol is http not https.
  4. https://…:8443/test/2 — SAME: protocol, host, and port all match. The different path doesn't matter.
  5. https://www.example.com:8443/… — different: the host is www.example.com, not api.example.com.
What the SOP actually blocks

The SOP is more subtle than "blocks everything cross-origin." It generally lets a request be sent to another origin, but stops your script from reading the response. Only simple requests are allowed cross-origin at all: GET, POST, or HEAD, with a tiny set of headers, and a Content-Type limited to application/x-www-form-urlencoded, multipart/form-data, or text/plain. Anything fancier gets blocked outright. Hold onto that fact — it's exactly what makes CSRF possible later, and also what some defences rely on.

The quick fix for development is to have the server serve the UI from the same origin as the API, with one line: Spark.staticFiles.location("/public"). Now the HTML, the JavaScript, and the API all live at https://localhost:4567, so the SOP is satisfied. (Chapter 5 covers the proper cross-origin solution, CORS.)

3

Why Basic auth hurts

With the UI working, the pain of HTTP Basic authentication becomes obvious — and it hurts in two directions at once: it's bad for users, and it's bad for engineering. Understanding exactly why motivates everything that follows.

🔑
The password rides along on every request.

Each API call carries the raw password, multiplying the chances a bug leaks it. In a microservice architecture, every service then has to handle that password carefully.

🐌
Verifying a password is deliberately slow.

Remember from chapter 3 that a secure hash like Scrypt takes ~100ms on purpose. Doing that on every request caps you at roughly 10 operations per CPU core per second — brutal for scaling.

😕
The login box is ugly and unbrandable.

The browser's built-in Basic auth dialog can't be styled. Your carefully designed product suddenly shows a generic system prompt.

🔒
There's no real way to log out.

Browsers cache Basic credentials and resend them automatically. On a shared computer, the next person can hit Back and they're still logged in as you — a serious problem.

None of these are fatal for a script or a service-to-service call where there's no human and the password is strong. But for a browser-facing API, they're disqualifying. What we actually want is for the user to log in once and then be trusted for a while — which is exactly what token-based authentication gives us.

4

Token-based authentication

The idea is simple and decades old. Instead of proving who you are on every request, you prove it once to a dedicated login endpoint, which hands you a short, random token. You then present that token on subsequent requests. The server looks it up to see who you are. When it expires or you log out, it's deleted, and you log in again.

Definition · Token-based authentication

The user's real credentials are presented once; the client then receives a short-lived token — typically a short, random string — that authenticates subsequent API calls until it expires.

Reading the diagram

Compare this to Basic auth and the win is clear: the expensive, sensitive password step happens once instead of on every call.

  1. Login. The client sends the username and password — just this once — to a dedicated login endpoint (in Natter, POST /sessions).
  2. Get token. The endpoint verifies the credentials, creates a time-limited token, saves it in the token store, and returns the token ID to the client.
  3. Reuse token. On every subsequent request the client sends the cheap, disposable token instead of the password. The API looks it up in the store to find out who's calling — no expensive password hashing required.

A short-lived token used while a user is actively on a site is usually called a session token or just a session.

Natter implements this with a clean TokenStore interface (just create and read for now, with revoke added at logout). Keeping it an interface matters: across this chapter and the next two we'll swap in three different backends — session cookies here, database tokens and encrypted client-side tokens later — without touching the controller.

TokenController.java — the login endpoint
public JSONObject login(Request request, Response response) {
  // UserController already verified the password via HTTP Basic, and set "subject"
  String subject = request.attribute("subject");
  var expiry  = now().plus(10, ChronoUnit.MINUTES);   // controls re-login frequency

  var token   = new TokenStore.Token(expiry, subject);
  var tokenId = tokenStore.create(request, token);         // store backend decides the ID

  response.status(201);
  return new JSONObject().put("token", tokenId);
}

Elegantly, login reuses the existing Basic-auth filter to verify the password — so this method just reads the already-authenticated subject and mints a token. Logging in is modelled RESTfully as creating a session resource (POST /sessions).

5

Session cookies

Now we need somewhere to keep that token in the browser. The oldest and most battle-tested answer is the session cookie. After login, the server sends a Set-Cookie header; the browser stores the token and automatically attaches it as a Cookie header on every later request to the same site. The server looks it up to know who you are.

Reading the diagram

The key thing to notice is step 3 — the browser sends the cookie by itself, with no work from your JavaScript.

  1. Login. The client posts credentials to the login endpoint.
  2. Set-Cookie. The server creates a session, stores the token's attributes server-side, and returns a Set-Cookie: JSESSIONID=… header. The browser tucks the cookie away.
  3. Auto-sent. On every later request to the same site, the browser automatically attaches the cookie. The server looks up the session to recover who the user is and what their token allows.

Cookies bring real security benefits for free: they're scoped to a domain (so they won't leak to other sites), and can be flagged Secure (HTTPS-only) and HttpOnly (hidden from JavaScript). That's why they remain a great choice for a first-party UI served from the same origin as its API.

Because Spark has built-in session support, the cookie-backed TokenStore is almost trivial — request.session(true) creates a session and the Set-Cookie header; request.session(false) reads an existing one. But the obvious implementation hides a subtle, dangerous bug.

6

The session fixation attack

Here's the flaw, and it affects every form of token-based auth, not just cookies. If you call request.session(true) and a session cookie already exists, Spark reuses it instead of making a new one. That sounds harmless — until you realise an attacker can plant a session cookie in a victim's browser before they log in.

Definition · Session fixation

A session fixation attack happens when an API fails to generate a new session token after a user authenticates. The attacker obtains a valid token on their own device, injects it into the victim's browser, and once the victim logs in, that same token is now tied to the victim's account — which the attacker still holds.

Interactive · Figure 4.8
How session fixation works — and the fix

Step through the attack, then reveal the one-line fix that defeats it. The whole exploit hinges on a session token that doesn't change at login.

The attacker never steals the victim's password — they don't need to. They just make sure the victim logs in using a session the attacker already knows. Step through to see it, then the fix.

Reading the diagram

Notice that at no point does the attacker learn the victim's password. The vulnerability is purely that the token stays the same across the login boundary.

  1. Attacker logs in first on their own device to obtain a valid session token from the API.
  2. They plant that token in the victim's browser — for example via an XSS flaw on a sub-domain, or by luring the victim to a link with the session ID embedded in the URL.
  3. The victim logs in. Because the server reuses the existing session instead of creating a fresh one, it simply flips the session's username from "attacker" to "victim" — and the attacker's token now opens the victim's account.
  4. The fix: always invalidate any existing session and create a brand-new one immediately after authentication. The new token is unguessable to the attacker, and their planted session is logged out.
CookieTokenStore.java — defeating session fixation
public String create(Request request, Token token) {
  var session = request.session(false);   // is there an EXISTING session?
  if (session != null) {
    session.invalidate();                  // kill it — this is the crucial line
  }
  session = request.session(true);       // now mint a FRESH, unguessable session

  session.attribute("username", token.username);
  session.attribute("expiry",   token.expiry);
  session.attribute("attrs",    token.attributes);
  return session.id();
}

A related hardening step: make sure your servlet container's session tracking-mode is COOKIE, never URL. URL-based sessions (;JSESSIONID=… in the path) make fixation trivial because an attacker can just craft a link. Jetty — which Spark uses — defaults to cookies.

7

Cookie security attributes

A cookie isn't just a value — it carries a set of attributes that control how the browser treats it. Setting these correctly is most of cookie security. The golden rule: always set the most restrictive attributes you can get away with.

Secure

The cookie is only ever sent over HTTPS, so it can't be intercepted on an unencrypted connection. Essential for any auth cookie.

HttpOnly

The cookie can't be read by JavaScript, making it much harder to steal via an XSS attack. Also essential for session cookies.

SameSite

The cookie is only sent on requests originating from the same site — the core CSRF defence we'll dig into next.

Domain

Without it, the cookie is "host-only" (sent only to the exact host that set it). Set it and the cookie spreads to all sub-domains — so don't, unless you truly need to.

Path

Limits which URL paths receive the cookie. Weak as a security control (easy to bypass via a hidden iframe), so don't lean on it.

Expires / Max-Age

Sets when the cookie dies. Omit these on auth cookies so the session ends when the browser closes — important on shared machines.

Spark already sets Secure and HttpOnly by default. Two extra hardening ideas are worth knowing. First, beware the Domain attribute: sharing a cookie across sub-domains means a single compromised sub-domain can steal your sessions — and sub-domain hijacking (claiming an abandoned host that still has live DNS) is depressingly common. Second, modern browsers honour magic cookie-name prefixes:

__Secure-

Forces the cookie to have been set with the Secure attribute by a secure origin. The browser rejects it otherwise.

__Host-

Everything __Secure- does, plus requires it be host-only (no Domain). This stops a sub-domain overwriting the cookie — a strong defence against sub-domain hijacking.

Session vs persistent cookies

A cookie with Expires/Max-Age is persistent — it survives a browser restart. One without is a session cookie, deleted when the tab closes. Keep auth cookies as session cookies so closing the browser logs the user out. But never rely on this alone — some browsers restore tabs and cookies on restart, so always enforce a maximum session lifetime and an idle timeout on the server.

8

Cross-Site Request Forgery (CSRF)

Here's the twist: the very thing that makes cookies convenient — the browser sends them automatically — is also their greatest weakness. The browser attaches your session cookie to requests your API receives even when those requests were triggered by a completely different, malicious site. That's CSRF.

Picture it: you're logged into Natter, then you click a tempting link from "Polly in Marketing." The page that loads runs JavaScript that fires requests at the Natter API from your browser. Because you're still logged in, the browser cheerfully attaches your session cookie. To Natter, those forged requests look exactly like you — posting abuse to your friends, in your name.

Definition · Cross-Site Request Forgery

CSRF (pronounced "sea-surf") occurs when an attacker makes a cross-origin request to your API and the browser automatically attaches the user's cookies. Unless you add specific defences, the request is processed as if the user genuinely made it.

Reading the diagram

The mental trap is thinking "the same-origin policy protects me." It partly does — but only on the response, not the request.

  1. Legitimate login. The user logs into Natter and gets a session cookie, stored happily in the browser.
  2. Victim visits a malicious site — tricked by an enticing link — whose page contains hostile JavaScript.
  3. The malicious script fires a request at the Natter API. The browser, seeing a request to Natter's domain, automatically attaches the session cookie. The API sees a perfectly authenticated request and acts on it.
  4. The SOP blocks only the response. The attacker's script can't read what came back — but it didn't need to. The damage (posting, deleting, transferring) was done by sending the request.
A free first line of defence

Requiring an application/json Content-Type and a custom header (like X-CSRF-Token) already helps, because those trip the SOP's "simple request" limits and force the browser to block them cross-origin. But clever attackers have bypassed such tricks (e.g. via old Flash bugs), so you need a dedicated defence. And one ironclad rule: never change state in response to a GET request — browsers allow cross-origin GETs freely, and every CSRF defence assumes GETs are safe.

9

First defence: SameSite cookies

The most direct fix, when your UI and API share a domain, is the SameSite cookie attribute. Mark a cookie SameSite and the browser only sends it on requests that originate from the same site. So when Polly's malicious page fires its request, the browser sends it without the session cookie — and the API rejects it as unauthenticated.

A SameSite cookie is only attached to requests that originate from the same site that set it. Toggle above to watch the browser include the cookie for the legitimate UI but strip it from the attacker's cross-site request.

Reading the diagram — and the two modes

SameSite comes in two strengths, and the difference is about what happens when a user clicks a link from another site:

  1. SameSite=strict — the cookie is withheld on every cross-site request, including when the user simply clicks a link to your site from elsewhere. Strongest, but can feel jarring (the user lands logged-out).
  2. SameSite=lax — the cookie is sent when the user directly clicks a link to your site, but withheld on most other cross-site requests (like background scripts and POSTs). A practical middle ground; many single-page apps work fine in strict mode anyway.
Definition · "Same site" ≠ "same origin"

SameSite uses the registerable domain, which is looser than an origin. api.payments.example.com and www.example.com are the same site (both reduce to example.com), even though they're different origins. Protocol and port are ignored. Working out the registerable domain relies on Mozilla's public suffix list, which catches tricky cases like .co.uk and github.io.

Why SameSite isn't the whole answer

SameSite is excellent, but because "same site" includes all sub-domains, its protection is only as strong as your weakest sub-domain — one hijacked sub-domain and it collapses. Treat SameSite as a defense-in-depth layer, and add a more robust defence on top. That's the double-submit cookie.

10

The robust defence: hash-based double-submit cookies

The strongest CSRF defence forces the caller to prove they know something an attacker can't read — something tied to the session. Traditional web apps put a hidden random token in every form. An API can't do that, so we use a clever cookie trick instead. But the obvious version has a flaw, and watching it evolve is the best way to understand the final design.

Step one: the plain double-submit cookie

Store a second random token as a cookie that is not HttpOnly, so JavaScript on your own site can read it. On every request, the client reads that cookie and echoes it back in an X-CSRF-Token header. The server checks the cookie and header match. A cross-origin attacker can't read the cookie (SOP), so they can't produce the matching header.

The flaw · the cookie can be overwritten

An attacker can't read the CSRF cookie — but they might be able to overwrite it with a value they choose (via an XSS flaw or a compromised sub-domain). Then they simply send that same known value in the header, and the two match. The defence collapses. The __Host- prefix helps, but we can do something fundamentally stronger.

Step two: bind the token to the session with a hash

Instead of a random second token, compute the CSRF token as a cryptographic hash (SHA-256) of the session cookie. Now the two are cryptographically bound: changing either one breaks the match. The attacker can't read the session cookie, so they can't compute the correct hash — and they can't usefully overwrite the CSRF cookie either, because any value they pick won't be the hash of the real session.

Reading the diagram

The whole improvement is about removing the attacker's ability to choose a value that the server will accept.

  1. Plain double-submit: the CSRF token is an independent random value. If the attacker can overwrite the CSRF cookie with xyz and also send xyz in the header, the server sees a match and is fooled.
  2. Hash-based: the server expects the CSRF token to equal SHA-256(session cookie). The attacker doesn't know the session cookie (it's HttpOnly), so they can't compute the right hash. Any value they overwrite with simply won't match — the request is rejected.
  3. Because the input (a high-entropy session ID) is unguessable, the hash needn't be slow like a password hash — SHA-256 is perfect: fast, preimage-resistant, and collision-resistant.
CookieTokenStore.java — the hash-based check
// On create: the token ID we hand the client is the HASH of the real session
return Base64url.encode(sha256(session.id()));

// On read: recompute and compare in CONSTANT TIME (see warning below)
var provided = Base64url.decode(tokenId);          // from X-CSRF-Token header
var computed = sha256(session.id());               // hash of the real session cookie
if (!MessageDigest.isEqual(computed, provided)) {  // constant-time equality
  return Optional.empty();                         // mismatch → treat as unauthenticated
}

Bonus: the raw session ID is now never exposed to JavaScript — the client only ever sees the hash. The real session stays locked inside the HttpOnly cookie.

Critical · use constant-time comparison

Never compare the tokens with a normal String.equals(). Standard comparisons bail out at the first mismatched character, so they take fractionally longer the more leading characters match. A timing attack measures those tiny differences (yes, even over the internet) to recover the token one character at a time — turning ~6410 guesses into ~640. Always use a constant-time comparison like Java's MessageDigest.isEqual, which checks every byte regardless.

The client side is small: after login, the JavaScript reads the returned token and stores it in a (non-HttpOnly) csrfToken cookie, then sends it as X-CSRF-Token on every request. Leave the token out, and the API treats the request as unauthenticated — CSRF defused.

11

Logout and token revocation

Remember one of the original complaints about Basic auth: you couldn't really log out. Token-based authentication fixes that — but only if you do it properly. Just deleting the cookie from the browser isn't enough, because that deletion can fail, or the cookie might linger in a misconfigured cache. The token must also be revoked on the server.

We add a third method to the TokenStore interface — revoke — alongside create and read. For session cookies, revocation is wonderfully simple: call Spark's session.invalidate(), which both removes the server-side session and sends a Set-Cookie header with an expiry in the past, so the browser deletes its copy too.

CookieTokenStore.java — revoking a session
public void revoke(Request request, String tokenId) {
  var session = request.session(false);
  if (session == null) return;

  // Even logout needs CSRF protection — or an attacker could log you out to annoy you
  var provided = Base64url.decode(tokenId);
  var computed = sha256(session.id());
  if (!MessageDigest.isEqual(computed, provided)) return;

  session.invalidate();   // kills server session AND expires the browser cookie
}

Notice logout still verifies the anti-CSRF token. It's lower-stakes than other operations, but an attacker forcibly logging users out is still a nuisance worth preventing.

The endpoint itself follows the RESTful pattern: logging in created a session resource with POST /sessions, so logging out deletes it with DELETE /sessions. The token ID comes from the X-CSRF-Token header rather than the URL — deliberately, because putting tokens in URLs risks leaking them into browser history and server logs.

Never put tokens in the URL

A request to DELETE /sessions/xyz would leak the token xyz into browser history and server access logs. Keeping it in a header avoids that. This is a small habit with outsized payoff — URLs are logged and shared in far more places than people expect.

Key takeaways

  • The same-origin policy is the foundation: origin = protocol + host + port, all matching exactly. It lets cross-origin requests be sent but blocks reading the response.
  • HTTP Basic auth is poor for browsers — it resends the password every call, is slow to verify, looks ugly, and can't truly log out.
  • Token-based authentication trades the password once for a short-lived token, which is cheap to validate on each request.
  • Session cookies are a great token store for same-origin first-party UIs — the browser sends them automatically and they support Secure/HttpOnly/SameSite.
  • Always invalidate the old session and create a new one at login to defeat session fixation.
  • Automatic cookie-sending enables CSRF. Defend with SameSite cookies (defense-in-depth) plus hash-based double-submit cookies (robust), and never change state on a GET.
  • Binding the CSRF token to the session via SHA-256 stops cookie-overwrite attacks; compare tokens in constant time to block timing attacks.
  • Logout must revoke server-side, not just delete the cookie — and keep tokens out of URLs.

What's the best way to avoid session fixation attacks?
Invalidate any existing session and create a fresh one after the user authenticates. A new, unguessable token means the attacker's planted session is useless.
Which cookie attribute prevents the cookie being read from JavaScript?
HttpOnly. It hides the cookie from scripts, making it harder to steal via XSS.
A cookie from https://api.example.com:8443 is SameSite=strict. Which pages can send it? (http://www… · https://other.com… · https://www.example.com… · https://www.example.org… · https://api.example.com…)
The example.com pages — http://www.example.com, https://www.example.com, and https://api.example.com. SameSite looks only at the registerable domain (example.com); protocol, port, and path don't matter. other.com and example.org are different sites.
What problem with plain double-submit cookies does the hash-based approach solve?
An attacker overwriting the second cookie with a value they know. Binding the token to the session via a hash means any overwritten value won't match SHA-256(session), so it's rejected.
Why must HTTP Basic credentials only be sent over HTTPS?
Because they're only Base64-encoded, which anyone watching the network can trivially decode back to the plaintext password.
