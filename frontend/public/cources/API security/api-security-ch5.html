1

Why cookies stop scaling

In chapter 4 we made session cookies safe with SameSite and CSRF defences, and Natter felt great. But success brings new demands. Marketing has bought a flashy new domain, nat.tr, and wants logins to work across both it and the original site. Meanwhile the product is expanding to mobile and desktop apps. Both ambitions run straight into the wall that cookies build.

The trouble is that everything that made cookies convenient — the browser handling them automatically and scoping them tightly to one site — is exactly what makes them awkward the moment you cross a domain boundary or leave the browser entirely.

🌐
A second domain can't talk to the API.

The new nat.tr site is a different origin, so the same-origin policy blocks its JSON login requests, throws away any Set-Cookie header on the response, and hides the anti-CSRF token. Three separate roadblocks.

📱
Native apps don't love cookies.

Mobile and desktop apps have to manage cookies manually — the automatic browser machinery isn't there. Cookies feel unnatural outside the browser.

🚫
The web is turning against cross-site cookies.

SameSite cookies are fundamentally incompatible with cross-site CORS, and browsers from Apple, Google, and Mozilla are increasingly blocking cross-site cookies for privacy. The writing is on the wall.

The plan for this chapter is to outgrow cookies in three deliberate moves: first relax the same-origin policy with CORS so other domains can call the API at all; then replace cookies entirely with the Bearer scheme and Web Storage; and finally harden the database that now has to store those tokens ourselves.

2

Cross-origin requests and CORS

Recall from chapter 4 that the same-origin policy lets a request be sent cross-origin but usually blocks the script from reading the response — and outright blocks "non-simple" requests (like JSON POSTs). CORS is the standard, introduced in 2013, that lets a server selectively opt back in to some of those blocked requests.

Definition · CORS

Cross-Origin Resource Sharing (CORS) is a standard that allows certain cross-origin requests to be permitted by web browsers. It defines a set of HTTP headers an API can return to tell the browser which cross-origin requests it should allow — and how much of the response the calling script may read.

The mental shift is this: with CORS, the server gets a say. Before CORS, the browser simply blocked anything that violated the SOP. Now, when a request would have been blocked, the browser first asks the target server whether it's allowed — via a preflight request — and only proceeds if the server says yes.

3

The preflight handshake

The heart of CORS is the preflight request. Before sending a "non-simple" cross-origin request, the browser quietly sends an OPTIONS request to the target server, describing what it's about to do and asking permission. Only if the server approves does the real request go out.

Definition · Preflight request

A preflight request is an automatic HTTP OPTIONS request the browser sends before a cross-origin request that would otherwise violate the SOP. It carries the Origin, the intended method (Access-Control-Request-Method), and any non-standard headers (Access-Control-Request-Headers). The server replies allowing or denying the real request.

Reading the diagram

The key thing to internalise is that the preflight is a separate request the browser makes on your script's behalf, before the one you actually wrote.

  1. The OPTIONS preflight. The browser sends OPTIONS to the target, including the Origin (who's asking), the method it intends to use, and any custom headers it wants to send.
  2. The server replies. It returns CORS headers describing what it permits — which origins, methods, and headers are acceptable.
  3. Allowed: if the server's response covers what the request needs, the browser sends the real request and lets the script read the response.
  4. Blocked: if the server sends no CORS headers, or they don't match, the browser refuses — and your script sees only a cryptic "blocked by CORS policy" error, never the response.

Because preflights never carry credentials, you must handle them before authentication in your filter chain — otherwise every preflight would fail authentication and the real request would never happen.

4

The CORS response headers

CORS is really just a small vocabulary of response headers. The server uses them to answer the browser's "may I?" with precise terms. Two of them can appear on both the preflight and the real response; the rest only make sense on the preflight. Here's the whole set, with where each one belongs.

Header
Scope & purpose
Access-Control-Allow-Origin
BOTH One specific allowed origin, or * for any. Can't be * if credentials are allowed.
Access-Control-Allow-Headers
PREFLIGHT Which non-simple request headers are permitted (or *).
Access-Control-Allow-Methods
PREFLIGHT Which HTTP methods are permitted (or *).
Access-Control-Allow-Credentials
BOTH Whether the browser may send credentials (cookies, saved passwords, client certs). If true, no header may use *.
Access-Control-Max-Age
PREFLIGHT How many seconds to cache this preflight result. Browsers cap it (Chrome ~10 min).
Access-Control-Expose-Headers
ACTUAL Which non-basic response headers the script is allowed to read.
The single-origin gotcha

Access-Control-Allow-Origin accepts only one value, not a list. So to support several origins, your server keeps an allow-list, compares the request's Origin against it, and echoes back the exact match. That's safe here — unlike the header-reflection risks from chapter 2 — precisely because you only reflect after an exact equality check against trusted origins. Always pair it with Vary: Origin so caches don't serve one origin's response to another.

For Natter, the CORS filter runs as the very first thing after rate-limiting (and before authentication, since preflights carry no credentials). It checks the Origin against an allow-list, sets the basic headers when it matches, and short-circuits preflights with a 204 No Content (or 403 if the origin isn't allowed).

CorsFilter.java — the heart of the filter
var origin = request.headers("Origin");
if (origin != null && allowedOrigins.contains(origin)) {   // exact match against allow-list
  response.header("Access-Control-Allow-Origin", origin);    // echo it back
  response.header("Access-Control-Allow-Credentials", "true");
  response.header("Vary", "Origin");                          // caches must key on Origin
}
if (isPreflightRequest(request)) {
  if (origin == null || !allowedOrigins.contains(origin)) halt(403);
  response.header("Access-Control-Allow-Headers", "Content-Type, Authorization, X-CSRF-Token");
  response.header("Access-Control-Allow-Methods", "GET, POST, DELETE");
  halt(204);   // preflight handled — no body needed
}
CORS and SameSite cookies fundamentally clash

If a cookie is SameSite, it is never sent on a genuine cross-site request — CORS or no CORS, the Access-Control-Allow-Credentials header is simply ignored. (Sub-domains of the same site are the one exception.) So you can't have both cross-site cookies and SameSite protection. With browsers now defaulting cookies to SameSite=lax and aggressively blocking cross-site cookies for privacy, this is the clearest sign yet that cookies aren't the future for cross-origin APIs — which is exactly why we move on.

5

What cookies quietly gave us for free

Before we tear cookies out, it's worth appreciating exactly what they were doing. Cookies are seductive because they bundle three separate jobs of token-based authentication into one neat package the browser manages for you. To replace cookies, we have to rebuild all three — and that's the roadmap for the rest of the chapter.

Reading the diagram

Seeing these as three separate responsibilities is the key insight — it's why replacing cookies takes three distinct pieces of work rather than one.

  1. Transport. Cookies travelled via the automatic Cookie/Set-Cookie headers. We replace this with the standard Bearer scheme: the token rides in the Authorization header instead.
  2. Client storage. Cookies lived in the browser's cookie jar. We replace this with the Web Storage API (localStorage), which holds the token across reloads and restarts.
  3. Server storage. The framework persisted session state for us. We replace this with our own database token store — which also means we now own the job of generating, hashing, and expiring tokens securely.

There's an upside to losing cookies' automatic behaviour: because the browser no longer attaches the token by itself, CSRF attacks largely vanish. We trade one class of problem for another.

6

Storing token state in a database

Once we abandon cookies, we lose the framework's free server-side storage, so we build our own. A token is a wonderfully simple thing — an ID, the user it belongs to, an expiry, and a bag of attributes — so a single database table does the job nicely.

schema.sql — the tokens table
CREATE TABLE tokens(
    token_id   VARCHAR(100) PRIMARY KEY,
    user_id    VARCHAR(30) NOT NULL,        -- who the token belongs to
    expiry     TIMESTAMP NOT NULL,
    attributes VARCHAR(4096) NOT NULL       -- everything else, as a JSON string
);
GRANT SELECT, INSERT, DELETE ON tokens TO natter_api_user;

The indexed columns (token_id, user_id, expiry) are the ones you'll search by; the variable attributes bag is just serialized JSON, which avoids a sprawling schema for rarely-queried fields.

The single most important security decision here is how you generate the token ID. It is incredibly tempting to use a database sequence — but those are predictable, usually just an incrementing integer. An attacker could simply wait for someone to log in and then guess the next ID to hijack their session. Token IDs must be unguessable, which means generating them from a cryptographically secure random source.

Definition · Entropy

Entropy measures how hard a value is to guess. A value with 128 bits of entropy has a 1-in-2128 chance of being any particular value — a secure minimum for long-lived secrets. Natter uses 160 bits: 20 bytes from a SecureRandom, then URL-safe Base64-encoded into a string.

Myth-busting · you can't "run out" of entropy

A persistent myth says the OS can exhaust its randomness if you read too much, tempting developers into dangerous home-grown workarounds. Not so. Once the OS has gathered ~256 bits of entropy at startup, it can generate strong random data essentially forever. Use SecureRandom (Java), or read /dev/urandom — never /dev/random, which can block for no security benefit. Trust the OS; don't roll your own.

The DatabaseTokenStore then just inserts the random ID with the JSON-serialized attributes on create, runs a simple query on read, and a DELETE on revoke. There's also housekeeping: a scheduled task runs DELETE FROM tokens WHERE expiry < current_timestamp every few minutes to stop the table growing forever (which would itself become a DoS vector), backed by an index on the expiry column to keep it fast.

Which database? It depends.

A SQL database is fine for demos and low traffic, but tokens are validated on every request, so the per-lookup cost adds up. Because tokens are structurally simple and rarely change after issuance, many production systems reach for a fast key-value store like Redis or a NoSQL store instead. Whatever you choose, insist on one consistency guarantee: token deletion must stick. A revoked token must never flicker back to life due to a replication glitch.

7

The Bearer authentication scheme

With cookies gone, how does the token travel? Reusing the old X-CSRF-Token header would work but feels wrong — it has nothing to do with CSRF anymore. There's a proper standard for exactly this: the Bearer scheme (RFC 6750). Born in OAuth2, it's now the universal way to send a token to an API.

Definition · Bearer token

A bearer token is one that authorizes anyone who holds it — no further proof of identity needed. That makes it easy to delegate access to a third party, but equally easy for an attacker to use if stolen. Like cash: whoever bears it, spends it. The token simply rides in the Authorization header: Authorization: Bearer QDAmQ9TSt…

The Bearer scheme is more than just a header name — it also restores standards-compliance to Natter. Recall that in chapter 4 we had to strip out the WWW-Authenticate header to stop the browser's ugly Basic-auth popup. The Bearer scheme gives us a proper challenge to send instead, and a standard way to tell the client why a token was rejected.

A standard challenge for an expired token
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="users",
    error="invalid_token",
    error_description="Token has expired"

The realm can point clients at different login endpoints (e.g. users vs admins); invalid_token tells the client to re-authenticate and retry.

The implementation is small: validateToken checks for an Authorization header starting with "Bearer ", strips that 7-character prefix, and looks the rest up. If it's missing or uses a different scheme, it's ignored so Basic auth still works at the login endpoint.

TokenController.java — parsing a Bearer header
var tokenId = request.headers("Authorization");
if (tokenId == null || !tokenId.startsWith("Bearer ")) {
  return;                          // not a Bearer token — leave Basic auth to work
}
tokenId = tokenId.substring(7);     // strip "Bearer " prefix

tokenStore.read(request, tokenId).ifPresent(token -> {
  if (Instant.now().isBefore(token.expiry)) {
    request.attribute("subject", token.username);   // valid → tag the request
    token.attributes.forEach(request::attribute);
  } else {
    response.header("WWW-Authenticate",             // expired → standard challenge
        "Bearer error=\"invalid_token\",error_description=\"Expired\"");
    halt(401);
  }
});
8

Web Storage — and the XSS trade-off it brings

The final pillar to replace is client-side storage. HTML 5 gives us the Web Storage API, which is both far easier to use than cookies and offers far more room (cookies cap out around 4KB). It exposes two objects on window, and the choice between them matters.

localStorage

Persists until explicitly deleted — survives browser restarts. Shared across all tabs from the same origin. The practical choice for auth tokens, so the user stays logged in.

sessionStorage

Cleared when the tab closes, and not shared between tabs. Tidy, but it forces a fresh login in every new tab — usually too annoying for auth tokens.

The API is refreshingly simple compared to wrangling document.cookie — just localStorage.setItem('token', t) to save and localStorage.getItem('token') to read. And each origin gets its own private copy, so a script from example.com can never see example.org's storage. Moving to Web Storage plus Bearer also lets us tighten CORS: since the browser no longer adds cookies automatically, we can drop Access-Control-Allow-Credentials entirely.

But every security decision is a trade-off, and this one has a sharp edge. A session cookie could be marked HttpOnly to hide it from JavaScript. Web Storage is, by design, only accessible from JavaScript — so that protection simply isn't available. This makes Web Storage more exposed to exfiltration if you have an XSS flaw.

Definition · Exfiltration

Exfiltration is stealing tokens or sensitive data from a page and quietly sending them to the attacker — without the victim noticing. The attacker then replays the stolen token to log in as the user from their own device.

Reading the diagram

The point of showing both is to puncture a comforting myth: that HttpOnly cookies "solve" XSS. They raise the bar, but they don't close the door.

  1. Exfiltration (Web Storage). Because localStorage is readable from JavaScript, an injected script loops over it and ships every token to the attacker's server — classically by creating <img> tags whose src URL carries the token. Browsers load image URLs eagerly, even without adding the element to the page, so the theft is silent.
  2. Proxy through the browser (works against cookies too). If the token is locked in an HttpOnly cookie and can't be read, the attacker doesn't bother — they simply make the malicious script issue the requests from inside the victim's browser. The browser attaches the cookie automatically, the requests look same-origin, and CSRF defences don't help. Tools like BeEF automate exactly this.

The honest takeaway: there is no reasonable defence if an attacker can run script on your page. Preventing XSS in the first place (chapter 2) must come first; Content-Security-Policy and Trusted Types are strong hardening layers on top.

9

Hardening the token database

We now own a database full of live tokens — effectively a drawer of working keys to every logged-in account. If an attacker reaches that database (via SQL injection or direct access), the damage dwarfs stealing one user's token. We harden it in layers, each blunting a different kind of access.

First, frame the threat precisely. An attacker who breaches the token store gains one of three capabilities, and each is harmful in its own way.

✏ Write access

🗑 Delete access

Revoke other users' tokens, locking them out — a denial of service.

Even with TLS protecting the API-to-database link, a breach gives the attacker read, write, or delete power — each with its own consequences. Tap each to see the matching defence.

Reading the diagram — and the layered defences

We don't rely on a single wall. We stack defences so a breach of one capability is still contained:

  1. Read access → defeated by hashing tokens (next). The stored value is a one-way hash, so stolen rows can't be replayed.
  2. Write access → defeated by HMAC authentication tags (section 10). Without the secret key, an attacker can't forge a token the API will accept.
  3. Delete access → mitigated by least-privilege database accounts and row-level security — separate accounts for queries vs bulk deletes, so an injection through the query path can't wipe the table.

Layer one: hash the stored tokens

A token is a credential, just like a password — so just as we hashed passwords in chapter 3, we hash tokens. The crucial difference: tokens already have high entropy (they're random), so we don't need a slow algorithm like Scrypt. A fast cryptographic hash like SHA-256 is perfect. The client gets the original random ID; the database stores only its SHA-256 hash. Steal the database and you get hashes that can't be reversed into usable tokens.

1
Separate the database from the API

Don't expose the DB to external clients; secure the API-to-DB link with TLS. The baseline.

2
Hash tokens (SHA-256)

Store only the hash. Read access yields useless hashes, not live tokens.

3
Authenticate with HMAC

Append a keyed tag so forged or tampered tokens are rejected before any DB lookup.

4

Split DB accounts (CQRS), restrict rows per user — contain what any single breach can touch.

10

Authenticating tokens with HMAC

Hashing stops a thief reading usable tokens, but it does nothing against an attacker with write access — they can just insert their own token row and use it. The fix is to make every valid token carry proof that your server created it, using a secret only your server knows. That proof is a message authentication code, and the standard one is HMAC.

Definition · MAC and HMAC

A message authentication code (MAC) computes a short authentication tag from a message and a secret key. Anyone with the key can recompute the tag; anyone without it cannot. Change one bit of the message and the tag changes completely. HMAC is the well-studied, standard way to build a MAC from a hash function — e.g. HMAC-SHA256 uses SHA-256 with a 256-bit secret key.

Here's the clever structural choice: we don't store the tag in the database. The database keeps only the plain (hashed) token ID. Instead, the server computes the HMAC tag and appends it to the token ID before handing it to the client — producing a two-part token of the form tokenId.tag.

Reading the diagram

The asymmetry is the whole trick: the database never sees the tag, and the key never touches the database.

  1. Create. The DatabaseTokenStore generates and stores the random token ID (hashed). The wrapping HmacTokenStore then computes HMAC-SHA256(key, tokenId), Base64-encodes it, and appends it: the client receives tokenId.tag.
  2. Validate. On each request the server splits the token at the dot, recomputes the expected tag from the ID and its secret key, and compares. Only if they match does it pass the bare tokenId down to the database for lookup.
  3. Forgery fails. An attacker with database write access can insert any tokenId they like — but they can't compute a matching tag without the secret key. Their forged token is rejected at the HMAC check, before any database lookup even happens (which also neatly sidesteps timing attacks against the DB).
Critical · constant-time comparison (again)

Just like the anti-CSRF check in chapter 4, comparing the provided tag against the computed one with an ordinary equals() opens a timing attack — an attacker measures tiny response-time differences to recover the tag one byte at a time. Always use a constant-time comparison like Java's MessageDigest.isEqual, which inspects every byte regardless of where the first mismatch is.

Where does the secret key live?

The HMAC key is just a 256-bit random value — but it has a requirement the token IDs don't: every API server must share the same key. If one server signs a token with a different key, another server (or the same one after a restart) would reject it as a forgery. So the key can't be generated fresh per process; it lives in an external keystore.

Definition · Keystore

A keystore is an encrypted, password-protected file holding cryptographic keys and TLS certificates for your API. Natter uses Java's KeyStore in the secure PKCS#12 format, generated with keytool. The keystore password is supplied at startup via a system property — never hard-coded — so developers needn't know the production password.

The implementation is elegant: HmacTokenStore wraps the DatabaseTokenStore rather than replacing it. The database store does the storage; the HMAC store adds the tag on the way out and verifies it on the way in. This decorator pattern means the same HMAC protection can wrap other token stores too — which turns out to be useful in the next chapter.

Going further · contain the blast radius

Two more refinements limit damage from a breach. Separate database accounts (the CQRS pattern) give the query path no delete privileges, so an injection there can't wipe tokens. And row-level security policies (in databases that support them, like PostgreSQL) restrict each query to only the current user's rows — so even a SQL flaw can't read or alter anyone else's tokens. Encrypting sensitive attributes is possible too, but it's genuinely hard to do well; nail the access controls first.

Key takeaways

  • Cookies struggle across domains and outside browsers. CORS selectively relaxes the same-origin policy so other origins can call your API.
  • CORS works via a preflight OPTIONS request — handle it before authentication, since preflights carry no credentials.
  • Cross-site cookies and SameSite are incompatible, and browsers increasingly block cross-site cookies — a strong reason to move beyond cookies.
  • Cookies bundle three jobs — transport, client storage, server storage — replaced by the Bearer scheme, Web Storage, and a database token store.
  • Generate token IDs from a cryptographically secure RNG (e.g. 160 bits from SecureRandom), never a predictable database sequence.
  • Web Storage eliminates CSRF but is exposed to XSS exfiltration — and against real XSS, no storage choice is truly safe, so prevent XSS first.
  • Hash stored tokens (SHA-256) so a database read can't yield usable tokens; HMAC with a secret key stops forgery and tampering.
  • Always compare tags in constant time; keep the HMAC key in a keystore; contain breaches with least-privilege accounts and row-level security.
★

Test yourself

Pop quiz · tap to reveal
For a cookie-based cross-origin login, which CORS header is needed besides Access-Control-Allow-Origin?
Access-Control-Allow-Credentials: true on both the preflight response and the actual response. Without it on both, the browser strips or rejects the cookie.
Which is a secure way to generate a random token ID?
Base64-encoding 20 bytes from a SecureRandom. A plain Random, a counter, or hashing the current time are all predictable — a hash is only as unguessable as its input.
Which standard HTTP authentication scheme is designed for token-based auth? (NTLM · HOBA · Basic · Bearer · Digest)
Bearer. Defined in RFC 6750, originally for OAuth2, now the universal way to send tokens to an API.
Where should the HMAC secret key be stored?
In a keystore accessible only to your API servers. Not in the database with the tokens, not hard-coded in source, and a memorable password is not a suitable key.
Which line correctly compares two HMAC tags? (computed.equals(provided) · Arrays.equals(...) · MessageDigest.isEqual(provided, computed))
MessageDigest.isEqual(provided, computed). It's constant-time, defeating timing attacks. The others short-circuit on the first mismatched byte.
Which API design pattern helps reduce the impact of SQL injection on the token store?
CQRS (Command Query Responsibility Segregation). It lets you use a query-only database account (no delete) for reads, so an injection on that path can't wipe tokens.
