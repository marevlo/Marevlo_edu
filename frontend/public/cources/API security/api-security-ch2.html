Meet the Natter API

Imagine you've had the perfect business idea: Natter, a social network for coffee mornings, book groups, and other small gatherings. You've got funding and a dream — but your investors are nervous about security. Where do you even start?

You start by building a small, honest version of the thing and learning to attack it yourself. That's exactly what this chapter does. Natter is built around social spaces — invite-only groups. Anyone can create a space and invite friends; any member can post a message that any other member can read; and the creator becomes the first moderator, with the power to delete abusive messages.

The API splits into two REST endpoints: one for normal users and one for moderators. Both speak JSON over HTTP to web and mobile clients, and both talk to a single shared database using SQL over JDBC (Java's database API).

The operations Natter offers

POST /spaces

Create a new social space. The caller becomes its owner; a unique ID comes back in the response.

POST /spaces/:id/messages

Add a message to a space.

GET /spaces/:id/messages

Read messages in a space. An optional since=<timestamp> limits results to a recent window.

DELETE (moderators)

The moderation API's single operation: delete an abusive message by its URI.

In this chapter you implement just the first one — create a space — because that single operation is enough to demonstrate every security lesson that follows. It's built in Java 11 with the lightweight Spark framework, an in-memory H2 database, and the Dalesbred library for friendlier database access.

The database schema

Before any API code, Natter needs somewhere to store data. The schema is deliberately tiny — just two tables — but it's worth understanding precisely, because nearly every attack in this chapter is really an attack on the conversation between the API and these tables.

Reading the diagram

This is an entity-relationship sketch — boxes are tables, and the line shows how they relate.

  1. spaces — one row per social space, storing its name and its owner. The space_id is the primary key (the unique handle for that row).
  2. messages — one row per message, carrying the author, the text, and the time it was sent. Its space_id is a foreign key pointing back at the space it lives in.
  3. The relationship is one-to-many: a single space can have many messages, but each message belongs to exactly one space.
  4. Both IDs come from SQL sequences — think of a sequence as a little counter that hands out a fresh, never-repeated number each time you ask.
schema.sql
CREATE TABLE spaces(
    space_id INT PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    owner VARCHAR(30) NOT NULL
);
CREATE SEQUENCE space_id_seq;          -- hands out fresh space IDs

CREATE TABLE messages(
    space_id INT NOT NULL REFERENCES spaces(space_id),  -- foreign key
    msg_id INT PRIMARY KEY,
    author VARCHAR(30) NOT NULL,
    msg_time TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    msg_text VARCHAR(1024) NOT NULL
);
CREATE SEQUENCE msg_id_seq;
CREATE INDEX msg_timestamp_idx ON messages(msg_time);  -- fast "catch-up" queries

The VARCHAR(30) and VARCHAR(255) limits matter later — the database will reject over-long input, but as you'll see, you should never rely on the database to catch your mistakes.

3

Controllers and the three-phase shape

Rather than cram all the logic into one giant file, Natter pulls its operations into controller objects. The main file just maps incoming HTTP requests to methods on those controllers. This separation isn't only tidy — it's what lets us bolt on security filters later without touching the core logic, which in turn makes security far easier to review.

Definition · Controller

A controller is the piece of code that responds to a user's request. The name comes from the model-view-controller (MVC) pattern. In a REST API there's no real "view" beyond JSON formatting, but organizing code into controllers still keeps the HTTP details separate from your business logic.

Almost every API operation, it turns out, has the same three-phase shape. Recognising this rhythm makes both the code and its security review much easier.

Reading the diagram

Read it left to right — data flows in one side and a response comes out the other. The reason we care so much about this shape is that each phase has its own characteristic security risk.

  1. Parse input — read the request body, pull out the fields you need. Risk: trusting input you haven't validated (this is where injection and bad-data attacks begin).
  2. Perform operation — run the database query or business logic. Risk: mixing user input into a SQL command (SQL injection).
  3. Prepare output — set the status code and build the response. Risk: echoing user input back unsafely or with the wrong content type (XSS).

Here's the first version of the createSpace controller. Read the warning comment carefully — this code is deliberately broken so we can attack it in a moment.

SpaceController.java — VULNERABLE version
public JSONObject createSpace(Request request, Response response) {
  // PHASE 1 — parse input
  var json = new JSONObject(request.body());
  var spaceName = json.getString("name");
  var owner = json.getString("owner");

  return database.withTransaction(tx -> {
    // PHASE 2 — perform operation
    var spaceId = database.findUniqueLong("SELECT NEXT VALUE FOR space_id_seq;");

    // !! WARNING: this line has a SQL INJECTION vulnerability !!
    database.updateUnique(
        "INSERT INTO spaces(space_id, name, owner) " +
        "VALUES(" + spaceId + ", '" + spaceName + "', '" + owner + "');");

    // PHASE 3 — prepare output
    response.status(201);
    response.header("Location", "/spaces/" + spaceId);
    return new JSONObject()
        .put("name", spaceName)
        .put("uri", "/spaces/" + spaceId);
  });
}

See the three phases mapped right onto the code. The highlighted line — building SQL by gluing strings together — is the bug we'll exploit next.

4

Wiring up the routes — and Spark's filters

A controller method does nothing until something calls it. A route is the rule that says "when a POST arrives at /spaces, call createSpace." One line does it: post("/spaces", spaceController::createSpace).

Definition · Route

A route defines how to turn an HTTP request into a method call on one of your controllers — e.g. POST /spaces → SpaceController.createSpace.

Spark (the framework) also gives us filters — hooks that run before or after the request handler. These are where, in the next chapter, we'll add authentication, rate-limiting, and so on, all without editing controllers. But the timing of the three filter types is subtle and worth nailing down now, because security headers must land on every response — including error responses.

Spark runs filters in a fixed order around your handler. Press a button to trace each path. The key insight: afterAfter-filters run no matter what, so that's where security headers belong.

Reading the diagram

The whole point is to see which filters fire when things go wrong versus when they go right — because a header you only set on success is a header an attacker can dodge by triggering an error.

  1. Before-filters run first, every time — great for validation (e.g. "reject anything that isn't JSON").
  2. Request handler — your controller. It either finishes cleanly or throws an exception.
  3. If it succeeds, the after-filters run. If it throws, Spark skips the after-filters and runs the matching exception handler instead.
  4. afterAfter-filters run at the very end on both paths. That's exactly why we put security headers here — so they're present even on error responses, which are otherwise easy to forget.
Try it out

Run mvn clean compile exec:java, then call the API: curl -d '{"name":"test space","owner":"demo"}' http://localhost:4567/spaces. You'll get back a 201 Created. Now try unusual input — an owner over 30 characters, or a name with a single quote in it. Those experiments are exactly how we find the next two vulnerabilities.

5

SQL injection — the attack

The code you just wrote has a serious flaw: a SQL injection vulnerability. It's the number-one entry on the OWASP Top 10 for a reason — it's everywhere, and it's devastating.

Definition · Injection attack

An injection attack happens when unvalidated user input is dropped directly into a dynamic command or query that the application then executes — letting the attacker control the code that runs. It applies to SQL, LDAP lookups, OS commands, HTTP headers, even log files.

Why is Natter vulnerable? Because it builds the SQL command by gluing strings together, mixing trusted SQL syntax and untrusted user input into one string. To the database, it's all just one string — it has no way to tell which parts you wrote and which parts the user wrote. That confusion is the entire attack.

In SQL, text values sit inside single quotes. The code dutifully wraps the user's input in quotes. But… what if the user's input contains a single quote of its own? Watch what a carefully crafted owner value does.

Reading the diagram

The genius (and horror) of the attack is that every character in that owner string is doing precise surgery on your SQL. Walk through it:

  1. The single quote ' closes the quote your code opened — so the database thinks the owner value has ended early.
  2. ); closes the VALUES(...) parenthesis and terminates your INSERT statement cleanly.
  3. DROP TABLE spaces; is now read as a brand-new, second command — one that deletes the entire spaces table.
  4. -- starts a SQL comment, so the leftover ') your code tacks on at the end is ignored instead of causing a syntax error. The attack stays clean and silent.

The result: the API responds 201 Created as if nothing happened — but your spaces table is gone, and with it your whole network.

"Little Bobby Tables"

This exact attack is immortalised in the famous XKCD comic "Exploits of a Mom," where a mother names her son Robert'); DROP TABLE Students;-- and wipes the school's entire student database. Funny — until it's your production system.

6

The fix: prepared statements & least authority

There are weak fixes and there's the right fix. You could try to escape every dangerous character — but different databases treat different characters specially, and the rules change between versions, so escaping is a losing game. You could validate inputs strictly — good practice, but you can't ban single quotes outright, or you'd reject legitimate names like Mary O'Neill.

The real answer is to stop mixing data and code in the first place. A prepared statement keeps them in completely separate lanes.

Interactive · Figure 2.7
Prepared statements keep data and code apart

Toggle between the broken approach and the fix to see exactly why one is exploitable and the other isn't — even with identical malicious input.

With a prepared statement, the SQL is parsed and compiled with placeholders (?) first — before any user data is anywhere near it. The values are then handed over separately, so the database can never mistake them for commands. Toggle the buttons above to compare.

Reading the diagram

The difference is when the database sees the structure of your command versus the user's data.

  1. In the broken version, you build one big string — SQL and user input fused together — and hand the database a finished command. It parses the whole thing as code, so injected SQL runs.
  2. In the fixed version, you send the command first, with ? placeholders where values go. The database parses and compiles that skeleton before seeing any data.
  3. The actual values arrive afterwards, through a separate channel. The database slots them in as pure data — never re-parsing them as code.
  4. So even '); DROP TABLE spaces; -- just becomes a (very oddly named) space. It's stored harmlessly as text, exactly as intended.
SpaceController.java — FIXED with a prepared statement
database.updateUnique(
    "INSERT INTO spaces(space_id, name, owner) VALUES(?, ?, ?);",
    spaceId, spaceName, owner);   // values passed separately, never parsed as SQL

That's the whole fix. Note the bonus: the database can compile the query once and reuse it, so prepared statements are often faster too. There's genuinely no excuse not to use them.

Using an ORM?

If you use an object-relational mapper or any abstraction over raw SQL, check its docs to confirm it uses prepared statements under the hood. For non-SQL databases, look for parameterized calls so you never build commands by concatenation.

Defence in depth: the principle of least authority

Here's a question worth sitting with: why could the API drop a table at all? Deleting tables is not something Natter ever legitimately does — so it should never have had permission to. Prepared statements stop the injection, but limiting permissions contains the damage if any injection ever slips through. Two defences, stacked.

Principle · Least Authority (POLA)

The principle of least authority (also "least privilege") says every user and process should get exactly the permissions it needs to do its job — no more, no less. Natter never needs to drop tables, so don't grant it the ability.

schema.sql — a restricted database user
CREATE USER natter_api_user PASSWORD 'password';
GRANT SELECT, INSERT ON spaces, messages TO natter_api_user;
-- note: no DROP, no DELETE, no UPDATE — only what the API actually needs

The app loads the schema as a privileged user, then switches to this restricted user for all real traffic. Now if injection ever recurs, the DROP TABLE is denied for lack of permissions — the blast radius is contained.

7

Input validation

Most security flaws boil down to one thing: an attacker submits input that violates an assumption you quietly made. You assumed a username fits in 30 characters; you assumed it contains only letters. Check those assumptions, explicitly, before the input does any work.

In memory-unsafe languages like C, an unchecked oversized input can trigger a buffer overflow — and in the worst case, remote code execution (RCE), where the attacker runs their own code in your process. Java is memory-safe, so Natter is largely spared those. But validation still matters: an over-long username sails right past the API and only fails at the database with an ugly 500 error.

Two definitions

Buffer overflow — when input exceeds the memory allocated for it and overwrites adjacent memory. In unsafe languages this can let an attacker overwrite code.

Remote code execution (RCE) — when an attacker injects and runs their own code in your API, gaining the full permissions of your legitimate code.

Even "safe" Java isn't immune — insecure deserialization

Some libraries that turn data back into Java objects will run code inside an object's constructor as they rebuild it. A maliciously crafted message can abuse this to achieve RCE — this has hit Java's built-in serialization and popular JSON parsers like Jackson Databind. The defence: allowlist a known set of safe classes and refuse to deserialize anything else.

The golden rule of validation flips your instinct around: don't try to list the bad inputs — list the good ones.

Principle · Allowlist, never blocklist

Always define acceptable inputs, not unacceptable ones. An allowlist says exactly what's valid and rejects everything else. A blocklist tries to name everything bad — and fails the moment an attacker thinks of something you didn't. For huge input spaces like Unicode, allow general classes ("decimal digit") rather than listing values.

SpaceController.java — validating inputs
var spaceName = json.getString("name");
if (spaceName.length() > 255) {                         // length check
  throw new IllegalArgumentException("space name too long");
}

var owner = json.getString("owner");
if (!owner.matches("[a-zA-Z][a-zA-Z0-9]{1,29}")) {       // allowlist via regex
  throw new IllegalArgumentException("invalid username");
}

That regex reads: a letter, then 1–29 more letters or digits. So usernames are 2–30 characters, alphanumeric, never starting with a number — a safe basic alphabet.

Watch out · ReDoS

Regular expressions can themselves be attacked. A regular expression denial of service (ReDoS) forces a pattern to take ages by making it backtrack endlessly — e.g. ^(a|aa)+$ against a long string of as ending in b. Design patterns so there's only one way to match any input; if unsure, prefer simple string operations. (Java 11 is notably more resistant than older versions.)

Finally, wire up an exception handler so validation failures return a clean 400 Bad Request instead of a scary 500, by catching IllegalArgumentException and JSONException and formatting a tidy JSON error.

8

XSS — the attack on your clients

Here's a counter-intuitive truth: even a REST API that only ever consumes and produces valid JSON can still be used to attack a web browser. The vulnerability is cross-site scripting (XSS), and it sneaks in through how the browser interprets your response.

Definition · Cross-Site Scripting (XSS)

XSS lets an attacker run their script in the context of another site. A reflected XSS bounces a malicious input straight back in the response. Because browsers enforce the same-origin policy — scripts from a site can read that site's cookies and data — a successful XSS lets the attacker's script do everything the real site's scripts can: steal session cookies, read private messages, alter data.

Natter's current error handler hands an attacker three gifts at once: it echoes the bad input back, it doesn't escape it, and it serves the response as Content-Type: text/html instead of JSON. Put those together and a browser will happily execute an injected script. Here's the full chain.

Reading the diagram

The attack chains four small oversights into one serious breach. Follow them in order:

  1. Crafted form — the attacker hosts a page with a hidden form pointed at Natter, using enctype="text/plain" and a little JavaScript to auto-submit it the moment a victim loads the page.
  2. Looks like JSON — the form fields are cleverly named so that when the browser serialises them, the result is valid JSON: {"x":"=","name":"x","owner":"<script>...</script>"}.
  3. API reflects it — Natter parses the JSON, rejects the script-filled username as invalid, and helpfully echoes the offending value back in an error message — served as text/html.
  4. Script runs — because the browser was told the response is HTML, it parses the <script> tag and executes it. On a real site that script could steal the victim's session and impersonate them.
9

Producing safe output

The XSS attack worked because of how Natter built its output. Fixing it means cleaning up three separate bad habits in what the API sends back — and then adding a layer of standard security headers as a safety net.

Look closely at that vulnerable error response and you'll spot three distinct problems, each making the attack easier:

1
Leaks internals

It returns the exact Java exception class and the Jetty server version — a roadmap for an attacker hunting known vulnerabilities.

2
Echoes raw input

It reflects the user's bad input back without escaping — the very hole the script slips through.

3
Wrong Content-Type

It serves text/html instead of JSON, so the browser runs the reflected script.

Five steps to safe output

The book lays out a clear remedy. Take them together and the XSS attack simply has nowhere to land.

① Be strict in what you accept

Require Content-Type: application/json on all input. An HTML form physically cannot submit that type, killing the form trick.

② Build output with a real JSON library

Never assemble responses by gluing strings — use a proper library so output is always well-formed.

③ Set the correct Content-Type

Explicitly set application/json on every response — especially error responses, which often default to HTML.

④ Never echo the Accept header

If you negotiate output formats, construct the Content-Type from what you actually produced — don't copy the client's header.

And the fifth step: don't echo user input in error messages at all. Return a generic "invalid username" rather than parroting the attacker's payload back to them.

The security headers safety net

Even with clean output, you add a set of standard HTTP headers to every response (remember the afterAfter filter — that's why it runs on errors too). These tell the browser to stop second-guessing your intentions.

Header
What it does & why it matters
X-Content-Type-Options: nosniff
Stops the browser from ignoring your Content-Type and "sniffing" the content itself. Without it, JSON can get reinterpreted as HTML or JavaScript. Always set this.
X-Frame-Options: DENY
Prevents your responses being loaded inside a frame, blocking drag-and-drop clickjacking. Superseded by CSP in newer browsers, but worth setting both.
X-XSS-Protection: 0
Counter-intuitively, current guidance is to disable the old browser XSS filter on APIs — it has caused vulnerabilities of its own.
Cache-Control: no-store
Stops browsers and proxies caching potentially sensitive responses. Safest default is no caching, re-enabled selectively where needed.
Content-Security-Policy
For an API, a minimal default-src 'none'; frame-ancestors 'none'; sandbox tightly restricts what any injected script could even do.
Main.java — hardening every response
// Reject anything that isn't JSON, before the handler even runs
before((request, response) -> {
  if (request.requestMethod().equals("POST") &&
      !"application/json".equals(request.contentType())) {
    halt(415, "{\"error\":\"Only application/json supported\"}");
  }
});

// afterAfter → runs on EVERY response, including errors
afterAfter((request, response) -> {
  response.type("application/json;charset=utf-8");
  response.header("X-Content-Type-Options", "nosniff");
  response.header("X-Frame-Options", "DENY");
  response.header("X-XSS-Protection", "0");
  response.header("Cache-Control", "no-store");
  response.header("Content-Security-Policy",
      "default-src 'none'; frame-ancestors 'none'; sandbox");
  response.header("Server", "");   // hide the Jetty version
});

Notice the charset=utf-8 — specifying it explicitly blocks a sneaky trick where an attacker forces a different encoding (like UTF-16BE) to steal JSON data.

Key takeaways

  • SQL injection happens when user input is concatenated into a query; prepared statements with placeholders eliminate it by keeping data and code in separate lanes.
  • Apply the principle of least authority — give the database user only SELECT and INSERT, never DROP — so any future injection is contained.
  • Validate inputs before use, and always allowlist the good rather than blocklist the bad. Watch for ReDoS in your regexes.
  • Even a pure-JSON API can enable XSS against browser clients — the killer detail is serving the wrong Content-Type.
  • Produce safe output: require JSON input, build responses with a real library, set the correct Content-Type everywhere, and never echo raw input in errors.
  • Apply standard security headers on every response via an afterAfter filter — so they reach error responses too, which are the easiest to forget.
  • The OWASP Top 10 is a useful baseline, but real security comes from learning principles that defeat whole classes of attack, not memorising a checklist.

Test yourself

Pop quiz · tap to reveal
Which of these is not in the 2017 OWASP Top 10? (Injection · Broken Access Control · Security Misconfiguration · XSS · CSRF · Known-vuln components)
Cross-Site Request Forgery (CSRF). It was in the list for years but dropped out as web frameworks improved their defences. (CSRF is covered later, in chapter 4.)
An attacker supplies the author value john' UNION SELECT password FROM users; -- to a concatenated query. What comes back?
Any messages written by John and the passwords of all users. This is a UNION-based injection — it shows the attacker isn't limited to the original table; they can pull data from other tables too.
Given int msgLen = buf.getInt(); byte[] msg = new byte[msgLen]; — in memory-safe Java, what's the main risk?
Passing a very large message length. The attacker makes the program allocate a giant array (up to 2GB), and a few such requests exhaust all memory. Invalid values just throw exceptions, thanks to Java's memory safety.
Which header stops browsers ignoring your Content-Type and guessing instead?
X-Content-Type-Options: nosniff. It forces the browser to respect the Content-Type you declared.
If your API can return JSON or XML based on the Accept header, what should you never do?
Two things: never include un-sanitized input in error messages (XSS risk), and never copy the Accept header straight into the Content-Type — always build the Content-Type from what you actually produced.
