1

Five controls, one threat model

In Chapter 2 we made Natter functionally sound — no SQL injection, no XSS. But the API still trusts anyone who shows up. This chapter adds the proactive machinery that decides who may do what, and proves it later if something goes wrong.

The elegant part is that these five controls aren't an arbitrary grab-bag. Each one is the direct antidote to one of the six STRIDE threats we met in Chapter 1. If you remember nothing else from this chapter, remember this mapping — it's how a security engineer reasons backwards from "what could go wrong" to "what do I build."

Denial of ServiceSTRIDE — D
→
Rate-limiting — cap how fast anyone can hit the API, so a flood can't exhaust your resources.
SpoofingSTRIDE — S
→
Authentication — verify users really are who they claim, so nobody can impersonate another.
Information disclosureSTRIDE — I
→
Encryption (HTTPS) — hide data in transit and at rest from eavesdroppers.
TamperingSTRIDE — T
→
Encryption + access control — modern encryption also detects modification; ACLs stop unauthorised writes.
RepudiationSTRIDE — R
→
Audit logging — record who did what and when, so actions can't be denied later.
Elevation of privilegeSTRIDE — E
→
Access control — enforce that users can only do what they're permitted, nothing more.
A subtle but vital detail

Only two of these controls actually reject a request outright: rate-limiting and access control. Authentication does not — a failed login doesn't immediately bounce you. Why? Because we want every attempt logged, including the failed ones (those are often the first sign of an attack). So authentication just records whether you're verified, and access control later decides whether that's good enough.

The order matters — a guided pipeline

These controls run as a sequence of filters, and the sequence is deliberate. Put them in the wrong order and you either waste resources on requests you'll reject anyway, or you fail to log an attack. Step through the pipeline below to see exactly why each control sits where it does.

Reading the diagram

Read it strictly left to right, and think about cost at each step — the guiding idea is "spend the least effort to throw out a bad request."

  1. Rate-limiting first. It's the cheapest possible check, so it goes before anything that costs real CPU. If you're being flooded, you want to drop requests before spending effort authenticating them.
  2. Authentication second — but it never rejects. It just attaches "this is verified user demo" (or "anonymous") to the request, so everything downstream knows who's asking.
  3. Audit logging third, right after we know who but before we decide allow/deny. That ordering is what guarantees failed and denied attempts still get recorded.
  4. Access control fourth. Now — armed with identity and a logged record — we make the allow-or-deny call. Denied requests stop here with a 403.
  5. Application logic last. Only fully-vetted requests ever reach your actual business code. And the whole exchange travels inside HTTPS, so none of it is readable on the wire.
3

Rate-limiting for availability

Availability is genuinely hard to defend. An attacker can rent a botnet and generate enormous traffic for almost nothing, while you spend real money trying to absorb it. You can't make denial of service impossible — but you can make sure that when you're overwhelmed, you degrade gracefully instead of collapsing entirely.

The single most important habit: never let an unauthenticated request consume significant resources. A flood of anonymous requests should be shed as early and as cheaply as possible. That's the whole philosophy behind putting rate-limiting first.

Two flavours of DoS

Network-level DoS floods you with traffic unrelated to real requests (easy to spot, often filtered by a firewall). Application-layer DoS (L7) is sneakier — it sends syntactically valid requests, just far too many of them, so it looks like legitimate traffic until it buries you.

It's also worth understanding amplification, the trick that makes distributed DoS so devastatingly cheap for attackers. The classic example is DNS amplification — and seeing how it works explains why "small request in, huge response out" protocols are so dangerous.

Reading the diagram

The cleverness of amplification is that it abuses a protocol's asymmetry — a small question that triggers a big answer.

  1. Spoof the source. The attacker sends DNS queries but forges the "return address" so it reads as the victim's IP rather than their own.
  2. Small queries out. Each query is tiny and cheap, scattered across many DNS servers. Crucially, DNS is unauthenticated — the servers have no way to know the request is forged.
  3. Huge replies flood the victim. Each server dutifully sends a much larger response — to the victim. Multiply that across thousands of servers and a trickle of attacker traffic becomes a tidal wave hitting the target.

These exploit UDP-based protocols (common in IoT). Network-level attacks like this are filtered with firewalls or specialist DoS-protection services with the capacity to soak up the flood.

Implementing rate-limiting with Guava

For application-layer protection, you add a rate limiter right in the API. In production you'd also rate-limit at a load balancer or reverse proxy — but enforcing it in the server too is a deliberate choice. It's an example of defense in depth: if the proxy is misconfigured or bypassed, the server still protects itself.

Principle · Defense in depth

Use multiple independent layers of defence, so that the failure of any single layer doesn't breach the whole system. Rate-limit at the proxy and at the server. Belt and braces.

Main.java — the very first filter
// Allow just 2 requests/second so it's easy to see in action.
// In production, set this BELOW what one server can truly handle.
var rateLimiter = RateLimiter.create(2.0d);

before((request, response) -> {
  if (!rateLimiter.tryAcquire()) {        // non-blocking: false = over the limit
    response.header("Retry-After", "2");  // tell client when to come back
    halt(429);                            // 429 Too Many Requests
  }
});

The 429 status and Retry-After header are the polite, standard way to say "you're going too fast, try again in 2 seconds." Rejecting instantly costs almost nothing, which is exactly what you want when under load.

Sizing the limit

Per-server limits should be a fraction of your total target. Need 1,000 req/s across 10 servers? Set each server to ~100 req/s — and actually test that each one can sustain it. Always leave headroom below the true ceiling.

Authentication — proving who you are

Almost everything Natter does needs to know who is doing it. In person you recognise a friend by their face; online, you can't. So you have to ask people to tell you who they are — and then make them prove it. For a social app, weak proof means impersonation and rumour-spreading; for a bank, it means someone spends your money. Nearly all security begins here.

Currently, Natter just believes the owner field when you create a space and the author field when you post. That's pure spoofing waiting to happen. The fix is to require a verified login and then check that claimed identities match the authenticated user.

The most important conceptual point is one the book is careful about, and it's easy to get wrong: authentication itself must not reject the request. Look at how it sits in the pipeline.

Reading the diagram

Compare the two runs and notice what's the same: in both, the request reaches the audit log. That's the entire reason for this design.

  1. Valid credentials → authentication tags the request with the verified username (subject = "demo"). Audit logs it. Access control sees a valid subject and lets it through to the app.
  2. Wrong/missing credentials → authentication tags the request as anonymous (subject stays null) but does not reject it. Audit still logs the attempt. Only then does access control reject it — if the operation needs a login — with a 401.
  3. The payoff: failed and malicious attempts get recorded. If authentication rejected immediately, those attempts would vanish before the audit log ever saw them — blinding you to attacks in progress.

HTTP Basic authentication

APIs don't render login forms, so Natter uses the standard HTTP Basic scheme (RFC 7617): the client sends the username and password in an Authorization header. The credentials are joined with a colon (demo:changeit) and Base64-encoded.

An HTTP Basic auth header
Authorization: Basic ZGVtbzpjaGFuZ2VpdA==
                      ↑ just Base64 of "demo:changeit"
Critical · Base64 is NOT encryption

Base64 is just a reversible encoding — anyone watching the network can decode it back to demo:changeit in one step. So Basic auth credentials are sent in effectively plain text. You must only ever use Basic auth over an encrypted HTTPS connection (section 7). This is a perfect example of how threats chain: information disclosure on the wire instantly becomes a spoofing attack.

Storing passwords safely with Scrypt

Here's a rule with no exceptions: never store passwords directly. Instead you store a hash — a fixed-length, random-looking fingerprint. To check a login, you hash the supplied password the same way and compare fingerprints. The real password never touches your disk, so a database breach doesn't immediately hand over everyone's password.

But not just any hash will do. Fast hashes (like plain SHA-256) are a liability here, because attackers can compute billions of them per second. Modern password hashes — Scrypt, Argon2, Bcrypt, PBKDF2 — are deliberately slow and memory-hungry to cripple brute-force attacks. Two ideas make them work.

Interactive · Concept
Two ideas that make password hashing safe

Toggle the two defences to see what each one stops. Together they turn a stolen database from a disaster into merely a headache.

Scrypt combines both defences. Toggle above to see why each matters. The headline: a hash that takes ~100ms and 32MB of memory drops an attacker from billions of guesses per second to a few thousand.

Reading the diagram

These two ideas attack the two different ways a thief tries to crack stolen hashes.

  1. Slowness (work factor). Scrypt is tuned to burn ~100ms of CPU and ~32MB of RAM per hash. That's invisible to a real user logging in once, but it throttles an attacker from billions of guesses per second down to a few thousand — buying everyone time to change passwords after a breach.
  2. Salting. A salt is a unique random value mixed into each password before hashing. It means two users with the same password get totally different hashes — which defeats rainbow tables (giant precomputed lookup tables of common-password hashes). The attacker can no longer crack everyone at once; they must attack each password individually.

Scrypt stores its parameters and the salt inside the hash string itself, so verification later can reproduce the exact same computation automatically.

Principle · Secure defaults

Establish secure defaults for every security-sensitive parameter — like Scrypt's cost factors — and only relax them if you've proven they genuinely hurt performance. Start safe; loosen only with evidence.

UserController.java — registering a user
var username = json.getString("username");
var password = json.getString("password");

if (!username.matches(USERNAME_PATTERN))     // same allowlist validation as ch.2
  throw new IllegalArgumentException("invalid username");
if (password.length() < 8)                     // minimum length
  throw new IllegalArgumentException("password must be at least 8 characters");

// Scrypt params (recommended ~2019): 32768 = CPU/mem cost, 8 = block, 1 = parallelism
var hash = SCryptUtil.scrypt(password, 32768, 8, 1);
database.updateUnique(
    "INSERT INTO users(user_id, pw_hash) VALUES(?, ?)", username, hash);

Note the prepared statement (chapter 2's lesson) and the deliberately expensive hash. The salt is generated automatically and baked into the stored hash string.

Authentication then runs as a before filter on every request: pull the Base64 credentials from the header, look up the user's stored hash, and let Scrypt verify the password. On success it tags the request — request.attribute("subject", username) — and on failure it simply leaves that attribute null. (Remember: it never rejects here.)

6

Encryption with HTTPS

Authentication stops people lying about who they are — but it does nothing to protect the request itself as it crosses the network. Picture checking your work gossip over coffee-shop wifi: without encryption, anyone on that hotspot can read every message you send and receive. Worse, they can read your Base64 password sailing past and then simply log in as you.

This is exactly the threat-chaining we keep seeing. One weakness (unencrypted traffic = information disclosure) directly enables another (stolen credentials = spoofing). The fix is to encrypt everything in transit with HTTPS — which is just ordinary HTTP running over TLS (Transport Layer Security; older versions were called SSL).

Reading the diagram

The mental model is "data is safe at the endpoints but exposed in between" — and there are two separate "in betweens" to worry about.

  1. In transit (HTTPS/TLS). Protects data on the wire between client and API. The first time a client connects, TLS performs a handshake: the server proves its identity with a certificate, then both sides agree on fresh encryption keys for that session. After that, every request and response is encrypted and tamper-checked.
  2. At rest (database encryption). Protects data sitting in storage from anyone with database access — whether a curious employee or an attacker who got in another way. This chapter focuses on transit; at-rest encryption comes later.
  3. Inside the API, data is decrypted — and that's expected. TLS operates at a lower layer, so your code just sees normal plain requests. The encryption is invisible to your application logic.

Enabling it in Spark is genuinely a one-liner once you have a certificate. For local development the mkcert tool generates a certificate trusted by your browser; Java wants it in PKCS#12 format.

Main.java — turning on HTTPS
import static spark.Spark.secure;

// First two args: keystore file + its password. Last two (client certs) → null for now.
secure("localhost.p12", "changeit", null, null);
Never disable certificate validation

It's tempting to pass -k / --insecure to curl (or the equivalent in a library) to silence certificate errors. Don't make a habit of it. In production, disabling certificate checks throws away the entire identity guarantee of TLS — an attacker can then impersonate your server. Get comfortable generating real certificates instead.

Strict Transport Security (HSTS)

For browser clients, you can add a Strict-Transport-Security header so the browser refuses to ever use plain HTTP for your domain again — closing the small window where a first request might go out unencrypted. (Just don't set it on localhost, or you'll lock yourself out of plain-HTTP dev servers until it expires.) And note from the CIA triad: TLS gives you confidentiality and integrity, but not availability — encryption alone won't save you from a DoS.

Audit logging for accountability

Accountability is simply being able to answer "who did what, and when?" — and the way you answer it is an audit log. It's hard to overstate how important this is. A good audit log is what lets you reconstruct an attack after the fact, prove who was responsible, and — analysed in real time — spot an attack while it's still happening.

Two design decisions make audit logging genuinely useful, and both are about when you log.

Reading the diagram

Two placement decisions do all the work here, and both are about capturing things you'd otherwise lose.

  1. Log START (before the request runs). A matching start and end record (tied together by a shared audit ID) means that if the server crashes mid-request — say, on an input an attacker crafted to crash it — you still have a record that the request was attempted. Log only at the end, and a crash erases all trace.
  2. After authentication — so the log knows who made each request. Logging before authentication would leave you with anonymous, near-useless entries.
  3. Before access control — so that denied and failed attempts are recorded too. Those rejected requests are frequently the most interesting ones from a security standpoint; logging after access control would silently discard them.
  4. Log END (after the response). Captures the outcome — the status code, whether it succeeded or was forbidden — completing the picture.
Principle · Separation of duties

Audit logs should be readable by different people than the system administrators. Why? Because admin accounts are the most powerful — and therefore the most in need of monitoring. The watchers shouldn't be able to quietly edit the record of what they watched. (In finance, the same principle means whoever requests a payment can't also approve it — a guard against fraud.)

Practically, audit logs go to durable storage (a database table or file) so they survive a crash, and the audit table deliberately has no foreign-key constraints — you want to record exactly what was attempted, even if it's inconsistent with other data. In production you'd ship these to a centralised SIEM system that correlates logs across many services to hunt for threats.

8

Access control with ACLs

We now know who every caller is — but Natter still lets anyone do anything. Any user can read any space, post anywhere, even delete other people's messages as a self-appointed moderator. The final control, access control, fixes that by deciding, for each request, whether this authenticated user is actually allowed to do this particular thing.

Access control comes after authentication (you must know who someone is before deciding what they can do) and is one of only two controls that reject requests outright. When it does reject, the right HTTP status matters — and the two relevant codes are endlessly confused, so let's pin them down.

401
Unauthorized

Despite the name, this means not authenticated. The server doesn't know who you are — you sent no credentials, or wrong ones. You might fix it by trying different credentials. Comes with a WWW-Authenticate header telling you how to log in.

403
Forbidden

This means authenticated, but not allowed. We know exactly who you are — you're just not permitted to do this. Trying different credentials won't help; you need someone to grant you access.

A useful one-liner: 401 = "who are you?" · 403 = "I know who you are, and no."

Access control lists (ACLs)

Natter uses one of the simplest access-control models: an access control list. For each space, we keep a list of which users may access it and what they're permitted to do. Permissions are just three letters — r (read), w (write/post), and d (delete, for moderators).

Reading the diagram

An ACL check is mechanical and runs as a before filter, keeping the access decision cleanly separated from the business logic.

  1. The space owner is automatically granted rwd when they create the space — full control, in the same database transaction that creates the space.
  2. Each protected route declares the permission it needs — posting needs w, reading needs r, deleting needs d.
  3. The filter looks up the caller's permission string for that space and checks whether the required letter is present. Missing letter → 403 Forbidden, and the controller never even runs.
  4. This mirrors the principle of least authority from chapter 1: a reader gets exactly r and no more, so they can't accidentally (or maliciously) do damage.
UserController.java — a permission-checking filter factory
// Returns a filter that demands `permission` for a given HTTP method.
public Filter requirePermission(String method, String permission) {
  return (request, response) -> {
    if (!method.equalsIgnoreCase(request.requestMethod())) return;  // not our method
    requireAuthentication(request, response);                          // must be logged in first

    var spaceId = Long.parseLong(request.params(":spaceId"));
    var username = (String) request.attribute("subject");

    var perms = database.findOptional(String.class,
        "SELECT perms FROM permissions WHERE space_id = ? AND user_id = ?",
        spaceId, username).orElse("");            // default: no permissions

    if (!perms.contains(permission)) halt(403);   // missing → Forbidden
  };
}

Wiring it up is then just one before(...) line per route — e.g. posting a message requires "w", reading requires "r". The controllers stay blissfully unaware of access control.

9

The privilege-escalation trap

Here's a subtle bug that's easy to ship and nasty in practice. To make Natter social, we add an operation that lets a member invite others into a space. The natural choice is to require only read permission to add a member — after all, any member should be able to bring a friend, right? That tiny decision opens a hole.

Definition · Privilege escalation

A privilege escalation (or elevation of privilege — the "E" in STRIDE) happens when a user with limited permissions exploits a flaw to grant themselves, or someone else, more permissions than they were given.

Interactive · Attack walkthrough
How a read-only user grabs full control

Step through the exploit to watch a user with only r permission escalate all the way to deleting other people's messages.

demo2 was granted only read access — the weakest permission. Step through to see how that's enough to seize total control, and how a one-line change shuts it down.

Reading the diagram

The exploit chains the "add member" operation against itself, and it works because the operation could grant more power than the caller holds.

  1. demo2 has only r. They can read messages in the space — nothing else, supposedly.
  2. They create a fresh account (evildemo2) — registration is open to anyone.
  3. They add that account to the space with rwd. The "add member" route only required r to call, but let the caller choose any permissions for the new member — including ones the caller never had.
  4. Now they delete messages through their new full-power account. A read-only user just escalated to moderator.
  5. The fix: require rwd (full permissions) to add a member — so only owners and moderators can hand out access. The safer general rule is to never let anyone grant permissions they don't themselves possess.
The deeper lesson

Whenever an operation hands out authority, ask: "could the caller use this to gain power they don't already have?" Two safe patterns: only let users grant permissions they themselves hold, or restrict the granting operation to fully-privileged users. The flaw here wasn't in any one line of SQL — it was in the permission required to call the operation. Access-control bugs hide in policy, not just code.

Key takeaways

  • The five controls map cleanly onto STRIDE: rate-limiting→DoS, authentication→spoofing, encryption→disclosure/tampering, audit logging→repudiation, access control→elevation of privilege.
  • Order is deliberate: rate-limit → authenticate → audit log → access control → logic. Only rate-limiting and access control reject; the rest observe.
  • Rate-limiting goes first and rejects cheaply, ideally at a proxy and the server (defense in depth). Use 429 + Retry-After.
  • Authentication never rejects — it tags identity so failed attempts still get logged. Use HTTP Basic, but only over HTTPS, since Base64 is trivially decoded.
  • Store passwords with a slow, salted hash like Scrypt — slowness defeats brute force, salt defeats rainbow tables. Establish secure defaults.
  • HTTPS/TLS protects data in transit (and encrypt sensitive data at rest too). It gives confidentiality and integrity, but not availability.
  • Audit log before and after each request, positioned after authentication but before access control, so denied and crashed requests are still recorded. Apply separation of duties.
  • ACLs enforce least authority; beware privilege escalation — never let users grant permissions they don't hold. 401 = not authenticated, 403 = not allowed.

Pop quiz · tap to reveal
Which statement about rate-limiting is true? (after access control · stops all DoS · enforced as early as possible · only for high-traffic APIs)
Rate-limiting should be enforced as early as possible. The whole point is to reject excess requests before they consume resources — so it runs before even authentication.
Which response header tells a client how long to wait before retrying?
Retry-After. Sent alongside a 429 to say "come back in N seconds."
Which are desirable properties of a secure password hash? (parallelizable · lots of disk · lots of bandwidth · lots of memory · random salt · lots of CPU)
Lots of memory, a random salt per password, and lots of CPU. High CPU/memory cost slows brute-force and dictionary attacks; the per-password salt defeats precomputed rainbow tables.
Why must HTTP Basic auth only be used over HTTPS?
The credentials are only Base64-encoded, which anyone snooping the network can trivially decode back to the plaintext password.
Which CIA-triad goal does TLS not provide?
Availability. TLS gives confidentiality and integrity, but does nothing to stop a denial-of-service attack.
Which principle says audit logs should be managed by different people than the system admins?
The principle of separation of duties — the most privileged accounts are the ones most in need of independent monitoring.
Which status code means "authenticated, but not permitted"?
403 Forbidden. (401 Unauthorized, confusingly, actually means not authenticated.)
