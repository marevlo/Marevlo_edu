OAuth2 & OpenID Connect

Scoped tokens: delegating part of your access

Picture the bad old days of sharing access. To let an accounting app read your bank transactions, you handed it your banking username and password — and prayed. Now it could read transactions, sure, but also transfer funds, cancel payments, do anything you could do. Tokens fixed the password-sharing part, but the tokens we've built so far still grant full access. A third party with your token can do everything you can. That's the problem this chapter opens with.

The fix is to restrict what a token can do, confining it to a well-defined scope. Your accounting app should get a token that reads recent transactions and nothing else — no payments, no transfers. We represent this as one or more string labels stored on the token, like transactions:read or payment:create. Because a token usually carries several, we just call them scopes.

Definition · scoped token

A scoped token limits the operations that can be performed with it. The permitted set of operations is the token's scope, specified by one or more scope labels (collectively, "scopes"). A token scoped to read_message simply cannot post or delete.

For Natter, we define one scope per API operation and have the login endpoint attach whatever scope was requested (defaulting to all of them):

create_space post_message read_message list_messages delete_message add_member

Enforcement is a new access-control filter, requireScope, that runs before each operation. It checks the HTTP method matches, reads the token's scope attribute, and rejects the request with 403 Forbidden (and a helpful insufficient_scope challenge) if the required scope is missing.

TokenController.java — enforcing a required scope
public Filter requireScope(String method, String requiredScope) {
  return (request, response) -> {
    if (!method.equalsIgnoreCase(request.requestMethod())) return;  // wrong method? skip this rule
    var tokenScope = request.<String>attribute("scope");
    if (tokenScope == null) return;                  // unscoped (direct password login) → allow
    if (!Set.of(tokenScope.split(" ")).contains(requiredScope)) {
      response.header("WWW-Authenticate",
          "Bearer error=\"insufficient_scope\",scope=\"" + requiredScope + "\"");
      halt(403);                              // missing the needed scope → forbidden
    }
  };
}
Watch out · the privilege-escalation trap

Notice the login endpoint itself can mint a token with any scope. So a client holding a narrowly-scoped token could just call login again and exchange it for a broader one — quietly escalating its own privileges and bypassing every scope check. The fix: require a special restricted scope (e.g. full_access) to call login, so a limited token can't be used to widen itself. Revoking a token via logout, by contrast, needs no scope at all.

2

Scopes vs permissions: who decides?

Scopes and permissions look similar — both gate what you can do — but they answer to different masters, and confusing them leads to muddled security design. The cleanest way to see the difference is to ask: who gets to decide?

Mandatory Access Control
Permissions

Set and enforced by a central authority — the company that owns the API. Users can't grant themselves permissions or hand them to others. Think of the classified-document model: SECRET clearance can't read TOP SECRET, no matter who wants to share it. This is what the chapter 3 ACLs implemented.

Discretionary Access Control
Scopes

Chosen by the user, at their discretion, when delegating part of their own authority to a third-party app. It's up to you how much of your access to grant. OAuth scopes are fundamentally about this kind of voluntary, user-driven delegation.

Definition · MAC vs DAC

Under mandatory access control (MAC), a central authority sets permissions and users cannot grant them away. Under discretionary access control (DAC), users can delegate some of their own permissions to others. OAuth2 provides DAC — "delegated authorization" — layered on top of an existing MAC permission system.

Beyond the theory, there are two practical design differences worth holding onto. They explain why you don't just reuse your permission names as scopes.

Different designers, different goals

An admin designs permissions around organizational policy ("which employees may touch which documents"). You design scopes by anticipating how users will want to delegate to apps. Google's pattern is telling: coarse, all-or-nothing scopes for system APIs like key management, but fine-grained scopes for the Fitness API so users share exactly which health stats they want.

Operations vs objects

Scopes typically restrict only which API operations a client may call, judged from the HTTP request alone. Permissions also restrict which specific objects are touched. A list_files scope lets a client call the list endpoint; the user's permissions then decide which files come back.

3

Introducing OAuth2

Letting users hand-craft scoped tokens is better than sharing passwords, but it's confusing — a user rarely knows which scopes an app actually needs, so they grant too few (and it breaks) or too many (and it's risky). The better idea: let the app request the scopes it needs, and let the API ask the user to consent. That request-and-consent dance is exactly what OAuth2 standardises.

OAuth introduces a central Authorization Server (AS) that handles user authentication, consent, and token issuance for all your APIs at once. The tokens it issues for accessing APIs are called access tokens. This centralisation turns out to be valuable even if you have no third-party clients at all — it's a big part of why OAuth2 became the dominant API-security standard.

Reading the cast

The names are historical and not always intuitive, so it helps to anchor each one to Natter:

  1. Resource Owner (RO) — the user. Called "owner" because it's their resources (messages, spaces) the app wants. The name isn't always accurate, but it stuck.
  2. Client — the third-party app or service trying to act on the user's behalf.
  3. Authorization Server (AS) — the dedicated service that authenticates the user, asks for their consent, and issues access tokens.
  4. Resource Server (RS) — the API hosting the resources. That's the Natter API. It receives access tokens and must validate them.

The flow in one sentence: the client asks the AS for a token of a given scope, the AS checks the user consents, issues an access token, and the client presents that token to the RS.

4

Client types and authorization grants

Before a client can ask for a token, it registers with the AS and gets a client ID. But not all clients can keep a secret — and that single fact splits them into two categories that shape everything about how they authenticate.

Confidential clients

Run in a protected place the user can't poke at — a server you control. They can hold a client secret (or stronger credentials like a JWT or TLS cert) and authenticate to the AS, proving they're the real client and blocking impersonation/phishing.

Public clients

Run entirely on the user's device — mobile apps, SPAs in the browser. They cannot keep a secret, because anyone who downloads the app can extract it. So they get no client secret and lean on other defences (like PKCE) instead.

To get an access token, the client needs the user's consent in the form of an authorization grant, which it then trades at the AS's token endpoint. OAuth2 defines several grant types for different situations — and importantly, two of them are now discouraged:

preferred
Authorization Code (+ PKCE)

The browser-redirect flow covered next. The right choice for almost every client type — web apps, SPAs, mobile, desktop, CLI. This is where we'll spend our time.

service-to-service
Client Credentials

The client uses its own credentials to get a token, with no user involved at all. Useful for microservice calls (chapter 11).

discouraged
Resource Owner Password Credentials (ROPC)

The user gives their password to the app, which forwards it to the AS — reintroducing the very password-sharing OAuth set out to kill. Fine for testing; may be deprecated. Avoid.

discouraged
Implicit grant

Returned the access token straight from the authorization endpoint in the URL fragment — a 2012 workaround for the pre-CORS era. The token could leak via browser history or scripts. Now that CORS exists, the security best-practice doc advises against it.

Finding the endpoints · discovery

OAuth2 doesn't fix the paths of the token and authorization endpoints, so they vary by AS. Rather than hard-coding them, a conforming AS publishes a JSON discovery document at /.well-known/oauth-authorization-server (OIDC providers may use /.well-known/openid-configuration). It lists the endpoint locations and supported features. Because clients send credentials and tokens to these endpoints, only ever fetch the discovery document over HTTPS from a trusted URL.

5

The Authorization Code grant

This is the heart of OAuth2 — the flow nearly every client should use. Its cleverness is that the user's credentials never touch the client at all; authentication happens directly between the user's browser and the AS. The client only ever receives a short-lived code, which it then exchanges for a token over a secure back-channel. Let's walk the whole dance.

Interactive · Figure 7.5
The Authorization Code grant, step by step

Step through the flow. Notice where each message travels — browser redirects in the front channel, the code exchange over a direct HTTPS back channel.

The client redirects the user's browser to the AS, the user logs in and consents, the AS redirects back with a short-lived code, and the client swaps that code for an access token over a direct HTTPS call. Step through to see each move and which channel it uses.

Reading the diagram

The deep idea is the split between two channels: the untrusted front channel (browser redirects, where things can leak) carries only a one-time code, while the trusted back channel (direct HTTPS) carries the actual token.

  1. Redirect to authorize. The client sends the browser to the AS's authorization endpoint with its client_id, requested scope, response_type=code, a redirect_uri, and a random state value (a CSRF defence, like the anti-CSRF token in chapter 4).
  2. Authenticate & consent. The AS authenticates the user directly in the browser and shows a consent screen naming the client and the scope it wants. The user approves or declines.
  3. Redirect back with a code. On approval, the AS redirects to the client's redirect_uri with a short-lived, single-use code and echoes the state. The client checks state matches what it sent.
  4. Exchange code for token. The client calls the AS token endpoint directly over HTTPS — not a redirect — sending the code, redirect URI, and (if confidential) its client credentials. This back channel keeps the token safe from browser-based theft.
  5. Use the access token. The AS returns the access token (with scope and expiry). The client presents it to the Natter API in an Authorization: Bearer header, exactly as in earlier chapters.

Why a code at all, rather than just returning the token? Because the redirect travels through the browser, where it can leak (history, logs, the Referer header, malicious scripts). The code is short-lived and single-use, and exchanging it requires a direct call — so even a stolen code is far less useful than a stolen token.

Definition · open redirect & pre-registered URIs

An open redirect is when a server can be tricked into redirecting the browser to an attacker's URI — great for phishing, since the link starts at a trusted site. To prevent the AS becoming an open redirect (and to stop codes being sent to the wrong place), clients must pre-register their exact redirect URIs; the AS refuses any it doesn't recognise.

The right redirect URI depends on the client. A web app uses a dedicated server endpoint. A mobile app has two options, and the difference matters for security: a private-use URI scheme like myapp://callback (simple, but any app can claim the same scheme — a malicious app could intercept the code), or a claimed HTTPS URI (Android App Links / iOS Universal Links), where the app proves it owns the web domain via a published JSON file, so no other app can hijack it. Claimed HTTPS URIs are recommended. Desktop and CLI apps that can't register a scheme typically spin up a temporary http://127.0.0.1 loopback server to catch the redirect.

6

Hardening code exchange with PKCE

There's still a gap. A public client has no secret, so what stops a malicious app that intercepted the authorization code from simply exchanging it for a token? The answer is PKCE (Proof Key for Code Exchange, pronounced "pixy") — and it's now recommended for every client type, not just mobile.

Interactive · RFC 7636
How PKCE ties the code to whoever started the flow

Step through the verifier/challenge mechanism, then see why an intercepted code is useless to an attacker.

The client invents a random secret (the verifier), sends only its hash (the challenge) to the AS up front, then reveals the original verifier when exchanging the code. The AS checks they match. Step through to see why this defeats code interception.

Reading the diagram

PKCE is essentially a per-request, dynamically-generated secret — one that never travels through the leaky front channel in usable form.

  1. Verifier. Before redirecting, the client generates a high-entropy random string — the code_verifier (e.g. 32 bytes from SecureRandom) — and stores it locally alongside state.
  2. Challenge. It does not send the verifier. It sends the SHA-256 hash of it — the code_challenge — as a query parameter on the authorize redirect. A hash can't be reversed, so seeing the challenge tells an attacker nothing useful.
  3. Exchange. When swapping the code for a token, the client now sends the original verifier. The AS hashes it and checks it matches the challenge from step 2. Match → issue the token.
  4. Attacker fails. An attacker who intercepts both the redirect and the code still can't exchange it — they never saw the verifier, and they can't derive it from the challenge's hash. The code is worthless to them.

Most OAuth client libraries compute PKCE for you, and AS's that don't support it simply ignore the extra parameters — so there's no reason not to always use it.

Computing a PKCE challenge from a verifier
var verifierBytes = new byte[32];
secureRandom.nextBytes(verifierBytes);                  // high-entropy random secret
var verifier = encoder.encodeToString(verifierBytes);
request.session(true).attribute("verifier", verifier);  // keep it locally, like state

var sha256 = MessageDigest.getInstance("SHA-256");
var challenge = encoder.encodeToString(
    sha256.digest(verifier.getBytes("UTF-8")));        // send only the HASH up front
return authorizeRequest + "&code_challenge=" + challenge + "&code_challenge_method=S256";
7

Refresh tokens

Alongside the access token, the AS often hands the client a second token: a refresh token. The idea is to keep access tokens short-lived without nagging the user to re-approve every few minutes. When the access token expires, the client quietly trades the refresh token for a fresh one — no user involvement.

Definition · refresh token

A refresh token lets a client obtain a new access token when the old one expires, without asking the user again. It's sent only over the secure back channel between client and AS, so it's considered safer than an access token that gets sprayed across many APIs.

The real payoff connects directly to chapter 6. Remember the revocation headache with stateless JWTs? Short access-token lifetimes plus refresh tokens solve it elegantly: the access token can be a self-contained JWT that the API validates locally with no database lookup, and because it expires quickly, the client is forced back to the AS regularly — which is exactly where revocation can be enforced. The complexity of revocation moves to the client (which must refresh) and the AS, instead of bloating a blocklist. Many AS's also rotate the refresh token on each use, so if one is stolen, the theft is detected when the legitimate client's copy stops working.

8

Validating an access token

Now the API's side of the story. In earlier chapters we just looked the token up in our own database. But OAuth tokens are issued by the AS, not the API — and sharing a database between them is a bad idea (one SQL injection anywhere compromises everything). So how does the Resource Server check a token it didn't create? Two standard answers, with a classic trade-off between them.

Interactive · two ways to validate
Token introspection vs JWT access tokens

Toggle between the two approaches to weigh the network-call-vs-local-check trade-off.

Introspection asks the AS about every token over HTTPS — always current, but a network call each time. JWT access tokens are verified locally with the AS's public key — fast and offline, but revocation is harder. Toggle to compare.

Reading the comparison

The whole choice comes down to a single tension: freshness versus speed.

  1. Token introspection (RFC 7662). The API POSTs the token to the AS's introspection endpoint (authenticating as a registered client) and gets back a JSON {"active": true, "scope": ..., "sub": ...}. Always reflects the token's current state — including instant revocation — but costs a network round-trip per request (you can cache briefly to soften this).
  2. JWT access tokens. The AS signs the token; the API verifies the signature locally using the AS's public key, with no call to the AS. Fast and resilient, but a JWT stays valid until it expires — revocation is harder (which is where short lifetimes + refresh tokens from section 7 come in).

The crucial detail for JWTs: use public-key signatures, not the shared HMAC keys of chapter 6. Only the AS holds the private key and can mint tokens; every API gets the public key and can only verify them. That's the principle of least authority — a compromised API can't forge tokens. The AS publishes its public keys as a JWK Set at an HTTPS URL the API fetches and refreshes.

Principle · least authority via asymmetric keys

With a shared HMAC key, anyone who can verify a token can also forge one — fine inside one trust boundary, dangerous across many. Public-key (asymmetric) signing splits the capability: the AS's private key signs, the widely-shared public key only verifies. So an attacker who breaches an API gains the power to check tokens, never to create them. Use a separate key for signing and (if you also encrypt tokens) for encryption.

Two more practical notes. When verifying a JWT access token, always strictly validate the alg header against an expected algorithm (recall the chapter 6 attacks), and confirm the iss and aud claims so a token minted for one API can't be replayed at another. And because the API now trusts the AS completely about who a token belongs to, the HTTPS connection between them must be locked down: pin the trust to your AS's own CA rather than every public CA, and restrict to TLS 1.2/1.3 with strong, forward-secret cipher suites.

Choosing a signature algorithm

For JWT signatures, EdDSA (Ed25519) is the best pick where supported — fast, compact, and designed to resist side-channel and nonce-reuse bugs. ECDSA (ES256) is more widely supported and shares many benefits, but beware: a repeated or weakly-random nonce leaks the private key (this very bug hacked the PS3 and stole Bitcoin). RSA works and verifies quickly but signs slowly and produces large tokens; prefer PSS padding over the older PKCS#1 v1.5. For encrypted tokens, prefer ECDH-ES and avoid the insecure RSA1_5 padding.

9

Single sign-on

Here's a benefit that falls out of OAuth almost for free. Because every client now delegates login to the same AS, the AS becomes the one place users authenticate — and that gives you single sign-on across all your apps and APIs without building anything extra.

The mechanism is simple. When a client redirects the user to the AS to get a token, the AS authenticates them and — because this all happens in a browser — drops a session cookie. The next time any client redirects to that same AS, it sees the existing cookie and skips the login prompt entirely. The user glides from app to app without re-entering credentials. The AS can also remember previously-granted consent, so it can skip the consent screen too. This even works across mobile apps from different developers, as long as they use the system browser (which is exactly why section 5 recommended it). When the user logs out, the client revokes the token at the AS's revocation endpoint to cut off further access.

One token per API, please

It's tempting to reuse a single access token across many of your APIs for convenience. Resist it. The more places a token travels, the more places it can be stolen — and a token good everywhere is a master key. Issue a separate access token per API, scoped to just what that API needs.

10

OpenID Connect

OAuth2 gives you basic SSO, but its real focus is delegated access to APIs — not answering "who is this user?" In fact, getting an access token doesn't even guarantee the user was present, let alone how they logged in. OpenID Connect (OIDC) fills that gap, extending OAuth2 into a full authentication protocol. If you've ever clicked "Log in with Google," you've used OIDC.

OIDC rearranges the OAuth pieces slightly. The API the client wants — a UserInfo endpoint that returns identity claims (name, email, and so on) — lives on the AS itself. So instead of three parties there are effectively two, with new names to match.

Definition · OP and RP

In OIDC, the combined AS-and-identity-API is the OpenID Provider (OP) — e.g. Google. The client is the Relying Party (RP) — the app delegating authentication to the OP. OIDC adds three things to OAuth: a standard UserInfo endpoint, a way to demand fresh or strong authentication, and session/logout management.

The signature feature is the ID token — a signed (optionally encrypted) JWT carrying identity and authentication claims. Rather than making three round-trips just to learn the user's name, the OP can return an ID token directly. The client validates it like any JWT (signature, exp, iss, aud) plus OIDC extras like nonce (anti-replay), auth_time (when they logged in), and acr/amr (how strongly — e.g. whether two-factor was used).

The most important rule · ID tokens ≠ access tokens

It is tempting to use an ID token (it's a nice signed JWT identifying the user!) to access your API. Don't — for third-party clients. An ID token authenticates the user to the client; it isn't scoped and carries no notion of delegated consent. If APIs accepted ID tokens, then any site you logged into with Google could replay your Google ID token back at Google's own APIs and impersonate you. The memorable framing: ID tokens are like usernames; access tokens are like passwords. Use access tokens for access, ID tokens for identity.

OIDC also hardens the redirect flow when ID tokens come back through the browser (implicit/hybrid). A nonce the client generates and checks in the returned token defeats replay; optional encryption protects personal data in transit; and c_hash/at_hash claims let the client confirm the code and access token weren't swapped. But the simplest defence is the one we've been recommending all along: use the authorization code flow with PKCE, where the OP does all these checks server-side when the code is exchanged.

When an API legitimately needs the ID token

Sometimes an API genuinely needs identity claims or proof of how a user authenticated (say, a payments API wanting assurance of strong auth). In that case the client may pass the ID token in addition to a normal access token — never instead of it. The API makes all access-control decisions on the access token, then verifies the ID token's signature and issuer, checks its own ID is in the audience and the client is the authorized party (azp), and confirms the ID token's subject matches the access token's resource owner.

Key takeaways

  • Scoped tokens let users delegate limited access to third-party apps — granting some operations but not others — instead of sharing a password or an all-powerful token.
  • Scopes are discretionary (user-delegated, designed around delegation), while permissions are mandatory (central-authority, designed around policy). OAuth layers DAC on top of MAC.
  • OAuth2 centralises authentication and consent at an Authorization Server, with four roles: resource owner (user), client (app), AS, and resource server (API).
  • Clients are confidential (can keep a secret) or public (can't). The Authorization Code grant is the right choice for all of them; the implicit and ROPC grants are discouraged.
  • The auth code flow splits a leaky front channel (browser redirects carrying a one-time code) from a secure back channel (direct HTTPS carrying the token); always pre-register redirect URIs.
  • PKCE ties the code to whoever started the flow via a verifier/challenge, defeating code interception — use it for every client type.
  • Refresh tokens keep access tokens short-lived without re-prompting the user, making stateless JWT access tokens practical despite the revocation problem.
  • Validate access tokens by introspection (a call to the AS, always current) or as public-key-signed JWTs (verified locally, fast) — the latter using asymmetric keys so APIs can verify but never forge.
  • OpenID Connect adds a full authentication layer with ID tokens — but ID tokens are for identity, access tokens are for access; never swap them for third-party clients.

Test yourself

Pop quiz · tap to reveal
What are typical differences between scopes and permissions?
Permissions are often set by a central authority, while scopes are designed for delegating access; and scopes typically restrict only which API operations can be called, whereas permissions also restrict which objects can be accessed. (Whether scopes or permissions are more fine-grained varies case by case.)
Which two standard OAuth grants are now discouraged?
The implicit grant (risk of access tokens leaking from the URL) and the ROPC grant (the client learns the user's password — the very thing OAuth avoids).
Which type of client should a mobile app be, and why?
A public client. Any credential embedded in an app download can be extracted by a user, so a mobile app can't keep a client secret — it relies on PKCE and claimed-HTTPS redirect URIs instead.
Which redirect URI type should a mobile client prefer? And should the auth code grant always use PKCE?
Prefer a claimed HTTPS URI (App Links / Universal Links) — a private-use scheme can be hijacked by a malicious app. And yes, always use PKCE; it improves security in all cases and is ignored by servers that don't support it.
Which endpoint determines if an access token is valid, and who may revoke a token via the standard revocation endpoint?
The token introspection endpoint validates a token. Only the client the token was issued to may revoke it via the standard revocation endpoint — a resource server cannot revoke on the client's behalf.
Which key validates a public-key signature, and may you use an ID token to access an API?
The public key validates the signature (the private key creates it). And no — never use ID tokens for access control with third-party clients; ID tokens are for identity, access tokens are for access.
