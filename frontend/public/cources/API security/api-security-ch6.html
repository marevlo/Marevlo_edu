Self-Contained Tokens & JWTs

Natter has millions of users and the token database is buckling. So we move the state onto the client itself — encoding it into the token — then layer on HMAC, encryption, and authenticated encryption to make that safe, meet the JWT standard, and solve the revocation problem stateless tokens create.

1

Moving the state onto the client

Natter has succeeded its way into a problem. Millions of users now hammer the token database on every single request, and it's becoming the bottleneck that throttles the whole platform. The tantalising idea this chapter explores: what if the token didn't need a database lookup at all — what if the token carried its own state?

The core trick is disarmingly simple. Instead of storing the token's attributes in a database row keyed by a random ID, we serialise those attributes — the username, the expiry, any extra claims — into a JSON object, Base64url-encode it, and hand that to the client as the token. When the client presents it back, we just decode and parse the JSON to recover everything we need. No database, no lookup, no bottleneck.

JsonTokenStore.java — state encoded into the token itself
// create: pack the token's fields into JSON, then Base64url-encode the whole thing
var json = new JSONObject();
json.put("sub", token.username);                 // "sub" = subject (standard JWT claim)
json.put("exp", token.expiry.getEpochSecond());   // "exp" = expiry, as UNIX seconds
json.put("attrs", token.attributes);
return Base64url.encode(json.toString().getBytes(UTF_8));

// read: just decode and parse — no database involved at all
var decoded = Base64url.decode(tokenId);
var json = new JSONObject(new String(decoded, UTF_8));
var expiry = Instant.ofEpochSecond(json.getInt("exp"));
var username = json.getString("sub");

The short key names (sub, exp) aren't arbitrary — they're standard JWT claim names, which we'll meet properly in section 3. They save bytes and set us up for the standard format.

This is wildly insecure as written

Stop and notice the gaping hole: the token is just Base64-encoded JSON, which anyone can decode, edit, and re-encode. A user could log in, change "sub":"test" to "sub":"admin", and re-encode — or skip logging in entirely and forge a token from scratch. Base64 is encoding, not encryption; it provides exactly zero security. We have to add cryptographic protection before this is usable at all. That's the entire arc of this chapter.

2

Protecting JSON tokens with HMAC

The fix for forgery is one we already built in chapter 5: the HmacTokenStore. By computing an HMAC-SHA256 authentication tag over the encoded JSON — using a secret key only the server knows — and appending it to the token, we make tampering and forgery detectable. Change one byte of the JSON and the tag no longer matches.

Because the HmacTokenStore is a wrapper, enabling this is a two-line change: build the JsonTokenStore, then wrap it. The decorator pattern from last chapter pays off again — the same HMAC code that protected database tokens now protects client-side JSON tokens without modification.

Reading the diagram

The key insight is that the tag is computed from the claims plus a secret — so the two are bound together inseparably.

  1. Encode. The token's claims become a JSON object, Base64url-encoded into the first segment. At this point it's readable and forgeable by anyone.
  2. Tag & append. The server computes HMAC-SHA256(secret, encodedClaims) and appends it after a period. A period is the perfect separator because it's not a valid Base64url character, so you can always find where the claims end and the tag begins.
  3. Attacker edits. An attacker changes the claims (say, sub to admin) and re-encodes — but they can't recompute the tag without the secret key. The server recomputes the expected tag, finds it doesn't match the attached one, and rejects the token.

In STRIDE terms, HMAC defends against spoofing (forging a token) and tampering (altering one) — but note it does not hide anything: the claims are still plainly readable. That's the next problem.

3

JSON Web Tokens

What we just hand-built — encoded JSON claims plus an authentication tag — is, almost exactly, a JSON Web Token. JWTs (pronounced "jots") standardised this idea in 2015, and they've become the most widely used token format in the world. The only thing our home-made version is missing is a standard header describing how the token was produced.

Definition · JSON Web Token

A JWT is a standard format for a self-contained security token. It carries a set of claims about a user as a JSON object, plus a header describing the token's format and algorithm. JWTs are cryptographically protected against tampering, and can optionally be encrypted.

A signed JWT in the common "JWS Compact Serialization" is three Base64url segments joined by periods: header.payload.signature. Tap through the parts below to see what each holds.

A JWS Compact Serialization is exactly three URL-safe Base64 chunks separated by periods: the header (metadata), the payload (the claims), and the signature (HMAC tag or digital signature). Tap any segment to decode it.

Reading the diagram

Compare this to our hand-rolled token from section 2 and the family resemblance is obvious — JWT just adds the header segment at the front.

  1. Header — decodes to {"alg":"HS256"}: metadata saying how the token was protected. This is the part our home-made token lacked.
  2. Payload — the claims set: {"sub":"test","aud":"https://localhost:4567","exp":...}. The actual statements about the user.
  3. Signature — the HMAC tag (or, with other algorithms, a real digital signature) computed over header.payload. The header is included in the tag, so it can't be tampered with either.

JWT is just one spec in a family called JOSE (JSON Object Signing and Encryption): JWS for signing, JWE for encryption, JWK for keys, JWA for algorithms. People say "JWT" loosely to mean the whole bundle.

The standardised claim names are genuinely useful — every JWT library and many API frameworks understand them, so they interoperate across languages and services. Here are the standard claims; most describe constraints on use rather than facts about the user.

Why audience matters · replay & reflection

The aud and iss claims aren't bureaucratic box-ticking — they stop real attacks. A replay attack captures a valid token and re-sends it; checking aud stops a token stolen from one API being replayed against a different one, even if both share a key. A reflection attack bounces a token back at its own issuer to confuse an auth protocol; verifying issuer and audience defeats it. Always check that your API's URI is in the audience.

4

The JOSE header — flexibility that bites

The JOSE header is where most of JWT's power lives — and most of its danger. Its whole job is to tell the recipient how the token was protected. But that creates a deep problem: you're being asked to trust instructions from the token about how to verify the token, before you've verified anything. That circularity is the root of several famous attacks.

The infamous alg: none attack

In 2015, researcher Tim McClean found that many JWT libraries blindly obeyed the alg header. An attacker could change it to influence verification — or set it to none, which told the library to skip signature checking entirely, accepting any forged token. The lesson: the header is attacker-controlled data, and trusting it to tell you how to authenticate violates the basic rule "never trust a claim you haven't authenticated."

The defining feature of JOSE is cryptographic agility — the ability to swap algorithms later. Noble goal, dangerous implementation: JOSE puts the algorithm in the message itself. The safer approach is key-driven agility: store the algorithm as metadata with the key on the server, and change it by changing the key. An attacker can't touch your server-side keys, so they can't confuse the algorithm.

Principle · never trust an unauthenticated claim

You can't authenticate a JWT until you've processed the alg header — but the header is part of the unauthenticated, attacker-controllable token. Resolve the paradox by deciding the algorithm from your trusted key store, not from the token. This is why headers are best stripped or strictly validated.

The same trust problem haunts the headers that say which key to use. JOSE offers one safe option and two dangerous ones:

kid — safe ✅

Just a key ID string. You look it up in your own server-side set of keys, so the attacker only picks which of your trusted keys, never supplies one.

jwk — unsafe ❌

The full key, embedded in the token. Trusting the sender to provide the verification key throws away all security — they sign with their key and hand it to you.

jku — unsafe ❌

A URL to fetch the key from. Same flaw as jwk, plus it makes your server fetch attacker-chosen URLs — an SSRF risk (chapter 10).

Definition · key rotation & headless JWTs

Key rotation is periodically changing your keys so no single key is overused or stays compromised for long; kid lets a recipient quickly find the right one. A headless JWT strips the header entirely and rebuilds it from expected values on receipt — smaller and lower-attack-surface, though nonstandard. The tokens we built in section 2 were effectively headless JWTs.

From here on, we use a real, well-maintained library (Nimbus JOSE + JWT) rather than hand-rolling crypto — it handles the alg-confusion checks, rejects unknown critical headers, and does constant-time tag comparison for us. The rule of thumb: always prefer a vetted library for anything cryptographic.

5

Why we need to encrypt attributes

HMAC stopped forgery and tampering — but remember, the claims are still plainly readable. That was fine in a locked-down datacenter database. It is not fine once the token lives on the client, where it can leak through XSS, phishing, or simple carelessness. Anything sensitive in the token — a user's name, location, role, or internal implementation details — is now exposed.

So we add encryption: transforming the readable plaintext into obscured ciphertext using a secret key, reversible only by someone holding that key. The guiding rule is Kerckhoff's Principle — the scheme must stay secure even if the attacker knows every detail of the algorithm, as long as the key is secret. Which is exactly why you only ever use openly-studied, public algorithms, never home-grown secrecy.

The dominant algorithm is AES, a block cipher: it encrypts a fixed 16-byte block at a time. To encrypt anything longer, you pick a mode of operation — and that choice is do-or-die. The classic demonstration is encrypting an image of a penguin two ways:

Reading the diagram

This single image is the most famous lesson in applied cryptography, and it's worth internalising why it happens.

  1. ECB (Electronic Code Book) encrypts every 16-byte block independently. Identical plaintext blocks — like the large areas of identical white background — produce identical ciphertext blocks. The pattern of the image survives encryption, so you can still see the penguin. ECB is essentially never safe.
  2. CTR (Counter Mode) mixes a per-block counter into the encryption, so identical plaintext blocks encrypt to completely different ciphertext. The output looks like pure noise, revealing nothing.

The takeaway: "I used AES" tells you almost nothing about whether data is safe. The mode — and, as we'll see next, whether it's authenticated — is what matters.

6

Authenticated encryption

Here's a subtlety that catches many people out: encryption alone protects confidentiality but not integrity. An attacker who can't read your CTR-mode ciphertext might still be able to change it in predictable ways — without ever decrypting it.

The example is chilling in its simplicity. Suppose a CTR-mode token decrypts to the string user=brian, and the attacker knows that. Because of how CTR works, flipping specific bits of the ciphertext flips the corresponding bits of the plaintext — so they can surgically change it to user=admin without knowing the key at all. This is a chosen ciphertext attack.

Definition · authenticated encryption

Authenticated encryption combines an encryption algorithm (for confidentiality) with a MAC (for integrity). It guarantees both that an attacker can't read the data and that they can't alter or forge it. In STRIDE terms it covers information disclosure plus spoofing and tampering — the full set.

You can build authenticated encryption by combining AES-CTR with HMAC — but how you combine them matters enormously. There are two ways, and only one is reliably safe:

Interactive · combining encryption + MAC
Encrypt-then-MAC vs MAC-then-Encrypt

Toggle between the two orderings. The ordering is not a style choice — one is secure and one is fragile.

The safe ordering is Encrypt-then-MAC (EtM): encrypt the plaintext, then MAC the ciphertext. Toggle to compare it with the riskier MAC-then-Encrypt.

Reading the diagram

The principle behind EtM is "authenticate exactly what you transmit," so you can verify integrity before doing anything risky.

  1. Encrypt-then-MAC (safe). Encrypt the plaintext, then compute the MAC over the ciphertext. On receipt you check the MAC first; if it fails you reject immediately and never even attempt decryption. The attacker can't probe the decryption logic at all.
  2. MAC-then-Encrypt (fragile). MAC the plaintext, then encrypt both together. Now you must decrypt before you can check the MAC — which means attacker-tampered ciphertext gets fed through your decryption code, and the error behaviour can leak information (the classic padding-oracle family of attacks).

Because getting this right is fiddly, cryptographers built dedicated authenticated encryption modes — like AES-GCM — that do it correctly in one primitive. Better still is a high-level library that removes the choice entirely.

That high-level approach is exactly what NaCl (pronounced "salt") provides. Its SecretBox gives you authenticated encryption in a single call — no mode selection, no ordering decision, no IV management exposed. The EncryptedTokenStore wraps any other store and is strikingly short:

EncryptedTokenStore.java — authenticated encryption via NaCl SecretBox
public String create(Request request, Token token) {
  var tokenId = delegate.create(request, token);              // inner store builds the token
  return SecretBox.encrypt(encryptionKey, tokenId).toString(); // encrypt + authenticate in one step
}

public Optional<Token> read(Request request, String tokenId) {
  var box = SecretBox.fromString(tokenId);
  var original = box.decryptToString(encryptionKey);          // fails loudly if tampered
  return delegate.read(request, original);
}
Principle · one key, one purpose

Don't reuse your HMAC key for encryption. A cryptographic key should serve a single purpose; mixing uses can interact in subtle, dangerous ways. Generate a separate AES key (256-bit for SecretBox) and store it alongside the HMAC key in your keystore.

7

Encrypted JWTs (JWE)

NaCl's SecretBox is hard to beat for simplicity, but it has no standard wire format — fine when one API both issues and consumes the token, awkward when many teams and languages must interoperate. For that, JOSE offers JSON Web Encryption (JWE): a standardised encrypted token. It looks like a signed JWT but has five parts instead of three, reflecting encryption's extra moving pieces.

header encrypted key IV / nonce ciphertext auth tag
A JWE has five Base64url parts: header, encrypted key, IV/nonce, ciphertext, and authentication tag. Tap each to see its role — and why the encrypted-key slot is blank for direct encryption.

Reading the diagram

The extra two segments over a JWS are exactly what encryption adds: a way to deliver the key, and the randomness that keeps repeated messages distinct.

  1. Header — like JWS but adds enc (which encryption algorithm) alongside alg (how the key is agreed). Here alg:dir means "direct" — we already share the secret key.
  2. Encrypted key — used by fancier key-management modes to ship a one-time key. With dir there's nothing to ship, so it's empty.
  3. IV / nonce — the random value that makes identical plaintexts encrypt differently each time. It needn't be secret, but must be unpredictable.
  4. Ciphertext — the encrypted claims.
  5. Auth tag — the MAC that authenticates the ciphertext, header, and IV together.
Definition · initialization vector (IV / nonce)

An IV or nonce (number-used-once) is a unique value fed to the cipher so the ciphertext differs even when the same message is encrypted twice. It doesn't need to be secret, but it must come from a secure random source (SecureRandom). Algorithms that fold the header and IV into the auth tag without encrypting them are called AEAD (authenticated encryption with associated data).

JOSE offers two families of direct authenticated-encryption methods, and the choice has a sharp safety edge worth understanding:

AES-GCM (A128GCM…)

Fast, single-primitive AEAD. But reusing a nonce with GCM is catastrophic — it leaks the MAC key and lets an attacker forge tokens. Great when you can guarantee unique nonces.

AES-CBC + HMAC (A128CBC-HS256…)

AES-CBC encrypted, then HMAC (an EtM construction). More forgiving of nonce mistakes, which is why it's often preferred for directly-encrypted JWTs.

Never reveal why decryption failed

CBC mode historically enabled padding oracle attacks, where an attacker recovers the full plaintext just by observing different error messages when they tamper with a token. JOSE's HMAC step largely shuts this down — but defend in depth: if decryption fails for any reason, return the same generic empty response. Never tell the caller whether it was the padding, the tag, or the audience that failed. Log details only to an admin-only debug log.

Don't compress-then-encrypt

JWE can compress claims before encrypting, which sounds like a free size win. It isn't safe in general: encryption hides content but not length, and compression makes length depend on content — so an attacker who can influence part of a token may guess the rest by watching how well it compresses. This is the CRIME/BREACH family of attacks. Leave compression off unless you've thought it through carefully.

8

Using types for secure design

We've built a lovely kit of parts — JsonTokenStore, HmacTokenStore, EncryptedTokenStore — that snap together. But that very modularity is a trap: if someone comments out the encryption wrapper, or combines the parts in one of the insecure orderings from section 6, the code still compiles and runs while silently losing its protection. A kit of parts is dangerous when most combinations are insecure and only a few are safe.

Principle · make insecure code hard to write

Secure API design should make it very hard to write insecure code — not merely possible to write secure code. Developers will make mistakes, so the safe path must be the path of least resistance. Ideally, an insecure combination shouldn't even compile.

The elegant fix is to encode the security properties into the type system using marker interfaces — interfaces with no methods, existing purely to label what a store guarantees. We define ConfidentialTokenStore (keeps state secret) and AuthenticatedTokenStore (prevents tampering), then a SecureTokenStore that is simply both. The TokenController then demands a SecureTokenStore — so a store that only confidential, or only authenticated, won't type-check.

A SecureTokenStore is defined as the subtype that is both confidential and authenticated. Because the controller requires one, the compiler rejects any store that doesn't provide both properties — turning a security rule into a type error.

How each store gets labelled

Each implementation declares exactly the guarantees it actually provides, and the compiler does the rest:

  1. JsonTokenStore — insecure alone, so it stays a plain TokenStore. It can never be handed to the controller directly.
  2. SignedJwtTokenStore — authenticates but doesn't hide claims, so it's only an AuthenticatedTokenStore.
  3. EncryptedTokenStore and EncryptedJwtTokenStore — authenticated encryption gives both properties, so both are SecureTokenStores.
  4. HmacTokenStore — cleverly, it returns a SecureTokenStore when wrapping something already confidential, but only an AuthenticatedTokenStore otherwise. Two static factory methods encode this difference, so the type you get back reflects what you put in.

Beyond catching mistakes, these interfaces double as living documentation — a reviewer or auditor can read the security intent straight from the types.

9

The Achilles' heel: revocation

Stateless tokens scale beautifully — until you need to cancel one. With all the state on the client, there's no database row to delete. How do you log a user out, or kill a stolen token, when the server remembers nothing? This is the central tension of self-contained tokens, and almost every answer involves sneaking some state back onto the server.

ignore it
Don't allow revocation

Only viable for short-lived, low-sensitivity tokens. Almost no real API qualifies — all data is sensitive to someone — so this is rarely acceptable.

allowlist
Store valid token IDs

Keep a minimal database record (just a unique ID) for each issued token. Revoke by deleting it; validate by checking it's still present. Secure default — but reintroduces a per-request lookup.

blocklist
Store only revoked token IDs

The inverse: record an ID only when revoked, and reject matches. The entry need only live until the token would expire anyway, so short expiries keep the list tiny.

attribute
Block by attribute

Invalidate a set of tokens at once — e.g. "all of Mary's tokens issued before Friday noon," the standard move after a password change. Saves space, costs query complexity.

short-lived
Just expire quickly

Issue brief tokens and force frequent re-authentication. No server state, but a poorer user experience — OAuth2 refresh tokens (chapter 7) make this smoother.

Definition · replay & idle timeout

A replay attack re-sends a captured valid token; short expiries and the jti claim limit it. An idle timeout (inactivity logout) revokes a token that hasn't been used for a while — handy for logging out users who wandered off without logging out, and naturally implemented with a last-activity timestamp stored server-side.

Hybrid tokens: the best of both

The recommended default sounds paradoxical at first: wrap a stateless JWT around a database allowlist entry. It seems like the worst of both worlds — you've kept the database dependency and taken on client-side state. In practice it's the best of both, because each layer covers the other's weakness.

Reading the diagram

The trick is that each layer answers a different question: the JWT answers "is this token authentic and unexpired?" and the database answers "is it still allowed to exist?"

  1. Create. The store saves a near-empty row in the database (just enough to mint a unique ID), then embeds that ID in the JWT as the standard jti claim. The heavy attributes ride in the encrypted JWT, not the database.
  2. Validate. Decrypt and verify the JWT as usual, then look up its jti in the allowlist. Present → valid; absent → revoked, so reject. Less-sensitive operations can even skip the lookup and trust the JWT alone, saving a query.
  3. Revoke. Simply delete the jti row. The next validation finds nothing in the allowlist and rejects the token — instant, immediate logout, no waiting for expiry.

This is why it beats either approach alone: you get instant revocation (when Facebook had to kill 90 million tokens in 2018, waiting for expiry wasn't an option), forgery protection even if the database is breached (the JWT's crypto still holds), and the freedom to place each attribute where it fits — stable data in the JWT, fast-changing data like last-activity in the database.

Key takeaways

  • Token state can live on the client by encoding it as JSON — eliminating the database bottleneck — but raw encoded JSON is forgeable and must be cryptographically protected.
  • HMAC over the encoded claims prevents spoofing and tampering; the period separator works because it's not a Base64url character.
  • JWTs standardise this (header.payload.signature) with well-defined claims like sub, exp, and crucially aud, which stops cross-API replay.
  • The JOSE header is the riskiest part: never trust the alg header (remember alg:none), prefer key-driven agility, use only kid for key selection, and lean on a vetted library.
  • HMAC doesn't hide claims — on the client you often need encryption. With AES the mode is everything (the ECB penguin), and you need authenticated encryption for integrity, not just confidentiality.
  • Combine encryption and MAC as Encrypt-then-MAC; better, use a single AEAD primitive or a high-level library like NaCl's SecretBox. Use a separate key per purpose.
  • JWE gives a five-part standard encrypted token; never reveal why decryption failed (padding oracles), and avoid compress-then-encrypt (CRIME/BREACH).
  • Encode security properties as marker-interface types so insecure combinations fail to compile.
  • Stateless tokens make revocation hard; a hybrid allowlist (JWT + a jti row in the database) is the secure default, giving instant revocation plus forgery protection.

Test yourself

Pop quiz · tap to reveal
Which STRIDE threats does the HmacTokenStore protect against?
Spoofing and Tampering. HMAC stops an attacker forging a brand-new token (spoofing) or altering an existing one (tampering). It does not provide confidentiality.
Which JWT claim indicates the API server a token is intended for? (iss · sub · iat · exp · aud · jti)
aud (audience). Your API must reject any JWT whose audience doesn't list its own identifier — this prevents a token being replayed against a different API.
True or false: the JWT alg header can be safely trusted to choose the validation algorithm.
False. The header is attacker-controllable and was the basis of the alg:none attacks. Decide the algorithm from the key on your server instead.
Which STRIDE threats does authenticated encryption protect against?
Spoofing, Tampering, and Information disclosure. It includes a MAC (so spoofing + tampering, like HMAC) and hides the data (information disclosure).
What is the purpose of the initialization vector (IV)?
It ensures the ciphertext is always different even if the same message is encrypted twice. It needn't be secret, but should come from a secure random generator.
Where should the HMAC/encryption keys be stored, and how should you compare auth tags?
Keys belong in a keystore accessible only to your API servers — never in the token database, never hard-coded. Compare tags with a constant-time check like MessageDigest.isEqual.
