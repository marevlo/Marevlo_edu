1

APIs are quietly running your whole life

Open any app on your phone. Almost every one of them is constantly chatting with a remote API in the background — downloading messages, checking for notifications, uploading your photos, and doing things on your behalf.

Load a web page with developer tools open and you'll see dozens of API calls firing off to build a page perfectly tailored to you. Behind that single page, the server itself may be a swarm of microservices, each talking to the others over internal APIs. Even your fridge, your smart speaker, and your lightbulbs now phone home to APIs in the cloud — that's the Internet of Things (IoT).

Here's the catch. The very thing that makes APIs wonderful for developers — they're easy to use — also makes them an easy target for attackers. The more we depend on APIs for important things, the more damage an attacker can do by breaking one.

Real-world stakes · GDPR

Europe's General Data Protection Regulation (GDPR), live since 2018, legally forces companies to protect users' personal data. Breaches must be reported within 72 hours, and getting it wrong can cost up to €20 million or 4% of global annual turnover — whichever is bigger. Securing your API is no longer optional; it's the law.

This whole series asks one question: how do you confidently expose an API to the world without getting burned?

2

The day you took your driving test

The book opens with a deliberately silly story. Stay with it — every odd moment maps onto a real security concept you'll use forever.

The story, in five scenes

1. Bumping into Alice in the park. You instantly know it's your friend — no ID needed. A long history of trust is enough.

2. The driving examiner. A stranger. He demands your licence and checks the photo. Formal proof, issued by a trusted authority.

3. Sneaking into first class. A ticket inspector blocks you. Your standard ticket only grants standard access.

4. Lending Bob the greenhouse key. Bob can water the plants — but that key doesn't open your house or your whisky cabinet.

5. The surveillance footage. Bob denies the party until you show the cameras. Now he can't wriggle out of it.

Each scene is a security mechanism in disguise:

①
Identification

Knowing who someone is — by history (Alice) or formal proof (the licence).

②
Authentication

The licence is trusted because a trusted body issued it and the photo matches you.

③
Authorization

Your ticket / key decides what you may do — first class? the whisky? Access is scoped.

④
Capabilities

The greenhouse key grants narrow power — lose it and only the greenhouse is at risk.

⑤
Audit logging

The cameras record who did what, when — proof that survives Bob's denial.

⑥
Levels of assurance

A casual hello needs little proof; a high-stakes action needs stronger ID. Match effort to risk.

Definition · Audit log

An audit log records the details of significant actions taken on a system, so you can later work out who did what, and when. It's the crucial evidence when investigating a breach.

3

So… what is an API, really?

An API (Application Programming Interface) is a boundary between one part of a software system and another. It defines the set of operations one component offers for others to use.

Think of a photography archive. It might expose operations like "list albums," "view a photo," and "add a comment." Then completely different programs can build on top of it: an online gallery uses it to display photos, while a word processor uses the same API to embed a photo into a document. One API, many clients.

API vs UI — what's the difference? Both are boundaries to a system. But a UI is designed for a human to interact with directly — rich, pretty, pleasant to read. An API is designed for other software to interact with — a stripped-back, highly regular view of raw data that a program can easily parse.

This is the single most important picture in the chapter, because it shows where an API lives: in the middle, as a gatekeeper between the outside world and your valuable data.

  1. Users are real people. They never touch your API directly — they touch a client.
  2. Clients are programs acting for the user: a browser, a phone app, an IoT gadget, or another API with no screen at all. The client turns a human intention ("show my photos") into a structured request.
  3. Your API receives that request, runs its business logic (the rules that decide what's allowed and what to do), and may call one or more backend systems — databases, processing engines, or further APIs — to get the job done.
  4. Finally a response travels back out the same path to the client, which renders it for the user.

Every security mechanism in this chapter is really about controlling what happens at that middle box — because that's the one boundary every request must cross.

4

Four flavours of API

There's no single "right" way to expose an API. Each style trades off efficiency, flexibility, and how tightly the client is coupled to the server.

RPC — Remote Procedure Call

Call remote functions as if they were local. Compact, fast binary messages, but clients need special "stub" libraries. Modern example: Google's gRPC. Older: SOAP (XML).

RMI — Remote Method Invocation

An object-oriented twist on RPC — call methods on remote objects. Once huge (CORBA, EJB), now in decline because the frameworks got too complex.

REST — REpresentational State Transfer

Roy Fielding's principles behind the web. Few generic operations, standard formats, hyperlinks to navigate. Loosely coupled, so clients survive as the API evolves.

Query-style — GraphQL / SQL

Few operations, but a rich query language lets the client decide exactly what data comes back. Great for filtering large data sets. Example: Facebook's GraphQL.

Example · Picking a style

A company running microservices it fully controls might pick the fast, efficient RPC style — it can push new stub libraries to every service. A public API used by strangers is better off with REST + JSON, maximising compatibility with every kind of client.

This series — like the book — focuses on APIs over HTTP using a loosely RESTful approach, because that's the dominant style today. But most of the advice transfers to any style, right down to designing a plain library.

Security sits where three worlds overlap

API security isn't one discipline — it's the intersection of three. You don't need to master all of each; you take the most critical pieces and blend them.

Interactive · Figure 1.2
API security = the centre of three fields

Tap a circle to see what API security borrows from it. The dark centre is where all three overlap — that's our subject.

API security borrows the most critical pieces from three established fields. Tap any circle to see exactly what we take from it.

Reading the diagram

Three big fields each have entire bookshelves devoted to them. The good news: you only need the slice of each that touches APIs — the small dark zone in the middle.

  1. Network security (left) — protecting data as it travels and stopping unauthorised access to the network. From it we take firewalls, load balancers, reverse proxies, and HTTPS / TLS.
  2. Application security (right) — building software that withstands attack. From it we take secure coding, common-vulnerability awareness, and safe credential handling.
  3. Information security (bottom) — protecting data across its whole life. From it we take access control and applied cryptography.
Two definitions to bank

Cryptography — the science of protecting information so two or more parties can communicate without anyone else reading or tampering with the messages (also great for data on disk).

HTTPS — HTTP running over a secure connection. Plain HTTP is visible to anyone watching the wire; HTTPS hides and protects it using TLS (formerly SSL).

Reading the diagram

Read it like water flowing downhill — a request enters at the top from the internet and sinks down through each protective layer before it ever reaches your code.

  1. Firewall (first dashed line) — works at the low TCP/IP level. If your API serves ports 80 and 443, the firewall blocks every other port outright.
  2. Load balancer — routes each request to a free server so one machine isn't buried while others sit idle.
  3. Reverse proxies — do the heavy, expensive work: terminating TLS encryption and checking credentials, so the API servers don't have to.
  4. API servers (second dashed line) — finally, your actual code runs here. Everything above existed just to deliver a clean, safe request to this point.
  5. Databases — the data store, usually guarded by even more layers.

The dashed lines are trust boundaries: everything inside one is managed by the same owner. As you'll see in the threat-modelling section, boundaries are exactly where attacks tend to happen.

Definition · SSL termination

SSL termination (or offloading) is when a load balancer or reverse proxy handles the client's TLS connection for the API server. It then opens a separate connection to the backend — either plain HTTP, or re-encrypted as a fresh TLS connection (SSL re-encryption).

The specialist services you'll meet

API Gateway

A specialized reverse proxy that makes many APIs look like one. Common in microservices; often handles auth and rate-limiting too.

Web App Firewall (WAF)

Inspects traffic at a higher level than a basic firewall — detects and blocks common attacks against HTTP services.

IDS / IPS

Intrusion detection / prevention systems watch internal traffic for suspicious patterns and either alert or actively block.

Important nuance

These boxes overlap a lot in practice — a load balancer may terminate TLS, a reverse proxy may act as a gateway. But beware: poor security in your API can undermine even the fanciest gateway, and a badly configured gateway can add new risks. Knowing the basics lets you judge whether a product actually fits your needs.

7

If an API only offers safe operations, why secure it?

A fair question — surely you just leave out the operations you don't want? Three reasons that isn't enough:

1
Different users, different authority

The same API serves admins, normal users, and total strangers on the internet. Without access control, anyone can do anything.

2
Safe operations, unsafe combos

Each call may be fine alone, but combinations can break the rules. (See the banking example below.)

3
Implementation bugs

E.g. not checking input size lets an attacker send a giant payload that eats all memory — a DoS attack.

Example · The banking trap

Imagine a banking API with separate withdraw and deposit operations, each carefully checking limits. Looks fine — but deposit has no way to know the money being deposited came from a real account. A clever attacker deposits money that never existed. The fix is a single transfer operation that moves money atomically, so the total always balances. Security must be judged across the whole API, not one operation at a time.

Definition · Denial of Service (DoS)

A DoS attack happens when an attacker stops legitimate users from accessing a service — often by flooding it with traffic, but also by cutting connections or exploiting bugs to crash the server.

There's no such thing as a perfectly secure system — and no single definition of "security." It always depends on context.

For a healthcare provider, letting you discover whether your friends have accounts is a serious privacy violation. For a social network, that exact feature is the whole point. Same capability, opposite verdict. Designing a secure API means weighing four things: the assets to protect, the goals that matter, the mechanisms available, and the environment & threats.

Assets what you protect

Mostly information — customer names, addresses, card numbers, database contents. Sensitive data like sexual orientation or political affiliation absolutely counts. There are physical assets too (servers, devices). The simple test: if anyone would suffer real or perceived harm should a part of the system be compromised, that part is an asset. Harm can be direct (lost money) or abstract (lost reputation when it emerges you didn't hash passwords properly).

8

Security goals & the CIA triad

Security goals define what "secure" actually means for your assets. The three most famous — true for almost every system — form the CIA Triad.

C
Confidentiality

Information can only be read by its intended audience.

I
Integrity

No unauthorized creation, modification, or destruction of data.

A
Availability

Legitimate users can access the API when they need it.

Other goals matter in context too — accountability (who did what) and non-repudiation (you can't deny having done it). Security goals are best thought of as non-functional requirements (NFRs), sitting alongside performance and reliability. Like those, they're hard to pin down — it's tough to prove a negative ("confidentiality is never violated") or to define "good enough."

Making goals precise — the cryptography game

Cryptography uses a clever trick: turn the goal into a game between an attacker and the system. The classic confidentiality game is indistinguishability.

In this game the attacker hands over two equal-length messages. The system flips a coin, encrypts one of them, and returns the ciphertext. The encryption is secure if the attacker can't guess which message it was with better than 50:50 odds. Play a round to see it in action.

Reading the diagram

This is how cryptographers turn a fuzzy idea ("the message is confidential") into something you can actually measure.

  1. The attacker chooses two messages of equal length, A and B, and hands them to the system.
  2. The system makes a secret, random choice — a coin flip — and encrypts just one of them with its key.
  3. It returns the resulting ciphertext to the attacker, who must guess: was that A or B?
  4. If no realistic attacker can do meaningfully better than a coin flip (50:50), the scheme is declared secure for confidentiality. Even seeing the encrypted output tells them nothing.

Most goals can't be made that precise. The everyday approach is to refine abstract goals into testable constraints. For a messaging API: the function is "users can read their messages." To preserve confidentiality, add constraints — "users can read only their own messages" and "a user must be logged in first." Now you can write concrete tests:

Testable constraints

① Create two users with dummy messages. ② Check user 1 cannot read user 2's messages. ③ Check a logged-out user can read nothing.

Follow the arrows clockwise. The crucial insight is the loop-back: finishing one lap doesn't end the work — it reveals new assets and restarts the cycle.

  1. Identify assets — list what's worth protecting (data, resources, devices).
  2. Define security goals — decide what "secure" means for each asset (confidentiality? integrity?).
  3. Refine into constraints — turn each goal into concrete, testable rules.
  4. Develop & test — build and verify those rules. This often creates new assets (e.g. a login system hands out session cookies — now those cookies need protecting), so you loop back to step 1.
9

Threats, threat models & STRIDE

A threat is any way a security goal could be violated for one of your assets. You can't defend against everything, so you decide which threats are worth worrying about — your threat model.

Definitions

Threat — an event or set of circumstances that defeats your API's security goals (e.g. an attacker stealing names from your customer DB threatens confidentiality).

Threat modeling — systematically identifying threats so they can be recorded, tracked, and mitigated.

Example · Right-sizing the threat

An API recording lap times for a local cycling club doesn't need to fear a nation-state spy agency. But it probably does need to stop riders quietly "improving" their own times or sabotaging rivals'. Realistic threats focus your effort where it matters.

As Eisenhower put it: "Plans are worthless, but planning is everything."

It matters far less how you do threat modeling than that you do it — the act of thinking through weaknesses almost always improves the API. The process: draw the components → mark trust boundaries → draw data flows → hunt for threats (especially on flows crossing boundaries) → record them. Steps 1–3 produce a dataflow diagram.

STRIDE — six classes of threat

Most attacks fall into a handful of categories. The popular STRIDE methodology names six, and each maps to a general defensive mechanism (e.g. spoofing → require authentication).

S
SpoofingPretending to be somebody else. → Defend with authentication.
T
TamperingAltering data, messages, or settings you shouldn't. → Integrity protections.
R
RepudiationDenying you did something you really did. → Audit logging.
I
Information disclosureRevealing data that should stay private. → Encryption & access control.
D
Denial of serviceStopping others reaching the service. → Rate-limiting & quotas.
E
Elevation of privilegeGaining access you're not supposed to have. → Strong authorization.

Reading the diagram

Read strictly left to right. The request from the browser or app travels along the encrypted HTTPS link and then has to clear four gates in a fixed order before the fifth box — your actual code — gets to run.

  1. Rate-limiting comes first and is deliberately cheap: if the API is being flooded, drop excess requests immediately, before wasting effort on them.
  2. Authentication establishes who is calling. Survive this and the request carries a verified identity.
  3. Audit log records the attempt — who, what, when — whether or not it ultimately succeeds.
  4. Access control is the final gate: given who you are, are you allowed to do this specific thing? If not, you're rejected here.
  5. Application logic — only requests that clear all four filters reach your real business logic. The whole conversation, in both directions, stays wrapped in HTTPS encryption.

Each gate can be built by you or handed to an external API gateway — but the order matters: there's no point authenticating a request you're about to drop for flooding, and no point checking permissions before you know who's asking.

① Encryption

Protects data outside your API. Two cases: in transit (over networks — use TLS) and at rest (on disk — protects against anyone with storage access). Modern encryption also stops tampering, not just snooping.

② Identification & Authentication

Authentication verifies a user is who they claim to be. Strictly, we never authenticate the user — we authenticate a claim about their identity (like a username), usually by asking for credentials only they should have (the word comes from giving credence to the claim). Remember Alice (history = no formal ID) vs the examiner (stranger = show the licence).

The three authentication factors

Something you know — a password. · Something you have — a key or device. · Something you are — biometrics (fingerprint, iris).

Combine two different factors and you get 2FA / MFA, so compromising one isn't enough. Note: two passwords is still one factor (both "something you know"); a password + a phone code is genuine 2FA.

Real use case · WebAuthn keys

A hardware security key (the WebAuthn standard) that plugs into your USB port is a "something you have" factor — a small device you keep on your key ring. Pair it with your password and you've got strong 2FA resistant to phishing.

③ Access Control / Authorization

Decides who can do what, to preserve confidentiality and integrity. Two main approaches:

Identity-based

First identify the user, then decide based on who they are. You can request any resource but may be denied by the rules. (Like the celebrity waltzing into the club.)

Capability-based

A token/key (a capability) names a resource and the permissions on it. You simply can't reach what you hold no capability for. (Like Bob's greenhouse key.)

Example · The no-access-control API

A wiki (think Wikipedia, invented by Ward Cunningham) famously has almost no access control — anyone can edit any page. Instead it relies on version control to undo bad edits, an audit log for accountability, and social norms. Even so, big wikis lock hot articles to stop "edit wars" and vandalism. Sometimes the best access control is a great audit trail.

④ Audit Logging

A record of every operation, for accountability. Used both after a breach (forensics) and in real time (spotting attacks in progress). A good log answers: who did it and with what client? when? what kind of request? on what resource? did it succeed — and if not, why? what else did they do nearby? Logs must be tamper-proof and often hold PII, so they need protecting too.

Definition · PII

Personally Identifiable Information — any data relating to an individual that helps identify them (name, address, date & place of birth). Laws like GDPR strictly control how PII is stored and used.

⑤ Rate-Limiting

Preserves availability against DoS attacks, which work by exhausting a finite resource (CPU, memory, connections). The key insight: spot when a client — or group — is using more than their fair share, and cap it. Before login you apply broad rate-limiting (per IP, per range); after login you can enforce per-user quotas.

Definitions · Quota & Throttling

Quota — a limit on resources one account may consume (e.g. five messages per day).

Throttling — slowing a client's requests without fully disconnecting them: queue requests for later, or reply "please slow down." Keep ignoring it and later requests get rejected.

The golden rule of rate-limiting

It must cost less to reject a request than to process it — otherwise the defence becomes part of the attack. That's why it usually runs in highly optimized load balancers, reverse proxies, or gateways out in front of your API. Some companies even sell DoS protection as a service, absorbing floods with global infrastructure.

Key takeaways

  • An API is a boundary built for software to use; it handles requests from clients (browsers, apps, IoT, other APIs) on behalf of users.
  • API security is the blend of information, network, and application security — you take the critical bits of each.
  • Define security through the CIA triad: Confidentiality, Integrity, Availability (plus accountability & non-repudiation when relevant).
  • Security is a loop: identify assets → define goals → refine into testable constraints → develop & test → repeat.
  • Model threats with STRIDE; pay closest attention to data flows that cross trust boundaries.
  • Five mechanisms layer as filters before your logic runs: Encryption, Authentication, Access control, Audit logging, Rate-limiting.
  • An API gateway is a specialized reverse proxy — useful, but it never excuses insecure API code.
