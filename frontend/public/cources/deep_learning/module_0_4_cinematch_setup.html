The CineMatch project — set up like a professional.

Theory's done. Now we build the codebase you'll work in for the rest of the track. Project structure, environment, synthetic dataset, version control — the unglamorous scaffolding that separates engineers who ship from people with a notebook full of dead experiments.

01 — Why structure
Why a notebook is not enough.

Most people who learn deep learning learn it in Jupyter notebooks. Notebooks are wonderful for exploration — you can see a tensor, plot it, modify a cell, re-run. They are also, for any project larger than a one-off experiment, a productivity trap.

The pattern repeats: a notebook starts clean, accumulates 200 cells over a week, becomes impossible to follow, gets copied into final_v3.ipynb, then final_v3_actually.ipynb, and a month later nobody remembers which version produced the model that's serving in production. We are not going to do that.

The Marevlo discipline

Notebooks are for exploration and presentation. Python files are for everything you'll run more than once. The dividing line: if a piece of code defines a reusable function, a model class, a dataset, or a training loop — it lives in a .py file. If it's a one-off chart or a sanity check, it can stay in a notebook.

Every module in this course has a paired notebook for type-along learning. But the real CineMatch codebase — the thing you'll deploy at the end — is a structured Python project. We start it now, and grow it module by module.

02 — Environment
Environment — a clean room to build in.

Three rules that will save you hours of debugging later:

  1. One project, one virtual environment. Never install project dependencies into your system Python.
  2. Pin your versions. A model trained against PyTorch 2.1 may behave differently against 2.4. Lock what works.
  3. Reproducibility is a feature. Anyone with your repo and your requirements.txt should get a working setup in one command.

Three categories that should never enter your git repo: secrets (API keys, credentials), generated artifacts (caches, compiled files, virtual envs), and large data (raw datasets, trained model weights — those belong in S3 or similar, with paths in your config).

04 — Synthetic data
Generating the CineMatch synthetic dataset.

We could use a public ratings dataset. We won't, for two reasons. First, public datasets come with their own quirks and license issues. Second — and more importantly — generating synthetic data ourselves means we know the ground truth. When we train a model in Phase 1 to predict watch-time, we'll know exactly which feature drives the prediction, because we wrote the function. That visibility is invaluable for learning.

The CineMatch synthetic universe

What we are simulating.

Five thousand fictional users with personality profiles (genre preferences, attention span, novelty appetite). A thousand fictional movies with attributes (genre, runtime, release year, critical rating). A function that, given a user and a movie, produces a probabilistic watch-time. We sample interactions from this function with realistic noise — and that becomes our training data.

The clean separation: the generator is the ground truth we'll never show the model. The samples are what the model sees. Our goal will be to train models that recover the generator's behavior just from the samples — exactly like real ML, where the true function exists but is hidden.

Step 1 — The generator script.

Save this as scripts/generate_synthetic.py:

"""
Generate the CineMatch synthetic ratings dataset.
Run: python scripts/generate_synthetic.py
"""
import numpy as np
import pandas as pd
from pathlib import Path

RNG = np.random.default_rng(seed=42)
N_USERS = 5000
N_MOVIES = 1000
N_INTERACTIONS = 200_000
GENRES = ["action", "drama", "comedy", "thriller", "romance", "scifi"]


def make_users(n):
    """Each user has a 6-dim genre preference vector and an attention span."""
    return pd.DataFrame({
        "user_id": np.arange(n),
        "age": RNG.integers(16, 75, n),
        "attention_span_min": RNG.normal(90, 25, n).clip(15, 200),
        **{f"pref_{g}": RNG.beta(2, 5, n) for g in GENRES},
    })


def make_movies(n):
    """Each movie has one primary genre, a runtime, and a critic score."""
    return pd.DataFrame({
        "movie_id": np.arange(n),
        "genre": RNG.choice(GENRES, n),
        "runtime_min": RNG.normal(110, 25, n).clip(70, 200).round(),
        "critic_score": RNG.beta(5, 3, n),
        "release_year": RNG.integers(1970, 2025, n),
    })


def simulate_watch_time(user, movie):
    """
    Ground truth: how long the user watches before stopping.
    Driven by genre alignment, runtime vs attention span, and critic score.
    Returns a fraction of runtime watched in [0, 1].
    """
    affinity = user[f"pref_{movie['genre']}"]
    runtime_fit = 1.0 - abs(movie["runtime_min"] - user["attention_span_min"]) / 200
    quality = movie["critic_score"]
    score = 0.5 * affinity + 0.3 * runtime_fit + 0.2 * quality
    noise = RNG.normal(0, 0.1)
    return np.clip(score + noise, 0, 1)


def main():
    out_dir = Path("data/raw")
    out_dir.mkdir(parents=True, exist_ok=True)

    users = make_users(N_USERS)
    movies = make_movies(N_MOVIES)

    # Sample interactions
    user_idx = RNG.integers(0, N_USERS, N_INTERACTIONS)
    movie_idx = RNG.integers(0, N_MOVIES, N_INTERACTIONS)
    rows = []
    for u, m in zip(user_idx, movie_idx):
        watch_frac = simulate_watch_time(users.iloc[u], movies.iloc[m])
        rows.append({
            "user_id": int(u),
            "movie_id": int(m),
            "watch_fraction": float(watch_frac),
            "finished": int(watch_frac >= 0.85),
        })
    interactions = pd.DataFrame(rows)

    users.to_parquet(out_dir / "users.parquet")
    movies.to_parquet(out_dir / "movies.parquet")
    interactions.to_parquet(out_dir / "interactions.parquet")
    print(f"Wrote {len(users)} users, {len(movies)} movies, {len(interactions)} interactions")


if __name__ == "__main__":
    main()

Run it:

python scripts/generate_synthetic.py
# Wrote 5000 users, 1000 movies, 200000 interactions

Step 2 — A PyTorch Dataset wrapper.

Save this as cinematch/data/dataset.py:

import pandas as pd
import torch
from torch.utils.data import Dataset


class CineMatchDataset(Dataset):
    """
    Loads merged user-movie-interaction rows and exposes them as tensors.
    Phase 1 will add proper feature engineering; for now we expose raw values.
    """
    def __init__(self, data_dir="data/raw"):
        users = pd.read_parquet(f"{data_dir}/users.parquet")
        movies = pd.read_parquet(f"{data_dir}/movies.parquet")
        inter = pd.read_parquet(f"{data_dir}/interactions.parquet")
        self.df = inter.merge(users, on="user_id").merge(movies, on="movie_id")
        self.feature_cols = ["age", "attention_span_min", "runtime_min", "critic_score"]

    def __len__(self):
        return len(self.df)

    def __getitem__(self, idx):
        row = self.df.iloc[idx]
        x = torch.tensor(row[self.feature_cols].values.astype("float32"))
        y = torch.tensor(row["watch_fraction"], dtype=torch.float32)
        return x, y

05 — Seed, config, ready
Seeding, configuration, and the ready-state.

Two more files and Phase 0 is complete. Both are tiny, both will save you hours.

cinematch/utils/seed.py — global determinism.

import random
import numpy as np
import torch


def set_seed(seed: int = 42) -> None:
    """Seed every source of randomness we use, for reproducible runs."""
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    if torch.cuda.is_available():
        torch.cuda.manual_seed_all(seed)
    # Required for full determinism on CUDA — costs ~10% speed
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False

cinematch/utils/device.py — pick the best device.

import torch


def get_device() -> torch.device:
    """Return CUDA if available, else MPS (Apple Silicon), else CPU."""
    if torch.cuda.is_available():
        return torch.device("cuda")
    if torch.backends.mps.is_available():
        return torch.device("mps")
    return torch.device("cpu")

Why a config file

Hyperparameters scattered across notebooks become impossible to manage. With a single config module, "what learning rate did that experiment use?" becomes trivial to answer — you just check the config at that git commit. From Phase 2 onward we'll replace this with proper experiment-tracking via TensorBoard, but the principle stays: centralize what varies, decentralize what doesn't.
