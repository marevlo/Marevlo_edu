Calculus, autograd, and the small math of learning.

A tensor by itself is just a pile of numbers. What turns it into a learning system is the ability to ask: "if I nudge this number, how does the loss change?" That question has a name — the gradient — and PyTorch will compute it for you, automatically, for any computation you can write.

01 — Why calculus
Why calculus is the engine of learning.

Before we open PyTorch, we need to understand what training a neural network actually is. At the highest level, it is exactly four steps, repeated millions of times:

The training loop, in one paragraph

Predict, measure, blame, adjust.

Show CineMatch a user and a movie. Predict a watch-time. Measure how wrong the prediction was — that number is the loss. Blame each of the model's parameters in proportion to how much it contributed to the error — those numbers are the gradients. Adjust each parameter slightly in the direction that would have reduced the loss. Repeat with the next user-movie pair. Twenty million times.

The third step — assigning blame — is the entire mathematical content of training. A model with a million parameters needs to know, for each one of those parameters individually, "if I had been a tiny bit larger, would the loss have gone up or down, and by how much?"

That question is what a derivative answers. The collection of all those derivatives, one per parameter, is what we call the gradient. And the algorithm that computes the gradient efficiently — without us having to derive it by hand — is automatic differentiation, or autograd for short.

The single most important fact in deep learning

Training is an optimization problem. The thing being optimized is the loss. The lever you have to optimize it is the parameters. The compass that tells you which way to turn that lever is the gradient. Everything else — every architecture, every trick — is in service of computing that compass faster, more stably, or more cleverly.

02 — Derivatives
Derivatives, intuitively.

The derivative of a function f(x) at a point x is, plainly, the slope of the function at that point. If f measures the loss and x is a parameter, the derivative says: "if you nudge x up by a tiny amount, the loss changes by approximately this much."

Formally, the derivative is defined as a limit:

f′(x) = limh→0 f(x + h) − f(x) h (2.1)

Read this slowly. We pick a tiny step h. We see how much f changes when x moves by that step. We divide the change by h to get a slope. Then we let h shrink to zero — what slope does it converge to?

Let's verify this on a small example. Take f(x) = 3x² − 4x and try to find the derivative at x = 1 by plugging in shrinking values of h:

def f(x): return 3*x**2 - 4*x

for h in [0.1, 0.01, 0.001, 0.0001]:
    slope = (f(1 + h) - f(1)) / h
    print(f"h={h}, slope ≈ {slope:.4f}")

# h=0.1,    slope ≈ 2.3000
# h=0.01,   slope ≈ 2.0300
# h=0.001,  slope ≈ 2.0030
# h=0.0001, slope ≈ 2.0003

The slope is converging to 2. We can verify this with the analytical rule (power rule): f′(x) = 6x − 4, so f′(1) = 2. Same answer. The numerical limit and the analytical formula agree, which is what we'd hope.

Partial derivatives — when you have many parameters.

A neural network is a function of millions of parameters, not one. So when we ask "how does the loss change when I nudge parameter w17?" we are holding all the other parameters fixed and only varying that one. That is what a partial derivative means.

∂L ∂wi = "how much does the loss change if I nudge only wi" (2.2)

The collection of all the partial derivatives — stacked into a vector — is what we call the gradient:

∇L = [ ∂L∂w1, ∂L∂w2, ..., ∂L∂wn ] (2.3)

The gradient has a beautiful geometric meaning: it points in the direction of steepest ascent. To minimize the loss, we step in the opposite direction — −∇L. That single line is the heart of gradient descent, and we'll see it in code in Phase 1.

03 — Chain rule
The chain rule — backprop's only trick.

Neural networks are deep precisely because they are compositions — function applied to function applied to function. A 10-layer network is just f10(f9(...f1(x)...)). And the chain rule is the calculus trick that lets us differentiate compositions.

The single-variable form: if y = f(u) and u = g(x), then

dydx = dydu × dudx (3.1)

Read it as: "to find how y changes per unit of x, multiply how y changes per unit of u by how u changes per unit of x." It's just compounding rates. If you ride a bike at 20 km/h and burn 30 calories/km, you burn 600 calories/h — that's the chain rule.

Why this is everything in deep learning.

Imagine a CineMatch network with three layers. The output prediction is ŷ = f3(f2(f1(x))), and the loss L is some function of the prediction. We want to know how a weight buried in the first layer affects the final loss. The chain rule says:

∂L∂w1 = ∂L∂ŷ · ∂ŷ∂f2 · ∂f2∂f1 · ∂f1∂w1 (3.2)

That is — literally — what backpropagation does. It applies the chain rule layer by layer, starting from the loss at the output and working backwards toward the inputs. At each layer it asks, "given the gradient flowing in from the layer above, what's the gradient flowing out to the layer below?" The math is mechanical. The cleverness is in doing it efficiently, which is what autograd is.

Worked example — chain rule through a 3-function composition
f₁(x) = 2x + 1, f₂(u) = u², f₃(v) = −log(v). Composed: L = f₃(f₂(f₁(x)))This is a toy model: linear → square → negative log — similar shape to a single-neuron BCE loss
Forward pass at x = 1: u = f₁(1) = 3, v = f₂(3) = 9, L = f₃(9) = −log(9) = −2.197
Backward pass — compute each local derivative:∂f₃/∂v = −1/v = −1/9 = −0.111. ∂f₂/∂u = 2u = 6. ∂f₁/∂x = 2.
Chain rule: ∂L/∂x = (∂f₃/∂v) · (∂f₂/∂u) · (∂f₁/∂x) = (−0.111) × 6 × 2 = −1.333This is exactly what autograd computes — but for millions of operations, automatically.
Verify numerically: L(1.001) = −log((2·1.001+1)²) = −log(9.012) = −2.198. ΔL/Δx = (−2.198 − (−2.197))/0.001 = −1.33 ✓Finite difference matches our chain-rule answer. The chain rule works!
Figure 1. Forward pass computes the loss left-to-right. Backward pass computes gradients right-to-left, multiplying chain-rule terms along the way.

The reason deep networks were considered "untrainable" in the 1990s is that those chain-rule products can shrink to zero (vanishing gradients) or blow up to infinity (exploding gradients) when chained through many layers. Modern architectures — ReLU activations, residual connections, batch normalization — exist almost entirely to keep this product behaving well. We'll meet each fix in Phase 2.

04 — Autograd
Autograd — calculus that does itself.

Here is the good news, the news that made deep learning practical for non-mathematicians: you do not have to compute any of these derivatives by hand. PyTorch tracks every operation you perform on a tensor and, when you ask, applies the chain rule automatically — no matter how complicated the computation.

Three pieces of vocabulary, then we'll see it work:

Three concepts you must know cold

Computational graph — the silent record PyTorch keeps of every operation. Nodes are operations, edges are tensors. Forward pass — running the computation to get a result, while the graph builds itself. Backward pass — walking the graph in reverse to compute every gradient via the chain rule. Call .backward() on a scalar and the magic happens.

The minimum example.

import torch

# Tell PyTorch we want gradients with respect to x
x = torch.tensor(2.0, requires_grad=True)

# Define a function — PyTorch silently records each step
y = 3 * x**2 - 4 * x          # y = 3x² − 4x

# Compute the gradient dy/dx by walking backward through the graph
y.backward()

# The gradient is now stored in x.grad
print(x.grad)                 # tensor(8.)
# Check: dy/dx = 6x − 4, so at x=2 the answer is 8 ✓

Three lines of code replaced what would have been a hand-derived derivative. Now scale this up: instead of one variable x, imagine a million parameters. The pattern is identical. Set requires_grad=True on your parameters, run the forward pass, call .backward() on the loss, and every parameter has its gradient sitting in its .grad attribute, ready for the next step of optimization.

The vector case — gradients of many things at once.

# A vector input — typical of a parameter tensor
x = torch.arange(4.0, requires_grad=True)
# x = tensor([0., 1., 2., 3.])

y = 2 * torch.dot(x, x)        # y = 2(x₁² + x₂² + x₃² + x₄²)
y.backward()

print(x.grad)                  # tensor([ 0.,  4.,  8., 12.])
# Check: dy/dx_i = 4x_i, so we get [0, 4, 8, 12] ✓

The output y here is a scalar — a single number. PyTorch requires this for .backward() because the gradient of a vector with respect to a vector is a matrix (the Jacobian), and computing the full Jacobian for million-parameter networks is too expensive. The convention: collapse to a scalar (typically by summing or taking the mean, which is what loss functions do anyway), then call backward.

Detaching — when you want to break the chain.

Sometimes you want a tensor's value but don't want gradients to flow through it. The classic case: in reinforcement learning or in target-network setups, you want to use a tensor as a constant target, not as a parameter to be optimized.

x = torch.tensor(3.0, requires_grad=True)
y = x ** 2
u = y.detach()             # u has the value 9.0 but no graph history
z = u * x                  # PyTorch sees z as u (a constant) times x
z.backward()

print(x.grad)             # tensor(9.) — treats u as constant, so dz/dx = u = 9
# Without detach: z = x³, dz/dx = 3x² = 27 — different answer!
A subtle gotcha

Gradients accumulate by default.

Each call to .backward() adds to the existing .grad — it does not overwrite. This is useful for some advanced patterns (RNNs, gradient accumulation across batches) but it means: in a normal training loop, you must zero the gradients before each new backward pass. Forget this and your gradients will be the sum of every batch you've seen — wildly wrong. The line is optimizer.zero_grad() at the top of every loop iteration.

05 — Probability
Probability — reasoning under uncertainty.

Calculus tells us how to optimize. Probability tells us what to optimize. Almost every loss function in deep learning has a probabilistic interpretation, and almost every prediction can be expressed as "how likely is this outcome." A practitioner who skips probability ends up using the right tools for the wrong reasons.

You do not need a graduate-level treatment to do deep learning well. You need four ideas, fluently.

1. A probability is a number in [0, 1] that sums to 1 across all outcomes.

If a CineMatch user can either finish a movie or not, those two probabilities must sum to 1. If they could pick one of five tour types, the five probabilities must sum to 1. This is why we'll use the softmax function in classification — it takes any vector of real numbers and squashes it into a valid probability distribution.

2. Conditional probability — when one thing depends on another.

Notation: P(A | B) reads "the probability of A, given that B happened." This is the bedrock of recommendation. The whole CineMatch question is: P(user finishes movie | user's history, movie metadata, time of day).

P(A | B) = P(A, B) P(B) (5.1)

3. Bayes' rule — flipping a conditional.

Often we know P(B | A) but want P(A | B). Bayes' rule converts between them:

P(A | B) = P(B | A) P(A) P(B) (5.2)

The famous medical-test example: a test is 99% accurate, you test positive, what's the probability you actually have the disease? If the disease is rare (say 0.1% of the population), the answer is not 99% — it's around 9%. Most positive results are false positives because most people don't have the disease. This intuition matters in DL too: a 99%-confident model trained on a class that's 0.1% of the data is almost certainly overfit to noise.

4. Expectation — the average of what you'd see.

𝔼[X] = Σx x · P(X = x) (5.3)

Read it as a weighted average: each possible value times the probability of seeing that value, summed up. Almost every loss in deep learning is an expectation — the expected error over the data distribution. We can't compute it exactly (we don't know the true distribution), so we approximate it with the average over our training set. That approximation is called the empirical risk, and minimizing it is called empirical risk minimization — which is what training a neural network literally is.

Probability in CineMatch

Why every loss in this course is an expectation.

When we say "minimize the loss on the training data," we mean: minimize the average per-example loss across all training examples. That average is our best estimate of the expected loss across all possible users and movies in the world — examples we have not seen and never will. Training is the act of using a finite, observed sample to estimate and minimize an infinite, unobserved expectation. This framing is why concepts like overfitting, generalization, and validation sets exist at all.

06 — Synthesis
Putting it all together — and the checkpoint.

Let's tie the threads. A neural network is a function from inputs to outputs. The output is compared to a true target via a loss function — almost always derived from a probabilistic argument (cross-entropy from log-likelihood, mean squared error from Gaussian noise). The loss is a scalar. PyTorch's autograd traces every operation that produced that scalar, then walks backward through the computational graph applying the chain rule, depositing one gradient per parameter into each parameter's .grad attribute. An optimizer (which we'll build in Phase 1) uses those gradients to nudge each parameter in the direction that would have reduced the loss.

That paragraph is, more or less, the entirety of deep learning. Everything else we will study in this course — convolutions, attention, generative models — is a different choice of function, but the calculus, autograd, and probability are unchanged.
