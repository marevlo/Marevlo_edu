LSTM & GRU. Gating saves the RNN.

Plain RNNs have a beautiful idea — one cell, applied repeatedly — crippled by vanishing gradients. The fix, published by Hochreiter and Schmidhuber in 1997 and not fully appreciated for fifteen years, is gating: small learned networks at each step that decide what to remember, what to forget, and what to expose. The resulting architectures — LSTM and its simpler cousin GRU — were the workhorse of sequence modeling from 2014 until transformers replaced them for large-scale language.

01 — The gating idea
The gating idea.

Go back to where the plain RNN failed. The issue was the one-step Jacobian ∂hs / ∂hs−1 = diag(1 − hs2) · Whh. Multiply many of these together for long sequences and gradients vanish (or explode). The state update was purely multiplicative — each new state is a function of the weighted previous state — which is exactly what causes the exponential decay.

The insight behind LSTM: what if part of the state were updated additively — where ht = ht−1 + (something small) — so the Jacobian has an identity contribution, just like ResNet's skip connections had for CNNs? Then gradients flowing back through the state wouldn't have to multiply by a potentially-shrinking matrix each step.

The gating principle in one sentence

Maintain a long-term memory that updates mostly by addition (cheap gradient flow through many steps), controlled by small learned gates that decide — at every timestep and every dimension — what to add, what to forget, and what to expose. The gates are the learnable part; the addition is what keeps gradients alive.

That principle generates a family of architectures. We'll cover the two that matter: the full LSTM (three gates, separate cell state) and the simpler GRU (two gates, no separate cell state).

02 — The LSTM cell
The LSTM cell.

An LSTM cell maintains two things at each timestep, not just one. Alongside the hidden state ht there's a cell state ct. The hidden state is what the network outputs at each step; the cell state is the long-term memory that carries information across many steps.

Three learned gates — each a small sigmoid-activated linear layer — control what happens to the cell state:

  • Forget gate ft. For each dimension of the cell state, outputs a number in [0, 1]: multiply the previous cell state by this to decide how much to keep.
  • Input gate it. For each dimension, outputs in [0, 1]: multiply by the candidate update to decide how much new information to write.
  • Output gate ot. For each dimension, outputs in [0, 1]: multiply by the tanh of the cell state to decide how much of the cell state to expose as the hidden state this step.

The equations.

All gates are functions of the concatenation of the current input xt and the previous hidden state ht−1:

ft = σ(Wf · [xt, ht−1] + bf) (2.1)
it = σ(Wi · [xt, ht−1] + bi) (2.2)
ot = σ(Wo · [xt, ht−1] + bo) (2.3)

A candidate new state gt, computed like a plain RNN's next state:

gt = tanh(Wg · [xt, ht−1] + bg) (2.4)

The cell-state update — the one that matters — combines forget gate and input gate with elementwise multiplication:

ct = ft ⊙ ct−1 + it ⊙ gt (2.5)

And the hidden state is the cell state "filtered" through the output gate:

ht = ot ⊙ tanh(ct) (2.6)
Figure 1. The LSTM cell. The top warm line is the cell state c, updated additively: ct = ft·ct−1 + it·gt. Because the update is addition (not matrix multiplication), gradients flow backward through c without multiplicative decay — as long as the forget gate ft stays near 1, information (and gradient) passes through cleanly.

Worked example — one LSTM timestep by hand.

Tiny LSTM: d_input=2, d_hidden=3. Processing one timestep.
Previous state: h_{t-1} = [0.2, −0.1, 0.5], c_{t-1} = [1.0, −0.5, 0.8]. Input: x_t = [0.7, −0.3]Concatenated [x_t, h_{t-1}] = [0.7, −0.3, 0.2, −0.1, 0.5] — length 5
Forget gate: f_t = σ(W_f · [x, h] + b_f). Suppose result = σ([0.8, 1.2, 0.9]) = [0.69, 0.77, 0.71]All near 0.7 — retains ~70% of each cell state dimension
Input gate: i_t = σ(W_i · [x, h] + b_i) = σ([−0.5, 1.5, 0.3]) = [0.38, 0.82, 0.57]Dimension 1 is strongly gated open — the LSTM wants to write new info there

Cell update: c_t = f_t ⊙ c_{t-1} + i_t ⊙ g_t = [0.69·1.0, 0.77·(−0.5), 0.71·0.8] + [0.38·0.29, 0.82·(−0.66), 0.57·0.83]
= [0.69, −0.39, 0.57] + [0.11, −0.54, 0.47] = [0.80, −0.93, 1.04]
Output gate: o_t = σ(W_o · [x, h] + b_o) = σ([1.0, 0.2, −0.4]) = [0.73, 0.55, 0.40]
Hidden state: h_t = o_t ⊙ tanh(c_t) = [0.73, 0.55, 0.40] ⊙ tanh([0.80, −0.93, 1.04]) = [0.73·0.66, 0.55·(−0.73), 0.40·0.78] = [0.48, −0.40, 0.31]
Key observation: c_{t-1} dim 0 was 1.0, now 0.80 (f=0.69 retained 69%, plus new input). The cell state changes additively — no matrix multiply, just element-wise gate-and-add.This is why gradients flow: ∂c_t/∂c_{t-1} = diag(f_t) = diag([0.69, 0.77, 0.71]) — no vanishing!
Debugging tip

If LSTM training stalls, check the forget gate bias initialization.

At initialization, gate biases are zero, so f_t ≈ σ(0) = 0.5 — the LSTM immediately forgets half of c_{t-1} every step. After 10 steps: 0.5¹⁰ = 0.001 of the original signal remains. The fix (Jozefowicz et al., 2015): initialize the forget gate bias to 1.0 or 2.0, so f_t starts near σ(1) = 0.73 or σ(2) = 0.88. This simple trick dramatically improves LSTM training on long sequences. PyTorch doesn't do this by default — you must set it manually.

Why this solves vanishing gradients.

The derivative of ct with respect to ct−1 is:

∂ct / ∂ct−1 = diag(ft) (2.7)

Just the forget gate, elementwise. The product over k timesteps is Πs diag(fs). If the network learns to keep fs near 1 for dimensions where long-range memory is needed, the product stays near 1 and gradients flow through all those timesteps. The network can choose how long to remember each dimension — an astonishingly flexible property.

Compare to plain RNN's ∂h/∂h term: diag(1 − h2) · Whh — a shared weight matrix that has the same multiplicative effect regardless of the data. The LSTM's gate is data-dependent and per-dimension. That's the qualitative difference.

03 — LSTM implementation
LSTM implementation.

PyTorch's nn.LSTM fuses all the gate computations into one big linear layer for efficiency. But it helps to see the explicit version first:

import torch
import torch.nn as nn

class LSTMCell(nn.Module):
    """Explicit LSTM cell — one timestep.

    For production use nn.LSTMCell or nn.LSTM (fused, CUDA-optimized).
    This is for understanding the mechanics.
    """
    def __init__(self, input_size, hidden_size):
        super().__init__()
        self.hidden_size = hidden_size
        # One big linear layer computing all four gate pre-activations at once.
        # Input is concat of x_t and h_{t-1}; output is 4*hidden stacked gates.
        self.gates = nn.Linear(input_size + hidden_size, 4 * hidden_size)

    def forward(self, x, state):
        h_prev, c_prev = state
        combined = torch.cat([x, h_prev], dim=-1)
        gates = self.gates(combined)
        # Split into four chunks: forget, input, candidate, output
        f, i, g, o = gates.chunk(4, dim=-1)
        f = torch.sigmoid(f)
        i = torch.sigmoid(i)
        g = torch.tanh(g)
        o = torch.sigmoid(o)
        c_new = f * c_prev + i * g
        h_new = o * torch.tanh(c_new)
        return h_new, (h_new, c_new)


class LSTMModel(nn.Module):
    def __init__(self, input_size, hidden_size):
        super().__init__()
        self.hidden_size = hidden_size
        self.cell = LSTMCell(input_size, hidden_size)

    def forward(self, x):
        # x: (B, T, input_size)
        B, T, _ = x.shape
        h = torch.zeros(B, self.hidden_size, device=x.device)
        c = torch.zeros(B, self.hidden_size, device=x.device)
        outputs = []
        for t in range(T):
            out, (h, c) = self.cell(x[:, t], (h, c))
            outputs.append(out)
        return torch.stack(outputs, dim=1), (h, c)

Production: use nn.LSTM.

PyTorch's built-in nn.LSTM calls a fused cuDNN kernel that computes the full sequence in one optimized GPU call — several times faster than a Python for-loop. Use it unless you have a specific reason not to:

lstm = nn.LSTM(
    input_size=64,          # embedding dim
    hidden_size=128,
    num_layers=2,           # stacked LSTM
    batch_first=True,
    dropout=0.2,            # dropout between layers
    bidirectional=False,    # True for sentiment-style tasks
)

# x: (B, T, 64)
output, (h_final, c_final) = lstm(x)
# output: (B, T, 128) — hidden state at every timestep
# h_final, c_final: (num_layers, B, 128) — final states

Parameter count.

For one LSTM layer with input size D and hidden size H:

params = 4 · H · (D + H) + 4 · H (3.1)

The 4 comes from four gate-weight matrices (forget, input, candidate, output). Compare to a plain RNN's H·(D+H) + H — an LSTM has roughly 4× the parameters for the same D and H. That's the cost of the gating machinery. Almost always worth it.

04 — GRU
GRU — the simpler cousin.

The LSTM has three gates plus a separate cell state. Can we simplify? Cho et al. (2014) proposed the Gated Recurrent Unit (GRU), which achieves most of the LSTM's benefit with less machinery: two gates, no separate cell state.

The GRU equations.

A reset gate rt controls how much of the previous hidden state to use when computing the candidate:

rt = σ(Wr · [xt, ht−1] + br) (4.1)

An update gate zt that plays the combined role of forget and input — it controls how much to keep from the previous hidden state versus how much to replace with the candidate:

zt = σ(Wz · [xt, ht−1] + bz) (4.2)

The candidate hidden state, computed from the input and a reset-gated version of the previous hidden state:

h̃t = tanh(Wh · [xt, rt ⊙ ht−1] + bh) (4.3)

Finally the update — a convex combination of old and new, mediated by the update gate:

ht = (1 − zt) ⊙ ht−1 + zt ⊙ h̃t (4.4)

What's removed and what's kept.

  • Removed: separate cell state. GRU has only the hidden state; there's no long-term memory track distinct from the output.
  • Removed: output gate. The hidden state is exposed directly, not filtered.
  • Kept: additive update structure. Equation (4.4) is still ht = something · ht−1 + something · h̃t, so ∂h_t / ∂h_{t-1} has an identity-like contribution (1 − z_t), preserving the gradient-flow fix.

Parameter count vs LSTM.

A GRU layer has three weight matrices (reset, update, candidate) instead of an LSTM's four. Cost:

GRU params = 3 · H · (D + H) + 3 · H (4.5)

So GRU is 3/4 the size of an LSTM with the same dimensions. On many tasks they perform nearly identically; on tasks requiring very long-range memory, the LSTM's separate cell state can help marginally.

gru = nn.GRU(
    input_size=64,
    hidden_size=128,
    num_layers=2,
    batch_first=True,
    dropout=0.2,
)

# x: (B, T, 64)
output, h_final = gru(x)
# output: (B, T, 128). Only one state (no cell state).
Practical choice between LSTM and GRU

Start with GRU. If it underfits, try LSTM. If either underfits, add capacity (more layers, larger hidden). On most sequence tasks they perform within a percentage point of each other. GRU trains slightly faster and has fewer knobs. LSTM has a small edge on very long or highly structured sequences where the separate cell state's added capacity matters.

Check your understandingQ3 / 06
GRU's update equation is h_t = (1 − z_t) ⊙ h_{t−1} + z_t ⊙ h̃_t. What does this mean when the update gate z_t is close to 0 across the whole vector?
AThe hidden state is replaced by the candidate.
BThe hidden state becomes zero.
CThe candidate is multiplied by (1 − 0) = 1.
DThe hidden state stays equal to the previous timestep: h_t ≈ h_{t−1}. When z_t is near 0, the weight on h̃_t is near 0 and the weight on h_{t−1} is near 1. The gradient ∂h_t/∂h_{t−1} ≈ identity, so gradients pass through this step unchanged. This is how the GRU can carry information (and gradients) across arbitrary numbers of timesteps without vanishing.
05 — CineMatch upgrade
CineMatch — upgrading the RNN to an LSTM.

Back to the CineMatch next-movie predictor from Module 3.3. We built it with a plain RNN. Let's swap in an LSTM and see what changes.

import torch
import torch.nn as nn

class CineMatchLSTM(nn.Module):
    """Next-movie predictor with stacked LSTM."""
    def __init__(self, vocab_size=50000, embed_dim=64,
                 hidden_dim=128, num_layers=2, dropout=0.2):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.lstm = nn.LSTM(
            input_size=embed_dim,
            hidden_size=hidden_dim,
            num_layers=num_layers,
            batch_first=True,
            dropout=dropout if num_layers > 1 else 0.0,
        )
        self.output = nn.Linear(hidden_dim, vocab_size)

    def forward(self, movie_ids):
        emb = self.embedding(movie_ids)           # (B, T, embed_dim)
        out, (h_final, c_final) = self.lstm(emb)  # out: (B, T, hidden_dim)
        last_hidden = out[:, -1]                   # (B, hidden_dim) — final step
        logits = self.output(last_hidden)          # (B, vocab_size)
        return logits

Three lines changed: nn.RNN → nn.LSTM, the tuple state unpacking, and num_layers=2 to give us some stacking. The training loop is completely unchanged.

Expected behavior change.

MetricPlain RNNLSTM (2-layer)
Params (without embed/output)~25K~430K
Hits@10 on next-movie (CineMatch)~18%~26%
Max useful history length~20 movies~100+ movies
Training wall time1×~1.3× (cuDNN fused is fast)

The headline: the LSTM uses longer histories effectively. The plain RNN's accuracy plateaus once input history exceeds ~20 movies — beyond that, gradients can't reach the early tokens, so including them adds noise but not signal. An LSTM keeps extracting signal from longer histories up to ~100+ movies.

Stacking and dropout.

PyTorch's nn.LSTM with num_layers=2 stacks two LSTMs: the output sequence of the first becomes the input sequence of the second. Dropout is applied to the output of each layer except the last. This is a simple way to add capacity; 2-3 layers is typical, beyond that training becomes unstable even with LSTMs.

Common tricks that generalize:

  • Layer normalization between layers — stabilizes training at larger hidden dims.
  • Residual connections across layers — same idea as ResNet applied to the temporal axis.
  • Weight tying — share the embedding and output-projection matrices (both are V×H). Saves millions of parameters on large vocabularies, often with no accuracy loss.
CineMatch · 2020-era recipe

The recommender that shipped when LSTMs ruled.

From roughly 2016 to 2019, the standard for next-item recommendation (Netflix, YouTube, Spotify, and similar) was a 2-3 layer LSTM on user interaction sequences, sometimes with attention over the hidden states, combined with user/item embeddings. Models like GRU4Rec (2016) were the academic standard; production systems were variations on the same theme. These were replaced around 2019-2021 by transformer-based sequential recommenders (SASRec, BERT4Rec). But for offline analysis or smaller-scale deployments, a well-tuned LSTM recommender is still competitive and significantly cheaper to serve.

06 — The bigger picture
Where LSTMs fit, and why they got replaced.

What LSTMs did well.

Between 2014 and 2018, LSTMs were the default for:

  • Machine translation (Google Translate shipped LSTM-based systems in 2016).
  • Speech recognition (stacked bidirectional LSTMs dominated acoustic models).
  • Language modeling (used for character-level and word-level language models).
  • Sequence labeling (POS tagging, named entity recognition, part-of-speech).
  • Sequential recommendation (CineMatch-style next-item prediction).

What broke the dominance.

LSTMs have a structural limitation that became binding as scale grew: they're inherently sequential in time. To compute h_t you need h_{t-1}, which needs h_{t-2}, and so on. This means training cannot be parallelized across the time axis — the T forward passes happen one after another, no matter how many GPUs you have. For sequences of length 500 or 5000, this is a severe bottleneck on modern hardware that wants to do many independent operations in parallel.

Transformers (Module 4.2) replaced the recurrence with self-attention, which computes the relationships between all pairs of positions in parallel. Every timestep sees every other timestep directly — no chain of 100 one-step applications, just one attention operation over the whole sequence. The compute is O(T2) in sequence length instead of O(T), but crucially that O(T2) work can be fully parallelized on modern GPUs. On sequences where T is manageable (hundreds to thousands), transformers train dramatically faster.

Why 2017 was sequence modeling's ResNet moment

"Attention is all you need" — the clarifying realization.

For years, sequence models combined LSTMs with attention: encoder-decoder architectures with attention mechanisms over the encoder's hidden states. The 2017 transformer paper asked — crucially — what if the LSTM isn't needed? What if attention alone, with positional encodings, is enough? The answer was yes, and the consequence was that sequence models became massively more parallelizable, which enabled the scale that produced GPT, BERT, and everything after. LSTMs aren't gone — they still power latency-critical or resource-constrained sequence systems — but for anything large-scale, transformers have replaced them.

When to still reach for LSTM/GRU in 2026.

  • On-device / low-latency systems where a transformer's O(T2) memory doesn't fit, but an LSTM's O(T) does.
  • Streaming tasks where you process data token-by-token without access to future tokens. LSTMs are natively causal and streaming-friendly.
  • Very long sequences where transformer attention scales quadratically. For 10,000+ token contexts, efficient attention variants or state-space models (which are loosely RNN-inspired) are competitive.
  • Small-data regimes where LSTM's inductive bias (recurrent structure) helps more than transformer's flexibility.
Phase 3's closing thought

CNNs, RNNs, LSTMs, and GRUs all share one deep design principle: bake the structure of the data into the architecture. Grid structure → convolutions. Sequential structure → recurrence with gating. Inductive biases beat raw capacity, at scales where data is limited. That principle stops being binding only when scale is large enough for transformers to rediscover the structure from data. Phase 4 is that story.

Phase 3 — recap and what's next.

Phase 3 covered specialized architectures for two of the most common data modalities: CNNs for grid data (images), RNNs/LSTMs/GRUs for sequential data. Both share the deep principle: encode the structure of the data into the architecture itself. CNNs do this with convolution (locality + translation equivariance); RNNs do this with recurrence (parameter sharing over time); LSTMs add gating to make long-range dependencies learnable.

Phase 4 turns to the modern stack: attention, transformers, embeddings, and pretraining. The thread that connects them: at scale, flexibility beats inductive bias. Transformers have less built-in structure than CNNs or LSTMs but learn the necessary structure from enormous datasets. Understanding how — and when flexibility finally becomes the right choice — is Phase 4's story.
