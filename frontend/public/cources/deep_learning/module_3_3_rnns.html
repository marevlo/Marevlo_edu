Recurrent networks. Processing order.

CNNs exploit that images are grids: nearby pixels relate. Sequences — movie watch histories, sentences, audio — have different structure. The events are ordered, their lengths vary, and what comes later depends on what came before. RNNs encode that temporal structure into the architecture: a single cell visited once per timestep, carrying state forward. Understanding RNNs — including why they ultimately failed and were replaced — is essential for understanding transformers.

01 — Why sequences
Why sequences need their own architecture.

CineMatch has a new feature to build: given a user's last 20 movies watched, predict what they'll watch next. The input is a sequence — an ordered list — of 20 movie IDs. The output is a probability distribution over the catalog of ~50,000 movies.

Take a step back and ask: what's structurally different about sequence data compared to the tabular features from Phase 2 or the images from 3.1?

Three distinctive properties of sequences.

  1. Order matters. Watching [Inception, Interstellar, Tenet] conveys something different from [Tenet, Interstellar, Inception]. The first looks like chronological Nolan-fandom; the second looks like reverse-chronological exploration. Permutation-invariant architectures (MLPs on flattened inputs) can't see this at all.
  2. Variable length. One user has 4 watches in their history; another has 400. A CNN or MLP expects fixed-size input and needs awkward padding/cropping to handle this. A sequence model should natively accept any length.
  3. Long-range dependencies. The movie a user watches next might depend on what they watched yesterday or what they watched five years ago. The model needs mechanisms to carry relevant information across many timesteps.

The RNN idea: a cell visited repeatedly.

An RNN defines a single cell — a small neural network — and applies it once per timestep. The twist: each invocation receives not just the current input but also the cell's own output from the previous step, called the hidden state. That hidden state is how information from earlier timesteps reaches later ones.

This is the sequence analog of CNN weight sharing. A CNN shares the same 3×3 kernel across every spatial location of an image. An RNN shares the same cell weights across every timestep of a sequence. Both are examples of parameter sharing over structured dimensions — space for CNN, time for RNN — and both dramatically improve sample efficiency.

The structural analogy

Think of an RNN as a CNN that convolves over time with a 1D kernel, but with an additional feedback loop: the output of each step is fed back as input to the next. That feedback loop is what lets information accumulate across the sequence — and also what causes every hard problem RNNs face.

02 — The cell
The RNN cell.

At each timestep t, the RNN cell receives two inputs — the current observation xt and the previous hidden state ht−1 — and produces a new hidden state ht:

ht = tanh(Wxh xt + Whh ht−1 + bh) (2.1)

Three learnable parameters, all shared across every timestep:

  • Wxh (shape H×D): how the current input maps into hidden space, with D the input dim and H the hidden dim.
  • Whh (shape H×H): how the previous hidden state influences the next. This is the "recurrence" that makes it recurrent.
  • bh (shape H): bias.

For prediction, there's also an output weight Why (shape V×H, where V is the output vocabulary size) that projects the hidden state to predictions:

yt = Why ht + by (2.2)

Initial state.

At t=0 there's no previous hidden state — by convention it's initialized to zeros (or sometimes to a learned parameter). From there the recurrence unrolls naturally.

Why tanh?

Classical RNNs use tanh as the activation function. It's bounded in [−1, 1], which keeps hidden states from blowing up during the forward pass (an unbounded activation like ReLU in a recurrent loop can produce arbitrarily large values that destabilize training). Modern variants use different activations inside gating structures (next module), but the plain RNN uses tanh for good reason.

A concrete forward pass.

03 — Unrolling
Unrolling through time.

To understand how RNNs train, we need to view them as a computational graph. The recurrence seems to involve a loop, but for a fixed-length sequence of T steps we can unroll it into T copies of the cell — each copy sharing weights with the others.

Figure 1. Folded vs unrolled views of an RNN. The folded view shows the recurrent definition — one cell with a self-loop. The unrolled view shows the same computation as a feed-forward network with T cell-copies chained in sequence. The cells share weights: W_xh, W_hh, b_h are identical across all copies. Unrolling is not a modeling choice — it's how we can apply standard feed-forward backpropagation to a recurrent model.

What unrolling lets us do.

Once unrolled, the RNN is a feed-forward network — just one with T times the depth of a single cell and with the interesting property that several layers share weights. Standard backpropagation applies. The gradient at one step flows back to earlier steps through the chain of hidden-state connections. We just need to be careful about weight sharing during gradient accumulation: the same weight appears in multiple copies, so its gradient is the sum of gradients over all copies.

One cell, many applications

Unrolling is purely a conceptual and computational tool. The model has a single cell; unrolling is the finite slice of that cell's trajectory over the specific sequence you're processing. In code, you use a Python for loop (or, more efficiently, a specialized cuDNN kernel) that applies the same layer object to each timestep.

import torch
import torch.nn as nn

class SimpleRNN(nn.Module):
    def __init__(self, input_dim, hidden_dim):
        super().__init__()
        self.hidden_dim = hidden_dim
        self.cell = nn.Linear(input_dim + hidden_dim, hidden_dim)

    def forward(self, x):
        # x: (batch, seq_len, input_dim)
        B, T, _ = x.shape
        h = torch.zeros(B, self.hidden_dim, device=x.device)
        outputs = []
        for t in range(T):
            combined = torch.cat([x[:, t], h], dim=1)
            h = torch.tanh(self.cell(combined))
            outputs.append(h)
        return torch.stack(outputs, dim=1), h

That's the core of every RNN. The nn.Linear is applied T times — once per timestep — with the hidden state threaded through. PyTorch's built-in nn.RNN does the same thing but with a fused CUDA kernel for speed.

04 — BPTT
Backpropagation through time (BPTT).

Now for training. We have a sequence of T hidden states h1, h2, ..., hT and (typically) a loss computed from the last hidden state (classification) or from every hidden state (language modeling):

𝓛 = Σt=1T 𝓛t(yt, ŷt) (4.1)

We need to compute the gradient of 𝓛 with respect to the shared weights Wxh, Whh, bh. Since these weights appear in every cell-copy of the unrolled graph, their total gradient is summed over all copies. This is Backpropagation Through Time — standard backprop on the unrolled graph, with the subtlety that shared weights accumulate gradients from every timestep they participate in.

The key gradient.

The derivative of the total loss with respect to any hidden state ht involves contributions from all loss terms at time t or later:

∂𝓛 / ∂ht = Σk≥t (∂𝓛k / ∂hk) · (∂hk / ∂ht) (4.2)

The critical term is ∂hk / ∂ht — how sensitive a later hidden state is to an earlier one. By chain rule:

∂hk / ∂ht = Πs=t+1k (∂hs / ∂hs−1) (4.3)

And each one-step Jacobian, from the cell equation hs = tanh(Wxh xs + Whh hs−1 + b), works out to:

∂hs / ∂hs−1 = diag(1 − hs2) · Whh (4.4)

The diag(1 − h2) term is tanh's derivative evaluated at the pre-activation (since d/dz tanh(z) = 1 − tanh(z)2). Its entries are always in (0, 1]. So the Jacobian is Whh scaled by numbers less than 1.

Worked example — 3 timesteps.

Continuing with the tiny RNN from § 2, suppose our loss depends only on h3 (say, classification from the final state). To update Whh, we need ∂𝓛/∂Whh, which involves gradients flowing back through all three timesteps.

This is ∂h₃/∂h₁. Every entry is small — at most ~0.15. That's across only two timesteps. Extrapolate to 50 timesteps and you're multiplying 49 such matrices together. If the dominant singular value of each Jacobian is well below 1, the product decays exponentially toward zero. That's the vanishing gradient problem, which § 5 spells out.

Truncated BPTT (TBPTT).

For long sequences, backpropagating through all T steps is expensive in both compute and memory (you have to store activations for every step). The standard practical fix: truncated BPTT. Run the forward pass for a chunk of length k, backpropagate through that chunk, detach the hidden state, continue forward for the next chunk. This approximates true BPTT by assuming long-range gradients are small (which they usually are anyway, due to vanishing gradients) and trades gradient accuracy for constant memory.

# Truncated BPTT idiom
h = torch.zeros(batch, hidden_dim).to(device)
for chunk_start in range(0, total_len, chunk_size):
    x_chunk = x[:, chunk_start : chunk_start + chunk_size]
    out, h = rnn(x_chunk, h)
    loss = loss_fn(out, targets[:, chunk_start : chunk_start + chunk_size])
    loss.backward()
    optimizer.step(); optimizer.zero_grad()
    h = h.detach()   # key line: cut the gradient graph at chunk boundary

05 — Vanishing & exploding
The vanishing and exploding gradient problem.

Recall the core gradient expression from BPTT (formula 4.3):

∂hk / ∂ht = Πs=t+1k diag(1 − hs2) · Whh (5.1)

We're multiplying k − t matrices together. For long sequences, this is a large product. What happens depends on the matrices' magnitudes.

Case 1 — Vanishing gradients.

If Whh's largest singular value is less than 1, and because diag(1 − h2) has entries at most 1, the product of many such matrices has entries that shrink exponentially toward zero. Gradients at early timesteps become negligible — the network can't learn long-range dependencies because the learning signal doesn't reach those steps.

Concretely: if each one-step Jacobian has norm ≈ 0.5 (typical for a well-initialized tanh RNN), then after 50 timesteps the gradient is scaled by 0.550 ≈ 10−15. Effectively zero. The gradient cannot travel long distances.

Case 2 — Exploding gradients.

If Whh's largest singular value is greater than 1 and the activations aren't saturating, the product grows exponentially. Gradients at early timesteps become enormous, and the optimizer takes huge steps in random directions, producing NaN loss or oscillating training.

Numeric proof: gradient vanishing in a 2-dim hidden state RNN over 10 steps 1. Plain RNNs are fragile: even slight shifts in weight magnitude during training can tip from one regime to the other. This is why gated architectures (LSTM, GRU) were invented.

Gradient clipping in practice.

# After backward(), before optimizer.step()
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()

Clipping doesn't help vanishing gradients — scaling a near-zero gradient gives another near-zero gradient. It only fixes the exploding direction. The vanishing problem is structural and requires architectural changes.

Why the plain RNN underperformed

The vanishing gradient is not a training artifact — it's a property of the architecture.

You can use better initialization, lower learning rates, more compute — the math in (5.1) still holds. Plain RNNs fundamentally struggle to connect events more than ~20 timesteps apart. A model of a 200-token sentence with a plain RNN literally cannot propagate information from the start to the end during training. This is why plain RNNs were rapidly superseded by LSTMs (1997, popularized ~2014) and then transformers (2017).

06 — CineMatch end-to-end
CineMatch next-movie prediction — RNN from end to end.

Now let's wire it into a real task. CineMatch wants a model: given a user's last T=20 watches, predict the probability distribution over the catalog of V=50,000 movies for "what do they watch next?"

The components.

A full RNN-based recommender is three layers:

  1. Embedding layer. Each movie ID (an integer from 0 to V−1) is mapped to a dense d-dimensional vector. The embedding is learnable. This turns the sparse categorical input into something a neural network can process.
  2. RNN layer. Processes the sequence of embeddings, producing a hidden state at each timestep.
  3. Output projection. A linear layer from H to V, followed by softmax, giving a probability distribution over the catalog. Predict the next movie by taking argmax or sampling.
import torch
import torch.nn as nn

class CineMatchRNN(nn.Module):
    """Next-movie predictor using a basic RNN.

    Inputs: (batch, seq_len) integer movie IDs
    Outputs: (batch, vocab_size) logits over catalog
    """
    def __init__(self, vocab_size=50000, embed_dim=64, hidden_dim=128):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.rnn = nn.RNN(
            input_size=embed_dim,
            hidden_size=hidden_dim,
            batch_first=True,
            nonlinearity="tanh",
        )
        self.output = nn.Linear(hidden_dim, vocab_size)

    def forward(self, movie_ids):
        # movie_ids: (batch, seq_len)
        emb = self.embedding(movie_ids)       # (batch, seq_len, embed_dim)
        _, h_final = self.rnn(emb)              # h_final: (1, batch, hidden_dim)
        h_final = h_final.squeeze(0)           # (batch, hidden_dim)
        logits = self.output(h_final)          # (batch, vocab_size)
        return logits

Training.

Standard cross-entropy loss. Each training example is a (history, next_movie) pair — slide a 20-movie window across every user's history and use the 20-movie prefix as input and the 21st movie as the target.

model = CineMatchRNN().to(device)
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)
loss_fn = nn.CrossEntropyLoss()

for epoch in range(n_epochs):
    for history, target in train_loader:
        # history: (B, 20) int, target: (B,) int
        history, target = history.to(device), target.to(device)
        logits = model(history)
        loss = loss_fn(logits, target)

        optimizer.zero_grad(set_to_none=True)
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
        optimizer.step()

Both the embedding table and the output projection scale with vocabulary size; tricks to reduce them (weight tying, sampled softmax, hierarchical softmax) are standard in large-vocabulary settings.

What to expect on this task

RNN baselines on next-item prediction.

On a CineMatch-sized dataset (100K users, 50K movies, avg 40 watches per user), a well-tuned RNN with length-20 windows typically reaches Hits@10 (probability the true next movie is in the top 10 predictions) around 15-20%. A plain RNN hits a plateau here; switching to LSTM (Module 3.4) usually pushes it to 22-28%. A transformer-based sequential recommender (Phase 4) can push to 30%+. Strong baselines come from combining watch-history RNN outputs with collaborative-filtering features — which is what modern production recommenders do.

07 — Variants
RNN variants — one-to-one, many-to-one, many-to-many.

One reason RNNs dominated sequence modeling for decades is their flexibility. The same cell can be configured for very different task shapes by choosing where inputs arrive and where outputs come out.

VariantShapeExample task
One-to-one1 input, 1 outputImage classification (not really an RNN task).
One-to-many1 input, many outputsImage captioning (one image → a sentence).
Many-to-oneMany inputs, 1 outputSentiment classification, CineMatch next-movie (uses final hidden state).
Many-to-many (aligned)Input per step, output per stepVideo frame classification, part-of-speech tagging.
Many-to-many (seq2seq)Input sequence → output sequence of different lengthTranslation, summarization. Two RNNs chained: encoder + decoder.

The CineMatch model we just built is many-to-one — it eats 20 movies and emits one distribution. An RNN that produces a new distribution at every step (language-model style: given the first t movies, predict the (t+1)th) is many-to-many. Both use the same cell — only the output-projection placement changes.

Bidirectional RNNs.

So far we've assumed causal processing: the output at time t depends only on x1, ..., xt. For many tasks (classifying a full sentence, labeling a full sequence), you want to use context from both directions. A bidirectional RNN runs two RNNs: one forward (left-to-right) and one backward (right-to-left). At each position, the two hidden states are concatenated. The output at position t then depends on the entire sequence.

For CineMatch's next-movie prediction, bidirectional doesn't apply — the task is inherently causal (we don't have the future to condition on). For sentiment classification on a completed review, bidirectional helps. The rule: bidirectional is only useful when the entire sequence is observable at inference time.

Deep (stacked) RNNs.

Like CNNs, RNNs can be stacked. A 2-layer RNN uses the hidden state sequence of the first RNN as the input sequence of the second. PyTorch's nn.RNN supports this via num_layers. Empirically, 2-3 layers is often the sweet spot — deeper plain RNNs compound the vanishing-gradient problem. (LSTMs handle depth better; we'll see in 3.4.)

The arc of sequence modeling, 1997–2017

RNN → LSTM → bi-LSTM → attention → transformer.

Jürgen Schmidhuber and Sepp Hochreiter published LSTM in 1997. It was quietly influential through the 2000s but didn't dominate — compute and data weren't there. Around 2014, LSTMs became the standard for machine translation (Google's production translation system used them). In 2015, Bahdanau et al. added attention on top of a seq2seq LSTM. In 2017, Vaswani et al. asked "do we need the recurrence at all if we have attention?" and invented the transformer. Every step was a response to the previous step's failure mode: LSTMs fix RNN's vanishing gradient; attention fixes the bottleneck of compressing a whole input into a fixed-size vector; transformers fix attention-over-LSTM's sequential-computation bottleneck.

Module 3.3 — Mastery checkpoint

Where this leads.

The plain RNN has a clean idea — weight-shared cell over time — but a fatal flaw: its gradients don't travel far. The fix, discovered by Hochreiter and Schmidhuber in 1997 but widely adopted only after 2014, is gating: small networks that decide at each timestep what to remember, what to forget, and what to expose. The Long Short-Term Memory (LSTM) and Gated Recurrent Unit (GRU) are what made RNNs actually work for language, translation, and — for a while — recommendation. That's Module 3.4.
