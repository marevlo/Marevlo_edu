Convolutional networks. Geometry as an inductive bias.

An MLP treats every pixel independently. A convolutional network knows two deep facts about images: nearby pixels are related, and what a cat looks like in the top-left of a frame is exactly what a cat looks like in the bottom-right. Wiring those two priors into the architecture — rather than hoping it discovers them — is the single biggest leap in deep learning. This module explains why, with CineMatch movie posters as the concrete task.

01 — Why CNNs
Why MLPs fail on images.

Here's a CineMatch movie poster. It's 224 pixels wide, 224 pixels tall, 3 color channels — 150,528 numbers. Now try to classify it as one of 10 genres with the MLPs from Phase 2.

An MLP's first hidden layer — say 1,024 units — needs a weight for every input connected to every hidden unit. That's 150,528 × 1,024 ≈ 154 million parameters in the first layer alone. For a problem we might realistically have 50,000 training posters for. The parameter-to-data ratio is hopeless.

Two deeper problems than parameter count.

Even if you had the compute and the data, the MLP is still structurally wrong for images. Two problems:

  1. No locality prior. Pixels are arranged on a 2D grid. Pixel (10, 20) is close to pixel (10, 21); far from pixel (200, 180). An MLP doesn't know this — to it, all input dimensions are exchangeable. It has to learn from data that nearby pixels tend to be related, which wastes enormous model capacity.
  2. No translation equivariance. A cat in the top-left of an image is the same object as a cat in the bottom-right. The features needed to detect it — whiskers, fur, eyes — look identical. An MLP has to learn a separate detector for every possible position of the cat, since each position lights up different input dimensions. Again, massively wasteful.
The core CNN insight

Don't make the network learn that "nearby pixels matter" and "a cat is a cat anywhere" from data. Bake those priors into the architecture. The result is fewer parameters, better sample efficiency, and a sharper learning target — because the network only has to discover patterns, not also discover that patterns are translation-invariant.

Convolutional networks bake in both priors simultaneously through a single primitive: the convolution operation. Everything else — pooling, feature maps, channels, modern architectures — is organization around that primitive.

A quiet revolution

Convolutional networks were invented in the late 1980s.

Yann LeCun's LeNet-5 (1998) classified handwritten digits on the MNIST dataset using a CNN. It was the right idea fifteen years too early — no one had enough compute or labeled data to scale it. For most of the 1990s and 2000s, classical computer vision methods (SIFT, HOG, SVMs) dominated the benchmarks. Then in 2012, AlexNet — essentially LeCun's idea with more layers, more data (ImageNet), and GPU training — beat every classical method by an enormous margin. The CNN era began. Every modern image model still uses the convolution operation LeCun formalized.

02 — The operation
The convolution operation.

Convolution is deceptively simple. You have an input (a 2D image, say) and a small kernel (a tiny 2D weight matrix, typically 3×3). To produce an output, you slide the kernel over every position of the input, compute an elementwise product with the patch it overlaps, sum the result, and write it into the output at that position.

(I ∗ K)(i, j) = Σm,n I(i+m, j+n) · K(m, n) (2.1)

That's the whole definition. Everything else is consequence. Let's make it concrete with a 4×4 input and a 3×3 kernel, which produces a 2×2 output.

Why this wires in both priors at once.

Look at what just happened. To compute any output cell, only a local 3×3 region of the input mattered — that's the locality prior. And the same 9 kernel numbers were used at every output position — that's the translation equivariance prior encoded as weight sharing. Same kernel, every location; if a feature is detected at position (5,5), the same feature would have been detected at (100,100).

The count of independent parameters? For a 3×3 kernel: 9 weights plus 1 bias. That's it. Regardless of whether the input is 28×28 or 1024×1024.

Parameter count, dramatic reduction

An MLP with one hidden unit connected to every pixel of a 224×224 image: 50,176 weights. A 3×3 convolution kernel that produces an entire feature map over the same image: 9 weights. The convolution has 0.018% of the parameters — and yet captures more because it generalizes across positions.

03 — Padding & stride
Padding, stride, and output size.

Three knobs control how a convolution maps input shape to output shape. Understanding them is the difference between confidently designing networks and randomly trying numbers until something fits.

Padding.

A naive 3×3 convolution shrinks the spatial dimensions by 2 each layer (−1 on each side). Stack 10 layers, lose 20 pixels — your 32×32 image becomes 12×12. Usually undesirable. The fix is padding: add zeros around the input before convolving.

With padding p of zeros on each side, the effective input is larger by 2p, so the output is larger by 2p. For a k×k kernel, padding p = (k−1)/2 keeps the output size equal to the input. "Same padding" is what this is usually called:

  • 3×3 kernel → padding 1 keeps spatial size identical.
  • 5×5 kernel → padding 2 keeps spatial size identical.
  • 7×7 kernel → padding 3 keeps spatial size identical.

Stride.

Stride controls how many pixels the kernel jumps between output positions. Stride 1 covers every position. Stride 2 skips every other position — the output is half the spatial size. Stride is the mechanism that lets us downsample without explicit pooling (more on pooling in § 4).

The full output-size formula.

For an input of size H, a kernel of size k, padding p, and stride s:

Hout = ⌊(H + 2p − k) / s⌋ + 1 (3.1)

Memorize this. It's the single most useful formula in CNN design. Check it on a familiar case: H=28, k=3, p=0, s=1 → (28+0−3)/1 + 1 = 26. Matches what we just computed.

Why 3×3 kernels dominated.

Modern CNNs (VGG, ResNet, everything after ~2014) use almost exclusively 3×3 kernels. Why not 5×5 or 7×7?

  • Parameter efficiency. Two stacked 3×3 kernels have the same receptive field as one 5×5 (both cover a 5×5 patch of the input after two layers) but use 2·9=18 parameters vs 25 — and one more nonlinearity in between.
  • Compute efficiency. Smaller kernels do fewer multiply-adds per output.
  • Hardware efficiency. 3×3 kernels are aggressively optimized in every CUDA library and neural processor.

The off-by-one that haunts every CNN implementer

The floor in equation (3.1) matters.

If H + 2p − k isn't divisible by s, the floor silently rounds down. A 7×7 input with a 3×3 kernel and stride 2 and padding 0 gives ⌊(7−3)/2⌋ + 1 = 3 — but ⌊(8−3)/2⌋ + 1 = 3 also (different input, same output). Feeding slightly different-sized images to the same architecture can produce identically-sized feature maps but with different content at the right edge. When designing architectures, pick paddings/strides so spatial sizes come out to clean numbers (powers of 2 if possible).

04 — Channels
Channels — the third dimension.

A real image isn't a 2D matrix — it's a 3D tensor: height × width × channels. A color image has 3 channels (R, G, B). A single grayscale image has 1. And after the first convolution layer, the output has as many channels as the number of kernels you used.

Multi-channel convolution.

When the input has Cin channels, a single convolution "kernel" is actually a 3D block of weights: k × k × Cin. You convolve it across all input channels simultaneously and sum — producing a single output channel. To produce Cout output channels, you use Cout such 3D kernels:

parameters per conv layer = (k · k · Cin + 1) · Cout (4.1)

The +1 is the bias per output channel. A 3×3 conv from 3→64 channels has (3·3·3 + 1) · 64 = 1,792 parameters. A 3×3 conv from 64→128 has (3·3·64 + 1) · 128 = 73,856 parameters. Parameter count grows fast with channel depth — channel count is the main capacity lever in a CNN.

What do channels represent?

Early layers learn simple features — edges at various orientations, color blobs. Deeper layers compose these into more complex features — textures, parts of objects, eventually whole-object detectors. Each channel of each layer represents a different learned feature. After the first convolution layer of a CNN trained on posters, you might have 64 channels, each lighting up for a different edge orientation or color.

Figure 1. Channel counts grow through the network: 3 input (RGB) → 64 → 128 → 256 → 512. Spatial dimensions typically shrink as channels expand, keeping total activation count (H · W · C) roughly constant or slightly decreasing. This is the canonical CNN shape evolution: wide-and-shallow input → narrow-and-deep features.

The 1×1 convolution.

A 1×1 kernel looks trivial — it just rescales a single pixel. But with multiple channels, a 1×1 conv from Cin to Cout channels is effectively a per-pixel MLP: it learns a linear combination of the input channels at each spatial location, with shared weights across all locations. This turns out to be extremely useful for mixing channels (the only way to let different channels communicate) and for cheap dimensionality reduction. Modern architectures (ResNet, MobileNet, ViT) use 1×1 convolutions constantly.

05 — Pooling
Pooling — downsampling and invariance.

Convolution followed by activation produces feature maps at roughly the input resolution (with same-padding). To build deeper, more abstract features, we need to reduce spatial resolution — so each subsequent layer can "see" more of the image through its receptive field. There are two ways to downsample: strided convolution (§ 3) and pooling.

Max pooling.

Max pooling slides a small window (typically 2×2) across the feature map. For each window position, it outputs the maximum value within the window. There are no learned parameters — it's a pure reduction operation. With window size 2 and stride 2, a 56×56 feature map becomes 28×28.

import torch.nn as nn

# Spatial halving via pooling
pool = nn.MaxPool2d(kernel_size=2, stride=2)

# Applied inside a conv block
block = nn.Sequential(
    nn.Conv2d(64, 128, kernel_size=3, padding=1),
    nn.BatchNorm2d(128),
    nn.ReLU(inplace=True),
    nn.MaxPool2d(2, 2),
)

Why max pooling makes sense.

  1. Downsampling. Halves spatial dimensions, reducing compute and memory in subsequent layers.
  2. Small-translation invariance. If a feature appears anywhere in the 2×2 window, the max captures it — the exact position within the window is thrown away. This provides robustness to small spatial shifts, which matches how we perceive objects (a whisker at pixel (50,50) vs pixel (51,50) should classify the same way).
  3. Non-linearity. Max is non-linear, so pooling adds expressive power beyond linear downsampling.

Average pooling.

Instead of max, take the average of values in the window. Historically common in LeNet; mostly replaced by max pooling after AlexNet. One place average pooling still dominates: global average pooling — reduce each entire feature map to a single scalar by averaging across all spatial locations. It's what most modern CNNs use instead of flattening before the classifier, because it's translation-invariant and parameter-free.

Strided conv vs pooling.

Modern architectures often skip pooling entirely and use strided convolutions for downsampling. Why? A stride-2 conv learns how to downsample, while max pooling applies a fixed rule. ResNet-50 uses one early max pool plus stride-2 convolutions for subsequent downsampling; ViT and most newer architectures use strided convs or patch-based downsampling exclusively.

The modern downsampling default

Use one initial max pool (after the first conv block) for fast early spatial reduction, then strided convolutions for the rest. Pure-pooling architectures (LeNet-era) and pure-strided architectures (ResNet-era) both work. What matters is that spatial resolution decreases as channels increase — the total activation volume stays roughly bounded while abstraction increases.

06 — Receptive fields
Receptive fields — what each neuron 'sees'.

A key question: what part of the original input influences each position of each feature map? This is the receptive field. Shallow feature maps have small receptive fields; deeper feature maps have larger ones. At some layer deep in a well-designed network, a single neuron's receptive field covers the entire input image — that's when the network has "seen" the whole picture.

How receptive fields grow.

For a stack of convolutions with kernel size ki and stride si, the receptive field after layer L is computed recursively:

RFL = RFL−1 + (kL − 1) · Πi<L si (6.1)

Concretely: after one 3×3 conv, RF = 3. After two stacked 3×3 convs with stride 1, RF = 5 (the second conv sees a 3×3 patch of the first layer's 3×3 patches, which overlap to cover 5×5 input). Add a stride-2 pool or conv, and subsequent layers double their effective receptive-field expansion.

This is why modern classifiers need ~20+ layers to handle 224×224 inputs — the receptive field needs to grow to span the whole image. Two 3×3 convs give RF = 5 at most; you need many layers of accumulation to get to 224.

Dilated (atrous) convolutions.

If you want large receptive fields without deep networks, dilated convolutions spread the kernel apart. A 3×3 kernel with dilation 2 still has 9 weights, but they're applied at positions (0,0), (0,2), (0,4), (2,0), ..., covering a 5×5 region. Dilation d multiplies the effective receptive-field growth by d. Useful for segmentation, audio, and other tasks where you need dense predictions at high resolution.

07 — LeNet end-to-end
LeNet — the first CNN, built end-to-end.

Let's put everything together in the simplest CNN that actually works. LeNet-5 (1998, LeCun et al.) was designed for 28×28 grayscale MNIST digits. It's small, well-understood, and introduces every core pattern still used in modern CNNs.

import torch
import torch.nn as nn

class LeNet(nn.Module):
    """LeNet-5 style, modernized with ReLU and max pool."""
    def __init__(self, num_classes=10):
        super().__init__()
        self.features = nn.Sequential(
            nn.Conv2d(1, 6, kernel_size=5, padding=2),   # 28x28x6
            nn.ReLU(inplace=True),
            nn.MaxPool2d(2, 2),                              # 14x14x6
            nn.Conv2d(6, 16, kernel_size=5),              # 10x10x16
            nn.ReLU(inplace=True),
            nn.MaxPool2d(2, 2),                              # 5x5x16
        )
        self.classifier = nn.Sequential(
            nn.Flatten(),                                   # 400
            nn.Linear(16 * 5 * 5, 120),
            nn.ReLU(inplace=True),
            nn.Linear(120, 84),
            nn.ReLU(inplace=True),
            nn.Linear(84, num_classes),
        )

    def forward(self, x):
        x = self.features(x)
        x = self.classifier(x)
        return x

model = LeNet(num_classes=10)
print(sum(p.numel() for p in model.parameters()))   # 61,706

62K parameters — smaller than most of the MLPs we built in Phase 2. And it works on MNIST. Let's understand every shape.

Modern networks flip this — their conv stacks are much deeper and dominate parameters, while global average pooling reduces the head to nothing.

From LeNet to a CineMatch poster classifier

Scaling up: 28×28 digits vs 224×224 movie posters.

A movie poster is ~65× larger in pixel count than MNIST. It's 3-channel color, not grayscale. And posters have vastly more visual complexity — faces, text, typography, lighting. LeNet as-is would work poorly. The fix isn't to change the idea — just to scale it up and add modern tricks. That's what Module 3.2 is about. A VGG, ResNet, or DenseNet trained on 224×224 posters uses exactly the patterns above, just deeper (30+ layers), wider (64→512 channels), regularized (BN + dropout), and with better init. The LeNet template is the blueprint.

Training a CNN — what changes from Phase 2.

Training a CNN uses exactly the machinery from Phase 2 (AdamW, BF16 AMP, weight decay, data augmentation, early stopping). The architecture changes, not the training loop. One adjustment: for image training, always use augmentation — the single biggest accuracy lever. Horizontal flip, random crop, color jitter are the baseline. MixUp or CutMix buy you additional gains for well-trained models.
