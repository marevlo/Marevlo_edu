Phase 2 Capstone. The regularized MLP that beats Phase 1.

Phase 1 built linear baselines. Phase 2 gave us everything needed to go deeper — MLPs, backprop, optimization, regularization, computational performance. This capstone puts it all together: a fully-regularized, AMP-trained CineMatch MLP with BatchNorm, dropout, AdamW with parameter groups, cosine warmup, early stopping, and gradient clipping. The ablation table proves that each piece contributes, and the final model beats every Phase 1 baseline by a clear margin.

01 — Architecture
The architecture — every Phase 2 technique in one model.

The CineMatch MLP from Module 2.3 had 2,433 parameters across a 4→64→32→1 architecture. The capstone version keeps the same depth but integrates every regularization and training technique from Phase 2:

# cinematch/models/mlp.py — Phase 2 capstone
import torch
import torch.nn as nn

class CineMatchMLP(nn.Module):
    """
    Production CineMatch MLP. Integrates:
      - BatchNorm (2.6) for training stability + mild regularization
      - Dropout (2.6) for generalization
      - GELU activation (2.3) — smoother than ReLU
      - Proper parameter grouping for weight decay (2.6)

    Architecture: 4 → 64 → 32 → 1
    Total params: 2,433 (weights) + 192 (BN) = 2,625
    """
    def __init__(self, in_dim=4, hidden=(64, 32), p_drop=0.2):
        super().__init__()
        layers = []
        d = in_dim
        for h in hidden:
            layers += [
                nn.Linear(d, h),
                nn.BatchNorm1d(h),
                nn.GELU(),
                nn.Dropout(p_drop),
            ]
            d = h
        layers.append(nn.Linear(d, 1))
        self.net = nn.Sequential(*layers)

    def forward(self, x):
        return self.net(x)


def get_param_groups(model, wd=0.01):
    """Weight decay on weight matrices only — not biases or BN params (2.6)."""
    decay, no_decay = [], []
    for name, p in model.named_parameters():
        if "bias" in name or "bn" in name.lower() or "norm" in name.lower():
            no_decay.append(p)
        else:
            decay.append(p)
    return [
        {"params": decay,    "weight_decay": wd},
        {"params": no_decay, "weight_decay": 0.0},
    ]
ArchitectureQuestion 01 / 07
The capstone MLP uses nn.BatchNorm1d before GELU and nn.Dropout after GELU. Why this ordering?
AThe ordering doesn't matter — all permutations are equivalent.
BBatchNorm must always come last in a block.
CBatchNorm normalizes the pre-activation distribution (Module 2.6 convention: Linear → BN → activation). Dropout goes after the activation because we want to drop activated outputs, not the raw linear outputs — dropping before activation would be partially undone by BN's renormalization.
DDropout must always come before BatchNorm to prevent data leakage.
02 — Training
The training script — production-grade.

This is the full training loop from Module 2.7, instantiated for the capstone. Every line references a specific Phase 2 module:

# cinematch/training/train_mlp.py — Phase 2 capstone training
import torch, math, json
from pathlib import Path
from copy import deepcopy

# Config
cfg = {
    "seed": 42, "batch_size": 64, "lr": 1e-3,
    "epochs": 100, "patience": 10, "p_drop": 0.2,
    "weight_decay": 0.01, "warmup_steps": 100,
}

torch.manual_seed(cfg["seed"])
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

# Model (2.1, 2.3, 2.6)
model = CineMatchMLP(in_dim=4, hidden=(64, 32), p_drop=cfg["p_drop"]).to(device)

# Optimizer: AdamW with param groups (2.4, 2.6)
opt = torch.optim.AdamW(get_param_groups(model, cfg["weight_decay"]), lr=cfg["lr"])

# LR schedule: linear warmup → cosine decay (2.4, 2.7)
steps_per_epoch = len(train_loader)
total_steps = cfg["epochs"] * steps_per_epoch

def get_lr(step):
    if step < cfg["warmup_steps"]:
        return cfg["lr"] * step / cfg["warmup_steps"]
    t = (step - cfg["warmup_steps"]) / max(1, total_steps - cfg["warmup_steps"])
    return 1e-6 + 0.5 * (cfg["lr"] - 1e-6) * (1 + math.cos(math.pi * t))

# BF16 AMP (2.7)
scaler = torch.cuda.amp.GradScaler()

# Early stopping (2.6)
best_val = float("inf"); best_state = None; patience_ctr = 0
step = 0

for epoch in range(cfg["epochs"]):
    model.train()
    for x, y in train_loader:
        x, y = x.to(device, non_blocking=True), y.to(device, non_blocking=True)

        # Set LR for this step
        lr = get_lr(step)
        for g in opt.param_groups: g["lr"] = lr

        with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
            logits = model(x).squeeze(-1)
            loss = torch.nn.functional.binary_cross_entropy_with_logits(logits, y.float())

        opt.zero_grad(set_to_none=True)
        scaler.scale(loss).backward()
        scaler.unscale_(opt)
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)  # (2.7)
        scaler.step(opt)
        scaler.update()
        step += 1

    # Evaluate
    val_loss, val_acc = evaluate(model, val_loader, device)
    if val_loss < best_val - 1e-4:
        best_val = val_loss
        best_state = deepcopy(model.state_dict())
        patience_ctr = 0
    else:
        patience_ctr += 1
    if patience_ctr >= cfg["patience"]: break

model.load_state_dict(best_state)
torch.save(model.state_dict(), "cinematch/checkpoints/mlp_v1.pt")
TrainingQuestion 02 / 07
The script calls scaler.unscale_(opt) before clip_grad_norm_. Why?
AIt's required by PyTorch — clip_grad_norm_ won't work without it.
BGradScaler multiplies the loss by a large scale factor, which means gradients are also scaled. clip_grad_norm_ needs to see the true (unscaled) gradient magnitudes to clip correctly. unscale_() divides gradients by the scale factor so the clipping threshold (1.0) means what we intend.
Cunscale_() converts gradients from BF16 to FP32.
DIt prevents gradient accumulation across steps.
03 — Ablation
The ablation — proving each technique's contribution.

The capstone's real value is in the ablation table: starting from a bare MLP, we add one technique at a time and measure the impact on both training and validation metrics. This is the discipline that separates cargo-cult ML from engineering.

ConfigurationTrain accVal accGapΔ val
Phase 1 logistic regression89.2%89.0%0.2—
Bare MLP 4→64→32→1, ReLU, SGD97.1%90.8%6.3+1.8
+ AdamW (lr=1e-3)96.4%91.3%5.1+0.5
+ Weight decay 0.01 (param groups)95.1%92.6%2.5+1.3
+ Dropout p=0.294.2%93.1%1.1+0.5
+ BatchNorm94.5%93.4%1.1+0.3
+ GELU (replace ReLU)94.3%93.5%0.8+0.1
+ Cosine warmup schedule93.9%93.6%0.3+0.1
+ Early stopping (patience 10)93.8%93.8%0.0+0.2
+ BF16 AMP + grad clip93.8%93.8%0.00.0

Read the table carefully. Four observations dominate:

1. The bare MLP overfits massively. Train 97.1% vs val 90.8% — a 6.3-point gap. It beats Phase 1's logistic regression on val (90.8 vs 89.0), but barely. The capacity is there; the regularization is not.

2. Weight decay is the single biggest regularizer. +1.3 points on val, gap closes from 5.1 to 2.5. Free — just a config flag in the optimizer.

3. Each technique contributes, but with diminishing returns. The first three additions (AdamW, weight decay, dropout) account for ~2.3 of the 2.8-point total improvement. The last four (BN, GELU, schedule, early stopping) add only ~0.7 combined. This is why the ablation matters — it tells you when to stop adding complexity.

4. BF16 AMP + grad clip add no accuracy. They're speed and stability techniques, not accuracy techniques. But they reduce training time by ~30% and prevent rare gradient explosions. In a production training script, they're non-negotiable.

The ablation discipline

Never ship a model with techniques you haven't individually validated. Every technique in the capstone MLP is there because the ablation proves it helps. If you add Mixup and val accuracy doesn't improve, take it out — unnecessary complexity is a liability in production (more code to maintain, more failure modes to debug, no benefit).

Ablation readingQuestion 03 / 07
Looking at the ablation, a colleague argues: "GELU only adds +0.1 over ReLU — not worth the complexity." Are they right?
AYes — always use ReLU because it's simpler.
BNo — GELU is always superior to ReLU.
CThe comparison is unfair because GELU has more parameters.
DIt depends on context. For this small CineMatch MLP, +0.1 is within noise — the colleague has a point. But GELU adds zero code complexity (replace nn.ReLU() with nn.GELU()) and zero runtime cost on modern hardware. For Phase 3+ CNNs and Phase 4 transformers, GELU's smoother gradient profile matters more. The principle: if the cost is zero and the benefit is non-negative, include it.

CineMatch Integration

This MLP is the production tabular model.

The capstone MLP serves as CineMatch's tabular feature scorer. It takes the 4 hand-engineered features (watch_time_norm, genre_match, recency, popularity) and produces a "will rate 4+" probability. It runs alongside the poster CNN (Phase 3) and the transformer recommender (Phase 4) in the ranking stage — each model contributes a feature to the final ranking ensemble. The MLP is the fastest of the three (sub-millisecond inference), serves as a fallback when the transformer is slow or unavailable, and its simplicity makes it the easiest to debug in production.

05 — What's next
What the capstone proves — and what comes next.

The Phase 2 capstone demonstrates three principles that carry through the rest of the track:

1. Depth beats width when properly regularized. A 2-hidden-layer MLP with 2,625 params beats a linear model with 5 params — not because it's bigger, but because it can compose features nonlinearly (the Universal Approximation Theorem from 2.1). Regularization ensures the extra capacity captures real patterns, not noise.

2. Every technique must justify its inclusion via ablation. The ablation table is the engineering discipline. If a technique doesn't improve val metrics, it doesn't ship. This discipline scales: in Phase 3 we ablated transfer learning; in Phase 4 we'll ablate pretraining.

3. The training script is reusable infrastructure. The capstone's train_mlp.py pattern — AdamW + param groups + cosine warmup + AMP + early stopping + grad clip — is the same template used for Phase 3's CNN, Phase 3's LSTM, and Phase 4's transformer. Write it once, validate it once, reuse everywhere.

Production inference cost — why 2,625 parameters is "free"
Model size: 2,625 params × 4 bytes (FP32) = 10.5 KBFits in L1 cache of any CPU. No GPU needed for inference.
Forward pass: layer 1 = 4×64 = 256 multiply-adds. Layer 2 = 64×32 = 2,048. Output = 32×1 = 32. Total: 2,336 FLOPsA single CPU core at 10 GFLOPS does this in 0.23 microseconds.
Batch of 500 candidates (ranking stage): 500 × 2,336 = 1.17M FLOPsStill < 0.12 ms on CPU. The MLP scoring is invisible in the 14.5ms pipeline budget from Module 3.5.
Compare: Phase 4 transformer scores 500 candidates in ~4ms. Phase 3 CNN extracts one poster embedding in ~2ms.The MLP is 30× faster than the CNN and 3000× faster than the transformer per inference. That's why it's the fallback model.
