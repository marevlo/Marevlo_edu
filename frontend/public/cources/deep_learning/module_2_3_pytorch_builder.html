The PyTorch builder's guide. Idioms that scale.

You've built an MLP and derived its gradients. Now we cover the scaffolding around models — the idioms that let you compose, initialize, save, and share neural networks at any scale. Every pattern in this module shows up constantly from here forward, including in Phase 3's convolutional and recurrent networks.

01 — The container
nn.Module — the container abstraction.

nn.Module is the base class for every neural network component in PyTorch, from a single Linear layer to a 175-billion-parameter transformer. A module is a self-contained computational unit that takes tensors in, produces tensors out, and manages its own parameters. Modules compose: a module can contain other modules, which can contain more modules, recursively. This is the LEGO-brick principle of deep learning.

Every custom module you write implements exactly two methods:

  1. __init__ — declare the submodules and parameters. This is where the architecture lives.
  2. forward — wire those pieces together into a computation. This is where data flow lives.

The separation is deliberate: architecture can be inspected, printed, serialized, compared, and reused independently of the particular tensors flowing through it. When you print a module, you see the submodule tree — its architecture. When you call it, you run the forward pass.

import torch
import torch.nn as nn

class CineMatchMLP(nn.Module):
    def __init__(self, in_features=4, hidden=32, out_features=1):
        super().__init__()     # required: registers module bookkeeping
        self.fc1 = nn.Linear(in_features, hidden)
        self.fc2 = nn.Linear(hidden, out_features)
        self.act = nn.ReLU()

    def forward(self, x):
        return self.fc2(self.act(self.fc1(x)))

model = CineMatchMLP()
print(model)
# CineMatchMLP(
#   (fc1): Linear(in_features=4, out_features=32, bias=True)
#   (fc2): Linear(in_features=32, out_features=1, bias=True)
#   (act): ReLU()
# )

What super().__init__() actually does.

The call to super().__init__() is not optional. It initializes the parent nn.Module's internal registries — the Python dictionaries that track submodules, parameters, and buffers. If you skip it, assigning self.fc1 = nn.Linear(...) later will appear to succeed but none of the framework's magic (parameter tracking, .parameters(), .to(device), state_dict) will work. Every custom module's __init__ must begin with this call.

Forgot super().__init__()?

Your model "runs" but doesn't actually train.

If you forget super().__init__(), the model forward pass will appear to work, but model.parameters() returns an empty list. The optimizer has nothing to update; loss.backward() computes gradients but they're never applied. Training looks like it's running but the weights don't change. The signature: loss is constant across iterations. Fix: add super().__init__() as the first line of __init__.

02 — Parameters & buffers
Parameters — the things that change during training.

Not every tensor stored on a module should be learned. Weights and biases should — they're updated by backprop. Running statistics for batch normalization, positional encodings, and constant lookup tables should not — they're fixed (or updated by other rules). PyTorch distinguishes these with two different tensor types:

TypePurposeUpdated by
nn.ParameterLearnable weights and biasesOptimizer (via gradient)
Buffer (self.register_buffer)Non-learnable state that should move with the modelManual update or not at all

When you assign nn.Parameter(...) as an attribute of a Module, it gets automatically registered — so model.parameters() will find it, optimizer will update it, and state_dict() will save it. Buffers do the same, minus the optimizer updates.

import torch
import torch.nn as nn

class NormalizedLinear(nn.Module):
    """A Linear layer that also stores feature means/stds for normalization."""
    def __init__(self, in_features, out_features, feature_mean, feature_std):
        super().__init__()
        # Learnable — optimizer updates these every step
        self.weight = nn.Parameter(torch.randn(out_features, in_features) * 0.02)
        self.bias = nn.Parameter(torch.zeros(out_features))
        # Non-learnable — moves with the model to GPU but isn't updated
        self.register_buffer("mean", feature_mean)
        self.register_buffer("std", feature_std)

    def forward(self, x):
        x_norm = (x - self.mean) / (self.std + 1e-8)
        return x_norm @ self.weight.T + self.bias

Inspecting parameters.

PyTorch gives you a handful of methods for introspecting what's inside a module. These are routine parts of debugging, checkpoint inspection, and parameter-specific optimizer configuration.

model = CineMatchMLP()

# All learnable parameters (passed to the optimizer)
for p in model.parameters():
    print(p.shape)
# torch.Size([32, 4])  <- fc1.weight
# torch.Size([32])     <- fc1.bias
# torch.Size([1, 32])  <- fc2.weight
# torch.Size([1])      <- fc2.bias

# Named parameters — useful for per-layer configuration
for name, p in model.named_parameters():
    print(f"{name:20s} {p.shape} grad={p.requires_grad}")

# Total parameter count
total = sum(p.numel() for p in model.parameters() if p.requires_grad)
print(f"Total learnable: {total:,}")  # 193

# state_dict — the full snapshot (parameters + buffers), used for saving
for k, v in model.state_dict().items():
    print(f"{k:20s} {v.shape}")

Freezing parameters.

Sometimes you want to train only part of a model. Maybe you're fine-tuning a large pretrained network and only the last few layers should update. The mechanism is requires_grad = False:

# Freeze the first layer
for p in model.fc1.parameters():
    p.requires_grad = False

# Pass only the still-trainable parameters to the optimizer
trainable = [p for p in model.parameters() if p.requires_grad]
optimizer = torch.optim.Adam(trainable, lr=1e-3)

Freezing is central to transfer learning, fine-tuning, and parameter-efficient methods like LoRA. Get comfortable with it now.

03 — Two styles
Sequential vs. functional — two valid styles.

For simple feedforward architectures, PyTorch offers a shortcut: nn.Sequential. It's a container that chains modules together with no custom forward logic — the output of each becomes the input of the next. Compare:

# Long form — subclass nn.Module
class CineMatchMLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(4, 32)
        self.act = nn.ReLU()
        self.fc2 = nn.Linear(32, 1)
    def forward(self, x):
        return self.fc2(self.act(self.fc1(x)))

# Short form — Sequential
mlp = nn.Sequential(
    nn.Linear(4, 32),
    nn.ReLU(),
    nn.Linear(32, 1),
)

# Both have identical behavior for this simple case:
x = torch.randn(8, 4)
print(mlp(x).shape)    # torch.Size([8, 1])

When to use which.

  • Use nn.Sequential when the computation is a linear chain — each layer's output is the next layer's input, no branching, no conditional logic, no reuse of intermediate values.
  • Use nn.Module subclass when you need any flexibility — skip connections, conditional routes, shared weights, reuse of an intermediate in multiple places, or branching logic. The custom forward method can contain arbitrary Python.

Because production architectures almost always need some branching — residual connections in ResNet, attention's query/key/value split in transformers, the encoder-decoder structure in sequence-to-sequence — most real models subclass nn.Module. Sequential is great for blocks within a larger model (a single stack of Linear+ReLU+Dropout, say) but rarely the whole thing.

CineMatch style guide

Use nn.Module subclasses in the main codebase.

Every model in the cinematch/models/ folder should be a named nn.Module subclass — LinearBaseline, CineMatchMLP, CineMatchCNN, etc. This makes them debuggable (you can print them and see a named structure), inspectable (you can iterate over layers by name), and reusable (importable by experiment scripts). Save Sequential for small intra-model blocks (like a "two-layer Linear+ReLU sub-head"). Not just stylistic — it saves hours in debugging, because when something breaks, named layers are much easier to reason about than indexed ones.

04 — Initialization
Initialization — the unlikely make-or-break.

An untrained network's weights have to start somewhere. The distribution you sample them from at initialization has a surprisingly large effect on whether training converges at all, let alone quickly. Bad initialization is one of the easiest ways to turn a perfectly sensible architecture into a non-trainable one.

What goes wrong with naive choices.

  • All zeros. Every neuron in a layer computes the same output and receives the same gradient. The layer acts like a single neuron forever, regardless of its nominal width. Training is permanently stuck in this symmetric state.
  • Too small (e.g. randn * 0.001). Forward-pass activations shrink with each layer, approaching zero. Gradients on the way back also vanish. Nothing learns.
  • Too large (e.g. randn * 5.0). Forward activations grow exponentially with depth. Neurons saturate their activations (sigmoid → 1; ReLU → huge). Gradients explode or vanish unpredictably. Loss is NaN within two steps.

The scaled-initialization principle.

The core idea: choose weight variance so that activations and gradients stay roughly unit-scale throughout the network. For a Linear layer with fan_in inputs, you want the pre-activation variance to be bounded. Two standard schemes:

SchemeVarianceBest for
Xavier / Glorot (2010)2 / (fan_in + fan_out)Sigmoid, tanh
Kaiming / He (2015)2 / fan_inReLU family

Xavier targets both forward-activation variance and backward-gradient variance being roughly 1. Kaiming is a correction for ReLU — since ReLU zeros out roughly half of its inputs, the variance needs to be 2× larger to compensate. This is why you use Kaiming with ReLU and Xavier with sigmoid/tanh. The wrong choice loses a factor of 2 per layer, which across a deep network can destroy training.

Variance propagation — why Kaiming init matters for ReLU networks
Layer 1: fan_in = 4 (CineMatch features). Kaiming: Var(w) = 2/4 = 0.5Weights drawn from N(0, √0.5) ≈ N(0, 0.707)
Pre-activation variance: Var(z₁) = fan_in × Var(w) × Var(x) = 4 × 0.5 × 1.0 = 2.0After ReLU (zeros ~half): Var(h₁) ≈ 2.0 / 2 = 1.0. Unit variance preserved! ✓
With wrong init (Var(w) = 1.0 instead of 0.5): Var(z₁) = 4 × 1.0 × 1.0 = 4.0. After ReLU: 2.0Doubles every layer. After 10 layers: 2¹⁰ = 1024× variance. Activations explode.
With too-small init (Var(w) = 0.01): Var(z₁) = 4 × 0.01 × 1.0 = 0.04. After ReLU: 0.02Halves every layer. After 10 layers: 0.02¹⁰/... ≈ 10⁻¹⁷. Activations vanish to zero.
import torch.nn as nn
import torch.nn.init as init

# PyTorch's defaults for nn.Linear are actually Kaiming-uniform variants,
# which work fine for ReLU networks in most cases.

# Explicit Kaiming init for a ReLU network:
def init_weights(m):
    if isinstance(m, nn.Linear):
        init.kaiming_normal_(m.weight, nonlinearity="relu")
        if m.bias is not None:
            init.zeros_(m.bias)

model = CineMatchMLP()
model.apply(init_weights)   # recursively applies fn to every submodule
The hidden default

PyTorch's built-in layers already use sensible defaults (Kaiming-uniform for Linear and Conv). For simple MLPs with ReLU, you usually don't need to call init.kaiming_normal_ explicitly — the out-of-box behavior is good. Custom init only becomes necessary when you're using unusual activations, building very deep networks, or deliberately trying to match a reference implementation's initialization. The key skill is knowing when your diagnostics (vanishing/exploding activations) indicate that you should override the default.

05 — Custom layers
Custom layers — when built-ins aren't enough.

PyTorch ships most of the layers you'll need. But eventually you'll hit a novel architecture, a research paper's quirky component, or a domain-specific operation — and you'll want to write your own layer. The mechanics are exactly the same as writing a model: subclass nn.Module, declare parameters in __init__, wire up the computation in forward.

A layer without parameters.

Some layers are pure transformations — they modify their input deterministically without any learned state. These are trivial:

class CenteredLayer(nn.Module):
    """Subtract the batch mean from each input."""
    def forward(self, x):
        return x - x.mean(dim=0, keepdim=True)

# Use it anywhere a layer is expected
block = nn.Sequential(
    nn.Linear(4, 32),
    CenteredLayer(),
    nn.ReLU(),
    nn.Linear(32, 1),
)

There's no __init__ needed if you're not declaring parameters or submodules. If you do need state, add an __init__ and remember super().__init__().

A layer with parameters.

Here's a more interesting example — a learnable "gating" layer that scales each input feature by a learned weight, with no bias. You could achieve this with an element-wise multiplication, but wrapping it in a layer makes it reusable, inspectable, and part of the module tree:

class CineMatchFeatureGate(nn.Module):
    """Learns a per-feature multiplicative gate in [0, 1]."""
    def __init__(self, num_features):
        super().__init__()
        # Initialize near 1 (no gating) so the layer starts as near-identity
        self.logit = nn.Parameter(torch.zeros(num_features))

    def forward(self, x):
        gate = torch.sigmoid(self.logit)   # [num_features]
        return x * gate                     # broadcasts over batch dim

# Used as part of a larger model
class GatedMLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.gate = CineMatchFeatureGate(num_features=4)
        self.fc1 = nn.Linear(4, 32)
        self.fc2 = nn.Linear(32, 1)

    def forward(self, x):
        x = self.gate(x)
        h = torch.relu(self.fc1(x))
        return self.fc2(h)

After training, you can inspect model.gate.logit, apply sigmoid, and see which features the model learned to rely on (gate ≈ 1) versus ignore (gate ≈ 0). This kind of interpretability is a gift; custom layers are easy to build and easy to reason about.

Lazy initialization — skip the dimension bookkeeping.

One annoying part of model design is calculating input dimensions for each layer. If you have a stack of Linear → Conv → Pool → Linear, you have to compute the spatial output shape of Conv+Pool to know the input size of the next Linear. PyTorch's lazy modules fix this — they infer their input dimensions from the first forward pass:

# No input dim specified — inferred from the first forward pass
model = nn.Sequential(
    nn.LazyLinear(32),
    nn.ReLU(),
    nn.LazyLinear(1),
)

x = torch.randn(8, 4)
print(model(x).shape)  # Now dims are set — first LazyLinear is Linear(4, 32)

Use lazy modules during prototyping. Before finalizing a model, replace them with explicit dimensions so the architecture is fully documented in the code.

06 — Save, load, refactor
Save, load, and the final CineMatch refactor.

Training a model you can't persist is training in vain. PyTorch's serialization is based on state dicts — ordered dictionaries mapping parameter names to tensor values. This is simpler and more portable than pickling the entire model object, which is why it's the recommended practice.

# Save
torch.save(model.state_dict(), "cinematch_mlp.pt")

# Load — must recreate the model first, then load weights into it
model = CineMatchMLP(hidden=32)
model.load_state_dict(torch.load("cinematch_mlp.pt"))
model.eval()  # disable dropout, freeze batch norm stats

The two-step load pattern (construct empty model → load weights into it) is a feature, not a bug. It means the architecture must be available as code — reproducibly — for the checkpoint to be useful. No mystery serialization, no deprecation breakage across PyTorch versions, no loading 500 MB of Python closures by accident.

Checkpointing for training resumption.

For long training runs, you'll want to save more than just weights — you'll want the optimizer state (Adam's momentum estimates, learning rate scheduler's step count), the epoch number, and possibly random number generator state. The convention:

# Save a full checkpoint
torch.save({
    "epoch": epoch,
    "model_state": model.state_dict(),
    "optimizer_state": optimizer.state_dict(),
    "scheduler_state": scheduler.state_dict(),
    "best_val_acc": best_val_acc,
}, "checkpoint.pt")

# Resume from checkpoint
ckpt = torch.load("checkpoint.pt")
model.load_state_dict(ckpt["model_state"])
optimizer.load_state_dict(ckpt["optimizer_state"])
scheduler.load_state_dict(ckpt["scheduler_state"])
start_epoch = ckpt["epoch"] + 1

The final CineMatch MLP — locked down.

Here's the clean, production-grade version that lives in cinematch/models/mlp.py. It uses every pattern from this module: named subclass, super().__init__(), parameter registration, explicit dims, Kaiming initialization, and dropout-aware mode handling (we'll use dropout properly in Module 2.6).

"""cinematch/models/mlp.py"""
import torch
import torch.nn as nn
import torch.nn.init as init


class CineMatchMLP(nn.Module):
    """MLP classifier: 4-feature input → hidden layers with ReLU → 1 logit.

    Args:
        in_features: number of input features (default 4 for CineMatch)
        hidden_dims: iterable of hidden layer widths
        dropout: dropout probability between layers (0.0 = off)
    """
    def __init__(self, in_features=4, hidden_dims=(64, 32), dropout=0.0):
        super().__init__()
        layers = []
        prev = in_features
        for h in hidden_dims:
            layers.append(nn.Linear(prev, h))
            layers.append(nn.ReLU())
            if dropout > 0:
                layers.append(nn.Dropout(dropout))
            prev = h
        layers.append(nn.Linear(prev, 1))
        self.net = nn.Sequential(*layers)
        self._init_weights()

    def _init_weights(self):
        for m in self.modules():
            if isinstance(m, nn.Linear):
                init.kaiming_normal_(m.weight, nonlinearity="relu")
                init.zeros_(m.bias)

    def forward(self, x):
        return self.net(x)
