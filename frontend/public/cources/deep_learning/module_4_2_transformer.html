The Transformer. Attention is all you need.

Module 4.1 gave us the attention mechanism — a single operation that lets any position query any other. This module turns that mechanism into a complete architecture: stacked layers of multi-head attention and feed-forward networks, held together by residual connections and layer normalization, with positional encoding to supply the order information that attention alone lacks. The result is the transformer — the architecture behind GPT, BERT, ViT, and every production recommender built since 2018, including CineMatch's next-generation ranking model.

01 — Position
The position problem — why attention needs help.

Attention treats its input as a set, not a sequence. The operation softmax(QKT/√d)V is permutation-equivariant — if you shuffle the input positions, the output shuffles in exactly the same way. That's a feature for set-structured data, but a disaster for sequences. In CineMatch, the order a user watched movies matters — watching three horror films in a row means something different from three horror films scattered across months.

The fix: positional encoding. Before feeding the sequence into attention, we add a position-dependent signal to each embedding so the model can distinguish "this is position 3" from "this is position 198."

Sinusoidal positional encoding.

The original transformer uses a deterministic encoding based on sine and cosine waves at different frequencies:

PE(pos, 2i) = sin(pos / 100002i/d)     PE(pos, 2i+1) = cos(pos / 100002i/d) (1.1)

Each dimension oscillates at a different frequency — the low-index dimensions change rapidly (high frequency, encodes fine position), the high-index dimensions change slowly (low frequency, encodes coarse position). The model reads position like a clock with multiple hands spinning at different speeds.

import torch, math

def sinusoidal_pe(max_len, d_model):
    """Create sinusoidal positional encoding matrix (max_len, d_model)."""
    pe = torch.zeros(max_len, d_model)
    pos = torch.arange(max_len).unsqueeze(1).float()
    div = torch.exp(torch.arange(0, d_model, 2).float() * -(math.log(10000.0) / d_model))
    pe[:, 0::2] = torch.sin(pos * div)
    pe[:, 1::2] = torch.cos(pos * div)
    return pe  # (max_len, d_model) — add to input embeddings

Learned positional embeddings — the modern default.

Most models since GPT-2 (2019) use learned positional embeddings: a standard nn.Embedding(max_len, d_model) table, trained end-to-end. Simpler, equally effective in practice. The only downside: the model can't generalize to positions longer than max_len seen during training (sinusoidal has no such limit in theory, though in practice both degrade on unseen lengths).

class TransformerEmbedding(nn.Module):
    def __init__(self, vocab_size, d_model, max_len=512, dropout=0.1):
        super().__init__()
        self.tok_emb = nn.Embedding(vocab_size, d_model)
        self.pos_emb = nn.Embedding(max_len, d_model)  # learned positions
        self.drop = nn.Dropout(dropout)
        self.scale = math.sqrt(d_model)  # scale embeddings before adding PE

    def forward(self, x):
        # x: (B, T) long tensor of token/movie IDs
        T = x.size(1)
        positions = torch.arange(T, device=x.device).unsqueeze(0)
        return self.drop(self.tok_emb(x) * self.scale + self.pos_emb(positions))

02 — Encoder
The encoder block — the transformer's building unit.

The transformer encoder is a stack of identical blocks. Each block has exactly two sub-layers, each wrapped in a residual connection and layer normalization:

x' = LayerNorm(x + MultiHeadAttention(x, x, x))
output = LayerNorm(x' + FFN(x')) (2.1)

Sub-layer 1: multi-head self-attention.

Exactly what we built in 4.1. Q, K, V all come from the same input — each position attends to every other position. The residual connection adds the original input back to the attention output, so the network learns what to add rather than the full representation from scratch. Layer normalization stabilizes the magnitude.

Sub-layer 2: position-wise feed-forward network (FFN).

After attention mixes information across positions, a simple MLP processes each position independently — the same weights applied at every position, like a 1×1 convolution over the sequence dimension:

FFN(x) = ReLU(x · W₁ + b₁) · W₂ + b₂ (2.2)

The inner dimension dff is typically 4× dmodel. For dmodel=256, dff=1024. This is where most of the transformer's parameters live — the FFN has 2 × dmodel × dff parameters per block, while attention has 4 × dmodel² per block. At dff=4d, the FFN is 2× the size of the attention layers.

Why residual + LayerNorm?

This is Phase 2 and 3's lessons applied at scale. The residual connection (Module 3.2: ∂y/∂x = ∂F/∂x + I) ensures gradients flow through 6, 12, or 96 stacked blocks without vanishing. LayerNorm (preferred over BatchNorm because it normalizes per-example, not per-batch — no serving-time mismatch, no batch-size dependency) keeps activations in a stable range.

class TransformerEncoderBlock(nn.Module):
    def __init__(self, d_model, n_heads, d_ff, dropout=0.1):
        super().__init__()
        self.attn = MultiHeadAttention(d_model, n_heads)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff),
            nn.GELU(),  # modern transformers use GELU over ReLU
            nn.Dropout(dropout),
            nn.Linear(d_ff, d_model),
            nn.Dropout(dropout),
        )
        self.ln1 = nn.LayerNorm(d_model)
        self.ln2 = nn.LayerNorm(d_model)
        self.drop = nn.Dropout(dropout)

    def forward(self, x, mask=None):
        # Sub-layer 1: self-attention + residual + norm
        x = self.ln1(x + self.drop(self.attn(x, mask=mask)))
        # Sub-layer 2: FFN + residual + norm
        x = self.ln2(x + self.ffn(x))
        return x
Pre-norm vs post-norm

The original paper placed LayerNorm after the residual add (post-norm). Most modern implementations place it before the sub-layer (pre-norm): x = x + Sublayer(LayerNorm(x)). Pre-norm trains more stably for deep stacks (24+ layers) because the residual stream flows through unnormalized, preserving gradient magnitude. Our implementation above uses post-norm for clarity; production code typically uses pre-norm.

Encoder architectureQuestion 02 / 07
In a transformer encoder block, the FFN applies the same weights at every position. Why not use different FFN weights per position?
APyTorch doesn't support per-position linear layers.
BPer-position weights would make the model too small.
CSharing FFN weights across positions is the same principle as convolution (weight sharing) and recurrence (same cell applied at every timestep). It gives the model a fixed parameter budget regardless of sequence length, enables arbitrary-length inputs, and regularizes by forcing the FFN to learn transformations that are useful at any position rather than memorizing position-specific patterns.
DThe positional encoding already encodes all position-dependent information.
03 — Decoder
The decoder block — autoregressive generation.

The encoder processes an entire input sequence in parallel. The decoder generates output one token at a time, using three sub-layers per block:

Sub-layer 1: masked self-attention. The decoder attends to its own prior outputs, but with a causal mask that prevents attending to future positions. When predicting position t, the model can only see positions 1 through t−1. This is enforced by setting future entries in the attention score matrix to −∞ before softmax, so their weights become zero.

def causal_mask(T, device):
    """Upper-triangular mask: 1 = attend, 0 = block."""
    return torch.tril(torch.ones(T, T, device=device)).unsqueeze(0).unsqueeze(0)
    # shape: (1, 1, T, T) — broadcasts over batch and heads

Worked example — causal masking on a 4-position sequence.

Causal mask for T=4, showing how attention scores are zeroed
Raw attention scores S = QK^T/√d (before masking): [[1.2, 0.8, −0.3, 0.5],
 [0.4, 1.5, 0.2, −0.1],
 [0.7, 0.3, 1.8, 0.9],
 [0.2, 0.6, 0.4, 2.0]]
Causal mask M = tril(ones(4,4)): [[1, 0, 0, 0],
 [1, 1, 0, 0],
 [1, 1, 1, 0],
 [1, 1, 1, 1]]
Masked scores: where M=0, set to −∞: [[1.2, −∞, −∞, −∞],
 [0.4, 1.5, −∞, −∞],
 [0.7, 0.3, 1.8, −∞],
 [0.2, 0.6, 0.4, 2.0]]
After softmax (each row sums to 1, exp(−∞) = 0): Row 0: [1.00, 0, 0, 0] — position 0 can only see itself
Row 1: [0.25, 0.75, 0, 0] — position 1 sees positions 0-1
Row 2: [0.18, 0.12, 0.70, 0] — position 2 sees 0-2
Row 3: [0.07, 0.11, 0.09, 0.73] — position 3 sees all
Position 2's output = 0.18·V₀ + 0.12·V₁ + 0.70·V₂. No contribution from V₃ — it's from the "future."At inference, position 3 doesn't exist yet when predicting position 2. The mask enforces this during training.

Sub-layer 2: encoder-decoder cross-attention. Q comes from the decoder's current state; K and V come from the encoder's output. This is how the decoder "reads" the input. In a translation model: the decoder generating French queries the English encoder. In CineMatch: the ranking decoder queries the user-profile encoder.

Sub-layer 3: position-wise FFN. Same as in the encoder — GELU, dff=4d, residual+norm.

class TransformerDecoderBlock(nn.Module):
    def __init__(self, d_model, n_heads, d_ff, dropout=0.1):
        super().__init__()
        self.self_attn = MultiHeadAttention(d_model, n_heads)
        self.cross_attn = MultiHeadAttention(d_model, n_heads)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff), nn.GELU(), nn.Dropout(dropout),
            nn.Linear(d_ff, d_model), nn.Dropout(dropout),
        )
        self.ln1 = nn.LayerNorm(d_model)
        self.ln2 = nn.LayerNorm(d_model)
        self.ln3 = nn.LayerNorm(d_model)
        self.drop = nn.Dropout(dropout)

    def forward(self, x, enc_out, causal_mask=None):
        # Sub-layer 1: masked self-attention
        x = self.ln1(x + self.drop(self.self_attn(x, mask=causal_mask)))
        # Sub-layer 2: cross-attention (Q from decoder, K/V from encoder)
        x = self.ln2(x + self.drop(self.cross_attn(x, context=enc_out)))
        # Sub-layer 3: FFN
        x = self.ln3(x + self.ffn(x))
        return x
Classic bug

Forgetting the causal mask in the decoder.

Without the mask, position t can see position t+1 during training — the model "cheats" by looking at the answer. Training loss drops fast, but the model is useless at inference when future tokens don't exist. This is a data leakage bug with a very specific signature: train loss plummets to near-zero suspiciously fast, and inference quality is terrible.

Causal maskingQuestion 03 / 07
In the causal mask, entry (i, j) is 0 when j > i. What does this accomplish inside the attention computation?
AIt doubles the attention weights on diagonal entries.
BIt removes positional encoding from future positions.
CIt skips the FFN for future positions.
DThe masked positions get score = −∞ before softmax, which makes their attention weight exactly 0 after softmax. Position i therefore receives zero information from any position j > i — it can only attend to itself and earlier positions. This enforces autoregressive (left-to-right) generation.
04 — Full architecture
The full transformer — architecture diagram.
The transformer architecture. Left: N encoder blocks (self-attention + FFN, each with residual + LayerNorm). Right: N decoder blocks (masked self-attention + cross-attention to encoder + FFN). Cross-attention feeds encoder output as K/V into the decoder. The output head produces next-token probabilities.

05 — Decoder-only
Decoder-only transformers — what GPT and CineMatch actually use.

The original transformer is an encoder-decoder model designed for sequence-to-sequence tasks (translation: English in, French out). But the two most common modern patterns are simpler:

PatternArchitectureUse caseExamples
Encoder-onlyStack of encoder blocks, no decoderClassification, embeddingsBERT, sentence transformers
Decoder-onlyStack of decoder blocks (masked self-attn only, no cross-attn)Generation, next-token predictionGPT, LLaMA, CineMatch recommender
Encoder-decoderFull original architectureSeq2seq (translation, summarization)T5, original transformer

For CineMatch's next-movie prediction, decoder-only is the natural fit: given watch-history [m₁, ..., m_T], predict m_{T+1}. Each position attends only to earlier positions (causal mask), exactly like a language model predicting the next word. No encoder needed — the input is the history.

class CineMatchTransformer(nn.Module):
    """
    Decoder-only transformer for next-movie prediction.
    Replaces the Phase 3 LSTM recommender.
    """
    def __init__(self, vocab=50_000, d=256, heads=8, layers=4,
                 d_ff=1024, max_len=200, dropout=0.1):
        super().__init__()
        self.embed = TransformerEmbedding(vocab, d, max_len, dropout)
        self.blocks = nn.ModuleList([
            TransformerEncoderBlock(d, heads, d_ff, dropout)
            for _ in range(layers)
        ])
        self.ln_f = nn.LayerNorm(d)
        self.head = nn.Linear(d, vocab, bias=False)
        # Weight tying (same trick as Phase 3 LSTM)
        self.head.weight = self.embed.tok_emb.weight

    def forward(self, x):
        # x: (B, T) long tensor of movie IDs
        T = x.size(1)
        mask = torch.tril(torch.ones(T, T, device=x.device))
        mask = mask.unsqueeze(0).unsqueeze(0)  # (1, 1, T, T)

        h = self.embed(x)              # (B, T, d)
        for block in self.blocks:
            h = block(h, mask=mask)    # causal self-attention per block
        h = self.ln_f(h)               # final layer norm
        logits = self.head(h)          # (B, T, vocab)
        return logits

The expected Hits@10 improvement: 26% (LSTM) → ~35% (transformer), primarily from better long-range context utilization.

CineMatch Integration

The recommendation transformer in production.

This CineMatchTransformer becomes the retrieval model in the two-stage pipeline from Module 3.5. At serving time: the user's watch-history [m₁, ..., m_T] feeds through 4 transformer blocks with causal masking. The logits at position T give probabilities over all 50K movies. Top-500 candidates go to the ranking stage. Training uses the same teacher-forcing recipe as the LSTM — predict m_{t+1} from [m₁, ..., m_t] at every position — but now all T positions train in parallel, making training ~4× faster wall-clock for the same number of epochs.

Decoder-onlyQuestion 05 / 07
Why does CineMatch use a decoder-only transformer instead of the full encoder-decoder architecture?
ADecoder-only has fewer parameters.
BCineMatch's task is next-movie prediction from a single sequence (the watch-history). There's no separate "source" and "target" sequence — the input and output are the same sequence shifted by one position, exactly like language modeling. The causal mask ensures each position only sees earlier items, matching inference conditions. An encoder would have nothing to encode that the decoder doesn't already have.
CThe encoder-decoder architecture doesn't support attention.
DDecoder-only is always better than encoder-decoder for every task.
06 — Training
Training the transformer — what changes from the LSTM.

The training recipe is almost identical to Phase 3's LSTM loop — teacher forcing with cross-entropy loss, AdamW with cosine schedule, BF16 AMP. Three things change:

1. The learning rate warmup is non-optional.

Transformers are notoriously sensitive to the initial learning rate. The original paper uses a warmup-then-decay schedule where LR increases linearly for the first few thousand steps, then decays. Without warmup, the large initial gradients from random attention weights can blow up training. The LSTM was more forgiving; the transformer is not.

2. Dropout placement.

Dropout in transformers goes in three places: after the attention weights (before the value projection), after each sub-layer (before the residual add), and inside the FFN (between the two linear layers). Typical rate: 0.1 for all three. For CineMatch's moderately-sized data, 0.1 is a good starting point.

3. Parallelism changes training speed dramatically.

The LSTM processes a 200-item sequence in 200 sequential steps. The transformer processes all 200 in one parallel forward pass. On a GPU with enough memory, transformer training is 4–10× faster wall-clock than LSTM training for the same number of epochs — because the GPU's massively parallel compute units are actually utilized instead of sitting idle waiting for the sequential LSTM steps.

Historical note

2017–2020: the transformer takeover.

Within three years of "Attention is all you need" (June 2017), transformers had replaced LSTMs as the dominant architecture in NLP (BERT, October 2018; GPT-2, February 2019), speech recognition (wav2vec, 2019), and were beginning to challenge CNNs in vision (ViT, October 2020). The reason wasn't just better accuracy — it was the parallelism advantage. The same model quality could be reached 10× faster, which meant researchers could iterate 10× more experiments, which meant transformers improved 10× faster than anything else. The architecture won in large part because it was faster to train.

07 — Place in the stack
The transformer's place in the stack.

Step back and see where we are. Phase 0 established foundations. Phase 1 built single-neuron models. Phase 2 built deep networks with the full training stack. Phase 3 specialized for images (CNN) and sequences (LSTM). Phase 4 has now introduced the mechanism (attention, 4.1) and the architecture (transformer, 4.2) that unified both specializations into a single framework.

The transformer is not just another architecture — it's the foundation for every frontier model since 2018. GPT-4, Claude, Gemini, LLaMA, DALL-E, Stable Diffusion, AlphaFold, ViT — all transformers at their core. What varies is scale (number of layers, dmodel, context length), training data, and task-specific heads. The attention mechanism from 4.1 and the block structure from 4.2 are the same in every one of them.

The breakthrough wasn't a new component — it was the composition: remove recurrence, let attention do all the mixing, and the architecture suddenly trains in parallel.
