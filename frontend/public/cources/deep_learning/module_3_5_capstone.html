Phase 3 Capstone. Two production models for CineMatch.

Phase 3 gave us two specialized architectures — CNNs for grid data, LSTMs for sequential data. This capstone turns both into actual CineMatch features: a poster genre classifier built by transfer-learning from a pretrained ResNet-50, and a next-movie LSTM recommender trained on user watch-histories. Both use the full Phase 2 training stack — AdamW with parameter groups, cosine warmup, BF16 AMP, gradient clipping, early stopping. When we ship them, they beat every Phase 1 baseline. And when Phase 4 opens next, we'll finally understand what the transformer fixes in both at once.

01 — What Phase 3 gave us
What Phase 3 gave us.

Before we build, a one-paragraph recap of the three lessons that made this capstone possible.

CNNs (3.1, 3.2). Convolution encodes two structural priors — locality (pixels far apart are weakly related) and translation equivariance (a cat is a cat whether it sits at coordinate 40 or coordinate 200). A single 3×3 kernel with 10 parameters outperforms a fully-connected layer with millions, because those 10 parameters are shared across every spatial position. The architecture history — LeNet → AlexNet → VGG → Inception → ResNet → DenseNet → ViT — is the story of stacking these priors deeper and deeper without the gradient collapsing, culminating in skip connections that made 152-layer networks trainable.

RNNs and gated variants (3.3, 3.4). Recurrence encodes a different prior — temporal parameter sharing. The same cell is applied at every timestep, so the network handles sequences of arbitrary length with a fixed parameter budget. Plain RNNs have vanishing gradients that make long-range dependencies almost unlearnable. LSTMs fix this with a cell-state highway and four learned gates; GRU is a lighter variant with roughly 75% of the LSTM parameter count and comparable performance in most tasks.

The common thread. Both CNNs and RNNs work because they build structure into the architecture. The MLP from Phase 2 is a universal function approximator, but asking it to learn translation equivariance from pixel data by itself is like asking a student to derive calculus from arithmetic — it's possible in theory, useless in practice. Architectural priors are how deep learning stays sample-efficient. Keep this principle in mind — Phase 4 will replace both priors with a more general one (attention), but the lesson stays the same: match the architecture to the data.

Phase 3's one-liner

Convolution is weight-sharing over space. Recurrence is weight-sharing over time. Both turn an intractable MLP into a learnable specialized network — each tuned to the geometry of its data.

02 — Poster genre classifier
Model 1 — Poster classifier.

CineMatch has 142,000 movies in its catalog. Each one has a poster. We want to predict genre from the poster alone — not because we lack genre metadata (we have it), but because the learned poster embedding becomes a powerful feature for the ranking models downstream. A horror poster and a romantic-comedy poster look fundamentally different; a 512-dimensional vector from a vision network captures that difference and can be fed into the recommender as one more feature alongside watch-history, rating, and demographic signals.

Ten genres: action, comedy, drama, horror, romance, sci-fi, documentary, animation, thriller, fantasy. About 8,000 posters per genre after filtering — 80,000 total — split 70/15/15 into train/val/test.

Why transfer learning is non-negotiable here.

A ResNet-50 trained from scratch on 80,000 images will converge, but badly. The problem isn't capacity — ResNet-50 has 25 million parameters, our dataset can support that. The problem is features. Early layers of a good vision model learn edge detectors, color gradients, texture primitives — features that are shared across every vision task ever. A network trained from 80K posters will learn noisier versions of those same features than a network trained on ImageNet's 1.2 million diverse images. Why relearn what has already been learned?

The transfer learning recipe is the standard one: load pretrained weights, freeze early layers, replace the classification head, fine-tune. The only decision is how much to freeze — which we'll ablate.

import torch, torch.nn as nn
from torchvision import models, transforms
from torch.utils.data import DataLoader

class PosterGenreClassifier(nn.Module):
    """ResNet-50 backbone + 10-way genre head for CineMatch posters."""
    def __init__(self, n_genres=10, freeze_backbone=True):
        super().__init__()
        # Pretrained ResNet-50 from torchvision (ImageNet weights)
        backbone = models.resnet50(weights=models.ResNet50_Weights.IMAGENET1K_V2)

        # Strip the ImageNet 1000-way head, keep everything else
        self.features = nn.Sequential(*list(backbone.children())[:-1])  # ends with AvgPool
        feat_dim = backbone.fc.in_features  # 2048 for ResNet-50

        # Freeze backbone? Early training phase = True, later = False
        if freeze_backbone:
            for p in self.features.parameters():
                p.requires_grad = False

        # Small MLP head: 2048 -> 512 -> 10
        # Dropout here, not in backbone (we're not training backbone anyway initially)
        self.head = nn.Sequential(
            nn.Linear(feat_dim, 512),
            nn.ReLU(inplace=True),
            nn.Dropout(0.3),
            nn.Linear(512, n_genres),
        )

    def forward(self, x):
        h = self.features(x).flatten(1)  # (B, 2048)
        return self.head(h)              # (B, 10) — logits

    def unfreeze_layer4(self):
        """After warmup, unfreeze the last ResNet stage for fine-tuning."""
        for p in self.features[-2].parameters():  # layer4 block
            p.requires_grad = True

Data augmentation — aggressive, because posters are small.

80,000 posters is tight for a 25M-parameter network. Augmentation is how we multiply the effective dataset size without actually having more data.

Common mistake

Using training augmentation on validation or test data.

Validation accuracy is meant to measure what the model has learned, not what random crop it happened to see. Always use deterministic transforms at eval time — resize, center crop, normalize. Nothing random. Same rule applies at inference.

The two-phase training recipe.

This is the recipe that consistently wins on transfer learning problems. The idea: the pretrained features are good but not adapted to posters. If we unfreeze the backbone on epoch 1, the large gradients from a random head will destroy the pretrained features before they have a chance to help. Instead, we train the head alone first, then unfreeze and fine-tune everything at a lower learning rate.

# ============ PHASE A: train head only (5 epochs) ============
model = PosterGenreClassifier(freeze_backbone=True).to(device)

# Only head parameters have requires_grad=True
opt = torch.optim.AdamW(
    [p for p in model.parameters() if p.requires_grad],
    lr=1e-3, weight_decay=0.05,
)
sched = torch.optim.lr_scheduler.OneCycleLR(opt, max_lr=1e-3,
                                             total_steps=5*steps_per_epoch)

for epoch in range(5):
    train_one_epoch(model, loader, opt, sched, criterion, scaler)
    evaluate(model, val_loader)

# ============ PHASE B: unfreeze layer4 + head (15 epochs) ============
model.unfreeze_layer4()

# Parameter groups: backbone layer4 gets 10x smaller LR than head
opt = torch.optim.AdamW([
    {'params': model.features[-2].parameters(), 'lr': 1e-5},
    {'params': model.head.parameters(),          'lr': 1e-4},
], weight_decay=0.05)

sched = torch.optim.lr_scheduler.CosineAnnealingLR(opt, T_max=15*steps_per_epoch)

scaler = torch.cuda.amp.GradScaler()
criterion = nn.CrossEntropyLoss(label_smoothing=0.1)

for epoch in range(15):

CineMatch Integration

What the recommender gets.

Beyond the 10-way classification, CineMatch also extracts the penultimate 512-dimensional vector from the head (the output before the final Linear(512, 10)). Every poster in the catalog gets embedded once, stored in a vector database, and queried at ranking time. Similar-looking posters cluster — a cluster of washed-out blues and red text will group horror titles, a cluster of soft pastels will group romantic comedies. This embedding becomes one of ~40 features feeding the ranking transformer in Phase 4.

03 — LSTM recommender
Model 2 — LSTM recommender.

The sequential side. A user has watched movies [m₁, m₂, m₃, ..., m_{t-1}]. Predict m_t. The catalog has V = 50,000 active titles (we restrict to titles with at least 100 views in the past year — the long tail is handled by a separate cold-start model). Watch-windows are capped at 200 movies; users with longer histories are truncated to their most recent 200.

Architecture decisions.

We use a 2-layer stacked LSTM, as built in 3.4 — but with one production refinement that halves the parameter count: weight tying. The embedding matrix (V × H) and the output projection (H × V) are made to share weights. This is a standard trick from language modeling; the intuition is that a movie's embedding and its "probability as a next item given context" logit row are two views of the same object, so they should share parameters.

class CineMatchLSTMRecommender(nn.Module):
    """
    Next-movie prediction: given a sequence of movie IDs, predict the next one.

    Key architectural choices:
      - 2 stacked LSTM layers, dropout between layers only
      - Embedding dim = hidden dim (required for weight tying)
      - Weight tying between input embedding and output projection
          (saves V * H parameters = 50K * 256 = 12.8M params)
    """
    def __init__(self, vocab_size=50_000, hidden=256, num_layers=2, dropout=0.3):
        super().__init__()
        self.vocab_size = vocab_size
        self.hidden = hidden

        # Movie ID -> embedding
        self.embed = nn.Embedding(vocab_size, hidden, padding_idx=0)

        # Stacked LSTM
        self.lstm = nn.LSTM(
            input_size=hidden,
            hidden_size=hidden,
            num_layers=num_layers,
            dropout=dropout if num_layers > 1 else 0,
            batch_first=True,
        )

        # Output projection — will be tied to embedding weights
        self.proj = nn.Linear(hidden, vocab_size, bias=False)

        # Weight tying: share parameters between embed and proj
        self.proj.weight = self.embed.weight  # THIS line saves 12.8M params

        # Init embeddings (tied, so init only once)
        nn.init.normal_(self.embed.weight, mean=0, std=0.02)

    def forward(self, x, lengths=None):
        """
        x: (B, T) long tensor of movie IDs. 0 = padding.
        returns: (B, T, V) logits over next-movie vocabulary.
        """
        e = self.embed(x)                   # (B, T, H)
        h, _ = self.lstm(e)                 # (B, T, H)
        logits = self.proj(h)               # (B, T, V)
        return logits

    def predict_next(self, history, top_k=10):
        """Inference: given [m1,...,mn], return top-k candidates for m_{n+1}."""
        self.eval()
        with torch.no_grad():
            x = torch.tensor(history, dtype=torch.long).unsqueeze(0)
            logits = self.forward(x)[:, -1, :]  # (1, V) — last timestep only
            # Mask out movies already in history (don't recommend what user watched)
            logits[0, history] = float('-inf')
            scores, ids = logits.topk(top_k)
        return ids.squeeze(0).tolist(), scores.squeeze(0).tolist()

Training loop — teacher forcing on watch-sequences.

Training is language-modeling-style teacher forcing: given history [m₁, m₂, ..., m_{T-1}], predict [m₂, m₃, ..., m_T] in parallel at every timestep. One forward pass produces T-1 supervised predictions, so every watch event trains the model.

model = CineMatchLSTMRecommender(vocab_size=50_000, hidden=256).to(device)
opt = torch.optim.AdamW(model.parameters(), lr=1e-3, weight_decay=0.01)
sched = torch.optim.lr_scheduler.CosineAnnealingLR(opt, T_max=20*steps_per_epoch)
criterion = nn.CrossEntropyLoss(ignore_index=0)  # 0 is padding

for epoch in range(20):
    model.train()
    for batch in loader:
        # batch: (B, T+1) where column 0..T-1 is input, 1..T is target
        x = batch[:, :-1].to(device)  # (B, T)
        y = batch[:, 1:].to(device)   # (B, T)

        with torch.autocast(device_type='cuda', dtype=torch.bfloat16):
            logits = model(x)  # (B, T, V)
            # Flatten for cross-entropy: (B*T, V) vs (B*T,)
            loss = criterion(logits.reshape(-1, 50_000), y.reshape(-1))

        opt.zero_grad(set_to_none=True)
        loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        opt.step()
        sched.step()

Evaluation — not accuracy, but ranking metrics.

"Is the top-1 prediction exactly right?" is the wrong question for a recommender. Users don't see one slot — they see a ranked list of 10, 20, or 50 candidates. So we evaluate with ranking metrics:

Hits@k. Fraction of test sequences where the true next-movie appears somewhere in the top-k predictions. Hits@10 = 0.26 means the right answer is in the top-10 list 26% of the time.

MRR (Mean Reciprocal Rank). Average of 1/rank, where rank is the position of the true movie in the sorted predictions. If true movie ranks 1st → contributes 1.0. If 5th → contributes 0.2. Rewards getting the right answer closer to the top.

NDCG@k (Normalized Discounted Cumulative Gain). Like MRR but weights multiple correct items (used when ground truth is a set, not a single next item).

What to notice

The LSTM beats Phase 1's linear baseline on every metric by 2–3×. But Phase 4's transformer will beat this by another 2–3× on long watch-histories — because attention handles length-200 context in a way the LSTM's hidden state simply can't. The LSTM's useful context is ~100 items; the transformer's is the full 200.

04 — Integration
Integration — two-stage pipeline.

Neither model is the "recommender" on its own. Production CineMatch combines both into a two-stage pipeline:

Two-stage CineMatch recommender. Stage 1 (LSTM) narrows 50K titles → 500 candidates using watch-history sequence. Stage 2 (Phase 4 transformer) ranks those 500 using ~40 features, one of which is the poster CNN's 512-dim embedding.

This two-stage pattern — retrieval then ranking — is standard across every production recommender: YouTube, Spotify, streaming platforms generally. The retrieval stage is fast but approximate (must shortlist from a 50K+ catalog in <10ms); the ranking stage is slow but precise (~100ms to re-score 500 candidates with rich features). Our LSTM handles retrieval, our CNN contributes one feature to ranking, and Phase 4's transformer will replace the LSTM in retrieval and form the core of the ranking model.

Production failure mode

The poster CNN embedding cache goes stale after a catalog update.

The poster CNN runs offline (batch inference on all 50K posters, store 512-d vectors in Redis). When new movies are added to the catalog, their poster embeddings don't exist in the cache — the ranking model receives a zero vector, and the new movie gets systematically under-ranked. Symptoms: new movies never appear in recommendations for the first 24 hours after release. Fix: trigger poster CNN inference on new catalog entries immediately on insert, or use a fallback embedding (average of genre neighbors) until the CNN result is cached. This "stale feature" bug is one of the most common production ML failures.

CineMatch Integration

The hand-off to Phase 4.

Today: LSTM retrieves 500 candidates, a gradient-boosted tree ranks them, we ship Hits@10 ≈ 26%. In Phase 4, we replace the LSTM with an attention-based retrieval model (Hits@10 → ~35%) and replace the GBT with a ranking transformer that handles all 40 features jointly, including the poster embedding (final recommendation quality +15-20% lift in watch-completion A/B tests). Every piece we built in Phase 3 gets upgraded — but the patterns (sequence modeling, visual features, two-stage ranking) stay exactly the same.

06 — Phase 3 complete
Phase 3 — complete.

Take stock. Five modules ago, we started Phase 3 with a question: the MLPs from Phase 2 work beautifully on tabular data — why aren't they enough? The answer, stretched across five modules, was that real-world data has structure, and architectures that ignore that structure pay for it with sample efficiency they can't afford.

We built two of the most important specialized architectures end-to-end. For images: convolution's locality and translation equivariance, stacked deeper and deeper through AlexNet, VGG, and Inception, finally made trainable at depth 152 by ResNet's skip connections. For sequences: recurrence's temporal parameter sharing, rescued from the vanishing gradient by LSTM's gating mechanism. And in this capstone, we turned both into actual CineMatch features — a poster classifier that beats training-from-scratch by 32 points, a recommender that beats the Phase 1 linear baseline by 12.

Phase 3 — Complete

Structural priors, internalized.

CNNs encode spatial structure. RNNs encode temporal structure. Both win because they match the geometry of the data — and both have sharp edges (CNNs struggle with long-range spatial dependencies; LSTMs struggle with contexts past ~100 tokens). These sharp edges are exactly what Phase 4 resolves.

Phase 4 opens with attention — a mechanism that lets a network directly query any prior position, bypassing both CNN's locality limitation and LSTM's sequential bottleneck. Stacked attention layers become the transformer, the architecture behind every large model shipped since 2018 — GPT, BERT, CLIP, ViT, and every modern recommender at scale. We'll build one from scratch (4.1 attention, 4.2 full transformer block), then pretrain word embeddings (4.3 word2vec), then scale to pretrained language models (4.4), then explore generative models (4.5 GANs). Phase 5 will close with uncertainty quantification using Gaussian Processes.

What changes is the architecture. What stays exactly the same is everything else: backprop, AdamW, cosine schedules, BF16 AMP, parameter groups, gradient clipping, dropout, weight decay, the train/val/test split, the evaluation metrics. Every piece of infrastructure from Phases 0 through 2 carries forward. Phase 3 was about specializing the architecture; Phase 4 is about generalizing it to one architecture that handles almost any sequence task — at scale, in parallel, with context lengths the LSTM couldn't dream of.

Historical note — the crossover

2012–2017, the five years between two revolutions.

AlexNet (2012) made CNNs the unquestioned approach for vision. LSTMs, published in 1997, dominated sequence modeling from 2014 onward through ELMo and seq2seq translation. Both felt permanent. And then in 2017, 'Attention is all you need' showed that for language, attention alone — without any recurrence — outperformed every LSTM benchmark while training 10× faster in parallel. By 2020, ViT had done the same for vision: a transformer with image patches as tokens, no convolutions at all, matched the best ResNets. The specialized architectures didn't disappear — they're still optimal for their narrow domains, on-device deployment, and data-scarce regimes — but the frontier moved decisively to attention. You're about to see why.

The next module opens Phase 4. Attention from first principles, no transformer yet — just the attention operation itself, why it has the form softmax(QK^T/√d)V, what the three projections (query, key, value) semantically represent, and how to implement it in ~30 lines of PyTorch. That's 4.1.
