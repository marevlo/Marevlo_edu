From perceptron to MLP. Depth, finally.

Phase 1 ended with an impossibility proof — no single neuron can represent XOR. This module shows the escape route: stack neurons into layers, insert a non-linear activation between them, and the model's representational power explodes. By the end you'll build a multi-layer perceptron on CineMatch and watch it break through the linear baseline's ceiling.

01 — Affine collapse
Why stacking linear layers isn't enough.

Before we add anything new, let's understand precisely why the obvious idea — "just stack more nn.Linear layers together" — does not work. This proof takes one line of algebra and saves you from a class of architectural mistakes that new practitioners make constantly.

Consider a two-layer stack with no activation between them:

h = W1 x + b1,   y = W2 h + b2 (1.1)

Substitute:

y = W2 (W1 x + b1) + b2 = (W2 W1) x + (W2 b1 + b2) (1.2)

Let W' = W2 W1 and b' = W2 b1 + b2. Then y = W' x + b'. A single linear layer. The composition of two linear functions is itself a linear function, and we've burned a bunch of parameters on a detour that provably cannot represent anything richer than what a single layer already could.

The central architectural rule

Two layers of the same kind of function, stacked, give no more expressive power than one — as long as the function class is closed under composition. Linear functions are closed under composition. So are affine maps. So are polynomials up to fixed degree. The only way to escape this trap is to insert a function between layers that isn't in the closed class — in our case, a non-linear activation. That's the entire content of Phase 2's architecture: Linear → non-linearity → Linear → non-linearity → ... where the non-linearity is what makes depth meaningful.

The minimum functional pattern.

The smallest meaningful "deep" network is two linear layers with a single non-linearity between them:

y = W2 · σ(W1 x + b1) + b2 (1.3)

where σ is some non-linear function applied element-wise. This is a multi-layer perceptron (MLP) with one hidden layer. Despite its simplicity, (1.3) can approximate any continuous function if you make the hidden layer wide enough — a theorem we'll state precisely in Section 4.

02 — The hidden layer
The hidden layer — and what it learns.

A hidden layer is any layer between the input and the output. It is "hidden" only in the sense that its values are not directly observed — we supervise the output, not the intermediate computations. What happens in the hidden layer is up to the network to figure out from the loss signal.

Geometrically, the hidden layer performs a representation change. It takes the input and projects it into a new feature space — one where the problem becomes linearly separable (in classification) or linearly fittable (in regression). The output layer is then just a linear model applied to those new features.

CineMatch's Phase 2 problem

"Will the user rate this 4+ stars?"

Throughout Phase 1, we asked two questions about CineMatch data — watch fraction (regression) and finished-or-not (classification). Both capped out because the underlying relationship between features and target is non-linear. Phase 2 asks a third, richer question: given the user's profile and the movie's attributes, will they rate it 4 stars or higher? This is a binary classification problem, but the decision surface involves genuine interaction terms — taste compatibility, attention-runtime fit, genre preferences. A linear model plateaus early; an MLP with one hidden layer closes most of the gap.

The MLP architecture, spelled out.

An MLP with one hidden layer of h neurons, taking d features and producing K outputs:

hidden: h = σ(W(1) x + b(1))     h ∈ ℝh (2.1)
output: y = W(2) h + b(2)     y ∈ ℝK (2.2)

where W(1) ∈ ℝh×d, W(2) ∈ ℝK×h, and σ is applied element-wise. Every parameter — all of W(1), b(1), W(2), b(2) — is learned by gradient descent. The architecture — the dimensions, the count of layers, the choice of activation — is picked by you before training. These architectural choices are hyperparameters.

Forward pass through CineMatch MLP: 4 → 64 → 32 → 1
Input: x = [0.6, 0.9, 0.2, 0.7] — one user-movie pair, 4 featuresShape: (4,) — or (B, 4) for a batch
Layer 1: z₁ = W₁·x + b₁. W₁ ∈ ℝ^{64×4}, b₁ ∈ ℝ^{64}. z₁ ∈ ℝ^{64}Params: 64×4 + 64 = 320. Each of 64 neurons computes a dot product with the 4 features.
Activation: h₁ = ReLU(z₁). 64 values → keep positives, zero out negatives.If z₁ = [1.2, −0.3, 0.8, ...], then h₁ = [1.2, 0.0, 0.8, ...]. About 50% are zero (typical).

What do the hidden units represent?

This is a question people ask a lot, and the answer is honest: whatever the gradient signal incentivizes. There is no prescription — the network learns whatever feature combinations reduce the loss fastest given the architecture and data. On CineMatch, one hidden unit might end up responding to "long attention span + long runtime" (a match detector), another to "high critic score + young user" (a popularity detector), and others to combinations we wouldn't have named. We don't get to choose; we get to shape the architecture so that some combination of useful features is learnable.

This non-interpretability is a cost of the power. Linear regression is fully interpretable — each weight tells you exactly how much one feature affects the output. An MLP's weights are not individually meaningful; only the network's behavior as a whole matters. Tools for interpreting trained networks (attribution, probing, mechanistic interpretability) are active research areas, and none of them recover the clean per-feature meaning of a linear model.

03 — Activations
Activation functions — three choices and why ReLU won.

The non-linearity between layers can be almost any differentiable function, but three choices dominate. Each has its shape, its derivative, and its historical moment.

Sigmoid.

σ(z) = 1 / (1 + e−z)   σ′(z) = σ(z)(1 − σ(z)) (3.1)

Smooth, output in (0, 1), the natural choice in the 1980s. But it has a brutal flaw: when |z| is large, σ′(z) is near zero. Stacked across many layers, the gradient vanishes. Training deep sigmoid networks was famously difficult throughout the 1990s — this vanishing-gradient problem was one of the reasons the field stalled.

Tanh.

tanh(z) = (ez − e−z) / (ez + e−z)   tanh′(z) = 1 − tanh²(z) (3.2)

A shifted, scaled sigmoid: output in (−1, 1), zero-centered. Zero-centered outputs often help gradient flow (successive layers don't need to cancel a mean), so tanh was preferred over sigmoid for hidden layers through the 1990s and early 2000s. But it has the same vanishing-gradient problem in saturation.

ReLU — rectified linear unit.

ReLU(z) = max(0, z)   ReLU′(z) = { 1 if z > 0, 0 if z < 0 } (3.3)

The 2010 innovation that changed everything. ReLU is piecewise linear — no exponentials to compute, no saturation in the positive regime. For z > 0, its gradient is exactly 1, so the signal flows through unchanged. For z < 0 it is 0, so some neurons become inactive — which turns out to be useful sparsity. ReLU networks trained dramatically faster than sigmoid/tanh networks, enabling the deep architectures that eventually won ImageNet 2012.

The dying ReLU problem

Some units can get stuck outputting 0 forever.

If a ReLU unit's pre-activation z becomes strongly negative, its output is 0 and its gradient is 0. There's no signal to change its weights, so it stays dead. A poorly initialized network can have a significant fraction of its units dead from the start, reducing effective capacity. The modern fix: Leaky ReLU or GELU — variants that have a small non-zero slope in the negative regime, preventing permanent death. For this course we'll use ReLU throughout because its pathologies are instructive; in production, a GELU or SiLU is often slightly better.

04 — Universal approximation
The universal approximation theorem — and its fine print.

Here is a result that sounds like magic:

Universal approximation (Cybenko 1989, Hornik 1991)

A feedforward network with a single hidden layer and a non-constant, bounded, continuous activation function can approximate any continuous function on a compact subset of ℝd, to arbitrary accuracy, given sufficient hidden units.

In plain English: an MLP with one hidden layer — if you make it wide enough — can represent any continuous function. Not linear functions, not polynomials, not any particular family: any continuous function. The architecture you just learned is already maximally expressive.

This is a mathematically strong statement and an oft-misunderstood one. Three important pieces of fine print:

  1. Existence, not findability. The theorem says suitable weights exist; it does not say gradient descent can find them. Learning is a separate problem from representation.
  2. "Sufficient" hidden units can be astronomical. For practical problems, a one-hidden-layer network might need millions of units to approximate what a deeper network does with a few hundred.
  3. Depth is exponentially more parameter-efficient. Deep narrow networks often need dramatically fewer parameters than shallow wide ones to represent the same function. This is the empirical justification for going deep — not expressivity, but efficiency of expressivity.
Why we still go deep

Parameter efficiency is the real argument for depth.

UAT tells us one hidden layer is enough in principle. Modern architectures — MLPs with 6 layers, CNNs with 50 layers, transformers with 100 — are deep because depth compounds representational power: each layer builds on features from the layer before it, producing hierarchies of abstraction. Trying to do the same job with one enormous layer is possible in theory and a disaster in practice — parameter counts explode, optimization becomes hopeless, generalization collapses. Phase 3 and 4 of this course build progressively deeper architectures precisely because depth is the efficient path to the expressivity UAT promises.

05 — Build it
Your first MLP — beating the Phase 1 baseline.

Time to build. We'll use the same four-line training loop from Phase 1; the only change is the model definition itself. And we'll evaluate against the linear baseline you built in Phase 1's capstone — that's the whole point. Any improvement is attributable to the new hidden layer.

import torch
import torch.nn as nn
from torch.utils.data import DataLoader, random_split
from cinematch.data.dataset import CineMatchClassDataset
from cinematch.utils.seed import set_seed
from cinematch.utils.device import get_device


class CineMatchMLP(nn.Module):
    """One hidden layer, ReLU, logits output. 4 → 32 → 1."""
    def __init__(self, hidden=32):
        super().__init__()
        self.fc1 = nn.Linear(4, hidden)
        self.fc2 = nn.Linear(hidden, 1)
        self.act = nn.ReLU()

    def forward(self, x):
        h = self.act(self.fc1(x))
        return self.fc2(h)   # return logits; loss applies sigmoid


set_seed(42); device = get_device()
ds = CineMatchClassDataset()
train_ds, val_ds = random_split(ds, [0.8, 0.2])
train_loader = DataLoader(train_ds, batch_size=256, shuffle=True)
val_loader = DataLoader(val_ds, batch_size=256)

model = CineMatchMLP(hidden=32).to(device)
criterion = nn.BCEWithLogitsLoss()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)   # Adam in Module 2.4

for epoch in range(20):
    model.train()
    train_loss, train_correct, n = 0.0, 0, 0
    for xb, yb in train_loader:
        xb, yb = xb.to(device), yb.to(device).unsqueeze(1)
        optimizer.zero_grad()
        logits = model(xb)
        loss = criterion(logits, yb)
        loss.backward()
        optimizer.step()
        train_loss += loss.item() * xb.size(0)
        preds = (torch.sigmoid(logits) > 0.5).int()
        train_correct += (preds == yb.int()).sum().item()
        n += xb.size(0)

    model.eval()
    val_correct, n_val = 0, 0
    with torch.no_grad():
        for xb, yb in val_loader:
            xb, yb = xb.to(device), yb.to(device).unsqueeze(1)
            preds = (torch.sigmoid(model(xb)) > 0.5).int()
            val_correct += (preds == yb.int()).sum().item()
            n_val += xb.size(0)

    print(f"epoch {epoch+1:2d}  "
          f"train_loss={train_loss/n:.4f} train_acc={train_correct/n:.3f}  "
          f"val_acc={val_correct/n_val:.3f}")

What changed compared to Phase 1's logistic regression? Exactly two things:

  1. The model definition. We subclass nn.Module, declare two Linear layers and a ReLU, and wire them together in forward. Two lines of computation instead of one.
  2. The optimizer. We use Adam instead of plain SGD. Adam is a smarter optimizer that adapts per-parameter learning rates — we'll derive it in Module 2.4. For now, treat it as a sensible default that trains MLPs faster than SGD.

Everything else is identical. Same loss, same training loop, same evaluation. This consistency is not an accident — it's the whole reason PyTorch's API is designed the way it is. The architecture plugs into a general training system.
