Backpropagation by hand. What autograd is doing.

You've been calling loss.backward() since Module 0.3 without knowing exactly what happens. This module demystifies it completely: we'll take a 2-2-1 network with specific numbers, compute the forward pass by hand, derive every gradient of the loss with respect to every weight by applying the chain rule carefully, and confirm autograd produces the same numbers. By the end there will be no magic.

01 — Why backprop
Backpropagation is the chain rule, applied carefully.

Backpropagation has a reputation for being mysterious. It shouldn't. Backprop is a single mathematical idea — the chain rule from calculus — applied systematically to a computation graph. Everything novel about the algorithm is about efficiency, not about the math itself.

Recall the chain rule from Module 0.3. If L = f(g(w)), then:

∂L/∂w = (∂L/∂g) · (∂g/∂w) (1.1)

Read it: the gradient of L with respect to w is "how L changes with g" times "how g changes with w." A neural network's loss is a long chain of such compositions — input → linear → activation → linear → activation → ... → loss. Backprop is what you get when you apply (1.1) to every link in that chain, in order, from the loss backward to the weights.

Forward pass, then backward pass.

Training a network has two passes per step:

  1. Forward. Starting from the input, compute every intermediate value up through the loss. Along the way, save any intermediate value you'll need later (so autograd can use them).
  2. Backward. Starting from the loss, propagate gradients backward through the same graph, applying the chain rule at each node. You end up with the gradient of the loss with respect to every parameter.

Then the optimizer takes one step — subtract the gradients scaled by the learning rate — and you repeat. Every neural network's training loop is this cycle.

Why backprop is efficient — reverse-mode autodiff

Computing each parameter's gradient independently would cost O(parameters × forward pass) per example. Backprop computes all gradients with exactly one forward pass and one backward pass — total cost O(2 × forward pass), regardless of how many parameters there are. For a network with billions of weights, this is the difference between computable in milliseconds and infeasible. This efficiency is why backprop unlocked deep learning, even though the algorithm itself was understood (as reverse-mode automatic differentiation) decades before neural networks needed it.

02 — Forward pass
Forward pass on a 2-2-1 network, by hand.

Our working example is the smallest network that still shows the full structure of backprop: 2 inputs, 2 hidden units (sigmoid activation), 1 output. We'll use squared-error loss so every derivative stays clean. You should be able to redo this entire section without looking up anything.

The network.

Two input features x1, x2. Two hidden units with pre-activations z1, z2 and activations h1 = σ(z1), h2 = σ(z2). One output ŷ. Squared error loss L = (ŷ − y)2.

Figure 1. The 2-2-1 network we'll derive backprop for. Weights are labeled by layer and connection (wij(1) is input layer i to hidden unit j; wj(2) is hidden unit j to output). We omit biases to keep the first pass clean; they'll appear in code later.

Forward equations.

z1 = w11(1)x1 + w21(1)x2    h1 = σ(z1)
z2 = w12(1)x1 + w22(1)x2    h2 = σ(z2)
ŷ = w1(2)h1 + w2(2)h2
L = (ŷ − y)2
(2.1)

Numerical example.

Let's plug in concrete numbers so the derivation below is unambiguous.

Inputs & target
x₁, x₂: 0.5, 0.8
y (target): 1.0
Weights — layer 1
w₁₁⁽¹⁾: 0.2
w₂₁⁽¹⁾: −0.4
w₁₂⁽¹⁾: 0.3
w₂₂⁽¹⁾: 0.1
Weights — layer 2
w₁⁽²⁾: 0.6
w₂⁽²⁾: −0.5

Forward computation.

z₁: 0.2·0.5 + (−0.4)·0.8 = 0.1 − 0.32 = −0.22
h₁ = σ(−0.22): 1 / (1 + e0.22) ≈ 0.4452
z₂: 0.3·0.5 + 0.1·0.8 = 0.15 + 0.08 = 0.23
h₂ = σ(0.23): 1 / (1 + e−0.23) ≈ 0.5572
ŷ: 0.6·0.4452 + (−0.5)·0.5572 ≈ 0.2671 − 0.2786 = −0.0115
L = (ŷ − y)²: (−0.0115 − 1)² ≈ 1.0232

Write these numbers down on paper. We'll reference every one of them during the backward pass. Notice: the prediction is close to zero but the target is 1, so the loss is large — the model is wrong in a specific, recoverable way. Our job now is to compute how each parameter should change to reduce this loss.

03 — Backward pass
Backward pass — deriving every gradient.

Now the main event. We'll compute ∂L/∂w for every weight w in the network by walking backward from the loss, applying the chain rule at each step. Take this slowly. The structure repeats and becomes familiar; once you've done it once, you've done it for any network.

Step 1 — Gradient of the loss with respect to the output.

∂L/∂ŷ = 2(ŷ − y) = 2·(−0.0115 − 1) ≈ −2.023 (3.1)

This is the "first derivative of the loss" — how the loss changes as the prediction changes. It's our upstream gradient. We'll propagate it backward through every earlier operation. Call this value δŷ for convenience.

Step 2 — Through the output layer to the hidden activations.

ŷ = w1(2)h1 + w2(2)h2, so:

∂ŷ/∂w1(2) = h1    ∂ŷ/∂w2(2) = h2
∂ŷ/∂h1 = w1(2)    ∂ŷ/∂h2 = w2(2)
(3.2)

By the chain rule:

∂L/∂w₁⁽²⁾: δ_ŷ · h₁ = −2.023 · 0.4452 ≈ −0.9007
∂L/∂w₂⁽²⁾: δ_ŷ · h₂ = −2.023 · 0.5572 ≈ −1.1272
∂L/∂h₁: δ_ŷ · w₁⁽²⁾ = −2.023 · 0.6 ≈ −1.2138
∂L/∂h₂: δ_ŷ · w₂⁽²⁾ = −2.023 · (−0.5) ≈ 1.0115

The gradient pattern for a linear layer is memorable: gradient w.r.t. weight = upstream gradient × input, gradient w.r.t. input = upstream gradient × weight. We just performed this for w(2) and now have gradients for h1, h2 — the new "upstream" values for the next step.

Step 3 — Through the sigmoid back to the pre-activations.

σ′(z) = σ(z)(1 − σ(z)) = h(1 − h). Using our computed h values:

σ′(z₁): 0.4452 · (1 − 0.4452) ≈ 0.2470
σ′(z₂): 0.5572 · (1 − 0.5572) ≈ 0.2467
∂L/∂z₁: ∂L/∂h₁ · σ′(z₁) = −1.2138 · 0.2470 ≈ −0.2998
∂L/∂z₂: ∂L/∂h₂ · σ′(z₂) = 1.0115 · 0.2467 ≈ 0.2496

Step 4 — Through the first linear layer to the input weights.

For z1 = w11(1)x1 + w21(1)x2, the partials are straightforward:

∂z1/∂w11(1) = x1,   ∂z1/∂w21(1) = x2 (3.3)

Applying the chain rule:

∂L/∂w₁₁⁽¹⁾: ∂L/∂z₁ · x₁ = −0.2998 · 0.5 ≈ −0.1499
∂L/∂w₂₁⁽¹⁾: ∂L/∂z₁ · x₂ = −0.2998 · 0.8 ≈ −0.2398
∂L/∂w₁₂⁽¹⁾: ∂L/∂z₂ · x₁ = 0.2496 · 0.5 ≈ 0.1248
∂L/∂w₂₂⁽¹⁾: ∂L/∂z₂ · x₂ = 0.2496 · 0.8 ≈ 0.1997

Done. We now have ∂L/∂w for all six weights. That's the complete backward pass for this network.

The pattern that generalizes

Backprop at every linear layer has exactly two "formulas" to remember: gradient with respect to weights = outer product of upstream-gradient and input. Gradient with respect to input = matrix multiplication of weight-transpose and upstream-gradient. At every activation layer: gradient with respect to pre-activation = upstream-gradient times activation's derivative. That's it. Every deep learning layer-type (convolution, attention, batch norm, embedding) has its own rules but they all look like these two patterns underneath.

The gradient flow, visualized.

Figure 2. Red arrows show the direction of gradient flow — opposite to the forward pass. At each node, the incoming gradient is multiplied by the local Jacobian (∂output/∂input) and passed upstream.

04 — Efficiency
Why backprop is cheap — reverse-mode autodiff.

Backprop is a special case of a more general algorithm called reverse-mode automatic differentiation. Understanding it briefly explains why training huge networks is feasible at all.

Consider a function f: ℝn → ℝ — many inputs, one output. The Jacobian of f is a 1×n row vector containing the partial derivatives ∂f/∂xi. Computing each partial independently ("forward-mode") costs one forward pass per input — n passes total. Computing them by reverse-mode is cheaper: one forward pass + one backward pass, total cost ≈ 2× one forward pass, regardless of n.

Loss functions fit this pattern exactly. The loss takes millions of inputs (all the parameters) and produces one scalar. Reverse-mode is perfectly matched: O(forward pass) time for all gradients. This is why backprop scales.

The matrix view — what autograd is actually doing.

If we treat every layer as a function taking a tensor input and producing a tensor output, backprop is a sequence of vector-Jacobian products (VJPs). Given an upstream gradient vector v and a Jacobian matrix J, the VJP computes vTJ in one step, without ever materializing J explicitly.

For a linear layer y = Wx, the Jacobian ∂y/∂x = W. The VJP vTW is just a matrix-vector product — computable with the same tools as the forward pass. Same for ∂y/∂W. Every layer in PyTorch implements a forward function and a backward function that computes these VJPs, and autograd chains them together.

# What autograd is roughly doing for our 2-2-1 network
# (simplified — real autograd tracks a full computation graph)

import torch

x = torch.tensor([0.5, 0.8])
y = torch.tensor([1.0])

W1 = torch.tensor([[0.2, -0.4], [0.3, 0.1]], requires_grad=True)
w2 = torch.tensor([0.6, -0.5], requires_grad=True)

# Forward
z = W1 @ x
h = torch.sigmoid(z)
y_hat = w2 @ h
L = (y_hat - y) ** 2

# Backward — autograd does chain rule on the graph
L.backward()

print("dL/dW1:", W1.grad)
print("dL/dw2:", w2.grad)

Run this and you'll see the numbers we derived by hand above. (The layout differs slightly because PyTorch stores weights in a slightly different convention — see pitfall below.)

Row-major vs. column-major

PyTorch weights are [out_features, in_features], not [in, out].

In our paper derivation, we wrote z = Wx with W as a 2×2 matrix where row i has weights into hidden unit i. PyTorch follows the same convention — nn.Linear(in=2, out=2) stores its weight as shape [2, 2] where W[i, j] is the weight from input j to output i. The call W @ x in the code above matches our derivation. The confusion arises mainly with reshapes and transposes during debugging — always check .shape when a matrix operation doesn't behave as expected.

Check your understandingQ4 / 08
PyTorch's autograd is said to use "reverse-mode" differentiation. Why is this faster for training than "forward-mode" for a network with many parameters and a scalar loss?
AForward-mode can't handle loops, reverse-mode can.
BReverse-mode uses less memory.
CReverse-mode's cost scales with the number of outputs, and there is exactly one output (the scalar loss) — so reverse-mode computes all input gradients in one backward pass. Forward-mode's cost scales with the number of inputs (millions of parameters), requiring one forward pass per parameter.
DReverse-mode is more numerically accurate.
05 — Autograd check
Autograd confirms our math, to the last decimal.

Let's write out a complete verification script. We'll run the same 2-2-1 network in PyTorch and compare its autograd gradients to our hand-derived values. This is a real practice in ML research — when you implement a novel layer or loss, gradient checking against a numerical or analytical reference is how you prove your backward pass is correct.

import torch

# Our exact inputs, target, and weights from the derivation
x = torch.tensor([0.5, 0.8], dtype=torch.float64)
y_true = torch.tensor([1.0], dtype=torch.float64)

W1 = torch.tensor(
    [[0.2, -0.4],     # row 0: weights into hidden unit 1
     [0.3, 0.1]],      # row 1: weights into hidden unit 2
    dtype=torch.float64, requires_grad=True
)
w2 = torch.tensor([0.6, -0.5], dtype=torch.float64, requires_grad=True)

# Forward pass
z = W1 @ x                  # [z1, z2]
h = torch.sigmoid(z)       # [h1, h2]
y_hat = w2 @ h              # scalar prediction
L = (y_hat - y_true) ** 2     # MSE loss

# Backward pass
L.backward()

print(f"Forward values:")
print(f"  z = {z.tolist()}")
print(f"  h = {h.tolist()}")
print(f"  y_hat = {y_hat.item():.6f}")
print(f"  L = {L.item():.6f}")
print(f"Gradients:")
print(f"  dL/dW1 =\n{W1.grad}")
print(f"  dL/dw2 = {w2.grad}")

Every value matches to within rounding error. That's because autograd is just applying the chain rule, the same way you would on paper — it's doing the same math, mechanically, on a graph it built during the forward pass. The small discrepancies come entirely from the three-decimal rounding we did by hand; with more decimals, the match is exact.

Why this exercise matters

You now trust the machine.

When you run loss.backward() in CineMatch's training loop, you're not trusting a black box — you've just verified the exact algorithm it uses, on a specific example, by hand. That trust generalizes: whatever layer you build later (custom attention, novel loss function, exotic activation), autograd is doing the same thing, the same way. And when your gradients look wrong, you now know how to debug: add print statements, check shapes, compute a small example by hand. Those skills compound throughout this course and your career.

06 — Pitfalls & checkpoint
The bugs backprop actually causes — and the checkpoint.

Backprop is correct when implemented correctly. The failure modes in practice are about its interaction with the rest of your training setup, not errors in the chain rule. Three bugs worth naming:

Vanishing gradients.

In deep networks with saturating activations, each layer's σ′ is small (≤ 0.25 for sigmoid). Across many layers, the gradient shrinks exponentially. By the time it reaches early layers, it's effectively zero, and those layers stop learning. Cures: ReLU-family activations (Module 2.1), careful initialization (Module 2.3), architectures that route signal through residual connections (Phase 3's ResNet).

Exploding gradients.

The opposite failure — gradients grow unboundedly across layers and blow up the parameters. Most common in recurrent networks (Phase 3.3). Cure: gradient clipping — cap the total norm of all gradients to a fixed maximum. A one-line intervention that prevents NaN losses in RNN training.

Dead neurons.

A ReLU unit with strongly negative pre-activation outputs 0 and has gradient 0. Its weights don't change, so it stays dead forever. Cures: careful initialization, Leaky ReLU, monitoring dead-unit rates during training. We'll see diagnostics in Module 2.6.

Check your understandingQ8 / 08
Why do we call optimizer.zero_grad() at the start of every training iteration?
ATo reset the model's weights.
BTo prevent memory leaks.
CPyTorch requires it for thread safety.
