Computational performance. Making training fast.

A CineMatch MLP trains in seconds on a laptop. A production recommender fine-tunes in hours on a single GPU. A foundation model runs for weeks on thousands of GPUs. The difference isn't the algorithm — it's how carefully the code uses the hardware. This module covers the lever that turns a 60-hour run into a 6-hour one: compilation, mixed precision, gradient accumulation, profiling, and multi-GPU training.

01 — Hardware
The hardware picture — compute, memory, bandwidth.

Up to this module we've treated "the GPU" as a magic box that makes matrix multiplications fast. To make training genuinely efficient, we need to understand what that box actually does — and, more importantly, what its bottlenecks are. Almost every inefficient training script is bottlenecked on one of three things, and knowing which one is half the battle.

The three numbers that describe a GPU.

Every accelerator has three headline numbers that determine what workloads it handles well:

  • Peak compute (FLOPs). How many multiply-add operations per second. A modern NVIDIA H100 delivers ~1 PFLOP/s in FP16; a consumer RTX 4090 delivers ~80 TFLOP/s FP16.
  • Memory bandwidth. How many bytes per second can be moved between GPU memory (HBM) and the compute units. H100 ≈ 3 TB/s, RTX 4090 ≈ 1 TB/s.
  • Memory capacity. Total GPU memory — 80 GB on an H100, 24 GB on a 4090. This is what limits model size and batch size.

The key observation: compute grows roughly 10× faster than bandwidth with each GPU generation. In 2013 they were balanced. In 2026, compute dwarfs bandwidth by a factor of ~200× on modern chips. That asymmetry means modern GPUs are often idle waiting for data — the compute units are starved.

The roofline model.

The arithmetic intensity of a kernel is the ratio of FLOPs performed to bytes moved. A matrix multiply of two (N × N) matrices performs 2N3 FLOPs and moves 3N2 · 4 bytes (for FP32) — so intensity scales with N. An element-wise addition of two vectors, by contrast, performs N operations and moves 3N · 4 bytes — intensity is constant regardless of size.

Figure 1. The roofline model. Low-intensity operations (elementwise ops, norms) are limited by memory bandwidth — they sit on the purple slope. High-intensity operations (large matmuls, convs) are limited by compute throughput — they sit on the red roof. The goal of performance engineering is to keep your operations on the compute-bound side of the ridge.
The performance-engineer's lens

When you see a GPU utilization of 30%, you're usually not actually utilizing 30% of its compute. You're bottlenecked on memory bandwidth — the compute sits idle waiting for data. Every technique in this module is about pushing more work into the compute-bound regime: larger batches, smaller data types, fused kernels, and hiding data movement behind compute.

02 — Mixed precision
Mixed precision — the biggest free win.

By default, PyTorch trains in FP32 — 32-bit floating point. Each tensor element is 4 bytes. Modern GPUs support lower-precision formats that use half the memory and run 2× to 10× faster on matmul-heavy kernels. The two relevant ones:

FormatBitsRangePrecisionNotes
FP3232~10−38 to 1038~7 decimal digitsDefault; no performance advantage
FP1616~10−5 to 65504~3 decimal digitsFastest; range issues with small gradients
BF1616~10−38 to 1038~2 decimal digitsWide range; lower precision but fewer range issues

FP16 has the same precision as FP32's mantissa but 1/10000th the range. BF16 matches FP32's range but has lower precision. For deep learning, both usually work but BF16 is easier (no loss scaling needed). Modern A100/H100 GPUs have dedicated BF16 tensor cores.

What breaks — and how we fix it.

Naively training in FP16 usually fails. Gradient values during training span a huge dynamic range, and the smallest ones (~10−8) round to zero in FP16. Those zeroed gradients mean parameters don't update — training stalls.

The fix is loss scaling: multiply the loss by a large constant (e.g., 65,536) before calling backward(). This scales up all gradients uniformly, pushing them into the representable range of FP16. Before the optimizer step, divide gradients back down.

PyTorch's torch.cuda.amp automates all of this. You wrap the forward pass in autocast (automatically uses FP16 where safe, FP32 where needed — think softmax, norms), and wrap the backward+optimizer in GradScaler (handles loss scaling automatically).

import torch
from torch.cuda.amp import autocast, GradScaler

model = model.cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)
scaler = GradScaler()

for x, y in train_loader:
    x, y = x.cuda(non_blocking=True), y.cuda(non_blocking=True)
    optimizer.zero_grad(set_to_none=True)

    # Forward: FP16 where safe, FP32 where not
    with autocast(dtype=torch.float16):
        preds = model(x)
        loss = loss_fn(preds, y)

    # Scale loss → backward → unscale → step
    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()

FP32 vs FP16 vs BF16 — what changes at the bit level
FP32: 1 sign + 8 exponent + 23 mantissa = 32 bits. Range: ±3.4×10³⁸. Precision: ~7 decimal digits.4 bytes per parameter. A 25M-param ResNet-50: 100 MB in FP32.
FP16: 1 sign + 5 exponent + 10 mantissa = 16 bits. Range: ±65,504. Precision: ~3 digits.2 bytes per param → ResNet-50: 50 MB. But max value is only 65K — gradients can easily overflow!
BF16: 1 sign + 8 exponent + 7 mantissa = 16 bits. Range: ±3.4×10³⁸ (same as FP32!). Precision: ~2 digits.2 bytes per param → same memory savings as FP16, but FP32's range → no overflow. This is why BF16 is the default.
AMP memory savings: forward activations stored in FP16/BF16 (half size) + weights in FP32 (master copy). Net effect: ~40% memory reduction → can double batch size → faster training.CineMatch transformer: FP32 batch=32 → BF16 batch=64 at same GPU memory
AMP failure modes

Loss going to NaN is the classic AMP debugging moment.

If loss NaN's a few hundred steps in, you've probably hit an FP16 underflow or overflow. First, try BF16 instead of FP16 (dtype=torch.bfloat16 in autocast, no scaler needed) — it handles range better. If that's not available, ensure loss values and key activations aren't astronomical. Certain ops (log, exp, softmax) are force-upcast to FP32 by autocast; if you've written custom ops outside PyTorch's autocast whitelist, you may need to manually cast to FP32 inside them.

03 — Gradient accumulation
Gradient accumulation — simulating large batches.

Why do we ever want a large batch size? Three reasons. First, larger batches give better gradient estimates (less noise per step), which matters for some workloads. Second, some training recipes are designed for specific batch sizes and don't work with smaller ones. Third, batch normalization needs enough examples per batch for its statistics to be reliable.

The problem: batch size is limited by GPU memory. A batch of 256 images in a deep CNN can easily eat 20 GB — if you only have 8 GB, you can't use that batch. Gradient accumulation solves this: simulate a large batch by accumulating gradients over several small batches, then taking one optimizer step.

effective batch size = per-step batch size × accumulation steps (3.1) 
accumulation_steps = 4  # simulate 4× larger batch
per_step_batch = 32     # what fits in memory
# Effective batch = 128

optimizer.zero_grad()
for i, (x, y) in enumerate(train_loader):
    x, y = x.cuda(), y.cuda()

    with autocast(dtype=torch.bfloat16):
        preds = model(x)
        loss = loss_fn(preds, y) / accumulation_steps  # normalize!

    loss.backward()  # accumulates gradients

    if (i + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)

Two subtleties:

  1. Divide loss by accumulation_steps. Gradients from N mini-batches sum; without dividing, you'd effectively multiply the learning rate by N.
  2. BatchNorm still uses per-step statistics. Gradient accumulation gives you a larger effective batch for optimization but not for BN — BN sees only the per-step batch. If BN statistics need the full effective batch, use LayerNorm instead or use SyncBN across GPUs (discussed in § 6).
When to use gradient accumulation

Large-model training with limited memory. Rare for CineMatch-scale problems. Standard for ViT-L, BERT-large, and most single-GPU fine-tuning of models that don't fit a full batch. Free from a wall-clock perspective (same total FLOPs), but unlocks batch sizes you couldn't otherwise reach.

04 — Profiling
Profiling — finding the actual bottleneck.

All the optimizations above are premature until you know which part of training is slow. Every experienced practitioner has a story about optimizing the wrong thing — spending a week on AMP when the real bottleneck was a stupid DataLoader, or rewriting kernels in CUDA when the model was CPU-bound. Profile first.

The built-in PyTorch profiler.

import torch
from torch.profiler import profile, ProfilerActivity, record_function

model, optimizer, loader = ...  # as usual

with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    schedule=torch.profiler.schedule(wait=1, warmup=1, active=3),
    on_trace_ready=torch.profiler.tensorboard_trace_handler("./logs"),
    record_shapes=True,
    with_stack=True,
) as prof:
    for step, (x, y) in enumerate(loader):
        with record_function("forward"):
            preds = model(x.cuda())
            loss = loss_fn(preds, y.cuda())
        with record_function("backward"):
            loss.backward()
        with record_function("optimizer_step"):
            optimizer.step()
            optimizer.zero_grad()
        prof.step()
        if step >= 5: break

print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=15))

The output is a table showing every operator, how much CPU and CUDA time it took, memory it used, and how often it was called. You open the saved trace in TensorBoard's Profiler plugin or in Chrome's trace viewer for a timeline visualization.

What to look for.

Pattern in the traceDiagnosisFix
Large gap between CUDA kernelsGPU idle waiting for dataMore DataLoader workers, pin_memory, non_blocking transfer
CPU bar saturated, GPU nearly idleCPU is the bottleneckCache preprocessing, simpler augmentation, more workers
Many tiny kernels back-to-backKernel launch overheadtorch.compile to fuse kernels
Long cudaMemcpyAsyncH2D transfer dominatespin_memory=True, non_blocking=True
Big gap at start of each epochData shuffling or loader resetPersistent workers, larger prefetch_factor
The profiler's paradox

The profiler itself slows down your training — especially with_stack=True and record_shapes=True.

Don't compare absolute numbers from a profiled run to an unprofiled one. Use the profiler to find relative bottlenecks, then turn it off and measure wall-clock separately. The schedule(wait=1, warmup=1, active=3) pattern keeps overhead bounded by only profiling a few steps after warmup.

05 — DataLoader
The DataLoader — where most training time actually goes.

For most realistic workloads, the DataLoader is the bottleneck. The GPU is fast; reading image files, decoding JPEGs, applying augmentations, and moving them to the GPU is slow. If that work happens serially with training, the GPU is idle most of the time.

PyTorch's DataLoader has a handful of settings that matter. Getting them right is often a 2-3× speedup.

train_loader = torch.utils.data.DataLoader(
    dataset,
    batch_size=64,
    shuffle=True,
    num_workers=8,           # parallel workers pre-fetch batches
    pin_memory=True,         # faster CPU→GPU transfer
    prefetch_factor=4,       # each worker prepares 4 batches ahead
    persistent_workers=True, # don't respawn workers each epoch
    drop_last=True,          # stable batch size (esp. for BN)
)

What each setting does.

  • num_workers. Number of parallel processes loading data. A common recipe is min(8, 4 × num_gpus) — too few and the GPU waits; too many and you oversubscribe CPU cores. Tune empirically.
  • pin_memory=True. Pages CPU-side batches into non-pageable ("pinned") memory, enabling faster asynchronous transfer to GPU. Essentially free win — always use it for GPU training.
  • prefetch_factor. How many batches each worker prepares ahead of the current one. Default 2 is often too low; 4–8 is common. Higher uses more RAM.
  • persistent_workers=True. Keeps workers alive between epochs instead of respawning. Avoids the big gap at epoch boundaries.
  • non_blocking transfer. Inside the loop: x = x.cuda(non_blocking=True). Combined with pin_memory, this lets the transfer overlap with GPU compute.

When the dataset is small.

For CineMatch's ~200K training examples, the whole dataset might fit in GPU memory as a single tensor. In that case, skip the DataLoader entirely — pre-load everything to GPU once, and slice batches from there. You can't beat zero data-loading overhead.

# Small-dataset shortcut: all-on-GPU
X_all = torch.tensor(X_numpy, dtype=torch.float32, device="cuda")
y_all = torch.tensor(y_numpy, dtype=torch.float32, device="cuda")

for epoch in range(n_epochs):
    perm = torch.randperm(len(X_all), device="cuda")
    for i in range(0, len(X_all), batch_size):
        idx = perm[i : i + batch_size]
        x, y = X_all[idx], y_all[idx]
        # ... training step ...

06 — Multi-GPU
Multi-GPU training — a primer.

When a single GPU isn't enough — either because the model is too big to fit or because training takes too long — you go multi-GPU. The principles generalize all the way up to thousand-GPU runs training foundation models. The basic idea depends on what you're trying to split across GPUs.

The three strategies.

StrategyWhat's splitWhen to use
Data parallelismBatch across GPUs, each has full modelMost cases; speeds up training
Model parallelismLayers across GPUsModel too big for one GPU
Tensor parallelismWeight matrices split across GPUsHuge single layers (LLMs)

For CineMatch and most everyday problems, data parallelism is all you need — and for years it's been the workhorse of deep learning. Each GPU holds a full copy of the model. Each step, the batch is split evenly across GPUs; each GPU does its forward/backward on its slice; then gradients are all-reduced (summed across GPUs so every GPU ends up with the same averaged gradient); finally each GPU runs the same optimizer step locally, keeping all model copies synchronized.

DataParallel vs DistributedDataParallel.

PyTorch has two data-parallel classes. Only one is recommended:

ClassModelRecommended?
nn.DataParallel (DP)Single process, scatter/gather on every stepNo — slow, deprecated for new code
nn.parallel.DistributedDataParallel (DDP)One process per GPU; gradients reduced via NCCLYes — standard for all modern multi-GPU training

DP has one Python process managing all GPUs via threads, which hits the GIL and scatters-gathers every batch. DDP has one process per GPU; inter-GPU communication uses NCCL (NVIDIA's collective communication library), which is dramatically faster. DDP also scales to multiple machines.

A minimal DDP training script.

import os, torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data.distributed import DistributedSampler

def main():
    dist.init_process_group(backend="nccl")
    rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(rank)

    model = MyModel().to(rank)
    model = DDP(model, device_ids=[rank])

    sampler = DistributedSampler(dataset, shuffle=True)
    loader = torch.utils.data.DataLoader(
        dataset, batch_size=64, sampler=sampler,
        num_workers=4, pin_memory=True,
    )

    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)

    for epoch in range(n_epochs):
        sampler.set_epoch(epoch)  # different shuffle per epoch
        for x, y in loader:
            x, y = x.to(rank, non_blocking=True), y.to(rank, non_blocking=True)
            optimizer.zero_grad(set_to_none=True)
            loss = loss_fn(model(x), y)
            loss.backward()  # gradients automatically all-reduced
            optimizer.step()

    dist.destroy_process_group()

if __name__ == "__main__":
    main()

You run this with torchrun:

torchrun --nproc_per_node=4 train.py

That launches 4 processes, one per GPU, each running main(). DDP handles all the gradient-reduction plumbing. DistributedSampler ensures each GPU sees a different slice of the dataset per epoch.

Scaling considerations.

  • Linear LR scaling. When you multiply effective batch size by N (by adding GPUs), multiply learning rate by N. This keeps the per-example update magnitude consistent.
  • Warmup with large batches. Large batches can blow up early in training; a linear LR warmup for the first ~1000 steps stabilizes things.
  • Gradient communication overlap. DDP overlaps backward computation with gradient all-reduce so communication hides behind compute. No code change needed — it's automatic.
  • SyncBN. For BatchNorm with small per-GPU batch, use SyncBatchNorm to compute BN statistics across all GPUs. For CineMatch's MLP with per-GPU batch ≥ 32, this isn't usually needed.
The usual progression

Single GPU + AMP → single GPU + AMP + grad accumulation → multi-GPU DDP + AMP → multi-machine DDP (large-scale). You rarely need more than the first two for problems at CineMatch's scale. All of Phase 2's models train comfortably on one GPU. Phases 3 and 4 start pushing into multi-GPU territory.

07 — Production loop
The production-grade CineMatch training loop.

This is the putting-it-all-together moment. Every concept from Phase 2 appears in the loop below: MLP with BN and dropout (2.1, 2.3, 2.6), AdamW with parameter groups (2.4, 2.6), cosine LR schedule (2.4), AMP with BF16 (this module), gradient accumulation (this module), early stopping (2.6), and the profiler (this module). This is the code you'd put in cinematch/training/train_mlp.py and actually ship.

import os, math, json
from pathlib import Path
from copy import deepcopy
import torch
import torch.nn as nn
from torch.cuda.amp import autocast, GradScaler
from torch.utils.data import DataLoader

from cinematch.models.mlp import CineMatchMLP
from cinematch.data import load_split


def get_param_groups(model, weight_decay):
    decay, no_decay = [], []
    for n, p in model.named_parameters():
        if not p.requires_grad: continue
        if "bias" in n or "bn" in n.lower() or "norm" in n.lower():
            no_decay.append(p)
        else:
            decay.append(p)
    return [
        {"params": decay,    "weight_decay": weight_decay},
        {"params": no_decay, "weight_decay": 0.0},
    ]


class CosineLR:
    """LR schedule: linear warmup then cosine decay."""
    def __init__(self, opt, base_lr, warmup_steps, total_steps, min_lr=1e-6):
        self.opt, self.base, self.warmup = opt, base_lr, warmup_steps
        self.total, self.min_lr, self.step_i = total_steps, min_lr, 0

    def step(self):
        self.step_i += 1
        if self.step_i < self.warmup:
            lr = self.base * self.step_i / self.warmup
        else:
            t = (self.step_i - self.warmup) / max(1, self.total - self.warmup)
            lr = self.min_lr + 0.5 * (self.base - self.min_lr) * (1 + math.cos(math.pi * t))
        for g in self.opt.param_groups:
            g["lr"] = lr
        return lr


def train_one_epoch(model, loader, optimizer, scheduler, scaler, device,
                    accumulation_steps=1):
    model.train()
    total_loss, n = 0.0, 0
    optimizer.zero_grad(set_to_none=True)
    for step, (x, y) in enumerate(loader):
        x = x.to(device, non_blocking=True)
        y = y.to(device, non_blocking=True)

        with autocast(dtype=torch.bfloat16):
            logits = model(x).squeeze(-1)
            loss = nn.functional.binary_cross_entropy_with_logits(logits, y.float())
            loss = loss / accumulation_steps

        scaler.scale(loss).backward()

        if (step + 1) % accumulation_steps == 0:
            scaler.unscale_(optimizer)
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            scaler.step(optimizer)
            scaler.update()
            scheduler.step()
            optimizer.zero_grad(set_to_none=True)

        total_loss += loss.item() * accumulation_steps * x.size(0)
        n += x.size(0)
    return total_loss / n


@torch.no_grad()
def evaluate(model, loader, device):
    model.eval()
    correct, total, total_loss = 0, 0, 0.0
    for x, y in loader:
        x = x.to(device, non_blocking=True)
        y = y.to(device, non_blocking=True)
        with autocast(dtype=torch.bfloat16):
            logits = model(x).squeeze(-1)
            loss = nn.functional.binary_cross_entropy_with_logits(logits, y.float())
        pred = (torch.sigmoid(logits) > 0.5).long()
        correct += (pred == y).sum().item()
        total += y.size(0)
        total_loss += loss.item() * y.size(0)
    return total_loss / total, correct / total


def main(cfg):
    device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
    torch.manual_seed(cfg["seed"])

    train_ds, val_ds = load_split("train"), load_split("val")

    train_loader = DataLoader(train_ds, batch_size=cfg["batch_size"],
                              shuffle=True, num_workers=4,
                              pin_memory=True, persistent_workers=True,
                              drop_last=True)
    val_loader = DataLoader(val_ds, batch_size=256, shuffle=False,
                            num_workers=2, pin_memory=True)

    model = CineMatchMLP(in_dim=4, hidden=(64, 32), p_drop=0.2).to(device)
    optimizer = torch.optim.AdamW(
        get_param_groups(model, weight_decay=0.01), lr=cfg["lr"]
    )

    steps_per_epoch = len(train_loader) // cfg["accum"]
    total_steps = cfg["epochs"] * steps_per_epoch
    scheduler = CosineLR(optimizer, cfg["lr"],
                           warmup_steps=100, total_steps=total_steps)
    scaler = GradScaler()

    best_val = float("inf"); best_state = None; patience_ctr = 0
    history = []

    for epoch in range(cfg["epochs"]):
        train_loss = train_one_epoch(
            model, train_loader, optimizer, scheduler, scaler, device,
            accumulation_steps=cfg["accum"]
        )
        val_loss, val_acc = evaluate(model, val_loader, device)

        if val_loss < best_val - 1e-4:
            best_val = val_loss
            best_state = deepcopy(model.state_dict())
            patience_ctr = 0
        else:
            patience_ctr += 1

        history.append({
            "epoch": epoch, "train_loss": train_loss,
            "val_loss": val_loss, "val_acc": val_acc
        })
        print(f"epoch {epoch:3d} | train {train_loss:.4f} | val {val_loss:.4f} | acc {val_acc*100:.2f}% | patience {patience_ctr}/{cfg['patience']}")

        if patience_ctr >= cfg["patience"]:
            print(f"early stopping at epoch {epoch}; best val = {best_val:.4f}")
            break

    model.load_state_dict(best_state)
    return model, history


if __name__ == "__main__":
    cfg = {
        "seed": 42, "batch_size": 64, "lr": 1e-3,
        "epochs": 100, "patience": 10, "accum": 1,
    }
    model, history = main(cfg)

    out_dir = Path("cinematch/checkpoints"); out_dir.mkdir(exist_ok=True, parents=True)
    torch.save(model.state_dict(), out_dir / "mlp_v1.pt")
    with open(out_dir / "mlp_v1_history.json", "w") as f:
        json.dump(history, f, indent=2)

What's in there (every line is load-bearing):

  • Parameter groups so weight decay only hits weights (not biases or BN).
  • Linear-warmup + cosine-decay LR schedule for stable training.
  • BF16 autocast for the forward and eval passes — free speedup.
  • GradScaler for numerical safety (even with BF16, unscale_ lets us do gradient clipping safely).
  • Gradient clipping at norm 1.0 to prevent rare explosions.
  • Gradient accumulation (accum=1 by default; change the config to simulate bigger batches).
  • Early stopping with patience and model-state restoration.
  • Per-epoch metrics logged to a JSON file alongside the checkpoint — reproducibility matters.
  • DataLoader configured with pin_memory, num_workers, persistent_workers, and non_blocking transfer.
Performance numbers on CineMatch

The cumulative speedup from naive to production.

A naive training script on this CineMatch MLP: ~120s/100 epochs on a laptop GPU. With AMP + proper DataLoader + async transfer: ~45s. The model is small enough that we never actually need DDP or gradient accumulation for CineMatch — but the loop above is ready for them when you hit Phase 3's larger CNN capstone. Writing it once, with all the hooks, means future scale-ups cost zero refactor effort.
