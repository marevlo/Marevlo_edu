Gaussian Processes. Predictions that know what they don't know.

Every model we've built produces a point estimate — "this user will rate this movie 4.2 stars." But how confident is that prediction? Is 4.2 a near-certainty, or a wild guess from sparse data? Gaussian Processes are the principled framework for predictions that come with calibrated uncertainty: a mean prediction and a variance that grows where data is sparse and shrinks where data is dense. For CineMatch, this means knowing when to recommend confidently and when to hedge with exploration.

01 — Functions, not parameters
A distribution over functions, not parameters.

A neural network learns parameters (weights) and uses them to define a function. A Gaussian Process does something conceptually different: it defines a distribution over functions directly. Instead of asking "what's the best function?", a GP asks "given the data we've seen, what set of functions is consistent with that data?"

The prior (before seeing data): infinitely many smooth functions, drawn from a multivariate Gaussian. The posterior (after seeing data): only the functions that pass through (or near) the observed points survive. The more data we see, the narrower the posterior band — and in regions with no data, the band stays wide. That width is our uncertainty.

The GP intuition

Think of a GP as placing a rubber band over the observed data points. Near the points, the band is tight (low uncertainty). Far from any point, the band is loose (high uncertainty). The shape of the band is determined by the kernel, which encodes our beliefs about function smoothness.

02 — Kernel
The kernel — encoding smoothness assumptions.

A GP is fully specified by a mean function m(x) (usually zero) and a kernel (covariance function) k(x, x') that defines how correlated two function values should be based on how similar their inputs are. The kernel is the GP's structural prior — it plays the same role as the architecture in a neural network.

The most common kernel is the Radial Basis Function (RBF), also called the squared exponential:

k(x, x') = σ² · exp(−||x − x'||² / 2ℓ²) (2.1)

Two hyperparameters: ℓ (length scale) controls how quickly correlation decays with distance — large ℓ means smooth, slowly-varying functions; small ℓ means wiggly, rapidly-varying ones. σ² (output variance) controls the amplitude of the function.

Math Core — why the RBF kernel guarantees smoothness.

The RBF kernel is positive definite — meaning any kernel matrix K built from it has all positive eigenvalues. This is required for K to be a valid covariance matrix (covariance matrices must be positive semi-definite). The proof: the RBF kernel can be written as an infinite sum of polynomial kernels, each of which is positive definite, and sums of positive definite functions are positive definite.

The smoothness property is stronger: functions drawn from a GP with an RBF kernel are infinitely differentiable (C∞). This follows from the kernel being infinitely differentiable in both arguments. In practice, this means RBF-GP predictions are always smooth curves — no sharp corners, no discontinuities. If your data has sharp transitions (e.g., a user's taste changing abruptly after a life event), the RBF kernel will smooth them out. Use Matérn-1/2 or -3/2 for rougher functions.

03 — Posterior
Posterior — conditioning on data.

The magic of GPs is closed-form posterior inference. Given training data (X, y) and test points X*, the posterior is still Gaussian with known mean and variance:

μ* = K(X*, X) · [K(X, X) + σ²nI]−1 · y (3.1)
Σ* = K(X*, X*) − K(X*, X) · [K(X, X) + σ²nI]−1 · K(X, X*) (3.2)

Where K(A, B) is the kernel matrix between sets A and B, and σ²n is the observation noise variance. The mean μ* is the prediction; the diagonal of Σ* gives the uncertainty at each test point.

import numpy as np

def rbf_kernel(X1, X2, l=1.0, sigma=1.0):
    """RBF (squared exponential) kernel matrix."""
    sq_dist = np.sum(X1**2, 1).reshape(-1,1) + np.sum(X2**2,1) - 2*X1@X2.T
    return sigma**2 * np.exp(-sq_dist / (2 * l**2))

def gp_predict(X_train, y_train, X_test, l=1.0, sigma=1.0, noise=0.1):
    """GP posterior mean and variance at test points."""
    K = rbf_kernel(X_train, X_train, l, sigma) + noise**2 * np.eye(len(X_train))
    K_s = rbf_kernel(X_train, X_test, l, sigma)
    K_ss = rbf_kernel(X_test, X_test, l, sigma)

    K_inv = np.linalg.solve(K, np.eye(len(K)))  # (K + σ²I)^{-1}

    mu = K_s.T @ K_inv @ y_train                  # posterior mean
    cov = K_ss - K_s.T @ K_inv @ K_s              # posterior covariance
    return mu, np.diag(cov)                        # mean, variance per point

Compare: at x* = 0.5 (close to training data), σ* would be ~0.12. At x* = 5.0 (far from all data), σ* → 1.0 (full prior uncertainty). The GP's uncertainty is a direct function of data density — this is the core value.

Math Core — learning hyperparameters via marginal likelihood.

How do we choose ℓ, σ², and σ²n? The GP provides a principled answer: maximize the log marginal likelihood (also called "model evidence"):

log p(y | X, θ) = −½ yᵀ Ky−1 y − ½ log|Ky| − N/2 · log(2π) (3.3)

Where Ky = K(X,X) + σ²nI and θ = {ℓ, σ², σ²n}. The three terms have intuitive meanings: the first is the data fit (how well the GP fits the training data), the second is the complexity penalty (−½ log|Ky| penalizes overly complex models — analogous to weight decay), and the third is a constant. Maximizing this balances fit and complexity automatically — no validation set needed for hyperparameter selection.

def neg_log_marginal_likelihood(X, y, l, sigma, noise):
    """Negative log marginal likelihood — minimize this to learn GP hyperparameters."""
    N = len(X)
    K_y = rbf_kernel(X, X, l, sigma) + noise**2 * np.eye(N)

    # Cholesky decomposition (numerically stable alternative to direct inverse)
    L = np.linalg.cholesky(K_y)
    alpha = np.linalg.solve(L.T, np.linalg.solve(L, y))

    # Three terms: data fit + complexity + constant
    data_fit = 0.5 * y @ alpha
    complexity = np.sum(np.log(np.diag(L)))  # = 0.5 * log|K_y|
    constant = 0.5 * N * np.log(2 * np.pi)

    return data_fit + complexity + constant

Numerical failure mode

Cholesky decomposition fails if Ky isn't positive definite.

If σ²n is too small (e.g., 1e-10), the kernel matrix becomes numerically singular — K has eigenvalues near zero, and adding a tiny noise term doesn't help. Cholesky raises LinAlgError: Matrix is not positive definite. The fix: add a jitter term (typically 1e-6 to 1e-4) to the diagonal: K_y += jitter * np.eye(N). This is standard in all GP libraries (GPyTorch, GPflow) and is not a hack — it's the numerical analyst's way of saying "I believe there's at least this much observation noise."

04 — Epistemic vs aleatoric
Epistemic vs aleatoric uncertainty.

The GP posterior variance captures two distinct types of uncertainty:

Epistemic uncertainty (model uncertainty): uncertainty from lack of data. "I haven't seen this region before, so I don't know." This shrinks as you collect more data in that region. In the GP, it's the part of the variance that comes from the posterior covariance Σ*.

Aleatoric uncertainty (data noise): irreducible uncertainty from noise in the observations themselves. "Even with infinite data, predictions here will vary because the data is inherently noisy." In the GP, it's σ²n — the observation noise variance, which doesn't shrink with more data.

Math Core — decomposing the posterior variance.

The total predictive variance at x* is:

Var[y*] = Var[f*] + σ²n = (σ² − K(x*,X) Ky−1 K(X,x*)) + σ²n (4.1)

The first term (Var[f*]) is epistemic — it depends on how far x* is from training data. As N → ∞ and data fills the input space, Var[f*] → 0 everywhere. The second term (σ²n) is aleatoric — it stays constant regardless of N. In practice:

Uncertainty decomposition for CineMatch
User A has rated 500 action movies. GP predicts their rating for a new action film:Var[f*] = 0.02 (lots of nearby data), σ²_n = 0.25 (inherent taste variation). Total: 0.27
Same user, a new documentary (0 documentary ratings):Var[f*] = 0.95 (no nearby data — nearly full prior), σ²_n = 0.25. Total: 1.20
After user rates 10 documentaries, re-predict:Var[f*] drops to 0.08. σ²_n still 0.25. Total: 0.33. Epistemic collapsed, aleatoric unchanged.
The decision: recommend the documentary in step 2 to learn (high epistemic = high information gain), even though it has higher total uncertainty. After step 3, switch to exploitation.This is the core of Bayesian optimization — Module 5.2
CineMatch Integration

Uncertainty-aware recommendations.

CineMatch predicts a user will rate a movie 4.2 stars with epistemic variance 0.8 (we have little data on this user-movie pairing — cold start) vs another movie rated 3.9 stars with epistemic variance 0.1 (thousands of similar users rated it). Which should we recommend? A purely exploitation-based ranker picks 4.2. An exploration-aware ranker might pick 3.9 — because it's a confident 3.9 vs an uncertain 4.2. Or it might pick 4.2 precisely because the high uncertainty means we'll learn a lot from the user's response. This is the exploration-exploitation trade-off that GPs enable and Module 5.2 will formalize.

05 — Scalability
The GP's computational bottleneck — and workarounds.

The posterior equations (3.1, 3.2) require inverting K(X, X) + σ²nI — an N × N matrix where N is the number of training points. Matrix inversion is O(N³) in compute and O(N²) in memory. For CineMatch with N = 200,000 rated interactions, this is 200K × 200K = 40 billion entries — completely infeasible.

Approximate GP methods.

For large N, exact GPs are replaced by approximations: sparse GPs (use M << N inducing points), random Fourier features (approximate the kernel with random projections), or deep kernel learning (use a neural network to learn a feature map, then place a GP on top). For CineMatch, the practical approach is to use GPs not for the main prediction model (that's the transformer) but for specific, small-scale problems where uncertainty matters — like the hyperparameter optimization in Module 5.2, which has N ≈ 20-100 evaluations.

Where this leads.

Module 5.2 applies GPs to their killer application in deep learning: Bayesian hyperparameter optimization. Instead of grid search or random search over learning rates, dropout rates, and layer sizes, we use a GP to model the relationship between hyperparameters and validation performance — then use the GP's uncertainty to decide which hyperparameters to try next. This is the exploration-exploitation trade-off from the CineMatch example above, applied to the ML engineering workflow itself.
