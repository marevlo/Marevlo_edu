Regularization. Techniques to generalize, not memorize.

Training a network to 100% accuracy on the training set is easy. Training one that still works on data it has never seen is the whole point. This module covers every major regularization tool — weight decay, dropout, early stopping, batch norm, data augmentation — and measures how much each is worth on CineMatch.

01 — The core problem
The core problem — overfitting.

Take a CineMatch MLP. Give it enough parameters. Train it long enough. It will drive its training loss toward zero — memorizing every user-movie interaction in the training set, including the noise. And then, on validation data it has never seen, its performance will be worse than that of a much simpler model. This is overfitting: finding patterns in the training set that don't exist in the real world.

Figure 1. The canonical overfitting pattern. Train loss decreases monotonically; validation loss decreases at first, then starts rising. The gap between them is the overfitting gap. Regularization's job is to make val loss keep going down — or at least not go up.

Why does overfitting happen in deep networks?

A modern MLP has way more parameters than you have training examples. CineMatch has 200,000 training interactions; a modest 4→64→32→1 MLP has 2,433 parameters. That's still more than one parameter per 100 training examples — tight but not absurd. Scale the model to 4→512→512→1 and the parameter count jumps to 266K — more parameters than training points. The network can now memorize the training set exactly.

The technical term for this is over-parameterization: a model with more capacity than the task strictly requires. All modern deep learning operates in this regime. The counter-intuitive fact is that over-parameterized networks often generalize well when properly regularized — even better than smaller models. The regularization is what makes that possible.

The regularization bargain

Bigger networks have more potential for overfitting — and more potential for fitting the true pattern. Regularization is what keeps the trade favorable: enough capacity to learn the real structure, enough constraint to not memorize the noise. The regularized big model almost always beats the un-regularized small one.

The three kinds of fit.

FitTrain lossVal lossDiagnosis
UnderfitHighHighModel too simple — add capacity
Just rightLowLow (and close to train)Ship it
OverfitVery lowHighAdd regularization or more data

Every regularizer in this module addresses the third row — pulling val loss back down when it's drifting above train loss. You never use all of them at once. Pick the ones that fit the problem, measure the gap they close, and stop adding when the gap stops shrinking.

02 — Weight decay
Weight decay — the universal starting regularizer.

The simplest regularization idea: large weights are suspicious. A model whose weights are all small can't produce wildly complex decision boundaries; a model with huge weights often has capacity to memorize. So add a term to the loss that penalizes large weights:

Ltotal = Ldata(w) + λ · ||w||22 (2.1)

||w||22 is the sum of squared weights. The coefficient λ (lambda) controls the strength — typical values are 10−5 to 10−2. Higher λ means stronger penalty, smaller weights, simpler model. This is called L2 regularization because it uses the L2 norm.

The gradient is elegantly simple.

Differentiating (2.1) with respect to w:

∇Ltotal = ∇Ldata + 2λw (2.2)

So the SGD update becomes:

wt+1 = wt − η(∇Ldata + 2λwt) = (1 − 2ηλ)wt − η∇Ldata (2.3)

Read that last form carefully. Every SGD step now multiplies w by (1 − 2ηλ) — a shrinking factor slightly less than 1 — before applying the data-driven update. This is why it's called "weight decay" — every step nudges weights toward zero by a fixed fraction. Without a gradient signal telling them to grow, they shrink; with one, they reach an equilibrium between the data pull and the regularization push.

PyTorch implementation — three ways to think about it.

import torch
import torch.nn as nn

model = nn.Sequential(
    nn.Linear(4, 64), nn.ReLU(),
    nn.Linear(64, 1)
)

# Way 1: weight_decay arg in the optimizer (cleanest, recommended)
opt = torch.optim.SGD(model.parameters(), lr=0.01, weight_decay=1e-4)

# Way 2: manual penalty added to loss (explicit, useful for custom losses)
l2 = sum(p.pow(2).sum() for p in model.parameters())
total_loss = data_loss + 1e-4 * l2

# Way 3: AdamW — decoupled weight decay (recommended for Adam-family)
opt = torch.optim.AdamW(model.parameters(), lr=1e-3, weight_decay=0.01)
Classic mistake

Don't apply weight decay to biases or normalization parameters.

Weight decay is designed for the weight matrices — the parameters that scale inputs. Bias terms don't cause overfitting the same way; they just shift outputs. Batch-norm and layer-norm parameters (γ, β) are scale/shift parameters that the network needs to produce nonzero activations. Applying weight decay to any of these three can hurt rather than help. Many PyTorch recipes use a parameter-group split: high weight decay on weight matrices, zero on biases/norm layers. We'll use this pattern in the capstone.

The L2-vs-L1 comparison.

RegularizerPenaltyEffect on weightsWhen to use
L2 (weight decay)λ · Σw2Shrinks all weights proportionallyDefault — almost always
L1 (LASSO)λ · Σ|w|Pushes many weights exactly to zeroWhen you want sparsity / feature selection
L1 + L2 (elastic)mix of bothBoth shrinks and sparsifiesLinear models with many noisy features

For deep networks, L2 is overwhelmingly the default. L1's sparsity benefit matters more for linear models with interpretable feature columns than for hidden layers where we don't care which specific neurons fire.

03 — Dropout
Dropout — the ensemble trick.

Dropout is the second most important regularizer, introduced by Hinton and collaborators in 2012. The rule is weirdly simple: during training, randomly set a fraction p of hidden-layer activations to zero on every forward pass. At test time, use all activations as normal. That's it.

Figure 2. Dropout in action. On each training step, a random subset of hidden units is temporarily zeroed (crossed out). The network learns to produce correct outputs even when any specific unit is missing — forcing it to spread knowledge across many units instead of concentrating it in a few.

Why this works.

Three interpretations, each correct and complementary:

  1. Ensemble view. Each dropout mask defines a different "thinned" sub-network. Training with random masks trains all 2H possible sub-networks simultaneously (sharing weights). At test time, using all activations is approximately an ensemble average over all these sub-networks. Ensemble averaging reduces variance, which is exactly what regularization is for.
  2. Co-adaptation prevention. Without dropout, neurons can develop brittle dependencies — unit 17 outputs 2× whatever unit 23 outputs, and unit 23 compensates. With dropout, unit 17 can't rely on unit 23 always being there. Each unit must learn something individually useful. The network becomes robust.
  3. Noise injection view. Dropout adds multiplicative Bernoulli noise to activations. Noise in training data effectively augments the dataset; noise in activations similarly "blurs" the loss surface, which biases the optimizer toward flatter, better-generalizing minima (connecting to 2.5's flat-minima story).

The inverted-dropout implementation.

One subtlety. If during training we zero out a fraction p of activations, the expected magnitude of the hidden layer output shrinks by (1 − p). At test time with no dropout, the magnitude is back to full. This mismatch would hurt. The fix is inverted dropout: during training, scale up the surviving activations by 1/(1 − p). Then train and test match up. PyTorch's nn.Dropout does this automatically — you just use it.

Inverted dropout — numeric example with p=0.5
Hidden layer output before dropout: h = [1.2, −0.8, 0.5, 2.0, −1.0, 0.3]6 hidden units, sum = 2.2, mean = 0.367
Random mask (p=0.5): mask = [1, 0, 1, 0, 1, 0] — 3 of 6 units survive
Without scaling: h_drop = [1.2, 0, 0.5, 0, −1.0, 0]. Sum = 0.7, mean = 0.117Expected output shrunk by ~50% — next layer receives weaker input during training than at test time
With inverted scaling (×1/(1−0.5) = ×2): h_drop = [2.4, 0, 1.0, 0, −2.0, 0]. Sum = 1.4, mean = 0.233E[sum] = 0.5 × 2 × original_sum = 1.0 × 2.2 ≈ 2.2. Expectation matches no-dropout!
At test time: no dropout, no scaling. h_test = [1.2, −0.8, 0.5, 2.0, −1.0, 0.3]Because training used inverted scaling, test-time activations have the same expected magnitude. No train/test mismatch.
import torch.nn as nn

class CineMatchMLP(nn.Module):
    def __init__(self, in_dim=4, hidden=(64, 32), p_drop=0.2):
        super().__init__()
        layers = []
        d = in_dim
        for h in hidden:
            layers += [nn.Linear(d, h), nn.ReLU(), nn.Dropout(p_drop)]
            d = h
        layers += [nn.Linear(d, 1)]
        self.net = nn.Sequential(*layers)

    def forward(self, x):
        return self.net(x)

# Training
model.train()  # enables dropout
# ... training loop ...

# Evaluation
model.eval()   # disables dropout; uses all activations
with torch.no_grad():
    val_preds = model(x_val)
Three classic dropout mistakes

model.train() / model.eval(), overusing on small nets, and on the output.

1. Forgetting model.eval() at validation is the #1 bug — validation loss looks noisier and worse than it should. 2. Using dropout on a small tabular MLP often hurts — the network doesn't have the capacity to benefit from the redundancy. Try p = 0.1 first, or skip it entirely on small nets. 3. Never put dropout after your output layer — you'll be zeroing the predictions themselves.

Typical dropout rates.

Layer typeTypical pNotes
Input layer0.0 — 0.1Rarely helps; use augmentation instead
Hidden layers (dense)0.2 — 0.5Sweet spot for wide MLPs
Hidden layers (after ReLU)0.2 — 0.3Most common pattern
Transformer attention/FFN0.1Enough for regularization without hurting training
Output layerNeverWould zero the predictions

04 — Early stopping
Early stopping — the free regularizer.

The simplest and most reliable regularizer of all: stop training as soon as validation loss stops improving. It requires no code changes to the model, no extra math, no new hyperparameter to tune (besides "how long to wait"). It's always a good idea.

Recall Figure 1. Validation loss decreases, reaches a minimum, then rises. The minimum of val loss — not the minimum of train loss — is where your best-generalizing model lives. Early stopping is the discipline of checkpointing the model at every epoch, then returning the version with best val performance.

Patience — not stopping at the first wobble.

Val loss curves have noise — small oscillations caused by stochastic gradients, minibatch order, dropout randomness. A single epoch where val loss ticks up doesn't mean "stop now" — it could just be noise. The canonical implementation uses a patience parameter: stop if val loss hasn't improved for k consecutive epochs. Common values are 5–20 depending on how noisy your validation metric is.

import torch
from copy import deepcopy

class EarlyStopping:
    def __init__(self, patience=10, min_delta=0.0):
        self.patience = patience
        self.min_delta = min_delta
        self.best_loss = float("inf")
        self.best_state = None
        self.counter = 0
        self.should_stop = False

    def step(self, val_loss, model):
        if val_loss < self.best_loss - self.min_delta:
            self.best_loss = val_loss
            self.best_state = deepcopy(model.state_dict())
            self.counter = 0
        else:
            self.counter += 1
            if self.counter >= self.patience:
                self.should_stop = True

    def restore_best(self, model):
        if self.best_state is not None:
            model.load_state_dict(self.best_state)


# Usage
es = EarlyStopping(patience=10)
for epoch in range(200):
    # ... train and compute val_loss ...
    es.step(val_loss, model)
    if es.should_stop:
        print(f"stopping at epoch {epoch+1}; best val loss = {es.best_loss:.4f}")
        break
es.restore_best(model)
Why early stopping is free

Unlike weight decay or dropout, early stopping adds no training overhead — you're just monitoring a metric you already compute. Unlike architectural choices, it doesn't require retraining to change. And it's always safe: the worst case is you trained too long and ignored the result, the best case is you get a better model and spend less compute. Every serious training script has some form of early stopping. Don't leave it out.

A subtle relationship to weight decay.

Early stopping and weight decay achieve similar ends via different means. Without weight decay, weights grow over time; early stopping catches the model before they grow too large. With weight decay, weights are actively pushed small; training can run longer without overfitting. Both push weights toward smaller magnitudes — one reactively, one proactively. They can compose, but the incremental benefit of adding one when you already have the other diminishes.

05 — Batch normalization
Batch normalization — regularizer by accident.

Batch normalization (BN, 2015) was designed to solve a different problem — speeding up training of deep networks by reducing the "internal covariate shift" between layers. It turned out to also provide a significant regularization effect, so much so that it's often listed alongside dropout and weight decay.

Here's what it does. Given a hidden layer's outputs z across a mini-batch of B examples, BN computes the batch mean μ and variance σ2 of each feature, then normalizes:

BN(z) = γ · z − μ √(σ2 + ε) + β (5.1)

The normalization forces each feature to have zero mean and unit variance within the batch. The learnable parameters γ and β then let the network rescale/shift to whatever distribution is actually useful. Crucially, during training the normalization is computed from this batch; during inference, a running average of μ and σ2 accumulated during training is used instead.

Why BN regularizes.

The regularization comes from the batch dependence. During training, a given example's normalization depends on which other examples happened to share its mini-batch. Different batches → different normalizations → noise injected into each example's hidden representations. This noise, like dropout's, biases training toward flatter minima and discourages co-adaptation.

You can see this empirically: networks with BN often generalize better than networks without BN, even when BN isn't strictly necessary for convergence speed. And when people use very large batch sizes (thousands of examples), BN's regularization effect weakens and val accuracy can suffer — another signature of noise-driven generalization.

Subtlety that bites everyone

Put BN before the activation, and watch out for small batch sizes.

1. Standard pattern is Linear → BatchNorm1d → ReLU, not ReLU first. The idea is to normalize the pre-activation — although in practice both orderings work and there's ongoing research on which is preferred. 2. BN needs a non-tiny batch size to estimate μ and σ2 reliably. With batch=1 or 2, BN is noise — use LayerNorm or GroupNorm instead. 3. Don't forget model.eval() at validation, same as with dropout — it switches BN from batch-statistics to running-statistics.

import torch.nn as nn

class CineMatchMLP_BN(nn.Module):
    def __init__(self, in_dim=4, hidden=(64, 32)):
        super().__init__()
        layers = []
        d = in_dim
        for h in hidden:
            layers += [
                nn.Linear(d, h),
                nn.BatchNorm1d(h),
                nn.ReLU()
            ]
            d = h
        layers += [nn.Linear(d, 1)]
        self.net = nn.Sequential(*layers)

    def forward(self, x):
        return self.net(x)

The sibling norms.

Norm typeNormalizes overUse for
BatchNormBatch dimensionCNNs; most dense layers with batch ≥ 16
LayerNormFeatures in one exampleTransformers; anything with batch = 1 or variable batch
GroupNormGroups of channels in one exampleSmall-batch CNN training
InstanceNormSpatial dimensions in one channelStyle transfer and image generation

The choice depends on what the batch dimension represents and how large it is. Transformers use LayerNorm universally because sequence lengths vary and attention doesn't play nicely with BN's batch statistics. CNNs still use BN everywhere. MLPs can use either.

06 — Data augmentation
Data augmentation — the most powerful regularizer (when available).

All the previous techniques regularize by changing the model or loss. Data augmentation regularizes by changing the data — creating artificial training examples by applying label-preserving transformations to real ones.

The canonical example is image classification. A cat is still a cat if you flip it horizontally, rotate it by 10°, crop it differently, shift its colors slightly, or add small amounts of noise. Training on augmented images is like training on a much larger dataset — the model learns invariances rather than memorizing pixel patterns.

Why it's so effective.

  • Implicit dataset scaling. Each original example becomes hundreds of augmented variants. Effective dataset size multiplies, and more data is the strongest known regularizer.
  • Encodes priors. The augmentations are the invariances you want the model to have. Horizontal flips encode "left/right orientation shouldn't matter." Rotations encode "rotation shouldn't matter" — if true. Choose augmentations that match the real-world invariances of your problem.
  • No inference cost. Augmentation happens only during training. At inference, you run the model on raw inputs. Unlike dropout, there's no train/test discrepancy to manage.
CineMatch and augmentation

Tabular data is harder to augment — and this is a real limitation.

Augmentation is the strongest tool in the toolbox when available. For images and text, whole pipelines exist (flips, crops, back-translation, dropout of words). For tabular data like CineMatch's user/movie features, augmentations are much more subtle — you can add small Gaussian noise to numeric features, swap values across similar users, or use techniques like SMOTE for class imbalance. The effect is more modest than on images. This is one reason tabular deep learning often underperforms gradient-boosted trees: the regularization toolbox is smaller.

Image augmentation in practice.

import torchvision.transforms as T

# Training transforms — augment
train_tfm = T.Compose([
    T.RandomResizedCrop(224),
    T.RandomHorizontalFlip(),
    T.ColorJitter(brightness=0.2, contrast=0.2),
    T.ToTensor(),
    T.Normalize(mean=[0.485, 0.456, 0.406],
                std=[0.229, 0.224, 0.225])
])

# Validation transforms — only resize + normalize, no augmentation
val_tfm = T.Compose([
    T.Resize(256),
    T.CenterCrop(224),
    T.ToTensor(),
    T.Normalize(mean=[0.485, 0.456, 0.406],
                std=[0.229, 0.224, 0.225])
])

When we reach Phase 3 and train CNNs on movie posters, every serious run will use augmentation. It's the single biggest lever after "use a reasonable architecture."

The 2020s wave: MixUp, CutMix, RandAugment.

Recent augmentation methods go beyond simple transforms:

  • MixUp: Linearly blend two training images and their labels — e.g., 0.7 × cat + 0.3 × dog. Surprisingly effective.
  • CutMix: Cut a rectangle out of one image and paste it onto another; mix labels proportionally to the pasted area.
  • RandAugment: Apply N random augmentations from a fixed list with random magnitudes; searchless alternative to AutoAugment.

These methods regularize not just by exposing the model to more data but also by forcing it to learn smooth decision boundaries — MixUp, for instance, teaches the model that intermediate predictions between classes should interpolate smoothly. On ImageNet, RandAugment or MixUp often closes 1–2 percentage points of top-1 accuracy gap.

07 — The recipe
The regularization recipe, and which ones to reach for first.

A practical CineMatch MLP recipe.

import torch
import torch.nn as nn

class CineMatchMLP(nn.Module):
    """Regularized MLP: BN + Dropout + weight decay + early stopping."""
    def __init__(self, in_dim=4, hidden=(64, 32), p_drop=0.2):
        super().__init__()
        layers = []
        d = in_dim
        for h in hidden:
            layers += [
                nn.Linear(d, h),
                nn.BatchNorm1d(h),
                nn.ReLU(),
                nn.Dropout(p_drop),
            ]
            d = h
        layers += [nn.Linear(d, 1)]
        self.net = nn.Sequential(*layers)

    def forward(self, x):
        return self.net(x)


# Separate params so weight decay only hits weight matrices
def get_param_groups(model, weight_decay=1e-4):
    decay, no_decay = [], []
    for name, p in model.named_parameters():
        if "bias" in name or "bn" in name.lower() or "norm" in name.lower():
            no_decay.append(p)
        else:
            decay.append(p)
    return [
        {"params": decay, "weight_decay": weight_decay},
        {"params": no_decay, "weight_decay": 0.0},
    ]


model = CineMatchMLP(p_drop=0.2)
optimizer = torch.optim.AdamW(get_param_groups(model), lr=1e-3)
es = EarlyStopping(patience=10)
# ... then the usual training loop, calling es.step(val_loss, model) each epoch ...

The CineMatch regularization ablation — quantifying what each is worth.

Below is the kind of ablation you should always run on a new problem. Each row adds one regularizer to the baseline. The numbers are representative for CineMatch's scale and shape — your real values will be similar directionally.

ConfigurationTrain accVal accGap
Baseline MLP (no reg)96.4%91.3%5.1
+ Weight decay 1e-495.1%92.6%2.5
+ Dropout p=0.294.2%93.1%1.1
+ BatchNorm94.5%93.4%1.1
+ Early stopping (patience 10)93.8%93.5%0.3

Note what's happening. Train accuracy decreases monotonically — every regularizer makes the model less good at memorizing the training set. Val accuracy increases monotonically — each regularizer adds incremental generalization. The gap closes from 5.1 to 0.3. That's the trajectory you want to see.

The regularization ordering that usually works

Start with weight decay — it's free in the optimizer and always helps. Add early stopping — no cost, always helps. Measure your gap. If > 2%, add dropout. If still > 2%, add BN. If you're in a domain where augmentation exists (images, text), always use it — it's usually the biggest lever. Stop adding regularization as soon as the validation gap closes to roughly your validation noise floor. More regularization than needed just underfits.
