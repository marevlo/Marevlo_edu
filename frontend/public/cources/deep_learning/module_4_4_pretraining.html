Pretraining. Learning everything, then specializing.

word2vec showed that training on context alone produces useful embeddings. The transformer showed that attention stacks scale beautifully. Pretraining combines both insights at scale: train a massive transformer on a huge, unlabeled corpus to learn general representations, then fine-tune on your specific task with a small labeled dataset. This pretrain-then-fine-tune paradigm is behind every frontier model — BERT, GPT, LLaMA — and the same idea applies directly to CineMatch's recommender.

01 — The pretraining insight
Labels are expensive, text is free.

CineMatch has 200K labeled interactions (user rated movie 4+ stars). That's enough for an MLP or even an LSTM, but it's tiny for a transformer with millions of parameters. Meanwhile, CineMatch has 50 million watch events — raw sequences of "user watched movie X, then Y, then Z" — with no rating labels at all. Pretraining turns this unlabeled behavioral data into a training signal.

The idea: train the CineMatchTransformer to predict the next movie in a watch-sequence (no labels needed — the next movie is the label). This pretraining phase learns general user-behavior patterns — genre preferences, temporal viewing rhythms, sequel-watching tendencies — from all 50M events. Then fine-tune on the 200K labeled interactions for the specific rating-prediction task. The pretrained model starts with a huge advantage over random initialization because it already understands the domain.

The pretrain-then-fine-tune paradigm

Phase 1: pretrain on a large, unlabeled dataset with a self-supervised objective (predict masked or next tokens). Phase 2: fine-tune on a small, labeled dataset for the target task. The pretrained weights encode general domain knowledge; fine-tuning adapts that knowledge to the specific problem. This is the same two-phase pattern as transfer learning in Module 3.5 (freeze backbone, train head, then unfreeze) — applied at the level of the entire model.

02 — BERT
BERT — bidirectional masked language modeling.

BERT (Devlin et al., 2018) is an encoder-only transformer pretrained with a simple objective: randomly mask 15% of input tokens and predict them from the surrounding context. Because the encoder uses bidirectional self-attention (no causal mask), each masked position sees both left and right context — unlike GPT, which only sees the left.

LMLM = −Σi ∈ masked log P(xi | x¬masked) (2.1)

The masking protocol from the paper: of the 15% chosen positions, 80% are replaced with a special [MASK] token, 10% with a random token, 10% left unchanged. This prevents the model from learning that [MASK] is always the signal to predict — it must learn to reconstruct from any corrupted input.

Math Core — the MLM gradient.

Let hi be the transformer's output vector at masked position i (shape: dmodel). The prediction head is a linear projection to the vocabulary:

P(xi = w | context) = softmax(Whead · hi + b)w    where Whead ∈ ℝV × d (2.2)

The gradient of the MLM loss with respect to hi is the familiar softmax cross-entropy gradient:

∂LMLM/∂hi = WheadT · (p̂ − ex_i) (2.3)

Where p̂ is the predicted probability vector (shape V) and ex_i is the one-hot vector for the true token. This gradient flows backward through all transformer layers via the standard backprop chain — but only at masked positions. Unmasked positions contribute no gradient to the loss, which means only ~15% of positions provide training signal per forward pass. This is why MLM is less sample-efficient than autoregressive pretraining (which uses 100% of positions).

Build It — MLM masking from scratch.

import torch

def create_mlm_batch(token_ids, vocab_size, mask_token_id, mask_prob=0.15):
    """
    Apply BERT-style masking to a batch of token IDs.
    token_ids: (B, T) long tensor
    Returns: masked_ids, labels (−100 at unmasked positions for ignore_index)
    """
    labels = token_ids.clone()

    # Step 1: decide which positions to mask (15%)
    prob_matrix = torch.full(token_ids.shape, mask_prob)
    mask = torch.bernoulli(prob_matrix).bool()

    # Don't mask padding or special tokens (positions 0, 1 typically)
    mask[:, :2] = False

    # Labels: -100 at unmasked positions (ignored by CrossEntropyLoss)
    labels[~mask] = -100

    # Step 2: of the masked positions...
    masked_ids = token_ids.clone()

    # 80% → replace with [MASK]
    replace_mask = mask & (torch.rand_like(prob_matrix) < 0.8)
    masked_ids[replace_mask] = mask_token_id

    # 10% → replace with random token
    random_mask = mask & ~replace_mask & (torch.rand_like(prob_matrix) < 0.5)
    random_tokens = torch.randint(vocab_size, token_ids.shape, device=token_ids.device)
    masked_ids[random_mask] = random_tokens[random_mask]

    # 10% → keep original (already done — masked_ids started as clone)

    return masked_ids, labels

Failure case — the masking distribution shift.

Subtle pretraining bug

The [MASK] token never appears at inference — creating a train/test mismatch.

During pretraining, 80% of selected positions see [MASK]. During fine-tuning and inference, [MASK] never appears. If the model learns to rely on [MASK] as a "predict here" signal, its representations at unmasked positions will be subtly different from what downstream tasks need. The 80/10/10 split (10% random, 10% unchanged) mitigates this — it forces the model to produce good representations even at non-[MASK] positions, because it can't be sure which positions are masked. This is not a cosmetic detail — ablations show that using 100% [MASK] (no random/unchanged) degrades downstream performance by 1-2% on GLUE benchmarks.

Fine-tuning BERT.

After pretraining, BERT is fine-tuned by adding a task-specific head — a small MLP on top of the encoder output. For classification, the output at the special [CLS] token feeds a linear classifier. For token-level tasks (NER, QA), each token's output feeds a per-token classifier. The entire model (encoder + head) is fine-tuned end-to-end with a small learning rate (2e-5 to 5e-5) for a few epochs.

BERTQuestion 02 / 07
BERT masks 15% of tokens during pretraining. Why not mask 50% for more training signal per sequence?
APyTorch can't handle more than 15% masking.
B50% masking would make the model too large.
C15% was chosen arbitrarily and higher rates work equally well.
DHigher masking rates remove too much context — the remaining tokens don't provide enough information to predict the masked ones, making the objective too hard and the learned representations less useful. 15% is a sweet spot: enough signal per sequence to train efficiently, enough context remaining for the model to learn meaningful patterns. (Later work like SpanBERT and T5 explored different rates; the optimal is task-dependent but typically 15-25%.)
03 — GPT
GPT — autoregressive next-token prediction.

GPT (Radford et al., 2018) takes the opposite approach: decoder-only transformer, pretrained to predict the next token given all previous tokens. No masking, no corruption — just the language modeling objective from Module 4.2:

LGPT = −Σt=1..T log P(xt | x1, ..., xt−1) (3.1)

This is exactly what CineMatchTransformer does — predict the next movie from the watch-history. GPT scales this idea to internet-scale text.

Math Core — the autoregressive gradient and teacher forcing.

At each position t, the model predicts a distribution over the vocabulary V. The gradient at position t is identical in form to BERT's — softmax cross-entropy — but the key difference is in the attention mask, not the loss:

∂L/∂ht = WheadT · (p̂t − ex_t)    (same form as BERT — the architecture difference is in the causal mask, not the loss) (3.2)

Crucially, all T positions contribute gradient simultaneously during training (teacher forcing: the true token at each position is fed as input to the next). This makes GPT-style pretraining more sample-efficient per sequence than MLM — 100% of positions provide signal vs ~15% for BERT. The trade-off: GPT sees only left context at each position, while BERT sees both sides.

Worked example — perplexity computation.

Computing perplexity on a 5-movie sequence
Sequence: [Alien, Jaws, Psycho, Scream, Halloween]T = 5 movies, V = 50,000
Model predicts P(Jaws | Alien) = 0.004, P(Psycho | Alien, Jaws) = 0.008, P(Scream | ...) = 0.012, P(Halloween | ...) = 0.0254 predictions (position 1 has no prior context to predict from)
Log-likelihood: log(0.004) + log(0.008) + log(0.012) + log(0.025) = −5.52 − 4.83 − 4.42 − 3.69 = −18.46Average per-token: −18.46 / 4 = −4.62
Perplexity = exp(−average log-likelihood) = exp(4.62) = 101.5The model is "as confused as if choosing uniformly among ~102 movies per step"
After pretraining on 50M sequences: P(Halloween | horror sequence) might reach 0.15 → perplexity drops to ~20. After fine-tuning: ~12. Lower = better.Perplexity is the standard metric for language/sequence models

Build It — autoregressive pretraining loop from scratch.

# GPT-style pretraining for CineMatchTransformer
import torch, torch.nn.functional as F

def pretrain_step(model, batch, optimizer, scaler):
    """One pretraining step. batch: (B, T) token IDs."""
    # Input: positions 0..T-2   Target: positions 1..T-1
    x = batch[:, :-1]      # (B, T-1)
    y = batch[:, 1:]       # (B, T-1) — shifted right by 1

    with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
        logits = model(x)   # (B, T-1, V) — causal mask inside model
        # Reshape for cross_entropy: (B*(T-1), V) vs (B*(T-1),)
        loss = F.cross_entropy(
            logits.reshape(-1, logits.size(-1)),
            y.reshape(-1),
            ignore_index=0,  # padding token
        )

    optimizer.zero_grad(set_to_none=True)
    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(optimizer)
    scaler.update()

    # Compute perplexity for logging
    ppl = torch.exp(loss).item()
    return loss.item(), ppl

# Tensor shapes at each step (B=32, T=200, V=50000, d=256):
# x:      (32, 199)           — input token IDs
# logits: (32, 199, 50000)    — prediction over vocab at each position
# y:      (32, 199)           — target token IDs (shifted by 1)
# loss:   scalar              — average cross-entropy over all valid positions

Failure case — train/inference distribution mismatch (exposure bias).

GPT's training weakness

Teacher forcing creates a distribution it never sees at inference.

During training, each position receives the true previous token as input (teacher forcing). During inference, each position receives the model's own previous prediction — which may be wrong. If the model predicts a wrong token at step t, all subsequent predictions are conditioned on an input it was never trained on. Errors compound. This is exposure bias. Mitigation strategies: scheduled sampling (randomly use model predictions instead of ground truth during training, with increasing probability), or beam search at inference (maintain multiple hypothesis sequences). For CineMatch's next-movie prediction, exposure bias is less severe than for text generation because we typically only predict 1 step ahead, not a long sequence.

CineMatch Integration

CineMatch uses both paradigms.

GPT-style pretraining for the recommender: train CineMatchTransformer on 50M watch-sequences with next-movie prediction. This is the retrieval model from 3.5/4.2, now pretrained on vastly more data. BERT-style pretraining for the ranking model: mask random movies in the watch-history, predict them from context. The bidirectional encoder produces richer per-movie representations that feed the ranking stage's feature set. Both models start from the same unlabeled behavioral data; they just use different masking strategies optimized for their respective tasks.

04 — Fine-tuning recipe
The fine-tuning recipe — what actually transfers.

Pretraining gives you a model with rich, general representations. Fine-tuning adapts those representations to your specific task. The recipe mirrors Phase 3.5's transfer learning — with the same two-phase pattern:

Phase A: freeze most of the model, train the head.

Add a task-specific head (linear layer for classification, keep the existing LM head for generation). Train only the head for 1-3 epochs with a moderate learning rate (1e-3). This lets the head align with the pretrained features without destroying them.

Phase B: unfreeze everything, fine-tune end-to-end.

Unfreeze the transformer blocks. Use a much smaller learning rate (1e-5 to 5e-5) with warmup. The pretrained weights shift slightly to adapt to the task; the head continues to improve. Typical schedule: 3-10 epochs depending on dataset size.

# Fine-tuning CineMatchTransformer for rating prediction
# Phase A: train classification head only
pretrained = CineMatchTransformer.load('checkpoints/pretrained.pt')
for p in pretrained.parameters():
    p.requires_grad = False

# Replace the next-movie head with a rating classifier
pretrained.head = nn.Linear(256, 1)  # binary: will rate 4+?
opt = torch.optim.AdamW(pretrained.head.parameters(), lr=1e-3)

# Phase B: unfreeze and fine-tune
for p in pretrained.parameters():
    p.requires_grad = True
opt = torch.optim.AdamW([
    {'params': pretrained.blocks.parameters(), 'lr': 2e-5},
    {'params': pretrained.head.parameters(),   'lr': 1e-4},
], weight_decay=0.01)

Math Core — why differential LR works.

Consider a pretrained weight wpre in layer 2 of the transformer. After one SGD step with learning rate η:

wnew = wpre − η · ∂Ltask/∂w (4.1)

The displacement ||wnew − wpre|| = η · ||∂L/∂w||. If η = 1e-3 (training from scratch), the first update moves the weight by ~1e-3 × gradient_norm. For pretrained weights where the gradient norm is ~1.0, this is a 0.1% shift per step — sounds small, but over 1000 steps (one epoch) the cumulative displacement can be ~3% of the original weight magnitude. That's enough to destroy the pretrained representation.

At η = 2e-5 (50× lower), the same 1000 steps produce ~0.06% cumulative displacement — the pretrained features are gently nudged rather than overwritten. The new head at η = 1e-4 (5× higher than the backbone) moves faster because it was randomly initialized and needs large updates to align with the pretrained features.

Worked example — layer-wise LR decay (LLRD).

Layer-wise learning rate decay for a 4-layer CineMatch transformer
Base LR = 2e-5, decay factor γ = 0.8 per layer (from top to bottom)
Layer 4 (closest to output): LR = 2e-5 × 0.8⁰ = 2.00e-5Most task-specific, needs most adaptation
Layer 3: LR = 2e-5 × 0.8¹ = 1.60e-5
Layer 2: LR = 2e-5 × 0.8² = 1.28e-5
Layer 1 (closest to input): LR = 2e-5 × 0.8³ = 1.02e-5Most general features — smallest updates to preserve them
Embedding layer: LR = 2e-5 × 0.8⁴ = 0.82e-5Movie embeddings from pretraining — barely touched
New classification head: LR = 1e-4 (separate, 5× the backbone base)Random init — needs fast learning
# Layer-wise LR decay implementation
def get_llrd_groups(model, base_lr=2e-5, head_lr=1e-4, decay=0.8):
    groups = []
    # Backbone layers: lower LR for earlier layers
    for i, block in enumerate(reversed(list(model.blocks))):
        lr = base_lr * (decay ** i)
        groups.append({"params": block.parameters(), "lr": lr})
    # Embedding: lowest LR
    groups.append({"params": model.embed.parameters(),
                   "lr": base_lr * (decay ** len(model.blocks))})
    # New head: highest LR
    groups.append({"params": model.head.parameters(), "lr": head_lr})
    return groups
Catastrophic forgetting

Too-high LR during fine-tuning destroys pretrained features.

The pretrained weights encode millions of training examples worth of knowledge. A learning rate of 1e-3 (suitable for training from scratch) will overwrite this knowledge in a few steps. The standard fix: 10-100× lower LR for pretrained layers vs the new head, plus warmup. If val performance degrades after initial improvement, you're forgetting — reduce LR or freeze more layers.

05 — Scaling laws
Scaling laws — why bigger models keep getting better.

One of the most important empirical discoveries of the 2020s: transformer performance follows power-law scaling with three variables — model size (N parameters), dataset size (D tokens), and compute budget (C FLOPs). Double any one → predictable, smooth improvement in loss. No sudden plateaus, no diminishing returns — just a straight line on a log-log plot.

L(N) ∝ N−α     (α ≈ 0.076 for language models) (5.1)

The Chinchilla result (Hoffmann et al., 2022) sharpened this: for a fixed compute budget, there's an optimal ratio of model size to data size. Training a 70B-parameter model on 1.4T tokens outperforms a 280B-parameter model on 300B tokens — even though the larger model has 4× more parameters. The lesson: data scales as fast as parameters.

Math Core — the three scaling law equations.

The Kaplan et al. (2020) scaling laws are three power-law relationships — each holding the other two variables at sufficiency:

L(N) = (Nc/N)αN    where αN ≈ 0.076, Nc ≈ 8.8 × 10¹³ (5.2)
L(D) = (Dc/D)αD    where αD ≈ 0.095, Dc ≈ 5.4 × 10¹³ (5.3)
L(C) = (Cc/C)αC    where αC ≈ 0.050, C ≈ 6ND (FLOPs ≈ 6 × params × tokens) (5.4)

The Chinchilla-optimal allocation: given compute budget C, the optimal model size N* and data size D* satisfy N* ∝ C0.50 and D* ∝ C0.50 — they scale equally. The practical rule of thumb: ~20 tokens per parameter.

Worked example — is CineMatch's transformer right-sized?
CineMatch has D = 50M watch-events. Chinchilla-optimal: N* = D/20 = 2.5M params.Our 15.9M-param CineMatchTransformer is 6× larger than Chinchilla-optimal!
But wait — each "token" in CineMatch is a movie ID, and the vocabulary is much smaller (50K vs 50K+ for text). The effective information per token is higher, so the 20:1 ratio from language models doesn't transfer directly.
Empirical check: plot val loss vs training tokens at fixed N=15.9M. If the curve is still declining at D=50M, the model is data-hungry → collect more data. If it's flattened, the model is saturated → try larger model.
Compute budget: C ≈ 6 × 15.9M × 50M = 4.77 × 10¹⁵ FLOPs ≈ ~30 GPU-hours on an A100.Cheap enough to run scaling experiments: train 4 models (4M, 8M, 16M, 32M params) and plot the frontier

What this means for CineMatch.

CineMatch's 50M watch-events can support a transformer with roughly 15-50M parameters before overfitting. Scaling beyond that requires more data (user interactions, metadata, cross-platform signals) or data augmentation. The scaling laws tell us exactly how much improvement to expect: doubling data from 50M to 100M events at fixed model size should improve loss by ~5% (following the power law). This lets the CineMatch team plan data collection investments with predicted ROI.

06 — Modern landscape
The modern pretraining landscape.

The pretrain-then-fine-tune paradigm has evolved into three distinct patterns:

PatternDescriptionCineMatch example
Feature extractionFreeze pretrained model, use outputs as fixed featuresUse pretrained transformer's last-layer output as input to a gradient-boosted tree ranker
Full fine-tuningUnfreeze all layers, train end-to-end with small LRFine-tune pretrained recommender on rating data
Prompt-based / in-contextNo weight updates — condition model with examples at inferenceUse a large pretrained model to rank candidates by describing the task in natural language
Historical note

2018–2024: from fine-tuning to prompting to reasoning.

BERT (2018) introduced pretrain+fine-tune as the dominant paradigm. GPT-2 (2019) showed zero-shot capability — performing tasks without any fine-tuning, by framing them as text completion. GPT-3 (2020) made few-shot in-context learning practical. RLHF (2022) added human-preference alignment. GPT-4 and Claude (2023-2024) added multimodal input, long context, and tool use. Each step reduced the amount of task-specific engineering needed — from training a new model per task (BERT era) to writing a prompt (GPT-4 era). For CineMatch, this means future recommenders may use general-purpose models prompted with user history rather than task-specific transformers trained from scratch.
