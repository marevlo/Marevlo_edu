Linear regression from first principles.

Linear regression is the simplest neural network — a single neuron with no activation function. Master it from scratch and you understand the skeleton of every deep model that follows. We'll predict CineMatch watch-times, derive the math by hand, then build the same model three ways: pure NumPy, PyTorch with autograd, and PyTorch with a proper nn.Module.

01 — The prediction problem
The prediction problem.

Every supervised learning problem has the same shape. You have inputs x. You want to predict an output y. You have a dataset of past examples where you know both. You want a function that, given a new x, produces a sensible y.

CineMatch's first ML problem

Predict watch-time from observable features.

Given a user (with age and attention span) and a movie (with runtime and critic score), predict what fraction of the movie the user will watch. The features are four numbers; the target is one number between 0 and 1. We have 200,000 such examples from the synthetic dataset we generated in Module 0.4. The function we want to learn — call it f — takes a 4-dimensional input and produces a scalar output.

We never observe f directly. What we observe are noisy samples:

y = f(x) + ε (1.1)

where ε is some small random noise (mood, network conditions, who knows what). The art of regression is to choose a family of candidate functions — a hypothesis class — and pick the member of that family that best fits the data we've seen.

Among all possible families, the simplest and most studied is the family of linear functions:

ŷ = w1x1 + w2x2 + ... + wdxd + b (1.2)

Read ŷ as "y-hat" — our prediction, distinct from the true y. The numbers w1, ..., wd are weights (one per feature), and b is the bias. Together, those d+1 numbers are the parameters of our model. Training is the act of finding values for them that make our predictions match the data well.

Why linear, even when we know the world isn't?

Three reasons that justify linear regression as our starting point:

  1. Tractability. The loss surface is convex — there is exactly one optimum, and any reasonable optimizer will find it.
  2. Interpretability. Each weight has a direct meaning: "if this feature increases by 1, the prediction changes by w".
  3. Pedagogical foundation. A linear regressor is a single neuron. Once you can train this, you can train any feedforward network — the loop is the same.
The single-neuron view

The equation ŷ = wTx + b is, drawn as a diagram, exactly one neuron. It takes weighted inputs, sums them, adds a bias, and outputs a number. Stack many of these in a row and call it a layer. Stack layers and call it a network. Add non-linear activations between layers and call it deep learning. Everything that follows in this course is a richer version of this single line.

02 — Vectorized form
From scalar to vectorized — speaking PyTorch's language.

The scalar equation we wrote — ŷ = w1x1 + ... + wdxd + b — is mathematically clear but computationally slow. To run on a GPU, we need to express the same idea in matrix form. Let's introduce notation that scales.

Stack the features of one example into a column vector: x ∈ ℝd. Stack the weights into another column vector: w ∈ ℝd. Then the prediction for that single example is the dot product:

ŷ = wTx + b (2.1)

For an entire dataset of n examples, stack the inputs into a matrix X ∈ ℝn×d (one example per row), and stack the targets into y ∈ ℝn. All n predictions at once become:

ŷ = Xw + b1n (2.2)

where 1n is a vector of ones — though in PyTorch you'll just write X @ w + b and broadcasting handles the bias automatically.

Why this matters for speed.

Equation (2.2) is one matrix multiplication followed by one broadcasted addition. On a GPU, that takes microseconds for a CineMatch batch of 64 examples, regardless of how many features each example has. Equivalent code as a Python loop over examples and features would take seconds — a thousand-times slowdown that makes deep learning infeasible.

import torch

# A CineMatch batch: 64 examples, 4 features each
X = torch.randn(64, 4)
w = torch.randn(4)
b = torch.tensor(0.5)

# Slow way (don't do this)
predictions_slow = torch.zeros(64)
for i in range(64):
    for j in range(4):
        predictions_slow[i] += w[j] * X[i, j]
    predictions_slow[i] += b

# Fast way (the only acceptable way in DL)
predictions_fast = X @ w + b           # shape [64]

# Same answer; the second is ~1000× faster on a GPU.

This is why every operation in deep learning is expressed as matrix multiplications. Convolutions are matmuls. Attention is matmuls. The forward pass of a 100-layer Transformer is, in the end, a few hundred matmuls strung together. Get comfortable with matrix shapes — you will be reading and writing them every day for the rest of this course.

03 — Choosing a loss
Choosing a loss — what does "best fit" mean?

We have a model with parameters w and b. We need a way to measure how wrong its predictions are, so we can search for parameters that make it less wrong. That measurement is the loss function.

For regression — predicting a continuous number — the most natural loss is mean squared error (MSE):

L(w, b) = 1 n Σi=1n (ŷi − yi)2 (3.1)

Read it as: average, across all n examples, the squared difference between prediction and truth. Squaring serves two purposes — it makes errors positive (so they don't cancel out), and it punishes large errors much more harshly than small ones. A single very wrong prediction contributes much more than 100 slightly-wrong ones.

Why squared, specifically?

You could also average absolute errors (the L1 loss). Why squared? Three reasons:

  1. Differentiability. The square function has a derivative everywhere. The absolute value has a kink at zero where the derivative is undefined.
  2. Convexity in linear models. MSE on a linear model produces a parabolic loss surface — one global minimum, no local traps.
  3. Probabilistic justification. If you assume Gaussian noise on your targets, minimizing MSE is exactly equivalent to maximum likelihood estimation. So MSE is not arbitrary — it has a principled probabilistic origin.
The deep connection — losses are likelihoods in disguise

Almost every loss function in deep learning corresponds to a probabilistic assumption about the data. MSE assumes Gaussian noise. Cross-entropy (which we'll meet in Module 1.3) assumes categorical outcomes. KL divergence measures distance between distributions. Choosing a loss is choosing a model of how your data is generated — and once you see this, loss functions stop being arbitrary recipes and become principled choices.

Computing MSE in PyTorch.

import torch

# Predictions and true values for a CineMatch batch
y_pred = torch.tensor([0.6, 0.8, 0.3, 0.9])
y_true = torch.tensor([0.7, 0.5, 0.4, 0.85])

# Manual computation
loss = ((y_pred - y_true) ** 2).mean()
print(loss.item())   # 0.0269

# Equivalent built-in
import torch.nn.functional as F
loss = F.mse_loss(y_pred, y_true)
print(loss.item())   # 0.0269

04 — Gradient descent
Gradient descent — making the loss smaller, one step at a time.

We have a model. We have a loss function. We have parameters w and b that determine the loss. The optimization problem is: find values of w and b that make the loss as small as possible.

For linear regression, this problem has a closed-form solution — the famous normal equations — but we will not use them, because that approach does not generalize to deep networks. Instead, we will use the algorithm that scales to billions of parameters: gradient descent.

The algorithm in three lines.

  1. Start with random initial values for w and b.
  2. Compute the gradient of the loss with respect to each parameter.
  3. Take a small step in the opposite direction. Repeat.

In math:

w ← w − η · ∇wL (4.1)
b ← b − η · ∂L/∂b (4.2)

The Greek letter η ("eta") is the learning rate — a small positive number, typically 0.001 to 0.1, that controls how big each step is. Too small and training crawls; too large and it overshoots and diverges. We saw both behaviors in the gradient descent visualizer in Module 0.3.

Computing the gradients by hand — once.

Just so we know what autograd is doing for us, let's derive the gradients of MSE for linear regression on a single example. Let ei = ŷi − yi be the error on example i. Then per-example loss is ei2, and:

∂L∂wj = 2n Σi ei · xi,j (4.3)
∂L∂b = 2n Σi ei (4.4)

Read these in plain English. The gradient with respect to weight j is two-over-n times the average of (error × that feature). The gradient with respect to bias is two-over-n times the average error. Both are simple averages. Both have a clear interpretation: if a feature was large on examples where you over-predicted, decrease that weight.

Worked example — one SGD step on CineMatch linear regression
Current weights: w = [0.003, 0.5, −0.2, 0.1], b = 0.4. Learning rate η = 0.014 features: runtime, genre_match, recency, popularity
Mini-batch of 2 examples: x₁ = [120, 0.8, 0.3, 0.6], y₁ = 0.75. x₂ = [90, 0.2, 0.9, 0.4], y₂ = 0.30
Predictions: ŷ₁ = 0.003·120 + 0.5·0.8 + (−0.2)·0.3 + 0.1·0.6 + 0.4 = 0.36+0.40−0.06+0.06+0.4 = 1.16
ŷ₂ = 0.003·90 + 0.5·0.2 + (−0.2)·0.9 + 0.1·0.4 + 0.4 = 0.27+0.10−0.18+0.04+0.4 = 0.63
Errors: e₁ = 1.16 − 0.75 = 0.41 (over-predicted). e₂ = 0.63 − 0.30 = 0.33 (over-predicted)MSE = (0.41² + 0.33²)/2 = (0.168 + 0.109)/2 = 0.139
Gradient ∂L/∂w₀ = (2/2) × (0.41·120 + 0.33·90) / 2 = (49.2 + 29.7) = 39.45Feature 0 (runtime) is large → gets a big gradient. This will shrink w₀.
Update: w₀_new = 0.003 − 0.01 × 39.45 = 0.003 − 0.395 = −0.392Massive update! The learning rate is too high for unnormalized features. This is why feature scaling matters — Module 0.4's StandardScaler prevents this.

And here is the punchline: autograd computes exactly these formulas, automatically, for any loss and any model. We derived them by hand once so you'd see what's underneath, but you will never need to derive them again.

Stochastic gradient descent (SGD) — and why we use it.

The formulas above sum over all n examples to compute one gradient step. For our 200,000-example CineMatch dataset, that's slow. The breakthrough idea, decades old, is: don't use all n examples for each step. Use a small random subset — a mini-batch, typically 32 to 256 examples — and take many more, smaller, noisier steps.

This is stochastic gradient descent, and it has two huge benefits: each step is fast (you only touch a handful of examples), and the noise actually helps escape bad regions of the loss surface. Every modern deep learning training loop is built on SGD or one of its descendants (Adam, AdamW — we'll meet those in Phase 2).

A subtle thing

The learning rate must shrink as the model gets close to a minimum.

SGD's noise means you'll always be jittering around the optimum, never reaching it exactly. Production training schedules usually start with a moderate learning rate (0.01 or so) and decrease it as training proceeds. We'll meet schedulers in Phase 2; for now, be aware that constant learning rates are a starting point, not a final answer.

05 — Three implementations
Linear regression, three ways.

Time to build it. We'll implement the same model three times, each one more idiomatic than the last. By the end, you'll have written every line and you'll see exactly what PyTorch is hiding in each abstraction.

Way 1 — Pure NumPy, gradients by hand.

This version uses the formulas we derived in Section 4. No frameworks, no autograd. This is what every linear regression has been since the 1800s.

import numpy as np

# Synthetic data: 200 examples, 1 feature
np.random.seed(42)
N = 200
X = np.random.uniform(70, 200, size=(N, 1))    # runtime in minutes
y_true_w, y_true_b = 0.003, 0.2
y = y_true_w * X.flatten() + y_true_b + np.random.normal(0, 0.05, N)

# Initialize parameters randomly
w = np.random.randn(1) * 0.01
b = 0.0

# Hyperparameters
lr = 0.0001
n_epochs = 100

for epoch in range(n_epochs):
    # Forward pass — compute predictions and loss
    y_pred = X @ w + b               # shape [N]
    error = y_pred - y               # shape [N]
    loss = (error ** 2).mean()

    # Backward pass — compute gradients by hand
    grad_w = (2 / N) * (X.T @ error)   # shape [1]
    grad_b = (2 / N) * error.sum()

    # Update parameters
    w -= lr * grad_w
    b -= lr * grad_b

    if epoch % 20 == 0:
        print(f"epoch {epoch:3d}  loss={loss:.5f}  w={w[0]:.5f}  b={b:.4f}")

print(f"True w={y_true_w}, learned w={w[0]:.5f}")
print(f"True b={y_true_b}, learned b={b:.4f}")

Way 2 — PyTorch with autograd, manual training loop.

Same algorithm, same data, same answer. The difference: PyTorch's autograd computes the gradients for us. The model definition is one line, and we don't write any derivative formulas.

import torch

torch.manual_seed(42)
N = 200
X = torch.empty(N, 1).uniform_(70, 200)
y = 0.003 * X.squeeze() + 0.2 + torch.randn(N) * 0.05

# Parameters — requires_grad=True so autograd tracks them
w = torch.randn(1, requires_grad=True)
b = torch.zeros(1, requires_grad=True)

lr = 0.0001
for epoch in range(100):
    y_pred = X @ w + b                          # forward
    loss = ((y_pred - y) ** 2).mean()       # MSE

    loss.backward()                            # autograd computes gradients

    with torch.no_grad():                     # manually update — no gradient tracking
        w -= lr * w.grad
        b -= lr * b.grad
        w.grad.zero_()                         # reset for next iteration
        b.grad.zero_()

    if epoch % 20 == 0:
        print(f"epoch {epoch:3d}  loss={loss.item():.5f}")

Notice the torch.no_grad() block — when we update parameters by subtracting their gradients, we don't want PyTorch to track those operations as part of the graph. The w.grad.zero_() call is the manual version of optimizer.zero_grad() that we discussed in Module 0.3.

Way 3 — PyTorch with nn.Module and an optimizer.

This is the version you'll use in production. Same model, same algorithm, but expressed in PyTorch's idioms — clean, composable, and ready to scale up to deep networks.

import torch
import torch.nn as nn

torch.manual_seed(42)
N = 200
X = torch.empty(N, 1).uniform_(70, 200)
y = (0.003 * X.squeeze() + 0.2 + torch.randn(N) * 0.05).unsqueeze(1)

# Define the model — one line for linear regression
model = nn.Linear(in_features=1, out_features=1)

# Loss and optimizer
criterion = nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.0001)

# Training loop — the canonical four lines
for epoch in range(100):
    optimizer.zero_grad()                # 1. clear old gradients
    y_pred = model(X)                     # 2. forward pass
    loss = criterion(y_pred, y)         # 3. compute loss
    loss.backward()                       # 4. backward pass + step
    optimizer.step()

    if epoch % 20 == 0:
        print(f"epoch {epoch:3d}  loss={loss.item():.5f}")

print(f"learned w={model.weight.item():.5f}  b={model.bias.item():.4f}")
The four lines you'll write a million times

optimizer.zero_grad(), y_pred = model(x), loss = criterion(y_pred, y), loss.backward(), optimizer.step(). This is the canonical PyTorch training loop. Every model in this course — convolutional, recurrent, transformer, generative — has exactly this loop at its core. The only thing that changes is the definition of model and criterion.

06 — Practice & checkpoint
Putting it on CineMatch — and the checkpoint.

Now we apply Way 3 to the actual CineMatch dataset we generated in Module 0.4. The only difference: we have 4 features instead of 1, and we use a proper DataLoader for batching.

import torch
import torch.nn as nn
from torch.utils.data import DataLoader, random_split
from cinematch.data.dataset import CineMatchDataset
from cinematch.utils.seed import set_seed
from cinematch.utils.device import get_device

set_seed(42)
device = get_device()

# Load the CineMatch dataset and split into train / val
ds = CineMatchDataset()
train_ds, val_ds = random_split(ds, [0.8, 0.2])
train_loader = DataLoader(train_ds, batch_size=256, shuffle=True)
val_loader = DataLoader(val_ds, batch_size=256)

# Linear regression on 4 features → 1 output
model = nn.Linear(4, 1).to(device)
criterion = nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=1e-5)

for epoch in range(10):
    # Train
    model.train()
    train_loss = 0.0
    for xb, yb in train_loader:
        xb, yb = xb.to(device), yb.to(device).unsqueeze(1)
        optimizer.zero_grad()
        loss = criterion(model(xb), yb)
        loss.backward()
        optimizer.step()
        train_loss += loss.item() * xb.size(0)
    train_loss /= len(train_ds)

    # Validate
    model.eval()
    val_loss = 0.0
    with torch.no_grad():
        for xb, yb in val_loader:
            xb, yb = xb.to(device), yb.to(device).unsqueeze(1)
            val_loss += criterion(model(xb), yb).item() * xb.size(0)
    val_loss /= len(val_ds)

    print(f"epoch {epoch+1:2d}  train_mse={train_loss:.5f}  val_mse={val_loss:.5f}")

# Inspect what the model learned
for name, p in model.named_parameters():
    print(name, p.data)

When you run this, you'll see the loss drop quickly in the first few epochs and then plateau. The plateau is meaningful — it's where a linear model has done as well as it can on data that has non-linear structure (remember from Module 0.4 that our generator used abs(runtime - attention_span), which is non-linear). The gap between this baseline and what an MLP achieves in Module 2.1 will be your first concrete demonstration of what depth buys you.
