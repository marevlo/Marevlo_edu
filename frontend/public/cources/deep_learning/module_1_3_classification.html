From regression to classification. Predicting categories.

Most ML problems aren't "how much" — they're "which one." Will the user finish the movie? Is this email spam? Which genre is this poster? In this module we shift from regression to binary classification using the same single-neuron model — adding only a sigmoid output and a new loss function.

01 — The shift
The shift — from "how much" to "which one."

Linear regression predicts a number on the real line. CineMatch's watch-fraction is a number between 0 and 1; its predicted value is somewhere in (more or less) that range. But many real questions are different in kind. Not "how long will the user watch?" but "will they finish the movie or not?" Yes or no. Two categories. The output is a label, not a quantity.

This is classification, and it requires three changes from regression:

AspectRegressionClassification
Output typeReal number ŷ ∈ ℝProbability p ∈ [0, 1]
Loss functionMean squared errorCross-entropy
Final predictionŷ itself1 if p > 0.5, else 0

What does not change: the linear model itself, the gradient descent algorithm, the canonical four-line training loop. We're swapping the output layer and the loss — everything else from Module 1.1 carries over.

CineMatch's first classification problem

"Did the user finish the movie?"

Recall from Module 0.4 that our synthetic dataset includes a finished column — a 0/1 label that is 1 whenever the watch-fraction was at least 0.85. This gives us a perfectly clean binary classification problem on the same data: given the user's age, attention span, the movie's runtime, and its critic score, predict whether the user finished it. Same features, different question, completely different machinery on the output side.

02 — The sigmoid
The sigmoid — squashing a real number into a probability.

Our linear model produces a raw score z = wTx + b. That score lives on the real line — it can be anything from minus infinity to plus infinity. But we want a probability — a number between 0 and 1. We need a function that takes any real number and maps it into that range, smoothly. The standard choice is the sigmoid (also called the logistic function):

σ(z) = 1 1 + e−z (2.1)

It has three crucial properties: at z = 0, σ(0) = 0.5 — the natural decision boundary. As z → +∞, σ(z) → 1. As z → −∞, σ(z) → 0. Smoothly. So we can interpret z as the model's "confidence score" and σ(z) as the corresponding probability.

Figure 1. The sigmoid function. It takes any real number z and squashes it into a probability between 0 and 1, with σ(0) = 0.5 as the natural decision boundary.

The full classification model in one line.

p(y = 1 | x) = σ(wTx + b) (2.2)

Read it as: "the probability that y equals 1, given the input x, is the sigmoid of the linear score." This model is called logistic regression — a confusing historical name, since it's used for classification, not regression. Despite the name, it's the simplest neural network classifier that exists: a single neuron with a sigmoid activation.

import torch

# Linear score for one CineMatch user-movie pair
x = torch.tensor([35.0, 90.0, 120.0, 0.7])  # age, attn, runtime, critic
w = torch.tensor([0.01, 0.02, -0.005, 2.0])
b = torch.tensor(-2.5)

z = x @ w + b                # raw score
p = torch.sigmoid(z)         # probability of finishing
prediction = (p > 0.5).int()  # 0 or 1

print(f"score z = {z.item():.3f}")
print(f"P(finish) = {p.item():.3f}")
print(f"prediction = {prediction.item()}")

03 — The right loss
Why MSE is wrong here — and what's right.

You might think: now that we have a probability, just use MSE between the predicted probability and the true label (0 or 1). It would almost work. But there's a problem.

Sigmoid + MSE produces a non-convex loss surface in the parameters. Gradient descent can get stuck. Worse: when the model is confidently wrong (say, predicts p = 0.99 when the truth is 0), the gradient through MSE is small — because the sigmoid is in its flat region. So the model barely corrects. We'd want a loss that says "you are wrong with high confidence — wake up and fix this fast."

That loss exists, and it is binary cross-entropy — also called log-loss. For a single example with true label y ∈ {0, 1} and predicted probability p:

L = − [y · log(p) + (1 − y) · log(1 − p)] (3.1)

Read it term by term. If y = 1, only the first term survives: L = −log(p). If the model predicted p close to 1 (correct and confident), −log(p) is near 0 — small loss. If the model predicted p close to 0 (confidently wrong), −log(p) blows up to infinity — huge loss. Symmetrically for y = 0.

Why this loss is the right one — three reasons.

  1. It punishes confident wrongness severely. A loss that goes to infinity when you're wrong-with-conviction is exactly what you want during training.
  2. Combined with sigmoid, the loss surface is convex. Just like linear regression, training is well-behaved — gradient descent converges to the global optimum.
  3. Probabilistic justification. Cross-entropy is the negative log-likelihood of the data under a Bernoulli model. Minimizing it is exactly maximum-likelihood estimation — a principled probabilistic foundation, just like MSE corresponds to Gaussian noise.
The pairing matters

Sigmoid + MSE = wrong (non-convex, slow gradients). Sigmoid + cross-entropy = right (convex, well-behaved gradients). Linear + MSE = right (convex, well-behaved). Linear + cross-entropy = nonsensical (cross-entropy needs probabilities). The lesson: output activation and loss function come as a pair. Each loss has its natural output activation, and using the right pairing eliminates a whole class of training problems.

The miraculous gradient.

Here is something elegant. Despite the messy-looking definitions of sigmoid and cross-entropy individually, when you compose them and differentiate, an enormous amount cancels out. The gradient of binary cross-entropy with respect to the score z reduces to:

∂L∂z = σ(z) − y = p − y (3.2)

Just p − y: predicted probability minus true label. The error itself, as a number. Same shape as the gradient for MSE on linear regression. This is why classification with sigmoid + cross-entropy plugs into the same training loop with no surprises.

Worked example — full forward + loss + gradient for one CineMatch example
Weights: w = [0.8, 1.2, −0.5, 0.3], b = −0.2. Features: x = [0.6, 0.9, 0.2, 0.7]. True label: y = 1 (user will rate 4+)
Score: z = w·x + b = 0.48 + 1.08 − 0.10 + 0.21 − 0.2 = 1.47
Probability: p = σ(1.47) = 1/(1+exp(−1.47)) = 1/(1+0.229) = 0.813Model thinks 81.3% chance user will rate 4+ — reasonable but not certain
BCE loss: L = −[y·log(p) + (1−y)·log(1−p)] = −[1·log(0.813)] = −(−0.207) = 0.207Moderate loss — model is right but not maximally confident
Gradient: ∂L/∂z = p − y = 0.813 − 1 = −0.187. ∂L/∂w = (p−y)·x = −0.187·[0.6, 0.9, 0.2, 0.7] = [−0.112, −0.168, −0.037, −0.131]All negative → SGD will increase all weights → z increases → p increases toward 1. Correct direction!
Update (η=0.1): w_new = w − 0.1·∂L/∂w = [0.811, 1.217, −0.496, 0.313]Small nudge. After thousands of such steps on the full dataset, the weights converge.
import torch
import torch.nn.functional as F

# A small CineMatch batch — predicted probabilities and true labels
p = torch.tensor([0.92, 0.15, 0.40, 0.68])
y = torch.tensor([1, 0, 1, 1], dtype=torch.float32)

# Manual binary cross-entropy
loss = -(y * torch.log(p) + (1 - y) * torch.log(1 - p)).mean()
print(f"manual: {loss.item():.4f}")

# Built-in equivalent (much more numerically stable, see pitfall below)
loss = F.binary_cross_entropy(p, y)
print(f"built-in: {loss.item():.4f}")
Numerical stability

Use BCEWithLogitsLoss, not BCELoss(sigmoid(z)).

If the model produces a very large positive z, sigmoid(z) rounds to exactly 1.0 in float32. Then log(1 - 1.0) is log(0) = −∞, and the loss explodes to NaN. PyTorch's BCEWithLogitsLoss takes the raw z and computes sigmoid + cross-entropy together, with the log-sum-exp trick built in for numerical safety. Always use it. Pass logits — not probabilities — to your loss function.

04 — Decision boundary
The decision boundary — where the model switches its mind.

For a 2-feature classifier, we can visualize what the model has learned. The set of inputs where p = 0.5 — equivalently, where z = wTx + b = 0 — is the decision boundary. On one side of it the model predicts 1, on the other it predicts 0.

For a linear classifier with two features, the decision boundary is a straight line in feature space:

w1x1 + w2x2 + b = 0 (4.1)

For three features it's a plane. For d features it's a (d−1)-dimensional hyperplane. The geometry of the boundary tells you what kind of problems the model can solve. A linear classifier can only carve up its input space with a single straight cut.

Figure 2. A decision boundary learned for "did the user finish?" Purple dots are users who finished, red dots are those who didn't. The dashed line is where p = 0.5 — the model's switch point. A linear model can only draw a single straight line.

Linearly separable, and not.

The example above is linearly separable — there exists some line that perfectly separates the purple dots from the red ones. Linear classifiers can solve linearly separable problems perfectly. But many real classification problems aren't linearly separable. The classic non-separable example is XOR (we'll see it formally in Module 1.4), and most real-world data has some entanglement that no single line can resolve.

When that happens, you have two options. Engineer non-linear features by hand (compute things like x12, x1·x2, etc., and feed them in) — the pre-2010 approach. Or stack non-linear layers and let the network learn its own features — the deep learning approach. Phase 2 onward.

05 — On CineMatch
Building it on CineMatch.

Now we put everything together. The model is one line; the training loop is identical to Module 1.1's; only the loss and final activation change.

import torch
import torch.nn as nn
from torch.utils.data import DataLoader, random_split
import pandas as pd
from torch.utils.data import Dataset
from cinematch.utils.seed import set_seed
from cinematch.utils.device import get_device


class CineMatchClassDataset(Dataset):
    """Same features as before, but target is the binary 'finished' label."""
    def __init__(self, data_dir="data/raw"):
        users = pd.read_parquet(f"{data_dir}/users.parquet")
        movies = pd.read_parquet(f"{data_dir}/movies.parquet")
        inter = pd.read_parquet(f"{data_dir}/interactions.parquet")
        self.df = inter.merge(users, on="user_id").merge(movies, on="movie_id")
        self.feat = ["age", "attention_span_min", "runtime_min", "critic_score"]
        # Standardize features — important for classification convergence
        self.mean = self.df[self.feat].mean()
        self.std = self.df[self.feat].std()

    def __len__(self): return len(self.df)

    def __getitem__(self, idx):
        row = self.df.iloc[idx]
        x_raw = row[self.feat]
        x = ((x_raw - self.mean) / self.std).values.astype("float32")
        y = float(row["finished"])
        return torch.tensor(x), torch.tensor(y)


set_seed(42); device = get_device()
ds = CineMatchClassDataset()
train_ds, val_ds = random_split(ds, [0.8, 0.2])
train_loader = DataLoader(train_ds, batch_size=256, shuffle=True)
val_loader = DataLoader(val_ds, batch_size=256)

# Model — same as linear regression, but interpreted as classifier (no sigmoid here!)
model = nn.Linear(4, 1).to(device)

# Loss applies sigmoid internally for numerical stability — pass logits
criterion = nn.BCEWithLogitsLoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.05)

for epoch in range(10):
    # Train
    model.train()
    train_loss, train_correct, n_train = 0.0, 0, 0
    for xb, yb in train_loader:
        xb, yb = xb.to(device), yb.to(device).unsqueeze(1)
        optimizer.zero_grad()
        logits = model(xb)
        loss = criterion(logits, yb)
        loss.backward()
        optimizer.step()
        train_loss += loss.item() * xb.size(0)
        preds = (torch.sigmoid(logits) > 0.5).int()
        train_correct += (preds == yb.int()).sum().item()
        n_train += xb.size(0)

    # Validate
    model.eval()
    val_loss, val_correct, n_val = 0.0, 0, 0
    with torch.no_grad():
        for xb, yb in val_loader:
            xb, yb = xb.to(device), yb.to(device).unsqueeze(1)
            logits = model(xb)
            val_loss += criterion(logits, yb).item() * xb.size(0)
            preds = (torch.sigmoid(logits) > 0.5).int()
            val_correct += (preds == yb.int()).sum().item()
            n_val += xb.size(0)

    print(f"epoch {epoch+1:2d}  "
          f"train: loss={train_loss/n_train:.4f} acc={train_correct/n_train:.3f}  "
          f"val: loss={val_loss/n_val:.4f} acc={val_correct/n_val:.3f}")

Run this and you'll see two new things compared to Module 1.1's regression. First, the loss is now reported in cross-entropy (not MSE), which is in different units. Second, we now also report accuracy — the fraction of predictions that matched the true label after thresholding at 0.5. Accuracy is what humans understand; loss is what the model optimizes.

What you'll observe

The pattern is similar to regression — and so is the ceiling.

Train and validation accuracy will rise together (no overfitting on this much data), and they'll plateau around 88-92% depending on initialization. That ceiling exists for the same reason as the regression ceiling: the underlying generator includes non-linear effects (the absolute-value attention/runtime fit) that a linear classifier cannot perfectly capture. In Phase 2 you will add a hidden layer with ReLU activations and watch this ceiling rise — concrete proof, again, that depth helps.

Beyond accuracy — what to actually measure.

Accuracy is intuitive but can be misleading. If 95% of CineMatch users didn't finish a particular movie, a "predict 0 always" model is 95% accurate but useless. Two metrics that capture more:

  • Precision — of the times the model said "finished," how often was it right? Captures false-positive rate.
  • Recall — of the actual "finished" cases, how many did the model catch? Captures false-negative rate.

For balanced datasets like CineMatch, accuracy is fine. For imbalanced ones (fraud detection, rare disease diagnosis), precision and recall — and their combination, F1 — are essential. We'll add these to the metrics module in Phase 2.

06 — Multi-class
From two classes to many — and the checkpoint.

Two-class classification covers a lot, but many problems have more categories. Genre prediction has six classes. MNIST digit classification has ten. Generalizing from binary requires two small extensions:

Softmax — sigmoid for many classes.

For K classes, the model produces K raw scores (one per class), and we squash them into a probability distribution that sums to 1:

softmax(z)k = ezk Σj ezj (6.1)

Read it as: take the exponential of each score, then normalize so they sum to 1. The result is a vector of probabilities — the highest one is the predicted class. For two classes, softmax reduces to sigmoid (try the algebra). For one class, both reduce to identity. The hierarchy is consistent.

Categorical cross-entropy — for multi-class.

The multi-class generalization of binary cross-entropy. For a true class k and predicted distribution p:

L = − log(pk) (6.2)

It's just the negative log probability assigned to the correct class. If the model gives the correct class a high probability, the loss is small; if it gives it a low probability, the loss is large. PyTorch's nn.CrossEntropyLoss combines softmax + categorical cross-entropy with the same numerical-stability trick — pass it logits, get back a stable loss.

import torch
import torch.nn as nn

# 6-class CineMatch genre classifier — predict primary genre from features
model = nn.Linear(4, 6)         # 6 logits per example
criterion = nn.CrossEntropyLoss()  # softmax + log, internally

x = torch.randn(32, 4)            # batch of 32
y = torch.randint(0, 6, (32,))    # integer class labels 0..5
logits = model(x)                # [32, 6]
loss = criterion(logits, y)      # scalar

Notice: the labels y are integer indices, not one-hot vectors. PyTorch's CrossEntropyLoss handles the one-hot conversion internally, which saves a lot of memory at scale. Some other frameworks differ — always check what label format your loss expects.

The output-loss pairings to memorize

Linear output + MSE → regression (real-valued output). Sigmoid output + binary cross-entropy → binary classification. Softmax output + categorical cross-entropy → multi-class classification. Three patterns cover ≈ 90% of supervised learning tasks. The architecture and training loop barely change between them — only the last layer and the loss.
