Handling Categories
and Mixed Data

K-Means speaks numbers. K-Medoids is more robust, but it still needs a usable distance. Real databases are rarely that cooperative. They contain colors, cities, device types, plans, symptoms, and ratings sitting beside age, income, or usage. This part rebuilds partitioning for that reality through K-Modes and K-Prototypes.

Modes
01 — The New Problem

What happens when subtraction stops making sense

In numerical data, distance feels natural. You can subtract one value from another, square it, compare magnitudes, and move forward. But categories do not behave that way. You cannot subtract Blue from Brown. You cannot average London and Tokyo.

This is why K-Means fails immediately on pure categorical tables. K-Medoids can sometimes be forced to work through a custom dissimilarity table, but that quickly becomes awkward or expensive. K-Modes and K-Prototypes were proposed precisely for this gap.

Mental Model
When numbers disappear, the center itself has to change.
K-Means uses a mean. K-Medoids uses an actual point. K-Modes uses the most frequent category pattern. K-Prototypes combines that categorical logic with ordinary numerical averaging.

The whole family is still partitioning. What changes is the language used to describe “similar” and “central.”

02 — K-Modes

Clustering when the data is made of labels, not magnitudes

K-Modes keeps the assign-and-update rhythm of K-Means, but changes two core ingredients: the center becomes a mode rather than a mean, and the distance becomes a mismatch count rather than Euclidean distance.

A mode is simply the most frequent value in a set. If a cluster contains many rows and the most common value in the Color column is Red, then Red becomes the cluster's representative value for that column.

The mismatch function

K-Modes measures dissimilarity by counting how many categorical attributes disagree between a row and a cluster mode. Every mismatch contributes one unit. Every match contributes zero.

Simple Matching Dissimilarity
Not “how far,” but “how many places do these differ?”
For purely categorical data, this is often the cleanest possible starting point. The model is not trying to invent a numerical geometry. It is simply counting disagreement across fields.

Algorithm steps

1

Initialize

Pick k starting rows or starting modes for the clusters.

2

Assign

For every row, count mismatches to each mode and assign it to the cluster with the smallest mismatch score.

3

Update

Within each cluster, compute the most frequent value in every categorical column. Those values form the new mode.

4

Repeat

Continue reassigning and updating until the modes stop changing or the objective stops improving.

Walkthrough for a categorical table
Dataset: (Red, Small), (Red, Large), (Green, Large), (Green, Large), (Red, Small). Let k = 2. Start with Mode 1 = (Red, Small) and Mode 2 = (Green, Large).
Step 1 — Assignment
Row 2 = (Red, Large) gives 1 mismatch with Mode 1 and 1 mismatch with Mode 2.
Row 4 = (Green, Large) gives 2 mismatches with Mode 1 and 0 with Mode 2.
Row 5 = (Red, Small) gives 0 with Mode 1 and 2 with Mode 2.
Step 2 — Cluster formation
Cluster 1 contains Rows 1, 2, 5.
Cluster 2 contains Rows 3, 4.
Step 3 — Update the modes
Cluster 1 colors = Red, Red, Red → mode is Red.
Cluster 1 sizes = Small, Large, Small → mode is Small.
Cluster 2 colors = Green, Green → mode is Green.
Cluster 2 sizes = Large, Large → mode is Large.
The modes do not change, so the procedure stops. This is the categorical analogue of the familiar K-Means loop.

Python implementation

python
# pip install kmodes
import numpy as np
from kmodes.kmodes import KModes

data = np.array([
    ['Red', 'Small', 'Square'],
    ['Red', 'Small', 'Square'],
    ['Green', 'Large', 'Circle'],
    ['Green', 'Large', 'Circle'],
    ['Red', 'Large', 'Square']
])

km = KModes(n_clusters=2, init='Huang', n_init=5, verbose=1)
labels = km.fit_predict(data)

print(labels)
print(km.cluster_centroids_)

Strengths and limitations

StrengthWhy it mattersLimitation
Works directly on categorical fieldsNo need to force everything into fake numeric formAll mismatches are treated equally
Fast and scalableKeeps the efficient partitioning spirit of K-MeansInitialization can affect the result
Easy to interpretCluster centers are described column by column through frequent valuesOrdinal information like Low, Medium, High is ignored unless encoded carefully

Two details worth knowing

The basic version counts all mismatches equally. That makes it simple and fast, but also blunt. In real tables, not every attribute deserves the same weight.

Initialization matters too. Starting with poor modes can push the algorithm into weaker local solutions, which is why practitioners often try several restarts.

03 — Real Uses for K-Modes

Where pure categories are the whole story

K-Modes becomes useful whenever the columns themselves are labels, flags, or choices rather than measurements. It is especially natural in segmentation problems where each field is a descriptor of state or identity.

Customer profiles by country, profession, device type, and plan
Medical symptom patterns such as fever, cough, fatigue, and diagnosis markers
Survey response grouping across preference categories
Operational incident grouping by type, source, and severity label

If your categories carry order, such as Low, Medium, High, then plain mismatch counting throws that order away completely. Low vs Medium and Low vs High are both treated as one mismatch.

04 — K-Prototypes

When the table has both numbers and labels

Real business data is usually mixed. A single row may contain Age, Income, Plan Type, and City together. K-Means can only hear the numbers. K-Modes can only hear the labels. K-Prototypes combines both worlds in one distance expression.

Its idea is direct. Use a numerical distance for numerical columns. Use a mismatch score for categorical columns. Add them together. Because these two parts live on very different scales, introduce a balancing weight called gamma.

The Combined Distance
Numerical difference + gamma × categorical mismatch
The numerical part is usually Euclidean-style. The categorical part is usually simple matching. Gamma decides how loudly the categorical side speaks relative to the numerical side.

Algorithm steps

1

Initialize prototypes

Choose k starting rows to act as hybrid centers.

2

Assign using the combined distance

For each row, compute numerical difference plus gamma-weighted categorical mismatch to every prototype.

3

Update the prototype

Take the mean of the numerical columns and the mode of the categorical columns within each cluster.

4

Repeat to convergence

Continue until assignments or prototypes stop changing materially.

Walkthrough for mixed data
Dataset: (20, Red), (22, Red), (60, Blue), (62, Blue). Let k = 2 and gamma = 10. Start with Prototype 1 = (20, Red) and Prototype 2 = (60, Blue).
Step 1 — Assign Row 2 = (22, Red)
Distance to Prototype 1 = (22 − 20)^2 + 10 × 0 = 4.
Distance to Prototype 2 = (60 − 22)^2 + 10 × 1 = 1444 + 10 = 1454.
Step 2 — Form the clusters
Rows 1 and 2 stay together.
Rows 3 and 4 stay together.
Step 3 — Update the prototypes
Cluster 1 prototype becomes (21, Red).
Cluster 2 prototype becomes (61, Blue).
This is the defining pattern of K-Prototypes: mean on the numeric side, mode on the categorical side, tied together through gamma.

Python implementation

python
# pip install kmodes
import pandas as pd
from kmodes.kprototypes import KPrototypes

data = [
    [25, 50000, 'Basic'],
    [26, 52000, 'Basic'],
    [45, 120000, 'Premium'],
    [48, 125000, 'Premium'],
    [22, 48000, 'Basic']
]

df = pd.DataFrame(data, columns=['Age', 'Income', 'Plan'])

kp = KPrototypes(n_clusters=2, init='Cao', n_init=5, verbose=1)
labels = kp.fit_predict(df, categorical=[2])

print(labels)
print(kp.cluster_centroids_)

What gamma really controls

Gamma is not just a tuning constant. It is a statement about what you want the algorithm to care about. If gamma is too small, the numerical features dominate and categories barely matter. If gamma is too large, the categorical mismatches dominate and the numbers fade into the background.

There is no universal perfect gamma. It often requires trial, validation, domain knowledge, or scaling logic to make the mixed distance behave sensibly.

05 — K-Modes vs K-Prototypes

The split is simple once you see the table clearly

SituationUseWhy
All columns are categoriesK-ModesThe mismatch logic and mode update are designed exactly for this case
Columns are mixed: numeric and categoricalK-PrototypesIt combines numerical and categorical comparisons in one objective
Data has only numbersK-Means or K-MedoidsNo need for categorical machinery
One-line summary
K-Modes speaks labels. K-Prototypes speaks labels and measurements together.
That is the cleanest practical distinction to keep in mind when choosing between them.
06 — Use Cases

Where these methods become genuinely useful

K-Modes and K-Prototypes are not niche curiosities. They become useful exactly where many business and medical datasets live: rows made of mixed descriptive attributes and measured values.

Demographic segmentation with country, profession, device, and plan
Healthcare grouping with blood pressure, cholesterol, blood type, and symptom flags
Retail analysis with annual spend and membership tier
Operations data with severity levels, source type, and numeric usage counters

In a retail setting, for instance, pure numeric clustering may split customers by spend alone while missing the fact that a strong behavior boundary is actually driven by membership tier or product preference. K-Prototypes lets both views speak at once.

07 — Research Layer

Where later work improved the basic story

The original ideas solved the core problem. Later work kept pushing on three recurring pain points: better initialization, better categorical dissimilarity, and better handling of uncertainty or fuzziness.

Initialization matters more than people expect

Density-aware starting points can reduce the instability of random initialization. This is why many implementations offer alternatives to purely random seeds.

Not all mismatches should count equally

Plain simple matching can be too blunt. In real data, some columns matter much more than others, and some rare category agreements are more informative than common ones.

Mixed data can also be approached other ways

K-Prototypes is attractive because it is direct, scalable, and interpretable, but it is not the only route. Some workflows build mixed-data dissimilarities first and cluster afterward through a different method.

08 — Decision Framework

Choosing carefully before you run anything

If your data looks like…Use…Watch out for…
Only categories like city, profession, plan, or symptom statusK-ModesOrdinal categories and initialization sensitivity
Numbers plus categories in the same tableK-PrototypesGamma tuning and feature scaling
Only numerical featuresK-Means or K-MedoidsOutliers, non-convex structure, and scaling

Most errors in mixed-data clustering happen before the algorithm starts: poor feature treatment, poor scaling, or pretending that categories are ordinary numbers.
