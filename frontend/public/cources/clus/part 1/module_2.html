Measuring Distance

Clustering is fundamentally about which points are "close" and which are "far." But "close" has many definitions — and choosing wrong means your clusters are meaningless.

Explore
01 — Euclidean Distance

The Straight Line Between Two Points

The most intuitive distance: the length of a straight line connecting two points. It's the default in most clustering implementations — and often the wrong choice.

Euclidean Distance
d(a,b) = √( Σ (aᵢ − bᵢ)² )
Square each dimension's difference, sum them, take the square root. Works in any number of dimensions. This is the L₂ norm of the difference vector.

The scale trap: If feature A ranges 20–80 and feature B ranges 10,000–200,000, Euclidean distance is almost entirely determined by B. The squaring amplifies large-scale features.

📐 Math Workout — Euclidean Distance in 3D
Problem: Customer A = (25, 50000, 7) and Customer B = (30, 52000, 3).
Features: age, income (₹), satisfaction (1-10).
Compute Euclidean distance. Which feature dominates?
Step 1 — Per-dimension differences
Δ_age = (25−30)² = 25
Δ_income = (50000−52000)² = 4,000,000
Δ_satisfaction = (7−3)² = 16
Step 2 — Sum and root
d = √(25 + 4,000,000 + 16) = √4,000,041 ≈ 2000.01
Income contributes 99.999% of the distance! Age and satisfaction are invisible.
This is why you MUST scale before computing Euclidean distance in clustering.
02 — Manhattan Distance

The Grid-Walk Distance

Instead of a straight line, Manhattan distance travels along axes — like walking city blocks. It's the sum of absolute differences. Less sensitive to outliers than Euclidean because there's no squaring.

Manhattan Distance (L₁)
d(a,b) = Σ |aᵢ − bᵢ|
Also called taxicab distance or city block distance. One outlier dimension doesn't dominate as aggressively as with Euclidean.
📐 Math Workout — Manhattan vs Euclidean with Outlier
Problem: A = (2, 3), B = (5, 3), C = (2, 100).
Compute both Euclidean and Manhattan from A to B and from A to C. Compare the ratios d(A,C)/d(A,B) for each metric.
Euclidean
d_E(A,B) = √((5−2)²+(3−3)²) = √9 = 3
d_E(A,C) = √((2−2)²+(100−3)²) = √9409 = 97
Ratio = 97/3 = 32.3×
Manhattan
d_M(A,B) = |5−2|+|3−3| = 3
d_M(A,C) = |2−2|+|100−3| = 97
Ratio = 97/3 = 32.3×
Same ratio here because the outlier is in one dimension only. The real advantage of Manhattan shows when outliers affect MULTIPLE dimensions — Euclidean's squaring compounds the effect while Manhattan stays linear.
03 — Cosine Similarity

Direction, Not Magnitude

Cosine similarity measures the angle between two vectors, completely ignoring their magnitude. Two documents with the same word proportions but different lengths have cosine similarity ≈ 1. This makes it ideal for text, embeddings, and any high-dimensional sparse data.

Cosine Similarity
cos(a,b) = (a · b) / (‖a‖ · ‖b‖)
Range: −1 (opposite) to +1 (same direction). 0 = perpendicular. Cosine distance = 1 − cosine similarity.
📐 Hard Workout — Cosine vs Euclidean Disagreement
Problem: A = (1, 100), B = (2, 200), C = (100, 1).
Compute: Euclidean d(A,B) and d(A,C). Cosine similarity cos(A,B) and cos(A,C).
Show that the two metrics completely disagree about which pair is more similar.
Euclidean
d_E(A,B) = √((2−1)²+(200−100)²) = √(1+10000) = √10001 ≈ 100.0
d_E(A,C) = √((100−1)²+(1−100)²) = √(9801+9801) = √19602 ≈ 140.0
Euclidean says: A is closer to B ✓
Cosine Similarity
cos(A,B) = (1·2 + 100·200)/(√(1+10000)·√(4+40000)) = 20002/(100.0·200.0) ≈ 0.9999
cos(A,C) = (1·100 + 100·1)/(√10001·√10001) = 200/10001 ≈ 0.020
Cosine says: A and B point same direction (≈1), A and C are nearly perpendicular (≈0)
Both metrics agree A is closer to B here. But the KEY insight: A=(1,100) and B=(2,200) have Euclidean distance 100 (seems far!) but cosine similarity 0.9999 (identical direction). Cosine ignores that B is just a scaled version of A. For tasks where magnitude doesn't matter (document similarity, user preference direction), cosine is correct.

🎯 Scenario: Recommender System Distance Choice

You're building a movie recommender. Users rate movies 1–5 stars. User A rated 10 movies, User B rated 200 movies. Missing ratings are 0. User A's vector is mostly zeros with a few ratings; B's is dense.

Think: Euclidean distance would say these users are very different because of the magnitude gap (B's vector is much longer). But if they rated the same movies similarly, they have similar taste. Cosine similarity measures the angle — if they agree on direction, similarity is high regardless of how many movies each rated. This is why recommender systems almost universally use cosine or Pearson correlation.
04 — Minkowski Distance

The Unified Family

Euclidean and Manhattan are special cases of the Minkowski distance, controlled by parameter p.

Minkowski Distance
d(a,b) = ( Σ |aᵢ − bᵢ|ᵖ )^(1/p)
p=1 → Manhattan · p=2 → Euclidean · p→∞ → Chebyshev (max absolute difference in any single dimension)

When to use Chebyshev (p→∞): When you care about the worst-case dimension. Example: in manufacturing quality control, a product fails if ANY measurement is too far from spec — not the average.

📐 Hard Workout — Curse of Dimensionality
Problem: In d-dimensional space, randomly place 1000 points uniformly in [0,1]ᵈ.
For d=2: the ratio of max distance to min distance between any pair is typically ~5–10.
For d=100: this ratio drops to ~1.1–1.3.

Explain why this makes Euclidean distance unreliable for high-dimensional clustering.
The phenomenon: Distance Concentration
In high dimensions, Euclidean distance is the sum of many squared terms. By the Central Limit Theorem, this sum concentrates around its expected value. All pairwise distances become approximately equal.
For d=100: E[d(x,y)] ≈ √(d/6) ≈ 4.08 for uniform [0,1]ᵈ
Std of pairwise distances ≈ √(7d/180)/√d → 0 as d→∞
All points become "equidistant" from each other!
Impact on Clustering
If all pairwise distances are nearly equal, K-Means centroids have no meaningful "nearest" assignment — clusters become arbitrary.
This is the Curse of Dimensionality for distance metrics. Solutions: (1) dimensionality reduction (PCA, UMAP) before clustering, (2) use cosine similarity instead (measures direction, not magnitude), (3) use domain-specific metrics.
