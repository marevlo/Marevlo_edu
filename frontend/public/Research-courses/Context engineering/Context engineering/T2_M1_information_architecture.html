A context window isn't a document — it's a page the model reads top-to-bottom, once, with uneven attention. The layout of that page shapes the answer more than most teams realize. This module is about structure: how to section the context into consistent blocks, how to chunk retrieved content without severing meaning, how to order competing signals so the right one wins, and what delimiters actually do (and don't do) inside a modern transformer.

01 / Foundations
Reading like a model
Information architecture for LLMs starts with understanding how the reader processes the page. It isn't how humans read.

M0 set up the window as a budget with uneven attention. This module is about the structure you impose inside that budget — what web designers would call information architecture, adapted for a reader that is neither human nor traditional parser. To do that adaptation well, start by taking the model's perspective seriously and noticing where it differs from your own.

How humans read a document

A human reader scans first, decides where to focus, and can re-read. They skim headings, jump to the conclusion, skip boilerplate, and come back to sections as needed. Traditional document design is built around this behavior: headings exist so humans can navigate, whitespace exists so humans can pause, summaries exist because humans can choose to read them instead of the full body.

How an LLM "reads" a context

A model doesn't scan. It produces a single distribution over the next token given every token in the context, using attention to weight which earlier tokens matter. A few consequences fall out of this that guide everything in this module:

  • No re-reading. The model doesn't go back. Every decision uses the current context once. If the right fact is hard to find, the model won't try harder — it will guess.
  • Attention is the whole navigation system. What humans accomplish with scanning, the model accomplishes with attention weights. The model can only "pay attention" to things that stand out to its attention mechanism — which is a function of position, surrounding context, and statistical distinctiveness.
  • Format cues matter more than aesthetics. Headings aren't for navigation (there's no cursor to jump). They're signals that change the probability distribution of what follows. A well-placed heading changes what the model generates; a decorative heading wastes tokens.
  • The model learned from structured text. Pretraining saw lots of well-structured documents — code, markdown, XML, JSON. The model has strong priors for how certain structures signal meaning. Using those structures invokes those priors; ignoring them leaves quality on the table.
The reframe

Design your context like you're writing instructions for a very fast reader who will never look back, processes everything at once, has been trained on billions of structured documents, and pays uneven attention based on position and novelty. That's not a metaphor — that's literally the reader.

Why "structured" beats "natural"

A common early instinct is to write prompts as flowing natural prose, the way you'd write a human instruction. It feels more polite, more nuanced, and more faithful to how the task was described in someone's head. It also underperforms structured layouts — consistently, across models, across tasks. The model is pattern-matching against its training data, and its training data is full of documentation, code, and markup. The more your context looks like well-engineered documentation, the more reliably the model behaves.

Structure isn't about legibility to you. It's about invoking the right priors in the model. The rest of this module is the concrete vocabulary for doing that.

02 / Layout
The canonical layout — six blocks, one page
One layout works for ~80% of production LLM systems. Deviate only when you have a specific reason.

Every production LLM context you'll build is some variation of the same six-block layout. Knowing the canonical order — and what each block is for — removes most of the guesswork. Deviate when you have a reason; follow it when you don't.

Why this order

Each block's position is chosen deliberately, based on what we learned about attention in M0:

  • System at the top because the model's pretraining expects stable role framing upfront and because the primacy zone is strong.
  • Tool specs early because they're stable per session and because the model needs them as vocabulary before encountering a task that might use them.
  • Task-scope content in the middle-to-late region because it's the largest block, and because by placing it after tools but before history it's close enough to the question to matter.
  • History just above instruction because recent turns are most relevant and middle-turn forgetting can be mitigated by summarizing older ones up-front or promoting facts to a memory block.
  • Current-turn instruction just before the user message because this is the single highest-attention region in the entire layout. Reserve it for format-critical or turn-specific directives.
  • User message last because that's what the model is responding to, and because attention peaks at the immediate pre-generation position.
One-block rule

Each block should have one job. If you find yourself putting instructions inside retrieved content, or mixing history with tool specs, you're losing the attention boost that clean block separation provides. When blocks blur, the model's priors for each block type get confused.

Common acceptable deviations

The canonical layout is a prior, not a law. Situations where you'll reasonably diverge:

  • Very long retrieved content. Split into two passes — retrieved content above history for background, key findings re-stated as a short summary in the instruction block near the end.
  • No conversation. Single-shot calls drop block 4 entirely.
  • Agent loops. History becomes a "scratchpad" of tool calls + observations; same position but different content shape (covered in M7).
  • No tools. Drop block 2. Don't fill it with fake content to "maintain the pattern."
03 / Chunking
Chunking discipline — the cut decides everything downstream
When retrieved content gets chopped into chunks, the cut points determine both what's retrievable and what's coherent when retrieved.

Chunking is the upstream decision that silently shapes everything downstream. A bad chunk boundary can make a document effectively invisible to retrieval (because the query embedding doesn't match the awkwardly-cut span), or it can surface content that's technically a match but missing the half-sentence that gave it meaning. The good news: there is a small set of chunking principles that cover most production cases.

The goals of a chunking strategy

  • Semantic completeness. Each chunk should be interpretable on its own. A chunk that starts mid-paragraph or mid-thought underperforms at retrieval and synthesis time.
  • Retrieval granularity. Chunks should be sized to match the question shape. Too large: embeddings are diluted and many chunks contain the answer equally. Too small: individual chunks don't carry enough context to be useful.
  • Overlap where meaning bridges. Some amount of overlap at chunk boundaries prevents facts that straddle a cut from being lost.
  • Structure-aware cuts. Respect the document's natural seams — paragraphs, sections, code blocks, table rows — before imposing arbitrary length limits.

The five-rule chunking policy

A policy that covers most production corpora. Codify these explicitly in your ingestion pipeline — don't leave them as "best practices we try to follow."

  1. Respect document structure first. Prefer cuts at H2/H3 boundaries, blockquotes, code fences, or paragraph boundaries over arbitrary length boundaries. Most documents have natural cut points; use them.
  2. Target range, not fixed size. Aim for 400–900 tokens per chunk, with structural constraints. Better a 1100-token chunk that preserves a section than two 550-token chunks that split a table.
  3. Overlap at semantic boundaries. Include one preceding and one trailing sentence (or a small % of tokens) for context. Overlap protects against ideas that bridge chunks.
  4. Never split atomic structures. Code blocks, tables, lists, and definitions should never be cut mid-structure. If a table is 2000 tokens, it stays 2000 tokens — or you split it by rows with headers repeated.
  5. Attach metadata to every chunk. Source document, section path, position, timestamp. This metadata is what lets you rerank, filter, and cite later. Chunks without metadata are forensic dead-ends.
class Chunk(BaseModel):
    id: str
    content: str
    tokens: int

    # The metadata that makes chunks useful downstream
    doc_id: str
    doc_title: str
    section_path: list[str]          # ["Pipelines", "Retry policy"]
    char_start: int
    char_end: int
    preceding_summary: Optional[str]  # short context for standalone use
    timestamp: datetime                # when doc was last updated
    tags: list[str] = []

    # Structural context — what was above/below this chunk
    prev_chunk_id: Optional[str]
    next_chunk_id: Optional[str]
Mistake to avoid

Fixed-size token chunking with zero overlap and no metadata is the default in most starter RAG tutorials. It is also the root cause of more "our RAG system doesn't work" investigations than any other single decision. If you inherit a system with this pattern, fixing chunking is usually the single highest-ROI refactor.

04 / Ordering
Ordering within sections — run the experiment
Two layouts. Identical content. Different answers. Pick an ordering and watch the expected quality score shift.

Once blocks are decided and chunks are cut, you face an ordering problem inside each block. Retrieved content has k items — in what order do you place them? For a given conversation history, do you reverse-chronological or forward-chronological? When few-shot examples and retrieved documents coexist, which comes first? These choices measurably affect output quality; the rules for making them are less obvious than the canonical layout.

The three ordering principles

  1. Most-important-last within a block. The model's attention peaks closest to the end of context. If one chunk is clearly the most relevant, place it last in its block.
  2. Context before claim. When content needs prior framing to be interpretable, framing first, claim last. A date needs a year; a code snippet needs its imports above it.
  3. Break monotony. Five similar-looking chunks in a row produce attention fatigue — the model stops distinguishing them. Interleave with brief dividers or metadata lines so each chunk has a visual fingerprint.

Conversation history — forward or reverse?

The correct default is forward chronological (oldest first, newest last). This matches the model's strong prior from training data (conversations unfold top-down) and puts the most recent turn closest to generation. Reverse-chronological history is occasionally proposed as a way to "prioritize recency" but it breaks the causal reading — the model sees replies before questions. Avoid it unless you have specific evidence it helps your workload.

Few-shot examples and retrieved docs together

When both coexist in the same block, the general rule: few-shot examples first, retrieved content second, with a visible separator between them. Few-shots are there to teach format; if they come after the content the model is likely to interpret them as more content.

Operational note

A fixed ordering policy in code is worth more than a perfect one. Ordering decisions made one query at a time by different engineers drift; a single policy applied uniformly is easier to measure and improve. Make the ordering explicit in your context-assembly pipeline (M2 builds this).

05 / Delimiters
Delimiters and signposts — what actually helps the model parse
Not all delimiters are equal. The model learned some of them from pretraining; others are decorative.

Delimiters are the punctuation of context engineering — cheap tokens that signal where one thing ends and another begins. Used well they give the model strong priors for parsing structure. Used badly they add noise and waste budget. The key distinction: some delimiters exist in the model's pretraining as meaningful structural markers, and some don't. Using the former invokes learned behavior; using the latter just uses tokens.

Delimiter comparison — what does and doesn't work
Reference
Weak: arbitrary decoration
================================ YOUR TASK ================================
Looks strong to humans; the model has no strong prior for this pattern. Consumes tokens without proportional benefit.
Okay: markdown headings
## Retrieved documents ### Document 1 Company policy on refunds...
The model has strong priors for markdown. Good for hierarchy; less crisp than XML for block separation.
Best: XML-style tags
<document source="policy.md"> Company policy on refunds... </document> <instruction> Answer using only the above. </instruction>
Strong prior from pretraining. Models treat matched open/close tags as a real scope. Works especially well for retrieved content blocks and tool specs.

The delimiter priority order

When you need to separate something inside a context, reach for these in order. Lower-priority options are still okay; higher-priority ones invoke more learned behavior:

  1. XML tags (e.g., <document>...</document>). Best for block-level separation with scope. The model's pretraining includes large amounts of XML-marked structured content; it has strong priors for matched tags as meaningful scope.
  2. Markdown headings (## Section). Great for hierarchy. Use ## for major blocks, ### for sub-blocks. Don't abuse # — it usually looks like "this is the title of the whole document" to the model.
  3. Code fences for code. Backticks + language hint. Strong prior.
  4. Numbered lists for sequences where order matters.
  5. Blank lines as soft dividers between paragraphs within a block. Fine, just weaker than the above.
  6. Decorative dividers (===, ---, emoji). Mostly ceremonial. Not harmful in small doses but not worth budget.

The signpost trio

Beyond delimiters, three kinds of signposts inside a block substantially increase model parse quality:

  • Role labels. "Source:", "Instruction:", "Constraint:". Cheap labels that tell the model what each chunk is for.
  • Provenance tags. "<document source="x.md" updated="2025-09">" — the model can cite accurately when the source is visible in context, and cite poorly when it isn't.
  • Outcome anchors. "Respond as JSON with fields: X, Y, Z" at the end, immediately before generation. This is the single most reliable way to get a structured output.
# A well-signposted retrieved-content block
<retrieved_documents>

<document source="policies/refund.md" updated="2025-09-14">
## Refund policy
Customers can request a refund within 30 days of purchase...
</document>

<document source="policies/shipping.md" updated="2025-08-01">
## Shipping policy
Orders ship within 2 business days...
</document>

</retrieved_documents>

<instruction>
Answer the user's question using only the retrieved documents.
Cite the source filename for each claim.
</instruction>
06 / Hierarchy
Hierarchy and scope — nested blocks that carry meaning
When content has natural hierarchy, expose it. The model handles nesting well; flat context dilutes structure.

Most real content has hierarchy: a knowledge base has topics with sub-topics; a codebase has packages with modules with functions; a conversation has turns with tool calls within them. When you preserve the hierarchy in how you lay out context, the model uses it. When you flatten everything to a single level, you lose that signal and force the model to infer structure it didn't need to infer.

Hierarchical layout — nested blocks with scope
Example
<system>
  role, persona, policies
</system>

<tools>
  <tool name="search">...</tool>
  <tool name="compute">...</tool>
</tools>

<knowledge>
  <topic name="billing">
    <section title="refund policy">...</section>
    <section title="subscription tiers">...</section>
  </topic>
  <topic name="shipping">
    <section title="delivery windows">...</section>
  </topic>
</knowledge>

<conversation>
  <turn role="user">...</turn>
  <turn role="assistant">...</turn>
</conversation>

<instruction>
  // turn-specific directives for this response
</instruction>

<user> current message </user>

When hierarchy helps and when it hurts

Hierarchy is an information-conveying tool, not always a formatting requirement. Use it when:

  • The nested items are genuinely related. Grouping refund and subscription sections under "billing" helps the model resolve ambiguous queries to the right scope.
  • There are multiple peers at each level. Two topics, each with multiple sections — hierarchy disambiguates. A single topic with five sections — hierarchy is overkill.
  • You want citations by path. Nested structure makes "policy.md → refund policy → timing" a natural, unambiguous reference.

Don't use hierarchy when:

  • The nesting is artificial. Wrapping every chunk in a "chunk" element inside a "chunks" element inside a "data" element adds overhead without meaning.
  • Levels are redundant with delimiters elsewhere. If chunks already have <document> tags with provenance, an extra "retrieved content container" around them is noise.
  • The content is flat. Don't invent structure that isn't there. Three independent documents are a flat list, not a tree.

Scope discipline — what belongs to whom

One subtle benefit of well-scoped hierarchical blocks: the model respects the scope of instructions and constraints. "Cite sources" inside <knowledge> is clearly about knowledge citations. "Respond in JSON" inside <instruction> is clearly about this turn's output format. Mixing these in a flat context creates ambiguity about what applies where.

Pattern

Match context hierarchy to conceptual hierarchy, not to storage hierarchy. If your knowledge base has 50 folders but only 3 map to conceptually distinct areas for the user's question, flatten the other 47 into a single peer group. Hierarchy should communicate structure the model can use, not replicate your filesystem.

07 / Anti-Patterns
Seven anti-patterns and the fixes
If you build context systems long enough, you'll ship most of these at some point. The value is catching them early.
01 · The kitchen-sink system prompt
4000 tokens of role, policies, edge cases, response format, tool usage rules, error handling, and the Ten Commandments. The model either attends to the wrong parts or fatigues.
SymptomInstruction forgetting as context grows; instructions at turn 10 are lost by turn 15
FixSystem prompt holds stable role and persona. Turn-specific instructions go in block 5 right before the user message. Format reminders repeat at the end.
02 · Flat chunk soup
20 retrieved chunks concatenated with newlines. No provenance, no scope, no ordering. Model can't distinguish which are relevant or where they came from.
SymptomPoor citation accuracy; output references "the document" without specificity
FixWrap each chunk in <document source="...">. Rerank and order. Limit to top 5-7 unless you have eval evidence more helps.
03 · Ceremonial dividers everywhere
=========, emoji banners, ASCII art headers. Looks crisp to humans, burns tokens without model-parseable meaning.
SymptomContext is 15% decoration; no measurable quality improvement over minimal version
FixReplace with XML tags or markdown headings — structures with pretraining priors. Budget for meaning, not aesthetics.
04 · Instruction buried in the middle
Critical "respond as JSON" directive appears once, in paragraph 3 of the system prompt, 10K tokens before generation.
SymptomIntermittent format failures; model ignores format for some queries
FixFormat directives also appear in the instruction block immediately before user message. Recency zone is the most reliable position for "do X in your output."
05 · Mixed-role blocks
Retrieved documents mixed with few-shot examples, no clear separator. Model conflates examples-to-learn-from with evidence-to-reason-about.
SymptomModel sometimes treats few-shot outputs as facts about the user's situation
FixSeparate blocks with distinct tags: <examples> vs <knowledge>. Role labels on every chunk.
06 · Unbounded history retention
Every prior turn verbatim, forever, with no compression or summarization. At turn 30 the context is a 70K-token log.
SymptomQuality degrades noticeably in long sessions; cost per turn climbs linearly
FixSummarize older history into a "summary so far" block. Keep the last ~5 turns verbatim. Promoted facts move to a memory block (M5).
07 · Inconsistent layout across turns
Different code paths build the context in different orders. Turn 1 has retrieved docs before history; turn 2 has them after. The model can't learn a stable reading pattern.
SymptomInconsistent behavior between similar queries; hard to reproduce bugs
FixSingle context-assembly function, used by every code path. Treat the layout as part of the system's contract, not per-call choice. Exactly what M2 builds.
Diagnostic tip

When an LLM system is behaving oddly, log the exact context that produced the bad output and inspect it visually. Most "the model is confused" bugs are actually "the context is confused" — mixed roles, inconsistent delimiters, buried instructions. The model is usually doing the reasonable thing given what it received. Fix the input, and the output often fixes itself.

08 / Mastery
Mastery check
Six questions on the layout decisions that separate production contexts from prompt-engineering experiments.
Q1. You need the model to respond in a strict JSON schema. Where should that directive be placed for maximum reliability?
Only in the system prompt at the very top
Scattered throughout the retrieved documents as examples
In the system prompt AND in the instruction block just before the user message (block 5)
After the user message, as a postscript
Format directives in the system prompt set the baseline; reinforcement in the recency zone (block 5) is what keeps them reliable as context grows. System-prompt-only placement is vulnerable to instruction forgetting. Post-user-message placement breaks causal flow and hurts more than helps.
Q2. Your RAG system chunks documents every 512 tokens with no overlap and no metadata. Retrieval quality is inconsistent. The highest-leverage fix is:
Switch to a larger embedding model
Increase top_k from 5 to 20
Respect document structure when chunking (paragraphs, sections), add 20% overlap, attach source/section metadata to every chunk
Reduce chunk size to 256 tokens
Structure-aware chunking with overlap and metadata is the biggest single lever for RAG quality. A bigger embedding model or more retrieved chunks doesn't fix chunks that are mid-sentence or missing provenance. Smaller chunks without structure awareness just makes the fragmentation worse.
Q3. You have 5 retrieved chunks of varying relevance to the user's question. Best ordering within the retrieved-content block:
Alphabetical by source
Most relevant first, least relevant last
Least-relevant first, most-relevant last — so the best chunk is closest to the user's question
Random — ordering doesn't matter inside a block
Attention peaks near the end of context. Placing the most-relevant chunk last in its block — closest to the user's question — puts it in the highest-attention region. "Most-important-first" is the human-readable default but leaves the strongest chunk in a weaker attention zone.
Q4. Which delimiter style has the strongest pretraining prior for block-level scope in modern LLMs?
Decorative dividers like ========
XML-style tags with matched open/close (e.g., <document>...</document>)
Emoji headers
ALL-CAPS SECTION NAMES
XML tags with matched open/close appear extensively in pretraining corpora and carry strong scope semantics. The model treats them as meaningful containers. Decorative dividers and emoji look structured to humans but are mostly ignored by the model. ALL-CAPS signals emphasis in some contexts but not scope.
Q5. Your chatbot has a 4000-token system prompt that covers role, policies, response format, tool usage, error handling, and examples. Users report inconsistent format compliance. Most likely cause:
The model is too small
Kitchen-sink system prompt — turn-specific directives are drowning in the system prompt. Move format guidance to the instruction block near generation
The user's question is ambiguous
The temperature is too high
Classic anti-pattern #1. System prompts that try to hold stable role plus turn-specific directives plus format guidance lose the latter two as context grows. The system prompt holds stable persona; turn-specific reminders (format, current constraints) belong in block 5, right before the user message.
Q6. Two engineers on your team build context differently — one puts retrieved docs before history, the other puts them after. You're seeing inconsistent behavior across similar queries. The structural fix is:
Retrain on the inconsistent contexts so the model learns to handle both
Have both engineers document their choices so users understand
Centralize context assembly in one function/service that all code paths must use; treat layout as part of the system contract
Use a larger model that's more robust to layout variation
Anti-pattern #7. When context assembly is scattered across code paths, layout drift is inevitable and so are unreproducible bugs. A single assembly function with a canonical layout — the subject of M2 — is the structural fix. Layout is part of your system's contract, not a per-call choice, and should be as rigorously managed as any other contract.
