01 / Foundations
What in-context learning actually does
Before choosing examples, understand what examples are doing in the model's behavior.

In-context learning (ICL) is the ability of a pretrained model to change its behavior based on examples in the context — without any weight updates. You show the model a few demonstrations of input→output pairs, and on a new input it produces outputs that match the pattern. This is the mechanism behind what people casually call "few-shot prompting," and it is one of the most powerful levers in context engineering when used well, and a source of quality regressions when used badly.

The useful mental model is that examples do two things simultaneously, and you want to be deliberate about both:

  • They specify the output format. The shape of the answer — fields, order, level of detail, tone, length — is conditioned much more strongly by a few examples than by a paragraph of instructions. "Respond as JSON" is weak; one example of the JSON is strong.
  • They specify the task itself. Examples implicitly define what the input space looks like, what counts as a good mapping from input to output, and where the edges are. The model generalizes from them. Good examples cover the patterns you want handled; bad examples teach the model the wrong pattern.

Why demonstrations beat descriptions

A pretrained LLM has seen billions of examples of task-performing structure — IDE autocompletions, Q&A threads, code reviews, email replies, spreadsheet-like input/output rows. It doesn't need to be told what an input→output mapping looks like in the abstract; it needs to be shown the specific mapping you want. A well-chosen example invokes the right pattern-matching circuit. A paragraph of prose describing the same pattern tries to build it from scratch in natural language.

This is the reason teams often find that deleting 200 tokens of instruction and adding one good example improves quality. The instruction was trying to describe what the example could just show.

The shift

Instructions tell the model about the task. Examples tell the model how the task looks when done well. The model is dramatically better at learning from the second than the first. When both exist, short instruction + good examples outperforms long instruction + no examples almost every time.

Where ICL lives in the canonical layout

From M1, few-shot examples go in block 3 (task-scope context), with a clear separator from retrieved content. From M3, they do not go in the system prompt (anti-pattern 5: mixed-role blocks). The layout looks like:

<system>
# Role + rules (block 1)
</system>

<tools> ...block 2... </tools>

<examples>
  <example>
    <input>...</input>
    <output>...</output>
  </example>
</examples>

<retrieved_documents> ...block 3 continued... </retrieved_documents>

<history> ...block 4... </history>
<instruction> ...block 5... </instruction>
<user> current message </user>

Examples before retrieved documents is the standard ordering when both exist — examples set format/pattern, then retrieved docs serve as content. Reversing this risks the model treating examples as more retrieved content.

02 / k-Shot
Zero, one, many — how many shots is enough?
The performance curve isn't linear. Understanding its shape tells you when to stop adding examples.

The default question people ask is "how many few-shot examples should I use?" The better framing: k is a dial on a quality/cost curve, and the curve has a characteristic shape. For most tasks with modern frontier models, the curve looks like this: zero-shot is a baseline, the first example produces a large jump, the second and third produce diminishing but real gains, and past five-to-eight you're usually paying tokens without meaningful quality improvement. For some tasks a single example is the ceiling; for a few, hundreds of examples help. Knowing the shape for your task is the work.

The four regimes

k = 0 (zero-shot)

No examples, only description. Works when: the task is common and well-represented in pretraining (translation, summarization, sentiment classification), the model is large enough to infer the format from description alone, or the output format is natural (prose, not structured). When zero-shot is enough, use it — examples cost tokens for no benefit.

k = 1 (one-shot)

The single highest-ROI shot in most workflows. A single well-chosen example pins format, tone, level of detail, and edge-case handling in one move. For structured outputs (JSON, specific templates), one-shot is often sufficient. The example acts as a template the model completes.

k = 2–5 (few-shot)

The sweet spot for tasks with genuine variance in input or output. Multiple examples let the model see the range — what inputs look like, what the task does across different inputs, how to handle corner cases. This is the regime most production ICL lives in.

k = 6+ (many-shot)

Genuinely useful in two cases: tasks with a long tail of input patterns (many shot retrieval surfaces per-query examples), and tasks where format or schema is complex enough that seeing many variations helps the model generalize. Otherwise, past 5–8 you're usually paying tokens for marginal quality improvements. The dramatic many-shot gains reported in some research typically reflect tasks where this tail exists; for most product tasks, it doesn't.

The default

Start with k=1. Measure. If quality meets the bar, stop. If not, add one example at a time, re-measure. Most teams land at k = 2–3 for production workloads. The "five examples is the best-practice default" you'll see in tutorials is often over-prescription for tasks where three would be enough.

When zero-shot beats few-shot

Counter-intuitively, there are cases where zero-shot outperforms few-shot. Recognize them:

  • The task is well-represented in pretraining. Summarizing news articles is something the model has seen millions of times; your examples are just noise.
  • Your examples are poorly chosen. A few bad demonstrations teach the wrong pattern. If you don't have good examples, no examples is better.
  • The input space is too diverse for k examples to be representative. If every query is very different, a few chosen examples bias the model toward their specific shape.
  • Cost or latency matters enormously. Zero-shot saves k × example_tokens per call. At scale this is real money.
03 / Selection
How to pick examples — four strategies, one default
Example selection is where most few-shot systems leave performance on the table. The strategies vary in complexity and value.

Assuming you've decided to use k examples, the next question is which k. This is a more important decision than most teams treat it as. The wrong selection strategy can make few-shot perform worse than zero-shot; the right one can make k=2 match what k=8 of worse examples would produce. Four strategies cover the space, each with specific use cases.

Four example-selection strategies
Comparison
Strategy A
Random selection
Pick k examples at random from your example pool. Simplest; usually worse than any other strategy.
✗ Default choice of research benchmarks — almost never the right choice in production
Strategy B
Fixed set (hand-picked)
Same k examples on every call, chosen once by a human. Predictable, cacheable, covers common patterns.
✓ Right default for uniform tasks — format-specification, classification with stable categories
Strategy C
Similarity-based retrieval
Per query, retrieve the k most similar examples from a labeled pool. Shows the model how similar inputs were handled.
✓ Best for high-variance inputs — custom classification, domain-specific generation
Strategy D
Diversity-optimized selection
Pick k examples that together cover the input space — different categories, edge cases, formats. Manual or algorithmic.
✓ Best when input space has known structure you want to represent

When to use which — decision tree

  1. Is the task uniform? Does every call look similar in structure (classify into same categories, generate same output format)? → Strategy B (Fixed). Hand-pick k representative examples once, cache them.
  2. Is input variance high but you have labeled examples? Many possible input shapes, labeled example bank available → Strategy C (Similarity retrieval). Embed the query, retrieve top-k similar examples dynamically.
  3. Do you want explicit coverage of edge cases? You know your task has 4 distinct patterns and want each represented → Strategy D (Diversity). Pick one example per pattern.
  4. None of the above apply, or you're not sure? → Start with Strategy B, move to Strategy C only once you have evidence that similarity-based selection improves quality on your workload.

Similarity retrieval — done right

Strategy C is powerful but frequently implemented badly. Three details that separate a working system from a broken one:

  • Embed input, not output. Retrieve examples whose inputs are similar to the new query. Embedding the outputs retrieves examples whose answers are similar, which isn't the goal.
  • Rerank for diversity. Top-5 similar examples are often near-duplicates. Apply Maximum Marginal Relevance (MMR) or similar to pull in variety — one highly similar, a few moderately similar from different sub-patterns.
  • Mind example quality. Retrieval pulls from your example bank. A bank with bad examples retrieves bad examples. Curate the bank before optimizing the retriever.
def select_examples(query: str, k: int = 3) -> list[Example]:
    # Strategy C: similarity retrieval with diversity rerank
    q_emb = embed_model.embed(query)

    # Retrieve more than k to have rerank headroom
    candidates = example_store.search(q_emb, top_k=k * 4)

    # MMR: balance similarity to query with diversity among selected
    selected = mmr_rerank(
        candidates,
        query_emb=q_emb,
        lambda_diversity=0.3,   # 0 = pure similarity, 1 = pure diversity
        k=k
    )

    return selected
The example-bank trap

The quality of similarity-based few-shot is bounded by the quality of your example bank, not your retrieval. Teams often over-invest in reranking and under-invest in example curation. Fifty carefully hand-picked examples outperform five thousand auto-generated ones for most production tasks. Curate first, retrieve second.

04 / Ordering
Example ordering — yes, it matters
Same examples, different order, different outputs. The effect is real and measurable.

One of the most-replicated findings in ICL research: with identical examples, the order you present them in affects output quality. Sometimes dramatically. The underlying reason ties back to attention (M0) — examples near the end of the examples block have more influence on the model's pattern-match than ones at the start. Order is a lever, not a footnote.

The four ordering principles

  1. Progress from simple to complex. First example is clean and clear. Later examples show edge cases and variations. The model builds a prior from the simple cases and extends with the complex ones.
  2. Most-similar-to-query last. When using similarity-based selection, the example closest to the current query should be the last example — immediately before the new input. Recency zone, highest attention.
  3. Avoid clustering same-class examples consecutively. Three positive-sentiment examples in a row followed by three negative ones teaches the model that position predicts class. Interleave.
  4. Stable ordering across calls (when using fixed examples). Fixed ordering is cacheable (relevant to M2's prompt caching); jumbled ordering busts caches. Unless you have evaluation evidence the order is hurting, don't vary it for the sake of variety.

Label distribution matters more than you'd expect

A subtle but important finding: if you have k=4 examples and 3 are labeled "positive" and 1 "negative," the model tends to slightly over-predict "positive" — even when the new query is clearly "negative." The phenomenon is sometimes called majority label bias. Two mitigations:

  • Balance label distribution in fixed example sets. If classifying into N classes, include roughly equal examples of each.
  • Don't rely on label counting. Use a classifier head, structured output with confidence, or explicit reasoning — don't just rely on the frequency of labels in examples to carry the decision.
Pragmatic default

If you're using fixed examples (Strategy B), hand-order them once with the principles above and cache that ordering. Don't shuffle per call. If you're using similarity retrieval (Strategy C), sort retrieved examples by similarity ascending — least similar first, most similar immediately before the new query.

05 / Format
The format of your examples is the format of the answer
The model will match your demonstration format tightly. Treat it as a specification, because it is.

When you show the model examples, it copies the format. Exactly. If your examples use spaces after colons, the answer has spaces after colons. If your examples are in sentence case, the answer is in sentence case. If one example uses "yes/no" and another uses "Y/N," the model's output will flip between them in a way that looks random but isn't. Inconsistency in example format is the single most common cause of inconsistent few-shot output format.

Format variables to standardize

VariableWhat to standardizeExample pitfall
Case All labels in same case (all lowercase, all UPPER, all Title) Ex 1: "positive" · Ex 2: "Negative" → output mixes case
Punctuation Trailing period or not; consistent comma handling Ex 1: "answer." · Ex 2: "answer" → output flips randomly
Delimiters Same separator between fields (colon, pipe, newline) Ex 1: "Q: ... A: ..." · Ex 2: "Question: ... Answer: ..."
Whitespace Consistent spacing; trailing newlines One example has tabs, another has 2 spaces
Label vocabulary Exact same label strings across examples Ex 1: "positive" · Ex 2: "pos" · Ex 3: "Positive"
Order of fields Same field order in every structured example Ex 1: {name, price} · Ex 2: {price, name}
Null/absent handling Consistent representation of missing values Ex 1: "none" · Ex 2: null · Ex 3: "" → output varies

The linter rule

Build a tiny linter that validates your example bank for format consistency before any new example goes in. It catches 80% of the production inconsistencies that would otherwise appear in output. The linter doesn't need to be sophisticated — just enforce the rules you wrote down:

def lint_examples(examples: list[Example]) -> list[str]:
    issues = []

    # All outputs should use same case on label
    label_cases = {ex.label_case() for ex in examples}
    if len(label_cases) > 1:
        issues.append(f"inconsistent label case: {label_cases}")

    # Outputs should either all end with period or none do
    periods = {ex.output.endswith(".") for ex in examples}
    if len(periods) > 1:
        issues.append("inconsistent trailing period")

    # Same delimiter between input and output
    # Same field order if structured
    # ... etc

    return issues

Format strength by type

Different output types have different strengths of format-from-examples:

  • Structured outputs (JSON, XML, code). Very strong format transfer. One example essentially defines the schema the model will output. Use k=1 or k=2 for most structured outputs.
  • Constrained free text (classifications, short labels). Strong transfer on vocabulary and case. Use k=2–4 to cover label variety.
  • Long-form generation (summaries, responses, explanations). Moderate transfer — the model picks up tone and structure but the output varies in length and specificity per input. Use k=2–3.
  • Creative or open-ended tasks. Weak format transfer — the model treats examples as inspiration, not template. Often k=0 (description) works as well or better.
Pragmatic tip

If you want a specific JSON schema reliably, one example plus a short schema description is more reliable than a schema description alone. Put the example in block 3, the schema reminder in block 5 (instruction), and you get belt-and-suspenders format enforcement.

06 / CoT
Chain-of-thought demos — when to show reasoning
Examples can show answers, or they can show the process for getting to an answer. The second is stronger for reasoning-heavy tasks.

For tasks that require multi-step reasoning — math, logic puzzles, complex classification with interacting rules, structured information extraction — the most powerful form of few-shot is chain-of-thought demonstrations: examples that show the intermediate reasoning steps, not just the final answer. The model learns to produce the reasoning chain on its own input, which dramatically improves accuracy on tasks where direct answer prediction is brittle.

Chain-of-thought — direct vs reasoning demos
Comparison
Direct (no CoT)
Q: If Alice has 5 apples and gives 2 to Bob, then buys 3 more, how many does she have? A: 6 Q: If Bob has 8 pencils and gives half to Charlie, then finds 2 more, how many does he have? A: 6 Q: Dana has 12 cookies, eats 3, gives a quarter of the rest to Eve, then bakes 4 more. How many does she have? A: ?
Expected accuracy on this type of task: ~72%
Chain-of-thought
Q: If Alice has 5 apples and gives 2 to Bob, then buys 3 more, how many does she have? Reasoning: Starting: 5. After giving 2: 5 - 2 = 3. After buying 3: 3 + 3 = 6. A: 6 Q: If Bob has 8 pencils and gives half to Charlie, then finds 2 more, how many does he have? Reasoning: Starting: 8. Half given away: 8 / 2 = 4. After finding 2: 4 + 2 = 6. A: 6 Q: Dana has 12 cookies, eats 3, gives a quarter of the rest to Eve, then bakes 4 more. How many does she have? Reasoning: ? A: ?
Expected accuracy on this type of task: ~94%

When CoT demos help

Chain-of-thought demonstrations are a strong lever when any of these apply:

  • The task has intermediate steps. Arithmetic, multi-step logic, rule application, structured reasoning.
  • The model's direct-answer accuracy is low but the task is solvable. If the model can get there with reasoning, showing reasoning examples unlocks that capability.
  • The correct answer is derivable from inputs. CoT shines on tasks where working through the inputs leads to the answer, weaker on tasks that require external knowledge.
  • You can evaluate the reasoning, not just the answer. CoT gives you visibility into why the model got something wrong — which is itself valuable.

When CoT demos don't help (or hurt)

  • The task has a direct mapping. Sentiment classification from a clean review. The model isn't reasoning about it in steps; it's pattern-matching. Adding CoT just adds tokens.
  • Your reasoning demos are wrong. A CoT example that shows faulty reasoning teaches faulty reasoning. Validate reasoning in examples more carefully than you'd validate final answers.
  • The model is already strong on direct prediction. Modern frontier models on well-represented tasks often don't need CoT demonstrations; they'll produce reasoning internally if asked.

Modern variants — structured reasoning

Many modern models have "thinking" or "reasoning" modes that produce extended internal reasoning before output. For these, your demonstrations should often skip explicit CoT and focus on answer format — the model is handling reasoning internally. Check your model's documentation: if it has a reasoning mode, your examples might need to be direct-answer format even for multi-step tasks.

The "let's think step by step" trap

The classic prompt-engineering trick of appending "Let's think step by step" works, but it's a weak version of what good CoT examples give you. For production systems, well-constructed CoT demos outperform the zero-shot magic phrase consistently. Don't rely on the phrase if you can instead show the process.

07 / Anti-Patterns
Seven few-shot anti-patterns
Most ICL quality problems trace to a small number of recurring mistakes. Knowing them by name catches them fast.
01 · Too many examples
k=8 or more when k=2 would have worked. Token cost per call scales linearly; marginal quality gain is flat past some point for your task.
SymptomHigh per-call token cost with no measurable quality advantage over fewer examples
FixRun an ablation: measure quality at k = 0, 1, 2, 3, 5. Pick the smallest k that hits your quality bar.
02 · Inconsistent example format
Examples use slightly different label capitalization, punctuation, or field order. Output mirrors the inconsistency.
SymptomOutput format varies unpredictably across calls with identical instructions
FixBuild a linter for the example bank. Enforce case, punctuation, field order, and label vocabulary consistency.
03 · Biased label distribution
k=4 classification examples: 3 positive, 1 negative. Model over-predicts positive even on clearly negative queries.
SymptomModel skews toward the most common label in examples
FixBalance label representation in fixed example sets. For retrieval-based selection, rerank to include examples from diverse classes.
04 · Examples in the system prompt
Few-shot demonstrations packed into the system prompt alongside rules and persona. Model conflates examples-to-learn-from with authoritative instructions.
SymptomModel sometimes treats example output content as facts about the current user
FixMove examples to block 3 with <examples>...</examples> tags. Keep system prompt for role and rules only.
05 · Bad examples in the bank
Example bank grew over time with auto-generated or hastily-added items. Some demonstrations show subtly wrong answers or wrong format.
SymptomFew-shot quality is worse than zero-shot in some portion of queries
FixQuarterly audit of the example bank. Every example should be review-approved. When in doubt, remove. Ten gold examples outperform a hundred mixed ones.
06 · Similarity retrieval without diversity rerank
Top-k embedding retrieval returns near-duplicate examples. The model sees five variations of the same pattern and misses others.
SymptomOn edge-case queries, the model produces outputs matching the wrong sub-pattern
FixApply MMR or similar diversity rerank over similarity candidates. Retrieve ~4k, return k with diversity.
07 · Never-updated example set
Examples written at launch, never revised. Product evolved; examples now demonstrate outdated formats, expired policies, or old schemas.
SymptomOutput uses deprecated field names, old tone guidelines, or reference content that no longer applies
FixTie example reviews to schema or content changes. Treat examples as configuration that has a maintenance cost, not a one-time write.
Maintenance rhythm

Treat the example bank like documentation: it rots. Every quarter, audit. For each example, ask: is this still the right output? Does the format still match current product? Would a human reviewer approve this as a good demonstration? Drop anything that fails. This is the single most-neglected maintenance task in LLM systems, and one of the highest-ROI to restore.

08 / Mastery
Mastery check
Six questions on the few-shot decisions that separate production systems from lucky prompts.
Q1. Your team uses k=10 few-shot examples for a classification task. Ablation shows k=3 achieves 98% of the quality at 30% of the token cost. What's the right move?
Keep k=10 — "more examples can't hurt"
Move to k=3 — the 2% quality gap is almost certainly not worth 3.3× the token cost for production scale
Split the difference at k=6
Add more examples; maybe k=15 will improve further
Few-shot k is a quality/cost dial. "More examples can't hurt" is wrong — they cost tokens and can introduce distractor tax. The diminishing-returns curve is exactly what you should optimize against: find the smallest k that hits your quality bar. 2% quality for 3.3× cost almost never pencils out at production scale.
Q2. You have a custom classification task with a labeled example bank of 500 items. Each user query has high input variance. Best example-selection strategy:
Random selection — simpler to maintain
Fixed 5 hand-picked examples on every call
Similarity-based retrieval with diversity rerank (MMR) — matches example selection to query shape
All 500 examples in the context
High-variance inputs with a labeled bank is exactly Strategy C's use case. Similarity retrieval pulls query-relevant examples; MMR prevents near-duplicate dominance. Fixed examples work for uniform tasks but waste the bank's diversity when inputs vary. Random wastes the selection lever entirely. All 500 blows your context budget.
Q3. Same examples, two orderings: A) easy → hard, most-similar to query last, vs B) random shuffle. Expected effect:
No meaningful difference — order doesn't matter much in modern models
Ordering A performs noticeably better — progression from simple cases and recency zone placement of most-relevant example both help
Random shuffle is actually better because it prevents the model from over-indexing on position
Ordering matters only for chain-of-thought examples
Ordering is a real lever. Progressing from simple to complex lets the model build a pattern prior. Placing the most-similar example immediately before the new query puts it in the recency zone where attention is highest. These effects are well-documented and apply across modern models — smaller for frontier models than for older ones, but still meaningful at production scale.
Q4. Your few-shot examples produce outputs with inconsistent formatting — sometimes "positive", sometimes "Positive", sometimes "pos". Most likely cause:
Temperature is too high
The examples themselves use inconsistent label format — model matches the demonstration format exactly, including its inconsistencies
Need to add a "be consistent" rule to the system prompt
Model is hallucinating labels
Models match example format with high fidelity, including inconsistencies. If examples vary between "positive" / "Positive" / "pos", outputs will vary too. Rules in the system prompt ("be consistent") are weak against concrete demonstrations. Temperature affects variation within a format, not format selection. Fix the examples, lint them, and the output consistency follows.
Q5. For a multi-step math word-problem task, direct-answer few-shot achieves 72% accuracy. Best way to raise it:
Add more direct-answer examples (k=10 instead of k=3)
Switch examples to chain-of-thought format showing intermediate reasoning steps
Use a larger embedding model for retrieval
Phrase questions as commands instead of questions
CoT demonstrations are specifically powerful for multi-step reasoning tasks. Showing the process teaches the model to produce its own reasoning chain before the answer, which on tasks like math word problems typically moves accuracy from ~70% to ~95% range. More direct-answer examples produce diminishing returns because they don't address the underlying gap (missing intermediate reasoning). Retrieval quality and phrasing are weaker levers for this failure mode.
Q6. Your few-shot system quality has degraded over the past 6 months with no code changes. No one has touched the example bank either. Most likely explanation:
The model got worse
Users are asking harder questions
The example bank is stale — examples now demonstrate outdated formats, expired policies, or deprecated schemas, while the rest of the product evolved
Context window quality has degraded over time
Anti-pattern 7 — never-updated example set. An example bank is configuration that rots as the product evolves around it. Six months of product iteration with zero example updates is almost guaranteed to introduce drift between what examples demonstrate and what the current product expects. Quarterly audits prevent this; reactive debugging rediscovers it too late. Model drift is much rarer than example drift.
