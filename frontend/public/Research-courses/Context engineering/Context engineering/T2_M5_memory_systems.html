Memory is the art
of deciding what
to forget.

Conversations grow. Sessions stretch across days. Users reveal facts, decisions get made, preferences emerge — and every one of them has to fit inside the same finite token budget. Memory is the system that decides what gets carried forward verbatim, what gets summarized, what gets promoted to a persistent state block, and what gets dropped. Done well, memory is invisible — the assistant just remembers the right things. Done badly, it's the most common source of "why did it forget?" and "why does it keep bringing up that one thing from turn 3?"

01 / Foundations
Why memory is a separate problem from history
Conversation history is one mechanism, used badly. Memory is the discipline of deciding what persists and how.

The naive approach to long-running LLM interactions: concatenate every prior turn verbatim and feed the whole log to the model on every call. This works for the first few turns. Then it stops. At turn 15 your context is 60% history. At turn 30 the middle of the history is in the attention dip (M0). At turn 60 you've run out of window entirely. At session 2, the next day, the user's previous conversation is gone as if it never happened.

Memory is the response to those failures. It's not one technique — it's a system, built from several mechanisms that each address a different failure mode. Those mechanisms are the subject of this module. The foundational mental shift:

The shift

History is a log. Memory is curated. A log is append-only and grows without bound. Memory is a small, carefully maintained set of facts and summaries that serve the current turn. Your system's quality depends on treating these as different things and engineering them separately — not on finding a "conversation memory" library that handles both as one.

The four failure modes memory addresses

  • Token bloat. Raw history grows linearly. Memory compresses — older, less-relevant turns become a summary that carries the signal at a fraction of the tokens.
  • Middle-dip forgetting. Important facts mentioned at turn 5 are in the attention dip by turn 20. Memory promotes them to a persistent state block that sits near the end of context (recency zone).
  • Cross-session continuity. A user comes back tomorrow. Raw history doesn't persist across sessions; memory does — selectively.
  • User-specific knowledge. "The user's name is Priya, they prefer brief answers, they're working on a specific project." These belong in a persistent user profile, not re-established each session.

Where memory lives in the canonical layout

Memory sits in block 3 (task-scope context) when it's per-session content, and contributes to block 1 (system prompt) when it's stable user-profile facts. The per-session summary specifically lives between retrieved content and conversation history in the layout:

<system>
# Role + rules + OPTIONAL user-profile facts if stable
You are Ario. The user is Priya, a data scientist who prefers brief answers.
</system>

<tools> ... </tools>

<session_memory>
# Summary of conversation so far — promoted facts
Decided on refund processing approach. Still open: which payment gateway.
Priya clarified budget constraint is ₹50K for this quarter.
</session_memory>

<retrieved_documents> ... </retrieved_documents>

<history>
# Last 5 turns verbatim — older turns folded into session_memory above
</history>

<instruction> ... </instruction>
<user> current message </user>

Memory blocks are structured, labeled, and short. They carry the compressed signal from turns that would otherwise be lost. They're also explicitly marked (not blended with retrieved docs or history) so the model can tell "these are facts established earlier in the conversation" vs "this is external knowledge" vs "this is the raw last exchange."

02 / Types
Four kinds of memory, four jobs
Real systems use several memory mechanisms at once. Knowing which is which is how you choose what goes where.

Cognitive science distinguishes several kinds of memory; the distinction is useful for LLM systems too. Not because we're building brains, but because each kind has a different job, a different persistence, and a different location in the context. Treating them as one blob — the pattern behind most "conversation memory" libraries — makes every one of them worse.

The four memory types
Reference
Type 1 · in-session
Working memory
The last few conversation turns, verbatim. What's actively being discussed. Short-lived, high-fidelity. This is the only memory that looks like "conversation history" in the traditional sense.
scope: current session, last ~3–8 turnswhere: block 4 (history)lifetime: until compressed
Type 2 · in-session, compressed
Episodic memory
Summary of this session up to now. What was discussed, what was decided, what's still open. Replaces older verbatim turns once they age out of working memory.
scope: current sessionwhere: block 3 (session_memory)lifetime: session duration
Type 3 · cross-session, stable
Semantic memory
Facts about the user that persist across sessions. Name, role, preferences, persistent context (what project they're working on, what tier they're on).
scope: all sessions for a userwhere: user profile → block 1 or 3lifetime: until explicitly changed
Type 4 · cross-session, behavioral
Procedural memory
Learned patterns of how this user interacts — how they phrase questions, what formats work, what level of detail they want. Closest to what a "preference model" learns over time.
scope: all sessions for a userwhere: affects retrieval, ranking, personalifetime: long; updated slowly

Why the distinction matters operationally

Each type has a different store, a different writer, a different reader:

  • Working memory is just recent turns — read from the conversation log, written by the turn itself. No separate store.
  • Episodic memory is written by a summarizer (§3). Lives in a session-scoped cache. Read on every turn; regenerated periodically.
  • Semantic memory is written by an extractor (§4) that watches conversations for persistent facts and commits them to a user-profile store. Read on every session start; updated deliberately.
  • Procedural memory is rarely a first build — it's the thing you add once the first three are solid. It tunes retrieval rankings and persona dials based on interaction history, not conversation content.
Don't build all four at once

Most production systems only need types 1, 2, and 3 — in that order of priority. Working memory is free (it's just recent turns). Episodic memory is the single biggest quality win for long sessions. Semantic memory is what gives cross-session continuity. Procedural memory is a late-stage optimization that rarely pays off until the other three are rock solid. Start simple.

03 / Summarization
Summarization — how episodic memory gets made
Old turns don't disappear; they get compressed. Done well, one paragraph can preserve what 20 turns of verbatim log carried.

Summarization is the mechanism that turns long verbatim history into compact episodic memory. The basic loop: when working-memory length exceeds a threshold, summarize the oldest N turns into a short block, append the summary to episodic memory, drop the verbatim turns. The context shrinks without losing the signal.

The quality of the summarization determines the quality of every future turn that depends on what happened earlier. A good summary captures decisions, user preferences revealed, open threads, and factual content introduced. A bad summary loses specifics, generalizes too aggressively, or — worst — invents things that didn't happen. Get this right and long sessions feel coherent; get it wrong and the assistant seems to "remember things that didn't happen."

The summarization trigger — when, not just how

Three common triggers for running a summarization pass. Pick one and be explicit:

  • Turn-count threshold. "After every 6 turns, summarize turns 1–N-6." Simplest. Predictable cost.
  • Token threshold. "When history block exceeds 8K tokens, summarize oldest 50%." More adaptive to variable turn length.
  • Semantic boundary. "When topic changes or user closes a subtask." Higher quality but requires a classifier to detect boundaries.

For most production systems a hybrid works well: token-threshold as the main trigger, with a minimum of 3–5 verbatim turns always retained for recency and a maximum gap between summarizations to bound latency.

What a good summary contains

Summaries that preserve downstream quality tend to follow a consistent structure:

  1. Decisions made. What was agreed to, what approach was chosen, what was ruled out.
  2. Facts established. Specifics the user provided: names, numbers, constraints, requirements.
  3. Open threads. What's still being worked on, what's waiting on input, what was deferred.
  4. User preferences revealed. Tone, level of detail, specific dislikes — things that should persist into the rest of the session.
# Summarization prompt template
SUMMARIZE_PROMPT = """You are summarizing a conversation for context continuity.

The summarization rhythm

A good episodic memory system runs one summarization call per N-turn window, amortized across the session. Cost is low per-turn. Latency is only incurred on the summarization turn (typically <1s on a fast model). Quality gain: long sessions go from "forgets things from 30 turns ago" to "coherent across hundreds of turns." It's one of the highest-ROI infrastructure investments for long-running assistants.

04 / Promotion
Promotion — moving facts into persistent state
Summaries compress everything uniformly. Promotion picks out specific high-value facts and puts them somewhere they won't be forgotten.

Summarization handles the bulk of history compression. But some facts shouldn't be "in a summary" — they should be elevated to a persistent state block that lives near the end of context, in the recency zone, where the model is most likely to use them. That elevation is promotion. It's the mechanism that keeps a critical fact ("user is on enterprise tier," "they care about latency above all else") alive and weighted even as the session goes on.

What to promote — and what not to

Not every fact belongs in a promoted state block. Promote selectively:

Promote

  • Stable identity facts. User name, role, account tier, organization. Rarely change, constantly useful.
  • Hard constraints. Budget limits, compliance requirements, integration constraints. Affect many downstream decisions.
  • Explicit preferences. "Always use metric units." "Prefer concise answers." Persistent behavioral steering.
  • Current project or focus. "Working on Q4 pricing model." Scopes the whole session's relevance.

Don't promote

  • Single-turn context. "User just asked about X." That's what working memory is for.
  • Transient emotional state. "User is frustrated about the bug." Relevant right now, not persistently.
  • Speculative facts. Things the user said once that might be true but weren't clearly asserted.
  • Content that fits in summary. If it's captured well enough in episodic memory, it doesn't need to be duplicated in promoted state.

Where promoted facts live

Promoted state sits in two possible places depending on persistence:

  • Session-scoped promoted state. Appended to the system prompt for the rest of this session, or rendered as a short block in block 1. "Current context: user is Priya, on Pro tier, working on Q4 pricing model. Budget constraint: ₹50K."
  • User-scoped promoted state. Saved to the user profile store. Loaded into the system prompt or as a dedicated block on every future session. Effectively becomes semantic memory (type 3 from §2).

The confidence threshold matters

Promotion is powerful and risky — a falsely promoted fact affects every future turn. An extractor that promotes on confidence >0.5 will create spurious "persistent" facts from casual or misheard utterances. Set the threshold high (0.85+), make promotion visible (log what was promoted and when), and provide a mechanism to correct. A false promotion is worse than no promotion because it propagates silently.

05 / Retrieval
Memory retrieval — strategies for getting the right memory into context
Semantic memory scales. You can't put all of it in context. So you retrieve — just like documents.

Working memory is small (last few turns), so it all goes into context. Episodic memory for the current session is small enough to always include. But semantic memory — stable facts across potentially hundreds of past sessions — can get large. At scale, you have to treat semantic memory like any other retrievable content: store it, embed it, retrieve the relevant subset per query, and inject into context. The stack is familiar from M2, adapted for memory-shaped objects.

Four memory-retrieval strategies
Reference
Strategy A
Always-load
Load all user-profile memory on every session start. Simplest; works when memory fits easily in budget.
When: memory per user < 500 tok. Works for identity + preferences only.
Strategy B
Semantic retrieval
Embed each memory item. Per query, retrieve top-k relevant memories via similarity. Same pattern as RAG.
When: hundreds of memory items per user, most irrelevant to any given query.
Strategy C
Key-based lookup
Memory stored as structured key-value pairs. Look up by query intent or entity detection.
When: memory is structured (preferences, settings, identity) rather than free-form.
Strategy D
Hybrid (recommended default)
Small always-load block for core identity. Key-based lookup for structured preferences. Semantic retrieval for episodic history.
When: most real production systems with non-trivial memory.

Retrieval invariants

Memory retrieval has specific constraints that differ from document retrieval:

  • Scope by user, always. Memory retrieval must key on user ID. Cross-user memory leak is a critical security and privacy bug, not a quality issue. This is non-negotiable and should be enforced at the retrieval layer, not just the application layer.
  • Recency matters more than for documents. Recent interactions usually more relevant than year-old ones. Rank by similarity × recency, not similarity alone.
  • Memory can contradict itself. Users change preferences; decisions get reversed. Always return the most recent version when memories conflict; flag conflicts during retrieval so downstream layers can resolve.
  • Frequent write, frequent read. Unlike document corpora (read-mostly), memory stores are updated every session. Cache invalidation must handle this — the "updated on write" pattern matters more.

A concrete retrieval pipeline

def retrieve_memory(
    user_id: str,
    query: str,
    intent: str,
) -> MemoryBundle:
    # Scope by user — always first, never optional
    scope = {"user_id": user_id}

    # 1. Always-load core identity
    identity = identity_store.get(user_id)

    # 2. Structured preferences by key
    prefs = preference_store.lookup(
        user_id=user_id,
        keys=["tone", "verbosity", "units", "tier"]
    )

    # 3. Semantic retrieval over episodic memory
    episodic = episodic_store.search(
        query_embedding=embed(query),
        filter=scope,
        top_k=5,
        rank_by="similarity_recency",   # combined score
    )

    # 4. Bundle with provenance
    return MemoryBundle(
        identity=identity,
        preferences=prefs,
        episodic=episodic,
        retrieval_time=now(),
    )
The default to start with

For a new production system, Strategy D (hybrid) is the right target — but build it in stages. Ship Strategy A first (always-load identity and preferences). Once you have volume, add Strategy B for episodic. Strategy C falls out naturally as you formalize what you're already storing. Don't start with a fully-built four-way memory retrieval system; you'll over-engineer it before knowing what the real access patterns look like.

06 / Forgetting
Forgetting, scope, and explicit control
A memory system without forgetting is a leak. Explicit forgetting is a feature, not a bug.

A memory system that never drops anything has two problems. First, it grows without bound — even with compression, the episodic memory store bloats. Second, and more consequentially, it remembers things the user didn't want remembered: casual utterances elevated into persistent preferences, old facts that are no longer true, one-off context that was never meant to carry forward. A real memory system forgets on purpose, with policies.

The three forgetting mechanisms

1 · TTL (time-to-live)

Every memory item has an expiration. Session-scoped items expire at session end. Episodic items might expire after 90 days. User-profile items might persist indefinitely but have a "last confirmed" timestamp that triggers re-verification.

2 · Supersession

New facts can replace old ones explicitly. "I'm now on the Enterprise plan" supersedes a prior "I'm on Pro." The memory system must recognize this and update rather than append. Requires entity resolution — knowing that both statements refer to the same concept (plan_tier).

3 · User-initiated forgetting

Users can explicitly ask the system to forget. "Please don't remember that I mentioned X." Production systems need a mechanism to honor this — both a user-facing affordance ("delete this from memory") and an in-conversation trigger ("don't remember this"). This is not just a UX nicety; in many jurisdictions it's a regulatory requirement.

Memory scope — what belongs where

Memory contentScopeLifetimeLoaded into
current turn context turn single call user message (block 6)
recent exchanges session last ~5 turns history (block 4)
session summary session session end session_memory (block 3)

identity facts user until changed system prompt (block 1)
preferences user until changed system prompt or block 3

Privacy, consent, and the right to be forgotten

Memory that persists across sessions carries privacy implications that turn-by-turn context does not. Three obligations apply to most production systems:

  • Transparency. Users should be able to see what the system remembers about them. "What do you know about me?" is a reasonable user question.
  • Control. Users should be able to delete specific memories or clear memory entirely.
  • Scope respect. Memory should never cross tenant boundaries or user boundaries. Enforced at retrieval, not just application.
Cross-user leak — the critical failure

If your memory retrieval code doesn't scope by user ID at the storage layer, a bug or misconfiguration in the application layer can leak user A's preferences into user B's session. This is a P0 security bug — not a quality issue. The fix is to make user-ID scoping a storage-layer invariant. Retrieval functions should refuse to run without a user scope. Defense-in-depth matters here more than for document retrieval.

07 / Anti-Patterns
Seven memory-system anti-patterns
The recurring ways memory systems fail in production. Each has a signature symptom and a specific fix.
01 · Unbounded history retention
Every turn appended verbatim, forever. No summarization, no compression, no trim. At turn 40 the context is 80K tokens of log.
SymptomCost rises linearly with turn count; quality degrades after ~20 turns
FixSummarize older turns into episodic memory. Keep last 5–8 turns verbatim. Promote critical facts to a state block.
02 · One blob called "memory"
Working, episodic, semantic, and procedural memory all lumped into one undifferentiated store. Treated as "chat history with extra steps."
SymptomCross-session continuity is broken; hard to debug what the system "knows"
FixSeparate the four types (M5 §2). Different stores, different writers, different readers, different lifetimes.
03 · Aggressive promotion without confidence threshold
Extractor promotes any mentioned preference or fact to persistent state. One-off casual utterances become permanent "user preferences."
SymptomSystem seems to over-remember things the user said in passing
FixConfidence threshold ≥ 0.85 on promotion. Require pattern match + explicit assertion. Log promotions for audit.
04 · Summarization that invents
Summarizer model paraphrases liberally, inventing plausible content that wasn't in the conversation. The invented facts then carry forward as if they were true.
SymptomUser says "I never said that" about something in the session summary
FixSummarization prompt explicitly forbids invention ("Do NOT invent facts"). Evaluate summaries against their source conversations for faithfulness.
05 · No cross-user scope enforcement
Memory retrieval doesn't enforce user_id at the storage layer. Application code is assumed to handle scope correctly.
SymptomPotential cross-user leak (often undetected until a specific incident)
FixScope by user at storage layer as an invariant. Retrieval functions refuse to run without user scope. Test for leak in QA.
06 · No forgetting mechanism
Memory items never expire. Old, stale, or user-deleted content keeps reappearing in retrieved memory.
SymptomSystem references preferences that are no longer current; users can't "clean slate"
FixTTL per memory type. Explicit supersession on entity conflict. User-initiated deletion affordance.
07 · Summaries that grow forever
Every summarization appends to the previous summary instead of replacing it. At turn 200 the "summary" is 5K tokens.
SymptomEpisodic memory ceases to be "compressed" — it's just another growing log
FixSummarization re-generates the full summary from the conversation, capped at a max length. Old summary is input to new summary but not concatenated.
The memory mindset

The useful frame: memory is engineered forgetting. The question is not "how do I remember everything?" but "what policy decides what stays, what compresses, what expires, and what the user controls?" Systems that answer this explicitly, per-type, with clear scopes and TTLs, hold up. Systems that treat memory as "chat history with more sessions" don't.

08 / Mastery
Mastery check
Six questions on the memory decisions that separate a long-running assistant from a goldfish with good retrieval.
Q1. Your chatbot's quality degrades noticeably after turn 20, even with a 200K window. You profile: context is only 50K tokens used. Most likely root cause and fix:
Out of tokens — increase max_tokens
Middle-dip forgetting — facts from turn 5 are now in the attention dip. Summarize older turns into episodic memory and promote critical facts to a state block near the end.
Model is degrading over time
Need a faster model
Classic middle-dip (covered in M0). Space isn't the issue — attention is. The fix is memory, specifically: episodic summaries for older history (reduces tokens AND concentrates signal), and promotion of specific high-value facts to a state block that sits near generation. Bigger window without this discipline doesn't help.
Q2. A user says in turn 3: "I'm thinking about maybe switching to the Enterprise plan." At turn 8 they ask a pricing question. Should "plan_tier = enterprise" be a promoted fact?
Yes, promote immediately — they mentioned Enterprise
No — "thinking about maybe" is exploratory, not an assertion. Promotion requires high-confidence explicit statements; this should live in working/episodic memory, not promoted state
Yes, but mark as "tentative"
Let the model decide at inference time
This is exactly the promotion-confidence case from §4. Aggressive promotion of speculative utterances creates false persistent facts that affect every future turn. Confidence threshold ≥0.85 would reject "thinking about maybe" as too tentative. The information still lives in episodic memory and working memory; it just doesn't get elevated to the persistent state block until asserted definitively.
Q3. Best place for "user's name is Priya, they prefer brief answers" in the canonical layout:
In every user message as a prefix
In the system prompt (block 1) or a dedicated user_context block early in context — stable identity/preference facts belong in the primacy zone
In the session summary at the end of context
In a separate API call per turn
Identity and preferences are stable, cross-session, semantic memory — they belong in the primacy zone (system prompt area) so they anchor the entire interaction. Prefixing user messages is wasteful (repeats the fact on every call) and puts the anchor in the wrong place. Putting it only in the summary loses the primacy benefit.
Q4. Your summarizer is producing plausible-sounding summaries, but users are complaining the bot "remembers" things they never said. Most likely root cause:
Summarizer model is too small
Summarization prompt doesn't explicitly forbid invention. The model paraphrases generously and hallucinates plausible details. Add "Do NOT invent facts" and evaluate summaries for faithfulness.
Summaries are too short
Need more retrieval
Anti-pattern 4 — summarization that invents. Summarizer models without explicit anti-invention instructions tend to fill in plausible details. The fix is twofold: (1) the summarization prompt must explicitly forbid invention and paraphrasing, and (2) summaries should be evaluated for faithfulness against source conversation, especially early in a new summarizer's deployment.
Q5. Your memory retrieval function takes a query and returns top-k similar memories. A bug in application code passes through the wrong user_id. What's the right defense?
Trust that application code always passes correct user_id
Add a TODO to fix later
Enforce user-ID scoping at the storage layer so retrieval refuses to run without a user scope. Defense-in-depth — application bugs can't cause cross-user leaks.
Hash the memories so leaks are obfuscated
Cross-user leak is a P0 security bug, not a quality issue. Application-layer scoping is necessary but not sufficient — storage-layer enforcement is defense-in-depth. If the retrieval function can't execute without a user scope, application bugs become failures (retrieval errors) rather than leaks (wrong user's data returned). This is the right architecture for anything that persists user-specific content.
Q6. You're building memory for a new production assistant. Resources are limited. Which memory types to build first, in order?
Procedural first — it's the most sophisticated
All four types simultaneously in a unified system
Working memory (free — just recent turns), then episodic (summarization for long sessions), then semantic (cross-session identity/prefs). Procedural is last, if ever.
Semantic first — persistent memory is the differentiator
Start-order matters. Working memory is free (just last-N turns in history block). Episodic memory via summarization is the single biggest quality win for long sessions — ship this second. Semantic memory for cross-session continuity comes third once you know what's worth persisting. Procedural memory (learned interaction patterns) is a late-stage optimization and often doesn't pay off until the other three are rock solid. Building all four up front is classic premature optimization.
