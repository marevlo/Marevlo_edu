If you can't measure it,
you're not engineering.
You're guessing.

Every lesson in this track assumes there's a feedback loop — that you can tell whether your context changes improved things or made them worse. Most teams don't have one. They ship prompt changes, users don't complain louder than yesterday, and they call it a win. This module is about the measurement infrastructure that turns guesswork into engineering: what to measure, how to build eval sets, when ablations matter, how A/B tests work for LLM systems, and the closed-loop process that compounds improvement over time.

01 / Why
Why context engineering needs its own evaluation regime
LLM evals exist. So do retrieval evals. Neither of them measures what this track has been about — the assembly layer itself.

Two kinds of evaluation are already mature in the LLM ecosystem. Model evals measure intrinsic model capability on fixed benchmarks — MMLU, GSM8K, HumanEval. Retrieval evals measure whether your vector index returns the right chunks for a query — recall@k, MRR, NDCG. Both are necessary. Neither is sufficient for what we've been building across this track.

Context engineering sits between these layers and has its own failure surface. A model can be excellent, a retrieval system can be perfectly tuned, and the overall system can still produce bad answers — because the assembled context put the key chunk in the middle-dip zone, or trimmed the user's actual constraint from history, or spec-bloated tools pushed real content out, or the agent state didn't promote a failed attempt. None of these are visible to model evals or retrieval evals. They're visible only to end-to-end evals that test the assembled context in situ.

The three-layer model

Model evals answer: "does the model know the answer if given perfect context?"
Retrieval evals answer: "can we find the right chunks for the query?"
Context evals answer: "given everything we have, did the assembled context actually enable a good answer?"
The last one is what this module teaches. It's the only one that scores the thing you actually ship.

What you're actually evaluating

In context engineering, the unit of evaluation isn't "the model" and isn't "the retrieval." It's a template version (cf. M2 §5) — a specific configuration of the assembly pipeline. When you ask "did that change help?", the subject is always a template. A template commits to:

  • A system prompt and instruction template (M3)
  • A tool subset and their specs (M7 §2)
  • Retrieval corpora, top-k, rerank settings (M2)
  • Budget allocation and trim policy (M2 §3–§4)
  • Memory scopes and promotion thresholds (M5)
  • Few-shot example bank (M4)
  • Layout order and delimiters (M1)

Every one of these is a knob. Changing any of them produces a new template version. Evaluating the change means comparing template A to template B on a fixed eval set, with enough signal to tell whether the difference is real.

The three failure patterns of teams without evals

  1. "Works on my query." Someone tries three queries, they look good, the change ships. In production, a fourth kind of query that wasn't tested regresses 40%.
  2. "We'll know when users complain." Users don't complain — they just stop using the feature. Silent churn is the nastiest regression signal because it doesn't alert anything.
  3. "Looks good in the logs." Spot-checking produced responses looks fine to the engineer who wrote the prompt. It turns out they pattern-matched the good-looking output format and didn't notice the factual errors.

All three dissolve the moment there's a quantitative eval set with real ground truth, run on every template change, with results stored over time. That's the infrastructure this module is about.

02 / Metrics
The six metrics that matter for context quality
Task-specific metrics vary. These six apply to almost every context-engineered system.

Some metrics you'll need are task-specific — a support bot might measure "resolution rate," an agent might measure "task completion." Below those, six metrics apply almost universally and should be on every dashboard from day one.

MetricMeasuresHowTarget
answer_correctness Is the answer factually right? LLM-judge against ground truth, or human rating. Required: a reference answer in the eval set. > 0.90
groundedness Is every claim supported by context? LLM-judge: decompose answer into claims, check each against provided context. Catches hallucination. > 0.95
context_recall Did needed info make it into the window? Check if the ground-truth-supporting chunks were actually present in the assembled context (not just retrievable). > 0.90
context_precision How much context was noise? Of included chunks, what fraction were actually relevant to the answer. Low value → attention dilution. > 0.60
instruction_following Did the system honor format, tone, constraints? LLM-judge or regex checks against explicit constraints in the instruction block. > 0.95
latency_p95 Is the system fast enough? End-to-end time from request to final token. Break down by pipeline stage (M2 §7). < 3s (varies)

Why the pair of recall + precision matters

Context recall and context precision tell you completely different things. Recall says did you have what you needed? Precision says how much noise came along? A system can have high recall and terrible precision — every query drags 30 chunks in, the right one is buried. Or high precision and terrible recall — the top chunks are relevant but the actually-answering chunk got filtered out. Tracking both is how you tune your retrieval + rerank + relevance-floor settings with intention.

Groundedness is the hallucination detector

Groundedness is the one metric that specifically catches the "model invented something plausible" failure mode. The LLM-judge procedure: 

def score_groundedness(answer: str, context: str) -> float:
    # 1. Decompose answer into atomic claims
    claims = llm_decompose(answer)

    # 2. For each claim, check if it's supported by context
    supported = 0
    for claim in claims:
        verdict = llm_judge(
            f"Is this claim directly supported by the context below? "
            f"Claim: {claim}\nContext: {context}\n"
            f"Answer only SUPPORTED or NOT_SUPPORTED."
        )
        if verdict == "SUPPORTED":
            supported += 1

    return supported / len(claims)

Groundedness below 0.95 is a hair-on-fire problem in RAG systems. Every ungrounded claim is a potential factual error in front of a user.

Cost as an implicit metric

Not listed above, but essential in practice: cost per request. A template that scores 2% higher on correctness but costs 4× as much might not be the right template. Every eval run should produce correctness + cost together; the Pareto frontier of (quality, cost) is what you're actually optimizing.

Don't average what you should track separately

Collapsing these metrics into a single "quality score" loses the signal. A change that raises groundedness by dropping precision is very different from a change that raises both — and the aggregate metric might look identical. Always display per-metric values and track them over time separately.

03 / Eval sets
Building an eval set that actually reflects production
A bad eval set makes you confident about changes that don't work. A good one catches regressions before users do.

The most common failure in LLM system evaluation is an eval set that doesn't reflect production. It was written once, in a week, by the people who wrote the prompts, based on their mental model of what users ask. Production users ask different things. The eval passes; the product regresses. Eventually the team distrusts the eval and flies blind.

Five properties of a useful eval set

  1. Drawn from production. Sampled from actual user queries, not invented. Mask PII, categorize by intent, but preserve the distribution.
  2. Stratified by intent. Separate buckets for each major intent (lookup, chat, agent task, etc.). Changes can help one bucket while hurting another — you need per-bucket numbers.
  3. Includes edge cases. The tricky queries, the ambiguous ones, the ones that previously regressed. These are where context-engineering choices have biggest impact.
  4. Has ground truth. Every example has a reference answer (or at minimum, a reference set of supporting chunks + constraints the answer must satisfy).
  5. Versioned. When you expand or re-annotate the eval, you create eval_set_v3 — you don't rewrite v2 in place. Old template scores stay comparable against old eval versions.

Anatomy of an eval example

class EvalExample(BaseModel):
    id: str                                    # stable across versions
    intent: str                                # "lookup", "chat", "agent_task", ...
    user_query: str
    session_context: list[Message] | None    # for multi-turn cases

    # Ground truth (at least one of these)
    reference_answer: str | None
    supporting_chunk_ids: list[str] | None   # for context_recall
    constraints: list[str] | None          # "must not mention X", "must cite Y"

    # Metadata
    source: Literal["prod_sample", "synthetic", "regression"]
    difficulty: Literal["easy", "medium", "hard", "edge"]
    tags: list[str]                         # "multi-hop", "long-context", "numeric-reasoning"

    # Provenance
    author: str
    annotated_at: datetime
    notes: str | None                     # why this was added — regressions especially

How many examples, and how to grow the set

  • Start with 50–100. Enough to get directional signal on changes. Writing 100 carefully-constructed examples is usually 1–2 weeks of focused work.
  • Grow to 300–500 as the system matures. This is the sweet spot for most production systems. Enough to stratify by 5–8 intents while still being runnable in reasonable time.
  • Add a regression example every time a bug ships. The instant a user reports something broken, that query becomes an eval example with a tag regression. You never regress that bug again.
  • Refresh the prod sample quarterly. User behavior drifts. An eval set built from last year's queries gradually stops reflecting what users do today.

Synthetic examples are fine — with care

LLMs are good at generating diverse query variations. Using them to augment an eval set is reasonable, with the following constraints: mark synthetic examples explicitly (source: "synthetic"), track their scores separately from prod-sampled scores, and never let synthetic become >50% of the set. A synthetic-dominated eval starts to score "how well does the system do on things an LLM would ask" rather than "how well does the system do on things real users ask."

Circularity warning

Don't use the same LLM as judge that you used to generate synthetic examples, and don't use the same LLM as judge that you're evaluating. Both create circularity — the model's biases appear on both sides of the equation and mask real quality differences. Use a different model (or a human-graded subset for calibration) for judging.

04 / Pyramid
The eval pyramid — fast cheap tests at the bottom, slow comprehensive ones at the top
Not every eval runs every time. A tiered strategy gives you fast feedback for small changes and comprehensive coverage for release candidates.

Running 500 eval examples with full LLM-judged metrics takes 15 minutes and costs a few dollars. Running the full suite on every commit is wasteful; skipping it until release is reckless. The answer is a pyramid — many fast cheap evals at the bottom, progressively fewer, slower, more comprehensive evals above.

Tier 4 — Canary
Live prod on small slice, real users
2–5% traffic · 24–72h
Tier 3 — End-to-end
Full eval set, all metrics, full pipeline
300–500 cases · 10–30 min
Tier 2 — Component
Assembly pipeline tested in isolation
50–100 cases · 1–3 min
Tier 1 — Unit
Deterministic: trim, layout, budget math, schema
> 100 tests · < 10s

What runs where

Tier 1 · Unit tests (every commit)

These are ordinary unit tests — no LLM calls involved. Trim policy: "given these blocks and cap X, did we drop the right ones in the right order?" Layout: "does the template render blocks in the canonical order?" Budget math: "do caps sum to the effective context?" Schema validation: "does every AssembledContext pass the Pydantic validator?" These protect you from logic regressions in the pipeline code itself.

Tier 2 · Component tests (every PR)

These test the assembly pipeline without the model call. Fix a query and a template; run the assembly; assert properties of the result. "For this query, chunks A and B should be in the assembled context." "For this long conversation, history should be summarized to under 2K tokens." "For an agent with a failed-attempt in state, the failed attempt should render before the trace." Fast, deterministic, powerful — and crucially, they don't depend on model non-determinism.

Tier 3 · End-to-end evals (nightly + pre-release)

The full eval set, running against the full pipeline with actual model calls, scored against all the metrics from §2. This is the gold standard of "did this change the output quality?" — but it's slow and costs money per run, so reserve it for merges to main and release candidates.

Tier 4 · Canary / production A/B (release gate)

The change runs on 2–5% of real traffic, with production metrics compared between the control and treatment groups over 24–72 hours. Covered fully in §6. This is where reality meets your eval set — and where the gap between them (common!) gets exposed.

Tier choice is a cost/risk tradeoff

A small change to a system prompt might only need Tier 1 + Tier 2 before shipping. A change to the trim policy probably needs Tier 3 — it affects what content reaches the model in subtle ways unit tests won't catch. A change to the core assembly pipeline or a new template family needs Tier 4 before full rollout. The pyramid isn't a rigid gate; it's a tool for matching diligence to risk.

05 / Ablations
Ablations — the only way to know why something works
A/B tests tell you whether a change helped. Ablations tell you which part of the change did the work. Teams that run ablations compound their improvements.

An ablation is an experiment where you remove or change one component of the system and see what happens to the metrics. Unlike A/B testing a full change, ablations are about decomposing a system's quality into the contributions of its parts. Done seriously, they produce the institutional knowledge that tells your team why your system works — which is the foundation for confident iteration.

What the table tells you

  • The reranker is doing the heavy lifting (−0.23 when removed). Don't touch it without serious testing.
  • The memory summary and priority-based trim are important (−0.13, −0.12). They're not "nice-to-haves."
  • The few-shot examples help some (−0.05). Modest but real.
  • The XML delimiters help a little (−0.04). Probably worth keeping for the modest gain.

Without this table, someone on your team will spend a week elaborating the system prompt, convinced it will help, and end up worse. With it, everyone on the team has the same accurate mental model of where quality comes from.

When to run ablations

  • When onboarding a template. Before you start optimizing, know which components contribute what. Otherwise you optimize the wrong things.
  • Before cost-cutting. Tempted to turn off the reranker to save money? Ablate first — if it costs 20% of quality, it's not negotiable.
  • Quarterly health check. Components that used to matter might not anymore (models changed, corpus changed). Ablations surface drift.
  • After a significant model upgrade. Component contributions change with model capability. A better base model might not need your memory summary anymore — or might need it more.
The discipline

Running ablations a few times per year is a small investment that produces outsized clarity. It's one of the highest-leverage engineering practices in LLM systems — and one of the least commonly adopted. If you do only one thing from this module, run a baseline ablation on your main template this quarter.

06 / A/B testing
A/B testing templates in production
Offline evals score against a fixed set. Production A/B tests score against the thing that actually matters — real users on real queries.

Offline evals are fast and controlled, but they always have a gap to reality. Your eval set has 400 examples; your production traffic is 40,000 a day, with long-tail queries you never anticipated. The final check is the one where real users interact with the new template and real metrics move (or don't). A/B testing is how you run that check without betting the whole product on a guess.

The standard A/B setup

Route a small fraction of traffic to the new template (treatment), the rest to the existing one (control). Collect metrics from both groups. After enough traffic, compare.

On a per-resolved-session basis, cost is actually down (higher resolution rate means fewer re-queries). Recommend continuing to full rollout pending day-7 results and latency deeper investigation.

Four common mistakes in LLM A/B tests

  • Running without guardrail metrics. You're optimizing for correctness but don't track cost, latency, or safety signals. The winning variant is 3× more expensive and times out twice as often — and you ship it anyway because the primary metric moved.
  • Stopping too early. Day 1 looks great, ship it. Turns out day 1 was weekend traffic that happened to suit the new template; weekday traffic regresses. Run for at least a full week to cover weekly cycles.
  • Sample contamination. Sessions get bucketed per-request rather than per-user. User A gets some requests on v4.2 and some on v4.3 in the same session. Conversation memory breaks across variants; your metrics reflect confusion, not quality.
  • Comparing to the wrong control. You spin up a new v4.3 and compare it to the v4.2 being replaced. But v4.2 was last updated 3 months ago, during a different traffic mix. Always snapshot control at the same time as treatment.

Bucketing — who goes to which variant

Three options, in order of preference:

  • Bucket by user ID (hashed). A given user always gets the same variant. Preserves session memory, session-level metrics meaningful.
  • Bucket by session. New sessions randomly assigned; session memory within a single assignment. OK when users rarely return.
  • Bucket by request. Each request independently assigned. Only safe for fully stateless systems with no memory, conversation, or session continuity. Rare in production LLM systems; most have at least session state.

When you can't A/B test

Some product surfaces don't support A/B testing — low-traffic features, enterprise deployments where contractual uniformity matters, or systems in early development. For these, offline evals (§4 Tier 3) are your primary tool, supplemented by careful manual inspection of prod traces. You won't get the same confidence as a real A/B, but a strong eval set gets you most of the way.

07 / Loop
The closed optimization loop — how improvement compounds
Any individual change might help or hurt. What distinguishes teams that steadily improve from teams that churn in place is the loop they run.

Most teams have steps of this loop scattered across their workflow — some error triage, occasional eval runs, the odd A/B test. The teams whose systems consistently improve have made the loop explicit, cyclical, and weekly. The loop doesn't have to be elaborate. It has to exist.

The context-engineering optimization loop
Architecture

What lives in each stage of the loop

1 · Observe

Production traces with full provenance (M2 §7) flow into a review queue. Team members — rotating weekly — spend 30–60 minutes looking at sampled traces, flagged low-rating sessions, and high-cost outliers. Goal: surface the top 3–5 failure patterns of the week.

2 · Triage

For each failure pattern, categorize by layer: retrieval (wrong chunks), assembly (right chunks, wrong layout), model (right context, bad output), tool (bad tool call), state (stale or missing state). This map tells you where to invest.

3 · Hypothesize

For each triaged pattern, propose a specific template change. "Raise relevance_floor from 0.4 to 0.5 to cut low-precision chunks." "Add a failed_attempts block to agent state." Write it down as a candidate; give it a v-number.

4 · Eval offline

Run the candidate through Tier 1–3 evals. Compare to current baseline on all metrics. If it doesn't clearly improve (or at least hold neutral with a side benefit), kill it here — cheap to discard.

5 · A/B canary

Candidate survived offline? Put it behind a flag, route 2–5% of traffic, watch guardrail metrics for 3–7 days. If it holds up, promote. If not, roll back — and add the failing trace to the eval set so the next candidate must pass it.

6 · Ship

Full rollout, template version bumped, commit adds at least one eval example to capture what this change was meant to fix. Now there's one more regression test protecting the improvement forever.

Why the loop compounds

Each cycle of this loop does three things at once: ships a specific improvement, adds a specific regression test, and refines the team's mental model of the system. Over six months, a team running this loop weekly has shipped 20+ targeted improvements, added 50+ eval examples covering real failure modes, and has a map of where quality comes from. The gap between teams doing this and teams shipping whatever seemed reasonable is enormous — and it shows up in the system-level numbers after a quarter or two.

The essential insight of this entire track

Context engineering isn't a one-shot design exercise. It's a discipline. The modules in this track taught the design principles — budget, layout, assembly, system prompt, ICL, memory, multi-modal, agent state. This module is what turns those principles into a practice that gets better over time instead of drifting. Instrument the pipeline. Measure what matters. Close the loop. Everything else compounds from there.

08 / Track Finale
Mastery check & what comes next
Six final questions, and a note on where context engineering fits in the broader arc of your work.
Q1. Your model eval scores are excellent (0.92 on your internal benchmark) and your retrieval MRR is 0.85, yet users report frequent bad answers. Most likely layer to investigate:
The model — try a larger one
Retrieval — add more corpora
The assembly layer — right chunks retrieved, but possibly in wrong position, with wrong delimiters, or trimmed incorrectly before reaching the model
The UI is misrendering correct answers
This is exactly the pattern from §1. Strong model evals and strong retrieval evals with weak user outcomes means the problem is between them — the assembly layer. Context evals (groundedness, context_recall, context_precision on the assembled context) are the missing instruments. Model swaps and corpus changes don't address layout/ordering/trim issues. UI investigation comes after you've ruled out content.
Q2. Your eval set is 80% synthetic and 20% production-sampled. Your template changes score well on it but keep regressing in prod. Root cause:
Synthetic examples aren't creative enough
Distribution mismatch — synthetic-dominated evals score "LLM-asked" queries, not real-user queries; you're optimizing for the wrong distribution
The judge model is too strict
You need more synthetic examples
From §3: synthetic examples carry the bias of the generator model and drift away from real user behavior. When they dominate the eval set, improvements optimize for that biased distribution. Fix: flip the ratio — production-sampled should be majority, synthetic is a supplement. Track the two subsets separately so you can see when the gap widens.
Q3. You're considering turning off the reranker to save 30% of pipeline cost. Right move to validate first:
Run an A/B test with the reranker off — prod truth is the final word
Run an ablation first — remove the reranker and re-run offline evals; if quality drops significantly, the cost saving isn't worth it
Just turn it off — you can reverse if things go bad
Ask the model to rerank in-context instead
§5's logic: before making a cost decision, know what quality the component contributes. An ablation answers that cheaply, offline, in minutes. If it's carrying 20 points of answer-correctness, the 30% cost saving is a massive regression in disguise. A/B testing is appropriate once the offline ablation shows the change is viable — running directly to A/B risks a real user-facing regression. In-context reranking is an orthogonal architectural change, not a cost decision.
Q4. Which eval tier should run on every commit?
Tier 3 (end-to-end with LLM calls) — so you catch everything
Tier 1 (deterministic unit tests) — fast, free, protects against pipeline-code regressions
Tier 4 (canary) — it's the ground truth
No tests on commit; run everything nightly
From §4's pyramid: unit tests run everywhere because they're cheap and catch logic regressions in assembly code (trim math, layout ordering, schema validation). Tier 3 is too slow and expensive for every commit — reserve for merges to main. Tier 4 is a release gate, not a commit gate. Skipping all pre-merge testing means bugs ship before they're caught.
Q5. In an A/B test, answer_correctness lifts 3pt but latency p95 regresses 500ms and cost/request rises 25%. The test is otherwise clean. The right call is:
Ship — quality is what matters
Don't ship — regressions are always bad
Evaluate the tradeoff against product goals — if users care more about quality than speed/cost in this surface, ship; otherwise, iterate to reduce the regressions first
Revert and try something else entirely
From §6: guardrail metrics matter. A 3pt quality lift for 25% more cost is a clear win for a high-value, low-volume surface (premium enterprise); a clear loss for a high-volume, cost-sensitive surface (free tier). There's no universal answer — the decision is product-goal-dependent, which is why you track guardrail metrics at all. The worst outcome is shipping on one metric without considering the others.
Q6. Your team has been shipping template changes for six months but overall metrics haven't moved much. You notice: no regression tests are added after ships, no ablations have been run, and A/B winners sometimes get re-tested and fail a month later. Missing practice:
More frequent releases
The closed loop from §7 — observe, triage, hypothesize, eval, A/B, ship with a regression example added. Without the last step especially, wins don't stick and patterns don't compound
Better models
Bigger eval set
This is the diagnostic pattern from §7. Individual changes ship; without regression tests they silently erode. Without ablations, nobody knows which parts of the system matter. Without consistently closing the loop, improvements don't compound — each cycle starts roughly from scratch. The fix is the discipline of the loop itself, not any single component.

Track 2 · Finale

You started this track with a context window and a budget. You ended it with a production discipline: assembly pipelines you can reason about, templates you can version, evals that make improvement compound. Along the way: layouts that respect the U-curve (M1), dynamic assembly with priority-based trimming (M2), system prompts that earn their tokens (M3), in-context learning that specifies format as much as task (M4), memory systems that separate log from knowledge (M5), multi-modal context that treats tokens as tokens (M6), agent state that keeps long-horizon loops coherent (M7), and now the measurement infrastructure that closes the loop.

The principles don't end at the window boundary. The same habits — engineer the assembly, measure what you ship, close the loop — transfer to every AI system you build on top of these foundations. The context window is the canvas; what you put in it is the craft. You're ready.
