The context is
not always prose.

Production context carries images, tables, JSON blobs, PDFs, audio transcripts, screenshots, code, spreadsheet rows. Each modality has its own tokenization cost, its own attention behavior, its own placement rules, and its own failure modes. The principles from M0 through M5 all still apply — budget, layout, retrieval, memory — but each gets a modality-specific adaptation. This module is the adapted playbook: image budgets, structured-data rendering choices, mixed-modality ordering, and when rendering a complex artifact as plain text is more reliable than sending it native.

01 / Foundations
Tokens are tokens — even when they started as pixels
Multi-modal models don't "see" images the way humans do. They tokenize them. Internalizing that one fact changes every downstream decision.

It's tempting to think of multi-modal context as fundamentally different from text context. In the application layer it looks different — you pass an image URL, a base64 blob, a file reference — but inside the model, everything converges. Images get encoded by a vision encoder into visual tokens that occupy positions in the same context window as text tokens. Audio goes through an acoustic encoder to audio tokens. Structured data you pass as JSON is just text tokens. The attention mechanism doesn't know or care which modality a given token originated from.

Two consequences fall out of this and guide everything in this module:

  • All the M0 constraints still apply to every modality. Budget is shared. Attention is still uneven — the U-curve exists for visual tokens too. The middle-dip still happens on image-heavy contexts. Effective-context measurement still requires workload-specific testing.
  • Different modalities tokenize at different rates. A single image might be 1,000–3,000 tokens depending on resolution and model. A one-minute audio clip might be a few hundred. A row of tabular data might be 20. Knowing the rates is how you budget intelligently across modalities.
The reframe

"Multi-modal" is not a capability layered on top of text — it's a different tokenization of the same underlying window. When you send an image, you are spending tokens. When you send structured data, you are spending tokens. When you send audio, you are spending tokens. The discipline from M0–M5 applies directly; only the accounting changes.

Why attention still behaves the same way

The U-curve (M0 §3) and middle-dip effects are not a property of text. They're properties of how attention scales across position in the context. They apply just as strongly to visual tokens — a detail in the middle of a long multi-image context gets less attention than one near the beginning or end. The practical implication: the M1 layout rules still apply. The most important image-based evidence goes close to the instruction, not piled at the start of a long image sequence.

This is one of the most under-appreciated facts about multi-modal systems. Teams often treat images as if they're "seen all at once" and therefore ordering doesn't matter. The attention mechanism disagrees. Position effects on visual tokens are real and measurable.

02 / Image Budget
Image context budget — resolution, count, and effective attention
Images are dense. Five screenshots can burn 10K tokens before you've said anything. Know what you're spending.

The most common production surprise with images: they cost more tokens than teams expect. A "single screenshot" at typical resolutions is often 1,000–2,000 tokens — more than many retrieved chunks. A five-image comparison is 5,000–10,000 tokens. Add a conversation with images on every turn, and at turn 10 your context is mostly pixels.

The three levers

Image cost is driven by three independent factors; each can be tuned:

  • Count. How many images are in context. This is the most direct lever — fewer images, fewer tokens. Don't include images "just in case"; the distractor tax from M0 applies to visual tokens too.
  • Resolution. Image dimensions in pixels. Most providers scale down large images to a fixed internal resolution, but within that ceiling higher-resolution inputs cost more tokens. A 2048×2048 screenshot costs substantially more than a 1024×1024 one, and often doesn't carry more information the model can use.
  • Detail setting. Many APIs expose a "low" vs "high" detail knob. Low mode tokenizes the image once at low res for ~85–170 tokens; high mode tiles it and costs multiples of that. Low detail is correct for most "does this image contain X" questions; high detail is needed when fine features matter.

The resolution decision framework

Before uploading an image, ask what task the model is doing with it:

  1. Object presence / coarse classification? → Low detail, 512–1024 resolution is plenty. "Is there a car in this photo?" doesn't need high-res.
  2. Reading text from the image (OCR-style)? → High detail, and consider whether OCR extraction to text is a better path (§6). High-res screenshots of documents are often a failure mode.
  3. Fine visual detail matters (medical, technical diagrams)? → High detail, original resolution. Accept the token cost.
  4. Before/after comparison? → Crop to the relevant region first. Don't send two full images when the difference is in a 200px box.
The "just send the screenshot" trap

A common failure mode: users paste a full-screen screenshot when the relevant content is a small region. The model then has to find the needle in the haystack — and often misses. Crop before sending. 300 tokens of the relevant region outperforms 2,000 tokens of full screen almost every time.

Image-specific attention behavior

Beyond position effects (which apply normally), visual tokens have some specific attention patterns worth knowing:

  • The model attends to salient regions. High-contrast areas, text, faces, central objects — these get more visual attention. A critical detail in a low-contrast corner may be effectively invisible.
  • Image captions are surprisingly load-bearing. A one-line caption ("Screenshot of the billing page, Oct 2025") before an image significantly improves the model's use of it. The caption primes the visual encoding.
  • Multiple small images < one composed image. If you need to show three charts, composing them into one image with labels often works better than three separate image inputs. The model sees it as one visual scene.
03 / Structured
Structured data — how you render it changes what the model does with it
A spreadsheet as JSON, as markdown table, as CSV, or as pipe-delimited rows — all "the same data," wildly different model behavior.

Once data is structured — tabular rows, JSON objects, key-value pairs, spreadsheet cells — you have a rendering choice that materially affects quality. The model doesn't literally see the structure; it sees text tokens that suggest structure via their format. Different renderings invoke different pretraining priors, and the one you pick determines how well the model reasons over the data.

Four ways to render the same table — what the model does with each
Comparison
Raw CSV
id,name,price,stock 101,Widget A,1299,42 102,Widget B,2499,0 103,Widget C,599,156
Compact but the model has weaker priors for CSV than for tables. Parsing errors common on wide tables or quoted strings.
Markdown table
| id | name | price | stock | |-----|----------|-------|-------| | 101 | Widget A | 1299 | 42 | | 102 | Widget B | 2499 | 0 | | 103 | Widget C | 599 | 156 |
Strong pretraining prior. Model reasons about columns reliably. Best default for small-to-medium tables.
JSON (array of objects)
[ {"id": 101, "name": "Widget A", "price": 1299, "stock": 42}, {"id": 102, "name": "Widget B", "price": 2499, "stock": 0} ]
Best for nested or variable-schema data. More tokens than CSV but more structured. Default for API-like content.
XML tagged
<products> <product id="101"> <name>Widget A</name> <price>1299</price> </product> </products>
Verbose. Good for deeply nested hierarchies; overkill for flat tables. Costs meaningfully more tokens.

The format-by-shape guide

Data shapeBest formatAvoidWhy
Flat tabular (<30 cols) Markdown table XML Columns align visually; model treats rows as records
Wide tabular (30+ cols) JSON array Markdown Markdown columns break; JSON survives width
Nested / variable schema JSON CSV CSV can't express nesting; JSON maps naturally
Key-value pairs Markdown list or JSON XML (overkill) Simple K:V doesn't need tag overhead
Hierarchical content XML or indented markdown Flat list Scope and nesting encoded explicitly
Time series / many rows Compressed CSV JSON (token bloat) Repeated keys in JSON dominate; CSV amortizes

Token efficiency by format

For the same data, the token cost varies by format — sometimes dramatically:

  • Raw CSV: Most compact. Keys listed once as header row.
  • Markdown table: ~1.3–1.5× CSV due to padding and separators. Worth it for the prior invocation.
  • JSON array of objects: ~2–3× CSV. Keys repeat per row. Best when nested or variable schema.
  • XML: ~3–5× CSV. Tags add substantial overhead. Use when the structure itself is the point.
Pragmatic default

For most production structured data: markdown table if it fits cleanly; JSON array of objects if it doesn't. Don't use XML for tabular data unless the hierarchy is meaningful. Don't use raw CSV for anything the model needs to reason carefully about — the token savings aren't worth the parsing fragility. When in doubt, render to markdown table and measure.

Precision and number formatting

A subtle but high-impact detail: how you format numbers in structured data affects the reliability of arithmetic the model does over them. Three patterns that improve math-heavy output:

  • Consistent decimal places. "1299.00" / "2499.00" / "599.00" beats "1299" / "2499" / "599.00" — the model treats trailing zeros as meaningful.
  • Aligned units. All prices in the same currency, all distances in the same unit. Mixing "5 km" and "3 miles" in the same column produces errors.
  • Use clear separators, not conventions. "1,299" is unambiguous in en-US; "1.299" reads as 1.299 in en-US and as 1299 in de-DE. When rendering for the model, prefer plain "1299" with no separators, or explicit units like "INR 1299".
04 / Layout
Mixed-modality layout — where images, tables, and text go
The canonical layout from M1 extends cleanly when you know where each modality belongs. The extensions aren't obvious but they are principled.

Most production contexts mix modalities: a screenshot, a data table, some retrieved text, and a user question. Where each piece goes inside the canonical layout affects whether the model can connect them. The key principle: keep each modality adjacent to the text that describes it, and keep the most important modality closest to the instruction.

Placement rules for multi-modal content

1 · Caption before image

Every image should have a one-line caption immediately above it. This primes the vision encoder and tells the model what to look for. "Screenshot of the billing page, Oct 2025" or "Diagram showing the data pipeline" — concise, specific. This is one of the cheapest and most effective improvements in multi-modal quality.

2 · Query-critical image last

If the user's question is about a specific image, put that image immediately before the instruction block. Recency-zone placement works for visual tokens too. If you have one reference image and three supporting images, the reference goes last.

3 · Group modalities by topic, not by modality

Instead of "all images, then all tables, then all text," organize by the subject each describes. The screenshot of a dashboard, the table of numbers from that dashboard, and the policy text about that dashboard belong together — even though they're three modalities. The model connects them more reliably when they're adjacent than when they're separated by other content.

4 · History holds descriptions, not the images themselves

When a conversation has past turns that included images, keep the images only in working memory (last 2-3 turns). Older turns should retain a text description of the image ("user shared a screenshot showing error X at timestamp Y") rather than the image itself. Images don't compress to summaries well; their text descriptions do.

The adjacency principle

Multi-modal reasoning works best when the model can attend across related tokens without wading through unrelated content. Keep related modalities adjacent. Put the caption, the image, the table derived from the image, and the question about all three close together, not scattered across blocks. Adjacency creates semantic cohesion the model can use.

05 / Audio & Docs
Audio and documents — the pipeline-heavy modalities
Less commonly native-supported, more commonly processed through a pipeline. The pipeline choice is the engineering decision.

Audio and long-form documents (PDFs, presentations, Word files) are the two modalities where production systems most often make a pipeline choice — native model support exists but is frequently out-performed by "process the content first, send the result as text or image." Knowing the tradeoffs is how you avoid the expensive path when it isn't needed.

Audio — native vs transcription

Two paths for audio input, each with clear use cases:

  • Native audio. Pass the audio file directly to a model with audio support. The model gets tone, pauses, emphasis, non-speech sounds. Costs more tokens per minute than transcription; supports fewer models.
  • Transcribe first, send text. Use a dedicated ASR (automatic speech recognition) model to get a transcript, send the text. Cheaper, broader model support, but loses everything non-lexical.

Use native audio when: tone or emotion matters (support calls, interviews), non-speech sounds matter (background audio analysis), or the content is short and low-volume. Use transcription when: content is long, you're processing at scale, or downstream use only needs the words.

Documents — the extraction pipeline

PDFs and complex documents rarely belong in context as raw bytes. A production pipeline typically does one of three things, chosen per-document-type:

  1. Text extraction only (for text-heavy docs). Extract to markdown, chunk per M1, retrieve per M2. Works for contracts, articles, reports. Fast, cheap, loses diagrams.
  2. Text + page images (for mixed docs). Extract text for searchable content, rasterize specific pages as images when the user needs to see layout, diagrams, or handwritten marks. Dual-track pipeline.
  3. Pure vision (for layout-heavy docs). Rasterize pages as images, send to a vision model, skip text extraction. Works for technical drawings, forms, scanned documents. Expensive but preserves everything.

Extraction quality matters more than the model

For text extraction pipelines, the extractor's quality is often the limiting factor — not the LLM. A bad PDF-to-text extractor produces mis-ordered columns, merged words, stripped formatting, and the LLM inherits those flaws. Invest in the extractor before blaming the model. Three rules:

  • Test extraction on your corpus. Pick 10 representative documents. Visually inspect the extracted text. If it's wrong, fix the extractor before anything else.
  • Preserve structure in extraction. Headings should extract as markdown headings. Lists as markdown lists. Tables as markdown tables. A flat stream of words loses everything M1 relies on.
  • Keep page numbers and section paths. Every extracted chunk should carry provenance back to its source page. Citation quality depends on this.
The "just send the PDF" trap

Some modern APIs accept PDFs as direct inputs. This is convenient but rarely the right production choice for anything beyond one-off analysis. You lose chunking control, lose retrieval, pay high token cost per call, and get model behavior that's inconsistent across document types. For production systems, always run extraction through a deliberate pipeline. Native PDF support is for prototyping; pipeline extraction is for scale.

06 / Text-First
When to render non-text as text — the modality demotion decision
Sometimes the most reliable path for a chart, an image, or a document is to describe it as text and skip the native modality entirely.

A counter-intuitive decision that often improves quality: don't send the native modality at all. For many use cases, a well-crafted text description of the content outperforms passing the image or audio directly. The text route is cheaper, more debuggable, and cacheable. The native route is only worth the extra cost when the model genuinely needs the raw signal.

The six cases where text beats native

  1. Charts and graphs. If you already have the underlying data, send the data as a markdown table or JSON instead of a chart image. The model reasons better over structured data than over pixels of a chart.
  2. Tabular screenshots. A screenshot of a spreadsheet is a loss-compression: the data exists in structured form, you're shipping pixels of it. Extract and send as markdown.
  3. Code in an image. Always worse than the code as text. OCR from code screenshots has subtle errors; typed code is perfect.
  4. Long audio with lexical content. A transcript is typically better than native audio unless tone/emotion matters. Transcribe, review, send text.
  5. Simple diagrams with labels. A block diagram with 5 labeled boxes and arrows is often better as structured text ("Auth service → API gateway → Database; API gateway also → Cache") than as an image.
  6. Documents where only the words matter. Legal contracts, articles, reports — extract text, skip native PDF input.

The four cases where native is the right choice

  1. Visual layout is the content. "Is this UI well-designed?" "Where is the button that does X?" Position matters; pixels are the answer.
  2. The model needs to see something you can't easily describe. Medical images, technical drawings with fine detail, handwriting.
  3. Compare-and-contrast visual tasks. Before/after, authentic vs fake, spot-the-difference. The visual encoding is load-bearing.
  4. The data doesn't exist in text form. Photograph of a physical object, satellite imagery, camera feed. There's no text version to fall back to.
The question to ask

Before including a non-text modality, ask: can the model accomplish the task if I describe the content in text? If yes, describe it — you'll usually get better quality at lower cost. If no (the pixels or audio or layout are genuinely the signal), send native. This one question prevents most multi-modal token bloat in production systems.

The captioning-first pipeline

A useful production pattern for image-heavy workloads: pass every image through a small vision model first to produce a structured caption, then decide per-caption whether to include the image natively or just the caption text. For most queries, the caption is enough; for a few, the native image is needed.

def prepare_image_for_context(image: Image, query: str) -> ContextEntry:
    # Small vision model produces a detailed caption
    caption = vision_model.caption(image, detail="high")

    # Does the query need the raw pixels?
    needs_native = query_needs_visual_detail(query, caption)

    if needs_native:
        # Both: caption primes the model, image for fine detail
        return ContextEntry(
            caption=caption,
            image=image,
            cost_tokens=caption_tokens + image_tokens
        )
    else:
        # Just the caption — cheaper, usually as good
        return ContextEntry(
            caption=caption,
            image=None,
            cost_tokens=caption_tokens
        )
07 / Anti-Patterns
Seven multi-modal anti-patterns
The specific ways multi-modal context goes wrong in production. Each has a signature.
01 · Full-screen screenshots for small details
User question is about one button; uploaded image is a full 4K desktop screenshot. Model misses the detail because it's 2% of the pixels.
Symptom"I don't see that in the image" when the thing is clearly there, just small
FixCrop before sending. Or add a caption pointing to the region. Or switch to high detail at the cost of tokens.
02 · Image spam in history
Every past turn's images retained verbatim in conversation history. At turn 10 the context is 15K tokens of stale images no one's asking about anymore.
SymptomRapid context growth; old images in middle-dip attention zone
FixKeep images only in last 2-3 turns of history. Replace older ones with text descriptions ("user shared screenshot of X at turn 3").
03 · XML for flat tabular data
Simple product list rendered as deeply nested XML with tags for every field, tripling the token count over markdown.
SymptomStructured data blocks 3–5× larger than necessary
FixMarkdown table for flat data; JSON for nested or variable schema; XML only for deeply hierarchical content where scope matters.
04 · No caption on images
Images passed in context with no preceding text. Model has to infer what the image is for without priming.
SymptomModel describes the image correctly but misses its connection to the question
FixOne-line caption immediately before every image. "Caption: [what it is, when, why relevant]" primes the encoder.
05 · Sending PDFs raw to the model
Native PDF support used for 500-page documents, paying full token cost on every call, no retrieval possible.
SymptomExorbitant per-call cost; no way to cite specific pages; rework on every query
FixExtraction pipeline: text-heavy → extract + chunk + retrieve; layout-heavy → rasterize relevant pages only; mixed → both.
06 · Native image when text would work
Chart image sent to the model when the underlying data is available. Model reasons less reliably about pixels than about tables.
SymptomNumeric answers from charts are often slightly off or hedged
FixSend the underlying data as markdown table or JSON. Reserve image for cases where visual layout itself is the signal.
07 · Modality scatter
Images grouped together, tables grouped together, text grouped together — organized by modality instead of by topic. Related content separated.
SymptomModel fails to connect an image to the table derived from it or the text describing it
FixGroup by topic. The screenshot, its table, and its explanation belong adjacent — even though they're three modalities.
The multi-modal discipline

Most multi-modal quality problems are actually budget, placement, or modality-choice problems in disguise. The model is usually doing the reasonable thing given what it received. Before blaming the vision encoder or the audio model, check: is the modality necessary? Is the caption there? Is the placement correct? Is the format the right one for the data shape? Fix those first, and most multi-modal issues resolve.

08 / Mastery
Mastery check
Six questions on the multi-modal decisions that separate careful production systems from "we just send the image and hope."
Q1. Your RAG system gets a user question about a specific button in a complex UI. You have a 4K full-page screenshot available. Best approach:
Send the full screenshot at high detail — more pixels means more information
Send the full screenshot at low detail — save tokens
Crop to the relevant region around the button, add a caption identifying it, send at standard detail
Describe the screenshot in text and skip the image
Anti-pattern 1 — full-screen screenshots for small details. The relevant region is 2% of the pixels; the model has to find a needle in the haystack. Cropping preserves detail at a fraction of the token cost. The caption primes the vision encoder on what to look for. Text-only description loses the pixel-level evidence the user is asking about.
Q2. Which format is best for a 200-row flat tabular dataset with 8 columns of mixed types, where the model needs to reason over the values?
XML with one tag per cell
Markdown table — strong pretraining prior for row/column reasoning, good token efficiency for flat tabular
Send as a screenshot of the spreadsheet
Raw CSV with no headers
Markdown table is the right default for flat tabular data under ~30 columns. XML is massive overkill and costs 3-5× tokens. Screenshot-of-spreadsheet is anti-pattern 6 — sends pixels where structured text would work much better. Headerless CSV loses column semantics. Markdown hits the sweet spot of pretraining prior strength and token efficiency.
Q3. A user is having a 15-turn conversation, sharing a screenshot per turn. What should the multi-modal layout do with older screenshots?
Retain all images verbatim — context is shared history
Keep images only in last 2-3 turns; replace older images with text descriptions in the summary
Drop all images except the first one for anchoring
Re-send all images on every turn to ensure freshness
Anti-pattern 2 — image spam in history. Retaining 15 turns of images means 15K+ tokens of stale pixels in the middle-dip zone. The fix is the same pattern as text history compression from M5: keep recent turns verbatim, fold older content into summary — but for images, "summary" means replacing the image with a text description of what it showed. Drop-all and re-send are both worse than the compression approach.
Q4. You have a user question that requires reading numbers from a chart. You have the underlying data available. Best approach:
Send the chart image at high detail — the chart is visual content
Send the underlying data as a markdown table; skip the chart image — structured data beats pixels for numeric reasoning
Send both the chart and a description
Convert the chart to ASCII art
Anti-pattern 6 — native image when text would work. When the underlying data exists, sending the data as structured text outperforms sending the chart image for numeric reasoning. Models reason more reliably over markdown tables than over pixels of charts. The native chart wastes tokens AND produces less accurate numbers. Both-at-once is worse than just the data for the same reason. ASCII art of a chart is neither better pixels nor better data.
Q5. Your team ships a contract-analysis system. Users upload 200-page PDFs and ask questions. Current approach: pass PDF directly to the model's native PDF support. Quality is inconsistent and per-call cost is very high. Best architectural fix:
Switch to a larger model with better PDF support
Compress the PDFs before sending
Build an extraction pipeline: extract text per page with page numbers, chunk per M1, retrieve relevant chunks per query, cite by page
Send only the first 50 pages
Anti-pattern 5 — sending PDFs raw for production use. Native PDF on every call is expensive, loses chunking control, makes citation imprecise, and wastes 95% of content on any specific query. A proper extraction pipeline with retrieval is the production pattern: extract once, chunk, index, retrieve per query. This is just M1-M2 applied to documents. Truncation to first 50 pages loses content arbitrarily; bigger model doesn't address the architectural issue.
Q6. A multi-modal context has: a screenshot of a dashboard, a table of metrics from that dashboard, a policy document about reading dashboards, and a user question. Best arrangement:
All images first, then all tables, then all text, then the question
Group by topic: [caption + screenshot + table + policy snippet about that dashboard] → question → put the most query-critical piece closest to the question
Alphabetical by content type
The order doesn't matter — the model processes the context as a whole
Anti-pattern 7 — modality scatter. Organizing by modality (all images, all tables, all text) separates related content and makes it harder for the model to connect them. Topic-grouping keeps the screenshot, the derived table, and the relevant policy snippet adjacent so the model can attend across them. And as with M1: the most query-critical content goes last, closest to the question. Order absolutely matters — attention is position-dependent for multi-modal tokens just as for text (M0).
