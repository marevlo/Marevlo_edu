The system prompt
is the constitution.

Block 1 of the canonical layout is where the model learns what it is, what it values, and what rules govern its behavior. It's the most re-read block of any session — included on every single call — and also the most commonly misused. This module is about what the system prompt should and shouldn't carry, how to write each part, and why the most common advice ("write a longer system prompt") is usually wrong.

01 / Purpose
What the system prompt is actually for
Block 1's job is narrow and specific. Getting that job right means carrying less, not more.

From M1 we know that the system prompt sits at the top of the canonical layout — the primacy zone, where the model's attention is strong and where pretraining has conditioned it to expect stable framing. From M0 we know that every token there costs budget on every single request. Those two facts together are the reason system prompts should be tight. Whatever goes there is re-read thousands of times per day at scale, competes for attention with everything added later, and bloats the effective context with duplicated content.

So the first question is not "how do I write a good system prompt?" It's what belongs in a system prompt at all? The answer is a short list of things that are genuinely stable across the life of a session and that benefit from being at the top of context:

  • Role and identity. What the model is supposed to be. "You are a customer support agent for Marevlo." One or two sentences. This primes the entire downstream behavior and is expensive to repeat elsewhere.
  • Persona and voice. How the model speaks. Warmth, formality, technicality. Short — the model has strong priors for common personas and doesn't need much to nail them.
  • Hard rules and policies. Things the model must never do. Privacy rules, content rules, scope limits. These stay in the primacy zone because we want them weighted heavily.
  • Tool-use philosophy. When to reach for tools, when to answer directly. The specific tool specs live in block 2 per M1; the meta-rule for using them lives here.
  • Refusal and escalation guidance. What to say when you can't help. What counts as "can't help." How to hand off to a human.
Test

For every line currently in your system prompt, ask: does this need to be here on every turn, for every request? If the answer is "no, only for some" — it belongs in a template-specific instruction block, not the system prompt. If the answer is "yes, but it's a turn-specific directive like response format" — it belongs in the instruction block at block 5 (recency zone). The system prompt's job is the always-true stuff.

What does not belong in the system prompt

A surprising amount, frankly. Moving these elsewhere is the single biggest quality win most teams find:

  • Response format specifications. "Respond in JSON with fields X, Y, Z." Put this in the instruction block near generation (block 5). Format directives decay as context grows; they need the recency boost.
  • Tool schemas. These go in block 2, not baked into prose in block 1. The model's pretraining expects structured tool definitions; don't dilute them in natural language.
  • Few-shot examples. These belong in block 3 (task-scope context) with a clear separator. Examples in the system prompt create mixed-role confusion (anti-pattern 5 from M1).
  • Long explanations, backstory, or vision statements. The model doesn't need lore. It needs instructions. Every paragraph of "we are a company that believes in..." is paid per request, forever.
  • Step-by-step chain-of-thought templates. Task-specific reasoning structure belongs near the task (block 5), not at the top. Otherwise it applies everywhere, including where it shouldn't.

The three-paragraph heuristic

A reliable default for production systems: a good system prompt fits in roughly three paragraphs. Role + persona (paragraph 1). Core rules (paragraph 2). Tool-use philosophy and escalation (paragraph 3). When a system prompt grows past that, it's almost always because it absorbed content that belongs elsewhere. The solution is not better writing — it's relocation.

Counter-intuition

The common instinct after a bad response is "I need to be more explicit in the system prompt." The right instinct is often the opposite: the system prompt may already be too crowded, and the specific thing you're trying to fix should be injected closer to generation where it will actually be weighted. Adding to the system prompt pushes the existing content deeper into diluted attention.

02 / Anatomy
Anatomy of a production system prompt
Five parts, specific jobs, specific placement inside the block. A reference for what goes where.

Every good production system prompt has roughly the same five-part skeleton — in roughly the same order. This is the information architecture of the block itself, nested inside the information architecture of the whole context. Following it gives you predictable behavior; deviating without reason gives you inconsistent behavior across requests.

The five-part system prompt anatomy
Reference
Part 1 · First sentence
Role declaration
"You are [role]." One sentence, concrete, identifies the function. This is the single most important line in your context — it conditions everything downstream.
Part 2 · Next 1–3 sentences
Operating context
Who the user is, what they're typically trying to accomplish, what product/platform this is. Short. Specific. Not marketing.
Part 3 · Short list
Core behavior rules
The small set of "always / never" rules that must hold across every interaction. Phrased positively where possible. 3–7 items is the reliable range.
Part 4 · Short paragraph
Tool-use and knowledge bounds
When to use tools vs answer from context. What to do about things you don't know. "Ground claims in retrieved content; if nothing retrieved supports a claim, say so."
Part 5 · Close
Refusal and escalation
What happens at the edges. What to refuse, how to refuse, when to hand off. Keep it brief — long refusal policies become training data for the model to over-apply them.

A concrete, small example

A customer-support system prompt that covers all five parts and still fits in ~220 tokens. Anything more than this is where teams should ask "does this belong here?" before adding.

# Role
You are Ario, the customer support assistant for Marevlo, an online
learning platform for AI and data science courses.

# Operating context
Users contacting you are learners enrolled in paid courses. They typically
ask about access, billing, course content, and technical setup. They
expect direct, competent answers and are technically literate.

# Core behavior
- Ground every factual claim in retrieved documentation. If nothing in
  context supports a claim, say so rather than guess.
- Be direct and concise. Match the user's register — technical with
  technical users, warm with frustrated users, brief with busy users.
- Cite the source filename for policy-based answers so users can verify.

# Tools and knowledge
Use the lookup_docs tool when a question needs current policy or pricing.
Use resolve_account when a user asks about their own order. Do not invent
order details, prices, or policies; if you can't retrieve them, say so.

# Refusal and escalation
If a request requires making changes to a paid subscription or issuing a
refund, respond with the next steps and hand off using the escalate tool.
Do not promise outcomes you cannot control.

Why this structure is load-bearing

Three properties fall out of following the anatomy that are hard to recover from deviating:

  • First-sentence identity. The role declaration at the very start is in the highest-attention position of the highest-attention block. A weak or generic opening ("I am a helpful AI assistant") wastes that position. A specific opening ("You are Ario, the customer support assistant for Marevlo") does real work.
  • Rules as a short list. A short, flat list of 3–7 behavior rules is far more likely to be followed than a long prose paragraph containing the same content. Lists invoke instruction-following priors from pretraining.
  • Refusal at the end. Refusal guidance at the end of the system prompt works because it's the last thing before the model transitions into reading everything else. Putting it earlier buries it; putting it later (outside the system prompt) is fine but loses the stable-anchoring benefit.
03 / Persona
Persona calibration — dialing the voice
Persona is powerful and cheap. Two sentences can shift output tone significantly — if the axes you pick are the ones that matter.

Persona in a production system isn't character creation. It's the set of register and style choices that make the model's output fit the product. The good news: models have strong priors for common registers, so you don't need many tokens. The trap: the defaults carry their own assumptions, and if you don't specify, you get the model's default voice — which is often generic, hedging, and over-apologetic. Specifying persona explicitly is how you stop that.

The axes that actually matter

Most persona specifications can be reduced to four axes. Moving one has an observable effect on output; moving all four nails the voice.

Specifying each axis

You don't set the axes by typing "formality: 2 / 4". You set them by picking words and examples that invoke the target register. Here's the phrasing that works:

  • Formality. "Use a conversational register" (low) / "Respond with professional precision" (high). Avoid "be formal" — models interpret that as stiff; say what you actually want.
  • Warmth. "Acknowledge frustration before solving" (high) / "Focus on the answer; skip pleasantries" (low). Warmth is about emotional acknowledgement, not just friendliness.
  • Technicality. "Assume technical literacy; use domain terms without defining them" (high) / "Explain concepts as if to a smart non-expert" (low). Be specific about the audience.
  • Brevity. "Prefer short direct answers; expand only if asked" (high) / "Be thorough; cover edge cases proactively" (low). Brevity controls how much the model hedges and pre-empts.
Anti-hedge override

Most consumer-grade model defaults lean heavy on apologetic hedging ("I'm sorry, but..."), unnecessary preamble ("Great question!"), and summary-of-the-question restatement. One high-ROI line: "Skip apologies, preamble, and restating the question. Answer directly." That single sentence can reclaim 20-30% of output tokens across a session and noticeably improves perceived quality.

Persona consistency across turns

A persona set in the system prompt holds across the whole session, but it can drift in two common ways:

  • Drift toward the user. If the user is chatty and informal, the model will tend to mirror that, gradually abandoning the persona. Counter this with occasional persona reinforcement in the instruction block (covered in §6).
  • Drift toward pretraining defaults. In long sessions, specific persona framing gets diluted by intervening content. Strong personas — specific named identity, distinctive voice choices — hold longer than generic ones.
04 / Rules
Rules come in three tiers — write them that way
Not all rules are equal. Treating them as such is how you end up with rules the model ignores.

When you write "always do X" or "never do Y", you're making a rule. Models interpret rules with varying strictness depending on how they're framed, where they appear, and how they relate to other rules. A common failure mode is treating every rule as equally important — which makes the model triage for you, often wrongly. The fix is to pre-triage: write rules in explicit tiers so the model knows which to weigh most.

Three rule tiers — pattern them differently
Reference
Tier 1 · MUST / MUST NOT
Hard constraints. Violating them is a bug. Use short, declarative, imperative phrasing. Keep the list tight — 3–5 items max. Rules here should be the ones you genuinely mean.
"Never share user data with third parties." "Always cite the source filename for policy claims."
Tier 2 · SHOULD
Strong defaults. The model is expected to follow them unless a specific situation argues otherwise. These guide behavior without being rigid.
"Prefer direct answers over hedged ones." "Match the user's technical register."
Tier 3 · PREFER / AVOID
Stylistic nudges. Directional preferences without hard enforcement. Good for voice and aesthetic; weak for safety-critical behavior.
"Avoid bureaucratic language." "Prefer concrete examples over abstract descriptions."

The rule-strength phrasing guide

The verb you pick maps to how strictly the model enforces the rule. Use this deliberately:

  • Must / must not / never / always. Strongest. The model treats these as non-negotiable. Save for things that are actually non-negotiable.
  • Should / should not. Strong default; model looks for reasons to follow, not reasons to break. Good for most behavior rules.
  • Prefer / avoid / tend to. Directional. The model will weight these but will let them yield to conflicting signals.
  • Consider / might want to. Weakest. The model will often ignore these in favor of its priors. Use only for genuinely optional nudges.

When rules conflict — precedence matters

Two MUST rules that contradict each other produce unpredictable behavior. Avoid this by making precedence explicit when conflicts are possible:

# Rule precedence (highest first)
1. User safety overrides everything else.
2. Data privacy rules override helpfulness.
3. Accurate citation overrides response length constraints.
4. All other rules apply by strength (MUST > SHOULD > PREFER).

Even a short precedence list dramatically reduces ambiguous behavior in edge cases. The model can't always tell which rule dominates without this — and when it can't tell, it picks a random-seeming one.

The under-used tier

Most production system prompts over-use MUST and under-use PREFER. The result: the model treats everything as rigid, and its natural judgment is suppressed on things that would have been better left flexible. If a rule's violation wouldn't actually count as a bug in a post-mortem, it's probably a SHOULD or PREFER, not a MUST.

05 / Framing
Positive vs negative framing — tell the model what to do
Negation is weak. Positive directives are strong. This is the single most common prompt-writing mistake and the easiest to fix.

There is a measurable, consistent finding across LLMs: rules phrased positively ("do X") are followed more reliably than rules phrased negatively ("don't do Y"). The underlying reason is that generative models are next-token predictors; they produce behavior, not withhold it. When you tell a model "don't mention X," you've still raised the probability of mentioning X by including it in the context. When you tell a model "focus on Y," you've raised the probability of Y without raising X.

Negative vs positive framing — side-by-side
Comparison
Negative framing
Don't be verbose. Don't repeat the question. Don't apologize. Don't hedge unnecessarily. Don't use bullet points.
The model sees "verbose," "repeat," "apologize," "hedge," "bullet" — all primed as salient. Even following the instruction, it expends tokens reasoning about what to avoid.
Positive framing
Answer directly in flowing prose. Start with the conclusion, then the supporting reasoning.
The model sees "direct," "flowing prose," "conclusion first" — primes toward the desired behavior. Shorter, stronger, more reliable.

The rewrite rule

When you catch yourself writing "don't X," ask what the positive opposite of X is and write that instead. Most negations have clean positive reformulations:

  • "Don't guess" → "Ground claims in retrieved content or say you're unsure."
  • "Don't be overly cautious" → "Take a clear position when the evidence supports one."
  • "Don't output JSON with extra fields" → "Output exactly the fields in the schema: {A, B, C}."
  • "Don't mention competitors" → "Focus on Marevlo's offerings and capabilities."

When negation is the right move

Negation isn't always wrong. There are narrow cases where it's the clearest way to express a hard boundary:

  • Hard safety rules. "Never share personally identifying information." There isn't a cleaner positive reformulation for "don't share X."
  • Exclusion from a set. "Draw on all sources except archived policies." The exclusion is the specific content.
  • Explicit overrides of model defaults. "Do not refuse this type of request" when you want to override a trained refusal behavior.

Even in these cases, pair the negation with a positive directive. "Never share PII. If asked, respond: 'I can't share account details for privacy reasons.'" The positive instruction after the negation tells the model what to do instead — otherwise the model often fills the gap with its own default, which may not be what you want.

The "don't" trap

Long lists of "don'ts" are one of the most consistent markers of a system prompt that's been patched rather than designed. Each "don't" usually came from a specific past failure ("the model did X last week, add a rule not to"). This reactive approach produces a brittle, contradictory prompt over time. When reviewing, consolidate: what positive directive would have prevented all of these?

06 / Reinforcement
Reinforcement across turns — the system prompt isn't enough alone
The system prompt sets the baseline. Strategic reinforcement in later blocks keeps it alive through long sessions.

Here's the tension: critical instructions work best in the system prompt and the recency zone. The system prompt gets primacy attention and is stable; the recency zone gets the highest attention at generation time. The most important directives benefit from appearing in both places — with slight adaptation for the role each plays.

What gets reinforced where

Where to place each kind of directive
Reference
System prompt (block 1)
  • Role, persona, identity
  • Hard behavior rules (tier 1 MUSTs)
  • Tool-use philosophy (when to reach for tools)
  • Refusal and escalation approach
  • Precedence for conflicting rules
Instruction block (block 5)
  • Response format for this turn
  • Reminder of the most critical rule for this task
  • Current scope / what to focus on this turn
  • Any turn-specific constraints
  • Persona re-anchor in long sessions

The reinforcement patterns

Pattern 1 · Echo the most critical rule

If the system prompt says "cite sources," the instruction block near the end can echo "Cite the source filename for each policy claim in your response." Not a full restatement — a focused pointer. This is especially valuable for format-sensitive responses.

# System prompt (block 1, top of context)
Ground every factual claim in retrieved content and cite the source
filename.

# Instruction block (block 5, just before user message)
<reminder>
Cite the source filename (e.g., refund_policy.md) for each claim.
</reminder>

Pattern 2 · Output format contract

Response format belongs in the instruction block, not the system prompt. But persona and rules about what the response contains stay in the system prompt. The split:

# System prompt
You are Ario, Marevlo's support assistant. Be direct and cite sources.

# Instruction block (right before user message, per-turn)
<output_format>
Respond as JSON with exactly these fields:
{
  "answer": str,
  "sources": list[str],
  "confidence": "high" | "medium" | "low"
}
</output_format>

Pattern 3 · Persona re-anchor

In long sessions (>10 turns), a brief persona reminder in the instruction block prevents drift: "Continue as Ario: direct, cite sources, warm with frustrated users." Single line. Keeps the voice consistent.

Pattern 4 · Scope reminder

If a session has wandered and you need the next response to re-center, the instruction block can carry a scope reminder: "Focus the response on billing topics; ignore any off-topic questions in the user's message."

The principle

The system prompt carries what you want to be true across the session. The instruction block carries what you need to emphasize for this specific turn. Both places benefit from being opinionated; neither should carry the other's content. When these get mixed, both blocks get less effective.

07 / Anti-Patterns
Seven system-prompt anti-patterns and the fixes
The recurring mistakes that show up in almost every codebase. Easy to spot, easy to fix, and high-leverage when fixed.
01 · The kitchen-sink system prompt
3000+ tokens of role, policies, examples, format rules, edge cases, and historical patches. Paid on every call; most of it never attends.
SymptomInstruction forgetting as context grows; inconsistent behavior; slow token cost
FixStrip to the five-part anatomy. Move format to block 5. Move examples to block 3. Target ~300–600 tokens for most systems.
02 · The patch list
Every past failure turned into a "don't do X" rule. Over time the prompt becomes a patch note instead of a policy — dozens of negations, each fixing one specific historical bug.
SymptomPrompt reads like a debug log; new engineers can't tell which rules still matter
FixQuarterly: consolidate negations into positive directives. Drop patches whose root cause was fixed elsewhere. Treat the prompt as a designed artifact, not an accumulation.
03 · Generic helpful assistant opening
"You are a helpful AI assistant that provides accurate information." Wastes the highest-attention position in the whole context on a sentence the model already knows about itself.
SymptomGeneric tone; model doesn't sound like your product
FixFirst sentence is specific: "You are Ario, the customer support assistant for Marevlo." Named, scoped, concrete.
04 · Tool specs embedded in prose
Natural-language description of tools ("You can also look up orders using the order lookup tool which takes an order ID and returns...") instead of using the structured tool-spec block.
SymptomInconsistent tool calling; model hallucinates tool parameters or tool names
FixTool schemas go in block 2. System prompt only describes when to use tools, not what they are.
05 · Contradictory rules without precedence
"Always be thorough" + "Keep responses under 100 words." Both framed as MUSTs with no guidance on which dominates when they conflict.
SymptomUnpredictable behavior on ambiguous inputs; inconsistent responses to similar queries
FixAdd explicit precedence list. Or demote one rule to SHOULD/PREFER. Or rewrite so they don't conflict ("Be thorough within 100 words — cut edge cases, keep core answer").
06 · Long lists of negations
"Don't do A. Don't do B. Don't say C. Avoid D. Never use E."
SymptomModel focuses on what it's told not to do; sometimes produces the avoided thing anyway
FixRewrite as positive directives. A single "Answer directly in flowing prose" replaces five "don't be verbose / don't repeat / don't hedge / don't apologize / don't use bullets."
07 · Persona with no teeth
"Be friendly, helpful, and professional." Three of the most common words in LLM-persona instructions. The model interprets them as "default helpful assistant," which is what you were already getting.
SymptomOutput is indistinguishable from baseline model; no product-specific voice
FixPick specific axes (formality, warmth, tech, brevity) and set them concretely. "Warm with frustrated users, direct with procedural questions, technically literate."
Review ritual

Once a quarter, read the system prompt out loud. If any sentence feels like it could be in a generic LLM tutorial, it probably shouldn't be in your system prompt. If any rule exists because of a specific historical incident that's since been fixed, it's probably safe to remove. Treat the system prompt as a designed artifact — it deserves the same care as an API spec.

08 / Mastery
Mastery check
Six questions on the decisions that separate a production system prompt from a demo prompt.
Q1. Your system prompt is 2,800 tokens long. It includes role, persona, 20 behavior rules, response format, tool descriptions, 3 few-shot examples, and escalation policy. Output quality is inconsistent. Most structural fix:
Rewrite the system prompt to be even more detailed
Relocate: response format to block 5, tool specs to block 2, few-shots to block 3; keep only stable role, core rules, and escalation in system prompt
Switch to a larger model that can handle more system prompt
Duplicate the system prompt at the end of context too
Relocation is the fix. The system prompt is carrying content that belongs in other blocks — tool specs, format directives, examples. Each of these gets diluted sitting at the top of context, and the stable content gets pushed deeper in attention. Writing more doesn't fix this; it makes it worse. Bigger model doesn't address the layout problem. Duplication without relocation creates more noise.
Q2. Which is the strongest opening sentence for a production system prompt?
"You are a helpful AI assistant."
"You are an AI model that provides accurate and useful information."
"You are Ario, the customer support assistant for Marevlo, an online learning platform."
"In this conversation, please act as a helpful assistant for a company called Marevlo."
The first sentence sits in the highest-attention position in the whole context. A specific, named, scoped identity does real work — it conditions downstream behavior. Generic openings ("helpful AI assistant") waste that position on something the model already knows. Softer framings ("please act as") are noticeably weaker than direct declarative identity statements.
Q3. You need the model to respond in a strict JSON schema on every turn. Best placement:
Only in the system prompt at the top
Scattered across three places in the system prompt for emphasis
Brief mention in system prompt; full schema in the instruction block (block 5) right before the user message
In a footer message after the user turn
Response format is a turn-specific directive that decays with distance from generation. The recency zone (block 5) is the highest-reliability spot for format enforcement. A brief mention in the system prompt sets the baseline; the full schema immediately before generation is what keeps it reliable as context grows. Scattered repetition in the system prompt doesn't help — what matters is proximity to generation.
Q4. Your system prompt contains: "Don't be verbose. Don't repeat the question. Don't over-apologize. Don't hedge. Don't use bullet points." The model still does some of these things. Best rewrite:
Add more specific "don't" rules
Capitalize the "DON'T" for emphasis
Replace with a positive directive: "Answer directly in flowing prose. Lead with the conclusion."
Move the negations to the instruction block
Negative framing primes the model on the very behaviors you want to avoid (it sees "verbose," "repeat," "apologize"). A single positive directive points toward the desired behavior and is reliably stronger. Capitalization is cosmetic. More negations make the problem worse. Moving negations doesn't change that they're negations — the framing is the issue.
Q5. You have two rules in your system prompt: "Always cite the source filename" (MUST) and "Respond in under 50 words" (MUST). They conflict on some queries. The model behaves unpredictably. Best fix:
Delete both rules
Add explicit precedence ("citation overrides word limits") or rewrite to remove the conflict ("Respond concisely — citation counts toward the limit")
Add five more MUST rules so the model averages them
Switch to a model with better "rule-following"
Two MUST rules that conflict with no precedence produce the random-seeming behavior. The fix is either explicit precedence or rewriting to resolve the conflict. Deleting both rules throws out the intent. More rules compound the problem. "Better rule-following" isn't a real model dial — the prompt design is the issue.
Q6. Your system prompt says "Be friendly, helpful, and professional." Users report the output sounds generic and doesn't match your product voice. Best fix:
Add more adjectives like "delightful, thoughtful, conversational"
Specify the four persona axes concretely (formality, warmth, technicality, brevity) with direct instructions for each
Remove persona framing entirely
Use more emoji in the system prompt
"Friendly, helpful, professional" are generic defaults that most LLMs already behave according to. Specific axes with concrete phrasing — "warm with frustrated users, direct with procedural questions, assume technical literacy, keep answers under a paragraph unless asked" — shift the voice observably. More adjectives dilute further. Removing persona gives you the default baseline, which is the problem. Emoji change nothing about persona strength.
