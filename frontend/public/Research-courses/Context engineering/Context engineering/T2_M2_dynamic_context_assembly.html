The context is
built at request time.
Engineer the builder.

M0 defined the budget. M1 defined the layout. This module is about the process that turns a user query into the final assembled context — per request, reliably, with budgets enforced and provenance preserved. Most production failures happen here, not in the prompt: in scattered string concatenation, ad-hoc trimming, uncached stable blocks, and silent truncation. The fix is a pipeline that treats context assembly as a first-class system with stages, policies, and observability.

01 / Reframe
From static prompt to dynamic assembly
Why "just write the prompt" stops working the moment your system has more than one type of request.

The mental model most teams start with: there is a prompt, written in a file, maybe with a few variables interpolated, maybe prepended to some conversation history, and passed to the model. It's static. It lives in the repo. You iterate on it by editing the file.

That model holds up for roughly the first demo. It breaks the moment your system has to do any of the following: retrieve different documents for different queries, maintain memory across turns, use different tool subsets depending on user intent, handle users with different permissions, operate under variable budgets based on model tier. At that point there is no longer a prompt. There is a context assembled per request, and the quality of your system becomes the quality of the assembler.

The shift

In static-prompt systems, the prompt is the artifact. In dynamic systems, the assembly pipeline is the artifact — the prompt is one ephemeral output of it. Debugging, testing, versioning, and improvement all move from the prompt file to the pipeline.

What the assembler has to do, concretely

On every request — typically within a few hundred milliseconds — the assembler has to:

  1. Gather inputs from sources that don't know about each other: system prompt template, tool registry, retrieval results, memory store, conversation log, user's current turn.
  2. Decide relevance: which of those inputs matter for this request? A travel query doesn't need refund-policy chunks; a specific user doesn't need enterprise-only tools.
  3. Allocate budget across categories: given a 128K window and an effective ceiling of 80K, how much goes to retrieved content vs history vs output reservation?
  4. Trim to fit when things don't: drop lowest-priority blocks first, summarize rather than truncate when possible, never silently exceed the window.
  5. Lay out the remaining blocks in canonical order (per M1), with proper delimiters and provenance.
  6. Emit observability: per-category token usage, cache hit/miss, time spent per stage, block sources. So when something's wrong, you can see what went in.
  7. Return the assembled object — not a string, a structured object — to the code that calls the model.

Why this is where failures concentrate

If you talk to teams who've been shipping LLM products for a year, most of their production incidents clustered in the assembly layer, not in the model call or the prompt text itself. The top three failure classes:

  • Silent truncation. Context exceeded the window, the provider silently cut tokens, the model saw a different layout than expected, behavior changed without an error.
  • Stale stable blocks. System prompt was cached incorrectly, a policy update shipped, and 20% of requests were still running old policy for two days.
  • Layout drift. Two code paths built context differently; the same user question produced different answers depending on which path handled it.

All three are assembly-layer bugs. None of them are model bugs. None of them are prompt bugs. They're engineering bugs in a layer that, for many teams, is still a string-concat function hidden in a controller. This module is about building that layer properly.

02 / Pipeline
The assembly pipeline — seven stages, one flow
A canonical pipeline that covers the needs of most production LLM systems. Treat each stage as a swappable, testable unit.

The assembly pipeline looks the same at the shape level across almost every production system. The variation is inside the stages — what retrievers you use, what policies apply, what budgets you set. The pipeline itself is stable. Internalizing it gives you a map: when something's wrong, you can locate it by stage.

Context assembly pipeline — seven stages
Diagram

What each stage is responsible for

Stage 1 · Resolve

Classify the request. Is this a question-answering request? An agent task? A conversation continuation? Different kinds of requests get different templates (§5). This stage also handles routing — which model tier, which retrieval corpus, which tool subset.

Stage 2 · Gather

Fan out to every source that could contribute content: vector retrieval, keyword search, memory store, tool registry, conversation log. Do this in parallel — sequential fetches here are the most common unforced latency penalty in LLM systems.

Stage 3 · Rank & filter

Rerank retrieved candidates with a cross-encoder or scoring model. Dedupe. Apply a relevance floor (cut anything below it — don't include "just in case" — recall the distractor tax from M0). This is where the set of candidate chunks becomes an actual selection.

Stage 4 · Budget

Allocate your effective-context budget across categories (§3 of this module). Each category gets a cap. If every incoming block fits within its cap, you move on. If not, you proceed to trim.

Stage 5 · Trim

Drop or compress lowest-priority blocks until under budget (§4). Summarize older history if allowed. Never let the pipeline silently pass an oversized context downstream.

Stage 6 · Layout

Assemble the final blocks in canonical order (per M1), apply delimiters, add provenance, insert the turn-specific instruction near generation. This is deterministic templating — no decisions happen here, just formatting.

Stage 7 · Emit

Final validation (did we exceed window? do required blocks exist?). Emit metrics. Return the assembled AssembledContext object to the caller. Errors raised here are bugs, not runtime behavior — they indicate the pipeline itself is broken.

Invariant

There is one function in your codebase that runs this pipeline. Every code path that talks to the model must go through it. If a feature needs "just a little different" context, it gets a new template (stage 1 selection), not a parallel assembly path. This single-entry invariant is what kills layout-drift bugs permanently.

03 / Budget
Budget allocation — run a stage, see the tokens move
Walk through the pipeline one step at a time. Each stage produces a different shape of context.

The budget stage is where most of the real engineering happens. It takes the post-rank set of candidate blocks and decides, given an effective-context ceiling, how many tokens each category gets. The policy is simple to state; the values are the engineering work and depend on your workload.

The allocation equation

Given an effective context ceiling E, you allocate fixed caps to the stable categories (system, tools, output reservation) and distribute the remainder across the dynamic categories (retrieved content, history, memory) by ratio:

E = effective_context_ceiling           # from M0: measure this, don't use advertised

fixed = system_tokens + tool_tokens + output_reserve
dynamic_budget = E - fixed

# Distribute the dynamic budget by policy
retrieved_cap = dynamic_budget * 0.55   # typical for RAG-heavy apps
history_cap   = dynamic_budget * 0.30
memory_cap    = dynamic_budget * 0.15

The ratios are task-dependent. A long-running assistant with dense prior context might flip it to 55% history / 30% retrieved. An agentic workflow might split 40% retrieved / 40% history (scratchpad) / 20% memory. The point is not the numbers — it's that you set them explicitly as policy, rather than letting whichever category fetched most content win.

Three allocation strategies

Strategy A
Fixed caps
Hardcode per-category budgets. Simplest to reason about. Works when workload is uniform and you know the shape of each request.
✓ Predictable, easy to debug
✗ Wastes budget on some queries, starves others
Strategy B
Ratio-based
Fixed ratios of remaining budget after stable categories. Scales gracefully as effective context changes across models.
✓ Portable across models
✓ Good default
Strategy C
Intent-conditional
Different templates → different ratios. A "lookup" intent might use 75% retrieved; a "continue conversation" intent 75% history.
✓ Highest quality at scale
✗ More templates to maintain
Default recommendation

Start with Strategy B (ratio-based) with sensible defaults (e.g., 55/30/15 for retrieved/history/memory). Move to Strategy C (intent-conditional) only once you have evidence from evals that different intents have meaningfully different optimal allocations. Don't start with C — you'll spend weeks tuning ratios for intents that don't have enough traffic to warrant separation.

04 / Trimming
Priority-based trimming — what gets dropped when things don't fit
Drag the cap. Watch blocks drop in the right order. This is what saves you from silent truncation.

When the sum of allocated content exceeds the effective context, something has to go. Which "something" is the trimming policy. A good policy produces stable, predictable behavior; a bad one drops critical content and the system silently degrades. The right default is explicit priority per block, lowest-priority-first drop.

Priority tiers

In most systems, blocks fall into four priority tiers. Codify them once:

  • P0 · Must-keep. System prompt, current-turn instruction, user message. If any of these have to be trimmed, the pipeline should raise, not quietly cut.
  • P1 · High. Top-ranked retrieved chunks (1-3), recent conversation turns (last 3-5), active tool specs.
  • P2 · Medium. Mid-ranked retrieved content, mid-history turns, ancillary tool specs.
  • P3 · Low. Low-ranked retrieved chunks, older history, soft context (persona hints, examples marked optional).

Four trimming techniques — in order of preference

Dropping a whole block is the blunt instrument. Before reaching for it, try the subtler techniques:

  1. Summarize. Replace old conversation turns with a short summary block. 20 turns of conversation → one paragraph summarizing key decisions. Costs one LLM call per session boundary (amortized cheap), preserves information.
  2. Compress. Strip boilerplate, redundant metadata, or verbose formatting from blocks. A retrieved chunk with 200 tokens of footer can become 400 tokens of content.
  3. Truncate tail. For chunks where the earliest content is more relevant (typical of well-chunked documents), cut the tail.
  4. Drop entirely. When none of the above apply or are worth the complexity, drop the lowest-priority block whole. Record what was dropped in the observability trail.
def trim_to_fit(blocks: list[ContextBlock], cap: int) -> list[ContextBlock]:
    # Sort by priority ascending — lowest priority dropped first
    candidates = sorted(blocks, key=lambda b: -b.priority)
    total = sum(b.tokens for b in candidates)
    drops = []

    while total > cap and candidates:
        if candidates[0].priority == 0:
            raise ContextOverflowError(
                f"cannot fit: only P0 blocks remain ({total} > {cap})"
            )
        dropped = candidates.pop(0)
        total -= dropped.tokens
        drops.append(dropped)

    # Re-sort for layout — by canonical order, not priority
    kept = sorted(candidates, key=lambda b: canonical_order[b.kind])

    # Emit drops to observability — never silently lose blocks
    log_context_drops(drops)

    return kept
The silent-truncation trap

If your pipeline hands an oversized context to the API and lets the provider truncate, you don't know what the model saw. Different providers truncate differently; some cut from the front, some from the back, some error. Never rely on this. The pipeline must fit the context locally, before the call, and must log every drop. Silent truncation is the single nastiest bug class in LLM systems because it reproduces intermittently.

05 / Templates
Templates and policies — versioned, testable, composable
The pipeline is the engine. Templates are the configurations that let one engine serve many request types cleanly.

A template in this module isn't a string with placeholders. It's a named configuration of the assembly pipeline: which gather sources to use, what ratios to allocate, what system prompt and instruction block to render, what tool subset to expose. Templates are the right unit for versioning, A/B testing, and multi-tenancy. "We changed the customer-support template" is a much more useful sentence than "we changed the prompt somewhere."

What lives in a template

class AssemblyTemplate(BaseModel):
    name: str                                   # e.g., "customer_support_v4"
    version: str                                # "4.2.1" — semver

    # Stable content
    system_prompt: str
    instruction_template: str                   # jinja — rendered per turn

    # Source selection
    retrieval_corpora: list[str]              # which indices to query
    tool_subset: list[str]                   # tool names, not specs
    memory_scopes: list[str]                 # user / session / global

    # Budget policy
    effective_context: int                  # the ceiling for this model tier
    output_reserve: int
    allocation: dict[str, float]            # ratios for dynamic categories

    # Retrieval policy
    top_k: int
    rerank_model: str
    relevance_floor: float

    # Trim policy
    history_compression: Literal["none", "summarize", "truncate"]
    priority_defaults: dict[str, int]

    # Metadata
    owner: str
    last_modified: datetime
    eval_set: str                                # which eval suite this template is benchmarked against

Why templates matter in practice

  • Routing becomes declarative. Stage 1 (Resolve) now just picks the right template by intent + user segment. No branching logic in the assembly code itself.
  • Versioning becomes safe. Bump to v4.3.0, keep v4.2.1 around, route 5% of traffic to the new one, compare metrics.
  • Multi-tenancy becomes straightforward. Each tenant gets a template variant. No "if customer_id == X do this" code smell.
  • Evals target templates, not prompts. Your eval set runs against a template version, so you can regression-test before flipping traffic.

Policies — the things templates share

Some logic shouldn't duplicate across templates. That logic lives in policies — shared, reusable components templates reference:

  • Trim policy. How to handle overflow — which tiers, whether to summarize, when to raise.
  • Rerank policy. Which reranker, what threshold, whether to dedupe.
  • History compression policy. When to summarize, how many turns to keep verbatim, what model to use for summarization.
  • Safety filter policy. What to strip or flag in retrieved content before it reaches the model.
Architecture advice

Templates describe what assembly to do. Policies describe how certain classes of decisions get made. Keeping these separate is what lets you change "all templates now use the new trim policy" as a one-line change — rather than editing 12 templates.

06 / Cache
Caching vs freshness — where to draw the line
The cheapest token is the one you don't recompute. But stale cached context is worse than slow context.

Context assembly latency and cost are usually dominated by a few stages — retrieval, reranking, sometimes memory summarization. Caching lets you pay those costs once and reuse the result many times. Most providers also support prompt caching on their side, where identical prefixes across requests are cached on the inference servers, cutting cost by ~90% for the cached portion. Doing both right is a meaningful lever on cost and latency.

What to cache, and for how long

ContentCache?TTLInvalidate on
System prompt yes days template version bump
Tool specs yes days tool schema change
Retrieved chunks (per query) yes minutes–hours corpus update, query change
Rerank scores yes minutes reranker version, query change
Memory summary (per session) yes session lifetime new turn that triggers re-summary
Conversation history (raw) no — changes every turn
Current-turn instruction no — per-turn value
User message no — per-turn value

Ordering for provider-side prompt caching

Provider prompt caches key on the prefix of the request. For this to help, your context layout needs all-stable content at the front and all-dynamic content at the back. The canonical layout from M1 is already structured for this — system prompt, then tool specs, then the largest semi-stable content (task-scope retrieval), then fully-dynamic content (history, instruction, user message). Don't disrupt this with small ordering changes; a single different token in the prefix busts the cache for everything after it.

The freshness vs cache tension

Cache TTLs are a direct tradeoff between cost savings and staleness. Three principles for getting the balance right:

  • Invalidate on source change, not on time. For corpus retrieval, tie the cache key to the corpus version, not just to the query. A policy update triggers invalidation of the right queries automatically.
  • Shorter TTLs for dynamic content, longer for stable content. System prompts can cache for days; retrieved chunks should cache for minutes (unless your corpus is truly static).
  • When in doubt, default fresh. The cost of a stale answer in production is usually much higher than the cost of an extra retrieval call. Cache aggressively at the infrastructure level (provider prompt cache) and conservatively at the content level.
Caution

Caching retrieved content aggressively across users, without care, can leak information between them — if cache key is (query) without (tenant_id), User A's retrieval results can surface for User B. Always scope cache keys by the tenancy of the data being retrieved. This is a security failure, not just a quality one.

07 / Observability
Observability — the pipeline that tells you about itself
You cannot improve what you cannot measure. For context assembly, the right metrics make all the difference.

Observability isn't a bonus. For context assembly, it's the thing that turns "the model is behaving oddly" from a black-box mystery into a specific, addressable bug. Every stage should emit metrics; every assembled context should be reproducible from logs; every drop and cache miss should be countable. A pipeline without observability is indistinguishable from a pipeline that works.

Per-request logging

For every assembled context, log enough to reproduce it. At minimum:

  • Request ID — ties to the model call, the response, and any user-facing trace.
  • Template name + version — lets you answer "did this bug happen under the old template or the new one?"
  • Per-block provenance — source, retrieval score, priority, final inclusion status.
  • Drops — what got trimmed, from which tier, with total tokens dropped.
  • Token counts per category — system, tools, retrieved, history, memory, instruction, user.
  • Stage timings — latency per pipeline stage so you can find the slow stage when p99 spikes.
  • Cache hit/miss per cacheable stage.

Alerting on the right signals

Set alerts on signals that indicate the pipeline is degrading, not just that it's slow:

  • Drop rate > threshold. If P3 drops are happening on >5% of requests, your budgets are too tight or your retrieval is returning too much.
  • P1 drops occurring at all. A P1 drop means important content was cut — this should almost never happen. Alert on count > 0.
  • Cache hit rate collapse. A sudden drop in cache hits indicates a bust — often from an unintentional template change or non-deterministic prefix.
  • Assembly latency p99 > 2× baseline. Something in the pipeline is slow; check per-stage timings.
  • tokens_total / effective_context > 0.9. Running close to the ceiling means you're one retrieval spike from overflow.
The upstream discipline

Observability isn't a feature you add at the end. It's a pipeline invariant — every stage emits, no stage hides, one function owns the assembly. When you inherit or design an LLM system, the fastest proxy for "how mature is this?" is to ask: if a user reported a bad response, could you, within 5 minutes, pull up the exact context that produced it? If not, you have work to do at this layer before anywhere else.

08 / Mastery
Mastery check
Six questions on the pipeline decisions that separate production-grade LLM systems from glued-together prototypes.
Q1. Your team has context assembly scattered across three controllers. A bug report says the same user question gave different answers from two different entry points. Most structural fix:
Write a linting rule that requires all controllers to call a specific string-concat helper
Centralize all context assembly in one pipeline function; every code path must route through it; treat this as a system invariant
Add unit tests that check the final prompt matches a golden file
Switch to a more robust model
Single-entry pipeline is the only fix that prevents drift by construction. Linting is preferences, not enforcement. Golden-file tests catch specific regressions but don't prevent a fourth entry point from being added. Model robustness is unrelated — the bug is structural in the assembly layer.
Q2. Your context hits the effective context ceiling 10% of the time. Blocks are being silently truncated by the provider. The correct response is:
Reduce top_k on retrieval for all queries
Trim locally in the pipeline before the call using explicit priorities; never pass oversized context to the provider; log every drop
Switch to a model with a larger window and rely on its truncation behavior
Add a retry loop that halves content on 413 errors
Local, priority-based trimming is the only way to know exactly what the model saw. Provider truncation is unpredictable across providers and model versions. Reducing top_k globally sacrifices quality for 90% of requests to handle 10%. Retry-on-error is reactive and leaves the pipeline dependent on the provider's specific truncation behavior.
Q3. Which of the following should be cached across requests for maximum benefit?
User message (it might repeat)
Conversation history
System prompt + tool specs (stable across most requests, large enough to matter, safe to cache)
Current-turn instruction
System prompt + tool specs are exactly the kind of stable, large, prefix content that benefits from both application-level and provider prompt caching. User messages and current-turn instructions are per-turn and never benefit from cache. History changes every turn. The canonical layout in M1 is structured with stable-first specifically so provider caches work.
Q4. You have three request types (lookup, chat, agent). Each has meaningfully different optimal budget allocations. What's the right way to handle this?
One template with dynamic branching in its allocation logic
Three templates, each with its own allocation ratios, selected by stage 1 (resolve) based on intent
Let the model decide at runtime how to allocate
Pick the ratios that are a best compromise for all three
Three templates with intent-conditional selection is exactly Strategy C from §3 — cleaner than branching logic inside a single template, versionable separately, evaluatable separately. Best-compromise ratios waste budget on every request type. The model can't meaningfully allocate its own context at runtime without an extra model call — a significant cost/latency tax.
Q5. Your pipeline's cache hit rate on retrieval just dropped from 70% to 15% with no apparent change to the corpus. Most likely cause:
The embedding model got worse at retrieval
Users are asking harder questions
Cache key drift — the key now depends on a non-deterministic or per-request value, busting the cache on every call
The cache server is full
Sudden cache hit collapse with no corpus change is almost always cache key drift — someone added a per-request value (timestamp, session ID, user ID included unnecessarily) to the key. This is a common bug introduced by well-intentioned refactors. Embedding quality degradation wouldn't halve the hit rate overnight; user behavior shifts slowly; full caches evict but don't collapse hit rate to 15% unless sized far too small. Always log cache key composition to diagnose this quickly.
Q6. Your pipeline emits per-stage timings and token counts but doesn't log per-block provenance. A user reports a bad answer. You can identify the slow stage, but not which retrieval returned the problematic chunk. The missing observability investment is:
Faster logging infrastructure
Log per-block provenance (source, score, priority, inclusion status) on every assembled context — so any specific block can be traced back to its origin
More aggressive alerting on slow stages
Version-bump the template to match the bad response
Per-block provenance is the piece of observability that lets you reproduce exactly what the model saw — down to which retrieval call returned which chunk. Without it, you know the pipeline was slow but not why the answer was bad. Aggregate timings don't help with content-level investigations. The bar: given a request ID, could you reconstruct the exact assembled context in five minutes? If not, provenance is missing.
