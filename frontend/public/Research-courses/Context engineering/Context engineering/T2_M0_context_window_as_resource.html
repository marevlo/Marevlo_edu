The window is not
a blank page.
It's a budget.

Every token you put in front of a model costs you latency, money, and — most surprisingly to new builders — attention the model could have spent on something more important. A 200K-token window sounds like abundance, but it's a shared ledger: system prompt, tool specs, conversation history, retrieved documents, and the model's own output all draw from the same pool. This module reframes the context window from container to constraint, and sets up the discipline the rest of Track 2 builds on.

01 / Reframe
From "input" to "budget"
Why the mental model you bring to context determines the quality of every downstream decision.

Most people new to building with LLMs start with a mental model inherited from traditional software: the prompt is input, the completion is output, the context window is the space inside which this happens. It's a reasonable starting point. It is also the reason so many first attempts at agents, RAG systems, and long-running assistants fail in ways that feel confusing and arbitrary.

The mental model that ships production systems is different. The context window is not space, it is a budget. Every token you spend comes from a finite pool, and every token you spend has three costs: the dollars you pay per input token, the milliseconds the model takes to process it, and — the cost that blindsides new builders — the attention the model could have allocated elsewhere. Adding tokens is never free, and often it actively degrades quality.

The reframe

Stop asking "what can I fit in the context window?" Start asking "what should I spend my token budget on, given what this model does well with which positions?" The first question is a container question. The second is an engineering question, and it is the entire subject of context engineering.

Why larger windows didn't solve the problem

When context windows jumped from 4K to 32K to 200K to 1M+ tokens, a reasonable hope was that context engineering would become less important — you could just "throw everything in." In practice the opposite happened. Larger windows made the costs of how you use them more consequential, not less, because:

  • Attention spreads. The model still has finite attention capacity per layer; feeding it 500K tokens doesn't mean it pays equal attention to all of them. Most of the attention goes to the edges and to a few tokens that happened to grab it.
  • Latency scales worse than linearly. Doubling context often more than doubles time-to-first-token. At 200K inputs, latency can exceed the useful response time budget for many applications.
  • Cost scales linearly. A 1M-token input on frontier pricing is real money. An agent that fills context indiscriminately can 10× your unit costs without improving quality.
  • Noise wins ties. When retrieval surfaces 50 marginally-relevant documents instead of 3 highly-relevant ones, the model's answer is noticeably worse, not better.

Context engineering as a distinct discipline

Prompt engineering, as practiced in 2023, was mostly about crafting a good instruction. Context engineering is the larger discipline that contains prompt engineering and adds: what to retrieve, what to remember, what to strip out, what to compress, what order to place things in, and how to budget across all of it. The instruction is just one of several tenants sharing the window.

This track treats the context window the way a systems engineer treats RAM: a finite resource to be allocated deliberately, with tight feedback loops on what's wasted. By M8 you'll have the full toolkit. This module lays the foundation: what the resource actually is, and why models don't use it the way you'd expect.

02 / Budget
The token budget equation
Every production context is a mix of five categories. Drag the sliders to see how the mix plays out — including when you've overspent.

Every production system's context breaks down into five budget categories. If you don't track them as separate line items, you'll be surprised when one silently eats the others. The categories:

  • System / instructions. Role, persona, policies, response format guidance. Usually fixed per session.
  • Tool specs. Function definitions, schemas, tool descriptions. Grows linearly with tool count (cf. Track 1 M1).
  • Conversation history. Prior user turns and assistant responses. Unbounded by default — the most common budget eater.
  • Retrieved / injected content. RAG chunks, fetched documents, database rows, prior memory summaries.
  • Output reservation. The tokens you leave free so the model can actually respond. Not a number the model will warn you about — if you fill the window, generation just truncates.

The silent overruns

Three categories reliably blow past their intended budgets without anyone noticing:

  • History — grows one exchange at a time; each individual increment feels small; at turn 20 you suddenly have 60K tokens of back-and-forth.
  • Retrieved content — top_k=10 feels reasonable, until your chunk size doubled on a re-index and the same call is now returning 80K tokens instead of 40K.
  • Output reservation — until you ask for a long structured output and the model hits max_tokens because you forgot to leave room.
Invariant

In production, track the five categories as separate metrics, not as one aggregate "tokens used." The aggregate tells you that you have a problem; the per-category breakdown tells you which one.

03 / Attention
Attention is not uniform across the window
The model pays much more attention to some regions than others. Knowing the shape of that curve changes where you put what.

The most consequential fact about transformer models, for context engineering purposes, is that they do not treat all positions in the context window equally. Attention is distributed unevenly, and the distribution has a characteristic shape: strongest near the very beginning, strongest near the very end, weakest in the middle. This is the well-known "lost in the middle" phenomenon, and it is not a bug you can prompt your way around — it's a property of how these models are trained and how attention scales.

Approximate attention shape across a long context
Diagram

The curve shape is qualitatively consistent across frontier models, though the depth of the middle dip varies — older or smaller models have much sharper dips; newer frontier models are flatter but not flat. Even an "almost flat" curve means a 5–10% recall hit on middle content, which is often the difference between passing and failing an eval.

What this means operationally

  • Put load-bearing instructions at the edges. System prompt opens. Critical reminders close (right before generation).
  • Don't bury the lede. If a retrieved document is the single most important piece of evidence, don't put it at position 50 of 100.
  • Order matters as much as content. Two contexts with identical tokens in different orders can produce meaningfully different answers.
  • The flatter the claim, the more you should test it. "Our 1M-context model handles long context perfectly" is a marketing sentence, not a technical one. Run needle-in-haystack evals on your actual workload.

The token-level intuition

A stylized picture of how much attention each position gets when the model is answering a question about a specific retrieved fact:

[system prompt — you are a careful assistant...] [tool specs: search, calc, retrieve] [retrieved doc 1 — irrelevant company history — ...] [retrieved doc 2 — also irrelevant — 4000 tokens of Q1 earnings...] [retrieved doc 3 — THE KEY FACT sits here — 'refund window: 30 days' ...] [retrieved doc 4 — marginally relevant — ...] [history: user asked 3 turns ago about shipping...] [user: What's our refund window?]

Highlighted = high attention (system prompt + the user's actual question). The critical fact in doc 3 is in the low-attention zone. Re-ordering to put doc 3 right before the user question reliably improves answer correctness on this kind of query.

04 / Position
Position effects, measured — play with a stylized lookup
Drop a needle at different positions and watch the expected retrieval rate change. The effect is real, it is measurable, and it is what "lost in the middle" literally means.

The clearest way to build intuition for position effects is to run a needle-in-haystack test: plant a single fact ("the magic number is 42") at various positions in a long context, ask the model to retrieve it, and measure the accuracy at each position. Three patterns the experiment reveals

  1. Start and end are near-perfect. The first 5–10% and last 5–10% of positions retrieve the needle with accuracy close to 100% across frontier models.
  2. The middle dip is real and model-dependent. Between positions 30% and 70%, retrieval rates can drop substantially. Older or smaller models might dip to 50–60%; current frontier models typically stay above 85% but rarely hit 100%.
  3. Second-to-last position is often strongest. Not the very last token, which can get dominated by the question itself, but the section just before — the "recency" zone.
The actionable lesson

When a specific retrieved document is likely the answer, put it as close to the user's question as possible — typically the last item before the question, not the first item in a list. The strongest retrieval zone in most real-world context layouts is right before the instruction that asks the model to act on it.

What this does NOT mean

A few popular misinterpretations of position effects that lead teams astray:

  • "So we should only use small contexts." No — use large contexts when you need to, and engineer the layout. The dip is real but manageable.
  • "So we should put everything at the end." Only the single most important thing. If you pile everything at the end you just re-create the middle dip at the start.
  • "So the system prompt should be at the end." Role and persona are stable anchors. The model's pretraining expects them up front. Move the specific instruction near generation, not the whole system prompt.
05 / Effective
Effective context ≠ advertised context
The window size on the pricing page is an upper bound, not a capability guarantee. Know the gap.

Models advertise a maximum context length — 128K, 200K, 1M. This is a structural limit: the model can accept inputs up to that length without erroring. It is not the length at which the model performs well. Effective context is the length at which the model maintains its task quality. The gap between the two is typically large, often underreported, and matters enormously for architecture.

How to measure effective context for your workload

There is no universal number — effective context depends on the task. For retrieval-heavy tasks, it's typically <30% of advertised. For summarization, closer to 60%. For code generation across many files, closer to 40%. The way to know is to test:

  1. Pick a representative task. Not a toy needle test — something close to your actual production workload.
  2. Run it at 10%, 25%, 50%, 75%, 100% of the advertised window. Same content, padded to each length with varied distractors.
  3. Measure task-level quality at each length. Use your real eval metric, not a generic score.
  4. Find the knee. The length at which quality starts to drop meaningfully is your effective context for this workload.

A rough calibration across model classes

These are approximate, vary by model version, and should be treated as starting priors — not as substitutes for measuring your own workload:

Model class Advertised Effective (retrieval) Effective (summary) Notes
Small / efficient 200K ~20-40K ~60-80K Sharp middle dips; most sensitive to layout
Mid-tier 200K ~60-100K ~120-160K Good for most production workloads with care
Frontier 200K-1M ~100-200K ~200-400K Still position-sensitive; flat but not uniform
Trap

"Our model handles 1M tokens" and "our model does your task well at 1M tokens" are different claims. When a provider announces a context extension, the marketing often elides this distinction. Always run your own workload at the new size before committing architecture to it.

The budget-vs-effective relationship

Effective context is the ceiling on how much of your budget you can deploy productively. If a model's effective retrieval context is 80K tokens, putting 150K of retrieved content into the prompt will mostly waste the second half. The practical consequence: your per-category budgets in §2 should sum to less than your effective context, not less than your advertised context. This alone solves a lot of "the model seems to ignore half my retrieved docs" problems.

06 / Failures
The six failure modes of context overuse
Every team shipping an LLM product runs into these. Knowing them by name makes the first diagnosis faster than the second.

When context budgets slip, the failures have characteristic shapes. Being able to name what you're seeing turns "the model got worse" into a specific, fixable problem. Six patterns cover almost every production context failure.

01 · Attention dilution
Model outputs become hedged, generic, or miss specific details that are clearly in the context. Adding more retrieved docs made quality worse, not better.
FixRetrieve fewer, more relevant chunks. Rerank to put the best 3-5 near the question, not 20 in arbitrary order. Covered in M2 (Dynamic Context Assembly).
02 · Middle amnesia
The model remembers what was said at the start of the conversation and the last few turns, but "forgets" something from turn 12. Classic middle-dip.
FixSummarize + re-inject critical mid-history facts near the end. Don't rely on long history for recall; promote key facts to a persistent state block (covered in M5 — Memory Systems).
03 · Output starvation
The model produces a truncated or oddly short answer. Root cause: context is 99% full and the output reservation is tiny. The model has no room to actually respond.
FixAlways reserve an explicit output budget (e.g., 4K-8K tokens). Enforce it as a hard check before the call, not a post-hoc observation.
04 · Instruction forgetting
The system prompt said "respond in JSON." Fifteen turns later the model is producing prose. The instruction is technically in context but has been drowned out by everything between it and generation.
FixRe-inject critical instructions near the end, especially format-shape ones. "Respond as JSON per schema X" as the last thing before generation is far more reliable than only in the system prompt (covered in M3 — System Prompts).
05 · Context poisoning
A bad example or contradictory retrieved doc ends up in context and the model anchors on it. Subsequent turns compound the error because the bad content is now in history too.
FixTreat retrieval output and tool results as untrusted input. Validate before injection. Structure context with clear provenance markers so the model can tell a retrieved fact from an authoritative instruction.
06 · Distractor tax
Model performance on a task drops sharply even though the relevant information is still present — because you added distractor content that was irrelevant but plausible. This is the most expensive failure per token.
Fix"Relevance" is a budget filter, not just a ranking filter. Cut anything below a relevance floor; don't include it "just in case." The cost of plausible noise is much higher than the cost of a missing fact.
Meta-lesson

All six failures get worse as the context fills. None of them are caused by context size — they're caused by poor allocation within a given size. Doubling the window doesn't fix them; it just raises the ceiling before they start.

07 / First-Class
Treating context as first-class engineering state
The discipline Track 2 is built on. If you take one thing from M0, take this.

The through-line of this track is simple to state and deceptively hard to practice: context is state, and state deserves engineering. In most early LLM systems, the prompt is assembled as a string, somewhere deep in application code, by code that concatenates whatever happened to be available. There is no schema, no budget tracking, no validation, no observability. When things go wrong, there is no single place to look.

The production discipline treats context like a database schema or an API contract: it has a structure, it has per-category budgets, it has validators, and it emits metrics. Assembling context is a deliberate pipeline stage, not a string concatenation hidden in the call site.

Context as a first-class object — minimal schema
Code
from pydantic import BaseModel
from typing import Literal

class ContextBlock(BaseModel):
    kind: Literal["system", "tool_spec", "history",
                  "retrieved", "memory", "output_reserve"]
    source: str                  # where did this come from?
    tokens: int                 # measured, not estimated
    priority: int               # 0 = must-keep, 10 = first-to-drop
    content: str

class AssembledContext(BaseModel):
    blocks: list[ContextBlock]
    window_limit: int
    budgets: dict[str, int]    # per-kind caps

    def total_tokens(self) -> int:
        return sum(b.tokens for b in self.blocks)

    def usage_by_kind(self) -> dict[str, int]:
        out = {}
        for b in self.blocks:
            out[b.kind] = out.get(b.kind, 0) + b.tokens
        return out

    def validate_budgets(self) -> list[str]:
        issues = []
        usage = self.usage_by_kind()
        for kind, cap in self.budgets.items():
            if usage.get(kind, 0) > cap:
                issues.append(f"{kind} over budget: {usage[kind]} > {cap}")
        if self.total_tokens() > self.window_limit:
            issues.append(f"total over window: {self.total_tokens()} > {self.window_limit}")
        return issues

    def trim_to_fit(self) -> "AssembledContext":
        # Drop lowest-priority blocks until under window
        sorted_blocks = sorted(self.blocks, key=lambda b: -b.priority)
        kept, total = [], 0
        for b in sorted(self.blocks, key=lambda b: b.priority):
            if total + b.tokens <= self.window_limit:
                kept.append(b); total += b.tokens
        return AssembledContext(blocks=kept, window_limit=self.window_limit,
                                 budgets=self.budgets)

What you get when context is first-class

  • Observability. Emit the usage-by-kind breakdown on every call. You now know where your tokens are going without code-spelunking.
  • Regression safety. When a new feature adds a block, budget validation fails before you ship, not three weeks later when users notice degraded answers.
  • Debuggability. Every block has a source. When something's weird in the output, you can trace back which retrieval, which memory summary, which tool call introduced the problem.
  • Composability. Context assembly becomes a pipeline you can reorder, swap stages in, and A/B test — rather than a pile of string concatenation.
Road ahead

Everything in the rest of Track 2 is a refinement of this idea. M1 is about how to structure the blocks (information architecture). M2 is how to assemble them per-request (dynamic composition). M3-M7 each address a specific block kind (system, few-shot, memory, multi-modal, agent-state). M8 is how to evaluate the whole pipeline. All of it assumes context is state. If you stop here, at least take that with you.

08 / Mastery
Mastery check
Six questions on the mental models that make the rest of the track land.
Q1. A frontier model advertises a 1M-token context window. Your team's product is a RAG system that retrieves documents to answer questions. What's the correct target for how much context to actually fill?
As close to 1M as possible — more context means more information
Measure effective context on your actual workload; expect it to be much less than 1M for retrieval tasks, typically 10-30%
Exactly 50% of advertised — a safe halfway point
Always keep under 8K tokens regardless of model
Advertised context is a structural maximum, not a quality guarantee. Effective context for retrieval is typically 10-30% of advertised, varies by model and workload, and must be measured — not assumed. Filling to advertised size wastes tokens that aren't being attended to; the "50% rule" is a false precision; blanket 8K caps give up real capability when you need it.
Q2. Your chatbot's answer quality degrades noticeably around turn 15 of a conversation, even though context is only half-full. Most likely root cause:
Model is out of tokens
Critical facts are now in the middle of the context and attention is dropping there — "lost in the middle"
Model was fine-tuned for shorter contexts
Need to switch to a larger model
Classic middle-dip. At turn 15 the early-conversation facts that used to be "right before the question" are now buried mid-context, where attention is weakest. The fix isn't a bigger model or more tokens — it's promoting critical facts into a summary or state block that sits near the end (covered in M5). Running out of tokens produces truncation, not gradual degradation.
Q3. Which of the five budget categories most commonly blows past its intended allocation without anyone noticing?
System prompt
Tool specs
Conversation history — grows one exchange at a time, each increment feels small, aggregate is large
Output reservation
History is the silent overrun. System prompts are fixed; tool specs grow only when you add tools; output reservation is usually set explicitly. History accumulates invisibly and is the first thing to check when a session-scoped system starts behaving oddly. Retrieval is a close second — top_k + chunk_size interactions are a common silent grower.
Q4. You want to give the model a specific instruction it must follow in its very next response. Best placement:
In the system prompt at the very start
Spread across the system prompt and repeated three times
In the last user message, right before the model generates — the recency zone
In a separate tool call
The position with the highest attention at generation time is immediately before the model responds. System-prompt-only instructions are vulnerable to "instruction forgetting" when lots of context sits between them and generation. Critical, immediate, turn-specific instructions belong near the end. Note: stable role and persona still go in the system prompt — this is about specific, immediate directives.
Q5. Your retrieval component returns 30 relevant-ish chunks and you pass all of them to the model. Output quality is worse than when you used 5 highly-relevant chunks. Root cause:
Model can't handle 30 chunks technically — it errors silently
Distractor tax — plausible-but-irrelevant content consumes attention and dilutes focus on the chunks that matter
30 chunks is just too many tokens regardless
The retrieval is broken and none of the 30 are relevant
Distractor tax is the specific name for this: adding plausible-but-not-relevant content measurably degrades output quality even when the critical content is still present. Relevance is a budget filter, not just a ranking filter — cut below a threshold rather than including "just in case." Not a technical handling issue, not about raw token count (assuming you're under effective context), and often not a broken retriever.
Q6. Your team's prompt is currently assembled by string concatenation scattered across several functions in your app. Which investment most accelerates everything else in Track 2?
Switching to a larger model
Writing a longer system prompt
Treating context as first-class state — a structured object with typed blocks, per-category budgets, and observability
Adding more retrieval
This is the central discipline of Track 2. Without a structured context object, you can't measure per-category usage, can't enforce budgets, can't validate before calls, can't trace outputs back to blocks, and can't A/B test layout changes. All the techniques in M1-M8 assume this foundation. Bigger model, longer prompt, more retrieval each fix a specific problem but don't compose.
