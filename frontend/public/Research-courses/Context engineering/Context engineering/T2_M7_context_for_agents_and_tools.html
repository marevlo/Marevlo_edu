Agent context
grows with every step.
Control the growth.

A one-shot prompt is assembled once and discarded. An agent's context is cumulative — each thought, tool call, and observation joins the pile, then the whole pile goes back to the model for the next decision. By step 12 you're feeding the model 40K tokens of its own history to pick the 13th action. This module is about what to put in that pile, what to leave out, and how to keep the loop converging instead of drowning in its own trace.

01 / Reframe
Agents aren't one-shot — context accumulates by design
Everything you learned in M0–M6 still applies. But now there's a new failure surface: the context grows every turn, and you're responsible for its shape.

In a one-shot prompt, you assemble the context once — system, tools, retrieved, history, instruction, user — and send it. The model responds. Done. The assembly pipeline from M2 runs exactly once per user turn.

In an agent, the model's response is another input. It calls a tool. The tool returns a result. That result goes back into the context. The model thinks again, calls another tool. Another result. Another thought. By the time the agent finishes a task, the model has been called 5, 10, 30 times — and each call saw a longer context than the last, because every prior thought, action, and observation is in there.

The agent context equation

Context at step n = base context + Σ(thought + action + observation) for all prior steps. The assembly pipeline runs every step. The decision of what to keep and what to drop from the accumulated trace is now a live, per-step policy — not a per-request one.

What changes when you go agentic

  • Tool specs become first-class context. Not just in the system prompt as a capability list — structured, formal, always present, consuming meaningful tokens (often 3–8K for a real toolset).
  • History is no longer user↔assistant turns. It's a ReAct-style trace: thought → action → observation → thought → action → ... The roles expand from two to three or four.
  • Observations can be huge. A search tool returning 50 results, a code execution returning a stack trace, an API returning 20KB of JSON — each lands in context and has to be managed.
  • Errors in the trace aren't bugs, they're signal. A failed tool call tells the agent "don't do that again." Pruning it aggressively breaks learning-within-the-loop.
  • The U-curve (M0) is now dynamic. The "middle" of the context shifts every step. Something that was in the recency zone 3 steps ago is now deep in the middle dip.

Why most agent systems fail

The top three failure modes in production agent systems, in rough order of frequency:

  1. Context explosion. Step 1: 5K tokens. Step 20: 80K tokens and climbing. By the time the model is choosing action 21, most of its attention is on its own babble from five steps ago. Latency and cost follow the same curve.
  2. Looping on failed actions. The agent tried something, it failed, the error went into context — but then on the next step, the model forgot the error was related to the exact same action it's about to retry. Poor provenance on observations → infinite loops.
  3. Tool spec bloat crowds out everything else. Toolkit has 40 tools, each with a verbose spec, totaling 12K tokens, present on every step, pushing the actual work into the middle-dip zone.

All three are addressed by the same discipline: treat the agent loop as an assembly problem, not a chat problem. The context at every step is assembled — not just appended to. The right question is always: given where we are in the trace, what needs to be in the window for the next decision to be good?

02 / Tool specs
Tool specs — the new block 2, and how to write them
Tool descriptions are prompts that happen to be in JSON. The same clarity rules apply. They sit in block 2 of the canonical layout (cf. Track 1 M1) and define what the agent can even think to do.

Tool specs live in block 2 of the canonical layout (cf. M1 §2). They follow the system prompt and precede retrieved content. They are stable across a session — the same toolkit usually applies through the whole conversation — which makes them prime candidates for provider prompt caching (cf. M2 §6).

But: the spec isn't just a schema. It's a prompt. The model reads the tool description, the parameter descriptions, the examples — and decides whether this is the right tool for the situation, and what arguments to pass. A poorly-written spec silently degrades every action decision. A great spec is the difference between an agent that calls the right tool on the first try and one that cycles through wrong tools for three steps before landing.

Bad spec vs good spec

Bad — vague description, unnamed params
{ "name": "search", "description": "Search for things", "parameters": { "query": "string", "limit": "number" } }
The model has to guess: search what? Their docs? The web? Their private data? What's "limit"? Results? Characters? Time?
Good — scoped, typed, with guidance
{ "name": "search_knowledge_base", "description": "Search the internal knowledge base for articles, runbooks, and past resolved tickets. Returns up to 10 ranked results with titles, excerpts, and URLs. Use this first for support queries before falling back to `search_web`.", "parameters": { "query": { "type": "string", "description": "Natural-language search query. Works best with 4-10 keywords; full sentences are OK." }, "top_k": { "type": "integer", "description": "Max results to return, 1-10. Default 5.", "default": 5 } } }
Name is scoped. Description says what it does, what it returns, and when to prefer it. Parameters have semantic descriptions, not just types.

Six rules for writing tool specs

  1. Name it like a function that already exists. search_knowledge_base, not search. Namespace prevents collisions and tells the model what domain this operates in.
  2. Open the description with the action verb. "Search..." "Create..." "Cancel..." — the first token of the description is what the model pattern-matches against.
  3. State the return shape. "Returns up to 10 results with title, excerpt, url" — the model plans its next step based on what it expects to get back.
  4. Include routing guidance. "Use this first before falling back to X" — tell the model which tool to prefer in ambiguous cases. Without this, tool choice becomes a coin flip.
  5. Describe parameters semantically, not just structurally. Not "a string" — say what kind of string, what format, what's valid. For enums, list the values inline.
  6. Keep it under ~150 tokens per spec. If a tool needs 300 tokens to describe, either it's doing too much (split it) or the description is padded (tighten it).

Tool subsetting — don't ship the whole kitchen

If your system has 40 tools, the answer is almost never "put all 40 specs in every request." Tool specs consume tokens, and tokens consume attention. An agent with 40 tools in its context will have worse tool selection than one with the right 8 tools, because the right ones are buried.

Three subsetting strategies, in order of sophistication:

  • Fixed by template. The "customer support" template has tool set A; the "code review" template has tool set B. No runtime decision — comes from stage 1 of M2's pipeline. Works when user intents map cleanly to toolkits.
  • Capability filter. Exclude tools the user doesn't have permission for. Exclude tools the current session has already marked unavailable. Deterministic, safe, cheap.
  • Retrieval-based tool selection. Embed each tool spec, retrieve top-k most relevant to the user query. Only for systems with ~50+ tools where static templating doesn't cut it. Introduces its own failure modes — a misretrieved toolkit is worse than a broad-but-static one.
Default recommendation

Start with fixed-by-template. Most systems never need retrieval-based tool selection — they think they do because they have 40 tools, but a 40-tool agent is usually 3 agents pretending to be 1. Split by concern, use templates. Only retrieve tools when you genuinely have a flat toolkit of 50+ peers.

03 / Agent loop
The agent loop — watch context grow, step by step
ReAct is the canonical pattern. The model alternates Thought → Action → Observation until it produces a final answer. Each step appends to context. The widget below runs a real trace.

The dominant pattern for agent loops is ReAct (Reason + Act). The model produces a thought describing what it's going to do and why, an action (a tool call with arguments), and then the system executes the tool and provides the observation back. This cycle repeats until the model produces a "final answer" instead of another action.

Other patterns exist — Plan-then-Execute (produce a full plan upfront, then execute), Reflexion (add a self-critique step), tool-augmented generation — but they're all variations on the same loop shape. The context-engineering concerns carry across.

The prompt template that runs the loop

At each step, the full context assembled for the model looks like this. The trace is the variable part — everything above it is stable and cacheable:

<system>
  You are an investigation agent. Use the tools below to answer user requests.
  Think step by step. Call one tool at a time. Respond with `final_answer` when done.
</system>

<tools>
  search_knowledge_base, search_web, read_file, final_answer  # full specs
</tools>

<user_request>
  Why did deployment 847 fail last Tuesday?
</user_request>

<trace>
  Thought: I should check the deployment logs first.
  Action: search_knowledge_base(query="deployment 847 logs")
  Observation: Found 3 results. Top: runbook-deploy-failures.md mentions 847 failed at stage 4.
  Thought: Stage 4 is where database migrations run. Let me read the migration log.
  Action: read_file(path="/logs/847/stage4.log")
  Observation: [Error: migration 2024_10_15_add_index timed out after 300s]
  Thought: ...
</trace>

<next> Continue the trace. </next>

The key structural choice: the trace is one block of text, clearly delimited, with consistent Thought/Action/Observation prefixes. The model has been trained on this pattern; deviating just makes the loop less reliable.

Where the trace sits in canonical layout

In the canonical layout from M1, the agent trace lives in block 4–5 — after retrieved content but before the current-turn instruction. It is both history and task-scope content, because each step's observations ground the next step's decisions. This is why plain "chat history" treatment of it misses the point — the trace isn't a conversation log, it's the agent's working memory.

04 / Scratchpad
Scratchpad and agent state — structured vs free-form
The trace itself is unstructured text. But many agents need structured state too — a running plan, a set of facts gathered, a checklist of sub-goals. Where does that live?

A pure ReAct loop treats the trace as the only state. Every thought, every observation, every intermediate fact — all in the rolling text of the trace. This works for short tasks (≤10 steps). It falls apart on longer ones because the key information (the 2nd observation's critical fact) gets buried under 8 steps of trace text and lands in the middle-dip zone of the U-curve.

The fix is to introduce structured state alongside the trace — a scratchpad the agent can explicitly read from and write to. Concretely:

class AgentState(BaseModel):
    original_goal: str                        # never changes — pinned to top

    plan: list[str]                        # sub-steps the agent has committed to
    plan_progress: dict[int, str]          # step_idx -> "done" / "failed" / "pending"

    facts: list[Fact]                     # explicitly gathered key facts
    # Fact := {"claim": str, "source": str, "confidence": float}

    open_questions: list[str]              # known unknowns
    failed_attempts: list[FailedAttempt]   # "don't try X again because Y"

    step_count: int
    token_budget_remaining: int

How state is maintained

There are two workable patterns for keeping the state populated. Both have tradeoffs:

Pattern A
Agent writes state directly
Give the agent tools like add_fact, mark_plan_step_done, note_failed_attempt. The agent maintains its own scratchpad between thoughts.
✓ Agent has full control
✓ Explicit, inspectable
✗ More tools to learn
✗ Agent can forget to write
Pattern B
System extracts state post-hoc
After each step, run a small "state updater" LLM call that reads the latest thought/observation and updates the structured state. Agent doesn't know state exists explicitly.
✓ Agent prompt stays clean
✓ Consistent extraction
✗ Extra LLM call per step
✗ State can lag reality
Pattern C (hybrid)
Agent writes, system validates
Agent has explicit state-writing tools (Pattern A). System runs a validator after each step to catch drift (Pattern B). Best of both at higher complexity.
✓ Robust at scale
✗ Most moving parts

Where state sits in the context

The AgentState object is rendered into text and placed before the trace in block 4. It looks like:

<agent_state>
  GOAL: Diagnose why deployment 847 failed and propose a fix.

  PLAN:
    [done]    1. Find the failing stage
    [done]    2. Read the error from that stage's log
    [pending] 3. Identify root cause
    [pending] 4. Propose fix

  FACTS GATHERED:
    - Stage 4 (db migration) is where it failed [source: runbook]
    - Migration 2024_10_15_add_index timed out at 300s [source: stage4.log]

  FAILED ATTEMPTS:
    - (none so far)

  OPEN QUESTIONS:
    - Why did the migration take > 300s? Was the table unusually large?
</agent_state>

<trace>
  ...
</trace>

The model now has a compressed view of what's important, right near generation. It can scan the goal + plan + facts in ~150 tokens before it has to wade through the trace. This single move — pinning structured state near recency — has outsized impact on long-horizon agents.

The load-bearing observation

Agent state is compressed history. The trace has everything; the state has the part that matters for the next decision. Keeping both means you get the traceability of the full log for debugging, and the signal density of the summary for live reasoning.

05 / Observations
Observation integration — what tool results actually look like in context
A 20KB JSON blob from an API can't just be pasted into the trace. Observations need processing before they enter context.

Tool outputs in production are almost never the right shape for direct insertion into the trace. A database query returns rows. A web search returns HTML snippets. A file read returns raw text that might be 100KB. An API returns nested JSON with 80% metadata. If you drop these verbatim into the trace, you blow the budget in three steps.

The observation-processing pipeline

def integrate_observation(raw: Any, action: ToolCall, state: AgentState) -> str:
    # Step 1: Normalize to string
    normalized = normalize_for_llm(raw)   # JSON → compact, HTML → text, etc.

    # Step 2: Truncate if needed (by tool-specific policy)
    if len(normalized) > action.max_observation_tokens:
        normalized = truncate_smart(normalized, action.max_observation_tokens)

    # Step 3: Wrap with provenance
    wrapped = render_observation(
        tool=action.tool_name,
        args=action.args,
        result=normalized,
        success=True,
    )

    # Step 4: If observation implies facts, extract and add to state
    facts = extract_facts(normalized, context=state.original_goal)
    for f in facts:
        state.facts.append(f)

    return wrapped

Four decisions per observation type

For every tool in your toolkit, decide these four things once and codify them:

DecisionOptionsDefault
Max tokens 500 / 1500 / 4000 / unlimited 1500 for reads, 500 for lookups, 4000 for fetches
Truncation direction head / tail / middle / smart head (top-k results or start of doc)
Summarize on overflow? no / extractive / LLM-summary extractive for lists; LLM-summary for prose
Visible in trace? full / summary only / reference only summary + reference key to fetch full if needed

Observation rendering

Render observations with clear delimiters, provenance, and (if truncated) explicit markers:

<observation step="4" tool="search_knowledge_base" status="ok">
  Found 3 results (showing top 2, 1 omitted due to budget):

  [1] runbook-deploy-failures.md (score: 0.91)
      "...deployment 847 failed at stage 4 during the database migration..."

  [2] incident-postmortems/2024-10-15.md (score: 0.84)
      "...migration 2024_10_15_add_index ran against a 2.4B row table..."
</observation>

Error observations are load-bearing

When a tool call fails, the error goes into the observation. This is where agents learn not to repeat mistakes. Three rules:

  • Make errors loud. <observation status="error"> — distinct from a successful observation. The model's pattern-matching should fire on "error".
  • Include the actual reason, not just a code. "400 Bad Request" is useless. "Parameter `start_date` must be in ISO 8601 format; got '2024-10-15T10pm'" tells the agent what to fix.
  • Promote error into state. A recurring error ("called search with empty query 3 times") gets added to failed_attempts. The state-level summary is what prevents the loop from retrying the same mistake forever.
The looping trap

The #1 cause of infinite agent loops: a tool fails, the error goes into context, but the trace is 15 steps long and the error has drifted into the middle-dip zone. The model tries the same thing again. Fails again. Loops. Solution: errors get promoted to failed_attempts in structured state (§4), which sits near recency, which the model actually reads. Always.

06 / Growth
Controlling context growth across the loop
Unchecked, context grows linearly with steps. At some point it hits the effective ceiling. Before it does, you compress — the right way.

Agent loops have a signature pattern of context growth. Early steps are small (few hundred tokens added per step). As observations accumulate — especially large ones from retrieval or file reads — the per-step addition grows. Around step 10–15, many agents are doubling their context every few steps as observations compound. Without intervention, you hit the wall.

Four growth-control strategies

Strategy 1
Observation size caps
Per-tool max token limits on observations (from §5). Cheapest, most effective first-line defense. Cuts the common case of one giant observation swallowing the budget.
✓ Do this by default
Strategy 2
Rolling trace window
Keep the last N steps verbatim. Replace older steps with a compressed summary. Similar to M5's session memory, adapted for agent traces.
✓ Keeps recent context sharp
✗ Can lose important early observations
Strategy 3
State-over-trace
Keep the structured agent state (§4) verbatim. Compress the raw trace aggressively — keep only the last 3-5 steps. The state carries forward the essential information.
✓ Best for long-horizon agents
✗ Needs robust state extraction
Strategy 4
Sub-agent delegation
When a sub-task is self-contained, spawn a sub-agent with its own fresh context. Return just the answer back to the parent agent. Parent never sees the sub-agent's trace.
✓ Unbounded horizon possible
✗ Significant architectural complexity

The compression decision

Most production agents land on a combination: aggressive observation caps (Strategy 1) + state-over-trace (Strategy 3), with sub-agent delegation (Strategy 4) reserved for truly long-horizon tasks. A rough decision rule:

def apply_growth_controls(trace: Trace, state: AgentState, cap: int) -> Trace:
    # Always-on: observation size caps were applied at ingestion (§5)

    if trace.total_tokens() < cap * 0.6:
        return trace                                    # plenty of room, no action

    if trace.total_tokens() < cap * 0.85:
        # Approaching limit: compress older steps
        return compress_older_than(trace, keep_recent=5)

    # Over 85%: aggressive state-over-trace
    return render_state_and_last_n(state, trace, n=3)

When to stop the loop entirely

Some loops will not converge within budget. You need hard exits:

  • Step cap. Hard stop at N steps (20 is typical). Beyond that, force a final-answer attempt with whatever's been gathered.
  • Token cap. If context exceeds some ceiling (say 80% of effective), force a summarization-then-final-answer cycle.
  • Repetition detection. If the same action+args has been tried 3+ times with the same error, exit with a "could not complete" answer. Don't let the loop grind.
  • User-initiated halt. Long-running agents should be interruptible. Surface a "stop and report what you have" control.
The frame

An agent without step/token caps isn't an agent — it's a budget-burning machine. The caps aren't a safety net; they're part of the agent's definition. "You have at most 20 steps and 40K tokens to solve this" is a parameter of the task, just like the goal itself.

07 / Anti-patterns
Seven anti-patterns that tank agent systems
The specific failure modes that separate agents that work from agents that loop, burn tokens, and cost more than they return.
1 · Kitchen-sink toolkit

40 tools in the spec block because "maybe the agent will need them." Result: 12K tokens of tool specs on every step, pushing real content into the middle-dip zone, and worse tool selection than a focused 8-tool setup. Fix: split the agent into multiple focused agents, each with its own toolkit. Use templates to select toolkits by intent (§2).

2 · Raw observations dumped into trace

Tool returns 40KB of JSON; the whole thing lands in the trace verbatim. Three such observations blow the budget. Fix: per-tool observation policy — max tokens, truncation direction, summary strategy (§5). Never insert raw tool output into context.

3 · No structured state

Everything lives in the trace. By step 10, the initial goal has drifted into the middle-dip zone and the model is chasing tangents. Fix: introduce AgentState with pinned goal, running plan, key facts. Render near recency (§4). Short-horizon tasks can skip this; anything > 8 steps should not.

4 · Error observations without promotion to state

Tool fails. Error sits in the trace. Three steps later, the model retries the exact same failed call because the error isn't in the recency zone anymore. Fix: promote errors to failed_attempts in state (§5). State renders near recency. Failed attempts become sticky learnings.

5 · No step/token caps

Agent runs until it decides it's done. In production this means: runs until budget exhaustion or infinite loop. Fix: hard caps (step_cap=20, token_cap=80% of effective) as definition of the task, not afterthoughts. Force a final-answer cycle when caps approach.

6 · Unclear tool names and descriptions

Tools named search, get, update. Descriptions that say "does searching." The agent picks tools by guessing. Fix: scoped names (search_knowledge_base), action-first descriptions, stated return shape, routing guidance (§2).

7 · No provenance on observations

Observations appear as free-floating text in the trace. Which tool produced them? With what arguments? The agent can't verify its own work; the debugger can't reproduce a failure. Fix: every observation wrapped with <observation step="N" tool="..." status="..."> — provenance is a first-class requirement, not a nice-to-have.

Synthesis

Three disciplines fix most of the above: (1) focused toolkits with high-quality specs, (2) structured state rendered near recency, (3) observation policy that caps size and preserves provenance. If your agent codebase doesn't explicitly have all three, its quality at step 15 will be unrelated to its quality at step 2.

08 / Mastery
Mastery check
Six questions on the decisions that separate robust agent systems from ones that loop and burn tokens.
Q1. Your agent has 38 tools in its spec block, totaling 11K tokens. Users report "it sometimes picks weird tools for simple questions." Best first move:
Rewrite all tool descriptions to be longer and more explicit
Switch to a larger-context model so the specs don't crowd out other content
Split the agent into multiple focused agents or use templates to select tool subsets by intent — 8 tools per invocation, not 38
Add examples of correct tool usage in the system prompt
The problem is the kitchen-sink toolkit (anti-pattern 1). More verbose descriptions make the bloat worse. A larger window doesn't fix attention distribution — the U-curve still applies. Examples help but don't address the root cause. Subsetting by intent is the structural fix — focused toolkits measurably improve tool selection.
Q2. Your agent keeps retrying a tool call that failed 3 steps ago with the same arguments and the same error. The error is still in the trace. Root cause:
The error message isn't specific enough
The error drifted into the middle-dip zone of the U-curve; it needs promotion to structured state (failed_attempts) which renders near recency
The model is too small
You need to add a system prompt rule "never retry failed tools"
The information is in the context but not where the model is attending — this is the classic middle-dip failure. Structured state rendered near recency (§4, §5) keeps the "don't do this again" signal in the attention zone. Error message quality helps prevent the initial failure but doesn't prevent repetition. System-prompt rules against retries are over-broad and will suppress legitimate retries. Larger models mitigate but don't eliminate this.
Q3. A search tool typically returns 30-50 results at 400 tokens each (12-20K tokens). Best handling policy:
Pass the full result — models are getting better at long context
Cap to top-k by relevance (e.g., top 5), include ranks and scores, keep tool result under ~1500 tokens; let the agent call again with refined query if needed
Pass everything but wrap in "ignore if not relevant" language
Split into 5 separate observations across steps
Observation size caps with semantic truncation (top-k by rank) is the §5 default. Full-pass eats budget and triggers attention dilution from M0. "Ignore if not relevant" language doesn't make the model ignore tokens — they still consume attention. Splitting across steps multiplies the growth problem.
Q4. You're designing a long-horizon agent expected to run 20-30 steps. Most important architectural decision:
Pick a model with the largest context window available
Introduce structured state (goal, plan, facts, failed_attempts) rendered near recency, with aggressive trace compression behind it — so context stays ~constant-ish as steps grow
Add more examples in the system prompt
Use ReAct with the default 1-tool-per-step limit
State-over-trace (§4, §6 Strategy 3) is the one intervention that turns linear growth into bounded growth. Larger windows buy time but the U-curve still applies — attention to content 15 steps back is poor regardless of window size. Examples help one-shot tasks, not long horizons. ReAct default is fine but orthogonal to the growth problem.
Q5. Which of the following is the best structural home for the agent's "original goal"?
In the system prompt (block 1)
As the first user message, relying on history to carry it forward
In the structured agent state rendered near recency — pinned so it stays in the attention zone even as the trace grows
In every thought, the model re-stating the goal before acting
The goal is specific to this run and needs to stay in attention as the trace grows. System prompt is for stable rules, not per-run goals. Putting it only as the first user message lets it drift into the middle-dip zone by step 10. State-rendered-near-recency is exactly designed for pinned per-run context. Having the model restate the goal every thought works but wastes tokens the agent could use for actual reasoning.
Q6. Your agent's p99 step latency has crept from 800ms to 3.4s over two weeks. No code changes; traffic pattern stable. Most likely cause:
The model provider got slower
Context growth — traces are running longer on average, feeding more tokens into each model call; observation policy or state compression isn't kicking in
Tool specs got verbose
Cache hit rate dropped
Agent latency is dominated by how many tokens each model call processes. Creeping context without a corresponding traffic pattern change means the compression strategy isn't bounding growth as task complexity drifted upward. Check token counts per step over time — this is the telemetry that surfaces drift. Provider latency would affect all models, not just this agent. Tool spec bloat would be sudden (on deploy), not gradual. Cache hit rate drops usually show up as cost spikes, not latency drift.
