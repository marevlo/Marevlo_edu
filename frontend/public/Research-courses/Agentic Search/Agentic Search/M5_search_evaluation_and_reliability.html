The agent works.
How would you know?

Most agentic search systems ship without evals, get monitored by vibes, and degrade silently when models update or indices drift. The teams that ship reliably do the opposite — they instrument every layer, build eval sets that catch real failures, set circuit breakers that bound damage, and treat regressions as launch-blocking. This module is the engineering of trust: how to know your search agent is working, and how to keep knowing.

01 / Foundation
Why agentic search needs different evals
Classical IR metrics break down for agents. Build the right measurement stack from day one.

Classical retrieval evaluation has a clean shape: you have a query, a labeled set of relevant documents, and you measure precision, recall, NDCG. The metric tells you whether the retriever found the right things. This breaks down for agentic search in three ways.

  • The agent's queries are not the user's query. An agent doesn't search for what the user typed — it decomposes, rewrites, and refines. Measuring relevance of the original query against documents tells you nothing about whether the agent's actual searches worked.
  • Right answer ≠ right retrieval. An agent can retrieve correct documents and produce a wrong synthesis. Or retrieve weak documents and produce a correct answer. Document-level metrics miss the actual quality signal: did the user get the right answer?
  • Failures are multi-layer. An agent failure can come from decomposition, tool selection, query rewriting, retrieval, evaluation, synthesis, or stopping logic. Single metrics aggregate all of this into one number that doesn't tell you what to fix.

The mature pattern is to evaluate per layer, not just end-to-end. End-to-end accuracy tells you the system is broken; layer-level metrics tell you where it's broken. Both matter, and they require different instrumentation.

Mental model

Treat the agent like a multi-stage pipeline (M0). Each stage gets its own evaluation: decomposition quality, tool selection accuracy, retrieval relevance, synthesis groundedness, citation faithfulness. End-to-end accuracy is the headline metric; layer metrics are the diagnostic ones. You need both.

The four evaluation horizons

Production systems run evaluation at four different cadences, each with a different purpose. Conflating them is one of the most common mistakes.

  1. Pre-commit (offline, fast). A small fixed eval set (~50–200 cases) that runs in CI. Catches regressions before they merge. Should complete in <5 minutes.
  2. Pre-deploy (offline, comprehensive). The full eval set (~500–5000 cases) run before each model or prompt deployment. Catches subtler regressions. Can take an hour.
  3. Online sampling (production, continuous). Sample 1–5% of production traffic, run through eval pipelines, alert on metric drift. The earliest warning system.
  4. Periodic deep audit (offline, manual). Quarterly or per-incident, an engineer reviews 50–200 traces by hand. Surfaces failure modes that automated evals miss.
02 / Metrics
The metric stack — what to actually measure
Nine metrics cover virtually every production agentic search system. Track them per-layer and as composites.

The metric stack below is the minimum viable set for a production agent. Every team that ships reliable systems tracks something close to this. Below the table, an interactive dashboard shows what these look like in a healthy versus degrading agent.

End-to-end metrics

  • Answer correctness. Is the final answer right? Measured via labeled eval set + LLM-as-judge for open-ended cases. The headline metric. Track p50 and p10 (worst-decile correctness matters more than average).
  • Answer groundedness. Are claims supported by retrieved evidence? Measured by checking each claim against the cited source. Catches hallucinations even when the answer is incidentally correct.
  • Citation faithfulness. Do citations actually support what they're cited for? Most agents cite plausibly but inaccurately — this metric catches it.

Per-layer metrics

  • Tool selection accuracy. When the agent picks a tool, is it the right one? Measured against a labeled set of (query, correct_tool) pairs. The single most diagnostic metric for routing failures.
  • Retrieval recall@k. For each subquery, did the right document appear in the top-k results? Catches index quality and query-rewrite quality.
  • Decomposition shape match. Did the planner pick the right shape (parallel/sequential/etc per M2)? Catches over-decomposition and under-decomposition.
  • Sufficiency precision. When the agent says "I have enough evidence," does it actually? Measured by stopping the trace at sufficiency declaration and checking. Catches premature termination.

Operational metrics

  • Cost per resolved query. Total $ / queries answered correctly. The unit-economics metric. Almost always different from cost-per-call.
  • Tail latency p95/p99. Mean latency lies for agents; tails are where users churn. Track per-query-shape, not aggregated.

03 / Eval Sets
Building an eval set that catches real failures
A bad eval set tells you the system works while users churn. A good one is the most valuable artifact your team will build.

The eval set is the spec for what your agent should do. Most teams start with 20–30 cases generated by the engineer who built the system — and discover six months later that the eval set passes while production fails. The reason is selection bias: the cases the engineer wrote are the ones the engineer was already designing for. The cases that catch real bugs are the ones the engineer didn't think of.

The five sources of eval cases

A robust eval set is sourced from multiple channels, in roughly these proportions:

  • Production samples (40%). Real user queries, with a human-labeled correct answer. The most representative cases. Sample across query shapes, not just by frequency.
  • Failure replays (25%). Every production failure that gets debugged becomes a permanent eval case. Prevents the same regression twice.
  • Adversarial / edge cases (15%). Hand-crafted to stress specific layers — empty results, contradicting sources, ambiguous intent, unusually long queries.
  • Synthetic generation (10%). LLM-generated variations of seed queries to expand coverage cheaply. Always reviewed by a human before adding.
  • Domain-expert authored (10%). The hardest cases that only a subject-matter expert would think to write. Highest-value-per-case.

Common mistake

Don't measure "did the agent's text contain the right answer?" using substring matching. Agents produce paraphrased, hedged, or restructured outputs that fail substring checks despite being correct. Use LLM-as-judge for answer correctness on anything that isn't a single-token exact-match question. The setup is in §4.

04 / LLM-as-Judge
LLM-as-Judge — calibration is the whole game
A judge that's lenient inflates your numbers. A judge that's strict floods you with false alarms. Calibrate carefully.

LLM-as-judge is the workhorse of modern eval. You give a model the question, the agent's answer, and the expected answer, and ask it to score correctness. It scales — you can run thousands of cases in minutes — and handles paraphrase, hedging, and restructuring that substring matching breaks on. But naive LLM-judge implementations have a calibration problem: they tend to be too generous, scoring partially-correct answers as correct.

The four anchors of a good judge

  1. Reference answer included. Don't ask the judge to evaluate correctness from scratch — give it the gold answer to check against. Closed-form judging is far more reliable than open-form.
  2. Decomposed criteria. Instead of "is this answer correct (yes/no)?", ask "is each of these facts present (one boolean per fact)?" Catches partial-credit failures.
  3. Bounded output (rubric or enum). The judge outputs one of a fixed set of labels — never a freeform score. {"correct", "partially_correct", "incorrect", "not_attempted"} beats "score 1-10" by a wide margin.
  4. Pairwise comparison for ranking. When comparing two systems, ask the judge "which is better?" not "score each." Pairwise judgments are more reliable than absolute scores.
Production LLM-as-judge prompt
Code
JUDGE_SYSTEM = """You are evaluating an agent's answer against a reference.

You will be given:
- The original question
- The reference answer (assumed correct)
- A list of REQUIRED FACTS that the answer must contain
- A list of FORBIDDEN CLAIMS that must NOT appear
- The agent's answer

For each required fact, output one of:
- "present": the fact is clearly stated
- "partial": fact is alluded to but lacks specifics
- "missing": fact is absent or wrong

For each forbidden claim, output:
- "absent": claim does not appear (good)
- "present": claim appears (bad — agent hallucinated)

Then assign one overall verdict:
- "correct": all required facts present, no forbidden claims
- "partially_correct": some required facts missing or partial, no hallucination
- "incorrect": forbidden claim present OR most required facts missing
- "not_attempted": agent did not attempt to answer

Output JSON only."""

class JudgeVerdict(BaseModel):
    fact_results: dict[str, Literal["present", "partial", "missing"]]
    forbidden_results: dict[str, Literal["absent", "present"]]
    verdict: Literal["correct", "partially_correct",
                     "incorrect", "not_attempted"]
    rationale: str

Calibrate the judge against humans

Periodically (~quarterly), have a human label a sample of 50–100 cases independently. Compare to judge labels. The agreement rate is your judge calibration. If it falls below ~85%, the judge prompt needs revision — usually because the rubric is ambiguous or the required facts are too vague.

  • Judge model choice. Use a frontier model for judging, even if your agent uses a smaller one. The judge is run rarely (eval cadence) and accuracy matters more than cost.
  • Position bias. When pairwise-comparing, alternate which answer is presented first. LLMs have measurable position bias.
  • Self-evaluation trap. Don't use the same model for both agent and judge. The judge will systematically rate the agent's answers higher than they deserve.
The disagreement is the signal

When the judge and a human disagree, the disagreement is more interesting than the verdict. Either the rubric is wrong (rewrite it), the gold answer is wrong (fix it), or the case is genuinely ambiguous (mark it flaky). All three are eval-set improvements you couldn't have spotted otherwise.

05 / Observability
Production observability — instrument every layer
If you can't replay a failed query, you can't debug it. Build the trace pipeline first.

Observability for agentic systems is fundamentally different from microservice observability. You don't just need request/response — you need the full cognitive trace of the agent: every plan, every tool call, every observation, every decision to refine or stop. Without this, debugging a failed query is impossible. With it, root-causing takes minutes.

The trace schema

Every production query should produce a trace with this minimum content. Store it for at least 30 days; longer if you have the budget.

class QueryTrace(BaseModel):
    trace_id: str
    user_id: str
    query: str
    started_at: datetime
    completed_at: Optional[datetime]

    # Pipeline stages — each as its own object
    decomposition: Optional[DecompositionTrace]
    tool_calls: list[ToolCallTrace]      # every call, in order
    plan_states: list[PlanState]        # snapshot per turn
    sufficiency_checks: list[SufficiencyCheck]
    final_answer: Optional[str]

    # Outcomes
    stopping_reason: Literal["success", "max_iterations", "budget",
                             "diminishing_returns", "error"]
    total_cost_usd: float
    total_tokens: int
    total_latency_ms: int
    error: Optional[str]

class ToolCallTrace(BaseModel):
    iteration: int
    tool_name: str
    input: dict
    output: dict
    latency_ms: int
    cost_usd: float
    cache_hit: bool
    error: Optional[str]

The four observability pillars

  1. Traces. The above per-query record. Stored in a queryable database (Postgres, ClickHouse, or specialized like Langfuse/Honeycomb).
  2. Metrics. Aggregations from traces — the metric stack from §2 emitted to Prometheus / Datadog. Calculated on a rolling window.
  3. Logs. Structured (JSON) logs at every decision point. Lower-cardinality than traces, higher-cardinality than metrics. The "what was the agent thinking?" channel.
  4. Sampling. Don't store full traces for every query at scale — that's expensive. Sample 100% of failures, 100% of high-cost queries (>$0.10), 5% of normal queries.

The replay capability

The single most valuable instrumentation: be able to replay any past query through the current system version. When debugging, you take the trace_id, pull the stored query and inputs, and re-run with current model/prompts. This converts "what changed?" from a forensic guessing game into a deterministic experiment. 

async def replay(trace_id: str, override_model: Optional[str] = None):
    # Pull the original query and any user-context
    original = await traces.fetch(trace_id)

    # Optionally override model / prompts to test changes
    config = current_config()
    if override_model: config.model = override_model

    # Run through the current pipeline
    new_trace = await run_agent(original.query, config=config)

    # Diff old vs new — surface what changed
    return trace_diff(original, new_trace)
Operational pattern

Build the replay endpoint into your CI. When you change a prompt, replay the last 100 production failures against the new prompt. If any previously-correct query now fails, you've introduced a regression. This catches silent quality drops that aggregate metrics miss.

06 / Circuit Breakers
Circuit breakers — bounding damage when things go wrong
Move the sliders to model your own thresholds. The state machine adapts in real time.

Circuit breakers are the difference between "one bad query degrades the user's experience" and "one bad query takes down your unit economics." They are the runtime enforcement of the budgets we discussed conceptually in M2 §5 — but extended to handle systemic failures, not just per-query overruns.

The three circuit breaker layers

  1. Per-query breakers. Stop a single query when it exceeds its budget — max iterations, max tool calls, max cost. Returns a partial answer or an apology.
  2. Per-user breakers. Throttle a user who is consuming abnormally many resources. Often a sign of abuse or a stuck client; sometimes a power user.
  3. Per-tool breakers. When a downstream tool (web search API, vector DB, SQL DB) is failing or slow, stop calling it for a window. Classic circuit-breaker pattern from microservices.

Three states, three transitions

  • CLOSED (normal). Calls flow through. If failure rate or latency exceeds thresholds within the rolling window, transition to OPEN.
  • OPEN (broken). Calls fail fast without hitting the downstream tool. Agent receives a structured error and can route to a fallback or apologize. After a timeout (typically 30–120s), transition to HALF_OPEN.
  • HALF_OPEN (probing). A small number of probe calls flow through. If they succeed, transition back to CLOSED. If they fail, back to OPEN with a longer timeout.

Fallback behaviors

When a circuit is OPEN, what should the agent do? Three patterns, in order of preference:

  • Route to alternate tool. If web_search is down, can the agent use web_search_fallback? Best when you've architected redundancy.
  • Degrade gracefully. "I can't search the web right now, but here's what I know from training." Honest about the limitation.
  • Apologize and stop. "I need real-time data to answer this and the search service is unavailable. Please try again in a few minutes." Worst UX but honest.
The silent fallback trap

The worst possible fallback is to silently return whatever the agent can produce without telling the user. The user gets a confident-looking answer that's based on stale or incomplete data. Always surface the degradation: "results may be incomplete because [tool] was unavailable."

07 / Drift
Detecting drift — the slow-motion failures
Loud failures get fixed in hours. Silent drift erodes quality for months. Build the detection that catches both.

Drift is what happens when the world changes around your agent and your eval set doesn't notice. The model gets updated; the search index gets re-embedded; the corpus accumulates new content; user query distributions shift. Each is small. The aggregate is a system that worked great in Q1 and ships wrong answers in Q3.

Seven drift patterns to monitor

01 · Model drift

Provider updates the model behind a stable name (e.g., gpt-4-turbo's quiet revisions). Behavior changes; nothing in your code did.

SymptomEval accuracy drops on a fixed set without any deploy
FixPin to specific model versions when available; run eval on a daily schedule, not just on deploy
02 · Index drift

Your vector index is re-embedded with a new model; relevance scores shift; previously top-1 results drop to top-10.

SymptomRecall@k drops while index stats look normal
FixVersion your embedding model; A/B test re-indexes against a fixed eval set before swapping
03 · Corpus drift

Source corpus grows or changes; queries that used to have one good answer now have ten conflicting ones.

SymptomCitation faithfulness drops; sufficiency checks pass on contradictory evidence
FixTrack corpus size and contradiction rate as first-class metrics; refresh evals quarterly with new gold answers
04 · Query distribution drift

Users start asking questions in shapes your agent wasn't designed for. Aggregate metrics may stay flat while new-shape accuracy is poor.

SymptomUser-reported failures concentrate in a query category not in your eval set
FixCluster production queries monthly; surface emergent shapes; add to eval set
05 · Tool drift

Downstream APIs change response shape, snippet style, or ranking. The tool still works; your agent still calls it; the outputs are subtly different.

SymptomTool-result schema validation warnings; agent reasoning becomes noisier
FixSchema-validate every tool response; alert on validation warnings; pin SDK versions
06 · Prompt drift

Prompts get edited iteratively without re-running full evals. Each edit looks fine; aggregate is a regression.

SymptomQuality degraded over a quarter without a clear deploy to point to
FixVersion-control prompts; require eval-pass for every prompt change; not just code changes
07 · Cache drift

Cached results from web/structured sources go stale; cache hits return content that no longer reflects reality.

SymptomRecency-sensitive queries suddenly wrong; cache invalidation rates abnormally low
FixTrack cache age distribution; alert on stale-rate exceeding threshold (M3 §5)

The daily eval cadence

Most teams run evals only at deploy. The teams that catch drift run them daily, on a fixed eval set, against the production system. The setup is small but the payoff is large: you find out about model drift the day it happens, not the week your CEO escalates.

08 / Mastery
Mastery check
Six questions on the patterns that distinguish reliable agentic search systems from fragile ones.
Q1. Your end-to-end accuracy metric is steady at 78%, but users are complaining about wrong answers. The most likely root cause:
The model is broken — switch providers
Aggregate metrics hide failure modes — slice by query shape to find the broken category
Users are wrong about which queries are failing
78% is good enough; users are unrealistic
A flat 78% can be 95% on lookups and 30% on aggregations — averaged together, the broken category is invisible. Always slice metrics by query shape (lookup, comparison, multi-hop, investigation, aggregation). User complaints typically concentrate in one shape; aggregate metrics drown that signal in the well-performing shapes.
Q2. You're building an LLM-as-judge for answer correctness. Which choice gives the MOST reliable scoring?
Ask the judge to score 1-10 and average across runs
Provide a reference answer and a list of required facts; judge each fact as present/partial/missing
Use the same model as the agent for consistency
Have the judge generate its own correct answer first, then compare
Closed-form judging with a reference answer and decomposed criteria is far more reliable than open-form scoring. 1-10 scales are noisy and hard to calibrate. Same-model judging introduces self-evaluation bias. Having the judge generate its own answer compounds errors. The pattern is: anchor on a known-good answer, decompose into facts, judge each fact discretely.
Q3. Your team can store traces for 100% of queries (expensive) or sample. Best sampling strategy:
Sample 1% uniformly at random
100% of failures, 100% of high-cost queries (>$0.10), 5% of normal queries
Sample only weekday traffic
Store everything but compress aggressively
Failures and expensive queries are the ones you'll need to debug; you can't predict in advance which they'll be, so capture all of them. Random sampling of normal queries gives you distributional visibility. The 5% baseline lets you compute aggregate metrics. Uniform 1% loses too many failures; weekday-only loses weekend failure modes.
Q4. Your web search tool starts failing at 40% rate. The agent keeps calling it. The structural fix is:
Add retry logic with exponential backoff
Add a per-tool circuit breaker that trips OPEN above a failure threshold and routes to a fallback or degrades gracefully
Increase the agent's max tool calls so it can retry more
Switch web search providers permanently
Retries amplify the load on a failing service and don't help when the issue is systemic. Circuit breakers stop calling the broken tool, route to fallbacks where available, and surface the degradation to users honestly. Increasing tool-call budgets makes the problem worse. Provider-switching is a longer-term fix; the breaker handles the immediate incident.
Q5. Eval accuracy on your fixed set has dropped from 84% to 76% over a month, with no deploys to your code. Most likely cause:
The eval set is stale
Model drift — the provider updated the model behind your stable model name
Random variance — re-run and it'll be fine
Users have started gaming the eval
No-deploy quality drops on a fixed eval set are the textbook signal of model drift. Providers regularly update models behind stable names; your inputs and prompts didn't change but the model behind them did. The fix is to pin specific model versions when offered, run eval daily (not just on deploy) to catch drift early, and have a model-comparison harness ready for when you do need to upgrade.
Q6. Your circuit breaker is OPEN because web search is down. The agent gets a query that needs the web. The MOST appropriate behavior:
Silently produce the best answer it can from training knowledge
Block all queries until the breaker closes
Surface the degradation to the user — answer with a clear "results may be incomplete because web search was unavailable" caveat
Retry continuously hoping the breaker closes
Silent fallback is the worst pattern — users get a confident answer based on incomplete data and trust the system less when they discover. Blocking all queries is too aggressive; many queries don't need web search. Continuous retry defeats the breaker's purpose. Honest degradation maintains trust: the user gets the best available answer plus the context they need to evaluate it.
