Track 01 · Agentic Search Module 00 ~2h 20m Foundational

Search isn't a function call.
It's a reasoning loop.

For thirty years, search meant query → results. One shot. One ranking. The user did the rest. Agentic search inverts this — the model becomes the searcher, planning queries, evaluating results, refining its strategy, and stopping only when it has enough to answer. This module unpacks why that shift changes everything: cost structure, latency, evaluation, failure modes, and the architectures that make it work in production.

01 / Foundations
The shift from lookup to reasoning
A precise model of what changes when search becomes an agent capability — and why most teams still build it wrong.

Classical search treats the query as a contract. You hand it a string, it hands you back a ranked list, and you trust the ranking. The intelligence lives in the index — TF-IDF, BM25, dense embeddings, learned rankers. The user, or the application, is responsible for reformulation, follow-up queries, and synthesis. The retriever does not think.

Agentic search collapses this division. The model is no longer a consumer of search results — it is the agent doing the searching. It generates queries, reads results, decides whether they're sufficient, generates new queries against new tools, and assembles an answer from evidence it gathered itself. Search becomes a multi-turn reasoning process with the index as one of many tools.

The core distinction

Static search is a function: search(q) → results. Agentic search is a loop: while not done: plan → search → observe → refine. Everything in this track follows from that single change.

Why this matters in production

Teams that miss this distinction ship "RAG with extra steps" and wonder why their agent burns $40 per query for marginal accuracy gains. They've added the loop without earning it. Agentic search is justified when at least one of three conditions holds:

  • Query decomposition is required — the question naturally breaks into subquestions that need different evidence (comparisons, multi-hop facts, time-bounded synthesis).
  • The right tool is unknown a priori — sometimes you need vector search, sometimes SQL, sometimes web. The agent has to choose.
  • Result quality is unpredictable — the agent must inspect what came back and decide whether to refine or proceed.

If none of these hold, you're paying for cognition you don't need. Static retrieval is faster, cheaper, and easier to evaluate. The art of agentic search starts with knowing when not to use it.

02 / Mechanics
Anatomy of a search tool call
Strip away the framework abstractions. Here is the raw exchange between an LLM and a search tool, byte by byte.

Every "agent uses a tool" interaction is, mechanically, three things: a tool schema the model sees in its context, a tool call the model emits as structured output, and a tool result injected back into the context for the next turn. There is no magic. Frameworks like LangGraph, OpenAI Assistants, and Anthropic's tool use API are wrappers around this same primitive.

The tool-call exchange — explore each layer
Code
1. Schema (what model sees)
2. Tool call (what model emits)
3. Execution (your code runs)
4. Result injection (back to model)

The schema is part of the model's system context. Tool descriptions are prompts — bad descriptions cause bad tool selection. This is one of the most underrated leverage points in agent design.

# What the model actually sees in context
tools = [
  {
    "name": "web_search",
    "description": """Search the web for recent information.
      Use this for: news, prices, current events, anything
      published after January 2025. Do NOT use for: code
      reference, math, or stable factual knowledge.""",
    "input_schema": {
      "type": "object",
      "properties": {
        "query": {"type": "string", "description": "3-8 word query"},
        "recency_days": {"type": "integer", "default": 365}
      },
      "required": ["query"]
    }
  }
]

The model emits a structured tool call. This is just text — the API parses it into JSON for you. The model has decided, based on the user query and the tool descriptions, that this tool is the right next step.

# Model output (parsed)
{
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "I need recent benchmark numbers. Let me search."
    },
    {
      "type": "tool_use",
      "id": "toolu_01ABC...",
      "name": "web_search",
      "input": {
        "query": "HNSW recall benchmark 2024",
        "recency_days": 90
      }
    }
  ]
}

Your application receives the tool call and runs the actual function. The model is paused — it has no idea what your code is doing, only that it asked for results and is waiting.

async def execute_tool(tool_call):
    if tool_call.name == "web_search":
        # Real call to Tavily / Brave / Exa
        results = await tavily.search(
            query=tool_call.input["query"],
            max_results=5,
            days=tool_call.input.get("recency_days", 365)
        )
        return {
            "tool_use_id": tool_call.id,
            "content": format_results(results)
        }

The result is appended to the conversation as a tool_result message. The model now sees: original query, its own tool call, and the result. It will either call another tool or produce a final answer.

# What goes back into context for the next turn
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01ABC...",
      "content": """[1] Malkov 2024 — HNSW++ achieves 99.2% recall
        at 0.8ms p99 on SIFT1M, +12% over baseline...
        [2] Pinecone benchmark — graph indices now lead
        IVF by 3.4x latency at equal recall..."""
    }
  ]
}
# → Model now decides: more search, or answer?

The four properties of a good search tool

Across hundreds of production agent systems, search tools that work in agent loops share four properties. We will return to each in M1, but they're worth naming now.

  • Narrow scope. One tool, one job. search_codebase beats universal_search. The model picks correctly more often when scopes are clear.
  • Self-describing inputs. Parameter descriptions are part of the prompt. "query" with no description forces the model to guess format.
  • Predictable output schema. Always return the same shape. Optional fields with stable keys. The model learns the schema implicitly across calls.
  • Failure as a value. Empty results are not exceptions — they are signal. Return them clearly so the model can refine, not crash.
03 / Planning
Search plan generation
Before the first tool fires, the agent forms a plan. Watch one materialize in real time.

Planning is the cognitive work that happens before the first tool_use token is emitted. It is where the agent decides: what subquestions matter, what tools could answer them, what order to attempt them in, and what would constitute "enough." Implicit planning happens in the model's hidden state. Explicit planning is materialized as a structured artifact — a list, a graph, a JSON tree — that subsequent steps can reference and revise.

What lives inside a good plan

A production-quality search plan is not just "a list of queries." It is a structured object the agent can update mid-execution. The minimum useful schema:

# Anatomy of a search plan (revisable)
{
  "goal": "Compare HNSW and IVF for production",
  "subgoals": [
    {
      "id": "sg1",
      "question": "What are HNSW's recall/latency characteristics at 10M scale?",
      "candidate_tools": ["arxiv_search", "benchmark_db"],
      "sufficient_when": "≥2 sources with quantitative numbers",
      "status": "pending"
    },
    { "id": "sg2", "question": "Same for IVF", ... },
    { "id": "sg3", "question": "Operational tradeoffs: rebuild cost, memory", ... }
  ],
  "synthesis": "Side-by-side table once all subgoals complete",
  "budget": {"max_tool_calls": 8, "max_cost_usd": 0.40}
}
Practitioner insight

Plans should be revisable artifacts, not one-shot blueprints. The first plan is almost always wrong about what evidence will be available. The agent's job is to update the plan as observations come in — mark subgoals as satisfied, add new ones discovered during search, abandon dead ends. Static plans give you brittle agents.

04 / Architectures
ReAct vs Plan-and-Execute
The two dominant control flows for search agents — when each wins, and the hybrid that beats both in production.

Two architectures dominate the agentic search literature. ReAct interleaves reasoning and action in a tight loop: think, act, observe, think, act, observe. Plan-and-Execute separates them: produce a complete plan upfront, then execute steps, replanning only on failure. Most production systems are hybrids — but you cannot design a hybrid without understanding the pure forms first.

Visualization

Thought
I need quantitative latency numbers for both Pinecone and Weaviate at 10M scale. I should start with the most authoritative source — vendor benchmarks or independent comparisons.
Action
web_search(query="Pinecone vs Weaviate benchmark 10M p99 latency", recency_days=180)
Observation
Found ANN-Benchmarks 2024 results and a Pinecone-published benchmark. Independent source has both. Pinecone-published only covers their own product.
Thought
The independent benchmark is the right anchor. Let me fetch the full ANN-Benchmarks page to get exact numbers, since the search snippet is too brief.
Action
web_fetch(url="https://ann-benchmarks.com/results-2024")
Observation
Pinecone p99: 14ms at 95% recall. Weaviate p99: 22ms at 95% recall. Same hardware, same dataset (deep1M scaled to 10M).
Answer
At 10M vectors and 95% recall on commodity hardware, Pinecone's p99 latency is ~14ms vs Weaviate's ~22ms — a 1.6× difference, primarily because Pinecone uses a custom segmented index while Weaviate uses standard HNSW.

Plan-and-Execute architecture

Plan-and-Execute trades adaptivity for predictability. The agent produces a full plan upfront — a directed acyclic graph of subtasks — and then a separate executor (often a smaller, cheaper model or even deterministic code) runs each step. The planner is only re-invoked if a step fails or returns an unexpected result.

The hybrid that wins in production

Most mature systems: plan upfront with a big model, execute with ReAct loops scoped to each subgoal, and replan when a subgoal fails or returns evidence that invalidates other subgoals. You get plan inspectability, parallelism where possible, and adaptivity where needed. We build this exact pattern in M6.

05 / Decision
When to use agentic search vs RAG vs direct retrieval
An interactive decision tree built from production failure patterns. Answer the questions for your use case.

The question "should I use agentic search?" is the wrong frame. The right question is: what does my query distribution look like, and what is the cheapest architecture that handles it? Most teams overshoot — they reach for agentic loops when a single retrieval would do, or they ship pure RAG when 30% of their queries genuinely need decomposition.

Decide: which retrieval pattern fits your use case?
Decision tree
1. Does your query require evidence from multiple distinct sources or steps to answer correctly?
No — single document or chunk usually suffices
Yes — multi-hop or comparative
2. Is the right tool / index obvious before the query, or does it depend on the query content?
Obvious — same index every time
Depends — could be SQL, vector, web, code, etc.
3. Is per-query latency under 1 second a hard requirement?
Yes — user-facing chat / autocomplete
No — async, background, or research workflow
4. Is retrieval quality unpredictable enough that the model needs to inspect results before continuing?
No — same query → similar quality each time
Yes — quality varies wildly, needs adaptive refinement
Answer the four questions above to see the recommended pattern.

06 / Pitfalls
Failure modes you will absolutely encounter
Every team building agentic search hits the same eight failures. Recognize them before they cost you a quarter.

These are not hypothetical. Every one of these has shipped to production at a serious company in the last eighteen months. They share a pattern: the agent's reasoning looks reasonable in the trace, but the result is wrong, expensive, or never arrives. The cure starts with naming them.

01 · Infinite Refinement
Agent keeps refining queries because results "aren't quite right." No stopping criterion. Burns budget until killed by max-iteration cap.
02 · Hallucinated Queries
Model emits queries with fabricated specifics — invented paper titles, fake author names, made-up product versions. Search returns nothing or garbage.
03 · Tool Drift
Model uses the wrong tool because descriptions overlap. SQL agent calls vector search for a structured query. No error — just wrong answer.
04 · Result Overload
Tool returns 50 results, all dumped into context. Model loses track of which evidence supports which claim. Citations become unreliable.
05 · Premature Termination
Agent stops after one search returns plausible-looking results. Misses contradictions or gaps that a second search would have caught.
06 · Cost Blowup
Single complex query triggers 40 tool calls and $8 in API costs. No per-query budget. One bad query takes down the unit economics.
07 · Stale-Cache Confidence
Cached results returned for "what's new in X" queries. Agent confidently reports week-old news as current. Cache key didn't include freshness intent.
08 · Lost Provenance
By the time the agent synthesizes, it can no longer say which result supported which claim. Audit and debugging become impossible.
M5 preview

Each of these has a structural fix — circuit breakers for #1 and #6, query validators for #2, tool routing for #3, result re-ranking and summarization for #4, sufficiency checks for #5, freshness-aware cache keys for #7, and provenance threading through every tool result for #8. We build all eight defensive patterns in M5.

The five levers that actually move the bill

  1. Smaller model on inner loops. Use a Sonnet/Haiku split — Sonnet plans, Haiku executes individual tool turns. Often 60–70% cost reduction with negligible quality loss.
  2. Aggressive caching with freshness keys. If two queries semantically equivalent within the freshness window, serve from cache. Especially for web search results.
  3. Per-query budget enforcement. Hard cap on tool calls and tokens. The agent must learn to spend its budget — a soft suggestion is ignored.
  4. Result truncation before context insertion. Never inject raw search results. Summarize or extract the relevant span first. Cuts input tokens by 5–10×.
  5. Routing. 80% of queries don't need agentic search. Route them to direct retrieval. Reserve the loop for the 20% that genuinely benefit.
08 / Mastery
Mastery check
Six questions calibrated against the patterns we just covered. Honest test — answers are not obvious.
Q1. Your team is building a customer support assistant. 90% of queries are answered by the FAQ; 10% require pulling data from the orders DB and the shipping API. What architecture is most appropriate?
Pure agentic search for all queries — consistency wins
Router → direct retrieval for FAQ-shaped queries, agentic search with DB + API tools for the rest
Pure RAG with all sources embedded together
Plan-and-execute with all three tools available upfront
90% of your traffic is single-hop and answerable from FAQ — paying agentic-loop costs there is unjustifiable. The 10% genuinely benefits from tool selection and adaptive refinement. Routing per-query is the standard production pattern; we'll formalize it in M5.
Q2. Your ReAct agent burns through 25 tool calls on a query that should have taken 3. Looking at the trace, each refinement looks reasonable in isolation. What is the most likely root cause?
Model is too small — upgrade to a frontier model
No explicit sufficiency criterion — the agent has no definition of "enough evidence to answer"
Search results are too sparse — switch search providers
Temperature is too high — set it to 0
This is the "infinite refinement" failure mode (Section 6, #1). Without a stopping criterion, the agent will always find a way to refine. The fix is structural: make the plan include explicit sufficient_when conditions per subgoal, and check them before each new tool call.
Q3. Two tools in your agent's toolkit: search_docs(query) and search_codebase(query). The agent occasionally uses search_docs for code questions. The most effective fix is:
Add a router LLM that picks the tool before the main agent runs
Tighten the tool descriptions — explicit "use this for X, do NOT use for Y" guidance
Merge the tools into one universal_search
Add few-shot examples of correct tool selection to the system prompt
Tool descriptions ARE prompts. Most tool-selection errors come from descriptions that don't disambiguate. Adding "do NOT use for" guidance is consistently the highest-leverage fix. Router LLMs and few-shots are heavier interventions to try only after descriptions are tight.
Q4. Your plan-and-execute agent produces a 6-step plan. Step 3 returns a result that contradicts an assumption baked into steps 5 and 6. What should happen?
Continue executing — the executor is deterministic for predictability
Stop and return an error
Re-invoke the planner with the new evidence; let it produce a revised plan
Execute steps 5 and 6 in parallel to save time, then reconcile
Plans are revisable artifacts. The whole point of separating planner from executor is that you can re-invoke the planner cheaply when evidence invalidates assumptions. Continuing to execute a plan you know is wrong is the most expensive bug in plan-and-execute systems.
Q5. Which of these queries is the WEAKEST justification for an agentic search architecture?
"What's our refund policy?" — answered by a single FAQ paragraph
"Compare Q3 revenue across our top 5 markets and explain the variance"
"Find papers from the last 6 months that contradict the original HNSW results"
"Why did response times spike at 2am on Tuesday — check logs, alerts, and recent deploys"
The refund policy query is single-hop, single-source, latency-sensitive — direct retrieval territory. The other three each require either decomposition (compare, find, why) or multi-tool coordination (logs + alerts + deploys). Right architecture per query type.
Q6. You're given access to a single hyperparameter for your agentic system: max_tool_calls per query. Setting it too high causes cost blowup; too low causes premature termination. The right approach is:
Set it to a single global value tuned on your eval set
Set per-query budgets based on classified query complexity, with the agent aware of its remaining budget
Remove the cap — use cost-based circuit breakers instead
Set it generously (e.g., 30) and rely on the agent's judgment to stop early
A single global cap is a blunt instrument — your simple queries get unnecessary slack, your hard queries get cut off. The mature pattern is: classify query complexity upfront → assign a budget → expose the remaining budget to the agent in its context so it spends deliberately. We build this exact mechanism in M5.
