The tool description
is the prompt.

Most agent failures aren't model failures — they're tool-design failures. A vague description sends the agent down the wrong path before it ever emits a token. A bloated input schema forces the model to guess. A noisy result schema poisons the next ten turns. This module gives you the engineering discipline of designing tools that agents actually use correctly.

01 / Foundation
Tools are prompts wearing JSON costumes
Reframe tool design as the highest-leverage prompt engineering surface in any agent system.

When you register a tool with an LLM, you are not "giving it a function." You are injecting structured prompt content into every turn the agent runs. The tool name, the description, the parameter list, the parameter descriptions, even the order of properties in the schema — all of these are tokens the model attends to before it decides whether to call the tool, and what to pass as inputs.

This reframing has a hard consequence: you must engineer tool descriptions with the same care you engineer your system prompt. A sloppy description doesn't just make a tool harder to use — it changes the agent's behavior on every query, including queries where the tool isn't relevant. The model is constantly asking "is this the right tool?" and bad descriptions make that question harder to answer.

The prompt-engineering frame

If you wouldn't ship a system prompt that says "do something useful with the user's input", you shouldn't ship a tool description that says "searches the database". Same principle — be specific about when to use it, when not to use it, and what good inputs look like.

The four jobs a tool description must do

Every well-designed tool description simultaneously accomplishes four things. Miss any one and you create a class of failures that no amount of model upgrading will fix.

  • Capability framing. What can this tool do? Not "searches" — "searches our internal Confluence pages, returning the top 5 most relevant pages with titles, URLs, and 200-token excerpts."
  • Boundary framing. What this tool cannot do, and what nearby capabilities it does not replace. This is the most-skipped, highest-leverage line in any tool description.
  • Selection guidance. When should the agent prefer this tool over alternatives? Concrete trigger conditions, not vague suitability.
  • Input shape priors. What good inputs look like. Examples are gold. "Use 3-7 word queries; avoid full sentences" is worth a thousand parameter descriptions.
Anatomy of a tool description — bad vs good
Compare

✓ Good description

  "name": "search",

What goes wrong: The model has no idea which knowledge base, no idea what good queries look like, no boundary against using this when it shouldn't, and a parameter named "q" with no description. The agent will use it for everything, with full-sentence queries, and underperform mysteriously.

{
  "name": "search_engineering_docs",
  "description": """Searches our internal engineering documentation
    (Confluence + RFC archive). Returns top 5 pages with title,
    URL, and 200-token excerpt.

    USE FOR: questions about our internal systems, architecture
    decisions, runbooks, post-mortems.

    DO NOT USE FOR: external information, code lookup (use
    search_codebase), customer data (use search_zendesk).

    Best query format: 3-7 keywords. Examples:
      ✓ 'auth service rate limit'
      ✓ 'postgres failover runbook'
      ✗ 'how do I figure out why auth is slow'""",
  "input_schema": {
    "properties": {
      "query": {
        "type": "string",
        "description": "3-7 keywords; avoid full sentences"
      }
    },
    "required": ["query"]
  }
}

What this earns you: The model selects this tool only for the right queries, formats inputs correctly the first time, and stops conflating it with adjacent tools. In production this typically reduces tool-selection errors by 60–80%.

02 / Taxonomy
The six search tool archetypes
Every search tool you'll ever build is a variation on one of six archetypes. Knowing them shortcuts most design decisions.

There is no single "search tool." Production agent systems compose 3–8 different search tools, each specialized for a substrate that requires different query semantics. Treating them as one is the most common architectural mistake. Here are the six archetypes, with the design constraints that distinguish each.

⊙
Web search
External web via Tavily, Brave, Exa, SerpAPI. Best for current events, public data, anything published recently.

Structured queries over a database. Model emits SQL or fills in templates. Highest precision when query maps to schema.

latency10–500ms
cost / call~$0 (your infra)

Code search
AST-aware or grep-based. GitHub API, ripgrep wrappers, language-server queries. Returns symbols and definitions.
query typesymbol / regex
latency50–500ms
cost / call~$0–$0.001
⬡
Knowledge graph
Cypher / SPARQL over entity-relationship graphs. Strongest for traversal queries — paths, neighborhoods, multi-hop facts.
query typeentity + traversal
latency30–300ms
cost / call~$0 (your infra)
⌖
Hybrid search
Vector + BM25 fused. The default for most production text retrieval. Catches both semantic and keyword matches.

latency40–150ms
cost / call~$0.0002

How to choose the right archetype

The archetype is dictated by the shape of the question matched against the shape of the data. Two heuristics:

  • Structured data → structured search. If the answer lives in a database row, vector search is malpractice. SQL or graph traversal will be faster, cheaper, and exact.
  • Unstructured data → hybrid first, vector second. Pure vector search misses exact-match queries (product codes, error strings, names). Hybrid catches both classes with one tool.

Naming conventions that pay off

  • Verb_object_qualifier. search_codebase_python beats codeSearch. Three slots: action, target, scope.
  • Snake_case in tool names. Models trained on Python predict snake_case more reliably. Camel works but is fractionally worse in benchmarks.
  • Avoid generic verbs like "get", "do", "process". They convey nothing. Use search, fetch, list, find, lookup — each carries different semantics the model has internalized.
  • Differentiate near-neighbors with the qualifier slot. If you have search_docs and a code search, make them search_engineering_docs and search_codebase. The qualifier carries the discrimination.
04 / Validation
The input validation layer
Models hallucinate inputs. The validation layer is where you stop garbage from reaching your search infrastructure.

Even with perfect tool descriptions, models will occasionally emit invalid inputs — wrong types, out-of-range values, malformed regexes, queries that bypass intended scopes. The input validation layer is your last line of defense before a bad input becomes a bad search becomes a bad answer. Crucially, it is also a feedback channel back to the agent: a well-crafted validation error teaches the model how to retry correctly.

Validation layer pattern — production-grade
Code
from pydantic import BaseModel, Field, field_validator
from typing import Optional, Literal

class SearchDocsInput(BaseModel):
    query: str = Field(..., min_length=2, max_length=200,
                       description="3-7 keywords; avoid full sentences")
    top_k: int = Field(5, ge=1, le=20)
    space: Optional[Literal["engineering", "product", "ops"]] = None

    @field_validator("query")
    @classmethod
    def no_full_sentences(cls, v: str) -> str:
        word_count = len(v.split())
        if word_count > 12:
            raise ValueError(
                f"Query has {word_count} words; this tool works best "
                "with 3-7 keywords. Try extracting key terms only."
            )
        return v.strip()

async def execute_search_docs(raw_input: dict) -> dict:
    try:
        validated = SearchDocsInput(**raw_input)
    except ValidationError as e:
        # Surface validation errors as TOOL OUTPUT, not exceptions.
        # This lets the agent self-correct on next turn.
        return {"error": "validation_failed", "detail": str(e)}
    return await do_search(validated)
Critical pattern

Return validation errors as tool results, not as exceptions. An exception kills the turn; a tool result lets the agent see the error in its next context window and self-correct. The error message is a teaching signal — write it for the model, not for a human debugger.

The five validation surfaces

  1. Type and shape. Pydantic / zod / json-schema. The cheapest layer, catches the loudest failures.
  2. Range and length. top_k caps, query length caps, recency window caps. Stops cost-blowup queries before they fire.
  3. Enum and scope. Constrain spaces, indices, namespaces to known values. Prevents the agent from inventing scope names.
  4. Semantic shape. Custom validators — query word count, regex validity, SQL safety. Catches what type checkers miss.
  5. Authorization. Does this user / agent role have access to this scope? Inject auth context, enforce at the boundary.
05 / Result Schemas
What you return shapes the next turn
Every result becomes context. Every field becomes attention budget. Design the output schema with the model's next decision in mind.

Search tool outputs are not "data returned to a function caller." They are tokens injected into the next attention pass of a multi-billion-parameter model. Every field you include costs context budget and competes for attention. Every field you omit forces the model to guess or call again. The discipline is finding the minimum schema that supports the agent's next decision.

Result schema evolution — three iterations
Pattern
v1 — Raw dump
v2 — Trimmed
v3 — Decision-ready

What teams ship first. Dumps the full search response. Wastes 60–80% of context on metadata the agent will never use.

{
  "results": [
    {
      "id": "doc_8273",
      "score": 0.8472,
      "vector": [0.012, -0.034, /* 1534 more dims */],
      "metadata": {
        "source": "confluence",
        "created_at": "2024-03-12T08:14:22.000Z",
        "updated_at": "2024-08-30T11:02:11.000Z",
        "author_id": "u_44182",
        "space_id": "sp_eng",
        "page_id": "p_8273",
        "version": 12,
        "raw_html": "<div class='wiki-content'>...</div>"
      },
      "content": "... full 8KB page content ..."
    }
    // × 20 results
  ],
  "facets": { /* aggregations */ },
  "debug": { /* timings, query plan, etc */ }
}

The trimmed pass. Strip metadata the model can't use. Cap result count. Truncate content. Better — but still missing the affordances the agent needs to decide.

{
  "results": [
    {
      "title": "Auth Service Rate Limit Configuration",
      "url": "https://wiki.co/eng/auth-rate-limit",
      "excerpt": "... 200-token snippet ...",
      "updated": "2024-08-30"
    }
    // × 5 results
  ]
}

Decision-ready. Adds the affordances the agent needs to choose between "answer now" and "call again." Total result count, freshness signal, and a stable chunk_id the agent can reference for follow-up fetches.

{
  "results": [
    {
      "chunk_id": "doc_8273:c4",           # for follow-up fetch
      "title": "Auth Service Rate Limit Configuration",
      "url": "https://wiki.co/eng/auth-rate-limit",
      "excerpt": "... 200-token snippet ...",
      "updated": "2024-08-30",
      "freshness": "current",            # enum, not raw date diff
      "score": 0.85                          # normalized 0-1
    }
  ],
  "total_matches": 47,                   # tells agent if it should refine
  "query_used": "auth service rate limit",  # echo for logs + agent reflection
  "suggested_followup": null              # or a query hint if appropriate
}

Six rules for result schemas

  1. Stable shape across calls. Same keys, same nesting. Optional fields are fine; mutating shapes are not. The model learns the schema implicitly across turns and breaks if it shifts.
  2. Token economy. Every byte costs context. Strip everything the model can't or won't use to make decisions.
  3. Decision affordances. Include signals the agent needs to choose what to do next: result count, freshness, confidence.
  4. Stable IDs for follow-up. Always include a chunk_id or equivalent so the agent can ask for "more of this" without re-issuing the search.
  5. Echo the query used. Especially after server-side rewriting. The agent reflects on what it actually searched, not what it intended to search.
  6. Empty is information. Return {"results": [], "total_matches": 0, "hint": "..."} with optional refinement hints. Never throw on no results.
06 / Composition
Composing multiple search tools in one agent
Toggle tools into the canvas and watch how the model's selection accuracy degrades as you add more.

A common production milestone is the day you give your agent its fourth, fifth, sixth search tool. This is also the day tool-selection accuracy starts to drop. The model has more options to weigh, more descriptions to attend to, more chances to pick wrong. Composition is not "add tools until you cover all cases" — it is an architectural choice with measurable cost.

The composition principles

  • Orthogonality first. Tools should partition the search space cleanly. search_docs + search_codebase + search_orders_db have near-zero overlap and selection accuracy stays high.
  • Avoid redundant variants. Don't expose search_kb_vector AND search_kb_hybrid AND search_kb_bm25. Pick one, hide the rest behind it.
  • Never ship a "universal" tool alongside specialized ones. The universal tool absorbs queries that should have routed to the specialists. Either pure universal, or pure specialists.
  • Cap at 7±2. Past 7 specialized tools, consider a router pattern: a meta-tool that picks the right specialist based on a classification step.
Router pattern — when you have too many tools
Architecture

The router pattern keeps the main agent's tool-selection problem trivial (one search-shaped tool) and pushes the actual selection into a focused classifier that only thinks about routing. Used when you have 8+ specialized search tools.

07 / Anti-Patterns
Anti-patterns to recognize and refactor
Eleven specific tool-design mistakes that show up across production agent codebases. Each with the bad form, the fix, and the symptom that exposes it.
01 · The God Tool

A single tool with a flag parameter that switches between fundamentally different behaviors. The model has to choose both the tool AND the mode.

Badsearch(query, mode="vector"|"sql"|"web"|"code")
FixSplit into search_kb, search_orders_db, web_search, search_codebase. Each with its own description.
02 · The Empty Description

Tool description that describes what the tool is rather than when and how to use it.

Bad"description": "A search function that uses BM25 ranking."
Fix"description": "Use this when the user asks about... Do not use for... Best with 3-7 keyword queries..."
03 · The Anonymous Parameter

Parameters with no description, leaving format and intent ambiguous.

Bad"q": {"type": "string"}
Fix"query": {"type": "string", "description": "3-7 keywords; avoid full sentences"}
04 · The Mutable Schema

Output schema that changes shape based on result type — sometimes a list, sometimes an object, sometimes a single value.

BadReturns {"result": "..."} for one result, {"results": [...]} for many.
FixAlways return {"results": [...]} even when the list has 0 or 1 entries.
05 · The Exception Bomb

Tool throws on bad inputs or empty results, killing the agent's turn.

Badraise NoResultsError("nothing found")
Fixreturn {"results": [], "hint": "try broader keywords or different scope"}
06 · The Vector Dump

Returning raw embedding vectors, scores in 8 decimal places, internal IDs, and other metadata the model cannot use.

BadReturns full embedding arrays + every metadata field from the index
FixReturn only: title, url, excerpt, normalized score, freshness, chunk_id
07 · The Synonym Stack

Multiple tools that do near-identical things: search_docs, find_docs, lookup_docs, query_docs.

Bad4 tools with overlapping descriptions and no clear discriminator
FixOne tool. If you need variants, expose them as parameters with clear enums.
08 · The Truncation Surprise

Tool silently truncates long content without telling the agent. The agent thinks it has the full document and answers wrong.

Bad{"content": "first 500 chars..."} (no signal it's truncated)
Fix{"content": "...", "truncated": true, "full_size": 8200, "fetch_url": "..."}
09 · The Hidden Side Effect

A "search" tool that also writes — logs the query as a support ticket, increments a counter, sends a notification. The model can't see what it's triggering.

Baddef search_tickets(q): log_ticket(q); return tickets
FixSearch is read-only. Side effects belong in separately-named tools.
10 · The Stringly-Typed Filter

Filter parameters that take freeform strings instead of constrained enums. The model invents values that don't exist.

Bad"category": {"type": "string"} → model passes "engineerin" (typo)
Fix"category": {"enum": ["engineering", "product", "ops"]}
11 · The Unbounded Result Set

No top_k cap on results. A broad query returns 200 hits, blows out the context window, degrades synthesis.

BadReturns all matches by default
FixDefault top_k=5, max top_k=20, always include total_matches
08 / Mastery
Mastery check
Six questions on the patterns that distinguish production tool design from prototype tool design.
Q1. Your team has search_docs and search_codebase tools. The agent occasionally calls search_docs for code questions. The HIGHEST-leverage fix is:
Add a router LLM that pre-classifies the query type
Add explicit "DO NOT use for code; use search_codebase instead" to search_docs description
Merge them into a single search tool with a "type" parameter
Switch to a larger model
Boundary framing in tool descriptions is consistently the highest-ROI fix for tool-selection errors. It's a one-line change that addresses the root cause — the model didn't know there was a better option. Routers and merging are heavier interventions to try only after the boundary fix isn't enough.
Q2. You're designing a result schema for a vector search tool. You're debating whether to include the raw similarity score (0.8472) or a normalized confidence enum (high/medium/low). Which is better and why?
Raw score — more information for the model to reason with
Normalized enum — model uses categorical signals more reliably than calibrated floats
Both — let the model pick which to use
Neither — score is internal, never expose it
LLMs reason much more reliably about discrete categorical signals than about uncalibrated floats. A score of 0.84 means nothing without context — high for cosine similarity on text, terrible for face recognition. An enum like "high" carries the calibrated meaning. Bonus: enums save context tokens.
Q3. Your search tool currently throws ValidationError when the agent passes an out-of-range top_k. You're seeing the agent crash mid-task. The right fix is:
Catch the exception and silently clamp the value to the valid range
Return the validation error as a tool_result so the agent can self-correct on the next turn
Remove the validation — let the search infrastructure handle it
Add the constraint to the tool description and remove the runtime check
Validation errors are teaching signals, not failures. Returning them as tool results gives the agent visibility into what went wrong and a chance to fix it. Silent clamping hides the issue and erodes the agent's understanding of the tool. Description-only constraints don't survive contact with hallucinated inputs.
Q4. Your agent has 8 search tools. You're considering switching to a router pattern. The single strongest signal that the router is justified:
Tool count exceeded 7
Latency is too high
Tool-selection accuracy on your eval set is degrading as tools are added
The system prompt is too long
Tool count alone isn't the signal — well-designed orthogonal tools can scale further than redundant overlapping ones. The actionable signal is measured selection accuracy on representative queries. If accuracy holds at 8 tools, no router needed; if it drops at 5, the router is overdue.
Q5. You ship a search tool that returns full embedding vectors in its results. Within a week, your latency is up 30% and synthesis quality is down. Most likely root cause:
The embedding model is slow
Vector arrays bloat the context, costing tokens and degrading attention quality
The agent is trying to interpret the vectors and getting confused
JSON serialization is the bottleneck
A 1536-dim vector is ~3000 tokens of useless context. Across 5 results × multiple turns, you've burned tens of thousands of tokens on data the model cannot use. Both latency (more tokens to process) and quality (attention diluted across noise) suffer. The Vector Dump anti-pattern.
Q6. You need to expose both vector search and BM25 keyword search over the same corpus to your agent. The strongest design choice:
Two tools — let the agent choose per query
One hybrid_search tool that fuses both internally; expose only the unified result
A universal search tool with a method parameter
Two tools, but make BM25 the default
Vector vs BM25 is an implementation detail the agent shouldn't have to reason about. Hybrid retrieval (fusion of both) catches both semantic and exact-match queries with a single call. Exposing both forces an unnecessary decision the agent will get wrong some percentage of the time. The Synonym Stack anti-pattern.
