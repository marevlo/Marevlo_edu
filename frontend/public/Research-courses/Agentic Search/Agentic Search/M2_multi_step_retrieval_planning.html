One question.
Many searches.

A real question rarely maps to one query. "Compare these two databases at 10M scale" is four searches. "Why did latency spike Tuesday at 2am" is six. Multi-step retrieval is the discipline of decomposing a question into the searches that can answer it, executing them adaptively, and knowing when to stop. This module is where the agent stops feeling like a chatbot and starts feeling like a researcher.

01 / Foundation
Why one query is rarely enough
A precise model of what makes a question multi-step, and why agents that don't decompose end up either over-fetching or under-answering.

Single-query retrieval works when the question maps cleanly onto one slice of the index. "What's the capital of France?" is one query. "What's our refund policy?" is one query. But the moment the question requires multiple distinct pieces of evidence, evidence from different sources, or evidence whose retrieval depends on what previous searches returned, single-query retrieval breaks down in two predictable ways.

  • Under-answering. The agent retrieves something plausible, the LLM smooths over the gaps with confident-sounding prose, and the user gets an answer that's wrong but feels right. This is the worst class of failure because it's invisible without evals.
  • Over-fetching. The agent jams every keyword from the question into a single mega-query, retrieves 20 marginally-related chunks, and lets the LLM "figure it out" — burning context and quality for no real coverage gain.

Multi-step retrieval is the structural fix. Instead of one query, the agent produces a small set of focused subqueries, each targeting a specific slice of the answer, and assembles the response from the union of evidence. The cost is more tool calls; the payoff is answers that are both more complete and more grounded.

The core insight

A question that needs N pieces of evidence is best served by N focused queries, not by one query that hopes to retrieve everything at once. Specificity beats breadth in retrieval, every time.

The three signals that a question needs decomposition

Before reaching for multi-step machinery, recognize the linguistic patterns that signal it's needed. These are the heuristics a good agent's planner uses (and that you should bake into your own evals).

  • Comparison and contrast. "A vs B", "compare X and Y", "how does this differ from that". Each side of the comparison is its own subquery.
  • Multi-hop entity resolution. "The author of the paper that introduced HNSW". You need to find the paper first, then the author. Two searches, not one.
  • Conjunction with distinct subjects. "What did Anthropic announce at dev day AND how does it compare to OpenAI?" The "AND" with two different subjects almost always means two searches.
02 / Decomposition
The five query decomposition patterns
Every multi-step question follows one of five decomposition shapes. Recognizing the shape tells you how to plan, parallelize, and stop.

Decomposition is not "break the question into pieces." It is selecting from a small inventory of shapes, each with its own structure, parallelism profile, and stopping criterion. Below are the five that cover ~95% of real production queries.

01 · Parallel decomposition
"Compare HNSW vs IVF at 10M scale"
Subqueries are independent. Run all in parallel, then synthesize. Lowest latency. The default for comparisons and lists.
02 · Sequential decomposition
"Who wrote the paper that introduced HNSW?"
Each subquery's input depends on the previous result. Cannot parallelize. Multi-hop entity resolution.
03 · Iterative decomposition
"Find recent papers contradicting the original HNSW results"
First search seeds the topic; subsequent searches refine based on what you found. The classic ReAct loop.
04 · Hierarchical decomposition
"Build a market analysis of vector DB vendors"
Goal → subgoals → sub-subgoals. A tree where leaves are concrete searches. Used for research-grade tasks.
05 · Branching decomposition
"Why did latency spike Tuesday 2am?"
Multiple hypotheses pursued in parallel. Each hypothesis is its own search line. Investigative debugging.

The decomposition algorithm in production

Most teams hand-roll prompts that say "break the question into subquestions." This works for prototypes and breaks for production. The robust pattern uses a structured-output decomposer with explicit shape detection. 

from pydantic import BaseModel
from typing import Literal

class SubQuery(BaseModel):
    id: str
    question: str
    depends_on: list[str] = []         # for sequential / hierarchical
    suggested_tool: str
    sufficient_when: str            # stopping criterion

class DecomposedQuery(BaseModel):
    shape: Literal["parallel", "sequential", "iterative",
                   "hierarchical", "branching", "single"]
    rationale: str                  # why this shape
    subqueries: list[SubQuery]
    synthesis_strategy: str

# Use structured output to get clean decomposition
decomposed = await client.messages.create(
    model="claude-opus-4-6",
    system=DECOMPOSER_PROMPT,
    messages=[{"role": "user", "content": user_query}],
    response_format=DecomposedQuery
)
Production note

The shape field is the most valuable output here. It tells your executor whether to fan out in parallel, run sequentially, or kick off an iterative loop. Without an explicit shape, your executor has to guess from subquery count alone, and it will guess wrong on edge cases.

03 / Iteration
Iterative search loops — search, evaluate, refine
Watch the loop unfold step by step. The hardest part isn't searching — it's deciding when to refine and when to stop.

Iterative search is what most people imagine when they hear "agentic search." The agent issues a query, evaluates the results, decides they're insufficient, refines the query, tries again. The loop continues until the agent has enough evidence to answer — or until it gives up. The structure is simple. Getting the evaluation and refinement steps right is hard.

The four refinement strategies

When evaluation says "not sufficient," the agent must decide how to refine. Four strategies, each appropriate for different failure types.

  • Query rewrite. Same intent, different vocabulary. Used when the original query missed the language the corpus actually uses (e.g., "contradiction" → "limitations"). Highest hit rate per refinement.
  • Scope expansion. Same query, broader source. Used when the right corpus wasn't queried (e.g., add web_search to academic search).
  • Constraint relaxation. Same intent, looser filters. Drop the date range, expand the namespace, raise top_k. Used when the index is too narrow.
  • Decomposition. Realize mid-loop that the question was actually multi-step and split it. Used when one search returns hints of additional sub-questions.
Production-grade iterative loop
Code
async def iterative_search(question: str, max_iterations: int = 5):
    evidence = []
    query = await initial_query(question)

    for i in range(max_iterations):
        results = await search(query)
        evidence.extend(results)

        # Evaluation step — explicit, structured
        eval = await evaluate_sufficiency(
            question=question,
            evidence=evidence,
            iteration=i
        )

        if eval.sufficient:
            return synthesize(question, evidence)

        # Refine — pick a strategy and rewrite
        query = await refine_query(
            previous_query=query,
            evidence=evidence,
            gap=eval.what_is_missing,
            strategy=eval.suggested_strategy   # rewrite|expand|relax|decompose
        )

    # Out of iterations — return what we have, flagged as partial
    return synthesize(question, evidence, partial=True)
04 / Trees
Search trees and beam search over queries
When linear iteration isn't enough — explore multiple query branches in parallel, score them, prune the dead ends.

Sequential iteration explores one query path at a time. For complex investigative tasks — debugging, research, multi-hypothesis questions — you want to explore multiple paths in parallel and prune the unproductive ones. This is search over the space of queries, and the right tool is the same one used in classical AI: beam search.

The idea is straightforward: at each step, generate K candidate next-queries (the "beam width"), execute all of them, score the results by promise, and keep only the top K to expand further. Branches that return weak evidence get pruned. Branches that return strong evidence get expanded.

Three branch-scoring strategies

The quality of beam search rests entirely on how well you score branches. Three approaches, in increasing sophistication:

  1. Result-count scoring. More results = higher score. Cheap, surprisingly effective for many use cases. Fails when irrelevant queries return high counts.
  2. LLM-as-judge scoring. Ask a small model "rate this evidence's relevance to the original question 0-1." More accurate, costs an extra LLM call per branch.
  3. Embedding-similarity scoring. Embed the question and each result excerpt; score by cosine similarity. Fast, calibrated, and the standard production approach for branch scoring.
When NOT to use search trees

Beam search is overkill for queries that have a clear linear structure. If your decomposed query is parallel or sequential with known dependencies, just execute it — don't waste tokens scoring branches you already know you need. Trees earn their cost only when the search space is genuinely unknown a priori.

05 / Stopping
Sufficiency, stopping, and the budget problem
The single most-failed mechanism in agentic search. When have you searched enough?

The hardest engineering problem in multi-step retrieval isn't planning, decomposition, or refinement. It's knowing when to stop. An agent without a sufficiency criterion will keep refining indefinitely until killed by a timeout. An agent with a too-aggressive criterion will stop after one search and answer wrong. Production systems calibrate this carefully — and revisit the calibration constantly.

The four stopping criteria, layered

No single stopping criterion works alone. Production systems layer multiple, with hard caps as backstops to softer signals.

  1. Goal satisfaction. Per-subgoal: "sufficient_when ≥2 sources confirm the latency number with quantitative data." The most accurate, requires explicit upfront definition.
  2. Marginal information rate. Did this last search add anything new? If two consecutive searches returned no novel evidence, stop. Catches the infinite-refinement loop.
  3. Hard iteration cap. max_iterations = N. The dumb-but-essential backstop. Almost never hit if other layers work, but absolutely required.
  4. Hard cost cap. max_tool_calls or max_cost_usd per query. The unit-economics backstop. Without it, one bad query can take down your monthly margins.
Layered stopping logic
Code
class SearchBudget:
    max_iterations: int = 8
    max_tool_calls: int = 15
    max_cost_usd: float = 0.50
    no_progress_threshold: int = 2      # consecutive empty refinements

def should_stop(state: SearchState, budget: SearchBudget) -> tuple[bool, str]:
    # Layer 1: goal satisfaction (preferred outcome)
    if all(sg.satisfied for sg in state.subgoals):
        return True, "all_subgoals_satisfied"

    # Layer 2: no progress
    if state.consecutive_empty >= budget.no_progress_threshold:
        return True, "diminishing_returns"

    # Layer 3 & 4: hard caps
    if state.iteration >= budget.max_iterations:
        return True, "max_iterations_reached"
    if state.tool_calls >= budget.max_tool_calls:
        return True, "max_tool_calls_reached"
    if state.cost_usd >= budget.max_cost_usd:
        return True, "budget_exhausted"

    return False, ""
Critical pattern

Always log the stopping reason. "max_iterations_reached" is a different production problem than "all_subgoals_satisfied". The first means your decomposer or refinement is broken; the second means everything is working. You cannot improve what you cannot distinguish.

Expose the budget to the agent

The most underrated technique in stopping-criterion design: tell the agent its remaining budget, in its context. When the model sees "you have 3 tool calls remaining," it spends them deliberately. When it doesn't, it spends them as if they were free. This single change typically reduces wasted tool calls by 30-50% in production agents. 

# Inject this into the system context every turn
budget_status = f"""
<budget>
  Tool calls used: {state.tool_calls} / {budget.max_tool_calls}
  Iterations: {state.iteration} / {budget.max_iterations}
  Subgoals remaining: {n_unsatisfied} / {n_total}
  
  Spend remaining calls deliberately. Prefer fewer, more-targeted
  searches over broad refinement.
</budget>

06 / State
Plan state management — the revisable artifact
A plan is a living document. Track it explicitly. Mutate it deliberately. Surface its current state to the agent every turn.

In M0 we said plans should be revisable artifacts. This section is the engineering of that: how plan state is structured, when it gets mutated, and how to surface it back to the agent so subsequent decisions are informed by what already happened.

A plan is fundamentally a state machine over subgoals. Each subgoal moves through a small set of states: pending → in_progress → satisfied | failed | abandoned. The plan also tracks evidence collected, queries tried, and the rationale for any state change. This state object is what gets serialized between turns.

Plan state schema
Code
class SubgoalState(BaseModel):
    id: str
    question: str
    status: Literal["pending", "in_progress", "satisfied",
                    "failed", "abandoned"]
    evidence: list[Evidence] = []
    queries_tried: list[str] = []
    sufficient_when: str
    last_updated: datetime
    failure_reason: Optional[str] = None

class PlanState(BaseModel):
    goal: str
    shape: str                           # parallel | sequential | etc
    subgoals: list[SubgoalState]
    discovered_subgoals: list[SubgoalState] = []   # added mid-flight
    iteration: int = 0
    tool_calls: int = 0
    cost_usd: float = 0.0
    consecutive_empty: int = 0

    def render_for_agent(self) -> str:
        """Format current plan state as text the agent can reason over."""
        return render_template("plan_state.j2", plan=self)

The five legal plan mutations

To prevent state corruption, restrict plan mutations to a small set of explicit operations. Every mutation should leave a breadcrumb in the trace.

  • mark_subgoal_satisfied(id, evidence) — when sufficiency criterion met. Most common.
  • mark_subgoal_failed(id, reason) — when refinement budget exhausted without satisfaction.
  • add_discovered_subgoal(parent_id, question) — when search reveals a new sub-question that needs answering.
  • abandon_subgoal(id, reason) — when new evidence makes the subgoal moot. Trim the plan.
  • replan(rationale) — when evidence invalidates the plan structure entirely. Rare but essential.

Render the plan back to the agent every turn

The biggest mistake teams make: they track plan state in code but never show it to the model. The agent then has to reconstruct what's happened from scrolling back through tool calls — slow, error-prone, and token-expensive. The fix: render a compact plan summary into context every turn. 

# What goes into the agent's context every turn:
<plan_state>
GOAL: Compare HNSW vs IVF for 10M vectors

SUBGOALS:
  ✓ HNSW recall/latency at 10M (3 sources)
  ✓ IVF recall/latency at 10M (2 sources)
  ⏳ Memory footprint comparison (1 source — need 1 more)
  ⏸ Real-world deployment cases (pending)

DISCOVERED:
  ⏸ Build time at 10M scale (added at iteration 4)

BUDGET: 6/15 tool calls used · 3/8 iterations · $0.18/$0.50
</plan_state>
07 / Production
Production patterns and pitfalls
Six patterns that show up across mature multi-step retrieval systems, and three pitfalls that consistently bite teams in their first deployment.

Patterns that work

  1. Decomposer-executor split. Use a strong model (Opus-class) for decomposition; use a smaller, cheaper model for individual subquery execution. Decomposition happens once per question; execution happens many times.
  2. Parallel-then-sequential ordering. Even within a sequential decomposition, look for opportunities to parallelize independent legs. Most "sequential" plans have parallelizable subgraphs.
  3. Sufficiency as a separate model call. Don't let the executor decide if it's done. Use a dedicated sufficiency-check call that sees only the question and the evidence — no recency bias from the search trace.
  4. Subgoal-level caching. Cache by subgoal question + evidence retrieved, not by the original user query. Vastly higher hit rates because subgoals recur across user queries.
  5. Provenance threading. Every piece of evidence carries which subgoal it satisfied and which query produced it. Enables both citations and debugging.
  6. Trace replay for evals. Save the full plan + execution trace for every production query. Replay against new model versions to detect regressions before they ship.

Pitfalls that bite

  • Over-decomposition. Breaking a single-hop question into 4 subqueries. Costs 4× and produces a worse answer. Always have a "single" shape in your decomposition output and prefer it when in doubt.
  • Evidence amnesia. Subgoal A finds something relevant to subgoal B, but the agent doesn't notice because evidence is siloed per subgoal. Always check incoming evidence against all unsatisfied subgoals, not just the active one.
  • Replan thrashing. The plan gets rewritten every turn, never converging on stable execution. Cap replans (e.g., max 2 per query) to force the agent to commit.

08 / Mastery
Mastery check
Six questions calibrated against the patterns we just covered.
Q1. A user asks: "Compare the p99 latency of Pinecone and Weaviate at 10M vectors." The most appropriate decomposition shape is:
Single — one query covers it
Parallel — two independent subqueries (one per vendor) run concurrently
Sequential — find Pinecone first, then use that to query Weaviate
Iterative — start broad and refine until you find both
Comparison questions with independent subjects are the canonical parallel decomposition. Both subqueries can fire concurrently with no shared state, halving latency vs sequential. There's no dependency between Pinecone's numbers and Weaviate's numbers.
Q2. Your iterative search agent runs 10 iterations and never stops. Looking at the trace, each refinement returns slightly different but equally weak results. The most likely root cause:
The model temperature is too high
No marginal-information stopping check — agent doesn't notice it's not making progress
The search index is corrupted
Need a larger context window
This is the classic "diminishing returns" failure. The agent has goal-satisfaction logic but no "did this iteration add new information?" check. Adding the no_progress_threshold layer (Section 5) detects when consecutive iterations return no novel evidence and exits cleanly.
Q3. You're choosing between a plain iterative loop and beam search for an agent that investigates production incidents. The strongest signal that beam search is justified:
You have lots of compute budget
Multiple distinct hypotheses need to be evaluated in parallel before committing
Your agent is using a frontier model
Latency requirements are loose
Beam search earns its cost when there's genuine uncertainty about which path to pursue. Incident investigation has multiple plausible causes (deploys, infra, traffic, DB) — beam search lets you explore several in parallel and prune the unproductive ones. For known-linear queries, plain iteration is cheaper.
Q4. You're refining a query that returned 0 results. The query was "HNSW recall contradiction 2024". The strongest first refinement strategy:
Constraint relaxation — remove the date filter
Query rewrite — "contradiction" isn't language used in academic abstracts; try "limitations" or "failure modes"
Scope expansion — add web search alongside arxiv
Decomposition — break into "HNSW" + "recall" + "2024"
Zero-result queries are usually a vocabulary mismatch with the corpus. The word "contradiction" rarely appears in paper abstracts — researchers use "limitations," "failure modes," "negative results." Query rewriting to match the corpus's language is the highest-hit-rate first refinement. Other strategies are good follow-ups if rewriting still fails.
Q5. Your team wants to surface the agent's remaining budget into its context every turn. The strongest reason to do this:
It helps with debugging
The agent makes more deliberate spending decisions when it can see its own constraints
It satisfies compliance requirements
It's required by the API
Models don't reason about constraints they can't see. Without budget visibility, the agent treats every tool call as if it were free and tends to over-explore. With budget visibility — "you have 3 calls left" — the agent shifts to deliberate, targeted spending. Typically 30–50% reduction in wasted calls in production.
Q6. Your decomposer outputs a 5-subquery plan for: "What's our refund policy?" The right action is:
Execute all 5 in parallel — minimize latency
Tune the decomposer's temperature down
Add a "single" shape option to the decomposer and prefer it for simple queries — this is over-decomposition
Cache the result so future calls are fast
A single-FAQ question being decomposed into 5 subqueries is the over-decomposition pitfall (Section 7). The fix is structural — the decomposer must have a "single" shape as a first-class option and explicit guidance to prefer it for narrow questions. Tuning temperature might help, but the root cause is missing decomposition skepticism.
