01 / Architecture
The reference architecture
Eleven components, four layers, one shippable system. Every box maps to a module you've already learned.

A production agentic search system has more moving parts than most teams expect. The good news: the parts are well-understood and the interfaces between them are stable. Below is the reference architecture this module assembles. It is not the only valid architecture, but it is the one that consistently survives contact with production at the scale where teams actually ship.

Production reference architecture
Diagram

The four layers, top to bottom

  • Ingress. Receives the query, attaches user context and budget, classifies the query shape, picks the pipeline (direct retrieval / RAG / full agent), and checks the cache. Most queries should not even reach the reasoning layer — this is where you save 60–80% of cost.
  • Reasoning. The cognitive core: planner decomposes, plan-state-store tracks progress, executor runs the loop, sufficiency judge decides when to stop. Three different model classes for cost-efficiency.
  • Retrieval. The tool fleet — every search archetype from M1, plus domain-specific tools. Each behind validation, caching, and circuit breakers.
  • Reliability. The cross-cutting layer that spans everything: traces, circuit breakers, evals, drift detection. Without this layer, you ship blind.
The most-skipped piece

Teams build the reasoning and retrieval layers and then ship. The ingress layer (especially the classifier and router) and the reliability layer get treated as "phase 2." This is backwards: without ingress, your agent runs expensive loops on queries that don't need them. Without reliability, you can't tell when it's broken. Build them with the rest, not after.

02 / Components
Component walkthrough — what each box does
Eleven components. For each: its job, its interface, and the module it came from.
01 · Ingress
Query Ingress
Receives raw user query. Attaches user identity, organization, locale, and per-user budget. The single chokepoint where authentication and authorization get enforced before any cognitive work happens.
Patterns: M4 §4 (auth), M5 §6 (per-user breakers)
02 · Ingress
Query Classifier
Cheap (Haiku-class) classifier that labels each query by shape: lookup, comparison, multi-hop, investigation, aggregation. Output drives routing. Latency <300ms; runs on every query.
Patterns: M2 §2 decomposition shapes
03 · Ingress
Router
Maps query shape to pipeline. Lookup → direct retrieval. Single-hop synthesis → RAG. Multi-step → full agent. The 80/20 cost lever — most queries should never enter the agent loop.
Patterns: M0 §5 decision tree
04 · Ingress
Cache Lookup
Freshness-aware cache keyed on (rewritten_query, recency_bucket, source_filters). Cuts production traffic 30–50% for common patterns. Always check before hitting the planner.
Patterns: M3 §7 cache-key formula
05 · Reasoning
Planner
Frontier-model decomposer. Outputs a structured plan: shape, subgoals, sufficiency criteria, suggested tools per subgoal, budget. Runs once per agent-pipeline query.
Patterns: M2 §2 production decomposer
06 · Reasoning
Plan State Store
Persistent, mutable record of the plan. Tracks subgoal status, evidence accumulated, queries tried, budget consumed. Rendered into agent context every turn.
Patterns: M2 §6 plan state schema
07 · Reasoning
Executor
Smaller model running the ReAct loop scoped to one subgoal at a time. Calls tools, observes results, refines. Cheap per-turn, runs many times.
Patterns: M0 §4 ReAct, M2 §3 iterative loops
08 · Reasoning
Sufficiency Judge
Dedicated model call: given subgoal + accumulated evidence, is it sufficient? Separate from executor to avoid recency bias. Triggers stop or continue.
Patterns: M2 §5 layered stopping
09 · Retrieval
Tool Fleet
Every search tool from M1 (vector, hybrid, SQL, code, web, graph). Each behind validation, schema enforcement, and per-tool circuit breakers. Tool descriptions are versioned prompts.
Patterns: M1 (all), M3, M4
10 · Reliability
Trace + Eval Pipeline
Every query produces a structured trace. Sampled traces feed the eval harness. Eval results emit metrics. Metrics drive alerts. The flywheel that lets you ship safely.
Patterns: M5 §3-5
11 · Reliability
Circuit Breaker Mesh
Per-query breakers (budget caps), per-user breakers (rate limits), per-tool breakers (downstream failures). Each operates independently; together they bound damage.
Patterns: M5 §6 circuit breakers

The three model tiers

Cost-efficient agents use three tiers of model, not one. The split is the single largest cost reduction available without quality loss.

  • Frontier (Opus-class) for the Planner. Runs once per query. Quality of decomposition matters more than cost — a bad plan is unrecoverable.
  • Mid (Sonnet-class) for the Executor. Runs many times per query. The cost workhorse. Quality matters but doesn't need to be frontier.
  • Small (Haiku-class) for Classifier, Sufficiency Judge, Query Rewriter. Narrow tasks where small models match large ones. Frequency × cheapness = big savings.

What this trace teaches

  • Most of the time is in tool calls, not LLM turns. 1.33s in tool calls + 1.66s for planner+sufficiency+synth + ~70ms for ingress/router. The LLM is rarely the bottleneck.
  • Three model tiers in action. Haiku for classifier and sufficiency, Sonnet for executor and synthesis, Opus for the planner. Cost-mix breakdown: planner ~$0.06, executor+synth ~$0.05, classifier+sufficiency ~$0.01, retrieval ~$0.02.
  • Parallel beats sequential. Decomposition shape was parallel — sg1 and sg2 ran concurrently. Same total tokens, half the wall time.
  • Sufficiency stops cleanly. No infinite refinement. The judge fired once, said "done," and the executor moved to synthesis. This is what M2 §5 looks like in practice.
04 / Capabilities
Capability matrix — what this architecture handles
A summary of which question shapes the reference architecture answers well, which it tolerates, and which it doesn't try to.

No architecture handles everything. The reference architecture is tuned for a specific class of questions — multi-step, multi-source, evidence-grounded synthesis. Below is the honest capability assessment, mapped against common production query types.

Query type Example Handled Notes
Single-hop lookup "What's our refund policy?" ✓ excellent Routed to direct retrieval — never enters agent loop. ~50ms.
RAG-shaped synthesis "Summarize what we know about customer X" ✓ excellent Routed to RAG pipeline — single retrieval + synthesis. ~800ms.
Comparison "Compare HNSW vs IVF at scale" ✓ excellent Parallel decomposition; multi-source synthesis. The trace example.
Multi-hop / entity resolution "Who wrote the paper that introduced HNSW?" ✓ good Sequential decomposition; each hop a separate tool call.
Investigation "Why did p99 spike at 2am Tuesday?" ✓ good Branching decomposition with hypothesis pruning. Higher cost.
Aggregation "How many customers churned in Q3?" ✓ excellent Routed to SQL agent. Exact answer with provenance to schema + query.
Code lookup "Show the function that retries failed payments" ✓ excellent Routed to code search tools. ripgrep + LSP for symbol-level precision.
Open-ended creative "Write a limerick about HNSW" ✗ bypassed Classifier routes to direct LLM call without search. No agent overhead.
Action / write operations "Update the customer's tier to gold" ✗ out of scope This is a search system. Action agents need separate write-tool architecture (different safety story).
Long-running research (hours) "Build a market report on AI startups" ✗ out of scope Needs background-job architecture, not interactive agent. Different cost/latency profile.
Honest scope

An architecture's value is partly defined by what it explicitly does not try to handle. This one is for interactive agentic search — multi-step, evidence-grounded answers in a few seconds to under a minute. Long-running research and write-action systems are different architectures with different trade-offs. Trying to unify them into one system makes both worse.

05 / Maturity
The maturity ladder — where is your system?
Five stages from prototype to production-grade. Honestly assess your current rung; the next module section maps the migration path.

Not every team needs the full reference architecture. Some are early enough that simpler patterns are appropriate. The ladder below is calibrated against systems we've seen ship — find the rung that matches your reality and use it as the starting point for prioritization.

1
Prototype
Single-tool agent, no router, no eval set, no observability beyond logs. Works in demos. Quality and cost are vibes-based. Most agentic search systems live here longer than they should.
2
Beta
Multiple tools, basic router (heuristic, not classifier). Hand-written eval set of 20–50 cases, run before each deploy. Traces stored but rarely queried. Cost is tracked at provider level.
3
Production-validated
Classifier-based router. Three model tiers in use. Eval set of 200+ cases sourced from production. LLM-as-judge for correctness. Per-query budgets enforced. Per-tool circuit breakers in place. Daily eval runs.
4
Operationally mature
Full reference architecture deployed. Plan-state-store as a first-class system. Sufficiency judge separate from executor. Trace replay in CI. Drift monitoring across all seven drift patterns. Per-shape metric slicing. On-call playbooks.
5
Continuously improving
Eval set actively curated by domain experts; growing weekly. A/B harness for prompt and model changes. Cost-per-resolved-query as a North Star metric. Public reliability targets (e.g., 95% answer correctness on the eval set, <$0.20 cost-per-resolved). Quarterly red-team sessions.

What separates the rungs

The transition from rung 1→2 is mostly building. From 2→3 is mostly measuring. From 3→4 is mostly automating. From 4→5 is mostly culture. Each transition takes a similar amount of engineering time but unlocks a qualitatively different operating mode.

Honest assessment

Rate yourself on each dimension separately, not as one rung. Most teams have rung 4 reasoning (good planner, executor, tool fleet) with rung 1 reliability (no real evals, no drift detection, no replay). The reliability gap is what causes the silent quality decay six months in. Bring up the lagging dimension before adding new capabilities elsewhere.

06 / Roadmap
Migration roadmap — building toward the reference
Four phases, ordered by dependency and impact. Don't skip ahead — each phase enables the next.

The mistake teams make is trying to build the full architecture in one go. The right pattern is incremental, ordered by what unlocks what. Below is the migration sequence, with the specific deliverables per phase. Each phase typically takes a small team 2–6 weeks.

Phase 1 · Foundation
Observe what you have
  • Trace pipeline: log every query end-to-end
  • Hand-label 30 representative queries
  • Build minimum LLM-as-judge harness
  • Measure baseline correctness, cost, latency
  • Identify worst-performing query shape
Phase 2 · Routing
Stop overspending
  • Build query classifier (Haiku)
  • Add direct-retrieval bypass for lookup queries
  • Add per-query budget caps
  • Cache layer with freshness-aware keys
  • Re-measure: typically 40–70% cost reduction
Phase 3 · Reasoning
Add the agent loop
  • Planner with structured output
  • Plan state store (M2 §6)
  • Executor scoped per subgoal
  • Sufficiency judge as separate call
  • Three-model tier split (Opus/Sonnet/Haiku)
Phase 4 · Reliability
Bound damage and drift
  • Per-tool circuit breakers
  • Eval set grown to 200+ from production samples
  • Daily eval cadence + alerting
  • Drift monitors for all 7 patterns (M5 §7)
  • Trace replay in CI

The order matters

Why this order, specifically:

  • Phase 1 before everything. You can't optimize what you can't measure. Even a basic eval set + traces will surface 80% of the problems and tell you which phase 2-4 work to prioritize.
  • Phase 2 before Phase 3. Routing alone often gets you most of the way there for the cheapest queries. Building a fancier reasoning layer for queries that should have bypassed it entirely is expensive in both build time and ongoing cost.
  • Phase 3 before Phase 4. You need a stable target to instrument against. Building circuit breakers and drift monitors for an agent that's still architecturally fluid means rebuilding them.
  • Phase 4 last but never optional. The temptation is to ship after phase 3 and claim "we'll add observability later." Six months later, you're debugging a quality drop with no traces, no evals, and no replay capability. Don't.
Anti-pattern

Building all four phases in parallel feels productive but creates integration debt. Components built without their downstream consumers in mind tend to need rework when the consumer finally arrives. Sequential phases let each phase's outputs inform the next phase's design.

07 / Scaling
Scaling considerations — where the architecture stresses
The reference architecture works to ~1M queries/day on standard infrastructure. Above that, specific components stress in predictable ways.

The reference architecture as described scales comfortably to roughly 1M queries/day on commodity infrastructure (a small Postgres for state, Redis for cache, a few app servers). Above that scale, specific components become bottlenecks in predictable order. Knowing the order lets you anticipate rather than firefight.

~1M queries/day · Plan State Store

First component to stress. Read-heavy (one read per executor turn) and write-heavy (one write per plan mutation). Single Postgres can handle this with good indexing.

Threshold signalp99 plan-state read >100ms; lock contention on subgoal updates
MitigationMove to a sharded store (one shard per user/org). Or move plan state to Redis with periodic snapshot to Postgres for replay.
~5M queries/day · Trace Pipeline

Trace volume becomes the dominant write load. Sampled traces still produce GBs/day of structured data.

Threshold signalTrace storage cost approaches LLM API cost; query latency on traces table degrades
MitigationMove traces from Postgres to a columnar store (ClickHouse, BigQuery). Tighten sampling (failures and high-cost still 100%, normal queries to 1%).
~10M queries/day · Cache Backend

Cache hit rates plateau as the long tail of unique queries grows. Single Redis instance hits memory limits.

Threshold signalCache memory utilization >80%; eviction rate climbing; hit rate dropping despite same workload
MitigationTier caching: hot keys in Redis, warm keys in a larger but slower store (S3 + lookup index). Per-shape TTLs to prevent low-value queries from evicting valuable ones.
~50M queries/day · Provider Rate Limits

You're now a top-10 customer of your LLM provider. Rate limits start to bite during traffic spikes.

Threshold signalProvider 429s during peak hours; latency spikes during retries
MitigationMulti-region key pools. Negotiate dedicated capacity. Multi-provider failover for non-critical query types (classification, sufficiency).
100M+ queries/day · Eval Velocity

Now your eval set is the bottleneck on iteration speed. Expanding it and labeling production samples becomes a full-time function.

Threshold signalEval set last meaningfully grew >3 months ago; new failure modes consistently caught by users, not evals
MitigationDedicated data-quality team. Active learning to surface high-value cases for labeling. Domain-expert review queue. Customer-feedback ingestion as eval candidates.

What does NOT need to change at scale

Some components scale much further than people expect with no architectural changes:

  • The planner. One call per query at the agent-pipeline branch. Even at 100M/day, that's a few hundred Opus-class calls per second — well within provider capacity.
  • The classifier. Stateless and cheap. Linear scaling. Throw more containers at it.
  • The executor. Stateless per turn. The plan state store carries the state. Linear scaling.

The pattern: stateful components stress first, stateless components scale linearly. Build the stateful pieces with horizontal scaling in mind from the start, even if you're starting at 10K queries/day.

08 / Mastery
Mastery check
Six questions on the synthesis of everything from M0–M6.
Q1. You're building a search agent for a customer support team. 80% of queries are FAQ-shaped, 15% need order DB lookups, 5% need full investigation across logs and customer history. Which component delivers the largest cost win?
A frontier model planner for the 5% complex queries
A query classifier + router that bypasses the agent loop for the 80% of FAQ queries
Aggressive caching of all results
A larger context window
The 80/20 cost lever is the router (Phase 2). FAQ queries should never touch the agent loop — direct retrieval handles them at ~1% the cost of full agentic processing. Optimizing the planner for the 5% complex queries is real work but smaller absolute cost. Caching helps but only after routing has already cut the bulk traffic.
Q2. Your team has a working agent with no eval set, no traces, no router. They want to add three things in one quarter: (a) a router, (b) an eval set, (c) drift monitoring. Best order:
Router → drift monitoring → eval set
Eval set → router → drift monitoring
Drift monitoring → eval set → router
Router → eval set → drift monitoring
Phase 1 (foundation) is always first — you cannot optimize what you cannot measure. The eval set + traces tell you where the router will pay off most and lets you verify the router doesn't degrade quality. Drift monitoring needs both an eval set and a stable system to monitor against, so it comes last. Building a router without an eval set means you can't verify it improved anything.
Q3. Your reference architecture uses three model tiers (Opus/Sonnet/Haiku). Which assignment is correct?
Haiku for planner, Sonnet for executor, Opus for classifier
Opus for planner, Sonnet for executor, Haiku for classifier and sufficiency judge
Opus for everything to maximize quality
Haiku for everything to minimize cost
The planner runs once per query and a bad plan is unrecoverable — frontier model. The executor runs many times per query — mid-tier optimizes cost. Classifier and sufficiency judge are narrow tasks where small models match large ones, run on every query — small model. This split typically delivers 60–70% cost reduction vs all-Opus with negligible quality loss.
Q4. Your system passes all evals but users complain about wrong answers. You check the metric dashboard — answer correctness is 88%. What's the most likely diagnostic move?
Add more eval cases
Tune the sufficiency judge
Slice metrics by query shape — the failing category is being averaged out by the well-performing ones
Replace the planner with a larger model
88% aggregate could be 98% on lookups and 30% on a category that represents 20% of complaints but 5% of traffic. Slicing by query shape (lookup, comparison, multi-hop, investigation, aggregation) almost always reveals the failing slice. This is the single most common diagnostic miss in production agentic systems and was emphasized in M5 §2.
Q5. At what scale does the plan state store typically need to move off a single Postgres instance?
~10K queries/day
~100K queries/day
~1M queries/day
~100M queries/day
Plan state is the first stateful component to stress. Read-heavy plus write-heavy with locking around subgoal mutations. Single Postgres handles up to roughly 1M queries/day with careful indexing; above that, sharding by user/org or moving to Redis with snapshot persistence is the standard fix. Stateless components (planner, executor, classifier) scale linearly much further before needing intervention.
Q6. Your team is debating: build the full reference architecture in 6 months, or ship the prototype now and harden it later. What's the single strongest argument against "ship now, harden later"?
Users will encounter bugs
It will cost more in API calls
Without observability and evals from day one, you cannot detect or root-cause the quality drift that will accumulate silently over months
The architecture is more elegant
Ship-now systems can iterate on bugs and cost in real time. The thing they cannot do is detect silent drift — quality degradation from model updates, corpus changes, prompt edits, or shifting user query distributions. Six months in, the system is worse than launch and nobody can tell you why or when it happened. The reliability layer is the only one whose absence creates compounding, irrecoverable damage. This is why M5 was a full module and why Phase 4 is non-optional in the migration roadmap.
