Act III · Learn to RepresentModule 16~30 minRanking Losses

The right loss function
changes everything.

Module 15 trained MF with MSE — predict the exact rating, minimize squared error. But Kartify has implicit data: clicks and purchases, not star ratings. And even if it had ratings, the product needs ranking, not prediction. This module introduces BPR — Bayesian Personalized Ranking — the loss function that made MF production-ready for implicit feedback. Same architecture as Module 15. Different loss. Dramatically different results.

01 / THE PROBLEM
Why MSE is the wrong loss for Kartify
Module 04 proved nDCG ≠ RMSE. Module 15 trained with MSE anyway. Now we fix it.

Module 15's biased MF trains by minimizing $(r_{ui} - \hat{r}_{ui})^2$ — the squared difference between the predicted and actual rating. This has two fatal problems for Kartify:

Problem 1: Kartify has no ratings. Implicit data is binary: clicked (1) or not observed (0). Training MSE on 0/1 targets collapses to predicting approximately zero for everything (DEEP M01 showed this — 99.998% of targets are 0).

Problem 2: Even with ratings, MSE optimizes prediction accuracy, not ranking quality. Module 04 proved that a model with RMSE = 3.0 and nDCG = 1.0 (perfect ranking) is better than a model with RMSE = 1.4 and nDCG = 0.3 (good prediction, terrible ranking). We don't care about exact scores. We care about which items appear at the top.

We need a loss function that says: "I don't care what score you assign to each item. I only care that items the user did interact with score higher than items they didn't."

Pause — reframe

If you can't predict exact ratings, but you CAN learn that clicked items should rank above unclicked items, what kind of training data do you need?

You need pairs: (user, positive item, negative item). For each user, take an item they clicked (positive) and an item they didn't (negative). The training signal is: "the positive item should score higher than the negative item." No exact scores. No ratings. Just relative preferences. This is the pairwise ranking approach.
02 / THE SOLUTION
BPR — recommendation as a tournament
For each user: the clicked item should beat the unclicked item. Train millions of these matches. The embeddings learn to rank.

Think of recommendation as a round-robin tournament. Every item is a player. For each user, the items they clicked are "winners" and a random unclicked item is the "loser." The model's job: learn to predict the tournament outcomes — the winner should score higher than the loser.

Priya clicked on a Silk Saree. She did not click on Running Shoes (sampled at random from unclicked items). BPR says: "I don't care what score you give the Saree or the Shoes. I only care that score(Saree) > score(Shoes) for Priya."

Train on millions of such pairs. The embeddings gradually learn positions in latent space where, for each user, their clicked items consistently outscore random unclicked items. That's ranking.

# BPR training — the core idea in 6 lines
for user, pos_item, neg_item in sample_triples(data):
    pos_score = dot(user_emb[user], item_emb[pos_item])
    neg_score = dot(user_emb[user], item_emb[neg_item])

    # Loss: we want pos_score >> neg_score
    # If pos_score - neg_score is large → loss is small (good)
    # If pos_score ≤ neg_score → loss is large (bad)
    loss = -log(sigmoid(pos_score - neg_score))
    loss.backward()
Why this works for implicit data

BPR doesn't need ratings — it only needs to know "Priya clicked A but not B." Every implicit feedback dataset provides this: observed interactions are positives; unobserved items are sampled as negatives. No ratings required. No assumption that unobserved = disliked (just "we have no evidence the user likes it"). The asymmetry between "observed positive" and "sampled negative" naturally handles the MNAR problem from Module 03.

03 / THE DETAIL
Negative sampling — the make-or-break engineering detail
How you choose the "loser" in each tournament match determines whether BPR learns efficiently or wastes its training budget.

Uniform random sampling (default): for each (user, positive item), sample a random item the user hasn't interacted with. This works but is inefficient — most random items are "easy negatives." Priya clicked a Silk Saree; the random negative is a Car Battery. The model easily distinguishes them. No useful learning happens.

Hard negative mining: sample negatives that the model currently scores high but are actually unclicked. These are the "close losses" in the tournament — the items the model wrongly thinks the user would like. Training on these gives much stronger gradients. Example: the random negative is a Cotton Kurta that the model scores just below the Silk Saree — now the model has to work to distinguish them.

Popularity-based sampling: sample negatives proportional to item popularity. Popular items are more informative negatives because the user plausibly saw them and chose not to click. A negative sample of a popular item the user ignored is stronger evidence than a negative sample of an obscure item they never saw.

The 10x rule

In practice, hard negative mining can improve BPR-MF's nDCG@10 by 10–30% relative over uniform sampling — same model, same data, same training time, just different negative samples. This is why negative sampling strategy is considered more important than model architecture for retrieval models. Get the negatives right before making the model bigger.

04 / THE VERDICT
Why BPR won production
BPR-MF is the default first model for any implicit-feedback recsys. Here's why it's still dominant.

Fast to train. $O(k \cdot |\text{interactions}|)$ per epoch. With $k = 128$ and 18M interactions: ~2.3B operations per epoch. On a single GPU: ~2 minutes per epoch, 20 epochs = 40 minutes total. Comparable to item-kNN's offline cost, but produces full-catalog embeddings.

Easy to serve. The item embeddings $\mathbf{v}_i$ can be indexed with an Approximate Nearest Neighbor (ANN) structure (FAISS, ScaNN). At serving time, compute the user's embedding $\mathbf{u}_u$, query the ANN index for the top-1000 nearest items in ~10ms. This IS the modern retrieval architecture.

Surprisingly hard to beat. A 2019 study (Ferrari Dacrema et al.) showed that many "deep learning for recsys" papers from top venues failed to outperform properly-tuned BPR-MF on standard benchmarks. The baseline is stronger than people think. If your deep model doesn't beat BPR-MF, the deep model is probably broken, not BPR-MF.

Architecture → Two-tower

BPR-MF is literally: two embedding tables (user, item) + dot product + pairwise loss. Replace the embedding lookup with a neural network that takes features as input → you get the two-tower retrieval model used at YouTube, Pinterest, Spotify, and every major platform. Understanding BPR-MF deeply means understanding the skeleton of production retrieval. The architecture didn't change; the input features got richer.

05 / KARTIFY
BPR-MF as Kartify's primary retrieval model
Replacing MSE with BPR. Same embeddings, right loss, measurable improvement.
Kartify pipeline — Module 16 update

RETRIEVAL (updated — BPR-MF replaces biased MF):

• Content-based TF-IDF (M07) — cold-start items
