The bug that ruins
every first kNN.

You build item-kNN. You ship it. The #1 recommendation for every user is a random obscure product nobody's heard of — with a perfect similarity score of 1.0. Your model isn't broken. It's doing exactly what you asked. The problem is that you asked the wrong question. This module fixes it.

01 / THE BUG
The mysterious perfect score
Two items, one shared user, cosine similarity = 1.0. The math is correct. The recommendation is garbage.

You deploy Kartify's item-kNN from Module 09. The first user opens the app. Their top recommendation: "Hand-Painted Pichwai Art Frame — ₹12,499." Similarity score: 1.00.

Priya has never browsed art. She's an ethnic wear customer. Why is an art frame her #1 recommendation with a perfect score?

You dig into the data. The Pichwai Art Frame has been purchased by exactly 1 user — User #847291. That same user also purchased the Banarasi Silk Saree that Priya liked. Their interaction vectors are perfectly aligned — both items have a single "1" at position #847291, and zeros everywhere else. Cosine similarity: 1.0.

One shared user. Perfect score. Garbage recommendation.

This is the single most common failure mode of naive kNN: items with few co-interactions get inflated similarity scores. Two niche items that share one user have cosine = 1.0, outscoring items with hundreds of legitimate co-purchases that have cosine = 0.7. The noise from low-support pairs dominates the signal from high-support pairs.

Pause — diagnose

Before reading the fix, think: how would YOU solve this? The problem is that low co-interaction counts produce unreliable similarity estimates. What's a simple way to penalise those?

The intuition: multiply the similarity by a confidence factor that's small when co-interactions are few and approaches 1 when they're many. A pair with 1 shared user should have its similarity heavily discounted. A pair with 500 shared users should be trusted at face value. This is called shrinkage — and it's one line of code.
02 / THE FIX
Shrinkage — the one-line fix that changes everything
One multiplication. Typically improves kNN quality by 10–20% relative. Not optional.
# Shrinkage: penalize similarities based on few co-interactions
def shrunk_similarity(a, b, shrinkage=100):
    co_count = np.sum((a > 0) & (b > 0))  # number of users who interacted with BOTH
    raw_sim = cosine_similarity(a, b)
    return raw_sim * co_count / (co_count + shrinkage)

That's it. The formula is: shrunk_sim = raw_sim × n / (n + λ), where n is the number of co-interactions and λ is the shrinkage parameter (typically 50–200).

Let's see what this does to our bug:

The Pichwai Art Frame has 1 shared user with the Banarasi Saree. With λ = 100: shrunk_sim = 1.0 × 1 / (1 + 100) = 0.0099. The perfect score of 1.0 becomes 0.01 — effectively zero.

Meanwhile, the Gold Jhumka Earrings have 280 shared users with the Banarasi Saree, and raw cosine = 0.61. With λ = 100: shrunk_sim = 0.61 × 280 / (280 + 100) = 0.449. The earrings keep most of their score because the evidence is strong.

The Bayesian interpretation

Shrinkage has a Bayesian interpretation: the raw similarity is the "observed" estimate, and the shrinkage parameter represents your prior belief that the true similarity is zero. With few observations (low n), the prior dominates → similarity shrinks toward zero. With many observations (high n), the data dominates → similarity approaches the raw estimate. λ controls how many co-interactions you need before you trust the data.

The anti-pattern

If your item-kNN is returning bizarre, unrelated items with perfect scores — check for missing shrinkage. It's the #1 implementation bug in neighborhood CF. Shrinkage is not a nice-to-have. It's structural. Skip it and your model is fundamentally broken even though the math is correct.

04 / MORE FIXES
Four more fixes that make kNN production-grade
Shrinkage is the critical one. These four are the difference between "works" and "works well."
Fix 1: Popularity dampening
Popular items are "similar" to everything — they have co-interactions with every other item simply because everyone buys them. Solution: downweight popular items using an IDF-style term. Multiply each user's interaction by 1/log(1 + total_interactions_for_that_item). This reduces the influence of items like "Cotton T-shirt" that everyone buys.
Fix 2: User activity normalization
Heavy users interact with everything — their co-interactions carry less signal per item than a selective user's. Divide each user's contribution by their total interaction count. A user who clicked 500 items contributes 1/500 per co-interaction; a user who clicked 5 items contributes 1/5. This prevents power users from dominating similarity estimates.
Fix 3: k selection
The number of neighbors k (typically 20–100) controls the breadth of the similarity model. Too small (k=5): insufficient coverage, many items have no score. Too large (k=500): noisy neighbors dilute the signal. Cross-validate on a held-out ranking metric (nDCG@10). For Kartify: k=50 is a good starting point for 480K items.
Fix 4: Similarity threshold
Even after shrinkage, some item pairs have near-zero similarity (0.01, 0.02). These add noise to the scoring aggregation without meaningful signal. Drop any neighbor with shrunk similarity below a floor (e.g., 0.05). This reduces memory, speeds up inference, and marginally improves quality.
05 / KARTIFY
Kartify's production kNN — the complete config
After this module, Kartify's item-kNN is production-grade. Here's the full specification.
Kartify item-kNN — production config

Similarity metric: Cosine (for "similar items") / Conditional probability (for "complete your look")
Shrinkage λ: 100 (cross-validated on held-out nDCG@10)
Neighbors k: 50 per item
Popularity dampening: IDF weighting on item interaction counts
User normalization: Divide contributions by sqrt(user activity)
Similarity floor: 0.05 (drop pairs below this after shrinkage)
Recompute schedule: Nightly batch job (~45 min on Kartify's 480K items)
Online latency: ~3ms per request (sparse matrix lookup + filter)
