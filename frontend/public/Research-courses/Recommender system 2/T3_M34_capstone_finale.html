01 / INTEGRATION
Wire the complete system — every module connected
Retrieval + ranking + reranking + diversity + monitoring. One function call, 200ms, 20 items.
🛠 Task 1: Complete the full pipeline

1a. Integrate all components. Your system should execute in this order:

→ User request arrives
→ Retrieval (Part 1): BPR-MF + FAISS → 1,000 candidates (10ms)
→ Feature extraction: Join user, item, context features (7ms)
→ Ranking (Part 2): DeepFM scores 1,000 candidates → top 50 (80ms)
→ Reranking: MMR diversity (λ=0.7) on top 50 → final 20 (3ms)
→ Return 20 items with scores, categories, prices

1b. Add the fallback chain from Module 28: if FAISS fails → serve cached results. If ranking fails → serve retrieval-scored top-50. If everything fails → serve popularity top-20.

1c. Add monitoring hooks from Module 31: log recommendation diversity, latency per stage, candidate count per source. These are the signals you'd monitor in production.

Pause — the integration test

Run your full pipeline on 100 test users. What's the P50 and P99 end-to-end latency? Is it under 200ms? If not, which stage is the bottleneck?

The most common bottleneck is feature extraction — joining features for 1,000 candidates from multiple sources. If you're over 200ms, check: (a) are you doing 1,000 individual feature lookups instead of batched lookups? (b) is the ranking model running on CPU instead of GPU? (c) is FAISS using the right index type (IVF-PQ, not brute force)? In a real deployment, features would be pre-computed in a feature store (Redis) with batch lookup. For the capstone, simulating this with a dictionary lookup is acceptable.

2b. Compute total daily cost. Sum all components. Express as cost per recommendation and cost per user per day.

2c. Compare to revenue impact. If recommendations drive 30% of Kartify's GMV (typical for e-commerce), and daily GMV is ₹50 lakh, the recommendation system's revenue attribution is ₹15 lakh/day. Is the infrastructure cost justified? What's the ROI?

The cost-quality frontier

Module 26's "good enough" principle applies to infrastructure too. A BPR-MF retrieval-only system (no ranking model, no GPU serving) might achieve 80% of the quality at 20% of the cost. The ranking model adds 5–15% quality improvement but 5× the infrastructure cost. Is it worth it? The answer depends on absolute GMV: 5% of ₹15 lakh = ₹75K/day. If the ranker costs ₹3K/day, the ROI is 25:1 — obviously worth it. If the ranker costs ₹50K/day, the ROI is 1.5:1 — marginal. Always compute the ROI, not just the quality lift.

Pause — the honest projection

Your offline nDCG improved 4× over popularity. Does that mean CTR will improve 4×? Why or why not?

No. Module 29's offline-online gap applies directly here. Offline nDCG measures ranking quality on historical data. Online CTR depends on: display quality (thumbnails, layout), user attention patterns, novelty effects, and ecosystem dynamics. A typical rule of thumb: a 4× offline improvement translates to 1.5–2× online improvement. So if current CTR is 2%, expect 3–4% with personalization, not 8%. Always discount offline gains by 50–70% when projecting online impact. Being conservative earns trust; over-promising destroys it.
05 / REPORT
The final deliverable — your capstone report
Everything in one document. This is what you'd present to a hiring committee, an investor, or a product team.
🛠 Task 5: Assemble the capstone report

Your final report should have 7 sections:

Section 1: Executive Summary (1 page). What you built, key results, business recommendation. Non-technical language.

Section 2: Dataset & Problem Framing (1 page). Data description, sparsity analysis, evaluation methodology, temporal split rationale.

Section 3: Model Development (3 pages). Baselines → BPR-MF → DeepFM ranker. Each model's architecture, training details, and results. The progression story: each model beats the previous.

Section 4: Evaluation (2 pages). Complete leaderboard (all models, all metrics). Statistical significance tests. Error analysis (where the system fails and why). Embedding visualizations.

Section 5: System Architecture (1 page). The full pipeline diagram. Latency breakdown per stage. FAISS benchmark. Fallback chain.

Section 6: Production Plan (1 page). Deployment plan, monitoring, A/B test design, cost analysis, ROI projection.

Section 7: Limitations & Future Work (1 page). What the system can't do yet. What you'd add with more time (SASRec for sessions, LightGCN for sparse subcategories, multi-task ranking, LLM-powered explanations). Be honest about limitations — it shows maturity.

What separates good from great

A good capstone shows a working system with accurate evaluation. A great capstone shows the THINKING: why you chose BPR-MF over WRMF (Module 17), why DeepFM over DCN-v2 (Module 26's complexity-quality trade-off), why you'd deploy retrieval-only first and add ranking later (Module 26's phased approach), and where the system would break under 10× traffic (Module 28's scaling analysis). The code proves you can build. The reasoning proves you can engineer.
