The model zoo —
picking the right architecture.

You've now seen 15+ recommendation models across 20 modules. Popularity, CBF, kNN, MF, BPR, WRMF, EASE, FM, DeepFM, DCN-v2, SASRec, BERT4Rec, LightGCN, NCF, LLM-augmented. A learner finishing this list might feel paralyzed: "which one do I actually use?" This module gives you the decision framework.

01 / FIVE QUESTIONS

A startup spent 4 months building a graph neural network recommender with attention mechanisms, multi-task learning, and contrastive pre-training. On their benchmark, it achieved state-of-the-art nDCG. They deployed it. Revenue was flat. A consultant came in, tuned the hyperparameters of a BPR-MF model in 3 days, and achieved the same nDCG. The GNN was dismantled. The consultant's invoice was smaller than one month's GPU bill for the GNN training. The team learned: model sophistication is not the same as business value. This module exists so you never make that mistake.

Ask these before choosing a model
The right model depends on your situation, not the model's sophistication.

Question 1: What data do you have? Explicit ratings → biased MF (M15). Implicit clicks/purchases → BPR-MF (M16). Rich item features → FM/DeepFM (M19–20). Session sequences → SASRec (M22). Social graph → LightGCN (M23).

Question 2: What's your cold-start severity? Mostly returning users → pure CF (MF, kNN). 50%+ new users → need features (FM/DeepFM, CBF fallback). Frequent new items → need content features (two-tower, LLM embeddings).

Question 3: What's your latency budget? Real-time retrieval (< 20ms for full catalog) → must use dot product + ANN (BPR-MF, two-tower, LightGCN). Ranking (< 100ms for 1000 items) → can use MLP-based models (DeepFM, DCN-v2, NCF). Batch/offline → can use anything including LLMs.

Question 4: What's your team size? Solo or small team → BPR-MF + EASE. No deep learning needed. These baselines are strong. Medium team → add DeepFM ranking + SASRec retrieval. Large team → full stack: multiple retrieval sources, DCN-v2 ranking, multi-task learning, LLM augmentation.

Question 5: What's your data scale? < 100K interactions → item-kNN or EASE (MF overfits). 100K–10M → BPR-MF sweet spot. 10M–1B → DeepFM/DCN-v2 + multi-source retrieval. > 1B → distributed training, feature stores, streaming updates.

Pause & Predict

You're the first ML engineer at a startup with 500K users, 50K items, 2M interactions, and a 2-person engineering team. Using the five questions framework, what model stack would you choose?

Data: implicit clicks → BPR-MF (not biased MF). Cold-start: moderate (500K users, most have some history) → pure CF is viable, no urgent need for features. Latency: standard web (<200ms) → dot product + ANN for retrieval. Team: 2 people → keep it simple, no deep learning ranking yet. Scale: 2M interactions → BPR-MF sweet spot. Stack: popularity fallback + BPR-MF retrieval + FAISS ANN. That's Phase 2 from this module. Add DeepFM ranking only after BPR-MF is running and you've measured the baseline. A 2-person team should spend 80% of effort on data quality and evaluation, not model complexity.

03 / THE PRINCIPLE
The "good enough" principle — when to stop
Every additional model adds engineering complexity. The marginal gain decreases. Know when to stop.

The 80/20 of recsys model selection: BPR-MF for retrieval + DeepFM for ranking gives you 80% of the quality ceiling with 20% of the engineering effort. Adding LightGCN, SASRec, multi-task learning, and LLM features might push you from 80% to 95% — but each addition doubles the system complexity.

Start here, add if needed:

Phase 1 (week 1): Popularity + item-kNN + EASE. Zero ML infrastructure needed. Verify the evaluation pipeline works.

Phase 2 (month 1): Add BPR-MF + FAISS ANN. This is where the big jump happens — full-catalog personalization.

Phase 3 (month 3): Add DeepFM ranking model. Use impression logs, add features. This separates the retrieval and ranking stages.

Phase 4 (month 6+): Add SASRec for session-aware retrieval, LightGCN for graph-enhanced embeddings, multi-task learning, A/B test everything. Only add what measurably improves business metrics.

The anti-pattern

The most common failure: jumping to Phase 4 complexity before Phase 2 baselines are properly tuned. A team builds a transformer-based multi-task ranking model with 50 features, discovers it barely beats popularity, and concludes "our data is too sparse for ML." The real problem: they skipped BPR-MF (which would have worked) and went straight to a model that needs 10× more data to train.

Pause & Predict

The 'good enough' principle says BPR-MF + DeepFM gives 80% of quality at 20% of effort. When is it worth pushing to 95% with SASRec + LightGCN + multi-task learning?

When the marginal revenue justifies the engineering cost. If Kartify's recommendations drive ₹15 lakh/day GMV, a 5% improvement = ₹75K/day = ₹2.7 crore/year. If adding SASRec + LightGCN costs 2 engineers × 6 months (₹30-50 lakh) plus ₹5 lakh/year infrastructure, the ROI is 5:1 in the first year. At a small startup doing ₹50K/day GMV, the same 5% = ₹2.5K/day = ₹9 lakh/year. The engineering cost exceeds the revenue lift. The decision is always: does the marginal quality improvement generate more revenue than it costs to build and maintain?
04 / KARTIFY
Kartify's final model selection — with justification
Kartify's complete model stack — Act IV finale

RETRIEVAL (4 sources, 10ms total):
• Popularity (M06) — cold-start, trending, ~200 candidates
• BPR-MF (M16) + FAISS — primary personalization, ~500 candidates
• SASRec (M22) + FAISS — session-aware, ~200 candidates
• LightGCN (M23) + FAISS — graph-enhanced, ~100 candidates
→ Dedup & merge → 1000 unique candidates

RANKING (DCN-v2 multi-task, 80ms):
• 33 features + sentence-transformer embeddings
• 3 heads: P(click), P(purchase|click), E[dwell]
• Combined score: business-objective weighted
→ Top 50 scored items

RERANKING (10ms):
• MMR diversity enforcement
• Business rules (boost new items, suppress low-stock)
→ Final 20 items displayed

LLM AUGMENTATION:
• Sentence-transformer item embeddings (batch, nightly)
• Conversational gift recommender (high-AOV categories)
• Post-hoc explanations ("why this recommendation")

WHY THESE CHOICES:
• BPR-MF over WRMF for primary retrieval: better nDCG, GPU-trainable
• DCN-v2 over DeepFM for ranking: explicit higher-order crosses, better on production data
• SASRec over GRU4Rec: parallel training, selective attention, compatible with ANN
• LightGCN as supplemental: +5% on sparse subcategories, same serving infra as BPR-MF
