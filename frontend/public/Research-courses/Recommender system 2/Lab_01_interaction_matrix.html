Part 02

Build the Sparse Matrix

From DEEP M01 — the interaction matrix R is M×N. Most cells are empty. Never store it dense.

The interaction matrix R has M users as rows and N items as columns. R[u][i] holds user u's rating of item i — or zero if they never interacted. At MovieLens scale that's 24M cells; at Kartify scale it's 1.1 trillion. You must store it sparse.

Lab_01 · sparse CSR matrix 
from scipy.sparse import csr_matrix

# Step 1: contiguous integer ID mappings
uid_map = {uid: idx for idx, uid in enumerate(sorted(ratings.user_id.unique()))}
iid_map = {iid: idx for idx, iid in enumerate(sorted(ratings.item_id.unique()))}

M, N = len(uid_map), len(iid_map)

rows = ratings.user_id.map(uid_map).values
cols = ratings.item_id.map(iid_map).values
vals = ratings.rating.values.astype(np.float32)

# Step 2: build the sparse matrix
R = csr_matrix((vals, (rows, cols)), shape=(M, N))

# Step 3: measure compression
sparse_mb = (R.data.nbytes + R.indices.nbytes + R.indptr.nbytes) / 1e6
dense_mb  = M * N * 4 / 1e6

print(f"Sparsity:    {100 - R.nnz/(M*N)*100:.3f}%")
print(f"Sparse:      {sparse_mb:.1f} MB")
print(f"Dense:       {dense_mb:.1f} MB")
print(f"Compression: {dense_mb/sparse_mb:.0f}×")

Part 03

Temporal Train-Test Split

From DEEP M05 — never use random splits in recsys. Train on the past, test on the future.

Random splits leak future information into training. If a user rated a movie in January and you train on their February ratings, you've cheated. Always split by timestamp: the last 20% of interactions by time become the test set.

⚠ Common Mistake

Random 80/20 splits are the most common evaluation mistake in recsys. They inflate metrics by 5–15% compared to temporal splits — making every model look better than it is in production.

Lab_01 · temporal_split() 
def temporal_split(df, test_frac=0.2):
    """Split by time — last test_frac of timestamps → test."""
    cutoff = df.timestamp.quantile(1 - test_frac)
    train  = df[df.timestamp <  cutoff].copy()
    test   = df[df.timestamp >= cutoff].copy()

    # Only keep test interactions for users/items seen in training
    # (cold-start users are a separate problem)
    test = test[test.user_id.isin(train.user_id.unique())]
    test = test[test.item_id.isin(train.item_id.unique())]
    return train, test

train_df, test_df = temporal_split(ratings, test_frac=0.2)
print(f"Train: {len(train_df):,} ({len(train_df)/len(ratings):.0%})")
print(f"Test:  {len(test_df):,}  ({len(test_df)/len(ratings):.0%})")
Part 04

Ranking Metrics From Scratch

From DEEP M05 — every metric computed by hand first, then coded. No library shortcuts.

Four metrics, all measuring how well the system ranks relevant items at the top of the list. Each captures a different aspect of ranking quality.

Lab_01 · four ranking metrics 
def precision_at_k(ranked_list, relevant_set, k):
    """Fraction of top-k that are relevant."""
    return len(set(ranked_list[:k]) & relevant_set) / k

def recall_at_k(ranked_list, relevant_set, k):
    """Fraction of all relevant items found in top-k."""
    return len(set(ranked_list[:k]) & relevant_set) / max(len(relevant_set), 1)

def ap_at_k(ranked_list, relevant_set, k):
    """Average Precision: rewards finding relevant items early."""
    hits, sum_prec = 0, 0.0
    for i, item in enumerate(ranked_list[:k]):
        if item in relevant_set:
            hits += 1
            sum_prec += hits / (i + 1)   # precision at each hit position
    return sum_prec / max(len(relevant_set), 1)

def ndcg_at_k(ranked_list, relevant_set, k):
    """nDCG: log-discounted gain, normalized by ideal ranking."""
    dcg  = sum(1.0 / np.log2(i + 2)
               for i, item in enumerate(ranked_list[:k]) if item in relevant_set)
    idcg = sum(1.0 / np.log2(i + 2) for i in range(min(k, len(relevant_set))))
    return dcg / idcg if idcg > 0 else 0.0

Part 05

Popularity Baseline

From M06 — the simplest recommender. Rank items by total interaction count.

The popularity baseline recommends the globally most-popular items not already seen by the user. It's non-personalized — every user gets the same ranking — but it's surprisingly competitive on sparse data where collaborative signals are weak.

Lab_01 · popularity baseline 
def evaluate_model(recommend_fn, test_data, k=10):
    metrics = {'ndcg': [], 'precision': [], 'recall': [], 'map': []}
    for user_id, relevant_items in test_data.items():
        ranked = recommend_fn(user_id, k)
        rel    = set(relevant_items)
        metrics['ndcg'].append(ndcg_at_k(ranked, rel, k))
        metrics['precision'].append(precision_at_k(ranked, rel, k))
        metrics['recall'].append(recall_at_k(ranked, rel, k))
        metrics['map'].append(ap_at_k(ranked, rel, k))
    return {m: np.mean(v) for m, v in metrics.items()}

item_popularity = np.array(R_train.sum(axis=0)).flatten()
