Build Kartify's
retrieval system.

No more reading. No more worked examples on 3×4 matrices. This is the capstone project — you build a real recommendation system from raw interaction data to production-ready retrieval. By the end of this module, you'll have a trained BPR-MF model, a FAISS ANN index, and a complete evaluation against multiple baselines. Every technique from Modules 01–31 comes together here.

01 / THE BRIEF
Your assignment — the Kartify recommendation engine
You're the first ML engineer at Kartify. The CEO wants personalized recommendations on the homepage. Here's what you have and what you need to deliver.

What you have: 18 months of user interaction data — clicks, add-to-cart, purchases — for 2.3M users and 480K products. Product catalog with category, subcategory, price, brand, and seller metadata. No existing recommendation system. The homepage currently shows "new arrivals" and "trending" — both manually curated.

What you need to deliver:

1. A retrieval system that produces 1,000 personalized candidate items per user in under 10ms.

2. At least 3 retrieval sources: popularity baseline, collaborative filtering (BPR-MF), and one additional source of your choice.

3. A complete evaluation report comparing all methods on nDCG@10, Precision@10, Recall@10, and MAP.

4. An ANN serving prototype demonstrating sub-10ms latency.

The deliverable: Code + evaluation report + serving demo. Due in Part 1. Parts 2 (ranking) and 3 (full pipeline + business report) build on this.

🛠 Task 1: Set up your environment

Create a project directory. Install dependencies: numpy, scipy, pandas, torch, faiss-cpu (or faiss-gpu if available). Download the dataset (MovieLens 10M or Amazon Reviews for a public dataset, or use the Kartify simulation generator from Lab 1). Verify everything loads.

02 / EDA
Explore the data — before you touch a model
Module 05 says: "Before you touch an algorithm, answer these." This is where that lesson becomes real.
🛠 Task 2: Exploratory Data Analysis

Answer these questions with code and visualizations:

2a. How many users, items, interactions? What's the sparsity? Compare to the Kartify numbers from DEEP M01.

2b. Plot the interaction distribution: how many interactions per user (histogram)? Per item? What's the median, mean, and 95th percentile?

2c. How many users have fewer than 5 interactions (cold-start users)? What fraction of total users is this?

2d. Plot interactions over time. Is there seasonality? Growth? Any anomalies (data collection gaps, promotional spikes)?

2e. Perform the temporal train-test split from Lab 1 (last 20% of time → test). How many test users have ≥1 relevant item? This is your evaluation population.

Pause — the checkpoint question

Before proceeding: what is the sparsity of your interaction matrix? Based on DEEP M14's analysis, is MF likely to help significantly over kNN at this sparsity level?

If sparsity > 99%: MF will likely outperform kNN significantly because transitive generalization fills in the 99%+ missing entries. If sparsity is 95–99%: MF will help but the gap will be smaller. If sparsity < 95% (very dense): kNN might match MF because direct co-interaction data is sufficient. For most real-world datasets, sparsity is 99%+ and MF wins by 15–30% nDCG over kNN.
03 / BASELINES
Build baselines first — the floor and the bar
Module 26's "good enough" principle: start simple, measure, then add complexity only when the simple thing fails.
🛠 Task 3: Implement three baselines

3a. Random baseline. Recommend k random items (excluding seen). This is the absolute floor. If your model doesn't beat random, something is fundamentally wrong.

3b. Popularity baseline. Rank items by total interaction count in training data. Non-personalized. Use the evaluation harness from Lab 1. Record nDCG@10, P@10, R@10, MAP.

3c. Item-kNN with shrinkage. Build the item-item cosine similarity matrix from Module 09. Apply shrinkage (λ=100) from Module 11. Score each user by summing similarities to their history items. Record metrics. This is the classical CF baseline.

3d. EASE. Implement the 3-line EASE from Module 18. Compare to kNN. Record which wins — this tells you whether learned item-item weights outperform heuristic cosine similarity on your data.

Expected results

On a typical e-commerce dataset: Random nDCG@10 ≈ 0.001. Popularity ≈ 0.02–0.05. Item-kNN ≈ 0.04–0.08. EASE ≈ 0.05–0.10. If your baselines are significantly different, check: (a) your temporal split is correct (not random), (b) your "relevant" threshold is appropriate (rating ≥ 4 for explicit, any interaction for implicit), (c) you're excluding seen items from recommendations.

04 / BPR-MF
Train BPR-MF — the primary retrieval model
This is the model that earns its place in the stack by beating every baseline.
🛠 Task 4: Train and tune BPR-MF

4a. Implement BPR-MF. Use the from-scratch code from DEEP M16, or use PyTorch. k=128, BPR loss with negative sampling (uniform to start).

4b. Train for 20 epochs. Log training loss per epoch. Plot the loss curve. Does it converge? If not, reduce learning rate.

4c. Hyperparameter search. Try: k ∈ {64, 128, 256}, lr ∈ {0.001, 0.005, 0.01}, reg ∈ {0.001, 0.01, 0.1}. Evaluate each on a validation split (80% train / 10% val / 10% test from the training period). Pick the best by validation nDCG@10.

4d. Evaluate on test set. Train the best hyperparameters on train+val. Evaluate on the temporal test set. Compare to all baselines.

4e. Visualize embeddings. Run t-SNE on the item embeddings. Color by category. Do items from the same category cluster? If yes, the model learned meaningful latent structure. If not, investigate — the model might not be converging properly.

Pause — the moment of truth

Does BPR-MF beat EASE on your dataset? If not, what might be wrong? (Hint: review Module 18's "when each variant helps" and check your hyperparameters.)

If EASE beats BPR-MF: your MF might be under-regularized (try higher λ), under-trained (more epochs), or the data might be dense enough that item-item patterns are sufficient. Also check: is your negative sampling strategy working? Are you sampling true negatives (items not in user's history)? A common bug: accidentally sampling positive items as negatives, which corrupts the loss signal.
05 / SERVING
Build a FAISS ANN index — prove it serves in under 10ms
A trained model is useless until it can serve recommendations at production speed.
🛠 Task 5: FAISS index and latency benchmark

5a. Export item embeddings. Extract the trained item embedding matrix from BPR-MF. Shape: (n_items, 128). Normalize to unit length (for cosine similarity via inner product).

5b. Build FAISS index. Create an IVF-PQ index: faiss.IndexIVFPQ(quantizer, d=128, nlist=1024, m=32, nbits=8). Train the index on the item embeddings. Add all items.

5c. Benchmark latency. For 1000 random users: compute their user embedding, search the FAISS index for top-500 neighbors. Measure P50 and P99 latency. Target: P99 < 10ms.

5d. Benchmark recall. For 100 random users: compare FAISS top-500 with brute-force top-500. Compute Recall@500 (what fraction of true top-500 does FAISS find?). Target: Recall@500 > 95%.

5e. Demo. Build a simple function: recommend(user_id, k=10) that computes the user embedding, queries FAISS, excludes seen items, and returns top-k item IDs. Print the recommendations for 5 sample users with item names/categories.

Common FAISS mistakes

1. Forgetting to normalize embeddings before building the index (inner product ≠ cosine if unnormalized). 2. Using too few IVF clusters (nlist too small → slow search). 3. Not training the index before adding vectors (PQ quantization requires training). 4. Confusing FAISS inner product with L2 distance (use IndexFlatIP, not IndexFlatL2, for dot product search).

06 / REPORT
Assemble your evaluation report
The deliverable that proves your system works.
🛠 Task 6: Write the evaluation report

Your report should include:

6a. Dataset summary. Users, items, interactions, sparsity, temporal split dates, test set size.

6b. Leaderboard table. All models compared on nDCG@10, P@10, R@10, MAP. Include training time and serving latency for each.

6c. Analysis. Which model won? By how much? Is the improvement statistically significant? (Run the evaluation 5 times with different random seeds for negative sampling — report mean ± std.) Where does the winning model fail? (Check: cold-start users, niche categories, new items.)

6d. Embedding visualization. t-SNE plot of item embeddings colored by category. What structure did the model learn?

6e. Serving benchmark. FAISS latency (P50, P99) and Recall@500. Does it meet the 10ms / 95% targets?

6f. Recommendation for next steps. Based on your analysis, what should Kartify build next? (Ranking model? More retrieval sources? Better negative sampling?) Justify with data from your evaluation.

What a strong capstone report looks like

The best capstone reports don't just show numbers — they tell a story: "I started with popularity (nDCG 0.03), added item-kNN which doubled it (0.06), checked EASE which did better still (0.08), then BPR-MF which beat everything (0.11). The 4× improvement over popularity comes from collaborative generalization. But cold-start users (40% of my test set) still see near-popularity performance — the next step should be adding content features (M19) or SASRec for session-aware retrieval (M22). FAISS serves in 4ms P99 with 97% recall — production-ready."

After Part 1, you have a working retrieval system:

Trained BPR-MF model with tuned hyperparameters and verified convergence.
Three+ baselines (random, popularity, kNN, EASE) for rigorous comparison.
FAISS ANN index with verified latency (< 10ms P99) and recall (> 95%).
Evaluation report with leaderboard, analysis, embedding visualization, and next-step recommendations.
