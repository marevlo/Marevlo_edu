D1 / FORMALIZATION
Product Quantization — compression math
↳ corresponds to M28 S2: PQ
Real-life analogy

PQ is like describing a face using a book of noses, a book of eyes, and a book of mouths. Instead of storing the full face image (expensive), you store three page numbers: "nose #47, eyes #12, mouth #83." The codebook (collection of parts) is pre-learned. Any new face can be approximated by its three closest parts. The approximation isn't perfect, but it uses ~100× less memory. PQ does the same with embedding dimensions: split the 128-d vector into 32 sub-vectors of 4 dimensions each, and replace each sub-vector with its nearest centroid ID.

PQ memory compression
$$\text{Raw storage: } N \times d \times 4 \text{ bytes (float32)}$$ $$\text{PQ storage: } N \times m \times \lceil\log_2 k^* \rceil / 8 \text{ bytes}$$ $$\text{Compression ratio: } \frac{d \times 4}{m \times \lceil\log_2 k^*\rceil / 8}$$
$d = 128$Embedding dimension
$m = 32$Number of sub-quantizers (the vector is split into 32 sub-vectors of $d/m = 4$ dimensions each)
$k^* = 256$Codebook size per sub-quantizer (256 centroids = 8 bits per code)
CompressionRaw: $128 \times 4 = 512$ bytes. PQ: $32 \times 1 = 32$ bytes. Ratio: $16\times$ compression
D2 / WORKED
Kartify FAISS index sizing — complete calculation
↳ corresponds to M28 S6: Kartify's infra
Memory budget for 480K items × 128d
Production math
BRUTE FORCE (IndexFlatIP)
$$480{,}000 \times 128 \times 4 = 245\text{MB}$$

Fits in RAM easily. But search time = 38ms (too slow).

IVF4096-PQ32
$$\text{PQ codes: } 480{,}000 \times 32 = 15.4\text{MB}$$ $$\text{Cluster centroids: } 4096 \times 128 \times 4 = 2.1\text{MB}$$ $$\text{PQ codebooks: } 32 \times 256 \times 4 \times 4 = 0.13\text{MB}$$ $$\text{Total: } \approx 18\text{MB}$$

18MB vs 245MB raw = 13.6× memory reduction. Search with nprobe=16: scans 16/4096 = 0.4% of items = ~1900 PQ distance computations. At ~10ns per PQ distance: 0.019ms + overhead ≈ 3ms total.

HNSW32
$$\text{Vectors: } 480{,}000 \times 128 \times 4 = 245\text{MB}$$ $$\text{Graph edges: } 480{,}000 \times 32 \times 2 \times 4 \approx 123\text{MB}$$ $$\text{Total: } \approx 368\text{MB}$$

HNSW uses MORE memory than brute force (graph edges add 50%). But search is ~1ms with Recall@500 > 98%. The trade-off: memory for speed + recall.

Problem: When does brute force beat ANN?
Interview
PROBLEM

At what catalog size does ANN become necessary? If brute force dot product takes $N \times 80$ns and the budget is 10ms, what's the maximum catalog size for brute force?

SOLUTION
$$N_{\max} = \frac{10\text{ms}}{80\text{ns}} = \frac{10 \times 10^6}{80} = 125{,}000$$

For catalogs under ~125K items, brute force is fast enough. No need for FAISS, no recall loss, no index maintenance. Many startups and niche platforms have <100K items — they should use brute force and invest engineering time elsewhere. ANN becomes necessary at ~200K+ items. Kartify's 480K is firmly in ANN territory.

Problem: Embedding dimension reduction
Interview
PROBLEM

Your BPR-MF uses k=256 but serving latency is 8ms. The target is 5ms. You can reduce k to 128 (which halves search time) but nDCG drops 3%. Alternatively you can keep k=256 and use more aggressive PQ (m=16 instead of m=32). Which is better?

SOLUTION

Option A: k=128. Halves search time (8ms → 4ms ✅). But permanently loses 3% nDCG — this can never be recovered by the ranking model because the retrieval embeddings are less expressive.

Option B: PQ m=16. Each PQ distance uses 16 lookups instead of 32 → ~40% faster distance computation. Search time: ~5ms ✅. But more aggressive quantization increases approximation error → Recall@500 might drop from 97% to 92%. The ranking model can partially compensate for lower recall.

Answer: Option B is usually better. PQ recall loss is recoverable (increase nprobe slightly, or add a reranking pass on brute-force top-100 of the PQ results). Embedding dimension reduction is permanent quality loss. The principle: prefer approximate search over reduced representation quality.
