User-based kNN —
the road not taken.

Item-based kNN asks: "what items are similar to what you liked?" User-based kNN asks a different question: "what did people like you also like?" Amazon tried this first. They switched. But user-based CF isn't dead — it lives in specific niches where its unique strength matters: serendipity.

01 / THE FLIP

The original GroupLens system (1994) — one of the first recommender systems ever built — was a user-kNN system for Usenet newsgroup articles. The idea was radical for its time: instead of having an editor curate content, let users' collective behavior determine what's worth reading. "People like you found these articles interesting." Twenty years later, this principle powers every social media feed. But the user-kNN approach itself fell out of favor at scale — too many users, too volatile. This module explains why, and when user-kNN still makes sense.

Flip the perspective — find similar people, not similar items
User-kNN was the original collaborative filtering algorithm. It's conceptually simpler: find users like me, see what they liked, recommend that.

In Module 09, item-based kNN worked in two steps: (1) find items similar to what you liked, (2) recommend the most similar ones. User-based kNN flips both steps: (1) find users similar to you — based on overlapping interaction patterns, (2) recommend items those similar users liked that you haven't seen.

The intuition is natural: "People like you also liked…" We do this in real life. You ask a friend with similar taste for a restaurant recommendation. You trust their suggestion because you've agreed on restaurants before. User-based CF formalises this — it finds your algorithmic "friends" and aggregates their preferences.

Pause — think

User-based CF and item-based CF both use the same interaction matrix. The difference is which axis they compute similarity on. What do you think the practical consequences of that difference are?

The key difference: user behaviour changes constantly; item identity is stable. Every time Priya clicks something, her interaction vector changes — and so does her similarity to every other user. But the interaction vector of "Banarasi Silk Saree" only changes when a new user buys it, which is much less frequent. This makes item-item similarity cacheable (stable) and user-user similarity volatile (expensive to maintain). This single asymmetry is why industry overwhelmingly chose item-based CF.
02 / THE ALGORITHM
The user-kNN algorithm — find neighbors, aggregate their ratings
Conceptually identical to item-kNN, but operating on rows (users) instead of columns (items).
# User-based kNN prediction
def predict_user_knn(user_id, item_id, R, user_sim, k=20):
    # Find users who have interacted with this item
    rated_by = np.where(R[:, item_id] > 0)[0]

    # Get their similarity to our target user
    sims = user_sim[user_id, rated_by]
    top_k = np.argsort(sims)[-k:]
    neighbors = rated_by[top_k]
    weights = sims[top_k]

    # Weighted average of neighbor interactions
    if weights.sum() == 0:
        return 0.0
    return weights @ R[neighbors, item_id] / np.abs(weights).sum()

The structure mirrors item-kNN exactly: find k nearest neighbors (now users, not items), compute a weighted average of their interactions with the target item, and that's the predicted score. The key difference is computational: the user-user similarity matrix has dimensions (2.3M × 2.3M) for Kartify, compared to (480K × 480K) for item-item. That's 23× larger in each dimension.

03 / THE TRICK
Mean-centering — the trick that makes user-kNN work on ratings
Some users rate everything 4+. Others are harsh and rate everything 2–3. Mean-centering adjusts for this personal bias.

If User A gives everything 4–5 stars and User B gives everything 2–3 stars, their raw ratings aren't comparable. A "3" from User A is a negative signal; a "3" from User B is their average. Mean-centering subtracts each user's average rating before computing similarity and prediction:

# Mean-centered prediction
def predict_mean_centered(u, i, R, user_sim, k=20):
    mu_u = R[u][R[u] > 0].mean()        # target user's mean rating
    rated_by = np.where(R[:, i] > 0)[0]
    sims = user_sim[u, rated_by]
    top_k = np.argsort(sims)[-k:]
    neighbors = rated_by[top_k]
    weights = sims[top_k]

    # Subtract each neighbor's mean
    mu_neighbors = np.array([R[n][R[n]>0].mean() for n in neighbors])
    deviations = R[neighbors, i] - mu_neighbors

    return mu_u + weights @ deviations / np.abs(weights).sum()

This is important: the prediction is mu_u + weighted sum of deviations. We're saying "User A's rating for item X is their average plus the amount similar users deviated from THEIR averages." A harsh rater who gave a movie a 4 (when they usually give 2.5) is a strong positive signal — stronger than an easy rater who gave the same movie a 4 (when they usually give 4.2).

When this matters

Mean-centering is critical for explicit ratings (star ratings, numeric scores) where users have personal baselines. For implicit binary data (clicks — 0 or 1), there's no meaningful mean to subtract, so mean-centering isn't used. Since Kartify uses implicit data, user-kNN for Kartify wouldn't use mean-centering. But if you ever work with ratings data (Goodreads, IMDB, app store reviews), this trick is essential.

04 / THE COMPARISON
Item-based vs user-based — the full decision framework
Not a religious choice. Each wins in specific regimes. But one regime is far more common than the other.
DimensionItem-kNN (M09)User-kNN (this module)
Similarity matrixitems × items — smaller, pre-computable, cacheable for hoursusers × users — larger, volatile, changes with every click
ScalabilityBetter when |items| ≪ |users| (most consumer platforms)Better when |users| ≪ |items| (rare; some B2B, enterprise)
StabilityHigh — item relationships change slowly with new usersLow — user profiles change with every interaction
Explainability"Because you liked X, which is similar to Y" — intuitive"Because users like you also liked Y" — opaque
Cold-start: new userWorks after 1 interaction (look up that item's neighbors)Needs several interactions to find meaningful user neighbors
Cold-start: new itemNeeds interactions to build item-item similarityCan recommend to users with existing neighbors
SerendipityLower — recommends items similar to history (filter bubble risk)Higher — can surface unexpected items from diverse neighbors
The production default

Item-based is the default for almost all consumer platforms. The stability, cacheability, and scalability advantages are decisive. User-based is reserved for niche cases. When you hear "kNN-based CF" without qualification in industry, it means item-based.

05 / THE NICHE
When user-based kNN is the right choice
User-based CF isn't dead. It lives in three specific niches where its unique strength — serendipity — matters.

Niche 1: Small user bases. B2B platforms, enterprise tools, niche communities with fewer than 10,000 users. Here, user-user similarity is tractable and often more meaningful than item-item. A SaaS tool recommending features to 500 enterprise customers is a user-kNN problem.

Niche 2: Serendipity-focused surfaces. Discovery playlists, "explore" feeds, "you might not expect this but…" modules. User-based CF can surface genuinely surprising recommendations because your neighbor users have diverse tastes. If your neighbor likes both indie jazz and Korean cooking, both might surface for you — a cross-domain leap that item-kNN's conservative similarity can't make.

Niche 3: Explicit rating systems. Book clubs, wine rating communities, movie review platforms with dedicated raters. Mean-centered user-kNN leverages the fact that different users have different rating scales — something implicit-data item-kNN doesn't need to handle.

Would user-based work for Kartify?

Not as the primary CF method. Kartify has 2.3M users (too large for user-user matrix) using implicit data (no mean-centering benefit). Item-based wins on every dimension.

But: a user-based signal could serve as a secondary retrieval source specifically for Kartify's "Discover" tab — a serendipity-focused surface where surprising cross-category recommendations are the goal. In practice, this is usually done with matrix factorization (Module 14+) rather than explicit user-kNN, but the concept of "find similar users" is the same.

After this module, you have the complete neighborhood CF picture:

User-based kNN flips the item-based perspective: find similar users, aggregate their preferences.
Mean-centering adjusts for personal rating baselines — critical for explicit ratings, unnecessary for implicit data.
Item-based wins for production on stability, scalability, explainability, and cold-start handling.
User-based wins for serendipity in small communities, discovery surfaces, and explicit-rating domains.
