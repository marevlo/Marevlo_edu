01 / THE TRUTH

Kartify's product manager, Neha, calls an urgent meeting. "Homepage engagement is 1.8% CTR. The board wants 3% by Q3. What can we ship in one week?" The ML team looks at each other. One week. No training data pipeline, no model serving, no feature store. Someone says: "Sort items by purchase count. Show the top 20." Neha frowns. "That's not ML." "No," the engineer replies, "but it'll be live by Friday, and it'll beat random by 40%." That's the popularity baseline. It's not glamorous. It's not ML. But it ships in a week and it works. Everything else we build in this course gets measured against it.

The embarrassing truth about recsys baselines
In 2019, a landmark paper showed that most published neural recommenders couldn't beat well-tuned popularity baselines. This module is about building the baseline that embarrasses everyone.

In 2019, Ferrari Dacrema, Cremonesi, and Jannach published a paper at RecSys that shook the field: "Are We Really Making Much Progress?" They re-implemented 18 recently published neural recsys methods with careful hyperparameter tuning. The result: the majority did not reliably outperform well-tuned non-neural baselines — including simple popularity-weighted heuristics.

This doesn't mean neural methods don't work — several genuinely advance the state of the art, and you'll learn them in Act III. But it means: if your sophisticated model can't beat a popularity baseline, either your model has a bug, your evaluation is wrong, or the task doesn't need a sophisticated model. All three are common.

The baseline mandate

Every recsys project starts with a baseline hierarchy. Before any personalised model: global popularity (zero-personalization floor). Before any collaborative model: content-based similarity (item-features floor). Before any deep model: matrix factorization or item-kNN (classical floor). At each step, the more complex model must clear the simpler one by a meaningful margin. If it can't, ship the simpler one.

So let's build the first rung: the popularity baseline. It's four lines of code, requires zero user history, and is your permanent cold-start fallback for every version of Kartify's recsys, forever.

02 / THE VARIANTS
Four ways to count "popular" — and each gives different results
"Sort by popularity" sounds trivial. It isn't. The definition of "popular" is a design decision with real product consequences.

A popularity baseline recommends the same list to every user: the most popular items. But "popular" can mean four different things, and each gives different rankings:

Variant 1
Raw Count
score(i) = total_interactions(i)
Count all clicks/purchases ever. Simple but heavily biased toward old items that have accumulated interactions over months.
✓ One line of SQL
✗ Old items dominate forever
Variant 2
Time-Windowed
score(i) = interactions_last_N_days(i)
Count interactions in the last N days only. Balances popularity with recency. N is domain-dependent: news = 1 day, movies = 30 days, fashion = 7–14 days.
✓ Fresh content surfaces
✗ Hard boundary at day N
Variant 3
Exponential Decay
score(i) = Σ e^(−λ · age_days)
Each interaction contributes a weight that decays exponentially with age. λ controls the half-life — higher λ means faster decay. Smooth version of windowed.
✓ Smooth, tunable freshness
✗ One more hyperparameter
Variant 4
Bayesian Average
score(i) = (C·m + Σr) / (C + n)
Shrinks toward the global average for items with few interactions. Prevents a product with 2 purchases from outranking one with 500. C controls shrinkage strength.
✓ Handles low-data items fairly
✗ Needs explicit scores (ratings)
Pause — predict

Kartify is a fashion e-commerce platform. Products have seasonal relevance (summer dresses, winter coats) and trend cycles (~2 weeks). Which popularity variant would you choose for Kartify's homepage?

Exponential decay (Variant 3) is the best fit. Fashion trends move fast — a dress that went viral last week matters more than one that was popular three months ago. Time-windowed (Variant 2) would work but creates artificial cliffs at the window boundary. Exponential decay gives smooth, tunable freshness. For Kartify, λ ≈ 0.1 (half-life of ~7 days) is a reasonable starting point — tune it by offline eval against held-out conversion data.
03 / BUILD
Building Kartify's popularity baseline — four lines that actually work
Your first algorithm. No ML frameworks. No embeddings. No training. Just counting — but counting smartly.
# Kartify popularity baseline — exponential decay variant
import numpy as np
from datetime import datetime

def popularity_score(interactions, now, decay=0.1):
    """Score each item by decay-weighted interaction count."""
    scores = {}
    for item_id, timestamp in interactions:
        age_days = (now - timestamp).days
        scores[item_id] = scores.get(item_id, 0) + np.exp(-decay * age_days)
    return sorted(scores, key=scores.get, reverse=True)

# Usage: recommend top 10 to ANY user
top_10 = popularity_score(all_interactions, datetime.now())[:10]

That's it. The function takes every interaction in the system, weights each by recency, sums per item, and returns items sorted by score. Every user sees the same list. There's no personalisation, no user history, no embeddings. And yet — this already beats Kartify's current static "Trending" section, because the static section is manually curated and updated weekly. The popularity baseline updates in real time.

Kartify status update

Kartify now has its first recommender. It's not personalised — everyone sees the same list. But it's real-time (updates as clicks flow in), tunable (λ controls freshness), and reliable (no cold-start problem — new users get the same experience as returning users). This is Kartify's permanent cold-start fallback. Even when we build collaborative filtering and deep learning models, the popularity baseline will serve every new user until they generate enough history for personalisation.

04 / SURPRISE
Six situations where popularity beats ML
Popularity isn't just a baseline to beat — it's a production component. In these six regimes, it's the correct answer.

1. Cold-start users. A first-time visitor has no history. Popularity is the only thing you can show them. Every production system has this fallback.

2. Trending content. When something goes viral — a celebrity wears a Kartify dress, a product gets featured on Instagram — the popularity signal IS the recommendation signal. Personalisation would actually hurt here by not surfacing the trend.

3. Small catalogs. If you have fewer than 1,000 items, popularity-weighted ranking is often competitive with ML approaches, because there simply isn't enough diversity for personalisation to matter much.

4. High-frequency domains. News, social media, real-time content — when items have a lifespan of hours, a time-windowed popularity baseline captures the essential signal: what's hot right now.

5. Sparse user histories. When 80% of users have fewer than 5 interactions, collaborative filtering has almost nothing to work with. Popularity is often stronger than a CF model trained on thin data.

6. Category-specific pages. Within a narrow category (e.g., "gold earrings under ₹2,000"), there might only be 40 items. Popularity-within-category is a strong ranker because the browsing intent already narrows the space.

Production role

In a multi-stage pipeline (Module 04), popularity isn't a competitor to ML — it's a candidate source. The retrieval stage typically pulls candidates from multiple sources: the collaborative model, the content-based model, AND a popularity source. Popularity ensures that trending and broadly-appealing items always make the candidate pool, even if the personalised model wouldn't surface them.

05 / THE LIMIT
The limitation that creates Module 07
Popularity treats every user the same. That's its strength (simplicity, cold-start) and its fatal flaw.

Open Kartify right now. The popularity baseline shows the same 10 products to every user: the most clicked items this week. User Priya — who browses exclusively ethnic wear and jewellery — sees running shoes in position 3 because running shoes are globally popular. User Arjun — who only browses sportswear — sees silk sarees in position 5 because sarees are globally popular.

Neither user is being served. The popularity baseline knows nothing about who is looking — it knows only what's globally popular. To do better, we need to use some signal about the items themselves. If we know that Priya's past clicks are all in "ethnic wear" and "jewellery," and we know which products are ethnic wear and jewellery, we can at least filter the popular items by relevance.

That's the bridge to the next module: content-based filtering. Instead of asking "what's popular?", we ask "what's similar to what this user has already liked?" It's the first personalised algorithm — and it doesn't need any data from other users.
