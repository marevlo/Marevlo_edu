Six problems
wearing the same
name.

Two engineers at two companies both say "I built a recommender system." One built a model that predicts star ratings. The other built a conversational agent that helps users plan outfits. They solved fundamentally different problems using fundamentally different techniques. The only thing they share is the name. This confusion kills more recsys projects than bad algorithms ever will.

01 / THE TRAP

When YouTube launched recommendations in 2008, they had one problem: "what should we show next?" By 2012, they had six: scale (800M videos), cold start (400 hours uploaded per minute), sparsity (most users watch <0.001% of videos), filter bubbles (political radicalization concerns), evaluation (A/B tests showed watch-time gains but user surveys showed declining satisfaction), and privacy (congressional hearings about tracking minors). Every recommendation team discovers that one problem is actually six problems wearing a trench coat.

The most expensive word in ML: "recommend"
When your PM says "build me a recommender," they could mean six completely different things. If you build the wrong one, it doesn't matter how good your model is.

Picture this. Your PM says: "Build me a product recommender for Kartify's homepage." You nod. You spend two months building a beautiful collaborative filtering model. You evaluate on nDCG@10. You get 0.42. You ship it.

Day one: the homepage shows six similar kurtas in a row. The PM says "this looks terrible — I wanted diversity." You say "but the model is accurate." The PM says "accurate at what?"

The problem wasn't your algorithm. The problem was that "homepage recommender" is actually a slate recommendation problem — where the coordinated set matters, not just the individual items — and you built a top-N ranking model, which optimises individual items without caring about the set.

Same word. Different task. Dead project.

"Recommendation" isn't one task. It's a family of at least six distinct task patterns, each with different inputs, different outputs, different algorithms, different metrics, and different failure modes. Picking the wrong task formulation is the single most common framing error in recommender systems — more common than picking the wrong algorithm, the wrong metric, or the wrong architecture.

02 / THE SIX
Six distinct recommendation problems
Tap each task to see what it really means, where it shows up, and which algorithms serve it.
01
rating_prediction
Predict the exact score a user would give an item.
▸

The task that started the field. Given a user and an item, predict a numeric score — usually 1–5 stars. Evaluated by RMSE or MAE. This is what the Netflix Prize optimised for.

Real-world example

"Will Alice rate Inception 3.8 or 4.2?" — MovieLens, Goodreads, early Netflix.

When to use: Almost never in 2025. Most platforms don't have explicit ratings, and even when they do, the task that matters is ranking, not score prediction. Rating prediction is an Era 1/2 task that persists mainly in academic benchmarks.

Biased MFSVD++Regression models
02
top_n_ranking
Produce a ranked list of N items for a user.
▸

The workhorse task of modern recsys. Given a user and a catalog, produce an ordered list where the best items are at the top. Evaluated by nDCG@K, MAP@K, Recall@K. This is what most "recommendation" actually means.

Real-world example

"Show Priya the 20 products she's most likely to click on Kartify's homepage." — Amazon, Myntra, any e-commerce homepage feed.

When to use: When you need to fill a feed, a "recommended for you" section, or a search results page. This is the default task for most recsys projects. But beware: if the surface needs diversity or coordination (e.g., a Netflix row of tiles), top-N ranking alone isn't enough — see slate recommendation below.

BPR-MFTwo-tower + ANNNeural ranking
03
next_item_prediction
Predict what the user will interact with next, given their sequence.
▸

Sequential recommendation. The input isn't just "who is this user" but "what has this user done in order." The model predicts the next item in the sequence. This captures evolving intent — a user who just watched three action movies probably wants another action movie, not a documentary.

Real-world example

"After watching Inception, Interstellar, and Tenet, what should we recommend next?" — YouTube Up Next, Spotify's auto-queue, Netflix's "because you watched."

When to use: When temporal order matters. E-commerce sessions, music playlists, video queues, news feeds. Anywhere the user's recent actions are stronger signals than their lifetime history.

SASRecBERT4RecGRU4RecTransformers
04
slate_recommendation
Select a coordinated set of items where the group matters, not just the individuals.
▸

The crucial distinction from top-N: a slate isn't just a ranked list — it's a set where items interact with each other. A Netflix row with 8 action movies is worse than a row with 6 action movies + 1 comedy + 1 thriller, even if each action movie individually scores higher. The quality of the slate depends on the combination.

Real-world example

"Design the 10-row Netflix homepage where each row is thematic, rows don't repeat genres, and the top row maximises click probability." — Netflix homepage, Spotify Daily Mix set, Google News homepage.

When to use: When you control an entire page or a multi-slot surface. When diversity, coverage, or complementarity across items is a requirement, not just individual relevance.

DPP (Determinantal Point Processes)MMRCombinatorial bandits
05
session_continuation
Recommend within a live session as the user's intent evolves in real time.
▸

Different from next-item prediction because the focus is on anonymous or lightly-known users in real-time sessions. A first-time visitor on Kartify has no history — but after 3 clicks, you know something. After 8 clicks, you know a lot. Session-based models react to in-session behaviour without requiring historical user profiles.

Real-world example

"A user arrived from a Google ad for 'wedding lehenga.' They've now browsed 4 lehengas and 2 jewellery sets. What next?" — E-commerce browse sessions, travel booking funnels, news reading sessions.

When to use: When most users are anonymous or have thin profiles. When in-session intent is stronger than historical preference. E-commerce (especially fashion), travel, classifieds.

Session kNNGRU4RecSTAMPSR-GNN
06
conversational_recommendation
Multi-turn dialogue that elicits preferences and refines recommendations.
▸

The newest paradigm. Instead of silently inferring what the user wants, the system asks. "What's the occasion?" "What's your budget?" "Do you want something traditional or modern?" Each answer narrows the recommendation space. The "model" is an LLM agent with access to the catalog.

Real-world example

"Help me find an outfit for a friend's wedding — budget ₹5000, something Indo-western, not too flashy." — Perplexity Shopping, Amazon Rufus, personal styling assistants.

When to use: When the user's need is complex, multi-attribute, or hard to express through clicks alone. Fashion styling, travel planning, gift shopping, high-consideration purchases.

Pause & Predict

Kartify has 480K items but the average user has seen only 40. If you train a model on what users clicked, what fraction of the product space has the model NEVER observed for a typical user?

40/480,000 = 0.008% observed. The model has NEVER seen 99.992% of the catalog for this user. Every recommendation for this user is an extrapolation into unseen territory. This is why collaborative filtering is so powerful — it fills in the 99.99% gap using other users' experiences.
03 / THE STAKES
Why picking the wrong task kills the project
It's not that the algorithm fails. It's that a perfect solution to the wrong problem is still wrong.

Let's make this concrete. Your PM says "build me a product recommender for Kartify's homepage." That could mean at least four different tasks:

If the homepage has 6 slots that should be diverse (a kurta, running shoes, a bag, earrings...) — it's slate recommendation. A top-N model will fill all 6 slots with similar items because they individually score highest.

If the user just searched "running shoes" — it's session continuation. Showing them kurtas from their last month's browsing ignores the immediate intent that brought them to the page right now.

If you're building the "you might also like" widget on a product page — it's next-item prediction or complementary recommendation. The user is looking at a specific item; the system should suggest what naturally follows.

If you're building a style assistant chatbot — it's conversational recommendation. No amount of collaborative filtering can replace a dialogue that asks "what's the occasion?"

The diagnostic question

Before you write a line of code, write one sentence: "Given [this input], produce [this output], evaluated by [this metric]." If you can't fill in all three blanks, the project is pre-algorithmic — go resolve scope. If you can fill them in, the task type becomes obvious, and half the algorithm candidates are eliminated.

04 / THE ARCHITECTURE
The funnel that runs every production recsys
You can't score 10 million items in 200 milliseconds. So you don't. You build a funnel.

Here's something that trips up everyone coming from academic ML: in a research paper, the recsys scores every item for every user and picks the top-N. In production, with millions of items and a 200ms latency budget, that's physically impossible. Instead, every real system at scale uses a multi-stage funnel — each stage shrinks the candidate set while increasing the model's precision.

The multi-stage funnel — how real recsys work

Fast, cheap model scans the full catalog. Casts a wide net. Misses are expensive; false positives are cheap. Uses two-tower models, ANN indices.

This architecture is so universal that understanding it is a prerequisite for every module that follows. When you learn matrix factorization in Act II, know that MF is a retrieval-stage model. When you learn DeepFM in Act III, know that it's a ranking-stage model. When you learn MMR or DPP, know that they're re-ranking strategies. Keeping this funnel in your head prevents the most common mistake in recsys: building a model that's perfect for the wrong stage.

The key insight

Each stage optimises a different thing. Retrieval optimises recall — "don't miss good items." Ranking optimises precision — "get the order right at the top." Re-ranking optimises diversity + business value — "make the final set useful as a whole." A single model can't do all three. That's why it's a funnel, not a monolith.

Pause & Predict

You're building a system for Kartify that must handle all six problems from this module. Which single architectural decision addresses the most problems simultaneously?

The retrieval → ranking → reranking three-stage architecture (M21/M27). It addresses: scalability (retrieval narrows 480K→1000 cheaply), cold start (different retrieval sources handle different scenarios), the filter bubble (reranking enforces diversity), data sparsity (MF in retrieval handles sparsity via latent factors), and evaluation (impression logging at the ranking stage provides clean evaluation data).

06 / LANDING
What you now understand
You can now do something that eliminates half of all recsys project failures: name the problem before solving it.

After this module, you have a framework most engineers lack:

You know the six distinct recommendation tasks — rating prediction, top-N ranking, next-item, slate, session continuation, and conversational — and can identify which one applies to any given surface.
You understand why picking the wrong task kills projects more reliably than picking the wrong algorithm, and you have the one-sentence diagnostic to prevent it.
You know the multi-stage funnel — retrieval → ranking → re-ranking → serving — and can place any algorithm into the correct stage.
You've mapped Kartify's three surfaces to their correct tasks and know why one model can't serve all three.
You carry the key insight: retrieval optimises recall, ranking optimises precision, re-ranking optimises the set.
