The iceberg below
the model.

Google's famous 2015 paper "Hidden Technical Debt in Machine Learning Systems" has a diagram showing the ML model as a tiny box surrounded by a vast infrastructure: data collection, feature extraction, analysis tools, process management, serving infrastructure, monitoring. The model is maybe 5% of the system. This module is about the other 95% — the data pipelines, training automation, monitoring, and incident response that keep a recommendation system running 24/7 without a human babysitting it.

01 / THE ICEBERG
What nobody tells you about production ML
The model is the smallest component. Everything else is plumbing — and the plumbing breaks more often than the model.

A junior ML engineer spends 3 months building a beautiful DCN-v2 ranking model. nDCG improves 8% offline. They hand it to the platform team for deployment and discover: the training data pipeline takes 6 hours to run and silently drops 12% of events due to a schema mismatch. The feature store doesn't have 4 of the 33 features the model needs. The GPU serving setup costs 3× more than budgeted. The model checkpoint is 4GB and takes 45 seconds to load. The monitoring dashboard doesn't exist.

The model was the easy part. The infrastructure around it takes 10× longer to build and is where most production failures originate.

Pause — the 95/5 split

Why does the infrastructure take 95% of the effort? What makes ML infrastructure harder than regular software infrastructure?

Regular software is deterministic: the same input always produces the same output. ML systems are stochastic and data-dependent: the output changes as the training data changes, and data quality issues produce SILENT failures — the system keeps running but produces worse results. A schema change in the event logging pipeline doesn't crash the model; it corrupts the training data, which produces a slightly worse model, which degrades user experience over weeks. No error log, no crash report, no alert. By the time someone notices, the damage is done. This is why monitoring and data validation are harder in ML than in regular software.
02 / DATA PIPELINES
From raw events to training data — the three-layer pipeline
Every recommendation starts with data. If the data is wrong, everything downstream is wrong.

Layer 1: Event collection (real-time)

Every user action becomes an event: page_view, click, add_to_cart, purchase, scroll, dwell_time. These events flow through a message queue (Kafka) into a data lake (S3/GCS). Volume: Kartify generates ~500M events per day. Each event has: user_id, item_id, event_type, timestamp, session_id, device, and ~20 context fields.

What breaks: Schema changes (a new field is added, old events don't have it). Duplicate events (network retries cause the same click to be logged twice). Missing events (mobile SDK bug causes iOS users' clicks to not be logged for 3 days). Late-arriving events (a user's session data arrives 2 hours after the session ended due to network issues).

Layer 2: Feature computation (batch + streaming)

Raw events are transformed into features: "user's click count in the last 7 days," "item's conversion rate," "seller's average rating." Batch features are computed nightly in Spark. Streaming features are computed in real-time via Flink/Spark Streaming. The results are written to the feature store (Redis).

Layer 3: Training data assembly (daily)

Combine features with labels (click/purchase outcomes) to create training examples. Join user features, item features, context features, and the label into a single row. Output: a training dataset of ~50M examples (last 14 days of impression data).

Pause — the join problem

When assembling training data, you join user features at the time of impression with item features at the time of impression. Why does "at the time of impression" matter? What happens if you use current features instead?

Point-in-time correctness. If you use current features, you're leaking future information into training. Example: an item's current conversion rate is 5%, but at the time of the impression 2 weeks ago, it was 2% (the item became popular after a viral post). Training with the current 5% teaches the model "this item was always good" — but at serving time, new items won't have the future's conversion rate. The model trains on data it can never see at serving time. This is the ML equivalent of using tomorrow's stock prices to make today's trading decisions. Always use point-in-time features.
The most common data pipeline bug

Training-serving skew: the features computed during training don't match the features computed at serving time. Different code paths, different libraries, different rounding. The training pipeline uses Spark; the serving pipeline uses Python. A feature that is 3.8 in training becomes 3.79999 in serving. Over 33 features, these tiny discrepancies accumulate into meaningful prediction differences. Solution: use the SAME code for feature computation in both training and serving (the feature store pattern), or validate that training and serving features match within tolerance on a sample of live traffic.

03 / TRAINING PIPELINES
Automated daily retrain — no human in the loop
The model retrains every night. If it's good, it deploys. If it's bad, it doesn't. Automatically.
# Kartify's daily training pipeline (pseudocode)

# 01:00 — Assemble training data
training_data = assemble_training_set(
    events=last_14_days,
    features=point_in_time_features(),
    labels=['clicked', 'purchased', 'dwell_seconds']
)
validate_data(training_data, checks=[
    'row_count > 40M',         # minimum expected volume
    'null_rate < 0.01',         # feature completeness
    'label_rate_click in (0.02, 0.08)',  # label distribution sanity
    'feature_drift < 0.1',      # distribution shift from yesterday
])

# 02:00 — Train models (parallel)
bpr_mf = train_bpr_mf(training_data, warm_start=yesterday_model)  # 10 min
dcn_v2 = train_dcn_v2(training_data, warm_start=yesterday_model)  # 3 hr
sasrec = train_sasrec(session_sequences)                          # 1 hr

# 05:00 — Validate models (automatic quality gate)
for model in [bpr_mf, dcn_v2, sasrec]:
    holdout_metrics = evaluate(model, holdout_set)
    if holdout_metrics['ndcg@10'] < yesterday_metrics['ndcg@10'] * 0.95:
        alert(f'{model.name} degraded >5% — blocking deployment')
        continue  # keep yesterday's model in production
    deploy(model)  # push to serving infra

# 06:00 — Rebuild indexes
rebuild_faiss_index(bpr_mf.item_embeddings)
rebuild_faiss_index(sasrec.item_embeddings)
update_feature_store(offline_features)

The key principle: no model deploys without passing the quality gate. If today's model is worse than yesterday's (by >5% on holdout nDCG), yesterday's model stays in production. The team gets an alert. No human needs to approve a routine deployment — but a human MUST investigate a quality gate failure.

Pause — what could trigger a quality gate failure?

The daily model suddenly shows 8% lower nDCG than yesterday. No code changes. What are the most likely causes?

Data causes (most likely): A data pipeline bug dropped events for a segment of users (e.g., iOS users) → training data is incomplete → model is biased. A schema change in the event logging → a feature is suddenly all nulls → model loses signal. A promotional event (Diwali sale) created unusual interaction patterns → model overfit to sale behavior.

Infrastructure causes: The training job ran out of memory and truncated the dataset. A feature computation job failed silently and the feature store served stale features from 3 days ago. GPU memory pressure caused the training to use a smaller batch size, changing convergence behavior.
04 / MONITORING
What to monitor — the hierarchy of signals
Not everything is worth alerting on. A good monitoring system catches problems early without drowning the team in false alarms.

Tier 1: Business metrics (alert immediately)

GMV per hour, overall CTR, conversion rate, error rate. These are the business's vital signs. A 10% drop in hourly GMV warrants a wake-up call. Check every 15 minutes. Alert threshold: >15% deviation from the same hour last week.

Tier 2: Model quality signals (alert within 1 hour)

Prediction distribution (mean, std, p95 of ranking scores). If the model starts predicting all zeros or all ones, something is broken. Recommendation diversity (unique items per 1000 users). Feature freshness (timestamp of last feature store update). Check every 30 minutes.

Tier 3: Infrastructure metrics (alert within 4 hours)

P99 latency, FAISS index size, GPU memory utilization, feature store read latency, training pipeline duration. These indicate developing problems that aren't yet user-facing. Check every hour.

Tier 4: Data quality metrics (check daily)

Training data volume, feature distribution drift, label rate drift, null rates, duplicate rates. These catch slow-moving data quality issues that accumulate over days or weeks.

The metric that catches everything

Recommendation diversity (unique items recommended across all users per hour) is the single best "canary in the coal mine" for recsys systems. If it drops: the model might be broken (defaulting to popularity), a retrieval source might be down (fewer candidate sources → less diversity), or a feedback loop might be accelerating (model converging on a few popular items). A diversity drop precedes almost every other failure signal by hours. Monitor it aggressively.

05 / INCIDENTS
Incident response — what to do when it breaks at 3am
Every production recsys team will face incidents. The quality of the response determines the business impact.

3:17am. The on-call engineer's phone buzzes: "GMV per hour dropped 20% vs last week." They check the dashboard: CTR is normal, but conversion rate collapsed. They check the ranking model: prediction distribution looks normal. They check the feature store: seller_rating feature is returning 0.0 for all items (a feature store bug). The ranking model sees every seller as terrible → recommends only items with high non-seller features → users see weird recommendations → they click (CTR normal) but don't buy (CVR collapsed).

Fix: roll back the feature store to yesterday's snapshot (10 minutes). Root cause: a nightly feature computation job encountered a data type error and wrote 0.0 instead of null for the seller_rating field. Post-mortem: add a data validation check for "seller_rating mean should be between 3.0 and 5.0."

The incident playbook

Step 1: Detect (monitoring alert or user report). Confirm the problem is real, not a false alarm.

Step 2: Mitigate (stop the bleeding). Roll back to the last known good state. Don't diagnose first — mitigate first. Every minute of broken recommendations costs revenue.

Step 3: Diagnose (find the root cause). Check in order: data pipelines → feature store → model serving → infrastructure. Most incidents (>60%) are data quality issues.

Step 4: Fix (apply the permanent solution). Fix the root cause, not just the symptom.

Step 5: Post-mortem (prevent recurrence). Document what happened, why it wasn't caught earlier, and what monitoring/validation to add. Blameless — focus on systems, not people.

06 / KARTIFY
Kartify's complete operations runbook
Kartify MLOps specification

Daily pipeline schedule:
01:00 — Data assembly (Spark, 45 min)
02:00 — Model training (GPU cluster, 3 hr parallel)
05:00 — Quality gate evaluation
05:30 — Index rebuild + feature store refresh
06:00 — Shadow deployment (new model serves but doesn't display)
08:00 — Production swap (if shadow metrics are clean)

Monitoring tiers:
T1 (immediate): GMV/hr, CTR, CVR, error rate → PagerDuty
T2 (1 hr): Prediction distribution, diversity, feature freshness → Slack
T3 (4 hr): Latency P99, GPU util, index size → Dashboard
T4 (daily): Data volume, drift, null rates → Email digest

Rollback capability:
Model rollback: < 2 min (previous model checkpoint always warm)
Feature store rollback: < 10 min (daily snapshots retained 7 days)
Full system rollback: < 15 min (popularity fallback always available)

On-call rotation: 1 ML engineer, 1 platform engineer. Escalation: T1 → both paged. T2 → ML engineer only. T3/T4 → next business day.

Monitoring hierarchy: Business metrics (immediate) → model quality (1hr) → infrastructure (4hr) → data quality (daily). Diversity as the canary metric.
