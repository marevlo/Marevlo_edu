Every time you see "Customers who bought this also bought…" on any e-commerce site, you're looking at a descendant of one algorithm: item-based k-nearest neighbors. Published by Amazon in 2003, it's the most deployed recommendation algorithm in history. This module builds it from scratch for Kartify.

01 / THE TEMPLATE

The "Customers who bought this also bought" widget is the most profitable UI element in e-commerce history. Amazon reported that 35% of their revenue comes from recommendations, and the item-kNN "also bought" widget is the foundation. It was first deployed in 2003 and, two decades later, still runs on essentially the same algorithm you're about to learn. Many fancier models have been tried. Few have displaced it from this specific surface. There's a lesson in that: sometimes the simple, interpretable approach wins because it's exactly right for the job.

The "also bought" template — 23 years and counting
Linden, Smith, and York published this algorithm in 2003. Twenty-three years later, it's still a candidate source in Amazon's retrieval pipeline.
Case study · Amazon 2003
"Amazon.com Recommendations: Item-to-Item Collaborative Filtering"

Amazon's engineers tried user-based CF first. It was too slow — computing similarity between millions of users in real time was intractable. Their insight: compute item-item similarities offline, then at serving time, just look up the pre-computed neighbors of items in the user's history. The offline computation happens once (or nightly). The online lookup takes milliseconds.

This architecture — offline pre-computation of item similarities, online aggregation at serving time — became the structural template that every "also bought" and "similar items" module still follows. The algorithm itself is simple. The engineering insight that it can be pre-computed and cached is what made it serve at scale.

Twenty-three years later, item-item CF remains a candidate source in Amazon's retrieval pipeline — alongside two-tower models, GNNs, and LLM-based retrieval. It didn't get replaced. It got composed.

02 / THE ALGORITHM
Two phases — offline similarity, online scoring
The elegance of item-kNN is that the expensive computation happens once, offline. The real-time serving is a simple lookup + aggregation.
Phase 1 · Offline (runs nightly)
Build the item-item similarity matrix

For every pair of items in the catalog, compute a similarity score based on how many users interacted with both. Items that are frequently co-purchased have high similarity. Items that are never co-purchased have zero similarity. Store only the top-k most similar items for each item (k ≈ 50–200).

# Phase 1: Build item-item similarity (offline, nightly)
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity
from scipy.sparse import csr_matrix

# R: (num_users × num_items) sparse interaction matrix
# Each column = an item's interaction vector across all users
R_sparse = csr_matrix(R)
item_sim = cosine_similarity(R_sparse.T)   # (items × items)
np.fill_diagonal(item_sim, 0)               # no self-similarity

# Keep only top-k neighbors per item
k = 50
for i in range(num_items):
    top_k = np.argsort(item_sim[i])[-k:]
    mask = np.ones(num_items, dtype=bool)
    mask[top_k] = False
    item_sim[i, mask] = 0
Phase 2 · Online (runs per request, ~5ms)
Score candidate items for a specific user

For user Priya: look at every item she's interacted with. For each candidate item she hasn't seen, sum the pre-computed similarities between that candidate and her history items. High sum = candidate is similar to many things Priya liked. Rank by sum. Recommend the top ones.

# Phase 2: Score items for user (online, per request)
def recommend_item_knn(user_id, R, item_sim, top_n=10):
    user_vec = R[user_id]                        # user's interaction vector
    scores = item_sim.T @ user_vec               # weighted sum of similarities
    scores[user_vec > 0] = -np.inf              # exclude already-seen
    return np.argsort(scores)[-top_n:][::-1]
Why this is fast

Phase 1 runs once per day (or on a schedule). It's expensive — O(n² · m) for n items and m users — but it runs offline, so latency doesn't matter. Phase 2 runs per request but only involves a sparse matrix-vector multiplication — the user's interaction vector times the pre-computed similarity matrix. For a user who's interacted with 30 items and a similarity matrix truncated to k=50 neighbors per item, this is ~1,500 multiply-adds. Takes microseconds. That's why Amazon could deploy this in 2003 and still use it today.

03 / WHY ITEM-BASED
Why item-based beats user-based for production
Amazon tried user-based CF first. They switched. Here's why — and when user-based is still the right choice.
Item relationships are more stable
The similarity between "Silk Saree" and "Gold Earrings" doesn't change much as new users arrive. But the similarity between Priya and Meera changes every time either of them clicks something. Item-item similarity can be pre-computed and cached for hours or days. User-user similarity needs frequent recomputation.
Items < Users at most platforms
Kartify has 480K items and 2.3M users. An item-item similarity matrix is 480K × 480K (with k=50 truncation, very sparse). A user-user matrix would be 2.3M × 2.3M — 5× larger in each dimension. At Amazon scale (hundreds of millions of users), user-user is simply intractable.
Better explainability
"We recommend Gold Earrings because you liked the Silk Saree, and customers who buy sarees often also buy these earrings" — intuitive and trustworthy. "We recommend Gold Earrings because user #37291 is similar to you and bought them" — opaque and slightly creepy.
Works from one interaction
A user who has clicked on just ONE item can get recommendations — the algorithm looks up that item's pre-computed neighbors. User-based CF needs several interactions to find meaningful user neighbors.
Pause — think

Is there any scenario where user-based CF would be better than item-based?

Yes — two scenarios. First, serendipity: user-based CF can surface surprising recommendations because your "neighbor" users have diverse tastes. If your neighbor likes jazz AND Korean food, both might surface for you. Item-based CF tends to be more conservative — it stays close to items you've already liked. Second, small user bases: in B2B or enterprise settings where you have 500 users and 50,000 items, user-user is smaller and more tractable. But for consumer platforms, item-based is the default.
04 / KARTIFY
Building item-kNN for Kartify
Kartify's item-kNN runs nightly over 18M click events. Here's how it slots into the existing pipeline.
Kartify's retrieval pipeline — now 3 sources

Source 1 — Popularity (M06): Serves cold-start users. ~10 candidates.
Source 2 — Content-Based (M07): Serves cold-start items + tag-based personalisation. ~20 candidates.
Source 3 — Item-kNN (this module): Cross-category collaborative signal. "Also bought" and "similar items." ~30 candidates.

All three sources feed into the ranking stage. The ranker merges ~60 candidates and orders them. This is a real multi-source retrieval pipeline — the same architecture YouTube, Netflix, and Amazon use. The difference is model sophistication in each slot, not the architecture itself.

For Kartify's product page ("You might also like"), item-kNN is the primary source. When a user is viewing a Banarasi Silk Saree, the algorithm looks up the saree's top-50 pre-computed neighbors, filters out items the user has already seen, and returns the top 8. Response time: ~3ms. No model inference needed — just a lookup and a filter.

06 / WHAT'S NEXT
One critical choice remains
The algorithm works. But the similarity metric you choose in Phase 1 changes everything about what it recommends.

We used cosine similarity in the code above. But we could have used Jaccard, Pearson, conditional probability, or asymmetric cosine — and each would give different item-item similarities, different neighbor lists, and different recommendations. The next module dives into this choice, because the similarity metric matters more than the algorithm itself.
