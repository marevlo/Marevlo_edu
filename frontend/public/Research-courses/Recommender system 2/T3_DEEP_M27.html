D1 / FORMALIZATION
MMR — Maximal Marginal Relevance formalized
↳ corresponds to M27 S3: Reranking
Real-life analogy

Imagine building a music playlist for a road trip. Pure relevance: play the user's #1 favorite song on repeat — highest per-song satisfaction but terrible playlist. Pure diversity: play one song from every genre — diverse but mostly songs the user doesn't like. MMR is the DJ who plays mostly favorites but deliberately spaces them with complementary tracks: a high-energy rock song, then an acoustic breather, then another rocker. Each new song is chosen to be both good AND different from what just played.

MMR selection criterion
$$\text{MMR}(i) = \lambda \cdot \text{Rel}(i) - (1 - \lambda) \cdot \max_{j \in S} \text{Sim}(i, j)$$ $$i^* = \arg\max_{i \in C \setminus S} \text{MMR}(i)$$
$\text{Rel}(i)$Relevance score of item $i$ from the ranking model (normalized to [0,1])
$\text{Sim}(i,j)$Similarity between item $i$ and already-selected item $j$ (cosine similarity of embeddings)
$S$Set of already-selected items (starts empty, grows by 1 each step)
$C \setminus S$Remaining candidates not yet selected
$\lambda$Trade-off parameter. $\lambda = 1$: pure relevance (no diversity). $\lambda = 0$: pure diversity (no relevance). Typical: $\lambda = 0.6$–$0.8$ for e-commerce
Retrieval recall ceiling theorem
$$\text{nDCG@}k_{\text{system}} \leq \text{nDCG@}k \big|_{\text{Recall@}K = 1.0} \times \text{Recall@}K_{\text{retrieval}}$$
$K$Number of retrieval candidates (1000 for Kartify). End-to-end nDCG can NEVER exceed the retrieval recall — if a relevant item isn't in the top-K candidates, no amount of ranking quality can surface it
ImplicationIf Recall@1000 = 80%, the system's maximum achievable nDCG is 80% of perfect ranking quality. Improving the ranking model beyond this ceiling has ZERO effect. This is why retrieval recall is the first thing to optimize
D2 / WORKED
MMR on 6 items — step by step
↳ corresponds to M27 S3
Select 3 items from 6 candidates, λ=0.7
Every step
SETUP

6 candidates with relevance scores and pairwise cosine similarities:

$$\text{Rel} = [0.95, 0.90, 0.85, 0.80, 0.75, 0.70]$$ $$\text{Items: Silk Kurta, Cotton Kurta, Silk Saree, Leather Bag, Gold Earrings, Cotton Dupatta}$$ $$\text{Sim(Silk Kurta, Cotton Kurta)} = 0.92 \text{ (very similar — both kurtas)}$$ $$\text{Sim(Silk Kurta, Silk Saree)} = 0.65 \text{ (same material, different garment)}$$ $$\text{Sim(Silk Kurta, Leather Bag)} = 0.15 \text{ (completely different category)}$$
STEP 1: SELECT FIRST ITEM

$S = \emptyset$. No similarity penalty. Select by pure relevance: Silk Kurta (Rel=0.95).

$S = \{\text{Silk Kurta}\}$

STEP 2: SELECT SECOND ITEM
$$\text{Cotton Kurta: } 0.7 \times 0.90 - 0.3 \times 0.92 = 0.630 - 0.276 = 0.354$$ $$\text{Silk Saree: } 0.7 \times 0.85 - 0.3 \times 0.65 = 0.595 - 0.195 = 0.400$$ $$\text{Leather Bag: } 0.7 \times 0.80 - 0.3 \times 0.15 = 0.560 - 0.045 = \mathbf{0.515}$$ $$\text{Gold Earrings: } 0.7 \times 0.75 - 0.3 \times 0.20 = 0.525 - 0.060 = 0.465$$

Select Leather Bag (MMR=0.515). Despite being ranked 4th by relevance, it wins because it's maximally different from the selected Silk Kurta. Cotton Kurta (ranked 2nd) is penalized heavily for being too similar to Silk Kurta (Sim=0.92).

RESULT

Pure relevance would select: Silk Kurta, Cotton Kurta, Silk Saree (3 ethnic garments). MMR selects: Silk Kurta, Leather Bag, Gold Earrings (diverse categories). The user sees variety instead of a wall of similar items.

Problem: Optimal candidate count
Interview
PROBLEM

Increasing retrieval candidates from 500 to 2000 improves Recall@K from 85% to 97%, but ranking latency increases from 40ms to 160ms (linear in candidate count). Your total budget is 200ms. How do you find the optimal K?

SOLUTION
$$T_{\text{ranking}}(K) = \frac{K}{1000} \times 80\text{ms} \quad \text{(linear scaling)}$$ $$T_{\text{total}}(K) = T_{\text{retrieval}} + T_{\text{ranking}}(K) + T_{\text{rerank}} + T_{\text{network}}$$ $$200 \geq 10 + \frac{K}{1000} \times 80 + 10 + 100 \implies K \leq 1000$$

With the 200ms budget, K=1000 is the maximum. But Recall@1000 = 92% and Recall@500 = 85%. The 7% recall improvement from 500→1000 adds 40ms latency. Is it worth it? Calculate: 7% more relevant items in the candidate set → ~3-5% nDCG improvement at the output. If 3% nDCG = 1.5% GMV lift, that's ₹22K/day for Kartify. The 40ms latency cost is ~0.4% conversion loss (Amazon's 100ms rule). Net: positive. K=1000 is optimal under these constraints.

Problem: Multi-source blending weights
Interview
PROBLEM

You have 4 retrieval sources returning 500, 200, 200, 100 candidates respectively (with overlap). After dedup, you have 800 unique candidates. How would you decide the quota allocation for each source?

SOLUTION

Method 1: A/B test each source. Run the system with and without each source. Measure the nDCG drop when each source is removed. The source whose removal causes the largest drop deserves the largest quota. This directly measures marginal contribution.

Method 2: Recall contribution. For each source, compute how many of the final top-20 recommended items originated from ONLY that source (not from any other). Sources with more "unique winners" deserve larger quotas because they're finding items the other sources miss.

Method 3: Enforce minimum quotas, let ranking decide the rest. Set minimum quotas (e.g., 50 from popularity for cold-start safety, 100 from SASRec for session relevance). Fill the remaining 650 slots from BPR-MF (the strongest general-purpose source). The ranking model will sort them — the quotas just ensure each source gets representation in the candidate pool.
