Serving at scale —
the infrastructure
behind 10ms retrieval.

You've trained a BPR-MF model. You have 480,000 item embeddings. A user opens the app and you need to find the 1,000 nearest items to their embedding in under 10 milliseconds. Brute force takes 40ms — too slow. This module is about the infrastructure that makes recommendation serving possible: approximate nearest neighbor search, embedding refresh pipelines, feature stores, and the engineering that sits between "I trained a model" and "a user sees recommendations."

01 / THE PROBLEM
Why serving is harder than training
Training is batch, offline, and tolerates minutes. Serving is online, real-time, and must complete in milliseconds. Different engineering universe.

Training BPR-MF takes 40 minutes on a GPU. Nobody is waiting. The model trains overnight, embeddings are exported in the morning, and the system is updated before traffic peaks. The latency tolerance is hours.

Serving happens while a user stares at their screen. They opened the app 200ms ago. In the next 10ms, you need to: load their embedding, search 480,000 item embeddings for the nearest 1,000, and return results. If you're 50ms late, the user sees a loading spinner. If you're 500ms late, they leave.

Training is a compute problem. Serving is a latency problem. The tools, architectures, and thinking are completely different.

Pause — the brute force cost

If each dot product takes ~80 nanoseconds (128 multiplications + sum) and you have 480,000 items, how long does brute force take? Is it fast enough?

$480{,}000 \times 80\text{ns} = 38.4\text{ms}$. The retrieval budget is 10ms. Brute force is 4× too slow. And this is for ONE retrieval source — Kartify has four running in parallel. We need a sublinear search algorithm. That's what ANN (Approximate Nearest Neighbor) search provides.
02 / ANN SEARCH
Approximate Nearest Neighbor — trading exactness for speed
You don't need the exact top-1000. You need 950+ of the true top-1000, and you need them in 5ms.

The key insight behind all ANN algorithms: you can tolerate a small error in recall to get a massive speedup. Finding the exact top-1000 requires scanning all 480K items. Finding 95% of the top-1000 can be done by only scanning ~5% of items — a 20× speedup.

Three ANN families

IVF (Inverted File Index) — FAISS. Pre-cluster the item embeddings into ~1000 clusters using k-means. At query time, find the 10 nearest clusters to the user embedding (fast — only 1000 comparisons), then scan only those clusters' items (~5% of the catalog). Trade-off: more clusters → faster but lower recall. Fewer clusters → slower but higher recall.

HNSW (Hierarchical Navigable Small World). Build a multi-layer graph of item embeddings. Top layers are sparse (coarse navigation). Bottom layers are dense (fine-grained search). At query time, start at the top layer and "walk" to the nearest item, then descend to the next layer for finer resolution. Like navigating a city: first pick the neighborhood (highway), then the street (local roads), then the house (front door).

PQ (Product Quantization). Compress each 128-dimensional embedding into ~8 bytes by learning a codebook of sub-vector centroids. The approximate distance between two compressed vectors can be computed in ~10ns instead of ~80ns. Combine with IVF: IVF narrows to ~5% of items, PQ makes each comparison 8× faster. Together: 20× × 8× = 160× speedup over brute force.

Pause — recall vs speed trade-off

If your FAISS IVF-PQ index achieves Recall@1000 = 95% with a search time of 3ms, what does the 5% miss mean for the user? Should you optimize for higher recall or lower latency?

The 5% miss means ~50 of the true top-1000 items are replaced by items ranked ~1001–1050. These are very close substitutes — the ranking model will likely reorder them similarly to the exact top-1000. The user impact is negligible: maybe 1 of 20 displayed items changes. Optimizing for lower latency (→ lower recall) is almost always the right trade-off because the ranking model compensates for retrieval noise, but nothing compensates for a loading spinner.
Why the dot product constraint matters now

Module 25 asked: "why can't NCF use ANN retrieval?" Now you see why. ANN algorithms (IVF, HNSW, PQ) rely on the score being a dot product or L2 distance — geometric operations that have structure ANN can exploit (clustering, graph traversal, codebook lookup). An MLP score has no geometric structure. You can't cluster items by MLP output because the output depends on which USER is querying. The dot product constraint isn't a limitation — it's the price of admission to sublinear retrieval. Every model in the retrieval stage must produce embeddings scorable by dot product. No exceptions.

03 / REFRESH
Embedding refresh — keeping the model current
Embeddings trained yesterday don't know about today's new products, today's trending items, or the user who just signed up 5 minutes ago.

Kartify uploads 500 new products every day. Yesterday's FAISS index doesn't contain them — they're invisible to retrieval. A viral Instagram post sends 10,000 users to one product at 2pm — the model doesn't know it's trending until the next daily retrain at midnight. A new user signs up, browses 3 items, and gets popularity recommendations because their embedding doesn't exist yet.

The model is always stale. The question is: how stale can you tolerate?

Three refresh strategies

Daily batch retrain (simplest). Retrain the full model overnight. Export new embeddings. Rebuild FAISS index. Swap the old index for the new one. Total freshness lag: 12–24 hours. Good enough for stable catalogs with low item churn.

Warm-start daily retrain. Initialize today's training from yesterday's embeddings (don't start from random). This converges in 2–3 epochs instead of 20, cutting retrain time from 40min to ~10min. New items get embeddings from their first day of interactions. New users get embeddings from their first session (if they had enough interactions).

Streaming embedding updates. As new interactions arrive in real-time, update user embeddings incrementally (e.g., SASRec processes the new click and outputs an updated user state). Item embeddings are still batch-refreshed (new items join the FAISS index hourly). This gives user freshness in seconds and item freshness in hours.

Pause — the consistency problem

If user embeddings update in real-time but item embeddings update daily, the user embedding lives in a "newer" space than the item embeddings. What could go wrong?

The spaces can drift apart. If the user embedding was updated based on new items (whose embeddings are in today's space), but the FAISS index contains yesterday's item embeddings, the dot products may be miscalibrated. In practice, this drift is small for daily refreshes because embeddings change gradually. But for weekly refreshes or after major model architecture changes, the drift can be significant. Solution: ensure user and item embeddings are always from the same model version. Update both or neither.
04 / FEATURE STORES
Feature stores — serving features at ranking speed
The ranking model needs 33 features per user-item pair, for 1000 pairs, in 15ms. That's 33,000 feature lookups in 15ms.

The ranking model (DCN-v2, M21) uses features like user's average order value, item's seller rating, session click count, and days-to-festival. These features come from different sources: the user database, the product catalog, the session tracker, and a calendar. At serving time, all features must be assembled into a single feature vector fast.

A feature store pre-computes and caches feature values in a low-latency key-value store (Redis, DynamoDB). At serving time: look up user features by user_id (1 lookup, ~1ms), look up item features for 1000 items (batched, ~5ms), compute context features from the request metadata (~1ms), assemble into feature tensors (~2ms). Total: ~9ms.

Online vs offline features

Offline features change slowly: user demographics, item catalog attributes, seller ratings. Pre-computed in batch and stored in the feature store. Updated hourly or daily.

Online features change per-request: session click count, time since last click, current scroll position. Computed in real-time from the request and session state. Not stored — generated fresh each time.

Near-online features: user's click count in the last 1 hour, item's click velocity in the last 15 minutes. Computed by a streaming pipeline (Kafka → Flink/Spark Streaming → feature store). Updated every few minutes. These capture short-term trends that daily batch features miss.

Kartify's feature architecture

Offline features (21): Redis, updated nightly. User demographics, item catalog, seller stats, LLM-generated tags.
Near-online features (6): Redis, updated every 10 min via Kafka stream. User click velocity, item trending score, category engagement rates.
Online features (6): Computed at request time. Session signals, device info, time/day, request metadata.
Lookup time: User features: 0.8ms. Item features (1000 items, batched): 4ms. Context: 0.5ms. Assembly: 2ms. Total: 7.3ms.

05 / RESILIENCE
Caching, fallbacks, and graceful degradation
What happens when the model server is down? When the feature store is slow? When everything breaks at once?

Production systems fail. The FAISS server crashes. Redis returns a timeout. The ranking model GPU runs out of memory. A well-designed recommendation system degrades gracefully — never showing an empty page.

Cache the last successful result. For each user, cache their last retrieval results (TTL: 1 hour). If retrieval fails, serve the cached results. The user sees slightly stale but relevant recommendations instead of nothing.

Pre-compute popular items. Maintain a pre-computed list of the top-200 popular items, updated hourly. If ALL retrieval sources fail, serve popularity. Every user gets something reasonable.

Feature fallback chain. If the feature store is slow: try the cached value (might be stale), then try a default value (global mean for continuous, mode for categorical). The ranking model produces a slightly less accurate prediction — better than no prediction.

Pause — design the degradation ladder

Kartify's systems fail in order: (1) FAISS is down, (2) ranking model GPU timeout, (3) feature store slow, (4) everything is fine but latency exceeds 200ms. What should the system do at each level?

FAISS down: Serve cached retrieval results (last 1 hour). If no cache: serve pre-computed popularity top-200.
Ranking GPU timeout: Skip ranking. Serve retrieval results directly (ordered by retrieval score, not ranking score). Quality drops but latency is fine.
Feature store slow: Use cached/default features. Ranking quality degrades slightly but latency is maintained.
Latency > 200ms: Return partial results. If retrieval is done but ranking isn't, return retrieval-scored top-50. If retrieval is still running, return cached results. The key principle: always return something, always within the budget.
The most expensive bug in production recsys

A silent fallback to popularity. Everything appears to work — users see recommendations, pages load, no errors in the dashboard. But the personalization model is down and everyone is seeing the same 20 popular items. Revenue drops 15% before anyone notices. Detection: monitor recommendation diversity (how many unique items were recommended across all users in the last hour). If it drops below a threshold, something is wrong even if error rates are zero.

06 / KARTIFY
Kartify's complete serving infrastructure
Everything from the model to the user's screen.
Kartify serving architecture

ANN Index: FAISS IVF4096-PQ32 on 480K items × 128d. Index size: ~50MB. Rebuild: nightly. Search: 3ms for top-500.
Model Serving: DCN-v2 on NVIDIA T4 GPU. Batch inference: 80ms for 1000 candidates. 4 replicas for redundancy.
Feature Store: Redis cluster (3 shards). Offline features: nightly refresh. Near-online: 10-min Kafka stream. Total lookup: 7ms.
Caching: User result cache (Redis, TTL 1hr). Popularity fallback (pre-computed, TTL 1hr). Feature defaults (in-memory, static).
Monitoring: P99 latency, retrieval recall proxy (candidate count), recommendation diversity, model prediction distribution, feature freshness.
SLA: P99 response time < 200ms. Availability: 99.9%. Fallback to popularity within 50ms if primary path fails.

After this module, you understand recsys serving infrastructure:

ANN search: IVF (cluster + scan), HNSW (graph walk), PQ (compression). Trade exactness for 100× speedup. 95% recall is good enough.
Embedding refresh: Daily batch (simplest), warm-start (faster convergence), streaming (real-time users, batch items).
Feature stores: Offline (Redis, nightly), near-online (streaming, 10-min), online (computed at request). 33K feature lookups in 7ms.
Resilience: Cache results, popularity fallback, feature defaults, graceful degradation ladder. Never show an empty page.
The silent killer: Silent fallback to popularity — monitor recommendation diversity, not just error rates.
