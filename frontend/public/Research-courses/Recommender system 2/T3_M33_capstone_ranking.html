Build Kartify's
ranking model.

Part 1 gave you a retrieval system: BPR-MF produces 1,000 candidates via FAISS in under 10ms. But those candidates are scored by a dot product — a simple, fast, feature-blind mechanism. Now you add the ranking layer: a DeepFM model that scores each candidate using user features, item features, context, and learned feature interactions. The ranker sees what retrieval can't — and it's what determines the final 20 items the user actually sees.

01 / THE TASK
What the ranking model must do
Take 1,000 retrieval candidates. Score each with a feature-rich neural model. Return the best 50 for reranking.

Your retrieval system from Part 1 outputs 1,000 candidate items for each user. These candidates are "plausibly relevant" — their embeddings are close to the user's embedding in latent space. But "close in latent space" doesn't consider: the user's current session (they clicked 3 kurtas in the last 5 minutes), the item's price (is it in their budget?), the time of day (evening browsers prefer occasion wear), or the item's conversion history (does this product actually sell, or just get clicked?).

The ranking model ingests ALL of this information — retrieval score, user features, item features, context — and produces a refined score that predicts the actual probability of a meaningful interaction (click, purchase, or engagement). It transforms "probably relevant" into "here's exactly what you should see."

🛠 Task 1: Define your ranking objective

Before writing any code, answer: what is your ranking model predicting? Options from Module 21:

Option A: Click prediction. Target = 1 if user clicked, 0 otherwise. Simple. Lots of training data. But optimizes for engagement, not value.

Option B: Purchase prediction. Target = 1 if user purchased, 0 otherwise. Less data (purchases are rare). But directly optimizes for revenue.

Option C: Multi-task. Predict BOTH click and purchase simultaneously (Module 21's ESMM approach). Combined score: $\text{score} = 0.3 \times P(\text{click}) + 1.0 \times P(\text{click}) \times P(\text{purchase})$.

Recommendation: Start with Option A (click prediction) for your first iteration. It has the most training data and fastest feedback loop. Add purchase prediction in a second iteration once the pipeline is working.

02 / FEATURES
Feature engineering — the ranking model's superpower
The retrieval model uses only IDs. The ranking model uses everything.
🛠 Task 2: Build your feature set

Implement at least 15 features across three categories. For each feature, specify: name, type (categorical/continuous), source (catalog/computed/context), and how to compute it.

User features (minimum 5):

• user_id (categorical, embedding) — the learned user representation
• user_interaction_count (continuous) — total historical interactions
• user_avg_price (continuous) — average price of items they've interacted with
• user_top_category (categorical) — most frequent category in their history
• user_days_since_last_active (continuous) — recency signal

Item features (minimum 5):

• item_id (categorical, embedding) — the learned item representation
• item_category (categorical) — product category
• item_price_bin (categorical) — price bucket (budget/mid/premium)
• item_popularity_7d (continuous) — interactions in last 7 days
• item_ctr_30d (continuous) — historical click-through rate

Context & cross features (minimum 5):

• hour_of_day (categorical) — current hour
• is_weekend (binary) — weekend vs weekday
• retrieval_score (continuous) — the BPR-MF dot product score from retrieval
• retrieval_source_count (continuous) — how many retrieval sources surfaced this item
• price_match (continuous) — |item_price - user_avg_price| / user_avg_price

Pause — which feature matters most?

Before training the model, which of the 15 features above do you think will have the highest feature importance? Why?

On most e-commerce datasets, retrieval_score (the BPR-MF dot product) and item_popularity_7d dominate. The retrieval score is already a good relevance signal — the ranker's job is to REFINE it, not replace it. Item popularity captures a strong global quality signal. The cross-feature price_match (does this item's price match the user's typical spending?) is often the strongest personalization signal beyond the retrieval score. If retrieval_score alone achieves 90% of the ranker's performance, the ranker's marginal value is in the other 14 features — especially context and cross-features that retrieval can't see.
03 / BUILD
Implement DeepFM — the ranking architecture
FM for automatic pairwise interactions + MLP for higher-order patterns. Module 20's architecture, now you build it.
🛠 Task 3: Implement DeepFM

3a. Define the architecture. Use Module 20's DeepFM structure: shared embeddings (dim=16 per sparse feature), FM path for pairwise interactions, Deep path (MLP [256, 128, 64]), sigmoid output. Use the from-scratch structure from DEEP M20 or a library like DeepCTR/PyTorch.

3b. Prepare training data. Generate impression-level training examples: for each user in the training period, take their retrieval candidates (from BPR-MF Part 1) and label: clicked = 1, not clicked = 0. Join with all features from Task 2. This creates a dataset of (features, label) pairs.

3c. Handle class imbalance. Typical CTR is 2–5% → 95%+ negative examples. Options: downsample negatives to 1:3 ratio, use focal loss (weight hard negatives more), or use standard BCE with gradient accumulation. Start with downsampling — simplest.

3d. Train. Binary cross-entropy loss, Adam optimizer (lr=1e-3), batch size 4096. Train for 5 epochs on the training period. Monitor validation AUC and log-loss per epoch. If validation loss increases, reduce learning rate or add dropout (0.2).

The training data trap

Your ranking model trains on impression data — items that the retrieval system CHOSE to show. This means the model never sees items that retrieval filtered out. At serving time, the model must score retrieval candidates that might look different from training data (if the retrieval model changes). Solution: periodically inject random candidates into training data to expose the ranker to the full item distribution. This is the training-serving skew from Module 31, applied to the ranking stage.

04 / EVALUATE
Evaluate the ranker — does it beat retrieval-only?
The ranker must justify its existence. It adds 80ms of latency and significant complexity. Does the quality improvement warrant it?
🛠 Task 4: Evaluate end-to-end

4a. Retrieval-only baseline. Score all test users by retrieval score only (BPR-MF dot product). Compute nDCG@10, P@10, R@10, MAP. This is the "no ranker" baseline — the system from Part 1.

4b. Retrieval + ranking. For each test user: run retrieval (top-1000 candidates), then score all 1000 with the DeepFM ranker, re-order by ranking score. Compute the same metrics on the ranker's top-10.

4c. Compare. Does the ranker improve nDCG@10 over retrieval-only? By how much? Is the improvement statistically significant (paired t-test, p < 0.05)?

4d. Feature importance. Compute permutation importance: for each feature, shuffle it and measure the nDCG drop. Which features contribute most? Does the ranking match your prediction from Task 2's Pause & Predict?

4e. Error analysis. Find 10 test users where the ranker improved over retrieval, and 10 where it made things worse. Examine: what features drove the change? Are there systematic patterns (cold-start users, niche categories, price mismatches)?

Pause — the expected lift

Based on the model comparison table from Module 26, how much nDCG improvement should you expect from adding a DeepFM ranker on top of BPR-MF retrieval?

Module 26's table shows: BPR-MF retrieval alone: +25–40% over popularity. DeepFM ranking on top: typically adds another +5–15% relative over retrieval-only. The gain is smaller than adding MF to the first place because the retrieval model already captures the dominant collaborative signal. The ranker's value comes from feature interactions (price × user budget, category × time of day) that retrieval can't model. If you see < 3% improvement, check: are your features informative? Is the training data sufficient? Is the model converging?
05 / PIPELINE
Wire the end-to-end pipeline — retrieval → ranking → output
The complete recommendation function: user opens app → 20 items appear.
🛠 Task 5: Build the integrated pipeline

5a. Implement the end-to-end function:

def recommend_kartify(user_id, k=20):
    """Full pipeline: retrieval → ranking → output."""
    # Stage 1: Retrieval (your Part 1 code)
    user_emb = bpr_model.get_user_embedding(user_id)
    candidates = faiss_index.search(user_emb, n=1000)

    # Stage 2: Feature extraction
    features = extract_features(user_id, candidates)

    # Stage 3: Ranking
    scores = deepfm_model.predict(features)
    top_50 = candidates[np.argsort(scores)[-50:][::-1]]

    # Stage 4: Simple reranking (MMR diversity)
    final_items = mmr_rerank(top_50, item_embeddings, k=k, lam=0.7)

    return final_items
🛠 Task 5 (continued)

5b. Benchmark end-to-end latency. For 100 test users, measure total time from user_id input to final 20 items output. Break down: retrieval time, feature extraction time, ranking time, reranking time. Does the total stay under 200ms?

5c. Demonstrate for 5 sample users. Print the final 20 recommended items with: item name, category, price, ranking score, and retrieval source. Does the output look reasonable? Show to a non-technical person and ask: "would you click on these?"
