The full stack —
retrieval, ranking,
reranking.

Acts I–IV built the models. Act V ships them. This module describes the complete multi-stage recommendation architecture that every major platform uses — from the moment a user opens the app to the moment 20 items appear on their screen. Every stage has a job, a latency budget, and a failure mode. Understanding the full stack is what separates someone who "knows recsys algorithms" from someone who "can build a recommendation system."

01 / THE FUNNEL

Pinterest's recommendation team once ran an experiment where they doubled the number of retrieval candidates from 1000 to 2000. They expected a quality improvement — more candidates means the ranking model has a larger pool to choose from. Instead, latency spiked from 150ms to 280ms, P99 errors increased 3×, and the ranking model's GPU ran out of memory during peak traffic. The improvement in nDCG@10 was 0.3%. They rolled back within hours. The lesson: in production, every decision is a system decision, not a model decision. This module teaches you to think about the system.

480,000 → 1,000 → 50 → 20 — the recommendation funnel
Each stage reduces the candidate set while increasing the quality of scoring. The funnel is not optional — it's mathematically necessary.
# The recommendation funnel — what happens in 200ms

# STAGE 0: User opens app (t=0ms)
catalog_size = 480_000          # total items

# STAGE 1: RETRIEVAL (t=0 → t=10ms)
# 4 parallel retrieval sources, each returns candidates
popularity_candidates    = popular_items(n=200)              # 0.5ms
bpr_mf_candidates       = faiss_ann(user_emb, n=500)        # 5ms
sasrec_candidates       = faiss_ann(session_emb, n=200)     # 5ms
lightgcn_candidates     = faiss_ann(graph_emb, n=100)       # 5ms
# Run in PARALLEL → max 5ms, not sum
candidates = dedup_merge([popularity, bpr, sasrec, lightgcn])  # ~1000 unique

# STAGE 2: RANKING (t=10 → t=90ms)
features = extract_features(user, candidates, context)       # 15ms
scores = dcn_v2_multitask(features)                          # 55ms
top_50 = sort_by_combined_score(scores, n=50)               # 5ms

# STAGE 3: RERANKING (t=90 → t=100ms)
diversified = mmr_diversity(top_50, lambda_=0.3, n=20)      # 3ms
business_adjusted = apply_business_rules(diversified)          # 2ms
positioned = assign_positions(business_adjusted, layout)       # 1ms

# User sees 20 items at t≈100ms + network latency
Pause & Predict

The three-stage funnel narrows 480K → 1000 → 50 → 20. What happens if you skip the ranking stage and go directly from retrieval (1000) to reranking (20)?

You lose the feature-rich scoring that distinguishes good candidates from great ones. Retrieval scores are dot products — they capture latent preference but not context (time of day, session behavior, price sensitivity). Going from 1000 to 20 using only retrieval scores means the final 20 items are selected based on coarse similarity, not fine-grained relevance. In practice, this costs 5-15% nDCG — significant for a mature platform. However, for an MVP or a small team, retrieval-only with MMR reranking is a completely valid Phase 1 deployment. Many successful systems ran retrieval-only for months before adding a ranking layer.
02 / CANDIDATE MANAGEMENT
Overlap, dedup, and blending — the merge problem
Four retrieval sources produce overlapping candidate sets. How to combine them is a decision that affects diversity and quality.

Each retrieval source has a different strength: popularity catches trending items. BPR-MF captures long-term taste. SASRec captures current session intent. LightGCN captures graph-structural patterns. They overlap partially — maybe 30% of BPR-MF's top-500 also appear in SASRec's top-200.

The merge strategy

Step 1: Dedup. Union all candidates. Remove duplicates, keeping the highest retrieval score per item.

Step 2: Source tagging. Tag each candidate with its source(s). An item retrieved by 3 of 4 sources is likely more relevant than an item from 1 source — this "retrieval agreement" signal can be used as a feature in the ranking model.

Step 3: Quota enforcement. Ensure minimum representation from each source: at least 50 items from popularity (prevents cold-start failure), at least 100 from SASRec (preserves session relevance), etc. Without quotas, BPR-MF dominates the candidate set and session/graph signals get drowned.

Source agreement as a ranking feature

An item retrieved by 3 of 4 sources is strong evidence of relevance — three independent models agree. Feed the number of retrieval sources (and which ones) as features to the ranking model. This lets the ranker learn "items from multiple sources deserve a boost" without hard-coding the logic. In Kartify's DCN-v2, the "num_retrieval_sources" feature consistently ranks in the top 10 most important features by permutation importance.

03 / RERANKING
The final 50 → 20 — diversity, freshness, and business rules
The ranking model optimizes for relevance. But a page of 20 items that are all the same category is a bad user experience. Reranking fixes the slate.

MMR: Maximal Marginal Relevance

MMR (Carbonell & Goldstein, 1998) balances relevance and diversity. At each step, select the item that maximizes:

# MMR: select items one by one
selected = []
for _ in range(20):
    best_score = -inf
    for item in remaining:
        relevance = ranking_score[item]
        max_similarity = max(cosine(item_emb[item], item_emb[s]) for s in selected)
        mmr = lambda_ * relevance - (1 - lambda_) * max_similarity
        if mmr > best_score:
            best_score, best_item = mmr, item
    selected.append(best_item)

# λ=1.0: pure relevance (no diversity). λ=0.0: pure diversity (no relevance).
# Kartify uses λ=0.7: mostly relevant, but penalize items too similar to already-selected ones.

Business rules

Boost new items: Items listed in the last 48 hours get a score multiplier (1.2×) to ensure catalog freshness. Without this, new items are invisible until they accumulate enough interactions for the retrieval models to discover them.

Suppress low-stock: Items with < 5 units remaining get demoted (0.5× multiplier) to avoid showing items that will be out-of-stock by the time the user clicks.

Category diversity floor: The final 20 items must include at least 3 different categories. If the ranking model produces 20 kurtas, swap the lowest-ranked duplicates for the highest-ranked items from other categories.

Sponsored insertion: If the business has sponsored placements, insert them at designated positions (e.g., positions 3 and 8) and shift organic items down. Mark sponsored items clearly in the UI.

Reranking changes what users see — measure it

Every reranking rule changes the final slate. The ranking model says "show items A, B, C." Reranking might swap C for D (for diversity). This means you're overriding a model decision with a heuristic. ALWAYS A/B test reranking rules against the pure ranking output. Sometimes "improve diversity" hurts conversion (users wanted 20 kurtas). Sometimes it helps (users discover new categories). Only metrics tell you which.

Pause & Predict

MMR uses λ=0.7 to balance relevance and diversity. What observable user behavior would tell you λ is set too high (too much relevance) vs too low (too much diversity)?

λ too high (0.9+): Users scroll less (everything looks the same → they either click the first item or leave). Category distribution of clicked items becomes very narrow. Repeat visit rate stays flat or declines (users feel 'the platform doesn't show me anything new'). λ too low (0.3): Position-1 CTR drops (the 'best' item is displaced by a diverse-but-irrelevant one). Users click items deeper in the list (they're searching for what they actually want among the diverse filler). Conversion rate drops even if CTR stays stable (diverse items get clicked out of curiosity, not purchase intent). The optimal λ is found by A/B testing — not offline metrics.
04 / FAILURE MODES
What breaks — and how to detect it
Every stage can fail silently. Monitoring is not optional.

Retrieval failure: A retrieval source returns empty (FAISS index corrupted, model serving crash). The candidate set shrinks. nDCG drops but no alarm fires because the system still returns results — just fewer and worse ones. Monitor: candidate count per source. Alert if any source drops below 50% of expected count.

Ranking stale features: The feature store hasn't updated in 6 hours. The ranking model uses stale session features, user features. Predictions degrade gradually. Monitor: feature freshness timestamps. Alert if any critical feature is > 1 hour stale.

Reranking over-diversification: The MMR λ is set too low (0.3), producing a diverse but irrelevant slate. Users don't click because the items are interesting but not what they want. Monitor: position-weighted CTR. If position 1 CTR drops while position 10 CTR stays flat, the top items are being displaced by diversity.

The funnel: 480K → 1000 → 50 → 20. Each stage narrows while increasing scoring quality.
