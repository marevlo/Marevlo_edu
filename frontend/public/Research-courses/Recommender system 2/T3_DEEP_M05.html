D1 / FORMALIZATION
Composite objectives with guardrail constraints
↳ corresponds to M05 S3: The Goodhart Trap
Real-life analogy

You're a coach training an athlete. Your primary goal: maximize sprint speed. But you have guardrails: injury rate must stay below 2%, training load must not exceed 20 hours/week, and sleep quality must not drop below 7/10. You can't just maximize speed blindly — you need to stay within the guardrails. Composite objectives in recsys work the same way: optimize the primary metric, but STOP if guardrails break.

Constrained optimization formulation
$$\max_{\pi} \quad \mathcal{J}(\pi) = w_1 \cdot \text{CVR}(\pi) + w_2 \cdot \text{Diversity}(\pi) + w_3 \cdot \text{NewItemExposure}(\pi)$$ $$\text{subject to:}$$ $$g_1(\pi): \text{ReturnRate}(\pi) \leq 0.08 \quad \text{(must stay under 8%)}$$ $$g_2(\pi): \text{CategoryGini}(\pi) \leq 0.65 \quad \text{(diversity floor)}$$ $$g_3(\pi): \text{AvgItemAge}(\pi) \leq 14 \text{ days} \quad \text{(freshness)}$$
$\pi$The recommendation policy — the function that maps each (user, context) to a ranked list
$\text{CVR}(\pi)$Conversion rate — fraction of recommended items that get purchased. Kartify's primary metric
$w_1, w_2, w_3$Blend weights. Kartify: $w_1 = 0.7, w_2 = 0.2, w_3 = 0.1$. These are tuned by product team, not learned
$g_k(\pi) \leq \tau_k$Guardrails — hard constraints that must hold. A model that improves CVR but breaks a guardrail is rejected, no matter how good the CVR gain
CategoryGiniGini coefficient of category exposure. 0 = perfectly uniform, 1 = one category dominates. Lower is more diverse

In practice, the guardrails are checked during A/B testing, not during training. The model optimizes the primary objective during training; the guardrails are verified post-hoc. If a guardrail is violated, the model is not shipped — regardless of primary metric improvement.

D2 / WORKED
All ranking metrics — computed by hand
↳ corresponds to M05 S4: The Offline-Online Gap
Real-life analogy

Imagine grading a student's exam answers. Precision: "of the answers they wrote, how many were correct?" Recall: "of all the correct answers possible, how many did they find?" AP: "did they write correct answers EARLY, or were they scattered among wrong ones?" nDCG: "did they put the BEST answers first, where they matter most?"

All four metrics on one example — every arithmetic step
Complete workout
SETUP

User has 4 relevant items: $\mathcal{I}^+ = \{A, C, F, H\}$. Total catalog: 1000 items.

Model's top-5 ranking: $[C, X, A, Y, F]$ where $X, Y$ are irrelevant.

Relevance vector: $\text{rel} = [1, 0, 1, 0, 1]$

PRECISION@5
$$\text{Precision@5} = \frac{\text{relevant items in top-5}}{5} = \frac{|\{C, A, F\}|}{5} = \frac{3}{5} = 0.600$$

60% of the top-5 were relevant. Simple ratio — no position awareness.

RECALL@5
$$\text{Recall@5} = \frac{\text{relevant items found in top-5}}{|\mathcal{I}^+|} = \frac{3}{4} = 0.750$$

Found 3 of 4 relevant items. Missed item $H$. If $H$ was critical (e.g., the user's favorite product), this is a problem.

AP@5 (AVERAGE PRECISION)

Compute precision at each relevant hit position:

$$\text{Pos 1 (C, ✓):} \quad P@1 = \frac{1}{1} = 1.000$$ $$\text{Pos 2 (X, ✗):} \quad \text{skip — not relevant}$$ $$\text{Pos 3 (A, ✓):} \quad P@3 = \frac{2}{3} = 0.667$$ $$\text{Pos 4 (Y, ✗):} \quad \text{skip}$$ $$\text{Pos 5 (F, ✓):} \quad P@5 = \frac{3}{5} = 0.600$$
$$\text{AP@5} = \frac{1}{|\mathcal{I}^+|} \sum_{k: \text{rel}_k = 1} P@k = \frac{1}{4}\left(1.000 + 0.667 + 0.600\right) = \frac{2.267}{4} = 0.567$$

Crucial: divide by 4 (total relevant), not 3 (found). The missed item $H$ penalizes AP — it's as if $H$ was ranked at position $\infty$ with $P = 0$.

nDCG@5 — FULL DERIVATION

Step 1: DCG@5 (Discounted Cumulative Gain)

$$\text{DCG@5} = \sum_{k=1}^{5} \frac{\text{rel}_k}{\log_2(k+1)}$$ $$= \frac{1}{\log_2 2} + \frac{0}{\log_2 3} + \frac{1}{\log_2 4} + \frac{0}{\log_2 5} + \frac{1}{\log_2 6}$$ $$= \frac{1}{1.000} + 0 + \frac{1}{2.000} + 0 + \frac{1}{2.585}$$ $$= 1.000 + 0 + 0.500 + 0 + 0.387 = 1.887$$

Step 2: IDCG@5 (Ideal — all relevant items at the top)

Ideal ranking: $[A, C, F, H, \text{irrelevant}]$ → rel = $[1, 1, 1, 1, 0]$

$$\text{IDCG@5} = \frac{1}{\log_2 2} + \frac{1}{\log_2 3} + \frac{1}{\log_2 4} + \frac{1}{\log_2 5} + \frac{0}{\log_2 6}$$ $$= 1.000 + 0.631 + 0.500 + 0.431 + 0 = 2.562$$

Step 3: Normalize

$$\text{nDCG@5} = \frac{\text{DCG@5}}{\text{IDCG@5}} = \frac{1.887}{2.562} = 0.737$$

73.7% of ideal. The irrelevant items at positions 2 and 4 hurt — they push relevant items $A$ and $F$ to later positions where the logarithmic discount reduces their contribution.

D3 / WORKED
A/B test power calculation — Kartify's eval plan
↳ corresponds to M05 S5 Q10: How Will I Evaluate?
Real-life analogy

Imagine you're flipping two coins to check if one is biased. Coin A lands heads 50% of the time. You suspect Coin B lands heads 52%. How many flips do you need to detect that 2% difference? If you flip 10 times, random noise overwhelms the signal. If you flip 10,000 times, the difference becomes clear. Power calculation tells you the minimum number of flips (samples) to detect a given effect size with statistical confidence.

Sample size formula for proportions (two-sided test)
$$n = \frac{(z_{\alpha/2} + z_\beta)^2 \cdot (p_1(1-p_1) + p_2(1-p_2))}{(p_2 - p_1)^2}$$
$n$Required sample size PER ARM (control and treatment each need $n$ sessions)
$p_1$Baseline conversion rate (current system). Kartify: $p_1 = 0.021$ (2.1%)
$p_2$Expected conversion rate under new model. Target: $p_2 = 0.026$ (2.6%, a +0.5pp lift)
$z_{\alpha/2}$Z-score for significance level. At $\alpha = 0.05$ (95% confidence): $z_{0.025} = 1.96$
$z_\beta$Z-score for statistical power. At 80% power ($\beta = 0.20$): $z_{0.20} = 0.842$
Worked: How many sessions does Kartify's A/B test need?
Step by step
INPUTS
$$p_1 = 0.021 \quad (\text{current CVR = 2.1\%})$$ $$p_2 = 0.026 \quad (\text{target CVR = 2.6\%, +0.5pp minimum detectable effect})$$ $$\alpha = 0.05 \Rightarrow z_{\alpha/2} = 1.96$$ $$\text{power} = 0.80 \Rightarrow z_\beta = 0.842$$
COMPUTE
$$\text{Numerator} = (1.96 + 0.842)^2 \times (0.021 \times 0.979 + 0.026 \times 0.974)$$ $$= (2.802)^2 \times (0.02056 + 0.02532)$$ $$= 7.851 \times 0.04588 = 0.3602$$
$$\text{Denominator} = (0.026 - 0.021)^2 = (0.005)^2 = 0.000025$$
$$n = \frac{0.3602}{0.000025} = 14{,}408 \text{ sessions per arm}$$

Kartify needs ~14,400 sessions per arm (28,800 total). At Kartify's traffic of ~2.5M sessions/month, a 10% traffic allocation gives 250K sessions/month per arm. The test would reach significance in about 2 days. However, running for 14 days captures weekly patterns (weekend vs weekday behavior).

INTERPRETATION

If the new model truly lifts CVR by 0.5 percentage points, a 14-day test at 10% traffic has 80% probability of detecting it. If the test shows no significant difference, it means either: (a) the true effect is smaller than 0.5pp, or (b) you were unlucky (20% chance with 80% power).

Module 05's Kartify brief used "200K sessions per arm" — that's more conservative, giving power to detect even smaller effects (~0.3pp) or higher confidence (90%+ power at 0.5pp).

D4 / FROM SCRATCH
From-scratch evaluation harness — reuse throughout the track
↳ supports M05 S6: Act I Complete — the evaluation code for all future modules
# ═══ EVALUATION HARNESS — reuse in every subsequent deep dive ═══
import numpy as np

def precision_at_k(ranked_list, relevant_set, k):
    """Fraction of top-k items that are relevant."""
    hits = len(set(ranked_list[:k]) & relevant_set)
    return hits / k

def recall_at_k(ranked_list, relevant_set, k):
    """Fraction of relevant items found in top-k."""
    hits = len(set(ranked_list[:k]) & relevant_set)
    return hits / max(len(relevant_set), 1)

def ap_at_k(ranked_list, relevant_set, k):
    """Average Precision at k. Rewards relevant items appearing early."""
    hits, sum_prec = 0, 0.0
    for i, item in enumerate(ranked_list[:k]):
        if item in relevant_set:
            hits += 1
            sum_prec += hits / (i + 1)  # precision at this position
    return sum_prec / max(len(relevant_set), 1)

def ndcg_at_k(ranked_list, relevant_set, k):
    """nDCG@k — penalizes relevant items ranked lower."""
    dcg = sum(1.0/np.log2(i+2) for i, item in enumerate(ranked_list[:k]) if item in relevant_set)
    idcg = sum(1.0/np.log2(i+2) for i in range(min(k, len(relevant_set))))
    return dcg / idcg if idcg > 0 else 0.0

def evaluate_model(model, test_data, k=10):
    """Evaluate model across all test users. Returns dict of mean metrics."""
    results = {'ndcg': [], 'precision': [], 'recall': [], 'map': []}
    for user_id, relevant_items in test_data.items():
        ranked = model.recommend(user_id, n=k)
        rel = set(relevant_items)
        results['ndcg'].append(ndcg_at_k(ranked, rel, k))
        results['precision'].append(precision_at_k(ranked, rel, k))
        results['recall'].append(recall_at_k(ranked, rel, k))
        results['map'].append(ap_at_k(ranked, rel, k))
    return {m: np.mean(v) for m, v in results.items()}

# ═══ TRAIN-TEST SPLIT for recsys ═══
def temporal_split(interactions, test_days=30):
    """Split by time: last N days → test, rest → train."""
    cutoff = interactions.timestamp.max() - pd.Timedelta(days=test_days)
    train = interactions[interactions.timestamp < cutoff]
    test = interactions[interactions.timestamp >= cutoff]
    # Build test_data: {user_id: [list of items they interacted with in test period]}
    test_data = test.groupby('user_id')['item_id'].apply(list).to_dict()
    return train, test_data
Why temporal split, not random split

Random train-test splits are WRONG for recsys. If you randomly split interactions, the test set contains future AND past events — the model trains on data from October and is tested on data from September. This leaks future information. Always split by time: train on everything before date $T$, test on everything after $T$. This simulates the real deployment scenario: the model is trained on past data and serves future users.

Interview problems — metrics & evaluation

Problem 1: Compute MAP@3 for two users
Interview
PROBLEM

User 1: relevant = $\{A, B\}$, model ranking = $[B, X, A]$. User 2: relevant = $\{C\}$, model ranking = $[X, C, Y]$. Compute MAP@3.

SOLUTION
$$\text{User 1: AP@3} = \frac{1}{2}\left(\frac{1}{1} + \frac{2}{3}\right) = \frac{1}{2}(1.667) = 0.833$$ $$\text{User 2: AP@3} = \frac{1}{1}\left(\frac{1}{2}\right) = 0.500$$ $$\text{MAP@3} = \frac{0.833 + 0.500}{2} = 0.667$$
Problem 2: A/B test design
Interview
PROBLEM

Your platform has 500K daily sessions and current CTR = 10%. You want to detect a 0.3pp lift with 95% confidence and 80% power. How many days should the A/B test run with 50/50 traffic split?

SOLUTION
$$n = \frac{(1.96 + 0.842)^2 \times (0.10 \times 0.90 + 0.103 \times 0.897)}{(0.003)^2}$$ $$= \frac{7.851 \times (0.09 + 0.0924)}{0.000009} = \frac{7.851 \times 0.1824}{0.000009} = \frac{1.432}{0.000009} = 159{,}111$$

~159K sessions per arm. With 50/50 split on 500K daily sessions = 250K per arm per day. Test reaches required sample size in 1 day, but run for 7+ days to capture weekly patterns.

Problem 3: Guardrail violation
Interview
PROBLEM

Your A/B test shows: CVR +0.8pp (statistically significant, $p < 0.01$). But the return rate guardrail went from 5.2% to 6.8% ($p < 0.05$). Should you ship?

SOLUTION

No. The guardrail violation (return rate +1.6pp) suggests the model is driving impulse purchases that users regret. The CVR gain is real but unsustainable — higher returns erode customer trust, increase logistics costs, and damage long-term retention. The model is Goodharting on conversion by recommending items that are easy to click-and-buy but not genuinely satisfying. Ship only after investigating and fixing the return rate regression, even if it means sacrificing some of the CVR gain.
