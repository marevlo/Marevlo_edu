NeuMF formalized —
GMF, MLP, and why the dot product fights back.

Module 25 introduced NCF and its controversy. This companion formalizes the NeuMF architecture, proves that GMF generalizes MF, shows why MLP on concatenated embeddings is theoretically more expressive but practically harder to train, and provides a from-scratch PyTorch implementation.

D1 / FORMALIZATION
NeuMF — two-path architecture formalized
↳ corresponds to M25 S2: NeuMF
NeuMF prediction
$$\hat{y}_{ui} = \sigma\left(\mathbf{h}^\top \begin{bmatrix} \underbrace{\mathbf{p}_u^G \odot \mathbf{q}_i^G}_{\text{GMF path}} \\ \underbrace{\phi_L(\ldots\phi_2(\phi_1([\mathbf{p}_u^M; \mathbf{q}_i^M])))}_{\text{MLP path}} \end{bmatrix}\right)$$
$\mathbf{p}_u^G, \mathbf{q}_i^G$GMF embeddings (separate from MLP). Dimension $k_G$, typically 64
$\odot$Element-wise (Hadamard) product. Output is a $k_G$-dimensional vector where each dimension captures one latent factor's interaction independently. When followed by a sum (via $\mathbf{h}$), this reduces to the dot product. With learned $\mathbf{h}$ weights, it generalizes the dot product by weighting each dimension differently
$\mathbf{p}_u^M, \mathbf{q}_i^M$MLP embeddings (separate). Dimension $k_M$, typically 32
$[\cdot; \cdot]$Concatenation → $2k_M$-dimensional input to the MLP
$\phi_l$MLP layer: $\phi_l(\mathbf{z}) = \text{ReLU}(W_l \mathbf{z} + \mathbf{b}_l)$. Typical: 3 layers, halving dimension each layer: $2k_M \to k_M \to k_M/2$
$\mathbf{h}$Output weights: learned vector of size $k_G + k_M/2$ that combines the two paths into a single logit

Why GMF generalizes MF

GMF → MF reduction
$$\text{GMF: } \hat{y} = \sigma(\mathbf{h}^\top (\mathbf{p}_u \odot \mathbf{q}_i)) = \sigma\left(\sum_{d=1}^k h_d \cdot p_{u,d} \cdot q_{i,d}\right)$$ $$\text{If } h_d = 1 \; \forall d: \quad \hat{y} = \sigma\left(\sum_{d=1}^k p_{u,d} \cdot q_{i,d}\right) = \sigma(\mathbf{p}_u^\top \mathbf{q}_i)$$

With uniform weights $\mathbf{h} = \mathbf{1}$, GMF IS the dot product through a sigmoid. With learned $\mathbf{h}$, each latent dimension gets a different importance weight — GMF learns "dimension 3 matters more than dimension 7 for predicting interactions." This is a modest but real generalization over MF.

D2 / FROM SCRATCH
NeuMF — complete PyTorch implementation
↳ corresponds to M25 S2
# ═══ NeuMF — from scratch ═══
import torch, torch.nn as nn

class NeuMF(nn.Module):
    def __init__(self, n_users, n_items, k_gmf=64, k_mlp=32, mlp_layers=[128,64,32]):
        super().__init__()
        # GMF path — separate embeddings
        self.user_gmf = nn.Embedding(n_users, k_gmf)
        self.item_gmf = nn.Embedding(n_items, k_gmf)
        # MLP path — separate embeddings
        self.user_mlp = nn.Embedding(n_users, k_mlp)
        self.item_mlp = nn.Embedding(n_items, k_mlp)
        # MLP layers
        layers = []
        input_dim = k_mlp * 2
        for dim in mlp_layers:
            layers.append(nn.Linear(input_dim, dim))
            layers.append(nn.ReLU())
            input_dim = dim
        self.mlp = nn.Sequential(*layers)
        # Output: combine GMF (k_gmf) + MLP last layer (mlp_layers[-1])
        self.output = nn.Linear(k_gmf + mlp_layers[-1], 1)
        self._init_weights()

    def _init_weights(self):
        for emb in [self.user_gmf, self.item_gmf, self.user_mlp, self.item_mlp]:
            nn.init.normal_(emb.weight, std=0.01)

    def forward(self, users, items):
        # GMF path
        gmf = self.user_gmf(users) * self.item_gmf(items)  # element-wise
        # MLP path
        mlp_in = torch.cat([self.user_mlp(users), self.item_mlp(items)], dim=-1)
        mlp = self.mlp(mlp_in)
        # Fuse
        combined = torch.cat([gmf, mlp], dim=-1)
        return torch.sigmoid(self.output(combined)).squeeze()

Problem: Parameter count comparison
Interview
PROBLEM

Kartify: 2.3M users, 480K items. BPR-MF with k=64. NeuMF with k_gmf=64, k_mlp=32, MLP=[128,64,32]. Compare parameter counts. Is the extra capacity justified?

SOLUTION
$$\text{BPR-MF: } (2.3\text{M} + 480\text{K}) \times 64 = 178\text{M params}$$ $$\text{NeuMF embeddings: } (2.3\text{M} + 480\text{K}) \times (64 + 32) \times 2 = 534\text{M params}$$ $$\text{NeuMF MLP: } 64 \times 128 + 128 \times 64 + 64 \times 32 = 8\text{K} + 8\text{K} + 2\text{K} = 18\text{K params}$$ $$\text{NeuMF total: } 534\text{M} + 18\text{K} \approx 534\text{M params}$$

NeuMF uses 3× more parameters than BPR-MF — almost all in the doubled embedding tables (separate embeddings for GMF and MLP paths). The MLP itself is tiny (18K params). With 18M training interactions, that's 18M/534M ≈ 0.034 observations per parameter — severely overparameterized. The extra capacity is hard to justify unless the data is much denser or regularization is very aggressive.

Problem: Why can't NCF use ANN retrieval?
Interview
PROBLEM

NeuMF's scoring function involves concatenation, MLP layers, and a learned output weight. Why can't this be served via FAISS/ANN like BPR-MF?

SOLUTION

ANN search requires that the score between a query (user vector) and a candidate (item vector) is a dot product or distance metric that can be computed in the vector space. This enables sublinear search: build an index on item vectors, query with user vector, find nearest neighbors in $O(\log N)$.

NeuMF's score involves: (1) element-wise product (ok for ANN in isolation), but (2) concatenation + MLP (NOT expressible as a dot product). The MLP transforms the concatenated vector in ways that don't preserve distance structure — the "nearest neighbor" in embedding space may not have the highest MLP output.

To serve NeuMF for 480K items: you'd need to score EACH item individually → $480K \times \text{MLP forward pass}$ = ~minutes per user. That's why NeuMF is impractical for retrieval and why production systems use dot products (BPR-MF, two-tower) for retrieval and reserve neural interaction (DeepFM, DCN-v2, NCF-style) for ranking 1000 candidates.
