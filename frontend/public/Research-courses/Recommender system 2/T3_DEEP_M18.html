D1 / DERIVATION
EASE closed-form — from autoencoder to matrix inverse
↳ corresponds to M18 S2: EASE
Real-life analogy

Imagine a group of friends. You know who each person hangs out with. EASE asks: "given that Alice hangs out with Bob and Charlie, can we predict she'd also hang out with Diana — based on the fact that Bob and Charlie both hang out with Diana?" It learns these transitive friendship weights in one shot — no iterative training needed. The "weight" between items A and B is: how well does knowing someone interacted with A predict they'll interact with B, after accounting for all other items?

EASE objective
$$\min_W \|R - R \cdot W\|_F^2 + \lambda \|W\|_F^2 \quad \text{subject to: } \text{diag}(W) = 0$$
$R$Interaction matrix ($M \times N$). Each row is a user's interaction vector
$W$Item-item weight matrix ($N \times N$). $W_{ij}$ = how much item $j$ predicts item $i$. This is the learned "similarity" matrix
$R \cdot W$Prediction: each item's score for a user is a weighted sum of the user's interactions with all OTHER items
$\text{diag}(W) = 0$An item cannot predict itself — prevents trivial solution $W = I$ (identity). Without this constraint, the model would just say "if you liked A, I predict you'll like A" — useless

Derivation

Without the diagonal constraint, the solution is the standard ridge regression closed form:

Unconstrained solution
$$W_{\text{unconstrained}} = (R^\top R + \lambda I)^{-1} R^\top R = (G + \lambda I)^{-1} G$$ $$\text{where } G = R^\top R \text{ (the Gram matrix)}$$

Let $P = (G + \lambda I)^{-1}$. Then $W_{\text{unconstrained}} = P \cdot G = P \cdot (G + \lambda I - \lambda I) = I - \lambda P$.

To enforce the diagonal constraint, Steck (2019) showed that the optimal $W$ with zero diagonal is:

EASE closed-form solution
$$\hat{W}_{ij} = \begin{cases} 0 & \text{if } i = j \\ -\frac{P_{ij}}{P_{jj}} & \text{if } i \neq j \end{cases}$$ $$\text{where } P = (R^\top R + \lambda I)^{-1}$$
$P_{ij}$Element $(i,j)$ of the inverse Gram matrix. Captures the partial correlation between items $i$ and $j$ after controlling for all other items
$-P_{ij}/P_{jj}$Normalized partial correlation. The negative sign comes from the KKT conditions of the constrained optimization. Each item's weight is proportional to its partial correlation, scaled by its own "self-importance" $P_{jj}$
D2 / FORMALIZATION
CML triplet loss — metric learning for recommendation
↳ corresponds to M18 S3: CML variant
CML triplet loss
$$\mathcal{L}_{\text{CML}} = \sum_{(u,i,j)} \left[\|\mathbf{u}_u - \mathbf{v}_i\|^2 - \|\mathbf{u}_u - \mathbf{v}_j\|^2 + m\right]_+$$
$\|\mathbf{u} - \mathbf{v}_i\|^2$Squared Euclidean distance between user $u$ and positive item $i$ in the embedding space. We want this SMALL (user close to items they like)
$\|\mathbf{u} - \mathbf{v}_j\|^2$Distance between user and negative item. We want this LARGE (user far from items they don't like)
$m$Margin — minimum required gap between positive and negative distances. Typical: $m = 1.0$. Forces a "safety margin" between liked and disliked items
$[\cdot]_+$Hinge: $\max(0, \cdot)$. If the margin is already satisfied (positive item is close enough relative to negative), loss = 0 and no gradient flows. Like BPR's sigmoid factor — focuses learning on violated constraints

Key difference from BPR: BPR uses dot product (directionality matters, not distance). CML uses Euclidean distance (distance satisfies triangle inequality). In CML's space: if user U is close to item A, and item A is close to item B, then U is guaranteed to be somewhat close to B. With dot product, no such guarantee exists. This makes CML embeddings more "geometrically coherent" — useful when you need user-user or item-item distances to be meaningful alongside user-item scores.

D3 / ANALYSIS
Regularization — what L2 actually does to embeddings
↳ supports M14–M18 regularization choices
L2 regularization — geometric interpretation
$$\frac{\partial}{\partial \mathbf{u}_u} \lambda \|\mathbf{u}_u\|^2 = 2\lambda \mathbf{u}_u$$
$2\lambda \mathbf{u}_u$A constant force pulling the embedding toward the origin. At every training step, the embedding is "taxed" by $2\lambda$ times its current magnitude. Large embeddings pay more tax → they shrink unless the data gradient is strong enough to justify their size
EquilibriumEach embedding dimension settles where: data gradient (push from training examples) = regularization pull (push toward zero). Dimensions supported by MANY training examples resist the pull (strong data gradient). Dimensions supported by FEW examples get shrunk to near-zero (weak data gradient can't overcome the pull)
Effect on sparse usersA user with 3 interactions has a weak data gradient (only 3 training signals). L2 shrinks their embedding toward zero → predictions collapse toward biases ($\mu + b_i$) → graceful degradation to popularity ranking. This is automatic cold-start handling built into the regularization

Problem 1: Design the retrieval stack
System Design
PROBLEM

You're building a recsys for a new e-commerce platform. You have 5M users, 200K items, 50M implicit interactions, and a latency budget of 150ms. Design the retrieval stack using what you've learned in M14–M18. Which MF variant do you choose? How do you serve it?

SOLUTION

Model choice: BPR-MF ($k=128$). Justification: implicit data → BPR loss (not MSE). MF → full-catalog embeddings. Start simple before adding complexity.

Baseline check: Also train EASE ($\lambda=500$) and compare on validation nDCG@10. If EASE wins, use EASE for "similar items" and BPR-MF for personalized retrieval (different strengths).

Training: BPR-MF: ~20 epochs, popularity-weighted negative sampling, $\eta=0.001$, $\lambda=0.01$. ~30 min on GPU.

Serving architecture:

1. Pre-compute all 200K item embeddings → store in FAISS IVF-PQ index (~50MB)

2. At query time: compute user embedding from their history (~0.1ms) → query FAISS for top-1000 (~5ms) → pass to ranking model (~100ms on 1000 candidates) → rerank top-50 (~10ms). Total: ~115ms < 150ms budget ✓

Cold-start fallback: Users with <3 interactions → popularity + CBF. New items → content features through two-tower extension (Act IV).

Problem 2: Diagnose embedding quality
Debugging
PROBLEM

Your BPR-MF model has nDCG@10 = 0.12 — poor. You inspect the learned embeddings: most user embeddings have norm ~0.01 (very small), but a few power users have norm ~5.0. Most item embeddings cluster near the origin. What's wrong and how do you fix it?

SOLUTION

Diagnosis: Regularization is too strong. When $\lambda$ is too large, the L2 pull dominates the data gradient for users/items with few interactions. Most users have $\sim10$ interactions — not enough to overcome heavy regularization. Their embeddings get shrunk to near-zero → dot products are near-zero → no discrimination between items → poor nDCG.

The power users (thousands of interactions) have strong enough data gradients to resist the pull — their embeddings are large and meaningful. But the model only "works" for the top 1% of users.

Fix: Reduce $\lambda$ by 10×. Cross-validate: $\lambda \in \{10^{-4}, 10^{-3}, 10^{-2}\}$. Also check: learning rate might be too low (embeddings aren't moving enough), or the number of training epochs might be insufficient.

Verification: After reducing $\lambda$, check that (a) embedding norms are more uniform across users, (b) the spread of dot products is wider (not all near zero), (c) nDCG improves on validation set.

Problem 3: EASE vs BPR-MF — when each wins
Interview
PROBLEM

On your dataset, EASE achieves nDCG@10 = 0.35 and BPR-MF achieves nDCG@10 = 0.32. A colleague says "EASE is always better, let's drop MF." Why is this wrong? What should you do?

SOLUTION

EASE and BPR-MF have different strengths:

EASE learns item-item weights → excellent for "similar items" and "also viewed" widgets. But it produces an $N \times N$ weight matrix, not user/item embeddings → can't be served via ANN index → can't do personalized retrieval at scale. For Kartify with 480K items: the weight matrix is $480K \times 480K$ — 230 billion entries. Not practical to store or query densely.

BPR-MF produces user and item embeddings → servable via FAISS ANN in ~10ms → the only option for real-time personalized retrieval at scale.

Correct action: Use both. EASE for "similar items" widget (item-item lookup, cached, ~O(K) per query). BPR-MF for personalized homepage retrieval (user-to-item ANN, ~O(log N) per query). They serve different surfaces with different serving constraints. Comparing nDCG alone ignores the serving architecture requirements.

Additional interview problems — MF variants
Problem: EASE vs BPR-MF — when each wins
Interview
PROBLEM

On a dense dataset (95% sparsity), EASE matches BPR-MF in nDCG. On a sparse dataset (99.9% sparsity), BPR-MF wins by 15%. Why does sparsity affect which model wins?

SOLUTION

EASE learns a full N×N item-item weight matrix B. Each entry B[i,j] represents item j's influence on predicting item i. To learn B[i,j] accurately, items i and j need sufficient co-interaction data. At 95% sparsity, most popular item pairs have enough co-interactions — B is well-estimated and EASE works perfectly.

At 99.9% sparsity, most entries in B are estimated from very few (or zero) co-interactions. The closed-form solution produces noisy estimates for rare pairs. BPR-MF's low-rank constraint (k=128) acts as implicit regularization — it CANNOT overfit to individual item pairs because it only has 128 dimensions to represent all relationships. This dimensionality bottleneck is a feature, not a bug, on sparse data.

Rule of thumb: EASE wins when data is dense enough for the item-item matrix to be well-estimated (~95% sparsity or less). BPR-MF wins when data is too sparse for direct item-item estimation (~99%+ sparsity).

Problem: Two-tower serving advantage
Interview
PROBLEM

A two-tower model processes user and item features through separate networks, then scores via dot product. Why is this architecture required for retrieval but not for ranking?

SOLUTION

Retrieval must search 480K items in <10ms. The only way to do this is ANN search, which requires pre-computed item vectors. Two-tower architecture enables this: run the item tower offline for all 480K items, store the vectors in FAISS. At serving time, only the user tower runs (once), then FAISS finds the nearest items in O(log N). Total: user tower (1ms) + FAISS (5ms) = 6ms.

Ranking only scores 1000 candidates. You can afford 0.08ms per candidate × 1000 = 80ms. This budget allows feature crosses between user and item (DeepFM, DCN-v2) — the user and item features are processed TOGETHER, not separately. Cross-features like "user_price_preference × item_price" cannot be captured by two independent towers. The ranking model sacrifices ANN-compatibility for richer modeling — a trade-off only affordable at the 1000-candidate scale.
