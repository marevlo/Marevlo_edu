losses, ESMM, latency.

Module 21 introduced the ranking layer and multi-task learning. This companion formalizes the multi-task loss, derives ESMM's solution to the sample selection bias problem, works through a complete latency budget allocation, and provides the interview-grade analysis of ranking vs retrieval trade-offs.

D1 / FORMALIZATION
Multi-task loss — joint optimization of CTR + CVR
↳ corresponds to M21 S3: Multi-Task Learning
Real-life analogy

Imagine a teacher grading students on three skills simultaneously: reading, writing, and arithmetic. Instead of three separate teachers (who might grade inconsistently), one teacher evaluates all three using a shared understanding of each student's ability. The shared understanding (the backbone) improves all three grades because reading ability informs writing predictions, and vice versa. Multi-task ranking works the same way — predicting click, purchase, and engagement from shared feature representations.

Naive multi-task loss
$$\mathcal{L} = \alpha \cdot \mathcal{L}_{\text{CTR}} + \beta \cdot \mathcal{L}_{\text{CVR}} + \gamma \cdot \mathcal{L}_{\text{dwell}}$$ $$\mathcal{L}_{\text{CTR}} = -\sum_n \left[y_n^{\text{click}} \log \hat{p}_n^{\text{click}} + (1-y_n^{\text{click}})\log(1-\hat{p}_n^{\text{click}})\right]$$ $$\mathcal{L}_{\text{CVR}} = -\sum_{n: \text{clicked}} \left[y_n^{\text{buy}} \log \hat{p}_n^{\text{buy}} + (1-y_n^{\text{buy}})\log(1-\hat{p}_n^{\text{buy}})\right]$$
$\alpha, \beta, \gamma$Task weights — balance gradient magnitudes. If CTR has 100× more labels than CVR, it dominates training without weighting. Typical: $\alpha=1, \beta=1, \gamma=0.1$ (normalize by task sample size first)
$\mathcal{L}_{\text{CTR}}$Binary cross-entropy on ALL impressions (user saw the item). Label = 1 if clicked, 0 if not
$\mathcal{L}_{\text{CVR}}$Binary cross-entropy on CLICKED impressions only. Label = 1 if purchased, 0 if clicked but didn't buy. Problem: only trains on the clicked subset — sample selection bias!

The sample selection bias problem

CVR loss trains only on clicked items. But at serving time, the model must predict CVR for ALL items — including ones the user hasn't clicked yet. The training and serving distributions don't match. Items that are clicked tend to be more attractive → CVR trained on clicked items is biased upward.

ESMM — Entire Space Multi-Task Model (Ma et al., 2018)
$$P(\text{purchase} \mid \text{impression}) = P(\text{click} \mid \text{impression}) \times P(\text{purchase} \mid \text{click, impression})$$ $$\hat{p}_{\text{CTCVR}} = \hat{p}_{\text{CTR}} \times \hat{p}_{\text{CVR}}$$
$\hat{p}_{\text{CTCVR}}$Predicted probability of purchase given impression — the end-to-end conversion rate. This is what we actually want to optimize
Key insightTrain the CVR tower on the ENTIRE impression space (not just clicks) by using the product $\hat{p}_{\text{CTR}} \times \hat{p}_{\text{CVR}}$ as the target for the purchase label. The loss on $\hat{p}_{\text{CTCVR}}$ uses ALL impressions (with purchase = 1 for purchases, 0 for everything else). This eliminates the sample selection bias because CVR is trained on the full distribution
D2 / WORKED
Latency budget allocation — by the numbers
↳ corresponds to M21 S5: Kartify spec
Kartify: allocate 200ms across three stages
Production math
RETRIEVAL (10ms budget)
$$\text{BPR-MF user embedding compute: } 0.1\text{ms}$$ $$\text{FAISS IVF-PQ search over 480K items: } 5\text{ms}$$ $$\text{Merge 3 retrieval sources + dedup: } 3\text{ms}$$ $$\text{Total: } 8.1\text{ms} \quad \text{Output: } 1{,}000 \text{ candidates}$$
RANKING (80ms budget)
$$\text{Feature extraction for 1000 candidates: } 15\text{ms}$$ $$\text{ (embedding lookups + session feature computation)}$$ $$\text{DCN-v2 forward pass (batched, GPU): } 55\text{ms}$$ $$\text{ (1000 × [cross network + MLP] + multi-task heads)}$$ $$\text{Score combination + sort: } 5\text{ms}$$ $$\text{Total: } 75\text{ms} \quad \text{Output: } 50 \text{ items, scored and ordered}$$
RERANKING (10ms budget)
$$\text{Diversity enforcement (MMR on 50 items): } 3\text{ms}$$ $$\text{Business rules (boost new, suppress low-stock): } 2\text{ms}$$ $$\text{Position assignment (hero vs grid): } 1\text{ms}$$ $$\text{Total: } 6\text{ms} \quad \text{Output: } 20 \text{ items, positioned}$$
TOTAL
$$T_{\text{total}} = 8.1 + 75 + 6 = 89.1\text{ms} \quad \text{(+ ~100ms network/render = 189ms < 200ms ✓)}$$

Under budget with 11ms headroom. The ranking stage dominates (84% of compute). This is typical — the ranking model is where most of the computational and quality investment goes.

Interview problems — ranking
↳ tests M21
Problem 1: Multi-task weight tuning
Interview
PROBLEM

Your CTR task has 50M training examples and CVR has 500K (only clicked items). Both losses use BCE. If you set $\alpha = \beta = 1$, what happens? How do you fix it?

SOLUTION

CTR has 100× more examples → its gradients are 100× larger → the shared backbone optimizes for CTR and ignores CVR. The CVR head barely trains.

Fix 1: Weight by inverse sample size: $\alpha = 1/50\text{M}, \beta = 1/500\text{K}$ → equal gradient magnitude per task.

Fix 2: Use ESMM — train CVR on the full impression space (50M examples) by predicting $P(\text{purchase} | \text{impression})$ as $P(\text{click}) \times P(\text{purchase} | \text{click})$. This eliminates both the gradient imbalance AND the sample selection bias.

Fix 3: Uncertainty-weighted loss (Kendall et al., 2018) — learn the task weights $\alpha, \beta$ automatically by modeling each task's homoscedastic uncertainty.

Problem 2: Retrieval vs ranking responsibility
Interview
PROBLEM

Your ranking model (DCN-v2) has nDCG@10 = 0.52, but end-to-end system nDCG@10 = 0.38. Where is the quality being lost? How do you diagnose?

SOLUTION

The gap (0.52 → 0.38) means relevant items are being dropped at the retrieval stage before the ranker ever sees them. Diagnosis:

Measure retrieval Recall@1000: Of the user's true relevant items in the test set, how many are in the 1000 retrieval candidates? If Recall@1000 = 70%, then 30% of relevant items are invisible to the ranker — no amount of ranking quality can recover them.

Fix: Improve retrieval recall. Add more retrieval sources (CBF for cold-start items, user-kNN for serendipity). Increase candidate count from 1000 to 2000 (if latency allows). Use multiple embedding spaces (text + visual + behavioral). The principle from M04: retrieval recall is the ceiling for end-to-end quality.
