Act IV · Go DeepModule 23~30 minGraph Recsys

Graphs know structure —
LightGCN.

MF learns user and item embeddings independently — each embedding is trained from that entity's own interactions. But what if a user's embedding could absorb information from the items they interacted with, and those items could absorb information from other users who liked them? That's graph neural networks for recommendation: embeddings that know their neighborhood. LightGCN does this with stunning simplicity — no activation functions, no feature transforms, just repeated averaging over the graph.

01 / THE GRAPH
User-item bipartite graph — a new way to see the data
The interaction matrix R is a graph. Every non-zero entry is an edge.

You've been looking at the interaction matrix $R$ as a table with rows (users) and columns (items). Now look at it as a graph. Every user is a node. Every item is a node. Every interaction is an edge connecting a user node to an item node. This is a bipartite graph — edges only connect users to items, never user-to-user or item-to-item.

Kartify's graph: 2.3M user nodes, 480K item nodes, 18M edges. At this scale, the graph looks like a massive web of connections — and its structure contains information that MF doesn't fully exploit.

Pause — what does the graph structure tell us?

In Module 14, we showed MF's transitive generalization: items A and B can be similar in latent space even if they share zero users, because they're connected through intermediate items. How does this relate to graph structure?

MF's transitive generalization IS graph structure — it's using multi-hop paths in the bipartite graph. Item A → User X → Item C → User Y → Item B. If this path exists, MF can infer that A and B are related. But MF learns this implicitly through gradient descent. A graph neural network makes this EXPLICIT — it literally passes messages along edges, aggregating information from neighbors. The graph structure that MF discovers implicitly is what GCN exploits directly.
02 / MESSAGE PASSING
GCN message passing — embeddings that absorb their neighborhood
Each node's embedding is enriched by averaging the embeddings of its neighbors. Repeat this for multiple layers to propagate information across the graph.

Imagine Priya's embedding starts as a random vector. In layer 1, her embedding absorbs a weighted average of the embeddings of the items she interacted with — kurta, saree, earrings. Her embedding now "knows" about ethnic fashion. In layer 2, each of those items' embeddings has already absorbed information from OTHER users who liked them. So Priya's embedding now contains indirect signals from users similar to her — users she's never directly connected to but who are 2 hops away in the graph.

After 3 layers, information has traveled 3 hops. Priya's embedding reflects not just her own interactions, but the tastes of users who liked similar items, and the items THOSE users liked. The graph propagates collaborative signals without any explicit similarity computation — the structure does the work.

# GCN message passing — one layer
# For each user u:
#   new_embedding(u) = average of embeddings of items u interacted with
# For each item i:
#   new_embedding(i) = average of embeddings of users who interacted with i

# In matrix form (sparse matrix multiply!):
# E_new = normalize(A) @ E_old
# where A is the adjacency matrix and normalize = degree normalization
03 / LIGHTGCN
LightGCN — the simplification that won
Standard GCNs use weight matrices and activation functions at each layer. LightGCN removes both. Just average. That's it.

The original GCN for recommendation (NGCF, Wang et al. 2019) used trainable weight matrices and non-linear activations at each layer. He et al. (2020) showed that these additions actually HURT recommendation quality. The weight transforms and non-linearities are designed for node classification tasks (where node features are rich); in recommendation, the only input is an ID embedding — there are no rich features to transform.

LightGCN removes everything except the core: neighborhood aggregation through weighted averaging.

# LightGCN — the entire model

# Initialize: same as MF — random user/item embeddings
E_0 = concat(user_embeddings, item_embeddings)  # (M+N, k)

# Propagation: K layers of simple averaging
for l in range(K):
    E[l+1] = D_inv_sqrt @ A @ D_inv_sqrt @ E[l]  # normalized adjacency
    # No weight matrix. No activation function. Just multiply.

# Final embedding: weighted average of ALL layers (including layer 0)
E_final = (1/(K+1)) * (E[0] + E[1] + ... + E[K])

# Prediction: same dot product as MF
score(u, i) = E_final[u] @ E_final[i]

# Training: BPR loss — exactly the same as Module 16

Three insights that make LightGCN elegant: (1) no trainable parameters in the propagation — the ONLY parameters are the initial embeddings $E_0$, same as MF. (2) Multi-scale combination — averaging across layers 0, 1, ..., K means the final embedding captures both local signals (layer 0 = raw ID embedding) and multi-hop collaborative signals (layer K). (3) BPR training — the loss function is identical to Module 16. LightGCN is MF with a pre-processing step that enriches embeddings via graph propagation.

LightGCN = MF + graph smoothing

Mathematically, LightGCN's propagation is equivalent to running a few iterations of graph signal smoothing on the initial MF embeddings. The normalized adjacency multiplication is a low-pass filter on the graph — it makes connected nodes' embeddings more similar. This is exactly the transitive generalization from Module 14, but applied explicitly and controllably. MF discovers it through gradient descent; LightGCN applies it structurally.

04 / WHEN
When graphs beat MF — and when they don't
Graphs aren't universally better. They shine in specific conditions.

Graphs beat MF when: the data is very sparse (Kartify's 0.002% fill — multi-hop propagation discovers connections that MF's gradient descent might miss). The graph has rich structure — communities, clusters, bridge nodes. There are auxiliary graph signals — social connections, item co-purchase networks, category hierarchies.

MF is sufficient when: the data is relatively dense (MovieLens 1M at 4% fill — MF already captures most patterns). The graph is too sparse or too uniform for propagation to add value (random graphs with no community structure). Training speed matters — LightGCN's sparse matrix multiplies add overhead over pure MF.

Typical improvements: LightGCN over BPR-MF: +3–8% nDCG@10 on sparse datasets, +0–2% on dense datasets. The gain comes from multi-hop information propagation that overcomes sparsity. On already-dense data, MF already captures the same patterns through gradient descent.

Kartify graph retrieval

Model: LightGCN (K=3 layers, k=128, BPR loss)
Training: Same as BPR-MF + 3 sparse matrix multiplies per forward pass
Serving: Extract final embeddings → FAISS ANN → same retrieval infrastructure
Impact: +5% nDCG@10 over BPR-MF on Kartify's sparse data (0.002% fill). The multi-hop propagation is particularly valuable for connecting ethnic fashion subcategories that have few direct co-interactions
Cost: Training is ~2× slower than BPR-MF. Serving is identical (dot product + ANN)

05 / PERSPECTIVE
Where graph methods sit in the evolution

The evolution of collaborative representations: item-kNN (M09) — direct neighbors only, no learned representations. MF (M14) — learned embeddings, transitive generalization through gradient descent. LightGCN (this module) — learned embeddings enriched by explicit graph propagation. Each step adds more structural awareness to the embeddings.

But graphs are not the final answer. They capture static collaborative structure — "who interacted with what." They don't capture temporal dynamics (what SASRec handles), feature interactions (what DeepFM handles), or real-time context (what the ranking model handles). In a production system, LightGCN is one retrieval source among several — not a replacement for the full stack.

After this module, you understand graph recommendation:

Bipartite graph: The interaction matrix IS a graph. Users and items are nodes, interactions are edges.
Message passing: Each node absorbs information from its neighbors. After K layers, information has traveled K hops.
LightGCN: No weight matrices, no activations — just normalized averaging + multi-layer combination. The only parameters are the initial embeddings.
When it helps: Sparse data where multi-hop propagation discovers connections MF misses. +3–8% nDCG on sparse datasets.
