Wide & Deep —
memorization meets
generalization.

FM (Module 19) learned pairwise feature interactions. But some patterns need to be memorized exactly ("users who searched for 'diwali kurta' buy item #7892"), while others need generalization ("premium ethnic buyers also like accessories"). Google's Wide & Deep architecture splits these into two components and trains them jointly. This module teaches the model that powers Google Play, YouTube, and most production ranking systems.

01 / THE INSIGHT
Memorization vs generalization — you need both
Some recommendations come from exact pattern matching. Others come from learned abstraction. Combining them is the key to production quality.

Memorization: "Users who installed app X and then searched for Y often install app Z." This is a specific, frequently-observed pattern. A linear model with the right feature crosses captures it perfectly. It's the recommendation equivalent of "people who bought this also bought that" — but for complex, multi-feature patterns.

Generalization: "Budget-conscious users who browse ethnic wear in the evening tend to like kurta sets under ₹999." This pattern is abstract — it generalizes across specific users, items, and sessions. A deep neural network discovers these patterns by learning dense representations of features and their interactions.

The problem: a model that only memorizes can't handle new feature combinations it hasn't seen. A model that only generalizes can miss highly specific patterns that drive conversion. Wide & Deep gives you both.

02 / THE ARCHITECTURE
Two paths, one prediction
Wide = linear model with manually-crossed features. Deep = MLP on dense embeddings. The final prediction combines both.
# Wide & Deep — the complete architecture

# WIDE PATH: linear model with feature crosses
# Captures exact memorization patterns
wide_features = [
    user_id × item_id,          # exact user-item interaction memory
    search_query × item_category,  # "diwali kurta" → ethnic category
    user_city × item_brand,      # Jaipur users × FabIndia
]
wide_output = linear(wide_features)  # just w·x — no hidden layers

# DEEP PATH: MLP on concatenated feature embeddings
# Captures generalization and novel combinations
deep_features = concat([
    user_embedding,     # dense 64-d from user ID
    item_embedding,     # dense 64-d from item ID
    category_embedding, # dense 16-d
    brand_embedding,    # dense 16-d
    price_bucket,       # dense 8-d
    context_features,   # hour, device, session signals
])  # total: ~200-d dense vector
deep_output = MLP(deep_features, layers=[256, 128, 64])

# COMBINE: sigmoid of sum
prediction = sigmoid(wide_output + deep_output)

The wide path is a linear model — simple, interpretable, exact. It memorizes specific feature combinations that the training data shows are predictive. The deep path is a multi-layer neural network — flexible, generalizing, abstract. It learns dense patterns that transfer to unseen feature combinations.

The final prediction is the sigmoid of their sum. During training, both paths receive gradients and learn simultaneously. The wide path doesn't need to generalize (the deep path does that). The deep path doesn't need to memorize every specific pattern (the wide path does that). Each focuses on what it's good at.

Feature crosses — the wide path's power

A "feature cross" is a new feature created by combining two or more features. Example: user_city=Jaipur × item_category=ethnic creates a unique feature that fires only when a Jaipur user sees an ethnic item. The wide path learns a weight for this specific combination — "Jaipur + ethnic → +0.3 boost." Without the cross, a linear model can only learn "Jaipur → +0.1" and "ethnic → +0.2" separately — it can't learn that the COMBINATION has special value beyond the sum of parts.

03 / DEEPFM
DeepFM — let the model learn its own crosses
Wide & Deep requires manual feature cross engineering. DeepFM replaces the wide path with an FM layer that automatically learns all pairwise interactions.

The biggest weakness of Wide & Deep: someone has to manually design the feature crosses. "user_city × item_category" works well, but what about "user_age × item_price × hour_of_day"? There are combinatorially many possible crosses, and selecting the right ones is expensive feature engineering.

DeepFM (Guo et al., 2017) replaces the wide path with an FM layer from Module 19. The FM layer automatically learns all pairwise interactions between features — no manual engineering needed. The architecture becomes:

# DeepFM = FM (pairwise interactions) + Deep (higher-order)

# SHARED EMBEDDINGS — both paths use the same feature embeddings
embeddings = [embed(f) for f in features]

# FM PATH: automatic pairwise interactions (Module 19's formula)
fm_output = FM_interaction(embeddings)  # O(kn) via the trick from DEEP M19

# DEEP PATH: MLP for higher-order interactions
deep_input = concat(embeddings)
deep_output = MLP(deep_input, layers=[256, 128, 64])

# COMBINE
prediction = sigmoid(fm_output + deep_output)

The key advantage: shared embeddings. Both the FM path and the deep path use the same embedding vectors for each feature. This means the FM's learned pairwise interactions inform the deep path's input representations, and vice versa. The two paths co-train, reinforcing each other's learning.

Pause — connect the dots

Trace the evolution: MF (M14) → FM (M19) → DeepFM (this section). What changed at each step? What stayed the same?

MF → FM: Input changed. MF uses only IDs; FM uses IDs + features. Both use dot products for interactions. FM → DeepFM: Added a deep path. FM captures pairwise interactions; the deep path captures higher-order interactions. Both share the same embeddings. What stayed the same: Embeddings + interaction function. The core idea — represent entities as vectors and compute compatibility — is unchanged from M14. The shell got richer; the skeleton is the same.
04 / KARTIFY
Kartify's first neural ranking model
DeepFM replaces the weighted blend from M13. The ranking model scores 1000 retrieval candidates and produces the final ordered list.
Kartify ranking model — DeepFM specification

Input features (33 total from M19):
User: ID, city, age_group, device, avg_order_value, preferred_category, loyalty_tier
Item: ID, category, subcategory, brand, price_bin, material, seller_rating, image_emb(512d)
Context: hour, day_of_week, is_weekend, session_click_count

Architecture:
Embedding dim: 16 per sparse feature, 512 for image
FM path: all pairwise interactions via O(kn) trick
Deep path: MLP [512, 256, 128, 1] with ReLU, BatchNorm, Dropout(0.2)
Output: sigmoid → P(click)

Training:
Loss: binary cross-entropy on click/no-click labels
Optimizer: Adam, lr=1e-3, batch_size=4096
Data: last 30 days of impressions with click labels
~2 hours on single GPU

Serving:
Input: 1000 candidates from retrieval stage
Score all 1000 in one batched forward pass: ~80ms
Return top-50 for reranking

After this module, you can build a neural ranking model:

Wide & Deep: Memorization (linear + crosses) + generalization (MLP on embeddings). Google's production architecture.
Feature crosses: Manually-created combination features that capture specific co-occurrence patterns the deep path might miss.
DeepFM: Replace manual crosses with FM's automatic pairwise interactions. Shared embeddings between FM and deep paths.
The evolution: MF → FM → DeepFM. Same skeleton (embeddings + interaction), richer inputs, deeper computation at each step.
