Act IV · Go DeepModule 21~30 minProduction Ranking

The ranking layer —
scoring 1000 candidates.

Retrieval (M14–M18) finds 1000 plausible items in 10ms using dot products. But the final list of 20 items shown to the user needs much finer discrimination — considering feature interactions, context, and multiple business objectives simultaneously. That's the ranking layer. It's the most important model in the stack because it directly controls what the user sees.

01 / THE STACK
Where ranking sits — the three-stage architecture
Every production recommendation system has this structure. The stages are non-negotiable — the math forces the architecture.

Stage 1: Retrieval (Modules 14–18). Scans the full catalog (480K items) and produces ~1000 candidates in ~10ms. Uses cheap models: BPR-MF dot products + ANN search. Optimizes for recall — don't miss any good item.

Stage 2: Ranking (this module). Scores the 1000 candidates with an expensive feature-rich model (DeepFM, DCN-v2) in ~80ms. Uses all available features: user demographics, item attributes, context, and cross-feature interactions. Optimizes for precision — put the best items at the top.

Stage 3: Reranking (Module 28). Takes the top-50 ranked items and applies business rules, diversity constraints, and fairness adjustments in ~10ms. Optimizes for slate quality — the set should be diverse, fresh, and commercially balanced.

Pause — why three stages?

Why not just use the expensive ranking model on all 480K items directly?

Module 04 computed this: scoring 480K items at 0.08ms each = 38 seconds. The budget is 200ms. The funnel is not a design choice — it's a mathematical necessity. The ranking model is 100× more expensive per item than retrieval, so it can only afford to see 1000 candidates, not 480K. The three stages exist because no single model can be both cheap enough for full-catalog scan AND expressive enough for final ranking.
02 / THE JOB
What the ranking model actually does
It's not just "predict click probability." It predicts multiple outcomes and blends them.

Retrieval answers: "which 1000 items could this user plausibly interact with?" The ranking model answers a harder question: "of these 1000, which should appear first, considering everything we know?"

The ranking model sees features that retrieval can't use. Retrieval uses dot products (required for ANN search), which means it can only use features baked into the user and item embeddings. The ranking model sees cross-features — combinations that only exist at scoring time: "this user, on this device, at this hour, seeing this item at this price, having clicked 3 items in this session already." These context-dependent signals are the ranking model's superpower.

The inputs retrieval can't use

Real-time session signals: What the user clicked in the last 5 minutes. Retrieval uses pre-computed embeddings — they can't react to the current session. The ranker can.

Cross-features between user and item: "Price sensitivity of THIS user × price of THIS item." Retrieval models user and item independently (two towers). The ranker sees them together.

Position and display context: Is this item appearing as a large hero image or a small grid tile? The ranker can adjust for display format. Retrieval has no concept of display context.

03 / MULTI-TASK
Multi-task learning — predict CTR, CVR, and dwell time simultaneously
A click isn't enough. You also care about conversions, engagement quality, and long-term value.

Optimizing for CTR (click-through rate) alone is Goodhart's law from Module 05: the model learns to recommend clickbait — items with attractive thumbnails that users click but don't buy. Multi-task learning predicts multiple outcomes from the same feature representations:

# Multi-task ranking model — three heads, shared backbone

# Shared feature extraction (backbone)
shared_repr = DeepFM_backbone(user_features, item_features, context)

# Task-specific prediction heads
p_click = sigmoid(MLP_click(shared_repr))     # P(click)
p_convert = sigmoid(MLP_convert(shared_repr))  # P(purchase | click)
p_engage = MLP_engage(shared_repr)            # expected dwell time (seconds)

# Combined score — the BUSINESS OBJECTIVE
score = w_click * p_click + w_convert * p_click * p_convert + w_engage * p_engage
# Typical weights: w_click=0.3, w_convert=1.0, w_engage=0.1
# Conversions matter most; clicks are a means; engagement adds quality signal

The key insight: the three heads share the same backbone (feature embeddings + lower MLP layers). This means learning from click data also improves the conversion prediction (shared representations learn general user-item compatibility). But each head has its own final layers to specialize.

The combined score IS the business objective

The weights $w_\text{click}, w_\text{convert}, w_\text{engage}$ encode what the business cares about. A marketplace like Kartify weights conversion highest (purchases = revenue). A content platform like YouTube weights watch time highest (engagement = retention). A social feed weights clicks + shares (virality = growth). The SAME multi-task architecture serves different businesses by changing these weights. The model doesn't change — the objective weights do.

04 / IN PRACTICE
DCN-v2 — the production ranking model
Module 20 introduced Wide & Deep and DeepFM. In practice, DCN-v2 is the current state-of-the-art for production ranking.

DCN-v2 (Wang et al., 2021, Google) replaces DeepFM's FM layer with a cross network that explicitly computes feature interactions at every order — 2nd, 3rd, 4th, and beyond. Each cross layer takes the previous layer's output and crosses it with the original input, building progressively higher-order interactions.

The architecture: a cross network (3–6 layers) runs in parallel with a deep network (standard MLP). The cross network handles explicit feature interactions. The deep network handles implicit pattern learning. Their outputs are combined for the final prediction — exactly the same "two-path" philosophy as Wide & Deep, but with learned crosses instead of manual ones.

Why DCN-v2 won production: On Google's production datasets, DCN-v2 consistently outperformed DeepFM and W&D with less hyperparameter tuning. The cross network is parameter-efficient (each layer adds only $O(d^2)$ parameters where $d$ is the input dimension) and the explicit cross mechanism is more stable to train than relying on MLP to discover interactions implicitly.

05 / KARTIFY
Kartify's complete ranking specification
The production-grade ranking model that sits between retrieval and reranking.
Kartify ranking model — production spec

Architecture: DCN-v2 (3 cross layers) + Deep (MLP [512, 256, 128])
Multi-task heads: P(click), P(purchase|click), E[dwell_time]
Combined score: 0.3 × P(click) + 1.0 × P(click) × P(purchase) + 0.1 × E[dwell]

Features (33): All from M19 feature catalog + 5 real-time session features
Embedding dims: 16 per sparse feature, 512 for image embedding
Training data: Last 14 days of impression logs with click/purchase/dwell labels
Training time: ~3 hours on 4 GPUs (daily retrain)

Serving:
Input: 1000 candidates from retrieval
Batched inference: ~80ms for 1000 candidates on GPU
Output: top-50 scored and ordered → passed to reranking

Latency budget:
Total: 200ms | Retrieval: 10ms | Ranking: 80ms | Reranking: 10ms | Network: 100ms

After this module, you understand production ranking:

Three-stage stack: Retrieval (recall, 10ms) → Ranking (precision, 80ms) → Reranking (slate quality, 10ms). Non-negotiable architecture.
Ranking's superpower: Cross-features, session signals, and display context — information retrieval can't use because of the dot-product constraint.
Multi-task learning: Predict CTR + CVR + dwell simultaneously. Combined score = business objective. Same model, different objective weights for different businesses.
DCN-v2: Cross network (explicit feature interactions) + Deep network (implicit patterns). Current production state-of-the-art.
