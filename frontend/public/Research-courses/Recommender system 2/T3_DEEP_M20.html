architectures formalized.

Module 20 introduced memorization vs generalization. This companion derives the Wide & Deep objective, formalizes DeepFM's shared-embedding architecture, introduces DCN-v2's cross network, and works through a complete DeepFM forward pass with real feature values.

D1 / FORMALIZATION
Wide & Deep — joint objective
↳ corresponds to M20 S2: Two Paths, One Prediction
Wide & Deep prediction
$$P(y = 1 \mid \mathbf{x}) = \sigma\left(\mathbf{w}_{\text{wide}}^\top [\mathbf{x}, \phi(\mathbf{x})] + \mathbf{a}_{L}^\top \mathbf{h}_L + b\right)$$
$\mathbf{x}$Raw input features — categorical IDs, continuous values
$\phi(\mathbf{x})$Feature crosses — manually designed combination features. $\phi(x) = \prod_i x_i^{c_{ki}}$ where $c_{ki} \in \{0, 1\}$ selects which features to cross. Example: $\phi = x_\text{city} \times x_\text{category}$
$\mathbf{w}_{\text{wide}}$Wide weights — linear coefficients for raw features and crosses. Learned via FTRL (Follow The Regularized Leader) optimizer in Google's implementation
$\mathbf{h}_L$Deep path's final hidden layer output. Computed as $\mathbf{h}_l = \text{ReLU}(W_l \mathbf{h}_{l-1} + \mathbf{b}_l)$ for layers $l = 1, \ldots, L$
$\mathbf{a}_L$Output weights connecting deep path's last hidden layer to the final logit
$\sigma(\cdot)$Sigmoid — converts the combined logit to a click probability in $(0, 1)$
Training loss (binary cross-entropy)
$$\mathcal{L} = -\frac{1}{N}\sum_{n=1}^{N}\left[y_n \log \hat{p}_n + (1 - y_n)\log(1 - \hat{p}_n)\right]$$
$y_n$Label: 1 if user clicked, 0 if not. Each impression becomes a training example
$\hat{p}_n$Predicted click probability from the combined Wide + Deep model
Key difference from BPRBPR (M16) trains on pairwise preferences. BCE trains on pointwise labels — each example is independent. This is a ranking-as-classification formulation suitable when you have rich features and impression logs with labels
D2 / FORMALIZATION
DeepFM — architecture with shared embeddings
↳ corresponds to M20 S3: DeepFM
DeepFM prediction
$$\hat{y} = \sigma\left(\underbrace{w_0 + \sum_{i=1}^{n} w_i x_i}_{\text{linear (1st order)}} + \underbrace{\sum_{i=1}^{n}\sum_{j=i+1}^{n}\langle \mathbf{e}_i, \mathbf{e}_j \rangle x_i x_j}_{\text{FM (2nd order)}} + \underbrace{f_{\text{MLP}}(\text{concat}(\mathbf{e}_1 x_1, \ldots, \mathbf{e}_n x_n))}_{\text{Deep (higher order)}}\right)$$
$\mathbf{e}_i$SHARED embedding for feature $i$ — used by BOTH the FM and Deep paths. Key: no separate embedding tables. FM and MLP see the same representations, so they co-train and reinforce each other
FM pathExactly the FM from DEEP M19 — all pairwise dot products, computed in $O(kn)$. Captures 2nd-order interactions automatically (no manual feature crosses needed)
Deep pathMLP on concatenated embeddings. Captures higher-order interactions (3rd, 4th, ...) that FM misses. The MLP can learn "Bengaluru × ethnic × premium × evening" as a single pattern
Advantage over W&DNo manual feature cross engineering. FM discovers ALL pairwise interactions automatically. The Deep path handles everything above 2nd order. Human effort: zero feature engineering
D3 / EXTENSION
DCN-v2 — Cross Network for automatic high-order interactions
↳ extends M20 for production ranking
Real-life analogy

DeepFM's FM layer only captures pairs. The MLP CAN learn higher-order interactions, but it has to discover them from scratch in a high-dimensional space. DCN's cross network explicitly computes higher-order feature crosses layer by layer — like building "user × item" in layer 1, "user × item × context" in layer 2, "user × item × context × price" in layer 3. Each layer adds one more order of interaction, making it easier to learn complex patterns than a generic MLP.

Cross Network layer (DCN-v2)
$$\mathbf{x}_{l+1} = \mathbf{x}_0 \odot (W_l \mathbf{x}_l + \mathbf{b}_l) + \mathbf{x}_l$$
$\mathbf{x}_0$Original input (concatenated embeddings). Re-used at every layer — this ensures each layer crosses with the original features, not just the previous layer's output
$W_l$Per-layer weight matrix. In DCN-v2 (mix version), this is a full $d \times d$ matrix. In DCN-v1, it was a rank-1 outer product (much cheaper but less expressive)
$\odot$Element-wise multiplication — "cross" the transformed input with the original. This is what creates the feature interactions: each dimension of $\mathbf{x}_0$ modulates each dimension of the cross output
$+ \mathbf{x}_l$Residual connection — keeps all lower-order interactions from previous layers. Layer $L$ captures interactions of order up to $L+1$
Production choice

In industry: DCN-v2 is increasingly preferred over DeepFM for ranking because the cross network explicitly models feature interactions at each order, while FM only captures 2nd-order. Google's 2021 paper showed DCN-v2 outperforms DeepFM on production datasets. The choice hierarchy: DeepFM (simpler, good baseline) → DCN-v2 (better quality, slightly more complex) → custom architectures (when you have the team to maintain them).

Interview problems — neural ranking

Problem 1: Why shared embeddings matter
Interview
PROBLEM

DeepFM uses shared embeddings for both FM and Deep paths. An alternative design uses SEPARATE embeddings for each path. What's the trade-off?

SOLUTION

Shared embeddings (DeepFM): Fewer parameters (each feature has ONE embedding, not two). FM and Deep paths co-train — the FM's pairwise signal improves the Deep path's input quality, and vice versa. Risk: the two paths may need different representations (FM prefers embeddings optimized for dot products; MLP may prefer different geometry).

Separate embeddings: Each path gets its own representation — FM embeddings optimize for pairwise dot products; Deep embeddings optimize for MLP processing. More flexible but 2× parameter count. Risk of overfitting on sparse features.

Empirical finding: Shared embeddings generally win on most benchmarks, especially with sparse data (which describes most recsys datasets). The regularization benefit of sharing outweighs the flexibility benefit of separation. Use separate only when you have very dense data and clear evidence that the paths need different representations.

Problem 2: Ranking model capacity planning
Interview
PROBLEM

Kartify's DeepFM has 33 features with embedding dim 16, plus a 512-d image embedding. The MLP is [512, 256, 128, 1]. How many parameters total? What's the inference latency for 1000 candidates?

SOLUTION
$$\text{Embedding params} \approx (2.3\text{M} + 480\text{K} + 500 + 100 + 24 + 7) \times 16 + 512 = 44.5\text{M} \text{ (dominated by user/item IDs)}$$ $$\text{FM params: } 0 \text{ (uses embeddings directly)}$$ $$\text{MLP input dim} \approx 33 \times 16 + 512 = 1040$$ $$\text{MLP params} = 1040 \times 512 + 512 \times 256 + 256 \times 128 + 128 \times 1 = 532\text{K} + 131\text{K} + 33\text{K} + 128 \approx 697\text{K}$$ $$\text{Total} \approx 44.5\text{M} + 697\text{K} \approx 45.2\text{M parameters}$$

Inference: for 1000 candidates, the embedding lookups are $O(1)$ each (table lookup). The FM interaction is $O(kn)$ per candidate. The MLP forward pass is $O(\sum d_l \times d_{l+1})$ per candidate. Batched on GPU: ~0.08ms per candidate → 80ms for 1000 candidates. Fits in the latency budget from M04 (200ms total, ~100ms allocated to ranking).
