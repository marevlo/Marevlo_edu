01 / THE PROBLEM
The popularity baseline's blind spot
Module 06 gave us a recommender that works — but treats every user identically. Here's where that breaks.

Priya opens Kartify. She's browsed 23 ethnic wear products and 8 jewellery items this month. The popularity baseline shows her: Nike running shoes (#1 globally), Levi's jeans (#2), a Fossil watch (#3), and then — finally — a kurta at #4. Three of her top four recommendations are irrelevant.

The baseline doesn't know Priya likes ethnic wear. It doesn't know anything about Priya at all. But here's the thing: it does know what the products are. Every product on Kartify has a title, a description, a category, tags, a brand, a price, a colour, and a set of style attributes. If Priya clicked on a "Floral Anarkali Kurta" and we know that product is tagged [ethnic, kurta, floral, cotton, casual], we can find other products with similar tags — even if no other user has ever bought those products.

This is content-based filtering — the first personalised recommendation algorithm. It uses what items are (their features and descriptions) rather than what other users did (their interactions). And it solves two problems at once: personalising for Priya, and recommending new products that have zero interaction history.

02 / THE IDEA
Three steps — represent, profile, match
Content-based filtering is beautifully simple. Three steps, each intuitive, each buildable in a few lines of code.
Step 1 · Represent
Turn each item into a vector of features

Take every product on Kartify and represent it as a list of numbers — its feature vector. Each dimension represents a feature: is it ethnic wear? Is it cotton? Is it under ₹1,000? Is it floral? The vector captures what the item is in a format that math can work with.

Step 2 · Profile
Build a user preference vector from their history

Priya clicked on 23 ethnic wear products. Average their feature vectors. The result is Priya's preference vector — a summary of what she likes, expressed in the same feature space as the items. Her vector will be strong on "ethnic," "cotton," "floral" and weak on "sportswear," "running," "synthetic."

Step 3 · Match
Score every item by similarity to the user's profile

For each product Priya hasn't seen, compute the cosine similarity between Priya's preference vector and the product's feature vector. Products that are "close" to her profile get high scores. Sort by score. Recommend the top ones.

# Content-based filtering in 10 lines
import numpy as np

def cosine_sim(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-8)

def user_profile(liked_items, item_vectors):
    """Average feature vectors of items the user liked."""
    vecs = [item_vectors[i] for i in liked_items]
    return np.mean(vecs, axis=0)

def recommend_cbf(profile, item_vectors, seen, top_k=10):
    """Score unseen items by cosine similarity to profile."""
    scores = {iid: cosine_sim(profile, vec)
              for iid, vec in item_vectors.items()
              if iid not in seen}
    return sorted(scores, key=scores.get, reverse=True)[:top_k]
Why cosine similarity?

Cosine similarity measures the angle between two vectors, not the distance. This means it's insensitive to vector length — a user who clicked 100 items and a user who clicked 5 items can have the same preference direction. A silk saree and a cotton kurta might both point in the "ethnic" direction even though one is ₹5,000 and the other is ₹599. Cosine captures taste direction, not magnitude.

03 / SMART WEIGHTING
TF-IDF — not all features are equally informative
If every product is tagged "clothing," that tag tells you nothing. TF-IDF automatically down-weights common tags and up-weights distinctive ones.

Raw feature counts treat all tags equally. But "clothing" appears on 95% of Kartify's products — it's useless for distinguishing between them. "Banarasi weave" appears on 0.3% — it's highly informative. We need to weight features by how distinctive they are.

TF-IDF (Term Frequency × Inverse Document Frequency) does exactly this:

TF — how often does this tag appear on this product? (Most products have one instance of each tag, so TF is usually 1 or 0.)

IDF — how rare is this tag across the entire catalog? IDF = log(total products / products with this tag). Rare tags get high IDF. Common tags get low IDF.

TF-IDF = TF × IDF. The "Banarasi weave" tag has TF=1 and IDF=log(480,000/1,440) ≈ 5.8. The "clothing" tag has TF=1 and IDF=log(480,000/456,000) ≈ 0.05. Banarasi weave is 100× more informative. TF-IDF captures this automatically.

TF-IDF works on text tags. But what about features that aren't text — like colour ("red"), price (₹1,499), or brand ("FabIndia")? Can content-based filtering use those?

Yes — you encode them into the feature vector alongside TF-IDF text features. Colours become one-hot dimensions. Price becomes a normalised numeric feature (or binned: "budget," "mid-range," "premium"). Brand becomes a categorical dimension. The feature vector for each product is the concatenation of all these signals: text tags (TF-IDF weighted), colour (one-hot), price (normalised), brand (one-hot or embedded), and any other structured metadata. The richer the feature vector, the more nuanced the similarity becomes.

05 / SUPERPOWER
The cold-start superpower — new products work from day one
Remember Kartify gets ~2,000 new products every week. Content-based filtering can recommend them immediately. No interaction data needed.

A new hand-embroidered kurta arrives on Kartify today. Zero clicks. Zero purchases. Zero interaction data of any kind. The popularity baseline can't recommend it — it has no popularity. Collaborative filtering (which you'll learn in Module 08+) can't recommend it — it has no user-item interactions.

But content-based filtering can. The kurta has tags: [ethnic, kurta, embroidered, cotton, festive, ₹2,199, brand:FabIndia]. Its feature vector is computed from metadata alone. It's placed in the catalog's feature space immediately. Users who like similar products will see it in their recommendations from day one.

Kartify status update

Kartify now has two recommendation sources: the popularity baseline (Module 06) and content-based similarity (this module). The popularity baseline handles new users. Content-based handles new products AND provides personalised recommendations for users with even a few clicks of history. In Kartify's retrieval pipeline, both are candidate sources — their outputs are merged and sent to the ranking stage. The system is already better than static "Trending."

06 / THE LIMIT
The limitation — and the question that creates Module 08
Content-based filtering uses what items ARE. But it can't see what it doesn't know about.

Content-based filtering has a fundamental limitation: it can only recommend items similar to what the user has already liked. Priya likes ethnic wear, so it recommends more ethnic wear. It will never suggest running shoes — even if Priya secretly wants them — because running shoes have zero tag overlap with kurtas.

This is called the filter bubble problem. The system reinforces existing preferences and cannot discover new ones. A user who starts in ethnic wear stays in ethnic wear forever.

There's a second limitation: content features are shallow. TF-IDF knows that two kurtas share the tag "floral" — but it doesn't know that people who buy floral kurtas also tend to buy silver jewellery. That pattern exists in the interaction data, not in the product metadata. To find it, you need to look at what other users did.

That's the bridge to the next module. Instead of asking "what items are similar to what you liked?", we ask a different question: "what did users who are similar to you also like?"

That question is collaborative filtering — and it's the idea that changed everything.
