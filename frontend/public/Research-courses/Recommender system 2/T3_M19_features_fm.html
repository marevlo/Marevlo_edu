Act IV · Go DeepModule 19~30 minFeature-Rich Models

The feature revolution —
why IDs aren't enough.

Every model you've built so far treats users and items as featureless IDs. User #847291 is just a number — the model doesn't know they're a 28-year-old woman in Bengaluru who browses on mobile at night. Item #12045 is just a number — the model doesn't know it's a ₹2,499 cotton kurta in the ethnic category. This module changes that. When you add features, three things happen: cold start gets solved, predictions get richer, and the model becomes a neural network.

01 / THE CEILING
What MF can't see — the ID-only limitation
MF embeddings are powerful but blind to everything except interaction patterns.

Kartify launches a new product: "Hand-Embroidered Pashmina Shawl — ₹8,999." Zero interactions. MF assigns it a random embedding near the origin. BPR can't train on it (no positive examples). Item-kNN has zero co-interactions. The product is invisible to every model from Acts II and III.

But Kartify knows things about this product: it's in the "ethnic accessories" category, priced in the premium tier, made of pashmina wool, listed by a seller with a 4.6-star rating who also sells popular Banarasi sarees. A human merchandiser would immediately say: "show this to the customers who buy premium ethnic wear."

The models can't, because they only see IDs. They're blind to features.

The same blindness applies to users. A new user arrives — User #2,300,001. MF has no embedding for them. But Kartify knows: they're browsing from Jaipur at 9pm on a mobile device, they arrived via a Google search for "wedding lehenga," and their first click was on a premium silk product. A human would say: "show them ethnic wedding wear, premium tier." The model says: "show them popularity rankings."

Pause — the pattern

Module 07's CBF used item features (tags) but ignored interaction patterns. MF (Modules 14–18) used interaction patterns but ignored features. What if a model could use BOTH?

That's exactly what Factorization Machines do — and what the two-tower model from M18's bridge section does. FM is the mathematical framework; two-tower is the production architecture. Both combine ID embeddings WITH feature embeddings in a single model. This module teaches the concept; Module 20 teaches the production version.
02 / THE FEATURES
Three kinds of features that change everything
User features, item features, and context features — each solves a different problem.

User features — who is this person?

Demographics: age group, gender, location, device type. These help with cold-start users — a 25-year-old woman in Mumbai is more likely to buy ethnic wear than a 45-year-old man in Delhi searching for electronics. Behavioral aggregates: average session length, price range of past purchases, category distribution of browsing history. These compress a user's interaction history into dense signals. Context: time of day, day of week, device, referral source. These capture the moment — the same user at 10am on a weekday (browsing basics) vs 10pm on a Saturday (browsing occasion wear) should get different recommendations.

Item features — what is this product?

Catalog attributes: category, subcategory, brand, price, color, material, size range. These are available from day one — no interaction data needed. Content embeddings: pre-trained image embeddings (ResNet/CLIP) capture visual style; text embeddings (BERT) capture description semantics. A new product with no clicks but a product image and description can be positioned in feature space immediately. Seller/creator signals: seller rating, return rate, shipping speed. These proxy quality before the product has its own track record.

Context features — what's happening right now?

Temporal: hour of day, day of week, season, days-to-festival (Diwali, Eid, Christmas). Fashion recommendations should shift as festivals approach. Session: what the user browsed in this session so far. If they clicked 3 lehengas, the next recommendation should be wedding-adjacent, not random. Device/platform: mobile users browse differently than desktop users — shorter sessions, faster scrolling, smaller grids.

The cold-start solution

Features solve cold start because they're available before any interaction occurs. A new product gets positioned by its category, price, brand, and image embedding. A new user gets positioned by their referral source, device, location, and first click. No waiting for enough interactions to train an embedding — the features carry the load until interaction-based signals catch up.

03 / THE BRIDGE
Factorization Machines — MF generalized to features
FM is the mathematical framework that unifies MF, linear models, and feature interactions into one elegant equation.

Factorization Machines (Rendle, 2010) generalize MF to handle arbitrary features. Instead of two separate embedding tables (user, item), FM treats EVERYTHING as a feature and learns embeddings for each feature value.

# MF prediction (Module 15):
# r_hat = mu + b_user + b_item + dot(u_embedding, v_embedding)
# Input: just two IDs — user_id and item_id

# FM prediction:
# r_hat = w_0 + Σ w_i·x_i + Σ Σ dot(v_i, v_j)·x_i·x_j
# Input: a FEATURE VECTOR with potentially hundreds of features:
#   [user_id=847291, item_id=12045, category=ethnic, price_bin=premium,
#    city=bengaluru, hour=21, device=mobile, brand=fabindia, ...]
#
# Each feature gets its own embedding vector v_i
# The model learns ALL pairwise interactions between features
# "user from Bengaluru × ethnic category" gets a learned weight
# "premium price × mobile device" gets a learned weight

The key insight: FM doesn't just use user ID and item ID — it treats category, price, location, time, and every other feature as first-class inputs. Each feature value gets an embedding vector. The prediction includes all pairwise dot products between feature embeddings. This means the model learns interactions like "Bengaluru users prefer ethnic wear" and "premium items convert better on desktop" — automatically, from data.

When FM's only features are user_id and item_id, it reduces exactly to biased MF from Module 15. FM IS MF, generalized. Every additional feature is bonus information that MF couldn't use.

04 / COLD START
How features solve cold start — concretely
The new Pashmina Shawl gets recommendations from its features, not its (nonexistent) interaction history.

Return to the Pashmina Shawl. With FM, its feature vector includes: category=ethnic_accessories, price_bin=premium, material=pashmina, seller_rating=4.6, season=winter. Each of these features has a learned embedding from other products in the same category, price tier, and material. The FM predicts Priya's score for the shawl by computing dot products between Priya's feature embeddings and the shawl's feature embeddings — even though they've never interacted.

The model doesn't need the shawl's own interaction history. It infers interest from the features that the shawl shares with products that DO have history. "Other premium ethnic accessories sell well to users like Priya" → this shawl will too. Cold start solved.

Kartify feature catalog

User features (12): user_id, city, age_group, gender, device, avg_session_length, avg_order_value, preferred_category, account_age_days, loyalty_tier, last_active_days, referral_source

Item features (15): item_id, category, subcategory, brand, price, price_bin, color, material, seller_id, seller_rating, avg_review_score, image_embedding(512d), days_since_listing, return_rate, stock_level

Context features (6): hour_of_day, day_of_week, is_weekend, days_to_next_festival, session_click_count, session_category_distribution

05 / THE LEAP
From FM to neural — the architecture evolution
FM learns pairwise feature interactions. Neural networks learn arbitrary interactions. Same inputs, deeper computation.

FM's prediction has three components: a bias term, linear feature weights, and pairwise interaction terms (dot products between feature embeddings). This captures second-order interactions like "Bengaluru × ethnic" but misses higher-order ones like "Bengaluru × ethnic × premium × evening."

The leap to neural networks: instead of computing all pairwise dot products explicitly, concatenate all feature embeddings and pass them through a neural network that can learn ANY interaction pattern — second-order, third-order, or beyond. This is exactly what Wide & Deep (Module 20) and DeepFM do.

The progression: MF → FM → DeepFM → full neural ranking model. Each step adds expressiveness. MF uses IDs only. FM adds features with pairwise interactions. DeepFM adds a neural network for higher-order interactions. But the skeleton — feature embeddings combined with an interaction function — is the same at every level.

After this module, you understand the feature revolution:

The ID ceiling: MF is blind to demographics, categories, prices, and context. New users and items are invisible.
Three feature types: User (who), item (what), context (when/where/how). Each solves a different recommendation problem.
Factorization Machines: MF generalized to arbitrary features. Each feature value gets an embedding. All pairwise interactions are learned.
Cold-start solved: Products with zero interactions get scored by their feature embeddings — category, price, brand carry the signal.
