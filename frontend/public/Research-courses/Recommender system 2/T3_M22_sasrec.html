Act IV · Go DeepModule 22~30 minSequential Recsys

Sequences matter —
what you clicked last
changes what comes next.

Every model so far treats a user's history as a SET — {kurta, saree, earrings} with no order. But browsing has a narrative. Priya clicked a kurta, then a matching dupatta, then earrings — she's assembling an outfit. The next recommendation should be sandals, not a random new kurta. This module introduces sequential recommendation: models that understand the ORDER of interactions and predict what comes next.

01 / THE GAP
Why sets aren't enough — the lost signal in ordering
Two users with the same history set but different orderings want completely different things.

User A: Browsed kurta → dupatta → earrings (assembling an outfit → next: sandals?)

User B: Browsed earrings → kurta → dupatta (started with accessories, moved to clothing → next: more clothing?)

Same three items. Same set: {kurta, dupatta, earrings}. But the ORDER tells completely different stories. User A is in "outfit assembly" mode — each item follows logically from the previous. User B might be in "browse everything" mode with no clear direction. MF and item-kNN see these two users as identical. A sequential model sees them as different.

Pause — when does order matter?

In what product scenarios does the ORDER of browsing carry significant signal? When does it NOT matter?

Order matters most in: outfit assembly (fashion), playlist building (music), recipe discovery (food), multi-step research (shopping for electronics), binge-watching (video). These are sessions with intent evolution — the user's goal becomes clearer with each action. Order matters less in: weekly grocery shopping (habitual, not sequential), content browsing with no clear goal, repeat purchases. For Kartify: session order is very relevant for "continue browsing" and "complete your look" surfaces.
02 / GRU4REC
GRU4Rec — the first neural session model
An RNN that reads the browsing sequence item by item and predicts the next item at each step.

GRU4Rec (Hidasi et al., 2015) was the first successful application of deep learning to session-based recommendation. The idea: feed the user's item sequence through a Gated Recurrent Unit (GRU) network. At each step, the GRU produces a hidden state that encodes "everything the user has done so far." Use that hidden state to predict the next item.

# GRU4Rec — conceptual forward pass
h = zeros(hidden_dim)  # initial hidden state

for item_id in session_sequence:
    x = item_embedding[item_id]     # look up item embedding
    h = GRU_cell(x, h)              # update hidden state
    # h now encodes: "user browsed items 1...t"
    # h can be used to predict item at step t+1

# Predict: dot product of h with all item embeddings
scores = h @ item_embedding_matrix.T  # (n_items,)
next_item = argmax(scores)

Strengths: Simple, fast at inference (just one GRU step per new click), captures short-range dependencies well. Weaknesses: Struggles with long sequences (GRU forgets early items), can't attend to specific past items selectively (the hidden state is a fixed-size compression of the entire history).

03 / SASREC
SASRec — self-attention for sequential recommendation
Instead of compressing the sequence into a single hidden state, attend to ALL past items and let the model decide which ones matter for predicting the next.

SASRec (Kang & McAuley, 2018) applies the Transformer's self-attention mechanism to recommendation sequences. The key difference from GRU4Rec: instead of reading the sequence left-to-right through a recurrence, SASRec looks at ALL past items simultaneously and computes attention weights — "how relevant is each past item to predicting the next?"

# SASRec — self-attention on item sequences

# Input: sequence of item embeddings + positional encoding
E = item_embedding[sequence] + position_embedding  # (L, d)

# Self-attention (causal — can only attend to past items)
for layer in transformer_layers:
    E = layer(E)  # multi-head self-attention + FFN
    # Causal mask: position t can only attend to positions 1..t

# The representation at the LAST position encodes the full sequence
user_state = E[-1]  # (d,) — "what should come next?"

# Predict: dot product with all items (same as MF serving!)
scores = user_state @ item_embedding_matrix.T

The magic: causal self-attention lets the model decide, for each position, which past items are most relevant. If Priya is assembling an outfit, the attention at the earrings position might attend strongly to the kurta (matching style) and weakly to a random earlier browse. The model learns these attention patterns from data.

SASRec produces embeddings

The output of SASRec at the last position is a user state vector — exactly like a user embedding from MF. This means SASRec's output can be served via the same ANN infrastructure as BPR-MF (Module 16). At inference: run the user's last N items through SASRec → get a user vector → query FAISS for nearest items → done. The sequential model plugs into the existing retrieval architecture with zero serving changes.

04 / BERT4REC
BERT4Rec — bidirectional attention for richer context
SASRec is unidirectional (left-to-right). BERT4Rec uses masked prediction to see both past and future context.

BERT4Rec (Sun et al., 2019) applies BERT's masked language modeling objective to item sequences. During training, it randomly masks items in the sequence and predicts them from the surrounding context — both left AND right. This gives the model a richer understanding of the sequence structure.

At serving time, BERT4Rec appends a [MASK] token at the end of the sequence and predicts what item should fill it — effectively predicting the next item using bidirectional context from the entire history.

SASRec vs BERT4Rec: SASRec trains by predicting the next item at each position (causal, left-to-right). BERT4Rec trains by predicting masked items (bidirectional). Empirically: BERT4Rec often wins on offline benchmarks, but SASRec is simpler to serve (no [MASK] token logic) and often wins in production A/B tests due to better alignment between training (causal) and serving (predicting the next item).

Kartify session model

Model: SASRec (2 transformer layers, 2 attention heads, d=128)
Input: Last 50 items in user's browsing sequence (zero-padded if shorter)
Training: Next-item prediction loss (cross-entropy), last 30 days of session data
Serving: Run SASRec → user state vector → FAISS ANN → top-50 session-aware candidates
Surface: "Continue Browsing" row on homepage, "You Might Also Like" on product pages
Impact: +8–12% nDCG@10 on returning users over BPR-MF (which ignores sequence order)

05 / WHEN
When to use sequential models — and when not to
Sequence models add complexity. They're not always worth it.

Use sequential models when: sessions have clear intent evolution (outfit assembly, playlist building). Average session length > 5 items. The order of clicks carries signal that the set doesn't. The product has a "continue browsing" or "next episode" surface.

Stick with set-based models when: users interact rarely (< 5 total interactions). Sessions are short (1–2 clicks). The domain is habitual (grocery, repeat orders). The order is essentially random (passive browsing with no clear goal).

In practice: most teams run SASRec as an ADDITIONAL retrieval source alongside BPR-MF, not as a replacement. SASRec excels at short-term, session-aware predictions. BPR-MF excels at long-term preference modeling. Together, they cover both timescales.

After this module, you understand sequential recommendation:

The gap: Set-based models (MF, kNN) ignore the ORDER of interactions. Two users with the same item set but different orderings want different things.
GRU4Rec: RNN that reads sequences left-to-right. Simple, fast, but forgets early items.
SASRec: Transformer self-attention on sequences. Attends to ALL past items selectively. Produces a user vector compatible with ANN retrieval.
BERT4Rec: Bidirectional masked prediction. Richer training signal, but more complex serving.
