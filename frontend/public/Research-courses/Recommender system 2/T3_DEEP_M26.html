D1 / PARAMETER COUNTS
How many parameters does each model have?
↳ corresponds to M26 S2: The comparison table
Real-life analogy

Parameters are like the number of knobs on a mixing board. A 3-knob EQ (popularity: 0 parameters) can only adjust bass/mid/treble. A 128-channel studio console (BPR-MF: ~178M parameters) can shape every frequency independently. But a 10,000-channel board (NeuMF: 534M parameters) has more knobs than any engineer can meaningfully control — most end up at default positions. More parameters ≠ better sound. More parameters = more ways to overfit to the studio noise.

Parameter counts for Kartify (M=2.3M users, N=480K items, k=128)
$$\text{Popularity: } 0 \text{ parameters (just item counts)}$$ $$\text{Item-kNN: } 0 \text{ learned params (similarity computed from data)}$$ $$\text{EASE: } N^2 = 480\text{K}^2 = 230\text{B entries — but stored sparse} \approx 50\text{M non-zero}$$ $$\text{BPR-MF: } (M + N) \times k = 2.78\text{M} \times 128 = 356\text{M}$$ $$\text{LightGCN: } (M + N) \times k = 356\text{M (same as MF — no extra params!)}$$ $$\text{SASRec: } N \times k + L \times k + \text{transformer params} \approx 62\text{M} + 0.4\text{M} = 62\text{M}$$ $$\text{DeepFM: } \text{embeddings} + \text{MLP} \approx 50\text{M} + 0.1\text{M} = 50\text{M}$$ $$\text{NeuMF: } 2 \times (M + N) \times k + \text{MLP} = 712\text{M} + 0.02\text{M} \approx 712\text{M}$$
Key insightLightGCN has the SAME parameter count as BPR-MF — no weight matrices in the propagation layers. The only parameters are the initial embeddings. NeuMF has 2× the parameters of BPR-MF because it uses separate embedding tables for the GMF and MLP paths. SASRec and DeepFM are relatively parameter-efficient because they don't embed all 2.3M users — SASRec uses item embeddings only, DeepFM uses feature embeddings
D2 / WORKED
Cost-per-recommendation: the business math
↳ corresponds to M26 S3: Good enough principle
Kartify: cost per 1M recommendations for each model tier
Production math
TIER 1: POPULARITY + kNN (no GPU needed)
$$\text{CPU instance: } \$0.10/\text{hr} \times 2 \text{ instances} = \$0.20/\text{hr}$$ $$\text{Throughput: } 50{,}000 \text{ recs/sec} = 180\text{M recs/hr}$$ $$\text{Cost per 1M recs: } \$0.20 / 180 = \$0.0011$$
TIER 2: BPR-MF + FAISS (GPU for FAISS optional)
$$\text{CPU + FAISS: } \$0.50/\text{hr} \times 4 \text{ instances} = \$2.00/\text{hr}$$ $$\text{Throughput: } 10{,}000 \text{ recs/sec} = 36\text{M recs/hr}$$ $$\text{Cost per 1M recs: } \$2.00 / 36 = \$0.056$$
TIER 3: BPR-MF + DeepFM RANKING (GPU required)
$$\text{GPU instances: } \$8/\text{hr} \times 8 \text{ instances} = \$64/\text{hr}$$ $$\text{Throughput: } 2{,}500 \text{ recs/sec (ranking 1000 candidates each)} = 9\text{M recs/hr}$$ $$\text{Cost per 1M recs: } \$64 / 9 = \$7.11$$
COMPARISON

Tier 1 → Tier 2: 50× more expensive, but nDCG jumps +25-40%. Worth it at almost any scale.

Tier 2 → Tier 3: 127× more expensive, but nDCG only improves +5-15%. Worth it only if the absolute GMV gain (5% of ₹15 lakh/day = ₹75K/day) exceeds the infrastructure cost (₹~40K/day). At Kartify's scale: yes, ROI is ~2:1. At a small startup with ₹50K/day GMV: no, the ₹40K/day GPU bill eats 80% of the recommendation-attributed revenue.

Problem: Justify your model stack to a VP of Engineering
Interview
PROBLEM

You're presenting to the VP of Engineering. They ask: "We have BPR-MF running. The ML team wants to add SASRec, LightGCN, and DCN-v2 multi-task ranking. That's 6 months of engineering. Convince me." How do you frame the ROI?

SOLUTION

Frame it as incremental investment with measurable checkpoints:

Phase 1 (1 month): SASRec retrieval. Adds session-aware recommendations. Expected lift: +5-8% nDCG on returning users. Serving cost: zero incremental (same FAISS infrastructure). Engineering: 1 person × 1 month. Measurable via A/B test within 3 weeks of deployment.

Phase 2 (2 months): DCN-v2 ranking. Adds feature-rich scoring. Expected lift: +5-10% nDCG on top of SASRec. Serving cost: +$2K/day for GPU ranking. Engineering: 2 people × 2 months. The big investment — but also the biggest quality lever because it uses cross-features (price × user budget, time × category).

Phase 3 (1 month): LightGCN retrieval. Adds graph-enhanced embeddings. Expected lift: +3-5% on sparse subcategories. Serving cost: zero incremental. Engineering: 1 person × 1 month.

Key framing: "Each phase is independently shippable and measurable. If Phase 1 doesn't show lift in A/B testing, we stop. We're not committing 6 months blind — we're committing 1 month at a time with clear success criteria."

Problem: When NOT to add complexity
Interview
PROBLEM

Your team adds SASRec to the retrieval stack. Offline nDCG improves 7%. But the A/B test shows 0% GMV lift after 3 weeks. What happened, and what do you do?

SOLUTION

Diagnosis checklist:

1. Retrieval recall check: Is SASRec producing candidates that BPR-MF already produces? If 90% overlap, the new model isn't adding new information — the ranking model sees the same candidates regardless. Fix: check Recall@1000 of SASRec-only vs BPR-only and the union.

2. User segmentation: Does SASRec help for session-heavy users (5+ clicks) but hurt for light users (1-2 clicks)? If session-heavy users are only 20% of traffic, the overall lift is diluted. Fix: deploy SASRec only for users with 3+ session clicks.

3. Offline-online gap: The 7% nDCG improvement may be on the items SASRec newly surfaces — but users might not want those items even though they're "relevant." Session-aware ≠ purchase-intent-aware.

Decision: If none of these fix the gap → do NOT ship. A 0% GMV lift means the engineering cost has negative ROI. Revert to BPR-MF only. The "good enough" principle in action.
