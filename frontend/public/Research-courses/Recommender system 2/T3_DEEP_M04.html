Module 04 named six recommendation tasks and introduced the multi-stage funnel. This companion gives each task its mathematical formulation, proves why RMSE and nDCG disagree, and computes Kartify's latency budget step by step.

📖 Narrative: M04 — Six Problems Wearing the Same Name
D1 / FORMALIZATION
Six tasks — input, output, loss, metric
↳ corresponds to M04 S2: Six Distinct Tasks
Real-life analogy

Think of each task as a different exam question format. "Rating prediction" is fill-in-the-blank — predict the exact number. "Top-N ranking" is multiple choice with ordering — rank the best answers. "Next-item" is sentence completion — predict what comes next. "Slate" is essay composition — craft a coherent set. "Session" is live conversation — respond to evolving context. "Conversational" is oral exam — ask clarifying questions. Same subject, different test formats, different skills tested.

Task 1: Rating prediction
$$\text{Input: } (u, i) \quad \text{Output: } \hat{r}_{ui} \in \mathbb{R} \quad \text{Loss: MSE} \quad \text{Metric: RMSE}$$ $$\hat{r}_{ui} \approx r_{ui} \quad \text{(predict the exact numeric score)}$$
InputA specific (user, item) pair — "what will Priya rate this saree?"
OutputA single real number — the predicted rating (e.g., 4.2 stars)
When usedAlmost never in 2025. Legacy task from Era 1/2. Still in some academic benchmarks
Task 2: Top-N ranking (Kartify's primary task)
$$\text{Input: } u, \mathcal{I} \setminus \mathcal{H}_u \quad \text{Output: } L_u^K = (i_1, \ldots, i_K) \quad \text{Loss: BPR/BCE} \quad \text{Metric: nDCG@K}$$ $$f(u, i_1) \geq f(u, i_2) \geq \ldots \geq f(u, i_K) \quad \text{(produce an ordered list)}$$
InputA user + the entire unseen catalog. "What should Priya see on her homepage?"
OutputAn ORDERED list of $K$ items — position matters. Item at position 1 is the top recommendation
Key differenceWe don't need exact scores — only correct ORDERING. A model that scores items [0.9, 0.7, 0.3] and one that scores [90, 70, 30] produce the same ranking and get the same nDCG
Task 3: Next-item prediction
$$\text{Input: } (i_1, i_2, \ldots, i_t) \quad \text{Output: } \hat{i}_{t+1} \quad \text{Loss: CE} \quad \text{Metric: HR@K, MRR}$$ $$P(i_{t+1} \mid i_1, \ldots, i_t) \quad \text{(predict next item from sequence)}$$
$(i_1, \ldots, i_t)$The user's interaction SEQUENCE — ordered by time. "Priya browsed kurta → saree → earrings. What next?"
HR@KHit Rate at $K$ — was the actual next item in our top-$K$ predictions? Binary per user, averaged
MRRMean Reciprocal Rank — $1/\text{position}$ of the correct item. If the correct item is at position 3, MRR contribution = $1/3$
Task 4: Slate recommendation
$$\text{Input: } u, \text{surface layout} \quad \text{Output: } S = \{i_1, \ldots, i_K\} \quad \text{Metric: slate-level quality}$$ $$\text{Quality}(S) \neq \sum_{i \in S} f(u, i) \quad \text{(the SET matters, not just individual items)}$$
$S$A SET of items — the combination matters. {kurta, saree, kurta, kurta} is worse than {kurta, saree, earrings, shoes} even if each kurta scores higher individually
Key differenceCannot be solved by picking top-$K$ individually. Needs diversity-aware selection (DPP, MMR) or set-level optimization
D2 / WORKED
Why nDCG and RMSE disagree — concrete proof
↳ corresponds to M04 S3: Why Wrong Task = Dead Project
Real-life analogy

Imagine two librarians recommending books. Librarian A predicts your ratings perfectly (RMSE = 0) but shows them in random order. Librarian B gets the ratings slightly wrong (RMSE = 0.5) but always puts the best book first. You'd prefer Librarian B — the order matters more than the exact score. That's why ranking metrics (nDCG) and prediction metrics (RMSE) can give opposite verdicts.

Worked: Two models — Model A has better RMSE, Model B has better nDCG
Key example
SETUP

True ratings for user Priya on 5 items: $r = [5, 4, 3, 2, 1]$

Relevant items (rating ≥ 4): $\mathcal{I}^+ = \{A, B\}$ (items with ratings 5 and 4)

MODEL A — GOOD RMSE, BAD RANKING

Predictions: $\hat{r}_A = [4.8, 3.9, 3.1, 2.2, 1.1]$ → sorted ranking: $[A, B, C, D, E]$ ✓

Wait — this ranking is actually correct. Let's make it more interesting:

Predictions: $\hat{r}_A = [3.0, 3.0, 3.0, 3.0, 3.0]$ → predicts mean for everything.

$$\text{RMSE}_A = \sqrt{\frac{(5-3)^2 + (4-3)^2 + (3-3)^2 + (2-3)^2 + (1-3)^2}{5}}$$ $$= \sqrt{\frac{4 + 1 + 0 + 1 + 4}{5}} = \sqrt{2} = 1.414$$

All items get same score → random ordering → say the ranking is $[C, E, A, D, B]$.

$$\text{nDCG@3}: \text{rel} = [0, 0, 1] \quad \text{DCG} = \frac{1}{\log_2 4} = 0.500$$ $$\text{IDCG} = \frac{1}{\log_2 2} + \frac{1}{\log_2 3} = 1.631 \quad \text{nDCG@3} = \frac{0.500}{1.631} = 0.307$$
MODEL B — WORSE RMSE, PERFECT RANKING

Predictions: $\hat{r}_B = [10, 8, 5, 2, 0]$ → ranking: $[A, B, C, D, E]$ ✓ (correct order!)

$$\text{RMSE}_B = \sqrt{\frac{(5-10)^2 + (4-8)^2 + (3-5)^2 + (2-2)^2 + (1-0)^2}{5}}$$ $$= \sqrt{\frac{25 + 16 + 4 + 0 + 1}{5}} = \sqrt{9.2} = 3.033$$
$$\text{nDCG@3}: \text{rel} = [1, 1, 0] \quad \text{DCG} = \frac{1}{\log_2 2} + \frac{1}{\log_2 3} = 1.631$$ $$\text{nDCG@3} = \frac{1.631}{1.631} = 1.000 \quad \text{(perfect!)}$$
THE VERDICT

Model A: RMSE = 1.414 ✓ (better), nDCG = 0.307 ✗ (terrible ranking)

Model B: RMSE = 3.033 ✗ (worse), nDCG = 1.000 ✓ (perfect ranking)

The scores from Model B are "wrong" by RMSE standards — predicting 10 instead of 5 is a big error. But the ORDERING is perfect. And in a recsys, only the ordering matters. This is the mathematical proof of Module 02's lesson: the Netflix Prize optimized RMSE when it should have optimized nDCG.

D3 / WORKED
Latency budget — why the funnel is necessary
↳ corresponds to M04 S4: The Funnel That Runs Everything
Real-life analogy

You need to find the best 10 candidates out of 480,000 job applications. You have 15 minutes (the latency budget). Strategy 1: read each application carefully (2 minutes each) → 480,000 × 2 min = 960,000 minutes = 1.8 YEARS. Impossible. Strategy 2: spend 1 second scanning each resume for keywords (retrieval), keep 1,000. Then spend 30 seconds on each of 1,000 (ranking), keep 50. Then spend 3 minutes on each of 50 (interviews). Total: 8 min + 8 min + 2.5 hours. Still too long, but the principle works — each stage filters aggressively so the expensive evaluation runs on fewer candidates.

Worked: Kartify's latency budget per pipeline stage
Numerical
TOTAL BUDGET

Kartify homepage must load in under 250ms. The recommendation pipeline gets ~200ms of that (the remaining 50ms is network + rendering).

$$T_{\text{total}} = 200\text{ ms}$$
NAIVE APPROACH: SCORE EVERYTHING

A ranking model (DeepFM-class) takes ~0.5ms per candidate with batch inference.

$$T_{\text{naive}} = N \times t_{\text{rank}} = 480{,}000 \times 0.5\text{ ms} = 240{,}000\text{ ms} = 4\text{ minutes}$$

Four minutes to score every item. The budget is 200ms. Off by a factor of 1,200×. Scoring the full catalog is impossible.

RETRIEVAL STAGE

Retrieval uses a cheap model (two-tower dot product + ANN index). Cost: ~0.001ms per candidate with ANN.

$$\text{ANN search over 480K items: } T_{\text{retrieval}} \approx 10\text{–}30\text{ ms}$$ $$\text{Output: } \sim 1{,}000 \text{ candidates}$$ $$\text{Remaining budget: } 200 - 30 = 170\text{ ms}$$
RANKING STAGE

Score 1,000 candidates with the expensive ranking model (0.1ms per candidate with batching):

$$T_{\text{rank}} = 1{,}000 \times 0.1\text{ ms} = 100\text{ ms}$$ $$\text{Output: } \sim 50 \text{ items, ordered by score}$$ $$\text{Remaining budget: } 170 - 100 = 70\text{ ms}$$
RE-RANKING STAGE

Apply diversity/business rules to 50 items. Simple rule-based reranking:

$$T_{\text{rerank}} = 50 \times 0.2\text{ ms} + \text{diversity computation} \approx 20\text{ ms}$$ $$\text{Output: final 20 items for the homepage}$$ $$\text{Total: } 30 + 100 + 20 = 150\text{ ms} \quad \checkmark \text{ under 200ms budget}$$
The math proves the architecture

Without the funnel, scoring all 480K items costs 4 minutes. With the funnel (ANN retrieval → batch ranking → rule-based reranking), it costs 150ms. The 1,600× speedup comes from the funnel's core insight: $480{,}000 \rightarrow 1{,}000 \rightarrow 50 \rightarrow 20$. Each stage reduces candidates by ~10–20× while increasing model complexity. The math forces the architecture — there is no other way to serve recommendations at scale in real time.

D4 / DERIVATION
Retrieval recall — why misses at stage 1 are fatal
↳ corresponds to M04 S4: The Funnel (retrieval stage)
Retrieval stage recall
$$\text{Recall@}C = \frac{|\mathcal{I}_{\text{rel}} \cap \mathcal{I}_{\text{retrieved}}|}{|\mathcal{I}_{\text{rel}}|}$$
$\mathcal{I}_{\text{rel}}$The set of items that are truly relevant to this user (if they saw them, they'd click/buy)
$\mathcal{I}_{\text{retrieved}}$The $C$ candidates the retrieval stage outputs (e.g., $C = 1{,}000$ out of 480K)
Recall@CFraction of truly relevant items captured. If Priya would like 50 items and retrieval captures 45 of them → Recall = 90%

Why retrieval recall is the most critical metric in the pipeline: if retrieval misses a relevant item, the ranking and reranking stages can never recover it. The funnel only subtracts — it never adds. A relevant item dropped at retrieval is gone forever from this request's recommendations.

Worked: Impact of retrieval recall on final quality
Numerical
SCENARIO

Priya has 50 truly relevant items in the catalog. The ranking model is perfect (orders items correctly). We vary retrieval recall:

$$\text{Retrieval Recall = 90\%: } 45 \text{ relevant items in candidates → ranking can surface all 45}$$ $$\text{Retrieval Recall = 50\%: } 25 \text{ relevant items in candidates → ranking can surface at most 25}$$ $$\text{Retrieval Recall = 10\%: } 5 \text{ relevant items → even a perfect ranker can only show 5}$$

If the homepage shows 20 items, Recall=10% means at most 5 of 20 slots have relevant items — the other 15 are irrelevant regardless of ranking quality. No amount of ranking sophistication can fix bad retrieval. This is why retrieval optimizes recall (don't miss good items) while ranking optimizes precision (get the order right).

D5 / INTERVIEW
Interview problems — tasks & pipeline
↳ tests M04 concepts
Problem 1: Task identification
Interview
PROBLEM

Your PM asks: "Build a recommender for our Netflix-style homepage with 10 rows of 8 tiles each, where no two adjacent rows should have the same genre." Which task type is this? Write the one-sentence diagnostic from Module 04.

SOLUTION

This is slate recommendation (Task 4), not top-N ranking. The constraint "no two adjacent rows have the same genre" is a set-level constraint that independent top-$K$ ranking cannot satisfy.

One-sentence diagnostic: "Given a user and a 10×8 homepage grid, produce a coordinated set of 80 items where each row is thematically coherent and cross-row diversity is enforced, evaluated by slate-level engagement metrics."

If the engineer treats this as top-N (Task 2), they'll fill each row with individually high-scoring items — and the homepage will show 8 action movies in every row.

Problem 2: Latency constraint
Interview
PROBLEM

A platform has 50M items and a 100ms budget. Their ranking model takes 0.2ms per item. How many candidates can the ranking stage handle? What does this imply for the retrieval stage?

SOLUTION
$$\text{Max candidates for ranking} = \frac{100\text{ ms}}{0.2\text{ ms/item}} = 500 \text{ items}$$

The ranking stage can handle at most 500 candidates. The retrieval stage must reduce 50M items to 500 — a 100,000× reduction. This requires an extremely efficient retrieval model (two-tower + ANN). The retrieval model's primary metric is Recall@500: what fraction of truly relevant items are in the 500 candidates?

If retrieval uses a simple ANN index (~10ms for 50M items), the total pipeline is: 10ms retrieval + 100ms ranking + 20ms reranking = 130ms. Under budget.

Problem 3: Metric mismatch
Interview
PROBLEM

A team reports "our model achieves nDCG@10 = 0.45 on the test set." The PM asks: "Is that good?" What's the right answer, and what additional information do you need?

SOLUTION

nDCG@10 = 0.45 means "the model achieves 45% of the ideal ranking quality." But this number is meaningless without context:

1. What's the baseline? If a popularity baseline gets nDCG@10 = 0.42, the model is only 7% better — probably not worth the complexity. If the baseline gets 0.15, the model is 3× better — significant.

2. How many relevant items exist? If each user has 2 relevant items in a catalog of 480K, nDCG@10 = 0.45 is excellent. If each user has 100 relevant items, 0.45 is mediocre.

3. What's the offline-online correlation? Does a +0.05 nDCG improvement offline translate to measurable online metric improvement? Without this calibration (from historical A/B tests), the absolute nDCG number is uninterpretable.

The right answer to the PM: "It depends on the baseline and the domain. Let's compare to popularity and to the current production model."
