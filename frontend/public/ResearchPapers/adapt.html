Adaptation of Agentic AI the four-paradigm map of post-training

Lead authorsPengcheng Jiang · Jiacheng Lin · Zhiyi Shi + 31
YearDec 2025 → Mar 2026 v3

AffiliationsUIUC · Stanford · Harvard · Princeton · Berkeley · CMU · UW · 8 more

The agentic AI literature is a mess. Toolformer, DeepSeek-R1, OpenClaw, s3, AgentFlow, GRPO, ReTool papers everywhere, no map. This 34-author survey from Stanford / UIUC / Harvard / Berkeley introduces the field's first unified framework: every adaptation method maps onto a 2×2 grid of two binary choices what gets optimised (agent vs tool) and what's the supervision signal (tool execution vs agent output). The result: four paradigms (A1, A2, T1, T2) that cleanly classify the entire post-training landscape. The framework makes design space explicit, makes trade-offs visible, and gives concrete guidance on which paradigm to pick for which problem. The most important orientation document for anyone building agents in 2026.

Agent post-training is fragmented. RLVR papers, tool-learning papers, memory papers, skill-library papers all overlap, none use the same vocabulary. Practitioners can't tell whether ReTool is "the same as" DeepSeek-R1 with extra steps, or whether s3 belongs in the same family as Toolformer. The four-paradigm grid
Two binary axes: WHAT is optimised (agent / tool) × WHAT'S the supervision signal (tool execution / agent output). The 2×2 yields four paradigms A1 (agent + tool-execution), A2 (agent + agent-output), T1 (tool, agent-agnostic), T2 (tool, agent-supervised). Every method in the field maps to one cell.

The framework makes design choices visible. Fine-tune a retriever vs train a search agent? That's T1 vs T2 different cost, different generality. SFT vs RLVR? Both apply within A1 or A2 with different trade-offs. The framework also shows the frontier: agent-tool co-adaptation. Today's leading systems combine multiple paradigms, but the theory of how to combine them is open.
One-line intuition Every paper that "improves agents after pretraining" is doing one of four things and once you see which, the trade-offs become obvious. Adapt the agent or adapt the tool? Use tool execution as signal or use agent output as signal? That's it. Four cells. Cleanest map of the field that exists.

The combinatorial reality

Most leading systems use multiple paradigms. A modern deep research system has T1-style retrievers (pretrained dense + sparse), T2-style adaptive search agents (trained against frozen reasoning model), and A1-style reasoning agents (RLVR with code sandbox + retrieval feedback). The framework's value isn't that you pick exactly one it's that you can name what each component is doing and reason about its trade-offs separately. When a paper modifies both agent and tool, the survey assigns it to the dominant locus and notes the secondary explicitly.

Deep dive · the four questions you'll need to answer
What's the practical difference between RLVR (Reinforcement Learning with Verifiable Rewards) and SFT/DPO?▸
SFT learns to imitate good trajectories; DPO learns from preferences; RLVR learns from a binary verifiable signal (did the code run? did the SQL return correct rows?). RLVR works when you have a clean external verifier.

SFT needs labeled (input, good output) pairs. Cheap, stable, but bounded by the quality of demonstrations. Can't surpass the demonstrator. Works for most A1 / A2 tasks where you have curated trajectories.

DPO needs preference pairs (output A preferred over output B). Avoids learning a separate reward model. Less compute than full RLHF but still bounded by preference data quality. Good for A2 with subjective tasks (writing, dialogue).

RLVR needs a verifiable signal code passes tests, SQL returns expected rows, math answer matches ground truth. The reward is binary or near-binary, computed automatically. No human labelling needed at training time. This is what powers DeepSeek-R1 and the modern reasoning-model wave. The killer constraint: only works where verification is automatable. Math, code, retrieval relevance yes. Open-ended writing, dialogue quality no.

Combination: SFT first to bootstrap a competent base policy, then RLVR (often with PPO or GRPO) to push beyond imitation. KL-regularization keeps the policy close to the SFT baseline so it doesn't degenerate.

∫ Show the math · SFT, DPO, and RLVR objectives▸

SFT (maximum likelihood on demonstrations). Given a dataset $\mathcal{D} = \{(x_i, y_i^*)\}$ of input-output pairs, minimise:

$$\mathcal{L}_{\text{SFT}}(\theta) = -\mathbb{E}_{(x, y^*) \sim \mathcal{D}} \Big[\sum_{t} \log \pi_\theta(y^*_t \mid x, y^*_{

Pure imitation. Bounded above by the demonstrator. No mechanism to surpass demonstrations.

DPO (preference pairs, no reward model). Given pairs $(x, y_w, y_l)$ where $y_w \succ y_l$ (preferred vs dispreferred), DPO fits the policy directly via an implicit reward derived from a reference policy $\pi_{\text{ref}}$:

$$\mathcal{L}_{\text{DPO}}(\theta) = -\mathbb{E}\Big[\log \sigma\Big(\beta \log \tfrac{\pi_\theta(y_w \mid x)}{\pi_{\text{ref}}(y_w \mid x)} - \beta \log \tfrac{\pi_\theta(y_l \mid x)}{\pi_{\text{ref}}(y_l \mid x)}\Big)\Big]$$

Where $\beta$ controls how far $\pi_\theta$ can drift from $\pi_{\text{ref}}$. No explicit reward model is trained the preference comparison becomes the loss.

RLVR with PPO and KL regularisation. Given a verifiable reward $r(x, y) \in \{0, 1\}$ (did code pass tests? did SQL return correct rows?), the objective is:

$$\mathcal{J}_{\text{RLVR}}(\theta) = \mathbb{E}_{y \sim \pi_\theta(\cdot|x)}\big[r(x, y)\big] - \beta \cdot \mathrm{KL}\big[\pi_\theta(\cdot|x) \;\|\; \pi_{\text{ref}}(\cdot|x)\big]$$

Optimised via PPO's clipped surrogate. The KL term is critical: without it, the policy drifts away from the SFT initialisation into degenerate regions that happen to satisfy the verifier but produce garbled language. This is why RLVR is typically SFT-first, then RLVR-second the SFT policy anchors the KL reference.

GRPO variant. DeepSeek-R1's Group Relative Policy Optimisation samples a group of $G$ outputs per prompt and uses within-group ranking as the advantage, avoiding the need for a separate value network:

$$A_i = \frac{r_i - \mathrm{mean}(r_1, \dots, r_G)}{\mathrm{std}(r_1, \dots, r_G)}$$

Simpler, cheaper than PPO and the empirical driver behind the 2025 reasoning-model wave.

Why does pure A2 SFT fail to teach tool use?▸
A2 supervision is on the final answer only. SFT on final answers gives no incentive to call tools the agent can improve answer likelihood without ever invoking a tool.

The paper makes this explicit: "supervising only the final output o* is insufficient for learning tool-use behavior: the agent could improve its final-answer likelihood without ever invoking tools, since the supervision provides no incentive to produce the correct tool call."

Concretely: imagine training on (question, gold answer) pairs. SFT pushes up the probability of the gold answer. If the agent can produce the gold answer directly from its weights without tool calls, it will tools are extra tokens that make the trajectory longer and harder to fit. The model has no learned reason to invoke a calculator, retrieve a fact, or search the web.

The fix: A2-style SFT must combine final-output supervision with tool-call supervision, effectively integrating A2-style supervision with A1-style imitation of tool-use trajectories. Or move to RLVR where tool execution becomes part of the reward computation.

This is one of the cleanest practical insights in the survey explains why naive "fine-tune on QA pairs" approaches to tool use don't work and what to do instead.

∫ Show the math · why A2 SFT starves tool use of gradient▸

Let a trajectory be $\tau = (x, c_1, c_2, \dots, c_k, o)$ where $x$ is the input, $c_i$ are intermediate tool calls + results, and $o$ is the final output. Let $y^*$ be the gold final answer.

Pure A2 SFT on final output only optimises:

$$\mathcal{L}_{\text{A2-SFT}}(\theta) = -\mathbb{E}_{(x, y^*)}\big[\log \pi_\theta(y^* \mid x)\big] = -\mathbb{E}_{(x, y^*)}\Big[\log \sum_{\tau: \mathrm{out}(\tau)=y^*} \pi_\theta(\tau \mid x)\Big]$$

The marginalisation sums over all trajectories that happen to produce $y^*$. The model is rewarded for any trajectory reaching the gold answer including the no-tool trajectory $\tau = (x, o)$ if the model already has the knowledge in its weights.

Gradient pressure: the no-tool trajectory is shorter, simpler, and less risky. Under SFT's maximum-likelihood objective, the model learns to collapse onto it. Tool calls receive zero direct gradient they are never supervised.

The fix (combined A1+A2 SFT). Supervise the full trajectory, not just the output:

$$\mathcal{L}_{\text{combined}}(\theta) = -\mathbb{E}_{(x, \tau^*)}\Big[\sum_t \log \pi_\theta(\tau^*_t \mid x, \tau^*_{

Where $\tau^*$ includes the correct tool calls and the final answer. The model now has explicit supervision on when to call tools, what arguments to pass, and how to integrate results. Same objective form as SFT, different supervision target and that single change is what makes tool use learnable.

What makes T2 (agent-supervised tool adaptation) hard?▸
You're training a tool whose only feedback signal comes through a frozen agent's output. Indirect, sample-inefficient, prone to credit-assignment failures.

T1 is easy in the ML sense: train a retriever on (query, relevant doc) pairs, train a domain model on its native data. The frozen agent doesn't enter the loop.

T2 is harder. Example: s3 trains a 7B searcher whose reward is "Gain Beyond RAG" how much better does the frozen reasoning model do with the searcher's output vs without it. The searcher's gradient flows through:

  • The searcher generates a query and retrieves docs.
  • The frozen reasoning model uses those docs to produce an answer.
  • The answer is scored.
  • That score is the reward for the searcher.

Two problems: (1) the signal is heavily filtered through the frozen agent's idiosyncrasies (a great search result gets a bad reward if the reasoning model happens to ignore it), (2) credit assignment was a good final answer because of the search query? the retrieved doc? the reasoning model's prior knowledge? Sample-inefficient.

Solutions in practice: GRPO-style group relative scoring (compare multiple search queries on the same task), Flow GRPO (for orchestrators in AgentFlow), and warm-starting the tool with T1-style supervised pretraining before T2 fine-tuning. T2 is where the most active research is happening.

∫ Show the math · T2 with Gain-Beyond-RAG reward (s3-style)▸

Let $\phi$ parameterise the trainable tool (e.g. a searcher). Let $\pi_A$ be the frozen agent policy. For a query $x$ with gold answer $y^*$:

$$\text{GBR reward: }\; r(x, \phi) = \mathrm{acc}\big(\pi_A(x, t_\phi(x))\big) - \mathrm{acc}\big(\pi_A(x)\big)$$

Where $t_\phi(x)$ is the searcher's output (retrieved passages) and $\mathrm{acc}(\cdot)$ is a binary correctness check against $y^*$. Reward is non-zero only when the searcher helps the agent get the answer right when it would otherwise fail so the searcher can't trivially win by retrieving popular passages that the agent already knows.

The tool policy maximises:

$$\mathcal{J}_{\text{T2}}(\phi) = \mathbb{E}_{x \sim \mathcal{D}}\big[r(x, \phi)\big] - \beta \cdot \mathrm{KL}\big[\pi_\phi \;\|\; \pi_{\phi,\text{ref}}\big]$$

Typically optimised with GRPO: sample $G$ candidate searches per query, compute GBR for each, use within-group advantage, update via clipped PPO surrogate.

Credit assignment is the core difficulty. The reward $r(x, \phi)$ is a function of three things: (a) the searcher's query generation, (b) the retrieved documents (dependent on an often-frozen retrieval backbone), and (c) the frozen agent's ability to synthesise docs into a correct answer. Gradient attribution to $\phi$ alone is indirect. The signal is also sparse when the frozen agent just can't solve the task regardless of retrieval, $r = 0$ for every action, and the searcher learns nothing from that query. This is why T2 methods typically need either warm-start SFT or careful curriculum design pure RL from cold-start rarely converges.

The 70× data efficiency claim (s3 vs A2 monoliths). s3 reached parity with A2 monolithic agents using 2.4k training examples vs roughly 170k for the A2 baseline. The hypothesis: the tool is a smaller parameter space with a narrower training objective localising adaptation to the skill that needs it ("find good search queries") rather than training the entire reasoning model ("do everything"). The paper treats this as a data-efficiency claim. It's not universal it holds most strongly when the tool's objective is procedural and well-specified, and the gap may narrow or invert for tasks where the bottleneck is reasoning rather than tool use.

Why is agent-tool co-adaptation called the "frontier"?▸
All four paradigms freeze one component while training the other. Co-adaptation trains both jointly much harder, much more powerful, mostly unsolved.

A1, A2, T1, T2 all assume one side is frozen. This makes the optimisation problem manageable but leaves performance on the table the agent and tool can't co-evolve to suit each other's strengths.

The challenges with joint training:

  • Moving target. If both agent and tool change, gradient updates to one shift the optimal behaviour for the other. Without careful alternation, training oscillates.
  • Credit assignment, squared. When a final outcome is good, was it due to better agent reasoning or better tool output? Hard enough with one trainable component; doubled with two.
  • Computational cost. Training two components on long agent trajectories is brutally expensive.

Open research directions the survey calls out: (1) alternating optimisation with theoretical convergence guarantees, (2) bilevel optimisation framings (tool training as inner loop, agent training as outer loop, or vice versa), (3) shared latent representations that let agent and tool co-adapt through a common embedding space. None solved. This is where the next breakthrough probably comes from.

∫ Show the math · bilevel formulation of co-adaptation▸

Let $\theta$ be agent parameters, $\phi$ be tool parameters, $\mathcal{L}$ be a task loss. A1, A2, T1, T2 each fix one of $\{\theta, \phi\}$ and optimise the other. Co-adaptation wants to optimise both but naive joint optimisation $\min_{\theta, \phi} \mathcal{L}(\theta, \phi)$ is unstable because each update shifts the other's optimal point.

The bilevel framing makes the interaction explicit. Pick one as the "leader" and the other as the "follower":

$$\min_\theta \; \mathcal{L}_{\text{outer}}\big(\theta, \phi^*(\theta)\big) \quad \text{s.t.} \quad \phi^*(\theta) = \arg\min_\phi \mathcal{L}_{\text{inner}}(\theta, \phi)$$

The inner problem trains the tool $\phi$ given the current agent. The outer problem updates the agent $\theta$ accounting for how the tool would adapt to this new agent. Unlike naive alternation, bilevel has theoretical convergence guarantees under mild assumptions (smoothness, uniqueness of inner minimum).

The gradient challenge. The outer gradient requires backpropagating through the inner optimisation:

$$\nabla_\theta \mathcal{L}_{\text{outer}}(\theta, \phi^*(\theta)) = \partial_\theta \mathcal{L}_{\text{outer}} + \partial_\phi \mathcal{L}_{\text{outer}} \cdot \nabla_\theta \phi^*(\theta)$$

The second term the hypergradient is computed via the implicit function theorem:

$$\nabla_\theta \phi^*(\theta) = -\big[\nabla_\phi^2 \mathcal{L}_{\text{inner}}\big]^{-1} \cdot \nabla_{\phi\theta}^2 \mathcal{L}_{\text{inner}}$$

Inverting a Hessian over LLM-scale $\phi$ is infeasible. In practice people use truncated unrolling (run a few inner gradient steps, backprop through them) or first-order approximations (drop the second term). Both lose guarantees and can diverge.

Why the survey calls this a frontier. The agent-tool system has exactly the structure bilevel wants, but the scale (tens-of-billions of parameters per side) breaks every clean solution. The likely near-term winners are approximately-bilevel: alternating optimisation with careful step scheduling, warm-starts from A1/A2/T1/T2 baselines, and shared-embedding bottlenecks that reduce the effective dimensionality of the interaction. Nobody has shipped this at scale yet.

§1The fragmentation problem

The paper opens with a simple diagnosis: agentic AI's literature has grown faster than its conceptual structure. The result is a field where similar techniques get reinvented under different names, where comparing methods is hard because they don't share vocabulary, and where practitioners can't reason about which method to pick for a new problem.

Three concrete fragmentation symptoms:

Same idea, different names
RLVR, GRPO, RFT closely related ideas appear in different papers with different terminology, making comparison hard.
Hidden trade-offs
Papers compare against narrow baselines. The cost-vs-flexibility-vs-modularity trade-offs across paradigms are rarely made explicit.
No design map
A practitioner asking "should I fine-tune the LLM, train a retriever, or use a skill library?" has no principled framework to consult.
The unifying notion: adaptation

The survey reframes everything under one concept: adaptation = improving an agent, its tools, or their interaction after pretraining. Pretraining is fixed. Adaptation is everything that comes after and the question is: what adapts, and what's the signal that drives the adaptation?

§2The four-paradigm framework formal definitions

Two binary axes generate four paradigms. The axes:

Axis 1 · WHAT is optimised
Agent the LLM-based reasoner. Includes its weights or its prompt strategies. Tool external components. Retrievers, code interpreters, search engines, memory modules, domain models.
Locus of optimisation. Which component's parameters / behaviour are being changed? The agent is one possible target; tools are another. Both are valid; they have different cost profiles.
Axis 2 · WHAT'S the supervision signal
Tool execution feedback comes from running the tool. Code returns pass/fail. Retrieval has relevance scores. SQL executes correctly or not. Agent output feedback comes from evaluating the agent's final response. Final answer correctness. Preference scores.
Source of training signal. Where does the loss / reward come from? Tool execution gives intermediate, often verifiable signals. Agent output gives final, often subjective signals. Different signals have very different sample efficiencies and correctness properties.
The four cells
A1 = AGENT × tool execution A2 = AGENT × agent output T1 = TOOL × independent (agent-agnostic) T2 = TOOL × agent-supervised
Note that T1 doesn't fit perfectly into the "supervision signal" axis it's tool training that happens independently of the agent, with whatever supervision is natural for that tool (paired data, contrastive learning, etc.). The survey treats it as the "frozen agent" baseline against which T2 is compared.

Worked classification: map four well-known systems to paradigms.

1
Toolformer learns to insert tool calls (calculator, search, etc.) into its outputs by self-supervised training: it inserts candidate tool calls, runs them, and keeps the ones that reduce perplexity on the correct continuation. Reward = tool execution result reducing loss → A1.
2
DeepSeek-R1 is trained with RLVR on math + code: reward is whether the final answer matches ground truth or the code passes tests. Even with reasoning chains, the reward signal lives at the agent output level. → A2.
3
Dense Passage Retriever (DPR) is trained on (question, relevant passage) pairs without involving any agent. A frozen agent later uses it. → T1.
4
s3 trains a 7B searcher whose reward is "Gain Beyond RAG" how much better the frozen reasoning model does with the searcher's retrieved docs vs without. The tool (searcher) is trained, signal comes through the frozen agent's output → T2.
§3A1 deep dive tool-execution-signaled agent adaptation

A1 methods optimise the agent using verifiable outcomes from tool execution. Two sub-eras:

Earlier era · SFT & off-policy
Toolformer introduces self-supervised learning from tool-augmented language modeling. ToolAlpaca, Gorilla use SFT on curated tool-use trajectories. DeepRetrieval uses retrieval relevance as the reward signal. Cheap and stable but bounded by demonstration quality.
Modern era · RLVR with PPO/GRPO
KL-regularized PPO on tool-execution feedback (code sandbox results, SQL execution, retrieval scores). GRPO (Group Relative Policy Optimization) avoids the need for a value network by comparing groups of rollouts. Allows the agent to surpass demonstrators by exploring.
When A1 wins
When tool execution gives a clean verifiable signal AND the agent's tool use is the bottleneck. Code agents, math agents with calculators, retrieval-heavy QA agents.
When A1 loses
When success isn't reducible to per-tool-call signals. Multi-step tasks where tool calls are individually correct but the orchestration is wrong A2's final-output signal catches this; A1 doesn't.
The KL regularisation trick

RLVR-trained agents drift from their pretrained behaviour, which can degrade general capabilities. The standard fix: KL-regularize against the SFT baseline. The PPO objective adds a $\beta \cdot \text{KL}(\pi_\theta \| \pi_{\text{ref}})$ term that penalises drift. Tuning $\beta$ trades exploration vs preservation. Most modern RLVR setups (DeepSeek's GRPO, OpenAI's PPO variants) use this.

§4A2 deep dive agent-output-signaled agent adaptation

A2 methods optimise using evaluations of the agent's final output. Three technical lineages, each with distinct character:

Lineage 1 · Reasoning-centric RL (the R1 paradigm)
DeepSeek-R1, Kimi-1.5 Reward = final answer correctness on math + code. "Reasoning chains" emerge from RL pressure, not supervision.
The breakthrough. RL pressure on verifiable final-answer correctness induces long reasoning chains as a side-effect the model learns that "thinking longer" raises accuracy. Established that RLVR alone (no SFT chain-of-thought labels) is enough to elicit reasoning. Defines the modern reasoning-model recipe.
Lineage 2 · Self-refinement & meta-feedback
Self-Refine, TextGrad Parameter-agnostic adaptation: text-level "gradients" via LLM feedback. Works on black-box LLMs without weight access.
Test-time learning. No weight updates. Instead, the LLM critiques its own output and revises. TextGrad treats the LLM's natural-language critique as a "gradient" and uses it to iteratively improve. Crucial for closed models you can't fine-tune.
Lineage 3 · Multi-tool orchestration RL
ReTool RL-trains agents to optimize tool-use STRATEGY (not just single calls) via final outcome. Reflection + multi-stage rollouts.
Strategy-level optimisation. ReTool and similar work optimise which tools to use, in what order, with what arguments not just per-tool correctness. The reward is end-task success. Bridges A1 and A2: tools are involved, but the signal is on the agent's overall behaviour.

The pure-SFT failure mode. A common A2 mistake explained.

1
Setup: training a tool-using agent. Dataset is (question, gold final answer) pairs. SFT objective: maximise log-prob of gold answer given question.
2
What goes wrong: the model learns to produce gold answers directly from its weights. Tool calls are extra tokens that make the trajectory longer and harder to fit; SFT actively discourages them.
3
The fix: combine A2-style final-output supervision with A1-style tool-call supervision. Or move to RLVR where the reward integrates over tool execution. Pure A2 SFT is the cleanest example of "right paradigm, wrong objective."
§5Tool adaptation · T1 (independent) & T2 (agent-supervised)

Tool adaptation is the other half of the framework and arguably where the most exciting work happens because tools are typically smaller and cheaper to train than full agents.

T1 · Foundational architectures
Dense retrievers (DPR, ColBERT), sparse retrievers (BM25 + learned reweighting), domain-specific models (medical NER, code parsers), and increasingly MCP servers exposing curated tool capabilities. Trained on native data, used by a frozen agent. Plug-and-play.
T2 · The recent breakthroughs
s3: 7B searcher trained with "Gain Beyond RAG" reward. AgentFlow: planner trained with Flow GRPO to orchestrate frozen Qwen modules. Agentic memory modules trained to better support frozen agents (memory writes/reads as a learned policy).
Why T2 matters
Adapts the system to the agent's idiosyncrasies. A retriever trained on (query, doc) pairs (T1) gets generic relevance. A retriever trained against the frozen agent's downstream success (T2) gets relevance for that agent. Often dramatic data-efficiency gains.
The T1→T2 progression
Many production systems start with T1 (use a pretrained retriever), then graduate to T2 (fine-tune it against the deployed agent's feedback) as they accumulate logs. T1 is the cold-start; T2 is the steady-state optimisation.
The data-efficiency story

The survey notes that T2 methods often achieve dramatic data-efficiency gains over A1/A2: training a small focused tool (7B searcher in s3) to support a frozen big agent is much cheaper than training the agent itself. For practitioners, this is the practical takeaway if you're CPU/GPU-poor, train tools, not agents.

§6Comparison + applications + open challenges

The survey ends with a systematic comparison along four dimensions and a tour of applications.

Cost
A1 < A2 (verifiable signals are cheaper). T1 < T2 (independent training is cheaper). Full RL on huge agents is the most expensive; T2 with small tools is often the cheapest path to material gains.
Flexibility
A2 ≥ A1 (final-output rewards apply to more tasks than verifiable signals do). T2 ≥ T1 (tool can be tailored to agent's specific needs). Higher flexibility usually means higher cost.
Generalisation
A1 generalises within tool families. A2 generalises broadly but inherits LLM judge biases. T1 generalises across agents (plug-and-play). T2 specialises to one agent but generalises across that agent's tasks.
Modularity
T1 is most modular (truly plug-and-play). T2 binds tool to agent. A1/A2 modify the agent itself. Modularity matters for long-lived production systems where components get swapped.
Application · Deep research
T1 retrievers + T2 search agents + A1 reasoning agents in cascade. The "deep research" archetype is a multi-paradigm system; the framework lets you reason about each layer separately.
Application · Software development
A1 dominates (code execution = verifiable signal). RLVR with PPO + KL regularization on test-pass reward is the recipe behind Cursor, Devin, etc.
Application · Computer use
A2 with screenshots + UI-state rewards, plus T2 for adapting screen-understanding tools to specific agent prompting strategies. Mixed paradigm.
Application · Drug discovery
T1 dominates domain models (folding, docking, generative chemistry) trained on native scientific data, orchestrated by a frozen agent. The agent's role is workflow planning, not domain expertise.
Frontier · Co-adaptation
No method currently trains agent + tool jointly with theoretical guarantees. Open: bilevel optimisation framings, alternating optimisation, shared latent representations. Whoever cracks this gets the next big paper.
Think like a researcher · the survey's questions for the field
Pitfalls when applying the framework · four mistakes that waste months
Pitfall 1 · Paradigm-shopping
Picking a paradigm before naming the signal you actually have
symptom: team says "let's do RLVR" before identifying the verifier
Teams pick paradigms by reading recent papers "DeepSeek-R1 is hot, let's do A2 RLVR" and then scramble to find a verifiable reward. If your task doesn't have a clean verifier (code tests, SQL equality, retrieval relevance), RLVR will not save you. Fix: start from the signal you can actually compute automatically and work backwards to the paradigm. No verifier → no RLVR; live preference data → DPO or A2; execution-signaled tool use → A1. The paradigm is downstream of the signal, not the other way around.
Pitfall 2 · Agent-first tunnel vision
Spending six months on A1/A2 when T2 would ship in three weeks
symptom: closed-API team building a bespoke "RLHF substitute"
If the agent is a closed API (GPT-4o, Claude, Gemini), A1/A2 is structurally unavailable you cannot fine-tune weights you don't control. Teams still attempt it via increasingly baroque prompt-engineering and TextGrad-style workarounds. Meanwhile, their retriever, reranker, or memory module all clearly T2 targets sits untouched. Fix: audit your stack. If the agent is frozen, resources should flow to T1/T2 first. The survey's clearest practical lesson for resource-constrained teams is that agent-supervised tool adaptation (T2) is often the highest-ROI move.
Pitfall 3 · Single-cell labeling
Classifying a hybrid system into one paradigm, then optimising only that half
symptom: "we're a T2 shop" but nightly SFT on success trajectories is A2
Real production systems span multiple cells: a T1 retriever feeds a T2 reranker whose output feeds an A2-fine-tuned reasoning model. Teams often pick the most prominent component and label the whole system by it which hides the fact that the other components are also being trained, often with conflicting objectives. Fix: document each component's paradigm separately. When something breaks, the component-level paradigm tells you which training pipeline owns the bug. Don't collapse a multi-paradigm system into one label.
Pitfall 4 · Framework-as-law
Treating "this is T2" as a reason not to try something that doesn't fit
symptom: dismissing in-context Skills because "they're not in the 2×2"
The four-paradigm grid covers parametric adaptation methods that change weights. In-context adaptation (Skills, dynamic prompt construction, structured runtime configuration) is explicitly out of scope in the paper. That doesn't mean it's less important for many production stacks it's the dominant adaptation lever. Fix: use the framework as a map of parametric options, not as a list of all options. Your best move may be a well-designed Skill library that sits outside A1/A2/T1/T2 entirely. The framework orients you; it doesn't bound you.
How would you pick a paradigm for a brand new agent product?▸
Decide based on (1) verifiability of feedback, (2) compute budget, (3) how often you'll redeploy. Different combinations point to different cells.

If feedback is verifiable + you can afford LLM training: A1 with RLVR. Strong, generalisable agents that improve directly from execution signal. Software engineering agents typically land here.

If feedback is subjective + you can afford LLM training: A2 with DPO or RLAIF. Final-output preferences via human or LLM judge. Open-ended assistants typically land here.

If feedback is verifiable but you can't afford to train the LLM: T2. Train a small tool (retriever, search subagent, memory module) against the frozen LLM's downstream success. Often the best ROI for resource-constrained teams.

If you need plug-and-play tools across multiple agents: T1. Build well-specified tools (often with MCP) and let any agent orchestrate them. Lower performance ceiling than T2 but more reusable.

Most production deployments combine all four: T1 retrievers as a base, A2 RLVR fine-tuning of the agent on user feedback, T2 fine-tuning of search tools as logs accumulate. The framework just helps you name what each component is doing.

What's the strongest argument that this framework is missing something?▸
It doesn't capture in-context adaptation (prompt engineering, RAG context construction, in-context learning) as a first-class paradigm. Treats them as preliminaries.

The survey explicitly scopes "adaptation" to mean parameter-level changes (or systems where parameter-level changes are the alternative). Prompt engineering, dynamic RAG context, in-context learning, prompt tuning these get mentioned in §2 as background but don't enter the four-paradigm grid.

Counterargument: in 2026, much production "adaptation" is happening at the in-context layer. Anthropic Claude's Skills, OpenAI's structured outputs, custom GPTs these adapt agent behaviour without touching weights. They arguably deserve their own paradigm dimension (in-context vs parametric adaptation).

The survey would say in-context adaptation isn't really "adaptation" in the same sense it's runtime configuration. Whether you buy that depends on how you draw the line. Worth thinking about for your own work.

Three places this framework will get extended▸
(1) Add an in-context vs parametric axis. (2) Add a coordination layer for multi-agent systems. (3) Add explicit treatment of safety/alignment objectives that cut across paradigms.

1. In-context vs parametric. Today's framework only covers parameter-level adaptation. As Skills, MCP, and structured prompting mature, the field will need a third axis distinguishing what gets stored in weights vs in runtime configuration.

2. Coordination dimension. The framework treats single-agent systems. Multi-agent coordination (orchestrator + workers, debate, mixture-of-experts at agent level) doesn't have a clean home. Likely a fifth paradigm or a sub-axis.

3. Alignment objectives. Safety / refusal training cuts across A1/A2/T1/T2 every paradigm has an alignment dimension that's not currently surfaced. As deployment regulations tighten, this becomes a first-class concern that shapes which paradigm you can use for safety-critical applications.

Why blindly applying the framework can mislead
Failure: classifying a hybrid system into a single cell
Real case: a team labelled their system "T2" because they fine-tuned a retriever against agent feedback. But the agent itself was also being SFT'd nightly on logged successful trajectories that's A2. They were doing both, but only optimising one. The framework assignment hid this. Lesson: when a system spans paradigms, name each component's paradigm explicitly. Don't collapse a multi-paradigm system into one label.
Failure: choosing A1/A2 when you can't actually train the LLM
Real case: a startup wanted to "do A2 RLVR" on their agent but they were using GPT-4 via API. They couldn't fine-tune. After two months of trying to make TextGrad work as a substitute, they pivoted to T2 (training a small retriever and reranker against GPT-4's downstream success). Got most of the gains they wanted in two weeks. Lesson: if you can't touch the agent's weights, A1/A2 isn't on the table focus on T2.

Deployment thought experiment · speculative translation to real companies
Which paradigm would each of these companies actually lean into? Our best guesses.
OpenAI / Anthropic / Google DeepMind
Frontier labs · full weight access
All four paradigms, combined. These labs own their base models and can fine-tune. A1 (tool-execution RLVR) for coding/math reasoning; A2 (preference + output) for open-ended dialogue and safety; T1 for reusable retrievers and code-sandbox tooling; T2 for agent-specialised memory and subagents. The survey's framework probably describes their internal org structure more accurately than it describes any single paper each team owns a cell.
Cursor / Replit / Windsurf
Coding-agent startups · frozen base model, in-house tooling
Heavy T2 focus, with A1 where possible. These companies use closed-API base models (or open-weight fine-tunes) but own the full coding environment linters, test runners, static analysers, deployment tooling. That tooling is the natural T2 target: fine-tune retrieval, fine-tune context construction, fine-tune the code-indexing subagent against successful completion signals from the frozen agent. Their competitive moat is the accumulated (query → context → agent output → build-pass) training data, not the base LLM.
Harvey / Legora / Relevance AI
Vertical-domain agent startups (legal, sales, ops)
T1 first, T2 as data accumulates. Vertical startups typically start with off-the-shelf retrievers and frozen agents (T1), because they don't yet have enough in-domain logged interactions to fine-tune productively. As logs accumulate "this retrieval → this drafted contract → did the lawyer ship it unchanged?" T2 becomes tractable. Rushing to A1/A2 before the data foundation exists usually produces overfit agents on small datasets.
Enterprise IT (Salesforce, ServiceNow, Atlassian)
CRM / ticketing platforms adding agents
T1 heavily, T2 selectively, A1/A2 rarely. Enterprise platforms have deep proprietary data (customer records, historical tickets, workflow metadata) but usually treat the LLM as a commodity. Their natural play is T1 (purpose-built retrievers and classifiers over their data) and T2 (fine-tuning those tools against agent outcomes). Fine-tuning the base agent itself sits awkwardly with their "any LLM backend" promise to customers.
Academic labs / research groups
University teams publishing on this survey's topics
A1/A2 RLVR for publishable contribution, co-adaptation for ambitious programs. A PhD student wants a methods paper that will land at a top venue. A1 and A2 with novel reward designs or verification schemes (RLEF, s3, ReTool style) are the most publishable cells in 2026 clean experimental setup, verifiable signal, clear comparison to existing methods. The ambitious programs (Stanford, CMU, UIUC) are moving to co-adaptation, which is the frontier the survey names and where paradigm-defining papers will come from over the next 18-24 months.
Solo builder / small startup (pre-seed)
One person, closed API, no ML team
Mostly in-context, not in the framework at all. Solo builders should probably not attempt A1/A2/T1/T2 at all. The entire four-paradigm grid assumes training infrastructure, labeled data, and evaluation harnesses you don't have. The right play is Skills, MCP, and well-designed prompts in-context adaptation that the framework deliberately excludes. Revisit the framework when you have logged user interactions measured in tens of thousands and at least one person whose full-time job is evals.

The picking decision in one line

Start from the signal you can compute automatically (verifiable → A1 or T2; subjective → A2 or T2 with LLM-judge; in-context only → skip the framework). Then check whether you can touch the agent's weights (closed API → T1/T2 only). The framework is a filter, not a menu most teams have only one or two viable cells given their constraints.

Build it · five steps to apply this framework to your project
1
Audit what's frozen vs what's trainable in your stack
If you use a closed LLM API (Claude, GPT-4o), the agent is frozen A1/A2 are off the table. If you have an open-weight model on your own GPUs, all four paradigms are available. If you have small in-house tools (retrievers, classifiers), T1/T2 are easy. This decision determines the design space.
2
Identify your highest-quality reward signal
Is your task verifiable (code passes tests, retrieval relevance scored, math answer matches)? Then A1 or T2 with verifiable signals. Subjective task (writing, dialogue, support resolution)? Then A2 or T2 with LLM-as-judge or human preferences. The signal source dictates the paradigm choice as much as the optimisation locus does.
forcing A2 SFT on tool-using tasks pure A2 SFT doesn't teach tool use
3
Start with T1, graduate to T2 as logs accumulate
For most production systems, T1 is the cold-start: use pretrained retrievers, MCP servers, off-the-shelf domain models. As you accumulate (query, agent_decision, outcome) logs, T2 becomes feasible fine-tune those tools against the frozen agent's downstream success. This progression is cheaper than starting with A1/A2 and gets most of the practical gains.
4
Compose paradigms name each component's paradigm explicitly
A real system has multiple components, each potentially in a different cell. The retriever might be T1 (DPR off-the-shelf), the reranker T2 (fine-tuned against frozen LLM), the LLM A2 (RLHF'd quarterly), etc. Document each component's paradigm. When you debug a failure, you'll know which component's training to revisit.
5
Watch for the co-adaptation frontier
If you're building a multi-component system, the obvious next step is joint training. Today this is mostly research, but bilevel optimisation, alternating training, and shared-embedding co-adaptation are all under active investigation. Don't ship co-adaptation in production yet but track the literature so you can adopt it when it matures. This is where the next 10x gains probably come from.
Research idea
Add a fifth axis to the framework: in-context vs parametric adaptation. Show that for some tasks, well-designed Skills + structured prompting beat parametric A2 fine-tuning at much lower cost. Practical contribution + framework extension.
Startup idea
"Paradigm-aware AutoML for agents" a tool that audits your agent stack, identifies which paradigm each component is in, recommends which to switch (e.g., "your retriever is T1; with 10K logged sessions you'd benefit from T2 fine-tuning"). Hooks into existing eval/observability tools.
1-line improvement
Document every component of your agent stack with a paradigm tag (A1/A2/T1/T2 + frozen/trainable). On rollouts where something breaks, the tag immediately tells the on-call engineer which training pipeline owns the failure. Cheap; saves hours during incidents.

What you'll leave with. A study plan, not a summary. Below is a reading path through the survey's ecosystem organised by paradigm A1, A2, T1, T2, plus the cross-cutting frontier. Each paper listed is one the survey leans on heavily; each card explains why it's worth reading in this order. 1Foundations · read before any paradigm
InstructGPT ↗
Ouyang et al. · NeurIPS 2022
The RLHF original. Every A2 method descends from this. Why first: you can't understand what A2 is optimising without having seen the reward-model-and-PPO pattern.
Toolformer ↗
Schick et al. · NeurIPS 2023
The A1 original. Self-supervised tool-call insertion via perplexity reduction. Why: the cleanest early example of tool-execution-signaled agent adaptation, before RLVR existed.
DPO ↗
Rafailov et al. · NeurIPS 2023
Reward-model-free preference optimisation. Why: half the A2 papers after 2023 use DPO instead of PPO. Understand the derivation once, save yourself fifty paper-readings.
ReAct ↗
Yao et al. · ICLR 2023
The interleaved reasoning + acting pattern that made agents legible. Why: even though it's a prompting technique (outside the 2×2), most modern agent architectures still use its structure for trajectory representation.
2A1 · tool-execution-signaled agent adaptation
Gorilla ↗
Patil et al. · 2023
SFT on API documentation with AST-based correctness signals. Why: good example of A1 without RL shows how far you can get with careful SFT plus a structured verifier.
ToolAlpaca ↗
Tang et al. · 2023
SFT pipeline for tool use at scale. Why: pairs naturally with Gorilla; together they define the pre-RLVR state of A1.
DeepRetrieval ↗
Jiang et al. · 2025
RLVR applied to retrieval 3× recall improvement over SFT baselines. Why: one of the cleanest demonstrations that RLVR with a verifiable signal beats SFT decisively when the verifier is clean.
RLEF ↗
Gehring et al. · 2024
Grounding code generation in execution feedback via RL. Why: the software-engineering-specific A1 template exactly the pattern Cursor/Replit-style coding agents use.
3A2 · agent-output-signaled agent adaptation
DeepSeek-R1 ↗
Guo et al. · 2025
The reasoning-model breakthrough. Why: defines the "R1 paradigm" of RLVR on final-answer correctness with GRPO. Read the paper then read the model card both are short and essential.
Kimi-1.5 ↗
Kimi Team · 2025
Parallel contribution to DeepSeek-R1 different team, similar formula. Why: reading both shows which decisions are fundamental vs incidental to the 2025 reasoning-model family.
Self-Refine ↗
Madaan et al. · NeurIPS 2023
Iterative self-critique as adaptation signal. Why: the gateway paper to a whole A2 sub-lineage that uses LLM-generated feedback instead of external verifiers.
TextGrad ↗
Yuksekgonul et al. · 2024
"Textual gradients" LLM-generated feedback as a surrogate for actual gradient. Why: the most interesting attempt to do A2 without weight access. The idea may not scale; the framing is influential regardless.
ReTool ↗
Feng et al. · 2025
A2 RL with reflection and multi-stage rollouts for tool orchestration. Why: the state of the art for "A2 that also wants to use tools" which is almost everyone.
4T1 · agent-agnostic tool adaptation
CLIP ↗
Radford et al. · ICML 2021
Foundational vision-text encoder that became a plug-and-play tool. Why: the archetype T1 trained independently, used everywhere. Every multimodal agent stack has one of these.
SAM (Segment Anything) ↗
Kirillov et al. · 2023
The segmentation analogue. Why: CLIP + SAM together show what T1 looks like in 2025 a small library of frozen, task-specific tools that any agent can invoke.
DPR ↗
Karpukhin et al. · EMNLP 2020
Dense Passage Retrieval the retrieval-tool template. Why: the minimum reading to understand how a retriever becomes a T1 component in an agent stack. Short paper, still read.
Model Context Protocol ↗
Anthropic spec · 2024-2026
The production T1 story. Why: MCP is how T1 tools actually get plugged in at scale. Reading the spec is faster than reading three papers about tool-use architectures.
5T2 · agent-supervised tool adaptation (the paper's most actionable cell)
s3 ↗
Jiang et al. · 2025
7B search subagent trained against a frozen reasoning model with "Gain Beyond RAG" reward matches monolithic A2 with roughly 70× less data in the paper's reported setting. Why: the most-cited T2 reference paper, and the one to read first.
REPLUG ↗
Shi et al. · 2023
Perplexity-weighted retriever training against a frozen LM. Why: earliest clean T2 the gradient-through-frozen-agent trick before it had a name.
Memento ↗
Zhou et al. · 2025
Memory modules trained entirely from binary agent outcome rewards. Why: the T2 template for memory specifically which is where the 2026 frontier is moving.
AgentFlow ↗
Zhang et al. · 2025
Flow GRPO trains a planner tool against frozen agents with in-the-flow optimisation. Why: shows what T2 looks like when the "tool" is itself a complex orchestration component.
Awesome-Adaptation-of-Agentic-AI ↗
Jiang et al. · GitHub companion
The survey's official companion repo. Why: continuously updated timeline of every A1/A2/T1/T2 paper with method, tools, backbone, and tuning strategy. Keep this open in a tab while reading the above.
6Frontier · co-adaptation and open problems (where 2026-2027 papers will land)
GEM ↗
Chen et al. · 2025
A benchmark suite for agentic training environments. Why: co-adaptation research is bottlenecked on shared environments. GEM is one of the first serious attempts to provide them.
Self-Evolving Agents Survey ↗
2025
Complementary survey focused on continual and lifelong adaptation. Why: the two surveys together cover most of the 2025 adaptation literature read in sequence after Jiang et al.
Practical Guide for Agentic AI Workflows ↗
2025
Production-grade design guidance. Why: bridges the gap between the Jiang survey's paradigm language and the operational decisions you'll make when deploying.

Question 1 of 8 · The two axes

What two binary choices define the four-paradigm framework?

The 2×2 framework is the survey's central contribution. Memorise these axes they're the conceptual scaffold for everything else in the field.
Question 2 of 8 · Classifying papers

DeepSeek-R1 is trained with RLVR on math + code, where reward = whether the final answer matches ground truth or the code passes tests. Which paradigm?

Where does the reward live? If it's on the agent's final answer, it's A2 even if execution is involved in verifying that answer. The locus of supervision matters more than what the agent does internally.
Question 3 of 8 · The pure-A2-SFT trap

You SFT a tool-using agent on (question, gold final answer) pairs. The agent stops calling tools entirely after fine-tuning. Why?

The locus of supervision shapes what gets learned. Pure A2 SFT teaches "produce the right final answer" not "use tools to get there." Always combine A2 with A1-style tool-call supervision when training tool-using agents.
Question 4 of 8 · T1 vs T2

What's the key difference between T1 and T2?

T1 is plug-and-play; T2 is tailored. Most production systems start with T1 (cold start) and graduate to T2 as they accumulate logs of agent-tool interactions. T2 is where the most active research is.
Question 5 of 8 · KL regularisation

Why do RLVR training setups (DeepSeek's GRPO, OpenAI's PPO variants) typically include KL regularisation against an SFT baseline?

KL regularisation is the standard tool for keeping RL-trained agents close to their pretrained capabilities while still gaining from the verifiable signal. Tune $\beta$ carefully it's the most important hyperparameter in RLVR setups.
Question 6 of 8 · Production paradigm choice

You're a 5-person startup using GPT-4 via API (no fine-tuning). You have 3 months of usage logs. Which paradigm should you focus on?

If you can't touch the agent's weights, train the tools. T2 is the practical answer for resource-constrained teams using closed APIs. Often gets most of the gains of A2 at a fraction of the cost.
Question 7 of 8 · Hybrid systems

A modern "deep research" system uses pretrained dense retrievers + an RL-trained search subagent + an RLVR'd reasoning model. How do you classify this in the framework?

Real systems span paradigms. Use the framework component-by-component, not system-by-system. Each retriever, reranker, agent, etc. has its own paradigm. The architectural document of a production agent should tag every component.
Question 8 of 8 · The frontier

Why does the survey identify "agent-tool co-adaptation" as the main open challenge?

The frontier is joint training of agent + tool. Today everyone freezes one side. Cracking joint training with theoretical guarantees and practical efficiency is where the next 10x agent gains will probably come from.
