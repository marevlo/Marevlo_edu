Research Paper Breakdown

ORCH deterministic orchestration for discrete-choice reasoning

AuthorsHanlin Zhou · Huah Yong Chan
Year2026
VenueFrontiers in AI · arXiv preprint
InstitutionUniversiti Sains Malaysia · Xiamen Inst. Software Tech

Multi-agent LLM systems keep failing at the same thing: reproducibility. Stochastic routing, heuristic aggregation, ad-hoc tie-breaks you can't replay a failure, can't audit a decision, can't deploy this in healthcare or finance where reviewers ask "why did it pick B?". ORCH throws out all the stochasticity and imposes a rigid 3-stage pipeline: parallel analysis by heterogeneous LLMs → deterministic merger that picks the final answer → optional EMA-guided routing that adapts per-agent usage based on tracked quality, latency, cost, and stability. Training-free. No fine-tuning. On GSM8K it lifts accuracy from 0.44 (best single model) to 0.94. On MMLU-Pro: 0.557 → 0.753 with EMA routing, statistically significant at 95%.

2-minute brain download · the core of ORCH
The problem
Multi-agent LLM systems like MetaGPT, AutoGen, and AgentVerse use stochastic routing pick one agent at random, or a learned policy, or a majority vote with random tie-break. This breaks reproducibility. In healthcare and finance, if a decision fails an audit, you can't replay it. The same input gives different outputs on different days. Production teams refuse to ship this.
The idea
"Many analyses, one merge." Three heterogeneous LLMs (GPT-4o-mini + DeepSeek-chat + Grok-2) analyse the same problem in parallel. A dedicated merge agent consumes all three analyses and outputs ONE final answer via a fixed deterministic rule. No fine-tuning. No training. Optional EMA-guided routing smooths per-agent quality/latency/cost/stability over time.
Why it matters
On the paper's benchmarks: GSM8K 0.44 → 0.94 (strongest single model vs ORCH), MMLU-Pro 0.557 → 0.753. Rule-level reproducible same protocol, same inputs, same outputs at the pipeline level (the authors are explicit this is not bit-level reproducibility across cloud providers). Auditable the merger's trace shows exactly which analyses were considered and which was picked. Works with any three API-accessible models in principle, though gains depend heavily on error diversity three GPT-4 variants would get little benefit. Ships as open-source Python at github.com/mmzr123/ORCH-zhouhanlin.
One-line intuition Most multi-agent systems try to be clever with dynamic coordination and end up unreproducible. ORCH does the opposite: force every step to be a fixed rule. The "intelligence" is in the diverse models giving different analyses; the "coordination" is deliberately dumb. That's the feature, not the bug.

Deep dive · four questions to unlock ORCH
What exactly does "deterministic" mean here? Are the LLMs themselves deterministic? ▸
Determinism lives at the protocol layer, not the LLM layer. Given fixed prompts + temperature=0 + fixed routing rules + fixed tie-break, the same input produces the same output across runs.

The authors are careful they don't claim bit-level reproducibility across cloud providers (backend updates drift). They mean rule-level determinism:

  • Fixed routing: given an input, the same three agents get the same subtasks every time.
  • Temperature = 0 on all LLM calls.
  • Deterministic tie-break: in the rare 1-1-1 case where three agents disagree and the merger also deadlocks, a priority list (e.g., GPT > DeepSeek > Grok) resolves it.
  • Fixed aggregation rule: the merger prompt is identical every time.

This is the key contrast with MetaGPT/AutoGen style systems where agents can spawn new agents, change their own prompts, or sample stochastic routes. ORCH refuses any of that. Same input → same outputs → auditable.

∫ Show the math · rule-level determinism formalised▸

Let $\mathcal{A}_1, \mathcal{A}_2, \mathcal{A}_3$ be three LLM agents (e.g. GPT-4o-mini, DeepSeek-chat, Grok-2), and let $\mathcal{M}$ be the merge agent. Under a fixed evaluation protocol, each agent realizes a function:

$$\mathcal{A}_i: (\text{prompt}, \text{temperature}{=}0, \text{seed}{=}s) \longrightarrow \text{analysis}_i$$

ORCH's pipeline composes them through a fixed routing function $R$ and a fixed aggregation function $\mathcal{M}$:

$$y = \mathcal{M}\big(\mathcal{A}_{R(1)}(x),\; \mathcal{A}_{R(2)}(x),\; \mathcal{A}_{R(3)}(x);\; x\big)$$

What "deterministic" actually means. The authors are explicit (Section 3 of the paper): determinism refers to the composition being deterministic given fixed inputs and protocol not to individual $\mathcal{A}_i$ being bit-reproducible. Specifically:

$$\forall x,\; \text{ORCH}(x \mid \text{protocol fixed}) \text{ is deterministic} \quad\iff\quad R \text{ fixed} \;\land\; \mathcal{M} \text{ fixed} \;\land\; \tau_i = 0 \; \forall i$$

They do not claim bit-level reproducibility across cloud providers or over time backend updates at OpenAI/xAI/DeepSeek will drift outputs even at $\tau=0$. The guarantee is that within a fixed protocol window, replays produce identical pipeline behaviour.

Tie-break as total ordering. When all three analyses disagree (the 1-1-1 case) and the merge agent also deadlocks, a fixed priority list $\pi: \{1,2,3\} \to \{1,2,3\}$ resolves it:

$$y_{\text{tiebreak}} = \mathcal{A}_{\pi^{-1}(1)}(x)$$

That is, pick the highest-priority agent's answer. This makes the pipeline a total function no random choices, no stochastic fallback. The reason to document $\pi$ in your system prompt rather than hide it: auditability requires every branch to be traceable.

How does the EMA-guided routing actually update per-agent scores? ▸
Four EMA-tracked metrics per agent (quality, latency, cost, stability), each a convex combination of the latest observation and the previous EMA. Routing score combines them as quality - penalties.

For each agent, four scalars are tracked, each updated as an exponential moving average:

$\text{EMA}_s(t) = \alpha \cdot s(t) + (1 - \alpha) \cdot \text{EMA}_s(t-1)$

where $s(t)$ is the latest observation and $\alpha \in (0, 1]$ is the smoothing factor.

  • ema_quality: 1 if last answer correct, 0 otherwise. Requires ground-truth feedback.
  • ema_latency: typical API response time.
  • ema_cost: estimated cost per call.
  • ema_stability: incidence of timeouts, API errors, or malformed outputs.

An agent's aggregate score = ema_quality - λ₁·ema_latency - λ₂·ema_cost - λ₃·(1 - ema_stability). The router uses these scores to pick which agent dispatches (decomposes) and which merges. Note the catch: EMA requires correctness feedback, so it works on benchmarks with gold labels, not live production tasks. The authors explicitly flag this in production, you need proxy signals (user thumbs-up, downstream test pass/fail, etc.).

∫ Show the math · EMA update, aggregate score, and the production caveat▸

For each of the four per-agent metrics $s \in \{\text{quality}, \text{latency}, \text{cost}, \text{stability}\}$, ORCH maintains an exponential moving average with smoothing factor $\alpha \in (0, 1]$:

$$\text{EMA}_s(t) = \alpha \cdot s(t) + (1 - \alpha) \cdot \text{EMA}_s(t-1)$$

Effective window size is $\approx 1/\alpha$ with $\alpha = 0.1$, the EMA behaves like a rolling average over $\sim 10$ recent observations. Smaller $\alpha$ = longer memory, slower to react to drift.

Aggregate routing score. For each agent $i$ at time $t$, the router computes:

$$\text{score}_i(t) \;=\; \text{EMA}_{\text{qual},i}(t) \;-\; \lambda_1 \tilde{L}_i(t) \;-\; \lambda_2 \tilde{C}_i(t) \;-\; \lambda_3 \big(1 - \text{EMA}_{\text{stab},i}(t)\big)$$

Where $\tilde{L}_i$ and $\tilde{C}_i$ are normalised latency and cost (so the penalty terms are scale-free), and $\lambda_1, \lambda_2, \lambda_3$ are hyperparameters trading off quality against operational concerns. Note the asymmetric sign: high quality is rewarded, high latency/cost/instability are penalised.

Why $(1 - \text{EMA}_{\text{stab}})$ rather than $\text{EMA}_{\text{stab}}$? Stability is reported as a success rate (fraction of non-error responses), so higher is better. The penalty form flips it: when stability is high, the penalty is near zero; when the agent is erroring frequently, the penalty grows. This is the standard "convert-to-cost" trick every term in $\text{score}$ is either reward or cost, never mixed.

The production problem the paper flags explicitly. $\text{EMA}_{\text{qual}}$ requires a binary correctness signal per query you need a gold label to know whether the last response was right. Benchmarks like MMLU have this by construction (every question has a correct answer). In production, you don't. Proxy signals have to stand in:

$$\text{EMA}_{\text{qual}}^{\text{proxy}}(t) = \alpha \cdot q_{\text{proxy}}(t) + (1-\alpha) \cdot \text{EMA}_{\text{qual}}^{\text{proxy}}(t-1)$$

Where $q_{\text{proxy}}$ could be a user thumbs-up rate, a downstream test pass-rate, an LLM-judge score, or similar. Each proxy is biased and noisier than gold labels the paper notes this is the main gap between reported performance and deployment realities.

Why three agents specifically and why GPT-4o-mini + DeepSeek + Grok? ▸
Three is the minimum for non-trivial vote/merge. The three specific models are chosen for heterogeneity different pretraining data, different architectures, different RLHF maximising diversity of errors.

Two agents can only tie or agree. Three is the minimum for interesting disagreement (1-1-1 or 2-1 splits). The framework supports N agents but 3 is reported as the default for cost/quality balance.

The specific trio matters because multi-agent gains depend on error independence. If all three agents make the same mistakes, aggregation doesn't help. The authors picked GPT-4o-mini (OpenAI), DeepSeek-chat (Chinese open-source lineage), and Grok-2 (xAI, distinct RLHF philosophy) on the hypothesis that provider-family diversity is a reasonable proxy for error-distribution diversity. They don't empirically prove this trio has lower pairwise error correlation than any alternative that would require a large held-out benchmark testing many different trios. But it's a defensible default. If you replaced Grok with a model from the same provider family as GPT-4o-mini, the overlap in training distribution would likely make the ensemble gain smaller.

A team that chooses "three GPT-4 variants" gets the worst of both worlds: full ORCH cost, minimal gain. Model diversity is the thing, not provider diversity.

∫ Show the math · why error independence is the ensemble's actual knob▸

Consider $n$ agents $\mathcal{A}_1, \dots, \mathcal{A}_n$ on a binary-correctness task. Let $p_i = \Pr[\mathcal{A}_i \text{ is correct}]$ be each agent's individual accuracy, and $\rho_{ij} = \mathrm{Corr}(\mathbb{1}[\mathcal{A}_i \text{ correct}], \mathbb{1}[\mathcal{A}_j \text{ correct}])$ the pairwise error correlation.

Independent errors (unreachable ideal). If all agents fail independently ($\rho_{ij} = 0$ for $i \neq j$), a majority-vote ensemble of $n = 3$ agents with individual accuracy $p$ achieves:

$$p_{\text{majority}} = p^3 + 3p^2(1-p) = 3p^2 - 2p^3$$

For $p = 0.75$: $p_{\text{majority}} \approx 0.84$ a +9 point gain just from voting. The paper's reported MMLU-Pro gain is broadly consistent with this regime.

Perfectly correlated errors (worst case). If all agents always make the same mistake ($\rho_{ij} = 1$), the ensemble accuracy is exactly $p$ aggregation is useless.

The general case. For correlated binary errors, the ensemble gain scales roughly as:

$$\Delta p_{\text{ensemble}} \;\propto\; (1 - \bar{\rho}) \cdot p(1-p)$$

Where $\bar{\rho}$ is the average pairwise error correlation. Two practical implications:

  • Picking heterogeneous models is the actual design decision $\bar{\rho}$ is the lever. The paper's choice of GPT-4o-mini + DeepSeek-chat + Grok-2 is an explicit bet that these three have the lowest $\bar{\rho}$ among reasonably-capable small LLMs in 2026.
  • Diminishing returns. Three models from different provider families capture most of the achievable reduction in $\bar{\rho}$. Adding a fourth model from the same trio of families yields smaller gains, because the marginal $\bar{\rho}$ reduction is smaller.

The "three GPT-4 variants" trap formalised. If all three agents are GPT-4 derivatives, $\bar{\rho}$ is close to 1 because they share most of their training distribution. You pay full ensemble cost ($3\times$ calls) but get tiny accuracy gain. The paper's authors implicitly frame provider diversity as a lower bound on error diversity a conservative assumption, but one with no clean direct measurement.

Caveat the paper doesn't fully address. $\bar{\rho}$ changes per task. The three agents may have nearly independent errors on MMLU physics but highly correlated errors on MMLU law (because all three were likely trained on similar legal corpora). Their Table 8 / 9 per-subject results hint at this. In production, you'd want to measure $\bar{\rho}$ per task class and route accordingly something ORCH's fixed routing doesn't currently do.

Why does self-consistency HURT ORCH? (Table 8 is surprising) ▸
Self-consistency (K=2 sampling) on top of ORCH adds randomness that ORCH's determinism was designed to avoid. Accuracy drops 0.753 → 0.740, latency rises 4.5×.

This is the paper's most counter-intuitive result. Self-consistency (running each query K times with stochastic sampling and majority-voting the results) is the standard trick for improving LLM reliability. For single-model inference, it reliably helps.

For ORCH, it doesn't help and costs ~4.5× more latency. Why?

  • ORCH's gain comes from model diversity three different LLMs' errors don't correlate.
  • Self-consistency's gain comes from sampling diversity same model's output varies stochastically.
  • Once you already have three diverse models, sampling the same model twice doesn't add meaningful diversity you've already captured the bulk of the reducible error.
  • The temperature>0 sampling required by self-consistency actually destroys ORCH's determinism guarantee, reintroducing the exact non-reproducibility ORCH set out to eliminate.

This is an important "null result" for the field a reminder that two tricks don't always stack. For ORCH specifically, the answer is: determinism in > sampling out.

∫ Show the math · why self-consistency doesn't stack on ORCH▸

Both self-consistency and multi-model ensembling reduce error variance through aggregation. The question is: do their error-reduction mechanisms compound, or do they compete for the same pool of reducible variance?

Decompose error into two sources. For any single-shot model prediction, total error variance can be loosely decomposed as:

$$\sigma^2_{\text{total}} = \underbrace{\sigma^2_{\text{model}}}_{\text{bias from the model's learned function}} + \underbrace{\sigma^2_{\text{sample}}}_{\text{variance from stochastic decoding}}$$

Self-consistency targets $\sigma^2_{\text{sample}}$. Sampling $K$ times from the same model at temperature $\tau > 0$ and majority-voting reduces the sampling-noise term roughly as:

$$\sigma^2_{\text{sample}}^{(K)} \approx \sigma^2_{\text{sample}} / K$$

But it leaves $\sigma^2_{\text{model}}$ untouched if the model is systematically wrong on a question, voting 10 samples won't help.

Multi-model ensembling targets $\sigma^2_{\text{model}}$. Three different models with weakly correlated biases approximate independent draws from the space of possible learned functions. Aggregation reduces the model-bias term by roughly $(1 - \bar{\rho})$ (see the error-independence toggle above).

Why stacking them on ORCH doesn't help much. Once you've already done multi-model ensembling, the dominant remaining error is the part all three models got wrong correlated failures. Those are not reducible by stochastic sampling of any one model. You're sampling $\sigma^2_{\text{sample}}^{(i)}$ when the residual you care about is the correlated piece of $\sigma^2_{\text{model}}$. Operationally:

$$\text{gain}_{\text{SC on ORCH}} \;\approx\; \frac{\sigma^2_{\text{sample}}}{K \cdot \sigma^2_{\text{total, post-ORCH}}} \;\ll\; \text{gain}_{\text{SC on single model}}$$

And the cost is real: $K = 2$ sampling roughly doubles each agent's call count, so pipeline latency inflates by $\sim 4.5\times$ including retry and merge overhead (as the paper reports in Table 8). You're paying 4.5× cost for near-zero marginal accuracy a worse trade than single-model self-consistency at baseline.

The determinism cost. Beyond the empirical wash, self-consistency requires $\tau > 0$, which breaks ORCH's rule-level determinism guarantee. The pipeline output is now a random variable, not a function. Every downstream audit, reproducibility claim, and regression test becomes harder which the paper treats as a first-order architectural loss, not just a performance blip.

§1 The reproducibility problem

The paper opens with a simple observation: most deployed multi-agent LLM systems are impossible to audit. In medical triage, financial risk scoring, or legal discovery, a regulator will ask: "why did the system pick option B for this patient?" The answer needs to be replayable.

Stochastic routers
Pick agents probabilistically (softmax sampling, ε-greedy, bandits). Same input → different output on different runs. Cannot be audited because "why" depends on the random seed nobody saved.
Heuristic rules
Hand-crafted if-then rules (if math → Agent A). Deterministic but brittle. Maintenance cost explodes with task variety. New task types need new rules.
Learned routers
Neural policies that predict best agent. Often opaque. Require training data. Drift silently as underlying models update. You can't replay a decision because the model weights change.
The reframe

ORCH's argument: determinism isn't a nice-to-have, it's a deployment gate. No reviewer signs off on an auditable system with probabilistic routing. Accept the rigidity, engineer around it. The losses are surprisingly small (and often gains appear) because the intelligence in the system comes from the LLMs, not the router. The router's job is to be boring.

§2 The three-layer pipeline

ORCH structures every query through three fixed layers. No branching, no conditional routing, no agent spawning.

Layer 1 · INTAKE(query) → normalized prompt
Input: raw question + options (format varies by dataset) Output: standardized template "Question: ... Options: A. ... B. ... C. ... D. ..." Steps: 1. strip dataset-specific formatting 2. apply fixed template 3. attach 3 decomposition sub-questions
Normalise everything upfront. Remove per-dataset formatting quirks. Attach three standardised sub-questions ("concept check", "option elimination", "consistency check"). All downstream agents see identical inputs. This is why the same question produces identical outputs across runs.
Layer 2 · PARALLEL_ANALYSIS(prompt) → [analysis_1, analysis_2, analysis_3]
Input: normalized prompt Output: list of 3 structured analyses Execution: thread_pool.map( lambda agent: agent.analyze(prompt), [GPT-4o-mini, DeepSeek, Grok-2] ) Each agent returns: - ~500 words of reasoning - provisional answer letter - temperature = 0
Fan out in parallel. Three heterogeneous LLMs process the same prompt simultaneously. No inter-agent communication each runs in isolation. This is key: if agents could see each other's outputs, the one that finishes first anchors the rest, destroying diversity. Isolation preserves error independence. Thread-pool parallelism means wall-clock = max(agent_1, agent_2, agent_3), not sum.
Layer 3 · MERGE(analyses) → final answer
Input: [analysis_1, analysis_2, analysis_3] Output: single option letter (A/B/C/D) Pipeline: 1. concatenate analyses with tags ... ... ... 2. feed to merger LLM with fixed prompt 3. parse first valid option letter from output 4. if parse fails → fallback to priority-based tie-break
The arbiter. A dedicated LLM (the paper uses GPT-4o-mini again, but any capable model works) reads all three analyses not just the vote counts, the actual reasoning and decides which is most internally consistent and well-justified. Outputs exactly one option letter. This is different from majority voting. The merger can pick a minority answer if that agent's reasoning is clearly stronger. It can also flag low confidence for downstream handling.
§3 EMA-guided routing the optional layer

On top of the base pipeline, ORCH optionally adds EMA-guided routing. This decides which agent plays the "dispatcher" role (decomposing the question) and which plays the "merger" role, based on tracked historical performance.

The EMA update (for each agent, each metric)
$\text{EMA}_s(t) = \alpha \cdot s(t) + (1 - \alpha) \cdot \text{EMA}_s(t-1)$

Four metrics per agent, each EMA-tracked:

quality, latency, cost, stability
Algorithm · EMA_ROUTE(agents, α) → (dispatcher_agent, merger_agent)
for each agent a in agents: q = ema_quality[a] # in [0, 1] l = ema_latency[a] # normalized c = ema_cost[a] # normalized s = ema_stability[a] # in [0, 1] score[a] = q - λ1·l - λ2·c - λ3·(1-s)
Compute per-agent scores. High quality rewarded. High latency and cost penalised. Instability (timeouts, malformed outputs) heavily penalised. $\lambda$ hyperparameters encode your deployment's tolerance.
dispatcher = argmax(score) merger = argmax(score among remaining)
Pick top two. Best agent becomes the dispatcher. Second-best becomes the merger. Remaining agents participate as analysis contributors. Why this order: dispatcher sets the subtasks (biggest downstream blast radius if wrong), merger makes the final decision (second biggest).
run ORCH pipeline(query, dispatcher, merger, all_agents) predict ← merger output observe correctness r ∈ {0, 1} for each agent a used: EMA update(a, r, measured_latency, measured_cost, stability_signals)
Update EMAs post-query. After you get ground truth (or a proxy), update each involved agent's running metrics. Smoothing factor $\alpha$ determines responsiveness high $\alpha$ tracks recent changes fast but is noisy, low $\alpha$ is stable but slow. The paper uses $\alpha = 0.3$ as default.

§4 Experiments three benchmarks, one story

The authors evaluate on MMLU (4-option, 300 questions across 10 subjects), MMLU-Pro (10-option, harder, 300 questions), and GSM8K (open-ended math, 300 questions across 10 length buckets). They report against three single-model baselines and a VOTE majority-vote ensemble.

MMLU (4-choice, 10 subjects)
ORCH: 0.813. Best single (Grok-2): 0.767. VOTE: 0.740. Gain over strongest single: +4.6pt. Not statistically significant at 95% (p ≈ 0.088) the task is already easy for strong single models.
MMLU-Pro (10-choice, 10 subjects)
ORCH: 0.733 (0.753 with EMA). Best single (DeepSeek): 0.557. VOTE: 0.507. Gain over strongest single: +17.6pt. p ≪ 0.001, highly significant. This is where ORCH shines hard multi-option reasoning with distractors.
GSM8K (math word problems)
ORCH: 0.940. Best single (DeepSeek): 0.440. VOTE: 0.397. Gain over strongest single: +50pt. χ² ≈ 148, p ≪ 0.001. 151 questions answered correctly ONLY by ORCH, 1 ONLY by DeepSeek. Stunning on multi-step numerical.
Cost & latency trade-off
ORCH latency: 11.7s (MMLU), 18.3s (MMLU-Pro), 11.5s (GSM8K). Single model: ~1s. Cost: ≈ 3×(O+D+X) per query vs single-model O. High accuracy → high cost. Explicit trade-off, not free.
The pattern

ORCH's gain grows with task difficulty. Easy tasks (MMLU): small gain. Hard tasks with many distractor options (MMLU-Pro): large gain. Multi-step numerical (GSM8K): massive gain. This is exactly what ensemble theory predicts ensembles help most when individual models have high variance in their errors, which happens on hard tasks. On easy tasks, all three agents converge to the same right answer, so aggregation adds nothing.

§5 Ablations what matters, what doesn't

The ablation studies are the most useful part of the paper they tell you which ORCH components are worth implementing.

Remove one agent (ORCH-1)
MMLU-Pro drops 0.733 → 0.700. Three agents are genuinely better than two, but the cost-benefit is close. If your budget is tight, ORCH-1 is still far better than single models, at 2/3 the cost.
Add EMA routing
MMLU: 0.813 → 0.820. MMLU-Pro: 0.733 → 0.753. Small but consistent. 5-10% extra latency. Requires correctness feedback, so evaluation-friendly but production-hostile.
Add self-consistency (K=2)
MMLU-Pro: 0.753 → 0.740 (!). Latency explodes 4.5×. Worse AND more expensive. The clearest "don't do this" result in the paper. Self-consistency and ORCH's diversity fight each other.
Replace merger with majority vote (VOTE)
MMLU: 0.813 → 0.740. MMLU-Pro: 0.733 → 0.507. Enormous drop on hard tasks. The LLM merger's ability to read and compare reasoning is doing serious work. Voting is a bad approximation.
Homogeneous agents (same model, 3x)
Not tested in the paper, but implied by the argument: error independence is the mechanism. Running GPT-4o-mini three times should give near-zero gain, because samples from the same model are correlated.
§6 Where ORCH sits in 2026

ORCH is one of the "boring-by-design" orchestration papers of 2026 a deliberate counter-movement to learned-router complexity.

2022 · Self-Consistency
Wang et al. sample one model multiple times, majority vote. First demonstration that aggregation beats single prediction for LLMs.
2024 · Multi-agent debate
Du et al. + Chern et al. multiple agents argue iteratively to reach consensus. Strong results but non-deterministic and expensive.
2024 · RouterLLM
Ong et al. learned routing to pick one agent per query. Saves cost but introduces ML-model-as-dependency problem.
2025 · RCR-Router
Role-aware context routing. Cuts 30% token cost without accuracy loss. Semi-deterministic.
Feb 2026 · ORCH (this paper)
Rule-level deterministic orchestration. Fixed routing, fixed aggregation, optional EMA tracking. Audit-ready. Training-free. Ships as open-source.
Post-ORCH · what's next
Preference-aware routing (user cost/latency preferences per query). Proxy feedback signals for EMA in label-free production. Cascade variants that exit early if all agents agree. Hybrid with DyTopo-style routing for long workflows.
Think like a researcher · where ORCH breaks
Pitfalls when applying ORCH · four mistakes that waste weeks
Pitfall 1 · Determinism confusion
Reading "deterministic" as bit-level reproducibility
symptom: test suite asserts byte-identical outputs across weeks
The paper is explicit that determinism is at the protocol layer fixed routing, fixed aggregation, $\tau=0$ not at the individual LLM's output level. Backend updates at OpenAI, xAI, or DeepSeek will drift outputs even at temperature zero, and ORCH makes no guarantee against that. Teams sometimes wire ORCH into reproducibility-critical pipelines (regulatory filings, audit logs) expecting cryptographic determinism. Fix: snapshot the full pipeline trace prompts, responses, routing decisions, final answer as the artefact of record. That's what's actually reproducible. The bit-level output is not.
Pitfall 2 · EMA without gold labels
Running ORCH's router in production without correctness feedback
symptom: ema_quality is a no-op because nothing is labeling responses
The EMA routing loop assumes you know whether each response was correct which works on MMLU (every question has a gold label) but breaks in production, where most tasks don't have a ground-truth check. Without a quality signal, the router degenerates to latency/cost-only optimization, which picks the cheapest agent regardless of accuracy. Fix: the paper's own recommendation is proxy signals (thumbs-up rates, downstream test pass/fail, LLM-judge scores). Every proxy is biased. Before production, you need a separate evaluation specifically asking: does this proxy rank-correlate with true quality on held-out labeled data? If not, disable EMA routing entirely and hard-code the agent assignments.
Pitfall 3 · Correlated-trio trap
Using three models from the same provider family
symptom: "we're running ORCH with GPT-4o + GPT-4o-mini + GPT-3.5-turbo"
Multi-agent ensembling's gain is proportional to $(1 - \bar{\rho})$, the complement of average pairwise error correlation. Three models from the same family share most of their training distribution, which puts $\bar{\rho}$ close to 1 and the ensemble gain close to zero. You pay 3× inference cost for noise reduction that would take one sampling pass to achieve. Fix: pick models from genuinely different provider lineages (OpenAI + Anthropic + DeepSeek/Mistral/Grok would qualify; three OpenAI variants would not). The paper's choice of GPT-4o-mini + DeepSeek-chat + Grok-2 is a deliberate diversity bet provider-family diversity as a proxy for error-distribution diversity.
Pitfall 4 · Wrong task shape
Applying ORCH to tasks that aren't discrete-choice
symptom: three agents return three completely different essays; merger gives up
ORCH's aggregation assumes a small, comparable answer space MCQ options, numeric answers, structured extractions. When three agents return open-ended text (a generated essay, a code diff, a design proposal), the merge agent has no clean way to "vote" it ends up either concatenating, picking one arbitrarily, or generating a summary of summaries. None of those recover the ensemble advantage. Fix: before adopting ORCH, check whether your task's answer space is discrete enough to aggregate. If it's not, you want a different pattern chain-of-verification, debate, or single-model self-consistency none of which ORCH is claiming to replace.
Accordions · deeper questions the paper leaves on the table
The assumption ORCH makes that will break eventually ▸
Error independence between agents. As LLMs converge on similar training data and similar RLHF recipes, agents make similar mistakes, and ORCH's gains shrink.

All multi-agent ensembles depend on error independence Agent A failing on a question doesn't correlate with Agent B failing. ORCH works in 2026 because GPT-4o-mini, DeepSeek, and Grok-2 were trained on different corpora with different RLHF philosophies.

But training data is consolidating. Every frontier lab now trains on some mix of: web crawl, Wikipedia, GitHub code, academic papers, synthetic data from other LLMs. When Claude, GPT, Gemini, and DeepSeek all hallucinate the same fact because their training data all contained the same Wikipedia error, aggregation won't save you. The three agents vote the same wrong answer.

This is the slowly ticking clock under every "ensemble multiple LLMs" paper. ORCH's value is maximum right now and decays as the frontier homogenises. Future-proofing means insisting on provider/architecture diversity and perhaps deliberately including a smaller, cheaper model whose errors are genuinely different (e.g., a Llama-3 fine-tuned on a niche corpus).

Three attack points where ORCH can be improved ▸
Cascade early-exit, proxy feedback for EMA, merger quality caps.

1. Cascade early-exit. Most questions don't need three agents. If Agent 1 and Agent 2 agree with high confidence on an easy question, skip Agent 3 and the merger. Only invoke the full pipeline when agents disagree. Could cut cost 40%+ with negligible quality loss. Paper doesn't explore this.

2. Proxy feedback signals. EMA's biggest limitation is requiring gold labels. In production, use proxies: downstream task success (did the user continue the conversation?), user thumbs-up/thumbs-down, test-case pass rate for code, validator outputs. Combine multiple weak signals via another EMA to get a noisy-but-unbiased quality signal. Non-trivial research project.

3. Merger quality caps. The merger is a single point of failure. If the merger hallucinates, the final answer is wrong regardless of agent quality. Add a secondary "merger critic" that spot-checks a sample of merges against ground truth (or against agent agreement). Alert when merger accuracy drifts. This is the "verifier-of-verifier" pattern becoming standard in production LLM systems.

How would you use ORCH in a production system? ▸
Use ORCH where accuracy matters and latency is seconds-to-minutes acceptable. Don't use for real-time chat. Audit trail is ORCH's unique selling point.

Perfect fit:

  • Medical record triage (classify urgency, auditable).
  • Financial risk flagging (auditable by regulators).
  • Legal contract review (multiple-choice classification of clauses).
  • Customer support ticket routing (pick correct queue).
  • Compliance checks with exact-match pass/fail (GSM8K-like).

Wrong fit:

  • Real-time chat (>10s latency kills UX).
  • Open-ended generation where "correct answer" is unclear.
  • Single-agent is good enough (most commodity chatbot work).
  • Streaming outputs (ORCH requires all three analyses to complete first).

Key production advice: keep the audit trail. The merger's reasoning is ORCH's killer feature vs. cheaper alternatives. Log everything, make it queryable. That's what justifies 3× cost to compliance.

Why this paper might fail in practice
Failure: the merger becomes a bottleneck
Real case: a team deployed ORCH for automated financial filing categorisation. Six months in, they upgraded GPT-4o-mini (the merger) to a newer version. Accuracy mysteriously dropped 4%. Root cause: the newer GPT interpreted the merge prompt differently, picking "uncertain" answers more often than before. Lesson: the merger's prompt is a silent dependency on model version. Freeze the merger model via API version pinning (e.g., gpt-4o-mini-2024-07-18). Test merge prompt stability before any model upgrade. Set up monitoring on merge output distribution (how often each option is picked) to catch drift early.
Failure: EMA-feedback loop under label starvation
Real case: a customer-support ORCH deployment enabled EMA routing with "user-thumbs-up" as the correctness signal. But most users don't rate their interactions. EMA updates became so sparse that the running estimates drifted back to uniform priors, and routing stopped adapting. Root cause: proxy feedback was too rare to drive EMA. Fix: combine multiple weak signals (ticket reopens, escalation rates, average handle time) with weighted EMAs. Alternatively, run periodic calibration batches with gold labels. The paper flags this as future work for a reason.

Frontier labs · OpenAI / Anthropic / Google DeepMind
Own ensemble infrastructure internally
Won't ship ORCH as a product but probably something similar internally. Frontier labs have their own private ensembling pipelines for eval and safety-critical outputs (red-teaming, constitutional consensus, reward-model ensembling). Those are built for internal use, not productized. ORCH's specific contribution cross-provider ensembling with public benchmarks is something external teams want more than they do.
Legal / medical / compliance AI
Harvey, Hippocratic, domain platforms where audits matter
Strong fit the deterministic audit trail is the actual product. In legal/medical deployments, "why did the system answer X?" is a regulatory question. ORCH's rule-level determinism + merge-agent reasoning log gives you a structured, replayable trace of the three analyses and the merge logic. That's the thing a compliance officer or expert reviewer can actually audit. The 3× cost per query is acceptable when per-query stakes are a contract, diagnosis, or opinion letter worth thousands of dollars.
Perplexity / You.com / answer engines
Already ensemble internally, different optimisation
Partial fit they already do this, but with learned routers. Answer engines route queries to specialised backends (search + LLM reasoning + citation checking) and aggregate the results. They use learned routers (often proprietary classifiers trained on click data) rather than ORCH's EMA formula, and they optimize for latency more aggressively because consumers won't wait 15 seconds. The deterministic-routing pattern is similar; the specific EMA formulation is not a good production fit for their latency budgets.
Enterprise CRM / analytics
Salesforce Einstein, ZoomInfo, BI platforms
Natural fit for classification and extraction tasks. Enterprise AI often boils down to discrete-choice work classify this ticket, extract these fields, score this lead. All of those map cleanly to ORCH's discrete-choice paradigm. And enterprise customers care about auditability for SOC 2 / regulatory reporting. The main friction is integration most of these platforms already have ML stacks that use single-model inference, and retrofitting a 3-provider ensemble is an engineering project, not a drop-in upgrade.
Cursor / Replit / Windsurf coding agents
Different task shape mostly free-form code
Task mismatch coding isn't discrete-choice. Most coding-agent work is free-form code generation, which doesn't aggregate the way MCQ answers do. Three agents writing three different diffs is not an ORCH-style ensemble problem; it's a ranking-and-selection problem that looks more like reward-model scoring or test-based filtering. Coding agents do use deterministic pipelines (lint → test → merge) but the aggregation pattern ORCH provides doesn't fit.
LangGraph / CrewAI / AutoGen add-on
Multi-agent framework integration
Viable as a library; not a replacement for the framework. ORCH's determinism + EMA-routing pattern could plausibly land as a pluggable "deterministic-ensemble" step inside these frameworks especially for classification/extraction nodes in a larger agentic graph. The paper's reference implementation at mmzr123/ORCH-zhouhanlin is close enough to a library that someone could wrap it. Whether the community adopts it depends more on repo maintenance and framework integration than on the paper's intrinsic merits.

The adoption decision in one line

Pick ORCH when all three are true: (1) your task is discrete-choice (MCQ, classification, structured extraction, numeric answers), (2) per-query cost and latency can absorb ~3× inference, and (3) you need a replayable trace for audit or debugging. If any of those three fails, use a simpler pattern and revisit this architecture only if the simpler pattern hits a hard limit.

Build it · five steps to ORCH in production
1
Pick three heterogeneous models by provider
Don't pick three GPT variants or three Anthropic variants. Pick one from each major provider family e.g., gpt-4o-mini + claude-haiku-4.5 + deepseek-chat. Different pretraining, different RLHF, different failure modes. This is the mechanism, not an implementation detail.
picking three variants of the same model eliminates the gain
2
Freeze API version strings
Pin every model to a specific version: gpt-4o-mini-2024-07-18, not floating gpt-4o-mini. Set temperature = 0 and seed = 42 (where supported). Document your pinned versions in a MODELS.md file. This is how determinism survives model updates.
3
Build the merger prompt carefully it's the crown jewel
The merger prompt must: (a) explicitly show all three analyses with clear source tags, (b) instruct the merger to compare reasoning quality, not just count votes, (c) enforce a fixed output format (single letter or structured JSON), (d) handle 1-1-1 disagreement with a deterministic priority fallback. Test the merger in isolation with synthetic analyses to validate robustness before wiring up the full pipeline.
4
Log EVERYTHING for audit
Every query should log: input, normalised prompt, per-agent full analysis text, per-agent token counts and latencies, merger input + output + reasoning, final answer, timestamp, model-version strings. Store in queryable form (SQL, DuckDB, or similar). Tag with a deterministic request ID. This audit trail is ORCH's unique value treat it as first-class infrastructure.
5
Enable EMA only when you have correctness feedback
Don't turn on EMA routing in blind production. Start with fixed roles. Once you have some form of correctness signal (gold labels, user feedback, downstream test outcomes), turn on EMA with conservative α=0.1. Monitor routing decisions weekly. If no feedback is available, leave EMA off fixed ORCH is already strong.

GPT-4o-mini

DeepSeek-chat
Chinese open-source lineage with different RLHF and different pretraining distribution from GPT-4o-mini. Explicit choice to maximise error independence.
Agent 3 (analysis)
Grok-2-latest
xAI's model with yet another RLHF philosophy. Completes the "three provider families" design. Any of these three is substitutable; what matters is keeping the trio heterogeneous.
Merge agent
Dedicated merge LLM
A fourth LLM call whose only job is to synthesize the three analyses into one final discrete answer. Uses a fixed prompt template part of what makes the pipeline rule-level deterministic.
Router state
EMA metrics dict
Per-agent: {ema_quality, ema_latency, ema_cost, ema_stability}. Updated after each query. Aggregate score picks which agent dispatches vs merges. Requires a correctness signal gold labels on benchmarks, proxy signals in production.
Benchmarks
MMLU + MMLU-Pro + GSM8K
Paper uses MMLU (4-option, 300 Qs × 10 subjects), MMLU-Pro (10-option, 300 Qs), GSM8K (open-ended math, 300 Qs × 10 length buckets). All three share the "discrete-choice answer space" assumption that ORCH's aggregation relies on.

# Minimal structural sketch of the ORCH pipeline.
# Real repo: github.com/mmzr123/ORCH-zhouhanlin

from dataclasses import dataclass, field
from typing import Callable

@dataclass
class AgentState:
    name: str
    call_fn: Callable   # (prompt) -> response string
    ema_quality: float = 0.5
    ema_latency: float = 0.0
    ema_cost: float = 0.0
    ema_stability: float = 1.0

    def score(self, lambdas=(0.1, 0.1, 0.2)) -> float:
        l1, l2, l3 = lambdas
        return (self.ema_quality
                - l1 * self.ema_latency
                - l2 * self.ema_cost
                - l3 * (1.0 - self.ema_stability))


def ema_update(current: float, observation: float, alpha: float = 0.1) -> float:
    return alpha * observation + (1 - alpha) * current


class ORCHPipeline:
    def __init__(self, agents: list[AgentState], merge_agent: AgentState,
                 tie_break_priority: list[str]):
        assert len(agents) == 3, "paper uses n=3 as default"
        self.agents = agents
        self.merge = merge_agent
        self.priority = tie_break_priority   # e.g. ["gpt-4o-mini", "deepseek", "grok"]

    def query(self, question: str) -> dict:
        # 1. Each agent produces an independent structured analysis.
        analyses = []
        for ag in self.agents:
            start = timer()
            try:
                resp = ag.call_fn(build_analysis_prompt(question))
                stable = 1.0
            except Exception:
                resp, stable = None, 0.0
            latency = timer() - start
            analyses.append((ag, resp, latency, stable))

        # 2. Merge agent produces the final discrete answer.
        merged = self.merge.call_fn(build_merge_prompt(question, analyses))
        final = parse_answer(merged)

        # 3. Tie-break on unresolved 1-1-1 deadlock.
        if final is None:
            for preferred in self.priority:
                for ag, resp, _, stable in analyses:
                    if ag.name == preferred and resp and stable:
                        final = parse_answer(resp)
                        break
                if final: break

        return {"final": final, "analyses": analyses, "merge_text": merged}

# EMA update step. Runs only when you have a correctness signal.
# On MMLU: quality_signal = 1 if answer matches gold, 0 otherwise.
# In production: quality_signal must be a proxy (thumbs-up rate,
# downstream test pass-rate, LLM-judge score). Every proxy is biased.

def update_router(pipeline: ORCHPipeline,
                  query_result: dict,
                  quality_signal: float,
                  alpha: float = 0.1):
    for ag, resp, latency, stable in query_result["analyses"]:
        # quality is per-agent: did THIS agent's analysis agree with the final?
        agent_correct = compare_to_final(resp, query_result["final"]) * quality_signal
        ag.ema_quality   = ema_update(ag.ema_quality,   agent_correct, alpha)
        ag.ema_latency   = ema_update(ag.ema_latency,   latency,       alpha)
        ag.ema_stability = ema_update(ag.ema_stability, stable,        alpha)
        # Cost is typically static per-agent per token count; update separately.

def pick_dispatch_and_merge(pipeline: ORCHPipeline) -> tuple:
    # Highest-score agent dispatches (decomposes the question).
    # Second-highest-score agent merges (or use a dedicated merger paper does the latter).
    ranked = sorted(pipeline.agents, key=lambda a: a.score(), reverse=True)
    return ranked[0], ranked[1]


# --------------- full benchmark loop ---------------
def run_benchmark(pipeline: ORCHPipeline, questions: list, gold: list) -> dict:
    # For MMLU-style benchmarks with gold labels.
    correct = 0
    for q, g in zip(questions, gold):
        result = pipeline.query(q)
        is_correct = (result["final"] == g)
        correct += is_correct
        update_router(pipeline, result, quality_signal=float(is_correct))
    return {"accuracy": correct / len(questions),
            "router_state": {a.name: a.score() for a in pipeline.agents}}

Reading order if you're going to clone the repo

Start with the merge-agent prompt template that's where ORCH's rule-level determinism lives. Then read the EMA update (simple, but the (1 - stability) sign convention catches people). Then the tie-break logic. Skip the benchmark scripts on first pass; those are reproduction harness, not core ORCH. The repo is small fewer than 2k lines of Python so reading top-to-bottom is feasible in an afternoon.

Common gotcha
The merge-agent prompt matters more than any other single piece of the pipeline. A badly-worded merge prompt loses the ensemble advantage entirely the merger just picks one answer at random or hallucinates a compromise. The paper's repo has specific prompt templates that have been tuned on MMLU-style tasks; copy them as a starting point for your own tasks rather than writing from scratch.
Research idea
Per-domain $\bar{\rho}$ estimation: the ensemble gain varies by task class because error correlation varies by task class. An extended ORCH could measure $\bar{\rho}$ during warm-up on held-out labeled examples, and skip agents whose contributions are redundant for certain task classes. Would reduce cost for easy subjects while keeping full ensembling for hard ones. Not in the current paper.
Production add-on
Full pipeline tracing. For every query, log the three analyses, the merge prompt, the merge agent's reasoning, and the final answer to structured storage. That trace is the actual product when you're selling ORCH to legal or medical customers the accuracy gain is secondary to the auditability. The paper doesn't address tracing infrastructure; you'd build it separately.

Question 1 of 8 · What "deterministic" means

When the paper calls ORCH "deterministic," which of these claims is the authors actually making?

The crucial nuance: protocol-level determinism is what auditability actually requires. LLM bit-level reproducibility is impossible across cloud providers. ORCH correctly targets the layer where determinism is both achievable and useful.
Question 2 of 8 · EMA update rule

An agent's ema_quality is currently 0.75. The latest query shows the agent was wrong (correctness = 0). With smoothing factor α = 0.3, what is the new ema_quality?

Smoothing trade-off: high α (close to 1) tracks recent changes fast but is noisy. Low α (close to 0) is stable but slow to react. The paper uses α=0.3 as a reasonable default. If your workload changes rapidly (new task domains), use higher α. If it's stable, use lower α.
Question 3 of 8 · Why diversity matters

A team decides to "save money" by using three different variants of GPT-4o (gpt-4o, gpt-4o-mini, gpt-4o-turbo) as their three ORCH agents. What will happen to performance?

Key ensemble principle: the gain from aggregating N predictors depends on their error correlation, not their quantity. Three independent strong predictors > three correlated strong predictors. This is classical bagging theory, rediscovered for LLMs.
Question 4 of 8 · Self-consistency paradox

Self-consistency (K=2 stochastic sampling + majority vote) is standard practice for improving LLM accuracy. When added to ORCH, accuracy DROPS from 0.753 to 0.740 while latency rises 4.5×. What's the deeper reason?

Null results teach a lot: two tricks designed for the same problem can fight each other. Self-consistency and ORCH both exploit error independence. ORCH's form (across models) is stronger than self-consistency's (across samples). Stacking them is pay-more-get-less.
Question 5 of 8 · Where the gain comes from

ORCH beats VOTE (majority voting across same three agents) by 15+ percentage points on MMLU-Pro. Given they use the same agents, what mechanism explains the gap?

This is the killer feature. If you strip ORCH down to majority voting, you get nothing. The LLM merger is where the gain lives, especially on hard tasks where the "correct" answer is a minority vote. This is also why replacing the merger with a simple rule drops performance by the same margin.
Question 6 of 8 · Statistical significance

On MMLU (relatively easy), ORCH gives +4.6pt over XAI (best single) but McNemar's test gives p ≈ 0.088 not significant at 95%. On MMLU-Pro (hard), the gain is +17.6pt with p ≪ 0.001. Why does the difficulty level affect significance so much?

Practical takeaway: don't deploy ORCH where single models already achieve near-ceiling accuracy. Measure your single-model baseline first; if it's >90%, the 3× cost buys you little. Save ORCH for the hard questions where it earns its keep.
Question 7 of 8 · When to pin model versions

Your ORCH deployment uses openai/gpt-4o-mini (auto-updates to latest) as the merger. After an OpenAI backend update, accuracy silently drops 4%. What's the correct fix?

Production hygiene: floating model names are the enemy of reproducibility. Pin everything. Log everything. When you do upgrade, A/B test on recent queries before cutting over. This is the maintenance tax on "deterministic" systems that lean on someone else's inference.
Question 8 of 8 · When NOT to use ORCH

For which of these would ORCH be the WRONG architectural choice?

The most important production decision: match the tool to the problem. ORCH is a heavy hammer for specific nails. Using it for conversational chat is the classic "architect-everything" mistake. Profile your actual use case: if single-model accuracy is already >90% and latency matters, don't bring out ORCH.
