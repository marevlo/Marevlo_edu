Graph-based Agent Memory when structure beats vector search

Lead authorsChang Yang · Chuang Zhou · Yilin Xiao + 15
Year2026
VenuearXiv survey · Feb 2026
TypeComprehensive survey · 18 authors

Vector RAG forgets what humans wouldn't: relationships, temporal order, and contradictions. Two facts about the same entity stored as two embeddings have no link; an event that happened "before" another has no precedence; the same fact in two phrasings drifts into inconsistency. This survey of graph-based agent memory argues that structure is the missing ingredient. Treat memory as a typed graph entities as nodes, relations as edges, time as a quad and you unlock relational reasoning, hierarchical organisation, and bounded retrieval. The paper systematises the field along the memory life cycle: extraction → storage → retrieval → evolution. Plus taxonomy across temporal scope, cognitive structure, function, and representation. The reference document for anyone building agent memory in 2026.

2-minute brain download · why graph memory matters
The problem with flat memory
Vector RAG stores text chunks as independent points in embedding space. Relationships disappear ("X manages Y" becomes two ungrounded facts). Time disappears ("Alice changed jobs in March" has no temporal anchor). Contradictions silently coexist (two phrasings of the same fact drift apart). Multi-hop reasoning collapses because there are no hops to traverse.
The graph reframe
Memory becomes a typed graph: entities/events/concepts are nodes; relations (semantic, temporal, causal) are edges. Updates merge into existing nodes instead of duplicating. Time becomes a first-class dimension via temporal quadruples (s, r, o, t). Retrieval traverses neighbourhoods instead of grabbing top-k similar text. Reasoning walks the structure.
Why now
2025-2026 saw a wave of production graph-memory systems (Zep, A-MEM, Graphiti, MAGMA, AriGraph). The survey unifies them under one taxonomy + life cycle. Anyone shipping agent memory in 2026 needs this map: which graph structure for which use case, what failure modes, what's open. Reference document, not a one-shot read.
One-line intuition Vector memory is a filing cabinet drop documents in, search by similarity. Graph memory is a web of connections every memory item knows what it relates to, when it happened, and what it contradicts. The structure does the reasoning that flat search can't.

The big idea: pick the structure to fit the memory type

The survey's central insight is that different memory types want different graph structures. Long-term semantic facts → KG (RDF triples). Session histories with summaries → hierarchical. Time-changing facts (job, address, preference) → temporal graph. Group events with multiple participants → hypergraph. Most production systems use a hybrid KG for stable knowledge, temporal graph for dynamic state, hierarchical for chat history. One structure rarely fits all.

Deep dive · four questions the survey forces you to think through
Why does flat vector RAG actually fail in production agents?▸
Three structural blind spots: lost relationships, no temporal ordering, silent contradictions. Each compounds in long-running deployments.

Lost relationships. "X manages Y" gets stored as a chunk. Three months later, "Y was promoted to lead engineer" is stored as a separate chunk with no link to X. A query "who does X currently manage?" hits both chunks via similarity but has no way to connect them. Multi-hop reasoning collapses.

No temporal ordering. "Alice works at Acme" was true in 2024. By 2026 she's at Bigco. Both chunks coexist with no precedence. The agent will return either depending on retrieval ranking randomly inconsistent answers about state.

Silent contradictions. "User prefers dark mode" stored. Six months later, "User changed to light mode." Both chunks live in the index. Retrieval might surface either. The system has no concept of updates every fact is a new addendum, never an overwrite.

Graphs solve all three by being structured: a single canonical node per entity, typed temporal edges, and an explicit update mechanism that resolves conflicts at write time.

What's "bi-temporal" and why does it matter?▸
Two timestamps per fact: when it was true (valid time) and when you learned it (transaction time). Lets you ask "what did the agent know about X on March 1?" critical for auditability.

Single-timestamp temporal graphs answer "what's true now" but conflate two things: when did the fact become valid in the world, and when did the system record it. These differ when information arrives late.

Example: User tells the agent "I moved to Berlin last month" on March 15. Valid time = February 15 (when they actually moved). Transaction time = March 15 (when the system learned). A bi-temporal store records both as a quad-with-two-times. Queries can ask "where did the agent think the user lived on March 1?" → answer: still London, because the system didn't learn about Berlin until March 15.

This matters for: (a) audit trails ("why did the agent do X?"), (b) replaying past decisions to debug failures, (c) handling out-of-order updates from multiple sources, (d) personalisation that respects when preferences became active.

How does memory consolidation work and why is it the hardest part?▸
Consolidation = merging similar nodes, resolving conflicting edges, abstracting common patterns into higher-level concepts. Hard because every choice is irreversible and policy-dependent.

Without consolidation, the graph grows linearly with input duplicate nodes for "John Smith" / "John D. Smith" / "Smith, J.", redundant edges for the same relation phrased two ways, dead nodes from one-off mentions never used again.

Consolidation operations from the survey:

  • Node merging: identify entities that refer to the same real-world thing (entity resolution) and merge them. Hard: false merges are unrecoverable; missed merges fragment the graph.
  • Edge dedup + conflict resolution: two edges saying contradictory things about the same node-pair → keep newer? higher-confidence? both with conflict tag? Policy decision.
  • Abstraction / generalisation: ten facts about ten different users preferring dark mode → induce a "preferences[dark_mode]" pattern. Lossy but compresses.
  • Forgetting: nodes/edges below significance threshold get pruned. Significance can be use-frequency, recency-weighted, or task-relevance-driven. Critical for keeping the graph healthy long-term.

Every consolidation step is a write to the persistent store and is hard to undo. Production systems usually run consolidation in batch (nightly) with snapshots so they can roll back if a merge turns out to be wrong.

Why six retrieval operators (similarity, rule, temporal, graph, RL, agent)?▸
Each captures a distinct retrieval pattern. Real systems compose them similarity for candidate generation, then graph traversal for expansion, then rule-based filters for hard constraints.

The survey distinguishes these because no single operator covers all queries:

  • Similarity: "find memories about onboarding" fuzzy, semantic. Good for recall, bad for precision.
  • Rule-based: "memories from last Friday between 2 and 4pm tagged 'meeting'" sharp constraints, deterministic.
  • Temporal: "what did I last say about X" or "show changes between t1 and t2" time-aware retrieval.
  • Graph: "everyone Alice has worked with on projects involving Y" multi-hop traversal that vector search literally cannot do.
  • RL-based: learned policy that picks the right operator and search depth based on query and observed reward. Adaptive.
  • Agent-based: the retriever IS another agent that can call external tools, plan multi-step retrieval, recursively refine. Open-ended.

Production systems typically compose: similarity (top-k candidates) → graph traversal (1-hop expansion) → rule filter (recency window) → rerank. Each stage compensates for the others' weaknesses.

The paper · the four-stage life cycle
📄
What you're about to read. A survey, walked end-to-end. Six sections: motivation → taxonomy → extraction → storage → retrieval → evolution & applications. The structure mirrors the survey's central organising principle: the memory life cycle.
§1Why memory is the next bottleneck

The paper opens with what's now an obvious observation: LLMs solved short-context reasoning. The frontier is long-horizon tasks multi-session dialogues, scientific discovery, game agents, code agents working over weeks. All of these require persistent memory that supports actual reasoning, not just retrieval.

Three failure modes of memory-less agents:

Knowledge cutoff
LLM doesn't know about events after training. Without external memory, every conversation starts from scratch.
Tool incompetence drift
Agents re-derive how to use the same API every call, making same mistakes. No procedural memory of what worked.
Performance saturation
Without memory, agents can't compound. The 100th task of a kind is no easier than the 1st.
Why specifically graph memory?

Graphs naturally model relational dependencies (entities + relations), hierarchical organisation (parent/child structure), and efficient retrieval (traversal from anchor nodes). Vector stores can do similarity but not relations. Key-value stores can do exact lookup but not multi-hop. Graph is the structure that subsumes both.

§2A taxonomy across four dimensions

The survey's biggest contribution is a clean taxonomy organising the entire field along four axes. Use it as a checklist when designing your own memory system.

Axis 1 · Temporal scope
Short-term: current conversation context, active reasoning trace. Limited capacity. Long-term: persistent across sessions, accumulated knowledge, learned patterns.
Working vs persistent. Most production agents need both. Short-term memory is usually the LLM's context window or a small key-value buffer. Long-term needs an external store with explicit reads/writes.
Axis 2 · Cognitive structure (six types)
Semantic facts/definitions Procedural skills/routines Associative concept links Working scratchpad Episodic chronological events Sentiment emotional tone
Inspired by human memory psychology. Each type maps to different storage + retrieval needs. A customer-support agent needs episodic (past tickets) + sentiment (customer mood) + semantic (product facts) + procedural (resolution playbooks). Different graph structures fit different cognitive types.
Axis 3 · Functional role
Knowledge memory: passive, static, global verifiable facts (e.g., canonical product catalog). Experience memory: proactive, dynamic, specific interactions (e.g., user X's preferences, action outcomes).
The synergy is the point. Knowledge memory tells the agent what's true in general; experience memory tells it what's happened with this specific user. Combining both = personalised but factually grounded responses.
Axis 4 · Representational structure
Non-structural: linear buffer, vector DB, key-value store. Flat. No relations modelled. Structural (graph-based): KG, hierarchical, temporal, hypergraph, hybrid. Explicit relations.
The thesis. The survey argues that for agents with long-horizon tasks, structural (graph-based) memory dominates because it makes relations and time first-class. Non-structural memory remains useful as a layer (vector index over graph node embeddings, e.g.).
§3Memory extraction · turning text into nodes & edges

Memory has to come from somewhere. The paper categorises extraction by source and technique:

From textual sources
NER + relation extraction for triples (entity, relation, entity). Semantic embedding for similarity-indexable nodes. LLM-driven summarisation for condensed memory fragments.
From sequential trajectories
Event segmentation with timestamps. Dynamic state snapshots at key moments. Pattern mining to discover recurring routines (procedural memory).
From multimodal data
Vision-language captioning turns images into text-extractable content. Joint multimodal embeddings map across modalities into a shared space.
Quality control
Extraction is where most errors enter the graph. Bad triples become noisy nodes; bad timestamps poison temporal queries. Validation at extraction time pays off across the entire memory life cycle.
§4Storage · the four graph structures

Once you have triples / events / chunks, where do they go? Four canonical structures, each fitting a different memory type:

Structure 1 · Knowledge Graph (KG)
(head, relation, tail) triples Conflict resolution at write time Schema evolution
For: long-term semantic + associative memory. Ideal for stable facts about entities. Examples: Zep, Graphiti, A-MEM. Trade-off: rich relations, but n-ary facts get split awkwardly into multiple binary triples.
Structure 2 · Hierarchical (tree / DAG)
Multi-level summaries Semantic clustering Recursive abstraction
For: procedural + episodic memory, especially session histories. Lower levels = raw events; upper levels = summaries. Examples: MemTree, RAPTOR. Trade-off: great for hierarchical reasoning, weaker for cross-tree relational queries.
Structure 3 · Temporal Graph (TG)
(s, r, o, t) quadruples Bi-temporal: valid time + tx time "Tree of Time" hierarchical constraints to prevent chronological hallucinations
For: episodic + working memory where time matters. Tracks state changes, supports "as-of" queries. Examples: Zep's bi-temporal layer, AriGraph. Trade-off: richest temporal reasoning, complex to query.
Structure 4 · Hypergraph
Hyperedges connect N nodes Preserves n-ary relations ("meeting with A, B, C about topic X")
For: group events, complex facts that lose structure when split into binary triples. Examples: HyperGraphRAG. Trade-off: highest expressivity, most computational overhead, fewer mature tools.
Hybrid (most production systems)
KG for stable knowledge + Temporal graph for state + Vector index over nodes + Hierarchical summaries for chat sessions
The realistic answer. No single structure fits all memory types. MAGMA uses four orthogonal graphs (semantic, temporal, causal, entity); Zep combines KG + temporal + vector. The survey's takeaway: compose structures by memory type.
§5Retrieval & evolution · making memory adapt

Storage without retrieval is a write-only graveyard. The survey covers six retrieval operators (similarity, rule, temporal, graph, RL, agent) plus three enhancement strategies: multi-round retrieval, post-retrieval reformulation, hybrid-source merging. Real systems compose multiple operators.

Worked example: Customer asks "When did I last open a ticket about billing?"

1
Similarity retrieval over ticket node embeddings → top-20 candidates with terms like "billing", "invoice", "charge".
2
Rule filter: status = "opened" (not "viewed"), customer_id = current user. Drops to 8 candidates.
3
Temporal sort + take latest. Returns ticket #4827 from March 12.
4
Graph expansion (1-hop): pull related entities (the agent who handled it, the resolution status, linked ticket #4901). Now the agent can answer not just "when" but "what happened next."

The evolution stage is what separates static graphs from agent memory. Internal self-evolution: consolidation (merge similar nodes), graph reasoning (infer new edges from existing structure), graph reorganisation (forgetting low-significance nodes). External self-exploration: feedback-driven adaptation (RL on retrieval policy), active inquiry (agent generates queries to fill its own knowledge gaps).

The active-inquiry frontier

Most current systems are passive they record what happens. Active-inquiry agents detect uncertainty in their own graph and generate questions to fill the gaps ("I have facts about X's job but not their start date let me ask"). This is where graph memory becomes genuinely self-evolving rather than just a sophisticated log.

§6Applications + open challenges

The survey's last sections enumerate domains where graph memory is shipping today and what's still open.

Conversational agents
Cross-session memory of user preferences, history. Zep, A-MEM, Mem0 in production.
Code agents
Knowledge of codebase structure (call graphs are literally graphs), past bug-fix patterns. Aider, Cursor, Devin variants.
Recommender + financial
User-item bipartite graphs + temporal preferences. Time-aware recs.
Game / robotics / science
World-model graphs (AriGraph), procedural memory of routines, scientific knowledge graphs for hypothesis generation.
Memory graph quality
Extraction errors propagate. Hallucinated nodes contaminate retrieval. Validation pipelines under-developed.
Scalability + efficiency
Graph traversal cost scales with neighbourhood size. Production systems with millions of nodes need careful indexing.
Privacy + security
Persistent memory of personal facts is a regulatory minefield. GDPR right-to-be-forgotten on a graph is non-trivial.
Dynamic schema learning
Most KGs assume a fixed schema. Real agent memory needs to invent new relation types as it encounters new domains.
Multi-agent memory coordination
When N agents share a graph, who can write what? How are conflicts resolved? Mostly open.
Think like a researcher · the gaps the survey reveals
What's the one finding from this survey that should change how you build agent memory?▸
Stop using one structure for all memory types. Compose structures: KG for facts, temporal for state, hierarchical for sessions, hypergraph where you need n-ary.

The most common production mistake: pick one memory architecture (vector RAG, or one KG, or one summarisation tree) and try to make it serve every memory need. Every type of memory has different access patterns. Trying to force them through one structure leaves you with a system that's mediocre at everything.

The survey's hybrid-systems section makes this explicit. Real production systems (MAGMA's four orthogonal graphs, Zep's KG + temporal + vector) succeed because they recognise this. Each memory type gets its native structure; a router decides which to query. The router is where most of the engineering effort goes.

Where will the next breakthrough come from?▸
Active-inquiry memory + dynamic schema learning. Agents that ask their own clarifying questions and invent new relation types as they go.

Today's graph memory is mostly passive it records what happens. The next generation will be active: the agent inspects its own graph, finds gaps, and generates queries (to the user, to external sources) to fill them. This shifts memory from a log to an investigation.

Combined with dynamic schema learning the agent inventing new relation types when its current schema can't express something it observed this gets you genuinely open-ended memory growth. The fixed-schema KG is a placeholder; the schema-evolving graph is the future.

What's the survey's biggest blind spot?▸
Cost & latency analysis. The survey is structural, not operational. Building these systems in production is more about query latency, write throughput, and graph maintenance cost than about which structure is "best."

The survey covers what graph structures exist and what techniques are out there but is light on the engineering reality: a million-node temporal graph is slow to traverse; nightly consolidation can take hours; conflict resolution policies have opinions about your data. None of this is in the paper.

For your own thinking: when you read the survey, mentally annotate each technique with "what does this cost at 10M nodes?" and "what's the write latency?" The literature is ahead of the engineering choose techniques that have a path to production.

Why this paper might oversell graph memory
Failure: KG construction quality is the silent killer
Real case: a team built a KG-memory system using LLM-extracted triples from chat history. Six weeks in, the graph had thousands of nodes but ~30% were duplicates (different surface forms of the same entity), ~15% had hallucinated relations, and entity-resolution false-merges had merged two distinct customers into one node. Retrieval looked clever; the data was garbage. Lesson: graph quality > graph structure. Spend more on extraction validation, entity resolution, and conflict resolution than on which graph type to use.
Failure: graph traversal latency at scale
Real case: a customer-support agent on a graph with 5M nodes. Retrieval = "similarity top-20 + 2-hop expansion." Worked great at 50K nodes. At 5M nodes, 2-hop expansion blew up to 10K-node neighbourhoods, took 4 seconds per query. Users hated it. Fix: bounded-degree expansion (cap neighbours per hop), pre-computed common subgraphs, query-time pruning by edge type. The survey doesn't dwell on this it's an engineering problem you have to solve yourself.

Build it · five steps to production graph memory
1
Pick structures by memory type, not by preference
Audit your agent's memory needs first: list each type (facts, state, sessions, skills, preferences). Each gets its native structure. Don't pick "a knowledge graph" pick "KG for stable user facts, temporal graph for state changes, hierarchical for chat sessions." This is the single biggest design decision.
2
Invest in extraction quality before structure choice
Bad triples poison every downstream stage. Build extraction validators: schema constraints, entity-resolution scoring, edge-confidence thresholds. Reject low-quality extractions back to a human queue or quarantine. 30% of graph memory production failures trace to bad extraction.
trusting LLM extraction without validation your graph will rot
3
Compose retrieval operators, don't pick one
Production retrieval is always a pipeline: similarity (candidates) → graph traversal (1-2 hop expansion) → rule filter (recency, entity type) → rerank. Build it as a composable chain. Each stage has a concrete role. Don't try to make one operator do everything.
4
Run consolidation in batch with snapshots
Node merging, edge dedup, abstraction these are destructive writes. Run them nightly (or weekly) on a snapshot. Always keep the previous N snapshots so you can roll back if a consolidation choice was wrong. False merges are unrecoverable without a snapshot.
5
Measure graph health, not just retrieval accuracy
Standard metrics: node count, edge count, degree distribution, consolidation merge rate, dead-node rate, contradiction count. A graph that's growing too fast OR too slow is a sign extraction or consolidation is broken. Build dashboards for these before shipping. Treat your graph as production infrastructure with SLAs.
Research idea
Schema-evolving KG: agent that invents new relation types when it can't express what it observed. Triggered when extraction confidence is low + new relation candidate has high prevalence. Could close the dynamic-schema gap the survey calls out.
Startup idea
Graph memory observability: a Datadog-for-agent-memory product. Hooks into Zep / Graphiti / Mem0 deployments and surfaces graph health metrics, anomalous merges, hallucinated edges, growth-rate alarms. Sell to enterprises shipping agent memory who currently fly blind.
1-line improvement
Add a "contradiction count" metric to any KG-based memory. Whenever a write contradicts an existing edge, log it. The trend line tells you whether your extraction quality is degrading. Single number that catches a class of silent failures.
