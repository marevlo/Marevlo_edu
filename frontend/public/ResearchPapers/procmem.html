ProcMEM agents that save skills and reuse them without retraining

TitleLearning Reusable Procedural Memory from Experience via Non-Parametric PPO
Year2026
VenuearXiv preprint · 2602.01869
FamilyMemory & skills for LLM agents

Every time an agent solves a task, it throws the solution away. Next time a similar task arrives, it re-derives the same procedure from scratch the same web-search pattern, the same tool-call sequence, the same fix for the same edge case. Tokens wasted, errors re-made. ProcMEM treats procedural knowledge as storable state: past trajectories are distilled into reusable skills; a non-parametric PPO policy learns which skill to recall when, without modifying the LLM weights. The promise: agents that get measurably faster and more accurate over time on repeated task families, with no fine-tuning, no catastrophic forgetting, and full interpretability of what's been learned.

From a 2-minute brain download on why agents forget everything, to the non-parametric PPO algorithm that makes skill recall trainable without touching LLM weights, to a production decision about when procedural memory pays off vs the bookkeeping overhead.

2-minute brain download · why procedural memory matters
The problem
LLM agents are amnesic across tasks. Every "book a flight then email the team" is solved from scratch same web-search patterns, same form-filling heuristics, same API dance. Tokens, latency, and errors all scale with how often the same sub-procedure gets re-derived. Fine-tuning can embed the skill into weights but brings its own costs: compute, specialised infrastructure, and the well-documented risk of catastrophic forgetting on capabilities you didn't mean to touch.
The idea
Treat procedural knowledge as an external, retrievable store of skills. Distill successful trajectories into named procedures ("book flight on X site," "send structured email via Y tool"). At runtime, a non-parametric PPO policy decides which skill to fetch given the current task + state. LLM weights never change only the skill library and the recall policy grow.
Why it matters
On repeated task families in the paper's benchmarks, agents became cheaper and more reliable as the skill pool evolved. The policy is interpretable (you can inspect which skills fire for which queries). No fine-tuning is needed at the LLM level. The architecture is backbone-agnostic in principle the paper shows transfer across Gemma-2-9B, Qwen3-32B, and Llama-3.3-70B on its benchmarks. The design goal: agents whose skill pool compounds over time, so later runs can draw on what earlier runs figured out.
One-line intuition Humans don't re-invent "tying shoelaces" every morning the skill is compiled into procedural memory we recall automatically. ProcMEM gives agents the same faculty: a skill library outside the model's weights, plus a small policy that learns when to recall which skill, trained with a policy-gradient method that never touches the big LLM.

The compounding property

Episode 1 starts empty no skills, agent derives everything. Each successful episode adds skills to the library and tunes the recall policy via PPO. By episode 5, the library contains 4 high-reward skills + 1 deprecated one. Future tasks benefit immediately no retraining, no weight updates, just better routing into the growing skill library. This is what "compounding" means for agents.

Deep dive · four questions the paper makes you reconstruct
Why "non-parametric" PPO? What makes it different from standard PPO? ▸
The policy being updated is not the LLM's weights it's a small, external skill-recall policy. "Non-parametric" with respect to the LLM; parametric with respect to a tiny routing network.

Standard PPO on an LLM would update all its weights via policy gradient billions of parameters, expensive, causes catastrophic forgetting. ProcMEM's trick: the LLM is frozen. The only trainable component is a small network $\pi_\theta(\text{skill\_id} \mid \text{state})$ that takes the current task + agent state and outputs a distribution over available skills.

"Non-parametric" refers to the skill memory itself: the library of skills is a growing list of discrete entries, not compressed into weights. New skills are literally appended to the library; old skills are pruned based on observed reward statistics. The policy over this library is what PPO optimises. This hybrid (non-parametric skill store + parametric routing policy) gives you the compounding of learned behaviour without the fragility of weight-level updates.

∫ Show the math · Skill-MDP and non-parametric PPO objective▸

The paper (v2, renamed Skill-Pro) formalises the decision process as a Skill-MDP:

$$\mathcal{M}_\Omega = (\mathcal{S}, \mathcal{A}, \Omega, P, R, \gamma)$$

Where $\mathcal{S}$ is the semantic state space (natural language), $\mathcal{A}$ is the primitive action space, $\Omega = \{\omega^{(1)}, \dots, \omega^{(K)}\}$ is the dynamic skill pool (procedural memory), $P$ and $R$ are the transition and reward functions, and $\gamma \in (0, 1]$.

The agent picks a skill via a selection policy $\mu$, then the frozen LLM generates primitive actions conditioned on that skill:

$$\omega_t \sim \mu(\omega \mid s_t, \Omega) \qquad a_t \sim \pi_{\text{LLM}}(a \mid s_t, \omega_t)$$

The hierarchical policy factorises as:

$$\pi_\Omega(\omega_t, a_t \mid s_t) = \mu(\omega_t \mid s_t, \Omega) \cdot \pi_{\text{LLM}}(a_t \mid s_t, \omega_t)$$

What "non-parametric" really means. Both $\pi_{\text{LLM}}$ and $\mu$ are kept fixed. The only thing that changes is $\Omega$ itself the contents of the skill pool. Optimization of the expected discounted return $\max \mathbb{E}[\sum_t \gamma^t r_t]$ reduces to optimising the pool evolution operator $\mathcal{E}$, which synthesises, refines, and prunes skills given new trajectories:

$$\Omega_{\text{new}} = \mathcal{E}(\Omega_{\text{old}}, \mathcal{T}^{(B)})$$

This is the crucial distinction from standard PPO. Standard PPO updates parameters $\theta$ of a neural policy via gradient descent. Non-Parametric PPO updates the contents of an external symbolic library no gradients, no weight access, no catastrophic forgetting at the LLM level (though the skill pool itself can degrade, which is what score-based maintenance is for).

How is a "skill" actually represented? What goes into the library? ▸
A skill = (name, preconditions, step-by-step instructions/code, observed reward statistics, trigger context embedding). Essentially a typed, named, annotated procedure.

Each skill entry contains:

  • Name: short human-readable identifier (book_flight_on_X).
  • Trigger embedding: vector representation of the task context where this skill was useful used for nearest-neighbour retrieval.
  • Preconditions: conditions under which the skill is applicable ("user has travel dates and destination").
  • Body: the actual procedure either natural-language step instructions, executable code, or a structured action sequence.
  • Statistics: number of uses, average reward, last-used timestamp. Drives retention/deprecation.

This is similar to how Voyager's skill library works, but ProcMEM adds the learned retrieval policy on top. Voyager uses similarity-based retrieval with no learning; ProcMEM learns which skill to actually fire among semantically-similar candidates, based on what's historically worked.

∫ Show the math · the Skill triple activation, execution, termination▸

The paper formalises each skill as a three-tuple:

$$\omega = \langle \mathcal{I}_\omega, \pi_\omega, \beta_\omega \rangle$$

Where $\mathcal{I}_\omega$ is the activation condition, $\pi_\omega$ is the execution procedure, and $\beta_\omega$ is the termination condition. All three are natural-language strings.

(1) Activation condition $\mathcal{I}_\omega$. A description of when the skill is applicable. Used by $\mu$ to decide whether to fire this skill:

$$\omega_t = \arg\max_{\omega \in \Omega_t} \mathrm{Sim}(s_t, \mathcal{I}_\omega)$$

Where $\mathrm{Sim}$ is cosine similarity over embeddings or an LLM-as-judge. This is why the file's description says "trigger embedding" it's the vector form of $\mathcal{I}_\omega$.

(2) Execution procedure $\pi_\omega$. An ordered sequence of natural-language instructions. While active, the LLM generates actions conditioned on both state and the procedure:

$$a_t \sim \pi_{\text{LLM}}(a \mid s_t, \pi_\omega)$$

Note the overloaded notation $\pi_\omega$ here is a string containing instructions, not a probability distribution. The frozen LLM interprets it.

(3) Termination condition $\beta_\omega$. A predicate over states. When it evaluates true, the skill returns control and $\mu$ picks the next skill:

$$\beta_\omega(s_t) = 1 \iff s_t \text{ satisfies } \beta_\omega$$

Value-based selection variant. The paper also describes a top-$k$ + value variant: take the $k$ most similar skills, then pick the one with the highest estimated return:

$$\Omega_t^{(k)} = \mathrm{TopK}_{\omega \in \Omega_t} \mathrm{Sim}(s_t, \mathcal{I}_\omega) \qquad \omega_t = \arg\max_{\omega \in \Omega_t^{(k)}} Q(s_t, \omega)$$

Where $Q(s_t, \omega)$ is a running estimate of expected return from invoking $\omega$ at $s_t$. The "Statistics" field in the file's skill-entry schema (avg reward, num uses) is effectively the backing store for this $Q$.

What's the training signal? How does the policy get better? ▸
Task completion reward (binary or scored). PPO uses observed rewards to push up probability of skills that led to success, down for skills that led to failure.

The training loop per episode:

  • Agent receives task. Policy $\pi_\theta$ proposes skill to recall (or "derive fresh" as a null action).
  • Agent executes the skill. Observes outcome, receives reward $r$ (task success, efficiency, etc.).
  • PPO update on $\pi_\theta$: standard clipped-ratio objective, computed from $(state, skill, reward)$ triples. Advantage estimated by baseline (running average reward per skill).

Critically, the reward is end-task, not per-step. This avoids the paper's concern about brittle step-level supervision and aligns with how successful skills compound a skill that "looks right" mid-execution but fails the task gets down-weighted. A skill that "looks ugly" but consistently solves tasks gets up-weighted.

∫ Show the math · semantic gradients and the PPO Gate▸

Non-Parametric PPO has two components: a candidate generator using semantic gradients, and a verifier using PPO-style trust-region clipping. Together they refine existing skills without touching LLM weights.

Per-trajectory semantic gradient. For each trajectory $\tau_i$ where skill $\omega = \langle \mathcal{I}_\omega, \pi_\omega, \beta_\omega \rangle$ was invoked, hindsight attribution assigns blame/credit to each component of the skill:

$$g_i = \nabla_{\text{sem}}(\tau_i, \omega) = \big(g_i^{(\mathcal{I})},\, g_i^{(\pi)},\, g_i^{(\beta)}\big)$$

Each $g_i^{(\cdot)}$ is a natural-language refinement suggestion e.g. "the activation was too narrow, should also fire when …" or "step 3 of the procedure caused the failure". Unlike TextGrad (instantaneous response quality), semantic gradients are designed for sequential decision-making and are generated by an LLM that reasons over the full trajectory.

Batch aggregation. Individual gradients can be noisy or trajectory-specific. The algorithm aggregates across the batch to filter out inconsistent signals:

$$\bar{g}_\omega = \mathrm{Aggregate}(\{g_i\}_{i=1}^B)$$

Where $\mathrm{Aggregate}$ is an LLM consolidation that extracts recurring patterns. The candidate skill is then:

$$\omega' = \omega \oplus \bar{g}_\omega$$

Where $\oplus$ is an LLM-driven revision step. This is the analogue of a gradient ascent step except in natural language, on a discrete object, without any parameter updates.

PPO Gate trust-region verification. Semantic gradients can hallucinate. Before the candidate enters the pool, it must pass a trust-region check. For each step $t$ in the batch where behaviour-skill $\omega$ was active:

$$\rho_t(\omega') = \frac{\pi_{\text{LLM}}(a_t \mid s_t, \omega')}{\pi_{\text{LLM}}(a_t \mid s_t, \omega)}$$

This importance ratio measures how the likelihood of the historical action would change under the candidate skill. Advantages are estimated from returns-to-go with a running baseline:

$$G_t = \sum_{k=t}^{T-1} \gamma^{k-t} r_k \qquad \hat{A}_t = G_t - \bar{R}$$

The PPO-style clipped surrogate is then computed as the verification score:

$$L^{\text{CLIP}}(\omega') = \hat{\mathbb{E}}_{\tau \sim \mathcal{B}}\Big[\tfrac{1}{|\tau|}\sum_t \min\big(\rho_t(\omega')\hat{A}_t,\; \mathrm{clip}(\rho_t(\omega'), 1{-}\epsilon, 1{+}\epsilon)\hat{A}_t\big)\Big]$$

From $N_c$ candidates, the pool accepts the best positive-scoring one:

$$\omega_{\text{new}} = \arg\max_{\omega'} L^{\text{CLIP}}(\omega') \quad \text{subject to}\quad L^{\text{CLIP}}(\omega_{\text{new}}) > 0$$

Why this is the paper's key trick. Semantic gradients on their own (the w/o PPO Gate ablation) destabilise training hallucinated refinements get accepted, performance collapses. PPO Gate on its own (the w/o SG ablation) has nothing good to verify the 30% drop in PPO Gate pass rate proves that the semantic gradients are doing real work in candidate quality. The paper's ablation shows both are needed.

How does ProcMEM avoid skill library bloat? ▸
Three mechanisms: (1) similarity-based deduplication at insertion, (2) periodic deprecation of low-reward skills, (3) merging skills that activate on overlapping trigger contexts.

Without pruning, the library would grow linearly with episodes and become impossible to search effectively. The paper uses:

  • Insert-time dedup: before adding a new skill, check if its trigger embedding is within $\epsilon$ of an existing skill. If so, the new attempt is treated as another sample of the existing skill rather than a fresh entry.
  • Reward-based deprecation: skills whose average reward falls below threshold $r_{min}$ (after minimum sample size) are deprecated kept in the library with a "deprecated" flag but not retrieved. Prevents wholesale deletion of skills that might recover.
  • Skill merging: skills with high trigger-embedding overlap AND similar reward statistics get merged into a single more-general skill, with the original variants kept as sub-procedures.

These three mechanisms keep the library size approximately proportional to the task diversity encountered, not the raw episode count. Critical for long-lived deployments.

∫ Show the math · score-based maintenance and evolutionary pressure▸

The skill pool has fixed capacity $K$. Skills that enter via the PPO Gate still need to be pruned over time as newer, better skills are discovered. The paper defines an advantage-style gain $G(\omega; \tau)$ for each skill's contribution within a trajectory:

$$G(\omega; \tau) = \frac{1}{|\mathcal{T}_\omega(\tau)|} \sum_{t \in \mathcal{T}_\omega(\tau)} \tilde{r}_t$$

Where $\tilde{r}_t = r_t - \bar{r}$ is a centred reward (the running baseline $\bar{r}$ reduces variance), and $\mathcal{T}_\omega(\tau)$ is the set of time steps where $\omega$ was active. When only trajectory-level returns are available, the per-step advantage collapses to $\tilde{r}_t \equiv (R(\tau) - \bar{R})/|\tau|$.

Online score update. Each skill maintains a cumulative gain $G_b(\omega)$ and invocation count $N_b(\omega)$, updated after each batch:

$$G_{b+1} = G_b + \sum_\tau G(\omega; \tau) \qquad N_{b+1} = N_b + \sum_\tau c(\omega; \tau)$$

$$\text{Score}_{b+1}(\omega) = \frac{G_{b+1}(\omega)}{\max(1, N_{b+1}(\omega))}$$

Pruning rule. A skill is removed if $\text{Score}(\omega) \leq 0$ or if it is semantically redundant with another skill. If the pool still exceeds $K$, skills are pruned in ascending order of score.

Why this creates evolutionary pressure. $\bar{r}$ is a running baseline, not fixed. As the pool improves, $\bar{r}$ rises. A skill that scored positive against an early-stage baseline can flip negative against a later-stage baseline which triggers pruning. This is the paper's argument for why the pool doesn't stagnate: every skill is competing against the current mean performance, not against absolute thresholds.

The FIFO ablation is the clearest demonstration. The paper's w/o Score (FIFO) variant replaces score-based pruning with first-in-first-out. Its online score goes negative ($-0.0064$) meaning the pool actively accumulates below-average skills because high-performers can get evicted just for being older. Reuse rate collapses from 0.925 to 0.131. This is the single sharpest ablation in the paper, and it's evidence that what you prune matters more than what you generate.

§1The procedural-memory gap

The paper opens with a simple observation: LLM agents can store factual memory reasonably well (RAG over past conversations, key-value stores for user preferences) but have no standard mechanism for storing procedural memory the compiled "how-to" knowledge that humans accumulate implicitly through repetition.

Hand-coded skills
Brittle, don't generalise, require engineer time for every new task family. Doesn't scale.
Fine-tuning
Embeds skills in weights but causes catastrophic forgetting of other capabilities. Expensive, slow, hard to debug.
Prompt appending (AWM, Reflexion)
Stores summaries of past runs as prompt text. Grows unboundedly, no learning of when to retrieve what, no reward signal.
The key reframe

Procedural memory should be external + learned: an external library of discrete skills (non-parametric) combined with a learned retrieval policy (parametric, but small). This gets the best of both worlds interpretable persistent skills + adaptive recall that improves with experience.

§2Skill representation + trigger embedding

Every skill $s_i$ in the library $\mathcal{L}$ has five components: name, preconditions, body, statistics, and most importantly for retrieval a trigger embedding $e_i \in \mathbb{R}^d$.

Skill schema
$s_i = (\text{name}_i, \text{pre}_i, \text{body}_i, \text{stats}_i, e_i)$

The trigger embedding is computed from the task context that led to the skill's first successful execution. When a new task arrives with context embedding $e_q$, candidates are retrieved by cosine similarity $\cos(e_q, e_i)$.

The selection among candidates is where the learned policy comes in. Naive similarity retrieval returns the top-k nearest skills; $\pi_\theta$ then distributes probability over this candidate set based on task state and each skill's success statistics. This two-stage (retrieve → rank) design keeps inference cheap even as the library grows.

§3Non-parametric PPO algorithm

Here's the main algorithm. The policy network $\pi_\theta$ is typically a small MLP or small transformer (10-100M params), trained separately from the frozen base LLM.

Algorithm · PROCMEM-STEP(task t, state x, library 𝓛)
e_q ← encode(t, x) candidates ← top_k(𝓛, cos(e_q, ·)) # k usually 5-10
Retrieval. Semantic similarity over trigger embeddings picks a small candidate set. Includes a "null" action meaning "derive fresh, don't use any stored skill."
p ← π_θ(skill | e_q, candidates) s* ← sample or argmax p
Learned selection. The small policy distributes probability over candidates (+ null). Training uses PPO; inference uses argmax (exploit) or sample (explore).
if s* is null: trajectory ← LLM_agent(t, x) else: trajectory ← execute(s*, x)
Execution. If the policy chose a skill, execute it. Otherwise, fall back to the frozen LLM deriving from scratch. The LLM is never retrained it's only used for execution and (later) skill distillation.
r ← evaluate(trajectory, t) # episode reward update_stats(s*, r) # running mean for each skill
Reward + statistics. Whatever evaluation signal you have (task success, BLEU score, tool-call correctness). Update the skill's running reward statistics. This is the basis for later deprecation.
PPO_update(π_θ, (e_q, s*, r)) # clipped ratio objective # on the small policy only
Policy update. Standard PPO clipped-ratio objective. Computed only for $\pi_\theta$'s parameters the LLM's billions of parameters are untouched. Cheap; can run online.
if success(r) and s* is null: new_skill ← distill(trajectory, t) 𝓛 ← 𝓛 ∪ {new_skill}
Library growth. If the null action was chosen AND the LLM succeeded, distill the trajectory into a new skill and add to library. Uses the frozen LLM (no retraining) to write the skill's body. The library grows only when fresh derivations succeed.

Worked example: Agent solves "book a flight from SF to NYC for next Tuesday" for the first time, then again a week later.

1
Run 1: Library empty. Retrieval returns only the null action. Agent derives the procedure fresh via LLM searches sites, compares prices, books. Succeeds (reward = 1). Since null was chosen and succeeded, distill the trajectory into [s1] book_flight_standard. Add to library.
2
Run 2 (similar task): Query "book SJC to BOS for next Friday." Retrieval returns s1 (high cosine similarity) + null. Policy $\pi_\theta$ has no history yet → samples uniformly. Suppose it picks s1.
3
Execution: s1 executes in fewer tokens than deriving fresh (the skill body is already compact and tool-ready). Task succeeds. PPO Gate update: the refined candidate passes trust-region verification, the online score rises, and the skill stays in the pool.
4
Run 20 (after many episodes, illustrative): Query probability mass concentrates on s1 for flight tasks. The library has grown to include send_email_structured, scrape_product_table, etc. The expected shape: average episode cost drops, success rate rises. The exact gains here are illustrative for this worked example the paper's actual reported numbers are on ALFWorld and Mastermind-v0, summarised in the Experiments section below.
§4Experiments compounding gains over episodes

The paper evaluates on two multi-turn decision-making benchmarks: ALFWorld (for in-domain train + out-of-distribution splits) and Mastermind-v0 from TextArena (with three difficulty tiers: v0, Hard, Extreme, for cross-task evaluation). Cross-agent transfer is tested on Gemma-3-4B, Qwen3-32B, and Llama-3.3-70B. Results are averaged over 50 episodes per setting. Key reported numbers:

Memory reuse rate (paper Table 1)
On Mastermind-v0 in-domain, Skill-Pro reaches 0.925 vs the strongest baseline (AWM) at 0.080. Cross-task and cross-agent reuse stay above 0.80. These numbers are specific to the paper's benchmarks the gap may narrow on domains where existing memory methods were already well-tuned.
Storage cost (paper Table 1)
Full pool is 816 total tokens on the paper's evaluation vs hundreds of thousands for trajectory-RAG baselines the paper frames this as "extreme memory compression." The absolute number is specific to Mastermind-style tasks with short contexts; expect larger pools for richer domains.
Task success (paper Table 2)
On ALFWorld Train, 0.90 success vs AWM 0.70, ReAct 0.58, CoT 0.60 on the same frozen Qwen3-32B backbone. On OOD, Skill-Pro at 0.909 vs AWM 0.900 the OOD gap is smaller, suggesting the strongest gains come from repeated-task families rather than truly novel distributions.
Ablations (paper Table 3)
The sharpest ablation is w/o Score (FIFO), which drops reuse rate to 0.131 and produces a negative online score (−0.0064) pruning which skills to keep matters more than either semantic gradients or the PPO Gate alone. w/o Skill (no procedural memory at all) scores 0.388.
What to watch for

The paper reports these results on clean automatable reward signals (ALFWorld success, Mastermind return) and on a specific family of reasoning-heavy text tasks. Domains where the gains may be more modest: tasks where the "skill" doesn't cleanly abstract (creative writing, open-ended dialogue), tasks with noisy reward signals, and tasks where existing memory methods were already specifically tuned. Don't assume the 10× reuse or 100× compression numbers transfer unchanged treat them as upper bounds from favourable conditions until you've measured on your own workload.

§5Limitations
Cold-start penalty
No benefit until the library has useful entries and the policy has trained signal. Early deployments look worse than stateless agents.
Distill quality depends on LLM
When skills are written by the LLM, their quality is bounded by the LLM. A weak base model produces weak skills that propagate weakness into future runs.
Reward engineering
PPO needs a clean reward signal. Tasks where success is ambiguous (creative writing, open-ended dialogue) don't give good training signal for recall policy.
Distribution shift
If the task distribution drifts (new tools, new UI layouts, new user needs), old skills become stale. Deprecation handles some of this but can lag.
Skill interdependence not modelled
Each skill is treated as an independent unit. Skills that should chain (login → navigate → submit) aren't represented as a graph. This is left to future work.

Think like a researcher · where ProcMEM breaks and what comes next
Pitfalls when applying the framework · four mistakes that eat weeks
Pitfall 1 · Cold-start pollution
Early episodes lock in bad skills before the baseline stabilises
symptom: first 20 episodes produce weird skills that never get pruned
The score-based pruner compares skills against a running baseline $\bar{r}$. During the first few batches, $\bar{r}$ is unstable and easily dragged up by lucky early skills which then set a low bar and survive pruning long after they should have been replaced. The paper doesn't emphasize this; the empirical curves look stable because their benchmarks aren't that noisy. Fix: don't enable pruning until you have enough trajectories for a reliable baseline (paper uses batch-level aggregation for this reason). Seed the pool with a few hand-written skills if possible, so early cold-start skills have real competition from step one.
Pitfall 2 · Reward-hacking skills
A skill that games the evaluator becomes highly recalled and spreads
symptom: ALFWorld success rate jumps, but solutions get weirder each batch
Since skills with positive online scores survive pruning, a skill that happens to exploit a quirk in your reward signal will stick in the pool and get refined into progressively more exploitative variants via semantic gradients. The paper's Mastermind and ALFWorld benchmarks have clean reward functions that are hard to hack. Production reward signals are often not as clean LLM-as-judge, noisy click-through, user rating. Fix: hold out a validation set with a different reward signal (e.g. human review) and periodically sanity-check top-scoring skills. If a skill scores high on online reward but low on held-out, it's probably hacking.
Pitfall 3 · Agent lock-in
Skills learned on one backbone subtly fail on another
symptom: skills trained on Gemma-2-9B drop 10 points when moved to Qwen3-32B
The paper's cross-agent results are strong learning on Gemma-2-9B and reusing on Gemma-3-4B, Qwen3-32B, Llama-3.3-70B all transfer well. But the transfer tested within the same family of tasks (Mastermind-v0 across backbones). For a new task class on a new backbone, skills co-adapted to the original LLM's quirks may underperform. Fix: treat cross-agent transfer as an empirical question per deployment, not a property to assume. When you migrate backbones, re-run a batch of trajectories on the new model and let Non-Parametric PPO refine the pool against the new LLM's priors.
Pitfall 4 · Library-as-oracle
Treating the skill library as deterministic when it's a learned artefact
symptom: downstream tests hardcode expected skill names like "StratPlan"
Skills are named by the LLM aggregator, and the names can drift as semantic gradients refine them. If your test harness asserts specific skill names are present or a production system makes routing decisions based on skill identity you're building on top of an artefact that's allowed to evolve. Fix: route on skill function, not skill name. Use the activation-condition embedding for external lookups, and keep the skill names as human-readable labels for debugging only. Treat the library as a learned index, not a stable API.
Accordions · open questions the paper leaves on the table
What if the skill library gets poisoned by a bad episode?▸
Biggest safety risk. A high-reward skill that's wrong (reward hacking, gaming the eval) becomes highly recalled and damages future runs. Deprecation lags.

Imagine a skill get_answer_fast that happens to score high on a faulty evaluator but actually produces wrong answers. PPO will push $\pi_\theta$ to recall it frequently. Deprecation (low average reward) only kicks in after many low-reward samples but by then, the skill has caused real damage.

Production mitigation: (1) human review of newly-distilled skills before library insertion, (2) two-phase evaluation a shadow mode where the skill is recalled but not executed, used only for evaluator disagreement detection, (3) anomaly detection on reward distributions to flag suspicious skills. None of these are in the paper; all are necessary in deployment.

Three attack points where ProcMEM can be strengthened▸
Skill composition graphs, meta-policy that transfers across task families, adversarial skill validation.

1. Skill graphs. Most real tasks chain skills (search → filter → extract → format). Treating skills as independent loses this structure. A graph-structured library where edges encode "typically follows" relationships would let the policy plan at the skill level rather than pick-one-and-execute.

2. Meta-policy transfer. A recall policy trained on one user's task distribution should warm-start a new user's agent. This requires meta-learning training $\pi_\theta$ to adapt quickly rather than just maximise reward on one distribution. MAML-style meta-PPO would be the natural fit.

3. Adversarial skill validation. Before adding a distilled skill to the library, have an adversary LLM try to construct inputs where the skill fails. If it finds them, the skill is marked "conditional" or rejected. Analogous to property-based testing for software. Cheap, catches reward-hacking skills early.

How would you use ProcMEM in a production agent system?▸
Great for stable task families with clear rewards (customer support, data entry, repetitive coding). Wrong for creative/open-ended tasks or first-week deployments.

Good fits:

  • Customer support agents handling common request patterns (refunds, account issues, troubleshooting).
  • Code-editing agents on a specific codebase the codebase conventions become recalled skills.
  • Data-entry / scraping agents on a fixed set of sites UI patterns become skills.

Bad fits:

  • Creative writing or ideation (no clean reward).
  • Early-stage deployments without a library seed.
  • Highly volatile environments (sites change daily, skills obsolete too fast).

Why this paper might fail in practice
Failure: reward-hacked skills persist
Real case: an internal data-extraction agent using ProcMEM learned a skill quick_extract that scored high on the auto-evaluator (BLEU against expected output) but was actually returning templated text with some fields correct. Because BLEU rewarded the template structure, the skill's avg reward was 0.78 above deprecation threshold. Took 3 weeks to notice via user complaints. Root cause: the evaluator signal was subtly misaligned with real correctness. Fix: ensemble evaluators, human spot-checks of high-use skills, alerts when skill probability mass concentrates suspiciously fast.
Failure: cold-start UX cliff
Real case: a coding-assistant deployment with ProcMEM. First two weeks felt slower than vanilla agent because library was empty AND recall policy was exploring. Users submitted more complaints in week 1 than with the old system. Churn spiked. By week 4, the system was clearly better but many users had already disengaged. Root cause: ProcMEM's compounding benefit vs cold-start cost wasn't communicated, and no seed library. Fix: pre-populate with skills distilled from historical traces, OR run ProcMEM in shadow mode until library is warm, then flip to active.

Deployment thought experiment · speculative translation to real products
Who would actually build procedural memory on this architecture? Our best guesses.
Anthropic / Claude Agent Skills
Closest shipped analog · manually-authored
Already shipping a simpler version. The paper explicitly cites Claude Agent Skills as the manually-authored analog of what it automates. Skills today are hand-written Markdown with activation conditions and procedures exactly the $\langle \mathcal{I}_\omega, \pi_\omega, \beta_\omega \rangle$ schema. The obvious next step is letting Claude propose skill refinements from conversation logs. Whether Anthropic ships that specifically is unknown, but the architectural path from "manual skills" to "learned skills" runs through something very close to ProcMEM's mechanism.
OpenAI Custom GPTs + Actions
GPT Store architecture · manually-authored tool + instruction bundles
Similar architectural shape, different optimisation story. Custom GPTs bundle an instruction prompt with a fixed set of actions that's basically a single-skill library with no learning. The natural extension is letting a GPT accumulate new "skills" from user interactions and prune underused ones. OpenAI hasn't shipped this publicly at API level. It would fit.
Cursor / Replit / Windsurf
Coding-agent startups · rich per-user signal
Strong fit for codebase-specific procedural memory. Coding agents have exactly the setup ProcMEM assumes: frozen base model (GPT-4/Claude via API), clean reward signal (did the build pass, did tests pass, did the user accept the diff), lots of recurring sub-procedures ("add a test for this function", "refactor this class"). Learning a per-user or per-codebase skill library would compound as logs accumulate and cost less than continual fine-tuning. The barrier is mostly engineering discipline, not research novelty.
LangGraph / CrewAI / AutoGen
Agent frameworks · provide the runtime, not the learning
A natural add-on module. These frameworks already have the scaffolding for multi-step agents with tools the thing ProcMEM adds is learning which procedures to reuse. Ecosystem-wise, the natural home is as a library that plugs into any of these frameworks (a "skill-store" component that sits alongside the memory and tool modules). Whether the Skill-Pro codebase at Miracle1207/Skill-Pro becomes that library depends on how the maintainers evolve it.
Enterprise automation platforms
Salesforce, ServiceNow, ZoomInfo workflow-heavy agents
Fit is real but governance adds friction. Enterprise customers care about auditability a learned skill library that mutates based on user interactions is a compliance conversation. The right architecture might be: learn skills in staging, snapshot the pool, promote to production with explicit versioning, roll back if any skill misbehaves. Doable, but the deployment pattern isn't plug-and-play.
Solo builders / small teams
Prototype-stage agents on commodity APIs
Probably premature. At small scale, you don't have enough trajectories to let the pool evolve meaningfully the paper uses 50 episodes per setting just for evaluation, and learning needs orders of magnitude more. A better move for solo builders is to write a handful of hand-authored skills (in Anthropic's Agent Skills format or equivalent), measure whether they help, and graduate to learned skills only once you have real user volume and a real reward signal.
🎯

The picking decision in one line

You need ProcMEM when you have (1) a frozen base model you can't retrain, (2) a clean automatable reward signal, and (3) enough logged trajectories for the pool evolution to matter. If any of those three is missing, the architecture is overkill use a simpler memory design until the constraints bite.

Build it · five steps from blank slate to deployed procedural memory
1
Start with a seeded skill library
Do NOT deploy with an empty library. Mine the last 1-3 months of historical traces for successful task patterns. Distill the top 20-30 into skills manually (or via the LLM). Seed the library. Cold-start pain is avoidable skip it.
deploying with empty library users will notice the regression
2
Wire up a CLEAN reward signal before anything else
PPO is only as good as its reward. Before deploying, commit to one automated evaluator: task-completion check, user thumbs-up/down, A/B comparison, whatever but make it robust and hard to game. Reward-hacked skills are the main production failure mode. A fragile evaluator poisons the library.
3
Keep the policy network small
The recall policy should be a small MLP or tiny transformer (10-100M params, runs on CPU). Don't build a massive routing model you lose the "small, fast, interpretable" benefit. If you need more capacity, it's a sign you should be using more skill-embedding features, not a bigger policy.
4
Ship the policy in shadow mode first
Deploy ProcMEM in read-only mode: recall skills, log what would have been executed, but fall back to the non-memory baseline for actual execution. Run for 2 weeks. Inspect the skills being chosen. Fix issues. Then flip to active mode. Catches a class of bugs you'd otherwise find in production.
5
Build an observability dashboard for the library
Show: total skills, top 10 by usage, skills by trailing-week reward, newly deprecated, newly added. This is what lets an ops team diagnose "why did accuracy dip last Tuesday?" often because a new skill was distilled that shouldn't have been. The library is now part of your production surface.
Research idea
Meta-ProcMEM: train the recall policy to transfer across users. Take a user's first 20 tasks, use MAML to adapt the policy to their task distribution. Goal: reduce per-user cold start from weeks to hours.
Startup idea
Skill marketplace: companies can publish curated verified skill libraries for specific domains (Salesforce workflows, AWS ops, Shopify admin). Agents subscribe to skill libraries like npm packages. Monetisation: skill royalties.
1-line improvement
Add a "skill chaining bonus" to the reward: reward goes up when a skill's output is used as input to another skill within the same episode. Encourages the policy to learn skill composition implicitly. Expected: +3-5% on long-horizon benchmarks.

The paper's skill-learning backbone. Any frozen open-weights LLM with a chat template works; the skill pool is backbone-agnostic at the architectural level. Paper also evaluates on Qwen3-32B and Llama-3.3-70B for cross-agent reuse.
Embeddings

Used to embed activation conditions $\mathcal{I}_\omega$ and query states $s_t$ for similarity-based skill selection. Any small sentence encoder works the learning happens at the skill-pool level, not in embedding space.
Benchmarks
ALFWorld + TextArena
Paper uses ALFWorld (Shridhar et al. 2021) for train/OOD splits and Mastermind-v0 from TextArena (Guertler et al. 2025) for cross-task difficulty tiers. Both are text-based, which fits the Skill-MDP's natural-language state space.

Official repo with the full Non-Parametric PPO implementation, skill-pool pruning, and experiment scripts. MIT-licensed per the Zenodo citation. Start by reading the skill pool class, then the PPO Gate verification, then the run loop.
Reward signal
Binary success + centered
For ALFWorld and Mastermind, reward is a per-episode binary (did the agent succeed). Advantage estimation subtracts a running baseline $\bar{r}$ to produce centered $\tilde{r}_t$. For open-ended tasks you'll need LLM-as-judge scoring instead.
Storage
In-memory list, no DB
Skill pool is a Python list/dict the paper reports 816 total tokens for the full pool on ALFWorld, so no vector DB is needed. For production at scale, switch to a proper store keyed by activation-condition embeddings.
Core loop · one full iteration of Non-Parametric PPO

# Minimal structural sketch of the Skill-MDP loop.
# This mirrors the paper's Algorithm 1 without reproducing it verbatim.
# See github.com/Miracle1207/Skill-Pro for the full implementation.

from dataclasses import dataclass, field
from typing import Callable

@dataclass
class Skill:
    name: str
    activation: str     # I_omega natural language trigger
    procedure: str      # pi_omega natural language steps
    termination: str    # beta_omega natural language stop cond
    cum_gain: float = 0.0    # G_b(omega)
    invocations: int = 0    # N_b(omega)

    @property
    def score(self) -> float:
        return self.cum_gain / max(1, self.invocations)


class SkillPool:
    def __init__(self, capacity: int = 32, seeds: list[Skill] | None = None):
        self.capacity = capacity
        self.skills: list[Skill] = seeds or []

    def select(self, state: str, sim_fn: Callable, top_k: int = 3) -> Skill:
        # Similarity-based top-k then value-based argmax.
        scored = sorted(
            ((sim_fn(state, s.activation), s) for s in self.skills),
            key=lambda x: x[0], reverse=True,
        )[:top_k]
        return max(scored, key=lambda x: x[1].score)[1]

    def prune(self) -> None:
        # Drop non-positive-score skills; then trim to capacity by ascending score.
        self.skills = [s for s in self.skills if s.score > 0]
        self.skills.sort(key=lambda s: s.score, reverse=True)
        self.skills = self.skills[:self.capacity]


def run_episode(task: str, pool: SkillPool, llm, sim_fn, verifier):
    # Rollout: select a skill per active segment, have the LLM act under it.
    trajectory = []
    state = initial_state(task)
    while not terminal(state):
        skill = pool.select(state, sim_fn)
        while not beta_satisfied(state, skill.termination):
            action = llm.act(state, skill.procedure)
            reward, state = env_step(state, action)
            trajectory.append((state, skill, action, reward))
    return trajectory
Non-Parametric PPO update · where Skill-Pro differs from stock RL

import math
from typing import Iterable

def semantic_gradient(traj, skill: Skill, llm) -> dict[str, str]:
    # Hindsight attribution ask the LLM what, in this trajectory, should be
    # refined in the activation / procedure / termination of the active skill.
    prompt = build_attribution_prompt(traj, skill)
    resp = llm.generate(prompt)   # structured: {I: "...", pi: "...", beta: "..."}
    return resp


def aggregate_gradients(grads: list[dict], llm) -> dict[str, str]:
    # LLM-based consolidation surfaces recurring patterns, drops outliers.
    return llm.generate(build_consolidation_prompt(grads))


def apply_gradient(skill: Skill, grad: dict, llm) -> Skill:
    # Skill revision step operator `oplus` from the paper.
    new_text = llm.generate(build_revision_prompt(skill, grad))
    return Skill(
        name=skill.name,
        activation=new_text["I"],
        procedure=new_text["pi"],
        termination=new_text["beta"],
    )


def ppo_gate_score(candidate: Skill, behavior: Skill, batch, llm,
                    epsilon: float = 0.2, baseline: float = 0.0) -> float:
    # PPO-style clipped surrogate evaluated on historical trajectories.
    # rho_t is the importance ratio under the candidate vs behavior skill.
    total, n = 0.0, 0
    for traj in batch:
        returns_to_go = compute_returns(traj)
        for t, (s, _, a, _) in enumerate(traj):
            adv = returns_to_go[t] - baseline
            rho = llm.logp(a, s, candidate) / llm.logp(a, s, behavior)
            rho = math.exp(rho)  # if logp returns log-probs, convert to ratio
            clipped = max(min(rho, 1+epsilon), 1-epsilon)
            total += min(rho * adv, clipped * adv)
            n += 1
    return total / max(1, n)


def evolve_pool(pool: SkillPool, batch, llm, n_candidates: int = 4):
    # One full evolution step over a trajectory batch.
    for skill in pool.skills:
        active_trajs = [t for t in batch if used(skill, t)]
        if not active_trajs: continue

        # 1) Generate Nc candidates via semantic gradients
        grads = [semantic_gradient(t, skill, llm) for t in active_trajs]
        agg = aggregate_gradients(grads, llm)
        candidates = [apply_gradient(skill, agg, llm) for _ in range(n_candidates)]

        # 2) PPO Gate keep only candidates with positive clipped surrogate
        scored = [(c, ppo_gate_score(c, skill, active_trajs, llm)) for c in candidates]
        scored = [(c, s) for c, s in scored if s > 0]
        if scored:
            best, _ = max(scored, key=lambda x: x[1])
            pool.skills[pool.skills.index(skill)] = best  # in-place refine

    # 3) Score-based maintenance
    for s in pool.skills:
        s.cum_gain += sum(gain(s, t) for t in batch)
        s.invocations += sum(count(s, t) for t in batch)
    pool.prune()
    return pool

Reading order if you're going to clone the repo

Start at the skill-pool class (the data structure is the paper's core contribution). Then read the score-based pruning that's the 96%+ of the value per the paper's FIFO ablation. Then read the PPO Gate verification. Skip the semantic-gradient prompt engineering on first pass; those prompts are tuned to the paper's specific benchmarks and will need to be rewritten for your domain anyway.

Common gotcha
The PPO Gate computes $\pi_{\text{LLM}}(a \mid s, \omega')$ i.e., you need log-probabilities of past actions under a hypothetical skill. For hosted APIs (OpenAI, Anthropic) this is often not available or extremely expensive. The paper uses open-weights models where you can get logprobs directly. If you're locked into a closed API, you'll need an approximation e.g., rejection sampling with an LLM-judge proxy for the importance ratio.
Research idea
Replace the PPO Gate's importance ratio with an LLM-judge over the agreement between the candidate skill's prescribed action and the historical action. Might recover most of the trust-region benefit without needing logprobs. Would unlock this family of methods for closed-API setups where logprobs are unavailable.
Production add-on
Snapshot the skill pool after each evolution batch. Promote a snapshot to production only after holdout evaluation clears a bar. Keep the last $n$ snapshots for rollback. This isn't in the paper their evaluation is offline but it's the deployment pattern any serious user of this framework will need to build.

Question 1 of 8 · What's "non-parametric" about it

ProcMEM calls itself "non-parametric PPO." What specifically is non-parametric?

Memory substrate vs policy distinction: "non-parametric memory" is a term from ML that means "not parameterised into weights" the knowledge lives in a data structure, not a model. ProcMEM's library is this; the routing policy over it is parametric.
Question 2 of 8 · Why not fine-tune the LLM

Why does ProcMEM freeze the LLM weights instead of fine-tuning on successful trajectories?

Practical ML: weight-level RL on huge models is a research problem, not a production tool. External memory + small routing policy gets most of the benefit with none of the cost.
Question 3 of 8 · Skill schema

What does a single skill entry in the library contain?

Structured skill representation is what separates ProcMEM from earlier workflow-memory approaches like AWM or Synapse that stored skills as opaque NL summaries.
Question 4 of 8 · Library growth control

Your ProcMEM deployment has 1,000 skills after 6 months. Searches are slow and you notice many near-duplicate entries. What's wrong?

Three mechanisms keep the library bounded: dedup at insertion, reward-based deprecation, skill merging. Monitor library size vs episode count sub-linear growth means it's healthy; linear or super-linear means one of the mechanisms is broken.
Question 5 of 8 · Cold start

You deploy ProcMEM with an empty library to an internal tool. Users complain for two weeks about slower / worse responses than the old system. What happened?

Know the cold-start cliff. ProcMEM is a compounding system the benefit per episode grows over time but starts negative. This makes pre-seeding and shadow deployments critical for production UX.
Question 6 of 8 · Reward hacking

What's the biggest safety risk with ProcMEM in production?

Reward hacking in procedural memory is especially dangerous because it compounds: once a reward-hacked skill enters the library, the policy will amplify its use, each use generates more "high reward" observations, and the skill becomes harder to dislodge. Guard the evaluator.
Question 7 of 8 · Transfer property

The paper notes that skills built with a stronger LLM transfer well to a weaker LLM. Why?

The externalisation benefit: putting knowledge outside the model makes it portable. This is the ops-friendly story for procedural memory: train once with your best model, deploy with your cheapest model. Compare: fine-tuning doesn't transfer across model families at all.
Question 8 of 8 · When NOT to use ProcMEM

For which workload would ProcMEM be the WRONG choice?

The procedural-memory prerequisite: repetition + measurable success. When tasks are genuinely unique (creative work) or success is unmeasurable (subjective judgement), there's nothing to compound. A, B, and D all have strong repetition + measurable reward.
